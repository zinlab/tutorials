01Why radix sort fits a GPU

Comparison sorts are bound from below by Ω(n log n) comparisons, and on a GPU they pay an extra penalty: the partition and merge steps branch on data, so warps diverge and memory access turns irregular. Radix sort avoids the comparison model entirely. It is a distribution sort — it inspects keys digit by digit in a chosen base and runs in O(k·n), where k is the number of digit passes. For a fixed key width that is linear in n.

What makes it a natural GPU citizen is that every phase reduces to a primitive the hardware is exceptional at: counting a histogram is embarrassingly parallel — thousands of threads tally into shared-memory bins. The prefix sum that turns counts into output offsets is the canonical scan, the most studied GPU primitive there is. And the final scatter is a wide stream of global writes. Crucially, control flow never depends on a key's value — only on its current digit, which every thread handles uniformly, so warps stay coherent.

The regime matters. Radix sort is memory-bandwidth bound, not compute bound — exactly where GPUs hold their largest lead over CPUs. Production implementations (NVIDIA's cub::DeviceRadixSort and the newer onesweep algorithm) run at a large fraction of peak DRAM bandwidth, and thrust::sort on integer keys dispatches straight to them.

02The algorithm

Pick a radix — the base in which keys are read. With a 4-bit digit the radix is 16, so a 32-bit key is an 8-digit number in base 16. Radix sort processes those digits one at a time.

LSD: least-significant digit first

We start at the lowest digit and work up to the highest. Each pass performs a stable counting sort on the current digit only. The trick is that stability lets earlier (lower) digits survive: once the most-significant digit is placed, the array is fully sorted, because within any group of equal high digits the keys are still in the order the previous passes left them. (The mirror image, MSD, recurses high-to-low into buckets and suits variable-length keys and strings, but LSD's flat, branch-free structure is friendlier to GPU bandwidth, so it dominates for fixed-width integers.)

One pass is a counting sort

A single digit pass is three moves — precisely the three steps the demonstration animates. Histogram: count how many keys carry each digit value (0–F), giving 16 bucket counts. Prefix sum: take an exclusive scan of those counts; the result for digit d is the number of keys with a strictly smaller digit, which is exactly the index where bucket d begins in the output. Scatter: walk the keys in order and drop each into output[offset[digit]++] — incrementing the offset as you go keeps equal-digit keys in their original order, and that is the stability.

Done serially that's trivial. The interesting work is doing all three in parallel across a GPU without losing stability, which is the next section.

03The three GPU phases

The serial counting sort has a sequential dependency — that offset[digit]++ threads through every key. To parallelize, the array is cut into tiles, one per thread block, and the three moves become three kernels.

P1

Histogram — per-block counting

Each block reads its tile with coalesced, striped loads (thread t touches keys t, t+blockDim, … so consecutive threads hit consecutive addresses) and tallies digits into a 16-entry shared-memory histogram via atomicAdd. It then writes that histogram to a global table laid out as blockHist[digit·numBlocks + block] — digit-major on purpose.

P2

Scan — one device-wide exclusive prefix sum

A single exclusive scan runs over that whole flat table. Because the layout is digit-major then block-minor, the scanned value at (d, b) equals (keys with a smaller digit, across the entire array) plus (keys with digit d in earlier blocks) — i.e. the precise global slot where block b's run of digit-d keys must begin. One scan computes every block's every-digit offset at once. Production code uses cub::DeviceScan; the source here calls thrust::exclusive_scan, which sits on the same machinery.

P3

Scatter — stable global placement

Each block re-reads its tile with a blocked layout (thread t owns a contiguous run), so a thread sees its own keys in original order. Per-thread sub-histograms plus a block-wide column scan give every key its within-block rank among equal-digit keys. The final address is globalOffset[d,b] + rank — and because thread order and in-thread order are both preserved, equal keys never overtake one another. That is the stability LSD demands.

Coalescing footnote. Blocked loads in the scatter aren't coalesced, and naive global writes are scattered. The fastest sorts (CUB) first sort keys locally within the tile so each digit's writes are contiguous, and use a shared-memory transpose to keep loads coalesced. The source here keeps blocked global loads for legibility — the algorithm is identical, only the memory choreography differs.

04Stability is not optional

A sort is stable if equal elements keep their input order. For LSD radix sort this isn't a nicety — it is the correctness argument. Each pass sorts on one digit and trusts the previous passes to have ordered everything less significant. If a pass reorders two keys that share its current digit, it silently scrambles the lower-digit ordering those keys had already earned.

Concretely, take two keys written as two hex digits, high–low: A = 0x14 (decimal 20) and B = 0x12 (decimal 18). After the low-nibble pass, B precedes A because its low nibble (2) is smaller than A's (4) — correct, since 18 < 20. The high-nibble pass now compares high nibbles, and both are 1: a tie. A stable pass leaves B before A, giving 18, 20. An unstable pass might swap them to 20, 18 — undoing the low-nibble work and producing a wrong answer. Stability is the thread that stitches the passes together.

This is why the scatter kernel goes to such lengths — per-thread histograms, a block-wide column scan, contiguous within-thread runs — purely to guarantee that two keys with the same digit land in the output in the same order they arrived.

06Performance & tuning

The radix-bits knob

Choosing r bits per digit sets two opposing quantities: there are 2^r buckets and ⌈32/r⌉ passes. More bits mean fewer passes — and since each pass reads and writes the whole array, fewer passes means less DRAM traffic — but also a larger histogram, a larger scan, more shared memory, and a wider scatter. The sweet spots in practice are 4 and 8 bits.

Why it is bandwidth bound

Each pass moves every key twice — one read, one write — so total DRAM traffic is about 2 · passes · 4 · n bytes for 32-bit keys (key-value sorting doubles it). With 8 passes that's ~64n bytes streamed. The histogram and scan touch only 16 × numBlocks elements — negligible beside the key array — so wall-clock time tracks how efficiently those key reads and writes hit peak bandwidth. The whole optimization game is (1) cut passes via a larger radix, and (2) keep every key access coalesced.

One sweep instead of three kernels

The version here launches three kernels per pass and bounces through global memory between them. The current state of the art, onesweep (in CUB), fuses the global digit-offset computation into a single pass using decoupled look-back — a chained scan where each block publishes a partial result and peeks backward at its predecessors instead of waiting for a separate global scan. That collapses a pass to roughly one read and one write of the keys, close to the hardware floor.

In production, call CUB. cub::DeviceRadixSort::SortKeys (and SortPairs) handles key-value pairs, ascending or descending order, and sorting only a bit subrange via begin_bit/end_bit — and it is retuned per architecture. The kernels here are for understanding what CUB is doing, not for beating it.

07Variants & extensions

The 1-bit split (Blelloch's formulation)

Strip the radix down to a single bit and each pass becomes a stable split: keys with a 0 bit move to the front, keys with a 1 bit to the back, order preserved within each group. The whole thing is driven by one exclusive scan. It is the multi-bit algorithm in miniature — every bucket count is just “how many zeros so far” — and it is the clearest way to see why a scan is the heart of radix sort.

Signed integers

An unsigned radix sort orders negative two's-complement integers after positives, because their sign bit is 1. Fix it by flipping the sign bit before sorting and again after: key ^ 0x80000000. Now the bit patterns are monotonic in signed value.

IEEE-754 floats

Floats need a slightly richer transform because negative floats have descending bit patterns. Map each bit pattern to an order-preserving unsigned key: if the sign bit is set, flip all 32 bits; otherwise flip only the sign bit. Sort, then invert the map. This handles ±0 and orders everything correctly (NaNs sort to the ends).

Key–value pairs

To sort records, carry a parallel values array and scatter each value to the same destination its key computes — valuesOut[dest] = valuesIn[i] inside the scatter kernel. Everything else is unchanged; you simply move twice the bytes. For large payloads, sort an index array instead and gather once at the end.

Descending order

Either reverse the digit-to-offset mapping (assign bucket 15−d's region to digit d) or sort ascending and reverse the result. CUB exposes this directly as SortKeysDescending.

MSD for strings and variable-length keys

When keys vary in length or you want early termination, switch to most-significant-digit first: partition into buckets by the top digit, then recurse into each bucket independently. Buckets become independent subproblems — natural parallelism — and once a bucket holds one element it needs no further work, which LSD can't exploit.
