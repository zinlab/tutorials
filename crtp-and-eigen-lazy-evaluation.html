Static polymorphism · zero-overhead abstraction

The Curiously Recurring
Template Pattern & how Eigen
spends nothing at runtime

A base class parameterized on its own derived class. From that one twist falls compile-time dispatch, mixins, and the expression templates that let a + b + c run as a single fused loop with no temporaries. This is the whole tour — mechanics, the Eigen engine, the gotchas, and what replaces it in modern C++.

01 Orientation

CRTP is one line of structure with outsized consequences:

The base class takes the derived class as a template parameter, and the derived class passes itself as that parameter. The recursion looks paradoxical but isn't: when the compiler sees Widget, the name Widget already exists (it is being declared), so Base<Widget> is a perfectly legal base. Because Base statically knows the exact derived type, it can downcast and call derived members at compile time — no virtual table, no indirection, fully inlinable.

◆ The one idea

Inheritance normally lets a derived class reach up to the base. CRTP additionally lets the base reach down into a specific derived class — known at compile time. That inversion is the entire engine: it turns runtime polymorphism into compile-time polymorphism, and it is how a library can hand you an operator like + that builds a description of work instead of doing the work.

What it buys you, in one breath: static (vtable-free) polymorphism, reusable behavior mixed into many types, method chaining that returns the right derived type, and — the headline act — expression templates, the technique behind Eigen's lazy evaluation. We build to that.

02 The pattern & its mechanics

Why the downcast is safe

Inside Base<Derived>, this has type Base<Derived>*. The static_cast<Derived*>(this) is valid provided the actual object really is a Derived — which the pattern guarantees, because Derived publicly inherits Base<Derived> and is the most-derived type. There is no runtime check and none is needed; the type relationship is fixed at compile time.

You'll see this downcast factored into a tiny helper so the cast appears once:

▲ Protect the base destructor

There is no virtual dispatch here, so deleting a Derived through a Base<Derived>* would be undefined behavior. Make the base destructor protected and non-virtual (protected: ~Base() = default;). This forbids polymorphic deletion through the base entirely — exactly what you want, since you should never be holding a Base* as a runtime handle in the first place.

The "static interface" contract

The base defines the public surface (area(), describe()) and delegates the variable part to a named hook (area_impl()) that every derived type must supply. If a derived class forgets the hook, you get a compile error — but only at the point the templated member is instantiated (i.e. actually called), which is worth remembering when reading errors.

03 Static vs dynamic dispatch

The closest cousin to CRTP is the classic virtual-function interface. They solve overlapping problems with opposite trade-offs.

◆ The deciding question

Is the concrete type known at the call site when you compile? If yes, CRTP gives you the abstraction for free. If the type is only known at runtime (you're storing mixed objects in one container and iterating), you need virtual functions — CRTP simply cannot express that, because Base<Circle> and Base<Square> share no common base type.

04 The incomplete-type trap & the traits escape hatch

This is the single most important subtlety, and it is exactly why Eigen is built the way it is. When the compiler is parsing Base<Derived>, the type Derived is incomplete — its definition hasn't finished yet (the base class is being established as part of defining the derived class). So the base may not touch anything that requires Derived to be complete:

Member-function bodies of a class template are only instantiated when used, by which point Derived is fully defined — so calling derived methods from the body works. But anything evaluated while the base class is being defined (data members, base-class-scope typedefs, default template arguments depending on Derived::X) hits the incomplete type and fails.

The traits workaround

The fix is to move the compile-time facts the base needs out of the derived class and into a separate, independently-specialized traits template. The base reads traits<Derived>::Scalar instead of Derived::Scalar, and traits<Derived> is a complete type even when Derived is not.

◆ Why this matters for Eigen

Eigen's MatrixBase<Derived> must know each expression's scalar type, compile-time row/column counts, storage order, and dozens of other flags — all while Derived is incomplete. So Eigen keeps every one of those facts in Eigen::internal::traits<Derived>, specialized for Matrix, Block, CwiseBinaryOp, and every other expression node. The traits class isn't an Eigen quirk; it's the necessary consequence of CRTP meeting incomplete types.

06 Eigen: expression templates & lazy evaluation

The problem: naïve operators allocate temporaries

Write a vector class the obvious way and every binary operator returns a fresh result by value:

The arithmetic is cheap; the temporaries and the repeated trips through memory are what hurt. We want the compiler to see the whole expression a + b + c at once and emit a single loop: d[i] = a[i] + b[i] + c[i], no temporaries. That fusion is what expression templates deliver.

The idea: + returns a description, not a result

Instead of computing, operator+ returns a small node object that merely remembers its operands by reference. Each node is itself a CRTP expression that knows how to produce its i-th element on demand. Nothing is computed until the result is assigned into a real vector — at which point one loop pulls element i through the entire expression tree.

No data is moved building this tree — only references are captured. The whole tree is a type the compiler knows, so coeff(i) inlines top-to-bottom into one expression.

A minimal expression-template engine

Here is the entire mechanism in ~40 lines. VecExpr<E> is the CRTP base; VecSum is an expression node; Vector is the concrete storage that triggers evaluation on assignment.

Now trace d = a + b + c; The type of the right-hand side is VecSum<VecSum<Vector,Vector>, Vector> — built with zero arithmetic, just captured references. The assignment loop runs once; e[i] inlines into (a[i] + b[i]) + c[i]. No temporaries, one pass, identical to a hand-written loop. The compiler optimizes the entire tree away.

◆ This is exactly Eigen, scaled up

Replace VecExpr with MatrixBase<Derived>, VecSum with CwiseBinaryOp<op, Lhs, Rhs>, and operator[] with coeff(row,col), and you have Eigen's architecture. Every node — Matrix, Block, Transpose, CwiseBinaryOp, CwiseNullaryOp — derives from MatrixBase via CRTP and exposes coeff(). Eigen layers on top: SIMD vectorization of the fused loop, unrolling for fixed-size types, cache-aware traversal, and a cost model that decides when fusing actually pays.

Aliasing — the price of laziness

Because nodes hold references and evaluation is deferred, the result can overlap an input. For element-wise ops this is harmless: a = a + b reads and writes a[i] at the same index, in order, so it's fine. Eigen assumes no aliasing for coefficient-wise expressions by default.

But operations where output element i depends on input elements at other positions are dangerous when aliased:

▲ Products are the exception to "lazy by default"

Coefficient-wise expressions (+, -, scalar *, .array() ops) are fused and lazy. Matrix multiplication is not folded into the same fused loop — a lazy product would recompute each shared dot-product many times, turning an O(n³) operation into something far worse. So Eigen evaluates products eagerly (often via a temporary). .noalias() tells Eigen the destination is disjoint from the operands so it can write the product straight into the destination and skip that temporary. Use it only when you are certain there's no overlap.

✓ Practical controls

.eval() forces evaluation of a sub-expression into a concrete temporary — your escape hatch for aliasing and for stopping runaway expression types. .noalias() on the left side suppresses the safety temporary for products. auto on an Eigen expression captures the expression type, not a value — convenient for composition but a footgun if the referenced operands die (see §09).

07 When to reach for it

Good fits — all share "the concrete type is known at compile time and the abstraction is on a hot path or pervasive":

  • Numeric / array libraries where temporaries dominate cost — the Eigen scenario. Expression templates are the textbook win.
  • Static interfaces on hot paths — small methods called in tight loops where a virtual call's missed inlining is measurable.
  • Mixins that synthesize many operations from a few primitives (comparisons, arithmetic, hashing, serialization) without touching the derived type's storage.
  • Per-type static state — counters, type registries, type-indexed caches.
  • Fluent builders that must keep returning the most-derived type through inherited setters.
  • Policy injection where behavior is selected at compile time and you want it inlined.

Reach for virtual functions instead when you store heterogeneous objects together, dispatch on runtime data, cross a stable ABI boundary (plugins, shared libraries), or simply value readable code and fast builds over the last few percent of speed.

08 Limitations

  • No runtime polymorphism. Base<A> and Base<B> are unrelated types. You cannot put them in one vector<Base*> or dispatch on a value decided at runtime. This is the hard ceiling.
  • Code bloat. Each derived type instantiates its own copy of every base member. Many derived types × large bases = large binaries and longer link times.
  • Compile-time cost. Deep expression-template trees and heavy traits machinery stress the compiler; build times grow.
  • Incomplete-type constraints. The base can't use the derived type's members at class scope — you must route through traits (§04).
  • Diagnostics. A small mistake can yield pages of template error spew (§09).
  • Dangling references with auto. Lazy nodes hold references; storing an expression in auto and outliving its operands is undefined behavior.
  • Verbosity & learning curve. The pattern is non-obvious to readers unfamiliar with it; the recursion reads like a mistake at first glance.

09 Pitfalls & classic bugs

9.1 — The copy-paste parameter bug

The most common CRTP bug, and one the compiler often won't catch immediately:

✗ Defend against it

Guard with a check that fires at instantiation: in a base member, static_assert(std::is_base_of_v<Base, Derived>, "CRTP misuse"); or assert the dynamic type. Even better, make the base constructor private and friend Derived; so only the genuine derived type can construct the base — a wrong parameter then fails to compile.

9.2 — Dangling expression with auto

Rule of thumb: assign Eigen expressions into a concrete Matrix/Vector (or call .eval()) unless you fully control the lifetime of the operands.

9.3 — Two-phase lookup when the derived class is itself a template

If you build a templated derived class on a dependent base, unqualified names aren't found in the dependent base during the first parse phase. Qualify with this-> or the base name:

9.4 — Forgetting the cast

Calling a hook directly in the base (impl() instead of static_cast<Derived*>(this)->impl()) silently calls the base's own (often non-existent or default) version instead of the derived override — no error, wrong behavior. Always go through the self() accessor.

9.5 — Object slicing & accidental base copies

Copying through a Base<Derived> reference slices off the derived part. Since you shouldn't be holding base references as handles anyway, keep base constructors/destructor protected to make this hard to do by accident.

10 Debugging tactics

  • Read template errors bottom-up, then top-down. The last "required from here" line is usually your actual call site; the first error line names the real constraint that failed. Ignore the middle.
  • Plant static_asserts at boundaries. static_assert(std::is_base_of_v<Base<D>, D>) turns silent CRTP misuse into a one-line, readable message instead of UB.
  • Inspect deduced types. Force a compile error that prints the type: template<class> struct Show; Show<decltype(expr)> _; — the error message shows exactly what monstrous expression type expr became. Invaluable for Eigen.
  • Add .eval() to bisect. If a lazy chain misbehaves, sprinkle .eval() to materialize intermediates; if the bug disappears, you've found an aliasing or lifetime problem.
  • Build sanitizers. ASan/UBSan catch the dangling-auto and wrong-downcast bugs that compile cleanly. Run them.
  • Check the assembly. Drop the snippet into Compiler Explorer (godbolt) at -O2. Correct CRTP/expression-template code shows a direct call or a single inlined loop; if you see call through a register or extra loops, fusion didn't happen.
  • Eigen specifically: compile with -DEIGEN_NO_DEBUG off during development — Eigen's assertions catch size mismatches and many aliasing mistakes at runtime with clear messages. EIGEN_RUNTIME_NO_MALLOC + internal::set_is_malloc_allowed(false) lets you assert that a hot path performs zero heap allocations.
  • Tame error walls with C++20 concepts: constrain the hook so a missing method reports "constraint not satisfied" rather than a deep instantiation failure (next section).

11 Modern alternatives (C++20 / C++23)

CRTP predates better tools for some of its jobs. Use these where they fit; keep CRTP where they don't (expression templates still want CRTP).

Deducing this (C++23): CRTP, retired for most cases

An explicit object parameter lets a base member template deduce the most-derived type directly — no <Derived> parameter, no static_cast, no recursion. For static interfaces, mixins, and fluent builders this is a clean replacement:

◆ What still needs CRTP

Deducing this handles member-function dispatch and mixins beautifully, but it does not replace CRTP where the base needs the derived type as a type at class scope — base-class typedefs, return types, friend declarations, and the traits-driven expression-template machinery in Eigen. So: prefer deducing this for new static-interface and mixin code; keep CRTP + traits for expression templates and anything that reasons about the derived type at compile time beyond a single call.

Other adjacent tools worth knowing: std::variant + std::visit for closed sets of types (compile-time-known alternatives without inheritance), and operator<=> (C++20) which retired the comparison-mixin use case entirely.

✓ The one-sentence summary

CRTP gives a base class compile-time knowledge of its derived type, which converts runtime polymorphism into zero-cost static polymorphism — and, in its most powerful form, lets operators return descriptions of work (expression nodes) so a whole expression collapses into one fused, temporary-free loop. That is the engine inside Eigen.
