Part I — Single-GPU speed

Wringing the most from one GPU.

Before you reach for more hardware, make one GPU fly. Three moves — lower-precision math, a one-line compiler, and disciplined memory use — routinely cut training time by half or more with no change to your model's accuracy. And none of it is worth doing blind: the fourth chapter is the profiler that tells you which change will actually help.

01Mixed precision

By default every number is 32-bit float (FP32). But modern GPUs have dedicated hardware (tensor cores) that multiply 16-bit floats several times faster, and 16-bit values take half the memory. Mixed precision runs the bulk of the math — the matrix multiplies and convolutions — in 16-bit, while keeping a few precision-sensitive pieces (the master weights, the loss reduction) in FP32. The result is typically 1.5–3× faster training and roughly half the activation memory, at no cost to final accuracy.

Two 16-bit formats exist and the choice matters. bfloat16 (bf16) keeps FP32's wide exponent range, trading precision for stability — it "just works" without extra machinery and is the default on Ampere-and-newer GPUs (A100, H100, and later). float16 (fp16) has more precision but a narrow range, so small gradients underflow to zero; it needs a GradScaler that multiplies the loss up before backward and unscales before the step.

import torch

model = model.to("cuda")
opt = torch.optim.AdamW(model.parameters(), lr=3e-4)

# --- bf16: the modern default on A100/H100+. No scaler needed. ---
for xb, yb in loader:
    xb, yb = xb.to("cuda"), yb.to("cuda")
    with torch.autocast("cuda", dtype=torch.bfloat16):   # run ops in bf16
        loss = loss_fn(model(xb), yb)
    opt.zero_grad()
    loss.backward()
    opt.step()

# --- fp16: needs a GradScaler to stop tiny gradients underflowing ---
scaler = torch.amp.GradScaler("cuda")
for xb, yb in loader:
    xb, yb = xb.to("cuda"), yb.to("cuda")
    with torch.autocast("cuda", dtype=torch.float16):
        loss = loss_fn(model(xb), yb)
    opt.zero_grad()
    scaler.scale(loss).backward()     # scale up before backward
    scaler.step(opt)                  # unscale, then step (skips if inf/nan)
    scaler.update()                   # adapt the scale factor
Prefer bf16 when your GPU supports it

On Ampere or newer, use bfloat16: same speed as fp16, no GradScaler, far fewer instability headaches. Reserve fp16 (with a scaler) for older GPUs that lack bf16 tensor cores. Either way, autocast chooses per-operation which precision is safe, so you don't manually cast anything.

02torch.compile

Eager mode runs your model op by op, dispatching each through Python — flexible, but it leaves speed on the table by launching many tiny GPU kernels and never fusing them. torch.compile (the flagship feature of the PyTorch 2 line) traces your model into a graph, fuses operations, and generates optimized kernels with its backend compiler (TorchInductor). It's one line, it keeps eager's debuggability, and it commonly delivers 1.3–2× speedups — stacking on top of mixed precision.

import torch

model = torch.compile(model)      # that's it — compiles on the first call

# The first iteration is SLOW (it compiles); subsequent ones are fast.
for xb, yb in loader:
    loss = loss_fn(model(xb.cuda()), yb.cuda())   # compiled, fused kernels
    ...

# Modes trade compile time for runtime speed:
model = torch.compile(model, mode="reduce-overhead")  # best for small models
model = torch.compile(model, mode="max-autotune")     # slowest compile, fastest run

Two things to understand so it doesn't surprise you. First, the first call compiles and is much slower — this is normal; time your throughput after warmup, not on step one. Second, graph breaks: data-dependent Python control flow (an if on a tensor's value, printing a tensor, calling .item()) forces the compiler to split the graph, and shape changes trigger recompiles. The compiler handles breaks gracefully — it just optimizes each piece — but for maximum speed you minimize them, and fullgraph=True makes any break an error so you can find them.

Compiler internals move fast — the interface doesn't

torch.compile has been stable and recommended for years, but its internals (supported patterns, autotuning, determinism guarantees) improve every release — recent versions added run-to-run determinism under torch.use_deterministic_algorithms(True). Treat the one-line interface as durable; treat exact edge-case behavior and flags as current-but-evolving, and re-check the docs for your version.

03Memory & activation checkpointing

Running out of GPU memory is the most common wall in scaling. Four things consume it, and knowing the split tells you what to cut:

ConsumerSize · notes
parametersThe weights. Fixed by model size. (1 unit)
gradientsOne per parameter. (1 unit)
optimizer stateAdam/AdamW keep two extra buffers per parameter. (2 units) — this is why the optimizer alone can triple parameter memory.
activationsEverything saved during forward for the backward pass. Grows with batch size, sequence length, and depth — often the largest consumer, and the one you can trade away.

Activation checkpointing (gradient checkpointing) attacks the biggest consumer: instead of storing every intermediate activation for the backward pass, it stores only a few and recomputes the rest during backward. You spend roughly one extra forward pass of compute to slash activation memory — often enough to fit a 2–4× larger batch or model. It's the standard tool when a model almost fits.

from torch.utils.checkpoint import checkpoint

class Block(nn.Module):
    def forward(self, x):
        # Wrap an expensive sub-computation: its activations are NOT stored,
        # they're recomputed in the backward pass. Memory down, compute up.
        return checkpoint(self._heavy, x, use_reentrant=False)

    def _heavy(self, x):
        return self.mlp(self.attn(x))
The OOM escape ladder

Hit "CUDA out of memory"? Climb this ladder in order: (1) mixed precision (halves activations), (2) smaller batch size with gradient accumulation to keep the effective batch (Volume II), (3) activation checkpointing, (4) a more memory-efficient optimizer or 8-bit optimizer, and only then (5) more GPUs and sharding (Part III). Each step is cheaper than the next.

04Profiling: find the bottleneck

The cardinal rule of performance work: measure, don't guess. Intuition about what's slow is wrong more often than not — the culprit is frequently the data pipeline or a hidden CPU-GPU synchronization, not the math you assumed. PyTorch's profiler records where time actually goes, on both CPU and GPU.

import torch
from torch.profiler import profile, ProfilerActivity

with profile(activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
             record_shapes=True) as prof:
    for i, (xb, yb) in enumerate(loader):
        loss = loss_fn(model(xb.cuda()), yb.cuda())
        loss.backward(); opt.step(); opt.zero_grad()
        if i >= 10: break

# The heaviest operations, by GPU time:
print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=10))
prof.export_chrome_trace("trace.json")   # open in chrome://tracing or Perfetto

A handful of patterns account for most wasted time. If GPU utilization is low and spiky, you're input-bound — fix the data pipeline (Part II). If the timeline is full of many tiny kernels, you're launch-bound — torch.compile fuses them. Frequent .item(), print(tensor), or .cpu() calls force synchronization points that stall the pipeline — batch your logging and pull metrics off the GPU sparingly. And large host↔device copies signal a transfer problem — pin memory and use non_blocking=True.

Optimize the measured bottleneck, nothing else. Every hour spent speeding up code that wasn't the bottleneck is wasted, and worse, it adds complexity you'll maintain forever. Profile first, find the single biggest cost, fix that, then profile again. Performance work is a loop of measurement, not a checklist of tricks applied hopefully.

Part II — Throughput

Keeping the tensor cores saturated.

Peak speed comes from three habits: never let the GPU wait on data, use the memory layouts and fused kernels the hardware prefers, and size batches for the throughput curve rather than by feel. Small, unglamorous changes here often beat a fancier model — and they compound with Part I's precision and compilation wins.

05The input pipeline at scale

Volume II's num_workers and pin_memory are table stakes. At scale, the bottleneck shifts to how the data is stored. Millions of tiny individual files murder throughput: each incurs filesystem overhead, and random access thrashes the disk. The fix is to pack data into a smaller number of large sequential shards (tar archives, or formats like WebDataset), which stream efficiently and shard cleanly across machines.

  • Shard into big files. Group thousands of examples per archive; read them sequentially. Turns random small reads into fast streaming.
  • Push decode/augment to the GPU when the CPU can't keep up. JPEG decoding and heavy augmentation can bottleneck the CPU; libraries that decode on the GPU (e.g. NVIDIA DALI) move that work off the critical path.
  • Cache what's stable. If preprocessing is deterministic and fits, cache the processed tensors so you pay the cost once, not every epoch.
  • Keep __getitem__ cheap. Per-item Python overhead multiplies by millions. Do vectorized work in batches, not heavy per-example logic.
The test: is the GPU ever idle?

Watch GPU utilization during a real run. If it dips between batches, data is your bottleneck and the moves above pay off directly. If it's pinned near 100%, your pipeline is fine — stop optimizing it and look at the model (Part I). The pipeline only matters until it stops being the limit.

06Kernel wins: channels_last, TF32, fused

Several near-free switches let the same math run faster by matching what the hardware wants. None change your model or its accuracy meaningfully; they change layout and kernel choice.

SwitchWhat it does · when
channels_lastReorders conv tensors to (N,H,W,C) in memory, which tensor cores prefer — often a solid CNN speedup. model.to(memory_format=torch.channels_last) and same for inputs.
TF32Lets FP32 matmuls/convs use tensor cores at slightly reduced precision on Ampere+. Big speedup, negligible accuracy cost for training.
fused optimizerAdamW(..., fused=True) runs the whole update in one kernel instead of many — meaningful when you have thousands of parameter tensors.
SDPA / FlashAttentionF.scaled_dot_product_attention (Volume III) is the fused, memory-efficient attention — always prefer it over a hand-rolled softmax for transformers.
 
import torch

# Let FP32 matmuls use tensor cores (TF32) — set once at startup.
torch.set_float32_matmul_precision("high")

# channels_last for a CNN: convert both model and inputs.
model = model.to(memory_format=torch.channels_last)
xb = xb.to(memory_format=torch.channels_last)

# Fused optimizer step (one kernel for the whole update).
opt = torch.optim.AdamW(model.parameters(), lr=3e-4, fused=True)
Stack them

These compose with Part I. A modern recipe — bf16 autocast + torch.compile + channels_last + TF32 + fused AdamW — routinely runs several times faster than naive FP32 eager, with the same final accuracy. Turn them on early; they're the cheapest speed you'll ever buy.

07Effective batch size

Bigger batches use the GPU more efficiently — up to a point. Throughput climbs steeply at first, then flattens once the hardware is saturated, after which a larger batch only costs memory. The job is to find that knee, then hold the effective batch there regardless of what fits in memory.

THROUGHPUT vs BATCH SIZE (schematic): samples/sec │ ______________ ← saturated: bigger batch │ / just costs memory, not speed │ / │ / │ / │ / ← steep early gains └──────────────────────────────► batch size ↑ the "knee": aim here

Two levers set the effective batch independent of memory. Gradient accumulation (Volume II) sums several micro-batches before stepping, simulating a big batch on a small GPU. Data parallelism (Part III) splits a large batch across GPUs. When you raise the effective batch, remember the linear scaling rule: roughly scale the learning rate with it, and add warmup so the larger early steps stay stable.

Bigger isn't automatically better

Very large batches produce smoother gradients that can converge to sharper minima and generalize slightly worse, and they need careful LR scaling and warmup to match small-batch quality. Chase batch size for throughput, but validate that accuracy holds — don't assume a 4× batch is strictly an upgrade.

08Determinism & reproducible performance

Two different goals get confused here: reproducibility (same result every run) and fast benchmarking (trustworthy speed numbers). They pull in opposite directions, because the fastest kernels are often nondeterministic.

For reproducibility, seed everything and force deterministic kernels — accepting some slowdown, since a few fast GPU operations use nondeterministic atomic adds:

import torch

torch.manual_seed(0)
torch.use_deterministic_algorithms(True)   # error out on nondeterministic ops
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False     # benchmark picks fast, varying algos

For raw speed, do the opposite: let cuDNN benchmark and pick the fastest convolution algorithm for your (fixed) input shapes. It's nondeterministic and adds a warmup cost the first time each shape appears, but it's a real speedup for shape-stable training.

torch.backends.cudnn.benchmark = True      # fastest algos for fixed shapes
Benchmark correctly or your numbers lie

GPU work is asynchronous — the Python line returns before the kernel finishes. To time anything, warm up first (compile, cuDNN benchmark, and caches settle over the first iterations), then torch.cuda.synchronize() before reading the clock, and average over many iterations. Timing a single un-synchronized step measures Python dispatch, not the GPU, and will be off by orders of magnitude.

Reproducible or fast — pick per situation, because the fastest kernels are the ones that don't promise the same bits twice.

Part III — Distributed training

Many GPUs, one model.

One GPU eventually runs out — of time, or of memory. Spreading a job across many GPUs is how large models get trained, but it introduces a new problem: keeping replicas in sync without drowning in communication. This part walks the ladder from simple data parallelism (DDP) to full sharding (FSDP2), and names the pitfalls that turn a multi-GPU run into a hung one.

09The landscape: DP → DDP → FSDP

There are two fundamentally different reasons to go distributed, and they call for different tools. If your model fits on one GPU but training is slow, you want data parallelism: put a full copy of the model on each GPU, feed each a different slice of the batch, and average their gradients so all copies stay identical. If your model doesn't fit on one GPU, you need to shard it — split the parameters themselves across GPUs.

How do I scale this training run?
   │
   ├─ model fits on 1 GPU, want it faster → DistributedDataParallel (DDP)
   │        one full replica per GPU, split the batch, all-reduce gradients
   │
   ├─ model + optimizer states don't fit → FSDP2 (fully_shard)
   │        shard params/grads/optimizer state across GPUs, gather just-in-time
   │
   └─ model is enormous (many billions) → FSDP2 + tensor/pipeline
            parallelism — split individual layers and the depth across GPUs too

One historical note that saves confusion: PyTorch's old single-process nn.DataParallel (DP) is deprecated and slow — it bottlenecks on one process coordinating all GPUs. Always use DistributedDataParallel (DDP), which runs one process per GPU. When people say "data parallel" today, they mean DDP.

10DDP in depth

DistributedDataParallel launches one process per GPU. Each holds a complete copy of the model and trains on its own shard of the data. The magic is in the backward pass: as gradients are computed, DDP all-reduces them — averages each gradient across all processes — so every replica applies an identical update and they never drift apart. Better still, it overlaps that communication with the backward computation (bucketing gradients and reducing them as soon as each bucket is ready), so the syncing is largely hidden behind work you're doing anyway.

import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader
from torch.utils.data.distributed import DistributedSampler

def main():
    dist.init_process_group("nccl")            # NCCL: the GPU comm backend
    rank = dist.get_rank()
    local_rank = rank % torch.cuda.device_count()
    torch.cuda.set_device(local_rank)

    model = MyModel().to(local_rank)
    model = DDP(model, device_ids=[local_rank])   # wrap it

    # Each rank must see a DIFFERENT slice of the data:
    sampler = DistributedSampler(dataset)
    loader = DataLoader(dataset, batch_size=64, sampler=sampler)
    opt = torch.optim.AdamW(model.parameters(), lr=3e-4)

    for epoch in range(epochs):
        sampler.set_epoch(epoch)               # reshuffle differently each epoch
        for xb, yb in loader:
            loss = loss_fn(model(xb.to(local_rank)), yb.to(local_rank))
            opt.zero_grad(); loss.backward(); opt.step()   # all-reduce in backward

    dist.destroy_process_group()

You launch it with torchrun --nproc_per_node=8 train.py, which spawns the eight processes and sets the environment variables DDP reads. Two correctness essentials: use a DistributedSampler so each rank gets distinct data (otherwise every GPU trains on the same batch — pure waste), and call sampler.set_epoch(epoch) each epoch so the shuffle varies. The effective batch size is your per-GPU batch times the number of GPUs, so revisit the learning-rate scaling from Part II.

Log and checkpoint from rank 0 only

All processes run the same code, so guard side effects — printing, progress bars, saving checkpoints — with if rank == 0:. Otherwise eight processes fight over the same file and flood your logs eightfold. Since replicas are identical, rank 0's weights represent them all.

11FSDP2 for large models

DDP replicates the entire model — parameters, gradients, and optimizer states — on every GPU. For a large model that's wasteful, and eventually impossible: with AdamW, a model needs roughly its parameter memory plus gradients plus two optimizer buffers, so a model that barely fits leaves no room to train. Fully Sharded Data Parallel (FSDP) fixes this by sharding those tensors across GPUs — each GPU stores only a slice. When a layer runs, FSDP gathers its full parameters just-in-time (an all-gather), computes, then immediately frees the gathered copy. Memory per GPU drops toward the total divided by the number of GPUs, at the cost of extra communication.

The current API is FSDP2 — fully_shard — which shards at the level of individual parameters and composes cleanly with mixed precision, activation checkpointing, and torch.compile. You apply it to the sub-modules you want sharded (typically each transformer block), then to the whole model:

import torch
from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy

# (inside each process, after init_process_group as in DDP)
model = MyLargeTransformer()

mp = MixedPrecisionPolicy(param_dtype=torch.bfloat16,
                          reduce_dtype=torch.float32)   # bf16 compute, fp32 grads

# Shard each block, then the root. Blocks gather -> compute -> free in turn.
for block in model.blocks:
    fully_shard(block, mp_policy=mp)
fully_shard(model, mp_policy=mp)

opt = torch.optim.AdamW(model.parameters(), lr=3e-4)    # now over sharded params
# training loop is the same shape as DDP from here
The distributed frontier moves quickly

Large-scale training is the fastest-changing corner of PyTorch. FSDP2 (fully_shard) is the current, recommended path and replaced the earlier FullyShardedDataParallel wrapper; the exact policy classes and options evolve each release, and higher-level libraries (such as torchtitan) package best-practice recipes. The concepts here — shard params/grads/optimizer state, gather just-in-time, overlap comm with compute — are durable; verify the exact API against your version's docs.

12Multi-node & communication pitfalls

Scaling past one machine adds the network as a new bottleneck. Inside a node, GPUs talk over fast NVLink; between nodes, over slower InfiniBand or Ethernet. Because every step synchronizes gradients, communication can come to dominate wall-clock time, and the whole run moves at the speed of its slowest participant. The pitfalls are less about code than about the failure modes:

  • Uneven data hangs the job. Collective operations (all-reduce, all-gather) require every rank to participate. If one rank has fewer batches and exits early, the others wait forever. Ensure every rank runs the same number of steps (drop the ragged last batch, or use a join context).
  • BatchNorm needs syncing. Each GPU's batch-norm sees only its shard's statistics, so replicas diverge. Convert to SyncBatchNorm so stats are averaged across GPUs — or use batch-independent norms (LayerNorm/GroupNorm), which sidestep the issue.
  • Checkpointing sharded state is different. With FSDP each rank holds a slice; use distributed checkpointing to save and restore correctly rather than assuming rank 0 has the whole model.
  • Stragglers and non-determinism. One slow GPU, thermal throttle, or a noisy network link drags everyone. Monitor per-rank step times; large-scale runs live and die by their slowest link.

Communication is the tax on parallelism. Every distributed strategy is a trade between memory saved and bytes moved: DDP moves gradients once per step; FSDP additionally moves parameters every layer. Scaling well means keeping the GPUs computing while the bytes fly — overlapping communication with computation — and never adding a GPU whose communication cost outweighs its compute. Past a point, more GPUs make a step slower, not faster.

Part IV — Compression & deployment

Shrinking it, then shipping it.

A trained model is a research artifact; a deployed model is a product, and the gap between them is compression and export. This part makes models smaller and faster for inference — quantization, pruning, distillation — then takes them out of Python with the modern torch.export path and into a serving stack. Note the deprecation: TorchScript is retired; torch.export is the way now.

13Quantization

Training happens in 16- or 32-bit floats, but inference often doesn't need that precision. Quantization represents weights (and sometimes activations) in low-bit integers — usually int8 — which shrinks the model roughly 4× and runs faster, especially on CPUs and edge devices with integer-optimized hardware. The trade is a small, usually recoverable accuracy drop.

Two approaches, differing in when the quantization happens:

ApproachPost-training (PTQ)Quantization-aware (QAT)
HowQuantize an already-trained model; calibrate scales on a little data.Simulate quantization during training so the model adapts to it.
CostMinutes, no retraining.A full (or fine-tuning) training run.
AccuracyGood for many models; can slip on sensitive ones.Best — recovers most of the quantization loss.
Use whenFirst thing to try; often enough.PTQ lost too much accuracy and it matters.
 
# torchao is the current PyTorch library for quantization (and low-bit training).
import torchao
from torchao.quantization import quantize_, Int8WeightOnlyConfig

model.eval()
# Post-training, weight-only int8 — a fast, low-risk default for inference:
quantize_(model, Int8WeightOnlyConfig())
# The model now stores int8 weights: ~4x smaller, faster memory-bound inference.

For large language models, quantization is nearly universal at inference time — int8 and even 4-bit weight quantization make models fit on smaller GPUs with little quality loss, and pair with the fine-tuning methods in Volume V (QLoRA). The exact APIs here evolve, but the idea is stable: fewer bits per weight, big memory and speed wins, modest accuracy cost.

14Pruning & distillation

Two more ways to make a model smaller, orthogonal to quantization and often stacked with it.

Pruning removes parameters the model barely uses. Unstructured pruning zeros individual weights (high sparsity, but needs special hardware/kernels to actually speed up). Structured pruning removes whole channels or heads — less aggressive, but the smaller model is genuinely faster on any hardware because it's simply smaller. Pruning usually degrades accuracy, so you fine-tune afterward to recover.

Knowledge distillation takes a different tack: train a small "student" model to mimic a large "teacher." Instead of (or alongside) the hard labels, the student learns from the teacher's full output distribution — the "soft" probabilities that reveal how the teacher sees relationships between classes. A well-distilled student can retain most of the teacher's accuracy at a fraction of the size.

import torch.nn.functional as F

def distillation_loss(student_logits, teacher_logits, labels, T=2.0, alpha=0.5):
    # Soft targets: match the teacher's softened distribution (KL divergence).
    soft = F.kl_div(
        F.log_softmax(student_logits / T, dim=-1),
        F.softmax(teacher_logits / T, dim=-1),
        reduction="batchmean") * (T * T)
    hard = F.cross_entropy(student_logits, labels)   # plus the real labels
    return alpha * soft + (1 - alpha) * hard
Compression stacks

These techniques compose: distill a large model into a small one, prune the redundant channels, then quantize the result to int8. Each multiplies the savings. In practice, distillation + quantization is the most common production combo — a small student, quantized, serves cheaply while retaining most of the teacher's quality.

15Export: torch.export

Your training code is Python, but production often wants a portable, Python-free graph — for a C++ service, a mobile app, or a specialized runtime. torch.export captures your model as an ExportedProgram: a clean, self-contained graph representation that downstream tools can compile and run anywhere.

import torch

model.eval()
example = (torch.randn(1, 3, 224, 224),)          # a representative input

exported = torch.export.export(model, example)    # capture the graph
print(exported.graph_module)                       # inspect the captured ops

torch.export.save(exported, "model.pt2")           # portable artifact
# ...load and run elsewhere, or lower to a runtime (below).
TorchScript is deprecated — use torch.export

Older tutorials reach for torch.jit.script/torch.jit.trace (TorchScript). That path is now deprecated. New deployment work should target torch.export and the runtimes built on it. If you inherit a TorchScript model it still runs, but don't build anything new on it.

From an ExportedProgram you lower to whatever runs your model: ExecuTorch for phones and edge devices, AOTInductor to compile it to a standalone shared library for C++ serving, or ONNX as an interchange format for other runtimes. Always export in eval() mode (so dropout/batch-norm behave for inference), pass a representative example input, and declare dynamic dimensions (like variable batch size) explicitly if the graph must handle them.

16Serving & inference optimization

Serving is a different optimization problem than training. There's no backward pass, so no gradients or optimizer state — memory is just weights and activations. You care about two metrics that trade against each other: latency (how long one request takes, watched at the tail — p99, not just the average) and throughput (requests per second per dollar). The knobs:

  • Turn off gradients, switch to eval. Wrap inference in torch.no_grad() (or inference_mode(), which is stricter and faster) and call model.eval(). Free speed and memory.
  • Compile and quantize. torch.compile and int8 quantization apply to inference too, often for bigger relative wins than in training.
  • Batch requests. Group incoming requests into batches to use the GPU efficiently — the central trick of every serving system, trading a little latency for a lot of throughput.
  • Cache what repeats. For autoregressive generation, the KV cache (Volume V) avoids recomputing attention over the whole prefix each token — the single biggest LLM inference optimization.

You rarely build serving from scratch. Dedicated servers handle batching, queuing, and scaling: TorchServe or NVIDIA Triton for general models, and specialized engines like vLLM for LLMs (which build in continuous batching and paged KV caches). Your job is to hand them an optimized, exported model and choose the batching and precision that hit your latency target.

Training optimizes a number; serving optimizes a budget. In production the question isn't "how accurate?" but "how much accuracy per dollar per millisecond?" Compression and export exist to move a model along that curve — smaller, faster, cheaper — until it fits the latency and cost envelope the product actually has. A model that can't be served within budget hasn't shipped.

The last mile — quantize, export, serve — is where a model stops being a demo and starts being a product.
