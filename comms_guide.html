Section 00

The Layered Stack: where every library lives

Before walking the history, anchor everything to one mental model. "Getting bytes from process A to process B" is solved at several layers, and every tool you listed is a layer (or a span of layers) in this stack. The recurring theme of the last 30 years is pushing common functionality down into reusable layers so the layers above stop reinventing it. UCX is the most recent big consolidation of that idea.

The core answer to your last question, up front

Yes — your intuition about UCX is exactly right. UCX sits above the transport APIs (sockets, verbs, CUDA IPC) and below the programming models (MPI, NCCL, SHMEM). The contract becomes: MPI ↔ UCX ↔ physical. An MPI implementation that uses UCX no longer talks to InfiniBand verbs or TCP sockets directly — it hands UCX a buffer, a destination, and a tag, and UCX decides which wire and which protocol to use. We'll justify why that abstraction was worth building in Section 08.

Two axes that explain every design choice

As you read, classify each tool along two axes. Almost every "why is it designed this way" question collapses into these:

  • Two-sided vs. one-sided. Two-sided (MPI send/recv) means both peers participate: the sender posts a send, the receiver posts a matching receive, and the library matches them. One-sided / RMA (SHMEM put/get, MPI-3 RMA, RDMA verbs) means one peer reads or writes the other's memory without the remote CPU being involved at all. One-sided maps directly onto RDMA hardware; two-sided requires a matching/queueing layer on top.
  • Copy vs. zero-copy. Does the data get staged through an intermediate bounce buffer, or does the NIC DMA straight out of and into the user's buffer? Zero-copy is faster but needs the buffer to be registered (pinned and mapped for the NIC) and needs a handshake so the sender knows where to write. This single trade-off is the reason the eager/rendezvous protocol split exists, which you'll see in MPI, UCX, and NCCL alike.

Section 01

Evolution timeline: how we got here

This is the lineage you asked for, compressed. Each entry solved a pain point left by the previous one.

1980s
Sockets (BSD) + vendor message libraries
Every supercomputer vendor (Intel NX, IBM, Thinking Machines) shipped its own incompatible message-passing API. Portable code was impossible. TCP/IP sockets worked everywhere but were slow and low-level.
1989–1993
PVM (Parallel Virtual Machine)
First widely adopted portable model: turn a heap of heterogeneous workstations into one "virtual machine." Dynamic process spawn, built-in heterogeneity/data conversion. Proved the demand but was a single implementation, not a standard.
1994
MPI-1 standard
The community wrote a specification, not a product. Anyone could implement it; user code became portable across vendors. This "standard, not a binary" decision is the most important event on this timeline.
1990s–2000s
MPICH & LAM/MPI → Open MPI; InfiniBand arrives (1999)
Reference implementations matured. InfiniBand brought commodity RDMA, forcing MPI implementations to grow pluggable transport layers (BTL/MTL, CH3/CH4) instead of being hard-wired to TCP.
2000s
PGAS era: SHMEM, GASNet, UPC, OpenSHMEM (2010)
One-sided "just write into remote memory" models, riding RDMA directly. GASNet became the portable conduit beneath many of them — a conceptual ancestor of UCX.
2009→
GPUs enter the fabric: GPUDirect, CUDA-aware MPI (~2013)
GPUs became the compute, but data still detoured through host memory. GPUDirect RDMA let the NIC DMA straight from GPU memory; MPI implementations learned to accept device pointers.
2015–2016
NCCL (2015) and UCX (~2015)
NCCL: GPU-topology-aware collectives that MPI couldn't deliver. UCX: unify the duplicated transport code that MPICH, Open MPI, and SHMEM had each written separately, into one framework.
2017→
NVSHMEM, GPU-initiated comms, UCX everywhere
NVSHMEM put the OpenSHMEM one-sided model inside CUDA kernels (GPU initiates communication). UCX became the default transport under Open MPI, MPICH, and OpenSHMEM; NCCL and DeepSpeed/PyTorch built on these layers.
The repeating pattern

Notice the rhythm: a new hardware capability appears (RDMA, GPUDirect, NVLink) → each programming model bolts on support for it independently → the duplicated plumbing gets painful → a shared lower layer is extracted (GASNet, then UCX/libfabric) to absorb it once. Keep this in mind; it is the entire justification for UCX.

Section 02

The physical & transport layer

Everything above is built on a small set of transports. You cannot reason about MPI/NCCL behavior without understanding these, because the libraries' "design choices" are mostly which transport to use and how to feed it.

TCP/IP & Berkeley sockets — the universal baseline

The kernel sockets API (socket(), connect(), send(), recv()) is the lowest common denominator: it works on any Ethernet anywhere. Its mechanism is also its weakness for HPC:

  • Kernel involvement on every transfer. A send() copies your buffer into a kernel socket buffer (skb), the TCP/IP stack segments it, the driver hands it to the NIC. The receiver's NIC raises an interrupt, the kernel reassembles, copies into the socket buffer, and a final copy lands it in your recv() buffer. That's two memory copies plus a syscall plus an interrupt per message — the "data touches the CPU many times" problem.
  • Latency floor. Syscall + interrupt + TCP processing gives you ~10–50 µs half-round-trip on commodity gear. RDMA fabrics deliver ~1–2 µs. For tightly-coupled HPC where you exchange millions of small messages, that gap dominates.
  • It is two-sided and stream-oriented. TCP gives you a reliable byte stream with no message boundaries; message libraries add their own framing on top.
Why it still matters

Sockets are the fallback every library keeps. When MPI/NCCL/UCX can't find a fast fabric, they fall back to TCP — which is exactly why a job that "works but is mysteriously slow" is often silently running over TCP instead of InfiniBand. (Debugging that is a recurring theme below.)

Shared memory — intra-node fast path

When two ranks are on the same node, going through a NIC is wasteful. Transports here:

  • SysV / POSIX shared memory + copy. Both processes map a common region; sender copies into it, receiver copies out. Two copies, but no kernel networking. Classic intra-node MPI path.
  • Single-copy via kernel assist. CMA (process_vm_readv/writev), KNEM, and XPMEM let one process's pages be read/written directly by another, eliminating one copy for large messages. XPMEM can even expose another process's address space for true zero-copy.
  • CUDA IPC. For two GPUs on one node, cudaIpcGetMemHandle lets a peer map another process's GPU buffer; the copy then rides PCIe or NVLink without touching host memory.

InfiniBand & the verbs API — RDMA, properly

InfiniBand (IB) is a purpose-built HPC fabric. The software interface is verbs (libibverbs), and it is a fundamentally different model from sockets. This is the single most important transport to understand because UCX, MPI, and NCCL all ultimately drive it.

The verbs mechanism (kernel bypass)

  1. Queue Pairs (QP). You create a send queue and a receive queue. To communicate you post Work Requests (WR) directly into these queues from user space.
  2. Kernel bypass. After setup, posting a request and reaping a completion happen entirely in user space by writing/reading mapped device memory (a doorbell). No syscall, no interrupt on the data path. This is where the µs-latency comes from.
  3. Memory registration. Before the NIC can DMA a buffer, you must ibv_reg_mr() it: the buffer's pages are pinned (can't be swapped/moved) and the NIC's IOMMU mapping + a protection key (rkey/lkey) are created. Registration is expensive — this cost is why every higher layer obsesses over registration caches and bounce buffers.
  4. Completion Queues (CQ). You poll a CQ for completions (busy-poll for lowest latency) or arm it for an event.

Two flavors of operation

  • Channel semantics (SEND/RECV). Two-sided: receiver must pre-post a receive buffer; sender's SEND consumes it. This is what MPI eager messages map to.
  • Memory semantics (RDMA WRITE / READ). One-sided: sender already knows the remote address + rkey and writes/reads it with zero remote CPU involvement. This is what powers SHMEM put/get, MPI RMA, and MPI rendezvous bulk transfer.
GPUDirect RDMA

Normally the NIC can only DMA host memory. GPUDirect RDMA lets you register GPU memory with the HCA so the NIC DMAs straight out of / into GPU VRAM across the network — host memory and the CPU are never touched. This is the hardware feature that makes "CUDA-aware MPI" and NCCL-over-IB fast. It requires the NIC and GPU to share a PCIe path (ideally under the same switch) for full performance.

RoCE & iWARP — RDMA over Ethernet

RDMA's benefits (kernel bypass, one-sided ops) were so valuable that two standards brought them to Ethernet so you don't need an IB fabric:

  • RoCE (RDMA over Converged Ethernet): runs the IB transport over Ethernet. RoCEv1 is L2-only; RoCEv2 encapsulates in UDP/IP and is routable — the common one. Crucially it usually needs a lossless fabric via Priority Flow Control (PFC) / ECN, because the IB transport historically assumed no drops. Misconfigured PFC is the #1 RoCE pain point.
  • iWARP: RDMA layered over TCP. Tolerates lossy networks (TCP handles loss) but has higher latency and is less common today.

From the verbs API's perspective, IB, RoCE, and iWARP look nearly identical — same libibverbs calls — which is precisely why a transport-neutral layer like UCX can treat them uniformly.

NVLink & NVSwitch — the GPU-to-GPU fabric

PCIe (~16–64 GB/s per direction depending on gen/width) became the bottleneck between GPUs. NVIDIA's answer:

  • NVLink: direct high-bandwidth GPU↔GPU (and GPU↔CPU on some platforms) links — hundreds of GB/s, far above PCIe, with cache-coherent-style load/store access to peer GPU memory.
  • NVSwitch: a crossbar so that all GPUs in a node (e.g. 8× in a DGX) get full NVLink bandwidth to every other GPU, not just nearest neighbors.
  • Across nodes, you still need IB/RoCE; GPUDirect RDMA + NVLink intra-node is the combination NCCL exploits to build its topology-aware rings/trees.

Section 03

PVM — the ancestor worth knowing

PVM (Parallel Virtual Machine, Oak Ridge / Emory, ~1989) is rarely used now but explains why MPI is shaped the way it is — MPI's design was largely a reaction to PVM's strengths and weaknesses.

What PVM did

  • A "virtual machine" daemon model. A pvmd daemon ran on each host; together they presented your pile of (possibly heterogeneous) workstations as one machine. You'd pvm_spawn() tasks dynamically at runtime.
  • Built-in heterogeneity. PVM did data encoding/decoding (pvm_pkint, XDR-style) so a big-endian and little-endian machine could exchange data correctly — a real concern in the era of mixed Suns, Crays, and x86.
  • Dynamic & fault-aware. Tasks could be added/removed during a run, and PVM had notification of host failure.

Why MPI superseded it

  • PVM was effectively one implementation, controlled by its authors. MPI was a standard many vendors could implement and optimize for their hardware — instantly giving portable code real performance.
  • PVM routed much traffic through daemons (extra hops/copies); MPI's model allowed direct, highly optimized point-to-point and rich collectives.
  • MPI formalized communicators, datatypes, and a huge collective API that PVM lacked.
The lesson that shaped everything after

MPI winning over PVM established the pattern that recurs through this entire guide: a community standard with many competing implementations beats a single product. NCCL and NVSHMEM are the notable exceptions (single-vendor), and OpenSHMEM/MPI are the rule.

Section 04

MPI — the deep dive

MPI is the workhorse. We'll answer each of your questions in mechanism-level detail.

"MPI is a protocol / contract" — elaborated

This framing is exactly correct and is the key to the whole ecosystem. MPI is a specification document (currently MPI-4.x) that defines an API and its semantics — function signatures, what MPI_Send guarantees, completion rules, datatypes, communicators. It deliberately says nothing about wires, buffers, or protocols on the network. It is a contract between your code and "some implementation."

Because it's a contract, you get two independent things:

  • Source-level portability. Recompile the same code against any MPI library and it runs — on a laptop over TCP, on a supercomputer over Slingshot, on an IB cluster.
  • Implementation freedom. Each implementation is free to transform your MPI_Send into whatever physical communication is fastest on the given hardware. This "transform user code → physical communication" mapping is where implementations differ, not in the API you call.
Does the user interface change between implementations?

No — that is the whole point. MPI_Bcast, MPI_Isend, etc. are identical across Open MPI, MPICH, MVAPICH2, Intel MPI, Cray MPICH. What changes is everything below the API: which transport, which collective algorithm, which buffering strategy, whether GPU pointers are accepted, tuning knobs, and the launcher. You can usually swap implementations by just recompiling/relinking. (Caveat: the ABI historically differed, so you relink; an ABI-standardization effort is changing this, but treat them as "recompile to switch.")

Why do multiple alternatives exist? History (LAM vs. MPICH camps), vendor incentive (a vendor tunes an MPI for its own fabric/CPU and ships it), research (Open MPI's modular design is a platform for experiments), and licensing/support (some sites want commercial support). They converge on the same API but compete on performance, hardware coverage, and features.

Transform user code → physical communication (the differentiator)

This is the crux of your question. Internally each MPI library has a component framework that maps the abstract operation onto hardware:

  • Open MPI: the pml (point-to-point management layer, e.g. ob1 or cm) chooses an MTL or set of BTLs. vader/sm = shared memory, tcp = sockets, uct/ucx = RDMA via UCX.
  • MPICH (CH4): a netmod — ucx, ofi, or sockets — does the actual transport. CH4 also has a "shm" path for intra-node.

So when you call MPI_Send, the library (1) checks if the peer is local → shared memory; (2) else picks the best NIC transport; (3) chooses eager vs. rendezvous based on size; (4) registers/pins memory if needed; (5) drives verbs/UCX/sockets. Different implementations differ precisely in how cleverly they make those choices.

Running MPI without Slurm: how launch & wire-up actually work

Slurm is just one launcher. MPI predates and doesn't require it. Here's the real machinery on bare-metal Linux boxes.

Step 1 — you need a way to start N processes on M machines

An MPI "job" is just N OS processes (ranks) running your binary, possibly spread across machines. Something has to log into each machine and exec the binary. That something is a process manager / launcher, invoked by mpirun/mpiexec. Without Slurm, the default bootstrap is SSH (or rsh).

You give it a hostfile:

# hosts.txt — Open MPI style
node01 slots=8
node02 slots=8
node03 slots=8
node04 slots=8
# Launch 32 ranks across the 4 nodes
mpirun --hostfile hosts.txt -np 32 ./my_app

# MPICH/Hydra equivalent
mpiexec -f hosts.txt -n 32 ./my_app
Your direct question: "Do we SSH into machines and run the same job?"

Essentially yes, but you don't do it manually. mpirun uses its launcher (Open MPI's PRRTE/ORTE, MPICH's Hydra) to SSH into each host and exec your binary the right number of times. All ranks run the same binary (SPMD model) and discover which rank they are at runtime via MPI_Comm_rank. They are not independent identical jobs — they form one communicator and coordinate.

Step 2 — prerequisites for the SSH bootstrap to work

  • Passwordless SSH from the launch node to every compute node (and often between them). Set up keys + authorized_keys; SSH agent or host-based auth.
  • Same binary at the same path on every node — usually via a shared filesystem (NFS/Lustre) mounted identically, or copy the binary everywhere.
  • Consistent environment: LD_LIBRARY_PATH, the MPI libs, and any CUDA libs must be found on each node. Non-interactive SSH often doesn't source your .bashrc — a classic "works on head node, fails on compute node" bug. Use mpirun -x VAR (Open MPI) or set them in a sourced profile.
  • Clock & hostname resolution: nodes must resolve each other's hostnames (DNS or /etc/hosts) and reach each other on the fabric.

Step 3 — what happens at launch (the wire-up)

  1. mpirun starts a head/launcher process on your node.
  2. It SSHes to each host and starts a per-node daemon (Open MPI: prted/orted; MPICH: a Hydra proxy hydra_pmi_proxy).
  3. Each daemon forks/execs the requested number of rank processes, assigning each a global rank and setting environment so the MPI runtime knows the layout.
  4. Business-card exchange via PMI/PMIx. Each rank, inside MPI_Init, publishes its connection info (IB GID/LID + QP numbers, or TCP host:port) into a key-value store provided by the launcher (PMI, now PMIx). Ranks then look up peers' info and establish the actual transport connections (QPs, sockets). This out-of-band bootstrap is how RDMA peers find each other before any RDMA exists.
  5. From then on, data flows directly peer-to-peer over the fabric; the daemons mostly handle I/O forwarding (stdout/stderr) and teardown.
Under Slurm it's the same idea, just a different launcher

With Slurm you'd use srun ./my_app (or mpirun inside an allocation). Slurm replaces SSH-bootstrap with its own daemons (slurmd) and provides the PMI/PMIx key-value store itself. The MPI wire-up logic is identical — only the "who starts the processes and hosts the KVS" differs. This is why understanding the no-Slurm path makes Slurm obvious.

Alternative launchers / no-SSH options

  • Hydra (MPICH) supports bootstrap servers: ssh, rsh, fork (single node), slurm, pbs, lsf via -bootstrap.
  • Single machine: just mpirun -np 8 ./app — it forks locally, no SSH, all ranks talk over shared memory. Great for development.
  • Containers/cloud: same hostfile+SSH pattern inside the container network; or use the scheduler's PMIx integration.
# Single-node quick test (no SSH, shared-memory transport)
mpirun -np 4 ./my_app

# Pin ranks to cores & report binding (Open MPI) — perf hygiene
mpirun --hostfile hosts.txt -np 32 \
       --map-by socket --bind-to core --report-bindings ./my_app

The operators: what each one does at the low level

MPI operations split into point-to-point (two named ranks) and collective (all ranks in a communicator). Plus one-sided RMA. Here's each, with the mechanism underneath.

The matching engine (why two-sided is "expensive")

Every incoming message carries an envelope: (communicator, source rank, tag). The receiver maintains two queues: posted receive queue (receives you posted that haven't matched yet) and unexpected message queue (messages that arrived before a matching receive). On each event the engine walks the queue to find a match. This software matching is the cost two-sided messaging adds on top of raw RDMA — and is exactly what one-sided models (SHMEM/RMA) avoid. UCX implements tag-matching as a first-class primitive (ucp_tag_send) precisely so MPI doesn't have to reimplement it.

Collectives — and what they do underneath

Collectives are not magic hardware ops (usually); the library composes them from point-to-point sends using clever algorithms chosen by message size and topology. Knowing the algorithm explains the performance.

Why ring-allreduce is bandwidth-optimal (the idea NCCL borrowed)

Naive allreduce (everyone sends to a root, root sums, broadcasts back) makes the root a bottleneck: it moves O(P·N) data through one link. Ring-allreduce arranges P ranks in a logical ring and runs two phases:

  1. Reduce-scatter (P−1 steps): the array is split into P chunks. Each step, rank r sends one chunk to r+1 and receives + adds a chunk from r−1. After P−1 steps each rank owns the fully-reduced value for one chunk.
  2. Allgather (P−1 steps): circulate those finished chunks around the ring so everyone ends with the whole reduced array.

Each rank sends ≈2N(P−1)/P bytes total — essentially 2N regardless of P, and every link is busy simultaneously. That bandwidth-optimality is why both large-message MPI_Allreduce and NCCL use ring/tree variants. This is also why allreduce performance is the headline metric for distributed training.

One-sided / RMA (MPI-3)

MPI also has a one-sided API that maps onto RDMA: MPI_Win_create exposes a memory window; then MPI_Put, MPI_Get, MPI_Accumulate read/write remote memory with no matching receive, synchronized by MPI_Win_fence / lock-unlock / flush. This is MPI converging toward the SHMEM model — useful when access patterns are irregular and you don't want the remote rank to post receives.

Buffering: does send/isend/recv allocate buffers? (eager vs. rendezvous)

This is the heart of MPI's internal design. The answer is "it depends on message size, and that threshold is the whole game." MPI uses two protocols, chosen automatically by a tunable eager limit (often ~16–64 KB).

Eager protocol (small messages)

Goal: minimize latency. The sender just throws the data at the receiver without asking permission.

  1. Sender copies the payload into a pre-allocated internal eager buffer (or sends inline) and fires it with the envelope.
  2. If the receiver hasn't posted a matching Recv yet, the data lands in the receiver's unexpected-message buffer — the library does allocate/manage memory here, on the receive side.
  3. When the matching receive is posted, the lib copies from the unexpected buffer into the user's buffer.

So for eager: yes, the library allocates and copies through internal buffers. MPI_Send can return almost immediately because the data is already buffered. Cost: extra memory copies + bounded buffer space (flooding a receiver with unexpected eager messages can exhaust it — a real failure mode).

Rendezvous protocol (large messages)

Goal: maximize bandwidth and avoid copying megabytes. The sender asks permission first so it can do a zero-copy RDMA straight from user buffer to user buffer.

  1. Sender sends a small RTS (ready-to-send) control message with the envelope and its buffer address/rkey.
  2. Receiver, once it has a matching posted receive, replies CTS (clear-to-send) with its destination buffer address/rkey (and registers that buffer with the NIC if not already).
  3. One side issues an RDMA WRITE (or READ) moving the payload directly between the registered user buffers — no intermediate copy.
  4. A completion notification tells both sides it's done.

So for rendezvous: no payload buffering/copy — but it requires memory registration (pinning) of both buffers and an extra handshake (higher latency, amortized over the large transfer).

Now, your specific calls
  • MPI_Send / small: copies into internal eager buffer, returns fast. Allocates/uses internal buffer.
  • MPI_Send / large: rendezvous; no payload buffer, but blocks until handshake + transfer can proceed (buffer reusable on return).
  • MPI_Isend: same protocol choice, but returns a request immediately. The buffer is not safe until MPI_Wait. For large/rendezvous, the actual transfer happens during Wait or when the progress engine runs.
  • MPI_Recv / MPI_Irecv: posts your buffer into the matching engine. If data already arrived eagerly, it copies from the unexpected buffer; if rendezvous, it sends CTS and the RDMA targets your buffer directly. Irecv doesn't allocate payload buffers — it registers your buffer (for rendezvous) and queues it.
The design choice, stated plainly

The eager/rendezvous split is a latency-vs-bandwidth + copy-vs-registration trade. Small messages: a copy is cheaper than a handshake → eager. Large messages: a handshake + registration is cheaper than copying megabytes → rendezvous/zero-copy. The eager limit tunable (MPICH_*_EAGER_LIMIT, Open MPI BTL eager_limit, UCX UCX_*_ZCOPY_THRESH/RNDV_THRESH) lets you move the crossover for your workload.

Memory registration & the registration cache — why pinning dominates the discussion

RDMA needs buffers pinned and mapped in the NIC (ibv_reg_mr), which is a slow kernel operation. If you registered/deregistered on every large send, rendezvous would be miserable. So implementations keep a registration cache (rcache): once a user buffer is registered, the mapping is cached keyed by address range, and reused on subsequent transfers. This is fast but introduces the infamous bug class: if your allocator frees and remaps that virtual address to different physical pages, a stale cache entry sends to the wrong memory. UCX/MPICH hook the allocator and madvise to invalidate stale entries; this is also why some sites set MALLOC_TRIM_THRESHOLD_=-1 or use --mca mpi_leave_pinned 1.

Buffered send (MPI_Bsend) — the one case where you supply the buffer

With MPI_Bsend you call MPI_Buffer_attach(buf, size) to hand MPI a chunk of your memory; subsequent buffered sends copy into it and return instantly, decoupling you from the receiver entirely. You're responsible for sizing it (use MPI_BSEND_OVERHEAD per message). It's rarely needed but is the explicit, user-controlled buffering mode — contrast with the implicit internal buffering of eager MPI_Send.

TCP or InfiniBand by default? Why, and how to select

If a usable high-speed fabric is present, every serious MPI picks it automatically over TCP. The selection logic, roughly:

  1. For each peer, is it on the same node? → use shared memory (vader/sm/CMA/XPMEM). Fastest, always preferred intra-node.
  2. Else, probe available devices. If an InfiniBand/RoCE HCA (or UCX/OFI reports an RDMA-capable provider) is up and reachable → use it.
  3. Only if no RDMA fabric is found → fall back to TCP sockets over whatever Ethernet exists.

Why prefer IB? 5–25× lower latency, higher bandwidth, kernel-bypass (no CPU on the data path), and one-sided RDMA for rendezvous. TCP is the safety net, not the choice.

The trap: silently running over TCP

The most common "my cluster is slow" bug is MPI falling back to TCP because the IB device wasn't detected, the subnet manager (OpenSM) isn't running, the port is DOWN, or UCX/OFI wasn't compiled in. Always verify the chosen transport — don't assume.

GPU-aware MPI: passing device pointers

What "CUDA-aware" / "GPU-aware" means

A GPU-aware MPI lets you pass a device pointer (GPU VRAM address) directly to MPI_Send/Recv etc. Without it, you must cudaMemcpy to host memory, MPI the host buffer, then cudaMemcpy back on the other side — three steps, host bounce, no overlap. With it, MPI handles the GPU memory itself.

Under the hood, GPU-aware MPI chooses the best path based on where the two buffers live:

  • Same node, NVLink/PCIe peers: uses CUDA IPC to copy GPU↔GPU directly (over NVLink if present), never touching host memory.
  • Different nodes, GPUDirect RDMA available: registers the GPU buffer with the HCA and the NIC DMAs straight from VRAM across IB/RoCE — no host bounce. This is the fast path.
  • No GDR: falls back to staging through a pinned host bounce buffer (a pipelined cudaMemcpy + network), which the library does internally so your code still just passes the device pointer.
// GPU-aware MPI: device pointers passed directly
float *d_buf; cudaMalloc(&d_buf, n*sizeof(float));
// no cudaMemcpy to host needed:
MPI_Send(d_buf, n, MPI_FLOAT, peer, tag, MPI_COMM_WORLD);
# Enable / verify the GPU path
ompi_info --parsable | grep -i cuda          # is Open MPI built CUDA-aware?
UCX_TLS=rc,cuda,cuda_copy,cuda_ipc,gdr_copy ./app
mpirun -x UCX_RNDV_SCHEME=get_zcopy ...       # tune GPU rendezvous scheme
Why this still isn't enough for deep learning — foreshadowing NCCL

GPU-aware MPI moves GPU buffers efficiently, but it does not understand the GPU interconnect topology when building collectives, and it can't run the reduction on the GPU in a fused, kernel-driven way. For an 8-GPU NVSwitch node doing allreduce on gradients, you want an algorithm that maxes NVLink bandwidth and does the sum on-device. That gap is exactly what NCCL was built to fill (Section 06).

Complete worked example: 2D Jacobi stencil with halo exchange

This is the canonical MPI pattern. We solve a 2D heat/Laplace equation with a 5-point stencil, decomposing the grid into horizontal strips (one per rank). Each rank needs the boundary rows of its neighbors — the halo / ghost rows — exchanged every iteration. It's complete and compiles.

/* jacobi2d.c  —  build: mpicc -O3 jacobi2d.c -o jacobi2d
 *               run:   mpirun -np 4 ./jacobi2d
 * 5-point Jacobi on an NxN grid, 1-D row decomposition + halo exchange. */
#include <mpi.h>
#include <stdio.h>
#include <stdlib.h>
#include <math.h>

#define N 512          /* global grid is N x N (interior) */
#define MAXIT 10000
#define TOL 1e-5

int main(int argc, char **argv) {
    MPI_Init(&argc, &argv);
    int rank, nproc;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &nproc);

    /* --- domain decomposition: split N rows across nproc ranks --- */
    int base = N / nproc, rem = N % nproc;
    int nrows = base + (rank < rem ? 1 : 0);   /* local interior rows */

    /* allocate (nrows+2) x (N+2): +2 for halo rows (top/bottom) & side borders */
    int ld = N + 2;                          /* leading dim (row length) */
    double *u    = calloc((nrows+2)*ld, sizeof(double));
    double *unew = calloc((nrows+2)*ld, sizeof(double));

    /* boundary condition: top global edge = 1.0 (rank 0 owns it) */
    if (rank == 0)
        for (int j = 0; j < ld; j++) u[0*ld + j] = 1.0;

    /* who are my neighbors? MPI_PROC_NULL = no-op send/recv (clean edges) */
    int up   = (rank == 0)        ? MPI_PROC_NULL : rank - 1;
    int down = (rank == nproc-1) ? MPI_PROC_NULL : rank + 1;

    double gdiff = 1.0;
    int it = 0;
    while (it < MAXIT && gdiff > TOL) {
        /* ---- HALO EXCHANGE: trade boundary rows with neighbors ----
         * Sendrecv avoids deadlock: send my top interior row up,
         * receive into my bottom halo from down, and vice-versa.  */
        MPI_Sendrecv(&u[1*ld],       ld, MPI_DOUBLE, up,   0,
                     &u[(nrows+1)*ld], ld, MPI_DOUBLE, down, 0,
                     MPI_COMM_WORLD, MPI_STATUS_IGNORE);
        MPI_Sendrecv(&u[nrows*ld],   ld, MPI_DOUBLE, down, 1,
                     &u[0*ld],       ld, MPI_DOUBLE, up,   1,
                     MPI_COMM_WORLD, MPI_STATUS_IGNORE);

        /* ---- local 5-point update on interior cells ---- */
        double ldiff = 0.0;
        for (int i = 1; i <= nrows; i++)
            for (int j = 1; j <= N; j++) {
                double v = 0.25 * (u[(i-1)*ld+j] + u[(i+1)*ld+j] +
                                   u[i*ld+(j-1)] + u[i*ld+(j+1)]);
                ldiff += fabs(v - u[i*ld+j]);
                unew[i*ld+j] = v;
            }

        /* swap buffers */
        double *t = u; u = unew; unew = t;

        /* ---- global convergence test: sum local diffs across all ranks ---- */
        MPI_Allreduce(&ldiff, &gdiff, 1, MPI_DOUBLE, MPI_SUM, MPI_COMM_WORLD);
        it++;
    }

    if (rank == 0)
        printf("converged after %d iters, residual=%.3e\n", it, gdiff);

    free(u); free(unew);
    MPI_Finalize();
    return 0;
}
What to notice (the transferable lessons)
  • Ghost/halo rows are the universal stencil pattern: pad your local array by the stencil radius, fill the pad from neighbors each step.
  • MPI_Sendrecv is used to avoid the classic deadlock where every rank blocks in Send waiting for a matching Recv that no one has reached yet.
  • MPI_PROC_NULL elegantly handles physical boundaries — the call becomes a no-op, no special-casing edges.
  • MPI_Allreduce turns per-rank local residuals into one global convergence number every rank agrees on.
Performance upgrade: overlap halo exchange with interior compute (non-blocking)

The version above is correct but synchronous: ranks idle during the exchange. The optimized pattern posts non-blocking MPI_Irecv/MPI_Isend for halos, immediately computes the interior cells that don't depend on halos, then MPI_Waitall and finishes the boundary cells. This hides communication behind computation:

MPI_Request reqs[4];
MPI_Irecv(&u[(nrows+1)*ld], ld, MPI_DOUBLE, down, 0, comm, &reqs[0]);
MPI_Irecv(&u[0*ld],       ld, MPI_DOUBLE, up,   1, comm, &reqs[1]);
MPI_Isend(&u[1*ld],       ld, MPI_DOUBLE, up,   0, comm, &reqs[2]);
MPI_Isend(&u[nrows*ld],   ld, MPI_DOUBLE, down, 1, comm, &reqs[3]);
/* compute rows 2..nrows-1 here (independent of halos) ... */
MPI_Waitall(4, reqs, MPI_STATUSES_IGNORE);
/* now compute boundary rows 1 and nrows that needed halos ... */

For a 2-D decomposition (better surface-to-volume ratio at large scale) you'd use MPI_Cart_create + MPI_Cart_shift to get neighbors in both dimensions, and MPI_Type_vector to describe non-contiguous column halos as a derived datatype.

Section 05

PGAS & OpenSHMEM — the one-sided model

To understand NVSHMEM you must first understand the model it implements. PGAS (Partitioned Global Address Space) is the alternative to message passing: instead of "send me a message," you treat remote memory as if it were part of one big global address space — partitioned because each chunk physically lives on one node, but addressable by all.

The SHMEM mental model

  • Symmetric heap. Every process (called a PE, processing element) allocates the same variables at the same symmetric offsets via shmem_malloc. So a pointer's offset is meaningful on every PE — PE 3 knows where variable X lives on PE 7.
  • One-sided put/get. The core ops are shmem_put(dest, src, n, pe) (write into PE pe's memory) and shmem_get(dest, src, n, pe) (read from it). The target PE does nothing — no matching receive, no CPU involvement. This maps directly onto an RDMA WRITE/READ.
  • Explicit synchronization. Because there's no matching, you synchronize separately: shmem_quiet (ensure my puts completed), shmem_barrier_all, shmem_fence (order puts), and atomics (shmem_atomic_add, compare-swap) for coordination.
Why one-sided exists at all

For irregular or data-dependent communication — graph algorithms, sparse solvers, dynamic load balancing — you often don't know in advance who will send you what, so posting matching receives is awkward. One-sided "just reach into their memory" fits these patterns and removes the matching overhead. The cost is you manage consistency/ordering yourself. SHMEM put/get latency can beat MPI for fine-grained access because there's no envelope matching.

OpenSHMEM (standardized 2010) is to SHMEM what MPI is to PVM: a community spec with multiple implementations (Sandia SOS, Open MPI's OSHMEM, Cray SHMEM), most of which sit on UCX or libfabric underneath — because put/get/atomics are exactly the RMA primitives those layers expose. GASNet played the same "portable conduit" role for UPC and other PGAS languages and is a direct conceptual ancestor of UCX.

/* OpenSHMEM: one-sided write, no receive on the target */
int *data = shmem_malloc(n*sizeof(int));  /* symmetric allocation */
shmem_int_put(data, local, n, /*to PE*/ 3);   /* RDMA WRITE into PE 3 */
shmem_quiet();                              /* ensure completion */
shmem_barrier_all();

Section 06

NCCL — GPU collective communication

NCCL (NVIDIA Collective Communications Library, "nickel," 2015) is the comms backbone of essentially all multi-GPU deep learning (PyTorch DDP/FSDP, Megatron, DeepSpeed). Let's answer your questions precisely.

Is it GPU-to-GPU only, or more?

NCCL is specifically about moving and reducing data that lives in GPU memory, between GPUs — both intra-node (over NVLink/NVSwitch/PCIe) and inter-node (over IB/RoCE with GPUDirect RDMA). So it spans the whole network, but the endpoints are always GPU buffers. It is not a general CPU message-passing library; it has no rich point-to-point tag-matching API like MPI. It does have ncclSend/ncclRecv (added later) for point-to-point, but its reason for existing is collectives.

The API — is it like MPI?

Deliberately familiar, but narrower and GPU/stream-centric:

NCCLMPI analogueNote
ncclAllReduceMPI_AllreduceThe headline op for gradient averaging.
ncclBroadcastMPI_Bcast
ncclReduce / ncclReduceScatterMPI_Reduce / MPI_Reduce_scatterReductions run on the GPU.
ncclAllGatherMPI_Allgather
ncclSend / ncclRecvMPI_Send / MPI_RecvPoint-to-point; grouped for pipelines.

Key API differences from MPI:

  • Every call takes a CUDA stream and is asynchronous with respect to the host — it enqueues a comm kernel on the stream, ordered with your compute kernels. This stream integration is the big ergonomic win for GPU pipelines.
  • Setup uses a unique ID broadcast out-of-band (often via MPI or a file) + ncclCommInitRank, rather than MPI's communicator world. People frequently use MPI just to bootstrap NCCL, then do all data movement with NCCL.
  • No datatypes/derived-types machinery, no tags, no wildcard matching — it's a lean collective API.

Physical layer & topology awareness

This is NCCL's superpower. At init it detects the intra-node topology (which GPUs share NVLink, NVSwitch, PCIe switches, which NIC is closest to which GPU) and builds optimized rings and trees across the whole job that maximize bandwidth on the fastest links. Transports it picks among:

  • NVLink / NVSwitch — GPU↔GPU on the same node, the fast path.
  • PCIe / shared host memory — when no NVLink between a pair.
  • GPUDirect RDMA over IB/RoCE — inter-node, NIC DMAs straight from VRAM.
  • P2P via CUDA for direct peer access; SHM staging when needed.

It runs the reduction arithmetic inside CUDA kernels on the GPU, fused with the data movement, using ring-allreduce / tree-allreduce (the bandwidth-optimal algorithms from Section 04) sized to the detected topology.

Why NCCL when we already have GPU-aware MPI?

This is the crux. GPU-aware MPI can move GPU buffers, but NCCL solves different problems:

  • GPU-topology-aware collective construction. NCCL builds rings/trees that specifically exploit NVLink/NVSwitch bandwidth and NIC-GPU affinity. Generic MPI collectives were designed around CPU/network topology and rarely match this.
  • On-GPU, kernel-fused reduction. NCCL does the sum/max in a CUDA kernel as data streams, keeping the GPU busy; MPI would often reduce on the CPU or with less GPU integration.
  • CUDA stream integration. NCCL ops slot into your stream and overlap naturally with compute kernels — exactly what a training step needs. MPI lives on the host timeline.
  • Tuned for the DL access pattern: a few very large collectives (gradients) repeated millions of times, where squeezing allreduce bandwidth is everything.

So it's not "NCCL replaces MPI" — many apps use MPI for control/CPU + NCCL for GPU collectives. (AMD's RCCL is the ROCm port of the same design; Microsoft's MSCCL/MSCCL++ adds custom programmable collective algorithms on top.)

Section 07

NVSHMEM — OpenSHMEM, but inside the GPU kernel

NVSHMEM is NVIDIA's implementation of the OpenSHMEM (PGAS, one-sided) model for GPU clusters. Its defining feature: communication can be initiated from within a CUDA kernel by the GPU threads themselves, not just orchestrated from the host. That's a genuinely different execution model from MPI and NCCL.

The user API — does it have send/recv?

No — it's the SHMEM one-sided model, not the MPI two-sided one. There is no send/recv matching. The vocabulary is put/get/atomics over a symmetric heap that lives in GPU memory:

  • nvshmem_malloc — allocate a symmetric buffer in GPU memory across all PEs.
  • nvshmem_put / nvshmem_get — one-sided write/read of a remote PE's GPU memory.
  • Device-side variants callable from inside a kernel: nvshmem_p (single element), nvshmemx_putmem_block/warp (cooperative by a thread block/warp), nvshmem_fence, nvshmem_quiet, nvshmem_barrier_all, plus atomics and (newer) collectives like nvshmem_float_sum_reduce.
// NVSHMEM: a CUDA kernel that does its OWN communication.
// Thread writes directly into a neighbor GPU's memory — no host, no NCCL call.
__global__ void shift(float *dst, float *src, int n, int peer) {
    int i = blockIdx.x*blockDim.x + threadIdx.x;
    if (i < n)
        nvshmem_float_p(&dst[i], src[i], peer);  // one-sided put, from device
}
// host side:
nvshmem_init();
float *buf = (float*)nvshmem_malloc(n*sizeof(float)); // symmetric GPU heap
shift<<<blocks,threads>>>(buf, src, n, peer);
nvshmemx_barrier_all_on_stream(stream);
Why GPU-initiated communication matters (vs NCCL)

With NCCL/MPI, the host launches a kernel, waits, then issues a collective — communication and computation are separate phases on the timeline, with kernel-launch and synchronization overhead between them. With NVSHMEM, a long-running kernel can compute and exchange data with neighbors as it goes (fine-grained, data-driven), eliminating launch/sync overhead and enabling deep overlap. This is ideal for strong-scaling stencils, FFTs, graph traversal, and increasingly for fused communication in transformer layers (e.g. overlapping all-to-all in MoE / tensor-parallel matmuls).

Physical-layer selection

NVSHMEM maps put/get onto whatever moves GPU memory fastest between the two PEs:

  • NVLink / NVSwitch intra-node: a put becomes a direct load/store or copy over NVLink into the peer GPU's address space — extremely low latency, which is what makes in-kernel single-element puts viable.
  • PCIe P2P intra-node when no NVLink between the pair.
  • GPUDirect RDMA over IB/RoCE inter-node — and crucially, GPU-initiated RDMA via mechanisms like IBGDA (InfiniBand GPUDirect Async), where the GPU itself rings the NIC doorbell, so a kernel can fire network transfers without a host round-trip.

"MEM" — the lower-level memory angle

The "MEM" in the name signals that the abstraction is memory, not messages. Things that happen at this lower level:

  • The symmetric heap is a chunk of each GPU's VRAM, registered for peer/NIC access at allocation, with consistent offsets so addresses are computable across PEs.
  • Address translation: a remote reference (offset, PE) is resolved to either a peer GPU virtual address (NVLink) or a registered region + rkey (RDMA).
  • Memory ordering becomes your responsibility — nvshmem_fence orders puts to one PE, nvshmem_quiet waits for global completion. Because there's no receive to imply ordering, you reason about it like a relaxed memory model. Atomics (nvshmem_atomic_add, CAS) provide the synchronization primitives.
When NVSHMEM, when NCCL?

Use NCCL for big bulk collectives (gradient allreduce) — it's tuned and simple. Use NVSHMEM when you need fine-grained, in-kernel, one-sided communication fused with computation, or custom communication patterns NCCL doesn't express. They coexist; some libraries use NVSHMEM to build faster custom collectives than stock NCCL for specific shapes.

UCX — the unifying abstraction

Now the destination. UCX (Unified Communication X) is exactly the layer you described: above the transport APIs (TCP, verbs, CUDA), below the programming models (MPI, OpenSHMEM, NCCL-net plugins). It's an open-source framework (UCF Consortium: NVIDIA/Mellanox, ORNL, IBM, others) extracting the transport plumbing that every model used to write itself.

The internal architecture (UCP / UCT / UCS)

  • UCT (transport layer) — thin, uniform wrappers around each hardware mechanism: post a zero-copy send on RC verbs, do an RDMA put, a CUDA IPC copy, a TCP send. UCT exposes capabilities ("I can do zero-copy RMA," "max eager size is X"). One UCT module per transport.
  • UCP (protocol layer) — the brain. It implements the logic the models need on top of raw transports: tag matching, automatic eager vs. rendezvous selection, message fragmentation, multi-rail (striping one transfer across several NICs), the registration cache, and crucially picking the best transport per peer (SHM intra-node, NVLink for GPU peers, RC verbs inter-node) — all behind one API.
  • UCS (services) — common utilities: data structures, memory hooks, config, async.

So an MPI implementation calls ucp_tag_send_nbx(...) with a buffer, destination, and tag. UCP figures out: is the peer local? GPU memory? big enough to warrant rendezvous? which rail? — and drives the right UCT transport. MPI no longer contains any verbs or socket code.

Your exact question, answered

"Does that mean MPI doesn't have to talk to physical layers? MPI ↔ UCX ↔ physical?" — Yes, precisely. When Open MPI's pml ucx or MPICH's ch4:ucx netmod is used, MPI delegates all transport decisions to UCX. MPI's job shrinks to: implement MPI semantics (communicators, datatypes, collective algorithms) and translate each operation into UCP calls. UCX owns the wire. OpenSHMEM does the same (its put/get → UCP RMA). NCCL can use UCX through its network plugin for the inter-node leg.

Why abstract UCX out? (repeat code — and more)

You guessed "due to repeated code." That's the biggest reason, but not the only one:

  1. Eliminate duplicated, hard plumbing. Before UCX, MPICH, Open MPI, MVAPICH, and every SHMEM each maintained their own verbs driver, their own registration cache, their own eager/rendezvous engine, their own GPUDirect support. That's the same difficult, bug-prone code written 5+ times. UCX writes it once, well, and everyone shares it. (The really nasty bits — registration caches, RoCE quirks, multi-rail — benefit enormously from being centralized.)
  2. New hardware integrated once. When a new NIC, NVLink generation, or GPUDirect feature ships, the vendor adds one UCT module and every consumer (all MPIs, SHMEM, RAPIDS) gets it for free — instead of N separate porting efforts. This is the decisive practical reason vendors backed it.
  3. Consistent, expert performance. The protocol decisions (thresholds, multi-rail striping, transport selection) are subtle; concentrating them lets a focused team tune them better than each model could individually, and improvements lift everyone.
  4. A clean, model-neutral API. UCP deliberately offers both tag-matched two-sided (for MPI) and RMA/atomics one-sided (for SHMEM) plus active messages — so it can serve message-passing and PGAS models from one codebase. That generality is why it could replace the bespoke layers of both worlds.
The historical throughline

This is the same consolidation that happened with GASNet for PGAS languages in the 2000s, now done broadly and GPU-aware. UCX is the maturation of the pattern the whole timeline has been building toward: push the common, hardware-touching code into one shared, vendor-maintained layer.

The cousins: libfabric, GASNet, Gloo, RCCL, MSCCL

UCX isn't alone. These are worth knowing because you'll meet them in the wild and they sharpen what UCX is.

libfabric / OFI — UCX's main peer

libfabric (the Open Fabrics Interfaces project) occupies the same layer as UCX: a transport-neutral framework below MPI/SHMEM, above hardware. It exposes providers (verbs, tcp, sockets, psm2 for Omni-Path, cxi for HPE Slingshot, efa for AWS, shm). Intel MPI, Cray MPICH, and AWS's libraries are built on it.

They overlap heavily and compete; which one you get is mostly determined by your vendor/fabric. On NVIDIA-IB clusters you'll typically be on UCX; on HPE Cray EX or AWS you'll be on libfabric. Both answer your "MPI ↔ X ↔ physical" picture identically — they're interchangeable abstraction layers.

GASNet-EX — the PGAS conduit

The portable communication layer beneath UPC, Legion, Chapel, and other PGAS languages. Older sibling in spirit to UCX, focused on one-sided RMA + active messages for partitioned global address space runtimes. Conceptually it pioneered the "extract the conduit" idea UCX generalized.

Gloo, RCCL, MSCCL — the DL-collective neighbors of NCCL

  • Gloo (Meta): CPU- and GPU-capable collective library, the original CPU backend for PyTorch torch.distributed; simpler than NCCL, used where NCCL isn't available or for CPU tensors.
  • RCCL: AMD's drop-in NCCL clone for ROCm GPUs — same API, same ring/tree ideas, AMD interconnects (Infinity Fabric/xGMI).
  • MSCCL / MSCCL++ (Microsoft): a layer to express custom, programmable collective algorithms (and a GPU-initiated comms primitive set) when NCCL's built-ins aren't optimal for a given topology or shape.
Where PyTorch fits

torch.distributed is an even higher layer: it picks a backend — nccl (GPU, the default), gloo (CPU/fallback), or mpi — and exposes all_reduce, all_gather, etc. So the full stack for distributed training is often: PyTorch DDP → NCCL → (UCX or NCCL's own net) → GPUDirect RDMA → IB/RoCE + NVLink. Every layer in this guide is somewhere in that line.

The three bugs you'll actually hit most
  1. Silent TCP fallback — "slow but works." Always verify the chosen transport with INFO logs.
  2. Wrong interface picked — comms binding to docker0/lo instead of ib0. Set *_IFNAME/NET_DEVICES explicitly.
  3. Collective mismatch / one rank diverged — a hang where ranks disagree on what collective to run. Check control flow is identical across ranks.
