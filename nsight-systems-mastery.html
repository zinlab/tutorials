1 Capture
Launching Nsight Systems

Before you can interpret anything, you need a clean capture. The goal is to record 2–3 full training iterations (skip the first for warmup) with all the trace sources that matter for PyTorch workloads.

The baseline capture command Capture 
nsys profile \
  --trace=cuda,nvtx,osrt,cudnn,cublas  # what to trace
  --cuda-memory-usage=true              # track cudaMalloc/cudaFree
  --gpu-metrics-device=all              # SM activity, power, temp
  --gpu-metrics-frequency=10000         # 10kHz sampling (100μs)
  --capture-range=cudaProfilerApi      # start/stop from code
  --output=baseline                     # output filename
  --force-overwrite=true                # replace existing
  python train.py

Trace flags explained

nvtxNVTX annotations (user-defined ranges)Maps timeline regions to your code: "forward", "backward", "optimizer step"

If the training script doesn't have NVTX annotations, wrap the key phases with torch.cuda.nvtx.range_push("forward") / range_pop(), or use with torch.autograd.profiler.emit_nvtx(): as a quick alternative that auto-labels autograd operations.
Programmatic capture control Capture

Instead of capturing the entire run (which includes slow imports, data loading, model init), use the CUDA Profiler API to bracket just the iterations you care about:

import torch.cuda.profiler as profiler
import torch.cuda

for epoch in range(epochs):
    for i, batch in enumerate(loader):
        if i == 5:
            profiler.start()       # start capture
        if i == 8:
            profiler.stop()        # stop capture

        # ... training step ...

Pair this with --capture-range=cudaProfilerApi in the nsys command. This gives you exactly 3 clean iterations (5, 6, 7) without startup noise. The resulting .nsys-rep file will be much smaller and faster to open.

Always skip at least the first 2–3 iterations. PyTorch's CUDA caching allocator, cuDNN autotuner, and torch.compile all do one-time work on the first pass that is not representative of steady-state performance.
Headless (SSH) capture and transfer Capture

On HPC clusters and cloud GPU instances, you typically SSH in and don't have a GUI. The workflow is:

# On the remote GPU node:
nsys profile --output=/tmp/profile_run1 ... python train.py

# Generates /tmp/profile_run1.nsys-rep (typically 50–500MB)
# Transfer to your local machine:
scp gpu-node:/tmp/profile_run1.nsys-rep .

# Open in Nsight Systems GUI on your local workstation

Alternatively, for a quick summary without the GUI:

nsys stats profile_run1.nsys-rep

This prints the kernel statistics, CUDA API summary, and OS runtime summary directly in the terminal — useful for a fast check before pulling the full file.

3 Triage
Where to look first

Don't try to understand everything at once. Follow this 3-step triage:

Step 1

Find one full iteration

Step 2

Measure GPU idle time

Step 3

Classify the gaps

Step 1 — Use NVTX markers (or the repeating pattern of kernel clusters) to identify one full iteration: data load → forward → loss → backward → optimizer step. Zoom to fit exactly one iteration. Measure its wall time with the ruler — this is your baseline.

Step 2 — Look at the GPU kernel row and estimate (or measure) the fraction of the iteration where the GPU row is empty. This is your "GPU idle percentage." Right-click the GPU row and select "Show idle gaps" to get a sorted list. The sum of those gaps divided by the iteration wall time gives you the fraction of time the GPU is waiting.

Step 3 — Classify each significant gap by what's happening on the other rows during that gap:

4 Diagnosis
The six patterns

Every slow PyTorch training run falls into one (or a combination) of these six patterns. Each has a distinct visual signature in Nsight Systems.

Pattern 1: Data starvation / CPU bottleneck

The GPU finishes a training step and then sits idle while the CPU prepares the next batch. This is the most common problem in practice.

CPU thread data loading / preprocess data loading / preprocess
GPU kernels forward + backward forward + ...

Red hash marks = GPU idle time. CPU is busy with data loading while GPU waits.

What you see

Large gaps on the GPU row (10ms–1s+). During these gaps, the CPU row shows solid activity — read(), poll(), futex() in the OS Runtime trace, or Python DataLoader worker threads churning. The gap appears between iterations, aligned with the data loading phase of the NVTX markers.

Root cause

The DataLoader can't deliver batches fast enough. Common reasons: too few num_workers, slow disk I/O (spinning HDD, NFS), CPU-bound augmentations (random crop, color jitter on CPU), data decompression (JPEG decode), or the GIL serializing multi-threaded transforms.

Fixes

Increase num_workers — start with num_workers = num_cpus // num_gpus. Set pin_memory=True and persistent_workers=True.
Move transforms to GPU — use NVIDIA DALI, Kornia, or torchvision's v2 transforms with GPU support.
Use faster storage — local NVMe SSD over NFS. Pre-convert data to optimized formats (WebDataset, FFCV, TFRecord).
Prefetch — set prefetch_factor=2 (or higher) so the next batch is ready before the GPU finishes.
Profile the loader separately — time just the data loading loop without GPU work to isolate disk vs CPU bottleneck.

Pattern 2: Kernel launch overhead

Thousands of tiny CUDA kernels each take only a few microseconds, but the launch latency between them adds up. The GPU looks busy but achieves low throughput.

CUDA API
GPU kernels
Power (W) ~180W (should be 500W+)

Dense tiny kernels with gaps between each — GPU looks busy, power says otherwise

What you see

The GPU row looks fully packed at normal zoom. But zoom in and you see dozens of tiny kernels (2–10μs each) with 1–5μs gaps. The CUDA API row is a solid wall of cudaLaunchKernel. The GPU metrics power trace stays low (150–250W on H100). In the kernel statistics, sort by Count — the top entries have 10,000+ invocations with average duration under 10μs.

Root cause

PyTorch eager mode dispatches each operator as a separate CUDA kernel. Element-wise ops (ReLU, add, multiply, dropout), pointwise activations, and small reductions each launch their own kernel. On H100, the minimum kernel launch overhead is ~3–5μs. If the kernel itself takes 5μs, you spend 50% of time on overhead.

Fixes

torch.compile(model) — The primary fix. It fuses element-wise ops, reduces kernel count by 5–20x, and generates larger optimized kernels. Run torch._dynamo.explain(model)(input) to see what gets fused and what breaks out.
CUDA Graphs — For fixed-shape workloads, record the entire training step as a graph and replay it, amortizing all launch overhead to near-zero. Use torch.cuda.make_graphed_callables().
Manual fusion — If torch.compile can't fuse certain ops (custom kernels, dynamic shapes), write fused Triton kernels for the hot path.

Pattern 3: Synchronization stalls

The CPU forces the GPU pipeline to drain by calling a synchronization operation. Everything stops until all queued GPU work completes.

CPU thread .item() .cpu()
CUDA API cudaStreamSync cudaMemcpy D2H
GPU kernels forward kernels backward kernels more bwd

Each .item() / .cpu() call drains the GPU pipeline, creating idle gaps

What you see

Intermittent gaps on the GPU row, each preceded by a cudaStreamSynchronize or cudaDeviceSynchronize on the CUDA API row. During the gap, the CPU is blocked waiting inside the sync call. After the sync completes, the CPU briefly runs Python code (processing the value it pulled back), then resumes launching kernels. These gaps appear in the middle of forward or backward passes — not at iteration boundaries.

Root cause

Operations that transfer data from GPU to CPU force synchronization: loss.item(), tensor.cpu(), tensor.numpy(), print(tensor), Python-side if tensor > threshold: conditionals. Each one drains the asynchronous kernel queue, waits for all pending GPU work to finish, copies the value to CPU, and only then allows the CPU to continue queueing new work. This kills the overlap between CPU kernel dispatch and GPU execution.

Fixes

Eliminate .item() in the hot loop — accumulate loss as a tensor: running_loss += loss.detach(). Only call .item() every N iterations for logging.
Replace Python conditionals with torch.where — keep the branching on GPU: result = torch.where(tensor > threshold, a, b).
Defer metrics to callbacks — log metrics in a separate non-blocking callback rather than inline in the training step.
Use torch.cuda.Event for timing — avoid torch.cuda.synchronize() for benchmarking. Events measure GPU time without draining the pipeline.

Pattern 4: Memory allocation churn

The GPU stalls while CUDA allocates or frees memory, bypassing PyTorch's caching allocator.

CUDA API cudaMalloc cudaFree cudaMalloc cudaFree cudaMalloc
GPU kernels kernel kernel kernel kernel kernel kernel

cudaMalloc / cudaFree calls correlate with GPU gaps

What you see

GPU gaps that align precisely with cudaMalloc or cudaFree calls on the CUDA API row. These allocations take 1–100ms each (far longer than kernel launches). You may also see them in the Memory Usage row as sharp spikes/drops. Enable --cuda-memory-usage=true to get the memory operations trace.

Root cause

PyTorch's caching allocator normally holds onto freed memory and reuses it. Allocation churn bypasses this when: tensors with varying sizes are created each iteration (dynamic shapes), torch.cuda.empty_cache() is called unnecessarily inside the loop, or code creates new tensors via torch.cat/torch.stack with growing dimensions. The caching allocator can also fragment, forcing new physical allocations even when total free memory exists.

Fixes

Pre-allocate buffers — create output tensors once outside the loop and rewrite operations to use out= parameters.
Pad to fixed sizes — for variable-length sequences, pad to the nearest power of 2 or a fixed bucket size so the allocator can reuse blocks.
Remove torch.cuda.empty_cache() — this is almost never helpful in a training loop. It forces the allocator to release all cached memory back to CUDA, triggering expensive re-allocations.
Set PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:128 — controls allocator fragmentation behavior. Tune this if you see OOM despite available memory.
Inspect with torch.cuda.memory_summary() — shows the allocator's internal state: active blocks, cached blocks, fragmentation.

Pattern 5: Communication bottleneck (multi-GPU)

In DDP/FSDP training, GPUs wait for collective communication (AllReduce, ReduceScatter) to complete before proceeding to the next step.

GPU 0 kernels forward + backward ncclAllReduce optim next fwd
GPU 1 kernels forward + backward ncclAllReduce optim next fwd

NCCL AllReduce dominates the timeline — communication not overlapped with compute

What you see

Large NCCL kernel blocks (ncclAllReduceRingLL, ncclReduceScatter, etc.) that appear after the backward pass completes, occupying 30–60% of the iteration wall time. All GPUs idle until the slowest one finishes its backward pass and enters the collective. The NCCL kernels should ideally overlap with backward compute, but instead they're serialized.

Root cause

DDP communication not overlapping with backward compute. Possible reasons: find_unused_parameters=True (adds overhead to detect which gradients need syncing), improper gradient bucketing (buckets too large = late start, too small = excessive collectives), or the model has sequential bottleneck layers that delay backward completion on some GPUs. For FSDP: full parameter gather happening synchronously before each layer's forward pass.

Fixes

Verify overlap — in a healthy DDP setup, NCCL kernels should start firing on the GPU during the backward pass (gradient buckets are communicated as soon as they're ready). If they only appear after backward, check gradient_as_bucket_view=True.
Set find_unused_parameters=False — if all parameters get gradients every iteration.
Tune bucket size — bucket_cap_mb=25 (default) works for most models. Smaller buckets start communication earlier but add overhead. Profile with different values.
For FSDP: use limit_all_gathers=True — prevents all-gathers from piling up and exhausting memory/bandwidth.
Check NVLink topology — run nvidia-smi topo -m to verify GPUs are connected via NVLink, not PCIe. NCCL over PCIe is 5–10x slower.
NCCL environment tuning — NCCL_ALGO=Ring vs Tree, NCCL_MIN_NCHANNELS, NCCL_BUFFSIZE. Profile with NCCL_DEBUG=INFO to see which algorithm/protocol NCCL selects.

Pattern 6: Low Tensor Core utilization

The GPU runs large kernels (no launch overhead) and stays busy (no gaps), but throughput is far below the hardware ceiling because Tensor Cores aren't being used.

GPU kernels sgemm_128x128 (FP32) volta_sgemm (FP32) sgemm (FP32) sgemm (FP32)
Power (W) ~350W (vs 600W+ with Tensor Cores)

Large GEMM kernels but named "sgemm" (FP32) instead of Tensor Core variants

What you see

Large GEMM/convolution kernels that fill the timeline (no idle gaps). But their names contain sgemm (single-precision GEMM) rather than Tensor Core variants like h884gemm, s16816gemm, or names containing _fp16_/_bf16_. Power draw is moderate (300–400W on H100) rather than high (550W+). The Nsight Compute roofline shows operations far below the Tensor Core ceiling.

Root cause

The model is running in FP32 instead of using mixed precision (FP16/BF16). On H100, the Tensor Core throughput for BF16 is ~4x that of FP32 CUDA cores for matrix operations. Even if AMP is enabled, Tensor Cores require matrix dimensions (M, N, K) to be multiples of 8 — misaligned dimensions cause fallback to CUDA cores.

Fixes

Enable mixed precision — wrap the forward pass with torch.autocast('cuda', dtype=torch.bfloat16) and use GradScaler (for FP16) or just autocast (for BF16, which doesn't need scaling on H100).
Align dimensions to multiples of 8 — embedding dims, hidden sizes, vocabulary size, batch size. Padding a vocab from 30,522 to 30,528 costs negligible memory but enables Tensor Cores for the output projection.
Verify with kernel names — after enabling AMP, re-profile and check that GEMM kernel names now contain h884, 16816, bf16, or fp16. If they still say sgemm, some layers may be excluded from autocasting.
For H100: consider FP8 — using Transformer Engine with FP8 training gives another 2x over BF16 for compatible architectures.

6 Scale
Multi-GPU and NCCL analysis

When profiling 8xH100 DDP/FSDP training, Nsight Systems shows all GPUs on the same timeline. This enables critical cross-GPU analysis.

Multi-GPU capture setup Capture

For torchrun-based distributed training, profile the entire launcher:

nsys profile \
  --trace=cuda,nvtx,osrt,cudnn,cublas \
  --cuda-memory-usage=true \
  --gpu-metrics-device=all \
  --output=multi_gpu_baseline \
  torchrun --nproc_per_node=8 train.py

Each GPU gets its own set of swim lanes in the timeline. You can expand/collapse individual GPU sections. For cleaner analysis, you can also profile a single rank:

# Profile only rank 0 (reduces file size and noise)
CUDA_VISIBLE_DEVICES=0 nsys profile ... \
  torchrun --nproc_per_node=8 train.py

But note: this misses the cross-GPU timing relationships. For communication analysis, you need all ranks.

What to look for across GPUs Interpret

1. Compute skew

All GPUs should finish their backward pass at roughly the same time. If GPU 3 consistently finishes 20% later than the others, all GPUs are throttled to GPU 3's speed (because the AllReduce can't complete until the slowest GPU contributes its gradients). Common causes: thermal throttling on one GPU, data imbalance across ranks, or uneven model partitioning.

2. Communication / compute overlap

In a well-configured DDP setup, NCCL AllReduce kernels should start firing on each GPU during the backward pass, not after it finishes. DDP buckets gradients and launches communication as soon as each bucket is ready. Look for NCCL kernels (purple) interleaved with backward compute kernels (teal) on the same GPU row. If NCCL only appears after all backward kernels are done, the overlap isn't working.

3. Straggler barriers

Look for a gap between one GPU finishing its compute and the NCCL kernel starting. If GPU 0 finishes backward at t=100ms but NCCL doesn't start until t=130ms, something is blocking the collective launch. This could be a Python-side barrier, a logging call on one rank, or a load imbalance.

4. NCCL algorithm selection

NCCL kernel names encode the algorithm: Ring, Tree, CollNet. Ring is optimal for small messages on <= 8 GPUs. Tree is better for large messages. If you see suboptimal selection, override with NCCL_ALGO=Ring or NCCL_ALGO=Tree and re-profile.

7 Decide
The diagnostic decision tree

Start at the top and follow the arrows. This encodes the systematic triage from the sections above into a quick-reference flowchart.

Open the profile. Zoom to one training iteration. Look at the GPU kernel row. Is there significant white space (GPU idle time)?

YES →The GPU is idle for a significant portion of the iteration
NO →GPU row looks fully packed, but throughput is still low

YES path: During the GPU gaps, what's happening on the CPU thread row?

A →CPU is busy (solid blocks visible)
→ Pattern 1: Data starvation / CPU bottleneck. Check OS Runtime stats for read/futex. Fix: more workers, GPU transforms, faster storage.
B →CPU is idle too (both rows empty)
→ Check CUDA API row. Is there a cudaStreamSynchronize?
• Yes → Pattern 3: Sync stall. Find the .item()/.cpu() call.
• No, but cudaMalloc → Pattern 4: Alloc churn.
• No, but NCCL just finished → Pattern 5: Comms bottleneck. Check DDP bucket config.

NO path: GPU row is packed. Check power draw and kernel sizes.

C →Power is low (< 250W on H100) despite full occupancy
→ Zoom in. Are there thousands of tiny kernels (< 10μs)?
• Yes → Pattern 2: Launch overhead. Use torch.compile or CUDA Graphs.
• No, kernels are large but power still low → Memory-bandwidth-bound kernels. Move to Nsight Compute roofline analysis.
D →Power is moderate (300–450W), large kernels visible
→ Check kernel names. Do they contain sgemm (FP32)?
• Yes → Pattern 6: No Tensor Cores. Enable AMP / BF16. Align dimensions to multiples of 8.
• No, names show h884/bf16 → Good! You're Tensor Core-bound. This is close to optimal — consider FP8 (Transformer Engine) or algorithmic improvements.
E →Power is high (500W+), large Tensor Core kernels
→ You're near hardware peak. Remaining gains come from: higher batch size (gradient accumulation), algorithmic changes (FlashAttention, fused optimizers), or scaling to more GPUs with good communication overlap.
