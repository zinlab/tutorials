1 Vision vs language: what changes

The five fundamental differences from LLM trainingOverview

1. Data pipeline is the #1 bottleneck

LLM data is pre-tokenized binary (microseconds to load). Vision data requires JPEG decode (0.5–5ms), resize, augmentation (random crop, flip, color jitter, mixup), and normalization — all CPU-intensive. The data pipeline is the first thing to fix, not the last.

2. Memory layout matters enormously

Convolutions and vision ops are deeply sensitive to whether data is stored as NCHW (channels-first, PyTorch default) or NHWC (channels-last, cuDNN preference). Wrong layout = up to 30% slower. LLMs don't have this issue — their tensors are [batch, seq, hidden], always contiguous.

3. Spatial dimensions create variable compute

Unlike fixed-length token sequences, images have spatial dimensions (H×W) that affect GEMM shapes at every layer (through feature maps). Multi-scale training, variable resolutions, and progressive resizing change the compute profile dynamically.

4. ConvNets and ViTs have different profiles

ConvNets are dominated by implicit-GEMM convolutions with cuDNN algorithm selection. ViTs look like LLMs but with patch-based tokenization and no causal masking — different attention patterns, different memory characteristics.

The fifth difference: activation memory dominates even more. Vision models process high-resolution feature maps. A single conv layer's output at 256 channels × 56×56 spatial × batch 256 is already 400 MB in FP32. LLMs have large activations too, but vision activations scale with H×W×C per layer, not just sequence×hidden — and vision models often have more layers with gradual resolution reduction creating a pyramid of differently-sized feature maps.

Priority shift: For LLMs, the profiling priority is Tensor Cores → communication → batch size → fusion. For vision, it's data pipeline → memory layout → Tensor Cores → augmentation on GPU → communication. The data pipeline being first is the defining characteristic of vision model performance engineering.

3 Compute and memory math

FLOP counting for vision architecturesCompute

Convolution FLOPs

Conv2d FLOPs:
2 × B × C_out × H_out × W_out × C_in × K_h × K_w

ResNet-50 (B=1, 224×224 input):
Forward: ~4.1 GFLOP
Forward + Backward: ~12.3 GFLOP
Batch of 256: ~3.1 TFLOP per step

At 50% MFU on H100 BF16 (495 TFLOP/s effective):
3.1 TFLOP / 495 TFLOP/s ≈ 6.3ms per step ← extremely fast
Actual time is ~40–80ms because data pipeline is the bottleneck

ViT FLOPs

ViT FLOPs ≈ same formula as LLM Transformer layers:
6 × N × B × num_patches (forward + backward)

ViT-Huge (632M params), 224×224, 14×14 patches = 256 patches, B=256:
6 × 632×10⁶ × 256 × 256 ≈ 249 TFLOP per step

ViT-Giant (1.8B params), 224×224, B=128:
6 × 1.8×10⁹ × 128 × 256 ≈ 354 TFLOP per step
Vision models are often under-compute, over-data. A ResNet-50 training step has ~3 TFLOP of compute — an H100 can do that in 6ms. But loading, decoding, and augmenting 256 images takes 30–100ms on CPU. The GPU spends 80%+ of its time idle waiting for data. This is why the data pipeline chapter comes before everything else for vision.

Solution tier list: from quick fixes to full rewritesData

Tier 1: DataLoader tuning (10 minutes, no code changes)

loader = DataLoader(
    dataset,
    batch_size=256,
    num_workers=12,            # rule: num_cpus / num_gpus, typically 8–16
    pin_memory=True,            # async H2D via pinned host memory
    persistent_workers=True,    # don't respawn workers every epoch
    prefetch_factor=3,          # prefetch 3 batches ahead
    drop_last=True,             # avoid small last batch (uneven compute)
)

Expected improvement

Going from num_workers=2 to num_workers=12 on a 96-CPU node with 8 GPUs typically gives 2–4× data loading speedup. persistent_workers eliminates 1–3 seconds of worker respawn per epoch. prefetch_factor=3 pipelines the next batch during GPU compute.

Common mistake: too many workers

Setting num_workers=32 on a machine with 32 CPUs and 8 GPUs means each GPU claims 32 workers, totaling 256 processes competing for 32 cores. CPU contention makes things slower. Always divide: num_workers ≈ num_cpus // num_gpus.

Tier 2: Move augmentations to GPU (moderate code changes)

The insight: GPU SMs are 100× faster than CPU cores for pixel operations. Move everything after decode to the GPU.

# Using Kornia for GPU augmentations
import kornia.augmentation as K

gpu_transforms = nn.Sequential(
    K.Resize(256),
    K.RandomCrop((224, 224)),
    K.RandomHorizontalFlip(p=0.5),
    K.ColorJitter(brightness=0.4, contrast=0.4, saturation=0.4, hue=0.1),
    K.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),
).cuda()

# DataLoader only does: read → decode → to_tensor → H2D
# GPU does: resize → crop → flip → jitter → normalize
for images, labels in loader:
    images = images.cuda(non_blocking=True)
    images = gpu_transforms(images)     # augmentation on GPU — microseconds
    output = model(images)

Expected improvement

Removes ~50–70% of the CPU data pipeline time. JPEG decode remains on CPU (the remaining bottleneck). Total pipeline time drops from ~100ms to ~30–50ms for a batch of 256.

Tier 3: NVIDIA DALI — end-to-end GPU pipeline

DALI moves the entire pipeline (including JPEG decode) to GPU using nvJPEG hardware decoder, eliminating the CPU bottleneck entirely.

# DALI pipeline (simplified)
from nvidia.dali import pipeline_def, fn, types

@pipeline_def(batch_size=256, num_threads=8, device_id=0)
def training_pipeline():
    jpegs, labels = fn.readers.file(
        file_root="/data/imagenet/train", random_shuffle=True
    )
    images = fn.decoders.image(
        jpegs, device="mixed"    # nvJPEG decode on GPU
    )
    images = fn.resize(images, resize_x=256, resize_y=256)
    images = fn.crop_mirror_normalize(
        images,
        crop=[224, 224],
        mean=[0.485*255, 0.456*255, 0.406*255],
        std=[0.229*255, 0.224*255, 0.225*255],
        mirror=fn.random.coin_flip(probability=0.5),
        output_layout="CHW",
    )
    return images, labels

Expected improvement

Eliminates CPU bottleneck entirely. Data pipeline drops to ~2–5ms per batch (GPU decode + GPU augment), making it negligible compared to model compute. This is the gold standard for ImageNet-scale training. Throughput often improves 2–5× over PyTorch DataLoader for ConvNets.

5 Memory layout: NCHW vs NHWC

Why channels_last gives 15–30% speedup for ConvNetsLayout

PyTorch defaults to NCHW (batch, channels, height, width) — but cuDNN's fastest convolution implementations prefer NHWC (batch, height, width, channels). The mismatch means cuDNN internally transposes data on every convolution, wasting bandwidth and compute.

LayoutMemory orderGPU friendlinessWhy
NCHW (default)All pixels of channel 0, then all of channel 1, ...Poor for Tensor CoresTensor Cores operate on matrix tiles where the reduction dimension should be contiguous. In conv, the reduction is over C_in — which is the slowest-varying dimension in NCHW, causing uncoalesced access.
NHWC (channels_last)All channels of pixel (0,0), then all channels of pixel (0,1), ...Optimal for Tensor CoresChannels are contiguous per pixel — cuDNN can load aligned tiles directly into Tensor Core MMA instructions without transposing.
# Enable channels_last — one line of code
model = model.to(memory_format=torch.channels_last)

# Convert input data too:
images = images.to(memory_format=torch.channels_last)

# Or do it in the training loop:
for images, labels in loader:
    images = images.cuda(non_blocking=True).to(memory_format=torch.channels_last)
    output = model(images)

Expected speedup

ResNet-50: +20–30%. ConvNeXt: +15–25%. EfficientNet: +10–20%. The speedup comes from: (1) cuDNN selects Tensor Core–optimized algorithms, (2) no internal transposes, (3) better memory coalescing. This is the single highest-impact optimization for ConvNet training — literally one line of code.

Gotchas with channels_last

Not all ops support it natively. If an op doesn't support NHWC, PyTorch silently converts back to NCHW, computes, and converts back — incurring two transposes that negate the benefit. Check with torch.channels_last_strides_2d(tensor). Common offenders: some custom CUDA ops, older norm implementations, some interpolation modes.

ViTs don't benefit. ViTs work with [B, N, D] tensors after patch embedding — there's no spatial H/W dimension to reorder. The patch embedding conv benefits, but that's one layer out of 24+.

cuDNN offers multiple convolution algorithms (Winograd, FFT, implicit GEMM, tiled convolution). The fastest one depends on kernel size, stride, dilation, channels, spatial size, batch size, and data type. cuDNN's heuristic doesn't always pick the best one.

6 Precision and Tensor Cores

Mixed precision for vision modelsPrecision

The precision strategy is similar to LLMs but with important differences in which ops get autocast.

# Standard vision mixed precision
scaler = torch.amp.GradScaler('cuda')  # needed for FP16, optional for BF16

for images, labels in loader:
    images = images.cuda(non_blocking=True).to(memory_format=torch.channels_last)

    with torch.autocast('cuda', dtype=torch.float16):  # or torch.bfloat16
        output = model(images)
        loss = criterion(output, labels)

    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()
    optimizer.zero_grad()

# BF16 is simpler (no scaler needed):
with torch.autocast('cuda', dtype=torch.bfloat16):
    output = model(images)
    loss = criterion(output, labels)
loss.backward()
optimizer.step()

The RGB input problem

The first conv layer takes 3-channel (RGB) input. C_in=3 is not a multiple of 8, so Tensor Cores can't be used for this layer. cuDNN pads internally to 4 or 8 channels. This is one conv layer out of ~50+ in a ResNet — negligible impact. But for architectures with many early narrow-channel layers, the misalignment can accumulate. Modern architectures (ConvNeXt, EfficientNet) use stem layers that quickly project to ≥32 channels.

7 Memory optimization

Vision-specific memory techniquesMemory

Activation checkpointing — layer selection matters more

Vision models have heterogeneous layer costs. Unlike LLMs where every Transformer layer is similar, vision models have early high-resolution layers (large activations, cheap compute) and late low-resolution layers (small activations, expensive compute). Checkpoint the early layers first — they save the most memory with the least recomputation cost.

# Selective checkpointing for ResNet — checkpoint early (memory-heavy) stages
from torch.utils.checkpoint import checkpoint

class ResNet(nn.Module):
    def forward(self, x):
        x = self.stem(x)
        x = checkpoint(self.layer1, x, use_reentrant=False)  # 56×56 — huge
        x = checkpoint(self.layer2, x, use_reentrant=False)  # 28×28 — large
        x = self.layer3(x)                                    # 14×14 — moderate
        x = self.layer4(x)                                    # 7×7 — small
        return self.head(x)

Multi-scale training memory management

Training at progressively higher resolutions (common for detection/segmentation: 320→480→640→800) changes memory requirements mid-training. The caching allocator can fragment when resolution jumps.

TechniqueHow it helps
Pad to fixed resolution setInstead of arbitrary sizes, use a small set of sizes (320, 480, 640, 800). The caching allocator can reuse blocks across the same size.
torch.cuda.empty_cache() at resolution changeRare valid use case — when switching to a dramatically larger resolution, clear the cache to avoid fragmentation. Don't call within a fixed-resolution training loop.
Gradient accumulation at high resWhen resolution doubles, activation memory quadruples. Reduce micro-batch size and accumulate more steps.

Large-batch training considerations

Vision models are typically trained with very large global batch sizes (4K–32K images) across many GPUs. This introduces convergence challenges:

IssueSolutionPyTorch implementation
Learning rate scalingLinear scaling rule: LR × batch_size / 256. Or square root scaling for larger batches.Multiply lr in optimizer by global_batch_size / 256
WarmupLinear warmup over 5–10 epochs to the target LR. Prevents divergence at large LR.Use torch.optim.lr_scheduler.LinearLR for warmup phase

BatchNorm synchronizationWith DDP, each GPU computes BN stats on its local batch. With small per-GPU batch sizes, stats are noisy. SyncBatchNorm computes global stats.nn.SyncBatchNorm.convert_sync_batchnorm(model)

SyncBatchNorm performance cost

SyncBatchNorm adds an AllReduce for mean and variance at every BN layer (2 AllReduces × number of BN layers × per step). For ResNet-50 with 53 BN layers, that's 106 extra AllReduces per step. On NVLink this is fast (~0.5ms total), but across nodes on InfiniBand it can add 5–20ms. If per-GPU batch size ≥ 32, standard BatchNorm is usually sufficient. Use SyncBN only when per-GPU batch is small (< 16).

model = torch.compile(model, mode="max-autotune")

11 Multimodal (vision + language)

Performance engineering for vision-language modelsMultimodal

Models like LLaVA, Qwen-VL, and GPT-4V-class systems combine a vision encoder with an LLM. This creates unique performance challenges.

Architecture components and their cost

ComponentExampleFLOPs per sampleKey bottleneck

Performance pitfalls specific to multimodal

Pitfall 1: Image tokenization creates variable-length sequences

A single high-res image might produce 256–1024 vision tokens, while text might be 50–500 tokens. Each sample has a different total token count. This makes batching inefficient — either pad everything to the longest sequence (wasting compute on padding) or use dynamic batching (complex, but more efficient). Solutions: pack samples to fixed total lengths (like LLM sequence packing), or group by similar image+text length and pad within groups.

Pitfall 2: Frozen vision encoder throughput mismatch

The vision encoder runs with torch.no_grad() (no backward), which is much faster than the LLM's training forward+backward. If the vision encoder produces tokens faster than the LLM can process them, the encoder's GPU time is wasted. Solutions: (1) precompute vision embeddings offline (best), (2) run the vision encoder in a separate stream overlapped with the LLM's backward of the previous step, (3) use a smaller/faster vision encoder.

Pitfall 3: Mixed modality attention is memory-expensive

When image tokens (256–1024) are concatenated with text tokens (500+), the total sequence length can reach 1500+ tokens. Attention is O(s²), so multimodal attention is much more expensive than pure text at the same "content" amount. FlashAttention is critical here. Also consider: use cross-attention instead of concatenation (separate attention for text-to-image and image-to-text) to avoid the full s² cost.

Profiling workflow for vision modelsProfiling

The diagnostic sequence for vision is different from LLMs because the most likely bottleneck is the data pipeline, not compute.

Step 1: Is the data pipeline keeping up?

Step 3: Are Tensor Cores in use?

# Nsight Compute — profile a conv kernel

    -o conv_analysis python train.py
# Check: Tensor Core pipe utilization > 0%

Step 4: Standard Nsight Systems analysis

Same as the LLM workflow: look for GPU idle gaps, sync stalls, allocation churn, communication overhead. The patterns are identical — only the likelihood differs (data pipeline is the most common culprit for vision vs compute for LLMs).

The vision performance engineer's decision tree: (1) GPU-Util oscillating → fix data pipeline. (2) GPU-Util steady but power low → check channels_last + Tensor Cores. (3) GPU-Util and power both high but images/sec still low → profile kernels with Nsight Compute, check for algorithm misselection. (4) Single-GPU is fast but multi-GPU doesn't scale → check communication overlap and SyncBN overhead.

The vision performance engineer's mental model: The data pipeline is the first bottleneck, not the last. Fix it with DALI, GPU augmentation, or preprocessed formats before touching anything else. For ConvNets, channels_last + cuDNN benchmark + BF16 are the three transformative optimizations — each is one line of code. For ViTs, the approach converges with LLM engineering (FlashAttention, torch.compile, FSDP). For diffusion, precompute everything you can (latents, text embeddings) and use CUDA Graphs for the iterative denoiser. Measure throughput in images/sec, diagnose with nvidia-smi first (is the GPU even busy?), and only reach for Nsight Systems/Compute after the data pipeline and layout are confirmed correct.
