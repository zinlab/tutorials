Part I — The transformer, in full

From attention to a language model.

Volume III built a transformer that reads a whole sequence at once. A language model needs one twist — it must predict the next token from only the tokens before it — plus the engineering that makes attention fast enough and position-aware enough to scale. This part upgrades the transformer into the decoder that Part II turns into a GPT.

01Scaled dot-product & causal masks

A language model is trained on one deceptively simple objective: given a run of tokens, predict the next one. Do that well over trillions of tokens and the model must learn grammar, facts, reasoning, and style as a side effect. The architectural requirement this imposes is causality: when predicting the token at position i, the model may look only at positions ≤ i. If it could peek ahead, "predicting the next token" would be trivial and it would learn nothing.

Causality is enforced with a causal mask on the attention scores: before the softmax, every score from a position to a future position is set to −∞, so its weight becomes zero. Each token attends to itself and its past, never its future.

THE CAUSAL MASK (● = can attend, · = blocked to future): attends to → t1 t2 t3 t4 query t1 │ ● · · · row = one token's allowed view query t2 │ ● ● · · lower-triangular: past + self only query t3 │ ● ● ● · query t4 │ ● ● ● ● masked scores get -inf, so softmax gives them zero weight
import torch
import torch.nn.functional as F

def causal_self_attention(x):        # x: (B, T, d)
    B, T, d = x.shape
    q = k = v = x                     # (simplified: real code projects each)
    scores = (q @ k.transpose(-2, -1)) / d ** 0.5      # (B, T, T)
    # lower-triangular mask: block every future position
    mask = torch.tril(torch.ones(T, T, device=x.device)).bool()
    scores = scores.masked_fill(~mask, float("-inf"))
    return F.softmax(scores, dim=-1) @ v

This one mask is the entire difference between an autoregressive decoder (GPT-style, generates text) and a bidirectional encoder (BERT-style, reads text). Everything else — the attention, the MLP, the residuals — is identical to Volume III. In practice you don't build the mask by hand; the next chapter's fused kernel takes a single is_causal=True flag.

02Efficient attention: SDPA & FlashAttention

Naive attention has a memory problem that caps sequence length. The score matrix is T×T for a sequence of length T, so doubling the context quadruples the memory just to hold the scores — at 8K tokens that matrix is enormous, and it's created only to be softmaxed and discarded. FlashAttention is the breakthrough that removed this wall: it computes attention in tiles that stream through fast on-chip memory, never materializing the full T×T matrix, and it's "IO-aware" — optimized around memory movement, the real bottleneck. The payoff is large speedups and roughly linear memory in sequence length, which is what made long-context models practical.

You get all of this through one call — F.scaled_dot_product_attention — which dispatches to FlashAttention (or another fused, memory-efficient kernel) automatically on capable GPUs:

import torch
import torch.nn.functional as F

# q, k, v: (B, n_heads, T, head_dim)
out = F.scaled_dot_product_attention(q, k, v, is_causal=True)   # fused + causal
# - is_causal=True applies the triangular mask WITHOUT building a T x T tensor
# - dispatches to FlashAttention on supported GPUs; falls back safely otherwise
Always use the fused path

Never hand-roll the softmax attention in production — F.scaled_dot_product_attention is faster, uses far less memory, and handles causal masking and padding masks for you. The manual version from Volume III is for understanding only. For packed variable-length batches, recent PyTorch adds a dedicated variable-length attention operator that avoids padding waste entirely — check your version.

This efficiency is not a nicety; it's load-bearing. Long context windows, the memory budgets in Part III, and affordable inference in Part IV all rest on attention being sub-quadratic in memory. FlashAttention is one of the handful of systems ideas that made modern LLMs feasible.

03Positional information: RoPE & friends

Attention is permutation-invariant (Volume III): with no position signal, the model can't tell "dog bites man" from "man bites dog." Early transformers added absolute position information — fixed sinusoids or a learned vector per position — to the token embeddings. It works, but learned absolute positions can't handle sequences longer than those seen in training, and they encode position clumsily.

Modern LLMs almost all use Rotary Position Embedding (RoPE) instead. Rather than adding a position vector, RoPE rotates each query and key vector by an angle proportional to its position. The elegant consequence: when a query at position m dots with a key at position n, the result depends only on their relative distance m−n — so the model natively understands relative position, generalizes better to longer sequences, and needs no extra parameters.

import torch

def apply_rope(x, positions, base=10_000):
    # x: (..., T, head_dim). Rotate pairs of dims by position-scaled angles.
    d = x.shape[-1]
    freqs = 1.0 / (base ** (torch.arange(0, d, 2, device=x.device) / d))
    angles = positions[:, None] * freqs[None, :]      # (T, d/2)
    cos, sin = angles.cos(), angles.sin()
    x1, x2 = x[..., ::2], x[..., 1::2]                 # even / odd dims
    # rotate each (x1, x2) pair by its angle:
    return torch.stack([x1 * cos - x2 * sin,
                        x1 * sin + x2 * cos], dim=-1).flatten(-2)

RoPE is applied to the queries and keys inside attention (not to the values, and not to the token embeddings). A related scheme, ALiBi, instead adds a distance-proportional penalty directly to the attention scores — simpler, also extrapolates. You'll meet RoPE by far the most often; recognizing it in a codebase (a rotation applied to Q and K before scaled_dot_product_attention) tells you you're looking at a modern LLM.

04The decoder block & residual stream

A GPT is a stack of identical decoder blocks, each the pre-norm attention-then-MLP pattern from Volume III, now with causal attention. A useful mental model is the residual stream: because every sub-layer is x = x + sublayer(norm(x)), information flows straight down the stack along the + x skip, and each block reads from that stream, computes something, and writes its result back by addition. The stream is the model's working memory; blocks are edits to it.

import torch
import torch.nn as nn
import torch.nn.functional as F

class DecoderBlock(nn.Module):
    def __init__(self, dim, n_heads, mlp_ratio=4):
        super().__init__()
        self.n_heads = n_heads
        self.norm1 = nn.LayerNorm(dim)
        self.qkv = nn.Linear(dim, 3 * dim, bias=False)
        self.proj = nn.Linear(dim, dim, bias=False)
        self.norm2 = nn.LayerNorm(dim)
        self.mlp = nn.Sequential(
            nn.Linear(dim, mlp_ratio * dim), nn.GELU(),
            nn.Linear(mlp_ratio * dim, dim),
        )

    def forward(self, x):                       # x: (B, T, dim)
        B, T, D = x.shape
        h = self.norm1(x)
        qkv = self.qkv(h).view(B, T, 3, self.n_heads, D // self.n_heads)
        q, k, v = qkv.permute(2, 0, 3, 1, 4)    # (B, heads, T, head_dim)
        a = F.scaled_dot_product_attention(q, k, v, is_causal=True)
        a = a.transpose(1, 2).reshape(B, T, D)
        x = x + self.proj(a)                    # write attention to the stream
        x = x + self.mlp(self.norm2(x))         # write MLP to the stream
        return x

Frontier models refine a few details you'll see: RMSNorm (a cheaper LayerNorm variant) replaces LayerNorm, SwiGLU replaces the GELU MLP for a small quality gain, and the token-embedding and output-projection weights are often tied (shared) to save parameters. These are optimizations on the same skeleton — the block above is the honest core, and Part II wraps a stack of them into a complete GPT.

A GPT is one causal block, normalized and skipped, stacked N deep — reading and writing a shared residual stream.

Part II — Build a GPT

A language model, end to end.

Now assemble a real, trainable GPT: turn text into tokens, stack the decoder blocks into a model, train it on next-token prediction, and generate from it. This is the nanoGPT path — small enough to run on one GPU, structurally identical to the giants. Understand these four chapters and every LLM codebase becomes readable.

05Tokenization (BPE) & the data pipeline

Models work on integers, so text must first become a sequence of token IDs. Splitting on words is brittle (huge vocabulary, no way to handle unseen words); splitting on characters makes sequences painfully long. Byte-Pair Encoding (BPE) threads the needle: it learns a vocabulary of frequent subword pieces, so common words are one token, rare words split into a few pieces, and any string is representable — no out-of-vocabulary problem. Typical vocabularies are 50K–130K tokens.

# In practice you use a fast, pretrained tokenizer (tiktoken, HF tokenizers).
import tiktoken
enc = tiktoken.get_encoding("cl100k_base")

ids = enc.encode("PyTorch makes tensors sing.")
print(ids)                       # [list of integer token IDs]
print(enc.decode(ids))           # round-trips back to the original text
print("vocab size:", enc.n_vocab)

The data pipeline for pretraining is then blunt and effective: tokenize the entire corpus into one long stream of IDs, and cut it into fixed-length chunks of context_length tokens. The training target is the input shifted by one — at every position, predict the next token.

import torch

def get_batch(data, context_len, batch_size, device):
    # data: a 1-D tensor of the whole tokenized corpus.
    ix = torch.randint(len(data) - context_len - 1, (batch_size,))
    x = torch.stack([data[i:i + context_len] for i in ix])          # inputs
    y = torch.stack([data[i + 1:i + context_len + 1] for i in ix])  # shifted +1
    return x.to(device), y.to(device)    # x,y: (batch_size, context_len)

That one-token shift is the language-modeling objective, made concrete: for a chunk of length T you get T next-token predictions in parallel (thanks to the causal mask — every position predicts its successor at once). It's remarkably data-efficient, which is why a modest corpus already teaches a small model to produce fluent text.

06A GPT from scratch in PyTorch

Here is the whole model: embed tokens, run the decoder stack from Part I, normalize, and project to vocabulary-sized logits. The output projection shares weights with the token embedding (weight tying) — a standard trick that saves parameters and tends to help.

import torch
import torch.nn as nn
import torch.nn.functional as F

class GPT(nn.Module):
    def __init__(self, vocab, dim=384, depth=6, heads=6, ctx=256):
        super().__init__()
        self.ctx = ctx
        self.tok = nn.Embedding(vocab, dim)
        self.pos = nn.Embedding(ctx, dim)                 # learned positions
        self.blocks = nn.ModuleList(
            [DecoderBlock(dim, heads) for _ in range(depth)])
        self.norm = nn.LayerNorm(dim)
        self.head = nn.Linear(dim, vocab, bias=False)
        self.head.weight = self.tok.weight                # weight tying

    def forward(self, idx, targets=None):                 # idx: (B, T)
        B, T = idx.shape
        pos = torch.arange(T, device=idx.device)
        x = self.tok(idx) + self.pos(pos)                 # (B, T, dim)
        for blk in self.blocks:
            x = blk(x)
        logits = self.head(self.norm(x))                  # (B, T, vocab)
        if targets is None:
            return logits, None
        # cross-entropy over the next token, flattened across batch and time:
        loss = F.cross_entropy(logits.view(-1, logits.size(-1)),
                               targets.view(-1))
        return logits, loss

model = GPT(vocab=enc.n_vocab)
print("params:", sum(p.numel() for p in model.parameters()) / 1e6, "M")

That's a complete GPT in about thirty lines — the same architecture, scaled down, as the models that write essays and code. The only differences at the frontier are size (thousands of dimensions, dozens of blocks, longer context), the refinements from Chapter 04 (RoPE, RMSNorm, SwiGLU), and orders of magnitude more data and compute. The shape is exactly this.

07Training it: schedule, clipping, stability

Training a GPT is the Volume II loop with LLM-specific settings. AdamW, a cosine schedule with warmup, gradient clipping, and bf16 autocast are all essentially mandatory; the one new subtlety is not applying weight decay to the norm and bias parameters (only to the matmul weights).

import torch

model = model.to("cuda")
model = torch.compile(model)                       # Volume IV speedups
opt = torch.optim.AdamW(model.parameters(), lr=6e-4,
                        betas=(0.9, 0.95), weight_decay=0.1)

for step in range(max_steps):
    x, y = get_batch(train_data, model.ctx, batch_size=32, device="cuda")
    with torch.autocast("cuda", dtype=torch.bfloat16):
        logits, loss = model(x, y)
    opt.zero_grad(set_to_none=True)
    loss.backward()
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)   # essential for LLMs
    # (advance a cosine-with-warmup LR schedule here, per Volume II)
    opt.step()
    if step % 100 == 0:
        print(f"step {step}  loss {loss.item():.3f}")

The metric to watch is training loss, ideally reported as tokens-per-second alongside it so you know your throughput. LLM training is prone to loss spikes — sudden jumps that can derail a run — which is exactly why gradient clipping and warmup matter, and why large runs checkpoint often (Chapter 12). A healthy run shows loss falling smoothly and, converted to perplexity (the exponential of the loss), dropping steadily.

Overfit a tiny dataset first

Before a long run, confirm the plumbing: train on a few thousand tokens and verify the model can memorize them (loss → near zero) and echo them back. If it can't, you have a bug in the model, masking, or data shift — find it now, not after burning a day of GPU time. This is Volume II's "overfit one batch" applied to LLMs.

08Generation & the KV cache

Generation is autoregressive: feed the prompt, get a distribution over the next token, sample one, append it, and repeat. How you sample controls the character of the output.

StrategyWhat it does
greedyAlways take the highest-probability token. Deterministic, but repetitive and flat.
temperatureDivide logits by T before softmax. T<1 sharpens (safer, duller); T>1 flattens (more varied, riskier).
top-kSample only from the k most likely tokens — cuts off the long tail of nonsense.
top-p (nucleus)Sample from the smallest set whose probabilities sum to p — adapts the cutoff to how confident the model is. The common default.
 
@torch.no_grad()
def generate(model, idx, max_new=100, temperature=0.8, top_k=50):
    model.eval()
    for _ in range(max_new):
        idx_cond = idx[:, -model.ctx:]              # crop to context window
        logits, _ = model(idx_cond)
        logits = logits[:, -1, :] / temperature     # last position's logits
        if top_k is not None:                       # keep only top-k options
            v, _ = torch.topk(logits, top_k)
            logits[logits < v[:, [-1]]] = float("-inf")
        probs = torch.softmax(logits, dim=-1)
        next_id = torch.multinomial(probs, num_samples=1)   # sample one token
        idx = torch.cat([idx, next_id], dim=1)      # append and continue
    return idx

The loop above recomputes attention over the entire sequence for every new token — wasteful, since the past tokens' keys and values never change. The KV cache is the fix and the single most important LLM inference optimization: store each layer's keys and values as they're computed, and for each new token compute attention using the cached past plus just the one new key/value. That turns per-token cost from growing-with-length into roughly constant, making long generations affordable. Production serving engines (vLLM, from Volume IV) build sophisticated paged KV caches around exactly this idea.

You now have the whole loop: tokenize, model, train, generate. Everything a large language model does, your GPT does — at a smaller scale. The next part is about what breaks when you make it big: the memory won't fit, the compute must be budgeted, and stability gets harder. The architecture doesn't change; the engineering around it does.

Part III — Training at scale

What breaks when it gets big.

Your nanoGPT trains on one GPU. A useful LLM does not fit — not the parameters, not the optimizer state, not the activations. Scaling up is less about the model (it barely changes) and more about arithmetic: how much compute buys how much quality, where every byte of memory goes, and how to keep a run stable for days. This part is that arithmetic, and the recipe that pulls Volume IV together for language models.

09Scaling laws & compute budgeting

One of the most consequential findings in modern ML is that model quality improves predictably with scale. Plot loss against model size, dataset size, or total compute and you get smooth power laws — bigger models trained on more data reliably do better, and you can forecast how much better before you spend the money. This is why the field pours resources into scale: the returns are lawful, not lucky.

But size alone is the wrong target. The Chinchilla result showed that for a fixed compute budget, most large models had been undertrained — too many parameters, too few tokens. The rough guidance: scale parameters and training tokens together, on the order of ~20 tokens per parameter for compute-optimal training. A bigger model starved of data is a waste; the right move is to balance the two against your budget.

Training FLOPs
≈ 6 · N · D params × tokens
Compute-optimal
≈ 20 tokens / param
Forward pass
≈ 2 · N FLOPs / token

The handy estimate: training compute in FLOPs is about 6 · N · D, where N is parameters and D is training tokens (the 6 bundles the forward and backward passes). That single formula lets you convert a GPU budget into a feasible model-and-data size before committing — the first calculation any large run should do.

10Memory math for LLMs

The reason LLMs need many GPUs isn't compute — it's memory. Walk through what a parameter costs to train with AdamW in mixed precision, per parameter:

ItemBytes per parameter
parameter (bf16)2 — the weight itself
gradient (bf16)2 — one per parameter
AdamW state (fp32)8 — two buffers (momentum + variance), kept in fp32
fp32 master weight4 — a high-precision copy for stable updates
total≈ 16 bytes/param, before any activations

So a 7-billion-parameter model needs roughly 7e9 × 16 ≈ 112 GB just for parameters, gradients, and optimizer state — already more than a single 80 GB GPU, with nothing left for activations. This is the wall that forces sharding: FSDP2 (Volume IV) splits those 16 bytes across GPUs so each holds only a slice. On top of this sits activation memory, which grows with batch size × sequence length × layers and is often the largest term for long context — the reason activation checkpointing is standard for LLMs.

Inference is much cheaper — plus the KV cache

Serving needs only the parameters (≈ 2 bytes each in bf16, or ~1 with int8), no gradients or optimizer state — so that same 7B model infers comfortably where it couldn't train. The new cost at inference is the KV cache (Chapter 08), which grows with batch × sequence length and often dominates memory for long-context serving.

11FSDP2 + AMP + compile for LLMs

Training a large model is where every Volume IV technique combines into one recipe. There's nothing new here — it's the composition that matters: shard the model with FSDP2, compute in bf16, checkpoint activations per block, and compile. Each addresses a different resource, and together they make a model that wouldn't fit train efficiently.

import torch
from torch.distributed.fsdp import fully_shard, MixedPrecisionPolicy
from torch.utils.checkpoint import checkpoint

model = GPT(vocab, dim=4096, depth=32, heads=32, ctx=4096)   # a real size

mp = MixedPrecisionPolicy(param_dtype=torch.bfloat16,
                          reduce_dtype=torch.float32)
for block in model.blocks:
    fully_shard(block, mp_policy=mp)     # shard each block across GPUs
fully_shard(model, mp_policy=mp)

# activation checkpointing: wrap each block so its activations are recomputed
# (apply per-block; combine with the sharding above)

model = torch.compile(model)             # fuse kernels on top of it all
opt = torch.optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.1, fused=True)
# training loop is the Chapter 07 loop, now sharded across the whole cluster

This is roughly what production training frameworks (like torchtitan) package and tune for you, adding tensor and pipeline parallelism for the very largest models. The concepts are all ones you've met; the value of a framework is the battle-tested defaults and the launch/checkpoint plumbing. Reach for one before hand-rolling a multi-node LLM run.

12Data, checkpointing & loss spikes

At scale, three operational realities dominate the outcome as much as the architecture.

Data is the product. Beyond a point, what separates good models is data quality, not model tweaks: deduplication (repeated text wastes compute and hurts generalization), filtering out junk, and thoughtfully mixing sources and domains. Teams spend enormous effort here because it moves the needle more than another clever layer.

Checkpoint constantly. A run spanning days on hundreds of GPUs will hit a hardware failure, a preemption, or a loss spike. Save distributed checkpoints frequently (FSDP shards state across ranks, so use distributed checkpointing, not a naive rank-0 save) and make resuming routine — restart cost should be minutes, not a lost run.

Loss spikes happen — plan for them. Large runs periodically show the loss jump sharply. Causes range from a bad data batch to numerical instability to too-high a learning rate. Mitigations: gradient clipping (contains the damage), warmup and careful LR (prevent many), and — when a spike does derail training — rewind to the last good checkpoint, skip or reshuffle the offending data, and continue.

Watch the gradient norm, not just the loss

The gradient norm often spikes before the loss does — it's an early-warning signal. Log it every step alongside loss and learning rate. A creeping or spiking grad norm tells you instability is coming while there's still time to lower the LR or intervene, rather than discovering it after the loss has already blown up.

At scale, the model is the easy part; the data, the memory budget, and staying alive for a week are the job.

Part IV — Adapting pretrained models

Standing on a pretrained giant.

Almost no one pretrains an LLM from scratch — it costs millions. The real workflow starts from a pretrained model and adapts it: efficiently fine-tuning it on your task, aligning it to follow instructions, and serving it within a budget. This part is the practical LLM work most practitioners actually do.

13Full vs parameter-efficient fine-tuning

Pretraining teaches a model general language and knowledge; fine-tuning specializes it — to your domain, your task, your desired behavior. The question is how much of the model to touch.

Full fine-tuning updates every weight. It's the most powerful, but it costs as much memory as training from scratch (all those optimizer states from Chapter 10) and produces a whole new copy of the model per task. Parameter-efficient fine-tuning (PEFT) freezes the pretrained weights and trains only a tiny set of new parameters — often well under 1% of the model. You get most of the benefit for a fraction of the memory, you can keep many lightweight task-specific adapters over one shared base, and freezing the base largely avoids "catastrophic forgetting" of its general abilities.

Reach for PEFT when

You have limited GPU memory, modest task data, or want many specialized variants of one base model. This is the common case — LoRA (next chapter) is the default.

Full fine-tune when

You have ample compute and a large, high-quality dataset, and need the model to shift substantially — e.g. adapting to a very different domain or language.

In practice this work runs on the Hugging Face ecosystem — transformers for models, peft for adapters, datasets for data — which sits on top of PyTorch. Knowing the PyTorch underneath (which this volume gave you) is what lets you debug and extend those libraries rather than treating them as magic.

14LoRA & QLoRA

LoRA (Low-Rank Adaptation) is the dominant PEFT method, and the idea is elegant. Instead of updating a big weight matrix W, freeze it and learn a small additive correction expressed as the product of two skinny matrices: W + A·B, where A and B have a tiny inner "rank" r. A 4096×4096 matrix has 16M weights; its LoRA correction at rank 8 has only about 64K — a few thousandths as many — yet it's enough to steer behavior. At inference you can fold A·B back into W, so there's zero added latency.

import torch
import torch.nn as nn

class LoRALinear(nn.Module):
    def __init__(self, base: nn.Linear, r=8, alpha=16):
        super().__init__()
        self.base = base                      # the frozen pretrained layer
        for p in self.base.parameters():
            p.requires_grad = False
        d_in, d_out = base.in_features, base.out_features
        self.A = nn.Parameter(torch.randn(d_in, r) * 0.01)   # trainable, tiny
        self.B = nn.Parameter(torch.zeros(r, d_out))         # starts at 0
        self.scale = alpha / r

    def forward(self, x):
        return self.base(x) + (x @ self.A @ self.B) * self.scale   # W·x + ΔW·x

Note B starts at zero, so training begins exactly at the pretrained model (the correction is initially nothing) and grows only as needed — stable by construction. QLoRA pushes this further: it quantizes the frozen base model to 4-bit (Volume IV) and trains LoRA adapters on top, which shrinks memory so dramatically that you can fine-tune a large model on a single consumer GPU. LoRA and QLoRA are why fine-tuning big models is now accessible outside large labs.

15Alignment: SFT, RLHF, DPO

A freshly pretrained model is a next-token predictor, not an assistant — ask it a question and it might continue with more questions, because that's a plausible text continuation. Alignment is the process of turning that base model into something helpful, honest, and harmless that follows instructions. It proceeds in stages:

  • Supervised fine-tuning (SFT). Fine-tune on curated examples of instructions paired with good responses. This teaches the format and behavior of an assistant, and gets you most of the way. It's ordinary fine-tuning (often via LoRA) on a well-chosen dataset.
  • Learning from preferences (RLHF). To go further, collect human judgments of which of two responses is better, train a reward model to predict those preferences, then optimize the LLM against that reward with reinforcement learning (PPO). Powerful, but a complex, finicky multi-model pipeline.
  • Direct Preference Optimization (DPO). A newer method that gets much of RLHF's benefit by optimizing the preference data directly with a simple classification-style loss — no separate reward model, no RL loop. Because it's far simpler and more stable, DPO and its variants have become a common default for preference tuning.
Most projects stop at SFT

Full RLHF is the domain of teams building frontier assistants. For the large majority of applied work, supervised fine-tuning on good instruction data — optionally followed by a round of DPO on preference pairs — delivers an aligned, task-tuned model without the machinery of a reward model and PPO. Start simple; escalate only if SFT plateaus below what you need.

16Inference, quantization & serving for LLMs

Serving an LLM is its own discipline because generation is sequential and stateful. The costs and optimizations differ from any other model:

  • The KV cache dominates. As in Chapter 08, caching past keys/values makes per-token generation cheap — but the cache itself grows with batch size and sequence length and often becomes the memory bottleneck. Efficient cache management (paging, eviction) is central to serving throughput.
  • Quantize aggressively. Int8, and increasingly 4-bit, weight quantization (Volume IV) cut memory and boost throughput with little quality loss — near-universal for deployed LLMs.
  • Continuous batching. Requests arrive and finish at different times; a good server dynamically merges them so the GPU never idles between them — the key throughput trick, built into engines like vLLM.
  • Speculative decoding. A small draft model proposes several tokens that the big model verifies in one pass — a way to generate faster without changing outputs.

The metrics also differ: you track time-to-first-token (latency before output starts), tokens per second (generation speed), and cost per million tokens. You won't build this stack yourself — dedicated engines (vLLM and peers) handle KV-cache paging, continuous batching, and quantized kernels — but understanding the mechanics lets you choose settings, size hardware, and diagnose why a deployment is slow or expensive.

The whole arc, in one volume. You built a GPT, trained it, scaled the memory and compute math, adapted a pretrained one with LoRA, aligned it, and served it. That is the complete life of a language model in PyTorch. The frontier adds scale and refinements, but nothing here is a black box to you anymore — which is exactly the position from which to use the big libraries well.

Pretrain once, adapt many times — the leverage of the LLM era is in the fine-tune, not the from-scratch.
