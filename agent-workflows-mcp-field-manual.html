FRAME Chapter I

The frame · why agents need infrastructure.

The problem LLM calls fail. Tool calls fail. Networks blip. Rate limits hit. An agent that loses state on every transient error is not a product.

The answer Durable execution + observability + standardized tool protocols. Each addresses a specific failure class.

An agent in production is a distributed system with one extra-difficult component — a probabilistic function call that costs money, takes seconds, and sometimes returns garbage. Treating it like any other RPC works fine for prototypes. It collapses in production. Three classes of failure surface immediately at scale: state loss when a worker crashes mid-loop; opacity when something goes wrong and there's no trace to inspect; and tool sprawl when each agent grows its own ad-hoc integration with each external system. The 2026 production answer addresses each with a dedicated layer — Temporal for state and orchestration, Langfuse for observability, MCP for tools.

The three layers, named

Production agent systems decompose into three layers that solve specific failure classes. Mixing them — putting state in the agent, putting tool authentication in the orchestrator, putting observability in the tools — produces systems that work in demo and fail in production.

  • Orchestration. Durable execution. The agent loop, retry policies, child workflows, signals, scheduled runs. Temporal dominates 2026 production; alternatives include Inngest, Restate, AWS Step Functions, Dagster.
  • Observability. Hierarchical traces of LLM calls, tool invocations, retrievals. Prompt versioning. Evaluation pipelines. Langfuse is the OSS leader; alternatives include Phoenix, Laminar, LangSmith, Braintrust, Helicone.
  • Tool protocol. Standardized RPC surface between LLMs and external systems. MCP (Model Context Protocol) is the 2026 emergent standard, with FastMCP as the Python framework, OAuth 2.1 as the authorization model.

FRAME Chapter II

The three layers · orchestration, observability, tools.

The separation Each layer has a distinct concern, a distinct failure mode, and a distinct toolset. Mixing them collapses the abstractions.

The composition Orchestrator drives the agent loop. Observability captures every step. MCP exposes the tools. The LLM lives in the middle.

A production agent system has a recognizable shape in 2026. An orchestrator (Temporal worker) drives a workflow that invokes activities. Activities call the LLM, execute tools, write to databases, send notifications. Every step emits a span to the observability backend (Langfuse). Tools are exposed via MCP servers — separate processes with their own auth, deployment, scaling. Each layer is replaceable; collapsing any two of them produces a system that's harder to evolve.

The two-side rule — workflows vs activities

The single most consequential rule in Temporal-based agent systems: workflows are deterministic; activities are not. Anything that touches the outside world — an LLM call, a tool invocation, a database write, a system clock read — lives in an activity. The workflow only orchestrates. Per Temporal's framing: "LLMs are probabilistic by nature and can sometimes return incorrect or inconsistent responses." Temporal handles this by replaying the workflow from event history on restart, replaying past activity results, and only re-executing activities that hadn't completed. The workflow sees consistent state across restarts; the non-determinism stays isolated.

The observability layer's role

Per Langfuse: "Because AI is inherently non-deterministic, debugging your application without any observability tool is more like guesswork. Well-implemented observability gives you the tools to understand what's happening inside your application and why." The observability layer is orthogonal to orchestration — Temporal tells you what ran and in what order, Langfuse tells you what the prompts and outputs were and how much they cost. Both are required; neither replaces the other.

The MCP layer's role

Before MCP, every agent system grew custom integrations with each external system. Slack tool. GitHub tool. Salesforce tool. Each with its own auth flow, error handling, schema, deployment. The 2026 shift: MCP standardizes the tool surface. A single protocol speaks to all tools; OAuth 2.1 standardizes auth; each tool ships as a containerized server with consistent contracts. Per the MCP + Temporal pattern: "Durable AI agents that combine OpenAI Agents (runtime/coordination), Temporal (durable workflows), MCP, and Pydantic Logfire for tracing." The agent code doesn't change when you add a new tool — drop in another MCP server.

Why the Haus stack uses all three

Haus's Dockmaster (the worked example in Ch XVI) sits at this exact intersection. Temporal drives the dev loop workflow (selection → implementation → verification → automerge). Langfuse traces every Claude Code agent invocation with prompt versions tagged via git. MCP servers expose Linear, internal Haus data, Salesforce, Google Workspace, and Zoom to the agents. Each layer does one thing; the system is operable because of the separation, not despite it.

The wrong-layer trap (again)

The same wrong-layer trap from Issue 14 returns here in a different costume. Putting tool authentication inside Temporal activities (instead of MCP). Putting prompt versions in code (instead of Langfuse). Putting orchestration state in the LLM context window (instead of Temporal). Each collapse looks simpler in the short term and harder to operate in the long term. The discipline that makes these systems scale is keeping the layers separate even when a quick hack would mix them.

ORCH Chapter III

Temporal · the mechanism.

The model Workflow code is replayed from a durable event history on every restart. Activities (side-effect operations) are executed once; their results are recorded in history.

The components Temporal Server (history + matching service backed by Postgres/Cassandra), workers (your code), task queues (between them).

Temporal's core abstraction is durable execution — code that survives process crashes, machine restarts, and network partitions without losing progress. The mechanism is event sourcing applied to workflows: every state change is appended to a persistent event history; on restart, the workflow re-executes deterministically against that history, replaying past decisions until it reaches the point of failure, then continuing forward. The user writes code that looks like ordinary async functions. The platform makes it durable.

The architecture


      
The implication: an activity that took 30 seconds and returned a result doesn't get re-executed when the worker restarts. The result is in the history. The workflow code replays through it as a recorded fact.


      
Workflows are deterministic; activities are not


      
This is the single most consequential rule. Workflow code must produce the same sequence of activity invocations every time it's replayed against the same history. Any non-determinism — calling random(), reading datetime.now(), making a direct HTTP call — produces different decisions on replay and corrupts the history. The Temporal SDK enforces this by sandboxing workflow imports:


      
from temporalio import workflow
from datetime import timedelta

with workflow.unsafe.imports_passed_through():
    from orchestrator.activities import agent, github, linear

@workflow.defn
class DevLoopWorkflow:
    @workflow.run
    async def run(self, input: WorkflowInput) -> WorkflowResult:
        # OK: calling activities — deterministically scheduled
        issue = await workflow.execute_activity(
            linear.select_issue,
            args=[input.project],
            start_to_close_timeout=timedelta(minutes=5),
        )

        # OK: workflow time — read from history, not system clock
        deadline = workflow.now() + timedelta(hours=2)

        # FORBIDDEN: direct HTTP call — non-deterministic side effect
        # import requests  # would raise SandboxError
        # requests.get("https://api.linear.app/...")

        # FORBIDDEN: system time directly
        # import time
        # time.sleep(60)  # workflow.sleep(60) is the correct primitive

        # FORBIDDEN: random — different on replay
        # import random
        # pick = random.choice(...)  # use workflow.random() instead

        return WorkflowResult(issue_id=issue.id)


      
The workflow.unsafe.imports_passed_through() context is how you import activity modules — the workflow sees the types but can't accidentally execute the side-effect code inside them.

The mental model

        
Stop thinking of Temporal as "a job queue with retries." Think of it as "a programming language whose semantics include durability." You write ordinary-looking async code; the language runtime makes every step survive crashes. The cost is the determinism constraint; the benefit is that production agent code looks almost identical to a notebook prototype, while behaving entirely differently when things break.
ORCH Chapter IV

Workflows, activities, retry policies.

Three primitives Workflow (deterministic orchestration), Activity (side-effect operation), Child Workflow (composable sub-orchestration).

The retry policy Per-activity, per-error-class. Backoff, max-attempts, non-retryable errors. The single biggest reliability lever.

Three primitives compose everything. A workflow is the durable orchestration definition — what happens in what order, with what timeouts, with what error handling. An activity is the unit of side-effect work — call an LLM, query a database, hit an external API. A child workflow is a workflow that runs as part of another workflow, with its own event history but linked to the parent. Each has different timeout semantics, different retry behavior, different deployment implications. Picking the right primitive for each step is most of what good Temporal architecture is.

Workflows — the orchestration unit

A workflow is the top-level durable execution. It has an ID, an event history, a result type. It can run for milliseconds or for years. Inside it, you call activities and child workflows; you wait for signals; you handle queries. The workflow is replay-safe by construction.

class DevLoopWorkflow(_VerificationPhase, _ImplementationPhase, _SelectionPhase):
    def __init__(self) -> None:
        self._ctx: RunContext | None = None
        self._current_phase: str = "initialize"
        self._phase_progress: float = 0.0
        self._validation_override: str | None = None

    @workflow.run
    async def run(self, input: DevLoopInput) -> DevLoopResult:
        ctx = await self._initialize(input)
        ctx = await self._selection(ctx)
        ctx = await self._implementation(ctx)
        ctx = await self._verification(ctx)
        return await self._cleanup(ctx)

    @workflow.signal
    def skip_issue(self, reason: str) -> None:
        if self._ctx:
            self._ctx.mark_skipped(reason)

    @workflow.signal
    def override_validation(self, decision: str) -> None:
        self._validation_override = decision

    @workflow.query
    def phase(self) -> str:
        return self._current_phase

The pattern from Dockmaster: split a long workflow into phase mixins (selection, implementation, verification, cleanup), each implemented in its own file but composed into one workflow class. The event history captures the whole run as one history; the code stays readable.

Activities — the side-effect unit

An activity is the unit of work that touches the outside world. Activities can be retried independently of the workflow that scheduled them. They have their own timeouts, their own retry policies, their own resource requirements. Activities are where LLM calls, tool invocations, and database writes live.

Retry policies — the reliability lever

Per-activity retry policy is the single highest-impact reliability primitive Temporal offers. The components:

from temporalio.common import RetryPolicy
from datetime import timedelta

# From dockmaster/_base.py — production retry policies

def _agent_retry_policy() -> RetryPolicy:
    return RetryPolicy(
        initial_interval=timedelta(seconds=5),
        maximum_interval=timedelta(seconds=120),
        maximum_attempts=2,                              # agents are expensive — don't loop
        non_retryable_error_types=["AgentJobPermanentError"],
    )

def _state_retry_policy() -> RetryPolicy:
    return RetryPolicy(
        initial_interval=timedelta(seconds=1),
        maximum_interval=timedelta(seconds=30),
        maximum_attempts=3,                              # DB writes — retry briefly
    )

def _api_retry_policy() -> RetryPolicy:
    return RetryPolicy(
        initial_interval=timedelta(seconds=2),
        backoff_coefficient=2.0,                       # exponential
        maximum_interval=timedelta(minutes=5),
        maximum_attempts=5,                              # external APIs — patient
        non_retryable_error_types=["AuthenticationError", "NotFoundError"],
    )

Non-retryable error types — the most important field

Most retry-policy bugs come from retrying things that should never retry. An authentication error doesn't get better on retry. A 404 doesn't become a 200 by waiting. A content policy violation isn't transient. The non_retryable_error_types field lists exception type names; Temporal sees the exception class and refuses to retry. The Dockmaster pattern: an AgentJobPermanentError exists specifically so the agent activity can signal "this won't get better, fail the workflow."

Workflow-level error handling

When an activity exhausts its retries, the workflow sees an ApplicationError. The workflow code can catch and route. This is where business-level fallback logic lives — not in the activity itself.

try:
    pr_result = await workflow.execute_activity(
        agent.implement_with_claude_code,
        args=[input],
        start_to_close_timeout=timedelta(hours=2),
        retry_policy=self._agent_retry_policy(),
    )
except ApplicationError as err:
    if err.type == "AgentJobPermanentError":
        # Don't retry the agent — surface to humans via Slack
        await self._notify_slack(f"Agent permanently failed: {err.message}")
        ctx.mark_failed(err.message)
        return ctx
    raise                                  # let other errors propagate up

Child workflows

A child workflow is a separate workflow execution invoked from within a parent. It has its own event history, its own retry policy, its own ID. Useful when:

  • The child is independently reusable (other parents might also invoke it).
  • The child has its own scaling characteristics (different task queue, different worker pool).
  • The child's event history would otherwise bloat the parent's history.

The Dockmaster pattern uses a child workflow for the 5-agent review ensemble — DevLoopValidationWorkflow — keeping the parent's history clean while the child fans out to 5 parallel activity invocations.

The event history size trap

Per the 2026 platform comparison: "For AI and LLM workflows specifically, Temporal faces workflow history saturation issues due to large LLM payloads, requiring teams to implement payload codecs to offload data to external storage as a workaround." Each activity's input and output is stored in the event history. A workflow with 50 activity calls each carrying a 30KB LLM response writes 1.5MB to history. Temporal's hard cap is 50MB or 50,000 events per workflow. Large LLM payloads burn through this surprisingly fast.

The fixes:

  • Payload codecs. Store activity inputs/outputs in S3/GCS; pass references through Temporal.
  • Continue-as-new. When a workflow approaches the event limit, finish it and start a new one with the carry-over state. Long-running agents that loop forever should continue-as-new every N iterations.
  • Child workflows for fan-out. Move heavy-payload steps into child workflows that complete and don't burden the parent.
The biggest Temporal mistake

Treating workflow code as ordinary code. Mixing in HTTP calls, system time reads, file IO. The SDK sandbox catches most of this but not all. Non-determinism bugs are subtle — the workflow "works" when traffic is light, then fails after a worker restart because a replay produces different decisions. The discipline: workflows orchestrate, activities act. If a line of code in a workflow function would do anything other than schedule activities, wait for signals, evaluate conditions, or update workflow-local state, it probably shouldn't be there.

ORCH Chapter V

Signals, queries & the agent loop.

Signals Async messages into a running workflow. Used for human-in-the-loop, cancellation, dynamic input.

Queries Synchronous reads of workflow state. Used for status APIs, dashboards, debugging.

A workflow that just runs activities in sequence isn't dynamic enough for agents. Real agent workflows pause for human approval, accept mid-flight cancellations, expose status to dashboards, branch based on signals from external systems. Temporal's signal and query primitives are how the outside world talks to a running workflow — and how the workflow talks back without breaking determinism. This chapter walks them and the agent-loop patterns that emerge.

Signals — async messages into a workflow

A signal is a message sent to a running workflow. It triggers a handler defined on the workflow class; the handler can update workflow state, which the workflow's main run loop can observe. Signals are durable — they're recorded in event history, and a workflow can re-process them on replay.

def skip_issue(self, reason: str) -> None:
    # Pattern 1: user wants to abort — handler updates state
    if self._ctx:
        self._ctx.mark_skipped(reason)

@workflow.signal
def override_validation(self, decision: str) -> None:
    # Pattern 2: human approval — sets a flag the workflow waits on
    self._validation_override = decision

@workflow.signal
def cancel_workflow(self) -> None:
    # Pattern 3: graceful cancellation — handler raises CancelledError
    if self._ctx:
        self._ctx.mark_cancelled()

# Sending a signal from outside:
$ temporal workflow signal --workflow-id "dev-loop-ENA-42" \
    --name "skip_issue" --input '"superseded by ENA-43"'

The signal handler is called inline as part of workflow replay. It must be deterministic too — it can't do HTTP calls, it can't read the clock. It updates workflow-local state; the main run code observes the state change and reacts.

The human-in-the-loop pattern

The combination signals enable: a workflow that pauses for human review without holding open resources. The workflow calls an activity that posts a message to Slack (or Linear, or GitHub PR) with an approval URL. Then the workflow waits — using workflow.wait_condition — for a signal to arrive. The worker is free to handle other work while waiting.

async def run(self, input: ReviewInput) -> ReviewResult:
    # Step 1: post the PR for review
    await workflow.execute_activity(
        github.post_review_request,
        args=[input.pr_url, input.reviewer_team],
        start_to_close_timeout=timedelta(minutes=1),
    )

    # Step 2: wait for human decision (could be hours; worker is idle in the meantime)
    await workflow.wait_condition(
        lambda: self._validation_override is not None,
        timeout=timedelta(days=3),                  # auto-fail after 72h
    )

    # Step 3: act on the decision
    if self._validation_override == "approve":
        await workflow.execute_activity(github.merge_pr, args=[input.pr_url], ...)
        return ReviewResult(merged=True)
    else:
        await workflow.execute_activity(github.close_pr, args=[input.pr_url], ...)
        return ReviewResult(merged=False)

The implication for cost: a workflow that waits 36 hours for a human approval uses essentially zero compute during those 36 hours. The worker process polls the task queue and never gets a task for that workflow until a signal arrives. This is what "no more duct-taping Workflows around human input" per Temporal's marketing means in practice.

Queries — sync reads of workflow state

Where signals are writes, queries are reads. A query is a synchronous request that returns workflow-local state. Queries are also replay-safe and deterministic. They're how you build status dashboards on top of running workflows.

def phase(self) -> str:
    return self._current_phase

@workflow.query
def phase_progress(self) -> dict:
    return {
        "phase": self._current_phase,
        "progress": self._phase_progress,
        "iteration": self._validation_round,
    }

# Querying from outside (REST API, CLI, dashboard):
$ temporal workflow query --workflow-id "dev-loop-ENA-42" --type phase
# Output: "verification"

Queries are read-only by design. A query handler must not mutate workflow state. The Dockmaster pattern: every long-running workflow exposes phase, phase_progress, and current_findings_count queries so the Temporal UI and Slack notifications can show real-time progress.

The agent loop pattern

The agent loop — model decides, tool executes, result feeds back, repeat — fits neatly into Temporal's primitives. Each iteration is a sequence of activities; the workflow chains them; the agent loop terminates when the model emits a final answer or a hard limit is hit.

class AgentLoopWorkflow:
    @workflow.run
    async def run(self, input: AgentInput) -> AgentResult:
        messages = [input.system, input.user]
        max_iterations = 15

        for iteration in range(max_iterations):
            # Step 1: call the LLM as an activity
            response = await workflow.execute_activity(
                llm.chat_completion,
                args=[messages, input.tools],
                start_to_close_timeout=timedelta(minutes=5),
                retry_policy=RetryPolicy(maximum_attempts=3),
            )
            messages.append(response.message)

            # Step 2: if the model produced a final answer, exit
            if not response.tool_calls:
                return AgentResult(final=response.content, iterations=iteration + 1)

            # Step 3: execute each tool call as a separate activity
            for tool_call in response.tool_calls:
                tool_result = await workflow.execute_activity(
                    mcp_invoke,
                    args=[tool_call.name, tool_call.arguments],
                    start_to_close_timeout=timedelta(minutes=2),
                )
                messages.append({
                    "role": "tool",
                    "tool_call_id": tool_call.id,
                    "content": tool_result,
                })

        # Hard limit — fail loudly
        raise ApplicationError("Agent exceeded max iterations", non_retryable=True)

What durability buys you in the agent loop

  • Free retry on transient LLM errors. Rate limit hit on iteration 8 of 15? The activity retries; the workflow doesn't notice.
  • Free crash recovery. Worker dies on iteration 12. New worker picks up the workflow from history; iterations 1-11 don't re-run their LLM calls.
  • Iteration cap enforced by code, not config. The loop counter is workflow-local state; even if the workflow restarts, the counter is replayed correctly.
  • Spend bounded by retries × max_iterations × per-call cost. Predictable cost ceiling, even when models misbehave.
  • Tool call results in history. A tool that took 30 seconds and returned 50KB doesn't get re-executed on workflow restart.

Per the Temporal-MCP integration pattern

Per Temporal's framing: "Every @mcp.tool() in the system is backed by a deterministic Temporal Workflow, which gives us automatic retries and full replay for audit/compliance out of the box. The AI agents' tools became durable operations. I didn't have to worry about inconsistent state if an agent's tool call crashed midway — Temporal ensures either the whole Workflow completes or it doesn't happen at all, and we can always inspect what happened from the history. Using Temporal for tool implementations also kept the agent's reasoning loop deterministic." The pattern: not only is the agent loop a workflow, but each MCP tool's implementation can itself be a workflow. Tools that touch multiple external systems with their own consistency requirements (transactional state changes across two services) become durable by construction.

The override pattern

Production agent workflows benefit from a small set of "escape hatch" signals — skip, cancel, override_validation, force_advance. These let operators intervene without touching the database directly. The Dockmaster pattern documents these in the README so on-call engineers can find them: send a signal from the Temporal UI, the workflow handles it cleanly, audit history captures who did what. Treating these signals as part of the supported interface — rather than emergency-only — is what makes the system humane to operate over time.

Langfuse · LLM observability.

The model Hierarchical traces: trace → span → generation. Every LLM call, tool invocation, retrieval step captured with input, output, model parameters, tokens, cost.

The contract Async ingestion — never blocks the application. SDK queues events locally and flushes in batches.

If Temporal answers "what ran and in what order," Langfuse answers "what did the model see, what did it say, and what did it cost." The two layers are orthogonal — Temporal records workflow state transitions; Langfuse records prompt-and-response content. You need both. This chapter walks the Langfuse mechanism, the trace model, and the integration points that make it useful for production agent systems.

The trace model

Langfuse organizes events hierarchically. The top-level object is a trace — a complete user-facing interaction (one chat session, one workflow run, one agent invocation). Inside a trace, spans represent sub-operations (a retrieval, a tool call, an intermediate step). Generations are the leaf nodes — actual LLM calls with their prompts, completions, model parameters, tokens, and cost. Per the Langfuse docs: "Hierarchical traces capture every LLM call, tool invocation, and retrieval step. Filter by user, session, cost, latency, or custom metadata."


      
Why hierarchical matters for agents

      
A single agent workflow run involves dozens of LLM calls + dozens of tool invocations. A flat list of generations is unreadable. Per the 2026 observability landscape: "Langfuse traces are a list of observations (generations, spans, events). Great for prompt-centric apps. Slower to read when the 'trace' is a 30-minute agent run with nested tool calls and sub-agents." The hierarchical model lets you drill from the top-level trace down to a specific generation in three clicks. For Dockmaster, this means: "the dev loop failed → which phase → which agent → what prompt → what the model returned."


      
Async ingestion — the latency contract

      
The Langfuse SDK never blocks application code. Per the docs: "No. Langfuse SDKs send tracing data asynchronously in the background. Trace events are queued locally and flushed in batches, so your application's response time is not affected." The mechanism: the SDK accumulates events in an in-memory queue; a background thread flushes batches to the Langfuse server every few seconds; on shutdown, a final flush drains the queue. The application sees micro-second overhead at instrumentation points.

    pin_env = f"{role.upper().replace('-', '_')}_PROMPT_VERSION"
    version = os.environ.get(pin_env)

    if version:
        # Pinned: load that exact version
        path = Path(f"prompts/{category}/{role}/{version}.md")

    HTTP long-lived connection. Client opens GET to /sse stream; sends POSTs to /message.

    OAuth 2.1; bearer tokens; per-session identity.

    Networked servers reachable from Claude Desktop / Claude.ai. The 2024-2025 standard.

    Streamable HTTP

    Newer single-endpoint HTTP transport that streams responses. POST /mcp with optional SSE response.

    OAuth 2.1; same model as SSE.

    The 2026 default for networked MCP. Simpler than SSE; behind proxies more readily.

    The session lifecycle

    Every MCP connection follows the same handshake:

    1. Initialize. Client and server exchange protocol versions and capabilities. Negotiation: "I support these features; you support these; we agree on the intersection."
    2. Discover. Client asks for the available tools, resources, and prompts.
    3. Invoke. Client calls tools/call or resources/read as the model directs.
    4. Notify. Either side can send notifications — e.g., the server tells the client when its tool list changes.
    5. Terminate. Connection closes; session state on the server (if any) is cleaned up.

    Why standardization matters

    Before MCP, an agent that needed to call Slack, GitHub, Salesforce, and a custom internal API needed four different SDK integrations, four different auth flows, four different error-handling shapes. Adding a fifth integration was a multi-week project. With MCP:

    • One JSON-RPC client speaks to all five servers.
    • OAuth 2.1 standardizes auth — same flow for every server.
    • Tools, resources, prompts have consistent schemas — the client doesn't change when a server is added.
    • Servers can be developed and deployed independently — different teams, different languages, different release cadences.

    The 2026 ecosystem

    By mid-2026 the MCP ecosystem has matured significantly:

    • SDKs in every language. Python (modelcontextprotocol/python-sdk + FastMCP), TypeScript, Go, Rust, Java, .NET.
    • OAuth 2.1 specification finalized. RFC 8414 (auth server metadata) and RFC 9728 (protected resource metadata) are the discovery primitives.

    The mental model

    MCP is "OpenAPI for LLMs." The standardization is at the protocol layer (JSON-RPC primitives, OAuth model, discovery endpoints), not at the data model (each tool defines its own schema). The win is the protocol — once the agent client speaks MCP, every server is just a configuration entry.

    How the magic works

    FastMCP introspects each tool function via Python's typing module:

    • Parameter types → JSON schema. int → "type":"integer". str → "type":"string". Optional[X] → nullable. Pydantic models → nested object schema.
    • Default values → optional vs required. Parameters without defaults are required; with defaults are optional.
    • Docstrings → tool description. First line is the summary; reStructuredText / Google-style is preserved.
    • Return type → output hint. Used for client-side validation and rendering.

    When the client calls tools/list, FastMCP emits the schema for every registered tool. When the client calls tools/call, FastMCP routes to the function, validates arguments, executes, serializes the return.

    For production servers, tool functions need access to databases, HTTP clients, authentication context, configuration. FastMCP supports dependency injection via type annotations:

    Error contract

    MCP errors carry a code and a message back to the client. FastMCP surfaces Python exceptions automatically, but production servers should normalize:

    MCP Chapter XII

    OAuth 2.1, transports, security.

    The auth model OAuth 2.1 with PKCE S256. DCR for client registration. JWTs as bearer tokens. Refresh tokens rotated and SHA-256 hashed in Redis.

    The cardinal rule MCP spec forbids passing upstream tokens to the client. The MCP server is its own OAuth AS proxying to upstream providers.

    A networked MCP server exposed to Claude.ai or Claude Desktop carries the same security concerns as any internet-facing API — plus a few specific to the protocol. The MCP spec mandates OAuth 2.1 with PKCE for authorization; it forbids token passthrough (the MCP server can't hand Claude an upstream Salesforce token); it requires specific well-known endpoints for client discovery. Getting this right is non-trivial but well-trodden. The Haus salesforce MCP is the worked example.

    The threat model

    The reason MCP's spec is strict on auth: clients are LLMs operated by third parties (Anthropic). The model can be tricked. The conversation can be poisoned. Once the server hands Claude a Salesforce OAuth token, Claude could (in principle) be persuaded by injection to leak it. Therefore the protocol design forces server-side credential isolation: the MCP server holds the upstream tokens; Claude only sees server-issued JWTs scoped to MCP sessions; revoking access means rotating the server-side mapping.

    class BearerAuthMiddleware:
    async def __call__(self, scope, receive, send):
        # Skip auth for well-known and health endpoints
        if scope["path"].startswith(("/.well-known", "/health", "/ready", "/auth/")):
            return await self.app(scope, receive, send)

        request = Request(scope, receive)
        auth = request.headers.get("authorization", "")
        if not auth.startswith("Bearer "):
            return await _send_401(send)

        token = auth.removeprefix("Bearer ")
        try:
            payload = jwt.decode(
                token,
                self.secret,
                algorithms=["HS256"],
                audience=self.audience,
                issuer=self.issuer,
            )
        except jwt.InvalidTokenError:
            return await _send_401(send)

        # Attach session to request scope for downstream tools
        scope["session"] = await self.session_store.get(payload["sub"])
        if not scope["session"]:
            return await _send_401(send)

        return await self.app(scope, receive, send)

    Building custom MCP · auth & RBAC.

    Two layers Authentication (who is calling) vs Authorization (what can they do). MCP handles auth at the protocol; RBAC is your responsibility.

    The Haus pattern Firebase JWT for auth (haus-mcp-internal) + dedicated RBAC service (haus-rbac-api) called per-request for permission checks.

    MCP's OAuth flow establishes who is calling. It says nothing about what they're allowed to do. Production MCP servers need authorization in addition to authentication — fine-grained permissions per tool, per resource, per user. Haus's pattern: a dedicated RBAC microservice that every MCP server calls before executing a tool. This chapter walks the auth-vs-RBAC split, the Haus implementation, and the design decisions that make it operable across many MCP servers.

    Why a separate RBAC service

    Embedding permission rules in each MCP server has obvious appeal — fewer network hops, simpler deployment. The Haus choice to centralize anyway addresses three failure modes:

    • Consistency. Five MCP servers each implementing "is this user in the analytics team?" produces five subtly-different answers. A central service has one answer.
    • Auditability. Every permission check goes through one service that logs them. Compliance gets one source of truth.
    • Change velocity. Adding a new role or changing a permission is one PR to the RBAC service, not one PR per MCP server.

    The trade-off: a network call on every tool invocation. With a fast service (FastAPI on Python 3.13, MySQL + Postgres connection-pooled) the per-call latency is single-digit milliseconds. For tool calls that already take seconds (LLM-side), the RBAC overhead is invisible.

    The Haus RBAC API shape

    The haus-rbac-api README captures the design exactly:

    • Python 3.13, Poetry, FastAPI, Uvicorn. Standard Haus service shape.
    • Databases: MySQL (users, orgs, user-org membership) + PostgreSQL (roles, permissions, role bindings). Two databases because users live in the legacy system; roles/permissions are the new RBAC model.
    • Auth via pluggable user_sources adapters. Firebase implemented first; others (Okta, internal SSO) can plug in later.
    • PostgreSQL schema: roles, permissions, role_permissions (join table), role_bindings.
    • Migrations with dbmate. Files in haus_rbac/db/migrations/; schema dumped to schema_pg.sql.
    • The one endpoint that matters: POST /permissions/check with body {"token": "...", "org": "...", "permission": "entity:action", "adapter": "firebase"}.

    Fail-closed by default
    
      
    The most important RBAC client behavior: when the RBAC service is unreachable, deny the request. The opposite — allow on RBAC failure — is the security antipattern. The Haus client returns False on any exception; the MCP tool sees this as permission denied; the user gets an error rather than uncontrolled access. The trade-off is availability for security; pick security.
    

      
    Permission naming convention
    

      
    Haus uses entity:action as the canonical permission name format. The convention's value: clients can construct expected permission names without consulting documentation. experiments:list, experiments:read, experiments:create, experiments:delete. kpis:read, kpis:admin.     The four-questions test
    
        
    For any MCP tool that operates on user data, ask: (1) Who is calling? — authentication. (2) Are they who they say they are? — token validation. (3) Are they allowed to invoke this tool? — RBAC.     MCP Chapter XIV

    Tool design · read-only, scoped, idempotent.

    Three principles Start read-only. Scope writes narrowly. Make every action idempotent.

    The Haus salesforce-mcp Seven tools, all read-only. SOQL query, list objects, describe, get record, search, org limits, current user. No deletes, no updates.

    A well-designed MCP tool is a small, focused operation with a clear schema and predictable failure modes. A badly-designed MCP tool is a god-function that accepts a free-text "instruction" and tries to interpret it. The difference matters because the LLM is generating the tool call — clarity at the tool surface translates directly to model success at using the tool. This chapter walks the design principles that make tools reliable and the patterns Haus uses across its MCP servers.

    Principle 1 — start read-only

    Per the haus-salesforce-mcp README: "MCP Tools (all read-only)." All seven tools the server exposes — get_current_user, run_soql_query, list_objects, describe_object, get_record, search_records, get_org_limits — read data. None modify, delete, or create. The reasoning:

    • Blast radius is bounded. An LLM that misinterprets a user request and queries the wrong Salesforce objects produces wasted time, not data loss.
    • Prompt injection has less leverage. A poisoned conversation trying to "delete all opportunities" has no tool to do so.
    • Audit is simpler. Read access logs are noisy but not consequential. Write access logs are forensic-grade evidence.
    • Iteration is safer. Adjusting a read-only tool's behavior doesn't risk corrupting data.

    Add writes when the use case demands them, with explicit scoping and idempotency. Not before.

    Principle 2 — scope writes narrowly

    When write tools are needed, the question to answer first: what's the smallest possible blast radius that still solves the problem? Compare:

    # BAD — unbounded write surface
@mcp.tool()
async def execute_database_query(sql: str) -> dict:
    """Execute arbitrary SQL against the database."""
    return await db.execute(sql)
# The LLM can drop tables. The user can be tricked into pasting malicious SQL.
# Every database is now in the blast radius of one tool call.

# GOOD — specific operation, validated inputs
@mcp.tool()
async def assign_ticket(
    ticket_id: str = Field(pattern=r"^[A-Z]+-\d+$"),
    assignee_email: str = Field(pattern=r"^[^@]+@hausanalytics\.com$"),
) -> dict:
    """Assign a Linear ticket to a user."""
    return await linear.update_assignee(ticket_id, assignee_email)
# Pattern validation forces well-formed inputs.
# Only Linear ticket assignment changes; nothing else is touched.
# Email domain restricted to internal — no external assignments possible.

    Principle 3 — make actions idempotent

    An MCP tool can be called twice. By the model retrying (it didn't see the response). By the orchestrator retrying (the activity timed out). By the user. Idempotent tools survive duplicates without harm; non-idempotent tools require explicit deduplication.

    The tool minimization rule

    The instinct on building an MCP server is to expose every capability of the underlying system. Resist this. A server with 12 carefully-chosen tools outperforms one with 50 thinly-wrapped API operations. The agent has limited attention; every tool added is a distraction the model has to consider on every step. Add tools when missing one makes a real workflow impossible — not when one might be useful someday.

    The two-probe pattern

    Per the haus-mcp-internal README: "K8s health probes: Liveness: GET /health — always 200 if process is alive. Readiness: GET /ready — 200 when AlloyDB and BigQuery clients are initialised, 503 otherwise." The split is critical:

    • Liveness (/health). "Is the process responding to HTTP at all?" Returns 200 unconditionally. Used by kubelet to decide whether to kill+restart the pod. Should never fail except in catastrophic process states (deadlock, OOM not yet killed). Failing liveness during startup or transient upstream issues triggers unhelpful restart loops.
    • Readiness (/ready). "Is the pod ready to serve traffic?" Returns 200 only when upstream dependencies are reachable. Failing readiness removes the pod from the Service endpoints but doesn't restart it. Used to route traffic away during transient issues.

    Redis — session state location

    OAuth state, auth codes, MCP refresh tokens, session data — all of these need persistence across pod restarts and shared visibility across replicas. Redis is the natural fit. Options:

    • Cloud-managed Redis (Cloud Memorystore, ElastiCache, Upstash). The Haus pattern — production servers connect to GCP Memorystore via SSL using a CA cert mounted in the pod.
    • In-cluster Redis (Bitnami chart, KubeDB). Cheaper but you operate it.
    • Redis sidecar. Local to the pod. Useless for multi-replica deployments — each replica has its own state.

    For development, the haus pattern uses a local Redis Docker container started by Make targets (make redis). For production, cloud-managed with SSL.

    The deploy-via-PR-label pattern

    Per the haus-salesforce-mcp README: "Development preview: Label a PR with deploy-to-haus-development." Per the haus-mcp-internal: "Deployed to K8s via the deploy-to-haus-development label on a PR. See .github/workflows/deploy-pr-to-k8s.yaml."

    The mechanism: a GitHub Action watches for the label; on label-add, builds the PR's image, tags it with the PR number, applies a per-PR overlay to the K8s manifests, deploys to a development namespace. The preview environment runs the exact code from the PR. Reviewers can connect Claude Desktop to it and validate end-to-end before merge.

    Observability

    MCP servers should emit standard service metrics plus protocol-specific ones:

    • HTTP request metrics — path, method, status, duration histograms. Standard Prometheus.
    • MCP-specific metrics — tool_call_total by tool name and result, auth_failure_total by reason, session_active gauge.
    • OAuth flow metrics — token_issued_total, refresh_token_used_total, dcr_registration_total.
    • Upstream health — calls to underlying APIs (Salesforce REST, AlloyDB, BigQuery), latency p99, error rates.
    • Sentry for error capture. The Haus pattern includes Sentry DSN env vars in every MCP server for unhandled exception reporting with trace context.
    The single-region trap

    MCP servers are often deployed single-region for simplicity. The trap surfaces when Claude.ai users connect from anywhere in the world and experience high latency. For latency-sensitive tool workflows (autocomplete-like UX), put MCP servers behind global load balancers and replicate to multiple regions. Session state remains the hard problem — either replicate Redis cross-region (cost) or accept that re-auth on regional failover is acceptable (operational simplicity).

    HAUS Chapter XVI

    The Haus reference · Dockmaster end-to-end.

    What it does Picks a ready Linear ticket, assigns it to a Claude Code agent in a K8s Job, drives 5-agent review, enables automerge after CI passes.

    Why it matters here It composes Temporal + Langfuse + MCP into one operable system. The worked example for the whole issue.

    Dockmaster is Haus's autonomous agent orchestrator. It picks issues from the product backlog, assigns them to Claude Code agents running in Kubernetes Jobs, drives implementation through a 5-agent review ensemble, and merges the result after human approval. It's the worked example of the three-layer stack — Temporal for orchestration, Langfuse for observability, MCP for tool access — composed into a single operable system. This chapter walks the end-to-end flow.

    The trigger

    A Dockmaster run starts from one of three sources:

    • Linear webhook. A ticket is labeled ready-for-agent; Linear posts to POST /webhooks/linear; the webhook server parses the ticket description for a Repo: HausAnalytics/<name> marker; starts a DevLoopWorkflow.
    • Schedule. A Temporal Schedule runs on a cron expression for each configured project; the scheduler entrypoint reads config/registry.yaml and triggers runs.
    • Manual. ./scripts/trigger-workflow.sh — a shell script that runs inside the worker pod via kubectl exec, starts a workflow with chosen options.

    The workflow shape

    Dockmaster · DevLoopWorkflow
    From trigger to merged PR through four phases

    The phases as Temporal mixins

    The DevLoopWorkflow class composes phase mixins — each phase is a separate Python file with a single method on the workflow class:

    • _SelectionPhase._selection(ctx) — runs a Claude agent to pick a ticket from the Linear backlog via Linear MCP.
    • _ImplementationPhase._implementation(ctx) — spawns a Claude Code agent in a K8s Job; the agent writes code, runs tests, opens a PR.
    • _VerificationPhase._verification(ctx) — CI fix loop, then the 5-agent review ensemble as a child workflow, then automerge enablement.
    • _CleanupPhase._cleanup(ctx) — terminate any remaining K8s Jobs, record final audit events.

    Every agent invocation runs in its own K8s Job. The orchestrator's k8s/job_builder.py constructs the spec; activities spawn the Job and poll for completion.     The retry policy story

    Each activity has the appropriate retry policy from _base.py:

    • Agent activities (LLM-cost-heavy): 5s-120s, max 2 attempts, non_retryable = AgentJobPermanentError.
    • State writes (DB-bounded): 1s-30s, max 3 attempts.
    • External API calls (Linear, GitHub): exponential 2s up to 5min, max 5 attempts, non_retryable = AuthenticationError/NotFoundError.

    The signal escape hatches

    Three signals make every running workflow controllable from the Temporal UI:

    • skip_issue("reason string") — skips to cleanup without merging.
    • cancel_workflow — gracefully cancels the workflow.
    • override_validation("approve" | "reject") — force-overrides the review ensemble verdict.

    An on-call engineer can intervene on any running workflow without touching the database or killing pods.

    The MCP servers in use

    The agent-runtime's config/mcp_servers.json declares which MCP servers the Claude Code agent has access to. The current production list:

    • Linear MCP (via mcp-remote bridge to https://mcp.linear.app/mcp) — read issues, comment on them.
    • GitHub via the official GitHub App auth, surfaced as built-in tools rather than MCP.
    • Internal MCP servers available but not enabled for Dockmaster agents (haus-mcp-internal, haus-salesforce-mcp) — separate use cases.

    Langfuse correlation

    Every Claude Code invocation carries the Langfuse correlation envelope as env vars (chapter VII). The agent runtime initializes Langfuse tracing on startup and emits traces with workflow_id as session_id. Dashboards aggregate by retry round, by role, by prompt version, by Linear issue. The four boolean scores — pr_opened, tests_passed, review_approved, ci_passed — feed the production health metric.

    Prompt management in flight

    Prompts live in agent-runtime/prompts/{execution,review}/{role}/v{N}.md. CI syncs to Langfuse on merge. Per-role env var pins the version for rollback. The agent runtime auto-selects the highest version unless pinned.

    The end-to-end timeline of one run

    1. Linear webhook fires: ticket ENA-42 labeled ready-for-agent.
    2. Webhook server parses repo from description, starts DevLoopWorkflow on Temporal.
    3. Workflow enters Selection phase: spawns K8s Job running the selection agent; agent uses Linear MCP to scan candidates; returns ENA-42 as the pick.
    4. Workflow enters Implementation phase: spawns Implementation Job; Claude Code clones the repo, reads CLAUDE.md, implements the change, runs tests, opens PR #1234. Activity returns the PR number.
    5. Workflow enters Verification phase: polls CI status; on failure, spawns CI-fix Job (up to 3 attempts). When CI is green, invokes DevLoopValidationWorkflow as a child workflow.
    6. Validation child workflow fans out 5 parallel activities — one per review role. Each spawns its own K8s Job. Findings are aggregated by SHA-256 fingerprint to deduplicate across rounds.
    7. If reviewers request changes: spawn Fixer Job; address findings; re-run validation. Loop up to N rounds.
    8. When all 5 reviewers approve: enable automerge on the PR.
    9. Human pushes the GitHub "Approve" button; PR merges.
    10. Workflow's polling activity detects merged status; transitions to Cleanup; records final audit events.
    11. Every step has been traced to Langfuse with workflow_id as session_id; every state change is in the Temporal event history; every tool call went through MCP with bearer-token auth.
    Why this composes

    The three layers don't merely coexist — they fit. Temporal handles durability so the agent loop can recover from any crash mid-flight. Langfuse captures the LLM-side narrative that Temporal's history alone can't show. MCP standardizes external access so the same agent code works against Linear, GitHub, internal data, and Salesforce without integration-specific scaffolding. The Haus implementation is one realization of a pattern that's becoming standard across the 2026 agent-platform space; the chapters that follow give you the picker and checklist to assess your own implementation against it.

    The anti-patterns the picker prevents

    • Picking Inngest because TypeScript when most code is Python. The picker raises language first; Python teams should default to Temporal even if the JavaScript ecosystem is louder.
    • Building bespoke RBAC inside each MCP server. The picker forces the multi-server question; once multiple servers exist, central RBAC pays for itself within a quarter.

    Signal sent before workflow handler attached; signal lost during worker restart that wasn't graceful.
    Verify signal handler is decorated correctly; check Temporal UI shows the signal event in history; signals on running workflows are durable.
    Replay produces different decisions
    Workflow code has hidden non-determinism; third-party library updated between deploys; dict iteration order assumption.
    Pin all dependencies; sort dict keys when ordering matters; use Temporal's Patched API for intentional behavior changes.

    The 24-item production checklist

    Workflow code free of non-determinism
    No random, datetime.now(), network calls, file IO inside workflow functions. All side effects in activities.
    Per-activity retry policies defined
    Agent activities, state writes, external API calls each have explicit RetryPolicy with appropriate maximum_attempts.
    non_retryable_error_types populated
    AuthenticationError, NotFoundError, business-logic permanent errors listed; prevents retry storms on dead-end failures.
    Activity timeouts realistic
    start_to_close_timeout set per activity; heartbeats for long-running ones; heartbeat_timeout <= 2× heartbeat interval.
    Event history kept under 50MB
    Large LLM payloads offloaded via payload codec; long-running loops use continue-as-new every N iterations.
    Escape-hatch signals defined
    skip, cancel, override_validation on every long workflow. Documented in README.
    Agent iteration cap enforced
    Hard cap on agent loops (10-20 iterations); raises non-retryable ApplicationError when exceeded.
    Langfuse correlation IDs propagated
    LANGFUSE_SESSION_ID = workflow_id; LANGFUSE_USER_ID, plus business IDs (issue, PR, branch) injected into every agent process.
    init_tracing() called early
    Langfuse SDK initialized in agent main entrypoint before any @observe-decorated function executes.
    Outcome scores emitted
    Boolean scores (pr_opened, tests_passed, review_approved, ci_passed) tagged on traces; aggregated into a production health dashboard.
    Prompts versioned as files
    Numbered vN.md per role in a prompts/ directory; never edited in place once merged to main.
    CI sync workflow active
    .github/workflows/sync-prompts.yml pushes new versions to Langfuse with candidate + codebase-v{N} labels.
    Per-role rollback env vars documented
    {ROLE}_PROMPT_VERSION=vN mechanism in README; runbook for prompt-regression rollback.
    MCP servers use OAuth 2.1 + PKCE
    PKCE S256 required (not plain); DCR allowed redirect domains restricted; refresh tokens hashed in Redis.
    Well-known endpoints published
    RFC 8414 and RFC 9728 endpoints serve correct JSON; tested with curl from outside the VPC.
    JWT secrets in Secret Manager
    No JWT signing keys committed to git; rotated periodically; per-environment values.
    Tools start read-only
    Write tools added only when read-only can't solve the use case; scoped narrowly with input validation patterns.
    Tool actions idempotent
    Idempotency keys supplied by caller, or natural upsert semantics; double-invocation produces same final state.
    RBAC fail-closed
    When RBAC service unreachable, MCP tool returns Permission Denied — never allows by default on RBAC error.
    Permission names follow entity:action
    Consistent convention across all MCP servers (e.g., experiments:list, kpis:read).
    /health vs /ready split correctly
    Liveness always returns 200 when process alive; readiness 503 when upstream deps unreachable.
    Ingress configured for streaming
    proxy-buffering: off; proxy-read-timeout: 3600; tested with curl SSE on /mcp endpoint.
    Pentest run before production exposure
    OAuth flow, redirect URI validation, token storage, refresh-token reuse tested by security review.
    Disaster runbook documented
    Procedures for: prompt rollback, workflow cancellation, MCP server outage, RBAC degradation, spend anomaly. Rehearsed.
