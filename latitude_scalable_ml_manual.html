03Where the time actually goes

The central, counterintuitive fact of large-scale training: most training jobs are not compute-bound. They are input-bound, and the GPUs sit idle waiting for data. Internalize this and half the interview answers write themselves.

THE THROUGHPUT CHAIN — the slowest link sets the pace ─────────────────────────────────────────────────────────────── object store ──► network ──► CPU decode ──► augment ──► collate ──► pinned buffer ──► H2D copy ──► GPU compute │ ◄────┘ and for multi-GPU, add: gradient all-reduce, and the SLOWEST RANK ─────────────────────────────────────────────────────────────── TYPICAL AV NUMBERS, to reason with (illustrative): one training sample = 6 camera JPEGs + a lidar sweep ≈ 6 x 200 KB + 2 MB ≈ 3 MB at 100 samples/s/GPU x 8 GPUs = 2.4 GB/s from storage → this ALONE saturates many network/storage setups JPEG decode ≈ 5 ms/image on one core → 30 ms/sample → 100 samples/s needs ≈ 3 CPU CORES OF DECODE PER GPU ─────────────────────────────────────────────────────────────── THE THREE FIXES, in order of leverage: 1 MOVE FEWER BYTES — pre-resize, pre-crop, store at the resolution you train at, use a compact format 2 DECODE LESS — cache decoded tensors, use GPU decode (nvJPEG/DALI), or store pre-decoded shards 3 OVERLAP — enough workers, prefetch, pin_memory, non_blocking copies (§11) ─────────────────────────────────────────────────────────────── AND THE ONE PEOPLE MISS: the storage layer hates small files. Millions of individual JPEGs on object storage is a request-rate problem, not a bandwidth problem. SHARD into large sequential files (§15). This single change is often a 5-10x win.

The sentence to have ready: "Before I optimize a training loop, I'd find out whether the GPUs are actually busy." A profiler timeline or even nvidia-smi sampling answers it in minutes. If utilization is 40% and the CPUs are pinned, no amount of mixed precision or torch.compile will help — the fix is in the data path. That diagnosis-before-treatment instinct is the same one you'd bring from GPU kernel work, applied one layer up.

04The multi-sensor data problem

AV training data is unlike ImageNet in ways that break naive pipelines. The req names "multi-sensor data fusion and processing" as a nice-to-have; knowing why it's hard is worth more than knowing any specific fusion architecture.

PropertyConsequence for the pipeline
Heterogeneous shapesImages are fixed-size; a lidar sweep has a variable point count; detections vary per frame. Default collate can't stack them — you need padding plus masks, or ragged/nested structures (§12).
Time alignmentSensors run at different rates with different latencies. A "sample" is a window of synchronized measurements, and building it correctly is an indexing problem you solve once at dataset-build time, not per-epoch.
Enormous and imbalancedPetabytes of mostly-boring highway driving. The valuable frames — cut-ins, construction, unprotected turns, unusual weather — are rare. Sampling strategy is a first-class part of the pipeline, not an afterthought.
Temporal correlationConsecutive frames are nearly identical. Random shuffling across a whole log leaks information between train and val; you generally split by drive segment, not by frame.
Calibration and metadata travel with itIntrinsics, extrinsics, ego pose. A "sample" is a bundle, not a tensor — which shapes both the storage schema and the collate function.
Reprocessing is routineA new auto-labeller means regenerating derived data for the whole archive. That's the orchestration and versioning problem (§32–33), and it's why lineage matters.
The subtle one worth raising unprompted

Temporal leakage. If you shuffle frames and then split train/val, near-duplicate frames land on both sides and your validation metric becomes optimistic — sometimes dramatically. The fix is to split at the level of a drive or a log segment, and to be careful that any mined/rare-event sampling doesn't quietly reintroduce the same frames. Naming this shows you understand AV data rather than generic vision data, and it's a genuine correctness issue rather than a performance one.

05Tensors, dtypes, devices, layout

ConceptWhat matters in a training pipeline
dtypefloat32 for master weights, bfloat16 for compute (wider exponent than fp16, so it rarely needs loss scaling), float16 where bf16 isn't supported, uint8 for images until the last possible moment. Keeping images as uint8 through the DataLoader and normalizing on GPU cuts host→device bytes by 4×.

# Keep bytes small on the wire, cast late, on the device.
# DataLoader yields uint8 [B, 3, H, W]; normalize on GPU.
imgs = batch["img"].to(device, non_blocking=True)      # uint8, 1 byte/px
imgs = imgs.to(torch.float32).div_(255.0)              # cast AFTER transfer
imgs = (imgs - mean) / std                             # mean/std live on device
imgs = imgs.contiguous(memory_format=torch.channels_last)

# 4x less PCIe traffic than sending float32 from the worker, and the
# normalization is free on the GPU. On an input-bound job this is one of
# the highest-leverage one-line changes available.

06Views, contiguity, and hidden copies

Verified behaviours worth being precise about, because each one is a silent performance or correctness trap.

OperationBehaviour (executed and checked)
t.transpose(1,2)Returns a non-contiguous view — no copy, but strides are now permuted.
v.view(-1) on that viewRaises. view requires compatible strides. Use reshape (which copies if it must) or .contiguous().view(...) — and know which you chose, because one is free and one is a full copy.
.contiguous()Copies into a dense layout. Cheap to write, not cheap to run — in a hot loop it's a full tensor copy every iteration.
x[mask], torch.nonzero, .item()All force a host synchronization because the output shape (or value) depends on device data. One .item() per step for logging is usually fine; one inside an inner loop serializes your pipeline.
In-place ops (add_, div_)Save memory, but can break autograd if the value is needed for a backward. Safe on inputs and on things outside the graph; think twice inside it.
The accumulation bug that leaks memory

total_loss += loss keeps the entire autograd graph for every step alive. Over an epoch that is an out-of-memory failure with a confusing traceback. It must be total_loss += loss.item() (or .detach()). This is the single most common memory leak in training code and it is worth naming as such — see Tree 2 (§40).

08Autograd in practice

Four behaviours, all verified by execution, that a training-infrastructure engineer must have exactly right.

ConstructVerified behaviour
torch.no_grad()Outputs have requires_grad=False; no graph is built. The standard wrapper for validation and inference.
torch.inference_mode()Stronger: outputs are inference tensors (.is_inference() is True) with view/version tracking disabled. Confirmed: an inference tensor cannot later re-enter autograd — attempting to backward through one raises. Faster than no_grad, but don't produce anything you'll later want gradients through.
.detach()Cuts the graph while sharing storage. Use for logging, for metrics, and for anything you stash in a list.
zero_grad(set_to_none=True)The default in modern PyTorch. Frees the gradient tensors instead of filling them with zeros — less memory and one fewer kernel per parameter.

Gradient accumulation — the arithmetic, confirmed

Accumulating N micro-batches must divide the loss by N, because the loss function already averages within each micro-batch. Executed check: four micro-batches of 16 with loss/4 reproduced a single batch of 64 to 3.0e-08. Omitting the division scaled the gradients by exactly 4× — which in practice looks like a mysteriously bad learning rate.

ACC = 4                      # effective batch = ACC * per_step_batch * world_size
opt.zero_grad(set_to_none=True)
for i, batch in enumerate(loader):
    # DDP: skip the all-reduce on all but the last micro-batch, otherwise you
    # pay full gradient communication ACC times per optimizer step.
    sync = ((i + 1) % ACC == 0)
    ctx = contextlib.nullcontext() if sync else model.no_sync()
    with ctx:
        with torch.autocast("cuda", dtype=torch.bfloat16):
            loss = criterion(model(batch), batch["target"])
        (loss / ACC).backward()          # <-- the /ACC is not optional

    if sync:
        # clip_grad_norm_ returns the PRE-clip norm; it does NOT sanitise NaN
        gn = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
        if torch.isfinite(gn):           # verified: you must check yourself
            opt.step()
        else:
            log.warning("non-finite grad norm, skipping step")
        opt.zero_grad(set_to_none=True)
        sched.step()
Two details in that snippet worth saying out loud

model.no_sync(). Without it, DDP all-reduces gradients on every micro-batch — so accumulation costs you 4× the communication for no benefit. This is one of the most common real inefficiencies in distributed training code.

The finite check. Verified: clip_grad_norm_ happily returns nan and does not clean the gradients. Stepping on NaN gradients poisons the weights permanently, and the loss curve goes flat several hundred steps later, far from the cause (Tree 3).

Activation checkpointing

Trades compute for memory: discard intermediate activations in the forward pass and recompute them during backward. Typically ~30% slower per step for a large reduction in activation memory — which often lets you raise the batch size enough to come out ahead. torch.utils.checkpoint.checkpoint(fn, *args, use_reentrant=False); use the non-reentrant version, which composes properly with modern features.

09Optimizers, schedulers, param groups

# Param groups: the standard "no weight decay on norms and biases" split.
# Applying weight decay to LayerNorm/BatchNorm params measurably hurts.
decay, no_decay = [], []
for name, p in model.named_parameters():
    if not p.requires_grad:
        continue
    if p.ndim <= 1 or name.endswith(".bias"):      # norms, biases
        no_decay.append(p)
    else:
        decay.append(p)

opt = torch.optim.AdamW([
    {"params": decay,    "weight_decay": 0.05},
    {"params": no_decay, "weight_decay": 0.0},
], lr=base_lr, betas=(0.9, 0.95), fused=True)      # fused=True on CUDA

# Linear warmup + cosine decay, as a closed form. Verified monotonic in both
# phases, bounded by [0, base_lr], and reproduced exactly by LambdaLR.
def lr_lambda(step):
    if step < warmup_steps:
        return (step + 1) / warmup_steps
    p = (step - warmup_steps) / max(1, total_steps - warmup_steps)
    return 0.5 * (1.0 + math.cos(math.pi * p))

sched = torch.optim.lr_scheduler.LambdaLR(opt, lr_lambda)
PointDetail
fused=True / foreach=TrueFused optimizers do the whole update in one kernel instead of several per parameter. On a model with hundreds of tensors this is a real step-time win, and it's a one-word change.
Scheduler steppingPer optimizer step, not per micro-batch — otherwise your schedule runs ACC× too fast. Verified ordering: the scheduler advances after the first optimizer step, so param_groups[0]["lr"] read after sched.step() reflects step 1, not step 0.
State sizeAdamW keeps two extra fp32 tensors per parameter. For a 1 B-parameter model that's ~8 GB of optimizer state alone — the reason FSDP shards it (§18).
Scaling the LRWhen the effective batch grows with world size, the LR must grow too — linear scaling with warmup is the standard recipe (§21).

The practical middle ground

Keep a lightweight manifest (a Parquet index of sample IDs, shard locations, and metadata) that fits in memory, so you get map-style random access and sampler-based shuffling — but have __getitem__ read from large sequential shards rather than millions of small files. You get index-level flexibility with storage-friendly access patterns. This is the design most mature AV pipelines converge on, and it's a good answer to "how would you structure the dataset layer?"

Two failure modes worth naming

Too many workers. Each is a process with its own memory and its own share of the CPU. Past the point where decode saturates the cores, more workers add contention and RAM pressure and make throughput worse. It's a tuned number, not a maximized one.

Workers that die silently. An OOM-killed worker surfaces as a cryptic "DataLoader worker (pid N) is killed by signal" — usually caused by the copy-on-write trap above, or by a memory leak in decode. Setting num_workers=0 temporarily moves the traceback into the main process and makes the real error visible; that's the first debugging move.

12collate_fn for sensor data

The default collate stacks tensors, which requires identical shapes. Lidar sweeps have different point counts, and frames have different numbers of detections — so for AV data you almost always write your own. The verified pattern is padding plus an explicit mask.

def pad_collate(batch):
    # Variable-length point clouds -> padded tensor + boolean mask.
    # Verified: mask.sum(1) == true lengths, padded region is exactly zero,
    # and fixed-size modalities still stack normally.
    lens = torch.tensor([b["pts"].shape[0] for b in batch])
    max_n = int(lens.max())
    C = batch[0]["pts"].shape[1]

    pts  = torch.zeros(len(batch), max_n, C)
    mask = torch.zeros(len(batch), max_n, dtype=torch.bool)
    for i, b in enumerate(batch):
        n = b["pts"].shape[0]
        pts[i, :n] = b["pts"]
        mask[i, :n] = True

    return {
        "pts":  pts,                                    # [B, max_n, C]
        "mask": mask,                                   # [B, max_n]
        "lens": lens,                                   # [B]
        "img":  torch.stack([b["img"] for b in batch]), # fixed size: stacks
        "meta": [b["meta"] for b in batch],             # keep as a list
    }

# THE CONSEQUENCE THE MASK EXISTS FOR -- verified that these DIFFER:
masked_mean = (pts * mask.unsqueeze(-1)).sum(1) / lens.unsqueeze(1)   # correct
naive_mean  = pts.mean(1)                                            # WRONG:
#   averages the zero padding in, so the answer depends on the batch's
#   longest sample. Every reduction downstream of a padded tensor must be
#   mask-aware, and this is where that discipline starts.
AlternativeWhen it's better
Concatenate + batch indexInstead of padding, concatenate all points into [total_N, C] and carry a batch_idx: [total_N] column. No wasted memory and no masking — and it's the layout sparse/scatter ops want. This is what most production lidar pipelines actually do; padding wastes memory when point counts are highly variable.
Fixed-size samplingRandomly subsample or pad every sweep to exactly N points. Simplest, and it makes shapes static — which matters for torch.compile and for predictable memory. Costs some information.
NestedTensorPyTorch's native ragged type. Improving, but support across ops is still incomplete — mention it as a direction rather than a default.
Where the collate should run

Inside the worker (which is where collate_fn already runs) — so the padding and stacking cost is parallelized across num_workers rather than serialized in the training process. That's automatic with the DataLoader, but worth stating, because engineers sometimes "optimize" by moving collation into the main loop and quietly serialize it.

14Sharding, and the duplication bug

With IterableDataset, nothing shards for you. There are two axes — DataLoader workers and distributed ranks — and forgetting either silently multiplies your data.

MEASURED: a 100-element IterableDataset, num_workers=4 ─────────────────────────────────────────────────────────────── NAIVE __iter__ returns iter(range(100)) → DataLoader yielded 400 items. Every worker replayed the WHOLE dataset. Your "epoch" is 4 epochs of duplicated data, your loss curve looks fine, and your model has quietly seen each sample 4 times per epoch. SHARDED each worker takes a disjoint stride → DataLoader yielded exactly 100 items, all unique. ─────────────────────────────────────────────────────────────── AND IN DISTRIBUTED TRAINING THERE ARE TWO AXES: global_shards = world_size x num_workers shard_id = rank * num_workers + worker_id Forget the rank axis and every GPU trains on the same data — which looks like training working, and is not.
class ShardedIterableDataset(IterableDataset):
    # Shards across BOTH distributed ranks and DataLoader workers.
    # Verified: with 4 workers this yields each item exactly once
    # (the naive version yields the whole dataset 4 times over).
    def __init__(self, shard_paths):
        self.shard_paths = sorted(shard_paths)   # sorted => deterministic

    def __iter__(self):
        # --- axis 1: distributed rank ---
        if dist.is_available() and dist.is_initialized():
            rank, world = dist.get_rank(), dist.get_world_size()
        else:
            rank, world = 0, 1

        # --- axis 2: DataLoader worker ---
        info = torch.utils.data.get_worker_info()
        wid, nworkers = (info.id, info.num_workers) if info else (0, 1)

        shard_id, num_shards = rank * nworkers + wid, world * nworkers

        # shard at FILE granularity: keeps reads sequential, which is what
        # object storage is good at. Requires len(shards) >= num_shards,
        # ideally a comfortable multiple so the work divides evenly.
        for path in self.shard_paths[shard_id::num_shards]:
            yield from read_shard(path)
Three consequences to state

Shard count must exceed total workers. With 8 ranks × 8 workers you need ≥ 64 shards, and ideally a few hundred, or some workers idle while others carry the load. Shard count is a dataset-build decision that constrains your training topology — plan it deliberately.

Uneven shards cause hangs. If one rank runs out of data before the others, it stops participating in collectives and the job deadlocks (Tree 5). Either equalize shard sizes at build time, or agree on a global step count and stop everyone together.

Shuffling needs two levels. Shuffle the shard order per epoch and maintain a shuffle buffer within a shard — otherwise samples within a shard always appear in the same sequence, and consecutive AV frames are highly correlated (§04).

17DDP mechanics

DDP IS AN AUTOGRAD PLUGIN — the mechanism, not the API ─────────────────────────────────────────────────────────────── SETUP every rank holds a FULL replica, initialized identically (DDP broadcasts rank 0's weights at construction). Parameters are packed into BUCKETS (~25 MB), built in REVERSE registration order — because backward produces gradients roughly in reverse of the forward. BACKWARD DDP registers a post-accumulate-grad HOOK on every param. The autograd engine fires it the moment that parameter's gradient is final. When every param in a bucket is ready, DDP launches that bucket's ALL-REDUCE asynchronously. ⇒ COMMUNICATION OVERLAPS WITH THE REST OF BACKWARD. The overlap IS the design; it is why DDP scales. STEP wait for outstanding all-reduces; gradients are now the mean across ranks; every rank steps its optimizer identically, so replicas stay in sync with NO weight broadcast. ─────────────────────────────────────────────────────────────── THE FAILURE MODES THAT FALL OUT OF THIS DESIGN • a parameter that receives NO gradient on some rank (an unused head, a conditional branch) ⇒ its bucket never completes ⇒ that rank never launches the all-reduce ⇒ EVERYONE HANGS. find_unused_parameters=True fixes it by traversing the graph each step — at real cost. Better: fix the model. • ranks executing DIFFERENT NUMBERS of collectives (uneven data, a rank-dependent branch, early exit) ⇒ deadlock (Tree 5) • gradient accumulation without no_sync() ⇒ ACC x the comms (§08)
SettingWhat it does
gradient_as_bucket_view=TruePoints gradients directly at bucket memory instead of copying — saves a full model's worth of gradient memory. Cheap win, commonly missed.
static_graph=TruePromises the graph never changes across iterations, enabling extra optimizations and making find_unused_parameters unnecessary in many cases.
bucket_cap_mbTrades overlap granularity against all-reduce efficiency. Larger buckets = fewer, bigger collectives = better bandwidth utilization, but less overlap.
broadcast_buffersSyncs buffers (BatchNorm running stats) each forward. Disable if you don't need it; better still, use SyncBatchNorm when per-rank batches are small.
DistributedDataParallel vs DataParallelNever use DataParallel. Single-process multi-thread, GIL-bound, and it scatters/gathers every step. It exists for backwards compatibility.

18FSDP and sharding strategies

DDP replicates everything, so the largest model you can train is bounded by one GPU's memory. FSDP shards parameters, gradients, and optimizer state across ranks, materializing full weights only transiently.

The honest positioning for an AV perception team

Perception models are typically hundreds of millions of parameters, not hundreds of billions — so DDP is usually the right tool and FSDP is usually unnecessary. The scaling problem here is data, not model size. Saying that plainly — "I'd reach for DDP with a well-tuned input pipeline first, and only consider FSDP or activation checkpointing if the model genuinely doesn't fit" — is a better answer than reflexively naming the most sophisticated technique. It also happens to be exactly what the req describes: efficiency of training with large volumes of data.

Three setup mistakes that produce confusing failures

Not calling set_device first. Every rank then allocates on GPU 0, which OOMs and looks like a memory bug rather than a configuration one.

Logging, checkpointing, or metric-writing from every rank. 32 processes writing the same file corrupts it; 32 copies of every log line makes the log useless. Guard on rank 0 — and remember that all ranks must still participate in collectives, so never put a barrier() inside a rank-0-only branch.

Mismatched world size and data. If the number of batches differs per rank, the job deadlocks at the end of the epoch (Tree 5). drop_last=True plus equal sharding is the defence.

21Scaling: batch size, LR, warmup

The rule that has to be right: when you scale the effective batch, you must scale the learning rate, and you must warm up. Effective batch is per_gpu_batch × world_size × grad_accum. The standard recipe is linear scaling — multiply the base LR by the batch multiplier — with a warmup of a few hundred to a few thousand steps, because a large-batch run at full LR from step zero frequently diverges in the first few hundred steps. (The square-root rule is the alternative, more common with adaptive optimizers; either way, state that the LR must move with the batch.)

Consequence of scalingDetail
Fewer optimizer steps per epochDoubling the batch halves the steps. If you keep step count fixed you've also doubled the data seen — be explicit about whether you're comparing at equal steps or equal samples, or your ablations are meaningless.
BatchNorm gets weirdPer-GPU batch shrinks as you scale out. Below ~8–16 samples per GPU, BN statistics get noisy — use SyncBatchNorm (costs a collective) or switch to GroupNorm/LayerNorm.
Warmup interacts with the scheduleWarmup steps are counted in optimizer steps, so they change when you change accumulation. Verified: a linear-warmup-then-cosine lambda is monotonic in both phases and bounded by [0, base_lr].
Diminishing returnsThere is a batch size past which convergence per sample degrades. Beyond it, more GPUs buys wall-clock but wastes compute. Knowing this limit exists is a strong signal.

22Fault tolerance and elasticity

At 64 GPUs for three days, something will fail. Designing for that is a distributed-systems question, which is exactly what the req's "large-scale distributed systems" line is testing.

ConcernApproach
Node failureCheckpoint on a fixed step cadence; torchrun with --max-restarts relaunches the group and every rank resumes from the last checkpoint. Checkpoint frequency is a trade against re-work: aim to lose no more than ~15–30 minutes.
Preemption (spot instances)Same mechanism, higher frequency. If preemption gives a warning signal, catch it and checkpoint immediately.

23Mixed precision

# bf16 is the default choice on Ampere and later: same exponent range as
# fp32, so it does NOT need loss scaling. fp16 needs a GradScaler.
scaler = torch.amp.GradScaler("cuda", enabled=(dtype == torch.float16))

for batch in loader:
    opt.zero_grad(set_to_none=True)
    with torch.autocast("cuda", dtype=dtype):
        loss = criterion(model(batch), batch["target"])   # autocast region
    # keep the backward OUTSIDE the autocast context
    scaler.scale(loss).backward()
    scaler.unscale_(opt)                       # unscale BEFORE clipping
    torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
    scaler.step(opt)                           # skips the step if inf/nan
    scaler.update()
BehaviourVerified / stated
Autocast casts ops, not parametersConfirmed by execution: inside an autocast region a Linear's output is bf16 while layer.weight.dtype remains fp32. Master weights stay fp32; only the matmul runs low-precision.
Per-op policyMatmuls and convolutions run low-precision; reductions, softmax, and normalizations stay fp32. Confirmed: a sum inside an autocast region came back fp32.
bf16 vs fp16bf16 has fp32's exponent range and fewer mantissa bits, so it rarely underflows and needs no scaler. fp16 has more precision but a narrow range — hence GradScaler.
unscale_ before clippingClipping scaled gradients clips the wrong magnitude. This ordering is a genuine, common bug.
TF32On Ampere+, torch.set_float32_matmul_precision("high") lets fp32 matmuls use tensor cores. Nearly free speedup; a small precision change worth validating once.

28What a checkpoint must contain

Executed and confirmed: saving only model.state_dict() does not resume a run. Training 20 steps straight through, versus training 10, checkpointing, and resuming — with model + optimizer + scheduler state the resumed losses matched the uninterrupted run exactly (max difference 0.0). With model weights only and a freshly constructed optimizer, the runs diverged immediately (difference 1.8e-01). AdamW's moment estimates and the LR schedule position are part of the training state, not incidental.

def save_checkpoint(path, model, opt, sched, scaler, step, epoch, cfg, metrics):
    ckpt = {
        # --- required for an EXACT resume (verified) ---
        "model":  (model.module if hasattr(model, "module") else model).state_dict(),
        "opt":    opt.state_dict(),        # AdamW moments -- omitting this breaks it
        "sched":  sched.state_dict(),      # schedule position
        "scaler": scaler.state_dict(),     # AMP loss scale
        "step":   step,
        "epoch":  epoch,
        # --- required for PROVENANCE (the MLOps half of the req) ---
        "cfg":          cfg,               # full config, not a path to one
        "data_version": cfg.data_version,  # LakeFS commit / dataset digest
        "git_sha":      current_git_sha(),
        "torch":        torch.__version__,
        "world_size":   dist.get_world_size() if dist.is_initialized() else 1,
        "rng": {                           # for bitwise-reproducible resume
            "torch":  torch.get_rng_state(),
            "cuda":   torch.cuda.get_rng_state_all(),
            "numpy":  np.random.get_state(),
            "python": random.getstate(),
        },
        "metrics": metrics,
    }
    # write to a temp path then rename: an atomic swap, so a crash mid-write
    # never leaves a truncated checkpoint that fails to load at 3am
    tmp = f"{path}.tmp"
    torch.save(ckpt, tmp)
    os.replace(tmp, path)

Strip the DDP module. prefixOtherwise the checkpoint only loads back into a DDP-wrapped model, and single-GPU evaluation breaks.

The verified hazard: num_workers is part of your seed

Executed check — with the same seed and the same worker count, the batch stream reproduced exactly. Changing num_workers from 2 to 4 changed the data stream, because each worker derives its seed from the base seed and its own id. So "I tuned the DataLoader for throughput" silently changes your augmentation stream and your results. Anyone comparing two runs must hold num_workers fixed, or accept that they are not comparable. This is a genuinely non-obvious footgun and a very good thing to raise unprompted.

BitwiseDeterministic algorithms, fixed cuDNN, fixed world size, fixed worker count, fixed library versions. Real performance cost. Needed for debugging and for audit.

ProvenanceYou may not reproduce the bits, but you can say exactly what code, config, data version, and environment produced an artifact. For a safety-relevant system this is the non-negotiable one — and it's an MLOps property (Part VIII), not a numerics property.

            loss = criterion(model(batch), batch["target"])
            loss.backward()
            opt.step()
        # report drives checkpointing, Tune scheduling, and fault recovery
        train.report({"loss": loss.item()},
                     checkpoint=train.Checkpoint.from_dict({"model": model.state_dict()}))

    train_loop_per_worker,
    train_loop_config={"lr": 1e-3, "epochs": 10},
    scaling_config=ScalingConfig(num_workers=8, use_gpu=True),
    datasets={"train": ds},
)
result = trainer.fit()

The argument to make for Ray in this role, which is also your strongest personal story: the fundamental problem in §16 is that CPU-side data preparation and GPU-side training need different amounts of hardware, and a plain DataLoader couples them — your decode parallelism is limited to the cores on the training node. Ray Data breaks that coupling: the transformation stage scales to as many CPU workers as it needs, streaming into GPU trainers that scale separately. When you describe your 400× distributed PyTorch/Ray result, frame it in exactly those terms — what was the bottleneck, why did decoupling fix it, and how did you measure it — because that is this team's core problem stated in their own vocabulary.

The contrast worth being able to draw

Airflow is task-centric: it orchestrates operations, and the data artifacts are implicit. Dagster is asset-centric: it orchestrates artifacts, and the operations are implied. For an ML data platform the asset framing usually wins, because the questions you actually ask are about data — "is this dataset stale?", "what fed this model?", "rebuild everything downstream of the new labeller." Being able to state that trade-off is enough to hold the conversation without having shipped Dagster.

33LakeFS and data versioning

LakeFS puts git-like semantics — branches, commits, merges, atomic operations — over object storage. It exists to solve a problem that is acute in AV: datasets change, and you need to know which version trained which model.

CapabilityWhy it matters here
Commits over object storageAn immutable identifier for "the dataset as it was." Put that commit ID in the checkpoint (§28) and the provenance question is answered forever.
BranchesBuild a new dataset version on a branch, validate it, then merge atomically. Nobody trains on a half-written dataset — which is otherwise a real and maddening failure mode.
Atomic mergesObject storage has no transactions. Without a layer like this, "update the dataset" is a non-atomic pile of object writes, and any reader mid-update sees an inconsistent mixture.
RollbackA bad label batch shipped? Revert the commit rather than reconstructing the previous state by hand.
Zero-copy branchingBranches are metadata, not data copies — so you can fork a petabyte dataset cheaply.
The question they might actually ask

"How would you make an experiment reproducible six months later?" The complete answer has four parts, and only one of them is about seeds: code version (git SHA), data version (LakeFS commit or dataset digest), environment (container image digest, library versions), and config (the resolved config, embedded — not a path to a file that has since changed). Seeds and deterministic algorithms get you bitwise reproducibility on top of that, but without those four you cannot even attempt it. §30, §36.

35Cloud and object storage

PropertyConsequence for an ML data platform
High latency, high bandwidthTens of milliseconds per request, but enormous aggregate throughput if you parallelize. Sequential large reads win; millions of small reads lose. This is the single most important fact and it drives the whole sharding design (§15).
Request rate limitsObject stores throttle per prefix. Millions of small objects under one prefix will rate-limit before bandwidth is the issue — and the symptom looks like random slowness, not an error.
No atomic multi-object operationsHence LakeFS (§33), or a manifest-based commit protocol you build yourself.
Eventual consistency (historically)Modern S3 is strongly read-after-write consistent, but listing behaviour and cross-region replication still surprise people. Don't build correctness on listing.
Egress and cross-AZ costMoving data between zones costs money per byte, every epoch. Data locality is a budget line, not just a performance concern.
Caching tiersLocal NVMe on the training node as a read-through cache is often the highest-leverage infrastructure change available: the first epoch pays object-storage cost, every subsequent epoch reads locally.
Where your HPC background maps cleanly

Lustre and GPFS taught the same lessons in a different vocabulary: small-file access patterns destroy parallel filesystems too, striping is the on-prem analogue of sharding, and metadata servers throttle much like prefix rate limits. When they ask about cloud storage, saying "I've solved this shape of problem on Lustre and GPFS — the constraint names differ but the design rules are the same: large sequential reads, avoid metadata storms, and cache locally" is both true and more credible than reciting S3 features.

37Deployment and model testing

"Automated model deployment and testing strategies" in a safety-relevant domain means gates, not vibes.

The export hop is where silent breakage lives

Tracing-based export bakes in whatever branch the example input took and freezes Python-side scalars as constants. The model still runs; it just quietly computes something slightly different. It will pass a smoke test and fail on a rare frame. A numerical comparison at every hop — eager vs exported vs optimized, on real data, with a stated tolerance — is the only thing that catches it, and building that comparison is squarely "automated testing strategies."

38CI for machine learning

TierWhat it checks, and how fast
Unit (seconds)Dataset returns the expected keys and shapes; collate pads and masks correctly; the sampler shards without overlap; a loss function's gradient is finite. Cheap, and it catches the majority of pipeline bugs.
Smoke (minutes)Train two steps on a tiny fixture, on CPU. Catches shape errors, config breakage, and import problems on every PR.
Convergence (tens of minutes, nightly)Train a small model on a small dataset and assert the loss falls below a threshold. This is what catches "the pipeline runs but learns nothing" — the failure that unit tests cannot see.
Throughput regression (nightly)Samples/sec on fixed hardware, with an alert on distribution shift. A 20% pipeline slowdown is invisible to correctness tests and expensive at fleet scale.
Data validation (per dataset build)Schema checks, null rates, label-distribution drift, and per-slice counts. A silently broken upstream labeller should fail the dataset build, not the model evaluation three days later.

The test that would have caught it is the deliverable — not the fix.

Part IX — Decision trees

Six scripts.

Each tree is a diagnostic you can run out loud. They share one grammar — classify before you dig — because the expensive failure in every one is treating the wrong layer. Learn the first branch of each; the rest follows.

   │       inputs finite?  → a corrupt sample, a bad decode, a

  • Three findings worth restating. (1) DistributedSampler pads by wrapping the first indices onto trailing ranks, so with 100 samples across 8 ranks, indices 0–3 are seen twice per epoch — which silently double-counts in distributed evaluation; drop_last=True is what makes ranks truly disjoint. (2) A model-only checkpoint does not resume a run: with optimizer and scheduler state the resumed losses matched exactly, without them they diverged immediately. (3) Changing num_workers changes the RNG stream, so two runs with different worker counts are not comparable.

One habit runs through every part of this manual: find out what's actually limiting you before you fix anything. Run the job with synthetic data before optimizing the model.
