Coalescing & Bank Conflicts

Two memory-throughput problems that live in two different memory spaces. They sound like opposites because the spaces are laid out as opposites — and a shared-memory transpose has to satisfy both at once.

warp = 32 threads global memory → coalescing shared memory → banks 32 banks × 4 B
00 — orientation

Two spaces, two rules

Memory coalescing is about global memory (off-chip DRAM): how a warp's 32 simultaneous accesses get bundled into hardware transactions. Bank conflicts are about shared memory (on-chip SRAM): how 32 threads collide over a fixed set of banks.

Optimizing one does nothing for the other. The mental models even point in opposite directions: coalescing wants threads packed onto consecutive addresses; avoiding bank conflicts wants threads spread across banks. The reason they don't fight is that consecutive addresses already spread across banks — so the same access pattern can be good for both.

01 — global memory

Memory coalescing

space: global / DRAM

When a warp issues a load, the memory controller inspects the 32 requested addresses and groups them into the fewest possible transactions (32-byte sectors / 128-byte cache lines). If consecutive threads touch consecutive words, the whole warp folds into one transaction. If they stride, the same data spreads across more transactions and you pay for bytes you never use.

Coalesced vs strided access

warp thread (8 shown, 32 real) word actually used fetched but wasted

The fix is almost always to make threadIdx.x drive the contiguous dimension. In a row-major array A[row*N + col], you want the fastest-varying thread index on col:

coalescing.cu
int col = blockIdx.x * blockDim.x + threadIdx.x;   // varies fastest within warp
int row = blockIdx.y * blockDim.y + threadIdx.y;

float v = A[row * N + col];   // coalesced: T0..T31 -> A[..0], A[..1], ...
// float v = A[col * N + row]; // strided by N: each thread N words apart
02 — shared memory

Bank conflicts

space: shared / SRAM

Shared memory is physically split into 32 banks, each 4 bytes wide. Successive 32-bit words land in successive banks, so word w lives in bank w % 32. A warp resolves its access in a single cycle only if the 32 threads hit 32 distinct banks (or all read the identical word, which broadcasts for free). If k threads hit different words in the same bank, the hardware serializes them into k cycles — a k-way conflict.

Conflict-free vs bank conflict

bank active this cycle piled into one bank (serialized) idle bank

The bank a thread touches is pure address arithmetic. For a 2D shared tile tile[R][C] of floats, element tile[r][c] sits at word index r*C + c, so its bank is (r*C + c) % 32:

banks.cu
// Row access -- vary c across the warp, r fixed:
//   bank = (r*C + c) % 32  ->  c, c+1, c+2, ...   all distinct   [ok]
//
// Column access -- vary r across the warp, c fixed:
//   bank = (r*C + c) % 32
//   if C is a multiple of 32 (e.g. C = 32): r*32 % 32 = 0 always
//   -> every thread hits the SAME bank -> 32-way conflict        [bad]
03 — the payoff

The transpose trap & padding fix

both rules at once

A shared-memory transpose loads a tile coalesced, then reads it back transposed — which is a column access. With a square tile[32][32], every thread in the warp lands on bank 0. The classic fix costs one wasted column: pad the inner dimension to 33. The row stride becomes 33, and since 33 % 32 == 1, each successive row shifts the bank by one.

Why +1 padding works

Here is the canonical transpose with both optimizations in place — coalesced global reads and writes, plus the padded tile that removes the shared-memory conflict:

transpose.cu
#define TILE 32

__global__ void transpose(float* out, const float* in, int N) {
    __shared__ float tile[TILE][TILE + 1];   // +1 padding -> conflict-free columns

    int x = blockIdx.x * TILE + threadIdx.x;
    int y = blockIdx.y * TILE + threadIdx.y;

    // Coalesced global READ: consecutive threadIdx.x -> consecutive in[] addrs.
    // Write into shared by ROW (no bank conflict on the write).
    tile[threadIdx.y][threadIdx.x] = in[y * N + x];

    __syncthreads();

    // Remap to the transposed block so the global WRITE is also coalesced.
    x = blockIdx.y * TILE + threadIdx.x;
    y = blockIdx.x * TILE + threadIdx.y;

    // Read shared by COLUMN (tile[tx][ty]); padding makes this conflict-free.
    out[y * N + x] = tile[threadIdx.x][threadIdx.y];
}
The pitfall. Drop the + 1 and that final tile[threadIdx.x][threadIdx.y] read becomes a column access where all 32 threads collide on one bank — measurably the difference between a transpose that runs near memory bandwidth and one that crawls.
04 — at a glance

Opposite-sounding, same access pattern

Memory coalescing

global memory

Diagnosed from the global index expression: is threadIdx.x driving the contiguous dimension?

Hurt by strided or column-major indexing that scatters a warp across cache lines.

want: warp threads packed onto consecutive addresses

Bank conflicts

shared memory

Diagnosed by computing (linear_index) % 32 across the warp — do any two threads share a bank?

Hurt by power-of-two strides; the escape hatch is padding an array by one element.

want: warp threads spread across distinct banks
