1 Hardware foundations

Every optimization decision flows from the hardware. You must know the compute, memory, and interconnect ceilings to judge whether your training run is efficient.

H100 SXM specifications and what they meanHardware
SpecValuePerformance implication
BF16 Tensor Core TFLOP/s990Peak throughput for matrix multiply. All linear layers should target this. If your GEMMs hit 80%+ (790+ TFLOP/s), you're doing well.
FP8 (E4M3) Tensor Core1,979 TFLOP/s2× BF16. Requires Transformer Engine and careful loss scaling. The next frontier beyond BF16 training.
FP32 CUDA Core67 TFLOP/s~15× slower than BF16 Tensor Core. Any GEMM running in FP32 is leaving 93% of peak performance on the table.
HBM3 capacity80 GBSets the upper bound on model + optimizer + activation memory per GPU. A 7B parameter model in BF16 is ~14 GB of weights alone.
HBM3 bandwidth3.35 TB/sDetermines throughput for memory-bound kernels (element-wise, reductions, norm layers). 42 bytes/FLOP at BF16 peak — below this arithmetic intensity, you're memory-bound.
L2 cache50 MBFlashAttention and tiled algorithms exploit this heavily. Working sets that fit in L2 avoid HBM round-trips.

NVLink 4.0900 GB/s bidirectional18 links. Critical for multi-GPU collective communication (AllReduce in DDP, AllGather/ReduceScatter in FSDP). ~7× faster than PCIe Gen5.
PCIe Gen5128 GB/s bidirectionalHost ↔ device data transfer. Relevant for data loading and CPU offloading. Not for GPU-to-GPU communication on NVLink nodes.
TDP700 WMaximum power draw. Power is the best proxy for real GPU utilization — more informative than nvidia-smi's GPU-Util %.

The key ratios to internalize

Arithmetic intensity threshold (BF16)
990 TFLOP/s ÷ 3.35 TB/s = 295 FLOP/byte

This means:
If a kernel does fewer than 295 FLOPs per byte loaded from HBM, it's memory-bound.
If it does more, it's compute-bound.

Typical arithmetic intensities:
Large GEMM (M=4096, N=4096, K=4096, BF16): ~2048 FLOP/byte → compute-bound ✓
Small GEMM (M=32, N=4096, K=4096, BF16): ~64 FLOP/byte → memory-bound ✗
Element-wise ReLU: ~0.5 FLOP/byte → heavily memory-bound
LayerNorm: ~5 FLOP/byte → memory-bound
Softmax: ~4 FLOP/byte → memory-bound
The fundamental insight: In LLM training, GEMMs (from linear layers) dominate total FLOP count and should be compute-bound. Everything else (norm layers, activations, dropout, softmax) is memory-bound. The performance engineer's job is to: (1) ensure GEMMs actually hit the Tensor Core ceiling, and (2) fuse everything else to minimize HBM traffic.
Multi-node interconnect: NVLink, NVSwitch, InfiniBandHardware
InterconnectScopeBandwidthLatencyUsed for
NVLink 4.0GPU ↔ GPU (same node)900 GB/s bidirectional~1 μsTensor parallelism AllReduce, FSDP AllGather within a node
NVSwitchAll-to-all within 8 GPUsFull bisection bandwidth~1 μsEnables any GPU to communicate with any other at full NVLink speed. Non-blocking crossbar.
InfiniBand NDRNode ↔ node (across network)400 Gb/s (50 GB/s) per port, typically 8 ports/node~1–5 μsData parallelism gradient sync, pipeline parallelism micro-batch transfer across nodes

Parallelism placement rule: Place communication-heavy parallelism strategies on the fastest interconnect. Tensor parallelism (AllReduce every layer) goes within a node over NVLink. Data parallelism (AllReduce once per step) goes across nodes over InfiniBand. Pipeline parallelism (point-to-point per micro-batch) goes across nodes over IB. Getting this mapping wrong — e.g., tensor parallel across nodes — is the #1 multi-node performance mistake.

Performance implication: ~85% of FLOPs are in large GEMMs (QKV, output proj, MLP). These must use Tensor Cores in BF16/FP8 — any FP32 fallback decimates throughput. The remaining ~15% (softmax, norms, activations, residuals) are memory-bound and should be fused together wherever possible to minimize HBM round-trips.
Scaling laws: how FLOPs scale with model and sequenceArchitecture
Total FLOPs per training step (forward + backward ≈ 3× forward)
FLOPs ≈ 6 × N × B × S
where N = total parameters, B = batch size (sequences), S = sequence length

For a 70B model, batch=1M tokens, sequence=4096:
6 × 70×10⁹ × 256 × 4096 ≈ 4.4 × 10¹⁷ FLOPs per step

On 256 H100s at 50% MFU (Model FLOP Utilization):
256 × 990 TFLOP/s × 0.5 = 126,720 TFLOP/s
Time per step ≈ 4.4×10¹⁷ / 126,720×10¹² ≈ 3.47 seconds

The two scaling axes

The attention wall: At sequence length 32K+, the O(s²) attention compute and memory become dominant. Without FlashAttention, a 32K sequence produces a 32K×32K attention matrix per head — ~4 GB per layer in FP32 for a single sample. FlashAttention eliminates the materialized attention matrix entirely, making long-context training viable.

3 Memory math

Memory is typically the binding constraint for LLM training — not compute. Understanding exactly where every byte goes determines your parallelism strategy, batch size, and whether you can train at all.

The four memory consumersMemory

1. Model parameters

Parameter memory = N × bytes_per_param
BF16: 7B model → 7×10⁹ × 2 bytes = 14 GB
FP32: 7B model → 7×10⁹ × 4 bytes = 28 GB

2. Gradients

Gradient memory = N × bytes_per_grad
Same shape as parameters. BF16 training with BF16 grads:
7B model → 14 GB

3. Optimizer states

AdamW stores 2 additional states per parameter:
- First moment (m): FP32, N × 4 bytes
- Second moment (v): FP32, N × 4 bytes
- FP32 master copy of weights: N × 4 bytes (for mixed precision)

7B model, AdamW, mixed precision:
(4 + 4 + 4) × 7×10⁹ = 84 GB of optimizer state

Total weight-related memory (params + grads + optimizer):
14 (params) + 14 (grads) + 84 (optimizer) = 112 GB for 7B
This already exceeds a single H100's 80 GB → need sharding.

4. Activations (the variable cost)

Activation memory per Transformer layer (approximate, without recomputation):
≈ b × s × d × (34 + 5·s·h/d) bytes (BF16, no Flash)

For LLaMA-7B (d=4096, h=32), b=4, s=4096, without FlashAttention:
Per layer: ≈ 2.1 GB
32 layers: ≈ 67 GB of activations

With FlashAttention (no materialized attention matrix):
Per layer: ≈ 0.5 GB
32 layers: ≈ 16 GB of activations
FlashAttention is not optional for LLMs. It doesn't just save memory (4× reduction in activation memory from attention) — it also eliminates O(s²) HBM reads by keeping the attention computation tiled in SRAM/L2. Without it, attention is both memory-capacity and memory-bandwidth bound.

The mixed precision training recipe

# Standard BF16 mixed precision for LLMs
model = model.to(torch.bfloat16)  # weights in BF16
optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)
# AdamW internally keeps FP32 master weights + FP32 m,v states

for batch in loader:
    with torch.autocast('cuda', dtype=torch.bfloat16):
        output = model(batch)      # forward in BF16
        loss = criterion(output)   # loss in FP32 (autocast promotes)

    loss.backward()                # backward in BF16
    optimizer.step()               # update FP32 master weights
    optimizer.zero_grad()

# Note: No GradScaler needed for BF16 (same exponent range as FP32)
# GradScaler IS needed for FP16 to prevent underflow in gradients
BF16 is the default for modern LLM training. Use BF16 for all compute (forward + backward). Keep optimizer states and master weights in FP32. No grad scaler needed. Autocast handles the casting automatically — it promotes operations that need higher precision (loss, softmax, norm) to FP32 and keeps GEMMs in BF16.
Tensor Core alignment requirementsPrecision

Tensor Cores process matrix tiles of fixed sizes. If your matrix dimensions don't align, the kernel falls back to CUDA cores or pads internally (wasting compute).

BF16 / FP16Multiples of 816×8×16 (M×N×K MMA instruction)
TF32Multiples of 416×8×8
FP8Multiples of 1616×16×32

Dimensions to check in your model config

FFN intermediate (d_ff)11008✓ (÷8 = 1376)Common oversight with SwiGLU: (4d × 2/3) may not be aligned
Attention heads × head_dim32 × 128✓head_dim should be 64 or 128 (both ÷8)

Vocabulary size is the sneakiest misalignment. Many tokenizers produce vocab sizes like 32,001 or 50,257 (GPT-2). The output projection GEMM does [b×s, d] × [d, vocab]. If vocab isn't a multiple of 8, the N dimension is misaligned and every output projection falls back to CUDA cores. Pad the embedding and output projection to the next multiple of 64 (for best FP8 compatibility too).

5 Data pipeline

Building a data pipeline that never starves the GPUData

For LLM training, the data pipeline is simpler than vision (no augmentation) but can still be a bottleneck if done wrong, especially at scale.

LLM data pipeline components

StageWhat happensBottleneck riskSolution
Storage readRead tokenized data from disk/networkNFS or cloud storage I/O limitsLocal NVMe SSD. Pre-tokenize and store in memory-mapped binary format (numpy mmap, safetensors, or Arrow/Parquet).
TokenizationConvert text → token IDsCPU-bound for on-the-fly tokenizationPre-tokenize offline and save as binary. For 1T tokens at 4 bytes/token = 4 TB on disk — feasible on NVMe.

# Optimal LLM DataLoader configuration
loader = DataLoader(
    dataset,
    batch_size=micro_batch_size,
    num_workers=4,            # 4–8 usually sufficient for pre-tokenized data
    pin_memory=True,            # enables async H2D via pinned host memory
    persistent_workers=True,    # avoids worker respawn cost every epoch
    prefetch_factor=2,          # pipeline next batch while GPU trains
    shuffle=False,              # use pre-shuffled indices for reproducibility
)

# For memory-mapped datasets (ideal):
class MMapTokenDataset(torch.utils.data.Dataset):
    def __init__(self, path, seq_len):
        self.data = np.memmap(path, dtype=np.uint16, mode='r')
        self.seq_len = seq_len
    def __len__(self):
        return len(self.data) // self.seq_len
    def __getitem__(self, idx):
        start = idx * self.seq_len
        chunk = self.data[start:start + self.seq_len + 1]
        x = torch.from_numpy(chunk[:-1].astype(np.int64))
        y = torch.from_numpy(chunk[1:].astype(np.int64))
        return x, y
For LLM pretraining, the data pipeline is usually NOT the bottleneck (unlike vision training) because the data is pre-tokenized binary that loads in microseconds. If you see data loading gaps in Nsight Systems, it's almost certainly a misconfiguration — fix num_workers and persistent_workers first.

6 Memory optimization techniques

Activation checkpointing (gradient checkpointing)Memory

The biggest memory savings technique. Instead of storing all intermediate activations for backward, throw them away and recompute them during backward.

StrategyMemory savedCompute costWhen to use
No checkpointing0%1× forward + 1× backward = baselineSmall models that fit in memory
Per-layer checkpointing~60–70% of activation memory~33% more compute (re-forward each layer)Standard for large LLMs. Best tradeoff.
Selective checkpointing~40–50%~15–20% more computeKeep memory-cheap activations (after GEMM), recompute memory-expensive ones (attention). Tunable sweet spot.
Full recomputation~90%+~100% more computeExtreme memory pressure. Rarely used for LLMs.
# Per-layer checkpointing in PyTorch
from torch.utils.checkpoint import checkpoint

class TransformerBlock(nn.Module):
    def forward(self, x):
        # Without checkpointing: all intermediates stored
        # With checkpointing: only input x is stored, rest recomputed
        return checkpoint(self._forward, x, use_reentrant=False)

    def _forward(self, x):
        x = x + self.attn(self.norm1(x))
        x = x + self.mlp(self.norm2(x))
        return x

# Or apply to all layers at once:
from torch.distributed.algorithms._checkpoint.checkpoint_wrapper import (
    checkpoint_wrapper, CheckpointImpl
)
for layer in model.layers:
    checkpoint_wrapper(layer, checkpoint_impl=CheckpointImpl.NO_REENTRANT)
Always use use_reentrant=False (or CheckpointImpl.NO_REENTRANT). The legacy reentrant implementation has subtle bugs with torch.compile, FSDP, and non-deterministic ops. Non-reentrant is the default in newer PyTorch versions and handles all edge cases correctly.

7 Distributed training

No LLM fits on a single GPU for training. Distributed training combines multiple parallelism strategies, each with different trade-offs for compute efficiency, memory, and communication.

The four parallelism dimensionsDistributed

Data parallelism (DP / DDP / FSDP / ZeRO)

Each GPU holds a copy of the model and processes different data. Gradients are synchronized (AllReduced) across GPUs after each backward pass.

VariantWhat's replicatedWhat's shardedCommunicationMemory per GPU
DDPParams, optimizer, gradsNothingAllReduce gradientsFull model × 18 bytes/param
ZeRO Stage 1Params, gradsOptimizer statesAllReduce + ReduceScatter optimizer~6 bytes/param saved
ZeRO Stage 2ParamsOptimizer + gradsReduceScatter grads + AllGather params~8 bytes/param saved
ZeRO Stage 3 / FSDPNothing (fully sharded)EverythingAllGather before forward, ReduceScatter after backward~(18/N) bytes/param
FSDP is the standard for LLM training in PyTorch. It shards parameters, gradients, and optimizer states across GPUs. Each GPU holds only 1/N of the model. During forward, it AllGathers the full parameters for each layer just-in-time, computes, then frees the gathered parameters. During backward, it does the same plus ReduceScatter for gradients.
# FSDP2 (PyTorch 2.x preferred API)
from torch.distributed._composable.fsdp import fully_shard

# Shard each Transformer layer independently
for layer in model.layers:
    fully_shard(layer)
# Shard the entire model (top-level)
fully_shard(model)

# Key configuration for performance:
# - Wrap at the Transformer block level (not too fine, not too coarse)
# - Use BF16 for communication: reshard_after_forward=True

Tensor parallelism (TP)

Splits individual layer operations (GEMMs) across GPUs. Each GPU computes a slice of the weight matrix and they AllReduce to combine results.

LayerHow it's splitCommunication
Linear (column split)Weight [d, d] → each GPU gets [d, d/tp]. Output is partial sum.AllReduce after the layer to combine partial outputs.
Linear (row split)Weight [d, d] → each GPU gets [d/tp, d]. Input is split across GPUs.AllReduce after the layer.
Attention headsHeads distributed across GPUs (h/tp heads per GPU)AllReduce after output projection.
MLPUp-proj column-parallel, down-proj row-parallelOne AllReduce per MLP (not two).
TP requires AllReduce at every layer — twice per Transformer block (once after attention, once after MLP). This must go over NVLink (900 GB/s) — it's far too frequent for inter-node InfiniBand. Never use TP across nodes. Typical TP degree: 2, 4, or 8 (within one node's NVLink domain).

Pipeline parallelism (PP)

Splits the model by layers across GPUs. GPU 0 runs layers 0–7, GPU 1 runs layers 8–15, etc. Micro-batches flow through the pipeline.

AspectDetail
Communication patternPoint-to-point: each GPU sends activations to the next GPU in the pipeline. Much less bandwidth than AllReduce.
Pipeline bubbleGPUs at the start/end of the pipeline idle while waiting for micro-batches to flow through. Bubble fraction ≈ (pp_degree - 1) / num_microbatches.
MemoryEach GPU holds only L/pp_degree layers. But must buffer activations for all in-flight micro-batches.
Scheduling1F1B (one forward, one backward) minimizes bubble. Interleaved schedules (virtual stages) reduce bubble further at the cost of more communication.
Pipeline bubble overhead:
Bubble = (pp_stages - 1) / total_microbatches

Example: PP=4, 32 micro-batches → bubble = 3/32 = 9.4% overhead
Example: PP=4, 8 micro-batches → bubble = 3/8 = 37.5% overhead ← too high
Pipeline parallelism is used for models too large for TP alone (which is limited to 8 GPUs within a node). For 70B+ models on 64+ GPUs: use TP=8 within each node, PP=4–8 across nodes, and FSDP for data parallelism within each pipeline stage.

Sequence parallelism (SP) and context parallelism (CP)

These split along the sequence dimension rather than the model dimension.

TypeWhat it splitsWhen to use
Sequence parallelismThe non-tensor-parallel regions (LayerNorm, dropout, residual add) are split across the TP group. During TP, these ops are redundantly computed on all GPUs — SP eliminates this redundancy.Always use with TP. It's essentially free — reduces memory without adding communication (converts AllReduce to AllGather + ReduceScatter which has the same total volume).
Context parallelismSplits the sequence dimension of attention across GPUs. Each GPU handles a chunk of the sequence and uses ring attention to exchange KV chunks.Very long sequences (32K+) where even FlashAttention's memory is prohibitive for a single GPU. Adds AllToAll communication per attention layer.

Communication volume per training step

DDP (AllReduce): 2 × model_size per step
FSDP (AllGather + ReduceScatter): 3 × model_size per step (2× AllGather fwd+bwd, 1× ReduceScatter)
TP AllReduce per layer: 2 × b × s × d × 2 bytes × 2 (twice per block: after attn + after MLP)

For a 70B model, FSDP across 8 GPUs:
3 × 70B × 2 bytes = 420 GB of communication per step
Over NVLink (900 GB/s): ~0.47 seconds if not overlapped
With overlap (AllGather during compute): effectively free if compute > comm per layer
The overlap principle: Communication should be fully hidden behind compute. FSDP achieves this by AllGathering the next layer's parameters while the current layer is computing, and ReduceScattering the previous layer's gradients while the current layer's backward is running. When communication time per layer < compute time per layer, communication is "free." When it's not, you see it as a gap in the Nsight Systems timeline.

9 Kernel optimization

torch.compile: the primary optimization toolKernels

torch.compile is a JIT compiler that fuses operations, eliminates Python overhead, and generates optimized CUDA kernels via Triton or inductor.

What torch.compile does

OptimizationWhat happensTypical speedup
Operator fusionCombines element-wise ops (add, mul, ReLU, dropout) into single kernels, eliminating HBM round-trips for intermediates1.5–3× for memory-bound regions
Python overhead eliminationCaptures the computation graph once, replays it without Python dispatch overhead10–30% for overhead-dominated workloads
Layout optimizationChooses optimal memory layouts (channels_last, contiguous) for each operation5–15% for convolutions
AutotuningTries multiple Triton kernel configurations and selects the fastest5–20% for individual kernels
# Basic usage
model = torch.compile(model)  # default mode="default"

# Options:
model = torch.compile(model, mode="reduce-overhead")   # uses CUDA Graphs
model = torch.compile(model, mode="max-autotune")      # more tuning time, faster kernels
model = torch.compile(model, fullgraph=True)            # error if graph breaks

11 Throughput engineering

MFU: the metric that mattersMetric

Model FLOP Utilization (MFU) is the gold-standard efficiency metric for LLM training. It measures what fraction of the GPU's theoretical peak FLOP/s is actually used for model computation.

MFU = Model FLOPs per step / (GPU peak FLOP/s × time per step × num_GPUs)

Where Model FLOPs ≈ 6 × N × B × S (forward + backward)

Example: 7B model, 1M tokens/step, 8 H100s, step time = 2.5s
Model FLOPs = 6 × 7×10⁹ × 1×10⁶ = 4.2 × 10¹⁶
Peak = 8 × 990 × 10¹² = 7.92 × 10¹⁵ FLOP/s
MFU = 4.2×10¹⁶ / (7.92×10¹⁵ × 2.5) = ~21.2% ← poor

If step time improved to 0.6s:
MFU = 4.2×10¹⁶ / (7.92×10¹⁵ × 0.6) = ~88.4% ← excellent

MFU benchmarks (what to target)

MFU rangeAssessmentTypical cause if low
> 55%Excellent — near state-of-the-art—
40–55%Good — room for optimizationCommunication not fully overlapped, suboptimal batch size, missing fused kernels
25–40%Mediocre — significant wasteNo torch.compile, FP32 GEMMs, poor TP/PP configuration, data loading gaps
< 25%Poor — debugging requiredMajor bottleneck: data starvation, sync stalls, wrong parallelism config, single-GPU bottleneck
MFU vs HFU: MFU counts only model FLOPs (the theoretical minimum compute). Hardware FLOP Utilization (HFU) also counts recomputation from activation checkpointing. HFU is always higher than MFU (since recomputation adds ~33% FLOPs). MFU is the more honest metric — it tells you how much of the GPU's capacity goes to useful work.
Batch size tuning: the throughput curveThroughput

Batch size is the most important throughput lever. Larger batches amortize fixed costs (communication, kernel launches, pipeline bubbles) over more useful compute.

The batch size → throughput relationship

As batch size increases:

PhaseWhat happensThroughput behavior
Too small (bs < 8K tokens)GEMMs don't fill the GPU (M dimension too small). Communication overhead dominates. Pipeline bubbles are large.Low throughput, low MFU. Near-linear scaling with batch size.
Sweet spot (bs = 0.5–2M tokens)GEMMs saturate Tensor Cores. Communication is hidden behind compute. Pipeline is efficiently utilized.Peak throughput. MFU plateaus at maximum.
Too large (bs > 4M tokens)Activation memory grows, forcing more gradient accumulation or checkpointing. Diminishing returns on MFU. Potential convergence issues (learning rate needs adjustment).Throughput flat or declining. Memory pressure increases.
Batch size = micro_batch_size × gradient_accumulation_steps × DP_degree × seq_len

Tune micro_batch_size first (maximize GPU memory usage)
Then tune gradient_accumulation to reach target global batch size
Target: global batch size in tokens = 0.5M–4M for most LLMs
The micro-batch size should be the largest that fits in GPU memory. This maximizes the M dimension of GEMMs (= micro_batch_size × seq_len), which maximizes arithmetic intensity and Tensor Core utilization. Use gradient accumulation to reach the target global batch size without increasing memory.

12 Profiling workflow

The complete profiling loop for LLM trainingProfiling

This section ties together the Nsight Systems and Nsight Compute guides with the LLM-specific context above.

Step 1: Quick sanity check (2 minutes)

# While training is running:
nvidia-smi dmon -s pucvmet -d 1  # streaming GPU metrics
# Check: GPU-Util ~95%, Power ~500–650W, Temp < 80°C
# Check: All 8 GPUs similar (no straggler)

Step 2: Nsight Systems capture (5 minutes)

nsys profile --trace=cuda,nvtx,osrt,cudnn,cublas \
    --cuda-memory-usage=true --gpu-metrics-device=all \
    --capture-range=cudaProfilerApi --output=llm_baseline \
    torchrun --nproc_per_node=8 train.py

Step 3: Timeline analysis (10 minutes)

Open the .nsys-rep and answer these questions in order:

#QuestionWhere to lookIf bad →
1How long is one training iteration?NVTX markers → measure wall timeCompute MFU. If < 40%, continue diagnosis.
2Is the GPU idle between iterations?GPU kernel row → gaps between iterationsData pipeline issue (Chapter 5)
3Are GEMMs using Tensor Cores?Kernel names: h884 vs sgemmPrecision issue (Chapter 4)
4Are there many tiny kernels?Kernel stats → sort by count/durationMissing torch.compile (Chapter 9)
5Is communication overlapped?NCCL kernels during or after backward?FSDP/DDP config issue (Chapter 7)

Step 4: Nsight Compute for hot kernels (10 minutes)

If the timeline shows no idle gaps but MFU is still low, the bottleneck is inside the kernels themselves.

# Profile the top GEMM kernel
ncu --set full --kernel-name "regex:gemm" \
    --launch-skip 500 --launch-count 3 \
    -o gemm_deep_dive python train.py

# Check:
# - SOL: SM% > 80%? Memory% < 50%? (should be compute-bound)
# - Roofline: near the BF16 Tensor Core ceiling?
# - Tensor Core pipe utilization > 0%?

Step 5: Fix → re-profile → iterate

Apply the smallest targeted fix, re-profile, verify the bottleneck moved. Repeat until MFU is satisfactory or the remaining bottleneck is irreducible (near hardware limits).

The profiling priority stack for LLMs: (1) Is it using BF16 Tensor Cores? (2) Is communication overlapped? (3) Is batch size large enough? (4) Are kernels fused? (5) Is the data pipeline keeping up? Address them in this order — each subsequent item matters less if the previous ones aren't fixed.

The performance engineer's mental model for LLM training: Everything is either a GEMM or not. GEMMs (85% of FLOPs) must hit Tensor Cores at BF16/FP8, with aligned dimensions and large batch sizes. Non-GEMMs (15% of FLOPs but 50%+ of memory traffic) must be fused to minimize HBM round-trips. Communication must overlap with compute. Memory must be sharded (FSDP) and checkpointed. Then measure MFU — the single number that tells you if you've succeeded.
