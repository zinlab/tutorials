1 Architecture decisions

The modern LLM architecture and every design choiceArchitecture

Modern LLMs have converged on a near-standard recipe. Here's every choice, its options, and the current consensus.

ComponentOptionsCurrent consensusWhy
Overall architectureEncoder-only, decoder-only, encoder-decoderDecoder-only (autoregressive)Simplest to scale, best for generation, competitive on understanding tasks too. GPT, LLaMA, Mistral, Qwen all use this.
NormalizationLayerNorm, RMSNorm, pre-norm, post-normPre-RMSNormRMSNorm is simpler (no mean subtraction), ~5% faster. Pre-norm (normalize before attention/MLP, not after) stabilizes training at scale.
Positional encodingSinusoidal, learned, ALiBi, RoPE, NoPERoPE (Rotary Position Embedding)Relative position encoding applied directly to Q/K. Extrapolates to longer sequences better than learned absolute. Used by LLaMA, Mistral, Qwen.
AttentionMHA, MQA, GQAGQA (Grouped Query Attention)Shares KV heads across groups of query heads. Saves KV cache memory at inference (critical for long-context serving) with minimal quality loss vs MHA.
Activation functionReLU, GELU, SiLU, SwiGLU, GeGLUSwiGLUGated linear unit with SiLU activation. ~1% better than GELU on downstream tasks. Used by LLaMA, Mistral, PaLM. Note: adds an extra weight matrix (gate projection), so MLP has 3 matrices instead of 2.
MLP hidden dimension4×d, 8/3×d (SwiGLU adjusted)~8/3 × d (rounded to multiple of 256)With SwiGLU, the intermediate dimension is 2/3 of what it would be with a standard MLP, keeping FLOPs roughly equal.
Bias termsWith bias, without biasNo bias in linear layersRemoving biases simplifies sharding (no bias reduction needed in TP) and empirically doesn't hurt quality. LLaMA, Mistral, Qwen omit biases.
Embedding tyingTied (share input/output embeddings), untiedUntied for large modelsTying saves parameters but limits the output projection. Untied is standard for 7B+ models.

2 Tokenization

Tokenizer design and its training impactTokenization
AlgorithmExamplesHow it worksPros / cons
BPEGPT-2/3/4, LLaMAIteratively merges the most frequent byte pair into a single token. Bottom-up.Pro: handles any text (byte-level fallback). Con: tokenization is deterministic but can be sensitive to whitespace/casing.
UnigramT5, mBARTStarts with a large vocabulary, iteratively prunes tokens that minimally impact likelihood. Top-down.Pro: probabilistic (can produce multiple valid tokenizations). Con: slightly slower training of the tokenizer itself.
SentencePiece BPELLaMA, MistralBPE on raw byte sequences (language-agnostic). Treats text as raw bytes, then applies BPE.Pro: handles any language/script, no unknown tokens. Standard for multilingual models.
tiktokenGPT-4, ClaudeByte-level BPE with optimized Rust implementation. Fast encoding/decoding.Pro: 3–10× faster than SentencePiece. Con: OpenAI-specific tooling.

Tokenizer design decisions

DecisionGuidance
Vocabulary size32K is the minimum viable for English. 64K–128K for multilingual or code-heavy models. Larger vocab → fewer tokens per document → faster training (fewer steps for same data), but larger embedding table (~1 GB per 32K tokens at d=4096 in BF16).
Special tokensMinimum: <bos>, <eos>, <pad>. For instruction-tuned: add role tokens (<|user|>, <|assistant|>), tool call tokens. Plan these before training — adding special tokens later requires retraining the embedding layer.
Whitespace handlingSentencePiece prepends a special whitespace marker (▁). This means "hello" at the start of a sentence tokenizes differently from " hello" mid-sentence. Be consistent — many training bugs come from inconsistent whitespace treatment between tokenization and data processing.
Number tokenizationStandard BPE splits numbers inconsistently: "2024" might become ["20", "24"] or ["202", "4"]. This hurts arithmetic ability. Some approaches: digit-level tokenization (each digit is one token), or special number tokens. LLaMA-3 improved this with better BPE training on numeric data.
Padding for efficiencyPad vocab_size to a multiple of 64 for Tensor Core alignment on the output projection GEMM. E.g., 32001 → 32064.

Sequence packing and document boundariesData

Documents have variable lengths but training requires fixed-length sequences. Sequence packing combines multiple documents into a single training sequence to avoid padding waste.

# Naive approach (wasteful): pad each document to seq_len
# [doc1_tokens + PAD + PAD + PAD ...]  ← 40% of compute wasted on padding

# Packing approach: concatenate documents with separator
# [doc1_tokens <eos> doc2_tokens <eos> doc3_tokens <eos> ...]
# Exactly seq_len tokens, no padding, no waste.

# CRITICAL: prevent cross-document attention with a document mask
# Tokens from doc1 should NOT attend to tokens from doc2

# Two implementations:
# 1. Document attention mask (supported by FlashAttention via cu_seqlens)
from flash_attn import flash_attn_varlen_func
attn_output = flash_attn_varlen_func(
    q, k, v,
    cu_seqlens_q=cu_seqlens,   # cumulative sequence lengths within the pack
    cu_seqlens_k=cu_seqlens,
    max_seqlen_q=max_doc_len,
    max_seqlen_k=max_doc_len,
    causal=True,
)

# 2. Loss masking: compute loss only on non-separator tokens
# Mask out <eos> and any padding from loss calculation
loss_mask = (targets != tokenizer.pad_id) & (targets != tokenizer.eos_id)
loss = F.cross_entropy(logits.view(-1, vocab_size), targets.view(-1), reduction='none')
loss = (loss * loss_mask.view(-1)).sum() / loss_mask.sum()
Document boundary masking is not optional. Without it, the model learns to predict the beginning of doc2 from the content of doc1 — which teaches it spurious cross-document correlations and degrades downstream performance. All modern training codebases (Megatron, torchtitan, Llama-recipes) support document-aware packing.

# Standard pretraining hyperparameters
optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=3e-4,
    betas=(0.9, 0.95),
    weight_decay=0.1,
    fused=True,
)

# Apply weight decay only to weight matrices, not biases/norms/embeddings
decay_params = [p for n, p in model.named_parameters()
                if p.dim() >= 2]  # weight matrices are 2D+
no_decay_params = [p for n, p in model.named_parameters()
                   if p.dim() < 2]  # biases and norm params are 1D
optimizer = torch.optim.AdamW([
    {'params': decay_params, 'weight_decay': 0.1},
    {'params': no_decay_params, 'weight_decay': 0.0},
], lr=3e-4, betas=(0.9, 0.95), fused=True)

5 Learning rate schedules

Warmup + decay schedulesSchedule
ScheduleFormula (after warmup)Used byPros / cons
Cosine decayLR = min_lr + 0.5×(peak_lr - min_lr)×(1 + cos(π × step/total))LLaMA, GPT-3, MistralPro: smooth decay, well-studied. Con: must know total steps in advance.
Linear decayLR = peak_lr × (1 - step/total)Some smaller runsPro: simple. Con: decays too fast early on, too slow late.
Inverse sqrtLR = peak_lr × √(warmup_steps / step)Original Transformer, PaLMPro: doesn't need total steps. Con: decays very slowly — may be too high late in training.
WSD (Warmup-Stable-Decay)Warmup → constant LR → rapid cooldownMiniCPM, recent researchPro: can continue training at any point (just extend the stable phase). Most flexible for large runs with unknown total steps.
# Cosine schedule with warmup (most common)
import math

def get_lr(step, warmup_steps, max_steps, peak_lr, min_lr):
    # Linear warmup
    if step < warmup_steps:
        return peak_lr * step / warmup_steps
    # Cosine decay
    progress = (step - warmup_steps) / (max_steps - warmup_steps)
    return min_lr + 0.5 * (peak_lr - min_lr) * (1 + math.cos(math.pi * progress))

# Typical: min_lr = peak_lr / 10
# Warmup: 1000–2000 steps
The cooldown phase matters more than you think. Recent work (MiniCPM, DeepSeek) shows that a dedicated "cooldown" phase at the end of training — where LR drops rapidly to near-zero while training on higher-quality data — produces outsized quality improvements. The WSD schedule formalizes this: train at constant LR, then do a sharp decay over the final 10–20% of steps.

The three failure modes

FailureWhat it looks likeCommon causesFix

The z-loss trick

Large logit values before softmax can cause numerical instability. The z-loss adds a penalty on the log-partition function: z_loss = 1e-4 * (torch.logsumexp(logits, dim=-1) ** 2).mean(). This regularizes logit magnitudes and prevents softmax saturation. Used by PaLM and some other models. Small training quality cost for much better numerical stability.

Recovering from loss spikes

If a loss spike doesn't self-recover within 50–100 steps: (1) roll back to the last checkpoint before the spike, (2) skip the offending data batches, (3) reduce LR by 10–20%, (4) resume. This is why frequent checkpointing (every 500–1000 steps) is essential. At the scale of training a 70B model, a single unrecoverable spike can waste $100K+ of compute.

The µP (maximal update parameterization) alternative: Standard parameterization (SP) requires retuning hyperparameters when changing model width. µP defines a parameterization where optimal hyperparameters transfer across model sizes. You can tune LR on a tiny model and directly apply it to the full model. This saves enormous hyperparameter search cost at scale. Implementation: adjust initialization, LR per-layer, and attention scaling by width-dependent factors.

7 Scaling laws

Chinchilla scaling and compute-optimal trainingScaling
Kaplan (OpenAI) scaling law:
L(N, D) ≈ A/N^α + B/D^β + L∞
where N = parameters, D = training tokens, L = loss

Chinchilla optimal (Hoffmann et al.):
D_opt ≈ 20 × N
Train on ~20 tokens per parameter for compute-optimal loss.
7B model → ~140B tokens. 70B model → ~1.4T tokens.

But "compute-optimal" ≠ "inference-optimal":
LLaMA-3 8B was trained on 15T tokens (1875× params).
Massively over-trained relative to Chinchilla — because inference cost
depends only on model size, not training tokens. Training longer makes
the model better at fixed inference cost.

Practical implications of scaling laws

InsightWhat it means for you
Loss is predictable from N and DYou can estimate final loss (and roughly quality) before training by running small-scale experiments and extrapolating. This enables "scaling law runs" — train 100M, 300M, 1B models to predict 7B performance.
Over-training is increasingly commonIf you're deploying the model for inference, training well beyond Chinchilla-optimal is worth it. Training cost is one-time; inference cost is ongoing. Budget for 100–200× parameters in tokens.
Data becomes the bottleneckAt 15T tokens, you may exhaust available high-quality data. This drives interest in synthetic data, multi-epoch training (repeat data), and data quality improvements.
Hyperparameters don't transfer perfectly across scaleLR, batch size, and warmup need adjustment when going from small experiments to full-scale. µP can help, but empirically most teams still do some tuning at target scale.

9 The training loop

Complete pretraining loop with all best practicesCode
# Simplified but complete pretraining loop
import torch
from torch.distributed import init_process_group
from torch.distributed._composable.fsdp import fully_shard

# 1. Setup distributed
init_process_group(backend='nccl')
rank = torch.distributed.get_rank()
device = torch.device(f'cuda:{rank % torch.cuda.device_count()}')

# 2. Build model
model = build_llama_model(config).to(device).to(torch.bfloat16)

# 3. Apply FSDP sharding
for layer in model.layers:
    fully_shard(layer)
fully_shard(model)

# 4. Activation checkpointing
from torch.utils.checkpoint import checkpoint
for layer in model.layers:
    layer.forward = lambda *args, _layer=layer: \
        checkpoint(_layer._original_forward, *args, use_reentrant=False)

# 5. Optimizer
optimizer = torch.optim.AdamW(
    get_param_groups(model),  # separate decay / no-decay groups
    lr=3e-4, betas=(0.9, 0.95), weight_decay=0.1, fused=True
)

# 6. Training loop
for step in range(max_steps):
    # Gradient accumulation
    for micro_step in range(grad_accum_steps):
        batch = next(data_iter)
        input_ids = batch['input_ids'].to(device)
        labels = batch['labels'].to(device)

        with torch.autocast('cuda', dtype=torch.bfloat16):
            logits = model(input_ids)
            loss = F.cross_entropy(
                logits.view(-1, vocab_size),
                labels.view(-1),
                ignore_index=-100,   # ignore padding
            )
            loss = loss / grad_accum_steps  # normalize by accum steps

        loss.backward()

    # Gradient clipping (CRITICAL for stability)
    grad_norm = torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)

    # LR schedule
    lr = get_cosine_lr(step, warmup=2000, max_steps=max_steps,
                       peak_lr=3e-4, min_lr=3e-5)
    for pg in optimizer.param_groups:
        pg['lr'] = lr

    # Step
    optimizer.step()
    optimizer.zero_grad()

    # Logging
    if step % 10 == 0 and rank == 0:
        log({'loss': loss.item() * grad_accum_steps,
             'grad_norm': grad_norm.item(), 'lr': lr,
             'tokens': step * global_batch_tokens})

    # Checkpoint
    if step % 1000 == 0:
        save_checkpoint(model, optimizer, step)

    # Evaluation
    if step % 500 == 0:
        val_loss = evaluate(model, val_loader)
        if rank == 0:
            log({'val_loss': val_loss})

Common bugs and their symptoms

SymptomLikely bugDebug approach
Loss doesn't decrease at allLabels are wrong (e.g., shifted incorrectly), LR is zero, model is frozenVerify labels match shifted input_ids. Print LR. Check requires_grad on parameters.

Checkpointing strategyOperations
AspectRecommendationWhy
FrequencyEvery 500–1000 steps + keep last 3Balances I/O overhead with recovery granularity. At $10K+/day for large runs, losing 1000 steps is expensive.
What to saveModel state, optimizer state, scheduler state, RNG states, step count, data positionFull reproducibility requires all of these. Forgetting the data position means repeating or skipping data on resume.
FormatDistributed checkpoint (torch.distributed.checkpoint) or FSDP state dictSaves sharded checkpoints in parallel — much faster than gathering to rank 0 for large models.
Async savingSave to a background thread/processAvoids blocking training for 30–60s during checkpoint writes on large models.
Validation on loadAfter loading a checkpoint, run one forward pass and compare loss to the logged value at that stepCatches silent corruption in checkpoint files (bit rot, incomplete writes).
# Distributed checkpointing with PyTorch
import torch.distributed.checkpoint as dcp

# Save (parallel across all ranks)
state_dict = {
    "model": model.state_dict(),
    "optimizer": optimizer.state_dict(),
    "step": step,
    "rng": torch.cuda.get_rng_state(),
}
dcp.save(state_dict, checkpoint_id=f"checkpoints/step_{step}")

# Load
state_dict = {"model": model.state_dict(), "optimizer": optimizer.state_dict()}
dcp.load(state_dict, checkpoint_id=f"checkpoints/step_{step}")
model.load_state_dict(state_dict["model"])
optimizer.load_state_dict(state_dict["optimizer"])

The ML engineer's mental model for LLM training: Architecture is mostly solved (decoder-only + RoPE + GQA + SwiGLU + RMSNorm). Data quality is the primary differentiator — invest heavily in curation, deduplication, and mixing. Use standard hyperparameters (AdamW, cosine LR, gradient clipping) and adjust LR + batch size based on scaling law experiments. Monitor training obsessively — loss, gradient norms, parameter norms — and checkpoint frequently enough to survive spikes. After pretraining: SFT on high-quality instruction data (quality >> quantity), then DPO/RLHF for alignment. Think of the training pipeline as four stages — pretraining → SFT → alignment → serving — each with its own data, hyperparameters, and failure modes.
