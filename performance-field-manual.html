BOTH Chapter I

Why training & inference diverge.

Read time 6 min

The premise Same hardware, same model, same framework — but the question "is my GPU full?" has two opposite answers depending on which workload you're running.

Mantra Training fills GPUs by default. Inference empties them by default. The work is to invert both.

Training and inference look like the same workload to outside observers. They're not. Training is throughput-oriented, compute-bound, and synchronous. Inference is latency-oriented, memory-bound, and asynchronous. Every optimization decision branches off this divide.

Training
Throughput · synchronous · compute-bound
  • One coherent job, fixed world size, runs for hours-to-weeks
  • All ranks march in lockstep — collectives every step
  • Large batches; matmul-dominated; saturates Tensor Cores
  • Memory holds: weights + gradients + optimizer state + activations
  • Optimize for MFU (Model FLOPs Utilization)
  • Failure mode: a slow rank, a hung collective, a checkpoint stall
Inference
Latency · asynchronous · memory-bound
  • Thousands of small requests, variable shapes, p99 matters
  • Each request has two phases with opposite characteristics
  • Small per-request work; KV cache reads dominate
  • Memory holds: weights + KV cache (and the cache grows with traffic)
  • Optimize for tokens/sec/$ at a TTFT/TPOT SLO
  • Failure mode: tail-latency blow-ups, KV-cache eviction, queue depth

The single most important difference

Training computes one matmul per layer per step on a very large batch (often thousands of tokens) — the arithmetic intensity is enormous, so each weight matrix is reused across many tokens. The GPU's tensor cores run hot. Compute-bound.

Inference, after the prompt is processed, generates one token per request at a time. The model reads the same enormous weight matrices to produce a single output. The arithmetic intensity collapses. Memory-bandwidth-bound.

This is why every advanced inference technique you'll read about — paged KV cache, continuous batching, speculative decoding, prefill-decode disaggregation, MQA/GQA — ultimately tries to do one thing: raise arithmetic intensity back up by amortizing weight reads across many tokens.

The bottom line

You cannot reuse training optimization intuition for inference. A bigger batch helps training nearly always; for inference, a batch that's too big blows your TPOT SLO while one that's too small wastes the GPU. The whole problem is finding the dynamic batch size that maximizes throughput under a latency constraint.

BOTH Chapter II

The roofline & what it tells you.

Interactive Click workloads to see where each lands on the roofline.

The lesson The roofline tells you which physical resource is the ceiling for each operation — compute or memory bandwidth. Optimizing the wrong one wastes weeks.

The roofline model is the single most useful diagnostic in performance work. On one axis: arithmetic intensity (FLOPs per byte). On the other: achieved performance (FLOPs/sec). The chart's outline is a hard ceiling defined by hardware. Every kernel you run is a point on the chart.

Interactive · Roofline Plot · H100 SXM
Where does your workload land?

How to read it

Two ceilings: the slope (memory bandwidth × arithmetic intensity) and the flat top (peak compute). The slope tells you "if your kernel reads X bytes per FLOP, this is the fastest it can go regardless of how good your CUDA code is." The flat top tells you "if you're compute-bound, this is the peak the hardware can deliver."

Below the slope (memory-bound region). The kernel is limited by bandwidth. Adding more SMs, switching to fancier matmul instructions, increasing clock speed — none of these help. You need to read fewer bytes or read them more efficiently (fusion, caching, quantization).

Below the flat top (compute-bound region). The kernel can use the matmul units. Optimization means using them better — bigger tile sizes, Tensor Cores instead of FP32 CUDA cores, better SM occupancy.

On the roof. The kernel is saturating hardware. Stop optimizing this kernel; find the next bottleneck.

The diagnostic move

For any slow kernel, run ncu --set roofline (Nsight Compute) to get its exact arithmetic intensity. If it's left of the inflection (memory-bound), bandwidth tricks help — FlashAttention, fused kernels, quantization. If it's right (compute-bound), pump arithmetic — bigger tiles, FP8 vs BF16, Tensor Core paths. Optimizing the wrong axis is the most common mistake in performance work.

BOTH Chapter III

The universal investigation protocol.

The discipline Same five steps for any performance problem in either workload. Skip a step and you'll spend a week chasing the wrong fix.

Why steps matter Each step gates the next. Without baseline, you can't tell if changes help. Without isolation, you'll fix things that aren't broken.

The same five-step protocol resolves every performance problem in either workload. The protocol is boring; the discipline is everything.

Step 1 — Establish baseline

Before any change, measure: step time (training) or TTFT / TPOT / throughput (inference). Get a clean baseline at the exact scale and config you care about. Without this, you can't tell whether a change helped, hurt, or did nothing.

The trap people fall into: profiling on a tiny model "to iterate fast." Tiny models have different bottlenecks. Profile at production-scale — or at least at a scale where the dominant kernels match production.

Step 2 — Pick the right metric layer

Performance has four layers, and the bottleneck might be at any of them:

Cluster level — GPU-hours allocated vs. used, queue depth, scheduling latency.

Node level — GPU utilization (DCGM), memory bandwidth, NCCL bandwidth, NUMA stalls.

Framework level — PyTorch Profiler / Nsight Systems: which Python operator launched which kernel, when, and how long it took.

Kernel level — Nsight Compute / ncu: which SM stalled on what, achieved memory bandwidth vs. peak, Tensor Core utilization.

Start at cluster level, descend only when needed. Most issues are visible at framework level.

Step 3 — Profile, don't guess

Tools, in order of when to reach for them:

# Real-time, low-overhead, always-on
nvidia-smi dmon -s pucvmet -i 0          # GPU + power + mem
dcgmi dmon -e 203,1004,1005,1009,1010      # DCGM fields

# Framework-level (PyTorch native)
with torch.profiler.profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=torch.profiler.schedule(wait=1, warmup=2, active=3),
    on_trace_ready=tensorboard_trace_handler("./prof"),
    record_shapes=True,
    profile_memory=True,
):
    for step in range(10):
        train_step(); prof.step()

# System-wide timeline (NVIDIA)
nsys profile -o train --trace=cuda,nvtx,osrt python train.py
# View with Nsight Systems GUI

# Kernel deep-dive (NVIDIA)
ncu --set roofline --kernel-id ::regex:"gemm":5 ./train
# Roofline analysis for the 5th gemm kernel

# Inference-specific
vllm-cli serve <model> --enable-prefix-caching --max-num-batched-tokens 8192 \
  --enable-chunked-prefill --metrics-port 9090
# Scrape Prometheus metrics for TTFT, TPOT, queue depth

Step 4 — Isolate one variable

The cardinal sin of performance work: making two changes at once. You'll attribute the wrong outcome to the wrong cause and ship the wrong fix to production. Change one thing. Measure. Compare to baseline. Decide. Repeat.

Step 5 — Verify at scale

A fix that improves single-node performance often regresses at 64-node scale (e.g., a custom kernel that doesn't overlap with NCCL). Always re-measure at the target scale before declaring victory.

The most common failure

Skipping step 1 (baseline) and step 4 (isolation) — together. Without them, you optimize random pieces, watch numbers move, and ship "improvements" that don't reproduce in production. Discipline matters more than cleverness.

TRAINING Chapter IV

Training · metrics & their targets.

The five panels You need exactly five dashboards for a training run. More clutters; fewer leaves blind spots.

The ceiling Frontier models hit 38–50% MFU. If you're at 25%, there's something obvious to fix. If you're at 60%, you're doing exceptional work.

Training performance has a small, fixed set of metrics that matter. Hit the targets and your run is healthy; miss any of them and you have a specific knob to turn.

MFU
Model FLOPs Utilization
Ratio of model FLOPs you achieve vs. hardware's theoretical peak. The headline number. Computed from step time, model FLOPs/token, batch, world size.
target: 40–60%
SM occupancy
SM Active %
DCGM field 1003. What fraction of GPU SMs are busy across the step. Sawtoothing = data starvation. Flat-low = sync overhead or tiny kernels.
target: ≥ 80%
TC %
Tensor Core Active
DCGM 1004. Specifically the Tensor Cores (where matmul happens). High SM but low TC means your work isn't going to matmul units — likely fp32 kernels or non-Tensor-Core paths.
target: ≥ 50%
HBM BW
HBM Bandwidth
DCGM 1005. GB/s of HBM read+write. High BW + low TC = memory-bound (good for attention; bad for matmul). Low BW + low TC = idle (a sync bug).
target: workload-dependent
step time
Step time (P50, P99)
Wall-clock per training step. Watch both percentiles; if P99 is 2× P50, you have stragglers — one rank is dragging the rest.
target: stable, P99/P50 < 1.1
NCCL BW
All-reduce bandwidth
From nccl-tests at job scale. If <80% of theoretical, your network or topology is leaking — see Issue 02.
target: ≥ 80% of fabric peak

How to actually compute MFU

# Step 1: estimate model FLOPs per token (Kaplan/Chinchilla approx)
# For a dense transformer: ~6 × N (params) for fwd+bwd per token
model_flops_per_token = 6 * num_params

# Step 2: tokens per step
tokens_per_step = batch_size * seq_len * world_size

# Step 3: achieved FLOPs/sec/GPU
total_flops_per_step = model_flops_per_token * tokens_per_step
flops_per_sec_per_gpu = total_flops_per_step / step_time_seconds / world_size

# Step 4: ratio against hardware peak
# H100 SXM: 989 TFLOPs/s for BF16, 1979 TFLOPs/s for FP8 dense
hw_peak_bf16 = 989e12
mfu = flops_per_sec_per_gpu / hw_peak_bf16

Where MFU goes when it's not there

If you're at 25% MFU and you should be at 50%, the missing 25% lives somewhere. Profile to find it:

If TC% < 30%: Tensor cores aren't being used. Common cause: fp32 ops (loss computation, layernorm in fp32, fp32 optimizer state being touched in the hot path), or non-aligned shapes that fall back to slower paths.

If SM% sawtooths between 90% and 10%: Data starvation — the GPU finishes work and waits for the next batch.

If SM% is flat at 60–70% with TC% similar: Communication is in the critical path. NCCL all-reduce isn't overlapping with backward computation. Check that the gradient bucket size matches what FSDP/DDP expects.

If SM% is high but step time is high: The math is happening, you just have inefficient layers (small matmul shapes, repeated kernel launches). Use torch.compile, fuse with FlashAttention, raise micro-batch.

Real-world MFU numbers

For reference: Llama 3 405B reported ~38% MFU on 16,384 H100 GPUs. 70B-class dense models routinely hit 45–55% MFU with FlashAttention + FSDP. Models below 38% have something obvious to fix. Above 60% on a dense model is rare and impressive.

TRAINING Chapter V

Training · bottlenecks, in order.

Click Each timeline to see how the bottleneck shows up in the profiler.

The order matters Bottlenecks are ranked by likelihood × impact. Fix top of list first.

Eight bottlenecks cover 95% of training performance problems. Below, each rendered as a profiler timeline. Click to see what each looks like in practice.

Interactive · Profiler Timeline — what each bottleneck looks like

The full ranking, by typical impact

1. Data loading. Sawtooth GPU util. Cause: num_workers too low, /dev/shm too small, slow PVC, CPU-bound augmentation. Fix order: raise num_workers and prefetch_factor, mount /dev/shm as memory emptyDir, cache hot shards locally, move augmentation to GPU (DALI / GPU transforms).

2. Communication not overlapping. Forward + backward + NCCL appear sequentially in the profile instead of overlapped. Cause: gradient bucket too small (every parameter triggers a tiny all-reduce), or communicator hook ordering is wrong. Fix: raise bucket_cap_mb in DDP (default 25 MB — try 100–250). For FSDP, ensure HYBRID_SHARD or right unit boundaries.

3. Recomputation. Activation checkpointing saves memory at the cost of re-running the forward during backward. Cheap option: too aggressive a checkpoint policy doubles forward FLOPs. Fix: checkpoint only transformer blocks (not every layer), check that selective recomputation isn't replaying matmul.

4. Synchronous checkpoint I/O. Every N steps the run pauses for tens of seconds while a 100GB checkpoint writes. Fix: PyTorch DCP async checkpointing, sharded saves, write to local NVMe first then upload to object store.

5. Straggler rank. One rank is 2× slower than peers; all others wait at every collective. Cause: a slow GPU, NUMA misbinding, a CPU-pinned data loader on a busy CPU. Fix: enable kubelet topology-manager-policy=single-numa-node and cpu-manager-policy=static; use Flight Recorder timing to identify the rank.

6. Tensor Cores not engaged. Layernorm and loss in fp32; matmul shapes not multiple of 16 (or 64 for SM90); non-contiguous tensors. Fix: torch.compile with mode='reduce-overhead', use nn.LayerNorm with cast inputs, FlashAttention.

7. Python overhead. Many small ops, each with kernel-launch overhead. Visible as gaps between CUDA kernels in the profile. Fix: torch.compile to graph-capture, CUDA Graphs for static shapes.

8. NUMA / topology. Per-rank performance varies on the same node. Some GPUs see 3× the memory latency. Fix: pin processes to NUMA nodes co-located with their GPUs.

The order in which to investigate

1. Profile one step with PyTorch Profiler + Nsight Systems.
   → Open the timeline view. Find the longest gap or longest kernel.

2. Gap between data load & forward?     → bottleneck 1 (data)
   Forward → backward → comm sequential?  → bottleneck 2 (comm)
   Backward 2× expected length?           → bottleneck 3 (recompute)
   Long flat pause every N steps?         → bottleneck 4 (ckpt)
   One rank much slower in FR timing?     → bottleneck 5 (straggler)

3. After overlap fixed, drop into Nsight Compute for the dominant kernel.
   TC % < 30 on a matmul?                 → bottleneck 6 (Tensor Core)
   Many sub-µs kernels with gaps?         → bottleneck 7 (Python)
   Per-rank variance on same node?        → bottleneck 8 (NUMA)
Cross-reference

Bottlenecks 5 (stragglers) and 8 (NUMA) often show up first as NCCL hangs, not as slow steps. If you've been here from Issue 02 (NCCL Field Manual), the diagnostic path overlaps — Flight Recorder timing tells you the same thing as profiler stragglers.

INFERENCE Chapter VI

Inference · prefill, decode & the two latencies.

The fork Every request has two phases with opposite characteristics. The whole performance story is figuring out how to handle them together.

SLOs TTFT (Time To First Token) and TPOT (Time Per Output Token). The user feels both, but they trade off against each other.

An inference request has two phases. The prefill phase processes the input prompt in parallel and is compute-bound. The decode phase generates tokens one at a time autoregressively and is memory-bound. Optimizing one without the other ships a service that's fast at the wrong thing.

TTFT
Time To First Token
Wall-clock from request submission to first generated token. Dominated by prefill latency + queue wait. The "perceived responsiveness" metric.
target: P95 < 1s for chat
TPOT / ITL
Time Per Output Token
Wall-clock between successive generated tokens. Dominated by decode. The "feels fast while streaming" metric.
target: P95 < 50ms
throughput
Tokens/sec/GPU
Aggregate generated tokens per GPU per second across all concurrent requests. The cost metric — directly proportional to $ per million tokens served.
target: workload-dependent
goodput
SLO-attainment throughput
Tokens/sec/GPU only counting requests that hit their TTFT + TPOT SLOs. A server with 99% throughput at 50% SLO attainment is not delivering 99% — it's delivering 49.5%.
target: ≥ 95% attainment

Why prefill and decode interfere

If you run them naively on the same GPU pool, every time a long prefill arrives, decode tokens for other in-flight requests get blocked. One long-prompt request slows every other user's streaming. The user with a 30,000-token prompt is delaying the user with a 50-token chat reply.

Three architectural responses, in increasing sophistication:

1. Continuous batching (Orca / vLLM). Instead of fixed batches, dynamically add requests to in-flight batches whenever a slot opens. Massive throughput win. The single highest-impact change you can make.

2. Chunked prefill (Sarathi / vLLM). Break long prefills into chunks. Mix one prefill chunk with a batch of decodes per step — the prefill chunk saturates compute, the decodes "piggyback" using bandwidth that would otherwise be idle. Reduces TPOT spikes during long prefills.

3. Prefill-decode disaggregation (DistServe / Splitwise). Run prefill workers and decode workers on separate GPU pools. Each pool can be tuned independently — different parallelism, different hardware, different batch sizes. Adds operational complexity but lets you hit tight TTFT and tight TPOT simultaneously.

The TTFT/TPOT trade-off

Aggregated serving (one pool) optimizes TTFT when prefills are small and TPOT when decodes dominate, but cannot do both. Disaggregated serving optimizes both at the cost of more GPUs and added latency from KV-cache transfer between the two pools.

Decision heuristic: if your traffic is conversational chat (medium prompts, medium outputs) and you have tight TTFT SLOs, aggregated with chunked prefill is probably enough. If your traffic mixes long-context RAG (huge prompts) with chat, you need disaggregation.

The single most important inference setting

Enable continuous batching. In vLLM this is default. In hand-rolled servers, it's the difference between 200 tokens/sec/GPU and 2,000 tokens/sec/GPU. If your serving framework doesn't do continuous batching, switch frameworks before tuning anything else.

INFERENCE Chapter VII

Inference · the memory wall.

The math Inference is HBM bandwidth-bound. Period. Every optimization is some form of "read fewer bytes per generated token."

The single biggest waster KV cache. Often 2× the size of weights. Manage it like the precious resource it is.

Inference's decode phase is memory-bandwidth-bound. Every generated token requires reading every weight in the model and every KV-cache entry up to the current position. On H100, you have 3 TB/s of HBM bandwidth and ~140GB of memory; you must spend both wisely.

The KV cache, sized

For a request with L tokens in a model with n_layers, n_heads, head_dim, FP16:

# Per request, in bytes:
kv_bytes = 2 * n_layers * n_kv_heads * head_dim * L * 2
#         ↑ K and V    ↑ KV-specific (GQA)         ↑ FP16

# Concrete: Llama-3-70B, GQA with 8 KV heads, 80 layers, head_dim 128
# For L=8192: kv = 2 * 80 * 8 * 128 * 8192 * 2 = ~2.6 GB per request
# A 70B BF16 model is ~140GB — KV for 50 long requests is bigger than weights

Three implications:

1. KV cache memory is the binding constraint on batch size. Not weights, not activations. If you can hold 100 requests' worth of KV cache, your max batch is 100. PagedAttention (vLLM) reduces fragmentation; quantization (KV cache in INT8 or FP8) doubles capacity.

2. Long contexts blow up linearly. A 100k-token context is 12× the KV cache of an 8k context. Long-context serving is fundamentally more expensive per request, and the cost is in memory, not compute.

3. Reading KV every token is what makes decode slow. Decode bandwidth ≈ (model weights + KV cache for batch) × tokens_per_sec. On H100 with 3 TB/s and a 140GB model, peak decode without KV is ~21 tokens/sec/request. Increase batch to amortize.

The optimization stack, ranked by impact

1. PagedAttention. vLLM's contribution. Treats KV cache like virtual memory pages instead of contiguous allocations. Eliminates fragmentation. Practical batch sizes 2–4× larger.

2. Continuous batching (covered Chapter VI). Doubles or triples throughput.

3. Quantization. Weights to INT8 / FP8 / INT4 cuts memory bandwidth proportionally. Decode is bandwidth-bound, so FP8 weights run nearly 2× faster than BF16 weights with minor quality loss. KV cache to FP8 doubles batch capacity.

4. Speculative decoding. A small "draft" model proposes K tokens, the large model verifies them in one forward pass. If draft is decent (50%+ acceptance rate), 1.5–3× speedup on decode. Memory cost: draft model weights + a few KB of extra state.

5. Multi-Query Attention (MQA) / Grouped Query Attention (GQA). Architectural — share KV heads across query heads. Shrinks KV cache 4–8×. Done at model design time, not serving.

6. Prefix caching. If many requests share a prefix (system prompts, RAG retrieved context), cache the prefix KV across requests. Massive win for chat assistants and RAG workloads.

7. Tensor parallelism for memory headroom. Split model across N GPUs; KV cache also splits N ways. Lets you fit bigger batches at the cost of one all-reduce per layer. Helpful past a single GPU's memory ceiling.

The order to attempt them

Start with vLLM defaults (PagedAttention + continuous batching). Then enable prefix caching (free, big win for chat). Then chunked prefill if TTFT spikes during long prompts. Then quantization (FP8 first on H100/Blackwell; INT4 if you can spare the quality). Speculative decoding is the last optimization to add — significant operational complexity for a 1.5–2× win.

Nine real cases, annotated.

The cases Five training, four inference. All from production. Each: context, symptom, debug, fix.

Production cases, presented in the same forensic form as the K8s and NCCL field manuals. Read them once, you'll start recognizing the patterns from your own dashboards.

01
The 27% MFU mystery
Training Large-scale 512 H100
Context

70B-parameter pretraining at 512 H100. Targeting 50% MFU based on similar models. Actual: 27%.

Symptom

Step time stable, no hangs, no OOM. DCGM shows GPU SM% at 92% (great), but Tensor Core % at 28% (bad). HBM bandwidth at 1.8 TB/s of 3 TB/s peak.

Debug

The signature — high SM, low TC — means the GPU is busy but not on matmul. Nsight Compute on the slowest kernel reveals it's a fused fp32 layernorm + cast kernel, taking 8% of step time.

PyTorch Profiler shows a hot path: every transformer layer dispatches a Python-side conditional that casts to fp32 for numerical stability, then casts back. Each cast is its own kernel.

Fix
# 1. Use a fused layernorm (NVIDIA Apex or torch native)
from apex.normalization import FusedLayerNorm
# vs the default nn.LayerNorm with explicit casts

# 2. Compile the model so casts fuse into matmuls
model = torch.compile(model, mode="reduce-overhead")

# 3. Use FlashAttention-3 (FP8 on H100) for attention
from flash_attn import flash_attn_func

Result: TC% climbed from 28% to 61%, step time dropped 32%, MFU went from 27% to 44%. Six engineer-hours; saved ~$2M over the training run.

02
The data-loader sawtooth
Training Medium-scale 8× A100
Context

Vision transformer fine-tune, 8× A100 single node. Expected ~1.2s/step; getting 4.1s/step.

Symptom

nvidia-smi -l 1 shows GPU util oscillating between 4% and 97% on a ~3-second period. Classic data-loader sawtooth.

Debug

PyTorch Profiler with profile_memory=True shows long dataloader_iter_next spans between forward calls. num_workers was 2.

df -h /dev/shm on the pod: 64MB. PyTorch DataLoader uses shared memory for multiprocessing IPC. 64MB is K8s default and immediately exhausted.

Fix
# 1. Bump shm via emptyDir in K8s pod spec
volumes:
  - name: shm
    emptyDir: { medium: Memory, sizeLimit: 64Gi }
volumeMounts:
  - { name: shm, mountPath: /dev/shm }

# 2. Increase num_workers and prefetch
DataLoader(dataset, batch_size=64, num_workers=16,
           prefetch_factor=4, persistent_workers=True,
           pin_memory=True)

# 3. Move image augmentation to GPU
import kornia.augmentation as K
gpu_aug = K.AugmentationSequential(...).cuda()

Step time went from 4.1s to 1.3s. The shm fix alone was 70% of the win.

03
The straggler in a 4-node FSDP run
Training Medium-scale 4 nodes
Context

32-GPU FSDP fine-tune across 4 nodes. Step time variance: P50 = 2.1s, P99 = 4.8s. Costly tail.

Symptom

Flight Recorder timing (TORCH_NCCL_ENABLE_TIMING=1) shows ranks 24–31 (the 4th node) consistently arrive at AllReduce 200–600ms later than ranks 0–23.

Debug

Node 4's nvidia-smi reveals: 8 GPUs all at 96% util, but their NUMA affinity is wrong. numactl -H shows GPUs 0–3 on NUMA node 0, GPUs 4–7 on NUMA node 1, but all 8 trainer processes are scheduled on NUMA node 0. Half the GPUs are doing cross-socket memory access at 50GB/s instead of the local 200GB/s.

Root cause: the K8s pod was started before topology-manager-policy=single-numa-node was applied to the kubelet on node 4 — config drift from a recent upgrade.

Fix
# 1. Fix kubelet config drift (on the bad node)
# /var/lib/kubelet/config.yaml
topologyManagerPolicy: single-numa-node
cpuManagerPolicy: static
reservedSystemCPUs: "0-7"

# 2. Restart kubelet; cordon, drain, reboot, uncordon

# 3. Ensure pod requests Guaranteed QoS so topology manager engages
resources:
  requests: { cpu: "32", memory: 256Gi, nvidia.com/gpu: "8" }
  limits:   { cpu: "32", memory: 256Gi, nvidia.com/gpu: "8" }
# requests MUST equal limits for the topology manager to engage

P99 step time dropped from 4.8s to 2.3s.

04
The async checkpoint that wasn't
Training Large-scale 128 nodes
Context

1024-GPU run, sharded checkpoints written every 500 steps. Each checkpoint write supposedly async, but step time pauses 90 seconds during write.

Symptom

Step time graph: smooth 2.4s with periodic spikes to 92s — exactly when checkpoints happen.

Debug

The framework claims to use PyTorch DCP async checkpoint, but Nsight Systems shows a synchronous torch.cuda.synchronize() followed by tensor.to(cpu) in the main training stream, not an offload stream. The "async" path actually copied GPU→CPU synchronously before queuing the write.

Root cause: a custom training loop called the legacy torch.save API on the result of get_state_dict(), which doesn't pipeline.

Fix
import torch.distributed.checkpoint as dcp
from torch.distributed.checkpoint.state_dict import get_state_dict
from torch.distributed.checkpoint import async_save

# Get sharded state without copying
sd = get_state_dict(model, optimizer)

# Returns a Future. Don't await it — let it run in background.
future = async_save(state_dict=sd, checkpoint_id=f"step_{step}",
                    storage_writer=dcp.FileSystemWriter("/ckpt"))

# Optionally check completion at next checkpoint boundary
if step % 500 == 0 and last_future:
    last_future.result()                   # now block if not done
last_future = future

Step time pause went from 92s to 1.1s (just the launch overhead). Pipeline runs the GPU→CPU copy on a dedicated stream and the disk write on a CPU thread, both concurrent with training.

05
The communication that wouldn't overlap
Training Medium-scale 16 GPUs DDP
Context

Mid-size language model (1.3B) on 16 A100s, DDP. Achieving 35% MFU, expected 55%+.

Symptom

Nsight Systems timeline: forward, backward, then a 100ms NCCL all-reduce — all sequential. Expected: NCCL chunks should overlap with backward.

Debug

The default DDP bucket size is 25MB, but the model has 1.3B params × 2 bytes = 2.6GB of gradients. That's ~100 buckets × 25MB. The first bucket finishes ready-for-allreduce only at the very end of backward. Communication can't overlap with anything because there's nothing left to compute.

Verified with torch.distributed.DistributedDataParallel(..., bucket_cap_mb=25) in code — yes, defaults.

Fix
# Bigger buckets → fewer all-reduces → ones that start earlier
model = DDP(model, device_ids=[local_rank],
            bucket_cap_mb=200,           # up from 25
            gradient_as_bucket_view=True,  # reduce mem
            static_graph=True)             # enables better hooks

# Or move to FSDP HYBRID_SHARD for medium-scale
from torch.distributed.fsdp import ShardingStrategy
model = FSDP(model, sharding_strategy=ShardingStrategy.HYBRID_SHARD,
             auto_wrap_policy=transformer_auto_wrap_policy)

MFU climbed from 35% to 51%. The trick is bucket_cap_mb sized so the bucket completes ~30% through backward, giving 70% of backward time for the all-reduce to overlap.

06
The TTFT cliff at 50 QPS
Inference Medium-scale vLLM
Context

Llama-3-70B served via vLLM on 4× H100. Up to 50 QPS: TTFT P95 = 800ms. At 55 QPS: TTFT P95 = 4.2s.

Symptom

Sharp cliff in TTFT, not gradual. TPOT mostly stable. Throughput tokens/sec/GPU plateaus right at the cliff.

Debug

vLLM metrics endpoint (Prometheus): vllm:num_requests_waiting jumps from ~0 to 12+ above 50 QPS. vllm:gpu_cache_usage_perc approaches 100%. New prefills are queued because there's no KV memory to admit them.

Two things are happening: (1) KV cache is saturated, blocking admission; (2) chunked prefill is off, so each prefill takes a step "alone" and blocks decodes.

Fix
# Launch flags
vllm serve meta-llama/Llama-3-70B \
  --tensor-parallel-size 4 \
  --enable-chunked-prefill \                 # mix prefill + decode
  --max-num-batched-tokens 8192 \           # bigger budget
  --gpu-memory-utilization 0.92 \           # reclaim 7% headroom
  --kv-cache-dtype fp8 \                     # 2x KV capacity
  --enable-prefix-caching                    # reuse system prompts

Cliff moved from 50 QPS to 92 QPS. KV in FP8 alone doubled capacity; chunked prefill smoothed TTFT during long prompts.

07
The TPOT P99 spike
Inference Large-scale Mixed traffic
Context

Production chat service. TTFT P95 = 600ms (great). TPOT P50 = 28ms (great). TPOT P99 = 380ms (catastrophic — users see stuttering).

Symptom

The TPOT distribution is bimodal: most tokens in 20–35ms, a fat tail at 200–400ms. Correlates with 10–15 QPS arrival of long-context RAG requests (~20k token prompts).

Debug

Even with chunked prefill, a 20k-token prompt's prefill chunks each take ~80ms. During each chunk, decodes piggyback but at reduced rate. The user generating a chat response sees their TPOT briefly spike whenever a long-prompt RAG request is mid-prefill.

Fix

This is the disaggregation use case. Route long-prompt requests to a "prefill pool" and decode-only continuations to a "decode pool"; KV cache transferred over RDMA between them.

# Architecture sketch (vLLM disaggregated or NIXL-based)
# Two pools, each with their own parallelism:
#   prefill-pool:  4× H100, TP=4, optimized for prompt processing
#   decode-pool:   8× H100, TP=8, optimized for tokens/sec
# KV cache shipped prefill → decode via NVLink/IB
# Router: tag long-context requests; route by prompt length.

Result: TPOT P99 dropped from 380ms to 52ms. Total GPU footprint grew 30%, but goodput (SLO-attainment throughput) doubled.

08
The prefix cache that wasn't
Inference Medium-scale RAG service
Context

RAG application: every request has a 3,000-token system prompt + retrieved context, then a small user query. Throughput targets calculated assuming prefix caching saves 95% of prefill compute. Reality: 30% savings.

Symptom

vLLM metrics: gpu_prefix_cache_hit_rate at 22%, expected 90%+.

Debug

The system prompt is 3,000 tokens but ends with the user's session ID for traceability: ...<session_id={uuid}>. Every request's prompt is byte-different at the very end. Prefix caching matches at the page granularity (typically 16 tokens) — but the last few pages always mismatch, and vLLM doesn't currently support fractional-page prefix sharing across requests.

Fix

Move the session ID to the end of the user message, not the system prompt. The system prompt becomes byte-identical across requests; vLLM's prefix cache hits at 96%. Throughput up 2.4×; no model or infrastructure change needed.

Lesson: prefix caching's value is fragile to prompt template design. Audit any per-request token in the system prompt. RAG retrieved context is similarly sensitive — order documents canonically, not by score.

09
The speculative decoding regression
Inference Large-scale Speculative
Context

Speculative decoding deployed for 70B model with 7B draft. Single-user benchmarks: 2.1× speedup. Production at 60 QPS: 0.85× — actually slower.

Symptom

Per-request decoder generates more tokens/sec, but aggregate throughput drops. Goodput drops further because TPOT P99 worsens.

Debug

Speculative decoding wins on latency by trading extra compute for fewer bandwidth-bound steps. In a single-user scenario, GPU has plenty of compute headroom — the bandwidth wins. In a 60-QPS scenario, the GPU is already compute-saturated by other concurrent decodes, so the speculative extra compute slows everyone down. The draft tokens that get rejected are pure waste.

Also: at high QPS, the target model verifies large batches efficiently anyway — bandwidth-bound becomes less binding because each weight read serves many concurrent requests.

Fix

Make speculation adaptive: enable only when GPU compute is <70% utilized. When the server is saturated, switch back to standard decoding. Result: matches non-speculative throughput at peak load, gains 1.5–2× speedup during low-traffic windows where latency matters more.

# In the request scheduler:
def should_speculate(gpu_util, queue_depth):
    return gpu_util < 0.70 and queue_depth < 8

The general lesson: latency optimizations and throughput optimizations are often in tension. Always benchmark at production load, never at the trivial 1-QPS case.

What ties all nine together

Every case was diagnosed the same way: measure first, find the gap between expected and actual, isolate one variable, fix, verify. Six engineer-hours per case on average; multi-million-dollar impact per case. Profiling is the highest-leverage skill in distributed ML.

Medium-scale vs. large-scale: what changes.

The pivot Most performance work changes character at ~64 GPUs (training) or ~10 QPS (inference). New failure modes; new tools.

The cost Medium-scale wastes engineer time. Large-scale wastes hardware time. Tune your priorities accordingly.

Optimization techniques that dominate at one scale become irrelevant or harmful at another. The transition is sharp — at some point your single-node tricks stop working and a new class of problems takes over.

Concern
Medium-scale (1–32 GPU train; <10 QPS serve)
Large-scale (64+ GPU train; 100+ QPS serve)
What changes
Dominant bottleneck

Data loading, single-GPU kernel efficiency, framework overhead

Network bandwidth, fragmentation, stragglers, fault recovery

You move from being CPU-bound to being network-bound. New profiling tools required (NCCL telemetry, fabric counters).

Parallelism strategy (training)

DDP usually sufficient. FSDP HYBRID for fitting models.

FSDP FULL_SHARD + tensor parallel + pipeline parallel. 3D parallelism common.

From "one strategy" to "tuned mix of three." Plan parallelism per stage of training.

Failure handling

Restart from checkpoint manually. Hangs are rare and reproducible.

Auto-restart with fault tolerance. Hangs are common; need Flight Recorder always on.

Mean time between failures (MTBF) at 1024 GPUs is ~12 hours. You must engineer for failure, not against it.

Checkpoint strategy

Sync save every N steps to single PVC. Acceptable.

Async + sharded across all ranks. Multi-tier (local NVMe → object storage).

Checkpoint size grows with model; sync writes break the loop at scale.

Inference scheduling

Single vLLM instance per model. Continuous batching enough.

Multiple instances behind a router. Prefix-aware routing. Possible PD disaggregation.

Routing logic becomes a first-class concern. Prefix cache hit rate determines effective throughput.

Observability

nvidia-smi loops, occasional torch.profiler runs.

Always-on DCGM + Prometheus + Grafana. Flight Recorder. Distributed traces.

You can't manually inspect 1000 GPUs. Telemetry must aggregate and alert.

Cost model

Engineer-hours dominate. Cluster $/hour is small.

Hardware $/hour dominates. A 1% improvement saves real money.

What's worth optimizing flips. At 1024 H100, a 1% MFU gain ≈ $400k/year saved.

Topology awareness

Optional. All your GPUs are in one box anyway.

Mandatory. Cross-rack NCCL is 4× slower than intra-rack.

Scheduler must know about racks, IB switches, NVLink islands.

The transitions, in detail

From 1 node to 4 nodes (training). NCCL bandwidth becomes visible. DDP starts showing communication overhead. bucket_cap_mb matters. Topology of the IB fabric starts mattering. You need to verify GPUDirect RDMA is engaged.

From 8 nodes to 64+ nodes (training). Pipeline parallelism becomes attractive (smaller bubbles → less idle time). Gradient accumulation lets you trade compute for communication. Stragglers start dominating P99 step time. Fault tolerance becomes non-optional; MTBF starts being measured in hours, not days.

From 1 QPS to 50+ QPS (inference). Single-user latency stops being the dominant metric; tail latency takes over. Speculative decoding starts hurting more than helping (as in Case 9). Prefix caching becomes the highest-leverage optimization.

From 100 QPS to 1000+ QPS (inference). A single GPU pool isn't enough; you need routing. Prefix-cache-aware routing (route similar prompts to the same instance) becomes essential. PD disaggregation pays back its complexity cost.

The cost-flip insight

Medium-scale: optimize for engineer-time. Pick frameworks that "just work" (vLLM, PyTorch Lightning, HuggingFace Accelerate). Large-scale: optimize for hardware-time. Custom kernels, custom schedulers, in-house infrastructure start paying back. The exact scale where this flips depends on your $/hour rate vs engineer $/hour rate, but it usually happens between 64 and 256 GPUs.
