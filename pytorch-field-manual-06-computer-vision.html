Part I — Vision foundations

Pixels into predictions.

Vision work starts with getting images into tensors correctly and augmenting them well — unglamorous, but the difference between a model that generalizes and one that memorizes. This part nails the input side, surveys the modern CNN backbones you'll actually reach for, and applies the transfer-learning leverage from Volume III to images specifically.

01Images as tensors & transforms v2

An image is a 3-D tensor of shape (C, H, W) — channels, height, width — and a batch is 4-D, (B, C, H, W). Color images have 3 channels (RGB); grayscale, 1. Pixels arrive as 8-bit integers in [0, 255] and must become floats, typically scaled to [0, 1] then normalized. PyTorch's convention puts channels first — a frequent surprise for anyone used to libraries that put channels last.

import torch
from torchvision.io import read_image
from torchvision.transforms import v2

img = read_image("cat.jpg")      # uint8 tensor, shape (3, H, W), values 0-255
print(img.shape, img.dtype)      # torch.Size([3, H, W]) torch.uint8

tf = v2.Compose([
    v2.Resize((224, 224)),
    v2.ToDtype(torch.float32, scale=True),      # uint8 [0,255] -> float [0,1]
    v2.Normalize(mean=[0.485, 0.456, 0.406],    # ImageNet channel statistics
                 std=[0.229, 0.224, 0.225]),
])
x = tf(img).unsqueeze(0)         # (1, 3, 224, 224), ready for a model

Use the modern torchvision.transforms.v2 API (not the older v1): it's faster, works on batches and on GPU, and — crucially for Part II — transforms images and their bounding boxes/masks together, so a random flip flips the labels too. For loading whole datasets, torchvision.datasets.ImageFolder reads a directory of class-named subfolders into a labeled dataset in one line, and the built-in datasets (CIFAR, ImageNet) plug straight into a DataLoader.

Channel order and normalization must match the model

A pretrained model expects RGB (not BGR), channels-first, and the exact normalization it was trained with. Feed BGR, or skip normalization, and its features misfire silently — no error, just poor accuracy. When using pretrained weights, take the transforms from the weights object (Chapter 04) rather than hand-writing them.

02Augmentation: Mixup, CutMix, RandAug

Volume II covered basic augmentation (flips, crops, color jitter). Vision classification benefits from stronger, more creative schemes that have become standard in state-of-the-art training recipes:

TechniqueWhat it does
RandAugmentApplies a random sequence of image operations with a single strength knob — automates augmentation policy so you don't tune each transform.
MixupBlends two images (and their labels) into a weighted average. Forces smoother decision boundaries; a strong regularizer.
CutMixPastes a rectangular patch from one image onto another, mixing labels in proportion to area. Combines mixing with spatial locality.
RandomErasingBlanks out a random rectangle, forcing the model not to rely on any single region — cheap occlusion robustness.
 
from torchvision.transforms import v2

# Mixup/CutMix operate on a BATCH (they mix samples together), so apply them
# in the training loop or a collate_fn, not in per-image transforms.
mix = v2.RandomChoice([v2.MixUp(num_classes=1000),
                       v2.CutMix(num_classes=1000)])

for images, labels in loader:
    images, labels = mix(images, labels)   # labels become soft (blended) targets
    # train as usual; use a loss that accepts soft targets (e.g. cross-entropy
    # with class probabilities). These techniques often add a point or two of
    # accuracy on top of a strong baseline.
Strong augmentation buys generalization — and needs longer training

Mixup, CutMix, and RandAugment make the task harder each epoch, which curbs overfitting and lifts final accuracy — but the model needs more epochs to fit the harder targets. They're standard for training vision models from scratch; for small-data fine-tuning, dial them down or off, since a pretrained model needs less regularization.

03Modern CNNs: ResNet to ConvNeXt

You'll rarely design a backbone; you'll pick one. The lineage from Volume III's ResNet continued to refine accuracy-per-compute:

  • ResNet — the reliable baseline. Well-understood, widely pretrained, still an excellent default (ResNet-50 is the field's reference point).
  • EfficientNet — introduced compound scaling: grow depth, width, and input resolution together in balanced proportion, reaching high accuracy at low compute. Strong when efficiency matters.
  • RegNet — design-space-searched networks with clean accuracy/compute trade-offs and good scaling behavior.
  • ConvNeXt — a pure CNN modernized with ideas borrowed from transformers (larger kernels, LayerNorm, fewer activations), matching vision transformers while keeping convolutional efficiency.

All of these live in torchvision.models with pretrained weights, swappable in a line. The practical guidance: start with a pretrained ResNet-50 or ConvNeXt-Tiny, get a baseline, and only shop for a fancier backbone if profiling says the backbone (not your data or head) is the limit.

from torchvision import models

# Any of these is one line, pretrained, ready to fine-tune:
resnet   = models.resnet50(weights=models.ResNet50_Weights.DEFAULT)
convnext = models.convnext_tiny(weights=models.ConvNeXt_Tiny_Weights.DEFAULT)
effnet   = models.efficientnet_v2_s(weights="DEFAULT")

04Transfer learning for vision

Volume III made the case; vision is where transfer learning is most reliably transformative, because low-level visual features (edges, textures) are near-universal. The recipe, concretely:

import torch
import torch.nn as nn
from torchvision import models

weights = models.ResNet50_Weights.DEFAULT
model = models.resnet50(weights=weights)
preprocess = weights.transforms()      # the EXACT transforms the model expects

# Replace the 1000-class head with one for our task:
model.fc = nn.Linear(model.fc.in_features, num_classes)

# Feature-extraction phase: freeze the backbone, train only the head.
for name, p in model.named_parameters():
    p.requires_grad = name.startswith("fc")
opt = torch.optim.AdamW((p for p in model.parameters() if p.requires_grad),
                        lr=1e-3)

# Fine-tuning phase (optional, if you have enough data): unfreeze and use a
# much smaller LR so the pretrained features are nudged, not wrecked.
# for p in model.parameters(): p.requires_grad = True
# opt = torch.optim.AdamW(model.parameters(), lr=1e-5)

Two vision-specific refinements. Use discriminative learning rates when fine-tuning — a smaller rate for early layers (general features you want to preserve) and a larger one for later layers and the head. And consider freezing the batch-norm statistics of a small-data fine-tune, since tiny batches give noisy stats that can hurt. But the headline is unchanged: a pretrained backbone plus a fresh head, on a few thousand images, beats training from scratch nearly every time.

In vision, almost always start pretrained. The features learned on large image corpora transfer so well that from-scratch training is the exception, justified only when your images are truly unlike natural photos (medical, satellite, microscopy — and even there, pretraining often still helps). Your first move on any vision task should be a pretrained backbone.

Part II — Beyond classification

Where things are, not just what.

Classification answers "what's in this image?" Real applications also need "where, and how many?" — detection draws boxes around objects, segmentation labels every pixel. These tasks reuse the backbones of Part I but add new heads, new losses, and new metrics. This part covers the concepts (boxes, IoU, NMS, mAP) and the models that put them to work.

05Detection foundations: boxes, IoU, NMS

Object detection finds every object in an image and draws a bounding box around each, with a class label. A box is four numbers — usually (x1, y1, x2, y2), the top-left and bottom-right corners. Two operations underpin the whole field.

IoU (Intersection over Union) measures how well two boxes overlap: the area of their intersection divided by the area of their union. It's 1 for a perfect match, 0 for no overlap, and it's how you decide whether a predicted box "hit" a ground-truth object.

INTERSECTION OVER UNION: ┌───────────┐ │ predicted │ IoU = ▓ intersection │ ┌────┼──────┐ ───────────────── │ │ ▓▓ │ │ (union of both boxes) └──────┼────┘ │ │ ground │ IoU = 1.0 perfect overlap │ truth │ IoU > 0.5 usually counts as a "hit" └───────────┘ IoU = 0 no overlap

NMS (Non-Maximum Suppression) cleans up duplicates. Detectors emit many overlapping boxes for the same object; NMS keeps the highest-confidence box and removes others that overlap it above an IoU threshold, leaving one box per object. PyTorch gives you both operations in torchvision.ops:

import torch
from torchvision.ops import box_iou, nms

boxes = torch.tensor([[10, 10, 50, 50], [12, 12, 48, 52], [200, 200, 260, 260]])
scores = torch.tensor([0.9, 0.8, 0.75])           # confidence per box

iou = box_iou(boxes, boxes)                        # pairwise overlap matrix
keep = nms(boxes.float(), scores, iou_threshold=0.5)   # indices to keep
print(keep)   # drops box 1 (overlaps box 0 heavily); keeps 0 and 2

Detectors also traditionally rely on anchors — a grid of predefined boxes of various sizes and aspect ratios — and predict offsets from these anchors rather than absolute coordinates, which makes the regression easier. Newer designs are "anchor-free" (predicting box centers and sizes directly) or set-based (DETR, next chapter), but anchors remain common and worth recognizing.

06Detectors: Faster R-CNN, YOLO, DETR

Three families, trading accuracy against speed and simplicity:

FamilyHow it works · trade-off
Faster R-CNNTwo-stage: a region-proposal network suggests candidate boxes, then a second stage classifies and refines each. Accurate, the classic workhorse; heavier and slower.
YOLO / RetinaNetOne-stage: predict boxes and classes densely in a single pass over the image. Fast, real-time capable; historically a touch less accurate, though modern YOLO closes the gap.
DETRTransformer set-prediction: treats detection as directly predicting a set of boxes, using the attention of Volume III. Elegant — no hand-designed anchors or NMS — and increasingly competitive.

You use pretrained detectors from torchvision.models.detection much like classifiers. Note the interface differences: detectors take a list of images (which may vary in size) and, in training, a list of target dicts with boxes and labels; in eval they return per-image dicts of detections.

import torch
from torchvision.models.detection import (
    fasterrcnn_resnet50_fpn, FasterRCNN_ResNet50_FPN_Weights)

weights = FasterRCNN_ResNet50_FPN_Weights.DEFAULT
model = fasterrcnn_resnet50_fpn(weights=weights).eval()

img = weights.transforms()(read_image("street.jpg"))
with torch.no_grad():
    out = model([img])[0]          # dict: 'boxes', 'labels', 'scores'
keep = out["scores"] > 0.5         # threshold low-confidence detections
print(out["boxes"][keep], out["labels"][keep])
Pick by your latency budget

Real-time application (video, robotics, edge)? Start with a one-stage detector (YOLO-family). Accuracy-first with time to spare (offline analysis, medical)? Faster R-CNN. Want a clean, anchor-free, NMS-free design and have the data to train it? DETR. As always, begin from pretrained weights and fine-tune on your objects.

07Segmentation: U-Net & Mask R-CNN

Segmentation labels pixels, not boxes. Two flavors: semantic segmentation assigns every pixel a class (road, sky, person) without separating individual objects, while instance segmentation also distinguishes each object (person 1 vs person 2) — a per-object mask.

The archetype for semantic segmentation is U-Net: an encoder that downsamples the image into features (like a CNN backbone), a decoder that upsamples back to full resolution, and — the key idea — skip connections that feed high-resolution encoder features directly across to the decoder, so fine spatial detail lost in downsampling is recovered. It's the backbone of medical imaging and, notably, resurfaces inside diffusion models (Chapter 15).

U-NET: ENCODER DOWN, DECODER UP, SKIPS ACROSS: input ─► [down] ─────────────skip────────────► [up] ─► output mask └► [down] ──────skip──────► [up] ──►┘ └► [down] ─skip─► [up] ─►┘ └──► bottleneck ──►┘ skips carry fine detail past the bottleneck, restoring sharp edges

For instance segmentation, Mask R-CNN extends Faster R-CNN with a third head that predicts a binary mask for each detected box — detection and segmentation in one model. Both are available pretrained in torchvision (torchvision.models.segmentation for semantic, the detection module for Mask R-CNN), and follow the same load-and-fine-tune pattern.

08Metrics: IoU & mAP

Detection and segmentation need their own metrics, because "accuracy" doesn't capture localization. For segmentation, the standard is mean IoU — the IoU (Chapter 05) between predicted and true regions, averaged over classes. Simple and intuitive: how much do the predicted and actual regions overlap.

For detection, the metric is mAP (mean Average Precision), and it deserves unpacking because it's reported everywhere and confuses everyone at first. A detection counts as correct if its IoU with a ground-truth box exceeds a threshold. As you sweep the confidence threshold, you trace a precision–recall curve for each class; the Average Precision is the area under that curve, and mAP averages it across classes.

MetricMeaning
mAP@0.5mAP counting a hit at IoU ≥ 0.5. Lenient localization — the classic "PASCAL VOC" metric.
mAP@[.5:.95]mAP averaged over IoU thresholds from 0.5 to 0.95. Rewards tight boxes — the stricter "COCO" metric, the modern standard.
mean IoUSegmentation: average region overlap across classes.
Report the metric your benchmark uses

A number is meaningless without its protocol. "mAP 45" could be COCO mAP@[.5:.95] (strong) or VOC mAP@0.5 (mediocre) — very different achievements. Always state the IoU threshold and dataset convention, and compare only like with like. Use an established evaluator (e.g. the COCO tools, or torchmetrics) rather than rolling your own — mAP is subtle to compute correctly.

Boxes need IoU, detection needs mAP — classification accuracy stops telling the truth the moment location matters.

Part III — Vision transformers

Attention comes for images.

The transformer that conquered language also works on pixels — cut an image into patches, treat them as tokens, and attend. Vision transformers match or beat CNNs at scale, unlock label-free pretraining, and bridge to language through models like CLIP. This part carries Volume III's attention into vision and sets up the generative finale.

09ViT from scratch

The Vision Transformer (ViT) asks: what if we treat an image like a sentence? It cuts the image into a grid of fixed-size patches (say 16×16 pixels), flattens and linearly embeds each patch into a token vector, adds positional embeddings, and feeds the sequence to a standard transformer encoder (Volume III). A special learnable "class token" collects information for the final classification. That's it — no convolutions.

import torch
import torch.nn as nn

class PatchEmbed(nn.Module):
    def __init__(self, img=224, patch=16, in_ch=3, dim=768):
        super().__init__()
        self.n_patches = (img // patch) ** 2                 # e.g. 14*14 = 196
        # a strided conv IS the patch-embed: each stride lands on one patch
        self.proj = nn.Conv2d(in_ch, dim, kernel_size=patch, stride=patch)

    def forward(self, x):                    # x: (B, 3, 224, 224)
        x = self.proj(x)                     # (B, dim, 14, 14)
        return x.flatten(2).transpose(1, 2)  # (B, 196, dim): 196 patch tokens

# Then: prepend a class token, add positional embeddings, run TransformerBlocks
# (from Volume III), and classify from the class token's final vector.
patches = PatchEmbed()(torch.randn(2, 3, 224, 224))
print(patches.shape)                          # (2, 196, 768)

The trade against CNNs is inductive bias. A convolution builds in locality and translation-invariance — strong priors that let CNNs learn from modest data. A ViT assumes almost nothing about images, so it's more flexible but data-hungry: on small datasets a CNN wins, but given enough data (or large-scale pretraining), the ViT's flexibility lets it pull ahead. This is why ViTs took off alongside huge pretraining datasets.

Use a pretrained ViT; rarely train one from scratch

Because ViTs need enormous data to shine, you almost always start from a pretrained one (torchvision and timm both offer many) and fine-tune — exactly the Part I recipe. Training a ViT from scratch on a small dataset usually underperforms a pretrained ResNet.

10Hierarchical & hybrid: Swin

Plain ViT has two limits: it works at a single resolution (no feature pyramid, awkward for detection/segmentation), and its global attention costs grow quadratically with the number of patches, which is expensive for high-resolution images. The Swin Transformer fixes both by borrowing the CNN's hierarchy. It computes attention within local windows (cheap, linear in image size), shifts those windows between layers so information crosses window boundaries, and merges patches as it deepens to build a multi-scale pyramid — the pyramid that detection and segmentation heads expect.

Meanwhile, the story went full circle: ConvNeXt (Part I) showed that a pure CNN, modernized with the training recipes and design choices of transformers, matches ViTs. So the honest state of play is a genuine choice, not a clear winner:

ConsiderationCNN (ResNet/ConvNeXt)Transformer (ViT/Swin)
Small/medium dataStronger — built-in inductive bias.Weaker unless pretrained at scale.
Very large dataExcellent (ConvNeXt).Excellent — flexibility pays off.
Dense tasks (detect/seg)Natural feature pyramid.Use hierarchical variants (Swin).
Ecosystem maturityVery mature, easy to deploy.Mature and growing fast.

The pragmatic answer: try a pretrained model of either kind, and let your data and metrics decide. The architectures have converged more than the debate suggests.

11Self-supervised: MAE & DINO

Labels are expensive; unlabeled images are nearly infinite. Self-supervised learning pretrains a backbone with no labels at all, by inventing a task the data can grade itself on — then that backbone transfers to real tasks with little labeled data. Three influential approaches:

  • Contrastive (SimCLR, MoCo). Make two augmented views of the same image, and train so their embeddings are close while different images' embeddings are far apart. The model learns what's essential about an image (invariant to crops and color) — this is the metric learning of Volume III, self-supervised.
  • Masked autoencoding (MAE). Hide most of an image's patches and train the model to reconstruct them from the few visible ones — literally BERT's fill-in-the-blank, for pixels. Simple, scalable, and produces excellent ViT features.
  • Self-distillation (DINO). A student network is trained to match a teacher (an average of the student's own past weights) across views. Remarkably, DINO's attention maps segment objects with no segmentation labels ever provided — structure emerges for free.

The payoff mirrors language: a backbone pretrained self-supervised on millions of unlabeled images gives strong general features, which you fine-tune on your small labeled set. It's the vision analogue of the pretrain-then-adapt workflow that dominates Volume V.

12Multimodal: CLIP

CLIP connects images and text in one shared space, and it's one of the most consequential vision models of recent years. It trains two encoders together — one for images, one for text — on hundreds of millions of image-caption pairs, with a contrastive objective: the embedding of an image and the embedding of its true caption should be close, and far from mismatched captions. The result is a space where a picture of a dog and the words "a photo of a dog" land near each other.

That shared space enables zero-shot classification — recognizing categories the model was never explicitly trained to classify — by comparing an image's embedding to the embeddings of text prompts:

# Conceptual zero-shot classification with a CLIP-style model:
labels = ["a photo of a cat", "a photo of a dog", "a photo of a car"]

image_emb = image_encoder(image)            # (1, D)
text_emb  = text_encoder(tokenize(labels))  # (3, D)

# cosine similarity, then softmax -> a probability per label. No training needed.
sims = (image_emb @ text_emb.T)             # (1, 3)
probs = sims.softmax(dim=-1)
print(labels[probs.argmax()])               # the best-matching caption

Beyond classification and retrieval, CLIP's text–image alignment is what lets you steer image generation with words — its text embeddings condition the diffusion models of the next part. CLIP is the hinge between the vision track and the language track, and a foundation of modern multimodal AI.

Vision and language stopped being separate. Patches-as-tokens brought the transformer to images; CLIP tied images and text into one space; and generative models take text and produce pixels. The clean split between "vision models" and "language models" has largely dissolved into a shared toolkit — which is why Volumes V and VI increasingly rhyme.

Part IV — Generative & applied vision

Making images, and shipping them.

The final turn: from recognizing images to creating them. The autoencoder of Volume III becomes a generator (VAE), then adversarial and — the modern workhorse — diffusion models. We close where every model must end: deployed, on the constraints real vision products face. This is the last part of the last volume.

13Autoencoders & VAEs

Volume III's autoencoder compressed and reconstructed images, but its latent space was a grab-bag — pick a random latent vector and the decoder usually produces garbage, because the space has holes. The Variational Autoencoder (VAE) fixes this by forcing the latent space to be smooth and continuous, which turns the decoder into a generator: sample any point from a simple distribution and get a plausible image.

Two changes do it. The encoder outputs a distribution (a mean and variance) rather than a point, and a loss term (KL divergence) pulls that distribution toward a standard normal — so the whole latent space stays densely, meaningfully populated. To backpropagate through the random sampling, the reparameterization trick expresses the sample as mean + std · ε with the randomness in ε, keeping the gradient path through mean and std.

import torch
import torch.nn.functional as F

def vae_step(encoder, decoder, x):
    mu, logvar = encoder(x).chunk(2, dim=-1)        # distribution per input
    std = torch.exp(0.5 * logvar)
    z = mu + std * torch.randn_like(std)            # reparameterization trick
    recon = decoder(z)
    recon_loss = F.mse_loss(recon, x, reduction="sum")
    # KL pulls the latent distribution toward N(0, I) -> smooth, samplable space
    kl = -0.5 * torch.sum(1 + logvar - mu.pow(2) - logvar.exp())
    return recon_loss + kl

VAEs generate somewhat blurry images on their own, but the idea is foundational — and, importantly, a VAE's compact latent space is exactly what latent diffusion (Chapter 15) generates into, for efficiency. The VAE is a building block of the modern generative stack, not just a historical stepping stone.

14GANs

Generative Adversarial Networks frame generation as a contest between two networks. A generator turns random noise into images; a discriminator tries to tell generated images from real ones. They train together: the discriminator learns to spot fakes, and the generator learns to fool it — an arms race that drives the generator toward increasingly realistic images. For years GANs produced the sharpest image synthesis (the uncannily realistic faces of the late 2010s were GANs).

# The adversarial training loop, in essence (one step):
# 1. Train D: real images should score "real", fakes should score "fake".
d_loss = bce(D(real), ones) + bce(D(G(noise).detach()), zeros)
d_loss.backward(); opt_D.step()

# 2. Train G: make fakes that D scores as "real" (note: no detach).
g_loss = bce(D(G(noise)), ones)
g_loss.backward(); opt_G.step()
GANs are famously hard to train

The adversarial balance is delicate: if either network overpowers the other, training collapses. The classic failure is mode collapse, where the generator produces only a few image types regardless of input. GANs remain useful (fast, sharp, great for some image-to-image tasks), but for general high-quality generation they've largely been overtaken by the diffusion models next — which are far more stable to train.

15Diffusion models

Diffusion models are the engine behind modern image generation, and the idea is beautifully simple. Take training images and gradually add noise, step by step, until they're pure static — the "forward process." Then train a network to reverse one step: given a noisy image, predict the noise that was added. To generate, start from pure random noise and run that denoiser repeatedly, and a coherent image emerges from the static.

DIFFUSION: LEARN TO REVERSE A NOISING PROCESS forward (fixed): image ─►+noise─►+noise─► ... ─► pure noise x0 x1 x2 xT reverse (learned): noise ─►denoise─►denoise─► ... ─► image xT ... x0 train a U-Net to predict the noise at each step; generate by denoising

The training objective is startlingly plain: add a random amount of noise to an image, ask the network to predict that noise, and take the MSE between prediction and truth. The denoiser is almost always a U-Net (Chapter 07) — its skip connections preserve the detail needed to reconstruct sharp images.

import torch
import torch.nn.functional as F

def diffusion_loss(unet, x0, scheduler):
    # x0: a clean image batch. Pick a random noise level t for each.
    t = torch.randint(0, scheduler.T, (x0.size(0),), device=x0.device)
    noise = torch.randn_like(x0)
    xt = scheduler.add_noise(x0, noise, t)      # noise x0 up to level t
    pred = unet(xt, t)                          # predict the noise that was added
    return F.mse_loss(pred, noise)              # ...and that's the whole loss

Two refinements make this practical and powerful. Conditioning steers generation — feed a text embedding (from CLIP, Chapter 12) into the U-Net and it generates images matching the prompt; classifier-free guidance amplifies that steering. Latent diffusion (the basis of Stable Diffusion) runs the whole process in a VAE's compact latent space instead of on full-resolution pixels, cutting compute enormously. Together they're how text-to-image systems work: encode the prompt, denoise in latent space guided by it, decode to pixels.

Every idea in this series converges here. A diffusion image generator uses a U-Net (Volume III/VI) as its denoiser, a VAE (this part) for its latent space, CLIP (Part III) to follow text, attention (Volume III) inside the U-Net, and all of Volume IV's efficiency and serving machinery to run. The most impressive systems in AI are compositions of the primitives you now understand individually.

16Deploying vision models

Vision models ship to demanding places — phones, cameras, cars, factory lines — so deployment (Volume IV) has vision-specific edges. The path is the same: export with torch.export, quantize, and target a runtime.

  • Real-time throughput. Video and camera feeds need frames processed within a tight budget. Batch frames when latency allows, use half precision, and consider a compiled or vendor-optimized runtime (e.g. TensorRT) for the last drop of speed.
  • Edge and mobile. On-device vision (a phone camera, an embedded board) has hard memory and power limits. Quantize aggressively (int8), pick an efficient backbone (EfficientNet, MobileNet), and export to a mobile runtime like ExecuTorch.
  • Preprocessing parity. The resize, crop, and normalization at inference must exactly match training — a mismatch here silently wrecks accuracy in production. Ship the preprocessing with the model, or bake it into the exported graph.
  • Interchange. Export to ONNX when the deployment target is another framework or a hardware-specific runtime that consumes it.

The instinct from Volume IV holds: measure the real bottleneck (often preprocessing or data movement, not the model), optimize that, and validate accuracy after every compression step. A vision model that's accurate in a notebook but too slow or too heavy for its target hasn't solved the problem it was built for.
