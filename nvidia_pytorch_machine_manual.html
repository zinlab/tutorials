Part I — CUDA semantics in PyTorch

Async, by default.

Six chapters on the execution model that everything else in this volume sits on. Almost every "PyTorch is doing something weird on my GPU" question is an async-execution question wearing a costume.

01Async execution & the current stream

THE CONTRACT: • CUDA ops are ENQUEUED on the current stream and return immediately. Python races ahead; the GPU runs behind. `y = x.matmul(w)` returns a tensor whose CONTENTS DO NOT EXIST YET — and that is fine, because every subsequent op on the same stream is ordered after it. • Per-device CURRENT STREAM (thread-local). Default stream unless you enter `with torch.cuda.stream(s):`. • ORDERING IS FREE within a stream. ACROSS streams you must synchronize explicitly (events, wait_stream) — §04. ───────────────────────────────────────────────────────────── THREE CONSEQUENCES YOU WILL BE ASKED ABOUT: 1 TIMING: `t0=time(); op(); t1=time()` measures ENQUEUE, not compute. Correct timing needs torch.cuda.synchronize() around the region, or CUDA events (elapsed_time), or the profiler. 2 ERRORS: a kernel's error surfaces at the next sync — so the Python traceback points at an innocent line downstream (§18's whole tree). 3 PERF: the host must stay AHEAD of the GPU. If Python can't enqueue faster than the GPU drains, you're launch-bound / host-bound — and any accidental sync (§03) collapses the pipeline entirely.
The one-liner

"PyTorch's CUDA API is asynchronous: ops enqueue on a stream and return, so correctness comes from stream ordering, timing requires an explicit sync or events, and error reporting is deferred to the next synchronization point — which is why the traceback usually lies."

02The caching allocator, operationally

Vol. I §05–13 has the deep structure; this is the operator's view — the numbers you read and the levers you pull.

ConceptWhat it means when you're staring at an OOM
Why cache at allcudaMalloc/cudaFree are synchronizing and slow. PyTorch grabs big segments and sub-allocates blocks from them — so allocation is a pointer bump in steady state. This is what makes eager training viable at all.
memory_allocated vs memory_reservedThe single most diagnostic pair of numbers. Allocated = what your tensors hold. Reserved = what PyTorch took from the driver. reserved ≫ allocated ⇒ fragmentation, not a leak. allocated climbing ⇒ you're holding references (§20).
FragmentationFree memory exists but not contiguously in the right pool → OOM with gigabytes "free". Caused by variable shapes, alternating big/small allocations, long-lived blocks pinning segments.
expandable_segments:TrueThe modern fix (via PYTORCH_CUDA_ALLOC_CONF): virtual-memory-backed segments that grow, so blocks don't have to fit a pre-sized segment. Reach for it before max_split_size_mb tuning.
empty_cache()Returns free cached segments to the driver. It does not defragment, does not free live tensors, and costs a sync — useful between phases (train→eval, or before handing the GPU to another library), useless as an OOM cure.
Stream-awarenessBlocks are associated with the stream they were allocated on. Reusing memory across streams without telling the allocator is a real corruption bug — §04.
Your unfair advantage

You've gone deeper on this allocator (MemPool, private pools, capture interaction) than most candidates ever will. When OOM or CUDA graphs comes up in any hour, this is the topic to steer toward and go deep — it's core-team-relevant, it's Piotr's own team's code, and it's yours.

03Sync points: the complete hunt

An accidental sync in a training loop is the classic silent 30% — the GPU drains, the host waits, and the pipeline restarts from empty. Know the list, and know the tool that finds them for you.

Sync sourceNotes
.item(), float(t), int(t), .tolist()The big one. Any host-side read of a device value must wait for the value to exist. total_loss += loss.item() every step is a per-step sync — usually acceptable, but not in a tight profiling loop.
print(t), t.cpu(), t.numpy()Same mechanism. A stray debug print in an inner loop is a stall.
Python control flow on a tensorif loss > threshold:, while not converged:, assert t.isfinite().all() — each forces the value to the host. Also the reason such code breaks CUDA-graph capture (§06) and causes graph breaks under torch.compile.
Data-dependent shapestorch.nonzero, torch.unique, boolean-mask indexing t[mask], masked_select — the output shape depends on device data, so the host must learn it. Unavoidable syncs by construction; the fix is algorithmic (fixed-size buffers, padding).
D2H copies without pinned memorynon_blocking=True only actually overlaps from pinned host memory; otherwise it's synchronous regardless of the flag (§05).
torch.cuda.synchronize()Explicit. Correct for timing; poison in a hot loop.
The tool most people don't know exists — use it in the interview
torch.cuda.set_sync_debug_mode("warn")
PyTorch tells you every time an op synchronizes. "error" makes it raise, so you can hunt every accidental sync in a training step to zero. Mentioning this in a "why is my training slow" answer is a strong signal — it's the diagnostic that converts guessing into a list.
torch.profiler + trace view
Gaps on the GPU timeline with the CPU busy = host-bound; gaps with the CPU idle = you're waiting on a sync.

04Multi-stream correctness

THE BUG EVERYONE WRITES ONCE: s = torch.cuda.Stream() x = torch.randn(N, device='cuda') # allocated on the DEFAULT stream with torch.cuda.stream(s): y = big_op(x) # runs on s — may start before the # default stream finished producing x! del x # allocator may hand x's block to # someone else on the default stream # WHILE s is still reading it ───────────────────────────────────────────────────────────── TWO SEPARATE HAZARDS, TWO SEPARATE FIXES: 1 ORDERING s must wait for x's producer: s.wait_stream(torch.cuda.current_stream()) (or record an event on the producer and s.wait_event(e)) 2 LIFETIME the ALLOCATOR must know x is still in use on s: x.record_stream(s) otherwise freeing x on the default stream lets its memory be reused while s's kernels still read it → silent corruption (the worst kind of bug: no error, wrong numbers) ───────────────────────────────────────────────────────────── Also: the backward pass runs each op on the stream its FORWARD ran on (Vol. V §17) — so custom-stream forwards propagate their complexity into backward, which is where the genuinely exotic bugs live.

05Pinned memory & the input pipeline

MechanismWhy it matters
Pageable vs pinned host memoryThe DMA engine can only copy asynchronously from page-locked memory. From pageable memory the driver must stage through an internal pinned buffer — which serializes. So non_blocking=True on a pageable tensor is a no-op promise.
DataLoader(pin_memory=True) + .to(device, non_blocking=True)The pair. Together they let the H2D copy of batch N+1 overlap with the compute of batch N. One without the other buys nothing.
Cost of pinningPage-locked memory is a scarce OS resource; pinning everything can degrade the whole machine. It's a knob, not a religion.
Where this shows up in triage"GPU utilization is 40% and the CPU is busy" → dataloader-bound: workers, pinning, prefetch, decode cost, or a sync in the loop (§03). It's Tree 2's first branch (§16).

06CUDA graphs & the allocator contract

PieceMechanism
What capture recordsA DAG of kernel launches on a stream, replayable with one call — collapsing thousands of launches into one submission. The payoff is exactly proportional to how launch-bound you were (Vol. III §20's world).
The allocator contractReplay reuses the recorded addresses, so every allocation made during capture must remain valid and stable → capture runs against a private memory pool in the caching allocator. This is the MemPool machinery you know deeply; it's the load-bearing integration between graphs and memory.
What breaks captureHost syncs and host-side control flow (§03), CPU-dependent values, dynamic shapes, unsupported ops, and non-capture-safe RNG (handled via graph-safe generator state). This is precisely why capturable=True optimizers keep step counters and even the learning rate as device tensors (Vol. III §21).
Practical useStatic shapes, warmup on a side stream, static input/output buffers you copy into and out of; make_graphed_callables for partial capture; torch.compile(mode="reduce-overhead") applies it for you.
Part II — Numerics

Different, or wrong?

The question this team answers constantly — about a failing test, a user's bug report, a new arch, a library bump. Four chapters to make you fast and correct at it.

07The precision knobs

KnobWhat it actually changes
TF32Ampere+ tensor cores run fp32 matmul/conv with ~10-bit mantissa internally (fp32 range, reduced precision), for a large speedup. Historically torch.backends.cuda.matmul.allow_tf32 (default off for matmul) and torch.backends.cudnn.allow_tf32 (default on for conv) — with torch.set_float32_matmul_precision("high") as the friendlier switch. The API is consolidating toward unified fp32-precision settings, so state the mechanism, and check the current spelling before asserting a default.
Reduced-precision reductionstorch.backends.cuda.matmul.allow_fp16_reduced_precision_reduction (and the bf16 twin), default on: lets fp16 GEMMs accumulate partial sums in fp16 within split-k. Turning it off is a real diagnostic step when an fp16 matmul's error looks too large.
autocast / AMPPer-op casting policy at the dispatch layer above autograd (Vol. V §06); GradScaler for fp16's underflow range; bf16 usually needs no scaler.
Accumulate types in kernelsNot user-visible, but decisive: ATen accumulates fp16 reductions in fp32 by design (Vol. V §08, §20). This is why a GPU sum can be more accurate than a naive fp32 CPU loop.
FMA contractionThe compiler fuses a*b+c into one instruction with a single rounding — mathematically better, bitwise different from separate ops. A perpetual source of "the last digit changed."

08Why two GPUs disagree

SAME CODE, SAME INPUTS, TWO ARCHS, DIFFERENT LAST BITS. WHY: ├─ different REDUCTION TREE (occupancy/geometry differs per arch, │ so the summation order differs → fp is non-associative) ├─ different LIBRARY ALGORITHM chosen by heuristics (cuBLAS/cuDNN │ pick per arch, per shape, per version — §11–12) ├─ TF32 / tensor-core paths available on one arch, not another ├─ different SASS from a different toolkit (FMA contraction choices) └─ atomics-based kernels: order varies even RUN TO RUN (§19) ───────────────────────────────────────────────────────────── NONE of these is a bug. ALL of them can trip a tolerance-tuned test. Which is why "this test only fails on sm_XX" is a WEEKLY event on this team — and why the ability to adjudicate it (§10) is the skill they are actually hiring for.

09Determinism, and how to force it

The determinism checklist (and what each line buys)
torch.use_deterministic_algorithms(True)
The killer move. Nondeterministic ops now raise with the op's name — converting "my run isn't reproducible" from a mystery into a list. Use warn_only=True to survey without failing.
torch.manual_seed(s) (+ dataloader generator, worker_init_fn)
Seeds are necessary but never sufficient — most people stop here and stay confused.
torch.backends.cudnn.benchmark = False
Autotuning picks algorithms by timing → the choice itself varies. Turn it off for reproducibility (and pay in speed).
torch.backends.cudnn.deterministic = True
Restricts cuDNN to deterministic algorithms.
CUBLAS_WORKSPACE_CONFIG=:4096:8
Required for deterministic cuBLAS on modern CUDA — a genuinely obscure line that signals real experience.
DistributedSampler.set_epoch(e)
Data order is part of determinism too; in distributed runs it's a per-epoch shuffle seed everyone must agree on.
Inherently nondeterministicWhy
scatter_add/index_add with duplicate indices; embedding backward; index_select backward; bincount; some interpolate backwardsThey use atomicAdd: correct sum, arbitrary order, so the floating-point rounding differs per run. This family is the #1 cause of "training won't reproduce."
Some cuDNN conv algorithmsAtomics/split-k in the backward filter/data passes.
NCCL reductionsCross-rank summation order isn't guaranteed across runs/topologies.

10Wrong vs differently-right

The adjudication procedure — this is the single most valuable page in the volume, because it's the judgment call the whole team makes weekly. (1) Get a trustworthy reference: compute the same math in float64, on CPU. Not another GPU run — that's comparing two suspects. (2) Compute both candidates' error against the reference, not against each other. (3) Ask whether the error is within what the format and the reduction length can produce: fp16 with ~10-bit mantissa over a 10⁶-element sum has a predictable error budget; TF32 matmul has a known mantissa; an fp32 tree-sum differs from a sequential sum by O(√n·ε) at worst. (4) If both candidates sit within the budget on either side of the reference, they are differently right — the fix is the test's tolerance, and the justification is the number you just computed. (5) If one candidate is outside the budget, diverges with scale, produces NaN/Inf, or drifts in a systematic direction, it is wrong — and now you have a minimal repro, an arch, a dtype, and a library version to hand to the kernel owner. That last handoff is what makes you a colleague on this team rather than a ticket-filer.

The anti-pattern to name out loud

Loosening a tolerance until the test goes green, with no error budget computed, is how a real numerical regression gets buried — and it's exactly the failure a CI-owning engineer is positioned to prevent. Say that you'd never widen a tolerance without a number behind it, and you've articulated the ethic this team runs on (Vol. III §17's "disable is triage, not disposal," one layer deeper).

Part III — The backend libraries

Where the algorithms get chosen.

PyTorch doesn't compute a convolution — cuDNN does, having picked one of many algorithms by heuristic. That indirection is where performance, numerics, and flakiness all originate, and it is Eddie Yan's daily surface.

11cuDNN: benchmark & algo selection

MechanismConsequence
Algorithm zooImplicit GEMM, FFT, Winograd, direct… each with different speed, memory (workspace), and numerical characteristics. cuDNN picks per (shape, dtype, arch, layout) via heuristics.
cudnn.benchmark = TrueEmpirically times candidate algorithms on the first occurrence of each input shape and caches the winner. Great for fixed shapes; pathological when shapes vary every batch (it re-benchmarks constantly — a classic "why is my model slow with variable-length inputs" answer). Also makes algo choice timing-dependent ⇒ nondeterministic.
WorkspaceFaster algorithms often need scratch memory. Under memory pressure, cuDNN falls back to a slower, smaller-workspace algorithm — so an OOM elsewhere in your model can silently make your convs slower. Beautiful, non-obvious, and exactly the kind of cross-layer effect this team lives on.
LayoutTensor-core conv paths want NHWC (channels_last, Vol. V §02). Feed NCHW and the library transposes around the call.
Version sensitivityA cuDNN bump can change heuristics ⇒ different algorithm ⇒ different perf and different last bits ⇒ arch-scoped test failures. This is the fire drill in Vol. III §25 and the tree in §22.

12cuBLAS & the matmul path

PieceWhat to know
cuBLAS / cuBLASLtThe GEMM engines. Lt exposes richer epilogue fusion (bias, activation) and heuristic algorithm selection; PyTorch routes matmuls/linear through them with per-shape heuristics.
Split-kFor skinny GEMMs, the k-dimension is split across blocks and partial results summed — which introduces an extra reduction, hence allow_fp16_reduced_precision_reduction (§07) and a source of numerics variation across shapes.
Workspace & determinismCUBLAS_WORKSPACE_CONFIG matters for both performance and (in deterministic mode) reproducibility (§09).
Alignment & shapeTensor-core paths want aligned pointers and friendly dimensions (multiples of 8/16 depending on dtype). An unlucky feature dimension can silently cost you the tensor-core kernel — a real, checkable answer to "why is this layer slower than the FLOPs suggest?"

13SDPA: the backend zoo

Scaled-dot-product attention is one op with several completely different implementations, chosen at runtime. It is, for that reason, the single flakiest and most support-heavy surface in modern PyTorch — and it is where Eddie Yan's bug from the intro landed.

torch.nn.functional.scaled_dot_product_attention(q, k, v, ...) selects among: ├─ FLASH ATTENTION fastest; fp16/bf16; head-dim & alignment │ limits; causal/no-mask; sm80+ ├─ MEMORY-EFFICIENT broader support, still no S×S materialization ├─ cuDNN ATTENTION NVIDIA's library path (newer) └─ MATH (fallback) composite ops; MATERIALIZES the S×S score matrix ⇒ O(S²) memory ⇒ the OOM-and-slow cliff nobody notices until profiling ───────────────────────────────────────────────────────────── SELECTION depends on: dtype, head dim, mask type (arbitrary attn_mask often disqualifies flash), dropout, alignment, arch, train vs eval. Force/inspect with torch.nn.attention.sdpa_kernel(...) — and in triage, ALWAYS check which backend actually ran before blaming the model. ───────────────────────────────────────────────────────────── THE INTERVIEW-GRADE STORY: pytorch#165801 — the CI docker script installed cuDNN 9.8 while the shipped wheels carried 9.10.2, so the cuDNN attention backend was NOT selected as expected. A VERSION SKEW BETWEEN CI AND WHEELS silently changed which KERNEL runs for users. Filed by Eddie Yan — Technical Lead of PyTorch at NVIDIA, and one of Piotr's directs. This one issue is the entire thesis of this loop: the core internals and the delivery pipeline are the same job.

14NCCL touchpoints

SurfaceWhat a life-cycle engineer must know

Part IV — The decision trees

Eight scripts.

Each tree is a diagnostic you can execute out loud. They share one grammar — classify before you dig — because the senior failure mode in every one of these is debugging the wrong class. Learn the first branch of each; the rest follows.

15Tree 1 · CUDA out of memory

"CUDA out of memory. Tried to allocate 2.00 GiB…"
   │
   ├─ FIRST: read the numbers in the message + memory_summary()
   │     allocated vs reserved (§02) — this ONE comparison routes you:
   │
   ├─ reserved ≫ allocated (GBs "free" but the alloc fails)
   │     ⇒ FRAGMENTATION, not capacity
   │     → PYTORCH_CUDA_ALLOC_CONF=expandable_segments:True   (first move)
   │     → stabilize shapes (variable seq-len is the usual cause);
   │       bucket/pad; avoid alternating huge/tiny allocations
   │     → max_split_size_mb as a fallback knob
   │     → empty_cache() is NOT a fix (§02) — say so
   │
   ├─ allocated ≈ capacity (you genuinely need the memory)
   │     ⇒ CAPACITY. In cost order:
   │     → autocast/bf16 (activations halve)
   │     → activation checkpointing (Vol. V §16: recompute vs store)
   │     → micro-batching + grad accumulation (§Vol. V §11: it's free)
   │     → fused optimizer (no foreach intermediates, Vol. III §21)
   │     → FSDP/sharding if the PARAMS+OPTIM STATE are the problem
   │       (know the split: params/grads/optim vs activations —
   │        naming which one dominates IS the diagnosis)
   │
   ├─ allocated CLIMBS every iteration  ⇒ that's Tree 6 (§20), not this one
   │
   ├─ a single op spikes it
   │     ⇒ materialization: SDPA fell back to MATH (O(S²) scores! §13),
   │       a cat/stack of everything, a foreach temporary, an
   │       unexpected fp32 upcast (Vol. V §03), a cuDNN workspace
   │     → memory snapshot (§23) shows the allocating stack directly
   │
   └─ it only OOMs on ONE GPU/rank
         ⇒ imbalance (uneven shapes per rank), another tenant on the
           device, or rank 0 doing extra work (logging/eval/checkpoint)

16Tree 2 · It's slow

"Training is slower than it should be."
   │
   ├─ FIRST QUESTION: is the GPU actually busy?  (profiler timeline;
   │     nvidia-smi is a crude proxy — the timeline is the truth)
   │
   ├─ GPU IDLE, CPU BUSY  ⇒ host-bound
   │     ├ dataloader: too few workers, no pin_memory, heavy decode,
   │     │   no prefetch → §05
   │     ├ python/launch overhead: thousands of tiny kernels
   │     │   → foreach/fused optimizer · torch.compile · CUDA graphs
   │     │     (Vol. III §20-21, §06) — the launch-overhead war
   │     └ accidental SYNCS: run torch.cuda.set_sync_debug_mode("warn")
   │         and hunt them to zero (§03) ← the move most people miss
   │
   ├─ GPU IDLE, CPU IDLE  ⇒ you're waiting on something
   │     ⇒ a sync, an H2D copy that isn't overlapped, or (distributed)
   │       a collective waiting on a straggler rank
   │
   ├─ GPU BUSY but throughput is below the roofline  ⇒ kernel-level
   │     ├ not on tensor cores: fp32 without TF32 (§07); bad
   │     │   alignment/dims (§12); wrong dtype
   │     ├ layout thrash: NCHW vs channels_last transposes (§11)
   │     ├ SDPA fell back to MATH (§13) — CHECK WHICH BACKEND RAN
   │     ├ cudnn.benchmark off, or ON with varying shapes (§11)
   │     └ hidden copies: dtype promotion inserting a full extra pass
   │         (Vol. V §03/§18) — visible as stray copy_ kernels
   │
   └─ DISTRIBUTED  ⇒ comm-bound
         ├ allreduce not overlapping (bucket size; find_unused_parameters
         │   forcing a graph traversal every step) — Vol. V §25
         ├ stragglers: one slow rank sets the pace (profile ALL ranks)
         └ topology: NVLink vs PCIe; NCCL_DEBUG=INFO shows the rings
The framing that makes this answer senior

"Slow" is never one question — it's four (host-bound, sync-bound, kernel-bound, comm-bound), and the profiler timeline distinguishes them in about ninety seconds. Open with that, then walk the branch they care about. Volunteering set_sync_debug_mode and "check which SDPA backend actually ran" marks you as someone who has done this for real.

17Tree 3 · Numerics mismatch

"Results differ — CPU vs GPU, or across GPUs, or after an upgrade."
   │
   ├─ MAKE IT DETERMINISTIC FIRST (§09) — otherwise you're chasing noise
   │     seeds + use_deterministic_algorithms(True) + cudnn.benchmark=False
   │
   ├─ BUILD A TRUSTWORTHY REFERENCE: float64, on CPU.
   │     NEVER adjudicate by comparing two suspects to each other (§10)
   │
   ├─ COMPUTE THE ERROR BUDGET: what CAN this dtype, this reduction
   │     length, this algorithm produce? (§07-08) Now you have a number.
   │
   ├─ WITHIN BUDGET, both sides of the reference ⇒ DIFFERENTLY RIGHT
   │     → fix the TEST's tolerance, with the budget as justification
   │     → document the arch-delta. Never widen a tolerance blindly (§10)
   │
   ├─ OUTSIDE BUDGET / diverges with scale / NaN / systematic drift
   │     ⇒ WRONG. Now localize:
   │     ├ toggle suspects ONE at a time: TF32 off · fp16 reduced-precision
   │     │   reduction off · force SDPA math backend · autocast off ·
   │     │   cudnn.benchmark off
   │     │     → whichever toggle fixes it NAMES THE LAYER
   │     ├ bisect the MODEL: hook every module, compare activations to the
   │     │   fp64 reference layer by layer → the first layer that exceeds
   │     │   budget is your op
   │     └ bisect the VERSION (Vol. IV P8) if it changed after an upgrade
   │         → then §22's tree: WHICH layer moved (torch/cuDNN/driver)?
   │
   └─ HAND IT OFF WELL: minimal repro + arch + dtype + shapes + library
         versions + the error-vs-budget number. On this team, that hand-off
         goes to a colleague down the hall — make it a gift, not a ticket.

18Tree 4 · Device-side assert / illegal memory access

"RuntimeError: CUDA error: device-side assert triggered"
   │
   ├─ THE TRACEBACK IS LYING — launches are async (§01), so the error
   │     surfaced at the next sync, not at the faulting kernel.
   │     ⇒ RERUN with CUDA_LAUNCH_BLOCKING=1 → now the stack points at
   │       the real op. (Also: only the FIRST error is meaningful; the
   │       context is corrupted afterward, so everything downstream is
   │       noise. Say this — it's the ptrblck reflex.)
   │
   ├─ READ THE ASSERT TEXT — it names the mechanism:
   │     "Assertion `srcIndex < srcSelectDimSize` failed"
   │        ⇒ INDEX OUT OF RANGE in a gather/embedding/index op
   │     "Assertion `t >= 0 && t < n_classes` failed"
   │        ⇒ a LABEL outside [0, C) — nll_loss/cross_entropy
   │
   ├─ RANK THE CLASSIC CAUSES
   │     1 labels 1-indexed, or a sentinel (255 / -1) reaching the loss
   │     2 embedding index ≥ num_embeddings (vocab/tokenizer mismatch)
   │     3 class-count mismatch (model head vs dataset)
   │     4 a genuinely bad index computed upstream
   │
   ├─ CONFIRM ON CPU — the same op raises a readable, synchronous
   │     Python error with the offending value. Cheapest possible proof.
   │
   └─ NOT AN INDEX BUG? ⇒ "illegal memory access" territory
         → compute-sanitizer (memcheck) finds the offending kernel
         → suspect custom kernels/extensions, out-of-bounds writes,
           multi-stream lifetime bugs (§04 — freed memory still in use)
         → TORCH_USE_CUDA_DSA for device-side assertions in builds that
           enable them; TORCH_SHOW_CPP_STACKTRACES=1 for the C++ frame

19Tree 5 · Nondeterminism

"Two identical runs give different numbers."
   │
   ├─ Is the INPUT identical? seeds (python/numpy/torch/cuda), dataloader
   │     worker seeding, shuffle order, DistributedSampler.set_epoch,
   │     augmentation RNG. Most "nondeterminism" dies here.
   │
   ├─ Still differs? ⇒ turn on the oracle
   │     torch.use_deterministic_algorithms(True)
   │       → it RAISES AND NAMES the nondeterministic op (§09).
   │         Mystery → list, in one line. This is the whole trick.
   │
   ├─ The named op is atomics-based (scatter_add/index_add/embedding
   │     backward) ⇒ inherent: order of atomic accumulation varies.
   │     → use the deterministic variant, restructure (sort+segment-reduce),
   │       or accept and document
   │
   ├─ Nothing raises, but it still differs
   │     ⇒ cudnn.benchmark (timing-based algo choice) · TF32/precision
   │       flags differing between runs · a different GPU/arch in a
   │       multi-node pool (§08) · NCCL reduction order
   │
   └─ Ask the meta-question: do you need BITWISE determinism, or
         STATISTICAL reproducibility? The first costs real speed; the
         second usually just needs seeds + a fixed data order. Knowing
         which one the user actually needs is the senior move — because
         "make it deterministic" is often the wrong request.

20Tree 6 · Memory grows

"Memory climbs every iteration / every epoch until it OOMs."
   │
   ├─ WHICH number is climbing? (§02)
   │     allocated climbing  ⇒ you are HOLDING REFERENCES (a real leak)
   │     reserved climbing, allocated flat ⇒ FRAGMENTATION (Tree 1)
   │
   ├─ ALLOCATED climbing — the reference-leak suspects, in order
   │     1 accumulating a TENSOR instead of a float:
   │         total_loss += loss        ← keeps the whole autograd graph
   │         total_loss += loss.item() ← correct  (Vol. IV §13, Vol. V §10)
   │     2 stashing outputs/activations in a python list for logging
   │         → .detach() (and usually .cpu()) before you store
   │     3 retain_graph=True used to silence an error (Vol. V §13) —
   │         it doesn't fix the bug, it just makes it expensive
   │     4 hooks / closures capturing tensors; a module holding its own
   │         last output as an attribute
   │     5 an EMA/buffer updated with = instead of copy_/detach
   │     6 caching keyed by shape that never evicts (Vol. IV P7's LRU
   │         with no capacity)
   │
   ├─ PROVE IT rather than guess
   │     torch.cuda.memory._record_memory_history() → _dump_snapshot()
   │     → open in the memory visualizer → the surviving allocations show
   │       their ALLOCATION STACK. The leak names itself. (§23)
   │
   └─ Growth only across EPOCHS, not iterations?
         ⇒ per-epoch objects: a new dataloader/iterator each epoch holding
           workers, a validation pass without no_grad (Vol. V §15), a list
           of per-epoch metrics holding CUDA tensors.

21Tree 7 · Distributed hang

"The job hangs. No error, no progress, GPUs at 100% or 0%."
   │
   ├─ FIRST: WHO is stuck, and WHERE?
   │     py-spy dump --pid <each rank>   → python stacks for every rank
   │     → the rank whose stack differs is usually the culprit
   │     → NCCL flight recorder (a per-rank ring buffer of collectives,
   │       dumpable on timeout) shows WHICH collective each rank is in —
   │       a mismatch is immediately visible
   │
   ├─ THE UNIVERSAL CAUSE: ranks disagree about the collective sequence
   │     ⇒ every rank must issue the SAME collectives, in the SAME ORDER,
   │       with the SAME shapes. Any divergence = permanent wait.
   │     ├ DDP: a parameter that gets NO GRADIENT on some ranks (unused
   │     │   head, conditional branch) ⇒ its bucket never completes ⇒ that
   │     │   rank never launches its allreduce (Vol. V §25).
   │     │   → find_unused_parameters=True (costly) or fix the model
   │     ├ control flow that differs per rank (`if rank == 0: extra_step()`,
   │     │   an early `continue`, a rank-dependent branch)
   │     ├ UNEVEN DATA: one rank runs out of batches and exits the loop
   │     │   while others wait at the next allreduce (→ join/drop_last)
   │     ├ a rank CRASHED or was OOM-killed; the survivors block forever
   │     └ mixing a CPU barrier (gloo) with GPU collectives inconsistently
   │
   ├─ CONFIRM the class: TORCH_DISTRIBUTED_DEBUG=DETAIL names mismatched
   │     params/shapes; NCCL_DEBUG=INFO shows ring setup and where it stalls
   │
   └─ HARDEN: collective timeouts + async error handling so a hang
         becomes a LOUD, ATTRIBUTED FAILURE instead of a silent stall —
         which is the difference between a 10-minute debug and a lost night
         of cluster time. (This is a great thing to say in Wei's hour: hangs
         are a CI/infra problem as much as a code problem.)

22Tree 8 · Regression after an upgrade

The life-cycle tree — the one this role owns. Something changed; the stack has six layers; the user says "it broke."

"It worked last week. New torch (or driver, or image) — now it's broken/slow/different."
   │
   ├─ ENUMERATE what can move (Vol. III §01's vertical, as a checklist)
   │     model code · data · PyTorch · the nvidia-* dep wheels bundled with
   │     it (cuDNN/cuBLAS/NCCL!) · CUDA runtime · DRIVER · the GPU itself ·
   │     the container image
   │     ⇒ collect_env on BOTH the good and bad environments, diff them.
   │       Half of all "PyTorch regressions" die at this diff, because the
   │       thing that moved wasn't PyTorch.
   │
   ├─ CLASSIFY the symptom
   │     ├ CRASH / import error / undefined symbol ⇒ Vol. III §12 (linking),
   │     │   not this tree
   │     ├ SLOWER ⇒ an algorithm choice changed: cuDNN/cuBLAS heuristics
   │     │   picked differently (§11-12), TF32 default changed, an SDPA
   │     │   backend became unavailable and fell back to MATH (§13),
   │     │   cudnn.benchmark interacting with new shapes
   │     └ DIFFERENT NUMBERS ⇒ Tree 3 (§17) — with a version axis
   │
   ├─ CHANGE ONE VARIABLE AT A TIME — the whole discipline:
   │     new torch + OLD cuDNN wheel? old torch + NEW driver? same torch,
   │     old image? Each pin isolates one layer. The dependency web of
   │     Vol. III §05 is exactly the list of variables to pin.
   │
   ├─ BISECT once the layer is known: nightlies (Vol. IV P8's
   │     first_bad), or the dep wheel versions, or the image digests.
   │     Cost model matters: each check is a build+test, so log₂ is hours.
   │
   └─ CLOSE THE LOOP LIKE THIS TEAM DOES
         → if it's OURS: fix + a regression test that would have caught it
         → if it's a LIBRARY's: minimal repro to the cuDNN/cuBLAS team —
           internally, down the hall (the vendor advantage)
         → if it's a SKEW between CI and the shipped wheels: that's
           pytorch#165801's class — fix the pin AND add the check that
           makes the skew impossible. THIS is the life-cycle engineer's
           signature move, and the panel will recognize it instantly.

Part VI — Synthesis

Two days out.

25Three fire drills

DRILL"Users report the 2.x wheels got 30% slower on attention. Nothing in the model changed."
Classify

Regression after upgrade (Tree 8) with a perf symptom ⇒ an algorithm-selection change, not a code change. Attention ⇒ SDPA backend selection is the prime suspect (§13).

Diagnose

Check which backend actually runs in both versions (force each with the sdpa_kernel context and compare; profile to see whether an O(S²) score matrix is being materialized — the math-fallback signature). Then ask why selection changed: a dtype/head-dim/mask condition that newly disqualifies flash, an arch condition, or — the pytorch#165801 class — a version skew between what CI validated and what the wheel actually ships, so the intended cuDNN/flash path isn't available at runtime. Confirm by diffing collect_env and the bundled nvidia-* wheel versions between the two installs.

Fix & harden

Restore the intended backend (fix the pin/condition), then add the check that makes the class impossible: a CI assertion that the selected backend for representative shapes is the expected one — a behavioral test, not a version test. That's the difference between fixing a bug and closing a hole.

DRILL"One CUDA test fails only on one arch, only in fp16, off by 3e-3."
Classify

Tree 3, arch-scoped. The magnitude smells like a reduction/accumulation difference, not corruption — but "smells like" is not a verdict.

Diagnose

Determinize; build the float64 CPU reference; compute both sides' error against it. Establish the budget: fp16 mantissa × reduction length × whether split-k / reduced-precision reduction is in play (§07). Toggle: TF32 off, allow_fp16_reduced_precision_reduction=False, force a deterministic algorithm — whichever toggle moves the number names the mechanism. Check whether the failing arch selects a different library algorithm (§11–12) or a different launch geometry (Vol. V §20).

Adjudicate & act

Within budget on both sides ⇒ differently-right: fix the tolerance with the budget written into the test comment, and note the arch-delta. Outside budget or systematic ⇒ wrong: minimal repro + arch + dtype + shapes + library version, handed to the kernel owner. Either way the paper trail is the deliverable — and either way you just did the job this team does every week.

DRILL"A distributed CI job hangs, but only on the aarch64 runners, and only sometimes."
Classify

Two trees at once: hang (Tree 7) plus platform-scoped flakiness (Vol. III §17, §19). "Sometimes" + "one platform" is the signature of a race that different timing exposes — or of an environment difference that only that fleet has.

Diagnose

Get the stacks: py-spy every rank in a hung job; dump the NCCL flight recorder to see which collective each rank is inside — a mismatch localizes it instantly to a rank-divergence bug (Tree 7's universal cause). If all ranks are in the same collective, it's transport/environment: NCCL_DEBUG=INFO for ring setup, check the fleet's driver/NCCL versions against x86 (Vol. III §09), check for a recent runner image change (Vol. III §19's timeline correlation). Timing sensitivity on one platform often means a real race that x86's timing happens to hide — do not dismiss it as "arm is flaky."

Fix & harden

Root-cause the divergence or the environment drift; then make hangs loud — collective timeouts and async error handling so the next occurrence produces an attributed failure with a flight-recorder dump instead of a dead job (§21). Coverage-wise: this is the parity problem of Vol. III §08 — the arm lane needs the same distributed signal as x86, or its bugs are found by users.
