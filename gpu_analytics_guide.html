Two facts about this picture matter more than any other, and the whole guide hangs off them:

The two load-bearing ideas

1. Everything above CUDA is "just" composition of parallel primitives. A SQL GROUP BY is a hash table (cuCollections) plus a reduction (CUB) over columnar buffers (libcudf) scheduled by an operator (Velox-cuDF) inside a pipeline (Velox) driven by a plan (Presto). No magic — each layer is a thin, well-defined transform of the layer below.

2. The data never wants to leave the GPU. The entire performance thesis is "keep columns resident in GPU memory from Parquet read to final result." The moment data is forced back to host memory — for an unsupported operator, for a naive network shuffle, for spilling — you pay PCIe and serialization costs that can erase the GPU's advantage. Most of your roadmap (UCX exchange, spilling, streaming agg) is really about not breaking GPU residency.

How a query flows through it

Concretely, end-to-end on Presto-on-GPU: SQL arrives at the Presto coordinator (JVM), which parses, optimizes, and produces a distributed plan of stages. Each worker is a Presto C++ (Prestissimo) process embedding Velox. Velox translates its slice of the plan into pipelines of operators. The cuDF DriverAdapter then rewrites those pipelines to swap CPU operators for GPU ones. Each GPU operator calls libcudf, which calls CCCL/cuCollections/nvCOMP, which launch CUDA kernels on columns living in GPU memory. When a stage boundary requires repartitioning, the Exchange operator moves columns between workers — over HTTP in the legacy path, or GPU-to-GPU over UCX (NVLink intra-node, RoCE/InfiniBand inter-node) in the path you own.

Where your roadmap sits

UcxExchange and "GPU exchange in Velox-cuDF" live in the Operators / Exchange row (§09). "Multi-node performance" is about how that row behaves on different Platform fabrics — NVL72 vs DGX vs AWS (§10). "Spilling to host" is a libcudf + RMM memory-hierarchy concern (§11). "Streaming aggregation" is a Velox-cuDF operator design problem on top of libcudf groupby (§12). Four projects, four different layers, one system.

Section 01

Why GPUs for analytics — and when not

Start with the 101, because the rest only makes sense if you know why a GPU wins on analytical queries and, just as important, where it loses.

The three numbers that decide everything

Analytical (OLAP) queries are bandwidth-bound, not latency-bound. A typical query scans hundreds of millions of rows, touches a few columns, and produces a small result: filter, join, aggregate, sort. The bottleneck is "how fast can I stream bytes through arithmetic." That favors the GPU on all three axes that matter:

AxisServer CPU (e.g. 64-core)Data-center GPU (e.g. H100 / B200)Why it matters for OLAP
Memory BW~0.3–0.5 TB/s (DRAM)3–8 TB/s (HBM3/3e)Scans & hash probes are pure memory streaming → ~10× headroom
Parallelismtens of cores, SIMD lanestens of thousands of threads (SIMT)Per-row work is embarrassingly parallel
$/throughputbaselineoften 5–20× better on heavy joins/aggsOne GPU replaces a rack of CPU workers on the right workload

The published Velox-cuDF numbers make this concrete: on TPC-H SF1000 a single GH200 Grace-Hopper finished 21 of 22 queries in ~100 s versus ~1,246 s for Presto C++ on a high-end Threadripper — well over a 10× single-node speedup on compute-heavy queries.

The columnar + SIMT fit

The GPU advantage is unlocked only if data is laid out so that 32 threads in a warp read 32 adjacent values in one memory transaction. That is exactly what a columnar layout gives you (§02): a column is a contiguous array, so a warp scanning it gets perfectly coalesced access. Row-major storage would scatter each thread's read across cache lines and waste 90%+ of memory bandwidth. Columnar storage is not a nice-to-have for GPU analytics; it is the prerequisite.

Where the GPU loses (be honest about this)

GPU is the wrong tool when…

Data is tiny. Kernel launch + host↔device copy overhead (microseconds to milliseconds) dominates if you're processing a few thousand rows. CPU wins on small queries; this is why Velox-cuDF wants ~1 GiB batches, not the ~1K-row batches CPU Velox likes.

The query is latency-critical and trivial. A point lookup by primary key wants an index on a CPU, not a full-column scan on a GPU.

It doesn't fit in GPU memory and can't stream. A 5 TB hash-join build side on an 80 GB GPU forces spilling or partitioning (§11). Sometimes the host-staging cost wins.

Heavy data-dependent control flow / UDFs. Branchy row-at-a-time logic causes warp divergence. Some of this is fixable with JIT (Sirius and cuDF both JIT-compile expressions), but not all.

The mature mental model: GPUs win scan-heavy, join-heavy, aggregation-heavy, string-heavy analytical batch queries on large data. That's precisely the OLAP / data-lake / ETL workload these systems target — and precisely why the design fights so hard to keep big columnar batches resident on the device.

Section 02

Columnar data & Apache Arrow — the lingua franca

Every layer above CUDA agrees on one data shape: columnar, Arrow-compatible buffers. cuDF columns, Velox vectors, and Arrow arrays are deliberately near-identical in memory so that crossing between them is cheap or free. Understanding this layout is non-negotiable: it explains why conversions are sometimes zero-copy and sometimes not, why null handling is a recurring source of bugs, and why a "column" is the atom everything else manipulates.

Row-major vs column-major

Picture a table of (id, price, country). Row storage (Postgres heap, a CSV line) keeps each record's fields together: [id0,price0,country0][id1,price1,country1]…. Columnar storage keeps each field's values together: [id0,id1,id2,…][price0,price1,…][country0,…].

For SELECT avg(price) WHERE country='US', columnar reads only the price and country arrays — contiguously — and skips id entirely. On a GPU, a warp reading 32 consecutive price values pulls them in one coalesced 128-byte transaction. That is the whole ballgame.

The Arrow physical layout

An Arrow/cuDF column of a primitive type is, at minimum, two buffers plus metadata:

  • Data buffer — a flat, contiguous array of the values (e.g. int32 packed end to end). Fixed-width types only; this is what kernels stream over.
  • Validity bitmap — one bit per element: 1 = valid, 0 = null. So nulls cost 1/8 byte each, and "is this null" is a bit test, not a sentinel comparison. A column with no nulls can omit the bitmap entirely (a common fast path).
  • Null count + type + length — cheap metadata.

Variable-width types add an offsets buffer. A STRING column is an offsets array (int32/int64) plus one big chars buffer holding all the bytes concatenated. String i is chars[offsets[i] : offsets[i+1]]. No per-string allocation — just two arrays. Nested types (LIST, STRUCT) compose recursively: a list column is offsets + a child column.

# INT32 column: [10, 20, NULL, 40]
data:     [10][20][??][40]      # 4 x 4 bytes, contiguous; ?? is garbage, masked out
validity: 0b00001011            # bit i set = row i valid → rows 0,1,3 valid, row 2 null
null_count = 1

# STRING column: ["gpu", "", NULL, "cudf"]
offsets:  [0, 3, 3, 3, 7]       # length = offsets[i+1]-offsets[i]
chars:    "gpucudf"             # row 1 is empty, row 2 null (still occupies an offset slot)
validity: 0b00001011
Why this layout is everywhere

Because it makes operators buffer transforms. A filter doesn't move strings around — it computes a new offsets array and gathers bytes. A "select these rows" is a gather: read an array of row indices, produce new data+validity by indexed copy. Once you see columns as buffers and operators as buffer math, the entire libcudf API (§05) becomes predictable.

Vectorized (batch-at-a-time) execution

Classic databases used the Volcano model: each operator's next() returns one row, pulled up the tree. That's death on a GPU (and slow even on modern CPUs) — one row can't fill a warp. Modern engines (Velox, DuckDB, cuDF, Photon) are vectorized: operators pass batches of a few thousand rows (CPU) up to ~1 GiB (GPU) as column vectors. Each operator does its whole job on the batch with tight, branch-light kernels, then hands the batch onward. This amortizes per-call overhead and exposes the data parallelism the GPU needs.

Dictionary encoding & run-length — compression that operators understand

Low-cardinality columns (e.g. country) are often dictionary-encoded: a small dictionary of distinct values plus an array of integer codes. Velox has DictionaryVector; Arrow has dictionary arrays; Parquet stores dictionary pages. Operators can sometimes run directly on the codes (compare integers, not strings), which is both less memory and faster. Knowing whether your data arrives dictionary-encoded matters for join/group-by performance — decoding too early throws away a free optimization.

The null-semantics trap

Presto, Spark, and SQL each have subtly different null rules (three-valued logic, null ordering in sorts, null handling in aggregates, NULLIF, etc.). A GPU operator that ignores the bitmap, or treats null as a sentinel value, will produce fast wrong answers — the worst kind. Much of the Velox-cuDF effort is exactly "make the GPU operator's null semantics match what Presto/Spark users expect." When you diff GPU vs CPU results (§15), null edge cases are the first place divergence shows up.

Section 03

CUDA, memory & RMM — the platform floor

You know CUDA kernels already, so this section skips the SIMT basics and focuses on the parts that actually govern analytics performance: memory and streams. In OLAP, allocation and data movement — not arithmetic — are usually what you're fighting.

On GH200 and GB200, the CPU and GPU are joined by NVLink-C2C at 900 GB/s (≈7× PCIe Gen5) with cache-coherent shared memory. "Managed memory" and "spill to host" stop being cliffs and become gentle slopes — host memory is almost a slower tier of GPU memory. This is why "spilling to host" (§11) is far more attractive on Grace-Hopper than on a PCIe-attached A100, and why your hardware target (§10) changes the whole spilling calculus.

Streams and stream-ordering — the soul of cuDF

A CUDA stream is an ordered queue of GPU work. Operations in one stream run in issue order; operations in different streams may overlap. Analytics engines lean on this for two reasons: (1) overlap copy with compute (while one batch computes, the next is copying in), and (2) run independent pipeline branches concurrently.

Crucially, libcudf is stream-ordered: nearly every API takes an rmm::cuda_stream_view, and both the kernel and its memory allocation/free are ordered on that stream. This is why Velox's CudfVector carries a stream alongside its cudf::table — so that asynchronously launched GPU work on that batch stays correctly ordered as it flows between operators.

The #1 correctness bug in async GPU code

Reading a result before its stream has produced it. If operator A launches work on stream s and operator B (on a different stream, or the host) reads A's output without synchronizing, you get a race — intermittent wrong answers that vanish under a debugger. Stream-ordered allocation prevents the allocation race; you still must order the data dependency (event wait, or share the stream). When a Velox-cuDF result looks nondeterministic, suspect stream ordering first.

RMM — why you never call cudaMalloc in a hot path

cudaMalloc/cudaFree are synchronous — they stall the whole device and serialize streams. A query that allocates per-batch (and every operator allocates its output column) would spend more time in the allocator than in compute. The fix is the RAPIDS Memory Manager (RMM): a pluggable allocator that everything (libcudf, cuCollections, Velox-cuDF) routes through.

RMM's workhorse is the pool allocator: grab a big slab from the driver once, then satisfy per-batch allocations by sub-allocating from the pool in microseconds with no device sync. Other resources stack on top (statistics/logging adaptors, a cuda_async_memory_resource wrapping the driver's cudaMallocAsync pool, a managed-memory resource for oversubscription, etc.) via the memory_resource interface — composable like decorators.

// Configure the process-wide RMM allocator ONCE at startup.
// A pool pre-reserves device memory so per-batch allocs don't hit the driver.
auto cuda_mr = std::make_shared<rmm::mr::cuda_memory_resource>();
auto pool_mr = std::make_shared<
    rmm::mr::pool_memory_resource<rmm::mr::cuda_memory_resource>>(
        cuda_mr.get(),
        60UL << 30);   // initial pool: 60 GiB
rmm::mr::set_current_device_resource(pool_mr.get());

// From here on, every libcudf output column allocates from the pool — fast,
// stream-ordered, no global device sync. This single line is often the
// difference between "GPU is slower than CPU" and "GPU is 10x faster".
Fragmentation — the silent OOM

Pool allocators can fragment: enough free bytes exist, but not contiguously, so a large column allocation fails with an out-of-memory error even though nvidia-smi shows free space. Big-batch GPU analytics (1 GiB allocations) is especially prone to it. Mitigations: the cuda_async_memory_resource (driver-managed pool defragments better), an arena resource, capping batch size, or fewer concurrent drivers. When you see "OOM but there's plenty free," it's fragmentation, not a leak — this is a frequent multi-driver tuning issue (§06, §15).

Section 04

CCCL · cuCollections · nvCOMP — the primitives

This is the Parallel Algorithms row. libcudf does not implement sort or hash from scratch; it composes these three libraries. Knowing them tells you what libcudf can do cheaply and where its operators' performance and limits come from.

CCCL = CUB + Thrust + libcu++

The CUDA Core Compute Libraries bundle three layers of generic parallel building blocks:

  • Thrust — high-level, STL-like algorithms over device vectors: sort, reduce, scan (prefix sum), transform, gather/scatter, unique, partition. Productivity layer; picks reasonable kernels for you.
  • CUB — lower-level, performance-tunable primitives at device, block, and warp scope: DeviceRadixSort, DeviceReduce, DeviceScan, DeviceSelect, segmented variants. When libcudf needs a tuned sort or scan, it calls CUB directly.
  • libcu++ (libcudacxx) — the CUDA C++ standard library: cuda::std:: types, cuda::atomic, cuda::barrier, memory-order primitives that work on device. The plumbing the other two stand on.
Scan & reduce are the two primitives behind half of SQL

Reduction (combine N values → 1) implements SUM/MIN/MAX/COUNT/AVG. Prefix-sum / scan (running totals) is how the GPU does compaction: to filter a column, compute a boolean mask, scan it to get output positions, then scatter surviving rows into place — all data-parallel, no locks. FILTER, WHERE, building offset arrays, computing partition boundaries: all scans underneath. Internalize "filter = mask + scan + scatter" and libcudf's filter implementation stops being mysterious.

cuCollections (cuco) — GPU hash tables

Joins and hash aggregations need a hash table that thousands of threads can probe and insert into concurrently. CPU hash maps (chaining, locks) don't translate. cuCollections provides lock-free, open-addressing GPU hash tables — static_map, static_multimap, static_set — built on cuda::atomic compare-and-swap.

Design points that leak upward into operator behavior:

  • Open addressing, not chaining. Fixed slot array; collisions probe to the next slot. No pointer chasing → coalesced, cache-friendly. But the table must be sized up front (a load factor below 1.0), so libcudf must estimate cardinality before building — over-allocate and you waste memory, under-allocate and you must rehash.
  • Two-phase use. Build phase: insert all build-side keys (massively parallel CAS). Probe phase: every probe-side thread looks up its key. This is exactly a hash join's build/probe split.
  • Multimap for joins. Equi-joins need duplicate keys → static_multimap. Aggregations map key → accumulator slot, often combined with atomic updates for the running aggregate.
// Sketch of a GPU hash inner-join, the shape libcudf::inner_join uses:
// 1) BUILD — insert (key, row_index) for every build-side row, in parallel
cuco::static_multimap<key_type, size_type> hash_table{
    capacity, cuco::empty_key{-1}, cuco::empty_value{-1}};
hash_table.insert(build_pairs.begin(), build_pairs.end(), stream);

// 2) PROBE — each probe row finds all matching build rows; emit index pairs
hash_table.retrieve(probe_keys.begin(), probe_keys.end(),
                     out_probe_idx, out_build_idx, stream);

// 3) GATHER — materialize the joined columns from the two index arrays
//    (cudf::gather on each side — see SS05). The join is "find indices,
//    then gather"; cuco does step 1-2, libcudf gather does step 3.

nvCOMP — GPU (de)compression

Decompression and compression run on the GPU via nvCOMP: LZ4, Snappy, Deflate/GZIP, Zstd, plus cascaded/bitpacking schemes tuned for columnar data. Three places it matters to you directly:

  • I/O. Parquet/ORC pages are compressed (Snappy/Zstd). cuDF's GPU reader decompresses on-device so the CPU never touches the bytes — pairs with GPUDirect Storage to go SSD/S3 → GPU directly.
  • Spilling (§11). Compress a column before evicting it to host/disk to cut the bytes moved across PCIe — trade GPU cycles (cheap) for bandwidth (scarce).
  • Exchange/shuffle (§09). Compressing partitions before they cross the network reduces wire bytes; on a fat NVLink fabric the compute may not be worth it, on a thin TCP link it usually is. A real decision knob.
Hardware decompression on Blackwell

Blackwell (B200/GB200) adds a dedicated hardware decompression engine that offloads formats like Snappy/LZ4/Deflate from the SM cores entirely — NVIDIA cites large speedups on database scan/decompress paths. If your TableScan is decompress-bound, the platform generation underneath you (§10) changes the answer.

The composition, in one line: cuCollections gives you the hash table, CUB/Thrust give you sort/scan/reduce, nvCOMP gives you (de)compression, RMM gives you fast memory — and libcudf (next) stitches them into relational operators with correct SQL semantics over Arrow columns.

Section 05

libcudf / cuDF — relational ops on columns

libcudf is the CUDA/C++ library that turns the primitives of §04 into the columnar relational operators of a query engine. (The Python cudf dataframe is a thin binding over it; Velox-cuDF, Sirius, and Spark-RAPIDS all use the C++ layer.) It is "the ETL algorithms" box.

The data model: column, column_view, table

  • cudf::column — owns its GPU memory (an rmm::device_buffer for data, another for the null mask, child columns for nested types). Move-only, like a unique_ptr to device data.
  • cudf::column_view — a non-owning, read-only window into a column (pointer + size + null mask + type + offset). Operators take column_views in and return owning columns out. mutable_column_view for in-place writes.
  • cudf::table / table_view — an ordered collection of columns of equal length: a batch of rows in columnar form. This is what flows between operators.
Operators are pure functions: views in, owned table out

Almost every libcudf op has the shape table f(table_view input, ..., rmm::cuda_stream_view, rmm::device_async_resource_ref mr). It reads inputs through views, allocates outputs from the memory resource on the given stream, and returns new owning columns. Inputs are never mutated. This functional style is why GPU pipelines are easy to reason about and parallelize — but it also means every operator allocates its full output, which is the root of GPU memory pressure (§11). There is no in-place "append"; a filter that keeps 90% of rows still allocates a fresh 90%-sized column.

gather & scatter — the two primitives under everything

If you learn one libcudf concept, learn gather. Given a table and an array of row indices, gather produces a new table containing those rows in that order. It is how filter, join materialization, sort, and shuffle-partition all actually move rows:

// gather: output[i] = input[ gather_map[i] ] for every column, in parallel.
// One indexed copy per output element — coalesced, no branching.
std::unique_ptr<cudf::table> selected = cudf::gather(
    input_table_view,        // the columns to pull from
    gather_map_view,         // INT32 column of row indices to keep, in order
    cudf::out_of_bounds_policy::NULLIFY,  // negative index → null (used by outer joins)
    stream, mr);

// FILTER is literally: build boolean mask → apply_boolean_mask
//   (internally: mask → scan → gather of surviving positions)
auto mask = cudf::compute_column(t, /*expr:*/ price > 100, stream, mr);
auto result = cudf::apply_boolean_mask(t, mask->view(), stream, mr);

scatter is the inverse (write input rows to given output positions) and underlies updates and some partition writes.

groupby: hash vs sort (this matters for §12)

libcudf groupby has two strategies. Hash groupby builds a key→accumulator hash table (cuco) and atomically updates running aggregates — one pass, great when the number of distinct groups is modest. Sort groupby sorts by key so equal keys are adjacent, then does a segmented reduce — handles huge cardinality and any aggregate, but pays for the sort. The engine picks based on cardinality and aggregate type. Streaming aggregation (§12) is built on the partial → merge decomposition of these.

GPU-direct I/O

The TableScan path can read Parquet/ORC straight from NVMe or S3 into GPU memory via GPUDirect Storage (GDS), bypassing the CPU bounce buffer, then decompress and decode on-device with nvCOMP + cuDF kernels. This keeps the "never leave the GPU" promise from the very first byte read — important for your AWS+S3 target (§10), where scan cost can dominate.

Type & overflow gotchas when matching Presto/Spark

libcudf is a general columnar library; Presto/Spark have specific type rules. Watch: decimal precision/scale (fixed-point columns), integer overflow in SUM (Spark may promote, a naive int64 accumulate wraps), timestamp units/timezones, and string collation. These are exactly the "fast but wrong" risks — the GPU op runs fine and returns a number that disagrees with CPU in the last digit or on an edge row. §15's CPU-vs-GPU diffing exists to catch these.

Section 06

Velox & Velox-cuDF — the execution engine

Velox is a C++ library, not a database. It provides the reusable guts of a query engine — vectors, expression evaluation, and a catalog of vectorized operators with a scheduler — so that systems like Presto, Spark (via Gluten), and others don't each re-implement them. "Composable data management": write the operators once, reuse everywhere. Velox-cuDF adds a GPU backend to that shared engine, which is why one effort accelerates Presto and Spark at once.

The Velox execution model

The objects you'll live in:

  • PlanNode tree — the logical/physical plan Velox receives (TableScan → Filter → Project → HashJoin → Aggregation → …).
  • Task — the execution of a plan fragment on one worker. A Task is split into Pipelines at blocking boundaries (e.g. a hash-join build).
  • Driver — a thread of execution running one pipeline over a slice of data. Velox runs multiple drivers per pipeline for intra-node parallelism. On CPU you want ~one driver per core; on GPU, far fewer (below).
  • Operator — a node in a pipeline (TableScan, FilterProject, HashProbe, HashAggregation, PartitionedOutput, Exchange…). Each consumes and produces RowVector batches.
  • Split — a unit of input (e.g. a Parquet file range) handed to a TableScan; how work is distributed.
  • Vector — Velox's Arrow-compatible columnar batch: FlatVector (plain column), DictionaryVector, ConstantVector, RowVector (a struct/row of child vectors = a table). Encodings are lazy and composable, like libcudf's dictionary handling.

Execution is push-based within a pipeline and connected across pipelines/workers by Exchange. The driver pulls a batch from its source operator, pushes it through the chain, and the last operator either yields results or hands them to a PartitionedOutput that feeds the next stage's Exchange (§08–09).

How cuDF plugs in: the DriverAdapter

Velox exposes a DriverAdapter hook: after Velox builds executable pipelines, a backend may rewrite them. The cuDF backend's adapter walks the operator chain and rewrites it for GPU. It can:

  • Replace an operator one-to-one (Velox HashJoin → CudfHashJoin).
  • Fuse adjacent operators into one GPU operator (fewer kernel launches, fewer intermediate columns).
  • Add a bridge operator where needed.

The first GPU operators include TableScan, CudfHashJoin, CudfHashAggregation, CudfFilterProject, CudfOrderBy, and the cuDF Exchange. When an operator has no GPU implementation yet, the adapter inserts a CudfConversion operator that copies the batch GPU→CPU, runs the CPU operator, and copies back — graceful fallback at the cost of a round trip.

CudfVector: a RowVector that owns a cudf::table

The clever bit. CudfVector inherits from Velox's RowVector but internally holds a std::unique_ptr<cudf::table> (the GPU-resident data) plus an rmm::cuda_stream_view (so async work stays stream-ordered as the batch flows between operators). To the rest of Velox it is a RowVector — it satisfies the standard "operators produce and consume RowVector" contract — but the bytes live on the GPU. This is what lets a whole pipeline stay on-device while still using Velox's unmodified scheduling, Task/Driver model, and operator interfaces. The CPU↔GPU boundary is exactly where a CudfVector is unwrapped or a CudfConversion fires.

The "GPU is barely faster" anti-pattern

Three classic misconfigurations flatten the GPU win: (1) batch too small — left at CPU defaults, so kernels are launch-bound; (2) accidental CPU fallback — one unsupported operator (or expression, or type) inserts CudfConversion mid-pipeline and every batch ping-pongs across PCIe; (3) too many drivers fragmenting memory into OOM. When a query underperforms, check those three before profiling kernels. The fix for (2) is to read the rewritten plan and confirm it's end-to-end GPU (no conversion nodes).

Section 07

Presto C++ · Gluten · Substrait — the compatibility layer

This row answers "how does a query written for Presto or Spark end up as a Velox pipeline?" Two different bridges, because Presto and Spark have different histories.

Presto C++ (Prestissimo)

Classic Presto workers are JVM processes. Presto C++ (a.k.a. Prestissimo) replaces the worker with a native C++ process built on Velox, while keeping the Java coordinator for parsing, planning, and optimization. The coordinator emits its plan as JSON/Thrift fragments; the C++ worker translates each fragment into a Velox PlanNode tree and executes it. For GPU, the cuDF DriverAdapter then rewrites those pipelines. Because the coordinator is unchanged, you inherit Presto's full SQL surface, optimizer, and connector ecosystem "for free" — the GPU work is entirely in the worker. This is why Presto-on-GPU targets end-to-end GPU execution.

Gluten — the Spark bridge

Spark's engine lives in the JVM and produces Spark physical plans. Gluten (an Apache incubator project) intercepts those plans and offloads supported operators to a native backend (Velox) instead of running them in the JVM. The plan is conveyed in Substrait, a cross-engine serialized plan format, which Gluten translates into Velox plan nodes. Where an operator isn't offloadable, Gluten falls back to vanilla Spark — so Spark-on-Velox is naturally hybrid.

Why Spark integration is "hybrid", Presto is "end-to-end"

The published results reflect this: Presto-on-GPU runs entire query plans on the GPU, while Spark-via-Gluten currently offloads specific compute-heavy stages (e.g. a costly second stage of a TPC-DS query) to the GPU and leaves the rest on CPU. Same Velox-cuDF operators underneath; different integration maturity and different scheduling philosophy. For your roadmap, Presto-on-GPU is the deeper integration and where the exchange work lands first.

Substrait — the plan interchange

Substrait is to query plans what Arrow is to data: a neutral, serializable representation of relational operations (read, filter, project, join, aggregate) that any engine can produce or consume. Gluten uses it (Spark→Velox); Sirius uses it (DuckDB/Doris→GPU, §13). Knowing Substrait exists explains how a GPU engine can be a "drop-in" for a host database without rewriting that database's SQL front-end.

Section 08

The distributed runtime — stages, tasks, splits

Everything so far was single-node. To run on NVL72 or a fleet of EC2 instances, the query is cut into pieces that run on many workers and exchange data between them. This is the structure your multi-node performance work lives inside.

Coordinator → stages → tasks → splits

  • The coordinator compiles SQL into a tree of stages. A stage boundary is wherever data must be repartitioned — e.g. before a distributed join or a global aggregation, rows must be grouped by key onto the right worker.
  • Each stage runs as parallel tasks, one per worker (per GPU, in practice). A task is the Velox Task of §06.
  • Input is divided into splits (file ranges, partitions) distributed across tasks so they work in parallel.
  • Between stages sits the shuffle / exchange: stage N's tasks each produce partitioned output; stage N+1's tasks each consume the partitions destined for them. This all-to-all data movement is the subject of §09.
# Distributed hash join, two stages, three workers:
Stage 1 (scan+partition)          Stage 2 (join+aggregate)
  task A ─ scan shard ─┐            ┌─> task X  (gets hash-partition 0)
  task B ─ scan shard ─┼─ EXCHANGE ─┼─> task Y  (gets hash-partition 1)
  task C ─ scan shard ─┘  (shuffle) └─> task Z  (gets hash-partition 2)
            every producer sends rows to the consumer that owns
            hash(key) % numPartitions  →  N producers x M consumers
Why the exchange is the scaling wall

Single-node, you scale by adding GPUs and the data stays on-device. The moment a join or aggregation key isn't co-located, you must move data between GPUs/workers. With N producers and M consumers, that's an all-to-all with up to N×M streams. If each byte detours GPU→host→TCP→host→GPU, the network and PCIe — not the GPUs — set the query time. Amdahl's law bites: you can make the operators infinitely fast and still be gated by shuffle. That is exactly why a GPU-native, NVLink/RDMA exchange (next) is worth a person's full attention.

Section 09 · your mandate

The Exchange — GPU-to-GPU shuffle over UCX

This is the operator you own. The Exchange moves data between workers at stage boundaries; on a GPU cluster it is usually the difference between linear scaling and a flat line. The job: keep columnar data on the GPU from the moment it's partitioned to the moment it lands in a peer GPU, using the fabric directly. Here's the full mechanism, bottom to top, tying back to the interconnect guide.

The three Velox exchange types

TypeWhat it doesUsed for
PartitionedHash-partition input by key; send each partition to the worker that owns itdistributed joins & group-bys (the common case)
MergeReceive multiple sorted input partitions and merge into one sorted outputdistributed ORDER BY / merge sort
BroadcastReplicate one worker's data to all workersbroadcast (small-side) joins

On the producer side this is the PartitionedOutput operator (it calls libcudf hash_partition to split a batch into per-destination sub-tables); on the consumer side it's the Exchange operator pulling its partitions from every producer.

The legacy path, and why it's slow

Baseline Presto shuffle is HTTP over TCP. On GPU data, a single partition's journey is brutal:

producer GPU ─D2H copy─> host ─serialize─> HTTP/TCP ─> host ─deserialize─> H2D copy ─> consumer GPU
            \_____ PCIe _____/   \__ CPU __/   \_ NIC _/    \__ CPU __/   \_____ PCIe _____/
   every byte crosses PCIe twice, is touched by the CPU twice, and pays TCP/kernel overhead

The GPUs sit idle through all of it. With many producers and consumers (all-to-all), this serializes the whole query behind PCIe and the CPU network stack. This is the wall §08 warned about.

The GPU-native UCX path (what you're building)

Replace that with a path where columns never leave device memory and the transport reads/writes GPU memory directly:

producer GPU buffers ──UCX (GPUDirect RDMA / NVLink)──> consumer GPU buffers
   no D2H, no CPU touch of payload, no TCP copy — the NIC or NVSwitch DMAs GPU→GPU

Step by step, and where each piece comes from:

  1. Partition on GPU. libcudf hash_partition(table, key_cols, num_partitions) produces, in one kernel pass, the reordered table plus the offset where each destination partition begins. No host involvement.
  2. Describe the partition as buffers. A cudf partition is just its column buffers (data, validity, offsets, children) plus small metadata (types, lengths, null counts). The metadata goes in a header; the buffers are the payload — already contiguous in GPU memory, so they're RDMA-ready as-is.
  3. Hand the GPU pointers to UCX. UCX's ucp_put/ucp_tag_send accept device pointers when built with CUDA support. UCX selects the transport itself (this is the whole point of UCX from the last guide): NVLink for same-node peers, RoCE/InfiniBand for remote peers, with GPUDirect RDMA so the HCA reads GPU memory without staging through host.
  4. Land directly in the consumer's GPU. The bytes arrive in a pre-registered device buffer on the consumer; the Exchange operator wraps them back into a cudf::table / CudfVector and the downstream pipeline continues — still on-device.
This is the exact seam where this guide meets the interconnect guide

The Exchange is a libcudf + UCX operator. libcudf gives you the partitioned device buffers; UCX (UCP API on top of UCT transports) moves them; GPUDirect RDMA lets the NIC touch GPU memory; NVLink/NVSwitch carries intra-node traffic; RoCE/InfiniBand carries inter-node. Everything in the comms guide's §02 (RDMA verbs), §06 (NCCL's transports), and §08 (UCX architecture) is the machinery underneath this one operator. You are, quite literally, the person making the query engine speak the fabric.

// Producer side of a GPU-native partitioned exchange (sketch).
// 1) Partition the batch on-device — reorders rows; returns split offsets.
auto [partitioned, offsets] = cudf::hash_partition(
    batch->view(), key_indices, num_workers, stream, mr);

// 2) For each destination worker, send its slice's buffers over UCX.
for (int dst = 0; dst < num_workers; ++dst) {
  auto slice = cudf::slice(partitioned->view(),
                          {offsets[dst], offsets[dst + 1]});
  WireHeader hdr = describe(slice);  // types, lengths, null counts, buffer sizes

  ucp_tag_send_nbx(eps[dst], &hdr, sizeof(hdr), header_tag, &params);
  for (auto buf : device_buffers_of(slice))   // DEVICE pointers
    ucp_tag_send_nbx(eps[dst], buf.data(), buf.size(), data_tag, &params);
  // UCX picks NVLink vs RDMA per endpoint; GPUDirect RDMA = no host bounce.
}
// Consumer: ucp_tag_recv_nbx into pre-registered device buffers, then
// reassemble a cudf::table view over them and push downstream. Zero D2H.

Serialization: less than you'd think

Because cuDF/Arrow buffers are already the wire format, "serialization" is mostly assembling a small header and sending the buffers verbatim — close to zero-copy. Contrast with row formats (serialize each row) or with engines that must transcode. Pitfalls: variable-width columns (strings/lists) carry offset buffers that must travel too and be rebased on arrival; null masks must travel; and the header must be enough to reconstruct exact types (decimal scale, timestamp unit) so the consumer rebuilds a semantically identical column.

Correctness & robustness checklist for the exchange (your "performance and correctness" mandate)

Buffer registration: RDMA needs pinned/registered memory; registration is expensive, so cache/pool registered buffers (UCX rcache) rather than registering per send. Backpressure & flow control: a fast producer must not OOM a slow consumer — bound in-flight bytes. Ordering & completion: don't reuse/free a send buffer before UCX signals completion (mirror of the stream-ordering rule in §03). Partition skew: a hot key sends one consumer 10× the data — detect and, where possible, split. Failure semantics: a dropped worker mid-shuffle must surface as a query error, not a hang. Empty/!found partitions: handle zero-row sends. These are the bugs that separate a demo from production.

The result that justifies the project

On an 8-GPU DGX A100, swapping the baseline HTTP exchange for the UCX-based cuDF exchange (using NVLink intra-node) delivered >6× speedup on the TPC-H suite at SF1000 — and let the cluster finish all 22 queries with default async allocation, no managed-memory fallback. The exchange wasn't a tweak; it moved the whole multi-GPU regime from "shuffle-bound" to "compute-bound."

NVL72 · DGX/HGX · AWS+S3 — topology decides the plan

"Drive multi-node performance, NVL72 first then AWS EC2 to S3" is really one question asked on three different fabrics: how fast can GPUs exchange data, and how fast can they read it? The same exchange operator behaves completely differently depending on the interconnect and storage underneath. Here's the landscape, ordered from the friendliest to the harshest.

Why NVL72 changes the exchange's character

On NVL72 the rack is the NVLink domain — every one of the 72 GPUs reaches every other at NVLink speed, ~14× PCIe Gen5, with uniform ~300 ns latency. The producer→consumer detour that dominates HTTP shuffle effectively vanishes; your UCX exchange runs over NVLink for the whole "cluster." So on NVL72 the limiter shifts to (a) how balanced your hash partitions are (skew now matters more than bandwidth) and (b) raw operator throughput. It also makes spilling to host cheap: the Grace CPU's memory hangs off NVLink-C2C at 900 GB/s, so host is just a slower memory tier, not a PCIe exile (§11). This is why the roadmap starts here — it's the regime where the GPU engine is least constrained, so you validate correctness and operator perf before the network gets hard.

…and why AWS+S3 is the hard mode you graduate to

On EC2, cross-instance traffic has no NVLink — it's EFA/RoCE, far thinner than the NVL72 fabric — and your data lives in S3, an object store with tens-of-ms first-byte latency and per-request cost. Two consequences: (1) the TableScan can dominate, so prefetching, parallel range reads, GPUDirect Storage, and keeping Parquet compressed until the GPU decompresses it all matter; (2) inter-instance shuffle is precious, so compress partitions (nvCOMP) before they hit the wire and partition topology-aware to minimize cross-instance bytes. The exchange that looked "free" on NVL72 is now the thing you fight. Same operator, opposite tuning.

Topology-aware partitioning (the portable idea)

Across all three regimes the winning move is the same: make the cheap link carry the most traffic. Hierarchical/two-level shuffle — partition first to local peers over NVLink, then a reduced cross-node step over IB/EFA — mirrors how NCCL builds hierarchical collectives (comms guide §06). On NVL72 there's no hierarchy to exploit (it's flat NVLink), so you optimize differently than on a multi-node DGX or EC2 fleet. A production exchange should be aware of which peers are NVLink-local vs network-remote and route accordingly; UCX already discovers this, but the partition assignment is your operator's choice.

Benchmark hygiene that bites everyone

Hot-cache single-node numbers and cold-cache multi-node numbers are not comparable — the published Velox-cuDF figures explicitly separate them. When you report multi-node speedups, fix: cache state (cold S3 vs warm local), whether managed memory was needed, allocator (async vs managed), and which exchange (HTTP vs UCX). A "regression" is often just a cold cache or an accidental managed-memory fallback, not the operator.

Section 11 · your mandate

Spilling to host — surviving memory pressure

"Add spilling to host in Velox-cuDF." Here's the problem it solves and the design space. GPU memory is small (tens of GB) relative to the data, and §05 told us every libcudf operator allocates its full output. A hash-join build side, a high-cardinality aggregation's hash table, or a sort's working set can all exceed device memory. Without spilling, the query simply OOMs. With it, the query slows down gracefully instead of dying. Velox already has a mature CPU spilling framework; the work is making the GPU operators participate.

The memory hierarchy you're managing

GPU HBM        tens of GB     8 TB/s    <-- where columns want to live
   |  D2H / H2D copy (PCIe ~64 GB/s, or NVLink-C2C 900 GB/s on Grace-Hopper)
Pinned host    hundreds of GB  ~DRAM BW  <-- first spill tier (page-locked for fast DMA)
   |  disk I/O
Local NVMe     TBs            GB/s       <-- second spill tier (or skip on Grace)

Spilling = pushing the coldest data down this hierarchy when HBM is tight, and pulling it back up when needed. The art is choosing what and when with minimal damage to throughput.

Mechanism

  1. Detect pressure. Hook RMM: when an allocation would exceed a budget (or the pool fails), invoke a spill manager rather than throwing. Velox's MemoryPool/arbitrator drives this on CPU; the GPU backend needs the same trigger wired to RMM.
  2. Choose a victim. Spillable units are whole batches / build-side partitions. Pick the least-recently-used, or partitions not needed until later in the plan. Operator knowledge helps: a hash-join build partition that won't be probed for a while is a great victim.
  3. Evict. D2H-copy the victim's column buffers into a pinned host buffer (fast DMA). Optionally nvCOMP-compress first to cut bytes moved — GPU compute is cheaper than PCIe bandwidth. Free the device buffers back to RMM.
  4. Restore. When the operator needs the victim, H2D-copy (decompress) it back. Track residency so you never read a spilled buffer as if it were on-device (a stream-ordering-class bug, §03).
// Spill manager sketch: evict a build-side partition under memory pressure.
HostBuffer spill_to_host(const cudf::table_view& victim, rmm::cuda_stream_view s) {
  auto packed = cudf::pack(victim, s);          // contiguous {metadata, gpu_data}
  auto bytes  = nvcomp::compress(packed.gpu_data, s); // optional, bandwidth vs cycles
  HostBuffer h = alloc_pinned(bytes.size());
  cudaMemcpyAsync(h.data(), bytes.data(), bytes.size(), cudaMemcpyDeviceToHost, s);
  s.synchronize();                              // ensure copy done before freeing device mem
  return h;  // device buffers now freed back to RMM; metadata kept to restore later
}

Spill vs partition vs recompute — the real decision

Three ways to not OOM, and when each wins

Spill (evict to host/disk, bring back later): simplest, preserves the plan; cost is data movement. Best when the overflow is modest or host is close (Grace-Hopper).
Partition (a.k.a. grace hash join: split the join/agg into key-ranges that each fit, process one at a time): avoids ever holding everything; cost is extra passes over data. Best when the build side hugely exceeds memory.
Recompute (drop and regenerate from source): best when regeneration is cheaper than the round-trip (rare in analytics, common in ML).
A good operator combines them: partition to bound peak, spill within a partition for slack.

Grace-Hopper / NVL72 tilts the answer toward "spill freely"

On PCIe-attached GPUs, every spill costs a ~64 GB/s round trip and you spill reluctantly. On GH200/GB200, host memory hangs off NVLink-C2C at 900 GB/s and is cache-coherent — spilling to host is ~14× cheaper, so the threshold to spill drops dramatically and managed memory becomes a viable backstop. Your spilling design should be parameterized by the host link speed, not hard-coded — the same code should spill aggressively on Grace and conservatively on a cloud PCIe instance.

Spilling pitfalls

Pinned memory exhaustion: pinned host allocations are themselves limited and slow to allocate — pool them. Spill thrashing: evict-then-immediately-need causes ping-pong; victim selection must look at the plan, not just recency. Stream races on restore: a restored buffer must be fully copied before any kernel reads it. Correctness under spill: a query must return identical results spilled or not — diff them (§15). Compression CPU fallback: if nvCOMP doesn't support a path and you silently fall back to host compression, you've reintroduced the CPU bottleneck you were avoiding.

Section 12 · your mandate

Streaming aggregation — bounded-memory GROUP BY

"Improve streaming aggregation support in Velox-cuDF." The blog notes HashAggregation got "a streaming interface to manage partial and final aggregations." Here's what that means and why it's the natural partner to spilling.

The partial → final decomposition

A distributed/streaming aggregation splits every aggregate into two phases that exploit associativity:

  • Partial (pre-aggregate): each producer aggregates its local batches into per-group partial states (e.g. for AVG: keep (sum, count), not the average). Input shrinks from N rows to ~G groups.
  • Shuffle the small partial states by group key (the Exchange, §09) — far less data than shuffling raw rows.
  • Final (merge): each consumer merges partial states for its groups and finalizes (sum/count → avg).

This is why SUM/COUNT/MIN/MAX/AVG distribute cleanly (they're associative/mergeable) while MEDIAN / exact COUNT(DISTINCT) don't — they need approximate sketches (HLL, t-digest) to stream.

Why "streaming" specifically

Non-streaming aggregation buffers all input, builds one giant hash table, then emits — peak memory = whole input. Streaming aggregation processes batch-by-batch, maintaining a bounded running hash table of group states and emitting/merging as it goes, so memory ∝ number of groups, not number of rows. On a GPU with tens of GB, that distinction is the difference between fitting and OOMing on a billion-row scan.

// Streaming partial aggregation: fold each incoming batch into a running state.
GroupState state;  // key -> partial accumulators, lives on GPU, size ~ #groups
while (auto batch = source->next()) {            // ~1 GiB GPU batches
  auto partial = cudf::groupby(batch, keys)
                    .aggregate({SUM(v), COUNT(v)}, stream, mr);
  state = merge_states(state, partial, stream, mr);   // combine into running result
  if (groups_exceed_budget(state))                  // high-cardinality safety valve
    spill_or_flush(state);                          // ties straight into SS11
}
auto result = finalize(state, stream, mr);            // e.g. sum/count -> avg
The cardinality cliff

Streaming aggregation is bounded by group count, not row count — but a high-cardinality GROUP BY (e.g. by user_id over a billion users) makes the running hash table itself blow past memory. Then you need either (a) sort-based aggregation (sort by key, segmented reduce — handles unbounded cardinality, costs a sort), or (b) spilling the partial-state table (§11), or (c) repartitioning so each worker holds fewer groups. The streaming interface's job is to detect this and pick a path instead of dying. This is precisely where your streaming-agg and spilling projects intersect — they're two halves of "aggregate more data than fits."

Why this lands in Velox-cuDF, not raw libcudf

libcudf gives you one-shot groupby over a table. Turning that into a streaming operator — running state across batches, partial/final split, spill integration, correct null/overflow semantics for Presto — is operator-level work in the Velox-cuDF backend. It's the kind of "compose libcudf primitives into a correct, memory-safe Velox operator" task that defines the whole integration.

Decision trees

Should this query (or operator) run on GPU?

How big is the data the operator touches? ├─ small (< ~100 MB / few M rows) → CPU. Launch + H2D overhead dominates; GPU batch can't fill the device. └─ large → What's the operator mix? ├─ scan + filter + join + groupby + strings → GPU. The sweet spot — bandwidth-bound, parallel, mergeable. ├─ point lookup / tiny result by key → CPU + index. GPU full-scan wastes the device. └─ branchy row-at-a-time UDF → GPU only if JIT/AST-expressible; else warp divergence kills it. Does the whole pipeline stay on GPU? ├─ yes, no CudfConversion nodes → ship it — residency preserved end to end. └─ no, one op falls back → measure: a mid-pipeline GPU→CPU→GPU bounce can erase the win. Consider CPU for that whole stage.

Which shuffle / exchange strategy?

Are producer and consumer GPUs in the same NVLink domain? ├─ yes (intra-node NVSwitch, or anywhere on NVL72) → UCX over NVLink. ~1.8 TB/s, ~free. Optimize for partition balance, not bandwidth. └─ no (separate nodes / EC2 instances) → Is the link fat (InfiniBand/RoCE) or thin (TCP/EFA, cloud)? ├─ fat IB/RoCE → UCX + GPUDirect RDMA. Keep buffers registered (rcache); avoid host staging. └─ thin / cloud → UCX + nvCOMP-compress partitions first; trade GPU cycles for scarce wire bytes. Topology-aware partitioning. Still on the legacy HTTP exchange? └─ yes → that's the baseline you beat >6x on DGX A100 — every byte does GPU→host→TCP→host→GPU. Move to UCX cuDF exchange.

Out of GPU memory — what now?

Why is it OOM? ├─ "plenty free" in nvidia-smi but alloc fails → fragmentation. Use cuda_async resource / arena; cap batch size; fewer drivers. └─ genuinely too big → How much bigger than memory, and what op? ├─ slightly over, host link is fast (Grace-Hopper) → spill to pinned host (SS11). Cheap over NVLink-C2C. ├─ build side ≫ memory (join/agg) → partition / grace-hash: process key-ranges that each fit, in passes. ├─ high-cardinality GROUP BY → sort-based agg or spill the partial-state table (SS11+SS12). └─ last resort, must complete → CUDA managed memory — correctness backstop, expect page-fault slowdown.

Correctness: diff the GPU against the CPU

Your "performance and correctness" mandate has a single most-valuable technique: run the same query on the CPU backend and the GPU backend and diff the results. Velox makes this natural — same plan, flip the backend. Where they disagree, you've found a semantics bug. The usual suspects, in order:

  • Nulls — null ordering in sorts, null propagation in expressions, COUNT vs COUNT(*), null keys in joins/group-bys.
  • Numeric — integer overflow in SUM, decimal scale/precision, float associativity (GPU reduces in a different order → last-bit differences; use tolerance for floats, exact for ints/decimals).
  • Strings — collation, case, Unicode, empty-vs-null.
  • Time — timestamp units, timezone, date boundaries.

Spilling and exchange add a second axis: a query must return identical results with and without spilling, and across different partition counts. Make those equivalences part of CI.

The one habit that catches the most bugs

After any plan runs, dump the rewritten operator tree and read it. Most "mysterious" GPU performance and correctness issues are visible right there: a CudfConversion you didn't expect, an operator that didn't get replaced, a partition count that doesn't match the topology. The plan is the ground truth for what actually ran — not what you think you asked for.

The whole guide in five sentences

Columnar data in GPU memory (§02) is processed by libcudf operators (§05) composed from CUDA primitives (§03–04), scheduled by Velox pipelines that the cuDF DriverAdapter rewrites for the GPU (§06), driven by Presto/Spark plans through a compatibility layer (§07). Single-node, the goal is to keep data resident and batches huge; multi-node, the Exchange becomes the bottleneck (§08), which is why a GPU-native UCX shuffle over NVLink/RDMA (§09) is the highest-leverage thing to own. The fabric underneath — NVL72 vs DGX vs AWS+S3 (§10) — changes whether shuffle is free or fatal, and therefore changes every other decision. When data won't fit, spilling (§11) and streaming aggregation (§12) keep queries alive instead of OOMing. Everything else is reading the rewritten plan, diffing GPU against CPU, and making sure the data never accidentally leaves the GPU.
