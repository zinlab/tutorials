01The problem we are solving

The forward pass of a convolutional neural network is dominated, in both FLOPs and wall time, by its convolutional layers. In a 2D conv layer every output pixel is a dot product between a small filter and a local window of the input — and that dot product is performed at every spatial location, for every input channel, for every output channel, for every image in the batch. The arithmetic is embarrassingly parallel, but it is also memory bound if you do it carelessly: the same input pixel and the same filter weight are re-read from DRAM hundreds of times.

The starting point here is a textbook one-thread-per-output-pixel kernel (new-forward-original.cuh) that takes 13.59 s to run two convolution layers over 10,000 images. The goal is to make it fast without changing the mathematics or the model accuracy (which stays exactly at 0.7955 across every step — a good sanity guard).

Why this matters Every technique here — tiling, shared memory, constant memory, im2col, per-shape kernels — is the same toolkit production CUDA kernels (cuDNN, cuBLAS, TensorRT, FlashAttention) use. The shapes are toy; the mechanics are not.

02LeNet-5 in one page

LeNet-5 (Yann LeCun, 1998) was the first widely successful CNN. It was designed to recognise handwritten digits and reads like the prototype every modern CNN was carved out of: convolve → pool → convolve → pool → flatten → fully-connected → classify.

Input image ─▶ Conv1 ─▶ Pool ─▶ Conv2 ─▶ Pool ─▶ FC ─▶ FC ─▶ Softmax (H × W × 1) ↑ ↑ ↑ 12 filters 24 filters 10 classes (7 × 7) (7 × 7 × 12)

The "5" in LeNet-5 is the number of trainable layers. The genius of the architecture, then and now, is two ideas glued together:

  • Local receptive fields — each filter only looks at a tiny spatial neighbourhood (here 7 × 7), so it learns translation-equivariant features (an edge is an edge wherever it appears).
  • Weight sharing — the same filter slides across the whole image, dramatically reducing parameter count and acting as a strong prior that the world is spatially stationary.

The version used in this project is a slightly larger MXNet variant: input images are 72 × 72, kernels are 7 × 7, and the two conv layers produce 12 and 24 output feature maps respectively. The dimensions are not the canonical 1998 ones, but the structure is.

03What a convolution layer really is

Strip away the visual metaphor and a 2D convolution layer is a 6-level nested loop. Given an input tensor x[B, C, H, W] and a filter bank w[M, C, K, K], it produces output y[B, M, H_out, W_out] where H_out = H − K + 1 (no padding, stride 1): 

for b in 0..B:                // image in the batch
  for m in 0..M:              // output feature map
    for h in 0..H_out:        // output row
      for w in 0..W_out:      // output col
        acc = 0
        for c in 0..C:        // input channel
          for p in 0..K:      // kernel row
            for q in 0..K:    // kernel col
              acc += x[b,c,h+p,w+q] * w[m,c,p,q]
        y[b,m,h,w] = acc

Every output element is independent of every other output element — that is what makes the layer GPU-friendly. The challenge is that each element is also cheap: only C·K·K multiply-adds. With C=1, K=7 (Layer 1) that's 49 MACs per output, but those 49 MACs require 49 reads from x and 49 reads from w. A naïve implementation does all 98 reads from DRAM, where each read costs hundreds of cycles. The arithmetic intensity is approximately 1 FLOP / 4 bytes loaded — the classic memory-bound regime.

Mental model Convolution is "a lot of tiny dot products that share their operands with their neighbours." The whole optimization story is about not loading those shared operands twice.

04The actual tensor shapes

Knowing the exact shapes is essential: they decide which optimizations pay off and which don't. The authors instrumented the network and printed:

LayerBCMH, WKH_out, W_outOutput pixels / image
Conv 110 000112727664 356

Three things to notice in advance, because they drive every later choice:

  1. M is small — only 12 and 24. This is the row dimension of the GEMM we'll eventually build. A 32 × 32 tile wastes most of its rows on Layer 1. That's exactly why Step 5 picks TILE_WIDTH = 16 for Layer 1 and TILE_WIDTH = 24 for Layer 2.
  2. The weight tensor is tiny — Layer 1 is 12·1·49 = 588 floats, Layer 2 is 24·12·49 = 14 112 floats. Both fit in the 64 KB CUDA constant memory. That's the entire premise of Step 3.
  3. The batch is huge (10 000). There is no shortage of independent work to throw at the GPU — the question is whether the kernel can accept that work in parallel. Step 1 fixes that.

05The naïve kernel — and why it's slow

Two files, not one The project ships two starting points. new-forward-v0.cuh is the true baseline that produces the 13.59 s number in the report. new-forward-original.cuh is a partially-optimized version that already has Steps 1–3 applied (3D grid, 16×16 tiles, constant memory). The walkthrough below uses v0 — that's the version the 124× is measured against.

The v0 kernel is brutally simple: one thread handles one entire image and walks the full six-loop for it serially. 

// new-forward-v0.cuh — the actual 13.59 s baseline

dim3 gridDim((B + 511) / 512);   // ceil(B/512) blocks
dim3 blockDim(512);              // ← THIS is where "512" comes from
forward_kernel<<<gridDim, blockDim>>>(...);

// inside the kernel:
int b = blockDim.x * blockIdx.x + threadIdx.x;   // one thread == one image

if (b < B) {
    for (int m = 0; m < M; m++)               // every output map,
      for (int h = 0; h < H_out; h++)         // every output row,
        for (int w = 0; w < W_out; w++) {     // every output col,
            y4d(b,m,h,w) = 0;
            for (int c = 0; c < C; c++)
              for (int p = 0; p < K; p++)
                for (int q = 0; q < K; q++)
                    y4d(b,m,h,w) += x4d(b,c,h+p,w+q)
                                  * k4d(m,c,p,q);
        }
}

The 512 is just the chosen thread-block size. With B = 10 000 it launches only ⌈10 000/512⌉ = 20 blocks of 512 threads each — about 10 000 threads total — to do all the work.

What this kernel is actually doing

Each of those ~10 000 threads runs the full six-loop for its one image. For Layer 1 that's M·H_out·W_out·C·K·K = 12·66·66·1·49 ≈ 2.6 M multiply-adds, performed by a single thread, in sequence. For Layer 2 it's 24·27·27·12·49 ≈ 10.3 M per thread.

Why this is so slow, mechanistically

  • The GPU is almost empty. Volta has ~80 SMs × 2 048 resident threads = ~160 000 thread slots. v0 launches ~10 000 threads total — about 6 % occupancy. The remaining 94 % of the hardware sits idle.
  • Each thread's work is serial. The 2.6 M MACs for one Layer 1 image are done one after another — there is no parallelism within an image. You've taken an embarrassingly parallel problem and serialised the inside of it.
  • Accumulator lives in DRAM, not a register. Look at the inner body: y4d(b,m,h,w) += .... That += is a read-modify-write to global memory for every single MAC — 2.6 M DRAM round-trips per Layer 1 image. Hundreds of cycles each. This alone explains most of the 13.59 s.
  • Zero data reuse. Adjacent output pixels share ~6/7 of their input window when K = 7, but the kernel re-fetches each input pixel ~K² = 49 times. The filter weights are re-fetched H_out·W_out times per m.
  • Warp divergence is harmless here — but only because every thread is doing the same massive sequential work. The "warps" are effectively running 32 independent serial programs side by side.
The diagnosis in one line v0 is slow because it expresses the problem as ~10 000 independent serial programs instead of ~500 million parallel multiply-adds. Step 1 doesn't "optimize" anything — it just lets the GPU see all the parallelism that was always there.

06Step 1 · Exploit parallelism (3D grid) 73× from baseline

Time: 13.589 s → 0.186 s. This single step buys you ~99% of the eventual win — and it costs almost nothing. The trick is just to launch enough threads.

The mechanism

Reinterpret the work as a 3D grid: (tile_x_y, batch, output_map). Now the GPU sees B · M · ⌈H_out·W_out / TILE²⌉ blocks of work and can schedule them across all SMs simultaneously. Each block uses TILE×TILE threads, one thread per output pixel instead of one thread per image. 

// grid: spatial tiles along x, batch along y, output-map along z
dim3 gridDim(ceil(H/TILE_WIDTH) * ceil(W/TILE_WIDTH), B, M);
dim3 blockDim(TILE_WIDTH, TILE_WIDTH, 1);

forward_kernel<<<gridDim, blockDim>>>(y.dptr_, x.dptr_,
                                     B, M, C, H, W, K);

The parallelism multiplier

The whole improvement is captured in this one comparison:

v0 (baseline)Step 1Multiplier
Total threads launched ~10 000 (one per image) B·M·H_out·W_out ≈ 522 M (L1) / 175 M (L2) ~50 000× more
Serial work per thread M·H·W·C·K·K ≈ 2.6 M ops (L1) C·K·K = 49 ops (L1) 52 000× less
Accumulator location Global memory (R-M-W per MAC) Register (float pValue) ~100× cheaper per MAC

Why this is the biggest single win

  1. Occupancy explodes. v0 used ~6 % of the GPU's thread slots; Step 1 fills them many times over. The GPU is finally a parallel machine. Threads that stall on memory get hidden behind other threads that are issuing arithmetic.
  2. The accumulator moves to a register. v0 did y4d(...) += x*w — a read-modify-write to DRAM for every MAC. Step 1 keeps the running sum in float pValue and writes y exactly once at the end. That alone eliminates ~2.6 M global-memory round trips per image on Layer 1.
  3. Per-thread work becomes warp-friendly. Each thread now does 49 (Layer 1) or 588 (Layer 2) MACs instead of millions. Warps finish at predictable rates, the scheduler can pack them densely.
Why so much speedup so fast The naïve kernel wasn't slow because of bad memory access patterns — it was slow because almost no threads were running. The GPU is a latency-hiding machine; if there's nothing to switch to, every DRAM stall is exposed. Filling the machine recovers a 70× factor before any "real" optimization has been done.

Caveats

  • You now do more arithmetic — every thread re-reads the same input pixels its neighbours need. We haven't fixed reuse yet.
  • The 6-loop inner body is still pulling from global memory every iteration. That's the next four optimizations.

07Step 2 · Sweep the tile width 0.186 s → 0.144 s

"Choose the right block size" sounds boring; it isn't. Tile width controls three independent things at the same time:

  • Threads per block, which determines how many warps a block contributes to its SM.
  • Register pressure per thread (more threads ⇒ fewer registers each ⇒ possible spill).
  • Shared-memory footprint per block (only relevant once we add tiling — see Step 4).

The authors swept three sizes:

TileThreads / blockTotal timeVerdict
32 × 3210240.186 sHits the per-block max; only 2 blocks per SM ⇒ low occupancy.
16 × 162560.170 sSweet spot: 8 blocks per SM = 2048 threads — full occupancy.
8 × 8640.220 sToo small — warp scheduler runs out of work, launch overhead dominates.
The mechanism Volta SMs cap at 2 048 resident threads and 32 resident blocks. 16 × 16 blocks contain 256 threads (= 8 warps), so you can fit exactly 2 048 / 256 = 8 blocks per SM — the maximum useful number. At 32 × 32 you get only 2 blocks; at 8 × 8 the per-block overhead dominates.

The lesson is that "more threads per block" and "more total threads" are different axes. You want enough threads to amortise per-block scheduling cost, but small enough that several blocks fit per SM so that when one block stalls on memory another can run.

08Step 3 · Weights into constant memory 0.144 s → 0.141 s

The filter bank w has two properties that make it a perfect candidate for constant memory:

  • It's read-only during a forward pass.
  • Every thread reads exactly the same address at the same time — when thread (tx, ty) is processing output map m, every other thread on that block is also using w[m, c, p, q] at the same (c, p, q) step of the inner loop.

CUDA constant memory is 64 KB of off-chip DRAM with a dedicated on-chip cache. The cache is optimised for broadcast: when every thread in a warp reads the same constant address, the hardware serves it in a single cycle — like a register, but shared across all threads. 

// reserve room — 64 KB / 4 bytes − 20 spare floats for safety
#define CMem_SIZE (64 * 1024 / 4 - 20)
__constant__ float k[CMem_SIZE];

...

// once per forward pass, before the kernel launch
cudaMemcpyToSymbol(k, w.dptr_, M * C * K * K * sizeof(float));

Why it actually helps here

Layer 1's weights are 588 floats and Layer 2's are 14 112 — comfortably inside 64 KB. Every thread in the same warp reads the same k[m, c, p, q] at the same inner-loop step, so the constant cache broadcasts a single load to all 32 lanes. That's a factor of 32 in read traffic for the filter, and you avoid trashing the L1 cache (which is now free to hold input pixels).

Caveat Constant-memory broadcast only beats global memory when the access is uniform within a warp. If different threads in a warp read different constant addresses, the loads serialize — one cycle per distinct address. Constant memory is not "free fast memory"; it's a broadcast device.

The gain is small in absolute terms (3 ms) because by Step 3 the kernel is no longer filter-bandwidth-bound — but the path it opens, freeing the L1 cache and reducing pressure on global memory, is what makes Step 4 work.

09Step 4 · Unroll + tiled GEMM 0.141 s → 0.128 s The conceptual big idea

This is the technique that every production framework eventually settles on: convolution as matrix multiplication. It is also sometimes called im2col + GEMM. The trick: rearrange the input so that the convolution becomes a single matrix product, then apply 50 years of GEMM optimization to it.

Step 4a · The reshape (im2col, implicit)

Think of the filter bank as a matrix W of shape M × (C·K·K) — each row is one filter, flattened. Think of the input as an unrolled matrix X_unroll of shape (C·K·K) × (H_out·W_out) — each column is one receptive field (the K×K×C patch that produces one output pixel), flattened.

W X_unroll Y ┌─────────────────┐ ┌─────────────────────┐ ┌────────────────┐ │ M × (C·K·K) │ × │ (C·K·K) × (H_o·W_o)│ = │ M × (H_o·W_o) │ └─────────────────┘ └─────────────────────┘ └────────────────┘ each row = one each col = one each cell = one flattened filter flattened input window output pixel

The cleverness in this code: X_unroll is never materialised. There is no separate cudaMalloc for it. Instead, when a thread needs X_unroll[row, col] it computes back what the original x[b, c, h+p, w+q] coordinate would have been and loads from there directly: 

// reverse the im2col mapping on the fly
int X_c = row_t / Ksq;       // which input channel
int X_p = (row_t % Ksq) / K;  // kernel row
int X_q = (row_t % Ksq) % K;  // kernel col
int X_h = col / W_out;        // output row → input row base
int X_w = col % W_out;        // output col → input col base

if (row_t < unrollColumn && col < H_out*W_out)
    X_tile[ty][tx] = x4d(X_b, X_c, X_h + X_p, X_w + X_q);
else
    X_tile[ty][tx] = 0;

Layer-2 kernel · the four lines above are im2col compressed into address arithmetic.

This saves the memory and bandwidth that an explicit im2col would have spent rewriting x into a much larger buffer (the expansion factor is K² = 49×).

Step 4b · The tiled matrix multiply

Once you've expressed the work as a matrix product, you use the classic shared-memory tiled GEMM: each block computes a TILE × TILE tile of the output by streaming TILE × TILE sub-tiles of W and X_unroll through shared memory. 

__shared__ float W_tile[TILE_WIDTH][TILE_WIDTH];
__shared__ float X_tile[TILE_WIDTH][TILE_WIDTH];

float pValue = 0.0;
int num_blocks = ceil(unrollColumn / (1.0 * TILE_WIDTH));

for (int i = 0; i < num_blocks; i++) {
    // every thread loads ONE element of W and ONE of X_unroll
    // into shared memory (see im2col arithmetic above)
    ...
    __syncthreads();

    // each thread now performs TILE_WIDTH multiply-adds
    // against data that lives in shared memory, not DRAM
    for (int j = 0; j < TILE_WIDTH; j++)
        pValue += W_tile[ty][j] * X_tile[j][tx];

    __syncthreads();
}

Why this works — the arithmetic-intensity argument

Without tiling, each multiply-add reads 2 floats from global memory ⇒ 8 B per FLOP. With TILE_WIDTH = 16, each block loads 2·16·16 = 512 floats from DRAM and performs 16·16·16 = 4 096 FLOPs ⇒ 8 FLOPs per byte. That's a 32× reduction in DRAM traffic per FLOP, which moves the kernel from memory-bound toward compute-bound territory.

Without tiling With TILE = 16 ───────────────────── ────────────────────────── FLOP : byte = 1 : 8 FLOP : byte = 8 : 1 (32×) Each pixel re-loaded Each pixel re-loaded ~K² ≈ 49 times ~K²/TILE ≈ 3 times

Why "implicit im2col" beats explicit im2col here

  • No extra DRAM write/read pass to materialise X_unroll (which would be ~49× larger than X).
  • The address arithmetic is a few integer ops per shared-memory load — cheap compared to the avoided DRAM transfer.
  • The accumulator stays in a register through the whole inner loop.
Caveat: bank conflicts The expression W_tile[ty][j] * X_tile[j][tx] is a typical tiled-GEMM access pattern. For TILE_WIDTH = 16 on a 32-bank shared memory, this is conflict-free for X_tile but can have 2-way conflicts for W_tile depending on the stride. Padding the shared array to [TILE][TILE+1] is a common one-line fix not used here.

10Step 5 · Two kernels, two tile sizes 0.128 s → 0.110 s

With the GEMM in place, the kernel's performance now depends on how well the tile shape matches the matrix shape. The two layers have very different shapes:

Layer 1 GEMM
  • M = 12 rows
  • C·K·K = 49 inner
  • H·W = 4 356 cols
  • Best tile: TILE_WIDTH = 16
Layer 2 GEMM
  • M = 24 rows
  • C·K·K = 588 inner
  • H·W = 729 cols
  • Best tile: TILE_WIDTH = 24

A single 32 × 32 tile wastes 5/8 of its rows on Layer 1 (only 12 of 32 are real) and 1/4 on Layer 2. A 24 × 24 tile fits Layer 2 exactly: M = 24 means the row dimension is fully used and no boundary check is wasted. 

if (M <= 16) {
    // Layer 1: M = 12, use a 16x16 tile
    dim3 blockDim(TILE_WIDTH, TILE_WIDTH, 1);            // 16x16 = 256
    dim3 gridDim(ceil(H_out*W_out/16.0),
                 ceil(M/16.0), B);
    forward_kernel_layer2<<<gridDim, blockDim>>>(...);
} else {
    // Layer 2: M = 24, use a 24x24 tile that fits M exactly
    dim3 blockDim(TILE_WIDTH2, TILE_WIDTH2, 1);           // 24x24 = 576
    dim3 gridDim(ceil(H_out*W_out/24.0),
                 ceil(M/24.0), B);
    forward_kernel_layer1<<<gridDim, blockDim>>>(...);
}

Why 24 and not 32 for Layer 2?

  • Row fit: M = 24 fits in one row-tile. With 32, eight threads per row would be idle and you'd need to bounds-check them.
  • Occupancy: 24×24 = 576 threads/block lets ~3 blocks fit per SM (under 2048 max). 32×32 = 1024 lets only ~2 — less latency hiding.
  • Shared-mem footprint: 2·24·24·4 B = 4.5 KB/block fits comfortably; 32×32 would need 8 KB and bump up against per-SM limits.
The shape-matching principle Generic kernels are convenient; tuned kernels are fast. The 15% win in this step comes purely from acknowledging that the two layers are different. Production libraries take this to the extreme — cuDNN ships dozens of kernel variants and picks one at runtime based on the exact shape, dtype, and hardware generation.

11Caveats & what's left on the table

0.11 s is good but not where the hardware can take this. Concrete next steps, roughly in order of expected payoff:

TechniqueEstimated winWhat it changes
FP16 / TensorCore 3–8× Use wmma intrinsics to run the inner GEMM on Volta TensorCores. 4×4×4 FP16 MAC blocks at full throughput.
Register tiling 1.5–2× Make each thread compute 2×2 or 4×4 output elements instead of one. Re-uses register-held operands and reduces shared-memory pressure.
Double buffering 1.2–1.4× Two shared-memory tiles per buffer; load the next tile while computing on the current one. Hides DRAM latency more aggressively.
Padded shared memory 1.05–1.15× __shared__ float tile[TILE][TILE+1] to kill bank conflicts.
Asynchronous copy (cp.async) 1.1–1.3× Ampere+ only. DRAM → shared without going through registers.
Winograd / FFT conv 1.5–3× For 3×3 filters, Winograd F(2,3) reduces MAC count by 2.25×. Less useful at K = 7 but worth knowing.
Filter in registers 1.1–1.3× Layer 1's filter is 588 floats — small enough to cache entirely in a per-block register file. Eliminates the W shared-memory tile.

Hidden assumptions worth questioning

  • The current Step-4 kernel uses row-major X_tile loads driven by the unrolled index, which can lead to non-coalesced global reads when adjacent threads land on non-adjacent input pixels. Profiling with nvprof --metrics gld_efficiency would tell you exactly how much.
  • The if (M <= 16) dispatch is fragile — any future layer with M = 8 or M = 48 falls onto a wrong-sized kernel. A real production pipeline does either runtime auto-tuning or template specialization on common shapes.
  • Constant memory was kept for the simple kernels but dropped in the GEMM kernels — the GEMM versions read W from global memory through shared. Re-introducing constant memory for W would cost nothing and reduce global-load traffic by ~15% on Layer 1.

12Raw CUDA vs cuDNN / cuBLAS — which approach is better?

Short answer: for production, use the libraries. For learning, building a custom kernel, or for shapes the libraries handle poorly, write your own. Here is what's actually happening under the hood in each case.

cuBLAS — the GEMM monolith

cuBLAS provides industrial-strength GEMM (and a few cousins). What it actually does is essentially Step 4 of this report, but pushed orders of magnitude further:

  • Multi-level tiling: a hierarchy of block tile → warp tile → thread tile, each carrying its own register-resident operand cache.
  • TensorCore wmma instructions for FP16/BF16/TF32 — performs a 16×16×16 matrix MAC in one instruction.
  • Double / triple buffered cp.async pipeline so a tile's compute fully overlaps the next tile's load.
  • Heuristic kernel selection based on M, N, K, dtype, transpose state, and SM generation — picks one of dozens of pre-tuned variants.

To use cuBLAS you'd still need an explicit (or implicit) im2col first, because cuBLAS doesn't know about convolutions: 

// (1) materialise X_unroll, then (2) call SGEMM
im2col_kernel<<<...>>>(x, X_unroll, B, C, H, W, K);

cublasSgemm(handle,
            CUBLAS_OP_N, CUBLAS_OP_N,
            H_out*W_out, M, C*K*K,   // dims
            &alpha,
            X_unroll, H_out*W_out,
            W,        C*K*K,
            &beta,
            y,        H_out*W_out);

cuDNN — the convolution monolith

cuDNN is a convolution-specific library that picks which algorithm to use under the hood, including:

AlgorithmWhen it's chosenWhat it is
IMPLICIT_GEMMMost shapes — the safe default.Exactly Step 4 here: implicit im2col + GEMM, no extra buffer.
IMPLICIT_PRECOMP_GEMMWhen K is small and reused.Pre-computes index tables to skip address arithmetic.
GEMMLarge K, irregular shapes.Explicit im2col into a workspace buffer, then cuBLAS.
DIRECTTiny shapes where GEMM has too much overhead.A custom direct-conv kernel.

FFT_TILINGFFT but per-tile to fit memory.Same idea, tiled.

The selection is done either by heuristic (fast, can be suboptimal) or by the cudnnFindConvolutionForwardAlgorithm() benchmarker (slow, exhaustive — runs each algorithm on dummy data and times them). PyTorch's torch.backends.cudnn.benchmark = True is exactly this.

The honest answer Use cuDNN/cuBLAS by default. Drop to a hand-written kernel only when profiling shows the library leaving real performance on the table — usually for unusual shapes, fused epilogues (conv → bias → ReLU → quantize as one kernel), or when you need bit-for-bit determinism. The real research frontier is somewhere in between: tools like CUTLASS, Triton, and CUDA Graphs let you compose library-grade kernels with the fusion freedom of hand-written code.

What that means for LeNet-5 specifically

On this exact workload, cuDNN's IMPLICIT_PRECOMP_GEMM or WINOGRAD would land in the 30–60 ms range — a further 2–3× over the hand-tuned 110 ms here, mostly thanks to TensorCore use and aggressive double-buffering. The hand-tuned kernel's value isn't that it's faster — it's that you understand every cycle of what it's doing, and that the same five techniques are exactly what's hiding inside those library calls.
