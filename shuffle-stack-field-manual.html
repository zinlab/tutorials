Who plans it, who moves it, who computes it

A systematic walk through one distributed analytical query — from the SQL string to the all-to-all shuffle and back — across the exact stack you work in: a Presto coordinator on top, Velox executing underneath it, cuDF doing the GPU compute, and an experimental UCX exchange moving partition buffers GPU-to-GPU between workers. Who provides the node list, who hands out the work, who pulls which portion, how each layer calls the one below and serves the one above — and where all of this rhymes with, or departs from, MPI.

§0

OrientationThree planes, one shuffle

The hardest moment in a distributed analytical query is not the scan or the join — it is the shuffle: the all-to-all redistribution of data between stages, where every producer must hand a slice of its output to every consumer so that all rows sharing a key end up on the same machine. Everything in this manual exists to answer one question precisely — when a query reaches that moment, who does what? To keep the answer straight, hold onto a single organizing idea: the stack is split into three planes, and almost every component belongs cleanly to one of them.

The control plane decides who, what, and where. It knows the list of live machines, turns SQL into a plan, cuts the plan into stages, and tells each worker which fragment to run, over which input, and where to send its output. In this stack the control plane is a single actor: the Presto coordinator. The data plane moves the bytes — it is the machinery that physically carries partitioned data from producers to consumers. Here it is a layered substack: Velox's exchange operators on top, the experimental ucx-exchange adapter beneath them, and UCX/UCXX at the bottom driving the NIC and the GPU. The compute plane transforms the bytes — the actual filters, joins, and group-bys — and in a GPU build that is cuDF / libcudf, running CUDA kernels over columnar data resident in GPU memory.

The single most useful fact in this whole manual: the control plane is unchanged by any of the GPU and RDMA work. Swapping the shuffle transport from HTTP to UCX, and the compute from CPU to cuDF, touches only the data and compute planes. The coordinator still hands out the same node list, the same tasks, the same partition scheme. That clean seam is why a research transport like ucx-exchange can drop into a production engine at all.

The questions this manual answers, in order

Who provides the node list? Who fixes the partition count and the hash scheme? How does a downstream task learn which upstream tasks to pull from, and which portion of their output is its share? How does each layer call the one below it and serve the one above it? When data actually moves, who initiates — the sender or the receiver? And at each step, how does the answer compare to how MPI would do it? Sections build strictly upward: control plane first (§2–§3), then the engine (§4), then the shuffle in the abstract (§5), then the real UCX implementation (§6–§8), then the whole thing end-to-end (§9), then MPI (§10).

§1

The mapWho sits where — the position diagram

Before any procedure, fix the geography. The diagram below is a single worker, sliced top to bottom along the path a row of data travels: it enters at the execution engine, gets partitioned, is handed to the transport adapter, and leaves through the NIC or NVLink. Two things are layered on top of that spine — the control plane reaching in from above (dashed), and the compute plane (cuDF) hanging off the side of the engine.

Reading the diagram: calls down, serves up

The vertical axis is a literal call stack. Within a worker, each layer depends on (calls) the layer beneath it, and serves (returns results to) the layer above it. Top to bottom: the SQL client calls the coordinator; the coordinator calls each Prestissimo worker; Prestissimo builds and calls a Velox Task; the Task's drivers call operators; an operator that needs to compute calls into cuDF kernels; an operator that needs to move data (PartitionedOutput, or Exchange) calls the ucx-exchange adapter, which calls UCXX, which drives the NIC. Results flow back up the same chain in reverse — the NIC's bytes become a cuDF table, which an operator consumes, which a driver pulls, which the Task reports, which Prestissimo returns to the coordinator, which streams to the client.

There is one important place where the simple “calls down” picture inverts, and it is the crux of the whole shuffle. Between two workers, the data plane is consumer-initiated. The producer does not push its partitions outward; the downstream Exchange operator pulls — it opens a connection to each upstream worker and requests its share. So while control flows top-down from the coordinator, the cross-worker data edge is a demand-driven pull from the bottom of the consumer's stack to the bottom of the producer's. Holding those two directions apart — control pushes down from the coordinator, data is pulled sideways by consumers — is most of what it takes to understand the sections that follow.

Why split the planes this cleanly

Because each plane can then evolve independently. The coordinator's scheduling logic, Velox's operator set, the transport, and the compute kernels are four separate engineering surfaces with stable interfaces between them. ucx-exchange is the proof: it reimplements only the transport, leaving planning and operators untouched.

What it costs

Two networks and two membership notions to keep coherent — the HTTP control mesh and the UCX data mesh — plus a translation boundary (Presto plan → Velox plan) that must be kept in lock-step as both systems evolve. MPI, by contrast, collapses all of this into one static communicator (§10), trading flexibility for simplicity.

§2

Control planeThe Presto coordinator — the only actor that decides

Nothing below the coordinator decides anything about who or where. Workers run the fragment they are handed, over the input they are handed, and emit to the destinations they are handed. So every “who provides X” question in this manual has, at bottom, the same answer for the structural decisions: the coordinator. What follows is the precise order in which it makes those decisions for one query, and exactly what it hands to each side of a shuffle.

1 · Membership — where the node list comes from

The node list is not configured statically; it is discovered. Each worker, on startup and then on a heartbeat, announces itself to the coordinator's discovery service. The coordinator's node manager maintains the live set, adding workers that check in and evicting ones that go silent. When the scheduler needs “the workers,” it reads this set. That is the entire source of truth for membership — dynamic, heartbeat-maintained, and elastic: a worker can join mid-cluster-life and be scheduled on the next query, or drop out and be removed. (Hold this thought for §10: MPI's membership is the polar opposite — a fixed set of ranks frozen at launch.)

2 · From SQL to a fragmented plan

The coordinator parses the SQL, analyzes it against the catalog (resolving tables, columns, and types), builds a logical plan, and runs the cost-based optimizer (join ordering, predicate pushdown, partition pruning). The optimizer then decides where data must be repartitioned — at each such point it inserts an exchange boundary, and the plan is cut there into plan fragments, one per stage. A fragment that consumes a shuffle has a remote-source (exchange) node at its leaves; a fragment that produces one ends in a partitioned-output node. Crucially, for each adjacent pair of stages the coordinator fixes the partition scheme: the partition function (hash on the shuffle keys) and the partition count P.

3 · Tasks, splits, and the exchange wiring

For each stage the coordinator chooses a set of workers from the node list and creates one task per worker. It then feeds tasks their splits — the granular units of input work. For a leaf scan, a split is a file or byte-range; the coordinator's split source doles them out and hands more to a task as it drains its queue, which is how load naturally balances across uneven workers. For a shuffle-consuming stage, the “splits” are something subtler and central to this whole story: each is a remote split — the location (URI) of one upstream producer task. The diagram shows what the coordinator tells each side.

So the answer to “how does a downstream task know which portion of the work is its share” is fully a control-plane answer: the coordinator assigns each consumer a partition id i and the list of every producer's location. The consumer's share is partition i, gathered from every producer. The producers were independently told to bucket their output into P partitions by the agreed hash. Neither side negotiates this; both are configured by the coordinator.

4 · TaskUpdate, lifecycle, and results

All of the above is delivered as an HTTP TaskUpdate to each worker: the serialized plan fragment, the current batch of splits, the output-buffer layout (how many partitions, drained by whom), and session properties. Over the query's life the coordinator sends more splits, then a “no more splits” marker, then a finish signal; it polls each task's status over HTTP and collects the final stage's output to stream back to the client. Failure handling rides this same channel — and in classic Presto a lost task fails the query, since there is no mid-query checkpoint (Velox/Presto's recoverable-exchange work softens this, but the default model is restart-the-query). Every byte of this is HTTP/REST and entirely separate from the UCX data mesh below.

The coordinator is a choreographer, not a courier. It computes the node list, the fragments, the splits, and the partition map, and ships those instructions over HTTP — but the actual partitioned data never passes through it. That separation is exactly what lets the data plane be re-implemented on RDMA without the control plane noticing.
§3

The bridgePrestissimo — turning a Presto task into a Velox dataflow

The coordinator speaks “Presto.” The engine that actually runs speaks “Velox.” Prestissimo is the membrane between them — the C++ Presto worker (the “native worker”) that accepts the coordinator's HTTP TaskUpdates unchanged, and translates each Presto plan fragment into a Velox plan it can execute. Because Prestissimo honours the exact same REST protocol the old Java worker did, the coordinator is none the wiser that C++, Velox, and a GPU are now underneath.

When a TaskUpdate arrives, Prestissimo's task manager creates a task and runs the plan converter, walking the Presto node tree and emitting the corresponding Velox PlanNode tree. The mapping is mostly one-to-one, and two rows of it are the ones that matter for the shuffle:

Presto plan nodeVelox plan nodeBecomes the operator…
TableScanNodeTableScanNodeTableScan (+ a connector; cuDF scan in a GPU build)
FilterNode · ProjectNodeFilterNode · ProjectNodeFilterProject
AggregationNodeAggregationNodeHashAggregation (partial / final)
JoinNodeHashJoinNodeHashBuild + HashProbe
PartitionedOutputNodePartitionedOutputNodePartitionedOutput — the producer end of the shuffle
RemoteSourceNode (exchange)ExchangeNodeExchange — the consumer end of the shuffle

The split flow is the other half of the bridge. Splits arrive over HTTP and Prestissimo adds them to the Velox task. For a scan, they are file splits. For an exchange, each remote split carries an upstream producer URI — and these become Velox remote splits fed to the Exchange operator. This is the concrete mechanism by which a consumer learns its sources: the coordinator's “here are the producer URIs” (§2) is delivered through Prestissimo as splits, and the Exchange operator spins up one connection per split. Prestissimo then constructs the Velox Task, sets up its output buffers, and launches the driver threads that run the operators.

Where the GPU and UCX swap-ins are registered

A native worker built for this stack registers its extensions at startup — the cuDF-backed operators (so TableScan, HashAggregation, joins run as libcudf kernels) and the ucx-exchange transport (the UCX ExchangeSource factory and the UCX partitioned-output path). The plan converter is unchanged; it still emits a generic ExchangeNode and PartitionedOutputNode. What differs is which implementation those nodes resolve to at run time — the registered UCX and cuDF ones. The README's test binaries make this concrete: 1brc_server and 1brc_client launch with -velox_cudf_enabled=true and the UCX exchange compiled in, which is exactly this registration happening in a standalone harness rather than inside Prestissimo.

Net effect: Prestissimo converts control (an HTTP description of a fragment, splits, and destinations) into a live dataflow (a Velox task of driver threads running operators). From here down, the coordinator is out of the picture until results are reported — the next three sections live entirely inside the worker and on the wire between workers.

§4

The engineVelox — how one task runs, and where the shuffle attaches

Inside the worker, a stage is a Velox Task: a tree of Operators, run by a pool of Driver threads, grouped into Pipelines. Data moves between operators as RowVectors — columnar, Arrow-compatible batches of order a thousand rows, which is the vectorized-execution heritage that makes a per-row virtual call into a per-batch one. Understanding the shuffle means understanding the two operators that sit at a task's boundary: the one that emits a stage's output, and the one that sources the next stage's input.

A Driver runs a chain of operators in a tight loop — pull a RowVector from below, push it through addInput / getOutput, repeat — with back-pressure expressed as a future an operator returns when it must wait. Parallelism within the worker is just the driver count: k drivers run the same pipeline over different slices of the input. The plan's leaves are source operators (a TableScan, or an Exchange); its root is a terminal operator (a PartitionedOutput, or the final result collector).

The two shuffle operators, in essence
// PRODUCER end — terminal operator of stage S.
class PartitionedOutput {
  void addInput(RowVector batch) {
    for (row : batch) {
      auto p = hash(row[keys]) % P;        // P = partition count, fixed by coordinator
      partitionBuffer[p].append(row);        // route row into its bucket
    }
    for (p : 0..P) outputManager.enqueue(p, serialize(partitionBuffer[p]));
  }                                          // serialize = SerializedPage (HTTP) OR cuDF packed_columns (UCX)
};

// CONSUMER end — source operator of stage S+1, this task owns partition i.
class Exchange {
  RowVector getOutput() {
    page = exchangeClient.next();             // pull next page from ANY ready source
    return deserialize(page);                 // back into a RowVector / cuDF table
  }                                          // exchangeClient holds one ExchangeSource per upstream producer
};

The OutputBufferManager on the producer side holds the per-partition buffers and serves them; the ExchangeClient on the consumer side holds one ExchangeSource per upstream producer and pulls pages as its downstream is ready. Two properties of this loop matter enormously for the comparison to come. First, it is demand-driven: the consumer pulls; the producer fills buffers up to a memory budget and then blocks (back-pressure), so a slow consumer throttles a fast producer rather than overrunning it. Second, it is pipelined: a consumer can begin draining a producer's early output while that producer is still computing later batches — there is no global barrier between the stages. (Both properties are precisely where the MPI collective model diverges, §10.)

Memory, the GPU, and spilling

Every operator runs under a memory budget. When a working set exceeds it, Velox spills — and in a cuDF build, where the “fast tier” is finite GPU HBM allocated from an RMM pool, the pressure point is real: a build-side hash table or a pile of in-flight partition buffers can exceed device memory, forcing a spill to host memory. That host-spilling path — keeping the shuffle correct when the working set outgrows the GPU — is one of the genuinely hard parts of making this stack production-grade, and it lives right here at the boundary between the operators and the exchange buffers.

§5

The shuffleAll-to-all, abstractly — and the seam the transport plugs into

Zoom out from one task to the whole stage boundary and the shuffle's shape appears. A producer stage has N tasks; a consumer stage has P. Every producer buckets its rows into P partitions; every consumer must gather its one partition from all N producers. That is an all-to-all: up to N×P independent transfers, and it is the single most expensive, most network-bound moment in the query.

The seam: a pluggable transport behind a fixed operator

Here is the design decision that makes everything in §6–§8 possible. Velox provides the Exchange operator and an ExchangeClient, but it deliberately does not hard-code how bytes cross the network. Instead it defines an abstract ExchangeSource and a factory registry; the operator asks the registry to build a source for each remote split's URI. The default registered implementation pulls serialized pages over HTTP. Swap the registered factory, and the very same operator now moves data over something else entirely — which is exactly what ucx-exchange does.

Velox's transport seam — the extension point ucx-exchange targets
// Velox core — the abstraction the Exchange operator pulls through.
class ExchangeSource {
  virtual void request(...) = 0;       // ask one upstream producer for the next chunk
  virtual void close() = 0;
  // a factory turns a remote-split URL into a concrete source:
  static void registerFactory(Factory);
};

// Default build: HTTP. Each source GETs pages from the producer's endpoint.
// UCX build: ucx-exchange registers its own factory, so the SAME Exchange
// operator now creates UcxExchangeSource objects that move GPU buffers over UCX.
ExchangeSource::registerFactory(ucxExchangeSourceFactory);

The README for the module states this division exactly: in Velox the inter-worker exchange is only partially implemented — the operator and the abstraction are there, but the exchange server and the concrete source object are left out of Velox proper, to be supplied by a transport. ucx-exchange supplies both. Everything the coordinator decided (§2) and Prestissimo wired (§3) is unchanged; only the thing on the other side of this ExchangeSource interface is different. The next section is what lives there.

The whole GPU-RDMA effort threads through a single virtual interface. The Exchange operator never learns it is using RDMA — it just calls request() on an ExchangeSource and receives chunks. That is the payoff of the plane separation from §0: the data-plane transport is a swappable implementation detail behind a stable seam.

§6

The cruxucx-exchange — what actually lives behind the ExchangeSource

This is the module you pointed at, and the reason the rest of the manual is scaffolding. ucx-exchange is the concrete transport that fills the seam from §5: it supplies both halves Velox leaves out — the producer-side server and the consumer-side source — and it moves cuDF partition buffers directly from GPU memory to GPU memory, built on UCXX (the RAPIDS C++ wrapper over UCX that cuDF and dask-cudf also use). Its own README states the division precisely: in Velox the inter-worker exchange is only partially implemented, and the server and source objects are out of scope for Velox proper — so this module implements all of it.

The two chains, and the engine they share

The module factors into a producer chain, a consumer chain, and shared connection machinery:

SidePipeline of objectsJob
ProducerUcxPartitionedOutput → UcxOutputQueueManager → UcxExchangeServer (+ Acceptor)Partition rows into per-destination queues of packed cuDF columns; serve each partition to whichever consumer connects for it.
ConsumerUcxExchange → UcxExchangeClient → UcxExchangeSourceOne UcxExchangeSource per remote producer task; each pulls this consumer's partition from that producer.

The UcxPartitionedOutput operator is the UCX-flavoured replacement for Velox's PartitionedOutput (§4): instead of serializing rows into HTTP pages, it packs each partition into a contiguous cuDF buffer and enqueues it in the UcxOutputQueueManager, keyed by destination partition. On the other side, the consumer's Exchange resolves — through the registered factory — to a UcxExchangeClient holding one UcxExchangeSource for every upstream producer. The module's own comment is blunt about the cardinality: the source “is the client that communicates with the remote UcxExchangeServer; there is a distinct instance for every remote task.” That is the N-producers fan-in of §5 made literal — a consumer owning partition i instantiates N sources, one aimed at each producer.

Both sources and servers are CommElements registered with a single per-process Communicator — a singleton that wraps one UCXX worker, runs the progress loop, and calls process() on each element in its work queue. The Acceptor inside it listens on a port for incoming connections; an EndpointRef represents one established peer link. Notably, the Communicator also registers a UCXX active-message receiver callback (owner "velox", id 123) — so the module uses two UCX primitives in concert: active messages for connection-time control, and tag-matched send/recv for the metadata and data streams.

The wire protocol — tag matching, MPI-style

Every metadata and data message rides a UCXX tag-matched transfer, and the 64-bit tag is hand-packed into three fields. This is a direct re-creation of MPI's tag-matching, with the “tag” carrying producer identity, message kind, and sequence:

BitsWidthFieldPurpose
63..324 bytesFNV-1a-32 hash of the producing taskIdidentifies the producer task (unique in the cluster)
31..241 byteop-type — 0x02 metadata · 0x03 data · 0x04 handshake-responsewhich kind of message this is
23..03 byteschunk sequence numberorders the chunks streamed between this task-pair

There is no serialization framework: multi-byte fields are written with raw memcpy and matching endianness is assumed between peers (little-endian on x86/ARM; cross-endian transfers are explicitly unsupported). Each cuDF vector crosses as two tag-matched messages — first the metadata (the packed_columns layout/header), then the data (the contiguous GPU bytes) — which is a transfer in the same spirit as a rendezvous: describe, then move. The receiving source allocates the landing buffer from an RMM pool and streams the data into it on a CUDA stream.

Tag construction + handshake structs (from UcxExchangeProtocol.h)
// 64-bit tag = [ FNV1a-32(taskId) | op-type | sequence ]
inline uint64_t getMetadataTag(uint64_t taskHash, uint64_t seq) {
  return (taskHash << 32) | 0x02000000 | seq;     // METADATA_TAG
}
inline uint64_t getDataTag(uint64_t taskHash, uint64_t seq) {
  return (taskHash << 32) | 0x03000000 | seq;     // DATA_TAG
}

// Sent by the source (consumer) to the server (producer) right after connect.
struct HandshakeMsg {
  char     taskId[256];   // which producer task this source wants
  uint32_t destination;    // the partition this consumer owns (its "i")
  uint64_t workerId;       // random per-Communicator id => detects same process
};
struct HandshakeResponse {
  bool isIntraNodeTransfer;  // true => bypass UCXX, use IntraNodeTransferRegistry
};

The same-process fast path

Because a single Prestissimo worker runs many tasks, a producer and a consumer frequently live in the same process. The handshake detects this: each Communicator mints a random workerId at startup; if the server's workerId equals the source's, the two are co-resident and the response sets isIntraNodeTransfer = true. From then on the pair skips UCXX entirely and uses the IntraNodeTransferRegistry — a singleton where the server publish()es a std::shared_ptr<cudf::packed_columns> keyed by (taskId, destination, sequenceNumber) and the source poll()s for it, synchronised by promises/futures. No network, no copy: just a pointer hand-off of a GPU buffer. The one hard requirement is correctness of GPU ordering — the producer must synchronise its CUDA stream before publishing, so the device data is actually ready when the consumer reads it. (Across different processes on the same node, UCXX still handles it, choosing CUDA-IPC/NVLink transports; only the same-process case uses the registry.)

One transfer, start to finish

Both endpoints are driven as explicit state machines on the Communicator's work queue, which is what lets a single progress thread multiplex many concurrent transfers without blocking:

The two state machines (from the source & server headers)
// consumer — UcxExchangeSource::ReceiverState
Created → WaitingForHandshakeComplete → WaitingForHandshakeResponse
       → ReadyToReceive → WaitingForMetadata → WaitingForData
       → (WaitingForIntraNodeData) → Done

// producer — UcxExchangeServer::ServerState
Created → ReadyToTransfer → WaitingForDataFromQueue → DataReady
       → WaitingForSendComplete → (WaitingForIntraNodeRetrieve) → Done

The server is created per (endpoint, partition key), pulls cuDF vectors for its partition out of the UcxOutputQueueManager, and sends the metadata+data pair; it independently detects same-node peers by checking whether the peer IP is in the local-IP set. It depends on Velox's Task (it is serving that task's output). And the whole module is gated by a gflag, velox_ucx_exchange — flip it off and the stack falls back to the default HTTP exchange, the cleanest possible demonstration that this is a drop-in transport behind the §5 seam.

How it plugs in — and what it replaces

Two registrations make it live, both at worker startup. (1) the consumer side registers a UcxExchangeSource factory via Velox's ExchangeSource::registerFactory, so the unchanged Exchange operator builds UCX sources for its remote splits. (2) the producer side substitutes UcxPartitionedOutput + UcxOutputQueueManager + UcxExchangeServer for the stock PartitionedOutput + OutputBufferManager + HTTP page server. The coordinator's plan, the partition map, the remote-split URIs — all identical to the HTTP path. What changed is only what happens after request() is called: raw GPU buffers over UCXX instead of serialized pages over HTTP.

One real invocation, straight from the module's test harness, shows the assembled system: 1brc_server and 1brc_client (the One-Billion-Row Challenge, on a GPFS cluster) launched with -velox_cudf_enabled=true -velox_cudf_table_scan=true -velox_cudf_memory_resource=pool and UCX_TLS=^ib selecting transports — cuDF doing the scan and aggregation, ucx-exchange moving the partitions, UCX choosing the wire. The next two sections open the two boxes this one leaned on: UCX itself, and cuDF.

§7

The transportUCX & UCXX — the substrate that moves the bytes

Under ucx-exchange sits UCX — Unified Communication X, the HPC networking framework that has spent a decade moving data for supercomputers, and which MPI implementations themselves run on. ucx-exchange does not call UCX's C API directly; it uses UCXX, the RAPIDS C++ object wrapper (Context · Worker · Endpoint · Request) that also underpins cuDF's own distributed shuffle in dask. Knowing what UCX offers explains why the module is shaped the way it is.

Three layers

UCX is stratified much like the database stack above it. UCP, the protocol layer, is the high-level API: tag-matched send/recv, one-sided RMA, active messages, streams, atomics — and the logic that selects transports, chooses eager-vs-rendezvous, fragments, and stripes across rails. UCT, the transport layer, is a thin shim over each hardware path: RC/UD/DC verbs over InfiniBand, RoCE, plain TCP, shared-memory transports (CMA, xpmem, knem), and the CUDA transports (cuda_copy, cuda_ipc, GPUDirect gdr). UCS, the services layer, is the plumbing — most importantly the memory-registration cache that makes repeated zero-copy transfers cheap.

The three primitives — and which shuffle need each serves

UCP primitiveSemanticsWhere the shuffle uses it
Tag-matched send/recvTwo-sided; receiver matches an incoming message by a 64-bit tag (the MPI model)ucx-exchange's metadata + data streams — chunk seq of partition i from a given producer (§6)
One-sided RMA (put/get)Initiator reads/writes remote memory with no remote-CPU involvement, after a one-time rkey exchangeThe natural fit for a pull: a consumer could RMA-get a producer's registered buffer; also the data phase of any rendezvous

That last row is the interesting one for your line of work. A database shuffle is irregular — a consumer does not know in advance how many bytes each producer owes it — which is precisely the regime where active messages and one-sided RMA shine over a rigid pre-posted two-sided exchange. ucx-exchange currently leans on tag-matching for the bulk streams, but the AM hook and UCP's RMA surface are the levers for pushing irregular shuffle/exchange patterns further.

Rendezvous, pinning, and why RMM pools matter

UCP transfers split by size. Small messages go eager — copied inline. Large ones go rendezvous: the sender advertises the buffer's address and remote key, the receiver pulls it with an RMA get straight into the destination, then a completion fires — no intermediate copies. For RDMA to touch a buffer at all, that memory must be registered (pinned) with the NIC, which is expensive; so UCS keeps a registration cache. This is the concrete reason the stack allocates received partitions from an RMM pool: register the pool's memory once, and every subsequent GPU buffer carved from it is already pinned and ready to RDMA — the per-transfer registration cost vanishes.

GPUDirect — the reason “GPU→GPU” is literal

The phrase from the ucx-exchange README — raw vector data copied from GPU memory to GPU memory — rests entirely on GPUDirect RDMA: a kernel capability (via nvidia-peermem / dmabuf) that lets the NIC DMA directly to and from GPU HBM over PCIe peer-to-peer. With it, a partition buffer travels GPU → NIC → fabric → NIC → GPU and never touches host memory. Without it, every transfer pays a round trip through a host bounce buffer (device-to-host copy, send, host-to-device copy) — the exact tax that makes a naive HTTP shuffle of GPU data so slow. The transport UCX actually selects depends on where the two endpoints are:

UCXX is not a database technology that happens to use the GPU — it is the same RAPIDS transport that dask-cudf shuffles on, and UCX beneath it is the same substrate OpenMPI moves messages over. ucx-exchange is the point where a database engine reaches down and grabs the HPC community's transport stack wholesale. That shared substrate is what makes the MPI comparison in §10 more than an analogy.
§8

The computecuDF — what the operators delegate to, and the buffer that gets shipped

The third plane is compute, and in a GPU build it is cuDF / libcudf: the joins, group-bys, filters, and sorts run as CUDA kernels over columnar data resident in GPU HBM, with RMM managing device memory. The relationship to Velox is clean — Velox orchestrates; cuDF computes. Velox owns the task, the driver threads, the operator tree, the exchange, the memory budgets and spilling; each operator that has real work to do hands it to a libcudf kernel. A Velox HashAggregation becomes a libcudf group-by; a TableScan decodes Parquet into a cudf::table; the operator tree is the same, the kernels underneath are GPU.

What makes this matter for the shuffle is that the data being shuffled is cuDF's GPU columnar data, and cuDF supplies exactly the primitive needed to make a partition shippable in one shot: cudf::contiguous_split / cudf::pack. A cudf::table is normally a collection of separately allocated buffers — each column's values, its null mask, its offsets. That is awkward to send. pack (and contiguous_split, which partitions and packs in one pass) collapses a table into a packed_columns: a single contiguous device buffer with every column's bytes laid end-to-end, accompanied by a small host-side metadata header describing the layout.

That structure maps perfectly onto the two-message transfer of §6. The metadata message is the packed_columns header (types, sizes, offsets, null-mask positions); the data message is the one contiguous device buffer. One partition equals one packed_columns equals one (metadata, data) pair equals one tag-matched UCX transfer — no per-column transfers, and no CPU-side serialization, because the “serialization” is just a GPU contiguous repack. On the far side, the bytes land in an RMM-pool buffer and cudf::unpack reconstructs a cudf::table_view over them as a zero-copy view, ready for the next kernel.

The GPU shuffle payload — pack → ship → unpack (cuDF)
// PRODUCER — turn one partition's rows into a single shippable GPU blob.
cudf::table_view part = /* rows that hashed to partition i */;
cudf::packed_columns packed = cudf::pack(part);     // one contiguous device buffer + layout header
// contiguous_split() does this for ALL partitions at once:
//   auto parts = cudf::contiguous_split(table, splitOffsets);  // vector<packed_columns>

send_metadata(packed.metadata);   // → UCX tag [hash | 0x02 | seq] : the layout
send_data(packed.gpu_data);       // → UCX tag [hash | 0x03 | seq] : raw bytes, GPUDirect, no host bounce

// CONSUMER — receive into RMM pool, rebuild a zero-copy table_view.
rmm::device_buffer buf = recv_data_into_pool(seq);  // already-pinned pool memory (§7)
cudf::table_view tbl = cudf::unpack(metadata, buf); // no copy; just a view over received bytes
// → hand tbl to the next libcudf kernel (final aggregation, join probe, …)

RMM is the quiet enabler of all of this. A pool_memory_resource pre-grabs a large device slab and sub-allocates from it in microseconds, dodging a cudaMalloc per buffer; a cuda_stream_pool lets transfers and kernels overlap. And, as §7 noted, the pool is registered with UCX once, so every partition buffer carved from it is already pinned for RDMA. The columnar formats line up too — cuDF columns and Velox RowVectors are both Arrow-shaped — so the handoff between Velox's orchestration and cuDF's kernels is a columnar pass, not a re-encode.

What the GPU-native path buys

The CPU never touches the row data across a shuffle. Compute stays on the GPU (cuDF), the repack stays on the GPU (contiguous_split), and the move stays on the GPU (GPUDirect). The classic row-serialize-to-host-buffer-then-HTTP tax — the dominant cost when shuffling GPU data the naive way — is gone.

What it demands

Finite HBM. A pile of in-flight packed_columns plus build-side hash tables can exceed device memory, which forces the host-spilling path from §4. Keeping the GPU shuffle correct and fast under memory pressure — overlapping transfer, compute, and spill without thrashing — is the genuinely hard engineering, and it is squarely where this stack's frontier sits.

Read it alongside the diagram: control flows down from the coordinator over HTTP; data is pulled sideways between workers over UCX.

The same twelve steps, in words

  1. SQL → coordinator. The client submits SQL over HTTP. The control plane owns everything from here until results.
  2. Node list. The coordinator reads the live worker set from its discovery service — the sole source of membership.
  3. Plan & partition scheme. Parse, optimize, fragment into stages at exchange boundaries, and fix the hash partition function and count P between the producing and consuming stages.
  4. Schedule & wire. Pick workers from the node list; create a task per worker; hand producers their scan splits and “emit P”; hand consumers their remote splits (every producer's URI) and their identity (“you are partition i”). The coordinator provides all conditions, inputs, and parameters; nothing below it decides placement.
  5. Translate & launch (producer). Prestissimo converts the Presto fragment to a Velox plan, builds the Task, and starts drivers; the cuDF and ucx-exchange extensions were registered at startup. The producer scans and pre-aggregates with cuDF kernels, hashes rows mod P, packs each partition with contiguous_split, and enqueues the packed_columns in the UcxOutputQueueManager; its UcxExchangeServer + Acceptor wait.
  6. Wire sources (consumer). The consumer's Exchange operator, via the registered factory, builds one UcxExchangeSource per producer URI — the §5 fan-in made concrete.
  7. Connect & demand. Each source connects to a producer's Acceptor, sends the handshake (taskId, destination = i, workerId), learns whether the same-process bypass applies, and calls request() for the first chunk. The consumer initiates; this is a pull.
  8. Metadata. The server pulls partition i's next packed_columns from its queue and sends the layout header on a metadata-tagged message.
  9. Data. It sends the contiguous GPU buffer on a data-tagged message — GPU→GPU over GPUDirect, no host bounce (or a pointer hand-off via the registry if co-resident).
  10. Compute. The source lands the bytes in an RMM-pool buffer, unpacks a cudf::table, and the consumer runs its final aggregation or join probe with cuDF kernels on the now-gathered partition.
  11. Report. The consumer's output becomes the next stage's input (another shuffle) or, at the final stage, is reported to the coordinator over HTTP.
  12. Return. The coordinator streams the result rows to the client.

§10

The foilWhat MPI would do — and what this stack borrowed from it

An all-to-all redistribution of partitioned data is the canonical HPC operation: MPI_Alltoallv. So the sharpest way to understand this stack is to ask how MPI would run the same shuffle — and the answer splits cleanly in two. At the transport layer, the two are nearly the same system; at the control layer, they are opposites. ucx-exchange is best read as HPC's transport wearing the database's control model.

Where they are the same system

Almost everything in §6–§7 is HPC plumbing. The substrate is literally shared: OpenMPI moves its messages over UCX, and ucx-exchange moves partitions over UCX — both bottom out in UCP tag-matching, the same rendezvous protocol, the same memory-registration cache, the same GPUDirect RDMA, the same CUDA-IPC/NVLink intra-node paths. ucx-exchange's hand-packed 64-bit tag is MPI's matching mechanism reinvented — where MPI matches on (communicator, rank, tag), ucx-exchange matches on (taskHash, op-type, sequence). If you have written CUDA-aware MPI over InfiniBand, you have already used four-fifths of what this module touches.

Where they are opposites

The divergence is entirely in the control model, and it is fundamental. MPI's shuffle is bulk-synchronous: a static set of ranks all call one collective, having already exchanged the counts, and the call behaves like a barrier. Velox's shuffle is asynchronous and demand-driven: there is no global call and no barrier; each consumer independently connects to each producer and pulls its share when its own pipeline is ready, with sizes discovered on the fly and back-pressure throttling the producers. A central, asymmetric coordinator replaces MPI's symmetric collective plus static communicator — it, not a group consensus, is the source of membership, partitioning, and placement.

The database chose this for reasons MPI's scientific heritage never forced on it: query data sizes are data-dependent and skewed (you cannot pre-exchange exact counts cheaply), clusters are elastic and multi-tenant (membership shifts), commodity nodes fail routinely (a barrier-style abort is unacceptable), and latency must be hidden by pipelining (a consumer draining early output while producers still compute). MPI's model is superb for static, regular, tightly-coupled simulations; the database's for dynamic, irregular, loosely-coupled analytics. ucx-exchange is the deliberate synthesis of the two.

Why not just call a collective? (UCC and NCCL)

If a collective were wanted, UCC — the UCX collectives library, built on the very same UCP — offers an alltoallv that is the near-exact analogue of MPI_Alltoallv, and it would slot onto this transport. NCCL is the other GPU collective library, but it is tuned for the regular, dense collectives of ML training — ring/tree allreduce over equal-sized tensors — and fits the ragged, size-unknown alltoallv of a database shuffle poorly. That mismatch is precisely why a bespoke point-to-point exchange wins here: the shuffle is irregular, pipelined, and back-pressured in ways a synchronous collective cannot express. It is also the opening for your own thread of interest — active messages and one-sided RMA are strongest exactly where MPI's two-sided collective is weakest, on irregular exchange.

The lasting lesson of this whole stack: transport and control model are orthogonal. You can take HPC's transport — RDMA, tag-matching, GPUDirect, NVLink — without taking HPC's bulk-synchronous, static-membership control. ucx-exchange is the existence proof: the fastest plumbing the supercomputing world has, driven by the elastic, demand-driven, fault-tolerant brain of a distributed database.
