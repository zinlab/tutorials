1 Tensors: memory, strides, and views

How tensors are stored in memoryCore

A PyTorch tensor is a view into a contiguous block of memory (the storage) plus metadata: shape, strides, offset, and dtype. Two tensors can share the same storage (a view) — modifying one modifies the other.

# The three components of a tensor
t = torch.tensor([[1, 2, 3], [4, 5, 6]])

t.storage()    # → [1, 2, 3, 4, 5, 6] — flat contiguous memory
t.shape        # → torch.Size([2, 3])
t.stride()     # → (3, 1) — elements to skip for each dimension
t.storage_offset()  # → 0 — offset into storage

# Strides determine layout in memory:
# t[i, j] lives at storage[offset + i*stride[0] + j*stride[1]]
# t[1, 2] = storage[0 + 1*3 + 2*1] = storage[5] = 6 ✓

Views vs copies

OperationView (shares memory)?Notes
t.view(6)✓ ViewOnly works if tensor is contiguous. Fastest reshape.
t.reshape(6)View if possible, copy if notSafe fallback — always works.
t.T / t.transpose(0,1)✓ View (just swaps strides)No data movement! stride becomes (1, 3) instead of (3, 1).
t[0] (indexing)✓ View (adjusts offset + shape)Basic indexing returns views. Advanced indexing (boolean, fancy) returns copies.
t.contiguous()Copy if non-contiguous, no-op if already contiguousCreates new contiguous storage if strides don't match standard row-major order.

The stride trick to remember: A contiguous tensor has strides that are the product of dimensions to its right: for shape [2,3,4], strides are (12, 4, 1). Any tensor whose strides match this pattern is contiguous. .T swaps strides without moving data — it's just a different interpretation of the same bytes. cuDNN and many kernels require contiguous tensors, so .contiguous() calls appear frequently.

# Check format

The dtype promotion trap: Multiplying a BF16 tensor by a Python float creates a FP32 result: bf16_tensor * 0.5 → FP32. This can silently accumulate FP32 tensors inside your model, defeating mixed precision. Fix: use bf16_tensor * torch.tensor(0.5, dtype=torch.bfloat16) or rely on torch.autocast.

3 Autograd engine

The computational graph and backward passCore

Every operation on tensors with requires_grad=True records a computation graph. Calling .backward() traverses this graph in reverse to compute gradients via the chain rule.

# The graph in action
x = torch.tensor(2.0, requires_grad=True)
y = x ** 2 + 3 * x    # y = x² + 3x
y.backward()           # dy/dx = 2x + 3 = 7
print(x.grad)          # tensor(7.)

# The graph is DYNAMIC — rebuilt every forward pass
# This enables data-dependent control flow (if/else, loops)
# and is why PyTorch is "define-by-run"

# Key attributes:
y.grad_fn         # → <AddBackward0> — the operation that created y
y.requires_grad   # → True (inherited from x)
y.is_leaf         # → False (computed from another tensor)
x.is_leaf         # → True (created directly, not computed)

Critical gradient behaviors

BehaviorWhat happensImplication
Gradient accumulation.backward() adds to .grad, not replacesYou MUST call optimizer.zero_grad() before each backward. Otherwise gradients from previous steps accumulate. This is also what enables gradient accumulation by calling backward multiple times before stepping.
Graph destructionBy default, the computation graph is freed after .backward()Calling .backward() twice on the same graph raises an error. Use retain_graph=True if you need to backward through the same graph multiple times (rare — GAN training is one case).
Leaf tensors only accumulateOnly leaf tensors (parameters, inputs) have their .grad populatedIntermediate activations don't store gradients by default. Use x.retain_grad() or hooks to access intermediate gradients.
In-place operationsIn-place ops on tensors that require grad can invalidate the graphAvoid x += 1 or x.add_(1) on tensors in the graph. Use x = x + 1 instead. The error message is "one of the variables needed for gradient computation has been modified by an inplace operation."

handle.remove()  # always clean up

    print(f"Gradient norm: {grad_output[0].norm()}")

# Custom autograd function (for non-standard forward/backward)
class StraightThroughEstimator(torch.autograd.Function):
    @staticmethod
    def forward(ctx, x):
        return (x > 0).float()   # non-differentiable step function

    @staticmethod
    def backward(ctx, grad_output):
        return grad_output        # pass gradient straight through

# The module tree
class Block(nn.Module):
    def __init__(self, d):
        super().__init__()
        self.linear = nn.Linear(d, d)       # sub-module → parameters auto-registered
        self.norm = nn.LayerNorm(d)          # sub-module
        self.register_buffer('mask', torch.ones(d))  # buffer — moves with .to(device)
        self.scale = 0.5                     # plain Python attr — NOT a parameter/buffer

    def forward(self, x):
        return self.norm(self.linear(x) * self.mask * self.scale)

model = Block(512)

# Traversal methods
list(model.parameters())        # all learnable params (recursive)
list(model.named_parameters())  # (name, param) tuples
list(model.buffers())           # all registered buffers
list(model.children())          # immediate sub-modules
list(model.modules())           # all modules (recursive, including self)

# state_dict — the serializable snapshot
sd = model.state_dict()
# {'linear.weight': ..., 'linear.bias': ..., 'norm.weight': ..., 'norm.bias': ..., 'mask': ...}
# Note: 'scale' is NOT in state_dict (plain Python attribute)

# .to() moves EVERYTHING registered (params + buffers)
model.to('cuda')       # moves all parameters and buffers to GPU
model.to(torch.bfloat16)  # casts all params/buffers to BF16

The #1 module bug: unregistered tensors

If you write self.weight = torch.randn(d, d) instead of self.weight = nn.Parameter(torch.randn(d, d)), the tensor won't be in .parameters(), won't be updated by the optimizer, won't be saved in state_dict(), and won't move when you call .to('cuda'). Use nn.Parameter for learnable weights and register_buffer for non-learnable state.

    batch_size=32,

    drop_last=True,            # drop incomplete last batch

How workers work

SettingBehaviorWhen to use
num_workers=0Data loaded in main process (synchronous, blocks GPU)Debugging only. Never in production.
num_workers=NN separate processes load data in parallel. Batches are queued and prefetched.Always use >0. Rule of thumb: N = num_CPUs / num_GPUs.
pin_memory=TrueAllocates batches in page-locked (pinned) host memory → async GPU transfer via .cuda(non_blocking=True)Always when using GPU. Enables overlapping data transfer with compute.
persistent_workers=TrueWorkers stay alive after one epoch ends. Avoids 1–5s respawn cost per epoch.Always when num_workers > 0 and training for multiple epochs.

The non_blocking trap

pin_memory=True only helps if you also use .cuda(non_blocking=True) when transferring data to GPU. Without non_blocking=True, the transfer blocks the CPU, defeating the purpose of pinned memory. The pattern is: images = images.cuda(non_blocking=True). But be careful: if you access the tensor on CPU after this (before the GPU transfer completes), you'll get stale data. As long as the next operation is a GPU operation (which will synchronize on the same stream), this is safe.

7 CUDA integration

Streams, events, and asynchronous executionCUDA

GPU operations in PyTorch are asynchronous by default. The CPU queues operations on a CUDA stream and continues immediately. The GPU executes them in order on that stream.

# Default stream — all operations go here unless specified otherwise
s = torch.cuda.current_stream()

# Synchronization — wait for all GPU ops to complete
torch.cuda.synchronize()        # blocks CPU until GPU is done (expensive!)

# Events — measure GPU time without synchronizing
start = torch.cuda.Event(enable_timing=True)
end = torch.cuda.Event(enable_timing=True)
start.record()
output = model(input)
end.record()
torch.cuda.synchronize()
print(f"GPU time: {start.elapsed_time(end):.2f} ms")

# Multiple streams — overlap independent operations
s1 = torch.cuda.Stream()
s2 = torch.cuda.Stream()

with torch.cuda.stream(s1):
    a = torch.matmul(x, w1)   # runs on stream s1
with torch.cuda.stream(s2):
    b = torch.matmul(y, w2)   # runs on stream s2 (can overlap with s1)

# Synchronize streams before combining results
torch.cuda.current_stream().wait_stream(s1)
torch.cuda.current_stream().wait_stream(s2)
c = a + b  # safe — both a and b are ready
Why asynchronous matters: When you write output = model(input), the CPU returns immediately — it doesn't wait for the GPU to finish. The CPU can then prepare the next batch, queue more operations, etc. The GPU executes the queued operations in order. torch.cuda.synchronize() is the "wait for everything to finish" barrier. Calling it inside a training loop (e.g., via .item(), .cpu(), or print(tensor)) forces the CPU to wait for the GPU, destroying the overlap. This is why profiling shows sync stalls for .item() calls.

8 Memory management

The caching allocator and memory diagnosticsMemory

PyTorch uses a caching allocator that intercepts CUDA memory allocations. Instead of calling cudaMalloc/cudaFree every time (slow — 1–100ms each), it maintains a pool of pre-allocated memory blocks and reuses them.

# Memory queries
torch.cuda.memory_allocated()     # bytes currently used by tensors
torch.cuda.memory_reserved()      # bytes held by the caching allocator (≥ allocated)
torch.cuda.max_memory_allocated() # peak usage since last reset
torch.cuda.max_memory_reserved()  # peak cache since last reset
torch.cuda.reset_peak_memory_stats()  # reset peak tracking

# Detailed summary
print(torch.cuda.memory_summary())
# Shows: active allocations, inactive (cached) blocks,
# allocation retries (fragmentation), and OOM events

# When to use empty_cache()
torch.cuda.empty_cache()
# Releases ALL cached memory back to CUDA.
# ALMOST NEVER useful in a training loop — forces re-allocation.
# Valid uses: before inference after training (release training cache),
# or when switching to dramatically different tensor sizes.

# Environment variable for fragmentation control
# PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:128
# Prevents the allocator from splitting large blocks into pieces
# smaller than 128 MB. Reduces fragmentation at the cost of
# potentially wasted space.

The "OOM but memory is available" mystery

You see: "CUDA out of memory. Tried to allocate 2 GB (GPU 0; 80 GB total; 40 GB allocated; 35 GB free; 48 GB reserved)." There's 35 GB "free" but the 2 GB allocation fails. This is fragmentation: the 35 GB is split into many small non-contiguous blocks, none large enough for a single 2 GB allocation. Fix: PYTORCH_CUDA_ALLOC_CONF=max_split_size_mb:512, or use gradient checkpointing to reduce peak allocation size, or reduce batch size.

9 Mixed precision

autocast and GradScalerPrecision
# BF16 mixed precision (recommended on A100/H100)
with torch.autocast('cuda', dtype=torch.bfloat16):
    output = model(input)      # GEMMs in BF16, norms/softmax in FP32
    loss = criterion(output, target)  # loss promoted to FP32 automatically

loss.backward()                # backward in BF16 (matches forward dtypes)
optimizer.step()               # optimizer works in FP32 (master weights)

# FP16 mixed precision (needs GradScaler for stability)
scaler = torch.amp.GradScaler('cuda')

with torch.autocast('cuda', dtype=torch.float16):
    output = model(input)
    loss = criterion(output, target)

scaler.scale(loss).backward()       # scale loss to prevent gradient underflow
scaler.step(optimizer)              # unscale gradients, then step
scaler.update()                     # adjust scale factor
optimizer.zero_grad()

What autocast does to each operation

Cast to BF16/FP16 (fast)Keep in FP32 (precise)Run in widest input dtype
linear, mm, matmul, conv2d, bmmlayer_norm, softmax, cross_entropy, binary_cross_entropy, log_softmaxadd, mul, cat, sum, relu

BF16 over FP16 for modern GPUs. BF16 has the same exponent range as FP32, so gradients don't underflow — no GradScaler needed. The code is simpler and training is more stable. FP16 has more mantissa precision (10 vs 7 bits) but the narrow range causes real problems at scale. Use BF16 on anything ≥ Ampere (A100/RTX 3090+).

10 torch.compile

TorchDynamo, TorchInductor, and compilation modesCompile

torch.compile is PyTorch's JIT compiler. It traces the Python code, captures the computation graph, optimizes it, and generates efficient CUDA code.

ComponentRole
TorchDynamoPython bytecode analyzer that captures the computation graph by intercepting Python execution. Handles dynamic control flow by creating "graph breaks" at non-traceable points.
AOTAutogradAhead-of-time autograd — traces both forward and backward graphs, enabling whole-graph optimization of the backward pass.
TorchInductorThe default backend compiler. Generates optimized Triton kernels (GPU) or C++ code (CPU). Performs operator fusion, layout optimization, and autotuning.
# Basic usage
model = torch.compile(model)                        # default mode
model = torch.compile(model, mode="reduce-overhead") # CUDA Graphs
model = torch.compile(model, mode="max-autotune")   # more tuning, slower compile
model = torch.compile(model, fullgraph=True)        # error on graph breaks
model = torch.compile(model, dynamic=True)          # handle dynamic shapes

# Check what happened
import torch._dynamo as dynamo
explanation = dynamo.explain(model)(sample_input)
print(explanation)  # number of graphs, breaks, reasons

# Compile individual functions too
@torch.compile
def custom_loss(pred, target):
    return F.mse_loss(pred, target) + 0.1 * pred.abs().mean()

Collective operations

# All operations are in-place and synchronous by default

# AllReduce — sum across all ranks, result on all ranks
tensor = torch.tensor([rank], dtype=torch.float32, device='cuda')
dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
# All ranks now have: tensor = [0 + 1 + 2 + ... + (world_size-1)]

# Broadcast — send from one rank to all
dist.broadcast(tensor, src=0)  # rank 0 sends, all receive

# AllGather — gather tensors from all ranks
gathered = [torch.zeros_like(tensor) for _ in range(world_size)]
dist.all_gather(gathered, tensor)

# Barrier — wait for all ranks
dist.barrier()  # all ranks wait here until everyone arrives
# DDP — simplest multi-GPU training
model = nn.parallel.DistributedDataParallel(
    model.cuda(), device_ids=[local_rank]
)
# DDP automatically AllReduces gradients after each backward
# Each GPU processes different data (DistributedSampler handles this)

sampler = torch.utils.data.distributed.DistributedSampler(dataset)

# FSDP — sharded model state across GPUs
from torch.distributed._composable.fsdp import fully_shard
for layer in model.layers:
    fully_shard(layer)

Security: always use weights_only=True

torch.load() uses Python's pickle by default, which can execute arbitrary code during deserialization. A malicious .pt file can run code on your machine. Always use weights_only=True (restricts to tensor data only) or use SafeTensors format (which is inherently safe).

Triton kernels (Python-based GPU programming)

# pip install triton
import triton
import triton.language as tl

@triton.jit
def fused_add_relu_kernel(x_ptr, y_ptr, out_ptr, n,
                           BLOCK_SIZE: tl.constexpr):
    pid = tl.program_id(0)
    offsets = pid * BLOCK_SIZE + tl.arange(0, BLOCK_SIZE)
    mask = offsets < n
    x = tl.load(x_ptr + offsets, mask=mask)
    y = tl.load(y_ptr + offsets, mask=mask)
    result = tl.maximum(x + y, 0.0)  # fused add + relu
    tl.store(out_ptr + offsets, result, mask=mask)

# Launch
grid = lambda meta: (triton.cdiv(n, meta['BLOCK_SIZE']),)
fused_add_relu_kernel[grid](x, y, out, n, BLOCK_SIZE=1024)

15 The 50 most common PyTorch gotchas

Gotchas 1–17: Tensors and autogradGotchas
#GotchaFix
1Forgetting optimizer.zero_grad() — gradients accumulate across stepsAlways call before .backward(). Or use set_to_none=True.
2Calling .item() in the training loop — forces GPU syncAccumulate loss as a tensor, call .item() only for logging every N steps.
3In-place operations on tensors in the graphUse x = x + 1 not x += 1 for tensors with requires_grad=True.
4Creating tensors in a loop without torch.no_grad()Metric accumulation, logging — wrap in with torch.no_grad():.
5Forgetting .detach() when storing loss historylosses.append(loss.detach()) — otherwise the entire computation graph is retained in memory.
6.view() fails on non-contiguous tensorsUse .reshape() (works always) or .contiguous().view().
7Boolean indexing returns a copy, not a viewx[mask] always copies. Can't modify original via the result.
8BF16 tensor × Python float → FP32 resultUse typed constants or rely on autocast context.
9tensor.data modifications bypass autogradAvoid .data. Use with torch.no_grad(): tensor.copy_(new_data).
10torch.tensor() always copies; torch.as_tensor() may share memoryUse as_tensor() for numpy arrays when you want zero-copy.
11requires_grad defaults to False for new tensorsWrap in nn.Parameter() for learnable weights.
12Gradient of a scalar works; gradient of a non-scalar doesn't (without specifying gradient)Call .backward(gradient=torch.ones_like(y)) for non-scalar outputs, or sum first.
13.backward(retain_graph=True) leaks memory if done repeatedlyOnly use when absolutely needed (e.g., GAN with shared discriminator).
14torch.cat vs torch.stack confusioncat concatenates along existing dim; stack creates a new dim.
15expand() doesn't allocate memory; repeat() doesUse expand for broadcast (stride=0), repeat for actual copies.
16Default integer tensors are int64 — wasteful for indicesUse dtype=torch.int32 explicitly when int64 isn't needed.
17torch.randperm returns int64 on CPU, which must be transferredGenerate on CPU with pin_memory, or generate directly on GPU.
Gotchas 18–33: Modules, training, and dataGotchas
#GotchaFix
18Forgetting model.eval() during validationDropout stays active, BatchNorm uses batch stats. Always model.eval() + torch.no_grad().
19Forgetting model.train() after validationModel stays in eval mode during next training epoch. Dropout disabled, BN wrong.
20Assigning a tensor as self.x = tensor instead of nn.ParameterWon't appear in .parameters(), won't be optimized, won't be saved.
21Using a Python list of modules: self.layers = [nn.Linear(...)]Use nn.ModuleList — plain lists aren't registered as sub-modules.
22Using a Python dict of modules: self.blocks = {'a': nn.Linear(...)}Use nn.ModuleDict — plain dicts aren't registered.
23nn.CrossEntropyLoss expects raw logits, not probabilitiesDon't apply softmax before CrossEntropyLoss — it applies log_softmax internally.
24nn.CrossEntropyLoss default reduction='mean' divides by total elements, not just non-ignoredUse ignore_index=-100 for padding, and verify the denominator is correct for your loss masking.
25DataLoader workers and multiprocessing on Mac/WindowsWorker processes use fork (Linux) or spawn (Mac/Windows). Spawn requires picklable datasets. Wrap main code in if __name__ == '__main__':.
26DataLoader with shuffle=True and distributed trainingUse DistributedSampler instead — it handles per-rank sharding. Don't use DataLoader's shuffle.
27Not calling sampler.set_epoch(epoch) with DistributedSamplerWithout this, every epoch sees the same shuffled order on each rank.
28num_workers too high — more workers than CPUs causes contentionRule: num_workers ≈ num_cpus // num_gpus.
29BatchNorm with batch_size=1 fails (can't compute variance)Use GroupNorm or LayerNorm for batch_size=1 scenarios.
30nn.Embedding padding_idx gradients are zeroed but the embedding itself isn't zero-initializedManually zero: model.embed.weight.data[pad_idx].zero_().
31Gradient clipping before scaler.unscale_() in FP16 AMPClip after scaler.unscale_(optimizer), before scaler.step(optimizer).
32Saving the entire model with torch.save(model, ...)Saves the class definition too (fragile). Always save model.state_dict() instead.
33load_state_dict with strict=True fails on missing/extra keysUse strict=False for partial loading, but check the returned missing/unexpected keys.
Gotchas 34–50: CUDA, distributed, and performanceGotchas
#GotchaFix
34CUDA operations are async — timing with Python's time.time() is wrongUse torch.cuda.Event for accurate GPU timing, or torch.cuda.synchronize() before timing.
35Tensors on different devices can't be combineda.cuda() + b fails if b is on CPU. Move both to the same device first.

38DDP wrapping must happen before optimizer creationThe optimizer must reference the DDP-wrapped model's parameters.
39DDP with unused parameters errors outfind_unused_parameters=True, but this adds overhead. Better: ensure all params get gradients.
40FSDP with mixed precision requires explicit policySpecify MixedPrecision policy for param, reduce, and buffer dtypes.
41torch.compile recompiles on shape changeUse dynamic=True or pad to fixed shapes.
42torch.compile first call is extremely slow (compilation)Warmup 1–2 iterations before profiling or benchmarking.
43cudnn.benchmark=True with variable-size inputs is slowerRe-benchmarks every new shape. Use only with fixed input sizes.

47Vocab size not padded to multiple of 8 — kills Tensor Core usagePad embedding and head to next multiple of 64.

49torch.no_grad() doesn't affect model.train()/model.eval()They're independent. no_grad = no gradient computation. eval = changes module behavior (dropout/BN).
50Printing a CUDA tensor forces synchronization and is slowAvoid print(tensor) in training loops. Use .item() or log to file.

The PyTorch mental model: A tensor is a view into storage, described by shape + strides + dtype + device. Autograd builds a dynamic computation graph on every forward pass and destroys it on backward — gradients accumulate, so zero them. Modules are nested containers of Parameters (learnable) and Buffers (non-learnable state) that move together with .to(). CUDA operations are asynchronous — the CPU queues work and moves on; synchronization points (.item(), .cpu(), print) destroy this overlap. The caching allocator manages GPU memory via pools — fragmentation, not total usage, causes OOM. torch.compile captures the graph, fuses ops, and generates Triton kernels. Distributed training wraps everything in process groups that communicate via NCCL collectives. Master these pieces and the 50 gotchas, and you can debug any PyTorch program.
