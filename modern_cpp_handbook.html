Part I

Foundations: the object model

Before classes and templates, three concepts that quietly govern everything: how the compiler sees your program, when objects exist, and which expressions can be moved from.

§01The compilation model

C++ compiles each translation unit (TU) independently — typically one .cpp file with its #include'd headers expanded inline. The linker stitches TUs together by resolving symbols. This separation has consequences that ripple through every design decision.

The four stages

  1. Preprocessing — #include textual expansion, macro substitution, #if branches.
  2. Compilation — parses the TU, performs semantic analysis, emits an object file with a symbol table.
  3. Assembly — the compiler's output becomes machine code.
  4. Linking — resolves symbol references across TUs, applies relocations, produces an executable or library.

The One Definition Rule (ODR)

Every entity must have exactly one definition across the entire program, with narrow exceptions for inline functions, templates, and types (which may appear in multiple TUs provided their tokens are identical). ODR violations are usually undefined behavior with no diagnostic required — the linker may pick any one definition or fail silently.

// header.hpp
inline int helper() { return 42; }   // OK: inline, ODR-permissive

int non_inline() { return 42; }   // BAD: every TU including this gets a definition
                                  // → multiple definitions at link time

constexpr int N = 100;            // OK: constexpr implies inline since C++17
inline constexpr int M = 100;     // also OK

extern int g_counter;             // declaration only; definition lives in one .cpp

Headers and the include blast radius

Because #include is textual, every byte in a header is paid for by every TU that includes it. Three disciplines limit the damage: forward declarations for pointer/reference uses in headers, the pimpl idiom for hiding implementation, and include what you use (no transitive include dependence).

// widget.hpp — public header, minimal
#pragma once
#include <memory>

class Widget {
public:
  Widget();
  ~Widget();
  void render();
private:
  struct Impl;                       // forward-declared opaque type
  std::unique_ptr<Impl> pimpl_;       // only the pointer is in the header
};

// widget.cpp — heavy includes live here
#include "widget.hpp"
#include <heavy_graphics_lib.hpp>

struct Widget::Impl { heavy_graphics_lib::Renderer r; };

Widget::Widget() : pimpl_(std::make_unique<Impl>()) {}
Widget::~Widget() = default;   // defined here, where Impl is complete
Linkage in one paragraph

An entity has external linkage (visible across TUs — the default for non-const globals and non-inline functions), internal linkage (one TU only — static globals, anonymous namespaces, const globals by default), or no linkage (local variables). Internal linkage is the right choice for helpers that shouldn't pollute the global symbol table — anonymous namespaces (namespace { ... }) are the modern way.

§02Storage duration & object lifetime

Every object has a storage duration determining when memory is allocated, and a lifetime determining when the object exists (the window during which member access is valid). Related but not identical.

Storage durationAllocatedExamples
automaticStack frame entry → exitLocal variables, function parameters
staticProgram start → endGlobals, static locals, static class members
threadThread start → endthread_local variables
dynamicnew → deleteHeap allocations

Lifetime is finer than storage

Within a storage duration, an object's lifetime begins after its constructor finishes and ends when its destructor starts. Accessing a member before construction completes or after destruction begins is undefined behavior — even though storage is still valid.

struct Widget {
  Widget() {
    // During this body, the Widget's lifetime has NOT begun yet
    // — but storage exists. Calling virtual functions on `this` here
    // runs the base-class implementation, not the derived one.
  }
  ~Widget() {
    // After this body starts, lifetime has ended for the derived class.
    // Virtual calls run the base-class implementation again.
  }
};

Static initialization order fiasco

The order of dynamic initialization of static-duration objects across TUs is unspecified. If a static in a.cpp uses a static in b.cpp during its constructor, you have a 50/50 chance of using a not-yet-initialized object.

Vulnerable
// logger.cpp
Logger g_logger;

// config.cpp
Config g_config("settings.ini");
// Config ctor calls g_logger.log(...)
// — but g_logger may not be constructed yet!
Construct-on-first-use
Logger& logger() {
  static Logger instance;   // init on first call
  return instance;
}
// Thread-safe since C++11 (Magic Statics)
// Initialization is deterministic at first use.
Object lifetime ≠ storage availability

You can end an object's lifetime explicitly with obj.~T() and reuse its storage for another object via placement new. This is the foundation of std::variant, std::optional, small-buffer optimization, and many allocator schemes. Between destruction and re-construction, the storage holds no live object — accessing it through the old type is UB.

§03Value categories

Every C++ expression has both a type and a value category. The category tells the compiler whether the expression's result can be moved from, has an address, is being identified by name, or is a temporary. Move semantics, perfect forwarding, and overload resolution turn on these distinctions.

expression / \ glvalue rvalue / \ / \ lvalue xvalue prvalue (named) (expiring) (pure rvalue, temporary) glvalue = "generalized lvalue", has identity (an address) rvalue = can be moved from xvalue = both (a glvalue being treated as moveable)
CategoryHas identity?Can be moved from?Examples
lvalueyesnox, arr[i], *p, function returning T&
prvaluenoyesliterals, x + y, function returning T by value
xvalueyesyesstd::move(x), function returning T&&
 
std::string make() { return "hi"; }
std::string& first(std::vector<std::string>& v) { return v[0]; }

std::string s = "a";

s             // lvalue   — has a name
make()        // prvalue  — temporary from function returning by value
first(v)      // lvalue   — returns T&, the reference identifies an object
std::move(s)   // xvalue   — rvalue reference cast
42            // prvalue
s + "b"       // prvalue  — operator+ returns a temporary

Why this matters

void consume(std::string  x);        // by value: copies lvalue, moves rvalue
void consume(std::string& x);        // by lvalue ref: binds only to lvalues
void consume(std::string const& x);  // by const lvalue ref: binds to both
void consume(std::string&& x);       // by rvalue ref: binds only to rvalues

std::string s = "hello";
consume(s);                // s is lvalue → calls T& or const T& overload
consume("hello");          // prvalue → calls T or T&& or const T& overload
consume(std::move(s));     // xvalue → calls T or T&& overload

The category of an expression is determined at compile time, by syntax, not by what's "in" the object at runtime. std::move(x) doesn't move anything — it casts x to an rvalue reference, making the move constructor a viable overload at the call site. The actual moving is performed by that constructor.

Part II

Types, references, and pointers

The type system has grown teeth in modern C++. Type deduction, reference collapsing, and a precise understanding of how const interacts with pointers and references separate working code from correct code.

§04The type system — auto, decltype

auto uses template argument deduction; decltype uses different rules. They produce different types from the same expression. Knowing which to reach for is part of fluency.

auto rules

  • auto x = expr; — strips const, references, and top-level cv-qualifiers. Always a fresh, modifiable copy.
  • auto& x = expr; — retains const; the variable is a reference.
  • auto&& x = expr; — forwarding reference; binds to anything.
  • decltype(auto) x = expr; — uses decltype rules: preserves reference-ness and cv-qualification exactly.
int i = 0;
int const& cri = i;

auto          a = cri;   // int               — const and & stripped
auto&         b = cri;   // int const&        — const kept
auto const&   c = i;     // int const&        — explicitly const
decltype(auto) d = cri;   // int const&        — decltype keeps everything
decltype(auto) e = i;     // int               — name → its declared type
decltype(auto) f = (i);   // int&              — parenthesized name → lvalue ref!
The parens in line f turn i from an identifier expression into a parenthesized expression, which decltype sees as an lvalue → reference type. This is the most famous decltype gotcha.

decltype

decltype(E) for a name E gives the declared type. For an expression E, it gives T, T&, or T&& matching the value category (prvalue, lvalue, xvalue).

int x = 0;
int& rx = x;

decltype(x)    a;          // int
decltype(rx)   b = x;      // int&  — must initialize
decltype((x))  c = x;      // int&  — (x) is lvalue expression
decltype(x+1)  d;          // int   — prvalue

// Real-world use: return type that depends on arguments
template<typename A, typename B>
auto add(A a, B b) { return a + b; }   // C++14: just use auto

Type aliases — using is the modern typedef

using IntVec = std::vector<int>;
using Callback = int(*)(int, int);    // reads left-to-right, unlike typedef

// Aliases can be templates — typedef cannot
template<typename T>
using Vec = std::vector<T, MyAllocator<T>>;
Vec<int> v;   // std::vector<int, MyAllocator<int>>

§05References vs pointers

References and pointers both let you refer to an object without copying, but they're semantically different in ways that matter for design.

RebindableNo — binds once for its lifetimeYes — can repoint

ArithmeticNoneYes — p++, p + n

The three reference flavors

int i = 0;

int&        lr  = i;          // lvalue reference
int const&  clr = i;          // const lvalue reference — binds to lvalue AND rvalue
int const&  clr2 = 42;         // extends lifetime of the temporary 42

int&&       rr  = 42;         // rvalue reference — binds to rvalues only
int&&       rr2 = std::move(i); // xvalue is rvalue
// int&&    rr3 = i;            // ERROR: i is lvalue

Reference collapsing

"Reference to a reference" can arise via type aliases and template deduction. The language collapses them: T& & → T&, T& && → T&, T&& & → T&, T&& && → T&&. The rule: if any reference is an lvalue ref, collapse to lvalue ref; otherwise rvalue ref. This rule is the entire mechanism behind std::forward and forwarding references.

When to use which

  • Function parameters that must reference an existing object → reference. The non-null guarantee shifts an entire class of bugs to compile time.
  • Function parameters that are optional or rebindable → pointer.
  • Data members → almost never references. A reference member kills move and copy assignment, since references can't be rebound. Use T* or std::reference_wrapper<T>.
  • Range iteration → for (auto& x : v) for elements; auto const& for read-only; auto&& when generic.
Dangling references — the silent killer

A reference outliving its referent is undefined behavior. Common sources: returning a reference to a local; binding a reference to a temporary; storing references in containers that reallocate.

std::string const& bad() {
  std::string s = "hello";
  return s;                // UB: reference to local that dies on return
}

auto trap(std::string const& s) { return s + "!"; }    // returns by value (good)
std::string const& x = trap("a");    // UB: temporary's lifetime is NOT extended through
                                  //     a function return; x dangles immediately

§06const correctness in depth

const is the single most important type-system feature for documenting and enforcing intent. The senior-level skill is knowing precisely where it goes and what it changes.

Read pointers right-to-left

int                p1;   // int
int*               p2;   // pointer to int
int const*         p3;   // pointer to const int   — can repoint, can't modify *p3
const int*         p4;   // same as p3 — identical meaning
int* const         p5;   // const pointer to int   — can't repoint, can modify *p5
int const* const   p6;   // const pointer to const int

const member functions

class Buffer {
  std::vector<char> data_;
  mutable std::mutex mu_;        // mutable: modifiable in const methods
public:
  std::size_t size() const;       // can be called on const Buffer
  char&       at(std::size_t i);
  char const& at(std::size_t i) const;  // const overload

  void log() const {
    std::lock_guard lk{mu_};       // mu_ is mutable — OK
    // data_.push_back('x');         // ERROR: data_ not mutable
  }
};

const overloading for accessors — the deduplication trick

class Container {
  T* data_;
public:
  T const& at(std::size_t i) const {
    if (i >= size_) throw std::out_of_range{"i"};
    return data_[i];
  }

  // Non-const delegates to const, then casts away the const it added
  T& at(std::size_t i) {
    return const_cast<T&>(std::as_const(*this).at(i));
  }
};
const_cast: a flag, not a tool

const_cast is the one cast that can remove const. Use it only in the accessor-delegation pattern above (where you originated the const) or when interfacing with C APIs. Casting away const on an actually-const object and writing through the result is undefined behavior.

Ref-qualified member functions

C++11 added ref-qualifiers on member functions: overload on whether *this is an lvalue or rvalue. Useful for builders and for guiding moves:

class Builder {
  std::vector<std::string> parts_;
public:
  std::vector<std::string> result() &       { return parts_; }            // copy
  std::vector<std::string> result() &&      { return std::move(parts_); } // move
};

Builder b;
auto r1 = b.result();             // b is lvalue → copy
auto r2 = Builder{}.result();      // temporary → move

§07Initialization — the seven forms

C++ has seven initialization syntaxes, each with subtly different rules. The intended unification is brace initialization in C++11 — with one notorious exception involving std::initializer_list.

int a = 42;            // 1. copy initialization
int b(42);             // 2. direct initialization
int c{42};             // 3. direct-list initialization (brace init)
int d = {42};          // 4. copy-list initialization
int e;                 // 5. default initialization — UB to read for built-ins
int f{};               // 6. value initialization — zero for built-ins
static int g;          // 7. zero initialization — statics get 0 implicitly

Brace initialization — the default recommendation

Prefer {...}: forbids narrowing conversions, no most-vexing-parse, works uniformly. With one caveat:

int    i{3.14};                 // ERROR: narrowing — won't compile
int    j = 3.14;                // silently becomes 3

std::vector<int> v1(10, 7);       // 10 elements of value 7
std::vector<int> v2{10, 7};       // initializer_list — TWO elements: 10 and 7

// The std::initializer_list overload is preferred when braces match it.
// This is the one place braces and parens MEAN something different.

Aggregate initialization

An aggregate has no user-declared constructors, no private/protected non-static data members, no virtual functions. Aggregates can be brace-initialized member-by-member:

struct Point { double x, y, z; };
Point p{1, 2, 3};                    // aggregate init
Point q{.x = 1, .y = 2, .z = 3};       // designated init (C++20)

Point r{1, 2};                       // r.z == 0.0 (missing members value-initialized)

Default vs value vs zero initialization

struct Pod { int x; double y; };

int    a;        // default-init: indeterminate, UB to read
int    b{};      // value-init: 0
Pod    c;        // default-init: members default-initialized → indeterminate
Pod    d{};      // value-init: members value-initialized → x=0, y=0

auto p = new Pod;    // new T  → default init → indeterminate members
auto q = new Pod{};  // new T{} → value init → zero-filled members
The most vexing parse

Parens around what looks like a constructor call can parse as a function declaration:

Widget w();    // FUNCTION declaration: takes no args, returns Widget
Widget w{};    // Variable initialized with empty brace-init
Widget w2(int());  // FUNCTION declaration — int() is a function type
Widget w3{int()};  // Variable init with a value-initialized int

Braces solve this. Always use them for variable initialization where possible.

Part III

Classes and resource management

A class is a type that bundles data and operations; a well-designed class is one whose copy, move, and destruction semantics are exactly what its invariants demand. Most C++ design boils down to getting the special members right and letting RAII carry the rest.

§08Class anatomy

A class is the unification of: a memory layout (data members), an access policy (public/protected/private), an invariant (the predicate the class maintains across its lifetime), and a set of operations that respect that invariant.

class Stopwatch {
public:
  // Special members
  Stopwatch()                                = default;
  Stopwatch(Stopwatch const&)              = default;
  Stopwatch(Stopwatch&&) noexcept            = default;
  Stopwatch& operator=(Stopwatch const&)   = default;
  Stopwatch& operator=(Stopwatch&&) noexcept = default;
  ~Stopwatch()                               = default;

  // Public interface (verbs)
  void   start();
  void   stop();
  double elapsed_ms() const;

private:
  using Clock = std::chrono::steady_clock;
  Clock::time_point start_{};
  Clock::duration   elapsed_{};
  bool              running_{false};
};

The five-way layout discipline

  1. Special members first — ownership obvious at a glance.
  2. Public operations — the contract.
  3. Protected operations — extension points for derived classes.
  4. Private helpers — implementation details.
  5. Data members last (and private) — never expose representation; an invariant the class doesn't control is an invariant the class can't keep.

Class vs struct

The only language difference is default access (private for class, public for struct). By convention: struct for passive aggregates with no invariant; class for types with an invariant and a non-trivial interface.

Special member functions — the canonical six

MemberSignatureImplicitly generated when
Default constructorT()No other ctor declared
Destructor~T()Always (virtual or not per the class)
Copy constructorT(T const&)No copy/move declared
Copy assignmentT& operator=(T const&)No copy/move assignment declared

Move assignmentT& operator=(T&&) noexceptNo copy/move/destructor declared
The interaction matrix is subtle

Declaring a destructor suppresses implicit move generation. Declaring a copy operation suppresses implicit move generation. Declaring a move operation suppresses implicit copy. Get one wrong, and your type silently loses move semantics — a 100× performance regression you can't see at the call site. The Rule of Five (§12) is the discipline.

§09Constructors in depth

The five kinds

class Widget {
public:
  Widget();                                       // 1. default
  Widget(int id, std::string name);              // 2. parameterized
  Widget(Widget const& other);                    // 3. copy
  Widget(Widget&& other) noexcept;                // 4. move

  // 5. converting constructor (single-arg, non-explicit)
  Widget(int id);                                 // allows: Widget w = 42;
  explicit Widget(std::string name);             // requires: Widget w{"a"};
};

Member initialization list — not optional

Members are initialized in declaration order, not initializer-list order. Putting work in the constructor body initializes by default first and then assigns — two operations per member.

Don't
Widget::Widget(int id, std::string n) {
  id_   = id;      // default-init,
  name_ = n;       // then assign.
  // 2x the work for std::string
  // impossible for const/ref members
}
Do
Widget::Widget(int id, std::string n)
  : id_(id), name_(std::move(n)) {}
  // Direct init, no temporaries.
  // Required for const, references,
  // types with no default ctor,
  // and base-class init.

Default member initializers (C++11)

class Widget {
  int         id_   = 0;          // default member initializer
  std::string name_ = "unnamed";
  bool        ready_{false};

public:
  Widget() = default;             // uses the defaults above
  Widget(int id) : id_(id) {}     // overrides id_, name_ stays "unnamed"
};

Delegating constructors

class Widget {
  int id_; std::string name_;
public:
  Widget(int id, std::string name) : id_(id), name_(std::move(name)) {}

  Widget()           : Widget(0, "unnamed") {}      // delegate
  Widget(int id)     : Widget(id, "unnamed") {}
};

= default, = delete

class NonCopyable {
public:
  NonCopyable()                              = default;
  NonCopyable(NonCopyable const&)            = delete;
  NonCopyable& operator=(NonCopyable const&) = delete;
  NonCopyable(NonCopyable&&) noexcept        = default;
  NonCopyable& operator=(NonCopyable&&) noexcept = default;
};

// = delete also works for any function — useful for banning overloads
void handle(int);
void handle(char) = delete;          // reject char calls explicitly
void handle(double) = delete;        // reject implicit conversions from double

explicit — preventing surprises

struct Implicit { Implicit(int); };
struct Explicit { explicit Explicit(int); };

void a(Implicit);
void b(Explicit);

a(42);              // OK: implicit conversion int → Implicit
b(42);              // ERROR: no implicit conversion
b(Explicit{42});    // OK: explicit

// Rule: single-argument constructors should be `explicit` unless implicit
// conversion is genuinely useful (e.g., std::string from const char*).
Copy elision & RVO

When a function returns a temporary, the compiler is allowed (and since C++17 sometimes required) to construct that temporary directly into the caller's variable — no copy, no move. This is return value optimization (RVO). The implication: return by value, even for expensive types. Don't return references or pointers as a "performance optimization" — RVO almost always wins, and the resulting code is safer and clearer.

§10Destructors & deterministic cleanup

The destructor runs when an object's lifetime ends: at scope exit for automatic objects, at delete for dynamic objects, at program exit for statics, in reverse construction order for members and base classes. Deterministic destruction is C++'s defining feature — the engine behind RAII.

struct Resource {
  Resource(char const* name) : name_(name) { std::cout << "acquire " << name_ << "\n"; }
  ~Resource()                       { std::cout << "release " << name_ << "\n"; }
  char const* name_;
};

struct Widget {
  Resource a{"a"};
  Resource b{"b"};
  Resource c{"c"};
};
// Construction: a, b, c (declaration order)
// Destruction:  c, b, a (reverse)

Destructors must not throw

Since C++11, destructors are implicitly noexcept. Throwing from a destructor during stack unwinding (another exception in flight) calls std::terminate. Treat this as inviolable.

class FileHandle {
  FILE* f_;
public:
  ~FileHandle() noexcept {
    if (f_) {
      try {
        std::fclose(f_);          // fclose could fail — log, don't throw
      } catch (...) { /* swallow */ }
    }
  }
};

Virtual destructors

If a class might be deleted through a base pointer, its destructor must be virtual — otherwise only the base destructor runs and derived members leak. The reverse: classes not intended for polymorphic deletion shouldn't have virtual destructors (the vtable pointer costs memory and forbids trivially-destructible status).

struct Base    { virtual ~Base() = default; };
struct Derived : Base { std::vector<int> data_; };

Base* p = new Derived{};
delete p;        // virtual → ~Derived then ~Base. Without virtual, leak.

// Rule: any class with virtual functions should have a virtual destructor.
struct Interface {
  virtual ~Interface() = default;
  virtual void go() = 0;
};

final — sealing the hierarchy

class Logger final { };          // can't be inherited from

struct Base { virtual void go(); };
struct Mid : Base {
  void go() override final;   // no further override
};

§11Copy semantics & the Rule of Three

Copying a class means producing an independent duplicate that satisfies the same invariant. For classes that own resources, the implicit member-wise copy is wrong — two objects share the same resource and both try to free it.

The Rule of Three

If a class needs any one of (destructor, copy constructor, copy assignment), it almost certainly needs all three. A destructor that releases something implies the copy operations must duplicate that something.

class String {
  char*       data_;
  std::size_t size_;

public:
  String(char const* s = "")
    : size_(std::strlen(s)),
      data_(new char[size_ + 1]) {
    std::strcpy(data_, s);
  }

  // 1. Destructor
  ~String() { delete[] data_; }

  // 2. Copy constructor — DEEP copy
  String(String const& other)
    : size_(other.size_),
      data_(new char[size_ + 1]) {
    std::strcpy(data_, other.data_);
  }

  // 3. Copy assignment — copy-and-swap idiom
  String& operator=(String other) {     // take by VALUE: copy happens here
    swap(*this, other);
    return *this;                       // other (with old data) dies, cleaning up
  }

  friend void swap(String& a, String& b) noexcept {
    using std::swap;
    swap(a.data_, b.data_);
    swap(a.size_, b.size_);
  }
};
The copy-and-swap idiom

Taking the argument by value lets the compiler choose between copy (lvalue arg) and move (rvalue arg) construction at the call site. The function body swaps with the temporary and returns. Three benefits: strong exception safety (if copy throws, *this is untouched), self-assignment is automatically safe, and one function handles both copy and move assignment when the type has a move constructor.

Self-assignment

The naive "delete then copy" form has a famous bug:

String& operator=(String const& other) {
  delete[] data_;                  // boom
  size_ = other.size_;            // other == *this
  data_ = new char[size_ + 1];
  std::strcpy(data_, other.data_);  // reads freed memory
  return *this;
}
// Self-assignment (a = a) frees a's storage, then reads from it. UB.

§12Move semantics & the Rule of Five (and Zero)

Moving an object transfers ownership of its resources, leaving the source in a valid but unspecified state. The classic optimization: don't deep-copy a 100 MB buffer if the source is about to die anyway — steal its pointer.

class String {
  char* data_; std::size_t size_;
public:
  // Move constructor: steal source's pointer, leave it null
  String(String&& other) noexcept
    : data_(other.data_), size_(other.size_) {
    other.data_ = nullptr;
    other.size_ = 0;
  }

  String& operator=(String&& other) noexcept {
    if (this != &other) {
      delete[] data_;
      data_ = other.data_;
      size_ = other.size_;
      other.data_ = nullptr;
      other.size_ = 0;
    }
    return *this;
  }
};

noexcept on move — not optional

std::vector and other containers consult std::is_nothrow_move_constructible when reallocating. If your move can throw, the container falls back to copy on growth — a hidden performance cliff. Always declare moves noexcept.

Rule of Five

If you declare any of (destructor, copy ctor, copy assign, move ctor, move assign), declare all five. The defaults are interdependent: declaring a destructor disables implicit moves, and so on. Being explicit avoids silently losing move semantics.

Rule of Zero — the better goal

Don't write any special members. Compose your class out of types that already manage their own resources (std::vector, std::unique_ptr, std::string), and let the compiler synthesize correct, efficient operations.

Rule of Five — explicit
class Image {
  size_t w_, h_;
  unsigned char* pixels_;
public:
  Image(size_t w, size_t h);
  ~Image();
  Image(Image const&);
  Image& operator=(Image const&);
  Image(Image&&) noexcept;
  Image& operator=(Image&&) noexcept;
};
Rule of Zero — compose
class Image {
  size_t w_{}, h_{};
  std::vector<unsigned char> pixels_;
public:
  Image(size_t w, size_t h)
    : w_(w), h_(h), pixels_(w * h * 4) {}
  // No special members needed.
  // Compiler generates correct ones.
};

Moved-from state

A moved-from object must be destructible and assignable; the standard says little else. Library types are typically left "empty" but unspecified. Don't read values from moved-from objects beyond destroying or reassigning them.

Modern C++ ownership: composition over inheritance, value semantics over pointer semantics, Rule of Zero over Rule of Five. A class that doesn't manage raw resources directly doesn't need special members — the code becomes shorter, faster (RVO + move), and harder to get wrong.

§13RAII as a discipline

RAII — Resource Acquisition Is Initialization — is the principle that resource ownership lives in object lifetime. Acquire in the constructor; release in the destructor. The destructor is guaranteed to run (normal scope exit or exception unwinding), so cleanup is automatic.

The phrase is misnamed; the important half is "destruction is release." RAII works because C++ destructors are deterministic.

Beyond memory

The pattern applies to any resource with paired acquire/release semantics:

A minimal RAII wrapper

template<typename Release>
class ScopeGuard {
  Release release_;
  bool active_{true};
public:
  explicit ScopeGuard(Release release) : release_(std::move(release)) {}
  ~ScopeGuard() { if (active_) release_(); }

  void dismiss() noexcept { active_ = false; }

  ScopeGuard(ScopeGuard const&)            = delete;
  ScopeGuard& operator=(ScopeGuard const&) = delete;
};

// Usage
void work() {
  FILE* f = std::fopen("data.bin", "rb");
  if (!f) throw std::runtime_error{"open failed"};
  ScopeGuard closer{[f]{ std::fclose(f); }};
  // ... use f ...
  // fclose runs on normal exit AND on exception
}

Why RAII outperforms try/finally

  • Composable — multiple RAII objects in one scope unwind in reverse order automatically. try/finally requires nesting.
  • Exception-safe by default — no path forgets to call cleanup.
  • Local to ownership — cleanup lives next to acquisition, not 200 lines later.
  • Transfers with the object — return an RAII type by value and ownership moves with it.

Every paired operation in C++ should live behind an RAII wrapper. The cost is one small class; the benefit is correctness-by-construction.

Part IV

Operator overloading & conversions

Operator overloading is what makes std::complex read like math and std::string behave like text. Used well, it disappears into clarity.

§14Operator overloading

Most operators can be overloaded as either members or free functions. Rule of thumb: binary symmetric operators (+, -, ==, <) as free functions to allow conversion on the left-hand side; asymmetric or modifies-self operators (=, +=, [], ->, ()) as members.

class Vec3 {
  double x_{}, y_{}, z_{};
public:
  Vec3() = default;
  Vec3(double x, double y, double z) : x_(x), y_(y), z_(z) {}

  // Compound assignment as member — modifies self
  Vec3& operator+=(Vec3 const& rhs) {
    x_ += rhs.x_; y_ += rhs.y_; z_ += rhs.z_;
    return *this;
  }
  Vec3& operator*=(double s) {
    x_ *= s; y_ *= s; z_ *= s;
    return *this;
  }

  // Element access
  double&       operator[](int i)       { return (&x_)[i]; }
  double const& operator[](int i) const { return (&x_)[i]; }

  // C++20: synthesize all six comparisons
  auto operator<=>(Vec3 const&) const = default;
};

// Binary symmetric ops as free functions, expressed via the member
inline Vec3 operator+(Vec3 lhs, Vec3 const& rhs) { lhs += rhs; return lhs; }
inline Vec3 operator*(Vec3 lhs, double s)         { lhs *= s; return lhs; }
inline Vec3 operator*(double s, Vec3 rhs)         { rhs *= s; return rhs; }
// Now: 2.0 * v works (the left-hand argument can convert)

Three-way comparison (C++20)

operator<=> ("spaceship") synthesizes all six comparison operators from one definition. The return type signals the strength of the order: strong_ordering (antisymmetric), weak_ordering (equivalent but not equal), partial_ordering (NaN-like, two values may be incomparable).

struct Money {
  int64_t cents;
  auto operator<=>(Money const&) const = default;   // strong_ordering
};
if (a < b) ...      // works, synthesized
if (a == b) ...     // works, synthesized

struct Adder {

Operator overloading is API design

The bar: an operator should mean what the reader expects. + for vector addition is fine; + for string concatenation is fine; + for "insert into container" is a war crime. If in doubt, name the operation as a function.

§15Conversions & explicit

C++ has implicit conversions in two directions: converting constructors bring types into your class; conversion operators let your class produce other types. Both can surprise.

Converting constructors

struct Meters { double v; };
struct Feet   { double v; Feet(Meters m) : v(m.v * 3.281) {} };

void use_feet(Feet);
use_feet(Meters{10});            // implicit Meters → Feet conversion

struct SafeFeet { explicit SafeFeet(Meters); };    // require explicit cast

Conversion operators

class Temperature {
  double celsius_;
public:
  Temperature(double c) : celsius_(c) {}

  // Implicit conversion to double — questionable design
  operator double() const { return celsius_; }

  // Better: explicit conversion
  explicit operator double() const { return celsius_; }
};

explicit operator bool

The one place implicit-to-bool conversion is idiomatic is "is this object usable?" In C++11+ the type-safe way is explicit operator bool:

class FileHandle {
  FILE* f_;
public:
  explicit operator bool() const noexcept { return f_ != nullptr; }
};

FileHandle f{...};
if (f) {            // OK: contextual conversion to bool
}
// int x = f;       // ERROR: explicit blocks accidental int conversion

User-defined literals

constexpr Meters operator""_m(long double v) { return {static_cast<double>(v)}; }
constexpr Meters operator""_km(long double v) { return {static_cast<double>(v) * 1000}; }

auto distance = 5.0_km + 300.0_m;

using namespace std::chrono_literals;
auto timeout = 500ms;       // chrono uses this heavily
