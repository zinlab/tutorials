Part I

Container fundamentals, from the spec down

Containers feel obvious until you ship them at scale. The competence bar for this role is understanding the specs themselves — image, runtime, distribution — and the operational seams between them. Everything else builds on this layer.

§01OCI image internals & the registry contract

An OCI image is not "a thing" — it's a content-addressed graph rooted at a manifest. The manifest is a small JSON document that references a config blob and an ordered list of layer blobs, each by SHA-256 digest. Layers are gzipped (or zstd) tarballs of filesystem diffs. The runtime stacks them with an overlay filesystem (overlayfs on Linux) at start time. Multi-arch images add an image index (manifest list) on top — a manifest of manifests, one per platform.

The three OCI specs you must know cold

  • image-spec — manifest schema, config schema, layer media types (application/vnd.oci.image.layer.v1.tar+gzip and friends), platform fields.
  • runtime-spec — the config.json bundle that a runtime (runc, crun) actually executes. Defines namespaces, cgroups, mounts, hooks.
  • distribution-spec — the registry HTTP API. HEAD/GET/PUT /v2/{name}/blobs/{digest}, /manifests/{tag-or-digest}, upload sessions, mount-from-other-repo.
// truncated OCI manifest
{
  "schemaVersion": 2,
  "mediaType": "application/vnd.oci.image.manifest.v1+json",
  "config": { "digest": "sha256:9f1a...", "size": 7023 },
  "layers": [
    { "digest": "sha256:e7d3...", "size": 31415926 },
    { "digest": "sha256:b4f1...", "size": 2718281 }
  ]
}
Production gotcha — tag vs digest

Tags are mutable; :latest on Monday is not :latest on Tuesday. In any non-dev workflow — Helm values, manifests, CI promotion — pin by digest (nim/llama:24.10@sha256:...). Tag-by-digest is what makes rollbacks meaningful and what makes signing useful.

Containerd is the runtime; Docker is a developer UX

Kubernetes talks to containerd (or CRI-O) directly via CRI. Docker the daemon is no longer in the loop on modern clusters. Containerd owns the snapshotter (which prepares the rootfs from layers — see §21 on lazy snapshotters) and shells out to runc as the OCI-compliant runtime. Knowing where to look when a pod won't start is half the battle:

Where to inspect, on the node

§02BuildKit, multi-stage, and reproducible builds

BuildKit is the modern Docker builder. It compiles a Dockerfile (the "frontend") into a Low-Level Build graph (LLB), then executes operations in parallel where dependencies allow. Each operation produces a content-addressed cache key — change nothing, get nothing rebuilt. This is the heart of fast iteration on inference containers, where layers are huge and rebuilds can otherwise take 30+ minutes.

Multi-stage as a discipline, not a trick

For NIM-style containers the rule is firm: build dependencies (CUDA devel image, compilers, header files, build-time pip caches) never reach the runtime image. Stage them, then copy only the artifacts:

# syntax=docker/dockerfile:1.7
ARG CUDA_VERSION=12.6.0
FROM nvcr.io/nvidia/cuda:${CUDA_VERSION}-devel-ubuntu22.04 AS build
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install --prefix=/install vllm==0.6.3 ...

FROM nvcr.io/nvidia/cuda:${CUDA_VERSION}-runtime-ubuntu22.04
COPY --from=build --link /install /usr/local
COPY --link app/ /opt/nim/

Four things to internalize from this fragment:

  • --mount=type=cache persists pip's cache between builds without baking it into a layer. This is the biggest lever for Python-heavy images.
  • COPY --link creates the copy as its own independent layer rather than mutating the previous one. Subsequent edits don't invalidate that layer's children — a major cache win.
  • devel only in the build stage; the final image uses runtime, saving 2-4 GB.
  • Build args at the top, scoped to stages. Lets you generate a matrix of (CUDA × Python × backend) images from one Dockerfile.
Cache invalidation cascade — the #1 build-time bug

Layer order matters: anything edited frequently should appear last. A common mistake is COPY . /app early in the Dockerfile, followed by pip install -r requirements.txt. Now any source-code change invalidates the pip layer and forces re-resolution of dependencies. The corrected order is COPY requirements.txt → pip install → COPY . /app. Same principle scales up: pin CUDA wheels in a separate stage and copy in.

Reproducibility

"Reproducible" in the JD means bit-identical image digest given the same inputs. Three sources of nondeterminism dominate:

  1. Timestamps. Set SOURCE_DATE_EPOCH and pass --build-arg SOURCE_DATE_EPOCH=…; BuildKit honors it.
  2. File ordering inside tarballs. BuildKit normalizes this; non-BuildKit doesn't.
  3. Upstream package versions. Pin every apt package, every wheel, every git ref. Use lockfiles (uv / pip-tools / poetry) and a frozen apt snapshot mirror.
The orchestration loop you'll actually build

Python tooling that, given a model + a target GPU profile, generates a Dockerfile from a template, resolves pinned versions from a manifest, invokes buildctl or docker buildx with the right cache mounts, pushes to a staging registry by digest, runs a smoke test that exercises the model, and then promotes by digest to the public registry. This kind of pipeline is exactly what "Python tooling and services for build orchestration" in the JD describes.

§03Multi-arch builds — amd64, arm64, Grace

NVIDIA ships Grace-Hopper (arm64 + Hopper GPU) and Grace-Blackwell. Multi-arch images are mandatory and non-trivial. Two paths:

ApproachMechanismWhen to pick
QEMU emulationdocker buildx build --platform linux/amd64,linux/arm64 with binfmt_miscSmall images, dev iteration. CUDA-adjacent compilation is 10–50× slower under QEMU — usually unacceptable.
Native multi-builderTwo builders (one amd64, one arm64) wired into one buildx instance, then imagetools create merges manifestsProduction CUDA images. Required for any real inference container.
 
# 1. build per-arch on native builders (in parallel CI jobs)
docker buildx build --builder=amd64-runner \
  --platform linux/amd64 -t reg/nim:24.10-amd64 --push .

docker buildx build --builder=arm64-runner \
  --platform linux/arm64 -t reg/nim:24.10-arm64 --push .

# 2. fuse into a single multi-arch tag
docker buildx imagetools create -t reg/nim:24.10 \
  reg/nim:24.10-amd64 reg/nim:24.10-arm64
arm64 + CUDA gotchas
  • Not every CUDA library has an arm64 SBSA build. Verify for cuDNN, NCCL, TensorRT-LLM, vLLM wheels.
  • Python wheels: manylinux2014_aarch64 is the relevant tag. Many ML wheels still ship amd64-only and require source builds on arm64.
  • --platform in the FROM line lets you mix architectures within a build — useful for cross-compiling on amd64 hosts using the target's sysroot, but tricky.

§04Registry topology and OCI artifacts

The registry is the single biggest single point of failure in any container rollout. NIMs are big (5–30 GB) and pulled by many pods simultaneously; one bad day on the registry is one bad day for the entire fleet. Topology choices:

  • Hub-and-spoke — central authoritative registry, regional read replicas, pull-through caches in each cluster. Standard for Harbor + Registry deployments.
  • Geo-replicated active-active — registry software replicates manifests and blobs across regions. Cloud-managed (ECR, Artifact Registry) does this; self-hosted is doable with Harbor's replication policies.
  • In-cluster mirror — a registry runs inside each cluster and is preloaded. Best for air-gapped (§25).
  • Pull-through cache — a transparent proxy (Distribution, Harbor) caches upstream blobs. Saves egress and decouples from upstream rate limits.

OCI artifacts — registries store more than images

Modern registries store anything wrapped in an OCI manifest: Helm charts, SBOMs, cosign signatures, model weights, even arbitrary tarballs. This matters for NIMs because model weights as OCI artifacts let you reuse the registry's auth, mirroring, and CDN for distributing checkpoints. ORAS is the canonical tool.

# push model weights as an OCI artifact
oras push reg/models/llama3-70b:bf16 \
  ./model.safetensors:application/vnd.nvidia.nim.model.v1+octet-stream \
  ./config.json:application/vnd.nvidia.nim.config.v1+json \
  --artifact-type application/vnd.nvidia.nim.model.v1
Pull storm — the rollout-day failure

You roll out a new NIM image. Five hundred pods on five hundred nodes pull simultaneously. Each pulls 15 GB. Your registry, your network, or your storage backend dies. Mitigations: (a) image preload via a DaemonSet or kubelet's --image-pull-progress-deadline, (b) P2P distribution with Dragonfly or Spegel that lets nodes pull from each other, (c) lazy snapshotters (§21) so pods can start before the full image is local. The good architectures combine all three.

§05Supply chain — signing, SBOM, SLSA

NVIDIA ships software customers will run on their own infrastructure. The supply-chain story has to be airtight. Three artifacts to know:

  • Signatures — cosign produces a signature that lives alongside the image in the registry. Sigstore's keyless flow uses OIDC + Fulcio; you can also use long-lived keys. Verify in admission with policy-controller or Kyverno.
  • SBOMs — Software Bill of Materials, typically SPDX or CycloneDX. syft generates them; attach to the image with cosign attest. Critical for CVE response: when a CVE drops, you query SBOMs to find affected images in 60 seconds instead of 60 hours.
  • Provenance (SLSA) — a signed attestation describing how the image was built (which CI job, which commit, which builder). SLSA Level 3 means the build platform itself is hardened and produces non-forgeable provenance.
Admission-time verification

The point of all three is to fail closed: if a pod manifest references an unsigned image or an image with a critical-CVE SBOM, admission rejects it. cosign verify in a validating webhook, or Sigstore's policy-controller for declarative policies. Don't bolt this on at the end — bake it in from the first prototype.

Part II

The GPU container stack

This is the NVIDIA-specific layer where 70% of operational pain in this role lives. Driver, runtime, toolkit, libraries — each lives in a different place, has its own compatibility rules, and breaks in its own way.

§06The CUDA container stack

The model is straightforward once you internalize it: driver on the host, CUDA runtime in the container, library injection at start time. The injection is performed by nvidia-container-runtime, a thin shim layered on top of runc that consults the OCI spec, sees that the container wants GPUs, and mounts the host's driver libraries (libcuda.so, libnvidia-ml.so, etc.) into the container's filesystem just before runc create.

┌─────────────────────────────────────────────────────┐
│                  CONTAINER                          │
│  ┌──────────────────────────────────────────────┐   │
│  │  app  →  CUDA runtime (libcudart.so 12.6)    │   │
│  │              ↓ dlopen                         │   │
│  │  libcuda.so.<driver-version>  ← injected      │   │
│  └──────────────────────────────────────────────┘   │
│                       /dev/nvidia0  (bind-mounted)  │
└──────────────────────────────┬──────────────────────┘
                               ↓
┌──────────────────────────────┴──────────────────────┐
│                   HOST KERNEL                       │
│  nvidia.ko  (driver kernel module, owns the GPU)    │
│  /usr/lib/x86_64-linux-gnu/libcuda.so.X.Y.Z          │
└─────────────────────────────────────────────────────┘

The three components

  1. nvidia-container-toolkit — installed on the host. Contains the runtime hook and CLI utilities. Configures containerd or Docker.
  2. nvidia-container-runtime — the OCI-runtime shim. /etc/containerd/config.toml declares it as a runtime, and the device plugin assigns it to GPU pods via runtimeClassName.
  3. libnvidia-container — the C library that performs the actual mount injection. The thing that turns a manifest stanza into a working /dev/nvidia* bind and a populated /usr/lib.
# /etc/containerd/config.toml — relevant fragment
[plugins."io.containerd.grpc.v1.cri".containerd.runtimes.nvidia]
  runtime_type = "io.containerd.runc.v2"
  [plugins."io.containerd.grpc.v1.cri".containerd.runtimes.nvidia.options]
    BinaryName = "/usr/bin/nvidia-container-runtime"
The two failure modes you'll see weekly

1. Library mismatch: container ships its own libnvidia-ml.so in /usr/lib; the injected one from the host lands in /usr/lib64 (or vice versa); LD_LIBRARY_PATH resolves the wrong one; CUDA_ERROR_SYSTEM_DRIVER_MISMATCH. Fix: don't bundle nvidia libs in the image (let injection win) or set LD_LIBRARY_PATH explicitly. The nvidia/cuda base images get this right; rolling your own base requires care.

2. nvidia-smi works on the host but fails in pods. Almost always: containerd config doesn't reference nvidia-container-runtime as a runtime, or RuntimeClass object is missing in the cluster, or the device plugin isn't running on the node. Verify in order: nvidia-smi on host → nvidia-container-cli info → kubectl get runtimeclasses → kubectl describe node | grep nvidia.com/gpu.

§07CUDA, cuDNN, NCCL, TensorRT — the compatibility matrix

This is the most boring and most consequential topic in the entire stack. Wrong combinations don't crash at startup; they crash deep in a kernel two hours into a long-running job, or worse, silently produce numerically wrong outputs.

What pins what

LibraryPinned byForward-compat?
Driverhost OS / sysadminYes — CUDA forward-compat package lets newer CUDA run on older drivers in some cases
CUDA Toolkitimage (nvidia/cuda:X.Y)Minor versions usually backward-compatible

The TensorRT engine trap

TRT engines are compiled artifacts tied to (TRT version × CUDA version × GPU SM architecture × precision). An engine built for TRT 10.3 on H100 (SM 9.0) will refuse to load on TRT 10.4 or on an A100 (SM 8.0). This is why NIMs ship multiple "profiles" — pre-compiled engines per GPU class. Production debug: trtexec --loadEngine=… with verbose logging tells you the version skew immediately.

CUDA forward compatibility — when it saves you, when it doesn't

The CUDA Forward Compatibility package lets a CUDA 12.6 application run on a host with a CUDA 12.2-era driver. Conditions: only the data-center driver line (R-series), only forward (you can't go backward), and only between minor versions. Not magic, but enough to decouple your image cadence from the host's driver upgrade cadence — important when you don't own the customer's nodes.

Part III

Kubernetes for inference workloads

Kubernetes is generic; LLM inference is not. The default abstractions miss the fact that startup is slow, models are huge, GPUs are precious, and topology matters. The chapters below cover the actual deployment patterns NIMs (and any production inference service) need.

§08Core workload patterns for inference

Three things decide your workload choice: is the model state on PVC or downloaded fresh? does each replica need a stable identity (for sharding)? is there one replica per node?

PatternWhenNotes
DeploymentStateless replicas, model fetched at start from cache or registryDefault for most NIMs. HPA on it. Rolling update is straightforward.
StatefulSetTensor-parallel sharding needs stable pod identity, or each replica owns a PVC with cached weightsUse with a Headless Service for peer discovery. Slow rolling updates can be painful — consider partitioned updates.
DaemonSetPer-node agents: DCGM exporter, device plugin, image preloader, node-feature-discoveryOne per node by design. Use nodeAffinity to restrict to GPU nodes.
Job / IndexedJobBatch inference, embedding pipelines, async model compilationIndexedJob (1.27+) for sharded batch — each pod gets a stable index.
LeaderWorkerSetMulti-host TP inference (one leader, N workers per replica)Kubernetes SIG project; the right abstraction for cross-node tensor parallel.
The "one replica spans N nodes" problem

For a 405B-parameter model with tensor-parallel-8 across two H100 nodes, no native workload primitive handles this cleanly. The community answer is LeaderWorkerSet (LWS) — a custom workload type from Kubernetes SIG Apps that treats N pods as one replica with shared lifecycle. It's worth knowing about because it's the direction the ecosystem is heading for cross-node parallel inference.

§09Probes, lifecycle, graceful drain — tuned for LLMs

Default probe settings are wrong for inference. A model takes 30–300 seconds to load weights into GPU memory before it can serve traffic. With the default liveness probe (10s after start, 3 failures), the pod gets killed before it ever loads.

startupProbe:           # gates liveness/readiness until model loaded
  httpGet: { path: /v1/health/ready, port: 8000 }
  periodSeconds: 10
  failureThreshold: 60     # 10 minutes — generous for big models
readinessProbe:
  httpGet: { path: /v1/health/ready, port: 8000 }
  periodSeconds: 5
  failureThreshold: 2
livenessProbe:
  httpGet: { path: /v1/health/live, port: 8000 }
  periodSeconds: 30
  failureThreshold: 3
lifecycle:
  preStop:
    exec: { command: ["/opt/nim/drain.sh"] }
terminationGracePeriodSeconds: 120

Two endpoints, not one

Liveness ("am I crashed?") and readiness ("can I take traffic?") need separate signals. A pod with a healthy event loop but a wedged inference engine should fail readiness, not liveness — taking traffic off lets you debug; killing the pod loses state. NIMs expose both: /v1/health/live and /v1/health/ready.

Graceful drain for streaming endpoints

An inflight chat-completions request can take 30+ seconds. On pod termination, you need to: (1) fail readiness immediately (Service removes from endpoints), (2) finish in-flight streams, (3) only then SIGTERM the process. A preStop hook that hits a /shutdown endpoint, waits for inflight count to hit zero (with a cap), and exits is the right pattern. terminationGracePeriodSeconds must exceed your longest reasonable request.

§10NVIDIA device plugin, GPU Operator, and DRA

How does Kubernetes know a node has GPUs? Not through magic — through the NVIDIA device plugin, a DaemonSet that runs on every GPU node, talks to nvml, and advertises nvidia.com/gpu: 8 to the kubelet. The kubelet exposes that as a schedulable resource. The scheduler matches pod resources.limits.nvidia.com/gpu against node capacity. End of story.

GPU Operator — the integrated bundle

Running the device plugin alone is a 2018 setup. GPU Operator manages the full stack as a single Helm release: driver, container toolkit, device plugin, DCGM exporter, GFD (GPU Feature Discovery), MIG manager, and node-feature-discovery. Driver is containerized (a privileged DaemonSet pod) so you don't need to install anything on the host. This is what production NIM customers run.

  • node-feature-discovery (NFD) — labels nodes with GPU details (nvidia.com/gpu.product=H100-SXM5, nvidia.com/gpu.memory=80, nvidia.com/mig.capable=true). These labels become pod nodeSelector targets.
  • GPU Feature Discovery (GFD) — NVIDIA's variant of NFD, produces NVIDIA-specific labels via NVML.
  • DCGM exporter — Prometheus exporter for GPU metrics. The single most important observability component. See §24.
  • MIG manager — applies MIG profiles to nodes based on labels. Requires node drain + GPU reset.

Dynamic Resource Allocation (DRA) — the future

The device plugin model is a counted resource: nvidia.com/gpu: 1 is opaque. DRA, GA in recent Kubernetes, replaces counted devices with structured ResourceClaims that can express richer constraints: "give me a GPU with at least 40 GB memory, MIG profile 1g.10gb, with NVLink to the GPU allocated for sibling pod." NVIDIA's DRA driver implements this. Expect DRA to subsume the device plugin for sophisticated workloads (multi-host TP, MIG-aware scheduling) over the next year or two.

The decision today

For most users: GPU Operator with the classic device plugin. For multi-host inference, MIG-mixed clusters, or workloads needing topology constraints: evaluate the DRA driver. Don't roll your own — the surface area is enormous.

§11GPU sharing strategies — MIG, time-slicing, MPS, full

The most over-asked, most-misunderstood topic in this whole space. There are four mechanisms and they are not interchangeable; each has different isolation, performance, and operational properties.

MIG operational realities
  • MIG profiles are configured at the device level and require node drain + GPU reset to change. Mixed-profile clusters need careful capacity planning.
  • MIG instances appear as separate GPUs to the device plugin (nvidia.com/mig-1g.10gb: 7). Pods request a profile, not a count.
  • No NVLink between MIG instances on the same GPU — they're isolated. Tensor parallelism within MIG is not viable.
  • MPS and MIG can be combined (MPS within a MIG instance), but it's rarely worth the complexity.

§12Topology — NUMA, NVLink, PCIe, GPUDirect

Inside a DGX H100 node: 8 GPUs, 2 CPU sockets, 8 NICs. Pin the wrong GPU to the wrong NIC and your AllReduce latency doubles. Topology awareness matters from the scheduling layer down to the NCCL config.

The relevant hardware facts

  • NVLink/NVSwitch — GPU-to-GPU fabric. H100 NVLink Gen4 = 900 GB/s per GPU. Within-node tensor parallel must go over NVLink, not PCIe.
  • PCIe — GPU-to-CPU and (slow path) GPU-to-NIC. Gen5 x16 = 64 GB/s.
  • GPUDirect RDMA — NIC writes directly into GPU memory, bypassing CPU. Required for high-throughput cross-node TP. Needs ConnectX NICs and a kernel module.
  • NUMA — pin the inference process to the CPU socket whose PCIe root complex owns its GPUs. Wrong pinning = cache thrash + slow data movement.
# inspect topology
nvidia-smi topo -m              # GPU-to-GPU and GPU-to-NIC matrix
lstopo --of png > topo.png      # full hwloc map of the box
numactl --hardware              # NUMA layout

Kubernetes' Topology Manager (kubelet feature) aligns CPU, memory, and device allocations to the same NUMA node when policy is single-numa-node. Without it, the kernel may give you a GPU on socket 0 and CPUs on socket 1 — silently slow.

Smell test for cross-node TP

If multi-node inference is on the roadmap, verify three things on day one: (1) NICs are ConnectX-7 or better with GPUDirect RDMA capable, (2) the cluster has a non-blocking RDMA fabric (InfiniBand or RoCEv2), (3) NCCL can be configured to use the right HCA per GPU (NCCL_IB_HCA). One missing piece and cross-node TP latency is unworkable.

§13Autoscaling for inference

Three layers of autoscaling, and they don't compose automatically. Get each right separately.

LayerToolSignal
Pod replicasHPA / KEDACPU/GPU utilization is the wrong signal for LLMs. Use queue depth, request concurrency, TTFT p99, or GPU memory pressure via DCGM.
Cluster nodesCluster Autoscaler / KarpenterPending pods. Karpenter is faster and supports mixed instance types.
Vertical (within pod)VPA (rarely for inference)GPU pods get whole GPUs; vertical resizing doesn't translate. Skip.

Why GPU utilization is a trap as an HPA metric

The GPU's "utilization" reported by nvidia-smi is the fraction of time any kernel was running, not how loaded those kernels were. A model serving one request per second can show 80% utilization while having huge headroom. Use application-level signals: tokens-per-second, queue length, batch fill rate. KEDA shines here because it scales on Prometheus queries — so any DCGM or Triton metric works as a scale signal.

Scale-up lag is the hardest problem

A new replica needs: pod schedule (seconds) + image pull (minutes if cold) + model load (minutes) + warmup (seconds–minutes). 5–15 minutes total. That means you must overprovision against expected demand or accept latency cliffs at scale boundaries. Mitigations: (a) keep a warm reserve of pods at low capacity (b) preload images via DaemonSet (c) use lazy snapshotters to start before pull completes (d) prebuilt model caches on persistent local SSD per node.

Part IV

LLM inference engines & NIMs

You don't need to be an inference researcher, but you need to know what each backend does well, why, and how it's packaged. Container/infra decisions are downstream of these.

§14LLM inference primer — prefill, decode, KV cache

A transformer inference request has two phases with wildly different characteristics, and almost every interesting inference optimization is downstream of this asymmetry.

PhaseWorkBottleneck
PrefillProcess the prompt; compute K and V tensors for all input tokens in parallel; produce first output token.Compute-bound (matrix-matrix multiplies). High FLOPs, short wall-time.
DecodeOne token at a time, repeatedly. Each step reads the full KV cache and produces one new token.Memory-bandwidth bound. KV cache grows with sequence length.

The KV cache dominates memory in modern serving. For Llama-3-70B at sequence length 4k, KV cache per request is ~1.3 GB. With 80 GB GPUs and a model that takes 40 GB, you have 40 GB for KV — i.e., 30-ish concurrent requests at that length. Every cleverness in modern inference servers — paged attention, prefix caching, prefill/decode disaggregation — is fundamentally about KV cache management.

Continuous batching

Naïve batching waits for all requests in a batch to finish — bad, because outputs have different lengths. Continuous batching (a.k.a. in-flight batching, iteration-level batching) re-forms the batch at every decode step, swapping completed sequences out and new prefills in. This is the table-stakes feature; every modern engine implements it.

§15The four engines you need to know

vLLM

The reference open-source engine, originated at Berkeley. Key contribution: PagedAttention — KV cache stored in fixed-size blocks rather than contiguous arrays, so memory fragmentation is eliminated and prefix-sharing becomes cheap. Continuous batching, prefix caching, speculative decoding, FP8, AWQ/GPTQ/INT4 quantization. Python with CUDA kernels. Fast dev velocity, broad model coverage. The default choice for "I have a new HuggingFace model, serve it tomorrow."

SGLang

From the same lab lineage. Key contribution: RadixAttention — a radix tree over KV-cache prefixes, enabling aggressive cross-request prefix sharing. Particularly good for structured generation (constrained decoding, JSON mode) and agent workflows where many requests share a long system prompt. Often benches faster than vLLM at high concurrency.

TensorRT-LLM

NVIDIA's. The model is compiled into a TRT engine for a specific GPU + precision combination. Peak performance — usually the fastest single-request and single-batch on Hopper hardware — but the compilation step (5–30 minutes per model/GPU/precision combination) and engine-binary distribution are real operational costs. Supports FP8, INT8, INT4, multi-GPU TP/PP, in-flight batching. NIMs that target Hopper-class hardware ship TRT-LLM engines by default.

Triton Inference Server

NVIDIA's serving framework, not an inference engine itself. It hosts backends — TRT-LLM, vLLM, ONNX, PyTorch, Python custom — behind one HTTP/gRPC interface, with batching, model ensembles, and multi-model multiplexing. NIMs use Triton as the outermost server with TRT-LLM (or vLLM) as the backend.

Decision: pick the engine
  • Max throughput, one model, willing to compile → TRT-LLM via Triton
  • Broad model coverage, fast iteration, OSS → vLLM
  • Heavy prefix sharing, structured outputs → SGLang
  • Multiple models on one server, mixed modalities → Triton with multiple backends

§16NIM architecture & the OpenAI API surface

A NIM, conceptually: a container that bundles a model, the optimal inference engine for that model on the target hardware, an OpenAI-compatible HTTP server, observability instrumentation, and a license layer. From the customer's perspective: docker run nvcr.io/nim/llama3:24.10 and an OpenAI-compatible endpoint comes up at port 8000.

Surface area you should know

  • OpenAI-compatible endpoints — /v1/chat/completions, /v1/completions, /v1/embeddings, streaming via SSE, function calling. The point of OpenAI compat is that customer code doesn't change.
  • NIM-specific — /v1/models (lists profile metadata), /v1/health/ready, /v1/health/live, /v1/metrics (Prometheus).
  • Profiles — at start, the NIM inspects the GPU (model + count + memory) and selects the best precompiled engine. H100×8 picks a different profile than H100×1 picks a different profile than L40S×1.
  • Model cache — typically /opt/nim/.cache. Hydrated from NGC, a private mirror, or a local model store. For air-gapped, point NIM_MODEL_NAME + NIM_CACHE_PATH at a preloaded volume.

Profile selection failures

Mismatched profile is one of the more confusing failure modes. The NIM logs an exhaustive list of available profiles and which one it selected. A common pattern is "this NIM supports H100 and A100 but not L40S, container starts on an L40S node, prints a profile-match-failed error, and exits." Platform engineering — Helm, Operators, Python

§17Helm chart design at platform scale

One-off charts are easy. A chart design system — many charts shared across many teams with consistent values schemas, common helpers, version lifecycle — is what the JD is asking about. The mental model is "design system" the way frontend has design systems: opinionated primitives that compose.

Building blocks

  • Library charts — chart type library, shipped with helpers (_helpers.tpl) and partials but no renderable templates. Application charts pull these in as dependencies. This is how you enforce consistency: every NIM chart includes common-nim-1.x and gets standard labels, annotations, probes, security contexts for free.
  • Schema validation — values.schema.json. Catches misconfiguration at helm install time instead of at pod-crash time. The single best Helm hygiene practice.
  • Named templates — {{- define "nim.commonLabels" -}}. Consistency without copy-paste.
  • Hooks — pre-install (e.g., create CRDs), post-upgrade (e.g., trigger reload). Hooks run as Jobs and have their own lifecycle.
  • Subchart values — passing values down via .Values.subchart.foo or via global. Easy to mess up.
# common-nim/templates/_probes.tpl — example library partial
{{- define "common-nim.probes" -}}
startupProbe:
  httpGet: { path: /v1/health/ready, port: http }
  periodSeconds: {{ .startup.period | default 10 }}
  failureThreshold: {{ .startup.threshold | default 60 }}
readinessProbe:
  httpGet: { path: /v1/health/ready, port: http }
  periodSeconds: 5
livenessProbe:
  httpGet: { path: /v1/health/live, port: http }
  periodSeconds: 30
{{- end -}}
The immutability traps
  • StatefulSet volumeClaimTemplates are immutable post-create. Schema changes require helm uninstall + helm install, not helm upgrade.
  • Service type and clusterIP are immutable. Switching ClusterIP→LoadBalancer requires recreate.
  • PVCs from volumeClaimTemplates survive StatefulSet deletion (intentional — protects data) and conflict on reinstall.
  • Selector labels on Deployments are immutable. Any matchLabels change requires recreate.

Build a CI test that does helm template against a matrix of values and runs kubeconform + kube-linter. Catch breakage before customers do.

§18Operators — CRDs, controllers, reconcile loops

Helm is push-mode: you run helm upgrade, things happen. An Operator is reconcile-mode: a controller watches a CRD and continuously drives the cluster toward the desired state. The right tool when state is dynamic — autoscaling decisions, model lifecycle, lease management.

The reconcile loop

def reconcile(req: Request) -> Result:
    obj = client.get(req.namespaced_name)        # observe
    if obj.deletion_timestamp:                   # cleanup path
        return handle_finalizer(obj)
    desired = compute_desired_state(obj.spec)    # pure function
    current = read_actual_state(obj)             # observe children
    if desired == current:
        update_status(obj, ready=True)
        return Result(requeue_after=300)         # level-triggered
    apply_changes(desired - current)
    update_status(obj, ready=False, reason="converging")
    return Result(requeue=True)

Two principles to internalize:

  • Level-triggered, not edge-triggered. Reconcile examines current state and acts; it doesn't track events. Missed events don't cause incorrect state — the next reconcile finds the discrepancy.
  • Idempotent. The same input twice produces the same output. Reconcile may be called repeatedly; that's fine.

When to write an Operator vs a Helm chart

Build Helm chart ifBuild Operator if
Install / upgrade / uninstall is the lifecycleState changes over time without user intervention
Configuration is mostly staticConfiguration is computed from cluster state
Day-2 operations are manual (cron, kubectl)Day-2 operations need automation: rollouts, repair, scaling
One team owns deploymentsMany teams self-serve via a CRD API
Operator anti-patterns to avoid
  • Reconcile storms. An operator that updates its own CR triggers itself. Use predicate functions to filter spec-only changes; never reconcile on status updates from yourself.
  • Hot loops. Always set requeue intervals; never return Result(Requeue: true) from a normal-state path.
  • Cross-namespace coupling. Decide upfront: cluster-scoped operator (one install for everything) or namespace-scoped (one per tenant). Mixing makes RBAC a nightmare.
  • Schema drift. CRD versions are forever. Plan v1alpha1 → v1beta1 → v1 with conversion webhooks from day one.

§19Production-grade Python tooling

"Python tooling and services for build orchestration, CI/CD integrations, Helm/Operator automation, and test harnesses." This is the substrate that ties the rest together. The bar is engineering hygiene, not exotic features.

The non-negotiables

  • Strict typing — mypy --strict or pyright in CI. Public APIs annotated; Protocol for duck typing where genuinely useful.
  • Linting + formatting — ruff for both. One tool, fast, configured once.
  • Packaging — modern pyproject.toml, uv or pdm or poetry for dependency management. uv is currently the speed-of-light option.
  • Testing — pytest. Unit tests are pure; integration tests use testcontainers for real containerd / registry. Mock the Kubernetes API at the kubernetes_asyncio layer, not at HTTP.
  • Async by default — asyncio + httpx + kubernetes_asyncio for any service that hits multiple APIs. Blocking I/O is rarely OK in 2026.

The Kubernetes Python client — patterns

from kubernetes_asyncio import client, config, watch

Watching is a contract, not a feature

Kubernetes watches close periodically (server-side timeout), watches can miss events under load, and the resource version can age out. A robust watcher uses an informer pattern: list to bootstrap state, watch for incremental updates, re-list on 410 Gone. The kopf framework wraps this. For more control, build on kubernetes_asyncio directly but match the pattern.

Testing infrastructure code

  • Unit: mock kubernetes_asyncio objects directly. pytest-asyncio for async tests.
  • Integration: spin up kind or k3d in CI; install your operator; assert on real cluster state. pytest-kind helps.
  • End-to-end: a long-lived test cluster with GPU nodes (expensive). Run a smaller suite, gated on main-branch merges.
  • Container tests: testcontainers-python launches real containers and asserts on their behavior. Useful for build-tool tests where you want a real registry.

§20CI/CD for container pipelines

A NIM factory CI/CD pipeline isn't just "build and push." It's: resolve dependencies → matrix build (CUDA × Python × backend × arch) → unit test → smoke test the resulting container → push by digest → run e2e on a real GPU cluster → generate SBOM and signatures → promote across environments by digest → trigger rollout.

The cache problem

CUDA images are huge. Without effective caching, your CI is in a 30-minute build per change. BuildKit has three cache modes:

  • Registry cache (--cache-to type=registry,ref=…) — pushes cache manifests to a registry. Works across CI runners.
  • Inline cache (--cache-to type=inline) — embeds cache metadata in the image itself. Simple, lower hit rates.
  • S3/GCS cache (--cache-to type=s3,…) — for very large caches; bypasses registry limits.

For a build farm, combine: registry cache for shared layers + local builder cache for hot iterations + cache-mounts in Dockerfile for package managers (pip, apt). Cache hit rate >85% is the target.

Promotion by digest

Build once, promote everywhere. The artifact is the digest. CI builds :dev-2026-05-12-abc123 at digest sha256:…; tests run against that digest; staging deploy references that digest; production deploy references that same digest. No rebuilds across environments. This is the only way to be sure that what you tested is what you shipped.

The GitOps overlay

For multi-cluster delivery: ArgoCD or Flux watching a Git repo of manifests. Image promotion is a PR that bumps a digest in values.yaml. Argo Image Updater automates the bump on tag policy. Audit trail is your git history. Rollback is git revert.

Part VI

Performance — image, startup, runtime

"Optimize container performance: layer layout, startup time, build caching, runtime memory/IO, network, and GPU utilization." Five orthogonal axes. Two chapters cover image+startup and runtime+networking, plus one on disaggregated inference as the emerging frontier.

§21Image & startup optimization

Image size — order of operations

  1. Multi-stage to strip build-time-only artifacts. Often the single biggest win — devel→runtime saves 2–5 GB.
  2. Layer order by mutability: OS → CUDA → Python deps → app code. Volatile last.
  3. COPY --link for hardlink semantics — independent layers, better cache reuse.
  4. Single RUN per logical unit with && and cleanup in the same layer. Don't have apt-get install in one layer and rm -rf /var/lib/apt/lists/* in the next — the files persist in the lower layer.
  5. Distroless or slim base for runtime stage when CUDA-base isn't required. nvidia/cuda:runtime is already fairly slim.
  6. Separate model from image. A 15 GB image with weights baked in is a deployment anti-pattern. Ship a lean image; fetch weights on first start to a cache volume; subsequent restarts are fast.

Startup — the lazy-pull trick

Default snapshotters (overlayfs) require the entire image to be present before the container starts. Lazy snapshotters serve files on demand:

  • eStargz — stargz-snapshotter. Re-packs the image so files can be fetched individually from the registry.
  • SOCI (Seekable OCI) — AWS-led. Decouples lazy-load index from the image itself; doesn't require image re-packing. Currently the operationally cleanest option.
  • Nydus — block-level format with content-addressed chunks; very efficient for large images with shared chunks across images.

The win: an 18 GB image that previously took 4 minutes to pull before container start can now start in 8 seconds — files are fetched as the process opens them. For NIMs, this is transformative on scale-out events.

The supporting infrastructure

Lazy pull only works when the registry supports range requests (all OCI-compliant registries do) and when containerd has the corresponding snapshotter plugin configured. Network latency to the registry directly affects cold-cache request latency. Pair lazy pull with a regional registry or P2P distribution (Dragonfly, Spegel) for the full benefit.

Probe-tuning math

For a model that loads in T seconds (P99), set:

startupProbe.failureThreshold * periodSeconds >= 2 * T
readinessProbe.failureThreshold * periodSeconds >= warmup_p99
livenessProbe.failureThreshold * periodSeconds >= longest_legitimate_stall

terminationGracePeriodSeconds >= longest_inflight_request + drain_buffer

Before debugging deeply: nccl-tests/build/all_reduce_perf -b 8 -e 1G -f 2 -g 8. Compare to known-good numbers for your hardware (e.g., H100 all-reduce ~470 GB/s on 8 GPUs). If you're 5–10× off, the problem is topology or transport, not your model.

§23Disaggregated inference

The emerging deployment pattern mentioned explicitly in the JD. The idea: prefill and decode have such different resource profiles (§14) that running them on the same GPU is inefficient. Prefill saturates compute briefly; decode is memory-bandwidth bound for long stretches. Disaggregating them onto separate pools lets each be tuned independently and avoids head-of-line blocking.

┌──────────────┐    KV cache     ┌──────────────┐
│   PREFILL    │ ──────────────→ │    DECODE    │
│  GPU pool    │  (NVLink/RDMA)  │   GPU pool   │
│  H100 ×4     │                 │  H100 ×8     │
│ compute-bound│                 │ bw-bound     │
└──────┬───────┘                 └──────┬───────┘
       ↑                                ↓
   request in                      tokens out

What this implies for infrastructure

  • KV cache transfer — gigabytes of state move from prefill to decode pods per request. Needs RDMA between nodes or NVLink within. Plain TCP won't work at scale.
  • Pool sizing — independent autoscaling on prefill and decode. Decode demand is sticky (lives for the request's duration); prefill demand is bursty (lives for a millisecond per request).
  • Routing layer — a request scheduler that picks prefill and decode workers, manages KV handoff, and recovers from worker failures mid-request. This is non-trivial; NVIDIA Dynamo and similar frameworks are emerging here.
  • Failure semantics — if a decode worker dies mid-stream, do you retry from scratch (re-prefill) or migrate KV to another decode worker? Both have edges.
Why this matters for the role

The container/infra side of disaggregated inference is what the JD's "support for disaggregated LLM inference and other emerging deployment patterns" gestures at. Specifically: workload primitives for cohorted pods (LeaderWorkerSet, or successor), topology-aware scheduling (NVLink-affinity between prefill and decode), and the routing/control plane. This is greenfield in 2026.

Operations — observability, multi-tenancy, edge

§24Observability — DCGM, OpenTelemetry, SLIs

Three layers, three tools. Don't conflate them.

  • Infrastructure metrics — Prometheus + kube-state-metrics + node-exporter.
  • GPU metrics — DCGM exporter. This is the canonical NVIDIA observability tool. Exposes utilization, memory, temperature, power, ECC errors, NVLink throughput, XID errors. Run as a DaemonSet on GPU nodes; scrape via Prometheus.
  • Application metrics — Triton and most engines expose Prometheus endpoints. TTFT, TPOT, throughput, queue depth, batch fill rate.
  • Traces — OpenTelemetry. Track a request through gateway → router → backend. Crucial for understanding tail latency.
  • Logs — structured JSON, with trace IDs. Fluent Bit → Loki / OpenSearch / Cloudwatch.

The metrics that matter for inference SLOs

XID errors — the early warning system

XID is NVIDIA's GPU error reporting facility. XID 13 (graphics-engine error), XID 31 (MMU fault), XID 79 (GPU fell off the bus), XID 109 (CTX switch timeout) — each tells you something specific. nvidia-smi shows the recent XIDs. DCGM exporter exposes them as metrics. Alert on any non-zero XID rate. XID 79 in particular is the harbinger of imminent GPU loss — drain the node now.

Cardinality discipline

Per-model, per-tenant, per-region labels multiply fast. Prometheus storage and query latency degrade nonlinearly with cardinality. Use label_replace + recording rules to keep the high-cardinality joins in derived metrics, not the raw scrapes.

§25Multi-tenancy, multi-cluster, air-gapped

Multi-tenancy primitives

  • Namespaces — soft isolation. RBAC per namespace, ResourceQuotas to bound GPU consumption per tenant.
  • NetworkPolicies — restrict pod-to-pod traffic. Default-deny ingress in every namespace; allow only what's needed.
  • PodSecurity admission — restricted profile by default; baseline as the compromise for workloads that need NET_ADMIN, etc.
  • vClusters — virtual clusters within a host cluster. Stronger isolation than namespaces; weaker than separate clusters. Each tenant gets their own control plane, sharing the workload plane.
  • Hard multi-tenancy — separate clusters. The only safe option for adversarial workloads on GPU; MIG within a shared cluster is fine for trusted multi-tenancy.

Multi-cluster patterns

Federation v1 is dead. Modern options:

  • Karmada / Open Cluster Management — propagate manifests from a hub to spokes with placement policies.
  • Argo CD ApplicationSet — GitOps-driven multi-cluster delivery. Simpler than Karmada for most cases.
  • Cluster API — declarative cluster lifecycle. The cluster itself becomes a CRD.
  • Submariner / Cilium Cluster Mesh — pod-to-pod networking across clusters.

Air-gapped delivery

The hardest distribution problem. The customer has no internet egress, possibly no inbound either. Everything ships in: container images, Helm charts, model weights, dependencies, license artifacts, sometimes the cluster itself.

  • Local OCI registry seeded from a bundle (Harbor, Distribution, or zot)
  • Model weights mirrored as OCI artifacts in the local registry
  • Helm charts pulled from a local chart repo (or the same OCI registry — Helm v3 supports OCI)
  • OS package mirror for driver/toolkit dependencies (if the cluster manages its own OS)
  • Time-bound license artifacts pre-loaded; license renewal flow that works without phone-home
  • A relocate tool: takes a Helm chart's values.yaml referencing public registries and rewrites it to the local mirror. chart-syncer or relok8s.
The shipping format

A common pattern: produce a single tarball ("airgap bundle") containing all images (as docker save archives), all charts, all weights, a manifest of what's inside, signatures, and an install script. Customer copies tarball over their data diode, runs load-bundle.sh, and the cluster comes up. This bundle becomes a release artifact in its own right — versioned, signed, and tested in a sealed-environment CI job.
