FOUNDATION Chapter I

The OCI image model · what's actually in an image.

The premise Before you optimize an image, understand what it actually is. The OCI Image Specification is short and the entire ecosystem builds on it.

Three documents Manifest, config, layers. Plus an index for multi-arch.

A container image is not a file. It's a collection of tarballs (layers) plus two small JSON documents that describe how to use them. The Open Container Initiative codified this in the OCI Image Specification, which is the lingua franca that every container runtime, registry, and build tool now speaks.

The three documents (four for multi-arch)

Layers. Each layer is a tarball — usually gzip or zstd compressed — of filesystem changes. Add a file, you get a layer with that file. Delete a file, you get a layer with a "whiteout" marker. Modify a file, the whole file is copied into a new layer. Layers are content-addressed by sha256 of their compressed bytes.

Config. JSON describing what the image is: architecture, OS, environment variables, working directory, exposed ports, entrypoint, command, labels, history. The config is itself content-addressed; its digest is what people mean when they say "the image ID."

Manifest. JSON pointing at the config and the layers, by digest. The manifest is the image identity — when you sign or verify an image, you're signing its manifest digest.

Image index (manifest list). JSON pointing at multiple manifests, one per platform. myapp:v1 as a tag points at an index; the index says "for linux/amd64 use manifest A, for linux/arm64 use manifest B."

# Pull the manifest (or index)
$ crane manifest gcr.io/distroless/static:nonroot
{
  "schemaVersion": 2,
  "mediaType": "application/vnd.oci.image.index.v1+json",
  "manifests": [
    { "digest": "sha256:a1b2...", "platform": { "architecture": "amd64", ... } },
    { "digest": "sha256:c3d4...", "platform": { "architecture": "arm64", ... } }
  ]
}

# Resolve to a specific manifest
$ crane manifest --platform linux/amd64 gcr.io/distroless/static:nonroot
{
  "schemaVersion": 2,
  "mediaType": "application/vnd.oci.image.manifest.v1+json",
  "config": { "digest": "sha256:...", "size": 1827 },
  "layers": [
    { "digest": "sha256:...", "size": 115724 }
  ]
}

# Examine the config
$ crane config gcr.io/distroless/static:nonroot | jq .config
{ "User": "65532:65532", "Entrypoint": [...], ... }

Why "image digest" beats "image tag"

A tag is a label that can move; a digest is the cryptographic identity of the bytes. myapp:latest can point to a different image tomorrow without anyone noticing. myapp@sha256:a1b2c3d4... will always be the same bytes — and if a registry ever returns different bytes for that digest, your runtime detects it and refuses to run.

The implication: pin by digest in production, not by tag. In CI, build and push by tag; promote to staging and prod by digest. The artifact reviewed in CI is byte-for-byte the artifact running in production.

Media types and OCI artifacts

The image manifest declares a mediaType — historically Docker-specific (application/vnd.docker.distribution.manifest.v2+json), now standardized as OCI (application/vnd.oci.image.manifest.v1+json). Most modern tools emit OCI by default.

The interesting extension: OCI artifacts. Since the manifest is just a pointer to blobs by media type, you can store anything in a registry that follows the OCI Distribution spec — Helm charts, SBOMs, signatures, Terraform modules, AI model weights, even arbitrary tarballs. oras push is the standard tool. Registries become a universal artifact hub, not just an image hub.

The single most-useful inspection command

crane manifest IMAGE, crane config IMAGE, crane export IMAGE - | tar -t | less. Together these answer "what is in this image" without pulling and running it. Add dive for a visual layer-by-layer view; syft IMAGE -o table for the SBOM. These four tools are what separates "I built an image" from "I understand the image I built."

BASE Chapter II

Base image strategy · scratch → distroless → Wolfi.

The shift In 2026, the question isn't "Alpine or Ubuntu" — it's "scratch, distroless, or a maintained distroless distribution like Wolfi or Chiseled Ubuntu."

The tradeoff Smaller and more locked-down = harder to debug, less ecosystem support. Pick by application type.

Base image choice is the single biggest determinant of an image's security posture, attack surface, and CVE exposure. The shift over the last three years has been from full distributions (Debian, Ubuntu, Alpine) toward minimal-by-construction alternatives that ship with no shell, no package manager, and continuous rebuilds against the latest patched packages.

Interactive · The seven base-image options

The decision tree

Pure static binary (Go, Rust statically linked): scratch or gcr.io/distroless/static. Difference: static includes ca-certs, tzdata, /etc/passwd entries. Use static 90% of the time; scratch when you've explicitly handled those yourself.

glibc-linked binary, needs libc: gcr.io/distroless/base or cgr.dev/chainguard/glibc-dynamic. Wolfi-based images get rolling updates; Google distroless gets less frequent updates.

Interpreted language (Python, Node, Java, Ruby): distroless language images (gcr.io/distroless/python3, node, java) or Chainguard's equivalent. Avoid FROM python:3.X-slim for production — too many extras.

Greenfield, want continuous CVE patching: Wolfi (Chainguard). Wolfi uses a rolling release model built for container environments from the ground up; CVEs are patched as they are disclosed rather than on a versioned release cycle.

Enterprise compliance (FedRAMP, FIPS): Red Hat UBI or Iron Bank. Pay for the support.

Ecosystem fit with existing tooling: Chiseled Ubuntu (Microsoft .NET defaults to this). Ubuntu compatibility, distroless attack surface.

The "Alpine is not distroless" caveat

Alpine ships with a shell (busybox sh), a package manager (apk), and a working filesystem. It's small (~5 MB compressed), but Alpine is not distroless — it ships with a shell and the apk package manager. Alpine's musl libc also causes subtle compatibility issues for some glibc-dependent code (DNS resolution differences, threading edge cases, performance regressions on some libraries). For production runtime: prefer a true distroless or scratch. For build stages or dev environments where you need apk and a shell: Alpine is great.

The Wolfi argument

Wolfi was created by Chainguard specifically as a container-native distribution. Three things make it different from Alpine:

One — glibc, not musl. No mysterious compatibility breaks. glibc compatibility makes Wolfi suitable for applications that cannot tolerate musl libc quirks.

Two — rolling releases, daily rebuilds. Chainguard rebuilds images continuously, generates SBOMs natively using Sigstore and Cosign for cryptographic signing, and publishes SLSA Level 3 build attestations per image digest. The near-zero CVE count that Chainguard images maintain comes from continuous rebuilds rather than a one-time hardening pass.

Three — built with apko/melange, declarative tools that produce build-time SBOMs natively. Wolfi provides a high-quality, build-time SBOM as standard for all packages.

Limitation: Wolfi focuses on latest package versions; non-latest versions are retained for 12 months and then 6 months by default. No LTS concept — you ride the rolling train. For most cloud-native workloads, that's the right tradeoff.

The 2026 default recommendation

For new services: Wolfi or distroless. For Go/Rust statically compiled: scratch or distroless/static. For Python/Node/Java: the language-specific distroless or Chainguard image. Use Alpine only as a build stage, not as a runtime base. Use the full distros (Ubuntu, Debian) only when a specific dependency requires it and you've documented why. The vulnerability differential between a distroless runtime and a full distribution runtime is typically 15–50 CVEs to zero, even for the same exact application code.

BASE Chapter III

Pinning & the update cadence.

The dual pressure You want reproducible builds (pin everything) AND timely security patches (update often). These pull in opposite directions; tooling resolves the tension.

The resolution Pin by digest. Update by automation. Verify by tests.

A Dockerfile that starts with FROM python:3.12-slim is non-reproducible: the same Dockerfile builds different bytes today than it did yesterday, because the base tag's underlying digest has shifted. A Dockerfile that pins FROM python:3.12-slim@sha256:a1b2c3... is reproducible — but goes stale and accrues CVEs. The discipline is doing both: pin always, and update via automation that re-pins.

The four pinning levels

1. Pin by major version. FROM python:3. Worst — any 3.x change rebuilds you. Don't.

2. Pin by full version. FROM python:3.12.7-slim-bookworm. Better — same Python version. But the tag still moves: the slim-bookworm variant gets rebuilt with new system packages every few weeks.

3. Pin by digest. FROM python:3.12.7-slim-bookworm@sha256:f9d.... Reproducible. The same bytes today, tomorrow, next year. This is the production discipline.

4. Pin by signed digest. Same plus admission controller verifies the digest's signature at deploy time. Covered in Chapter X.

The automation layer

Pinned digests rot. CVEs are disclosed weekly; the base image you pinned six months ago is now full of unpatched issues. Three patterns to handle this:

Renovate / Dependabot for Docker. Configure to open PRs when a pinned base digest has a newer version. Renovate's docker manager natively pins-and-updates: opens PRs like "Update python:3.12 from @sha256:a1b... to @sha256:b2c...", with the changelog and a CVE diff in the description.

{
  "$schema": "https://docs.renovatebot.com/renovate-schema.json",
  "extends": ["config:recommended"],
  "docker": {
    "pinDigests": true            # Renovate pins by digest automatically
  },
  "packageRules": [
    {
      "matchManagers": ["dockerfile"],
      "matchUpdateTypes": ["digest"],
      "automerge": true,             # auto-merge digest updates after CI
      "schedule": ["before 9am every weekday"]
    }
  ]
}

Continuous rebuild against base. Chainguard's pattern: every Chainguard image is rebuilt nightly against the latest patched Wolfi packages. When a CVE is disclosed upstream, the affected package is patched and the image is rebuilt, typically within hours. You consume the image and inherit the freshness. If you build derivative images, configure your CI to rebuild them on a schedule (e.g., daily) so they pick up base updates without code changes.

SLAs by criticality. Critical CVEs (active exploit, severity 9+): patch in 24 hours. High: 7 days. Medium: 30 days. Low: best effort or "accept the risk" with documentation. Apply these to the image rebuild cadence, not just to application code.

Reproducibility — and when it matters

Two runs of the same Dockerfile should produce byte-identical images. Three sources of nondeterminism block this:

Timestamps. Files written during build get their mtime. Fix: SOURCE_DATE_EPOCH=$(git log -1 --pretty=%ct) and BuildKit's --output type=oci,rewrite-timestamp=true.

Network calls. apt-get update downloads whatever happens to be on the mirror today. Fix: lock files (requirements.txt with hashes, package-lock.json, go.sum) and pinned package versions. For apt, freeze the Debian snapshot URL.

Filesystem walk order. COPY . . processes files in the order the filesystem returns them. Fix: BuildKit normalizes this. If building outside BuildKit, sort explicitly.

Reproducibility matters most for: compliance (SLSA Level 3+, FIPS), supply-chain forensics ("rebuild this same image to verify no tampering"), and reproducible-research scenarios. For most teams, "deterministic enough" — pinned bases + lock files — is sufficient.

The pinning discipline in one sentence

Pin every base image by digest, automate the unpinning-and-re-pinning, and let your CI tell you when something broke. Manual base-image bumps are the #1 reason production images carry CVEs from 18 months ago.

BUILD Chapter IV

Layer layout · the cache topology.

The mental model Every Dockerfile line creates a layer. Every layer that changes invalidates every layer below it. Order the lines by how often they change.

The empirical test Touch an app source file, rebuild. If it takes more than 10s, your layer order is wrong.

A Dockerfile produces a sequence of layers, ordered from stable to volatile. Get the ordering right and a code change rebuilds in seconds; get it wrong and a code change invalidates the system-package install, eating minutes of CPU.

The five rules of layer ordering

1. Stable first, volatile last. Base → system packages → language deps → app code. App code changes every commit; deps weekly; system rarely. Reverse this and every commit invalidates dep install.

2. Split COPY for dependency files. Don't COPY . . before pip install. Copy requirements.txt alone, install, then copy the rest. The dep install layer must be invalidated only when its inputs change.

3. One RUN per logical action. Too many RUNs = layer count overhead. Too few = poor cache granularity. The right balance: install all system deps in one RUN, install all language deps in another, build the app in a third.

4. COPY only what's needed. A typical mistake: COPY . /app after WORKDIR /app drags in .git, node_modules, tests/. Use .dockerignore ruthlessly.

5. Never RUN apt-get update in isolation. Couple it with the install in the same RUN: RUN apt-get update && apt-get install -y --no-install-recommends ... && rm -rf /var/lib/apt/lists/*. Otherwise the update layer caches, the install gets stale package versions.

BuildKit's cache key, exactly

For every Dockerfile instruction, BuildKit computes a cache key from: (a) the base image digest (so changing FROM invalidates everything), (b) the previous step's cache key (transitive — any earlier change invalidates this), (c) the exact instruction text including whitespace, (d) for COPY, the hashes of the source files.

Implications: even reordering identical RUN commands invalidates the cache. Changing whitespace in a comment invalidates the cache. Adding a trailing newline to requirements.txt invalidates pip install. The cache is exact and unforgiving.

Cache mounts — work the layer cache can't do

Layer cache skips a whole step. Cache mounts let the step run but keep its scratch directory across builds. Critical for package managers:

# syntax=docker/dockerfile:1.7
# Python
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# Node
RUN --mount=type=cache,target=/root/.npm \
    npm ci --prefer-offline

# Go (module + build caches)
RUN --mount=type=cache,target=/root/.cache/go-build \
    --mount=type=cache,target=/go/pkg/mod \
    go build -o /app ./cmd/myapp

# Rust
RUN --mount=type=cache,target=/usr/local/cargo/registry \
    --mount=type=cache,target=/build/target \
    cargo build --release

# Apt — note the docker-clean removal
RUN --mount=type=cache,target=/var/cache/apt,sharing=locked \
    --mount=type=cache,target=/var/lib/apt,sharing=locked \
    rm -f /etc/apt/apt.conf.d/docker-clean && \
    apt-get update && apt-get install -y --no-install-recommends ...
The CI dividend

A team building 100 PRs per day on a 6-minute image saves 10 engineer-hours per day by getting to a 60-second image. The fix is usually a five-line reordering plus three cache-mount lines. Cross-reference Issue 06 Chapters II–III for the deeper performance discussion.

BUILD Chapter V

Multi-stage builds · build vs runtime.

The principle The build environment and the runtime environment are different. The runtime should ship neither the compiler nor the tests.

The shape Stage 1 (build): tooling-heavy. Stage 2 (runtime): minimal. Copy the artifact from 1 → 2.

Multi-stage builds are how you ship a distroless runtime while still using full toolchains during build. Stage 1 has the compiler, package manager, headers, debug tools. Stage 2 has only the compiled binary and what it actually needs to run. The two are connected by COPY --from=<stage>.

Stage 1 · buildBuilder image
Full toolchain: golang:1.22-alpine, gcr.io/distroless/python3-debian12:debug, cgr.dev/chainguard/wolfi-base with apk add gcc make. Heavy (hundreds of MB to several GB). Discarded at end of build.
Stage 2 · test (optional)Test image
If you want hermetic tests inside the container, run them in their own stage. Same base as build; ENTRYPOINT runs the test suite. Fails the build on failure.
Stage 3 · runtimeFinal image
Minimal: scratch, gcr.io/distroless/static, cgr.dev/chainguard/static. Contains only the binary and what it links to. No shell, no package manager. This is the image that ships.

The canonical Go multi-stage

# syntax=docker/dockerfile:1.7

# ===== Stage 1: build =====
FROM golang:1.22-alpine3.20@sha256:abc... AS build
WORKDIR /src
COPY go.mod go.sum .
RUN --mount=type=cache,target=/go/pkg/mod \
    go mod download
COPY . .
RUN --mount=type=cache,target=/root/.cache/go-build \
    --mount=type=cache,target=/go/pkg/mod \
    CGO_ENABLED=0 GOOS=linux \
    go build -trimpath -ldflags="-s -w" -o /out/app ./cmd/app

# ===== Stage 2: runtime =====
FROM gcr.io/distroless/static-debian12:nonroot@sha256:def...
COPY --from=build --chown=nonroot:nonroot /out/app /app
USER 65532:65532
EXPOSE 8080
ENTRYPOINT ["/app"]

Notes: CGO_ENABLED=0 produces a static binary that runs from scratch or distroless/static. -trimpath removes build paths from the binary (helps reproducibility). -ldflags="-s -w" strips debug info, shrinking the binary further.

The canonical Python multi-stage

# syntax=docker/dockerfile:1.7

# ===== Stage 1: build wheels =====
FROM cgr.dev/chainguard/python:3.12-dev@sha256:abc... AS build
WORKDIR /build
RUN python -m venv /venv
ENV PATH="/venv/bin:$PATH"
COPY requirements.txt .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install --no-deps -r requirements.txt
COPY . .
RUN pip install --no-deps .

# ===== Stage 2: runtime =====
FROM cgr.dev/chainguard/python:3.12@sha256:def...
COPY --from=build /venv /venv
ENV PATH="/venv/bin:$PATH"
USER 65532
ENTRYPOINT ["/venv/bin/python", "-m", "myapp"]

Note the -dev variant for the build stage (has pip, build tools, headers) and the non--dev variant for runtime (minimal). The /venv directory captures everything pip installed; copying it across stages is cleaner than re-installing.

Variants worth knowing

Single-stage with no compiler. If your CI builds the binary outside Docker (typical for Go monorepos), the Dockerfile is just FROM scratch; COPY ./bin/app /app; ENTRYPOINT ["/app"]. Fastest builds; tightest images.

Distroless base image with --target. BuildKit supports docker build --target build . to stop at a named stage. Useful for "give me an image with all the build tools for interactive debugging."

Build stage shared across services. If many services build similarly, define the build stage in a base image (push to your registry), and each service's Dockerfile starts with that. Reduces duplication.

Parallel stages. BuildKit runs independent stages concurrently. If stage A doesn't depend on stage B, both run in parallel. Restructure to maximize this.

The mental shift

Before multi-stage: "the Dockerfile is one long script." After: "the Dockerfile is a small build system." Stages map to pipeline phases (resolve deps → compile → test → package). Get used to thinking of FROM as defining a stage, not just a base.

BUILD Chapter VI

BuildKit · beyond the Dockerfile.

The modern builder BuildKit replaced the legacy Docker build engine. Parallel stages, cache mounts, secret mounts, multi-arch via emulation or native nodes, SBOM/provenance attestations, and frontends beyond Dockerfile.

The default in 2026 Always-on. docker buildx is the CLI; the daemon is BuildKit.

BuildKit is the engine; docker buildx is the CLI. The legacy Docker build (linear, no parallel stages, no cache mounts, no attestations) is deprecated. Anything substantial in 2026 uses BuildKit features that are simply unavailable in the legacy builder.

What BuildKit adds over the legacy builder

Parallel stage execution. If build-ui and build-api are independent stages, BuildKit runs them concurrently. Total time = max(stages) + final, not sum.

Cache mounts. --mount=type=cache,target=... — covered in Chapter IV.

Secret mounts. --mount=type=secret,id=npm,target=/root/.npmrc. Secret is available during the RUN; never enters a layer. Replaces the old anti-pattern of ENV NPM_TOKEN=... which leaks into image history.

SSH agent forwarding. --mount=type=ssh. For private git dependencies during build. The SSH socket forwards into the build; the key itself never enters the build context.

Multi-platform builds. docker buildx build --platform linux/amd64,linux/arm64 produces an image index covering both arches in one command. QEMU emulation or native builder nodes.

Attestations. --sbom=true --provenance=mode=max attaches SBOM and SLSA provenance to the image as referrers. Chapters IX–XI.

Pluggable frontends. The "Dockerfile" is one frontend; others exist (Buildpacks, Earthly, custom). Specified by # syntax=... directive.

Multi-platform — emulation vs native

Cross-arch builds have two implementations:

QEMU emulation. One amd64 builder emulates arm64 (or vice versa). Easy setup: docker run --privileged --rm tonistiigi/binfmt --install all. Slow: 10–30× slower than native for compute-heavy builds. Fine for static Go binaries; painful for large C++ compiles.

Cross-compilation. The amd64 builder compiles arm64 binaries directly using a cross-compiler. Fast (no emulation). Works for Go, Rust, and statically-linked C with appropriate flags. Doesn't work for languages or dependencies that require running tooling on the target architecture.

Native builder nodes. Register both real amd64 and real arm64 machines as buildx nodes; buildx routes platform-specific work to the native node. Fastest. Requires real arm64 hardware (AWS Graviton, etc.).

# Set up the builder with two nodes
docker buildx create --name multiarch --use \
    --driver docker-container # local amd64
docker buildx create --append --name multiarch \
    --driver remote --node arm64-node tcp://arm64-host:1234

# Build for both — each node handles its native arch
docker buildx build \
    --platform linux/amd64,linux/arm64 \
    --tag myorg/myapp:v1.2.3 --push .

Frontends — Dockerfile is one option

The # syntax=docker/dockerfile:1.7 directive selects a Dockerfile frontend version. Other frontends exist:

Buildpacks (Cloud Native Buildpacks). Infer the language and dependencies from source; produce an image without a Dockerfile. pack build myapp. Used by Heroku, Google Cloud Run.

Earthly. Earthfile syntax — Dockerfile-like but with explicit dependency graphs and parallelism. Good for monorepos.

apko + melange (Chainguard). Declarative YAML; no Dockerfile. apko build alpine.yaml .... The tools that build Chainguard images. Best for "build me a distroless image that contains exactly these apks." apko lets you bundle a collection of APKs into an OCI image using a declarative YAML manifest. melange lets you build apks using declarative YAML pipelines.

# app.yaml
contents:
  repositories:
    - https://packages.wolfi.dev/os
  packages:
    - ca-certificates-bundle
    - tzdata
    - my-app                            # your app packaged as a wolfi apk
entrypoint:
  command: /usr/bin/my-app
accounts:
  users:
    - username: nonroot
      uid: 65532
  run-as: 65532
archs: [amd64, arm64]

# Build it
apko build app.yaml myorg/myapp:v1 myorg-myapp-v1.tar
docker load < myorg-myapp-v1.tar
The 2026 default BuildKit invocation

docker buildx build --platform linux/amd64,linux/arm64 --provenance=mode=max --sbom=true --cache-to type=registry,ref=...,mode=max --cache-from type=registry,ref=... --push -t myorg/myapp:$TAG . — multi-arch, attestations, registry cache, push. Six flags that produce a production-grade image. Adopt the pattern in CI once; it pays off for every service forever.

BUILD Chapter VII

Dependency management · lockfiles & hashes.

The premise "I declared X" is not the same as "I installed X version Y from source Z verified by hash H." Lockfiles bridge the gap.

The new threat Supply-chain attacks via dependency substitution. Hashes are now the minimum bar.

Application dependencies are now the most likely vector for compromise — not the base image, not your own code. Lockfiles and hash verification turned what was "best practice" five years ago into table stakes after a series of high-profile incidents.

The dependency hierarchy

Three levels of pinning, each tighter than the last:

Version range. requests = "^2.28" (allows 2.28.x, possibly 2.29). The default in most ecosystems. Convenient; non-reproducible; vulnerable to a new patch shipping a backdoor.

Exact version. requests==2.28.2. Reproducible across runs. But: a malicious 2.28.2 could be uploaded to a private mirror; if the public package is yanked and re-uploaded with different content, you're vulnerable.

Version + hash. requests==2.28.2 --hash=sha256:62.... Fully reproducible AND tamper-evident. The package manager verifies the downloaded bytes match the hash before installing.

Lockfile generators by language

# Python — pip-tools (pip-compile) generates requirements.txt with hashes
pip-compile --generate-hashes --output-file requirements.txt requirements.in
# Or use Poetry / PDM / uv — they all support hashes natively

# Node — npm ci uses package-lock.json which includes integrity hashes
npm ci   # installs exact versions from lock, verifies integrity hash

# Go — go.sum already records hashes for all transitive deps
go mod tidy
# go build automatically verifies go.sum on every build

# Rust — Cargo.lock records exact versions + checksums
cargo build   # verifies checksums from Cargo.lock

# Java — Gradle/Maven via dependency-verification
./gradlew --write-verification-metadata sha256
# Produces gradle/verification-metadata.xml

The hashed install in the Dockerfile

# Python — pip refuses to install anything not matching the hash
COPY requirements.txt .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install --require-hashes -r requirements.txt

# Node — npm ci fails if package-lock.json doesn't match installed
COPY package.json package-lock.json .
RUN --mount=type=cache,target=/root/.npm \
    npm ci --omit=dev --ignore-scripts

# Go — go build verifies go.sum automatically
COPY go.mod go.sum .
RUN --mount=type=cache,target=/go/pkg/mod \
    go mod download -x

# --ignore-scripts is important — many npm supply-chain attacks
# run code in postinstall hooks. Disable unless you actively need them.

Private mirrors and proxies

For high-stakes environments: route all dependency fetches through a private mirror that you control. Three benefits:

One — cache. Public package registries (npm, PyPI, Maven Central) get slow, rate-limit, or go down. A local mirror is always available.

Two — vetting. The mirror can block packages your security team hasn't reviewed, or quarantine packages until vulnerability scans complete.

Three — audit. You can prove which packages your builds consumed and when. Critical for compliance and incident response.

Common implementations: Sonatype Nexus, JFrog Artifactory, Cloudsmith, GitHub Packages proxy mode, AWS CodeArtifact, Verdaccio (npm specific). Configure your build's package manager to use the mirror as upstream; never let prod builds reach the public internet.

The "axios" lesson

In March 2026, axios was compromised via a supply-chain attack. Teams pinning by version range pulled the malicious release; teams pinning by hash and validating against a mirror did not. The pattern repeats: a dependency goes through a maintainer transition, the new maintainer ships a malicious version, the world's CI pipelines auto-install it.

Three controls, in order of effort:

(1) Lockfile with hashes — every build verifies bytes. Cheap.

(2) Private mirror with vetting — the malicious package never reaches your builds. Moderate effort.

(3) Reproducible-build verification — independent rebuild matches your CI output. Expensive; only for the highest-stakes environments.

The hash-or-die principle

Every production image's build should fail closed when dependencies don't match their pinned hashes. pip install --require-hashes. npm ci not npm install. go.sum verified automatically. Cargo.lock in version control. Five lines of CI policy that prevent a category of attack that has cost the industry billions.

Vulnerability management · scan, triage, patch.

The CVE inversion A 5-year-old Alpine image has dozens of CVEs. A daily-rebuilt Wolfi image has zero. The base image you choose determines the CVE floor.

The discipline Scan in CI, alert in registry, patch by rebuild.

Container vulnerability management is the discipline of finding the CVEs in your images and acting on them. The ignore-unfixed flag is key. CVEs that have no fix yet aren't actionable; failing the build on them just causes alert fatigue and people disabling scans. Fail on fixable CVEs; record the rest in a tracking system.

The triage flow

For each CVE in your image, ask in order:

1. Is the vulnerable package even reachable? The CVE is in libxml2 but your service doesn't parse XML. Mark "not applicable" (formally, a VEX statement — see Chapter IX). Doesn't reduce risk on paper but separates real exposure from theatre.

2. Is there a fixed version available? If yes: bump and rebuild. Done in minutes for most language deps.

3. Is the base image at fault? Rebuild against a fresher base; many CVEs evaporate. If you're on Wolfi or Chainguard, this is automatic via nightly rebuild.

4. Can the application be reconfigured to avoid the vulnerability? CVE in TLS 1.0 handling; disable TLS 1.0. CVE in a legacy parser; disable the parser.

5. If none of the above: document and accept. Sometimes the right answer is "we accept this risk for X reason, here's the compensating control." Write it down. Review quarterly.

Continuous registry scanning

CI scans catch CVEs at push time. But a CVE disclosed tomorrow affects images already in your registry. Registries with built-in scanners (Harbor, Quay, ECR, ACR, Artifactory) periodically rescan stored images and alert when a newly-disclosed CVE matches an image you've already pushed.

The right alert rule: "image is being actively pulled AND has a new HIGH/CRITICAL CVE." Don't alert on every CVE in every dormant image — that's a quarterly cleanup task, not an interrupt.

The pragmatic policy

Fail CI on new fixable HIGH/CRITICAL CVEs. Alert on registry CVEs in actively-pulled images. Auto-rebuild nightly to pick up base updates. Triage unfixed CVEs via VEX so they don't fill the dashboard. The hardest part isn't the tooling — it's not failing builds on unfixed CVEs that no human can act on.

SUPPLY Chapter IX

SBOM & attestations · the provenance layer.

SBOM The parts list. Every package and every version that went into this image. Like the ingredient label on food.

Attestation Signed metadata about the image: who built it, how, when, from what source.

SBOMs answer "what's in the image"; attestations answer "where did it come from and how was it built." Together they make container images analyzable as software artifacts, not just opaque tarballs. Both became regulated assets after the 2021 US Executive Order 14028 and the corresponding European NIS2 directive.

SBOM formats

SPDX: ISO standard, sponsored by the Linux Foundation. JSON, YAML, RDF, or tag-value formats. Government and compliance default.

CycloneDX: OWASP project. JSON or XML. Vulnerability-management focused; ships with built-in VEX support. Vendor and tooling default.

Most tools generate either or both. The differences are mostly schema; the information overlaps heavily.

Generating SBOMs

# 1. BuildKit at build time — the easy path
docker buildx build \
    --sbom=true \                       # attaches SPDX SBOM as attestation
    --provenance=mode=max \             # attaches SLSA provenance
    -t myorg/myapp:v1 --push .

# 2. Syft — scan a built image, get SBOM
syft myorg/myapp:v1 -o spdx-json > sbom.spdx.json
syft myorg/myapp:v1 -o cyclonedx-json > sbom.cdx.json
syft myorg/myapp:v1 -o table              # human-readable

# 3. apko/melange — SBOM is a build artifact, not a scan
# apko emits SBOM natively as part of every build,
# cryptographically tied to the image bytes

The third option matters because of what the SBOM means. A scanned SBOM (option 2) is one tool's best guess about what's in the image. A build-time SBOM (option 1 or 3) is the build system's authoritative record of what it put in. Build-time SBOMs are more accurate; scan-time SBOMs are universal (work on any image, including ones you didn't build).

Attestations and the referrers API

An attestation is signed metadata attached to an image. Defined by the in-toto specification: a predicate (the claim) plus a subject (what it's about) plus signature.

Critical attestations for container images:

SBOM attestation: predicate type https://spdx.dev/Document or https://cyclonedx.org/bom. The SBOM, signed by the builder.

SLSA provenance: predicate type https://slsa.dev/provenance/v1. The build environment, source repository, commit, build command. Chapter XI.

VEX (Vulnerability Exploitability eXchange): predicate type https://openvex.dev/ns. Statements about which CVEs in the SBOM are actually exploitable in this product.

Storage: the OCI 1.1 referrers API stores these as separate OCI artifacts that reference the image by digest. Stores signatures and attestations using OCI 1.1 referrers mode to avoid tag pollution and work cleanly with immutable tags. No more dangling :sha256-...sig tags polluting the registry.

VEX — the missing piece

A scanned image might show 50 CVEs. Of those, maybe 5 are actually exploitable in your use of the affected libraries. VEX is how you formally declare the other 45 "not affected" with reasoning, and have that statement signed and verified the same way the SBOM is.

{
  "@context": "https://openvex.dev/ns/v0.2.0",
  "@id": "https://example.com/vex/2026-001",
  "author": "Security Team <security@example.com>",
  "timestamp": "2026-05-12T10:00:00Z",
  "statements": [{
    "vulnerability": { "name": "CVE-2025-12345" },
    "products": [{
      "@id": "pkg:oci/myapp?repository_url=myorg.io",
      "hashes": { "sha-256": "abc123..." }
    }],
    "status": "not_affected",
    "justification": "vulnerable_code_not_in_execute_path",
    "impact_statement": "The function in libxml2 is only called by feature X, which is disabled at build."
  }]
}

Attached to the image as a signed attestation, this is how you stop drowning in CVE noise. The scanner reports 50; the VEX says 45 don't apply; the dashboard shows 5 actionable.

SUPPLY Chapter X

Signing · cosign, sigstore, keyless.

The shift Long-lived signing keys → short-lived OIDC-issued certificates. No key management. Identity-based verification.

The verification The signature proves who built it, not just that the bytes haven't changed.

Image signing answers two questions: did these exact bytes come out of a trusted build, and who or what was the trusted build. The 2026 standard is Sigstore (cosign + Fulcio + Rekor) — keyless signing using short-lived OIDC certificates, with the signing event recorded in a public transparency log.

How keyless signing works

The classic approach: generate an RSA or ECDSA keypair, store the private key in HSM/Vault, sign images with it. Problems: key management is hard, the private key is a permanent compromise risk, rotation is painful.

The keyless approach (Sigstore Fulcio):

  1. CI gets an OIDC token from GitHub Actions / GitLab CI / Buildkite identifying the workflow.
  2. The OIDC token is sent to Fulcio (Sigstore's CA).
  3. Fulcio issues a short-lived (10 minute) X.509 certificate where the subject is the CI workflow identity (e.g., https://github.com/myorg/myrepo/.github/workflows/build.yml@refs/heads/main).
  4. cosign signs the image's manifest digest with the certificate.
  5. The signing event is recorded in Rekor, the public transparency log.
  6. The certificate expires; the private key never persisted to disk; the verifiable record is the Rekor entry plus the certificate.

Verification reverses this: cosign retrieves the certificate, verifies the OIDC identity matches what you expect, checks Rekor that the signing happened. cosign assumes we're using the Sigstore public good instance.

Sign and verify, in practice

# In CI (GitHub Actions) — sign by digest
- name: Sign the image
  env:
    DIGEST: ${{ steps.build.outputs.digest }}
  run: |
    cosign sign --yes \
      --identity-token=${{ steps.idtoken.outputs.token }} \
      ghcr.io/myorg/myapp@${DIGEST}

# Anywhere — verify before pull/run
$ cosign verify \
    --certificate-identity-regexp="^https://github.com/myorg/.+/\.github/workflows/build\.ya?ml@.+$" \
    --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
    ghcr.io/myorg/myapp@sha256:abc...

Verification for ghcr.io/myorg/myapp@sha256:abc... --
The cosign claims were validated
Existence of the claims in the transparency log was verified offline
The code-signing certificate was verified using trusted certificate authority

The verification command names the expected identity. Anyone trying to substitute a different image fails the identity check; anyone trying to substitute a signature from a different workflow fails too.

Admission control — block unsigned images at the cluster

Signing in CI is half the story. The other half is refusing to run unsigned images in production. Two common implementations:

Kyverno: K8s policy engine. verifyImages rule checks signatures at admission. If verification fails, the pod isn't created.

Sigstore Policy Controller: purpose-built for cosign verification. Lightweight; one job to do.

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-images
spec:
  validationFailureAction: Enforce            # block, don't just warn
  rules:
    - name: check-myorg-signed
      match:
        any:
          - resources:
              kinds: ["Pod"]
      verifyImages:
        - imageReferences: ["ghcr.io/myorg/*"]
          attestors:
            - entries:
                - keyless:
                    issuer: https://token.actions.githubusercontent.com
                    subject: "https://github.com/myorg/*"
                    rekor:
                      url: https://rekor.sigstore.dev

What signing does NOT prove

Signing proves "this CI workflow produced this image." It does not prove:

That the workflow itself is uncompromised. If the CI configuration is malicious, signing legitimizes whatever it builds.

That the source code is what you think. The workflow could be checking out a different branch, applying patches, or building from a fork.

That the image is free of vulnerabilities. Signing is identity, not safety.

That the SBOM is accurate. The same workflow could lie in its SBOM and sign the lie.

The mitigations: SLSA provenance (Chapter XI) ties the signature back to the source commit; VEX (Chapter IX) qualifies the SBOM; vulnerability scanning (Chapter VIII) provides the orthogonal safety check.

The keyless adoption order

Start by signing all production images in CI with cosign keyless. Stay there for a few months — get used to the signing identity, verify it works through promotion. Then add Kyverno verification at admission. Then add SLSA provenance generation (the next chapter). Each step takes a sprint; combined they raise your supply-chain posture from "we hope the registry isn't compromised" to "we cryptographically verify every byte that enters our cluster."

SLSA · build integrity levels.

What SLSA is Supply-chain Levels for Software Artifacts. A graduated set of requirements for what your build process can prove about itself.

The bar Level 1 is "we generate provenance." Level 3 is "the provenance is non-forgeable even if the build is compromised."

SLSA is the graduated framework for build-pipeline integrity. Each level adds requirements that make it harder for an attacker who compromises some part of the pipeline to forge the provenance. By Level 3, even a fully compromised build job cannot fake the attestation.

The four levels

level 0
No provenance
No claims about how the artifact was built. Every Docker Hub image with no attestations starts here.
starting point
level 1
Provenance exists
Build produces machine-readable provenance: source repo, commit, builder, inputs. Need not be tamper-evident.
documentation
level 2
Authenticated provenance
Provenance is signed. Hosted CI service (GitHub Actions, GitLab, Buildkite). Distrusts the user, trusts the platform.
most teams here
level 3
Non-forgeable provenance
Build runs on isolated infrastructure; provenance generated outside the build job. Compromised build can't fake its own attestation.
production target

What SLSA provenance contains

Anyone with the provenance can answer: which source commit, which workflow file, which CI run, which workflow version. Combined with the cosign signature, they can verify it's authentic.

The Level 3 architecture

The naive approach: build the image AND generate the provenance in the same CI job. Problem: if the build is compromised (malicious code in a build script), it can also forge a provenance that says everything's fine.

The SLSA Level 3 pattern: Job 1 (build & push) uses BuildKit to compile and push the image, with native provenance disabled. Job 2 (SLSA L3 provenance) runs in a completely isolated job using slsa-framework/slsa-github-generator. It receives only the final image digest, independently verifies the source repository, commit, and workflow via GitHub APIs, and generates a non-forgeable SLSA Level 3 attestation. Even if Job 1 is fully compromised, this provenance cannot be faked.

# .github/workflows/release.yml
jobs:
  build:                                    # Job 1: build + push image
    runs-on: ubuntu-latest
    outputs:
      digest: ${{ steps.push.outputs.digest }}
    steps:
      - uses: actions/checkout@v4
      - id: push
        run: |
          DIGEST=$(docker buildx build --push ... --metadata-file md.json \
                   | jq -r '.containerimage.digest')
          echo "digest=$DIGEST" >> $GITHUB_OUTPUT

  provenance:                               # Job 2: separate, non-forgeable provenance
    needs: [build]
    permissions:
      actions: read
      id-token: write
      packages: write
    uses: slsa-framework/slsa-github-generator/.github/workflows/generator_container_slsa3.yml@v2.0.0
    with:
      image: ghcr.io/myorg/myapp
      digest: ${{ needs.build.outputs.digest }}

Verifying SLSA at admission

The same Kyverno/Sigstore Policy Controller setup from Chapter X can verify SLSA provenance. The policy:

verifyImages:
  - imageReferences: ["ghcr.io/myorg/*"]
    attestations:
      - predicateType: https://slsa.dev/provenance/v1
        attestors:
          - entries:
              - keyless:
                  subject: "^https://github.com/slsa-framework/slsa-github-generator/.+$"
                  issuer: https://token.actions.githubusercontent.com
        conditions:
          - all:
              - key: "{{ buildDefinition.externalParameters.workflow.repository }}"
                operator: Equals
                value: "https://github.com/myorg/myrepo"
              - key: "{{ buildDefinition.externalParameters.workflow.ref }}"
                operator: Equals
                value: "refs/heads/main"

The admission controller refuses any image whose provenance says it was built from a fork, a feature branch, or a different repo. Combined with branch-protection rules (only certain people can merge to main), this is a strong supply-chain perimeter.

The SLSA progression

Most production teams should be at Level 2 today (BuildKit's --provenance=mode=max running in hosted CI). The jump to Level 3 — separating the provenance job from the build job — costs a sprint of work and a small CI runtime increase, and produces meaningfully stronger guarantees. For high-stakes services (financial, healthcare, critical infrastructure), L3 is the bar; for everything else, L2 + cosign signing is a strong baseline.

DISTRIBUTION Chapter XII

Registry topology · hub, mirror, cache.

One registry doesn't scale Past a few clusters in a few regions, a single registry becomes the bottleneck and the SPoF. Production topology is hierarchical.

Three roles Hub (source of truth), mirrors (replicas in regions), pull-through caches (on-demand cache of upstream).

A registry topology serves three purposes: minimize image-pull latency at the edge, isolate clusters from registry outages, and reduce egress cost from cloud providers. Below ~200 nodes a single registry is fine. Above that, you need a topology — and there are three established patterns.

The three-tier registry topology

Tier 1

Internal hub
  • Source of truth for your org
  • CI pushes here
  • Signing & SBOM happen here
  • Scanning & admission policies
  • Backed by object storage (S3)
  • Single region; replicated for HA

Tier 2

Regional mirrors
  • Active replication from hub
  • One per region/cloud
  • Same content as hub
  • Local-region latency for pulls
  • Survive hub outage
  • Harbor, Zot, ECR replication

Tier 3

Cluster pull-through caches
  • On-demand cache per cluster
  • Pulls from regional mirror
  • Survives mirror outage briefly
  • ~10× faster pod startup
  • Sub-second image resolution
  • Zot, Spegel, ociregistry

The hub — choosing the platform

AWS ECR / GCP AR / Azure ACR
Fully managed; auto-replication across regions; integrates with cloud IAM. Best when staying in one cloud. Per-GB storage + egress costs.

Replication from hub to regional mirrors

Harbor replicates with a pull-or-push policy you define: "everything matching prod/*, replicate to eu-west-1.myregistry.com on push, retain forever." Configure your OCI registry such as Harbor or Zot to replicate repositories across zones or regions, keeping images, charts, configurations and attachments in sync. When CI pushes to the hub, the replicate runs; within seconds or minutes (depending on size), the mirror has the image.

Zot supports periodic-polling and on-demand mirroring: As with git, wherein every clone is a full repository, you can configure your local zot instance to be a fully mirrored OCI registry. For this mode, configure zot for synchronization by periodic polling, not on-demand. Zot copies and caches a full copy of every image on the upstream registry.

The key invariant: the manifest digest is identical at hub and mirror. Same signatures, same attestations, same SBOM. Mirrors are read-only replicas; you never push to a mirror, you push to the hub.

The pull-through cache pattern

Even regional mirrors are too far for a node pulling a 5 GB image. The third tier — cluster-local pull-through cache — sits inside your cluster and caches what's been pulled. For a pull-through cache mirrored registry, configure zot for on-demand synchronization. When an image is first requested from the local zot registry, the image is downloaded from the upstream registry and cached in local storage. Subsequent requests for the same image are served from zot's cache.

Three implementations worth knowing:

Zot in pull-through mode: simplest. Configure each cluster's containerd to point at the cache; the cache pulls from upstream on first miss. Cached images stick around; LRU eviction.

Spegel (CNCF Sandbox): peer-to-peer image distribution. Every node serves the layers it has to neighbors. No central cache — the cluster itself is the cache. Massively reduces upstream bandwidth.

ociregistry: pull-only, pull-through, caching OCI distribution server providing caching pull-through capability to multiple upstream registries (internal, air-gapped, or public).

Containerd config for pull-through

# /etc/containerd/certs.d/_default/hosts.toml
server = "https://registry.example.com"

[host."http://cache.cluster-local:5000"]
  capabilities = ["pull", "resolve"]
  override_path = true
  skip_verify = true             # cache is internal; OK to skip TLS

Every docker pull registry.example.com/foo on a node now goes through the in-cluster cache. The cache pulls from the real registry once, serves to every subsequent puller within milliseconds. For a 1000-node cluster, the upstream registry sees one pull instead of a thousand.

Air-gapped patterns

For air-gapped environments (no internet from the cluster), the pull-through cache becomes a full mirror:

One — periodic sync. A pump moves images from the connected hub to the air-gapped mirror via approved transport (physical media, one-way diodes, scheduled VPN windows).

Two — tarball transport. crane export or skopeo copy --dest-tls-verify=false docker://... oci-archive:bundle.tar packages images for offline transport.

Three — policy at the gate. Air-gapped environments often need strict provenance verification — only signed images from approved sources cross the air gap.

The topology heuristic

Below 200 nodes / one region: single registry is fine. 200–2000 nodes / multiple regions: add regional mirrors. 2000+ nodes / latency-sensitive: add cluster-local pull-through caches or Spegel. Cross-reference Issue 06 Chapter V — the same scale breakpoints govern when to deploy lazy pulling.

            type=sha,prefix=sha-

      - id: push
        uses: docker/build-push-action@v5
        with:
          context: .
          platforms: linux/amd64,linux/arm64
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          provenance: mode=max
          sbom: true
          cache-from: type=registry,ref=ghcr.io/${{ github.repository }}:buildcache
          cache-to: type=registry,ref=ghcr.io/${{ github.repository }}:buildcache,mode=max

      - name: Scan image
        uses: aquasecurity/trivy-action@0.18.0
        with:
          image-ref: ghcr.io/${{ github.repository }}@${{ steps.push.outputs.digest }}
          severity: CRITICAL,HIGH
          exit-code: '1'
          ignore-unfixed: true

      - uses: sigstore/cosign-installer@v3
      - name: Sign image (keyless)
        run: |
          cosign sign --yes \
            ghcr.io/${{ github.repository }}@${{ steps.push.outputs.digest }}

    needs: [build]
    permissions:
      actions: read
      id-token: write

Build once, promote by digest

The single most-important promotion principle: never rebuild for staging or production. The image bytes that passed CI are the bytes that run in prod. Promotion is just a tag move (or a Git commit, in GitOps).

# BAD: rebuild for staging
docker build -t myorg/myapp:staging-v1.2.3 .
docker build -t myorg/myapp:prod-v1.2.3 .       # different bytes!

# GOOD: promote by digest
DIGEST=$(crane digest ghcr.io/myorg/myapp:v1.2.3)
crane tag ghcr.io/myorg/myapp@$DIGEST staging-v1.2.3
crane tag ghcr.io/myorg/myapp@$DIGEST prod-v1.2.3
# All three tags point at the same bytes

GitOps promotion

The 2026 norm: ArgoCD or Flux watches a Git repo; the repo contains Kubernetes manifests referencing images by digest. Promotion is a Git commit changing the digest reference.

# manifests/prod/myapp/values.yaml — before
image:
  repository: ghcr.io/myorg/myapp
  digest: sha256:abc123...

# PR: bump prod to the staging digest, after tests pass
image:
  repository: ghcr.io/myorg/myapp
  digest: sha256:def456...

The PR is reviewed; admission control verifies the signature at deploy; ArgoCD reconciles. Every change to production has a Git commit, a code review, a signature verification.

The eight stages, one principle

Every stage produces an artifact the next consumes. Lint produces a clean Dockerfile. Build produces an image. Test produces a passing result tied to a digest. Scan produces a passing CVE check tied to a digest. SBOM produces a parts list. Sign produces a signature. Provenance produces an attestation. Promote moves the digest forward. Every artifact references the same digest. Same digest, end to end.
