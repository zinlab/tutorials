MAP Chapter I

The performance surface · five questions.

The frame All container performance work reduces to five questions. Knowing which one applies tells you what to measure and what to change.

The discipline Measure before tuning. Tune one variable at a time. Stop at the SLO.

Container performance has five distinct surfaces, each with its own metrics, levers, and failure modes. Knowing which surface a problem lives on is more than half the work — most tuning failures come from optimizing the wrong layer.

The five questions

1. Is the image too large? Affects pull time, registry cost, attack surface, and how much filesystem you have to mount. Levers: multi-stage builds, distroless bases, layer ordering. Metric: image size after compression.

2. Is the build too slow? Affects developer iteration speed and CI cost. Levers: BuildKit cache mounts, registry-based cache, parallel stages. Metric: build wall-clock time, cache hit rate.

3. Is startup too slow? Affects autoscaler reaction time, deploy time, scale-from-zero latency, fault recovery. Levers: lazy pulling, image distribution, lazy initialization in the app. Metric: time from kubectl apply to readyz=200.

4. Is the runtime starved? Affects throughput and tail latency once steady-state. Levers: cgroup limits, page cache, CPU pinning, NUMA awareness, GPU sharing. Metric: throughput, p99 latency, throttling rate.

5. Are we measuring the wrong thing? Affects every decision above. Levers: metrics taxonomy, traces, profiles, eBPF. Metric: signal-to-noise ratio in dashboards.

The order of operations

The five questions aren't independent — they layer on each other:

  1. Measure first. Without instrumentation (Chapters XI–XII), every tuning effort is guesswork.
  2. Get image size and layer hygiene right. If the image is wrong, nothing downstream helps.
  3. Speed up build. Required for fast iteration on every other layer.
  4. Speed up startup. Determines how quickly the system reacts to load and failures.
  5. Tune steady-state runtime. This is where most tutorials start, and where most teams over-invest before the upstream layers are fixed.
The performance budget

The real goal isn't "make it faster" — it's "meet the SLO with minimum complexity." Most production services have far more headroom than they need; spending two engineer-weeks on cgroups v2 micro-tuning to reclaim 3% throughput is rarely the right trade. The right trade is fixing the thing that's actually limiting your reliability or your bill. The remaining 14 chapters are arranged so you can stop at the first chapter that solves your problem.

IMAGE Chapter II

Layer layout · the cache cliff.

The premise Every Dockerfile line creates a layer. Each layer that changes invalidates every layer below it. The "cache cliff" is when your build flips from 5 seconds (all cached) to 5 minutes (cache miss at the top).

The discipline Order by change frequency. Test the cliff by rebuilding without changes — it should take under 10s.

The cache cliff is the single most under-managed property of a Dockerfile. A well-ordered Dockerfile rebuilds in 5 seconds when only application code changed; a poorly-ordered one rebuilds in 5 minutes. The difference is one line of ordering, not any clever trick.

The five rules of layer ordering

1. Stable first, volatile last. Order Dockerfile lines so that things that change rarely come first, and things that change often come last. Concretely: base image → system packages → language deps → app code. App code changes every commit; system packages change monthly; the base changes quarterly.

2. Split COPY for dependency files first. Don't COPY . . before installing deps. Copy the lock file alone, install, then copy the rest. A code change shouldn't invalidate dep install.

# BAD — every code change re-runs pip install
COPY . .
RUN pip install -r requirements.txt

# GOOD — pip install only re-runs when requirements.txt changes
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .

3. Combine related RUN commands; separate unrelated ones. Each RUN is a layer. Too many layers = registry overhead; too few = poor cache granularity. The rule: one RUN per "logical action" (install deps, build app, configure).

4. Multi-stage to discard build context. Don't ship the compiler. Don't ship the test files. Don't ship .git. The build stage holds tools; the runtime stage copies only the artifacts.

5. Use .dockerignore ruthlessly. Every byte in the build context is hashed and sent to the builder. .git, node_modules, target/, build outputs — all should be excluded.

Cache invalidation, explained

BuildKit's cache key for each step is composed of: the base image digest (changes when FROM changes), the previous step's cache key (transitive — any earlier change invalidates this and everything below), the exact text of the Dockerfile instruction, and for COPY steps, the hashes of the source files. Three implications: changing the base image invalidates everything; changing any instruction text invalidates everything below it; changing a file in a COPY source invalidates the layer and below.

Verifying the cliff

The fastest way to test your Dockerfile's cache behavior:

# 1. Build cold (clears cache)
docker buildx prune -af
time docker buildx build --load -t myapp:test .
# Cold time: 90s (example)

# 2. Touch a source code file (no actual change)
touch src/main.py

# 3. Rebuild — should be fast if layer order is right
time docker buildx build --load -t myapp:test .
# Should be: < 10s on a well-ordered Dockerfile

# 4. Change requirements.txt (dep change)
echo "# bump" >> requirements.txt
time docker buildx build --load -t myapp:test .
# Should re-run pip but skip base; maybe 30s

If step 3 takes more than 10 seconds, your layer order is wrong. If step 4 fully rebuilds, your dep step isn't isolated.

The unspoken rule

A Dockerfile's layer order is not "polish" — it's the single biggest determinant of how fast your CI runs. A team building 100 PRs a day on a 60-second image (well-ordered) vs. a 6-minute image (poorly ordered) recovers 10 engineer-hours per day. The fix is a five-line reordering.

IMAGE Chapter III

Build caching · BuildKit deep cuts.

The premise Layer cache is one of three caching surfaces in modern BuildKit. The others — RUN cache mounts and remote layer cache — multiply the speedup.

The combination Layer cache (skip the step) + cache mount (skip work inside the step) + remote cache (share across machines).

BuildKit's caching has three independent surfaces: layer cache (skip an entire step if its inputs match), cache mounts (persist a directory across builds even when the step runs), and remote cache (share both kinds across machines and CI runs). Each is independent; combining all three is what separates a 4-minute CI build from a 20-second one.

Surface 1 — Cache mounts (the fastest win)

Package managers (pip, apt, npm, go, cargo) have their own caches that they normally write into the build's filesystem — and therefore into a layer. With cache mounts, the cache lives outside the image but persists across builds.

# syntax=docker/dockerfile:1.7

# Python — pip cache
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install -r requirements.txt

# APT (Debian/Ubuntu) — needs to disable docker-clean
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt \
    rm -f /etc/apt/apt.conf.d/docker-clean && \
    apt-get update && apt-get install -y --no-install-recommends curl

# npm — package cache
RUN --mount=type=cache,target=/root/.npm \
    npm ci --prefer-offline

# Go — module & build caches
RUN --mount=type=cache,target=/root/.cache/go-build \
    --mount=type=cache,target=/go/pkg/mod \
    go build -o /app ./cmd/myapp

# Cargo (Rust)
RUN --mount=type=cache,target=/usr/local/cargo/registry \
    --mount=type=cache,target=/build/target \
    cargo build --release

Cache mounts are surprisingly under-used. A Rust project building from scratch can take 8 minutes; with a cargo cache mount, an incremental rebuild after a one-line change is 12 seconds. The cache lives in the BuildKit daemon's storage, not in any image layer.

Surface 2 — Remote cache backends

The cache mounts above live in the local BuildKit instance. For CI/CD, you need the cache to be available on every runner. Four options, from simplest to most powerful:

Cache modes — min vs max

# mode=min (default): cache only final layers; smaller, faster to write
docker buildx build \
  --cache-to type=registry,ref=myorg/myapp:cache,mode=min ...

# mode=max: cache ALL intermediate layers; bigger, much more reusable
docker buildx build \
  --cache-to type=registry,ref=myorg/myapp:cache,mode=max ...

Use mode=max for CI/CD. Multi-stage builds have many intermediate layers (the build stage's outputs, for instance) that mode=min doesn't preserve. Recovery on a cold cache is much better with mode=max; the trade-off is registry storage.

Parallel stage execution

BuildKit runs multi-stage builds with parallelism automatically — independent stages run concurrently. Structure stages with minimal dependencies for maximum parallelism. Two stages that build independent artifacts (UI bundle + API binary) run side-by-side; the final stage combines them. Total time = max(stages) + final, not sum.

The compounding effect

Cache mounts alone might save 60% of build time. Registry cache alone might save 80% (cold CI runner). Combined, on a well-ordered Dockerfile, a CI build that took 6 minutes drops to under a minute. The cost is approximately three lines of CI config and one BuildKit pragma.

IMAGE Chapter IV

Startup time · the cold-start anatomy.

The classic finding Container image pull accounts for ~76% of total startup time on average.

The diagnosis Decompose the cold-start budget into pull / extract / create / start / warmup. Optimize the largest bar first.

A "cold start" is everything from "scheduler picks a node" to "pod serves first request." Most teams treat it as a single number; effective tuning decomposes it into five distinct phases, each with its own levers.

Interactive · Cold-start budget for four scenarios

Decomposing the budget

Pull (network-bound). Bytes from registry to node. Dominates for large images. Levers: registry mirrors per region/cluster, parallel layer pull, lazy pulling (Chapter V).

Extract (CPU/IO-bound). Decompress and unpack tarballs onto the filesystem. Often overlooked; can be 30–40% of an image's setup time. Levers: zstd compression (faster decompress than gzip), efficient snapshotters (overlay2, fuse-overlayfs).

Create (kernel-bound). Set up namespaces, mount the overlayfs, plumb cgroups. Usually 100–500ms regardless of image size. Levers: avoid complex security profiles that delay setup.

Start (app-bound). The process actually runs. Includes interpreter startup, library loads, configuration reads. Levers: ahead-of-time compilation (Native Image for JVM, AOT for Python), reducing imports.

Warmup (app-bound). Model loading, JIT compilation, prefilling caches, opening connections. The hidden cost in ML serving: a 70B model can take 8 minutes to load weights into GPU memory. Levers: tiered model loading, persistent caches (PVCs), warming a healthy pod before traffic.

The application-level startup

Even with perfect image distribution, the app itself often takes longer than the pull:

# In Python — log time at each phase
import time
T0 = time.time()
def stamp(label):
    print(f"{label}: {time.time() - T0:.2f}s")

stamp("process_start")
import torch                              # 1.5s
stamp("torch_imported")
from transformers import AutoModel       # 0.8s
stamp("transformers_imported")
model = AutoModel.from_pretrained("...")   # 45s — bulk of startup
stamp("model_loaded")
model.to("cuda")                           # 12s
stamp("model_on_gpu")
_ = model.generate(...)                       # 8s — JIT, kernels
stamp("first_inference")
# Total: ~70s after the image is on disk

The Kubernetes lifecycle clock

K8s pod startup, in order: kubectl apply → etcd write (~50ms) → scheduler picks node (~50–500ms) → kubelet pulls image (seconds-to-minutes) → image extract → create container (100–500ms) → ENTRYPOINT runs → process initializes → startupProbe passes → readinessProbe passes → Service endpoint updates → traffic flows.

The two big consumers are image pull (network + extract) and the application's own warmup (interpreter + library + model load). Effective optimization addresses whichever is bigger first; for large CUDA/NIM workloads, application warmup often dominates after pulls are optimized.

The right SLO for cold start

For autoscaling response: cold start under 30s. For deploy speed: under 2 minutes. For NIM-class workloads: under 5 minutes from scratch, under 90 seconds with engine cache. If you're past these, the rest of your scaling story is broken regardless of how much GPU you throw at it.

IMAGE Chapter V

Lazy pull · SOCI, eStargz, Nydus.

The trick Start the container before the image is fully pulled. Pull each file on first access. Works because most container images contain large amounts of bytes that are never read at runtime.

Production winners SOCI (AWS, no image conversion needed) and eStargz (broader support, requires image conversion).

A typical 5 GB image has 80% of its bytes never touched at runtime — library variants, locale data, debug symbols, files for code paths the container doesn't execute. Lazy pulling exploits this: start the container immediately, fetch each file when something actually opens it.

Why this works

Research from FAST '16 documents that image pull accounts for 76% of container startup time. Most of the time the pull is moving bytes the application doesn't need on the critical path. Lazy pulling fixes that by:

  1. Building an index of every file in the image: offset, size, hash.
  2. Mounting the image as a virtual filesystem backed by the registry, not local disk.
  3. When the kernel opens a file, fetching that file's bytes via HTTP range request.
  4. Caching what's fetched on the local disk, so subsequent reads are fast.

The net effect: lazy pulling can reduce overhead from minutes to milliseconds, achieving up to 95% faster startup for large container images on supported platforms.

The two production technologies

Axis
eStargz
SOCI (Seekable OCI)
Origin

Google / containerd · 2020. Predates SOCI; the foundation for the whole pattern.

AWS · 2023. Borrowed design principles from stargz, pushed via Fargate adoption.

Image conversion?

Required. eStargz is a backward-compatible extension of OCI tar.gz layers — files are individually compressed with a table-of-contents footer. Conversion changes the image digest, breaking signatures.

Not required. SOCI generates an index as a separate OCI Artifact in the registry, linked back to the unchanged image by OCI Reference Types. Image digests and signatures stay valid.

Performance

Excellent pull-time reduction. Application startup can be slower than baseline if many small files are accessed on the critical path (each is an HTTP fetch).

SOCI maintains the same application startup time as standard images while still dramatically improving pull. The win is asymmetric.

Snapshotter

stargz-snapshotter — containerd plugin. Requires running a daemon on every node.

soci-snapshotter — also containerd plugin. Natively supported on AWS Bottlerocket OS.

Best fit

Multi-cloud or on-prem K8s where you control image conversion in CI. Mature ecosystem.

AWS-heavy fleets, especially on Bottlerocket / ECS / Fargate. Unchanged image signatures = supply-chain friendly.

Nydus — the third option

Nydus (CNCF Sandbox, originally Alibaba) takes a different approach: convert the image to a chunked, content-addressable format where files are broken into 1 MB blocks, each deduplicated across images in the registry. Reads are by chunk, not by file. The result: better deduplication across many similar images, slightly more random-access overhead. Used at hyperscale where many similar images share a base.

The configuration story

# Install soci-snapshotter (binary or systemd service)
# /etc/containerd/config.toml additions:

[proxy_plugins]
  [proxy_plugins.soci]
    type = "snapshot"
    address = "/run/soci-snapshotter-grpc/soci-snapshotter-grpc.sock"

[plugins."io.containerd.grpc.v1.cri".containerd]
  snapshotter = "soci"            # use SOCI by default

# Restart containerd
systemctl restart containerd

# Generate SOCI index in CI after building image
soci create myorg/myapp:v1.2.3 --span-size 4194304
soci push myorg/myapp:v1.2.3

When lazy pull is and isn't a win

Big wins: autoscaling response time (pod ready in 5s instead of 60s), CI ephemeral environments (PR previews spin up instantly), batch jobs with many short tasks, scale-from-zero serverless patterns.

Marginal wins: long-running services where startup is a one-time cost amortized over hours of runtime, especially if the app touches most of the image at startup anyway.

The trade-off to understand: lazy loading doesn't eliminate the time required to download image data — it redistributes it from startup time to runtime. Tune SOCI's parallel pull mode settings for your workload; production teams have reported a 60% improvement just by tuning these parameters.

The deployment heuristic

If pod startup time matters and images are over 1 GB, lazy pulling is worth deploying. Below 500 MB or for long-lived services, the operational complexity (snapshotter daemon, index generation in CI) usually isn't justified. Mixed clusters (some lazy, some not) work fine — set the snapshotter via runtimeClass per pod.

RUNTIME Chapter VI

Memory · cgroup v2, OOM, page cache.

The shift cgroup v2 (default in all modern distros) introduces memory.high — a soft limit that triggers reclaim instead of OOM-kill. Used right, it prevents most "container died at midnight" pages.

The page cache trap Container memory includes file cache. RSS isn't enough to predict OOM behavior.

Container memory is the most mis-understood resource in modern containers. The kernel doesn't ramp up pressure as you approach the limit — it OOM-kills you the moment you exceed it. cgroup v2's memory.high changes this; understanding it is the single biggest reliability win for memory-bound workloads.

Interactive · The cgroup v2 memory interface

The page cache trap

A container's "memory" as the kernel accounts it includes both anonymous memory (your app's malloc'd RAM) and file-backed page cache (the kernel's cache of files the app has read). For a database or any heavy file reader, the page cache can dwarf the RSS — and counts toward memory.max.

# Get the container's cgroup path
CG="/sys/fs/cgroup/system.slice/docker-$(docker inspect --format '{{.Id}}' myapp).scope"

# All the memory breakdown
cat $CG/memory.stat
# anon          4194304    ← actual app memory
# file          16777216   ← page cache (file reads)
# kernel        524288     ← kernel slabs
# file_mapped   8388608    ← mmap'd files (also in file)

# Current usage vs limit
cat $CG/memory.current        # total bytes used right now
cat $CG/memory.max            # hard limit (or "max")

# Did anything get throttled or OOM-killed?
cat $CG/memory.events

The MemoryQoS feature (Kubernetes v1.36)

Kubernetes v1.36 (April 2026) graduated Memory QoS to a usable alpha that sets memory.high for you. Memory QoS uses the cgroup v2 memory controller to give the kernel better guidance on how to treat container memory:

apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
featureGates:
  MemoryQoS: true
memoryReservationPolicy: TieredReservation
memoryThrottlingFactor: 0.9             # memory.high = 90% of limit

With this enabled, pods get smooth degradation under memory pressure instead of sudden OOM kills. Caveat: kernel 5.9+ required — earlier kernels have a livelock bug where memory.high triggers indefinite hangs.

The three classes of memory problem

Symptom
Root cause
Fix
Pod OOMKilled on first request
Working set just slightly exceeds memory.max. No gradual pressure; kernel just oom-kills at the cliff.
Add headroom OR use MemoryQoS (memoryThrottlingFactor: 0.9) for early reclaim before max.

Tuning per workload type

Web service / API. Set memory.high = 80–90% of memory.max. Triggers reclaim before OOM, keeps response times stable.

Database / cache. Use memory.min (TieredReservation) to protect the page cache from being reclaimed under node pressure. Don't set high — these apps want to fill memory.

ML training. Guaranteed QoS, requests=limits, no memory.high. Training peaks are predictable; you want hard limits.

ML inference. Guaranteed QoS for the model server. The KV cache and activations are sized at startup; runtime memory should be flat. OOM here means a bug.

The single biggest reliability lever

Enable Memory QoS, set memory limits to peak × 1.5×, monitor container_memory_working_set_bytes. This eliminates ~80% of "the container died at 3 AM" pages. It's free; it requires kernel 5.9+. The trade-off (slightly slower operation when memory.high triggers reclaim) is almost always worth it.

RUNTIME Chapter VII

CPU · throttling, pinning, NUMA.

The trap CPU limits don't behave like memory limits. A container at "0% CPU" can still be heavily throttled if its quota is set wrong.

The watch metric container_cpu_cfs_throttled_seconds_total increasing while the app waits.

CPU is the most subtly mis-tuned resource in containers. Where memory has an unambiguous limit and a clear "OOM" failure mode, CPU operates in 100ms quota windows that produce invisible latency spikes. A perfectly healthy-looking pod can be losing 30% of its work to throttling without anyone noticing.

How CPU limits actually work

Linux's CFS bandwidth controller divides time into 100ms "periods." Each period, your container gets a quota proportional to its cpu.max setting. If cpu.max says "200ms per 100ms period" (i.e., 2 CPUs), your container can use 200ms of CPU time across all its threads in any 100ms window. Run over → throttled until the next period.

# Find throttling metrics in cgroup v2
CG="/sys/fs/cgroup/system.slice/docker-$(docker inspect --format '{{.Id}}' myapp).scope"
cat $CG/cpu.stat
# usage_usec         12000000      # total CPU used
# user_usec          10000000
# system_usec        2000000
# nr_periods         60000         # how many 100ms windows passed
# nr_throttled       8400          # how many of those throttled
# throttled_usec     420000000     # total time the container was throttled (μs)

# Throttling rate = nr_throttled / nr_periods = 8400/60000 = 14%
# In 14% of windows, the container hit its quota and had to wait

Throttling at 14% means at any moment, your container has a 14% chance of being suspended for tens of milliseconds. This shows up in p99 latency as huge tail spikes; the average barely moves.

The two CPU controls

cpu.weight (relative share). Proportional. If two containers have weights 200 and 100, they get CPU in a 2:1 ratio under contention. Doesn't cap; doesn't throttle. The right tool when you want fair sharing without hard limits.

cpu.max (hard limit, quota). Absolute cap. Throttles when exceeded. Necessary for predictable billing / multi-tenant isolation but introduces tail latency.

In Kubernetes terms: resources.requests.cpu sets cpu.weight, resources.limits.cpu sets cpu.max.

The "no CPU limits" debate

A controversial but increasingly common pattern: set requests but not limits for latency-sensitive workloads. Reasoning: limits cause throttling, which destroys tail latency; the node's total capacity already caps usage; pod priority and PDBs handle eviction when node is full; for latency-critical paths, throttle > eviction > OOM in pain order.

This works well in single-tenant or trusted-multi-tenant environments. Don't do it in untrusted multi-tenant clusters — a noisy neighbor with no limits will eat the node.

CPU pinning & NUMA

For latency-critical workloads (HFT, low-latency inference, real-time audio), CPU pinning matters: pin a process to specific CPUs so the scheduler doesn't move it between cores, losing L1/L2 cache locality every time.

Kubernetes' CPU Manager with static policy assigns exclusive CPUs to Guaranteed-QoS pods. The Topology Manager coordinates this with NUMA awareness — keeps your CPUs, memory, and GPUs in the same NUMA zone.

# /etc/kubernetes/kubelet-config.yaml
cpuManagerPolicy: static
topologyManagerPolicy: single-numa-node
reservedSystemCPUs: "0-3"            # pin system to first 4 CPUs

To engage: pod must be Guaranteed QoS (requests == limits) AND request integer CPU counts (e.g., cpu: "8" not "7.5"). The CPU Manager then gives the pod exclusive access to 8 specific cores; no other pod's threads run on them. (Cross-reference Issue 03 Case 03 — same machinery that fixes NUMA-related training stragglers.)

CPU governor & frequency scaling

On bare-metal nodes, the kernel's CPU frequency governor can introduce surprising latency. The default schedutil ramps frequency up under load; for low-latency apps you want performance mode (always run at max frequency):

# Set performance governor
for cpu in /sys/devices/system/cpu/cpu*/cpufreq/scaling_governor; do
    echo performance > $cpu
done

# Disable C-states for ultra-low latency (costs power)
for cpu in /sys/devices/system/cpu/cpu*/cpuidle/state*/disable; do
    echo 1 > $cpu
done
The diagnostic question

"Is my container being throttled?" should be the first question asked when investigating tail latency. nr_throttled / nr_periods > 1% means the answer is yes. The fix is either more CPU quota or removing the limit; rarely is it the application code.

RUNTIME Chapter VIII

I/O · volumes, FS layers, storage drivers.

The hierarchy Container filesystem < emptyDir < hostPath < PVC. Each has different performance characteristics; pick by access pattern, not by reflex.

The overlooked control io.max for I/O throttling. Most clusters don't set it; some workloads benefit a lot.

Container I/O has four distinct paths, each with its own performance profile. The container filesystem is the worst for performance; the right volume choice can be 100× faster for the right workload.

The four I/O paths

The container's own filesystem, layered on overlayfs. Writes go to the upperdir. Copy-on-write triggers on first write. Slow for write-heavy workloads. Discarded when container exits.

emptyDir: { medium: Memory } — backed by tmpfs (RAM). Counts toward memory limit. For shared-memory IPC (Triton, PyTorch DataLoader), this is the answer.

Default emptyDir uses the node's disk via the kubelet's volume directory. Fast on NVMe; slow on EBS gp2. Doesn't survive pod restart.
node-local

PersistentVolumeClaim with a CSI driver. Performance depends on the underlying storage class (NVMe, SSD, network FS). Survives pod restart and rescheduling.

The container filesystem is slow — proof

# Inside a container, write 1 GB to overlay FS
$ time dd if=/dev/zero of=/tmp/test bs=1M count=1024
# 1073741824 bytes (1.1 GB) in 4.2s = 256 MB/s  ← overlayfs COW

# Same on emptyDir mounted at /work
$ time dd if=/dev/zero of=/work/test bs=1M count=1024
# 1073741824 bytes (1.1 GB) in 1.1s = 980 MB/s  ← direct ext4

# Same on tmpfs emptyDir
$ time dd if=/dev/zero of=/dev/shm/test bs=1M count=1024
# 1073741824 bytes (1.1 GB) in 0.32s = 3.4 GB/s ← RAM

3–10× difference for the same write workload. For any container that writes more than a few MB at a time (logs, scratch data, checkpoint files), mount an emptyDir and write there.

Shared memory and /dev/shm

Containers default to 64 MB of /dev/shm. PyTorch's DataLoader uses shared memory for multi-worker IPC; ML inference servers (Triton) use it for tensor passing. Default size causes mystery hangs:

volumes:
  - name: dshm
    emptyDir:
      medium: Memory
      sizeLimit: 16Gi              # much larger than default 64 MB
volumeMounts:
  - mountPath: /dev/shm
    name: dshm

Storage classes & IOPS

On AWS, GCP, Azure: storage class choice dictates IOPS, latency, and cost. The fastest options:

AWS: io2 (provisioned IOPS), gp3 (baseline 3000 IOPS), instance-store NVMe for ephemeral. For LLM weights / training: instance-store or io2.

GCP: pd-extreme for high IOPS; local SSD for ephemeral. filestore for shared.

Azure: Premium SSD v2 (configurable IOPS); ultra disks for the highest tier.

The default volume pattern for ML

Model weights → PVC mounted read-only across replicas (or S3 with prefetch). Scratch space → emptyDir on local NVMe. Shared memory → emptyDir (medium: Memory) at /dev/shm with 16 GB+ size. Logs → stdout/stderr, harvested by Fluent Bit / Loki. Never write logs to the container filesystem.

RUNTIME Chapter IX

Network · CNI, NAT, MTU, eBPF.

The hierarchy Container → veth pair → bridge → eth0 → fabric. Every hop is a potential bottleneck.

The modern answer Cilium (eBPF) replaces iptables-based kube-proxy. Drops NAT overhead, gives observability for free.

Container networking is layered: each packet traverses a veth pair, a Linux bridge, an iptables (or eBPF) NAT chain, the host's network stack, and finally the wire. Every layer has a cost; for high-throughput services, those costs are measurable.

The packet path, layer by layer

[container app]
       ↓
[container netns]
       ↓ veth peer
[host bridge / linux interface]
       ↓ iptables (kube-proxy) OR eBPF (Cilium)
       ↓ NAT, mark, route
[host eth0]
       ↓
[ToR switch] → [fabric] → [destination]

Two big costs: iptables conntrack (every connection tracked in a kernel hash table; in services with millions of connections, this gets slow and lossy) and NAT (every packet rewritten; CPU cost grows with throughput).

CNI choice matters

simple · default
kube-proxy + iptables
Default in most clusters. Uses Linux iptables for service VIPs and NAT. Works fine up to ~5000 services / ~50000 pods. Beyond that, iptables rule count slows the kernel.
baseline
scale · IPVS
kube-proxy + IPVS
Same kube-proxy, but uses IPVS (in-kernel L4 LB) instead of iptables. Hash table lookup instead of linear iptables walk. Scales to 10× more services.
10× scale
modern · eBPF
Cilium
Replaces kube-proxy with eBPF programs. Drops iptables entirely. Connections via eBPF socket maps; service LB via XDP. ~20% throughput improvement plus deep observability (Hubble).
production winner
simple · routing-based
Calico
Pure L3 routing (no overlay) when run in BGP mode. Each pod gets a real IP routed across the fabric. Excellent performance; requires fabric cooperation (BGP peering with ToRs).
no encapsulation

MTU — the hidden 30% cost

Most CNIs default to a 1500-byte MTU minus VXLAN/IPinIP overhead, leaving the container with an MTU of ~1450. For high-throughput connections, every packet sends 50 fewer bytes than it could, costing 3% throughput.

If your fabric supports jumbo frames (9000-byte MTU), configure the CNI to use it. For TCP-heavy intra-cluster traffic, this can give 30% throughput improvement by reducing packet overhead per byte. Verify with ping -M do -s 8972 from one pod to another — if the ping works, the fabric supports it end-to-end.

Service mesh — when it's worth it

Service meshes (Istio, Linkerd) add sidecar proxies that intercept every connection. Benefits: mutual TLS, retries, traffic splitting, observability. Costs: 5–15% latency overhead per hop, more memory per pod, operational complexity.

The 2026 evolution: ambient mesh (Istio Ambient, Cilium Service Mesh) eliminates the sidecar. Network proxies run as DaemonSets, not per-pod. Same features, less overhead. This is replacing sidecar mesh for new deployments.

Network metrics that matter

# Per-pod RX/TX bytes — basic throughput
container_network_receive_bytes_total
container_network_transmit_bytes_total

# Packet drops — congestion or buffer too small
container_network_receive_packets_dropped_total
container_network_transmit_packets_dropped_total

# TCP retransmits — packet loss
node_netstat_TcpExt_TCPLostRetransmit
# > 0.1% of TCP segments = real problem

# Connection tracking — iptables capacity
node_nf_conntrack_entries        # used
node_nf_conntrack_entries_limit  # max (default 65536)
# If used / limit > 80% sustained, conntrack table is filling up
The Cilium-or-not decision

For new clusters, Cilium with eBPF is the right default. The throughput, observability, and policy story is significantly better than iptables. For existing clusters on kube-proxy + iptables: migrate when you start seeing iptables-related performance issues (conntrack table full, p99 latency spikes correlated with service deploys). Below ~3000 services, the default works fine.

GPU Chapter X

GPU · utilization that actually means something.

The lie nvidia-smi's "GPU utilization" lies. It says 100% when SMs are 5% busy if any kernel is running.

The truth SM activity, tensor core activity, HBM bandwidth — DCGM's profiling metrics. These are the ones that correlate with throughput.

"GPU utilization" is one of the most misleading metrics in the container ecosystem. nvidia-smi's DCGM_FI_DEV_GPU_UTIL reports "% of time at least one kernel was running" — not "% of compute being used." A GPU running a tiny copy kernel reports 100%; one running 5% of its tensor cores reports the same. Cross-reference Issue 03 — this trap costs production teams real money.

The DCGM metric hierarchy

NVIDIA's Data Center GPU Manager exposes a hierarchy of metrics. The deeper you go, the more truthful the answer:

Metric
What it measures
When to use
DCGM_FI_DEV_GPU_UTIL
% of time at least one kernel was running. Reports 100% even for tiny kernels.
Capacity allocation only ("is anyone using this GPU?"). Never as a throughput proxy.
DCGM_FI_PROF_SM_ACTIVE
% of cycles a Streaming Multiprocessor had at least one active warp.
First real-truth metric. Better than GPU_UTIL but still doesn't show how much work the SMs are doing.
DCGM_FI_PROF_SM_OCCUPANCY
Ratio of active warps to maximum possible. Shows how saturated the SM scheduler is.
Critical for understanding if you're SM-bound or memory-bound.
DCGM_FI_PROF_PIPE_TENSOR_ACTIVE
% of cycles the tensor cores were active. THE metric for ML workloads.
Always check for LLM training and inference. Should be 30–60%+ during real work.
DCGM_FI_PROF_DRAM_ACTIVE
% of cycles the HBM was reading/writing. Memory-bandwidth proxy.
High DRAM_ACTIVE + low TENSOR_ACTIVE = memory-bound (typical for decode in LLM serving).
DCGM_FI_DEV_FB_USED
Framebuffer (VRAM) used.
Capacity planning. For inference: KV cache + model weights + activations.

What "good utilization" looks like by workload

LLM training (forward + backward + optimizer): SM_ACTIVE 80–95%, TENSOR_ACTIVE 30–55%, DRAM_ACTIVE 40–70%. High variance is fine if MFU is right.

LLM inference, prefill phase: SM_ACTIVE 85–95%, TENSOR_ACTIVE 40–60%. Compute-bound, similar to training.

LLM inference, decode phase: SM_ACTIVE 30–60%, TENSOR_ACTIVE 5–15%, DRAM_ACTIVE 70–90%. Memory-bound — this is normal. Don't try to "fix" low SM utilization here; it's inherent.

Image generation (diffusion): SM_ACTIVE 80–95%, TENSOR_ACTIVE 30–50%. Iterative; clean compute pattern.

Multi-Instance GPU (MIG) and time-slicing

For workloads that don't fully utilize a GPU, partitioning is the answer:

MIG (Multi-Instance GPU): H100/A100/B200 can be split into up to 7 isolated partitions. Each has dedicated SMs, dedicated HBM, dedicated cache. True isolation; some compute capacity lost to partitioning overhead. Use for: serving many small models that don't fully use a GPU each.

Time-slicing: schedule multiple pods to the same GPU; the GPU context-switches between them. No isolation (one pod's bad behavior affects all); higher overall utilization. Use for: bursty workloads, dev/test environments.

The number that actually matters

If you can only watch one GPU metric in 2026, it's DCGM_FI_PROF_PIPE_TENSOR_ACTIVE for tensor-heavy workloads or DCGM_FI_PROF_DRAM_ACTIVE for memory-bound inference. DCGM_FI_DEV_GPU_UTIL is fine for "is this GPU idle?" capacity-planning queries; it tells you nothing about how well your work is using the chip. Cross-reference Issue 03 Chapter IV — this is the same set of metrics that drives training-throughput tuning.

Application instrumentation

The application's own metrics are usually the most useful — they're what your SLO depends on. Three rules:

1. RED everywhere. Rate, Errors, Duration for every meaningful unit of work (HTTP request, queue job, gRPC method, inference). Use Prometheus histograms for duration; the resulting bucket-based latency view is much better than averages.

2. USE for resources. Utilization, Saturation, Errors for every queue, pool, cache. Connection pool: in-use, max, waits. Queue: depth, enqueue rate, dequeue rate.

3. Domain-specific gauges. Things only your service knows: tokens generated/sec for an LLM, queue depth in a batcher, cache hit rate for a recommender. These tend to predict failures faster than RED.

from prometheus_client import Counter, Histogram, Gauge

# RED

                              "E2E request time", ["model"],
                              buckets=(0.1, 0.5, 1, 2, 5, 10, 30, 60))

# LLM-specific (OTel GenAI semantic conventions)
ttft = Histogram("gen_ai_server_time_to_first_token",
                 "TTFT", ["model"], buckets=(0.05, 0.1, 0.25, 0.5, 1, 2, 5))

                 "TPOT", ["model"], buckets=(0.005, 0.01, 0.02, 0.05, 0.1))
tokens_in = Counter("gen_ai_server_input_tokens", "Input tokens", ["model"])
tokens_out = Counter("gen_ai_server_output_tokens", "Output tokens", ["model"])

# USE — batch & KV cache state

Recording rules — pre-computed dashboards

For dashboards that query 10k+ time series, Prometheus computation can dominate browser load time. Recording rules pre-compute frequently-used queries at scrape time:

- name: container-perf
  interval: 30s
  rules:
  # Per-namespace CPU usage (sum across all pods)
  - record: namespace:container_cpu_usage_seconds:rate5m
    expr: sum by (namespace) (rate(container_cpu_usage_seconds_total[5m]))

OBSERVABILITY Chapter XII

Tracing & profiling · OTel + Pyroscope + eBPF.

Three signals Metrics tell you what. Traces tell you where. Profiles tell you why.

The 2026 norm OpenTelemetry for traces and metrics. Pyroscope for continuous profiling. eBPF (Cilium Hubble, Pixie) for kernel-level visibility.

Metrics tell you what happened (the rate of errors went up). Traces tell you where (this request spent 3 seconds in the database). Profiles tell you why (in that span, the database driver was busy in the connection pool waiting for a lock). The three signals are complementary; modern container observability uses all three.

OpenTelemetry — the unified collection layer

OpenTelemetry (OTel) is the CNCF's unified standard for emitting traces, metrics, and logs. The 2026 architecture has three pieces:

OTel SDK / auto-instrumentation: in your application, captures spans, metrics, and logs in the OTLP wire format.

OTel Collector (agent): runs as a DaemonSet on each node. Receives OTLP from local applications, batches, enriches with K8s metadata, forwards.

OTel Collector (gateway): centralized cluster-level. Filters, samples, routes to multiple backends (Prometheus, Tempo, Loki, vendor systems). The fan-out point.

receivers:
  otlp:
    protocols:
      grpc: { endpoint: 0.0.0.0:4317 }
      http: { endpoint: 0.0.0.0:4318 }
  prometheus:
    config:
      scrape_configs:
        - job_name: dcgm
          static_configs: [{ targets: ["localhost:9400"] }]

processors:
  batch: { timeout: 10s, send_batch_size: 1024 }
  k8sattributes:                         # enrich with pod/namespace/cluster
    auth_type: serviceAccount
    extract:
      metadata: [k8s.pod.name, k8s.namespace.name, k8s.node.name]
  tail_sampling:                         # keep errors, sample 1% of OK
    policies:
      - { name: errors, type: status_code, status_code: { status_codes: [ERROR] } }
      - { name: slow, type: latency, latency: { threshold_ms: 500 } }
      - { name: probabilistic, type: probabilistic, probabilistic: { sampling_percentage: 1 } }

exporters:
  otlp/tempo: { endpoint: tempo-gateway:4317 }
  prometheusremotewrite: { endpoint: http://prometheus-gateway/api/v1/write }

service:
  pipelines:
    traces: { receivers: [otlp], processors: [k8sattributes, tail_sampling, batch], exporters: [otlp/tempo] }
    metrics: { receivers: [otlp, prometheus], processors: [k8sattributes, batch], exporters: [prometheusremotewrite] }

Tracing — what to trace, when

Trace every request that crosses a service boundary, every meaningful internal operation. For an LLM inference pipeline:

[ request_received ]                            # 0ms
   [ auth_check ]                              # 5ms
   [ rate_limit ]                              # 1ms
   [ retrieve_context (RAG) ]
      [ embed_query ]                          # 12ms
      [ vector_search ]                        # 38ms
   [ queue_to_inference ]                      # 4ms wait in queue
   [ inference ]
      [ prefill ]                              # 320ms
      [ decode (streamed) ]                    # 2840ms
   [ post_process ]                            # 8ms
   [ response_sent ]                           # total: 3.23s

When the user reports "it was slow," a trace tells you immediately whether the time went into auth, retrieval, queue wait, prefill, or decode. Without traces, you guess.

Sampling — keep the budget tight

Head sampling: decide at trace start (typically 1–10% probability). Simple, deterministic per trace. Drawback: misses rare slow / error traces.

Tail sampling (preferred for high-traffic services): the OTel collector buffers the full trace; after it completes, decides whether to keep based on rules (always keep errors, always keep slow, sample 1% of healthy fast). Costs more memory in the collector but keeps the interesting traces.

Continuous profiling — Pyroscope

Sampling-based profiling captures stack traces of every running thread, every few seconds. Aggregated over time, you get a flame graph showing exactly where CPU is spent.

2026 production standard: Grafana Pyroscope. Low overhead (~1% CPU). Continuous (not just during an incident). Can profile from agents in K8s with no code change for Go, Java, Python, Node.

# Pyroscope server
helm install pyroscope grafana/pyroscope -n monitoring

# Python — wrap your entrypoint
RUN pip install pyroscope-io

# In your code
import pyroscope
pyroscope.configure(
    application_name="my-service",
    server_address="http://pyroscope.monitoring:4040",
    tags={"region": "us-east-1", "version": "v1.2.3"},
)

The output: flame graphs by service, by container, over time. When a service slows down, you see immediately which function changed — often even before the user reports the slowdown.

eBPF — kernel-level observability

eBPF lets you attach kernel-side probes that capture syscalls, network packets, file ops, scheduler events. Tools like Cilium Hubble (network) and Pixie (general L7 protocols) give visibility you can't get any other way.

The 2026 use cases:

  • CPU run-queue latency — measure threads waiting to be scheduled. Detects cgroup throttling that traditional metrics miss.
  • I/O latency histograms — find the rare 200ms fsync stalls that ruin p99 but average to nothing.
  • TCP retransmits per connection — narrow down which flows are seeing packet loss.
  • Service-mesh-free L7 tracing — Pixie captures HTTP/gRPC calls at the kernel without code changes or sidecars.
# BCC tools — install via apt or as a container

# Find slow block I/O
biolatency -m 10

# Find off-CPU time (threads waiting)
offcputime -K -f 30 > offcpu.txt

# Network — slow connect()s
tcpconnlat 10           # show connects taking >10ms

# Run-queue latency (scheduling delays)
runqlat -P -m
The four signals in 2026

Metrics (Prometheus + DCGM) for "is something wrong right now." Logs (Loki) for "what was happening when it broke." Traces (Tempo + OTel) for "which span owned the slowness." Profiles (Pyroscope) for "which function in that span." Each fills a gap the others can't. Skipping any of them means longer incidents.
