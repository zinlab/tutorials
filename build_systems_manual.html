Part I — The compilation model

What happens to your code.

Before you can reason about Make, CMake, or packaging, you need a precise mental model of what "building" does. Every build system is, fundamentally, an elaborate way to invoke a compiler and a linker in the right order. Understand that core transformation and the rest is orchestration.

01What "building" actually means

"Build" is a loaded word. For an interpreted language like pure Python, there's barely a build at all — the interpreter reads your source and runs it. For a compiled language like C++, "building" means a multi-stage transformation from human-readable text into machine code that the CPU executes, plus the gluing-together of all the pieces.

The whole job decomposes into two fundamental operations, repeated and orchestrated:

  • Compiling — translating one source file into one object file of machine code, in isolation, knowing nothing about the rest of the program.
  • Linking — stitching all the object files (and libraries) together into a final executable or library, resolving the cross-references between them.

Everything else — Make, CMake, Ninja, Bazel — exists to answer two questions efficiently: what needs compiling and linking, and in what order. A build system is a dependency graph plus a rule for turning inputs into outputs plus the machinery to do the minimum work when something changes.

A build system is a memoized function over a dependency graph. Its inputs are source files and configuration; its outputs are artifacts; its value is doing the least work to bring outputs up to date when inputs change. Make, CMake, Bazel, and the rest are all variations on this one idea — they differ in how they describe the graph, how precisely they track dependencies, and how aggressively they cache.

The layers, top to bottom

YOU WRITE main.cpp, utils.cpp, utils.h, CMakeLists.txt | v META-BUILD CMake / Meson reads your project description, (configure) generates the actual build files | v BUILD TOOL Make / Ninja reads generated files, figures out (orchestrate) what's stale, runs commands in order | v COMPILER DRIVER gcc / clang / cl one invocation per source file | (preprocess + compile + assemble) v OBJECT FILES main.o, utils.o machine code + symbol tables, | not yet runnable v LINKER ld / lld / link resolves symbols across objects, | combines into final artifact v ARTIFACT ./myapp or libfoo.so or foo.pyd the thing you actually run or ship

Most of this manual walks up and down these layers. Parts I–III cover the compiler and linker (the bottom). Parts IV–V cover Make and CMake (the middle). Parts VI–VII cover the Python-specific apex: bindings and packaging. Part VIII steps back to the alternatives and the cross-cutting concerns.

Why this is worth knowing deeply

You can build software for years by copy-pasting CMake incantations and hoping. It works until it doesn't — until you hit an "undefined reference" error, a version-mismatch crash at load time, an ABI incompatibility in a Python wheel, a circular dependency the linker can't resolve. Every one of those failures lives at a specific layer. Engineers who understand the layers diagnose in minutes what others stare at for hours.

Build errors are not random. Each one lives at a specific layer of the pipeline; knowing the layers turns "no idea" into "I know exactly which tool to interrogate."

02The C/C++ compilation pipeline

When you run gcc main.cpp -o main, that single command hides four distinct stages. Understanding them lets you intervene at any point — to inspect intermediate output, to diagnose where something broke, or to split the work for a build system.

main.cpp | | [1] PREPROCESS cpp / gcc -E | - expand #include (paste header contents in) | - expand #define macros | - process #if / #ifdef conditionals v main.ii (preprocessed translation unit — pure C++, no directives) | | [2] COMPILE gcc -S | - parse, type-check, optimize | - emit target assembly v main.s (assembly text) | | [3] ASSEMBLE as / gcc -c | - translate assembly to machine code | - produce relocatable object v main.o (object file — machine code + symbol table + relocations) | | [4] LINK ld / gcc (driver invokes linker) | - combine objects + libraries | - resolve symbols, apply relocations v main (executable)

Watching each stage

You can stop the compiler driver after any stage. This is the single most useful debugging technique most engineers never learn:

# [1] Preprocess only — see exactly what the compiler sees after #includes/macros
$ g++ -E main.cpp -o main.ii
$ wc -l main.ii        # a trivial file becomes thousands of lines after #include <iostream>

# [2] Compile to assembly — see what code was generated
$ g++ -S main.cpp -o main.s

# [3] Assemble to object, but do NOT link
$ g++ -c main.cpp -o main.o

# [4] Link object(s) into the final executable
$ g++ main.o -o main

Why the split matters for build systems

The crucial property: step 3 (producing main.o) only depends on main.cpp and the headers it includes. It knows nothing about utils.cpp. This independence is what makes incremental builds possible: change main.cpp and only main.o needs rebuilding; utils.o is untouched. The linker then re-combines them.

This is also why C++ builds parallelize so well: every .cpp → .o compile is independent and can run on its own core. The link step is the serialization point at the end.

main.cpp ---[compile]---> main.o --+ utils.cpp ---[compile]---> utils.o --+--[link]--> myapp math.cpp ---[compile]---> math.o --+ (parallel, independent) (serial, combines all)

The separate-compilation model and its cost

Because each translation unit is compiled in isolation, the compiler must be told about anything defined elsewhere — via header files containing declarations. This is the source of C++'s header/source split, its long compile times (every TU re-parses every header it includes), and a whole category of build pathologies. C++20 modules aim to fix this, but the header model still dominates and will for years.

The recompilation cascade

Because every .cpp that includes a header must recompile when that header changes, a change to a widely-included header (say, a core types.h included by 500 files) triggers 500 recompiles. This is why "put as little as possible in headers" and "use forward declarations" and "pimpl idiom" are not style preferences — they are compile-time-performance engineering. A build system tracks these header dependencies precisely (§18); getting that tracking wrong gives you either stale builds or full rebuilds.

03Translation units, declarations, the ODR

Three concepts form the bedrock of how C++ programs are assembled from pieces. Misunderstandings here cause the most baffling linker errors.

The translation unit

A translation unit (TU) is a single source file after preprocessing — i.e., the .cpp with all its #included headers textually pasted in and all macros expanded. The compiler processes exactly one TU at a time, producing exactly one object file. The TU is the atom of compilation.

Declaration vs definition

The distinction that everything hinges on:

  • A declaration tells the compiler something exists and what its type is: int add(int, int); or extern int counter;. It promises that a definition exists somewhere.
  • A definition provides the actual thing: the function body, the storage for the variable: int add(int a, int b) { return a + b; }.

You can declare something many times; you must define it exactly once (per the rules below). Headers contain declarations so that many TUs can use something; one source file contains the definition so the linker has one thing to point at.

utils.h int add(int, int); // DECLARATION — goes in header // (included by many TUs) utils.cpp int add(int a, int b) { // DEFINITION — goes in ONE source return a + b; // (compiled into utils.o) } main.cpp #include "utils.h" // sees the declaration int main() { return add(2, 3); // compiler trusts add exists; } // linker connects to utils.o

The One Definition Rule (ODR)

The ODR is C++'s most important — and most violated — linkage rule. In essence:

  • Every entity (function, variable) used in the program must be defined exactly once across the entire program.
  • Certain entities (classes, inline functions, templates) may be defined in multiple TUs, but every definition must be token-for-token identical.
  • Violating the ODR is undefined behavior — and crucially, the compiler often cannot detect it, because each TU is compiled separately. The linker catches some violations; others silently produce a broken program.

How linkage decides what's visible across TUs

LinkageMeaning
external linkageVisible to other TUs. Normal functions and non-const global variables. The linker can connect references across files.
internal linkageVisible only within its own TU. Achieved with static at file scope, or anonymous namespaces. Each TU gets its own private copy.
no linkageLocal variables, function parameters — exist only within their scope.
inline (since C++17 for variables)May be defined in multiple TUs; the linker merges them into one. How header-only libraries work.

Why this matters in practice

The two most common linker errors map directly to ODR violations:

  • "undefined reference to X" — you declared X and used it, but no TU defined it (or the TU that did wasn't linked in). The promise was made but not kept.
  • "multiple definition of X" — two TUs both defined X with external linkage. Usually because a definition (not just a declaration) ended up in a header that got included by multiple files. Fix: mark it inline, or move the definition to a single source file.
The header-only library trick

Header-only libraries (Eigen, much of Boost, fmt in header mode) put everything in headers. They get away with it because every function is inline or a template — both of which the ODR explicitly permits in multiple TUs, with the linker merging duplicates. The cost is compile time: every TU re-parses and re-instantiates everything it uses. This is the fundamental trade — header-only is easy to consume but slow to compile.

04The preprocessor in depth

The preprocessor is a text-substitution engine that runs before the compiler proper sees anything. It's primitive — it doesn't understand C++ at all, it just manipulates tokens — but it's load-bearing, and its quirks cause real bugs.

The four things it does

DirectiveEffect
#includeTextually paste the named file's contents at this point. Quotes search local dirs first; angle brackets search system/include paths.
#define / #undefDefine or remove a macro — an object-like constant or a function-like text template.
#if / #ifdef / #ifndef / #else / #endifConditional compilation — include or exclude code based on macro values.
#pragmaCompiler-specific directives (e.g., #pragma once, #pragma omp).

Include guards — the canonical pattern

Because #include is literal text pasting, including the same header twice would paste its contents twice — re-declaring everything and breaking the ODR for class definitions. Include guards prevent this:

// utils.h
#ifndef UTILS_H        // if not yet defined...
#define UTILS_H        // define it, so a second include is skipped

int add(int, int);

#endif  // UTILS_H

The modern alternative, supported by every real compiler though not technically standard:

// utils.h
#pragma once

int add(int, int);

#pragma once is cleaner and avoids the (rare) problem of guard-macro name collisions. Use it unless you need maximal portability to exotic compilers.

Macros: power and peril

Function-like macros look like functions but are pure text substitution, with no type checking and no scope:

// Looks fine...
#define SQUARE(x) x * x

// ...until you use it
int a = SQUARE(1 + 2);   // expands to 1 + 2 * 1 + 2 = 5, not 9!

// Defensive version (still imperfect — evaluates x twice)
#define SQUARE(x) ((x) * (x))
int b = SQUARE(1 + 2);   // ((1 + 2) * (1 + 2)) = 9, correct

int i = 5;
int c = SQUARE(i++);     // ((i++) * (i++)) — UB, i incremented twice
Prefer the language to the preprocessor

Modern C++ makes most macro uses obsolete: constexpr for constants, inline functions and templates for function-like macros, enum class for enumerations. The preprocessor has no concept of types, scope, or namespaces — it will happily substitute text into the middle of a string or a comment-adjacent context and produce nonsense. Reserve macros for include guards, conditional compilation, and the few things only the preprocessor can do (stringizing with #, token-pasting with ##, __FILE__/__LINE__).

Conditional compilation and feature detection

The preprocessor's #if family is how a single codebase targets multiple platforms and configurations:

// Platform branching
#if defined(_WIN32)
  #include <windows.h>
#elif defined(__linux__)
  #include <unistd.h>
#elif defined(__APPLE__)
  #include <TargetConditionals.h>
#endif

// Feature / version gating
#if __cplusplus >= 202002L
  // use C++20 features (concepts, ranges, ...)
#endif

// Build-configuration flags set by the build system via -D
#ifdef NDEBUG
  // release: assert() compiles to nothing
#else
  // debug: assert() active
#endif

The build system feeds these macros via compiler flags: -DNDEBUG, -DMY_FEATURE=1. This is the primary channel by which CMake or Make communicates configuration choices down into the source code. When you select a "Release" build type, CMake adds -DNDEBUG; your code reads it via #ifdef. The whole configuration system rests on this preprocessor mechanism.

Useful predefined macros

__LINE__Current line number (integer).

Part II — Compiling

The compiler, up close.

One source file in, one object file out. That single transformation is where optimization happens, where debug info is born, where warnings catch bugs, and where the standards and flags you choose silently shape everything downstream. This part is the compiler's-eye view.

05The compiler driver & its flags

When you type gcc or clang or cl, you're not talking to the compiler directly — you're talking to the compiler driver, a program that orchestrates the preprocessor, the actual compiler, the assembler, and the linker, passing the right flags to each. Understanding which flags go to which stage demystifies a lot.

The three drivers

DriverEcosystem
gcc / g++GNU Compiler Collection. The Linux default. gcc for C, g++ for C++ (links the C++ standard library).
clang / clang++LLVM-based. Default on macOS, increasingly common on Linux. Better diagnostics, faster compiles, the basis for many tools.
cl / cl.exeMicrosoft Visual C++ compiler. Windows-native. Different flag syntax entirely (/O2 not -O2).

GCC and Clang share almost all flag syntax (Clang was designed as a drop-in). MSVC is its own world. Cross-platform build systems exist largely to paper over this divide.

The flags that actually matter

You can spend a career and use perhaps thirty compiler flags regularly. Here are the load-bearing ones (GCC/Clang syntax):

-o FILEName the output file.
-I DIRAdd DIR to the header search path (for #include).

-L DIRAdd DIR to the library search path (for the linker).
-l NAMELink against libNAME.so or libNAME.a.

-O0/-O1/-O2/-O3/-Os/-OgOptimization level (§06).

-march=nativeOptimize for the exact CPU compiling the code (use carefully — non-portable binary).

The canonical separate-compilation invocation

# Compile each TU to an object (independently, in parallel)
$ g++ -std=c++20 -O2 -g -Wall -Wextra -Iinclude -c src/main.cpp  -o build/main.o
$ g++ -std=c++20 -O2 -g -Wall -Wextra -Iinclude -c src/utils.cpp -o build/utils.o

# Link the objects + a library into the executable
$ g++ build/main.o build/utils.o -Llibs -lfoo -o myapp

The compile lines carry -I, -D, -std, -O, -W — things the compiler needs. The link line carries -L, -l — things the linker needs. Mixing them up is a common confusion: -l on a compile-only (-c) line does nothing useful; -I on a link line is harmless but pointless.

Compile flags and link flags are different populations. Compile flags (-I -D -std -O -W -fPIC) configure the source→object translation and must be identical across all TUs that will be linked together (especially -std and ABI-affecting flags, or you get ODR/ABI violations). Link flags (-L -l -shared -rpath) configure how objects combine. Every build system, deep down, is sorting flags into these two buckets and feeding them to the right stage.

Seeing what the driver does

# Print every sub-command the driver runs (illuminating)
$ g++ -v main.cpp -o main
# ...shows the cc1plus invocation, the as invocation, the collect2/ld invocation

# Print the include search paths the preprocessor will use
$ g++ -E -x c++ - -v < /dev/null

# Print the library/startup file search dirs
$ g++ -print-search-dirs

06Optimization, debug info, sanitizers

Three orthogonal axes that the build type (Debug/Release) bundles together. Understanding them separately lets you make deliberate choices instead of accepting CMake's defaults blindly.

Optimization levels

LevelBehavior
-O0No optimization. Fastest compile, slowest code, most faithful debugging (every variable lives where you expect). The Debug default.
-O1Basic optimizations, modest compile cost.
-O2The standard production level. Aggressive but doesn't bloat code size much. The Release default.
-O3Everything in -O2 plus more aggressive inlining and vectorization. Sometimes faster, sometimes not (code bloat can hurt cache). Measure before assuming it helps.
-OsOptimize for size. Useful for embedded or when binary size matters.
-OgOptimize but keep debugging usable. The right choice for a debuggable-but-not-glacial build.
-Ofast-O3 plus -ffast-math (breaks strict IEEE float semantics). Dangerous for numerical code; never use without understanding the consequences.

Debug information

-g embeds DWARF debug info into the object/binary: the mapping from machine instructions back to source lines, variable names, types, scopes. This is what lets gdb/lldb show you source while stepping. Crucially:

  • -g is orthogonal to optimization. You can (and for production often should) build -O2 -g — optimized code with debug info. The debug info doesn't slow the program; it just makes the binary bigger.
  • At high optimization, debug info gets approximate — variables get optimized into registers or away entirely, so the debugger says "<optimized out>". This is the tension -Og resolves.
  • Debug info can be split out into separate .debug / .dSYM files (via objcopy --only-keep-debug or -gsplit-dwarf) so the shipped binary stays small but symbols are available for crash analysis.
Ship optimized binaries with debug info, then strip

The production pattern: build -O2 -g, archive the unstripped binary (or split debug info) somewhere safe, then strip the copy you ship. When a crash report arrives with a stack trace, you symbolicate it against the archived debug info. You get small shipped binaries and debuggable crashes. Throwing away debug info entirely means crash reports are useless hex addresses.

Sanitizers — the highest-ROI debugging tools in C++

Sanitizers are compiler-inserted runtime checks that catch entire categories of bugs that otherwise manifest as random crashes or silent corruption. They're enabled with -fsanitize= and require recompiling.

SanitizerCatches
-fsanitize=address (ASan)Heap/stack/global buffer overflows, use-after-free, double-free, memory leaks. ~2× slowdown. Catches the bugs that cause most C++ crashes.
-fsanitize=undefined (UBSan)Undefined behavior: signed overflow, null-deref, misaligned access, invalid casts, shifts out of range. Low overhead.
-fsanitize=thread (TSan)Data races between threads. ~5-15× slowdown but finds races static analysis can't.
-fsanitize=memory (MSan)Use of uninitialized memory. Clang-only; requires instrumenting all code including libc++.
-fsanitize=leak (LSan)Memory leaks (subset of ASan, also standalone).
 
# Build with ASan + UBSan (the standard combo) and run
$ g++ -std=c++20 -O1 -g -fsanitize=address,undefined -fno-omit-frame-pointer \
      main.cpp -o main_asan
$ ./main_asan
# On a bug, ASan prints the exact line, the allocation site, and the access —
# turning a mysterious segfault into a precise report.

Run your test suite under sanitizers in CI. The bugs they catch — use-after-free, races — are the ones that cost days to debug in production. -fno-omit-frame-pointer gives readable stack traces; pair it with sanitizer builds.

07Warnings, standards, language modes

Warnings are free bug-finding

Compiler warnings catch a remarkable fraction of bugs at zero runtime cost. The discipline of building warning-clean is one of the cheapest quality investments available.

FlagEffect
-WallDespite the name, NOT all warnings — a curated set of the most useful ones. Always on.
-WextraAdditional useful warnings beyond -Wall. Also always on.
-WpedanticWarn on non-standard extensions. Use for portable code.
-WerrorTurn warnings into errors. Forces a warning-clean build. Use in CI; optionally relax locally.

# The reasonable baseline for a serious project
$ g++ -std=c++20 -Wall -Wextra -Wpedantic -Werror ...
-Werror in CI, not necessarily locally

-Werror in CI guarantees no warning ever merges. But -Werror for every local developer build can be infuriating — a new compiler version adds a warning and suddenly nobody can build. The common compromise: -Werror only in CI (gated on a known compiler version), warnings-as-warnings locally. Some projects gate -Werror behind a CMake option defaulting OFF, ON in CI.

The standard version (-std)

-std=c++NN selects which version of the C++ standard the compiler enforces. This affects which language features and standard-library facilities are available, and subtly affects ABI.

StandardHeadline features
c++11auto, lambdas, move semantics, smart pointers, threads. The "modern C++" baseline.
c++14Generic lambdas, return-type deduction. Minor.
c++17Structured bindings, std::optional, std::variant, if constexpr, parallel algorithms, filesystem. Widely adopted baseline.
c++20Concepts, ranges, coroutines, modules, <format>, three-way comparison. Large release.
c++23std::expected, std::mdspan, more ranges, deducing this. Incremental.
The -std must match everywhere

Every TU linked into the same binary must use a compatible standard and standard-library version. Mixing -std=c++17 and -std=c++20 objects, or linking against a library built with a different libstdc++ ABI, can produce subtle ODR violations and crashes — because, e.g., std::string's layout differs between standard-library versions. This is the root of many "it links but crashes mysteriously" bugs. Build the whole dependency graph with one toolchain and one standard.

GNU extensions vs strict standard

-std=c++20 enables GNU extensions on top of the standard; -std=gnu++20 is the actual GCC default and includes more extensions. For portable code, prefer the non-gnu variant and add -Wpedantic to flag accidental extension use that won't compile on other compilers.

08Object files: symbols, sections, relocations

The object file is the output of compilation and the input to linking. Cracking one open demystifies linking entirely — most linker errors become obvious once you can see what's inside the .o files.

What's inside an object file

An object file (ELF on Linux, Mach-O on macOS, COFF/PE on Windows) contains:

  • Sections — chunks of the program: .text (machine code), .data (initialized globals), .bss (zero-initialized globals, no space used in the file), .rodata (constants, string literals).
  • A symbol table — the named entities this object defines or references. Each symbol is either defined here (the object provides it) or undefined (the object needs it from elsewhere).
  • Relocations — placeholders the linker must fill in. When main.o calls add(), it doesn't yet know add's address; it leaves a relocation entry saying "patch this spot with the address of add once you know it."

Reading symbols with nm

$ g++ -c utils.cpp -o utils.o
$ nm utils.o
0000000000000000 T _Z3addii        # T = defined in .text, the mangled name of add(int,int)

$ g++ -c main.cpp -o main.o
$ nm main.o
                 U _Z3addii        # U = UNDEFINED — main.o needs add() from elsewhere
0000000000000000 T main

The symbol-type letters from nm are diagnostic gold:

LetterMeaning
T / tDefined in text (code) section. Uppercase = external (visible to linker), lowercase = local.
D / dDefined in initialized data section.
B / bDefined in BSS (uninitialized data).
R / rDefined in read-only data.
UUndefined — referenced but not defined here. The linker must find it.
W / wWeak symbol — used if nothing stronger is found (templates, inline functions).

The "undefined reference" error, demystified

When the linker reports undefined reference to 'add(int, int)', it means: some object has a U _Z3addii entry, and no object or library in the link provided a matching T _Z3addii. The fix is always one of:

  • You forgot to compile and link the source file that defines it (add utils.o to the link line).
  • You forgot to link the library that provides it (add -lfoo).
  • A signature mismatch — you declared add(int, int) but defined add(long, long); the mangled names differ, so the linker sees no match.
  • A language-linkage mismatch — C++ code calling a C function without extern "C", so the names mangle differently.

Disassembly and section inspection

# Disassemble the machine code
$ objdump -d utils.o

# List sections and their sizes
$ size utils.o
   text    data     bss     dec     hex filename
     23       0       0      23      17 utils.o

# Full ELF structure (sections, symbols, relocations)
$ readelf -a utils.o

# Just the relocations — what the linker still needs to patch
$ readelf -r main.o

Name mangling preview

Notice add(int, int) appears as _Z3addii. C++ mangles function names to encode their full signature (namespace, class, parameter types) into a single linker symbol, because the linker only understands flat strings and C++ has overloading. _Z3addii = _Z (mangled), 3add (3-char name "add"), ii (two int params). We cover mangling fully in §12 — but seeing it here explains why a signature mismatch produces "undefined reference": the mangled names simply don't match.

# Demangle a symbol back to human-readable form
$ echo _Z3addii | c++filt
add(int, int)

# nm can demangle inline
$ nm -C main.o
                 U add(int, int)
0000000000000000 T main

When the linker complains, reach for nm. The undefined symbol it names, matched against what your objects and libraries define, is the entire diagnosis.

Part III — Linking

Stitching it together.

Linking is where most build mysteries live. The linker is a humble program with a narrow job — resolve symbols, apply relocations, combine sections — but its quirks (one-pass resolution, link order sensitivity, the static-vs-shared divide, the runtime loader) generate a disproportionate share of confusing errors. This part is the longest because it pays the highest diagnostic dividends.

09The linker's job: resolution & relocation

The linker takes a pile of object files and libraries and produces a single artifact (executable or shared library). It does three things:

  1. Symbol resolution — for every undefined symbol (U in nm), find the object or library that defines it, and connect the reference to the definition. If it can't find a definition: "undefined reference." If it finds two: "multiple definition."
  2. Relocation — now that every symbol has a final address, go back and patch every relocation entry (every place that referenced a symbol whose address wasn't known at compile time) with the real address.
  3. Section combination — merge all the .text sections into one, all the .data sections into one, etc., laying out the final memory image.
main.o utils.o +------------------+ +------------------+ | .text: main() | | .text: add() | | call add <----+--------+--> (defined here) | | [reloc: add?] | | | | symbols: | | symbols: | | T main | | T add | | U add --------+---+ | | +------------------+ | +------------------+ | ^ +--------------+ resolution: U add in main.o matched to T add in utils.o | v relocation: patch the call site with add's address +--------------------------------------------+ | myapp (executable) | | .text: main() { call 0x401126 } add() | +--------------------------------------------+

Two linkers you'll meet

The compiler driver invokes the linker for you

You rarely call ld directly. When you run g++ main.o utils.o -o myapp, the driver invokes ld with a great deal of added machinery: the C++ runtime startup files (crt0), the standard library (-lstdc++), the C library (-lc), and platform specifics. This is why you should link C++ programs with g++/clang++, not bare ld or gcc — the C++ driver knows to pull in the C++ runtime.

10Static libraries & archives

A static library (.a on Unix, .lib on Windows) is simply an archive of object files — a .tar-like bundle with an index. It is not magic; it's a convenience for shipping many .o files as one unit.

Creating and using one

# Compile the objects
$ g++ -c -fPIC math.cpp string.cpp io.cpp

# Archive them into a static library (ar = archiver)
$ ar rcs libmylib.a math.o string.o io.o
#         r = insert, c = create, s = write symbol index

# Inspect contents
$ ar t libmylib.a
math.o
string.o
io.o

# Link a program against it
$ g++ main.o -L. -lmylib -o myapp
#              ^      ^ links libmylib.a (drops "lib" prefix and ".a" suffix)
#              search current dir

What static linking actually does

When you link against a static library, the linker copies the needed object files out of the archive into your executable. The library is consumed at link time; the resulting binary is self-contained. After linking, libmylib.a is no longer needed — the relevant code is baked into myapp.

Crucially, the linker only pulls in the object files it needs to resolve undefined symbols. If your program uses add() from math.o but never touches io.o, only math.o gets copied in. This is object-file granularity, not function granularity (a single .o is all-or-nothing unless you compile with -ffunction-sections -fdata-sections and link with --gc-sections).

PropertyStatic linking
Final binarySelf-contained; includes the library code.
DistributionOne file; no runtime dependency on the library being present.
Binary sizeLarger — every program copies the code it uses.
Memory at runtimeEach process has its own copy of the code.
Updating the libraryRequires relinking every program that uses it.
Symbol resolutionAt link time. Errors surface during the build.
Static linking's appeal for deployment

A statically-linked binary "just runs" — no worrying about whether the right library version is installed on the target. This is why Go and Rust default to mostly-static binaries, and why static linking is popular for containers and CLI tools. The cost is binary size and the inability to patch the library without rebuilding. For Python extension modules, though, static linking of your C++ dependencies into the extension is often exactly right (§41).

11Shared libraries & dynamic linking

A shared library (.so on Linux, .dylib on macOS, .dll on Windows) is loaded at runtime, not copied into your binary at link time. Many programs share one copy on disk and (largely) in memory. This is the dominant model for system libraries and a frequent source of deployment pain.

Creating and using one

# Compile with -fPIC (Position-Independent Code) — REQUIRED for shared libs
$ g++ -c -fPIC math.cpp string.cpp io.cpp

# Link the objects into a shared library
$ g++ -shared math.o string.o io.o -o libmylib.so

# Link a program against it (same syntax as static!)
$ g++ main.o -L. -lmylib -o myapp

# But now the .so is NOT copied in — it must be found at RUNTIME
$ ./myapp
./myapp: error while loading shared libraries: libmylib.so:
cannot open shared object file: No such file or directory

# Tell the loader where to find it (one option — see §13 for better ones)
$ LD_LIBRARY_PATH=. ./myapp     # now it runs

Why -fPIC matters

Position-Independent Code can be loaded at any memory address without modification. Shared libraries need this because the OS may load the library at a different address in each process (and address-space layout randomization moves it around). Code that hardcodes addresses can't be shared this way. The compiler generates indirect references (through the GOT — Global Offset Table — and PLT — Procedure Linkage Table) instead. Forgetting -fPIC when building objects destined for a shared library produces a link error: relocation R_X86_64_32 against ... can not be used when making a shared object; recompile with -fPIC.

The two-phase symbol resolution

With shared libraries, symbol resolution splits across two times:

  • At link time, the linker checks that every symbol your program uses exists in some shared library, and records a dependency ("I need libmylib.so") plus the symbol names. It does not copy the code.
  • At load time, the dynamic loader (ld.so) finds the actual .so files, maps them into memory, and resolves the symbols to their real runtime addresses.

This is why a program can link fine but fail at startup with "cannot open shared object file" (the .so isn't where the loader looks) or "undefined symbol" (the .so is found but a newer version removed a symbol). The link-time and run-time worlds are different; bugs hide in the gap.

PropertyDynamic (shared) linking
Final binarySmall; contains only a reference to the library.
DistributionMust ship or guarantee the .so at the right version on the target.
Binary sizeSmaller — code lives once in the shared library.
Memory at runtimeOne copy of code shared across all processes using it.
Updating the libraryReplace the .so; all programs pick up the new version (if ABI-compatible) without relinking.
Symbol resolutionPartly at link time, finalized at load time. Errors can surface at startup.

SONAME and versioning

Shared libraries carry a version scheme to manage ABI compatibility. You'll see a chain of names:

libmylib.so.1.4.2     # the real file (major.minor.patch)
libmylib.so.1    ->   libmylib.so.1.4.2   # SONAME symlink (major version = ABI)
libmylib.so      ->   libmylib.so.1       # dev symlink used at link time (-lmylib)
  • The SONAME (libmylib.so.1) is embedded in the library and in programs that link against it. The major version is the ABI contract; bumping it signals an incompatible change.
  • Programs record the SONAME they need; the loader finds a file providing that SONAME at runtime. So you can upgrade 1.4.2 → 1.4.3 transparently, but 1.x → 2.x requires relinking.
ABI vs API — the distinction that bites

An API change breaks compilation (you must edit source). An ABI change breaks linking/loading (already-compiled code crashes or misbehaves). Adding a member to a class is ABI-breaking (the layout changed) even if the API looks compatible. The C++ ABI is fragile: changing a class's private members, adding a virtual function, changing inline function bodies in a header — all can break ABI silently. This is why mixing a library built against one version of a dependency with a program built against another causes "it links but crashes." The C ABI is far more stable, which is why stable plugin interfaces and FFI boundaries are usually C, not C++.

12Symbol visibility & name mangling

Name mangling, fully

C++ supports overloading: add(int, int) and add(double, double) coexist. But the linker works with flat symbol strings and has no concept of types. The compiler bridges this by mangling: encoding the full signature into a unique symbol name.

$ cat demo.cpp
namespace geo { struct Point { double dist() const; }; }
int    add(int, int);
double add(double, double);

$ g++ -c demo.cpp && nm -C demo.o   # -C demangles
# mangled            demangled (via c++filt / nm -C)
# _Z3addii        -> add(int, int)
# _Z3adddd        -> add(double, double)
# _ZNK3geo5Point4distEv -> geo::Point::dist() const

Consequences of mangling:

  • Overloaded functions get distinct symbols — that's the point.
  • A declaration/definition signature mismatch produces different mangled names → "undefined reference," even though the names "look the same."
  • Mangling schemes differ between compilers (and historically between versions), so C++ libraries built with incompatible compilers can't link against each other. The Itanium C++ ABI standardized this for GCC/Clang; MSVC mangles differently.

extern "C" — the FFI escape hatch

To expose a C++ function with an unmangled, stable name (so C code, Python ctypes, or another language can call it), wrap it in extern "C":

extern "C" {
  int my_api_add(int a, int b) { return a + b; }
}
// Symbol is plain "my_api_add", not mangled. No overloading allowed.
// This is THE mechanism for stable C ABI plugin interfaces and FFI.

Symbol visibility — controlling the public surface

By default on Linux/ELF, every external symbol in a shared library is exported and visible to anything that loads it. This is usually wrong: it bloats the dynamic symbol table (slow load), prevents the linker from optimizing, leaks implementation details, and invites symbol clashes between libraries.

The fix: hide everything by default, explicitly export the public API.

# Compile with hidden default visibility
$ g++ -fvisibility=hidden -fvisibility-inlines-hidden -fPIC -c lib.cpp
// Mark the public API explicitly with an export macro
#if defined(_WIN32)
  #define MYLIB_API __declspec(dllexport)
#else
  #define MYLIB_API __attribute__((visibility("default")))
#endif

MYLIB_API int public_function(int x);   // exported
int internal_helper(int x);             // hidden — not in the dynamic symbol table
Hidden visibility is the professional default

Building shared libraries with -fvisibility=hidden and an explicit export macro is the standard for any serious C++ library. It gives faster loading, smaller symbol tables, real encapsulation, and protection against symbol collisions when multiple libraries are loaded into one process — a real problem in Python, where dozens of extension modules share one address space. On Windows this is mandatory anyway (nothing is exported without __declspec(dllexport)), so a cross-platform export macro is unavoidable.

Inspecting exported symbols

# List dynamic symbols a shared library exports
$ nm -D --defined-only libmylib.so
$ readelf --dyn-syms libmylib.so

# What undefined symbols does it still need from elsewhere?
$ nm -D -u libmylib.so

13The dynamic loader: rpath & friends

At program startup, before main() runs, the dynamic loader (ld.so / ld-linux.so) finds and maps every shared library the program needs. Where it looks is a precise, ordered search — and getting it wrong is the single most common deployment failure for native code.

The loader's search order (Linux)

  1. DT_RPATH in the executable (deprecated, but checked first if present and no RUNPATH).
  2. LD_LIBRARY_PATH environment variable — a colon-separated list of directories.
  3. DT_RUNPATH in the executable (the modern rpath; checked after LD_LIBRARY_PATH).
  4. The ldconfig cache (/etc/ld.so.cache, built from /etc/ld.so.conf and standard dirs).
  5. Default system directories: /lib, /usr/lib, /lib64, /usr/lib64.

Inspecting dependencies and resolution

# What shared libraries does this binary need, and where does the loader find each?
$ ldd ./myapp
        linux-vdso.so.1 (0x00007fff...)
        libmylib.so => not found              # <-- the problem, right here
        libstdc++.so.6 => /usr/lib/x86_64-linux-gnu/libstdc++.so.6
        libc.so.6 => /usr/lib/x86_64-linux-gnu/libc.so.6

# What SONAMEs are recorded in the binary as needed?
$ readelf -d ./myapp | grep NEEDED
 0x0000000000000001 (NEEDED)  Shared library: [libmylib.so]
 0x0000000000000001 (NEEDED)  Shared library: [libstdc++.so.6]

# What rpath/runpath is baked in?
$ readelf -d ./myapp | grep -E 'RPATH|RUNPATH'
 0x000000000000001d (RUNPATH)  Library runpath: [$ORIGIN/../lib]

The four ways to make a library findable

MechanismWhen to use
LD_LIBRARY_PATHQuick local testing only. Fragile, global, affects all programs — never the production answer.
RUNPATH / rpath (-Wl,-rpath)Baked into the binary at link time. The right answer for relocatable app bundles. Use $ORIGIN for paths relative to the executable.
System install + ldconfigInstall the .so into a standard dir (or one in /etc/ld.so.conf.d/) and run ldconfig. The right answer for system-wide libraries.
Static linkingSidestep the problem entirely — no .so to find (§10).

The $ORIGIN trick for relocatable bundles

The professional way to ship an application with bundled libraries is rpath relative to the executable, so the bundle works wherever it's unpacked:

# Link with rpath = "the lib dir next to wherever this binary ends up"
$ g++ main.o -L. -lmylib -Wl,-rpath,'$ORIGIN/../lib' -o myapp
#                          ^ note single quotes — $ORIGIN must reach the linker literally

# Now this layout is fully relocatable:
#   myapp/bin/myapp           <- finds libs via $ORIGIN/../lib
#   myapp/lib/libmylib.so

This is exactly how Python wheels, AppImages, and most bundled native applications work. The auditwheel tool (§41) rewrites rpaths to $ORIGIN when repairing wheels.

dlopen — loading libraries at runtime by choice

Beyond automatic loading, a program can load a shared library explicitly with dlopen(), look up symbols with dlsym(), and unload with dlclose(). This is the mechanism behind plugin systems — and behind how Python imports C extension modules (CPython dlopens your .so and calls its init function).

#include <dlfcn.h>

void* handle = dlopen("libplugin.so", RTLD_LAZY);
if (!handle) { /* dlerror() explains why */ }

using fn_t = int(*)(int);
auto fn = reinterpret_cast<fn_t>(dlsym(handle, "plugin_entry"));
int result = fn(42);

dlclose(handle);
The #1 native-code deployment failure

"Works on my machine, fails on the server with cannot open shared object file" is almost always a loader-path problem: the .so exists somewhere on your dev box (often pulled in via LD_LIBRARY_PATH you forgot you set) but isn't findable on the target. The fix is never "set LD_LIBRARY_PATH in production" — it's bake an rpath, install to a standard location, or static-link. Diagnose with ldd on the target; the "not found" line points straight at the culprit.

14Link order & the one-pass linker

One of the most surprising linker behaviors — and a frequent source of "but the library is right there!" errors — is that link order matters. The traditional Unix linker is essentially single-pass and left-to-right.

The one-pass model

The linker processes its inputs left to right, maintaining a set of currently-undefined symbols. When it reaches a static library (.a), it pulls in only those member objects that satisfy currently-pending undefined symbols — and then moves on. If a later object introduces a new dependency on that library, it's too late: the linker already passed it.

# WRONG: library before the object that needs it
$ g++ -lmylib main.o -o myapp
main.o: undefined reference to `add'
# When the linker saw -lmylib, NOTHING was undefined yet, so it took nothing.
# Then main.o introduced "add" as undefined — but libmylib was already passed.

# RIGHT: the object that NEEDS the symbol comes BEFORE the library that provides it
$ g++ main.o -lmylib -o myapp     # works

Put libraries AFTER the objects and libraries that depend on them. The rule: dependencies flow left-to-right. If A needs B, A comes first. The final link line reads like a topological sort of the dependency graph: your own objects, then the libraries they need, then the libraries those need, ending with the most fundamental (libc).

Circular dependencies between static libraries

If libA.a needs a symbol from libB.a AND libB.a needs a symbol from libA.a, no single ordering works in one pass. Three fixes:

# Option 1: repeat a library so the linker revisits it
$ g++ main.o -lA -lB -lA -o myapp

# Option 2: group them so the linker iterates until stable (GNU ld)
$ g++ main.o -Wl,--start-group -lA -lB -Wl,--end-group -o myapp

# Option 3 (best): don't have circular library dependencies.
# It usually signals a design problem — merge the libraries or break the cycle.

Why shared libraries are less order-sensitive

Shared libraries record their own dependencies (the NEEDED entries) and the loader resolves everything at load time, so the strict left-to-right rule is relaxed for .so files. Most modern linkers are also more forgiving than the classic model. But static libraries still follow the one-pass rule, and link order remains a real source of confusion when mixing static archives.

--as-needed and over-linking

By default, listing -lfoo records a dependency on libfoo.so even if your program uses nothing from it ("over-linking"). This bloats the dependency list and slows loading. -Wl,--as-needed tells the linker to only record dependencies actually used:

$ g++ main.o -Wl,--as-needed -lfoo -lbar -o myapp
# libfoo/libbar only recorded as NEEDED if a symbol from them is actually used

Almost every linker error is one of a handful of patterns. Here's the decoder.

The universal linker-debugging recipe

Q: Is the error at LINK time or RUN time?
   |
   +-- LINK time ("undefined reference", "multiple definition")
   |     1. c++filt / nm -C the symbol to read it human-readably
   |     2. nm -C each object/library: who defines it? who needs it (U)?
   |     3. No definer anywhere  -> missing source/library in the link
   |     4. Definer exists but not found -> link ORDER (§14) or missing -l
   |     5. Two definers -> ODR violation, definition in a header (§03)
   |
   +-- RUN time ("cannot open shared object", "undefined symbol")
         1. ldd ./binary  -> which .so is "not found"?
         2. Found but undefined symbol -> ABI/version mismatch
         3. readelf -d  -> check NEEDED entries and RUNPATH
         4. Fix the loader path (rpath/ldconfig) or align versions

A worked example: the classic C/C++ mismatch

# A C library header without extern "C", included from C++
$ cat clib.h
int c_function(int);     # compiled by a C compiler -> symbol "c_function"

$ cat main.cpp
#include "clib.h"        # C++ sees this, mangles to "_Z10c_functioni"
int main(){ return c_function(1); }

$ gcc -c clib.c          # nm clib.o: T c_function
$ g++ -c main.cpp        # nm main.o: U _Z10c_functioni  <-- MANGLED, won't match
$ g++ main.o clib.o -o app
main.o: undefined reference to `c_function(int)'

# FIX: guard the C header so C++ sees extern "C"
$ cat clib.h
#ifdef __cplusplus
extern "C" {
#endif
int c_function(int);
#ifdef __cplusplus
}
#endif
# Now C++ does NOT mangle it; nm main.o shows U c_function; links cleanly.

Every linker error names a symbol or a file. Demangle the symbol, find who defines it and who needs it, and the fix names itself.

Part IV — Make

The original build tool.

Make is fifty years old, ubiquitous, and still the substrate beneath much of what you'll touch — CMake's default output is a Makefile. Understanding Make's model (a dependency DAG with timestamp-based staleness) clarifies every build system that followed, because they are all reactions to Make's strengths and limits.

16Make's model: the DAG

Make's entire conceptual model fits in one sentence: a Makefile declares a graph of targets, each with prerequisites and a recipe; Make rebuilds a target whenever any prerequisite is newer than the target.

The anatomy of a rule

target: prerequisite1 prerequisite2
	recipe-command-1
	recipe-command-2
# ^ recipes MUST be indented with a TAB, not spaces. The #1 Make beginner error.
  • target — usually a file to produce (e.g., main.o), sometimes a phony label (e.g., clean).
  • prerequisites — files the target depends on. If any is newer than the target, the target is stale.
  • recipe — shell commands that produce the target. Run only when the target is stale or missing.

A complete minimal Makefile

# The final program depends on two object files
myapp: main.o utils.o
	g++ main.o utils.o -o myapp

# Each object depends on its source AND the headers it includes
main.o: main.cpp utils.h
	g++ -std=c++20 -c main.cpp -o main.o

utils.o: utils.cpp utils.h
	g++ -std=c++20 -c utils.cpp -o utils.o

# A phony target (not a real file) for cleanup
.PHONY: clean
clean:
	rm -f myapp main.o utils.o

How Make traverses the graph

Run make myapp and Make works backwards from the goal:

goal: myapp | +-- needs main.o ----+-- needs main.cpp (leaf — is it newer than main.o?) | +-- needs utils.h (leaf — is it newer than main.o?) | +-- needs utils.o ---+-- needs utils.cpp (leaf — is it newer than utils.o?) +-- needs utils.h (leaf — is it newer than utils.o?) Make's logic, depth-first: 1. To build myapp, first ensure main.o and utils.o are up to date. 2. For main.o: is main.cpp or utils.h newer than main.o? If yes, run recipe. 3. For utils.o: same check. 4. If either .o was rebuilt (or myapp is missing/older), relink myapp. 5. If nothing changed, Make prints "nothing to be done" and does zero work.

The timestamp model and its one big flaw

Make decides staleness purely by file modification time (mtime): a target is stale if any prerequisite has a more recent mtime. This is simple and fast, but:

  • It can't tell if a file's contents actually changed — touching a file (updating mtime without changing content) triggers a rebuild. Content-hash-based systems (Bazel, §44) avoid this.
  • Clock skew across machines or filesystems breaks it — a file from the future is always "newer."
  • It only knows the dependencies you declared. If main.cpp includes config.h but your rule didn't list config.h as a prerequisite, changing config.h won't trigger a rebuild — a stale-build bug. Solving this is §18.
The TAB rule

Recipe lines must begin with a literal TAB character, not spaces. This is the single most common Make error, producing the cryptic Makefile:5: *** missing separator. Stop. Configure your editor to keep tabs in Makefiles. Many a lost hour traces to spaces-vs-tabs here.

17Makefile syntax in depth

Real Makefiles use variables, automatic variables, and pattern rules to avoid the repetition of the minimal example. These features are where Make becomes powerful — and cryptic.

Variables and the two assignment operators

# Recursive assignment (=): expanded each time it's USED (lazy, can cause surprises)
CXX = g++

# Simple assignment (:=): expanded ONCE, at definition (eager, predictable — prefer this)
CXXFLAGS := -std=c++20 -O2 -Wall -Wextra
INCLUDES := -Iinclude

# Append
CXXFLAGS += -g

# Conditional assignment (?=): set only if not already defined
BUILD_DIR ?= build

# Reference a variable with $(NAME) or ${NAME}
$(CXX) $(CXXFLAGS) $(INCLUDES) -c main.cpp
Prefer := over =

Recursive (=) variables are re-expanded on every use, which can cause infinite loops (X = $(X) foo) and confusing late-binding behavior. Simple (:=) variables expand once and behave like variables in normal languages. Use := by default; reach for = only when you specifically need deferred expansion.

Automatic variables — the cryptic but essential shorthands

Inside a recipe, Make provides automatic variables referring to the current rule's targets and prerequisites. These are what make pattern rules possible.

VariableMeans
$@The target name.
$<The first prerequisite.
$^All prerequisites (space-separated, duplicates removed).
$?Prerequisites newer than the target.
$*The stem matched by a pattern rule's %.
$(@D) / $(@F)Directory part / file part of the target.

Pattern rules — one rule for all object files

Instead of writing a rule per object file, a pattern rule with % handles all of them:

CXX      := g++
CXXFLAGS := -std=c++20 -O2 -Wall -Wextra -Iinclude

# Gather sources and derive object names
SRCS := $(wildcard src/*.cpp)
OBJS := $(SRCS:src/%.cpp=build/%.o)
#       ^ substitution reference: src/foo.cpp -> build/foo.o

# Link rule
myapp: $(OBJS)
	$(CXX) $^ -o $@
#       ^all objs  ^target

# Pattern rule: how to build ANY build/%.o from the matching src/%.cpp
build/%.o: src/%.cpp | build
	$(CXX) $(CXXFLAGS) -c $< -o $@
#                        ^first prereq (the .cpp)  ^target (the .o)

# Order-only prerequisite (after |): ensure build/ exists, but don't
# rebuild objects just because the directory's mtime changed
build:
	mkdir -p build

.PHONY: clean
clean:
	rm -rf build myapp

Useful functions

$(wildcard src/*.cpp)              # glob files
$(patsubst %.cpp,%.o,$(SRCS))      # pattern substitution
$(filter %.cpp,$(FILES))           # keep matching
$(filter-out main.cpp,$(SRCS))     # remove matching
$(notdir path/to/file.cpp)         # -> file.cpp
$(basename file.cpp)               # -> file
$(addprefix build/,$(NAMES))       # prepend to each
$(shell git rev-parse --short HEAD) # run a shell command
$(foreach v,$(LIST),$(v).o)        # iterate

18Real-world patterns & auto-deps

The pattern-rule Makefile in §17 has a lurking bug: it doesn't track header dependencies. If main.cpp includes utils.h and you only declared main.cpp as the prerequisite, editing utils.h won't rebuild main.o — a stale-build bug that produces baffling "I changed the header but nothing happened" behavior. The fix is automatic dependency generation.

The -MMD trick

GCC and Clang can emit, as a side effect of compilation, a tiny Makefile fragment listing every header the source actually included. We then include those fragments back into our Makefile.

CXX      := g++
CXXFLAGS := -std=c++20 -O2 -Wall -Wextra -Iinclude
DEPFLAGS := -MMD -MP
# -MMD : write a .d dependency file alongside the .o (user headers only)
# -MP  : add phony targets for headers, so deleting a header doesn't break the build

SRCS := $(wildcard src/*.cpp)
OBJS := $(SRCS:src/%.cpp=build/%.o)
DEPS := $(OBJS:.o=.d)

myapp: $(OBJS)
	$(CXX) $^ -o $@

build/%.o: src/%.cpp | build
	$(CXX) $(CXXFLAGS) $(DEPFLAGS) -c $< -o $@

build:
	mkdir -p build

# Pull in the auto-generated dependency files. The leading "-" means
# "don't error if they don't exist yet" (first build).
-include $(DEPS)

.PHONY: clean
clean:
	rm -rf build myapp

Now, the first time main.cpp compiles, GCC writes build/main.d containing something like:

build/main.o: src/main.cpp include/utils.h include/config.h

On subsequent runs, -include pulls this in, so Make knows main.o depends on those headers and rebuilds correctly when any of them changes. This is the canonical correct Make pattern for C/C++ — and it's exactly what CMake generates for you automatically, which is one reason people stop hand-writing Makefiles.

Parallelism

# Run up to N recipe jobs in parallel — essential for fast C++ builds
$ make -j8           # 8 parallel jobs
$ make -j$(nproc)    # one per CPU core
$ make -j            # unlimited (careful — can exhaust RAM on big projects)

Make parallelizes correctly only if your dependency graph is correct. Missing or wrong dependencies cause race conditions under -j that don't appear in serial builds — another reason precise auto-deps matter.

Recursive Make and its problems

A common (and problematic) pattern is "recursive Make": a top-level Makefile that invokes make in each subdirectory. The classic paper "Recursive Make Considered Harmful" explains why: each sub-make sees only its own piece of the graph, so cross-directory dependencies are invisible, parallelism is fragmented, and incorrect builds result. The modern guidance is "non-recursive Make" (one Makefile that includes fragments from subdirectories, building the whole graph at once) — but at that complexity, most projects switch to CMake or Meson, which solve the multi-directory problem properly.

When hand-written Make stops scaling

Make is excellent for small-to-medium single-directory projects, simple task automation, and as a thin wrapper around other tools. It strains when you need: multi-platform support (Windows has no native Make and different compiler flags), dependency discovery (finding installed libraries), IDE integration, or multi-directory builds with shared dependencies. At that point you want a meta-build system that generates Makefiles (or Ninja files) — which is exactly CMake's job (Part V).

19Make's limits

Understanding why Make wasn't enough motivates everything in Part V. Make's limitations are not bugs — they're consequences of its 1976 design meeting different needs than modern cross-platform C++.

LimitationConsequence
No portability layerMakefiles encode compiler flags directly (-O2, -std=c++20). These differ on MSVC (/O2, /std:c++20). A Makefile is effectively platform-specific. Cross-platform builds need per-platform branches or a generator.
No dependency discoveryMake can't find where Boost or OpenSSL is installed. You hardcode paths or write fragile shell probes. CMake's find_package solves exactly this.
Timestamp-only stalenessNo content hashing; touch triggers rebuilds; clock skew breaks correctness.
Manual dependency trackingYou must wire up -MMD yourself; getting it wrong gives stale builds.
No IDE / tooling integrationNo way to generate Visual Studio / Xcode / compile_commands.json without bolt-ons.
String-based, footgun-heavy languageTabs, recursive variables, shell-per-line, escaping rules — Make's language is notoriously error-prone.
Poor at large parallel/distributed buildsNinja was built specifically to be a faster, leaner build executor than Make for generated builds.

The division of labor that followed

The modern answer splits Make's job in two:

  • A meta-build system (CMake, Meson) handles configuration: detecting the platform, finding dependencies, expressing the project portably, and generating low-level build files.
  • A build executor (Ninja, or still Make) consumes those generated files and runs the commands fast and correctly.

CMake + Ninja is the dominant modern C++ stack: CMake configures and generates build.ninja; Ninja executes it with minimal overhead and maximal parallelism. Make survives as a build executor (CMake can target it) and thrives as a general task runner — but for building C++ itself, it has mostly become an implementation detail beneath CMake.

Make taught everyone the dependency-graph model. Its successors kept the model and replaced the language, the portability story, and the dependency-discovery gap.

Part V — CMake

The meta-build standard.

CMake is the de facto standard for C++ build configuration. It is also widely misunderstood — most CMake people write is "old-style" CMake that fights the tool. This part teaches the modern, target-centric model that makes CMake tractable, then goes deep on dependencies, installing, cross-compilation, and the anti-patterns to unlearn.

20What CMake is: a generator

The single most important fact about CMake, the one that explains all its design: CMake does not build your code. CMake generates build files for another tool, which then builds your code.

CMakeLists.txt (you write this — portable project description) | | cmake -S . -B build -G Ninja [CONFIGURE + GENERATE phase] | - detect compiler, platform, features | - find dependencies | - evaluate your CMakeLists.txt logic v build/build.ninja (generated — platform-specific build instructions) | | cmake --build build [BUILD phase] | - invokes Ninja (or Make, or MSBuild, or Xcode) v artifacts (your executables and libraries)

CMake is a translator. You describe your project once, portably; CMake generates native build files for whatever toolchain is present: build.ninja for Ninja, Makefile for Make, .vcxproj for Visual Studio, .xcodeproj for Xcode. The same CMakeLists.txt builds on Linux with GCC, macOS with Clang, and Windows with MSVC.

The standard workflow

# Configure: read CMakeLists.txt, generate build files into build/ (out-of-source)
$ cmake -S . -B build -G Ninja -DCMAKE_BUILD_TYPE=Release

# Build: invoke the underlying tool
$ cmake --build build -j8

# Install (optional)
$ cmake --install build --prefix /usr/local

# Run tests (if any)
$ ctest --test-dir build

The -S/-B split keeps build artifacts out of your source tree (an "out-of-source build") — always do this; in-source builds litter your source with generated files and are hard to clean.

Generators

Generator (-G)Produces
Ninjabuild.ninja. Fastest. The recommended default everywhere it's available.
"Unix Makefiles"Makefiles. The traditional default on Unix.
"Ninja Multi-Config"Ninja with multiple build types in one tree (Debug + Release without reconfiguring).
"Visual Studio 17 2022"MSBuild .sln/.vcxproj. The Windows IDE default.

Use Ninja unless you have a reason not to

cmake -G Ninja gives the fastest configure-to-build cycle, the best parallelism, and produces compile_commands.json (with -DCMAKE_EXPORT_COMPILE_COMMANDS=ON) that powers clangd, IDE autocomplete, and static analyzers. The Make generator is a fine fallback; the IDE generators are for when you genuinely want to develop in that IDE.

21The language & two-phase model

CMake has its own scripting language. It's quirky — everything is fundamentally a string or a list of strings — but you only need a small subset for 95% of work.

The minimal CMakeLists.txt

cmake_minimum_required(VERSION 3.20)
project(MyApp VERSION 1.0 LANGUAGES CXX)

set(CMAKE_CXX_STANDARD 20)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

add_executable(myapp src/main.cpp src/utils.cpp)
target_include_directories(myapp PRIVATE include)

That's a complete, working, cross-platform build for a small program. Four ideas: declare the minimum version, declare the project, declare the target (add_executable), and tell the target where its headers are.

Language essentials

# Variables are strings; set with set(), reference with ${}
set(MY_VAR "hello")
message(STATUS "value is ${MY_VAR}")

# Lists are semicolon-separated strings
set(SOURCES a.cpp b.cpp c.cpp)   # stored as "a.cpp;b.cpp;c.cpp"

# Conditionals
if(WIN32)
  message(STATUS "Building on Windows")
elseif(APPLE)
  message(STATUS "Building on macOS")
endif()

# Cache variables: persist across runs, settable from the command line with -D
option(BUILD_TESTS "Build the test suite" OFF)
set(MY_PATH "" CACHE PATH "Path to something")
# override at configure time:  cmake -B build -DBUILD_TESTS=ON

The two phases — configure vs build time

This distinction trips everyone up. CMake code runs at configure time (when you run cmake), not at build time. So:

  • if(WIN32), message(), find_package(), variable assignments — all evaluated once, when CMake runs, before any compilation.
  • The result is baked into the generated build files. Changing the build files' behavior based on something only known at build time requires generator expressions (§28), which are evaluated later.
The classic configure-vs-build confusion

People write if(CMAKE_BUILD_TYPE STREQUAL "Debug") and expect it to switch behavior per-build in a multi-config generator (Visual Studio, Ninja Multi-Config) — but those generators don't have a single build type at configure time; they choose it at build time. The if() evaluates to empty and the branch never fires. The fix is a generator expression like $<CONFIG:Debug> (§28), which is evaluated per-configuration at build time. This is the most common source of "my CMake conditional doesn't work" confusion.

22Targets: the modern model

The dividing line between "modern CMake" (good) and "old CMake" (painful) is: everything is a property of a target, not a global setting.

What a target is

A target is a buildable thing — an executable, a library — and a bag of properties: its sources, its include directories, its compile definitions, its compile options, the libraries it links, the language standard it requires. Modern CMake attaches everything to targets and lets dependencies propagate those properties automatically.

# Define a library target
add_library(mathlib
  src/vector.cpp
  src/matrix.cpp
)

# Attach properties TO THE TARGET (not globally)
target_include_directories(mathlib
  PUBLIC  include          # consumers need this too
  PRIVATE src              # only mathlib's own build needs this
)
target_compile_features(mathlib PUBLIC cxx_std_20)
target_compile_definitions(mathlib PRIVATE MATHLIB_INTERNAL=1)

# Define an executable that USES the library
add_executable(myapp src/main.cpp)

# Link it — this ONE line propagates mathlib's PUBLIC include dirs,
# compile features, and definitions to myapp automatically
target_link_libraries(myapp PRIVATE mathlib)

The power: when myapp links mathlib, it automatically inherits mathlib's PUBLIC include directories and compile features. You don't repeat them. The dependency graph carries the configuration. This is "usage requirements," covered next.

The old way, and why it's bad

Old CMake (avoid)

Global commands: include_directories(), add_definitions(), link_directories(), setting CMAKE_CXX_FLAGS by hand. These affect everything in the directory and below, indiscriminately. They don't propagate across targets correctly, leak settings to unrelated targets, and make large projects an entangled mess.

Modern CMake (do this)

Target commands: target_include_directories(), target_compile_definitions(), target_compile_options(), target_link_libraries(). Each attaches a property to one target with explicit visibility (PUBLIC/PRIVATE/INTERFACE). Properties propagate along the dependency graph exactly as far as you specify.

Library types

add_library(foo STATIC src/foo.cpp)   # libfoo.a / foo.lib
add_library(foo SHARED src/foo.cpp)   # libfoo.so / foo.dll
add_library(foo src/foo.cpp)          # STATIC or SHARED per BUILD_SHARED_LIBS

# Header-only library: no sources to compile, only usage requirements
add_library(headeronly INTERFACE)
target_include_directories(headeronly INTERFACE include)
target_compile_features(headeronly INTERFACE cxx_std_20)

# Object library: compiled once, reused by multiple targets without an archive
add_library(objs OBJECT src/a.cpp src/b.cpp)

23Usage requirements: PUBLIC/PRIVATE/INTERFACE

The three keywords PUBLIC, PRIVATE, and INTERFACE control how a target's properties propagate to things that link it. This is the conceptual core of modern CMake, and once it clicks, large projects become manageable.

The mental model

For any property you attach to a library, ask two questions: does building this library need it? Does using this library need it?

KeywordApplies to library's own build?
PRIVATEYes, but NOT propagated to consumers. "I need this to build, but my users don't."
INTERFACENo, but propagated to consumers. "I don't need this, but my users do." (Header-only libs.)
PUBLICBoth. "I need this AND my users need it." (= PRIVATE + INTERFACE.)

The canonical example

add_library(network src/socket.cpp src/http.cpp)

target_include_directories(network
  PUBLIC  include          # public headers — consumers #include <network/http.h>
  PRIVATE src              # internal headers — only network's .cpp files use them
)

target_link_libraries(network
  PUBLIC  OpenSSL::SSL      # network's public headers expose OpenSSL types -> consumers need it
  PRIVATE ZLIB::ZLIB        # zlib used only inside .cpp, not in public headers -> consumers don't
)

Now any target that does target_link_libraries(myapp PRIVATE network) automatically gets network's include directory and links OpenSSL::SSL — but does not see network's private src dir or link ZLIB. The visibility keywords encode exactly the right propagation.

The rule of thumb: if a dependency appears in your public headers, it's PUBLIC. If it's used only in your .cpp files, it's PRIVATE. A type from OpenSSL in a function signature in your header → PUBLIC (consumers compiling against your header need OpenSSL's headers too). A zlib call buried in an implementation file → PRIVATE (consumers never see it). Getting this right means consumers just target_link_libraries against you and everything they transitively need flows automatically.

Why this scales

In a project with 50 libraries, each declaring its usage requirements correctly, you never manually wire up transitive dependencies. Link the top-level executable against the few libraries it directly uses, and CMake walks the graph, collecting every PUBLIC/INTERFACE include dir, definition, and linked library along the way. Old-style CMake, where everything is global, collapses under this complexity; target-based CMake handles it by construction.

24Finding dependencies

The thing Make couldn't do: locate an installed library and tell your build where its headers and binaries are. CMake's find_package is the answer, and it has two distinct modes that confuse everyone.

find_package, two modes

# The modern, target-based usage
find_package(OpenSSL 3.0 REQUIRED)
target_link_libraries(myapp PRIVATE OpenSSL::SSL OpenSSL::Crypto)

find_package(Threads REQUIRED)
target_link_libraries(myapp PRIVATE Threads::Threads)

find_package(fmt REQUIRED)
target_link_libraries(myapp PRIVATE fmt::fmt)
ModeHow it works
Config mode (preferred)Looks for a <Package>Config.cmake file installed by the package itself. This file defines imported targets like fmt::fmt with all usage requirements baked in. Modern, accurate, the package author maintains it.
Module mode (legacy/fallback)Looks for a Find<Package>.cmake script (shipped with CMake or your project) that hunts for headers and libs heuristically and sets variables like OpenSSL_LIBRARIES. Used for packages that don't ship their own config files.

CMake tries config mode first, then module mode. When a package provides imported targets (fmt::fmt, OpenSSL::SSL), use them — they carry the include dirs and link flags automatically. Avoid the old variable style (${OpenSSL_LIBRARIES}, ${OpenSSL_INCLUDE_DIR}) when targets are available.

Imported targets are the goal

Variable style (legacy)
find_package(OpenSSL REQUIRED)
target_include_directories(myapp
  PRIVATE ${OPENSSL_INCLUDE_DIR})
target_link_libraries(myapp
  PRIVATE ${OPENSSL_LIBRARIES})

You manually wire includes and libs; error-prone, no transitive propagation.

Target style (modern)
find_package(OpenSSL REQUIRED)
target_link_libraries(myapp
  PRIVATE OpenSSL::SSL)

One line. The imported target carries its own include dirs, compile flags, and transitive deps.

pkg-config bridge

For libraries that ship a .pc file but no CMake config, the PkgConfig module bridges them:

find_package(PkgConfig REQUIRED)
pkg_check_modules(LIBAV REQUIRED IMPORTED_TARGET libavcodec libavformat)
target_link_libraries(myapp PRIVATE PkgConfig::LIBAV)
# IMPORTED_TARGET gives you a proper target instead of bare variables

Helping find_package find things

# Point CMake at a non-standard install prefix
$ cmake -B build -DCMAKE_PREFIX_PATH=/opt/mylib;/opt/other

# Or per-package hint
$ cmake -B build -DOpenSSL_ROOT=/opt/openssl-3

25FetchContent & dependency management

Beyond finding installed dependencies, modern projects often want to fetch and build dependencies as part of their own build. CMake offers two mechanisms, and the broader ecosystem offers package managers.

FetchContent — download and build at configure time

The built-in, increasingly popular approach. Declares a dependency by Git URL or archive; CMake downloads it and adds it to your build, so its targets become available as if they were yours.

include(FetchContent)

FetchContent_Declare(
  fmt
  GIT_REPOSITORY https://github.com/fmtlib/fmt.git
  GIT_TAG        10.2.1          # pin a tag/commit — never a branch
)
FetchContent_MakeAvailable(fmt)

# Now fmt::fmt is a target in your build, ready to link
target_link_libraries(myapp PRIVATE fmt::fmt)
  • Pros: zero external tooling; pin exact versions; works in CI without pre-installed deps; the dependency builds with your exact flags/toolchain (no ABI mismatch).
  • Cons: rebuilds dependencies from source (slow first build); can cause version conflicts if two deps want different versions of a third; no global cache by default.

ExternalProject — build at build time, fully isolated

The older mechanism. Builds a dependency as a separate sub-build at build time (not configure time), so its targets aren't directly usable; you reference its installed outputs. More isolation, more friction. FetchContent has largely superseded it for in-tree dependencies; ExternalProject still fits when you need a dependency built with a totally different toolchain or isolated entirely.

Package managers for C++

C++ historically lacked a standard package manager. Two now dominate, both integrate with CMake:

ManagerModel
vcpkg (Microsoft)Builds from source, caches binaries. A "manifest" (vcpkg.json) lists deps; a CMake toolchain file makes them findable via find_package. Huge catalog.
Conan (JFrog)Binary-first package manager with a Python-based recipe system. Generates CMake files; supports binary caching across machines. Strong in enterprise.
SpackHPC-focused; builds many variants/versions side by side. Common on supercomputers.
CPM.cmakeA thin wrapper over FetchContent adding a version cache. Lightweight, no external tool.
Choosing a dependency strategy

For a small project or one that must build anywhere with no setup: FetchContent (or CPM). For a team wanting fast builds with binary caching and a large curated catalog: vcpkg (manifest mode) or Conan. For HPC with many compiler/version combos: Spack. The key is to pick one and use it consistently — mixing system packages, FetchContent, and a package manager for the same dependency is how you get the ABI-mismatch crashes from §11.

26Toolchains, cross-compiling, presets

Two facilities that separate hobby CMake from production CMake: toolchain files (for controlling exactly which compiler and flags get used, especially when cross-compiling) and presets (for capturing reproducible configurations).

Toolchain files

A toolchain file is a small CMake script that tells CMake which compiler, linker, and target platform to use. It's passed at configure time and read before your project, so it can set the compiler. This is how cross-compilation works — building on an x86 host for an ARM target, say.

# arm-linux-toolchain.cmake
set(CMAKE_SYSTEM_NAME Linux)
set(CMAKE_SYSTEM_PROCESSOR aarch64)

set(CMAKE_C_COMPILER   aarch64-linux-gnu-gcc)
set(CMAKE_CXX_COMPILER aarch64-linux-gnu-g++)

# Where the target's headers/libs live (the sysroot)
set(CMAKE_SYSROOT /opt/aarch64-sysroot)

# Search for programs on the host, but libs/headers in the target sysroot
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
# Use it at configure time
$ cmake -B build -G Ninja \
    -DCMAKE_TOOLCHAIN_FILE=arm-linux-toolchain.cmake \
    -DCMAKE_BUILD_TYPE=Release

The CMAKE_FIND_ROOT_PATH_MODE_* settings are the subtle part of cross-compiling: you want find_package to find target libraries (in the sysroot), not the host's. Getting these wrong means linking your ARM binary against x86 host libraries — which fails confusingly. Package managers like vcpkg and Conan also deliver their integration via a toolchain file, which is why you sometimes can only pass one CMAKE_TOOLCHAIN_FILE (chain them if you need both).

Presets — reproducible configurations

CMakePresets.json (CMake 3.19+) captures configure/build/test settings in a checked-in file, so "how do I build this project" has one canonical answer instead of a wiki page of cmake -D... incantations.

// CMakePresets.json
{
  "version": 6,
  "configurePresets": [
    {
      "name": "default",
      "generator": "Ninja",
      "binaryDir": "${sourceDir}/build/${presetName}",
      "cacheVariables": {
        "CMAKE_BUILD_TYPE": "RelWithDebInfo",
        "CMAKE_EXPORT_COMPILE_COMMANDS": "ON"
      }
    },
    {
      "name": "asan",
      "inherits": "default",
      "cacheVariables": {
        "CMAKE_CXX_FLAGS": "-fsanitize=address,undefined -fno-omit-frame-pointer"
      }
    }
  ]
}
$ cmake --preset default        # configure using the named preset
$ cmake --build --preset default
$ cmake --preset asan           # a sanitizer build, one word
Presets are the modern onboarding story

Check in a CMakePresets.json and a new developer's first build is cmake --preset default && cmake --build --preset default — no tribal knowledge about which flags to pass. IDEs (VS Code, CLion, Visual Studio) read presets natively, so the same configurations populate the IDE's build menu. A personal CMakeUserPresets.json (gitignored) lets each developer add local presets without touching the shared file.

27Installing & exporting

Building a library is half the job; making it consumable by other projects is the other half. This is the chapter most tutorials skip, and the reason so many libraries are painful to depend on.

The install rules

install() commands describe what goes where when someone runs cmake --install. You install the binaries, the public headers, and — crucially — a CMake config file so others can find_package you.

include(GNUInstallDirs)   # gives CMAKE_INSTALL_LIBDIR, _INCLUDEDIR, _BINDIR

add_library(mathlib src/vector.cpp)
target_include_directories(mathlib PUBLIC
  "$<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>"
  "$<INSTALL_INTERFACE:${CMAKE_INSTALL_INCLUDEDIR}>"
)
# ^ the include dir differs between building in-tree and being installed —
   generator expressions (§28) pick the right one for each context

# Install the library binary, associating it with an export set
install(TARGETS mathlib
  EXPORT mathlibTargets
  LIBRARY DESTINATION ${CMAKE_INSTALL_LIBDIR}
  ARCHIVE DESTINATION ${CMAKE_INSTALL_LIBDIR}
  RUNTIME DESTINATION ${CMAKE_INSTALL_BINDIR}
)

# Install the public headers
install(DIRECTORY include/ DESTINATION ${CMAKE_INSTALL_INCLUDEDIR})

Exporting targets so find_package works

The magic that makes your library show up as an imported target (mathlib::mathlib) in someone else's find_package(mathlib): generate and install a config file.

# Install the export set as a *Targets.cmake file, namespaced
install(EXPORT mathlibTargets
  FILE mathlibTargets.cmake
  NAMESPACE mathlib::
  DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/mathlib
)

# Generate a version file and a config wrapper
include(CMakePackageConfigHelpers)
write_basic_package_version_file(
  mathlibConfigVersion.cmake
  VERSION ${PROJECT_VERSION}
  COMPATIBILITY SameMajorVersion
)
configure_package_config_file(
  Config.cmake.in mathlibConfig.cmake
  INSTALL_DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/mathlib
)
install(FILES
  ${CMAKE_CURRENT_BINARY_DIR}/mathlibConfig.cmake
  ${CMAKE_CURRENT_BINARY_DIR}/mathlibConfigVersion.cmake
  DESTINATION ${CMAKE_INSTALL_LIBDIR}/cmake/mathlib
)

Now a downstream project writes find_package(mathlib REQUIRED) and target_link_libraries(app PRIVATE mathlib::mathlib) — and inherits all your usage requirements. This is what "good CMake citizenship" means: your library is installable and findable with zero manual wiring on the consumer's side.

BUILD_INTERFACE vs INSTALL_INTERFACE

A library's include directory is different when building in its own tree (${SOURCE}/include) versus after installation (${prefix}/include). The $<BUILD_INTERFACE:...> / $<INSTALL_INTERFACE:...> generator expressions encode both so the right path is used in each context. Forgetting this is why some installed libraries export include paths pointing back into the original build tree — which breaks the moment that tree is deleted.

28Generator expressions & configs

Generator expressions (genexes) are the answer to "I need a value that depends on something not known until build time" — the build configuration, the target platform, the compiler. They have the cryptic syntax $<...> and are evaluated when build files are generated, after all the configure-time logic has run.

Why they exist

Recall the two-phase model (§21): ordinary CMake code runs once at configure time. But multi-config generators (Visual Studio, Ninja Multi-Config) decide Debug-vs-Release at build time, per build. An ordinary if(CMAKE_BUILD_TYPE STREQUAL Debug) can't see that choice. Genexes can, because they're evaluated later, per-configuration.

# Add a definition only in Debug builds — works in multi-config too
target_compile_definitions(myapp PRIVATE
  $<$<CONFIG:Debug>:DEBUG_LOGGING=1>
)

# Different optimization flags per compiler
target_compile_options(myapp PRIVATE
  $<$<CXX_COMPILER_ID:GNU,Clang>:-Wall -Wextra>
  $<$<CXX_COMPILER_ID:MSVC>:/W4>
)

# The BUILD/INSTALL_INTERFACE genexes from §27
target_include_directories(lib PUBLIC
  $<BUILD_INTERFACE:${CMAKE_CURRENT_SOURCE_DIR}/include>
  $<INSTALL_INTERFACE:include>
)

The common genex forms

ExpressionEvaluates to
$<CONFIG:Debug>1 if the build config is Debug, else 0.
$<CONFIG:Debug>:FLAGS> wrapped as $<$<CONFIG:Debug>:FLAGS>FLAGS if Debug, else nothing (conditional inclusion).
$<CXX_COMPILER_ID:GNU>1 if compiling with GCC.
$<PLATFORM_ID:Windows>1 on Windows.
$<TARGET_FILE:tgt>The full path to target tgt's output file.
$<BUILD_INTERFACE:x> / $<INSTALL_INTERFACE:x>x only when building in-tree / only when installed.

Build types, set properly

# Single-config generators (Ninja, Make): choose at configure time
$ cmake -B build -G Ninja -DCMAKE_BUILD_TYPE=RelWithDebInfo

# Multi-config generators (VS, Ninja Multi-Config): choose at build time
$ cmake -B build -G "Ninja Multi-Config"
$ cmake --build build --config Release    # pick here, not at configure
$ cmake --build build --config Debug      # same build tree, different config
Never set CMAKE_BUILD_TYPE from inside CMakeLists.txt unconditionally

Hardcoding set(CMAKE_BUILD_TYPE Release) in your CMakeLists overrides the user's choice and breaks multi-config generators (which ignore it). If you want a sensible default when the user didn't pick one, guard it: if(NOT CMAKE_BUILD_TYPE AND NOT CMAKE_CONFIGURATION_TYPES) set(CMAKE_BUILD_TYPE RelWithDebInfo) endif(). The build type is the user's decision, not the project's.

29Testing & packaging (CTest/CPack)

CTest — registering and running tests

CTest is CMake's test driver. You register tests with add_test; CTest runs them, reports pass/fail, supports parallelism, timeouts, and filtering. It doesn't care what a "test" is — any command that exits 0 (pass) or non-zero (fail) works, so it drives GoogleTest, Catch2, doctest, or a plain script equally.

enable_testing()

add_executable(unit_tests test/test_vector.cpp test/test_matrix.cpp)
target_link_libraries(unit_tests PRIVATE mathlib GTest::gtest_main)

# Register with CTest. With GoogleTest, this helper auto-discovers each TEST()
include(GoogleTest)
gtest_discover_tests(unit_tests)

# Or register a plain command as a test
add_test(NAME smoke COMMAND myapp --self-check)
set_tests_properties(smoke PROPERTIES TIMEOUT 30)
$ ctest --test-dir build -j8           # run all tests, 8 in parallel
$ ctest --test-dir build -R vector     # only tests matching "vector"
$ ctest --test-dir build --output-on-failure   # show output of failed tests
$ ctest --test-dir build --rerun-failed        # re-run only last failures

CPack — producing distributable packages

CPack turns your installed artifacts into a package: a .deb, .rpm, .tar.gz, a macOS .dmg, a Windows NSIS installer. It reuses your install() rules to decide what goes in.

set(CPACK_PACKAGE_NAME "mathlib")
set(CPACK_PACKAGE_VERSION ${PROJECT_VERSION})
set(CPACK_GENERATOR "TGZ;DEB")            # tarball + Debian package
set(CPACK_DEBIAN_PACKAGE_MAINTAINER "you@example.com")
include(CPack)
$ cmake --build build
$ cpack --config build/CPackConfig.cmake
# produces mathlib-1.0.0-Linux.tar.gz and mathlib_1.0.0_amd64.deb

Part VI — Python ↔ C++ bindings

Crossing the language line.

The reason most of this manual matters to an ML engineer: you have fast C++ or CUDA code and you want to call it from Python. This part covers how that bridge is built — from the raw CPython C-API up through pybind11, its modern successor nanobind, and the no-compile alternatives — and the GIL concerns that make threading across the boundary subtle.

31Why bind, and the landscape

Python is productive but slow for tight numerical loops. The universal pattern in scientific and ML software: write the hot path in C++/CUDA, expose it to Python, and get both productivity and speed. NumPy, PyTorch, TensorFlow, SciPy — all are thin-ish Python layers over large C/C++/CUDA cores, connected by binding code.

The fundamental mechanism

A Python extension module is a shared library (.so on Linux/macOS, .pyd on Windows) that exports a single initialization function with a special name. When you import mymodule, CPython dlopens the .so (§13) and calls PyInit_mymodule(), which registers the module's functions and types with the interpreter.

import mymodule | v CPython finds mymodule.so on sys.path | v dlopen(mymodule.so) [the dynamic loader, §13] | v call PyInit_mymodule() [the one required exported symbol] | v init function registers functions, classes, constants | with the interpreter as Python objects v module ready — Python calls now dispatch into your C++ code

Every binding tool — pybind11, nanobind, Cython, raw C-API — ultimately produces this: a .so with a PyInit_* function. They differ in how much boilerplate you write to get there and how the type conversions are expressed.

  • Binding an existing C library with a stable C ABI, no build step wanted → ctypes/cffi.
  • Binding C++ (classes, templates, overloads, STL) → pybind11 (mature) or nanobind (newer, leaner).
  • Accelerating Python code itself, or wrapping C with Python-like glue → Cython.
  • New project, C++17+, performance/size sensitive → nanobind.

32The CPython C-API foundation

You'll rarely write raw C-API code, but understanding it explains what pybind11 and nanobind do for you — and is essential when something goes wrong at the boundary.

Everything is a PyObject*

Every Python value — an int, a list, a function, your class instance — is, at the C level, a PyObject*: a pointer to a struct beginning with a reference count and a pointer to its type. The entire C-API is functions that take and return PyObject*.

// A trivial function exposed via the raw C-API
static PyObject* myadd(PyObject* self, PyObject* args) {
    int a, b;
    if (!PyArg_ParseTuple(args, "ii", &a, &b))   // parse Python args into C ints
        return NULL;                              // on error, return NULL (exception set)
    return PyLong_FromLong(a + b);                 // build a Python int from the result
}

static PyMethodDef Methods[] = {
    {"add", myadd, METH_VARARGS, "Add two ints"},
    {NULL, NULL, 0, NULL}
};
static PyModuleDef module = { PyModuleDef_HEAD_INIT, "mymod", NULL, -1, Methods };

PyMODINIT_FUNC PyInit_mymod(void) {        // the required init function
    return PyModule_Create(&module);
}

Even this trivial example shows the friction: manual argument parsing with format strings, manual object construction, manual error handling via NULL returns. Now imagine doing this for a class with methods, operator overloads, and STL containers. That boilerplate is exactly what pybind11 eliminates.

Reference counting and the GIL — the two things that bite

Two C-API realities that the binding libraries manage for you, and that you must understand when they leak through:

  • Reference counting: every PyObject* has a refcount. The C-API has rules about which functions return "new references" (you own it, must decref) vs "borrowed references" (you don't own it). Getting this wrong leaks memory or causes use-after-free crashes. pybind11/nanobind wrap objects in RAII handle types so refcounts are correct automatically.
  • The GIL (Global Interpreter Lock): only one thread may execute Python bytecode or touch PyObject*s at a time. Any C code that calls the Python API must hold the GIL; any long-running C computation that doesn't touch Python objects should release the GIL so other Python threads run. This is the crux of threading across the boundary (§37).
The C-API ties you to CPython internals

Code against the raw C-API depends on CPython's object layout and is sensitive to version changes and the GIL. This is why the ecosystem built abstractions on top (pybind11, nanobind) and why projects like the "stable ABI" / "limited API" and HPy exist — to decouple extensions from CPython internals so a single wheel works across Python versions. We return to this when discussing wheels and ABI in §41.

33pybind11 in depth

pybind11 is a header-only library that lets you expose C++ to Python with remarkably concise, declarative code. It infers type conversions, manages refcounts, translates exceptions, and supports the full richness of C++ (classes, inheritance, operators, STL, smart pointers).

The same add function, in pybind11

#include <pybind11/pybind11.h>
namespace py = pybind11;

int add(int a, int b) { return a + b; }

PYBIND11_MODULE(mymod, m) {            // defines PyInit_mymod for you
    m.doc() = "example module";
    m.def("add", &add, "Add two ints",
          py::arg("a"), py::arg("b"));   // named args, auto type conversion
}

Compare to the raw C-API version in §32: no argument parsing, no manual object construction, no NULL-return error handling. pybind11 sees int add(int, int) and generates all of it. Python ints convert to C++ ints and back automatically.

Binding a class

#include <pybind11/pybind11.h>
#include <pybind11/stl.h>      // auto-convert std::vector <-> list, etc.
namespace py = pybind11;

class Matrix {
public:
    Matrix(int rows, int cols);
    double at(int i, int j) const;
    void set(int i, int j, double v);
    int rows() const;
    std::vector<double> row(int i) const;
};

PYBIND11_MODULE(linalg, m) {
    py::class_<Matrix>(m, "Matrix")
        .def(py::init<int, int>(), py::arg("rows"), py::arg("cols"))
        .def("at", &Matrix::at)
        .def("set", &Matrix::set)
        .def("row", &Matrix::row)         // returns std::vector -> Python list
        .def_property_readonly("rows", &Matrix::rows)
        .def("__repr__", [](const Matrix& m) {
            return "<Matrix >";
        });
}

From Python this is a natural class: m = linalg.Matrix(3, 3); m.set(0, 0, 1.5); print(m.rows). The <pybind11/stl.h> header makes std::vector ↔ list, std::map ↔ dict, etc., conversions automatic.

The features that make it the standard

  • Automatic type conversion for primitives, strings, STL containers, and anything you teach it.
  • NumPy integration (pybind11/numpy.h): accept and return NumPy arrays, access buffers zero-copy — essential for ML.
  • Smart pointer support: std::shared_ptr, std::unique_ptr ownership transfer handled correctly.
  • Exception translation: C++ exceptions become Python exceptions automatically; you can map custom types.
  • Inheritance, virtual functions, even overriding C++ virtuals in Python (trampoline classes).
  • GIL management helpers: py::gil_scoped_release / py::gil_scoped_acquire (§37).

The NumPy buffer pattern (the ML workhorse)

#include <pybind11/numpy.h>

// Accept a NumPy array, operate on it in-place without copying
void scale(py::array_t<double> arr, double factor) {
    py::buffer_info buf = arr.request();
    double* ptr = static_cast<double*>(buf.ptr);
    py::gil_scoped_release nogil;          // release GIL for the hot loop
    for (py::ssize_t i = 0; i < buf.size; ++i)
        ptr[i] *= factor;                  // raw C++ speed, no Python overhead
}
Why pybind11 won

Before pybind11, binding C++ to Python meant SWIG (a separate code generator with its own interface files) or hand-written Boost.Python (heavyweight, slow to compile). pybind11 is header-only, needs no separate tool, produces small modules, and expresses bindings in plain modern C++. It became the default for new C++ projects almost immediately and powers a huge fraction of the scientific Python ecosystem.

34nanobind: the successor

nanobind is by Wenzel Jakob, the author of pybind11, built years later with the benefit of hindsight and a clean C++17 baseline. The API looks almost identical to pybind11 — deliberately — but the implementation is far leaner and faster.

The same module in nanobind

#include <nanobind/nanobind.h>
namespace nb = nanobind;

int add(int a, int b) { return a + b; }

Nearly identical to the pybind11 version — by design, so the migration is mechanical. The differences are under the hood.

35Cython, ctypes, cffi

Three approaches with different philosophies from the C++-first binding libraries. Knowing when each fits keeps you from reaching for a sledgehammer.

Cython — a Python-like compiled language

Cython is a superset of Python that compiles to C. You add type annotations to Python code, and Cython generates a C extension that runs much faster. It's also a clean way to wrap C libraries. The model is fundamentally different: you're not binding C++ from the C++ side, you're writing Python-ish code that compiles down.

# fib.pyx — compiled by Cython to C, then to a .so
def fib(int n):                 # typed parameter -> C-speed loop
    cdef int a = 0, b = 1, i
    for i in range(n):
        a, b = b, a + b
    return a

# Wrapping a C function:
cdef extern from "math.h":
    double sqrt(double x)

def c_sqrt(double x):
    return sqrt(x)

Cython shines when accelerating existing Python (incrementally add types to hot functions) and when the codebase is Python-centric. It's heavily used in scientific Python (scikit-learn, pandas internals). For wrapping a large modern C++ API, though, pybind11/nanobind are more natural — Cython's C++ support exists but is clunkier.

ctypes — call a shared library from pure Python, no compilation

ctypes is in the standard library. It loads a shared library at runtime and lets you call its C functions directly from Python by describing their signatures. No build step, no compiler — but you're responsible for getting the types exactly right, and it's a C ABI only (no C++ classes).

import ctypes

lib = ctypes.CDLL("./libmymath.so")        # dlopen the library
lib.add.argtypes = [ctypes.c_int, ctypes.c_int]  # declare signature
lib.add.restype  = ctypes.c_int
print(lib.add(2, 3))                       # calls the C function add()

Note: this only works if add is exported with an unmangled name — i.e., compiled as C or wrapped in extern "C" (§12). ctypes cannot call mangled C++ symbols.

cffi — a friendlier C FFI

cffi (C Foreign Function Interface) is a third-party library that's generally more robust than ctypes. You give it C declarations (literally copy-pasted from a header) and it builds the binding, in either an ABI mode (like ctypes, no compiler) or an API mode (compiles a small shim for safety and speed).

from cffi import FFI
ffi = FFI()
ffi.cdef("int add(int, int);")            # paste the C declaration
lib = ffi.dlopen("./libmymath.so")
print(lib.add(2, 3))

37The GIL, threading, free-threading

The Global Interpreter Lock is the single most important runtime concept when bridging C++ and Python, and the source of the most subtle bugs and the biggest performance opportunities.

What the GIL is

CPython's GIL is a mutex that allows only one thread to execute Python bytecode — or touch any PyObject* — at a time. It exists because CPython's reference counting isn't thread-safe; the GIL serializes access to make it safe cheaply. The consequence: pure-Python multithreading doesn't parallelize CPU work (threads take turns holding the GIL).

The two GIL operations in bindings

When your C++ code runs, it was called from Python, so it starts holding the GIL. Two moves matter:

  • Release the GIL around long C++ computations that don't touch Python objects. This lets other Python threads run in parallel while your C++ crunches numbers — the key to getting real multicore speedup from a C++ extension.
  • Re-acquire the GIL before touching any Python object again (creating return values, calling back into Python).
double heavy_compute(py::array_t<double> data) {
    auto buf = data.request();
    double* ptr = static_cast<double*>(buf.ptr);
    double result = 0;
    {
        py::gil_scoped_release nogil;        // GIL released for this block
        // Other Python threads run in parallel now.
        // MUST NOT touch any PyObject in here.
        for (py::ssize_t i = 0; i < buf.size; ++i)
            result += std::sqrt(ptr[i]);
    }   // GIL re-acquired here as nogil goes out of scope
    return result;                            // safe to build Python object now
}
The two GIL bugs

Touching Python objects without the GIL (e.g., after gil_scoped_release, calling a Python callback or creating a py::object) → data race, crash, or corruption. Holding the GIL during long C++ work → you've serialized everything; your "parallel" extension runs single-threaded and Python threads starve. The discipline: release the GIL for pure-C++ number crunching, hold it for anything touching the interpreter, and never cross the streams.

Free-threading: the GIL's twilight

Python 3.13 introduced an experimental free-threaded build (PEP 703) that removes the GIL entirely, making CPython genuinely multithreaded. This is a multi-year transition, but it changes the calculus for extension authors:

  • Extensions must be explicitly marked as free-threading-compatible, and must actually be thread-safe (no more relying on the GIL to serialize access to your C++ state).
  • Code that previously assumed "the GIL protects my global state" must add real locking.
  • nanobind was designed with free-threading in mind; pybind11 has added support. Both let you declare a module free-threading-safe.
  • The payoff: true parallelism for mixed Python/C++ workloads without the release/acquire dance — eventually.

What this means for an ML engineer today

For the foreseeable future you target the GIL-enabled build, so: release the GIL around heavy C++/CUDA work to get multicore and to overlap with other Python threads (this is exactly how PyTorch lets a Python training loop stay responsive while C++/CUDA kernels run). Design your C++ state to be thread-safe anyway — it future-proofs you for free-threading and prevents bugs when your extension is called from multiple Python threads.

Release the GIL for the math, hold it for the interpreter. That one rule separates an extension that scales across cores from one that quietly serializes everything.

pip reads these tags and picks the most specific wheel matching your interpreter, ABI, and platform. If none matches, it falls back to the sdist and builds from source.

The manylinux problem and solution

A Linux binary compiled on Ubuntu 24.04 may not run on CentOS 7 — different glibc versions, different system library versions. So how can one Linux wheel work everywhere? The answer is the manylinux standard: build inside a deliberately old Linux environment so the binary depends only on a baseline set of very stable, ancient system libraries that every modern distro still has.

TagBaseline
manylinux2014Based on CentOS 7 / glibc 2.17. Very broad compatibility. A common safe default.
manylinux_2_28glibc 2.28 (AlmaLinux 8 era). Newer baseline, still widely compatible.
manylinux_2_34glibc 2.34. Newest, for when you need recent toolchains.

The manylinux_X_Y naming directly encodes the minimum glibc version (X_Y = glibc 2.Y... e.g. 2_17). A wheel tagged manylinux_2_17 promises "I only need glibc 2.17 or newer," which virtually every live Linux system satisfies.

The manylinux trick is "build old, run everywhere." Compile your extension in an old-glibc container (the PyPA provides official manylinux Docker images), bundle any third-party shared libraries into the wheel itself (so you don't depend on the host having them), and the result runs on essentially any Linux from the last decade. The tools auditwheel (Linux), delocate (macOS), and delvewheel (Windows) automate the "bundle the libraries in" step (§41).

41Extension wheels & cibuildwheel

Shipping a compiled extension as wheels for every platform/Python combination is genuinely hard — you'd need Linux, macOS (Intel + ARM), and Windows machines, each with multiple Python versions. cibuildwheel turns this into a single CI configuration.

The matrix problem

A C++ extension supporting Python 3.9–3.13 on Linux (x86_64 + aarch64), macOS (x86_64 + arm64), and Windows (x86_64) is already 5 × 5 = 25 wheels. Building these by hand is untenable; doing it correctly (right glibc baseline, bundled libraries, right ABI tags) by hand is nearly impossible.

cibuildwheel

cibuildwheel runs in CI (GitHub Actions, etc.), spins up the right environments (including manylinux containers and emulation for cross-arch), builds your wheel for every target, repairs each wheel to bundle dependencies, and tests each one in a clean environment.

# .github/workflows/wheels.yml (essentials)
jobs:
  build_wheels:
    strategy:
      matrix:
        os: [ubuntu-latest, macos-latest, windows-latest]
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4
      - uses: pypa/cibuildwheel@v2.x
        env:
          CIBW_BUILD: "cp39-* cp310-* cp311-* cp312-* cp313-*"
          CIBW_ARCHS_LINUX: "x86_64 aarch64"
      - uses: actions/upload-artifact@v4
        with: { path: ./wheelhouse/*.whl }

That one config produces the whole matrix of tested, repaired wheels. cibuildwheel internally uses the manylinux images and the repair tools below.

The wheel-repair tools

ToolPlatform · job
auditwheelLinux. Checks the wheel only depends on the manylinux baseline; copies any extra shared libraries into the wheel and rewrites rpaths to $ORIGIN (§13). Re-tags the wheel as manylinux.
delocatemacOS. Bundles non-system .dylibs into the wheel and fixes load paths.
delvewheelWindows. Bundles dependent .dlls into the wheel.

These tools solve the loader-path problem (§13) for distribution: instead of hoping the user has libfoo.so installed, the wheel contains it, with rpaths rewired so the extension finds its bundled copy. This is why a pip install scipy just works without you installing BLAS/LAPACK system-wide — they're bundled in the wheel.

The ABI question for extensions

An extension compiled against CPython 3.11's ABI generally won't load in 3.12 (the cp311 ABI tag enforces this). Two mitigations:

  • The stable ABI / limited API (abi3 wheels): compile against a restricted, version-stable subset of the C-API. One abi3 wheel works across many Python versions — far fewer wheels to build. nanobind supports this well (§34); pybind11 has constraints.
  • Just build per-version wheels with cibuildwheel — simpler, more features available, at the cost of more artifacts. The common choice unless the version matrix is painful.
The libstdc++ ABI trap in wheels

A C++ extension dynamically links libstdc++. If your wheel's extension was built against a newer libstdc++ than the user's system provides, importing it fails with undefined symbol: ...GLIBCXX_3.4.30... at load time (§15). The manylinux + auditwheel flow handles this by building against an old baseline and/or bundling the right libstdc++. This is the single most common "my C++ wheel imports on my machine but not the user's" failure — and exactly why the manylinux discipline exists.

42Environments: pip, venv, uv, conda

Installing packages without isolating environments is how you get "dependency hell" — project A needs numpy 1.x, project B needs numpy 2.x, and a single global install can't serve both. Environment isolation is non-negotiable; the tools differ in speed and scope.

The tools

ToolWhat it does
venv (stdlib)Creates an isolated Python environment (a directory with its own interpreter link and site-packages). The baseline.
pipInstalls packages into the active environment from PyPI (or wheels/sdists). The default installer.
virtualenvThe older third-party venv; faster creation, more features. Largely superseded by stdlib venv for simple use.

The baseline workflow (venv + pip)

# Create and activate an isolated environment
$ python -m venv .venv
$ source .venv/bin/activate          # Windows: .venv\Scripts\activate

# Install into it
$ pip install numpy requests
$ pip install -e .                   # editable install of the current project

# Capture exact versions for reproducibility
$ pip freeze > requirements.txt

The modern fast workflow (uv)

# uv replaces venv+pip+pip-tools, an order of magnitude faster
$ uv venv                            # create .venv
$ uv pip install numpy requests      # install (fast)

# Or full project management with a lockfile
$ uv add numpy                       # adds to pyproject.toml + uv.lock
$ uv sync                            # reproduce the exact locked environment
$ uv run python script.py            # run in the project env without activating

pip vs conda — the key distinction

pip (+ venv/uv)

Installs Python packages from PyPI. For compiled extensions, relies on wheels that bundle their native deps. Lightweight, universal, the default. Struggles when you need specific system-level binaries (a particular CUDA toolkit, MKL, a Fortran BLAS) that aren't bundled.

conda (+ mamba)

Installs from conda channels (conda-forge, etc.), managing Python and arbitrary binary dependencies — CUDA, compilers, system libraries — as first-class packages with a real solver. Heavier, but the right tool for complex ML/scientific stacks where native deps must be coordinated.

Don't mix pip and conda carelessly

Within a conda environment, installing some things with conda and others with pip can produce inconsistent states (conda's solver doesn't know about pip-installed packages). The pragmatic rule: prefer conda for the environment's foundation (Python, CUDA, native libs), then pip-install pure-Python leaf packages on top, and avoid having both manage the same package. For pure-Python or wheel-friendly stacks, uv/pip alone is simpler and faster.

43Versioning, entry points, metadata

The finishing details that separate a sloppy package from a professional one.

Versioning

Python versioning follows PEP 440 (which is close to, but not identical to, SemVer). Key forms:

1.2.3            # release
1.2.3a1          # alpha pre-release
1.2.3rc1         # release candidate
1.2.3.dev4       # development release
1.2.3.post1      # post-release (packaging fix, no code change)
1.2.3+cu121      # local version label (e.g. a CUDA-specific build)

Two common version-management strategies:

  • Static: hardcode version = "1.2.3" in pyproject.toml. Simple, explicit.
  • Dynamic from VCS: tools like setuptools-scm or hatch-vcs derive the version from your git tags, so git tag v1.2.3 is the single source of truth. Declare dynamic = ["version"] in [project].

Dependency specifiers

dependencies = [
    "numpy>=1.24,<3",            # range
    "requests~=2.28",             # compatible release: >=2.28,<3.0
    "torch>=2.0; platform_system=='Linux'",  # environment marker
    "typing-extensions; python_version<'3.11'",  # conditional dep
]
Libraries pin loosely; applications pin tightly

If you publish a library to PyPI, use loose ranges (numpy>=1.24) so it composes with others' requirements — over-pinning a library causes resolution conflicts for everyone downstream. If you deploy an application, pin exactly (a lockfile: uv.lock, poetry.lock, or pinned requirements.txt) so deployments are reproducible. The lockfile is for the application; loose ranges are for the library. Confusing the two is a common source of either un-composable libraries or un-reproducible deployments.

Entry points — exposing commands and plugins

Entry points let a package register a console command or advertise a plugin to other packages.

# A console script: `pip install mytool` then run `mytool` from the shell
[project.scripts]
mytool = "mytool.cli:main"          # module:function

# A GUI script (no console window on Windows)
[project.gui-scripts]
mytool-gui = "mytool.gui:run"

# A plugin entry point: other packages can discover this
[project.entry-points."myapp.plugins"]
fancy = "mytool.plugins:FancyPlugin"

The console-script mechanism is how pip, pytest, black, and every other command-line Python tool puts an executable on your PATH — the installer generates a small launcher that imports the module and calls the function. The plugin mechanism is how pytest finds plugins, how flake8 finds extensions, etc.: a host package queries the entry-point group at runtime and loads whatever packages registered under it.

Inspecting installed metadata

$ pip show numpy                 # name, version, deps, location
$ pip show -f mytool             # also list installed files
$ python -c "import importlib.metadata as m; print(m.version('numpy'))"

# What does this wheel contain / claim?
$ unzip -l mytool-1.2.0-*.whl    # a wheel is just a zip
$ python -m zipfile -l mytool-1.2.0-*.whl

A wheel is a zip with metadata. Once that demystifies, packaging stops being magic and becomes a format you can inspect, build, and reason about like any other.

44Beyond CMake: Bazel, Meson, Ninja

CMake won the cross-platform C++ configuration war, but knowing the alternatives clarifies CMake's tradeoffs and prepares you for codebases that chose differently.

Ninja — the executor, not a configurator

First, a clarification: Ninja is not a CMake alternative. Ninja is a build executor — a faster, dumber replacement for Make, designed to be generated by a higher-level tool, not written by hand. CMake generates build.ninja; Meson generates build.ninja; you almost never write Ninja by hand. It exists to run generated builds with minimal overhead and maximal parallelism. So "CMake + Ninja" is the configurator + executor pair, not a choice between them.

The configurator alternatives

ToolCharacter
MesonA cleaner, more opinionated CMake alternative. Python-like syntax, fast, sane defaults, generates Ninja. Used by GNOME, systemd, SciPy, NumPy. Less ubiquitous than CMake but much more pleasant to write. Strong choice for new projects not constrained to CMake.
BazelGoogle's build system. Hermetic, reproducible, scales to giant monorepos, content-addressed caching, remote execution. Steep learning curve and heavy infrastructure; overkill for small projects, transformative for huge ones. Used by TensorFlow.
build2An integrated C++ build + package manager aiming to be a complete toolchain. Smaller community.
Buck2Meta's Bazel-like system (Rust-based). Similar niche to Bazel — large-scale, hermetic.
xmake / premakeLua-configured build systems; lighter than CMake, smaller ecosystems.

The Bazel difference: hermeticity

Bazel's defining idea is hermetic, content-addressed builds. Every action declares its exact inputs; outputs are cached by the hash of all inputs (compiler, flags, sources, dependencies). Given the same inputs, the output is reused — locally or from a shared remote cache, even across a whole company. This is far stronger than Make's timestamp model (§16): it's correct under any change, supports distributed caching, and makes builds reproducible by construction. The cost is rigidity — Bazel insists on knowing every input, which means wrapping every dependency in Bazel's model.

What to actually use

Default to CMake + Ninja — the broadest ecosystem, every dependency supports it, every IDE understands it. Consider Meson for a new project where you control all dependencies and want a saner language. Reach for Bazel/Buck2 only at monorepo scale where hermeticity and remote caching pay for the heavy onboarding. For Python-facing C++ extensions specifically, the configurator is wrapped by scikit-build-core (CMake) or meson-python (Meson) regardless (§36).

45Caching, reproducibility, speed

Build speed is engineering productivity. A 10-minute build that could be 1 minute taxes every iteration. The cross-cutting techniques apply regardless of build system.

Compiler caching

ToolWhat it does
ccacheCaches the result of each compilation keyed by a hash of the preprocessed source + flags. Recompiling an unchanged TU becomes a cache hit — near-instant. The single highest-ROI build-speed tool. Drop in via CMAKE_CXX_COMPILER_LAUNCHER=ccache.
sccacheccache's spiritual successor (Rust), with support for a shared cache backend (S3, Redis, GCS) so a whole team or CI fleet shares cache hits. Also caches Rust and CUDA.
 
# Wire ccache into a CMake build — one flag, big speedup on rebuilds
$ cmake -B build -G Ninja -DCMAKE_CXX_COMPILER_LAUNCHER=ccache
$ ccache --show-stats             # watch the hit rate climb

Reducing the work in the first place

  • Faster linker — switch to lld or mold (§09). Linking often dominates incremental builds; this is a one-flag win.
  • Precompiled headers — target_precompile_headers() in CMake compiles a heavy header set once. Helps codebases that include large headers (the STL, framework headers) everywhere.
  • Unity / jumbo builds — concatenate many TUs into one to amortize header parsing (CMAKE_UNITY_BUILD). Faster full builds, but hurts incremental builds and can surface ODR issues; use with judgment.
  • Reduce header coupling — forward declarations, the pimpl idiom, and (eventually) C++20 modules cut the recompilation cascade (§02).
  • Parallelism — -j$(nproc), and Ninja's automatic parallelism. Make sure your dependency graph is correct or parallelism causes races (§18).

Distributed compilation

For very large codebases, distribute compilation across many machines: distcc (classic), icecc/icecream (with scheduler), or the remote-execution facilities of Bazel/Buck2. Combined with a shared sccache, a fleet can build what one machine couldn't in reasonable time.

Reproducible builds

A reproducible build produces bit-for-bit identical output from the same source. It matters for security (verifying a binary matches its source), caching (identical inputs → identical outputs → cache hits), and debugging. The enemies of reproducibility:

  • Embedded timestamps — __DATE__/__TIME__ (§04), archive timestamps. Set SOURCE_DATE_EPOCH to pin them.
  • Absolute paths in debug info — use -ffile-prefix-map=$PWD=. to strip build-dir paths.
  • Nondeterministic ordering — file(GLOB) ordering, hash-map iteration in codegen, parallel-build races.
  • Toolchain drift — different compiler versions produce different code. Pin the toolchain (containers, toolchain files §26).

The cheapest big win is almost always ccache plus a faster linker. Before reaching for unity builds or distributed compilation, add CMAKE_CXX_COMPILER_LAUNCHER=ccache and -fuse-ld=mold. Together they often cut incremental build time by more than half with zero code changes — and ccache's effect compounds across every branch switch and CI run. Measure with ccache --show-stats and your build tool's timing before optimizing further.

46Cases, cheatsheets & closing

Six worked debugging cases spanning the whole pipeline, the cheatsheets worth keeping open, and the closing signals.

case 1 "undefined reference, but the function is right there"
Symptom

A C++ program calling into a C library fails to link: undefined reference to 'parse_config(char const*)'. The function is clearly defined in libconfig.a, which is on the link line.

Diagnosis

nm -C libconfig.a shows T parse_config (unmangled — it's a C symbol). nm -C main.o shows U parse_config(char const*) (mangled). The C++ TU mangled the name because the C header lacked extern "C" (§12). The two names don't match.

Resolution

Wrapped the C header's declarations in #ifdef __cplusplus / extern "C" { ... }. Now C++ emits the unmangled symbol, it matches the archive, and it links. (Had the names matched, the next suspect would have been link order — §14 — putting -lconfig after main.o.)

Takeaway

Every "undefined reference" is answerable with nm -C on both sides: who needs the symbol, who defines it, and whether the mangled forms match. The fix names itself.

case 2 "the wheel imports on my machine, crashes on the server"
Symptom

A pybind11 extension wheel built on the dev box (Ubuntu 24.04) fails on import on an older deployment host: ImportError: ... undefined symbol: ...GLIBCXX_3.4.32.

Diagnosis

The extension was linked against the dev box's newer libstdc++ and records a dependency on a recent GLIBCXX version symbol the server's older libstdc++ doesn't provide (§11, §15). ldd on the server confirmed the system libstdc++ was older.

Resolution

Rebuilt the wheel inside the official manylinux container (old glibc baseline) and ran it through auditwheel repair, which re-tagged it manylinux and bundled the needed runtime. The repaired wheel imported everywhere. Switched CI to cibuildwheel so every release wheel is built this way (§41).

Takeaway

"Build old, run everywhere." Never build distribution wheels on your latest-and-greatest dev environment; build them in manylinux and audit them. The whole manylinux/auditwheel apparatus exists precisely to prevent this failure.

case 3 "I changed a header but nothing rebuilt"
Symptom

In a hand-written Makefile project, editing config.h changes behavior, but make says "nothing to be done" and the old behavior persists until make clean.

Diagnosis

The object rules listed only the .cpp as a prerequisite, not the headers it includes. Make's dependency graph was incomplete, so it never learned that the object depended on config.h (§16, §18).

Resolution

Added automatic dependency generation: -MMD -MP on the compile line and -include $(DEPS) in the Makefile (§18). Now each .o depends on every header it actually includes, and header edits trigger correct rebuilds. (This is one reason teams move to CMake, which generates these dependencies automatically.)

Takeaway

A stale build is almost always an incomplete dependency graph. Timestamp-based tools only rebuild what you told them depends on the changed file; getting header deps right is mandatory, not optional.

case 4 "cannot open shared object file" only in production
Symptom

A binary runs in development but in the container fails at startup: error while loading shared libraries: libmyengine.so: cannot open shared object file.

Diagnosis

ldd ./app on the dev box found libmyengine.so via an LD_LIBRARY_PATH set in the developer's shell profile — invisible and forgotten. In the container, no such variable, and the .so wasn't in any standard search path or baked as an rpath (§13).

Resolution

Linked the app with -Wl,-rpath,'$ORIGIN/../lib' and shipped the .so in a lib/ dir next to the binary, making the bundle relocatable and self-contained. Removed reliance on LD_LIBRARY_PATH entirely.

Takeaway

"Works on my machine" loader failures are almost always a hidden LD_LIBRARY_PATH. The production answer is rpath ($ORIGIN), a system install + ldconfig, or static linking — never an environment variable.

case 5 "my CMake Debug conditional does nothing in Visual Studio"
Symptom

A project with if(CMAKE_BUILD_TYPE STREQUAL "Debug") adding a -DDEBUG_LOG definition works under Ninja but the definition never appears in Visual Studio builds.

Diagnosis

Visual Studio is a multi-config generator: it has no single CMAKE_BUILD_TYPE at configure time — the config is chosen at build time. So the if() evaluated against an empty string and never fired (§21, §28).

Resolution

Replaced the configure-time if() with a generator expression: target_compile_definitions(app PRIVATE $<$<CONFIG:Debug>:DEBUG_LOG>), which is evaluated per-configuration at build time and works in both single- and multi-config generators.

Takeaway

Anything that must vary per build configuration belongs in a generator expression, not a configure-time if(). The two-phase model (configure vs build) is the root of most "my CMake conditional doesn't work" puzzles.

case 6 "the extension is 'parallel' but uses one core"
Symptom

A pybind11 extension doing heavy numerical work is called from multiple Python threads, but CPU usage never exceeds one core and threads don't speed anything up.

Diagnosis

The C++ function held the GIL for its entire duration. Since only one thread can hold the GIL, every "parallel" call serialized behind it (§37). The compute didn't touch any Python objects after entry, so it had no need to hold the GIL during the loop.

Resolution

Wrapped the hot loop in py::gil_scoped_release, releasing the GIL for the pure-C++ work and re-acquiring it only to build the return value. CPU usage scaled across cores immediately. Audited the loop to confirm it touched no PyObject while the GIL was released.

Takeaway

An extension that doesn't release the GIL around heavy work is single-threaded no matter how many Python threads call it. Release for the math, hold for the interpreter — and never touch Python objects without the GIL.

The senior signals

What distinguishes someone who truly understands builds from someone who copies incantations:

  • You locate errors by layer. Given an error, you know instantly whether it's preprocess, compile, link, load, or packaging — and which tool to interrogate (§01).
  • You reach for nm, ldd, and readelf reflexively. A symbol error isn't a mystery; it's a two-minute investigation with the right tool (§08, §15).
  • You know the compile/link flag split and why -std and ABI flags must match across a whole binary (§05, §07).
  • You write target-centric CMake with correct PUBLIC/PRIVATE/INTERFACE, and you can explain why old global-state CMake doesn't scale (§22–23).
  • You understand the static/shared tradeoff and the runtime loader, so deployment "library not found" errors don't surprise you (§10–13).
  • You build distribution wheels in manylinux, not on your dev box, and you know why (§40–41).
  • You release the GIL for heavy work in extensions and keep your C++ state thread-safe (§37).
  • You optimize builds with measurement — ccache and a fast linker first, fancier techniques only when the numbers justify them (§45).
  • You pick the right tool for the boundary — ctypes for a C ABI, pybind11/nanobind for C++, Cython for accelerating Python (§35).

The through-line of this entire manual: a build is a transformation pipeline, and every tool is orchestration over that pipeline. Source becomes a translation unit becomes an object becomes — via the linker — an artifact, which the loader brings to life, which packaging wraps for distribution, which bindings expose to another language. Master the transformation and the orchestration tools (Make, CMake, pip, cibuildwheel) reveal themselves as what they are: convenient, fallible automation over a process you can always inspect and reason about by hand.

When the build breaks, you don't guess. You know which layer failed, which tool to ask, and what its answer means.
