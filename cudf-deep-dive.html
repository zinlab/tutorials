00

What cuDF Is & Why It Exists

cuDF (“KOO-dee-eff”) is a GPU DataFrame library for loading, joining, aggregating, filtering, and otherwise manipulating tabular data. It is one library in the broader RAPIDS suite (cuML for ML, cuGraph for graphs, cuSpatial, cuVS, etc.), all of which share the same memory manager and CUDA foundations.

The motivating problem: pandas is the lingua franca of data work but is single-threaded, CPU-bound, and row-oriented in spirit. As datasets grow into the tens of GBs, pandas spends its life waiting on a CPU. cuDF keeps the familiar API but executes on thousands of GPU threads against a columnar layout built for parallelism, frequently delivering 10–150× speedups on ETL-style workloads.

What lives under the "cuDF" umbrella

ComponentWhat it is
libcudfCUDA/C++ library: Arrow-compliant data structures + the actual algorithms (joins, groupby, sort, string ops, IO). The real engine.
pylibcudfThin Cython bindings exposing libcudf to Python with near-zero overhead. The new foundation layer.
cudf (classic)The pandas-like Python package: DataFrame, Series, Index. What you import cudf.
cudf.pandasZero-code-change accelerator: it transparently swaps pandas for cuDF with CPU fallback.
cudf.polarsA GPU execution engine for Polars LazyFrames.
dask-cudfMulti-GPU / multi-node scale-out via Dask, with cuDF partitions.
Mental model

Think of cuDF as "the database/ETL engine on the GPU," not "NumPy on the GPU." Its workloads are relational algebra (selects, joins, group-bys, sorts) over heterogeneous typed columns — a fundamentally different shape of computation than the dense linear algebra PyTorch is built for. Hold onto this distinction; it explains almost every design decision.

01

The Columnar Model (Apache Arrow)

Everything in cuDF rests on the Apache Arrow columnar memory format. Understanding a single column is the key that unlocks the whole library, because every algorithm is ultimately written against this physical layout.

A column stores values of one type contiguously (Struct-of-Arrays), so a GPU warp reading element i across 32 threads reads 32 adjacent addresses — a coalesced memory access. That coalescing is the physical reason columnar layouts are fast on GPUs.

Anatomy of a column

A libcudf column is a small bundle of GPU buffers:

  • Data buffer — the values, packed contiguously.
  • Validity (null) bitmask — 1 bit per element; 1 = valid, 0 = null. Lets cuDF represent missing data without sentinel values, and skip nulls in a branch-light way.
  • Null count — cached count of zeros in the mask.
  • Type + size — a runtime data_type and element count.
  • Child columns — for nested/variable-length types.
values 127×315×
null mask 110110

Above: an int column of length 6 with two nulls. The data slots under a null may hold garbage — only the mask defines validity.

Variable-length & nested types

Strings and lists can't be fixed-width, so they use offsets + a child buffer:

offsets 03810
chars G P U c u D F i s

A string column ["GPU","cuDF","is"]: the offsets child says string i spans chars[offset[i] : offset[i+1]]. Length is implied by the difference of consecutive offsets — computing all lengths is a single vectorized subtraction.

  • String = offsets child + character data.
  • List = offsets child + a child column of elements.
  • Struct = N child columns (one per field), sharing a null mask.
  • Dictionary / categorical = integer indices column + a keys child.
  • Decimal = fixed-point (decimal32/64/128) integers + a scale.
Why this matters in an interview

If asked “how does cuDF compute string lengths fast?” the answer is structural: it's offsets[i+1] - offsets[i], a fully parallel elementwise op — no scanning of characters needed. Most string-algorithm performance comes from exploiting the offsets layout.

The Architecture, Layer by Layer

cuDF is deliberately stratified. Data and control flow down through layers that each translate from a friendlier abstraction to a faster, narrower one. From the top (your Python) to the metal (CUDA):

Python / user-facing
Frame layer — cudf.DataFrame / Series / Index
pandas-compatible API, alignment, indexing, dtype coercion, the “feel” of pandas.
▼
Python / internal
Column layer — cudf.core.column
Internal typed-column objects; bridges pandas semantics to the GPU buffers. Being slimmed down over time.
▼
Cython
pylibcudf — near-zero-overhead bindings
Pythonic Table/Column/DataType wrappers calling straight into libcudf. Both cudf.pandas and cudf.polars are built directly on this.
▼
C++ / CUDA
libcudf — the engine
Owning types (column, table), views, the type_dispatcher, and every algorithm. This is where data actually gets processed.
▼
CUDA primitives
CCCL (Thrust + CUB) · cuco · cuda kernels
Device-wide sort/scan/reduce, block/warp primitives, concurrent hash tables, and hand-written kernels. The actual GPU execution.

The historical “Cython layer” has largely migrated into pylibcudf. The intent is that pylibcudf becomes a first-class, standalone way to drive libcudf from Python — even exposing libcudf features pandas has no concept of (e.g. specialized text ops) — while cuDF-classic becomes a thinner pandas-compatibility shell on top.

The one-sentence architecture

cuDF-classic gives you the pandas API → pylibcudf gives Python near-direct access to → libcudf, a C++/CUDA library of Arrow-shaped data structures and algorithms → built on CCCL/cuco/custom kernels → all memory allocated through RMM.

03

Inside libcudf: The C++ Core

This is the heart of the system and the part interviewers probe hardest. Three ideas define it: the owning-vs-view ownership model, the type_dispatcher, and the functional (immutable) algorithm style.

Owning types vs. views

libcudf uses modern C++ (RAII, smart pointers) and draws a hard line between memory owners and non-owning views:

TypeOwns memory?Role
rmm::device_bufferYesUntyped owned device memory.
cudf::columnYesA typed column: data buffer + null mask + children.
cudf::tableYesAn owned set of columns.
cudf::column_viewNoRead-only descriptor: pointer, size, type, null mask, offset, children.
cudf::mutable_column_viewNoWritable view.
cudf::table_viewNoNon-owning set of column_views (lets tables share columns cheaply).
cudf::scalarYesA single typed value living in device memory.

The convention that follows from this is the signature you'll see everywhere: algorithms take views as input and return freshly-allocated owning outputs.

C++ · libcudf// Canonical libcudf signature shape.
// Inputs are non-owning views; output is a newly-allocated owned column.
// The last two params are ALWAYS a stream and a memory resource.
std::unique_ptr<cudf::column> my_op(
    cudf::column_view const& input,
    rmm::cuda_stream_view stream      = cudf::get_default_stream(),
    rmm::device_async_resource_ref mr = cudf::get_current_device_resource_ref());
Interview-grade detail

Why views in, owners out? It makes ownership unambiguous, makes functions composable, and means operations don't mutate their inputs — which is exactly what enables safe parallelism and copy-on-write semantics. The cost is that nearly every op allocates, which is precisely why a fast allocator (RMM, §05) is not optional but load-bearing.

The type_dispatcher

A column's type is a runtime value (cudf::data_type), but CUDA kernels are compile-time templates. The type_dispatcher bridges that gap: given a runtime type, it instantiates and invokes a templated functor for the matching concrete type. This is how a single API like cudf::sort works across int32, float64, strings, timestamps, etc., without a giant switch in every algorithm.

C++ · libcudfstruct size_of_functor {
  template <typename T>
  __device__ int operator()() { return sizeof(T); }
};

// runtime data_type  ->  compile-time T  ->  functor invoked
int n = cudf::type_dispatcher(col.type(), size_of_functor{});

Internally it's a switch over the type id that maps each case to functor.operator()<ConcreteType>(). Write your kernel once as a template; the dispatcher selects the instantiation. This single mechanism keeps libcudf from drowning in per-type code.

Algorithms are compositions of primitives

libcudf rarely hand-rolls kernels from scratch. Most algorithms are compositions of parallel primitives over the columnar buffers:

DataFrame opHow it's realized on the GPU
filter / boolean maskelementwise predicate → stream compaction (copy_if / gather of surviving indices)
sortCUB device radix sort (fixed-width) or comparator sort; reorder via gather
groupby + agghash-based (build a cuco concurrent hash map keyed on group cols, scatter-reduce) or sort-based (sort by key, then segmented reduce)
joinbuild a hash table on the smaller side, probe with the other, gather matched row indices from both
reduction / scanCUB device-wide reduce / prefix-scan
gather / scatterthe universal “reorder/select rows” primitive underpinning sort, join, filter, take

Two primitives do an outsized amount of work: gather (output[i] = input[map[i]]) and scatter (output[map[i]] = input[i]). Once you see that selecting, sorting, joining, and shuffling are all “produce an index map, then gather,” libcudf's design clicks into place.

The core power, decoded

cuDF's speed isn't one magic kernel. It's: (1) a columnar layout that makes memory accesses coalesced and operations vectorizable, (2) a small set of extremely-optimized data-parallel primitives (sort/scan/reduce/gather/scatter/hash) from CCCL and cuco, (3) a type-dispatch mechanism to apply them generically, and (4) an allocator fast enough that the allocate-heavy functional style stays cheap.

04

How Computation Actually Reaches CUDA

There are two distinct paths from “a Python call” to “a running CUDA kernel.”

Path A — Precompiled libcudf kernels (the common case)

When you call df.groupby(...).mean(), the work routes Python → pylibcudf → a precompiled libcudf function. libcudf ships compiled CUDA kernels (and CCCL/cuco templates instantiated at libcudf build time). Your call just launches them on a stream against your column buffers. No compilation happens at runtime; it's a library call into shipped machine code.

Path B — JIT-compiled user code (UDFs & expressions)

When you write a custom Python function and apply it, cuDF must turn your Python into a GPU kernel at runtime. The pipeline:

pipelineYour Python UDF
   │  Numba parses + type-infers
   ▼
Numba IR  ──►  LLVM IR  ──►  PTX  (CUDA assembly)
   │
   ▼  (for row-wise libcudf integration)
Jitify JIT-compiles & links the PTX device function
into a native CUDA C++ kernel, then launches it.

So Numba lowers your Python to PTX, and Jitify (libcudf's runtime CUDA JIT) inlines that device function into a templated kernel and compiles/caches it. The first call pays a one-time compilation cost; subsequent calls reuse the cached kernel. The same JIT machinery powers transform with string expressions and some groupby-apply paths.

Streams & asynchrony

Every libcudf API takes a cuda_stream_view. Kernel launches and RMM allocations are stream-ordered: enqueued onto a stream and executed asynchronously with respect to the host. This lets independent operations overlap and lets allocation/free participate in the same ordering as compute (critical for correctness with async allocators). libcudf supports per-thread default stream (PTDS) so multi-threaded host code doesn't serialize on the legacy default stream.

Gotcha

Because GPU work is async, a microbenchmark that doesn't synchronize() before stopping the timer measures launch latency, not execution. Always synchronize when timing GPU code.

05

Memory Management: RMM

The RAPIDS Memory Manager (RMM) is the shared allocation layer for all of RAPIDS (cuDF, cuML, cuGraph) and interops with CuPy and Numba. It exists because raw cudaMalloc/cudaFree are slow and synchronizing — and recall from §03 that cuDF's functional style allocates constantly. A naive allocator would dominate runtime.

Memory resources (the core abstraction)

RMM is built around the device_memory_resource interface — an allocate(bytes, stream) / deallocate(...) pair. Concrete resources and composable adaptors let you assemble exactly the allocation behavior you want:

ResourceBehavior
cuda_memory_resourcePlain cudaMalloc/cudaFree. Simple, slow, synchronous.
pool_memory_resourceAllocates one big block up front, then sub-allocates from it. The classic perf win — avoids per-op driver calls.
cuda_async_memory_resourceUses cudaMallocAsync + the CUDA driver's built-in stream-ordered memory pool. Modern, integrates with stream ordering.
managed_memory_resourcecudaMallocManaged (unified memory): pages migrate between host and device on demand, enabling oversubscription beyond GPU RAM.
arena_memory_resourcePer-thread arenas to reduce contention in multi-threaded allocation.
pinned_memory_resourcePage-locked host memory for fast async H↔D transfers.
adaptorslogging, tracking, statistics, limiting, failure_callback, prefetch — wrap any resource to add behavior. They compose (a tracking pool with logging, etc.).

Turning on the pool (the single biggest knob)

Pythonimport rmm
# One large up-front allocation; sub-allocate from it thereafter.
rmm.reinitialize(
    pool_allocator=True,
    initial_pool_size="10GB",   # grab 10 GB once
)
import cudf            # cuDF now allocates from the RMM pool
# Real-world: switching the pool on alone can yield multi-x speedups,
# because alloc/free stop hitting the slow CUDA driver path.

cuDF doesn't manage memory itself — it asks RMM's current device resource. Swap that resource (pool, async, managed, logging) and all of cuDF's allocations change behavior with zero cuDF code changes. That decoupling is the whole point of RMM.

Running larger-than-GPU-memory

  • Managed memory — unified-memory pages migrate automatically; you can process datasets bigger than VRAM, paying paging cost when you exceed it.
  • Spilling — cuDF can spill device buffers to host (or disk) under pressure (e.g. CUDF_SPILL=on), and Dask-CUDA orchestrates spilling across workers.
  • Chunked / streaming readers — the Parquet/ORC readers can process files in chunks so you never materialize the whole table at once.
Python ↔ C++ lifetime trap

A subtle, interview-worthy detail: an rmm::device_buffer holds only a reference to its memory resource, not ownership of it. When Python takes ownership of a buffer allocated by a C++ library, you must ensure the memory resource outlives the buffer — otherwise you free against a dead resource. The guidance is to pass in a resource whose lifetime you control.

06

Communication & Scale-Out

A single cuDF DataFrame lives on one GPU. Going beyond one GPU — or beyond one node — is where communication becomes the dominant cost, and RAPIDS has a stack purpose-built to keep data on the GPU and off the slow CPU/PCIe path.

Multi-GPU / multi-node: dask-cudf

dask-cudf partitions a logical DataFrame into many cuDF DataFrames spread across GPUs/workers. Embarrassingly-parallel ops (elementwise, filters, partition-local reductions) need no communication. The expensive ones — joins, group-bys, sorts — require a shuffle: rows are repartitioned by key across all workers so that matching keys land together. Shuffle is the network-bound bottleneck of distributed dataframes, so the transport matters enormously.

The transport: UCX, NVLink, RDMA, GPUDirect

  • UCX-py / UCXX — unified communication layer that auto-selects the fastest path between two endpoints: shared memory, NVLink/NVSwitch for GPU↔GPU inside a node, or InfiniBand/RDMA across nodes.
  • GPUDirect RDMA — sends data directly from one GPU's memory to another over the network, bypassing host staging buffers (no GPU→CPU→NIC→CPU→GPU bounce).
  • Dask-CUDA — provides LocalCUDACluster, sets up one worker per GPU, configures the RMM pool per worker, enables spilling, and wires up UCX.

Storage & compression

  • KvikIO + GPUDirect Storage (GDS) — read files from NVMe/storage straight into GPU memory, skipping the CPU bounce buffer. cuDF's Parquet/ORC readers can use this path.
  • nvCOMP — GPU-accelerated (de)compression used inside the Parquet/ORC readers/writers, so decompression itself runs on the GPU rather than choking the CPU.
The throughline

Every layer of the comms/IO stack chases the same goal: keep bytes on the GPU and avoid the CPU/PCIe bottleneck. GDS for reads, nvCOMP for decompress, NVLink/RDMA + GPUDirect for transfers. In distributed cuDF, your performance story is almost always a shuffle/transport story.

The Three Front-Ends

One engine (libcudf), three ways to drive it — a key thing to articulate in an interview.

1 · cudf-classic — explicit GPU dataframes

You import cudf and write GPU-native code. Maximum control; you know everything is on the GPU; unsupported ops raise rather than silently fall back.

2 · cudf.pandas — zero code change

Accelerate existing pandas code (and third-party libraries that use pandas) with no edits:

Python# notebook:  %load_ext cudf.pandas
# script:    python -m cudf.pandas script.py
# explicit:  import cudf.pandas; cudf.pandas.install()   # BEFORE importing pandas
import pandas as pd     # pd is now a proxy module

How it works: cudf.pandas hijacks the import of pandas via a ModuleAccelerator(fast=cudf, slow=pandas). Every pandas object becomes a proxy wrapping both a GPU (cuDF) and CPU (pandas) implementation. Each attribute lookup / method call is attempted on the GPU first (copying from CPU if needed); if that fails (an unsupported op, an exotic dtype), it transparently falls back to CPU (copying back from GPU), synchronizing data under the hood. A profiler tells you which lines ran where.

Trade-offs to name

Fallback means hidden GPU↔CPU copies, so favor immutable operations (mutability isn't always preserved across a fallback). Join row-ordering may differ from pandas for performance. And cudf.pandas is not meant to be mixed with a direct import cudf.

3 · cudf.polars — GPU engine for Polars

Polars builds a lazy logical query plan; cudf.polars translates that plan (its IR) into pylibcudf operations and executes on the GPU, falling back to CPU Polars for unsupported nodes:

Pythonimport polars as pl
lf = pl.scan_parquet("events.parquet").filter(pl.col("spend") > 8)
lf.collect(engine="gpu")        # execute the plan on the GPU via libcudf

This is the most “query-optimizer” flavored front-end: because Polars is lazy, it can do plan-level optimization (predicate/projection pushdown, etc.) before lowering to kernels — closer to a real database query engine than imperative pandas.

cuDF vs. PyTorch: Two GPU Worldviews

Both move computation onto CUDA, both have caching allocators, both are async on streams — but they solve different problems and embody different paradigms. This comparison is gold for an interview because it forces you to articulate cuDF's identity.

DimensioncuDFPyTorch
Data modelTable of heterogeneous, typed columns (Struct-of-Arrays). Nulls, strings, lists, structs, categoricals, decimals, datetimes.Dense, homogeneous N-D Tensor of one numeric dtype, strided.
DomainETL / analytics: joins, group-bys, sorts, filters, string & time ops, IO.Deep learning / numerics: matmul, conv, activations, gradients.
Computation kindRelational algebra: gather/scatter, hashing, segmented & variable-length ops. Often memory-bandwidth-bound & irregular.Dense linear algebra: regular access patterns, often FLOP-bound.
Kernel librariesCCCL (Thrust/CUB), cuco hash tables, custom kernels.cuBLAS, cuDNN, CUTLASS, custom kernels.
ParadigmMostly declarative / functional: immutable columns, ops return new columns; the “graph” (in cudf.polars) is a query plan.Imperative tensor algebra: you compose differentiable ops; a dynamic compute graph is recorded.
AutodiffNone. No gradients; not its job.Core feature: autograd records ops for backprop.
MutabilityColumns largely immutable; copy-on-write; ops allocate outputs.In-place ops common (x.add_()); views share storage.
Runtime compilePrecompiled libcudf + Numba/Jitify JIT for UDFs & expressions; cudf.polars optimizes the plan.torch.compile / TorchInductor fuses op graphs.
AllocatorRMM (pluggable, shared across all RAPIDS).Built-in CUDA caching allocator.

The deep difference, in one line

PyTorch asks “what arithmetic do I do to these dense numbers (and how do I differentiate it)?” cuDF asks “how do I reorganize, match, and aggregate these heterogeneous rows?” One is calculus over tensors; the other is relational algebra over tables. That's why cuDF leans on sort/hash/gather and PyTorch leans on matmul/conv.

They're complementary — and interop is zero-copy

The real-world pattern is to use them together: GPU ETL/feature-engineering in cuDF, then feed a model in PyTorch — without ever leaving the GPU. Numeric columns can move zero-copy because both honor the __cuda_array_interface__ and DLPack protocols:

Pythonimport cudf, cupy as cp, torch

gdf = cudf.read_parquet("features.parquet")   # GPU ETL in cuDF
feats = gdf[["f0","f1","f2"]]

# zero-copy: cuDF -> CuPy -> torch, all on-device, no host round-trip
arr = feats.to_cupy()                         # or .values
t   = torch.as_tensor(arr, device="cuda")      # shares GPU memory
# ...train / infer in PyTorch...
Boundary of interop

Only numeric, fixed-width columns map cleanly to tensors. Strings, lists, structs, and nulls have no tensor equivalent — you encode/embed/impute them in cuDF first, then hand the resulting numeric block to PyTorch. That handoff point is itself a great thing to discuss: feature engineering is cuDF's job; learning is PyTorch's.

High-Value Topics for the Interview

If you can speak fluently to these, you'll demonstrate real depth beyond API familiarity.

Data & layout

  • Why columnar wins on GPUs — coalesced memory access, SIMT vectorization, better compression, projection pushdown.
  • Null handling — validity bitmask, cached null count, null-aware kernels; why this beats sentinel values.
  • String/list/struct representation — offsets + child buffers; computing lengths as offset differences; why nested types share/compose null masks.
  • Decimal / fixed-point — decimal32/64/128 as scaled integers (exactness for money).

Engine internals

  • Owning vs view types; the “views in, owned out” signature convention; the (stream, mr) trailing params.
  • type_dispatcher — runtime type → compile-time template; how one API covers all dtypes.
  • Primitive decomposition — express filter/sort/join/groupby in terms of gather/scatter/scan/reduce/hash; name CCCL and cuco.
  • Hash-based vs sort-based groupby/join trade-offs.
  • Stream ordering & PTDS — async execution; stream-ordered allocation; why timing needs synchronization.
  • JIT path — Numba → PTX → Jitify for UDFs/expressions; one-time compile cost + caching.

Memory & scale

  • RMM resources & adaptors — pool vs async vs managed; composing logging/tracking; why the pool is the headline knob.
  • Larger-than-memory — managed memory oversubscription, spilling, chunked Parquet reader.
  • Distributed cuDF — dask-cudf partitions; the shuffle as the bottleneck; UCX/NVLink/RDMA/GPUDirect.
  • IO acceleration — KvikIO + GPUDirect Storage; nvCOMP in Parquet/ORC.

Engineering & ecosystem

  • Build & deps — CMake, CCCL (Thrust + CUB + libcudacxx), conda + pip wheels, separate CUDA 11/12 builds, calendar versioning (YY.MM).
  • Testing — GoogleTest in libcudf (where algorithm correctness is validated), pytest + pandas-equivalence checks in Python, pylibcudf tests that only verify the bindings (not re-test the algorithm).
  • Interop protocols — __cuda_array_interface__, DLPack, Arrow C Data Interface; how cuDF plays with CuPy, Numba, PyTorch, Spark (Spark-RAPIDS plugin).
  • pandas-compatibility tax — the semantic gaps (index alignment, mutation, dtype edge cases) cuDF-classic must paper over, and why pylibcudf exists to escape that tax.
If you remember one framing

cuDF is a relational/analytical engine whose performance comes from a columnar Arrow layout, a compact set of world-class data-parallel primitives, a type-dispatch mechanism to apply them generically, and an allocator (RMM) fast enough to make its allocate-heavy functional style cheap — all wrapped in a pandas-shaped API, with two alternative front-ends (zero-code pandas, and a Polars GPU engine) sharing the same libcudf core.
