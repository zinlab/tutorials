MAP Chapter I

The engineering loop · code → cluster.

The premise Five surfaces of engineering discipline. Each one is independently necessary; together they form the loop that converts a Git commit into a production change without a human in the path.

The rule Automate the boring parts. Test the dangerous parts. Make the manual parts impossible.

There are five surfaces of engineering discipline that determine whether a team ships safely at scale: orchestration (how work runs), CI/CD (how it gets built and verified), GitOps and deployment (how it reaches the cluster), testing (how regressions get caught), and developer tooling (the IDE-to-CI loop that holds everything together). Each is independently necessary; all five must work for production velocity to be safe.

The five surfaces, in order of feedback latency

Developer tooling (sub-second feedback). Linter as-you-type, type checker on save, format on save. The fastest feedback loop. If broken, every other surface gets worse because bad code reaches CI.

Local tests (seconds). uv run pytest in the dev loop. Fast tests run on every save; slow tests run on demand. Should catch ~80% of regressions before push.

CI/CD (minutes). Lint, type-check, unit tests, build image, integration tests, security scan, sign. Gates everything else. Should run in under 10 minutes for normal-sized services.

GitOps deployment (minutes to hours). Argo CD or Flux reconciles a Git repo to cluster state. The cluster's state is whatever the repo says. Promotion is a PR.

Orchestration runtime (hours to days). Argo Workflows, Kubeflow Pipelines, Airflow run the actual work — ML training, batch jobs, periodic ETL. Long-lived; observable; resumable.

The order of operations

The five surfaces aren't independent. They layer:

  1. Developer tooling first. Without fast local feedback, every subsequent surface is slower.
  2. Tests second. Without tests, CI is just a build step. Without integration tests, deployments are guesses.
  3. CI/CD third. Once you have tests, CI gates them. Once you have artifacts, CI signs them.
  4. GitOps fourth. Once artifacts are signed and stored, GitOps moves them to clusters.
  5. Orchestration fifth. Once services are deployed, orchestrators run the actual work that consumes them.
The four-second test

The single most useful question to ask about engineering discipline: how fast does feedback reach the developer when they make a mistake? If a typo takes 4 seconds (linter in editor): excellent. 40 seconds (local test): good. 4 minutes (CI): tolerable. 40 minutes (failed deploy): broken. 4 hours (incident in prod): broken. The remaining 14 chapters are the controls that keep your feedback latency in the low range.

ORCH Chapter II

Workflow engines · Argo, Kubeflow, Airflow.

The three contenders Argo Workflows (K8s-native, YAML DAGs), Kubeflow Pipelines (Python, ML-focused, K8s-native), Airflow (Python, broad ecosystem).

The shift Pure Airflow is fading for ML/training workloads. Argo or Kubeflow dominate where workloads are GPU-heavy and K8s-native.

Workflow engines orchestrate long-running, multi-step jobs: ML training, data pipelines, batch ETL, model evaluation, CI for non-trivial builds. The 2026 choice is between Argo Workflows (general-purpose, K8s-native), Kubeflow Pipelines (ML-focused, built on Argo), and Apache Airflow (Python-centric, broad ecosystem). Argo Workflows — the K8s-native DAG engine

Argo is the workhorse. Each step is a pod; the workflow controller is a CRD. Argo Workflows is built on top of Kubernetes, and each task is run as a separate Kubernetes pod. Many reputable organizations use Argo Workflows for ML, ETL, Data Processing, and CI/CD Pipelines. At scale, Argo's throughput is essentially bounded by your Kubernetes control plane, API-server quotas, and cluster autoscaling behavior rather than by an external worker pool.

apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  generateName: train-sweep-
spec:
  entrypoint: sweep
  arguments:
    parameters:
      - { name: lrs, value: "[0.001, 0.003, 0.0003]" }
  templates:
    - name: sweep
      steps:
        - - name: prep
            template: prep-data
        - - name: train
            template: train-one
            arguments:
              parameters: [{ name: lr, value: "{{item}}" }]
            withParam: "{{workflow.parameters.lrs}}"     # fan-out
        - - name: pick-best
            template: select-best
            arguments:
              artifacts: [{ name: results, from: "{{steps.train.outputs.artifacts}}" }]

    - name: train-one
      inputs:
        parameters: [{ name: lr }]
      container:
        image: ghcr.io/myorg/trainer:v1.2.3@sha256:abc...
        command: [python, -m, trainer]
        args: ["--lr={{inputs.parameters.lr}}"]
        resources:
          limits: { nvidia.com/gpu: "8", memory: 128Gi }
      outputs:
        artifacts:
          - { name: ckpt, path: /out/ckpt.pt }                  # auto-uploaded to S3

Kubeflow Pipelines — Argo with ML batteries

Kubeflow is an end-to-end MLOps platform for Kubernetes, while Argo is the workflow engine for Kubernetes. KFP authors pipelines in Python; the compiler emits Argo YAML; the platform adds experiments, runs, comparisons, and integration with KServe (serving) and Katib (HPO).

from kfp import dsl
from kfp.dsl import Input, Output, Dataset, Model

@dsl.component(base_image="ghcr.io/myorg/trainer:v1")
def train(data: Input[Dataset], lr: float, model: Output[Model]) -> None:
    import torch
    # ... training logic ...
    torch.save(state, model.path)

@dsl.pipeline(name="sweep")
def sweep(lrs: list = [0.001, 0.003, 0.0003]) -> None:
    data = prep_data()
    with dsl.ParallelFor(lrs) as lr:
        train(data=data.outputs["data"], lr=lr) \
            .set_accelerator_type("nvidia.com/gpu") \
            .set_accelerator_limit(8)

# Compile to Argo YAML
from kfp import compiler
compiler.Compiler().compile(sweep, "sweep.yaml")

Airflow — when not to use it for ML

Airflow excels at data engineering: @dag/@task decorators, hundreds of integrations (Snowflake, BigQuery, Salesforce, SaaS APIs), strong scheduling UI, mature alerting. It struggles at the ML-on-K8s use case because Airflow relies on static scheduling and worker pools, which can limit massive parallelism.

The 2026 pattern is hybrid: Airflow for the data pipeline that feeds the training data; Argo/Kubeflow for the training itself; Airflow triggers Argo via the KubernetesPodOperator or a webhook. Use each where it's strong.

The orchestration-engine decision

K8s-native, ML-heavy, GPU workloads: Argo Workflows or Kubeflow Pipelines. Kubeflow if you want the platform features (experiments, KServe, Katib). Argo if you want lighter weight and you build the ML abstractions yourself.

Data engineering, many SaaS integrations: Airflow. The ecosystem of operators (S3, Snowflake, dbt, etc.) is unmatched.

Mixed workload: Airflow for orchestration + Argo for GPU training (Airflow triggers Argo). Airflow knows about your data; Argo knows about your cluster.

Greenfield, want to defer the orchestrator choice: ZenML or similar (ZenML uses pluggable orchestrators — it achieves the best of both worlds: Airflow's scheduling and Kubeflow's scalable execution. The benefit here is flexibility. If you start with Airflow and later want to move to Kubeflow, ZenML makes that easier. You swap out the orchestrator, and your pipeline code remains the same.).

The single most-useful Argo feature

withParam for fan-out. One workflow definition, N parallel pods, results aggregated. The same machinery that lets you sweep hyperparameters lets you fan out evaluations, regression tests, or backfills. If you're writing for-loops in shell scripts to launch K8s jobs, you should be writing Argo workflows instead.

CI/CD Chapter III

CI/CD · GitHub Actions, GitLab, Buildkite.

The 2026 norm Every commit runs a graph of jobs. The graph gates merge. The merge triggers a release. The release becomes a PR to the GitOps repo.

The discipline Fast lane (lint + unit) under 2 min. Full lane under 10 min. Nothing else is acceptable.

CI/CD is the bridge between code and cluster. The 2026 standard pattern: developer pushes a branch, CI runs lint + type-check + unit tests in parallel (under 2 minutes), then on PR merge runs build + integration tests + scan + sign + push (under 10 minutes), then the release job opens a PR against the GitOps repo bumping the image digest. Job graph, not job list

The naive CI is sequential: lint, then type, then test, then build. The right CI is a DAG with explicit dependencies. Lint and type-check have no dependencies; they run in parallel. Build and unit tests have no dependencies; they run in parallel. Integration tests need the built image and may need migrations to run first.

jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [{ uses: astral-sh/setup-uv@v3 }, { run: uv run ruff check . }]
  typecheck:
    runs-on: ubuntu-latest
    steps: [{ uses: astral-sh/setup-uv@v3 }, { run: uv run mypy --strict src/ }]
  unit:
    runs-on: ubuntu-latest
    steps: [{ uses: astral-sh/setup-uv@v3 }, { run: uv run pytest tests/unit -n auto }]
  build:
    runs-on: ubuntu-latest
    steps: [{ uses: docker/build-push-action@v5, with: { push: true, provenance: "mode=max" } }]
  integration:
    needs: [build]
    runs-on: ubuntu-latest
    steps: [{ run: ./scripts/integration-test.sh }]
  release:
    needs: [lint, typecheck, unit, integration]
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    steps: [{ run: ./scripts/open-gitops-pr.sh }]

The "fast lane" / "full lane" split

Branch CI runs the fast lane: lint + type + unit. Under 2 minutes. Optimized for "did I break something obvious." Main-branch CI runs the full lane: everything including integration, scan, sign, release. Optimized for "is this safe to deploy."

This means: open a PR, get feedback in 2 minutes. Approve and merge, full release pipeline runs. The two latencies are budgeted differently and optimized differently.

Self-hosted runners for GPU CI

For ML services that need GPU in CI (smoke-test a kernel, profile a model load), hosted runners don't work. Self-hosted GitHub Actions runners on K8s (via Actions Runner Controller) or Buildkite agents on GPU nodes solve this. Each PR can spawn a runner with H100s on demand, runs the test, terminates.

The CI budget

Branch CI under 2 minutes. Main CI under 10 minutes. Anything slower and developers wait for CI before context-switching, productivity drops. The two levers: parallelism (every independent job in parallel) and caching (Issue 06 Chapter III). Six lines of CI config save your team weeks of cumulative wait time per year.

CI/CD Chapter IV

GitOps · Argo CD & Flux.

The principle The cluster's state is whatever the Git repo says. The agent reconciles. Promotion is a PR.

The two tools Argo CD (UI-rich, app-centric). Flux (CLI-first, modular).

GitOps is the discipline of treating Git as the source of truth for cluster state. An agent inside the cluster watches a Git repo of YAML and reconciles cluster state to match. Promotion isn't kubectl apply; it's a Git commit. Rollback isn't a separate process; it's git revert.

The two-repo pattern

The mature pattern separates application source from deployment manifests:

# Repo A: myorg/myapp — application source
myapp/
├── src/                # the application
├── tests/
├── Dockerfile
├── .github/workflows/release.yml
└── pyproject.toml

# Repo B: myorg/deployments — GitOps manifests
deployments/
├── clusters/
│   ├── dev/myapp/values.yaml          # digest: sha256:abc...
│   ├── staging/myapp/values.yaml      # digest: sha256:abc...
│   └── prod/myapp/values.yaml         # digest: sha256:def... (older)
└── apps/
    └── myapp/                          # shared Helm chart or kustomization
        ├── Chart.yaml
        └── templates/

When the source repo's CI produces a new image, the release job opens a PR against the deployments repo bumping the digest in clusters/dev/myapp/values.yaml. The PR's own CI verifies the manifest is valid; it auto-merges. The cluster's agent sees the change and rolls out. Same digest promotes to staging via another PR (auto or manual); prod requires explicit approval.

An Argo CD Application

kind: Application

  name: myapp-prod
  namespace: argocd

  project: production
  source:
    repoURL: https://github.com/myorg/deployments

      valueFiles: [values.yaml]
  destination:
    server: https://prod-cluster.example.com
    namespace: myapp

    syncOptions:
      - CreateNamespace=true
      - ServerSideApply=true

Promotion patterns

Auto-promote dev. Every successful main-branch CI on the app repo opens a deployments-repo PR. Dev auto-merges. Developers see their change in dev within minutes.

Time-bake staging. After dev runs successfully for N hours with no alerts, a bot opens a PR promoting to staging. Auto-merge if all health checks green.

Manual prod. Staging-to-prod PRs are manual. Require explicit approval from CODEOWNERS. The PR includes a diff of what's changing and links to the new image's attestations.

Progressive delivery. Combine with Argo Rollouts or Flagger: prod PRs trigger a canary that's automatically aborted if SLO metrics degrade. The PR doesn't just say "deploy"; it says "begin progressive rollout."

Drift detection & self-heal

GitOps agents continuously reconcile. If someone kubectl edits a deployment in prod, Argo CD detects the drift and reverts within minutes (if selfHeal: true). The Git repo is the source of truth; the cluster is the projection.

This is uncomfortable at first ("I just need to debug something quick") and indispensable later ("how did this deployment get edited and why did no one know?"). The right escape hatch is a ConfigManagementPlugin or a SyncWindow that pauses reconciliation in a controlled way, not kubectl edit.

The cluster-as-projection mindset

Once GitOps is in place, "what's deployed in prod" is answered by a Git commit hash, not a cluster query. Every change has a PR; every PR has a reviewer; every reviewer sees the diff. Rollback is git revert. Audit is git log. The cluster becomes a derived artifact, not a state machine humans poke at.

CI/CD Chapter V

Helm · charts, values, releases.

What Helm is A templating engine + package manager for Kubernetes. Chart = template + default values + dependencies. Release = chart + override values + namespace.

What Helm is not A lifecycle manager. Helm renders YAML; it doesn't know how your app should behave at runtime.

Helm packages a Kubernetes application as a "chart" — a directory of YAML templates plus a values file. helm install renders the templates with values and applies the result. helm upgrade diffs and applies changes. Charts are versioned, dependency-aware, and distributable via OCI registries.

The chart anatomy

# helm create myapp produces:
myapp/
├── Chart.yaml              # metadata: name, version, appVersion, deps
├── values.yaml             # defaults (overridable)
├── values.schema.json      # JSON schema validating values
├── templates/
│   ├── deployment.yaml     # templated Deployment
│   ├── service.yaml
│   ├── ingress.yaml
│   ├── hpa.yaml
│   ├── _helpers.tpl        # reusable template snippets
│   └── NOTES.txt           # printed on install
└── charts/                 # bundled subcharts (if any)

Values schema — the often-skipped step

A values.schema.json file alongside values.yaml validates user-provided values at install time. Missing or wrong-typed values fail before any rendering happens. This is the single most-underused Helm feature.

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["image", "resources"],
  "properties": {
    "image": {
      "type": "object",
      "required": ["repository", "digest"],
      "properties": {
        "repository": { "type": "string" },
        "digest": {
          "type": "string",
          "pattern": "^sha256:[a-f0-9]{64}$"             # require digest, not tag
        }
      }
    },
    "replicas": { "type": "integer", "minimum": 1, "maximum": 100 }
  }
}

Helm in GitOps

The classic Helm CLI (helm install, helm upgrade) is imperative and stateful — it stores release state in K8s Secrets and you have to know to use it. The GitOps pattern is different: render and apply via Argo CD or Flux.

Argo CD's Application resource points at a Git repo containing a chart; Argo CD renders the chart with the values and applies the result. Flux's HelmRelease resource does the same. Both make releases declarative and Git-driven; neither uses helm install at runtime.

Library charts and umbrellas

Library charts hold reusable template snippets (the _helpers.tpl contents). Many services in one org share the same labels, probes, securityContext patterns — put them in a library chart and import.

Umbrella charts bundle multiple subcharts. myorg-platform as an umbrella chart with prometheus, grafana, loki, tempo as deps. One Helm install brings up the full observability stack.

The Helm value pattern

Three-layer values: chart defaults (values.yaml), per-env values (values-prod.yaml), per-instance overrides (helm install --set ...). The chart should be opinion-free; env values carry the opinion; instance overrides are escape hatches. Use values.schema.json to enforce structure. Pin image.digest in env values; reject tag-only images at install time.

CI/CD Chapter VI

Operators · CRDs & the reconcile loop.

The pattern Define a custom resource. Write a controller that watches it. Reconcile the cluster state to match the resource's spec.

The frameworks Kubebuilder / Operator SDK (Go), Kopf (Python), Java Operator SDK, kube-rs (Rust).

An Operator extends Kubernetes with new types of resources and the code to manage them. Where Helm renders YAML, an Operator runs forever — it watches the cluster, detects drift, performs backups, rotates credentials, scales replicas based on domain logic. Operators follow Kubernetes principles, notably the control loop. The operator pattern aims to capture the key aim of a human operator who is managing a service or set of services.

The two building blocks

The Operator pattern, introduced by CoreOS, relies on two key Kubernetes concepts: Custom Resource Definitions (CRDs) define new types of Kubernetes resources (e.g., PostgresCluster or KafkaTopic). Controllers watch for changes to those custom resources and take action to reconcile the state.

CRD: a schema definition. After applying a CRD, the K8s API server accepts and stores objects of that new kind. The schema is OpenAPI-validated; the API server rejects malformed requests just like for built-in types.

Controller: a long-running Go (or Python/Rust/Java) program that watches the API server for changes to the CRD and reconciles cluster state. Watches are event-driven; reconcile loops are idempotent.

The reconcile loop · watch → diff → act

Reconcile, in code

func (r *PostgresClusterReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    log := log.FromContext(ctx)

    // 1. Fetch the custom resource
    var cluster databasev1.PostgresCluster
    if err := r.Get(ctx, req.NamespacedName, &cluster); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    // 2. Compute what cluster state SHOULD be
    desiredSts := buildStatefulSet(&cluster)
    desiredSvc := buildService(&cluster)

    // 3. Diff against actual state; create or update as needed
    if err := r.upsert(ctx, desiredSts); err != nil {
        return ctrl.Result{}, err
    }
    if err := r.upsert(ctx, desiredSvc); err != nil {
        return ctrl.Result{}, err
    }

    // 4. Update status subresource — what's actually happening
    cluster.Status.ReadyReplicas = countReadyPods(ctx, r, &cluster)
    cluster.Status.Conditions = computeConditions(&cluster)
    if err := r.Status().Update(ctx, &cluster); err != nil {
        return ctrl.Result{}, err
    }

    // 5. Requeue periodically to re-check (drift detection)
    return ctrl.Result{RequeueAfter: 30 * time.Second}, nil
}

Three properties make this work: idempotency (Reconcile can run any number of times with the same result), level-triggered (act on current state, not on event history — if you missed an event, the next Reconcile still converges), and status separation (spec is desired, status is observed; never confuse them).

The status subresource

K8s differentiates between spec (user-written, desired state) and status (controller-written, observed state). They have separate API endpoints, separate RBAC. Always update status separately via r.Status().Update(); never via the main r.Update(). Two reasons: the API server is optimized for this split, and you avoid overwriting spec changes the user just made.

Finalizers and lifecycle

When the user deletes a custom resource, you may need to clean up external state (database snapshots, cloud resources, S3 buckets). Add a finalizer; reconcile checks for DeletionTimestamp and, if set, performs cleanup, then removes the finalizer. The object disappears only after the finalizer is removed.

The operator promise

An Operator codifies "how a human SRE manages this service" into a controller that runs forever, never sleeps, never forgets. The Prometheus Operator, the Strimzi Kafka Operator, the cert-manager controller, the External Secrets operator — these are how production K8s actually works. If your service has any non-trivial day-two operations (backups, failover, key rotation, scaling decisions), at some point you'll either write an operator or buy one. The discipline is recognizing which.

CI/CD Chapter VII

Helm vs Operator · the decision.

The framing Helm handles day-one (install, upgrade). Operators handle day-two (backup, scale, repair). Most production stacks use both.

The single test Does the workload need ongoing operational logic? Helm if no; Operator if yes.

The Helm-vs-Operator question gets asked weekly in K8s shops and answered badly more often than not. The right framing is what each tool covers in the application's lifecycle. Helm typically handles day-one operations (deployment and upgrades). Helm is about installation and updates. Operators cover the entire lifecycle, including operations, self-healing, and domain-specific intelligence.

The decision tree

Use Helm when: your application is stateless (use Helm — it handles templated deployment, version control, and environment configs without needing complex automation); you need package distribution and templating; the workload is simple enough that kubectl apply with some templating suffices; you want fast iteration without writing Go.

Use an Operator when: your application is stateful or distributed (use an Operator — these workloads need lifecycle control: backups, replica sync, safe scaling, and failover); you need ongoing reconciliation (drift detection, automatic recovery); you have domain-specific logic that humans repeatedly perform; you're shipping a product others install.

Use both when: you want the rendering and packaging of Helm AND the operational logic of an Operator. Helm and Operators aren't mutually exclusive. In fact, many teams use both, each handling different layers of the stack. A common pattern looks like this: Helm deploys stateless apps (e.g., APIs, web frontends, jobs). Operators manage infrastructure and services with lifecycle needs (e.g., databases, service meshes, certificate managers). This split keeps things clear: Helm handles the fast-moving app layer, while Operators manage the complex, stateful dependencies.

The classic combos

Prometheus stack. Install the Prometheus Operator via a Helm chart (kube-prometheus-stack). The chart provides one-shot install of the operator + CRDs + default Prometheus/Alertmanager/Grafana. The operator then runs the ongoing reconciliation. Helm for day-one; Operator for day-two.

Cert-manager. Install via Helm (the operator + CRDs). Use Certificate custom resources via your own Helm charts (or directly via app manifests) to request certificates. cert-manager-the-controller renews, reissues, rotates.

ArgoCD-managed Helm. ArgoCD as the operator that renders and applies Helm charts. The chart is the package; ArgoCD is the lifecycle manager. Flux supports HelmRelease CRs, allowing teams to manage Helm charts declaratively via GitOps.

The "I just want to deploy my app" case

For 80% of services in a typical org — stateless APIs, web frontends, simple workers — Helm + GitOps is enough. Don't write an Operator. The two-week investment in writing an Operator for something Helm can handle is two weeks not spent on actual features.

The reverse trap: trying to express stateful logic in Helm templates. Helm has no runtime; templates are pure text rendering. If you find yourself writing Helm hooks for ordered upgrades, or YAML conditionals that branch on cluster state, you've outgrown Helm.

When to ship an Operator

If you're shipping software others run, the question becomes commercial. An Operator absorbs operational complexity into your code, not your users' runbooks. Users with an Operator can run your software with less expertise — they declare intent (kind: MyAppCluster) and the controller handles the rest.

This is why nearly every CNCF database (PostgreSQL via Crunchy/CloudNativePG, MongoDB Community, Apache Cassandra via K8ssandra, ClickHouse, Kafka via Strimzi, etc.) ships an operator. The bar for users is "kubectl apply"; the operator handles backups, scaling, failover, version upgrades.

The pragmatic order

Start with raw YAML. Outgrow it; switch to Helm. Outgrow that (need lifecycle ops); add an Operator that consumes your Helm-rendered configs. Don't skip levels. Every "we should write an Operator for this" should first ask "did Helm fail us here, or do we just dislike YAML?"

TEST Chapter VIII

The test pyramid · unit → e2e.

The pyramid Many fast cheap tests; few slow expensive tests. The shape isn't arbitrary; it's an investment portfolio.

The new tier For ML: evaluation harnesses. Quality gates that test model behavior, not just code behavior.

The classic test pyramid says: lots of fast unit tests, fewer integration tests, fewer end-to-end tests. The shape is an investment portfolio — most of your testing time goes into the cheapest, fastest, most reliable tier, with progressively more expensive tiers on top for things only they can verify.

What each tier is good at

Unit tests catch logic bugs at the function level. Sub-millisecond; thousands per service; coverage gates live here. The goal is fast feedback on whether a function does what it claims.

Component tests verify that a module's contract holds — an API endpoint returns the right shape, a worker processes a job correctly, a parser handles malformed input. Use in-memory fakes; no real network or database. Hundreds per service.

Integration tests verify that components talk to real dependencies correctly. Real Postgres via testcontainers. Real K8s API via envtest or KIND. The "we mocked X and it lied" bugs only show up here. Tens per service.

E2E / eval tests verify that the whole system does what the user wants. For services: smoke test against a deployed staging environment. For ML: golden-set evaluations, regression tests on output quality, latency budgets. Few per service; slow; expensive to maintain.

The anti-patterns

Inverted pyramid. Lots of E2E tests, few unit tests. Slow CI, flaky tests, low coverage of edge cases. Common in teams that started without test discipline.

Hourglass. Lots of unit, lots of E2E, nothing in between. Misses the integration-bug class entirely. Symptom: "all tests pass but the deployment is broken."

Cube. Equal numbers at every tier. Slow CI without commensurate coverage gain.

Mock everything. Unit tests that mock so much they test only the mock framework. Symptom: refactoring breaks tests even when behavior is unchanged. The fix: smaller surface to mock, bigger contract to test.

The ML-specific tier

For ML systems, the E2E layer is split: integration E2E (does the inference server return well-formed responses?) and evaluation (does the model give correct/good answers?). The latter is its own discipline:

  • Golden-set regression. A frozen evaluation set with expected outputs (or output qualities). New model passes only if it doesn't regress on the golden set.
  • LLM-as-judge. Use a stronger model to grade your model's outputs on a held-out prompt set. Score gates the release.
  • Behavioral tests. Verify specific properties: refuses to answer X, follows instructions for Y, citation format matches Z. CheckList-style suites for NLP.
  • Latency & cost SLOs. TTFT, TPOT, tokens/sec/GPU all have numeric gates. Regression in throughput fails the release.
The pyramid as portfolio

Think of tests as an investment portfolio. You want most of your money in the highest-yield, lowest-risk asset (unit tests: fast, cheap, reliable). You want some in higher-risk assets for coverage that the safe asset can't provide (integration, E2E). You want very little in the speculative tier (manual exploratory testing, smoke tests). The shape isn't aesthetic; it's economics.

TEST Chapter IX

Unit testing · pytest, fakes, property tests.

The framework pytest. Has been the right answer for ten years; remains so.

The two superpowers Fixtures (composable setup) and parametrize (one test, many cases). Master these and you're 90% there.

pytest is the Python testing framework. It replaced unittest for new code circa 2018 and hasn't been seriously challenged since. The two reasons: fixtures (dependency-injection for tests, composable, scoped) and parametrize (one test definition expanded into many test cases).

The minimum that should be in every repo

import pytest

Parametrize — the table-driven test

import pytest
from myapp.parsing import parse_duration

@pytest.mark.parametrize("raw, expected_seconds", [
    ("30s", 30),
    ("5m", 300),
    ("1h", 3600),
    ("2h30m", 9000),
    pytest.param("invalid", 0, marks=pytest.mark.xfail(strict=True)),
])

    assert parse_duration(raw) == expected_seconds

Fakes > mocks for non-trivial cases

unittest.mock creates objects that have any method. They're easy to write but lie cheaply: tests pass even when the real interface changed underneath. Fakes are real implementations of an interface, simpler than the production one — an in-memory database that implements the same interface as your real database. Tests against fakes don't lie; refactors don't silently break tests.

from dataclasses import dataclass, field
from typing import Protocol

class UserRepo(Protocol):
    async def get(self, user_id: str) -> User | None: ...
    async def save(self, user: User) -> None: ...

# Production: backed by Postgres
class PostgresUserRepo:
    async def get(self, user_id: str) -> User | None:
        return await self.conn.fetchrow(...)
    ...

# Test: in-memory fake with same shape
@dataclass
class FakeUserRepo:
    users: dict[str, User] = field(default_factory=dict)

    async def get(self, user_id: str) -> User | None:
        return self.users.get(user_id)

    async def save(self, user: User) -> None:
        self.users[user.id] = user

# Both satisfy UserRepo; the test against the fake is fast AND honest

Property-based testing

Some properties hold for ALL inputs, not just the ones you thought to write. Hypothesis generates random inputs and tries to falsify them.

from hypothesis import given, strategies as st
from myapp.serialization import dumps, loads

# Property: serialize then deserialize returns the same object
@given(st.dictionaries(keys=st.text(), values=st.integers()))
def test_roundtrip(d: dict[str, int]) -> None:
    assert loads(dumps(d)) == d

# Hypothesis discovers edge cases you'd never write:
#   - empty dict
#   - keys with surrogate codepoints
#   - keys that are valid Unicode but break your JSON encoder
#   - integers near MAX_SAFE_INTEGER

Async tests

# pyproject.toml
[tool.pytest.ini_options]
asyncio_mode = "auto"          # every test is async-aware; no @asyncio mark needed

# Then:
async def test_concurrent_fetches(client) -> None:
    results = await asyncio.gather(
        client.fetch("/a"),
        client.fetch("/b"),
        client.fetch("/c"),
    )
    assert len(results) == 3
    assert all(r.status_code == 200 for r in results)

Coverage as a gate, not a goal

Coverage % is a leading indicator at low values (under 60% means real gaps) and a noise metric at high values (95% vs 98% rarely matters). Set a floor (70-80% for new code) and stop optimizing for it. Use pytest-cov with --cov-fail-under=80.

Better than chasing percentages: mutation testing with mutmut. Modifies your code to introduce bugs; if your tests still pass, your tests are incomplete. The actual quality signal coverage tries to approximate.

The minimum-viable pytest setup

pytest + pytest-cov + pytest-xdist (parallel) + pytest-asyncio + hypothesis. Add pytest-mock if you must but prefer real fakes. Configure in pyproject.toml under [tool.pytest.ini_options]. Run with uv run pytest -n auto --cov=src --cov-fail-under=80. That's the whole minimum-viable harness.

Integration testing · testcontainers, envtest.

The premise Test against real dependencies, not mocks. Containerize the dependency; tear down after.

The K8s case envtest spawns a real kube-apiserver + etcd locally. KIND spins up a full cluster in a container.

Integration tests verify that your code works with real dependencies. The 2026 standard is to spin up a real Postgres / Redis / Kafka / OPA / S3-compatible store as a container per test class, exercise the code, tear down. testcontainers makes this a 3-line addition to a pytest suite.

testcontainers — real databases in tests

import pytest
from testcontainers.postgres import PostgresContainer
from myapp.repository import UserRepository

@pytest.fixture(scope="session")
def postgres():
    with PostgresContainer("postgres:16-alpine") as pg:
        yield pg.get_connection_url()

@pytest.fixture
async def repo(postgres):
    # Each test gets a clean schema; reuses the session-scoped container

async def test_user_insert_and_fetch(repo: UserRepository) -> None:

The Postgres container starts once per test session (slow setup, ~3 seconds), shared by all tests. Each test gets a clean schema via the function-scoped fixture. ~30 ms per test after the first.

Testing K8s controllers — envtest

kubebuilder will create testing scaffolds for all API and controller it creates, named as _test.go. It uses gomega as testing framework, and uses controller-runtime/pkg/envtest as to bring up a testing environment. During testing of controller, it will bring up an actual kube-apiserver and etcd done by envtest, and doing the testing. No real K8s nodes; just the API surface. Fast (under 5 seconds to start) and faithful for testing CRD validation, reconciler behavior, and watch handling.

Contract testing

For service-to-service interactions, integration tests would require deploying every consumer alongside the producer. Contract tests (Pact, Schemathesis) decouple this: the producer publishes a contract (OpenAPI / Pact spec); consumers test against the contract; producer tests its implementation matches the contract.

This is the right pattern for microservices where you don't want every service's CI to depend on every other service being deployable.

The integration-test budget

Integration tests are expensive. They're slow (seconds to minutes per test), flaky if not careful, and brittle when dependencies change. Budget rules:

  • Total integration time under 5 minutes for branch CI; under 10 for main.
  • Each test must clean up after itself (truncate tables, delete namespaces).
  • Use session-scoped fixtures for expensive setup; function-scoped for state.
  • Retry only deterministic flakes (e.g., port-bind races); fail loud on test logic flakes.
  • If a test fails, dump the dependent service's logs and state automatically.
The "real dependency or fake" call

Real dependency when: the dependency has non-trivial behavior you're testing (query planner, lock semantics, network protocol quirks). Fake when: the dependency is a simple data store and you're testing your code, not its code (use a fake repository instead of testcontainers Postgres). The wrong calls: real dependency for trivial cases (slow CI), fake for complex cases (false confidence).

E2E & ML eval harnesses.

For services A handful of smoke tests against deployed staging. The "is the deploy actually working" gate.

For ML Eval suites. Golden sets, behavioral tests, LLM-as-judge. Quality gates that resemble engineering gates.

E2E tests are few, slow, and irreplaceable. They verify the property no other tier can: that the whole thing works in something resembling production. For ML, this tier expands into evaluation harnesses — the gating mechanism that prevents quality regressions from shipping.

Service E2E — smoke against staging

The deploy-time pattern: after a release lands in staging, run a tiny suite of "the system is alive and behaving" tests. Real HTTP against the staging URL. Real auth. Real downstream services. Failure rolls back the deployment.

import httpx
import pytest

@pytest.fixture(scope="session")
def client():

def test_p99_latency(client: httpx.Client) -> None:

    for _ in range(100):

        client.post("/v1/predict", json={"input": "warmup"})

Putting eval into CI

# .github/workflows/eval.yml
name: model-eval
on:
  pull_request:
    paths:
      - 'models/**'
      - 'evals/**'
jobs:
  golden-set:
    runs-on: [self-hosted, gpu, h100]
    steps:
      - uses: actions/checkout@v4
      - name: Pull base model
        run: huggingface-cli download ${{ vars.BASE_MODEL }}
      - name: Apply finetune from this PR
        run: ./scripts/apply-finetune.sh
      - name: Run eval suite
        run: |
          uv run python -m evals.run \
            --golden-set evals/golden/v3.jsonl \
            --output results.json
      - name: Compare against baseline
        run: |
          uv run python -m evals.compare \
            --baseline evals/baselines/prod.json \
            --candidate results.json \
            --fail-on-regression-pct 2.0             # 2% drop fails the PR
      - if: failure()
        uses: actions/upload-artifact@v4
        with: { name: eval-results, path: results.json }

Eval-as-code discipline

Treat your eval suite like your application code:

  • Versioned. Golden sets are committed to the repo with their date and provenance. v3 of the golden set is a different artifact than v2.
  • Reviewed. PRs that change the eval set get extra scrutiny — you don't want a PR to silently weaken the gate.
  • Reproducible. Same seed, same model, same data → same scores. Otherwise you can't tell signal from noise.
  • Triaged. A regression on the eval suite is not a "rerun and hope"; it's a Jira ticket with a root cause.
The eval discipline

For ML services, the eval suite IS the regression suite. A model release without an eval gate is a deploy with no test coverage. Build the harness early; budget for it like you budget for unit tests. Three months in, the time you spend on evals will pay back twentyfold the first time a "small refactor" silently degrades quality on a benchmark you weren't watching.

PYTHON Chapter XII

Python tooling · uv, Ruff, mypy.

The 2026 default stack uv for environments and dependencies. Ruff for lint+format. mypy or ty for types.

Why this matters The old stack (pip + venv + black + isort + flake8 + pylint) is 10 tools and 10 configs. The new stack is 3 tools sharing one config.

Python's developer toolchain went through a phase change in 2024-2026. The old stack — pip + venv + pyenv + pip-tools + black + isort + flake8 + pylint + bandit + mypy — has been collapsed into three tools that share one config file. For most new projects, the cleanest default stack is: uv for Python installation, environments, dependency management, locking, and command running. Ruff for linting and formatting. Ty for type checking. The migration is a weekend project; the payoff is permanent.

Interactive · The three-tool stack

The migration math

Migrating from pip+venv+black+isort+flake8 to uv+ruff is roughly 15-30 minutes per repo. Two thirds of that is browsing Ruff rules to decide which to keep. The payoff: 10-100x faster installs, milliseconds-instead-of-seconds linting, one config file instead of five, no more "did you activate the venv" conversations. The hardest part is the social one — convincing the team to stop using their muscle memory.

Python services · FastAPI, Pydantic, async.

The default stack FastAPI for HTTP. Pydantic for data. Async by default. Structlog for logs. OpenTelemetry for traces.

The principle Type-safe at the boundaries; async at the request handlers; observable everywhere.

For Python services in 2026, the boring choice is the right choice: FastAPI for HTTP, Pydantic for input/output validation, async I/O at the request handler, OpenTelemetry for tracing, structlog for logs. Each piece is mature, well-typed, and integrates with the others without surprises.

The minimum-viable FastAPI service

from contextlib import asynccontextmanager
from fastapi import FastAPI, Depends, HTTPException
from pydantic import BaseModel, Field
import structlog

log = structlog.get_logger()

# --- request/response models ---
class PredictRequest(BaseModel):
    text: str = Field(..., min_length=1, max_length=8192)
    temperature: float = Field(0.7, ge=0.0, le=2.0)
    max_tokens: int = Field(512, ge=1, le=4096)

class PredictResponse(BaseModel):
    output: str
    tokens_used: int
    model_version: str

# --- lifespan: startup & shutdown ---
@asynccontextmanager
async def lifespan(app: FastAPI):
    app.state.model = await load_model()
    log.info("model.loaded", version=app.state.model.version)
    yield
    await app.state.model.close()

app = FastAPI(lifespan=lifespan, title="prediction-api")

# --- health endpoints ---
@app.get("/healthz")
async def healthz() -> dict[str, str]:
    return {"status": "ok"}

@app.get("/readyz")
async def readyz(app: FastAPI = Depends()) -> dict[str, str]:
    if not app.state.model.is_ready():
        raise HTTPException(status_code=503)
    return {"status": "ready"}

# --- the actual endpoint ---
@app.post("/v1/predict", response_model=PredictResponse)
async def predict(req: PredictRequest) -> PredictResponse:
    try:
        result = await app.state.model.generate(
            text=req.text,
            temperature=req.temperature,
            max_tokens=req.max_tokens,
        )
    except ModelOverloadedError as e:
        log.warning("model.overloaded", queue_depth=e.queue_depth)
        raise HTTPException(status_code=503, detail="overloaded") from e

    return PredictResponse(
        output=result.text,
        tokens_used=result.tokens,
        model_version=app.state.model.version,
    )

Why Pydantic v2

Pydantic v2 (released 2023, the standard now) does input validation in Rust under the hood. 5-50x faster than v1. Same API ergonomics. The defaults match what services actually want: strict types, JSON schema generation, validators, computed fields. If you're still on v1, migrate — the speedup alone is worth a week.

Async, properly

Async Python is misunderstood. The simple version:

Use async for I/O-bound work where you'd otherwise block: HTTP calls, database queries, file reads. The event loop handles many concurrent in-flight operations.

Don't use async for CPU-bound work. The event loop only runs one thing at a time per process. Use asyncio.to_thread() or a process pool for CPU work.

Don't mix sync and async. requests.get() in an async handler blocks the event loop. Use httpx.AsyncClient. psycopg2 blocks; use asyncpg or psycopg v3 async. boto3 blocks; use aioboto3.

Observability — what every service must emit

# Structured logging — JSON output, K8s metadata picked up by Loki
import structlog
structlog.configure(
    processors=[
        structlog.contextvars.merge_contextvars,
        structlog.processors.add_log_level,
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)

# OpenTelemetry — traces auto-emitted from FastAPI & httpx
from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
from opentelemetry.instrumentation.httpx import HTTPXClientInstrumentor

FastAPIInstrumentor.instrument_app(app)
HTTPXClientInstrumentor().instrument()

# Prometheus metrics — /metrics endpoint, scraped by Prometheus
from prometheus_fastapi_instrumentator import Instrumentator
Instrumentator().instrument(app).expose(app, endpoint="/metrics")

# Result: every request emits a structured log + a trace span + metrics
# Total cost: ~3% latency overhead; visibility worth orders of magnitude more

The deploy-readable application

Three things every Python service must do that beginners often forget:

Run as the same user the image was built for. USER 65532 in Dockerfile; runAsNonRoot: true in podSpec. Cross-reference Issue 07.

Handle SIGTERM properly. K8s sends SIGTERM, waits terminationGracePeriodSeconds (default 30), then SIGKILL. Your service must finish in-flight requests, close DB pools, and exit cleanly. uvicorn does this; gunicorn does too. Make sure your shutdown handlers are wired.

Expose /healthz, /readyz, and /metrics. Three separate endpoints. healthz = process alive. readyz = ready to serve traffic. metrics = Prometheus scrape target.

The boring stack works

FastAPI + Pydantic + httpx + asyncpg + structlog + OpenTelemetry is the right stack for ~90% of Python services in 2026. It's boring; it's well-typed; it integrates without surprises; documentation is excellent; the entire ecosystem is async-first. Pick this and spend your design energy on the actual business logic, not on plumbing.
