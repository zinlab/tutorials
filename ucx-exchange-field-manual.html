00

Orientation & intent

In a massively-parallel query engine, a shuffle is the moment a query stage repartitions its output so the next stage can consume it: every producing task scatters its rows across N partitions, and every consuming task gathers the slice destined for it. It is the most network-intensive primitive in distributed SQL, and on CPUs Velox already has machinery for it.

The catch the README states up front: in Velox this exchange is only partially implemented. The framework defines the shape — a task ending in a PartitionedOutput operator feeding a task whose source is an Exchange operator — but the actual exchange server and the exchange-source object are deliberately left out of Velox core, to be supplied by the embedding system (Prestissimo supplies an HTTP one). This module supplies a different implementation, purpose-built for cuDF.

The problem with reusing the CPU pathwhy this exists

The stock exchange is built for row-oriented CPU vectors: it serializes rows into a wire format, segments them into pages, ships them over HTTP, then reassembles and deserializes on the far side. Run that against GPU-resident columnar data and every chunk pays a device→host copy, a serialize, a network hop, a host→device copy, and a deserialize. The compute is on the GPU; the data tax is paid on the CPU and the PCIe bus.

The UCX Exchange collapses that. As the README puts it, no serialization or deserialization is needed — only packing — and the data is never segmented and reassembled. A partition is laid out once as a single contiguous device buffer and that buffer's bytes are RDMA'd, GPU to GPU. The intent is bandwidth: keep columnar data in its native GPU layout from the producing kernel to the consuming kernel.

The thesis in one line

Treat a shuffle chunk as one opaque GPU blob plus a small host-side layout descriptor. Move the descriptor on the control plane, RDMA the blob on the data plane, and reconstruct a table_view over the received bytes with zero further copies.

01

Conceptual primer

Four vocabularies meet in this directory. You need a working model of each before the code reads cleanly.

cuDF packing & contiguous splitwhat is moved

A cuDF table is a set of columns, each with its own device allocations (data, validity bitmask, and for nested types, children and offsets). That scatter of allocations is great for compute but terrible for transport. cudf::contiguous_split solves it: given a table and a set of row offsets, it produces, for each partition, a packed_columns — a pair of (host metadata, single device buffer). The metadata is a compact CPU-side description of where every column lives within that one buffer; the device buffer is all the column bytes concatenated contiguously. cudf::pack is the one-partition case; cudf::unpack reverses it, rebuilding a table_view that points into the buffer without copying.

Key property

After contiguous_split, a whole partition is exactly one rmm::device_buffer plus a few hundred bytes of host metadata. That single base pointer and single length is the entire reason a GPU shuffle can be zero-copy.

UCX, UCXX, and the UCP/UCS splithow it moves

UCX (Unified Communication X) is a transport abstraction that picks the best available path between two endpoints — shared memory, CUDA IPC, NVLink, PCIe, InfiniBand, RoCE, plain TCP — behind one API. UCP is the high-level protocol layer (tag matching, active messages, RMA, streams); UCS is the services layer (status codes, memory types, config). UCXX is the modern C++ object wrapper RAPIDS uses: Context → Worker → Endpoint, with requests returned as shared_ptr<ucxx::Request> carrying completion callbacks. This module is written against UCXX, dropping to raw UCP handles only where it must (the active-message acceptor).

  • Tag matching — tagSend(buf,len,tag) rendezvouses with tagRecv(buf,len,tag,mask); UCX matches them by a 64-bit tag. Two peers can have many independent transfers in flight, demultiplexed purely by tag. The whole data path is built on this.
  • Active messages (AM) — a sender pushes a small message that triggers a registered callback on the receiver with no pre-posted receive. Used here exactly once per connection: the handshake.
  • Rendezvous — for large transfers UCX negotiates, then does a single zero-copy DMA. The sender's tagSend does not complete until a matching tagRecv is posted — a property this design quietly weaponizes for backpressure.

RMM, streams, and the sync boundarywhen it's safe

cuDF allocates through RMM and runs kernels on CUDA streams. A stream is an ordered queue of async GPU work; results are only guaranteed materialized after stream.synchronize(). This matters at the network edge: UCX is not stream-aware. Hand UCX a device pointer whose producing kernels are still queued and it may DMA stale bytes. Every send site synchronizes the stream before the buffer leaves; every receive site syncs after allocating. The stream is even carried alongside the data (PackedTableWithStream) so downstream cuDF ops reuse it.

The Velox operator contractwhere it plugs in

Velox runs a pipeline of Operators driven by Driver threads. Each implements addInput, getOutput, isBlocked, isFinished; backpressure is a returned BlockingReason plus a ContinueFuture the scheduler waits on. This module provides two operators — UcxPartitionedOutput (sink) and UcxExchange (source) — honoring that contract exactly, so the existing scheduler, split machinery and stats plumbing keep working unchanged.

03

Transport core

One Communicator object per process owns one UCXX worker, and every UCX call in the whole module happens on that worker's single run() thread. Dozens of Velox driver threads never touch UCX directly — they hand work across through a queue. This section builds up why that shape is almost forced, starting from the one UCX concept the rest depends on.

First, the one idea everything rests on: progressthe UCX engine

UCX is asynchronous. When your code asks to send or receive — tagSend, tagRecv, amSend — that call does not move any bytes. It writes down what you want (a buffer, a 64-bit tag, a callback to run when the operation finishes) and returns immediately. The operation is now posted, but completely inert. Nothing has happened on the network yet.

The thing that actually drives posted operations forward is a single function: worker->progress(). Calling it hands the worker a slice of CPU to do whatever is currently possible — poll the network for arrived data, match a posted receive against an incoming send, copy bytes into your buffer, and, when an operation finishes, run the completion callback you registered for it. Think of progress() as turning a crank: the machine only moves while you're turning it.

Two consequences fall out of this, and the entire rest of this section is downstream of them:

  • If nobody calls progress(), nothing ever finishes. A posted receive sits there forever no matter what arrives on the wire, because matching and completing it is work that only happens during a progress() call.
  • Your completion callbacks run inside progress(). onData, onMetadata, sendComplete, the handshake handler — every one is invoked from within some progress() call, on whatever thread made that call. This single fact is behind the re-entrancy rule we hit later.
The idea underneath everything · posting an operation is not progressing it

Why a single thread owns the workerand what an FSM is

A UCX worker can be progressed by several threads at once — but only if you create it in UCS_THREAD_MODE_MULTI, which makes the worker take an internal lock on every operation and brings a long list of subtle hazards. This module makes the opposite choice: exactly one thread ever calls progress(), or any other UCX function. That thread is the Communicator's run() loop. Because UCX state is only ever touched from one place, there are no data races on it and no locks needed inside UCX — the design buys simplicity and speed while giving up nothing it actually needs.

But one thread has to serve many conversations at once: every transfer flowing into or out of this worker, possibly dozens at a time. Each conversation is written as a finite state machine — FSM for short. An FSM is just a small, fixed set of named states with rules for moving between them. A receiver, for example, walks Created → ReadyToReceive → WaitingForMetadata → WaitingForData → … → Done. There are two kinds here: the UcxExchangeServer (sender side) and the UcxExchangeSource (receiver side); §06 and §08 walk their states in full. For now, all that matters is: an FSM does its work one step at a time, and the method that takes one step is called process().

The actors, and the one channel between themdrivers vs the worker

So there are two populations of threads. On one side, Velox's pool of driver threads runs the query operators — they call addInput, getOutput, isBlocked. On the other, the lone Communicator thread owns the worker and is the only one allowed to speak UCX. The two must never touch UCX state at the same time, so they don't share it: a driver that wants something to happen on the network describes the work and drops it into a thread-safe queue, and the Communicator thread picks it up. Data flows back the same way, through the exchange / output queues. Those queues are the only thing crossing the boundary.

“Block until a network event or a wake-up” — what that actually means

Phase 4 is worth slowing down on, because it is two different ways of calling progress for two different situations.

While there is work to do, the loop uses the non-blocking form, worker->progress(): do whatever is ready right now and return at once, even if that's nothing. In a tight loop that would spin the CPU at 100% — fine for the short burst while draining work, wasteful when idle. So once everything is drained — the work queue empty and every operation just waiting on the network — the loop switches to the blocking form, progressWorkerEvent(0), which puts the thread to sleep until there is genuinely something to do. An idle worker then costs ~0% CPU instead of spinning.

So what wakes it? Two kinds of event:

  • A network event — a packet arrives, a connection request comes in, an operation the worker is watching becomes ready. The operating system can wake a sleeping thread when the worker's underlying file descriptor becomes readable; that's what the blocking wait listens for.
  • An explicit wake-up — the subtle one. Suppose the Communicator thread is asleep, and meanwhile a different thread (a Velox driver) pushes a new element onto the work queue. The sleeper has no idea: it is waiting on network events, and a queue insertion is not a network event — it would sleep right through the new work. The fix is worker->signal(), a call any thread can make to forcibly rouse the sleeping worker. Under the hood it pokes the very same notification the blocking wait is watching (an eventfd-style nudge), so the worker wakes, loops around, sees the new queue item, and runs it. That is what “an explicit wake-up” is: signal() — called by addToWorkQueue whenever a driver hands over work, by the deferred-cleanup path, and by stop() when the loop must exit. Every place that gives the thread something new to notice also signals it, so it never sleeps through work.

What process() does, and why elements go dormantWorkQueue

process() takes one FSM from its current state to its next. It looks at the state, performs the single action that state calls for — in ReadyToReceive, post the tagRecv for the next chunk's metadata; in DataReady, send the data — and usually updates the state. It is deliberately written to do one step and return, never to loop or block inside. That's what lets one thread interleave many FSMs: step element A, progress, step element B, progress, come back to A.

The work queue therefore has a precise meaning: it holds exactly the elements that have work to do right now. A driver (or a callback) puts an element in by calling addToWorkQueue, which also signals the worker:

That meaning is what drives the two behaviors people find confusing:

  • After process() runs, if the element still has immediate work — it just moved to a state whose next step can run without waiting — it puts itself back with addToWorkQueue(self), to be popped and stepped again. This is what “re-enqueue itself” means.
  • But if the element's next step is to wait for the network — it just posted a receive and now needs the data to arrive — it does not re-enqueue. It goes dormant: it simply stops being in the queue. There is nothing for the thread to do on its behalf until the network responds, so keeping it in the queue would only burn cycles re-checking a state that can't advance.

And here is the loop-back that makes the whole scheme work: what brings a dormant element back is its own completion callback. When the data finally arrives — during some later progress() call — UCX fires the element's onData, and that callback calls addToWorkQueue(self), returning the element to the queue so it can take its next step. A dormant element is re-awakened only by the very callback that fires inside progress(). Hold onto that — it is exactly why the deadlock below happens.

The deadlock: why progress after every itemthe rule decoded

Now the paragraph that's easy to read past. First, which progress() is this about? The loop calls progress in two places: the non-blocking one after each work item in phase 2, and the blocking one in phase 4. The rule is about the per-item progress in phase 2 — not phase 1 (endpoint cleanup), which posts no receives. Here is the failure, walked with that per-item progress removed:

  1. An element is popped and its process() runs. It's in ReadyToReceive, so it posts a tagRecv for metadata and goes dormant — it does not re-enqueue, because it's now waiting on the network. Remember: posting a receive is inert; nothing has progressed it yet.
  2. The drain loop moves on. Nothing else is pending, so the work queue empties.
  3. With the queue empty, the loop reaches phase 4 and goes to sleep in the blocking wait.
  4. Why does “never progressed” matter? Because an operation only advances — and its callback only fires — during a progress() call. This element is counting on its onData callback to wake it back up, and that callback can only fire from progress. No progress ran on the freshly-posted receive, so the callback never fires.
  5. Why does it “never re-enqueue itself”? Because the only thing that would re-enqueue a dormant element is that same onData callback. Callback never fires ⇒ addToWorkQueue(self) is never called ⇒ the element stays dormant, off the queue, indefinitely.
  6. Meanwhile the loop is asleep waiting for a new network event — but there may be none coming. The data the element wants might already be sitting in UCX's internal buffers (it arrived, or rode along with the rendezvous), needing only a progress() call to be matched and delivered. The blocking wait fires on new readiness; it will happily sleep through data that has already arrived but hasn't been progressed. That's the chicken-and-egg: the element won't re-enter the queue until its callback fires, the callback won't fire until progress runs, and progress won't run until something wakes the loop — which nothing will. The worker hangs.

Why does “progress after every item” break the cycle? Because the instant process() posts that receive, the very next line progresses the worker — giving UCX a chance to advance the receive right then. Often it completes immediately: the callback fires synchronously, the element re-enqueues, and the drain loop simply keeps going. And if the data genuinely isn't here yet, that progress call still arms the operation correctly, so the network event that completes it will properly wake the blocking phase. Either way the vicious cycle never forms. That's why it must be “progress after every item,” not “progress once at the end.”

The deadlock the “progress after every item” rule prevents

Why callbacks must defer their own cleanupre-entrancy

This subsection is just the re-entrancy rule from earlier, cashed out. Recall the two facts: completion callbacks run inside progress(), and endpoint->closeBlocking() — the call that tears an endpoint down — itself calls progress() internally (it has to drive the close handshake to completion). Put them together and a callback that called closeBlocking() would be calling progress from inside progress. That is re-entrancy, and UCX forbids it: re-entering the worker corrupts the very state machine you're in the middle of advancing — depending on the build, undefined behavior or an outright deadlock.

So when an endpoint's own close callback fires (UCX telling you “this connection just dropped”), it cannot clean the endpoint up there and then — cleanup needs closeBlocking(), which needs progress, which it's already inside. The only safe move is to write the work down and get out:

“Defer” just means: don't do the dangerous thing now; leave a note for the main loop to do it when it's safe. That note is drained in phase 1 of the run loop — outside any callback, where calling progress is fine. The diagram makes the forbidden back-edge and the safe deferral explicit.

Where callbacks come from — and why one path out of them is forbidden

The same re-entrancy explains an otherwise-odd choice nearby: endpointsMutex_ is a recursive_mutex — a lock the same thread is allowed to acquire more than once without deadlocking itself (a plain mutex would deadlock on the second acquire). Why is that needed? removeEndpointRef() takes the lock and then calls closeBlocking(); that progresses the worker; that progress can fire the listener callback for a brand-new incoming connection; and that callback also wants the same lock. Same thread, lock already held, second acquisition — precisely the case a recursive mutex exists for.

A close cousin of deferred cleanup guards a different danger: freeing GPU memory out from under UCX. While a transfer is in flight, UCX may be actively DMA-ing into or out of the device buffer. If the task is cancelled and the code frees that buffer immediately, UCX could still be mid-transfer — a use-after-free on the GPU. So a cancelled request isn't dropped; it's moved to deferredRequests_ and held alive there (which keeps its device buffer alive, since the request owns a reference to it) until isCompleted() finally reports that UCX has fully wound the operation down. Only then are the request and its buffer released — phase 3 of the loop. The lifecycle below shows the buffer pinned in a “do not free” state until that moment.

Connection on demand & endpoint sharingAcceptor · listenerCallback

The last piece is how the connections themselves are managed. An endpoint in UCX is a handle to a connection to one specific peer — the object you call tagSend/tagRecv/amSend on. This module builds no fixed mesh of endpoints up front; there is no static topology. A consumer's source opens an endpoint to a producer's listener only when it actually has data to fetch, and the producer's server object doesn't even exist until the handshake lands and the Acceptor's active-message callback spawns it. Connections, like the servers behind them, are created on demand:

To keep connections from multiplying, endpoints are pooled by remote host:port. Outgoing endpoints to a given peer are shared — five sources all pulling from the same remote node reuse one endpoint rather than opening five — while incoming endpoints (ones the peer opened to us) are not. The result is the tidy invariant in the diagram: between any two nodes there are at most two endpoints, one per direction.

Each shared endpoint is wrapped in an EndpointRef that tracks every comm element currently using it, and it does so with weak_ptrs so a finished server or source can be destroyed without the EndpointRef keeping it artificially alive. When the underlying connection drops, the EndpointRef fans the close out to all of them, so every FSM that depended on that connection is told to wind down. (As an aside, the peer's real IP is read from the connection request itself rather than trusted from the handshake payload — a safety fix covered in §13.)

Wire protocol

The protocol is deliberately thin: a per-connection handshake over active messages, then, for each chunk, a two-message tag-matched exchange — small metadata first, opaque GPU blob second. A 64-bit tag scheme keeps every concurrent transfer distinct.

The 64-bit tagUcxExchangeProtocol.h

Every send/recv carries a tag that encodes who, what, and which chunk. The top half identifies the producing task (so two tasks never collide), the next byte is the operation, and the low 24 bits are the chunk sequence number.

Receivers post tagRecv with TagMaskFull (exact match), so UCX delivers precisely the chunk and operation expected. This is what lets one endpoint carry many in-flight transfers without a multiplexing layer of the module's own.

The handshakeHandshakeMsg / HandshakeResponse

The first thing a source does after connecting is send a HandshakeMsg as an active message. It names the partition the source wants (taskId + destination) and carries the source's random workerId. The server replies with a HandshakeResponse whose single meaningful bit tells the source whether the two are in the same process — if so, the source abandons UCX entirely for the in-process fast path (§9).

The metadata recordMetadataMsg

Because the receiver cannot know a chunk's size in advance, each chunk begins with a metadata message that carries the cuDF layout descriptor, the exact dataSizeBytes to expect, and an atEnd flag that doubles as the stream terminator. The receiver reads it, allocates exactly that many device bytes, then posts the data receive. Serialization is hand-rolled memcpy in host byte order behind a magic number and a length-prefixed layout — with an explicit, documented assumption that both peers are little-endian.

Receiver pre-allocation

Metadata is received into a fixed kMaxMetaBufSize = 1 MB buffer (≈10,000+ columns); the sender writes only the exact bytes. A VELOX_CHECK on the serialized size turns "table too wide" into a clean error rather than a buffer overrun. The deserializer bounds-checks every field against totalSize.

05

Producer path

On the producing worker, the UcxPartitionedOutput operator turns a stream of GPU CudfVectors into per-partition packed_columns and drops them into queues. A server reads from those queues and ships them. The operator's job is partition, pack, enqueue, and apply backpressure.

Accumulate, then partitionaddInput / flushPending

Rather than partition every tiny input batch (expensive: each contiguous_split launches kernels and produces metadata), the operator buffers CudfVectors until it has roughly a target row count — 10,000 by default, matching the HTTP path's chunking — then concatenates and partitions the merged table once. The single-input case is a fast path that skips the concat allocation entirely.

Peak-memory discipline

When merging multiple inputs, the operator frees the input GPU buffers before partitioning, holding peak device memory near 1× the data rather than 2×. On rmm::bad_alloc it drops every queued partition for the task and rethrows — shedding GPU memory rather than wedging.

The producer-side queuesOutputQueueManager → OutputQueue → DestinationQueue

These mirror Velox's own buffer hierarchy so closely that the stats even masquerade as OutputBuffer::Stats to reach the Presto layer unchanged. The manager maps taskId → one UcxOutputQueue; that queue holds a vector of UcxDestinationQueues, one per partition; each destination queue is a FIFO of packed_columns with a single pending notify callback. The server registers that callback; when data arrives it fires, handing the chunk to the server.

Backpressure on this side is a classic water-mark: once queuedBytes_ > maxSize_, the operator's checkBlocked hands the scheduler a kWaitForConsumer future, and producers resume only when the queue drains back below 90% (kContinuePct). A subtle lifecycle wrinkle: if a server asks for data before the producing task has even initialized (a sink connecting first), the manager creates an uninitialized placeholder queue just to hold the callback, and a removedTasks_ set stops it from resurrecting placeholders for tasks that are already dead.

06

Server FSM · the sender

A UcxExchangeServer is spawned per connecting source. It is a small state machine that loops: pull one chunk from the destination queue, send its metadata, send its data, repeat — until the queue signals end, at which point it sends an atEnd marker and finishes.

UcxExchangeServer · ServerState transitions (remote path solid, intra-node dashed)

Pulling one chunk, then sending two messagessendData

In ReadyToTransfer the server registers a callback with the destination queue and goes to WaitingForDataFromQueue. When the callback delivers a chunk, the state advances to DataReady and process() calls sendData(), which serializes and tagSends the metadata, then tagSends the GPU buffer's raw device pointer. A null chunk means "queue is finished" → send a metadata-only atEnd and go Done.

On sendComplete with UCS_OK, the server logs throughput, increments the sequence number, releases the buffer, and returns to ReadyToTransfer for the next chunk — a tight pull-send loop driven entirely by queue callbacks and UCX completions.

The context-shell trick

Why wrap the buffer in a DataSendContext instead of passing it directly? Two lifetimes that must be decoupled. The ucxx::Request must outlive UCP's wireup-replay machinery (which can re-fire callbacks on completed requests), but the multi-megabyte GPU buffer should be freed the instant the DMA finishes. The callback moves the buffer out of the context — freeing device memory — while the now-empty context shell stays alive with the Request, costing nothing.

07

Consumer path

On the consuming worker, the UcxExchange operator pulls reconstructed tables from a UcxExchangeClient, which fans in many UcxExchangeSources — one per producing task — into a single shared queue. The operator's job is split discovery, dequeue, and wrapping bytes back into a CudfVector without copying.

Splits become sourcesgetSplits / addRemoteTaskIds

Velox feeds the exchange its upstream task list as splits. Driver 0 pulls them, and each new remote task id becomes a UcxExchangeSource. The ids are shuffled first so that, across a cluster, many consumers don't all stampede the same producer at once.

Fan-in and completion accountingUcxExchangeQueue

All sources write into one UcxExchangeQueue. It tracks numSources_ and numCompleted_; the exchange is at end only once noMoreSources_ is set and every source has delivered its terminating nullptr. Consumer drivers that find the queue empty register a promise keyed by their driver id; an enqueue wakes exactly as many as there are newly-available tables.

Bytes back to a CudfVector, no copygetOutputFromPackedTable

The operator's output step is the mirror image of packing: the received packed_table already is a table view over a device buffer, so the operator simply wraps it in a CudfVector, reusing the very stream the source allocated on. No deserialization, no copy.

The operator's isBlocked combines two futures — "more splits available" and "more data available" — with folly::collectAny, so a driver parks until either a new upstream task appears or a chunk lands, then re-checks. This is the standard Velox exchange dance, faithfully reproduced.

08

Source FSM · the receiver

The UcxExchangeSource is the largest single file in the module and the protocol's center of gravity. It drives one producing task's connection: connect, handshake, receive the response, then loop metadata → allocate → data → unpack → enqueue until atEnd. A whole second branch handles the in-process fast path.

Handshake, then the responsesendHandshake / onHandshakeResponse

From Created, the source associates (or opens) an endpoint, fires the handshake as an active message, and waits. Notice the weak_from_this() capture — the universal defense against the object being destroyed while a callback is in flight.

When the HandshakeResponse arrives, its single bit decides the source's whole future: if isIntraNodeTransfer is set, every subsequent chunk comes from the in-process registry; otherwise it's the UCX metadata/data loop.

Metadata → allocate exactly → receive dataonMetadata / onData

This is the crux of the two-message protocol. The source receives metadata, reads dataSizeBytes, allocates precisely that many device bytes through RMM, synchronizes, then posts the data tagRecv straight into that device buffer. The atEnd flag short-circuits to Done.

In onData, the received buffer and the metadata become a packed_columns; cudf::unpack rebuilds the table_view; the result is bundled with its stream and enqueued. Then back to ReadyToReceive.

Backpressure as dormancyReadyToReceive · resumeFromBackpressure

This is the cleverest mechanism in the file. When the consumer queue is overloaded, the source does not post the next receive and does not re-enqueue itself — it simply goes dormant. Because no tagRecv is posted, the server's next tagSend blocks at rendezvous: UCX itself becomes the backpressure valve. The consumer thread re-arms the source via a CAS-guarded wake-up once the queue drains.

A trap in the client's other "flow control"

The client keeps a second, maxQueuedColumns_-based notion of flow control — and its own comment admits it is a no-op: the protocol is push-based, so there is no "don't request" lever. The real backpressure is the dormancy above. The dead path is a maintenance hazard flagged in §14.

Intra-node fast path

When a source and a server live in the same process — common when many tasks share a node and GPU — going through UCX at all is wasteful: the data already sits in the same address space. The IntraNodeTransferRegistry hands the packed_columns over by shared pointer, bypassing the network stack entirely.

Detecting same-process via workerIdwhy not IP

Each Communicator mints a random 64-bit workerId at startup. The source sends it in the handshake; the server compares it to its own. A match means the same singleton, hence the same process. The code comments record exactly why this replaced an earlier IP-based check: multiple Docker containers can share the host IP, which made IP comparison wrongly conclude "same node" for peers that were actually isolated. A random id has negligible collision probability and no such confusion.

publish / poll handoffIntraNodeTransferRegistry

The mechanism is a keyed map plus a promise/future pair. The server publish()es the packed_columns under a (taskId, destination, seq) key and gets back a future that resolves when the source has taken the data; the source poll()s the same key, moves the data out, and fulfils the promise. Ownership passes by pointer — no copy, no UCX, no DMA.

On the server side this collapses the FSM: from DataReady it publish()es and parks in WaitingForIntraNodeRetrieve, polling its future with a zero timeout each time the work queue revisits it; when the source has taken the chunk, it loops. The source mirrors this from WaitingForIntraNodeData, re-polling the registry until the entry appears.

Two guardrails — and why they exist

Broadcast is excluded. In broadcast mode the same packed_columns is shared across every destination server; the intra-node source's destructive move would yank the buffer out from under the others, corrupting their sends. canUseIntraNode() disables the fast path for broadcast and for not-yet-initialized (placeholder) tasks.

It busy-polls. Both sides re-enqueue themselves and poll rather than block on the promise via the worker's wake-up. It is correct but burns Communicator-thread cycles — the single clearest efficiency target in the module (§14).

Problems & approaches

Seven design challenges define this module. Each card states the problem, the approach taken, what it buys, and where it still bites.

P1Moving columnar GPU data without a host bounce
Problem

A cuDF table is a scatter of independent device allocations. Shipping it the CPU way means D2H copy → serialize → network → H2D → deserialize, paying the PCIe tax on data that started and ended on the GPU.

Approach

contiguous_split lays a partition out as one device buffer + small host metadata. The server tagSends the raw device pointer; the receiver tagRecvs into a device buffer; unpack rebuilds a view. UCX's CUDA support does the GPUDirect DMA.

Benefit

Genuinely zero-copy on the data plane. No serialization, no segmentation, no reassembly. Bandwidth is bounded by the link (NVLink / IB / RoCE), not by CPU marshalling.

Limitation

One contiguous_split per chunk still costs a kernel + a transient peak allocation. No on-wire compression, so bandwidth-bound links carry raw bytes.

P2Unknown chunk size & dynamic, late-bound membership
Problem

The receiver can't know a chunk's byte size until the producer has partitioned, and the set of producer/consumer pairs is decided at plan time but materialises dynamically as tasks launch, retry, and tear down.

Approach

Connection-on-demand: a source connects when it has work, and an active-message handshake spawns the server. A two-message protocol sends metadata (with exact size) first, then the data — so the receiver allocates precisely and demultiplexes by 64-bit tag.

Benefit

No global barrier, no fixed world size, no pre-wired topology. Tasks come and go freely; each transfer is independent and self-describing.

Limitation

A metadata round-trip per chunk adds latency, and the full cuDF layout is re-sent every chunk even when the schema is unchanged. The taskHash is only 32-bit FNV-1a — a latent collision risk in long-lived clusters.

P3The CUDA-stream / network ordering boundary
Problem

UCX is not stream-aware. Hand it a device pointer whose producing kernels are still queued on a stream and it may DMA stale or partial bytes — a silent data-corruption bug.

Approach

stream.synchronize() at every boundary: after contiguous_split before enqueue, after device allocation before posting a receive, before an intra-node publish. The stream travels with the data (PackedTableWithStream) so downstream ops reuse it.

Benefit

Correctness at the edge with a single, well-placed sync. Downstream cuDF work inherits the right stream, avoiding spurious default-stream serialization.

Limitation

A hard synchronize() is a coarse hammer — it stalls the stream rather than enqueuing an event-based dependency. Under high concurrency this serializes more than strictly necessary.

P4Lifetime safety across async UCX callbacks
Problem

A task can abort mid-transfer, freeing the source/server while a completion callback is still pending. Worse, UCP's wireup-replay can fire callbacks on already-completed requests, and a freed Request means the callback lambda lives in freed memory.

Approach

A three-part discipline: every callback captures weak_from_this() and no-ops if the object is gone; every callback guards on expected state to ignore replays; completed requests are retained in a completedRequests_ vector so their lambdas stay valid for the object's lifetime.

Benefit

No use-after-free under cancellation or replay. The pattern is textbook-correct for UCXX and applied uniformly across handshake, metadata, and data sends/recvs.

Limitation

Retained requests accumulate for the object's lifetime (bounded, but non-zero memory). The discipline is manual and easy to get subtly wrong in any future code path that forgets the weak-ptr or the state guard.

P5Many driver threads, one UCX worker
Problem

UCX workers are happiest progressed by one thread; Velox runs many driver threads. Letting them all touch the worker means UCS_THREAD_MODE_MULTI locking and a long tail of concurrency hazards.

Approach

A singleton Communicator owns the only worker and runs the only progress loop. Driver threads interact solely through a mutex-guarded WorkQueue of CommElements, signalling the worker to wake. Both FSMs are re-entrant step functions.

Benefit

No worker-level lock contention and none of the multi-thread UCX foot-guns. A clean, debuggable single-threaded core; callbacks always run in one known context.

Limitation

That one thread does all progress and all process() calls — a throughput ceiling at high message rates or many concurrent endpoints. Scaling means sharded workers, reintroducing the complexity this avoids.

P6Backpressure without a pull protocol
Problem

The data path is push-based: the server sends unconditionally. With no "please pause" message, a slow consumer would let the queue grow without bound and OOM the GPU.

Approach

Use rendezvous as the valve. When the consumer queue exceeds the high-water mark, the source goes dormant and stops posting receives; the server's next tagSend then blocks at rendezvous. A CAS-guarded wake re-arms the source once the queue drains to the low-water mark. The producer side adds an independent byte-water-mark block.

Benefit

End-to-end flow control with no extra protocol messages — UCX's own rendezvous semantics do the throttling. Bounded queue depth, bounded GPU memory.

Limitation

Thresholds are counts of tables (32/16), not bytes, so a few huge chunks can still spike memory. A vestigial second "flow control" in the client is a documented no-op and pure confusion.

P7Co-located peers paying the network tax
Problem

When producer and consumer tasks share a process (common with many tasks per node/GPU), routing data through UCX is pure overhead — the bytes are already in the same address space.

Approach

A random per-process workerId in the handshake detects same-process peers. The IntraNodeTransferRegistry then hands the packed_columns over by shared pointer via a keyed publish/poll with a promise/future, bypassing UCX entirely.

Benefit

Near-free intra-process transfer — a pointer move, no DMA, no serialization. Robust to shared host IPs (the workerId fix) that broke the earlier IP-based detection.

Limitation

Busy-polling on both sides burns Communicator-thread cycles. Disabled for broadcast (shared buffer + destructive move would corrupt other destinations) and for not-yet-initialized tasks.

Why not NCCL

NCCL is the obvious GPU-comms library, so the absence of it here is a design statement. The short version: NCCL optimises the wrong axis for this problem. It is a collective library for dense, symmetric, fixed-membership communication; a shuffle is sparse, irregular, dynamically-membered point-to-point with a control plane. UCX is a transport toolkit; the shuffle needs a transport toolkit.

Why UCX is the right shape instead

Everything the shuffle needs falls out of UCX naturally. Connection-on-demand maps to UCX endpoint creation per peer. Sparse, independent transfers map to tag matching — many in-flight sends/recvs on one endpoint, demultiplexed by tag, with no collective barrier. The control plane (handshake, metadata) rides the very same worker as the data plane, using active messages and host-memory tags. Late sizing is solved by the module's own two-message protocol layered on tags. And UCX still gives full GPUDirect coverage of the interconnect, so the data plane is no slower than NCCL's point-to-point would be for these irregular, one-shot transfers.

Where NCCL would, in fact, win

This is not "NCCL is bad." For dense, repeated, symmetric collectives on a fixed topology — gradient all-reduce in data-parallel training, embedding all-to-all in recsys with a fixed world size, tensor-parallel all-gather — NCCL's topology-aware ring/tree algorithms, NVLink/NVSwitch exploitation, and in-network reduction (SHARP) decisively beat a hand-rolled point-to-point library. The Velox shuffle is simply not that workload: it is irregular, dynamically-membered, and one-shot per chunk. Using a collective library for it would mean rebuilding communicators constantly, emulating variable all-to-all with grouped Send/Recv whose sizes you don't yet know, and fighting NCCL's lockstep ordering — paying setup and synchronization costs to get a data plane no faster than what UCX already provides here.

One nuance worth flagging for the design's own future: the module currently uses only tag send/recv and active messages (its destructor comment notes flush() was removed because there is no RMA). For the most irregular shuffles, UCX one-sided RMA (receiver exposes a buffer, sender puts once it knows the remote address) or a rendezvous-style active-message data path could amortise away the per-chunk metadata round-trip. That is a UCX-internal optimisation — and precisely the kind of thing NCCL's model can't express at all.

Evaluation & space for improvement

An experimental module with a clear, correct core and a frank set of rough edges. The strengths are structural; the gaps are mostly latency, scale, and polish — several already flagged in the code's own comments.

What's strong

  • Genuinely zero-copy data plane. The contiguous_split → tagSend(device ptr) → tagRecv(device buf) → unpack path keeps columnar data in GPU layout end to end. This is the whole point and it lands.
  • Clean control/data plane split. Small host-side metadata and handshakes on one plane; opaque device blobs on the other. The 64-bit tag scheme demultiplexes concurrent transfers without a bespoke mux layer.
  • Correct async lifetime discipline. The weak_ptr + state-guard + request-retention triad is exactly right for UCXX and applied uniformly. The deferred-cleanup design respects UCX's "no progress inside callbacks" rule.
  • Single-threaded progress is the right default. It sidesteps the multi-thread UCX hazards and keeps the system debuggable; callbacks always run in one known context.
  • Faithful Velox integration. Operators, queues, backpressure, and stats mirror their CPU counterparts so closely that the existing scheduler and Presto-layer plumbing work unchanged — even the stats masquerade as OutputBuffer::Stats.
  • Two elegant tricks. Rendezvous-as-backpressure (no extra protocol messages) and the in-process registry (pointer-handoff for co-located peers) are both genuinely clever.

Scaling past one progress thread

The single progress thread is §03's defining choice, so it's fair to ask whether more threads would go faster. The answer splits sharply on what “more threads” means: there are two designs hiding in the question and they have opposite verdicts.

The naive reading: many threads sharing one worker — don'ta de-optimization

Taken literally, this is several threads all calling progress() (and tagSend/tagRecv) on the same worker. To even be legal it requires building the worker in UCS_THREAD_MODE_MULTI, and it's the wrong move, for reasons baked into how UCX works:

  • A worker's progress engine is serial. UCX advances all of a worker's in-flight operations through one internal state machine. In MULTI mode that machine is guarded by a worker-wide lock, so concurrent progress() calls don't run in parallel — they queue on the lock and execute one at a time. You pay locking on every operation and get no parallel progress in return. UCX's own documentation says exactly this: a worker can be used from many threads, but because of lock contention it performs best driven mostly from one. People who have tried the shared-worker path have measured it running dramatically slower — in one reported case roughly 100× — than keeping progress on a single thread.
  • It detonates every lock-free invariant in §03. The module is correct because one thread touches UCX: handleToEndpointRef_ needs no mutex, the work queue has a single consumer, callbacks fire on a known thread. Let progress run on many threads and callbacks fire on any of them — every shared structure now needs locking, and the re-entrancy reasoning (which assumed one call stack) has to be redone. UCX is touchy here in practice too: registering active-message handlers or driving progress concurrently on one worker is a known source of races and crashes.

So “multiple communicator threads, one worker” is a de-optimization: more lock traffic, the same serial progress, and a far more fragile codebase.

Two ways to add threads · one is a de-optimization, one is how you actually scale

The right reading: one worker per thread — shard the transportthe design that scales

The version that does scale keeps each thread single-threaded over UCX but gives each its own worker. This is the standard high-performance UCX pattern — MPI implementations, dask-cuda, and the Fugaku/Tofu work all do variants: N independent progress domains, each a private worker in UCS_THREAD_MODE_SINGLE with no internal locking, driving N network flows genuinely in parallel. End to end it looks like this:

  • One context, N workers. All workers are created from a single ucp_context — the per-process object that owns hardware resources and, crucially, the memory domain. Because memory registration lives on the context, not the worker, you register each GPU buffer once and every worker uses the resulting rkey. Sharding doesn't multiply your registration cost.
  • A deterministic shard function. Both ends compute the same mapping — e.g. shard = hash(taskId, partition) % N — so a given transfer lands on matching workers on producer and consumer. Keeping a whole partition on one worker also means that partition's chunks stay ordered; there's no cross-worker reordering to reconcile.
  • Per-worker everything. Each worker is a full copy of the §03 machine: its own run() loop and eventfd, its own work queue and deferred queues, its own endpoint pool, its own listener. The single-thread invariants hold within a worker exactly as before.
  • Connection routing falls out of “endpoints are worker-bound.” In UCX every connection is created in the scope of one worker, and an endpoint cannot be moved to another worker. So each worker gets its own listener on a distinct port, and a node advertises a port per shard. The source connects source.worker[k] → dest.listener[k], the Acceptor on dest.worker[k] spawns the server there, and the tag is matched on worker k on both sides. As a bonus, tag matching is now per-worker, so the live-tag population per worker shrinks — the 32-bit tag-hash collision worry from §13 eases.
The sharded transport, end to end · one context, N single-thread workers, routed by a shard key

Low-level mechanismsthe concrete pieces

  • Worker creation. One shared ucp_context per process; ucp_worker_create with thread_mode = UCS_THREAD_MODE_SINGLE per thread.
  • CPU / NUMA pinning. Pin each progress thread (pthread affinity) to a core on the NUMA node nearest its NIC, so completion polling and the NIC's DMA stay local — the same topology logic the rest of the stack cares about.
  • GPU affinity. Each worker thread sets its device (cudaSetDevice) and primes the context (the cudaFree(0) trick) on that device. On a multi-GPU node the clean shard key is the GPU — one worker per device — so a worker only ever moves its own GPU's buffers.
  • Wake-up per worker. Each worker has its own eventfd; signal() wakes only the worker whose queue a driver touched. A driver enqueuing work computes the shard, pushes to that worker's queue, and signals that worker.
  • Thread-safe shared state. The process-wide singletons — the output-queue manager and the intra-node registry — are now touched by N worker threads, so they need real synchronization (a mutex or a concurrent map), or they get sharded by worker too. This is the tax for giving up the single-thread simplicity.
  • Registration sharing. One registration cache on the context serves all workers; the registration is the expensive part, and you do it once.

When it's actually worth itthe ceiling

Sharding buys throughput only when one progress core is the bottleneck — and on RDMA hardware that bottleneck is usually message rate, not bandwidth. The NIC offloads the byte movement; the progress thread's job is to post operations and process completions and callbacks, and that is CPU work bounded by one core. So the single thread saturates with many small messages or many concurrent flows, not when one big transfer is in flight (that's NIC-limited, and sharding a single flow wouldn't help). The cases that clearly justify the redesign:

  • High message rate — many partitions/peers, where per-message posting and completion processing maxes a core.
  • Multiple GPUs — you want a progress thread per device for locality anyway.
  • Multiple NICs / rails — one worker per NIC to drive them in parallel.
Verdict

The single-thread design is the right default — simplest, race-free, and adequate while RDMA offload keeps one core ahead of the message rate. Worker-per-thread sharding is the principled way to scale past that ceiling, and it is a real redesign (connection routing, a shard-aware protocol, thread-safe singletons, affinity), not a flag. The one option never to reach for is the middle one — many threads on a single shared worker — which adds lock overhead for no parallel progress.

Threading facts above (worker thread modes, per-worker progress serialization, the worker-per-thread scaling pattern, context-scoped memory registration) are drawn from UCX/NVIDIA UCX documentation and the OpenUCX community; the ~100× figure is a user-reported measurement, not a general benchmark.

Why cleanup leads the loop

A natural question about run(): phase 1 drains endpoint cleanups, but the other reclamation — freeing cancelled requests — is phase 3, after the work. Why does endpoint cleanup go first rather than last? Because the two reclamations have opposite jobs, and the ordering is the whole point.

Cleanup leads so the work loop never touches a dead endpointprevention

A deferred endpoint cleanup got onto the queue because a close callback fired during the previous iteration's progress — an endpoint dropped. The danger is what's still sitting in the work queue: comm elements (servers and sources) bound to that now-dead endpoint. If phase 2 ran first, the loop could pop one of those elements and call process() on it, which would post a tagSend/tagRecv on a closing or closed endpoint — wasted work at best, a use-after-free or hard error at worst. Draining cleanups first runs closeAndDrainCommunicators(), which cancels and finalizes exactly those elements before the work loop can step them. Putting it first turns “don't act on the dead” into a structural guarantee instead of something every element's process() has to defensively re-check in every state.

Two more reasons reinforce it. Reclaim before you allocate: phase 2 allocates fresh device buffers for incoming chunks, so releasing the closed endpoints' resources first keeps peak memory, endpoints, and file descriptors lower within the iteration. Minimize the close-to-teardown gap: the cleanup was deferred last iteration, so handling it at the very top is the soonest safe moment (outside any callback, where closeBlocking() may legally progress the worker), rather than after another full work drain. There's also a small bookkeeping win — removeEndpointRef prunes the handle→endpoint map up front, so if a new connection to the same peer arrives during this iteration's progress, the stale handle is already gone and won't trip the “handle already exists” check.

And why request-reaping trails — the mirror imagecollection

Phase 3, which frees cancelled requests, sits after the work loop on purpose. A cancelled request can only be released once isCompleted() reports UCX has finished with its buffer, and what drives a cancellation to completion is the per-item progress() inside phase 2. So reaping is placed immediately after the progress-heavy phase to harvest precisely the completions that phase just produced; reaping before phase 2 would mostly find nothing ready and simply defer everything another iteration.

So the work loop is bracketed by two reclamations doing opposite things: purge-before (phase 1, prevent — don't run work on endpoints that are already gone) and reap-after (phase 3, harvest — collect the cancellations this iteration's progress just finished). Endpoint cleanup leads because its job is prevention; request cleanup trails because its job is collection.

Purge before work, reap after · the two reclamations bracket the loop for opposite reasons

The exchange as a system

Zoom out from the FSMs and the progress loop: what is this thing architecturally? It is a dynamic, point-to-point all-to-all shuffle service for GPU-resident columnar data. Almost every design question an interviewer asks is one of a handful of forces acting on that service — this section is the lens, and §18–§22 walk each force in turn.

The design

The service is a shuffle: producers partition their output by key and push each partition toward the consumer that owns it; consumers receive and reassemble their partition. Mapped onto the operators (§02, §05, §07), UcxPartitionedOutput accumulates, concats, hash/equal-partitions, then contiguous_split + packs; the exchange moves the packed partitions; UcxExchangeSource receives, unpacks, and feeds the downstream operator. The defining choice is the split into two planes: a control plane (the handshake plus per-chunk metadata — exact byte size, the cuDF layout, the 64-bit tag, the PartitionKey) and a data plane (the device buffer itself, moved once and zero-copy). Membership is dynamic: connection-on-demand, no fixed world size, no global barrier (§06, §11·P2).

two planes · negotiate on a thin control path, move bulk once on a zero-copy data path

Why it’s shaped this way

Separating the planes lets the consumer reserve the exact device buffer before the data arrives (the size rides on the control plane) and lets flow control act through the control/rendezvous path without ever touching data bandwidth. Connection-on-demand makes the system elastic — tasks launch, retry, and tear down freely with nothing pre-wired. And a zero-copy data plane keeps cost at link bandwidth (NVLink / IB / RoCE), not CPU marshalling.

Flow control & backpressure

The single most likely distributed-systems question here, and one a reviewer flagged directly. The data path is push-based with no native pull, so absent backpressure a slow consumer grows an unbounded queue and OOMs the GPU. How you bound it is a design choice with real trade-offs.

The design

The server tagSends partitions unconditionally; the protocol has no “pause” message. The valve is UCX rendezvous plus watermarks (§06, §08, §11·P6). Rendezvous requires a posted receive before a bulk transfer can proceed, so when the consumer’s queue crosses the high-water mark the source goes dormant and stops posting receives — the server’s next tagSend then blocks at rendezvous. A CAS-guarded wake re-arms the source once the queue drains to the low-water mark, and the producer side adds an independent count/byte block. This is implicit, credit-free backpressure: the absence of a posted receive is the stop signal.

backpressure without a pull protocol · rendezvous + high/low watermarks as the valve

The design space worth naming: credit-based flow control (the consumer explicitly grants N buffers of credit; precise, but adds messages and accounting), a bounded sliding window of in-flight chunks, or rendezvous-as-valve as used here (zero extra messages, coarser granularity).

Why it’s shaped this way

It buys end-to-end flow control with no extra protocol messages — UCX’s own rendezvous semantics do the throttling — giving bounded queue depth and therefore bounded GPU memory, with no credit accounting to get wrong.

Memory, registration & spilling

GPU memory is the scarce resource, and RDMA cannot touch unregistered memory. The exchange’s memory design is three moves: register once, pool aggressively, spill before you OOM.

The design

Registration. UCX RDMA requires registered (pinned) memory and a remote key (rkey); registration pins pages and programs the NIC, so it is expensive — a registration cache amortizes it across reuses of the same buffer. Pooling. An RMM pool pre-allocates a large device arena and sub-allocates in O(1), avoiding a cudaMalloc (and its implicit sync) per chunk; pooled buffers stay registered, so reuse skips re-registration. Zero-copy. contiguous_split lays a partition out as one device buffer; the server tagSends the raw pointer and GPUDirect RDMA / NVLink moves it (§11·P1); when GPUDirect is unavailable (topology or driver), the fallback stages through pinned host memory. Spilling. When the device pool is exhausted — skew, or a slow drain — packed buffers spill to pinned host memory (or disk) and reload on demand, turning a memory spike into graceful degradation instead of an OOM.

Why it’s shaped this way

The cache and pool drive steady-state allocation and registration cost to near-zero, so bandwidth stays link-bound rather than allocator-bound; the zero-copy path keeps the data plane off the CPU; and spilling converts the worst case (memory exhaustion under skew) from a crash into a slowdown.

Connections, topology & scaling

Scaling the exchange across many GPUs and nodes turns three knobs: connection count, progress throughput, and hardware locality.

The design

Connection-on-demand. Endpoints are worker-bound; a source connects only when it has work, and the Acceptor’s active-message handshake spawns the server (§06, §11·P2) — no pre-wired mesh. All-to-all topology. N participants imply up to N² endpoints; connection count and setup cost grow quadratically, mitigated by endpoint reuse and a 2-level shuffle that aggregates intra-node first, then exchanges a few inter-node links. Progress throughput. One worker per thread, sharded by hash(taskId, partition) % T, sharing a ucp_context for registration, with per-worker listeners (§15) — the single-core ceiling is message rate, not bandwidth. Hardware locality. GPU↔NIC affinity (use the NIC on the GPU’s PCIe/NVLink rail), NUMA-local progress threads, NVLink intra-node, rail-aligned routing.

connection topology · flat all-to-all (~N² endpoints) vs a 2-level hierarchical shuffle

Why it’s shaped this way

Connection-on-demand keeps membership elastic with no global barrier; sharded workers lift the single-core message-rate ceiling; affinity makes effective bandwidth approach the link rather than a cross-NUMA fraction of it; and the hierarchy cuts the quadratic inter-node connection count toward linear.

Fault tolerance & delivery semantics

Distributed means partial failure. What does the exchange guarantee when a task aborts mid-transfer, a UCP callback replays, or a peer goes silent?

The design

Fire-and-forget at the edge. The handshake response is fire-and-forget; completions drive state, not acknowledgements (§06). Lifetime discipline. Every async callback captures weak_from_this() and no-ops if the object is gone, guards on expected state to ignore UCP wireup replays, and retains completed requests so their lambdas stay valid for the object’s lifetime (§11·P4). Cancellation / drain. closeAndDrainCommunicators cancels endpoints first (purge-before), then reaps completions (reap-after) — the loop ordering of §16. Delivery semantics. At this layer you get effectively at-least-once within a task’s lifetime; exactly-once requires idempotent consumption or dedup (sequence numbers / tags). Timeouts detect dead peers; transient errors get bounded retries.

Why it’s shaped this way

The discipline makes the system safe under cancellation and UCP replay (no use-after-free); drain ordering makes teardown clean (no work on dead endpoints, no leaked requests); and pushing exactly-once up to an idempotent unpack keeps the transport itself simple rather than baking in a heavyweight reliability protocol.

The trade-off space: why UCX

The “what transport would you use?” question. The honest answer is a matching argument: the workload is a dynamic, point-to-point, zero-copy GPU shuffle, and that shape selects UCX over the collective and RPC alternatives.

The design — the options and where they sit

UCX is a transport abstraction (its UCT layer selects rc / ud / cuda_ipc / cuda_copy / tcp) with point-to-point flexibility, first-class GPU support (GPUDirect), dynamic membership, and tag matching — it fits irregular, late-bound shuffles. NCCL is collective-optimized (allreduce, all-to-all) and topology-aware, superb for dense regular collectives and training, but rigid: a fixed communicator/world and collective semantics fit a static dense all-to-all, not a dynamic point-to-point shuffle with tasks coming and going (§12). MPI is the HPC standard but its world is fixed at launch — awkward for an elastic, task-based engine. gRPC / Thrift / TCP are general and easy, but a host bounce plus serialization make them slow for GPU bulk (fine for the control plane). Raw RDMA verbs give maximal control at maximal complexity — UCX is the abstraction over them.

the trade-off space · the workload (dynamic, point-to-point, zero-copy GPU shuffle) selects UCX

Why UCX here — and the cost

The workload is dynamic membership + point-to-point GPU shuffle + zero-copy, which is precisely UCX’s sweet spot. NCCL’s collective model does not match the irregular, late-bound pattern; gRPC is too slow on the data plane (though it could carry control). The cost is real: UCX is lower-level, so you build flow control, lifetime safety, and routing yourself — which is exactly why §11 and §18–§21 exist.
