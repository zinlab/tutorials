1 Context
Nsight Systems vs Nsight Compute

These two tools answer fundamentally different questions. Getting clear on their roles prevents wasted time.

Nsight Systems

System-level profiler. Shows the timeline across CPU, GPU, and communication. Answers: where is time being spent? Which kernels dominate? Where is the GPU idle? Is communication overlapped?

Scope: entire application, all kernels, all threads.
Overhead: low (1–5%).
Output: timeline + summary statistics.

Nsight Compute

Kernel-level profiler. Deep-dives into a single CUDA kernel. Answers: why is this kernel slow? Is it compute-bound or memory-bound? What are the warps stalled on? What limits occupancy?

Scope: 1 kernel (or a few), re-runs it multiple times.
Overhead: very high (10–100x slowdown per kernel).
Output: roofline, counters, source correlation.

The workflow: Use Nsight Systems first to identify the top 3–5 kernels by total GPU time. Then use Nsight Compute to deep-dive into each one. Never start with Nsight Compute on the full application — it will take hours and produce an overwhelming amount of data.

2 Capture
Launching ncu

Basic ncu capture commands Capture 
# Profile ALL kernels (careful — very slow)
ncu --set full \
    --launch-count 3 \
    -o baseline_kernel \
    python train.py

# Profile only GEMM kernels (most common target)
ncu --set full \
    --kernel-name "regex:gemm" \
    --launch-skip 100 \
    --launch-count 5 \
    -o gemm_analysis \
    python train.py

# Profile a specific kernel by index
ncu --set full \
    --launch-skip 50 \
    --launch-count 1 \
    -o single_kernel \
    python train.py
FlagPurposeGuidance
--set fullCollect all hardware countersUse this unless you know exactly which section you need. The basic and detailed sets miss important metrics.
--kernel-name "regex:..."Filter kernels by name patternTarget specific kernels: gemm, conv, elementwise, flash. Avoids profiling thousands of irrelevant tiny kernels.
--launch-skip NSkip the first N kernel launchesSkip warmup iterations. If your forward pass launches 200 kernels and you want iteration 3's GEMM, skip ~400+.
--launch-count NProfile only N kernel launchesKeep small (1–10). Each profiled kernel is replayed 5–20+ times internally to collect all counters.
-o filenameOutput file (.ncu-rep)Transfer to local workstation for GUI analysis, like Nsight Systems.
--section "regex:..."Collect only specific counter sectionsSpeeds up collection if you only need e.g. memory or compute metrics. Use ncu --list-sections to see options.
Performance impact: Nsight Compute replays each kernel multiple times (once per counter group). A kernel that takes 1ms will be replayed 10–20 times, taking 10–20ms per profiled launch. With --set full, profiling even 5 kernel launches can slow a training step from 100ms to 30+ seconds. Always use --launch-skip and --launch-count.
Programmatic capture from Python Capture

For more precise control, use --capture-range with the CUDA Profiler API:

import torch.cuda.profiler as profiler

for i, batch in enumerate(loader):
    # ... forward, backward ...

    if i == 5:
        profiler.start()    # ncu starts collecting
    if i == 6:
        profiler.stop()     # ncu stops
# Launch with capture-range mode
ncu --set full \
    --replay-mode application \
    --target-processes all \
    python train.py
Replay modes matter. Nsight Compute has three replay modes: kernel (default, replays each kernel in isolation — most accurate but changes memory state), application (reruns the whole app per counter pass — slow but preserves memory state), and range (replays a marked range). For PyTorch, kernel mode is usually fine. Use application if you suspect the kernel's behavior depends on prior GPU state.
Comparing two implementations (A/B diff) Capture

One of Nsight Compute's most powerful features is side-by-side kernel comparison. Profile the baseline and optimized version separately, then open both in the GUI:

# Baseline
ncu --set full --kernel-name "regex:my_kernel" \
    --launch-count 3 -o baseline python train_v1.py

# Optimized
ncu --set full --kernel-name "regex:my_kernel" \
    --launch-count 3 -o optimized python train_v2.py

# In the GUI: File → Open → baseline.ncu-rep
# Then: Add Baseline → optimized.ncu-rep
# Every metric shows absolute and % difference

The diff view highlights regressions in red and improvements in green across every counter. This is invaluable for validating that a change (e.g., adding torch.compile, changing data layout, adjusting tile sizes in a Triton kernel) actually improved hardware utilization rather than just changing where time is spent.

3 Foundation
The roofline model

The roofline model is the single most important concept in Nsight Compute. It tells you whether a kernel is limited by compute throughput or memory bandwidth — and how far it is from the hardware's theoretical maximum.

Understanding the roofline chart Concept

Roofline model for H100 SXM (BF16 Tensor Core). Points below the roofline have room to improve.

How to read it

The X-axis is arithmetic intensity — FLOPs performed per byte of data moved from memory. The Y-axis is achieved performance in FLOP/s. The roofline has two parts:

The sloped line (left) represents the memory bandwidth ceiling. Kernels in this region are memory-bound — they can't compute faster because they're waiting for data from HBM. The slope equals the GPU's memory bandwidth (3.35 TB/s on H100 SXM).

The horizontal line (right) represents the compute ceiling. Kernels here are compute-bound — they have enough data but can't execute instructions faster. The height equals peak FLOP/s (990 TFLOP/s BF16 Tensor Core on H100).

The ridge point is where the two lines meet. Its X-coordinate tells you the arithmetic intensity needed to fully utilize both compute and memory — about 295 FLOP/Byte for H100 BF16.

What the example points mean

PointPositionInterpretation
A: BatchNormLow AI, near bandwidth ceilingMemory-bound and near the memory roofline — this kernel is already efficient given its arithmetic intensity. The only way to speed it up is to fuse it with adjacent ops to increase AI.
B: Small GEMMNear ridge, slightly below roofNear the transition point. Has moderate AI but doesn't quite reach the ceiling — may have occupancy issues or suboptimal tiling.
C: Large GEMMHigh AI, near compute ceilingCompute-bound and close to peak — this is the goal. Large matrix multiplies with Tensor Cores should land here.
D: UnoptimizedHigh AI, far below compute ceilingHas enough arithmetic intensity to be compute-bound but achieves far less than the hardware can deliver. The vertical gap to the ceiling is the optimization opportunity — look at warp stalls, occupancy, and instruction mix.
Multiple rooflines: Nsight Compute shows separate rooflines for FP32 CUDA cores, FP16 Tensor Cores, BF16 Tensor Cores, and FP8. If your kernel appears near the FP32 ceiling but far below the BF16 ceiling, it's using the wrong precision — switch to mixed precision to jump to the higher roofline.

4 Overview
Speed of Light (SOL)

The "Speed of Light" section is the first thing to check in any Nsight Compute report. It summarizes how close the kernel is to the hardware's theoretical peak across compute and memory, expressed as percentages.

Reading the SOL summary Interpret

Here's what a typical SOL display looks like for different kernel types. The percentage tells you how close to the hardware maximum the kernel achieves.

Example: Well-optimized large GEMM (BF16 Tensor Core)

SM Throughput87%

Compute pipe utilization — close to peak Tensor Core throughput

Memory Throughput42%

HBM bandwidth utilization — not the bottleneck, compute-bound

Example: Memory-bound element-wise kernel

SM Throughput12%

Very low compute utilization — SMs mostly waiting for data

Memory Throughput89%

Near peak HBM bandwidth — memory-bound kernel working as expected

Example: Poorly optimized kernel

SM Throughput18%

Low compute — SMs are underutilized

Memory Throughput23%

Low memory — not bandwidth-bound either. Something else is wrong.

The diagnostic rule: High SM + low Mem = compute-bound (good for GEMMs). Low SM + high Mem = memory-bound (expected for element-wise, reductions). Low SM + low Mem = neither pipeline is saturated — look at occupancy, warp stalls, or instruction-level inefficiencies. This is the quadrant where the biggest gains live.

5 Compute
Compute analysis

Instruction mix and pipe utilization Interpret

The Compute Workload Analysis section breaks down which functional units (pipes) inside the SM are active:

PipeWhat it doesWhat to look for
FMA (FP32)Fused multiply-add on CUDA coresIf this dominates and Tensor pipe is idle, the kernel isn't using mixed precision. Switch to BF16/FP16.
Tensor CoreMatrix multiply-accumulate unitsShould be the dominant pipe for GEMM/convolution kernels. If idle, check data types and matrix alignment (multiples of 8).
ALU (INT)Integer arithmetic (indexing, address computation)High ALU with low FMA/Tensor = the kernel spends most time computing addresses rather than doing useful math. Indicates memory access pattern issues.
FP64Double-precision floating pointShould be zero for training workloads. If non-zero, something is accidentally in FP64 — torch.set_default_dtype(torch.float32) and check for Python float constants in kernels.
LSULoad/store unitHigh LSU utilization means the kernel is memory-access heavy. Expected for data movement kernels (transpose, copy), a red flag for compute kernels.
SFUSpecial function unit (sin, cos, exp, rsqrt)High SFU utilization in activation functions (GELU, SiLU) is expected. If it dominates, consider using polynomial approximations or fusing with adjacent ops.

Common problem: Tensor Core pipe shows 0% utilization on a GEMM kernel

This means the kernel fell back to FP32 CUDA cores. Check: is autocast enabled? Are the input tensors actually in FP16/BF16 (print tensor.dtype)? Are matrix dimensions M, N, K multiples of 8? Are you using a cuBLAS GEMM or a custom kernel that doesn't use wmma/mma instructions?

Warp execution efficiency Interpret

A warp is 32 threads that execute in lockstep. Warp execution efficiency measures what fraction of threads in a warp are active (not masked off by divergent branches).

Ideal

100%

Acceptable

> 80%

Concerning

< 50%

Low efficiency means branch divergence — different threads in a warp take different paths in an if/else, so the GPU serializes both paths. Common in: custom kernels with data-dependent branching, sparse operations, irregular workloads. Less common in standard PyTorch ops (cuBLAS/cuDNN are well-optimized).

6 Memory
Memory analysis

The memory hierarchy and where bottlenecks occur Concept

The GPU memory hierarchy has multiple levels, each with different bandwidth and latency. Nsight Compute shows utilization at each level.

LevelH100 SXM specsTypical access pattern
Registers~80 TB/s effective, per-threadFastest. Compiler-managed. Exceeding 255 registers per thread spills to local memory (slow).
Shared memory (SMEM)228 KB per SM, ~30 TB/s aggregateProgrammer-managed scratchpad. Used for tiling in GEMM, reductions, inter-thread communication. Under-utilization here often means suboptimal tiling.
L1 cache256 KB per SM (combined with SMEM)Automatic caching. High L1 hit rate = good spatial/temporal locality. Low hit rate = scattered access patterns.
L2 cache50 MB total, ~12 TB/sShared across all SMs. Working sets that fit in L2 see huge speedups. Flash attention exploits this aggressively.
HBM (DRAM)80 GB, 3.35 TB/sGlobal memory — slowest. Every byte you can avoid reading from HBM is a win. Fusion reduces HBM traffic by keeping intermediates in registers/SMEM.

Key metrics to check

MetricWhat it tells youAction
DRAM Throughput %How close to peak HBM bandwidthIf near 100%, kernel is memory-bound. Reduce data movement (fusion, smaller dtypes, tiling).
L2 Hit RateFraction of memory requests served by L2High hit rate (>80%) means data fits in L2 — good. Low hit rate means every access goes to HBM.
L1 Hit RateFraction served by L1/shared memoryLow hit rate = poor spatial locality or insufficient tiling. Consider shared memory tiling or data layout changes (AoS → SoA).
SMEM UtilizationHow much shared memory is used vs availableNear 100% may limit occupancy (fewer blocks per SM). Near 0% on a memory-bound kernel means SMEM tiling isn't being used when it could be.
Sector EfficiencyFraction of fetched bytes actually usedLow efficiency means uncoalesced access — threads request scattered addresses, forcing the GPU to fetch full cache lines but only use part. Fix data layout or access patterns.
Memory access pattern problems Issue

Uncoalesced global memory access

In the Memory Workload Analysis, you see high "Sectors per Request" (ideally 1 for coalesced, can be up to 32 for fully scattered). The GPU fetches 128-byte sectors from HBM — if 32 threads in a warp each want bytes from different sectors, you fetch 32 sectors but only use 4 bytes from each, wasting 97% of bandwidth.

Fix

Ensure consecutive threads access consecutive memory addresses (coalesced access). For matrix operations: use column-major layout for column-accessed data, row-major for row-accessed. For custom kernels: reindex thread → data mapping so thread i accesses address base + i * sizeof(element). For PyTorch: use .contiguous() on tensors before passing to CUDA operations, and prefer channels_last memory format for convolutions.

Shared memory bank conflicts

In the Shared Memory section, you see high "Bank Conflicts per Access" (ideally 0). Shared memory has 32 banks — if multiple threads in a warp access different addresses in the same bank simultaneously, the accesses serialize (a 32-way conflict is 32x slower).

Fix

Pad shared memory arrays to avoid stride-conflicts: instead of __shared__ float tile[32][32], use __shared__ float tile[32][33] (adds one column of padding to shift bank alignment). In Triton kernels, the compiler usually handles this, but check the generated PTX if stalls persist.

Register spilling

In the Launch Statistics, you see "Local Memory per Thread" is non-zero. This means registers overflowed to local memory (which is actually global memory with L1 caching) — 100–400x slower than register access.

Fix

Reduce register pressure: simplify per-thread computation, break large kernels into smaller fused stages, use --maxrregcount=N compiler flag to force register limits (trades spilling for occupancy). In Triton: reduce BLOCK_SIZE or the number of accumulator variables.

7 Occupancy
Occupancy analysis

What occupancy means and when it matters Concept

Occupancy is the ratio of active warps on an SM to the maximum number of warps the SM can hold (64 on H100). Higher occupancy gives the GPU more warps to schedule when one warp stalls (e.g., on a memory fetch), hiding latency through parallelism.

Max warps per SM (H100)

64

Max threads per SM

2048

Max blocks per SM

32

Three resources that limit occupancy

ResourceHow it limitsNsight Compute shows
Registers per threadH100 has 65,536 registers per SM. If a kernel uses 128 registers/thread, each block of 256 threads needs 32,768 registers — only 2 blocks fit (= 16 warps = 25% occupancy).The Occupancy section shows "Registers per Thread" and "Theoretical Occupancy" limited by registers. If this is the limiter, you'll see "Registers" highlighted in red.
Shared memory per block228 KB per SM on H100. If each block uses 100 KB of SMEM, only 2 blocks fit. At 256 threads/block, that's 16 warps = 25% occupancy.The Occupancy section shows "Shared Memory per Block" and flags it if it's the limiter.
Block size (threads per block)If the block is too large (e.g., 1024 threads) and resources per block are high, fewer blocks fit. If too small (e.g., 32 threads = 1 warp), you need 64 blocks to fill an SM but the max is 32.The "Block Limit" row in the occupancy calculator. Increasing block size is often the easiest lever.
Occupancy isn't everything. A memory-bound kernel at 25% occupancy might be performing optimally if it's already saturating HBM bandwidth — more warps won't help because there's no more bandwidth to use. A compute-bound Tensor Core GEMM at 50% occupancy can still hit 85% of peak FLOP/s because Tensor Core instructions have high throughput. Use occupancy as a diagnostic for latency-sensitive kernels (those with many memory stalls), not as a universal optimization target.
The occupancy calculator Interpret

Nsight Compute includes an interactive occupancy calculator (also available as a standalone Excel spreadsheet from NVIDIA). It shows three curves:

Occupancy vs Registers per Thread — shows how occupancy drops as register usage increases. The achieved register count is marked. If you're on a steep part of the curve, reducing registers by even a few can jump occupancy significantly.

Occupancy vs Shared Memory per Block — shows how occupancy drops as SMEM usage increases. If the achieved SMEM is right at a step boundary, a small reduction can unlock an additional block per SM.

Occupancy vs Block Size — shows how occupancy scales with threads per block. Generally, 128–256 threads per block is the sweet spot. Below 64, you can't fill enough warp slots; above 512, resource pressure usually kills occupancy.

Achieved vs theoretical occupancy

The calculator shows theoretical maximum occupancy given the resource constraints. But achieved occupancy (measured at runtime) can be lower if there aren't enough blocks to fill all SMs (the grid is too small). This happens with small batch sizes or small spatial dimensions — the problem decomposes into too few blocks to keep 132 SMs on H100 busy. Fix: increase batch size, or restructure the kernel to launch more blocks.

Interpreting stall patterns for PyTorch workloads Issue

GEMM kernels (linear layers, convolutions)

Healthy GEMM stall profile: dominated by Wait (Tensor Core pipeline latency) and Not Selected (scheduler has options). If instead you see high Long Scoreboard, the GEMM is memory-bound — likely because the matrices are too small to achieve high arithmetic intensity.

Common GEMM problems

Small M dimension (batch_size × seq_len is small) — not enough work to fill the GPU. Increase batch size or use gradient accumulation.
Misaligned K dimension — K not a multiple of 8 or 16 prevents efficient Tensor Core tiling. Pad weights/inputs.
FP32 fallback — Tensor pipe shows 0%. Enable autocast.
Suboptimal algorithm — cuBLAS heuristic picked the wrong GEMM implementation. Use torch.backends.cuda.matmul.allow_tf32 = True and ensure torch.backends.cudnn.benchmark = True.

Element-wise operations (ReLU, add, multiply, dropout)

These are almost always memory-bound — low arithmetic intensity (1–2 FLOPs per byte loaded). Healthy stall profile: dominated by Long Scoreboard (waiting for HBM loads) with high memory throughput in SOL. The achieved performance should be near the memory bandwidth roofline.

When element-wise ops are a problem

Individual element-wise kernels are rarely worth optimizing in isolation — they're inherently memory-bound. The fix is fusion: combine multiple element-wise ops into a single kernel so intermediate results stay in registers instead of round-tripping through HBM. This is exactly what torch.compile does. If you see 20 separate element-wise kernels between two GEMMs, that's 20 HBM round-trips that a fused kernel reduces to 1.

Reductions (sum, mean, softmax, layer norm)

Reductions are tricky because they have two phases: a memory-bound parallel reduction across elements, then a communication phase where partial results are combined. Stall profile typically shows Long Scoreboard (memory loads) and Barrier (synchronization between reduction stages).

When reductions are a problem

Small reduction dimension — reducing over a small axis means each block has little work. This shows up as low occupancy and high "Drain" stall (kernels finishing too quickly).
Multiple separate reductions — LayerNorm computes mean, then variance, then normalizes — 3 separate HBM reads of the same data. Fused LayerNorm kernels (like Apex's) do it in a single pass. Check if you're using the fused version.

Attention kernels

If you see standard (non-Flash) attention, the softmax + matmul chain is memory-bound with massive HBM traffic (O(N²) for sequence length N). The stall profile shows extreme Long Scoreboard.

Fix

Switch to FlashAttention (torch.nn.functional.scaled_dot_product_attention in PyTorch 2.0+). Flash attention tiles the attention computation to keep data in SMEM/registers, reducing HBM traffic from O(N²) to O(N). In Nsight Compute, a FlashAttention kernel should show high SMEM utilization, moderate Long Scoreboard (SMEM latency is much lower than HBM), and throughput much closer to the compute roofline.

9 Decide
Decision tree

Given a kernel's Nsight Compute report, follow this tree to diagnose and fix.

Step 1: Look at the Speed of Light summary. Which pipeline is closer to its ceiling?

A →SM Throughput high, Memory Throughput low → Compute-bound
B →SM Throughput low, Memory Throughput high → Memory-bound
C →Both low → Latency-bound (neither pipeline saturated)

Path A (Compute-bound): Check the roofline position and instruction mix.

A1 →Near the compute ceiling (>75% SOL)
→ The kernel is well-optimized. Remaining gains: switch to a higher roofline (FP32→BF16→FP8), increase matrix dimensions, or use a more efficient algorithm.
A2 →Far below the compute ceiling (<50% SOL)
→ Check pipe utilization: is Tensor Core pipe active? If not → enable mixed precision, align dims to 8.
Check warp stall reasons: high Wait → pipeline bubbles, need software pipelining.
Check warp execution efficiency: low (<80%) → branch divergence in the kernel.

Path B (Memory-bound): Check if the kernel is near the memory bandwidth ceiling.

B1 →Near the memory ceiling (>75% DRAM throughput)
→ The kernel is efficiently memory-bound. Only way to improve: reduce data movement through fusion (combine with adjacent kernels), use smaller data types (BF16→FP8 halves memory traffic), or improve cache hit rates (tiling for L2).
B2 →Below the memory ceiling (<50% DRAM throughput)
→ Memory is the bottleneck but you're not hitting the bandwidth wall. Check:
• Sector efficiency: low → uncoalesced access. Fix data layout.
• L2 hit rate: low → working set doesn't fit in L2. Consider tiling or blocking.
• SMEM bank conflicts: high → pad shared memory arrays.
• MIO/LG Throttle stalls: high → too many outstanding loads. Use vector loads or reduce loads per thread.

Path C (Latency-bound — both pipelines underutilized): This is where the biggest gains live.

C1 →Low achieved occupancy (< 25%)
→ Check the occupancy limiter: registers, shared memory, or block size. Reduce the dominant limiter. Consider --maxrregcount to trade register pressure for occupancy.
C2 →Reasonable occupancy (> 40%) but still latency-bound
→ Check warp stalls: dominant Long Scoreboard with moderate occupancy means dependent memory access chains. Restructure the algorithm to have independent loads that can overlap. Consider prefetching or double-buffering.
C3 →The grid is too small (fewer blocks than SMs)
→ The kernel doesn't launch enough blocks to fill all 132 SMs on H100. This is a problem geometry, not a kernel optimization problem. Increase batch size, restructure the decomposition to launch more blocks, or fuse this kernel into a larger one.

The systematic workflow: Nsight Systems finds the which — which kernels dominate, where the GPU is idle. Nsight Compute finds the why — why a specific kernel underperforms. Start with SOL (compute-bound or memory-bound?), check the roofline (how far from the ceiling?), read the stall reasons (what are warps waiting on?), check occupancy (is there enough parallelism?), then translate findings to PyTorch-level actions. Profile → diagnose → fix → re-profile. The gap between your kernel's achieved throughput and the hardware ceiling is your optimization budget.
