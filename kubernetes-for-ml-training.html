Chapter I

The anatomy of a cluster.

Read time 6 min

Prereq Containers (Docker) basics. A vague sense that "kubectl" is involved.

Outcome You can sketch the control plane on a napkin and explain what each piece does.

A Kubernetes cluster is not one program. It is half a dozen programs in a polite, paranoid conversation, persisting their consensus into a single key-value store that pretends to be the truth.

Every action — create a pod, scale a deployment, mount a volume — is a write to etcd. Every other process is a control loop watching etcd for drift between what users declared they wanted and what's actually running. This is the only sentence about Kubernetes you really need to internalize.

Fig. 1 — The two planes

The four control-plane parts

API Server. The only door into the cluster. Every kubectl command, every controller, every kubelet — they all talk to the API server over HTTPS. It validates requests, authenticates them, writes accepted state to etcd, and emits watch events to everyone subscribed. If the API server is down, the cluster is, for all practical purposes, frozen — though existing workloads keep running.

etcd. A distributed key-value store with strong consistency (Raft consensus). It holds every object you've ever created. Lose etcd and you lose the cluster. Back it up.

Scheduler. Watches for pods that have a spec but no nodeName. For each one, it filters nodes that can run the pod (enough CPU, memory, GPU; matches affinities), then scores the survivors, then writes the choice back to the API server. That's the entire job.

Controller Manager. A single binary running dozens of control loops: the Deployment controller, the Job controller, the Node lifecycle controller, the Endpoint controller, and so on. Each watches some kind of object and writes other objects to compensate for drift.

What runs on every node

kubelet. The local agent. Watches the API for pods bound to its node, then talks to the container runtime to start them. Reports node and pod status back upward.

Container runtime. Almost always containerd now. Pulls images, manages containers' lifecycles, attaches storage and network.

kube-proxy. Programs iptables (or IPVS, or eBPF) rules so that traffic to a Service's virtual IP lands on a healthy pod.

Device plugins. For GPUs, nvidia-device-plugin tells the kubelet how many GPUs the node has and how to attach them to a container. Without it, the cluster cannot see GPUs at all.

Why this matters for ML

The control plane is small and cheap; the data plane is the entire cost. On a real training cluster, control-plane nodes might be 3× m6i.xlarge ($200/mo) and the data plane is 64× p5.48xlarge at $98/hr each. Every optimization that matters is about keeping GPUs hot — control-plane tuning is a rounding error.

Rule of thumb

When something feels missing in Kubernetes for ML — fair-share quotas, gang scheduling, queueing, distributed run lifecycle — there is almost certainly a Custom Resource Definition (CRD) for it. The big four to know: Kubeflow Training Operator (PyTorchJob, TFJob, MPIJob), Volcano (batch + gang), Kueue (queues + quotas), Ray on K8s (RayCluster, RayJob).

Actor: kubectl  ·  Target: kube-apiserver
01 — You run kubectl apply

Your YAML manifest is shipped to the API server over HTTPS. The API server authenticates you (cert, token, OIDC), authorizes the action (RBAC), and runs admission controllers — mutating webhooks may inject sidecars, validating webhooks may reject pods that don't meet policy (e.g., missing resource requests).

The pod's status field shows Pending, with no nodeName yet. It exists conceptually but is running on nobody.

Why this matters operationally

When a pod is stuck, you debug it by asking: which actor in the conversation is silent?

Stuck in Pending? The scheduler couldn't find a node. Run kubectl describe pod and read the Events — you'll see "0/8 nodes available: insufficient nvidia.com/gpu". The fix is more nodes, smaller pods, or different node selectors.

Stuck in ContainerCreating? The kubelet accepted the binding but can't start the container. Almost always: image pull failure (wrong registry credentials), volume mount failure (PVC not bound), or device plugin not ready.

Stuck in CrashLoopBackOff? The container started and died. kubectl logs --previous shows you the last words. Most common: OOM kill, missing config, wrong CUDA version inside the image vs. driver on the host.

The single most useful debug command

kubectl describe pod <name> prints the entire conversation: every state transition, every probe failure, every scheduler verdict. Always read this before reading logs.

Chapter V

The scheduler & why it fails you.

Try it Click each pending pod to attempt scheduling. Watch which ones can't fit, and why.

The lesson Vanilla kube-scheduler is greedy and per-pod. It will happily place pod #1 of a distributed run, then leave pods #2–#7 pending while pod #1 burns money.

The default scheduler is a one-pod-at-a-time matchmaker. That's fine for web services. For distributed training, it's a recipe for partial scheduling — the scenario where some workers start, can't reach their peers, and stall until somebody manually kills the run.

What the default scheduler does, step by step

Filter. Discard nodes that fail any predicate: insufficient CPU/memory/GPU, taints not tolerated, node-selector mismatch, port conflicts, volume zone mismatch. After this, you have a candidate set.

Score. Rank survivors by built-in scoring plugins: spread pods of the same replica set (resilience), prefer nodes with the image already pulled (speed), prefer less-utilized nodes (or more-utilized, with bin-packing). Highest score wins.

Bind. Write spec.nodeName on the pod. The kubelet on that node takes over.

The three ways this hurts ML

1. Partial scheduling. Your WORLD_SIZE=8 run gets 5 workers on GPUs and 3 stuck in Pending. The 5 running pods sit at zero utilization waiting for rendezvous. Cost: ~$500/hour for nothing. Fix: a gang scheduler (Volcano, Kueue with WorkloadPriorityClass + queueing, KAI Scheduler) that holds the whole job until it fits.

2. Topology-blind placement. The scheduler sees "this node has 8 GPUs free" and "that node has 8 GPUs free" — but doesn't know that nodes in rack A1 share a NDR InfiniBand switch and nodes in rack B7 do not. Cross-rack all-reduce is 4× slower. Fix: topology labels on nodes plus topology-aware scheduling (NVIDIA's Topology Aware Scheduling, or affinity rules keyed on topology.kubernetes.io/zone and rack labels).

3. Fragmentation. After a week of starts and stops, your 8-GPU-per-node cluster has lots of nodes with 1, 2, or 4 free GPUs and very few with 8. New 8-GPU jobs queue indefinitely. Fix: bin-packing scheduling policy (set MostAllocated in scheduler config) so small jobs cluster onto the same nodes, leaving full-node holes for big jobs.

In production ML clusters

Replace or augment the default scheduler. The most common choices in 2025–26: Kueue (lightweight, official-ish, ABi-compatible with native scheduler) for queueing & quotas; Volcano for full gang scheduling & batch semantics; NVIDIA KAI Scheduler if you're on NVIDIA-heavy infrastructure and want topology awareness baked in.

Chapter VI

Topology & the NCCL tax.

Click Each topology to see how it changes the cost of an all-reduce.

Stack ranking NVLink > NVSwitch > IB-Fat-Tree > IB-Rail-Optimized > RoCE > TCP. Bandwidth differs by 10–50×.

Every step of synchronous training ends in an all-reduce: every worker sums its gradients with every other worker's. The speed of that all-reduce is gated by the slowest link in the topology. The scheduler's job, for distributed training, is to keep that slowest link as fast as possible.

Interactive · Topology Cost

How to make the scheduler topology-aware

Step 1: label your nodes. Most cloud providers add zone labels automatically (topology.kubernetes.io/zone). You add the rest:

kubectl label node node-gpu-001 \
  rack=A1 \
  ib-switch=switch-A1-leaf-3 \
  gpu-type=H100-SXM5

Step 2: in your pod spec, require co-location using topologyKey:

spec:
  affinity:
    podAffinity:
      requiredDuringSchedulingIgnoredDuringExecution:
        - labelSelector:
            matchExpressions:
              - { key: app, operator: In, values: [my-training-run] }
          topologyKey: ib-switch     # all peers share an IB switch

Step 3 (for serious setups): install the NVIDIA GPU Operator and the Topology-Aware Scheduling components. These auto-discover NVLink islands, NUMA topology, and PCIe trees, and surface them as labels the scheduler can constrain on.

Measure, don't guess

Run nccl-tests as a one-off PyTorchJob across the placement your scheduler chose. If all_reduce_perf reports under 80% of the link's theoretical bandwidth, topology is leaking. Common causes: cross-zone scheduling on cloud, missing GPUDirect RDMA, MTU mismatch (set it to 9000 on the data network).

Chapter VII

Training manifests, annotated.

Form Two manifests: a single-node multi-GPU Job, then a multi-node PyTorchJob.

Read Every line. There are no decorative fields — each one is doing work.

Below is the smallest training manifest worth deploying, then the canonical multi-node version. Both compile in real clusters today.

Single-node, 8-GPU run

apiVersion: batch/v1
kind: Job
metadata:
  name: llama-finetune-001
  namespace: ml-training
spec:
  backoffLimit: 2                          # retry up to 2x on failure
  ttlSecondsAfterFinished: 86400          # auto-clean after 24h
  template:
    spec:
      restartPolicy: OnFailure
      priorityClassName: training-high
      tolerations:                          # schedule onto GPU nodes
        - { key: gpu, operator: Exists, effect: NoSchedule }
      nodeSelector:
        gpu-type: H100-SXM5
      containers:
        - name: trainer
          image: registry.internal/llama-trainer:2026.04.01
          command: ["torchrun", "--standalone", "--nproc_per_node=8", "train.py"]
          env:
            - { name: NCCL_DEBUG, value: WARN }
            - { name: NCCL_IB_DISABLE, value: "0" }
            - { name: CUDA_DEVICE_MAX_CONNECTIONS, value: "1" }
          resources:
            requests:
              cpu: "32"
              memory: 256Gi
              # requests & limits MUST match for guaranteed QoS class
              nvidia.com/gpu: "8"
            limits:
              cpu: "32"
              memory: 256Gi
              nvidia.com/gpu: "8"
          volumeMounts:
            - { name: data,        mountPath: /data,    readOnly: true }
            - { name: checkpoints, mountPath: /ckpt }
            - { name: shm,         mountPath: /dev/shm } # bigger /dev/shm for DataLoader
      volumes:
        - name: data
          persistentVolumeClaim: { claimName: training-data-readonly }
        - name: checkpoints
          persistentVolumeClaim: { claimName: ckpt-llama-001 }
        - name: shm
          emptyDir: { medium: Memory, sizeLimit: 128Gi }

Multi-node, 4×8=32 GPU PyTorchJob

The Kubeflow PyTorchJob CRD handles rank assignment, master discovery, and per-worker pod creation. You don't manually set RANK or MASTER_ADDR — the operator injects them.

apiVersion: kubeflow.org/v1
kind: PyTorchJob
metadata:
  name: llama-pretrain
  namespace: ml-training
spec:
  runPolicy:
    cleanPodPolicy: Running
    schedulingPolicy:
      minAvailable: 4                       # gang: all 4 or nothing
      queue: research-prio
  pytorchReplicaSpecs:
    Master:
      replicas: 1
      restartPolicy: OnFailure
      template:
        spec: { /* same container spec as below */ }
    Worker:
      replicas: 3                          # total world size = 4 * 8 = 32
      restartPolicy: OnFailure
      template:
        metadata:
          annotations:
            scheduling.volcano.sh/queue: research-prio
        spec:
          schedulerName: volcano
          affinity:
            podAntiAffinity:               # spread peers across nodes
              requiredDuringSchedulingIgnoredDuringExecution:
                - labelSelector: { matchLabels: { job-name: llama-pretrain } }
                  topologyKey: kubernetes.io/hostname
            podAffinity:                    # but keep them in one IB island
              preferredDuringSchedulingIgnoredDuringExecution:
                - weight: 100
                  podAffinityTerm:
                    labelSelector: { matchLabels: { job-name: llama-pretrain } }
                    topologyKey: ib-island
          containers:
            - name: pytorch
              image: registry.internal/llama-trainer:2026.04.01
              resources:
                limits:
                  nvidia.com/gpu: "8"
                  rdma/hca: "1"            # request RDMA NIC
The QoS gotcha

Kubernetes assigns each pod a QoS class. Guaranteed (requests == limits on every resource) is the only class that prevents eviction under node pressure. Set requests = limits on training pods or you will lose work to bursty neighbors.

Chapter VIII

Performance: the knobs that matter.

Reading order The list is ordered by typical impact, not setup difficulty. Top items deliver 2–10× speedups; bottom items are 5–15% tuning.

Method Profile first (DCGM + Nsight), then optimize the slowest stage.

There is one rule of training performance: find the bottleneck, fix it, find the next one. Below are the bottlenecks ranked by how often they show up in real clusters.

Bottleneck
Symptom
Fix
Cross-node bandwidth

Per-step time scales with world size instead of being constant. nccl-tests reports <30 GB/s on 200G IB.

Verify GPUDirect RDMA is enabled. Set NCCL_IB_HCA to the right HCA. MTU 9000. Pin workers to one IB island via topology-aware scheduling.

Data loading

GPU util sawtooths: 95% then 0% then 95%. DataLoader workers can't keep up.

Cache hot shards on local NVMe via emptyDir. Use a parallel filesystem (Lustre, FSx, Weka) for cold data. Increase num_workers and prefetch_factor. Pin memory.

Checkpoint I/O

Run pauses every N steps for minutes. Throughput drops to zero during save.

Async checkpointing (PyTorch DCP). Sharded checkpoints. Write to local NVMe first, then upload to object storage. Use rsync-style incremental saves.

CPU starvation

GPUs idle waiting for CPU pre-processing. nvidia-smi shows low SM occupancy.

Increase pod CPU requests. Use DALI or Pillow-SIMD for image pipelines. Move augmentation to GPU. Pin CPU affinity per GPU (use the cpu-manager-policy=static kubelet flag).

/dev/shm too small

PyTorch DataLoader crashes with "bus error" under multiprocessing.

Mount an emptyDir: { medium: Memory } at /dev/shm sized to ~50% of node RAM. K8s default is 64MB.

NUMA misbinding

Some GPUs are 20% slower than identical peers on the same node.

Enable kubelet's topology-manager-policy=single-numa-node. Verify each GPU's NUMA affinity matches its CPU set.

Scheduler latency

Pods sit pending 30–120s after submission, even with capacity.

Scale up scheduler replicas (single by default). Check the queue depth in Kueue. Disable per-pod admission webhooks if any are slow.

DNS & control plane

Inference pods occasionally fail to resolve service names. Scheduler API latency > 100ms.

Scale CoreDNS replicas with cluster size. Move etcd to dedicated SSD. Increase API server --max-requests-inflight. Use endpoint slices.

The metrics dashboard you should have

Five panels, in this order, are enough to find 90% of training issues:

1. GPU SM occupancy (DCGM exporter → Prometheus → Grafana). Should sit at 80–95% on a healthy training run. Sawtooths mean data starvation; flat low values mean small batch sizes or sync overhead.

2. GPU memory used. Climbing memory across steps = a leak (usually from cached tensors not being freed). Flat at 90% = healthy; flat at 50% = you could increase batch size.

3. NCCL all-reduce time per step. Log this from inside the trainer. Sudden jumps = topology change (a pod got rescheduled away from the IB island).

4. Disk read throughput on the data PVC. If this maxes out, you're I/O bound.

5. Pod restart count. Anything > 0 deserves investigation — you may be losing work and not noticing.

The one-line health check

If kubectl top pod shows your trainer using 100% of its CPU request and DCGM shows GPU at 40%, the CPU is starving the GPU. Add more CPU. If both are 100%, you're balanced — increase the model size or batch.

Chapter IX

Operating the cluster.

Daily Check failed jobs, GPU util, queue depth.

Weekly Drift on node taints, evictions, image sizes.

Quarterly Upgrade K8s minor version. Test failover. Rotate certs.

A training cluster is a machine with three demands: keep the GPUs hot, keep the jobs honest, and keep the cluster boring. Boring is the highest praise.

Day-to-day: triage

Start every morning with three queries:

# 1. Anything restarting more than it should?
kubectl get pods -A --field-selector=status.phase!=Succeeded \
  | awk '$5 > 2'

# 2. Anything pending more than 10 minutes?
kubectl get pods -A --field-selector=status.phase=Pending \
  -o custom-columns=NS:.metadata.namespace,NAME:.metadata.name,AGE:.metadata.creationTimestamp

# 3. Any node throwing pressure events?
kubectl get events -A --field-selector reason=NodeNotReady,reason=DiskPressure

Capacity & cost

A GPU is idle in three ways, each with a different remedy:

Allocated but unused — the pod is running, the GPU is reserved, the trainer is not using it (rendezvous stuck, data loader stuck, eval phase). Fix: training-side. The cluster did its job.

Allocatable but unallocated — fragmentation. The node has 3 GPUs free, but no job wants exactly 3. Fix: bin-packing scheduler policy, or a smaller-shape job class.

Not allocatable — the node has unschedulable taints, a NotReady status, or no device plugin. Fix: kubectl describe node, look at conditions and taints.

The metric you want is GPU-hours allocated / GPU-hours available, measured weekly. Healthy clusters sustain 75–90%. Below 60%, queue policy is the problem (jobs requesting odd shapes, no preemption). Above 95%, you're starving queued work — add nodes or lower priority for low-value experiments.

Upgrading

K8s ships a minor version every ~4 months. Don't skip more than two. The order is fixed: control plane first, then nodes, then add-ons. For managed clusters (EKS, GKE, AKS), the cloud handles the first step; you handle node pools yourself.

Before any upgrade:

# Backup etcd (self-managed)
ETCDCTL_API=3 etcdctl snapshot save backup.db

# Check API deprecations
kubectl convert -f manifests/ --output-version=v2

# Drain a single node first as a canary
kubectl drain node-gpu-001 --ignore-daemonsets --delete-emptydir-data
# ... upgrade kubelet, reboot ...
kubectl uncordon node-gpu-001

Security primitives you should have on day one

RBAC. Each namespace gets a default Role: read pods/logs, create Jobs, no node access. Cluster-admin is for two people, not twenty.

NetworkPolicy. Default-deny on every namespace, then allow only what the workload needs. Otherwise any compromised pod can hit etcd.

Pod Security Standards. Set baseline as the floor and restricted in non-training namespaces. Trainer pods often need privileged-ish access to GPUs and IB devices — keep those exceptions narrow.

Image provenance. Sign images with cosign; the admission webhook rejects unsigned ones.

Backups

Three things to back up:

etcd. Hourly snapshots, kept 7 days. This is the cluster.

PVCs. Use Velero with cloud-provider CSI snapshots. Datasets you can re-fetch; checkpoints you cannot.

Manifests. Live in Git. Always. The cluster is not a source of truth.
