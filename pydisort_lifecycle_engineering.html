Part I

The system & the stack

Before any single competency makes sense, you have to see the whole vertical and the build graph that produces it. This is the orientation: what pydisort is, as a life-cycle artifact, and why a one-line import is a multi-team supply chain in miniature.

The NVIDIA Senior Life Cycle Engineer brief names a specific competency set: continuous build and release for Linux, cloud-hosted CI/CD, C++ build systems with dependency, symbol-resolution, and linking problems, the full vertical “from end-users, through layers of open-source software, down to math libraries, compilers, and drivers,” and the discipline of shipping and supporting at scale. pydisort is a near-perfect miniature of that brief: a single repository that nonetheless exercises every layer.

The deep truth the brief is pointing at: a native Python extension is not a program, it is a contract — between a compiler ABI, a set of shared libraries, a Python ABI, an OS loader, and a hardware/driver pair. The life-cycle engineer’s job is to keep every clause of that contract true on every platform, on every release, forever. Most production incidents are a clause silently violated: an ABI flag flipped, a symbol unregistered, a library found at the wrong path.

Throughout, each chapter follows the same four-beat structure the brief implies: (1) the production issue you actually hit, (2) how pydisort configures it today, (3) whether and how to improve it, and (4) the decision tree / debugging reflex that makes you fast.

§01The software vertical, from import to libm

The brief’s phrase “through layers of open-source software, down to math libraries, compilers, and drivers” is the mental model to internalize first. When a user types import pydisort, the loader walks a dependency tower. Knowing the tower cold is what lets you place any bug at the right layer in seconds.

user code import pydisort; Disort(opts).forward(prop, ...) │ ┌────▼─────────────────────────────────────────────────────────┐ │ pydisort/__init__.py import torch → from .pydisort * │ Python ├────▼─────────────────────────────────────────────────────────┤ │ pydisort/pydisort*.so pybind11 module (PYBIND11_MODULE) │ C++ / Python C-API ├────▼─────────────────────────────────────────────────────────┤ │ libdisort_release.so DisortImpl, TensorIterator glue │ C++17 + LibTorch │ ├── libcdisort (OBJECT) the 1980s DISORT 2.1.3 solver │ C99 + libm │ └── (CUDA) libdisort_cuda_release.so gpu_kernel<12> │ CUDA / nvcc ├────▼─────────────────────────────────────────────────────────┤ │ libtorch_cpu.so libc10.so the USER's installed torch │ resolved by RPATH │ (+ libtorch_cuda, libc10_cuda when GPU) │ ├────▼─────────────────────────────────────────────────────────┤ │ libstdc++ / libgomp / libm toolchain runtime + math │ compiler / glibc ├────▼─────────────────────────────────────────────────────────┤ │ CUDA driver (libcuda.so) JIT-compiles PTX for the GPU │ driver / hardware └────────────────────────────────────────────────────────────────┘

Three properties of this tower drive everything later:

  • It is half-vendored, half-borrowed. libcdisort and libdisort_release are ours and ship inside the wheel. libtorch/libc10 are the user’s — they are deliberately not in the wheel (§11). The boundary between “ship it” and “find it at runtime” is the single most consequential packaging decision.
  • Each arrow is an ABI. Python↔C++ is the CPython ABI (why wheels are tagged cp310, cp311…). C++↔libtorch is the libstdc++ ABI (§04). C++↔CUDA↔driver is the PTX/SASS contract (§08). A mismatch at any arrow is a load-time crash, not a compile error.
  • The bottom layer is not in your control. The CUDA driver, the user’s glibc, the user’s torch build — production support means reasoning about their machine from your stack trace.
The placement reflex

When something breaks, name the layer before you name the bug. “ImportError: undefined symbol” is the C++↔libtorch arrow (§04/§05). “cannot open shared object file” is the loader/RPATH arrow (§06). “no kernel image is available” is the CUDA arch arrow (§08). Numerically-wrong-but-runs is the solver/precision arrow (§16). The layer tells you which tool to reach for.

§02The two-stage build graph

pydisort does not build with a single command. It builds in two stages with two different build systems, and understanding why is the foundation for every CI and packaging chapter.

STAGE 1 — CMake STAGE 2 — setuptools + torch cmake -B build pip install --no-build-isolation . cmake --build build --parallel └─ setup.py: torch.utils.cpp_extension ├─ libcdisort (OBJECT, C, libm) ├─ CppExtension (CPU) OR ├─ libdisort_release.so (C++/torch) └─ CUDAExtension (if torch.cuda) └─ build/lib/*.so ◄───────────────────┐ sources: python/csrc/*.cpp └────── links: build/lib/*.so (parsed!) output: pydisort/pydisort*.so

Stage 1 is plain CMake. It compiles the C solver as an OBJECT library, links it into libdisort_release.so against LibTorch, and drops the result in build/lib/. Stage 2 is setup.py, which uses PyTorch’s own cpp_extension.BuildExtension to compile the pybind11 module and link it against whatever Stage 1 produced. The bridge between the stages is literally a directory scan:

def parse_library_names(libdir):
    library_names = []
    for root, _, files in os.walk(libdir):
        for file in files:
            if file.endswith((".a", ".so", ".dylib")):
                file_name = os.path.basename(file)
                library_names.append(file_name[3:].rsplit(".", 1)[0])  # lib.so → NAME
    return library_names

current_dir = os.getenv("WORKSPACE", Path().absolute())
lib_dirs    = [f"{current_dir}/build/lib"]
libraries   = parse_library_names(f"{current_dir}/build/lib")   # whatever Stage 1 built

The Python ext links against the artifacts of the CMake stage by discovering them on disk — not against CMake targets. The WORKSPACE env var lets CI relocate the source tree.

Why two stages instead of one? Two opposing forces:

Why CMake for the libs

The C/C++/CUDA libraries need real dependency management: find_package(Torch), CUDA architecture handling, FetchContent for fmt/pybind11, per-compiler flags. CMake is the lingua franca here, and it is what the C++ unit tests link against too.

Why torch’s BuildExtension for the ext

cpp_extension knows the exact include paths, library paths, and — critically — the ABI and compile flags of the installed torch. It guarantees the Python ext is compiled with the same contract as the torch it will load against. Reinventing that in CMake is a maintenance trap.

The fragility this introduces

Stage 2 depends on Stage 1 having already run and on the build dir being co-located. That is why CI does cmake --build then pip install --no-build-isolation --no-deps . — --no-build-isolation so the install reuses the already-built build/lib and the already-installed torch, rather than spinning up a pristine build env that knows nothing about Stage 1. Get the order wrong, or build in isolation, and Stage 2 links against an empty build/lib and produces a useless .so.

Improve

The modern alternative is scikit-build-core, which makes CMake the build backend directly — one stage, declarative, no on-disk handshake, proper editable installs. The cost is teaching scikit-build-core to use torch’s exact flags (via find_package(Torch) + the cmake_prefix_path that cd.yml already computes). For a project this size the current hybrid is pragmatic; the moment a third build target appears, scikit-build-core pays for itself.

Decision tree — build backend

Pure-CMake project, no Python ext → CMake. Python ext over a torch op → torch.utils.cpp_extension (this repo’s Stage 2). Python ext + substantial non-torch C++ libs + want one command → scikit-build-core. Header-only / pybind-only, no torch → pybind11’s own setup_helpers or scikit-build-core.

Part II

C++ build systems: dependencies, symbols, linking

The brief calls this out by name as a way to stand out — “dependency issues, symbol resolution, and linking problems.” It is also where native extensions actually break in production. Four chapters, one per failure mode.

§03Dependency acquisition & pinning

pydisort acquires its C++ dependencies three different ways, and each represents a distinct point on the reproducibility-vs-convenience curve.

DepMechanismPin & tradeoff
cdisort 2.1.3vendored in-tree (cdisort213/)maximally reproducible; the solver never moves under you. Cost: you own its security/patching.
fmt 12.1.0FetchContent + git tagpinned to a tag, cached as a tarball; reproducible unless the tag is re-pointed upstream
pybind11 2.11.1FetchContent + git tagsame; must stay compatible with the torch pybind it co-exists with
torch 2.10.0pip, exact == pinthe hard constraint — both a build requirement and a runtime dependency (§12)

The FetchContent path is wrapped in a small macro that adds a tarball cache so repeated CI configures don’t re-clone:

if(EXISTS ${CACHE_FILE})                              # .cache/-.tar.gz
  FetchContent_Declare(${name} DOWNLOAD_COMMAND tar -xzf ${CACHE_FILE} -C ...)
else()
  FetchContent_Declare(${name}
    GIT_REPOSITORY ${url} GIT_TAG ${tag}
    PATCH_COMMAND ${patch} UPDATE_DISCONNECTED TRUE)               # don't re-fetch on reconfigure
endif()
FetchContent_MakeAvailable(${name})

Production issues

  • Git tags are mutable. A tag is a pointer; upstream can move it. For true reproducibility, pin a commit SHA or add FetchContent hash verification. The tarball cache helps only after the first fetch.
  • Network in CI. A first-time configure clones two repos from GitHub. A GitHub outage breaks your build. The cache tarball + UPDATE_DISCONNECTED mitigates; FETCHCONTENT_FULLY_DISCONNECTED=ON with a pre-seeded cache eliminates it.
  • The diamond. torch also bundles a pybind11. pydisort fetches its own 2.11.1. If those two pybind11s disagree on the internal type registry layout, you get subtle cross-module cast failures. The fix is to keep your fetched pybind close to torch’s, or build the ext purely against torch’s pybind (it ships the headers).
  • Vendored cdisort is a security surface. A 1980s C codebase frozen in-tree never gets upstream CVE fixes because there is no upstream cadence. That is acceptable for a numerics kernel with no untrusted input, but it must be a conscious decision recorded somewhere.
Decision tree — how to acquire a C++ dep

Tiny, stable, no upstream cadence (a numerics kernel) → vendor. Active library you track by version (fmt, pybind11) → FetchContent + pinned SHA + cache. Heavy, ABI-defining, the user already has it (torch) → depend on it, pin exactly, do not bundle (§11). System primitive (libm, libstdc++) → find, never ship.

One small but real detail: the top-level CMake sets cmake_policy(SET CMP0074 NEW) so that <Package>_ROOT environment variables are honored by find_package — the standard escape hatch for telling CMake where a hand-installed dependency lives. Knowing the CMP policy numbers is exactly the kind of build-system fluency the brief rewards.

§04The dual C++11 ABI problem

This is the single most famous linking problem in the PyTorch-extension world, and pydisort handles it explicitly. In 2015 GCC 5 changed the libstdc++ ABI for std::string and std::list (the “C++11 ABI”), gated behind the macro _GLIBCXX_USE_CXX11_ABI. Two object files compiled with different values of that macro have different mangled names for any function passing a std::string — they will fail to link, or worse, link and corrupt memory.

PyTorch ships wheels built with a specific ABI value. Every line of C++ you link into the same process must agree, or symbols silently fail to resolve.

pydisort does not guess; it interrogates the installed torch and propagates its answer:

execute_process(
  COMMAND ${Python3_EXECUTABLE} -c
    "import torch; print(int(torch._C._GLIBCXX_USE_CXX11_ABI))"
  OUTPUT_VARIABLE _ABI_OUTPUT  OUTPUT_STRIP_TRAILING_WHITESPACE)

add_compile_definitions(_GLIBCXX_USE_CXX11_ABI=${_ABI_OUTPUT})   # match torch, whatever it is

The FindTorch.cmake module reinforces this with a phony interface target that propagates the flag to anything that links it — so the ABI travels with the dependency, not as a global:

add_library(torch_cxx11_abi INTERFACE IMPORTED)
execute_process(COMMAND ${Python3_EXECUTABLE} -c
  "import torch; print(torch.compiled_with_cxx11_abi())" OUTPUT_VARIABLE TORCH_CXX11_ABI ...)
target_compile_definitions(torch_cxx11_abi
  INTERFACE _GLIBCXX_USE_CXX11_ABI=$<BOOL:${TORCH_CXX11_ABI}>)

Stage 2 gets the same guarantee for free: torch’s own BuildExtension compiles the pybind module with torch’s ABI flags. Both stages therefore agree by construction.

How it manifests in production

  • The classic symptom: undefined symbol: _ZN3c107WarningC1ENSt7__cxx1112basic_stringE... on import. The __cxx11 token in the mangled name is the tell — one side wants the new ABI, the other the old.
  • The nastier symptom: it links and imports, then segfaults or returns garbage when a std::string crosses the boundary, because the two ABIs lay the string out differently. This is the “links but corrupts” failure the brief implies.
  • macOS is immune — it uses libc++ (Clang’s STL), not libstdc++, so there is no _GLIBCXX_USE_CXX11_ABI. This is why the detection is guarded by CMAKE_SYSTEM_NAME STREQUAL "Linux" in FindTorch.
The trap

Modern PyTorch (2.x on manylinux_2_28) ships with the new ABI (=1), reversing a decade of =0 defaults. A stale tutorial that hard-codes add_definitions(-D_GLIBCXX_USE_CXX11_ABI=0) — note that exact line is commented out in this repo’s CMakeLists, a fossil of the old advice — will produce undefined-symbol errors against a modern torch. Always derive the value; never hard-code it.

Debugging reflex

# what ABI does the user's torch want?
python -c "import torch; print(torch.compiled_with_cxx11_abi())"

# does the offending symbol carry the __cxx11 tag?
nm -D -C build/lib/libdisort_release.so | grep -i string | head
c++filt _ZN3c107Warning...                 # demangle the undefined symbol from the error

§05Symbols: the device-dispatch registry

The most interesting symbol-resolution problem in pydisort is not a missing library — it is ATen’s DispatchStub, the mechanism by which one logical operation (call_disort) resolves to a CPU or CUDA implementation at runtime. It is a small registry built entirely out of the C++ static-initialization order, and it is a beautiful, fragile example of cross-translation-unit symbol wiring.

The pattern has three keywords in three different files:

// disort_dispatch.hpp — DECLARE the stub (a global function-pointer table)
using disort_fn = void (*)(at::TensorIterator&, int, disort_state*, disort_output*);
DECLARE_DISPATCH(disort_fn, call_disort);

// disort_dispatch.cpp — DEFINE the stub, then REGISTER the CPU impl
DEFINE_DISPATCH(call_disort);
REGISTER_ALL_CPU_DISPATCH(call_disort, &disort::call_disort_cpu);

// disort_dispatch.cu — REGISTER the CUDA impl into the same table
REGISTER_CUDA_DISPATCH(call_disort, &disort::call_disort_cuda);

At call time, forward() simply hands the table the tensor’s device and lets it pick:

at::native::call_disort(flx.device().type(),  // CPU or CUDA — chosen at runtime
                        iter, options->upward(), ds_.data(), ds_out_.data());

The REGISTER_* macros expand to a static object whose constructor writes the function pointer into the table. That is the crux: registration happens as a side effect of loading the object file. And that is exactly where the symbol-resolution problems live.

The dead-strip / never-loaded trap

If the CUDA registration lives in a separate library (libdisort_cuda_release.so) that the process never actually loads — or if the linker dead-strips the registration object because “nothing references it” — then the static constructor never runs, the table entry stays null, and you get RuntimeError: Could not run 'call_disort' with arguments from the 'CUDA' backend at runtime, with a perfectly successful build. The fix is to force the registration TU to be kept: --whole-archive / -Wl,--no-as-needed on the library, or ensure the symbol is referenced. This is the canonical “builds fine, fails at runtime” symbol bug.

Production issues & how pydisort is exposed to them

  • Private-header coupling. ATen/native/DispatchStub.h is a torch-internal header with no stability guarantee. pydisort reaches into at::native — a deliberate, powerful choice that buys zero-overhead dispatch but couples the build to torch internals that can change between minor versions. This is a direct contributor to the exact-pin in §12.
  • One process, one registry. The dispatch table is a process-global singleton inside libtorch. If two copies of libtorch are loaded (the vendoring mistake in §11), you get two registries and half your registrations land in the wrong one. This connects symbol resolution directly to the packaging decision.
  • CPU vs all-CPU. REGISTER_ALL_CPU_DISPATCH registers for every CPU capability tier (AVX2/AVX512); the HAVE_AVX512_CPU_DEFINITION/HAVE_AVX2_CPU_DEFINITION defines in compilers.cmake are what make those tiers real. Drop them and the AVX-tier stub may resolve to a generic path or fail to register the SIMD variant.

Debugging reflex

# Is the dispatch symbol even in the lib?
nm -D -C build/lib/libdisort_release.so | grep call_disort

# Watch which libs the loader pulls in — did the CUDA reg lib load at all?
LD_DEBUG=libs python -c "import pydisort" 2>&1 | grep disort

# Confirm the registration object survived the link (not dead-stripped)
nm build/lib/libdisort_cuda_release.so | grep -i register

§06Runtime linking: RPATH, RUNPATH & $ORIGIN

Once symbols resolve at build time, they still have to be found at load time. A shared library records where to look for its dependencies in its dynamic RPATH/RUNPATH entries. pydisort encodes a deliberate two-part search policy:

if sys.platform == "darwin":
    extra_link_args = ["-Wl,-rpath,@loader_path/lib",
                       "-Wl,-rpath,@loader_path/../torch/lib"]
else:  # linux
    extra_link_args = ["-Wl,-rpath,$ORIGIN/lib",           # OUR libdisort, shipped in the wheel
                       "-Wl,-rpath,$ORIGIN/../torch/lib"]   # the USER's torch, next to us in site-packages

Read those two RPATH entries as a sentence: “find my own helper libs beside me, and find torch in the sibling torch package.” The layout that makes it work:

site-packages/ ├── pydisort/ │ ├── pydisort.cpython-311-x86_64-linux-gnu.so ← has RPATH $ORIGIN/lib, $ORIGIN/../torch/lib │ └── lib/ │ └── libdisort_release.so ← found via $ORIGIN/lib └── torch/ └── lib/ ├── libtorch_cpu.so ← found via $ORIGIN/../torch/lib └── libc10.so

$ORIGIN (Linux) and @loader_path (macOS) both mean “the directory of the file doing the loading,” resolved by the dynamic loader at runtime. This is what makes a relocatable wheel possible: no absolute paths are baked in.

RPATH vs RUNPATH — the non-transitivity gotcha

Modern GNU linkers default to --enable-new-dtags, so -Wl,-rpath actually emits DT_RUNPATH, not the legacy DT_RPATH. The difference is fatal to debug if you don’t know it: RUNPATH is not transitive. It is consulted only for the direct dependencies of the object that carries it. So pydisort.so’s RUNPATH finds libtorch_cpu.so — but libtorch_cpu.so’s own dependencies are not searched via pydisort’s RUNPATH. They rely on torch’s own RPATH (which torch sets up) or LD_LIBRARY_PATH. When a torch sub-dependency goes missing, this is why.

This is also why the CI test step sets LD_LIBRARY_PATH/DYLD_LIBRARY_PATH to build/lib: the C++ test binaries are built without the wheel’s package-relative layout, so they need an explicit search path to find libdisort_release.so.

Production issues

  • macOS uses install-names, not just RPATH. Each dylib records an “install name” (often @rpath/libfoo.dylib); delocate rewrites these during wheel repair (§11). A mismatched install-name yields Library not loaded: @rpath/... — the mac analogue of the Linux RUNPATH miss.
  • SIP & DYLD_* stripping. macOS System Integrity Protection strips DYLD_LIBRARY_PATH from the environment of protected processes, so you cannot always paper over a bad install-name with an env var the way you can on Linux. The repo works around this with DYLD_FALLBACK_LIBRARY_PATH in the CD repair step, which SIP does not strip.
  • The @loader_path/../torch/lib bet. It assumes torch is installed as a sibling package in the same environment. In a conda env where torch lives elsewhere, or a vendored-torch wheel, the bet fails. This is acceptable because the wheel declares a torch dependency, so pip will always co-install torch in the same site-packages.

Debugging reflex

# Linux: what's recorded, and what's missing?
patchelf --print-rpath pydisort/pydisort*.so       # shows $ORIGIN entries
ldd pydisort/pydisort*.so | grep 'not found'        # the smoking gun
readelf -d pydisort/pydisort*.so | grep -E 'RPATH|RUNPATH|NEEDED'

# macOS: install names & rpaths
otool -L pydisort/pydisort*.so                      # dependent dylibs + install names
otool -l pydisort/pydisort*.so | grep -A2 LC_RPATH  # the rpath entries
Decision tree — “cannot open shared object file”

Is the lib in the wheel? → unzip -l the wheel; if not, it should be in package-data or repaired in (§11). Is the RPATH right? → patchelf --print-rpath; fix in setup.py link args. Is it a transitive dep of torch? → not your RUNPATH’s job; the user’s torch install is broken. Is it macOS @rpath? → check install-names with otool -L; delocate should have rewritten them.

Part III

Performance portability

The reason torch is in the stack at all: one kernel, two devices, and a path to the GPU. This is where deep-learning-framework engineering and classical numerics meet — and where pydisort is most ambitious and most unfinished.

§07The dispatch architecture: TensorIterator

pydisort’s performance-portability strategy is to lean entirely on ATen’s TensorIterator — the same machine that powers element-wise ops in PyTorch. The insight: a radiative-transfer problem is embarrassingly parallel across (wavelength, column) pairs. Each pair is one independent DISORT solve. So pydisort frames the whole batch as a single broadcasted element-wise operation whose “element” is one atmospheric column.

auto iter = at::TensorIteratorConfig()
    .resize_outputs(false)
    .check_all_same_dtype(true)
    .declare_static_shape({nwave, ncol, ds().ntau, 2}, /*squash_dims=*/{2, 3})
    .add_output(flx)
    .add_input(prop)
    .add_owned_input(bc->at("umu0").view({1, ncol, 1, 1}).expand({nwave, ncol, 1, 1}))
    // ... 10 more broadcast boundary-condition inputs ...
    .add_input(index)                  // flat (wave,col) index → picks the right ds_[]
    .build();

at::native::call_disort(flx.device().type(), iter, options->upward(), ds_.data(), ds_out_.data());

The cleverness is the index input: each element carries its own flat index, which the kernel uses to look up a preallocated, per-(wave,col) DISORT state struct. That makes the kernel reentrant — no shared mutable state across elements — which is precisely what lets the same loop body run single-threaded, multi-threaded, or on the GPU.

The CPU kernel then loops with ATen’s grain-sized parallelism:

int grain_size = iter.numel() / at::get_num_threads();
iter.for_each([&](char** data, const int64_t* strides, int64_t n) {
    for (int i = 0; i < n; i++) {
      int idx = static_cast<int>(*idxf);
      disort_impl(out, prop, umu0, /*...*/, ds[idx], ds_out[idx], nprop);  // the C solver
    }
  }, grain_size);

Production issues

  • Thread safety hinges on per-element state. Because each element owns its own ds_[idx], the parallel for_each is safe. The instant someone makes cdisort touch a shared scratch buffer, the parallelism becomes a data race. The per-(wave,col) allocation in reset() is therefore load-bearing, not just tidy.
  • Grain size = numel/num_threads gives one chunk per thread — fine when columns cost roughly the same, poor when optical depths vary wildly (a thick column is far more expensive than a thin one), causing load imbalance. A smaller grain with work-stealing would balance better at the cost of more scheduling overhead.
  • Precision boundary. cdisort is internally double; gather_flx/gather_rad wrap the C output structs with torch::from_blob(..., kFloat64) and copy into the result tensor’s dtype. Run the module in float32 and you silently downcast double results — correct, but a precision cliff to document.
  • Memory: nwave × ncol fully-allocated DISORT states can be large; reset() frees and reallocates the whole array whenever options change, which is O(nwave·ncol) churn if called in a hot loop.
Improve

Profile column-cost variance; if high, switch to a finer grain or sort columns by optical depth for better cache/branch behavior. Expose grain_size as a tunable. Add a fast-path that skips reset() reallocation when dimensions are unchanged.

§08The CUDA build matrix

The GPU story is where the brief’s “compilers and drivers” layer becomes concrete. A CUDA build must decide which GPU architectures to compile for, and that decision is a direct binary-size / compatibility / build-time tradeoff. pydisort encodes a version-aware policy:

# 60:P100 61:1080Ti 70:V100 75:RTX4000 80:A100 86:3090 89:4090 90:H100/H200 120:5090
set(_cuda_arch_list 60 61 70 75 80 86 89)

if(CUDAToolkit_VERSION VERSION_GREATER_EQUAL "13.0")
  list(REMOVE_ITEM _cuda_arch_list 60 61 70)   # nvcc 13 drops sm_<75
endif()
if(CUDAToolkit_VERSION VERSION_GREATER_EQUAL "12.0")
  list(APPEND _cuda_arch_list 90 120)          # Hopper + Blackwell
endif()
set(CMAKE_CUDA_ARCHITECTURES ${_cuda_arch_list})

Each architecture in the list becomes a SASS (native machine code) blob baked into a fat binary. The mental model every GPU life-cycle engineer needs:

ConceptWhat it is & why it matters
SASSarch-specific machine code (sm_80). Runs only on that arch. Compiled ahead of time.
PTXarch-neutral virtual ISA. The driver JIT-compiles it to SASS at first launch on an unknown GPU — this is forward compatibility.
fatbinthe container holding SASS for listed archs + (optionally) PTX. Bigger arch list = bigger binary + longer nvcc compile.
compute capabilitythe GPU’s arch number. The runtime picks the matching SASS, or JITs PTX, or errors.
The error every CUDA shipper sees once

CUDA error: no kernel image is available for execution on the device means the fatbin has no SASS for the GPU’s arch and no PTX to JIT from. The defense is to always include a PTX for the highest arch (so newer GPUs JIT forward) — e.g. set CMAKE_CUDA_ARCHITECTURES to include a 90-virtual entry. pydisort lists real arches only, so a GPU newer than the toolkit knows about would hit this; adding a virtual/PTX tier closes the gap.

The honest state of pydisort’s GPU path

Two things a life-cycle engineer must surface plainly: (1) the CUDA kernel in disort_dispatch.cu is a scaffold — the dispatch is wired with gpu_kernel<12> and CUDAGuard, but the actual disort_impl(...) call inside the GPU lambda is commented out. The device path registers but does not yet compute. (2) The released wheels are CPU-only: setup.py chooses CUDAExtension vs CppExtension from torch.cuda.is_available() at build time, and the CD runners have no GPU, so every published wheel is the CPU build. The GPU capability exists in the source and the CMake option, but it is not in the product yet.

Improve — the path to shipping GPU wheels

  • Decouple the build decision from torch.cuda.is_available() (a builder with the toolkit but no device should still produce a CUDA wheel) — key off the CUDA toolkit presence or an explicit flag.
  • Add GPU-equipped CI runners (self-hosted or a cloud GPU pool) so the CUDA kernel can be tested for parity against the CPU goldens (§16).
  • Adopt the PyTorch convention of +cu124-style local version labels and a separate package index, because a CUDA wheel’s torch dependency is itself CUDA-versioned — multiplying the matrix.
  • Trim the arch list per release channel: a debug/dev wheel can be native (one arch, fast build); a release wheel needs the broad list + PTX.

Debugging reflex

# list the SASS/PTX images baked into the fatbin
cuobjdump --list-elf  build/lib/libdisort_cuda_release.so
cuobjdump --list-ptx  build/lib/libdisort_cuda_release.so

# memory errors / illegal accesses in the kernel

§09The torch integration surface

pydisort presents itself as a first-class torch module, which buys interoperability (it composes with nn.Sequential, gets .to(device), prints nicely) but also imports torch’s harder semantics. The class is a Cloneable module bound through torch’s pybind helper:

class DisortImpl : public torch::nn::Cloneable<DisortImpl> {
  DisortOptions options;                  // = std::shared_ptr<DisortOptionsImpl>
  void reset() override;                   // Cloneable requires this; (re)allocates ds_
  torch::Tensor forward(torch::Tensor prop, std::map<std::string, torch::Tensor>* bc, ...);
};
TORCH_MODULE(Disort);                       // the shared_ptr wrapper type

The pybind layer (ADD_DISORT_MODULE macro + a kwargs-driven forward) turns this into the ergonomic Python API seen in tests:

from pydisort import DisortOptions, Disort, scattering_moments
op = DisortOptions().header("Isotropic Scattering Case 1")
op.flags("usrtau,usrang,lamber,quiet,intensity_correction,...")
op.ds().nlyr = 1; op.ds().nmom = 16
ds = Disort(op)
flux = ds.forward(prop, fbeam=..., umu0=...)   # boundary conditions as kwargs

Production issues unique to the torch surface

  • Shared options are a shared mutable singleton. DisortOptions is a shared_ptr deliberately shared across instances; the header itself warns that reset() “modifies the shared options object — all instances sharing it will see the change.” That is a documented footgun: cloning a module does not clone its config, so a reset on one clone reconfigures its siblings. Powerful for ensembles, dangerous by accident.
  • No backward = no gradients. forward() exists; there is no autograd backward. The cdisort solver is not differentiable, so even though the result is a torch tensor, gradients will not flow through a DISORT solve. Anyone wiring pydisort into an optimization loop expecting loss.backward() to reach the optical properties will get zeros / a graph break. Surfacing this is a support necessity.
  • Type erasure with default args. The FORWARD_HAS_DEFAULT_ARGS macro is required so torch’s AnyModule type erasure can carry the default bname/temf arguments — a torch-version-sensitive incantation.
  • Cloneable contract. Cloneable<T> deep-copies parameters/buffers and calls reset() on the clone. Since pydisort holds its real state in plain std::vector<disort_state> (not registered buffers), the clone’s correctness depends entirely on reset() rebuilding that state from options — which it does, but only because options are shared.
Improve

If a differentiable RT solve is ever needed, wrap forward in a custom torch::autograd::Function with an analytic or finite-difference adjoint. Short of that, document loudly that the module is forward-only. Consider a clone_options() path for users who want independent configs, removing the shared-mutation surprise.

Part IV

Packaging & release management

“Continuous build and release management for Linux” and “shipping multi-cloud large-scale systems” — the part of the brief about turning a build into an artifact the world can pip install. For a torch extension, this is the hardest packaging problem in the ecosystem.

§10Wheels, manylinux & cibuildwheel

A wheel is a zip with a compatibility tag. For a native extension the tag is a promise: “this binary will load on any machine matching {python}-{abi}-{platform}.” pydisort uses cibuildwheel to manufacture and validate that promise across the matrix.

CIBW_PLATFORM: linux
CIBW_MANYLINUX_X86_64_IMAGE: manylinux_2_28      # glibc 2.28 floor (≈RHEL/Alma 8)
CIBW_BUILD: cp${{ env.PY_VERSION }}*             # one Python at a time: cp310..cp313
CIBW_ARCHS: x86_64
CIBW_SKIP: "*musllinux*"                       # no Alpine/musl builds
CIBW_BEFORE_BUILD: |
  pip install numpy 'torch==2.10.0'
  cmake -B build -DCMAKE_PREFIX_PATH="$TORCH_CMAKE_PREFIX_PATH" ...   # STAGE 1 inside the container
  cmake --build build --parallel

The structure to internalize:

KnobMeaning & production consequence
manylinux_2_28builds inside a container with glibc 2.28 so the wheel runs on any newer glibc. A higher floor = simpler build, fewer old-distro users. The single most important portability lever on Linux.
CIBW_BEFORE_BUILDruns Stage 1 (CMake) inside the manylinux container, so the native libs are built against the container’s old toolchain — matching the wheel’s glibc promise.
per-Python loopa torch ext is not abi3 (it uses the version-specific CPython API via pybind + torch), so you build cp310, cp311, cp312, cp313 separately — 4× the work.
HOST_PYTHON 3.11the Python that runs cibuildwheel is fixed; the matrix Python is what gets built for. Conflating the two is a classic CD bug.

macOS mirrors this with CIBW_PLATFORM: macos, CIBW_ARCHS: arm64, and a per-Python matrix.

A real inconsistency worth fixing

The macOS deployment target is set in two places that disagree: CMakeLists.txt says CMAKE_OSX_DEPLOYMENT_TARGET "12.0", while cd.yml exports MACOSX_DEPLOYMENT_TARGET: 15.0. The wheel’s platform tag is derived from the env var (15.0), but the CMake-built lib targets 12.0. They happen to be compatible (a 12.0 lib runs on 15.0), but the wheel will claim a 15.0 minimum and exclude macOS 12–14 users unnecessarily. Pick one floor and set it in both places.

Decision tree — the glibc / OS floor

Need the broadest reach → lowest practical manylinux (2_17), accept an older toolchain. Build needs C++17/modern CUDA → 2_28 (pydisort’s choice). Internal/HPC only → build for the exact target image and skip manylinux entirely. Apple Silicon only → arm64 + a deliberate min-OS; add x86_64 only if Intel Macs still matter to you.

§11auditwheel / delocate: the exclude strategy

This is the most important packaging decision in the whole repo, and it is a single flag. After cibuildwheel builds the wheel, it runs a “repair” step that normally vendors every external shared library into the wheel to make it self-contained. pydisort deliberately tells it not to:

# Linux: do NOT copy any lib*.so into the wheel
CIBW_REPAIR_WHEEL_COMMAND_LINUX:
  auditwheel repair --exclude lib*.so -w {dest_dir} {wheel}

# macOS: copy nothing for torch or our own disort lib
CIBW_REPAIR_WHEEL_COMMAND_MACOS:
  delocate-wheel -e torch -e disort_release -w {dest_dir} {wheel}

Why on earth would you ship a non-self-contained wheel? Because the alternative is catastrophic:

If you vendored torch

The wheel would balloon by ~2–3 GB. Worse: the process would now have two copies of libtorch/libc10 — the user’s and yours. libtorch’s dispatch table, CUDA caching allocator, and type registry are process-global singletons. Two copies means two registries, half-initialized state, and crashes like c10::Error or duplicate-registration aborts. (§05 is exactly this failure.)

So instead

Exclude torch entirely; resolve it at runtime from the user’s install via the $ORIGIN/../torch/lib RPATH (§06). Ship only our own libdisort_release.so — but not even via the repair tool: via package-data, placed in pydisort/lib/ and found via $ORIGIN/lib.

The package-data half of the deal, from pyproject.toml:

[tool.setuptools.package-data]
"pydisort" = ["disort/**/*", "cdisort213/*", "lib/*.so", "lib/*.dylib", "*.pyi", "py.typed"]

The whole strategy in one sentence: vendor what is uniquely yours and small (libdisort), borrow what is the user’s and huge and singleton-bearing (libtorch), and ship nothing that the user must already have. The exact torch==2.10.0 dependency (§12) is the contract that makes “borrow at runtime” safe.

The consequence to own in support

This wheel is not a standalone manylinux wheel in spirit, even though it carries the tag. It will fail to import in any environment where a compatible torch is not present at the expected sibling path. That is by design, but it means “pip install pydisort then import fails” tickets are almost always “torch missing / wrong version / wrong location” — check that first.

Decision tree — vendor vs. depend (the general rule)

  • Library is small + uniquely yours + no global singletons → vendor it into the wheel (libdisort).
  • Library is huge, ABI-coupled, the user already has it, and holds process-global state → depend on it, exclude from repair, find via RPATH (libtorch).
  • Library is a system primitive (libm, libstdc++, libgomp) → never ship; rely on the manylinux policy / the user’s OS.

§12Versioning & ABI compatibility windows

Everything above resolves into one line that appears three times in the repo:

# [build-system].requires
"torch==2.10.0"
# [project].dependencies
"torch==2.10.0"
# and in CI/CD: pip install 'torch==2.10.0'

Why == and not >=? Because LibTorch has no stable C++ ABI across releases. The internal headers pydisort relies on — ATen/native/DispatchStub.h, TensorIterator, the gpu_kernel machinery — change shape between minor versions. A pydisort built against 2.10.0’s internals, loaded against a user’s 2.11.0, can hit undefined symbols, changed vtables, or altered struct layouts — the §04/§05 failures, but caused by version drift instead of ABI flags.

Stability tierWhat you can depend on
Python APIfairly stable across torch minors — the safe surface
Stable C++ (torch::stable)a small, explicitly-stable subset; pydisort does not restrict itself to it
ATen internals (at::native)no stability guarantee — pydisort lives here, hence the exact pin
CPython ABIstable within a minor (cp311), changes across (why the per-Python matrix)
The cost & the cadence

An exact pin means every torch release forces a pydisort rebuild-and-republish, and users cannot upgrade torch independently. That is the price of using ATen internals for zero-overhead dispatch. The release engineering implication: a torch release is a trigger event for your pipeline. Track the torch release calendar; treat “bump torch” as a first-class, tested change (re-run the full goldens in §16, because numerics can shift with their kernels too).

Improve

  • Run a compatibility matrix in CI against the last N torch minors to learn your true window empirically, even if you ship one pin.
  • Where feasible, migrate hot interfaces to the torch::stable ABI to widen the window (and reduce republish toil) — at the cost of the convenience of at::native.
  • Consider ~=2.10 only after proving compatibility; never blindly loosen a native-ABI pin.
Part V

CI / CD

The literal center of the role: “continuous integration, testing, and delivery.” pydisort has three workflows — a gate, a publisher, and a release robot. Here is each, with the production gaps a senior engineer would close.

§13Continuous integration: the gate

ci.yml runs on every PR and push to main, as a two-job pipeline: a fast style gate, then a build-and-test matrix that depends on it.

on: pull_request / push(main) │ ├─ job: pre-commit (ubuntu) ── cpplint, cppcheck, pre-commit hooks │ │ needs: ✓ ├─ job: build-and-test matrix = { ubuntu-latest, macos-latest } × py3.11 │ ├─ install torch==2.10.0 (CPU wheel on Linux via download.pytorch.org/whl/cpu) │ ├─ cmake -B build -DBUILD_TESTS=ON ← Stage 1 │ ├─ cmake --build build --parallel │ ├─ pip install --no-build-isolation --no-deps . ← Stage 2 │ └─ ctest --output-on-failure -E test_disort_09 ← C++ + Python tests │ env: LD_LIBRARY_PATH / DYLD_LIBRARY_PATH = build/lib

Several decisions here are textbook, and a few are gaps:

DecisionRead

--no-build-isolationgood — reuses Stage-1 artifacts & the installed torch (the only correct way given §02)

Improve

  • Close the CI/CD Python gap: run the test matrix over the same {3.10, 3.11, 3.12, 3.13} CD ships. If minutes are tight, test the endpoints (3.10 + 3.13) on PRs and the full set nightly.
  • Cache Stage 1: the CMake .cache tarballs and build/ are re-created every run; an actions/cache keyed on the lockfile + CMake files cuts minutes.
  • Add a wheel-smoke job that builds a wheel and pip installs it into a clean venv with only the declared deps — this is the test that would have caught a broken RPATH or a missing package-data entry before users do.
  • Set fail-fast: false for the OS matrix so you see all platform results from one run.

§14Continuous delivery to PyPI

cd.yml is the publisher. It is workflow_dispatch — manually triggered with branch and OS inputs — and fans out into a small DAG:

workflow_dispatch (branch, build_os ∈ {Both, MacOS, Ubuntu}) │ concurrency: deploy-pypi-, cancel-in-progress ├─ prepare (resolve branch) ├─ build-macos matrix py3.10–3.13 → cibuildwheel → upload artifact (per py) ├─ build-linux matrix py3.10–3.13 → cibuildwheel → upload artifact (per py) └─ publish-pypi (needs both; if: always()) download all wheels-* → dist/ → pypa/gh-action-pypi-publish

The OIDC upgrade (free security win)

PyPI Trusted Publishing exchanges GitHub’s OIDC token for a short-lived, scoped upload credential — no secret to leak, rotate, or scope-creep. The id-token: write permission is already present; the change is to configure the publisher on PyPI and drop the password: line. For a project that ships to a global index, this is the highest-leverage supply-chain hardening available.

Improve — the release safety net

  • Insert a TestPyPI dry-run + a clean-venv install/import/smoke-test job between build-* and publish-pypi. Publishing untested binaries to a permanent, immutable index (you cannot overwrite a released version) is the riskiest step in the whole life cycle.
  • Generate and attach build provenance / attestations (the publish action supports it) so consumers can verify the wheels came from this workflow.
  • Make the publish tag-triggered (see §15) so “merge → tag → publish” is one chain, not a manual hand-off that can be forgotten.

§15Release automation

The third workflow, bump-and-tag.yaml, fires when a PR is closed (merged) into main or nightly. It bumps the version, tags it, and writes the release notes:

on:
  pull_request:
    types: [closed]
    branches: [main, nightly]
steps:
  - uses: jasonamyers/github-bumpversion-action   # bumps __init__.py via .bumpversion.cfg
  - uses: ad-m/github-push-action                  # pushes the bump commit + tag
  - uses: octokit/request-action                   # POST .../releases/generate-notes
  - uses: actions/create-release                   # cut the GitHub release
  - uses: actions/github-script                    # comment the notes back on the PR

The single source of version truth is small and clean:

[bumpversion]
current_version = 1.7.3
commit = True
tag = True
[bumpversion:file:python/__init__.py]
search  = __version__ = "{current_version}"
replace = __version__ = "{new_version}"

The git log confirms the model in action: a steady stream of “Bump version: 1.7.2 → 1.7.3” commits, one per merged PR.

Production issues

  • Every merge is a patch bump. The automation always increments the patch component, so semantic intent (was this a feature? a breaking change?) is lost. SemVer becomes “build counter.” A label-driven bump (major/minor/patch from PR labels) restores meaning.
  • The bot pushes back to main. A workflow that commits to the branch that triggers it risks a loop and definitely re-triggers push-gated CI. The standard guards: a [skip ci] in the bump commit and/or path filters; verify those are present.
  • Tag ≠ publish. bump-and-tag creates a tag and a GitHub release, but cd.yml is manual. So a tagged release is not on PyPI until a human dispatches the deploy. That decoupling is a deliberate safety choice, but it is also where “we tagged 1.7.3 but forgot to publish it” happens.
Improve — close the loop

Wire cd.yml to also trigger on: push: tags: ['v*'], gated behind the wheel-smoke job from §14. Then the chain is fully automatic and safe: merge → bump & tag → build → test wheels → publish via OIDC. Keep the manual workflow_dispatch as the break-glass path for re-releases.

Part VI

Quality, docs, debugging & operations

“Drives software quality and reliability” and “debugs issues as they arise, fixes problems across the entire stack.” The disciplines that keep a shipped extension trustworthy — and the toolkit for when it isn’t.

§16Testing & numerical goldens

pydisort tests at two levels, both wired through ctest:

  • C++ unit tests — test_disort, test_clone, built by the setup_test macro and linked against pydisort::disort. These exercise the C++ API and the module-clone semantics (§09) directly, below the Python layer.
  • Python parity tests — test_disort_01..09, globbed and registered as ctest cases run with python3. Each mirrors a reference case from the original cdisort C suite. test_disort_01 reproduces the six Van de Hulst isotropic-scattering cases (a 1980 standard reference table) — these are numerical golden tests.

Golden tests are the right tool here because radiative transfer has known analytic/tabulated answers, so correctness is checkable against published physics with assert_allclose:

from numpy.testing import assert_allclose, assert_equal
# Reference: Van de Hulst (1980), Multiple Light Scattering, Tables — VH1 Table 12.
assert_allclose(flux_up, reference_up, rtol=...)   # physics is the oracle

Production issues unique to numerical goldens

  • Tolerance is a contract. Too tight and harmless compiler/optimization changes turn the suite red; too loose and real regressions slip. The build uses -O3 -funroll-loops -funroll-all-loops -fstrict-aliasing (no -ffast-math, mercifully), so reassociation is bounded — but cross-compiler and cross-arch float results will differ in the last bits. Tolerances must be set per-quantity, not globally.
  • CPU/GPU parity (future). The moment the CUDA kernel (§08) computes, it needs to match the CPU goldens within tolerance — a parity suite, not just a self-consistency check. GPU floating-point ordering differs from CPU, so the tolerance story gets harder.
  • The excluded test. test_disort_09 is skipped in CI (§13). A golden test that “doesn’t pass on CI’s hardware” is itself a finding — it usually means a tolerance that is platform-sensitive, which is worth root-causing rather than excluding permanently.
Improve

Add property-based tests (e.g. energy conservation: out-flux ≤ in-flux for a non-emitting layer; reciprocity relations) that hold for any input, complementing the fixed goldens. Record tolerances and their justification next to each assertion. Run the goldens as part of the “bump torch” change (§12), since their kernels can shift your last bits.

§17Static analysis & style gates

The pre-commit config is the quality floor enforced before code merges. It is multi-language and layered:

LayerHooks
global hygienetrailing-whitespace, end-of-file, merge-conflict, case-conflict, detect-private-key, mixed-line-ending, codespell, check-github-workflows

Production-honest reads

  • The exclude list is pragmatic but lossy. cdisort.c/.h (vendored, not yours to reformat), disort.hpp, pydisort.cpp, and python/__init__.py are globally excluded. Excluding vendored C is correct; excluding your own disort.hpp/pydisort.cpp (large, macro-heavy, lint-noisy) is a maintenance shortcut that also turns off real checks on those files. Worth periodically re-including with targeted suppressions instead of a blanket skip.
  • cppcheck --check-config only. That flag checks cppcheck’s own include configuration — it is not a full static analysis pass. The hook name promises “find warnings/errors” but the args under-deliver. A real pass (--enable=warning,performance,portability) would catch more.
  • The cpplint filter list disables ~10 categories (include-order, fn-size, braces/parens whitespace, …). Each suppression is a deliberate tradeoff against Google-style strictness; the list should be revisited when the codebase stabilizes.
Improve — the analysis ladder

Add clang-tidy (bugprone-, performance-, modernize- checks) for real semantic analysis; run ASAN/UBSAN builds of the C++ tests in a dedicated CI job (native extensions are exactly where use-after-free and UB hide); and add compute-sanitizer once the GPU path is live. Static gates catch style; sanitizers catch the bugs that crash users.

§18Docs & typing life cycle

Docs are built on Read the Docs with Sphinx, and the package ships type information — both are life-cycle artifacts that drift if not gated.

build:
  os: ubuntu-22.04
  tools:
    python: "3.9"                 # ← note: package requires-python >=3.10

  configuration: docs/source/conf.py

  install:

A latent docs/runtime mismatch

RTD builds docs on Python 3.9, but pyproject.toml sets requires-python = ">=3.10". This works only because the docs build evidently does not import the compiled extension (which couldn’t install on 3.9 anyway, and certainly couldn’t find a 3.9 torch + native lib). It is a smell: the docs describe a package they cannot actually import. Bump RTD to 3.10+ and, ideally, build the extension in the docs env so autodoc reflects the real signatures.

Typing: PEP 561

pydisort ships a py.typed marker and a pydisort.pyi stub via package-data, so downstream users get autocomplete and mypy coverage over a C-extension that has no introspectable signatures. The risk is stub drift: the .pyi is hand-maintained and nothing checks it against the actual pybind-exported signatures.

Improve

  • Generate stubs with pybind11-stubgen in CI and diff against the committed .pyi — fail the build on drift.
  • Pin docs/requirements.txt for reproducible doc builds (RTD’s own advice).
  • Run a docs build in CI (not just on RTD) so doc breakage is a PR signal, not a post-merge surprise.

§19Debugging the native boundary

When a pure-Python bug appears, you read a traceback. When a native extension breaks, the traceback ends at import pydisort and the real story is in the loader, the symbol table, or a signal. This is the toolkit, mapped to the failure it diagnoses.

Decoding the three import-time crashes

(a) ImportError: undefined symbol: _ZN...__cxx11... → ABI mismatch (§04) or torch version drift (§12); confirm with nm + c++filt. (b) ImportError: libdisort_release.so: cannot open shared object file → RPATH/package-data problem (§06/§11); confirm with patchelf --print-rpath and unzip -l the wheel. (c) Segfault immediately on import torch; import pydisort → almost always two libtorch copies (the vendoring mistake, §11) or an ABI-incompatible torch; confirm with LD_DEBUG=libs.

§20Supply chain & developer environments

The brief asks for “cloud-hosted developer environments” and “multi-cloud large-scale systems.” Two life-cycle concerns close out the operational picture: can you trust what you shipped, and can a new contributor build it.

Supply chain

  • Pinning posture: torch is exact-pinned (strong); fmt/pybind11 are tag-pinned (medium — tags are mutable, §03); the .cache tarballs are unverified by hash. Adding SHA pins + hash verification + an SBOM (e.g. via cyclonedx) would make the dependency set tamper-evident.
  • Publish credentials: the long-lived PYPI_API_TOKEN should become OIDC Trusted Publishing (§14) — the biggest single supply-chain improvement available.
  • Provenance: attach build attestations so a consumer can prove a wheel came from this repo’s CD, not a typosquat.

Reproducible developer environments

The reproducibility primitive already exists: the exact two-stage sequence (cmake build → pip install --no-build-isolation --no-deps .) plus the exact torch pin means a contributor’s local build matches CI’s. To make that a cloud-hosted dev environment in the brief’s sense:

  • Ship a devcontainer / Dockerfile based on the same manylinux_2_28 image CD uses, so “works in my Codespace” equals “works in the wheel.”
  • Capture the “scripting glue” (the bash in the workflows, the WORKSPACE env handshake, the cmake-prefix-path computation) into committed scripts so local and CI builds share one code path — today some of that logic lives only inside YAML run: blocks.
  • For GPU work, a cloud GPU runner pool (the “multi-cloud” angle) is the prerequisite to ever testing §08.
Improve — one build path

The strongest reproducibility move is to have exactly one build script that CI, CD, and humans all call. Right now Stage-1 CMake flags differ subtly between ci.yml (no explicit prefix path) and cd.yml (computes TORCH_CMAKE_PREFIX_PATH). Converging them removes a class of “passes in CI, fails in CD” surprises.

§22The decision trees

Vendor vs. depend (the packaging fork)

Is the library uniquely yours and small, with no process-global singletons? ├─ YES → VENDOR into the wheel (package-data + $ORIGIN/lib) e.g. libdisort └─ NO ↓ Does the user already have it, is it huge, does it hold global state (dispatch/allocator)? ├─ YES → DEPEND: exact-pin, exclude from repair, find via RPATH e.g. libtorch └─ NO ↓ Is it a system primitive (libm, libstdc++, libgomp)? └─ YES → NEVER ship; rely on manylinux policy / the OS

Which C++ ABI / which torch?

Linux build? ├─ YES → derive _GLIBCXX_USE_CXX11_ABI from torch._C._GLIBCXX_USE_CXX11_ABI (never hard-code) └─ NO (macOS) → libc++, no GLIBCXX ABI flag — skip the detection Always: build against the SAME torch version you will run against (exact pin).

CUDA architecture list

Dev / local iteration? → native (one arch, fastest nvcc) Internal cluster, known GPUs? → just those archs (smallest fatbin that covers the fleet) Public release wheel? → broad SASS list + PTX for the newest arch (forward-compat JIT) Toolkit older than target GPU? → must include a virtual/PTX tier or hit "no kernel image"

undefined symbol? → nm + c++filt → __cxx11 ? Glossary

TermMeaning in this stack
ABIbinary contract between separately-compiled objects (name mangling, struct layout, calling convention). Mismatch = link failure or silent corruption.

TensorIteratorATen’s broadcasting element-wise execution engine; pydisort frames each (wave,col) DISORT solve as one element.
RPATH / RUNPATHdynamic library search paths baked into an object. RUNPATH (new default) is non-transitive.
$ORIGIN / @loader_path“directory of the loading object,” resolved at runtime — what makes a relocatable wheel possible.
manylinux_2_28build standard: link against glibc 2.28 so the wheel runs on any newer glibc.
cibuildwheeltool that builds + repairs wheels across the Python/OS/arch matrix in CI.
auditwheel / delocateLinux/macOS wheel-repair tools. pydisort uses them to exclude torch, not bundle it.
fatbin / SASS / PTXCUDA fat binary holding per-arch machine code (SASS) and/or virtual ISA (PTX, JIT’d by the driver).

OIDC Trusted Publishingsecretless PyPI upload via short-lived GitHub-issued tokens. The recommended replacement for API tokens.
