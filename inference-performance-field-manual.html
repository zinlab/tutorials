FRAME Chapter I

The frame · serving as a product surface.

The shift Inference is not deployment. It's an SLO surface — every percentile is a product number.

The rule Optimize what users experience. p99 latency, not p50. Cold-start, not warm-only. Cost per million requests, not cost per pod-hour.

Training produces a model. Inference produces revenue. The two have entirely different operational shapes — training is throughput-bound, batch-friendly, fault-tolerant via checkpointing; inference is latency-bound, request-driven, every percentile is a user-facing number. This issue is about the discipline of treating an inference container as a product surface with its own SLOs, benchmarks, optimization stages, and budgets. The boundary the manual walks is the four-stage path from "pod object created" to "useful response served": image pull, container start, model load, warm. Each stage has its own time budget, its own failure modes, its own optimization techniques.

Why the discipline shifts from training to inference

Issue 01-05 covered training infrastructure — Kubernetes for ML training, NCCL debugging, the AI/ML landscape, container engineering, container performance. The frame was throughput-first: maximize tokens-per-second across distributed GPUs, tolerate stragglers via gang scheduling, restart on failure from checkpoint. Inference inverts almost every assumption:

  • Latency, not throughput. A 30-second p99 ruins the user experience even if p50 is fine.
  • Request-driven, not batch-driven. Traffic is bursty, often diurnal; the optimization target is "respond well under variable load," not "process the dataset."
  • Cold-start matters. Training pods run for hours; an inference pod that takes 3 minutes to come up makes scale-out useless during demand spikes.
  • Cost-per-request, not cost-per-job. The financial axis is dollars per million inference requests. GPU utilization translates directly to unit economics.

The four-stage cold-start budget

Every inference container ships through the same pipeline:

  • Image pull. The image is fetched from the registry to the node. Sequential layer download is the default; lazy pulling (Chapter VII) and pre-pulling (Chapter VIII) change this dramatically.
  • Container start. Runtime extracts the layers, sets up the rootfs, runs the entrypoint. Dominated by image size on cold nodes.
  • Model load. The model artifact (weights, tokenizer, config) gets loaded into memory or GPU memory. Often hits a remote object store; mmap, streaming, FUSE strategies in Chapter IX.
  • Warmup. First requests are slow because caches are cold, JIT hasn't compiled, CUDA kernels haven't been autotuned. The "readiness" signal must wait for warmup completion, not just process startup.

The naive cold-start is the sum of these — pull + extract + start + load + warm. A serious inference platform brings each one down independently, plus parallelizes where possible.

The cost stack

Inference cost has four lines that platform teams should monitor separately. Provisioned base capacity: pods always running, paying even at zero load. Scale-out marginal cost: cost of each extra replica spun up during a spike, dominated by cold-start length × probability of being requested during start. Per-request compute: the actual GPU/CPU seconds per inference. Tail-latency penalty: the opportunity cost of slow tail responses (dropped users, retries amplifying load).

Most teams optimize per-request compute first because it's the most visible. The real leverage is usually in cold-start (which determines how aggressively you can scale to zero) and tail-latency control (which determines how much capacity you waste on safety margin). A 60% reduction in cold-start might allow scale-to-zero for a workload that previously had to keep 3 hot replicas, which is a 100% cost reduction on the base capacity line.

The optimization order that actually works

Measure first. The biggest mistake in inference optimization is reaching for vLLM or SOCI before knowing where the seconds actually live. Benchmark the four stages independently. The stage that dominates is the one to optimize first. For most teams the first surprise: model load (Chapter IX) takes longer than image pull. The second surprise: warmup (Chapter X) takes longer than model load. The third surprise: a Python import-time side effect adds 12 seconds to startup that nobody noticed.

Stage-by-stage

Image pull (often the largest share). The container runtime pulls layers sequentially from the registry. On a cold node with no cached layers, a 5GB ML image over a 1 Gbps network is 40+ seconds of pure download, plus decompression. Research shows that time to take for pull operation accounts for 76% of container startup time — that statistic from FAST '16 is even more dramatic for ML images, which are 5-50× larger than typical web service images.

Image extraction. Once layers are pulled, they must be unpacked. The default overlayfs snapshotter decompresses every byte before the container starts. Large layers with millions of small files (Python site-packages classic) take longer than fewer-but-bigger layers of similar total size.

Container start & Python import. The container starts; the entrypoint runs; Python (most inference containers in 2026) begins importing. import torch alone takes 2-5 seconds. Add transformers, numpy, pandas, custom modules — easily 8-15 seconds before user code even runs. This stage is invisible to most monitoring; it lives between "container started" and "process listening on port."

Model load. The model artifact (weights, tokenizer, config, scalers, feature transformers) must be loaded. Three sub-paths matter:

  • Where it lives: baked into the image, in a sidecar PVC, in S3/GCS, in a model registry.
  • How it's read: cold copy (download then load), mmap, streaming, FUSE-mounted.
  • Where it ends up: CPU RAM, GPU VRAM, both (for KV cache + weights).

A 10GB model from S3 to GPU on a fresh pod can easily be 60 seconds without optimization. With streaming and mmap, the same artifact can be ready in <10 seconds.

Warmup. CUDA lazy-compiles kernels on first use. Frameworks like PyTorch trigger autotune on the first batch. Caches are cold. The first few requests are 5-30× slower than warm requests. If readiness probes pass too early, real user traffic hits a half-warm pod and experiences pathological tail latency.

Where the surprises are

Three patterns repeat across teams in 2026:

  • Model load is bigger than people expect. A 5GB safetensors file downloaded sequentially from S3 to a node, then memcopied to GPU, can take longer than the image pull. For traditional ML (XGBoost, scikit-learn, custom Python), the load is often tiny (MBs) but Python deserialization, pickle scanning, or pandas DataFrame construction can take seconds.
  • Python import time matters more than expected. Especially with PyTorch + transformers + CUDA bindings; the kernel/CUDA registration handshake at import dominates for some workloads.
  • Warmup is invisible. The pod passes the readiness probe (HTTP 200 on /health) but the first user requests still hit pathological latency. The fix is making readiness actually mean ready (Chapter X), with a warmup endpoint that exercises the model.
The "image is small but startup is slow" paradox

A 300 MB image that still takes 90 seconds to be ready is normal in inference. Most of those seconds are NOT in the image pull — they're in Python import time, model loading, and warmup. Shrinking the image further has near-zero payoff; the real optimizations are post-pull. Cross-reference Issue 07 Chapter VIII on image-size measurement; the budget for an inference image is different from a stateless web service.

BENCH Chapter III

Latency & throughput · the vocabulary.

The five numbers p50, p95, p99, p999, max. The average lies; the tail tells the truth.

The decomposition Time-to-first-token vs time-to-completion. Server-side latency vs end-to-end. Cold vs warm. Per-request vs per-batch.

Before you can optimize, you have to agree on what you're measuring. Inference serving has a richer measurement vocabulary than typical request/response services — particularly for streaming responses, batched compute, and GPU-bound workloads. This chapter pins the terms.

The percentile discipline

Inference SLOs live in percentiles. Quote in this order — p50, p95, p99, p999, max — and never report the mean. The mean of a long-tail distribution hides exactly what users experience.

  • p50 (median). Half of requests faster, half slower. A reasonable feel-good number; usually not the SLO.
  • p95. The slowest 5% are above this. The common "user-facing" SLO target for interactive workloads.
  • p99. The slowest 1%. For high-volume APIs, this is the percentile that determines retries, timeouts, and queue accumulation.
  • p999 (three-nines). The slowest 0.1%. Where cold-start, GC pauses, and pathological GPU stalls show up. Often where the real problems hide.
  • Max. The absolute worst. Useful as a smoke signal; not a sane SLO target alone.

Latency components for inference

# Time from request received to response complete
total_latency = (
    queue_time          # waiting in the pre-model queue (batching, backpressure)
    + parse_time        # JSON deserialization, validation
    + preprocess_time   # tokenization, feature engineering, image decode
    + inference_time    # the model forward pass
    + postprocess_time  # detokenize, format response, apply business logic
    + serialize_time    # JSON encode the response
)
# For streaming (LLMs):
time_to_first_token = (queue_time + parse + preprocess + first_inference_step)
time_to_completion  = TTFT + (output_tokens * inter_token_latency)

For non-streaming workloads (classification, regression, embeddings) the relevant number is end-to-end latency. For streaming LLMs, separately track time-to-first-token (TTFT) — what determines whether the UI feels alive — and inter-token latency (ITL) — what determines whether the stream feels smooth. Don't conflate them; they have different optimization paths.

Throughput vocabulary

  • Requests-per-second (RPS). The blunt instrument. Fine for stateless, fixed-cost-per-request services.
  • Queries-per-second (QPS). Synonymous with RPS in this context.
  • Tokens-per-second (TPS). For LLMs, the meaningful capacity number. Decoder throughput is the bottleneck; quoting RPS without TPS hides huge variance from different input/output lengths.
  • Goodput. Requests-per-second that met the SLO. The number that matters: capacity that breaks SLO is not capacity.

Cold vs warm vs steady-state

Benchmarking discipline requires reporting which regime you measured:

  • Cold-start latency. Time from a fresh pod with no cached layers to first successful request. The most painful number; usually 10-100× worse than steady state.
  • Warm-start latency. Time from a fresh pod on a node where the image is already cached, all the way to ready. Measures Python import + model load + warmup. The relevant number after the first replica.
  • First-request latency. A warm pod's first inference. Measures CUDA kernel autotune, JIT, autocache warming.
  • Steady-state latency. 100+ requests in, everything cached and warmed. The number marketing materials usually quote.

The four numbers tell separate stories. Cold-start determines how aggressively you can scale to zero. Warm-start determines how aggressively you can autoscale during a spike. First-request determines whether to add explicit warmup (Chapter X). Steady-state determines per-request economics.

Saturation and tail amplification

Two failure modes that show up only above ~70% capacity:

  • Queueing latency explosion. Per queueing theory, latency rises asymptotically as utilization approaches 100%. At 80% utilization, p99 is often 2× p50; at 95% utilization, p99 is 10×+ p50. The fix is keeping utilization below ~70%, not optimizing the model further.
  • Tail amplification by retries. Client timeouts trigger retries, which add to load, which slows responses, which trigger more retries. The result is a metastable failure where the system is "fine" at 60% load but collapses at 75%. The fix is retry budgets, circuit breakers, and load shedding — never naive client-side retry.
The minimum useful benchmark

For any inference workload, report: model + hardware + input shape + output shape, p50/p95/p99 TTFT (or end-to-end for non-streaming), p50/p95/p99 ITL (for streaming), peak goodput (RPS or TPS where p99 still meets SLO), cold-start time, warm-start time. Six tuples; nothing less is publishable. In 2026 the deployment landscape is fragmented; frameworks like vLLM, TensorRT-LLM running on Triton and Hugging Face's Text Generation Inference (TGI) each promise different benefits — only standardized benchmarking makes them comparable.

BENCH Chapter IV

Benchmarking methodology · k6, vegeta, locust.

The tools k6 for general HTTP, vegeta for simple rate sweeps, locust for behavior-modeled load, GenAI-Perf for LLM-specific token-level metrics.

The rule Open-loop, not closed-loop. Measure where the bottleneck moves, not just where it is now.

The most common benchmark methodology error: closed-loop testing. The tool sends a request, waits for the response, sends the next. This measures end-to-end round-trip per worker, but it never observes queueing because there's never queue — the next request only arrives after the previous completes. To find the real shape, you need open-loop: requests fire at a target rate regardless of response time, and queueing emerges naturally.

The minimum k6 inference benchmark

import http from 'k6/http';
import { check, sleep } from 'k6';
import { Trend } from 'k6/metrics';

const ttft = new Trend('ttft_ms');
const total = new Trend('total_ms');

export const options = {
  scenarios: {
    // Open-loop arrival: fixed rate independent of response time.
    // As load grows, queue grows; this is what you actually want to measure.
    sweep: {
      executor: 'ramping-arrival-rate',
      startRate: 10,
      timeUnit: '1s',
      preAllocatedVUs: 200,
      maxVUs: 800,
      stages: [
        { target:  10, duration: '1m' },   // warm pool
        { target:  50, duration: '2m' },
        { target: 100, duration: '2m' },
        { target: 200, duration: '2m' },
        { target: 400, duration: '2m' },   // expect SLO break around here
        { target:   0, duration: '30s' },
      ],
    },
  },
  thresholds: {
    // Hard SLO — k6 returns non-zero exit if violated
    'http_req_duration{status:200}': ['p(95)<500', 'p(99)<1500'],
    'ttft_ms':                       ['p(95)<200', 'p(99)<800'],
    'http_req_failed':               ['rate<0.001'],
  },
};

export default function () {
  const payload = JSON.stringify({
    text: "sample input from a realistic distribution, not a fixed string",
    options: { max_tokens: 128 }
  });
  const res = http.post(__ENV.URL, payload, {
    headers: { 'Content-Type': 'application/json' },
    tags:    { endpoint: 'inference' },
  });
  check(res, { '2xx': (r) => r.status < 300 });
  total.add(res.timings.duration);
  // no sleep() — open-loop means arrivals are time-driven, not response-driven
}

The four phases of a useful benchmark

  1. Cold-start. Delete all pods. Start the service from scratch. Measure time-to-first-request-success. This is what users experience during scale-from-zero or new revisions.
  2. Step-load. Step the rate up in stages and let each stage soak for ≥2 minutes. The slope of the latency-vs-throughput curve tells you where the saturation knee is.
  3. Steady-state. Run at 70% of the saturation knee for ≥10 minutes. Verify p99 stays flat over time (memory leaks, KV cache pressure, GC pauses surface here).
  4. Burst. Sudden step from 10 RPS to 200 RPS. Measure how long the SLO stays broken before autoscaler catches up. This is the most important number for diurnal workloads.

The input-distribution gotcha

Benchmarking with the same payload repeatedly produces optimistic results. Real traffic has variable input shapes — short prompts and long prompts, easy and hard requests, the occasional pathological outlier. The bench should sample from a realistic input distribution:

  • For LLMs: sample input lengths from your prod distribution (typically log-normal). A bench using only 100-token inputs will under-estimate p99 by 3-10×.
  • For embeddings/classification: include the edge cases your prod traffic has — very short, very long, empty, malformed, multilingual.
  • For vision: vary image sizes and aspect ratios. Resize-and-pad cost varies non-trivially.
  • For recommenders: sample real user IDs with their actual feature-vector shapes (sparse vs dense), not synthetic uniform IDs.

Coordinated omission, briefly

The closed-loop trap has a name: coordinated omission. If a tool fires the next request only after the previous response, then a 10-second response delays all subsequent requests; the tool reports ten 10-second responses but the queue depth that would have built up under open-loop is invisible. Always use open-loop generators (k6's ramping-arrival-rate, vegeta's constant rate, wrk2's -R flag). For visual proof, the same workload under closed-loop will show p99 of ~10s; under open-loop, the same workload may show p99 of 30+s as queue depth amplifies the tail.

Bench in the staging cluster, not in dev

The image registry distance, the network path, the node type, the storage class — all matter for inference. A bench on a developer laptop tells you about the model; a bench on the actual staging cluster tells you about production. Cross-reference Issue 06 Chapter X on profiling under load. Use the same observability stack in the bench that you use in prod, so the latency numbers you see in Grafana during the bench are the same shape as the ones you'll see during a real incident.

BENCH Chapter V

The latency–throughput frontier.

The shape Latency rises gently with load up to the knee, then explodes. Cost-per-request falls until the knee, then rises.

The decision Run at ~70% of the knee. Closer to 100% wins cost-per-request but pays in tail amplification.

The fundamental shape of an inference system: as throughput grows, latency stays roughly flat, then sharply rises near the saturation point, then explodes. Cost-per-request falls inversely until the same knee, then rises (because tail amplification, retries, and dropped traffic eat the savings). Knowing your knee — and operating at a safe distance from it — is the single most important capacity-planning skill.

Building the frontier curve

import json, pandas as pd, matplotlib.pyplot as plt

# Run k6 at constant rates: 10, 50, 100, 200, 400, 600, 800 rps
# Capture JSON output per run with: k6 run --out json=run_N.json bench.js

rates = [10, 50, 100, 200, 400, 600, 800]
rows = []
for rps in rates:
    with open(f"run_{rps}.json") as f:
        durations = [json.loads(l)["data"]["value"]
                     for l in f if json.loads(l)["metric"] == "http_req_duration"]
    s = pd.Series(durations)
    rows.append({
        "rps":  rps,
        "p50":  s.quantile(0.50),
        "p95":  s.quantile(0.95),
        "p99":  s.quantile(0.99),
        "p999": s.quantile(0.999),
    })

df = pd.DataFrame(rows)
print(df)
# Plot: x = rps, y = latency in log scale, multiple lines for p50/p95/p99/p999.
# The knee is where p99 starts rising sharply (typically inflects at 70-85% of max capacity).

The frontier shape, narrated

  • Below 30% utilization: latency is dominated by the model itself; the curve is essentially flat. Throughput-vs-cost is wasteful but predictable.
  • 30-70% utilization: latency rises slightly; cost-per-request improves substantially. The sweet spot for production operation.
  • 70-90% utilization: latency rises sharply; p99 may be 3-5× p50. Cost-per-request continues to improve but you're trading cost against user pain.
  • Above 90% utilization: queueing explodes; p99 is 10-30× p50; small load variations cause SLO violations.
  • Above the knee: requests time out; retries amplify; metastable failure follows.

The knee is workload-specific

Three factors shift the knee:

  • Variance in input cost. Workloads with high variance (LLMs with mixed prompt lengths, classifiers with variable preprocessing) saturate earlier because long requests block short ones. The knee can be at 50-60% for high-variance workloads vs 80-90% for uniform workloads.
  • Batching scheme. Continuous batching (Chapter XII) pushes the knee higher because it doesn't have to wait for a full batch to start. Static batching pushes it lower.
  • Parallelism within the replica. Pods with many CPU workers (Gunicorn N workers) blur the knee because individual workers saturate independently. Pods with a single GPU worker have a sharper knee.

Operating-point selection

Once you have the frontier curve, pick the operating point:

  • Target utilization = 65-70% for interactive workloads where SLO violations are visible to users.
  • Target utilization = 80-85% for batch/async workloads where queue depth is acceptable.
  • Set HPA target metric to half of saturation point: if 100 RPS is the knee, the HPA averageValue per pod should be 50 RPS. The autoscaler is reactive (Chapter XIV); the headroom absorbs the lag.

What the latency curve doesn't tell you

  • Cold-start tail. The latency curve assumes warm replicas. If scale-out cold-start is 90s, every spike will produce a wave of p999 violations regardless of where you set the operating point.
  • Memory pressure. Some workloads (vLLM KV cache, embedding caches) saturate memory before they saturate compute. The knee on the latency curve is hit, but the actual failure mode is OOM, not slow response.
  • Cost. The frontier is useful for capacity sizing; cost optimization comes from combining it with autoscaling (Chapter XIV) and scale-to-zero (Chapter XV) policies.
The frontier is a living document

Re-run the benchmark on every meaningful change: model version, runtime version (vLLM/Triton upgrade), hardware change, batch-size config change, image-size change. The shape can move dramatically. The knee on a vLLM v0.6 was different from v0.7's; H100 frontier is different from A100's; a quantized model has different shape from full-precision. Treat the frontier curve as a checked-in artifact in the service repo, not a slide deck.

IMAGE Chapter VI

Image optimization · layers, distroless, budgets.

The target ML inference image: <2 GB if at all possible, <1 GB if you can. Model artifact lives outside the image (Chapter IX).

The principle Layer order matters as much as image size. Stable layers go early; volatile layers go late.

Cross-reference Issue 07's full treatment of image building. This chapter is the inference-specific extract: which optimizations matter most for cold-start, where the typical ML image goes wrong, and what the production target looks like.

The ML inference image, dissected

A naive PyTorch inference image is often 8-12 GB. The breakdown:

# for one C extension
NVIDIA CUDA toolkit (cuda:12.4-devel base)            4.2 GB     # mostly unused at runtime
Python 3.11                                            55 MB
pip install torch (with CUDA)                         2.8 GB
pip install transformers, sentencepiece, ...          1.2 GB
pip install pandas, numpy, scikit-learn, scipy         320 MB
pip install fastapi, uvicorn, gunicorn, redis...      130 MB
COPY ./src /app/src                                    8 MB
COPY ./model.safetensors /app/model.safetensors       5.2 GB    # DON'T do this
                                                  ----------
total                                                14.5 GB

Two killing patterns:

  • CUDA toolkit in runtime image. The nvidia/cuda:12.4-devel base includes nvcc, headers, libraries needed for compilation but not for inference. Use nvidia/cuda:12.4-runtime or cuda:12.4-base in runtime; multi-stage from devel for any compile step.
  • Model baked into the image. The 5GB model in the final layer means every code change re-pulls 5GB across the network. Mount the model from a sidecar/PVC/object store; the image stays small and stable.

The target image, multi-stage

# ============ stage 1: build ============
FROM python:3.11-slim AS build
WORKDIR /build

# UV for fast resolves; cache mount for reuse
RUN --mount=type=cache,target=/root/.cache/pip \
    pip install uv

COPY pyproject.toml uv.lock ./
RUN --mount=type=cache,target=/root/.cache/uv \
    uv pip install --system --no-deps -r uv.lock

# ============ stage 2: runtime ============
FROM nvidia/cuda:12.4.0-runtime-ubuntu22.04 AS runtime

# Single layer: install python + tiny runtime deps
RUN apt-get update && \
    apt-get install -y --no-install-recommends \
      python3.11 python3.11-distutils libgomp1 ca-certificates && \
    rm -rf /var/lib/apt/lists/*

# Stable: copy site-packages from build stage
COPY --from=build /usr/lib/python3.11/site-packages /usr/lib/python3.11/site-packages

# Volatile: app code last, so code changes only invalidate the final layer
WORKDIR /app
COPY src/ src/
COPY entrypoint.sh ./

# Pin the digest of the base in CI; never use mutable tags
USER 65532:65532
EXPOSE 8000
ENTRYPOINT ["./entrypoint.sh"]

Layer ordering — the cache-hit lever

The order of COPY and RUN statements determines which layers can be reused. The principle: stable things early, volatile things late.

  • Base image, OS packages, system libs — change rarely. Top of the file.
  • Python dependencies (pyproject.toml + uv.lock) — change weekly. Middle.
  • Application code — changes per commit. Bottom.

A code-only change with this ordering means only the final layer (8 MB) is rebuilt and pushed. With reversed ordering, every code change invalidates every subsequent layer including the 2.8 GB torch install.

Distroless considerations for inference

Distroless (Issue 07 Chapter VI) is great for stateless services and even most ML inference. Two complications for GPU inference:

  • NVIDIA runtime needs the right libraries. gcr.io/distroless/python3-debian12 does not include libcuda.so; that's fine because the NVIDIA container toolkit mounts it from the host at runtime. But it must include libgomp1, sometimes libgl1 for image-handling models. Test thoroughly on the actual node type.
  • Debugging is harder. No shell means kubectl exec doesn't get you a bash prompt. Plan ephemeral debug containers (Issue 09's debugging chapter) and structured logging in advance.

The image-size budget for inference

A defensible budget for different inference types in 2026:

  • Classic ML (XGBoost, sklearn, custom Python): target 300-500 MB. Anything beyond 1 GB is unnecessary.
  • Deep-learning CPU inference: target 800 MB - 1.5 GB. The torch/onnx/triton runtime alone is the bulk.
  • Deep-learning GPU inference (no LLM): target 1.5-3 GB. CUDA runtime, torch with CUDA, model framework.
  • LLM inference (vLLM, TGI, Triton): target 4-8 GB. These have substantial deps; some images approach 10 GB. The pull cost is real; lazy pulling (Chapter VII) is the workaround.

Quick wins audit

  1. Multi-stage. If your Dockerfile is single-stage, you're shipping build tooling to prod. Always multi-stage.
  2. Slim or distroless base. Anything ending in :slim beats anything ending in :latest; distroless beats slim when feasible.
  3. --no-install-recommends on apt. Cuts 30-50% of typical apt install bloat.
  4. Combine RUN steps with && and rm -rf /var/lib/apt/lists/*. Each RUN is a layer; smaller layers compress less and pull slower.
  5. BuildKit cache mounts. --mount=type=cache,target=/root/.cache/pip — wheels are reused across builds.
  6. UV instead of pip. Resolution alone is 10-50× faster; install is 3-10× faster. Build-time, not runtime, but builds happen often.
  7. Strip __pycache__ and .pyc files if you're aggressively trimming.
  8. Audit site-packages: bundled tests/, docs/, examples/ directories in dependency packages can be 100s of MB.
The 80/20 fix order

For most teams, three changes get you 80% of the savings: 1. Move model out of image (Chapter IX). 2. Switch CUDA base from devel to runtime. 3. Multi-stage build with proper layer ordering. These typically move an image from 12 GB to 2 GB without touching dependency choice. Only then is it worth deeper optimization (distroless, lazy pulling, etc).

IMAGE Chapter VII

Lazy pulling · SOCI, eStargz, snapshotters.

The shift Don't pull the whole image before starting; pull bytes on demand. Container starts while download is still in progress.

The choices eStargz (image conversion), SOCI (separate index, no image change), Nydus/EROFS (high-performance), cloud-managed (GCP/Azure streaming).

For large inference images, lazy pulling is the most consequential cold-start optimization available in 2026. The premise: most parts of a container image are unnecessary for most container runs, so instead of eagerly fetching the entire image upfront, pieces of the image's layers can be retrieved by the container runtime as needed. The result for ML workloads is dramatic — Grab reported a 60% improvement after lazy-loading tuning; BuildBuddy reported a 10× improvement, from 30s to 4s pull time on cold starts.

The two production-ready formats

Lazy pulling · choose the format

How SOCI works, briefly

In containerd the component that manages the container's filesystem is called a snapshotter. The default snapshotter, overlayfs, pulls and decompresses the entire container image before a container can be started. With lazy loading snapshotters (such as stargz or SOCI snapshotter), the container starts without downloading the entire container image and instead lazily loads files from an OCI compatible registry, like Amazon Elastic Container Registry.

concurrent_download_chunk_size = "16mb"      # balance parallelism vs HTTP overhead
max_concurrent_unpacks = -1

Per Grab's writeup, we achieved a 60% improvement by optimizing SOCI's parallel pull mode settings, demonstrating the value of proper configuration tuning. The defaults are too conservative for ML; bump max_concurrent_downloads_per_image to 20 on high-bandwidth networks.

The trade-off that surprises teams

Lazy loading doesn't eliminate download time — it redistributes it. The clean number from a 2026 deep-dive on lazy-pulling readiness: Pull is 65× faster with FUSE (0.088s vs 5.782s). Only the TOC and image manifest are downloaded, not the actual layer data. Readiness is 20× slower with FUSE (0.271s vs 0.013s). When nginx tries to serve /healthz, the FUSE filesystem must fetch the nginx binary, shared libraries, and config files from the registry over HTTP.

Net: container starts faster, but first requests are slower until the working set is fetched. For ML inference, where startup includes Python import + model load anyway, the trade-off usually wins big — but you must measure readiness, not just pull. The right benchmark phrase is "time from Pod create to /healthz returning 200," not "time to pull the image."

Required cluster-level setup

  • Snapshotter installed on every node. SOCI runs as a containerd plugin; it must be on every node that will run lazy-loaded images. Bottlerocket OS includes SOCI support natively (the reason Grab chose it for production); on Ubuntu / AL2 you install via DaemonSet.
  • Containerd configured. The config.toml must reference the SOCI snapshotter for the plugin path.
  • Images indexed. Run soci create && soci push in CI for every image. The index is a separate OCI artifact in the same registry. Image digests remain unchanged (this is the key advantage over eStargz).
  • Registry support. ECR has full support; ghcr.io, GAR, Quay support OCI Referrers and work. Some self-hosted registries lag.

The migration order

  1. Start with the biggest image (vLLM, TGI, large PyTorch). The gain scales with image size.
  2. Roll out the snapshotter to a single node pool first; bench cold-start with and without.
  3. Wire SOCI index creation into CI so every new image build creates a new index.
  4. Monitor for stale FUSE mounts, registry-side rate limits, and credential expiry under load (BuildBuddy hit all three).
The "registry as runtime dependency" risk

With lazy pulling, your container registry is now in the critical path for running pods, not just for starting them. A registry outage no longer just blocks new deploys — it can degrade running pods that haven't fetched their full working set yet. Plan for: 1) registry HA and read replicas, 2) background prefetch policies, 3) alerts on registry latency spikes, 4) graceful degradation if a span fetch fails.

IMAGE Chapter VIII

Pre-pull strategies · DaemonSets & warmers.

The orthogonal axis Even with lazy pulling, getting the image already cached on the node beats any pull strategy. Pre-pull when you know which images you'll need.

The two patterns DaemonSet pre-pullers for "every node needs this," node-pool affinity for "specific nodes need this."

Lazy pulling makes the first run fast. Pre-pulling makes it instantaneous. The strategies are complementary, not competitive. For an inference workload where you can predict which images need to be available, getting the image pulled to the node before the first replica needs it converts cold-start to warm-start.

The DaemonSet pre-puller

apiVersion: apps/v1
kind: DaemonSet
metadata:
  name: inference-image-prepuller
  namespace: platform
spec:
  selector:
    matchLabels: { app: inference-image-prepuller }
  template:
    metadata:
      labels: { app: inference-image-prepuller }
    spec:
      nodeSelector:
        # Only run on inference node pools, not on system or training nodes
        workload-class: inference
      tolerations:
        - key: nvidia.com/gpu
          operator: Exists
      # Each init container pulls one image we want pre-cached
      initContainers:
        - name: pull-vllm
          image: ghcr.io/myorg/vllm-server:v1.42.0@sha256:abc...
          command: ["sh", "-c", "echo pre-pulled"]
        - name: pull-cmmm-serving
          image: ghcr.io/myorg/cmmm-serving:v2.18.0@sha256:def...
          command: ["sh", "-c", "echo pre-pulled"]
        - name: pull-classifier
          image: ghcr.io/myorg/classifier:v3.7.0@sha256:ghi...
          command: ["sh", "-c", "echo pre-pulled"]
      # The actual pod just sleeps — it exists to keep the images "owned" by
      # a running pod, preventing kubelet from garbage-collecting them.
      containers:
        - name: pause
          image: gcr.io/google_containers/pause:3.9
          resources: { requests: { cpu: "5m", memory: "10Mi" } }

The pattern: init containers pull the images we want cached; the pod transitions to Running with a pause sidecar that keeps it alive. Kubelet's image garbage collection respects "in use by a running pod" and won't evict pre-pulled images even under disk pressure (until the DaemonSet itself is evicted, which is rare).

Pull-through caches and registries

An orthogonal approach: regional pull-through caches. Instead of pulling from ghcr.io directly, every node points at registry-cache.region.svc.cluster.local which fetches and caches on the first request, serves from cache thereafter.

  • Registry mirrors: containerd config can specify mirror registries via endpoint list. Pull tries the mirror first, falls back to upstream.
  • Distribution Spreader / Harbor / Zot / Docker Distribution: lightweight read-through proxies. Bandwidth into the cluster is paid once; subsequent pulls are intra-cluster traffic.
  • Native cloud caches: ECR has VPC endpoints; GAR has regional mirrors. Use them.

The image-ownership pattern with Kyverno or admission policies

For larger orgs, codify a policy that only pre-pulled images can run on inference node pools. This forces discipline: a new image must be added to the pre-puller DaemonSet (or registered via a platform-level mechanism) before any pod on the inference pool can use it.

Node pre-warming via Karpenter / cluster-autoscaler

For workloads with predictable demand patterns, pre-warm at the node level: schedule the autoscaler to add capacity 10-15 minutes before the expected spike. Cross-reference Issue 09's autoscaling section. Karpenter supports custom NodeClaim creation; you can have a CronJob create overprovisioner pods that hold spare capacity, then evict them once the actual demand materializes.

The tradeoffs

  • Pre-pull uses disk on every node. A 4 GB image times 100 nodes = 400 GB. Cost is real; track it.
  • Pre-pull is reactive to image updates. A new image version is pulled to nodes only after the DaemonSet update. Plan ~5-10 min lag between image push and node-readiness.
  • Pre-pull doesn't help cold-node-add. A brand-new node added by the autoscaler still has to pull. Combine with lazy pulling (Chapter VII) to cover this case.
The combined pattern that wins

Production-grade inference cold-start in 2026 stacks three optimizations: 1. Lazy pulling (SOCI) so cold-node-add is fast. 2. DaemonSet pre-pull so existing nodes always have current images. 3. Pull-through cache so even uncached fetches are intra-cluster. With all three, an image-pull time of 30+ seconds on cold nodes drops to <5 seconds. Compounded with model-load optimization (Chapter IX), the whole cold-start envelope can fall below 15 seconds.

MODEL Chapter IX

Model loading · mmap, streaming, FUSE.

The rule Never bake the model into the image. Mount it, stream it, or fetch it on init.

The four strategies Init container download, sidecar with shared volume, S3-FUSE mount, mmap from a shared filesystem (read-only).

For most inference workloads, model loading is a larger share of cold-start than the image pull. A 5 GB safetensors model downloaded from S3 to a node, deserialized, and copied to GPU VRAM can take 60+ seconds with naive code. With the right strategy, the same artifact reaches GPU in 5-15 seconds. This chapter walks the four production strategies, when each fits, and the gotchas.

Anti-pattern: model in image

FROM python:3.11-slim
COPY ./model_v2.safetensors /app/model.safetensors    # 5 GB
COPY ./src /app/src
CMD ["python", "/app/src/serve.py"]

Three problems compound. First, every model version forces a new image build + push + pull cycle — even though code didn't change. Second, image-pull time scales with model size. Third, multiple replicas pull the same 5GB across the network independently (no shared cache). For anything beyond toy services, separate model from image.

Strategy 1 — init container download

An init container fetches the model from object storage into a shared emptyDir volume; the main container mounts the same volume read-only.

apiVersion: v1
kind: Pod
spec:
  volumes:
    - name: model-cache
      emptyDir: { sizeLimit: "20Gi", medium: "Memory" }    # tmpfs — fast reads
  initContainers:
    - name: model-fetcher
      image: ghcr.io/myorg/s5cmd:v2.2.2
      command:
        - sh
        - -c
        - |
          s5cmd --concurrency 32 cp \
            "s3://my-models/cmmm/v2.18.0/*" \
            "/cache/"
      env:
        - { name: AWS_REGION, value: us-east-1 }
      volumeMounts:
        - { name: model-cache, mountPath: /cache }
      resources:
        requests: { cpu: "2", memory: "4Gi" }
  containers:
    - name: inference
      image: ghcr.io/myorg/cmmm-serving:v2.18.0
      env:
        - { name: MODEL_PATH, value: /cache/ }     # container reads from emptyDir
      volumeMounts:
        - { name: model-cache, mountPath: /cache, readOnly: true }

Why this beats baking in image:

  • Image stays stable. Image version reflects code; model version is a parameter. The image works for many models.
  • Parallel download. s5cmd --concurrency 32 uses multiple connections; a 5 GB model can pull in 10-15 seconds vs sequential 45+ seconds.
  • tmpfs cache. medium: Memory makes the emptyDir a tmpfs — RAM-backed, ~10 GB/s read speed. Worth the RAM cost for fast model loading.

Strategy 3 — S3-FUSE mount (mountpoint-s3, goofys)

It's worse when: 1) Every pod needs the full model (FUSE doesn't beat parallel S3 GETs). 2) The model is read repeatedly during inference (page-cache misses per request). 3) Network to S3 is unreliable (FUSE retries on every miss).


      
Strategy 4 — mmap from shared filesystem

      
For traditional-ML and small-DL models, the gold standard: a shared read-only filesystem (NFS, EFS, FSx) that contains all promoted model versions. The inference code uses mmap (or safetensors' load_file with device='cpu' then .to('cuda')) and pages are demand-loaded.

      
from safetensors.torch import load_file
import torch

# Safetensors uses mmap under the hood; tensor metadata read instantly,
# actual weights are paged in lazily as the GPU copy reads them.
def load_model(path: str, device: str = "cuda:0"):
    # Step 1: instantaneous — only reads the safetensors header (offsets table)
    state_dict = load_file(path, device="cpu")
    # Step 2: explicit copy to GPU — this is where actual bytes are touched
    for k, v in state_dict.items():
        state_dict[k] = v.to(device, non_blocking=True)
    torch.cuda.synchronize()
    return state_dict

MODEL Chapter X

Init & warmup · probes that don't lie.

The trap Process running ≠ ready to serve. Default probes return 200 the moment the HTTP server binds; real users hit a half-warm pod.

The fix Three probes: startup, liveness, readiness. Readiness must run a real inference path, not just a /health endpoint.

A pod that passes its readiness probe but isn't actually ready inflicts pathological tail latency on real users. The fix is making readiness mean readiness — including model loaded, GPU caches warmed, CUDA kernels JIT'd, downstream connections established.

The three probes, properly

containers:
  - name: inference
    image: ghcr.io/myorg/cmmm-serving:v2.18.0
    ports: [{ containerPort: 8000, name: http }]

    # Startup: gives the pod time to fully initialize.
    # Kubelet does not run liveness/readiness while startup is failing.
    # Failure here = pod is killed and restarted.
    startupProbe:
      httpGet: { path: /startup, port: 8000 }
      periodSeconds: 5
      failureThreshold: 60            # up to 5 min for cold-start

    # Liveness: pod is alive (process responsive, not deadlocked).
    # Failure = pod restarted. Use the CHEAPEST possible check.
    livenessProbe:
      httpGet: { path: /live, port: 8000 }
      periodSeconds: 10
      failureThreshold: 3
      timeoutSeconds: 2

    # Readiness: pod can accept traffic.
    # Failure = pod removed from Service endpoints. NO restart.
    # MUST run a real inference path.
    readinessProbe:
      httpGet: { path: /ready, port: 8000 }
      periodSeconds: 5
      failureThreshold: 3
      timeoutSeconds: 3

Three probes have three distinct semantics. Mixing them up is the source of most probe-related production bugs.

The /ready endpoint that actually validates readiness

from fastapi import APIRouter, HTTPException
import torch

router = APIRouter()
_warmed = False
_model = None

def warm_model(model):
    """Run inference once on a representative payload to JIT kernels and
    populate caches. Called once after model load completes."""
    global _warmed, _model
    _model = model
    with torch.no_grad():
        # Use real input shapes the model will see in prod
        dummy = build_representative_input()
        for _ in range(3):                       # multiple passes; first does CUDA autotune
            _ = model(dummy)
    torch.cuda.synchronize()
    _warmed = True

# /startup — returns OK once the container is past Python import + model load,
# but not yet warmed. Kubelet stops killing the pod for slow startup.
@router.get("/startup")
def startup():
    if _model is None: raise HTTPException(503, detail="model not loaded")
    return {"status": "started"}

# /live — cheapest possible check. Just "the process is alive."
@router.get("/live")
def live():
    return {"status": "alive"}

# /ready — runs a real inference path. Only returns 200 when warm.
@router.get("/ready")
def ready():
    if not _warmed:
        raise HTTPException(503, detail="not warmed")
    try:
        # Quick smoke: tiny inference, must complete < 200ms
        with torch.no_grad():
            _ = _model(build_smoke_input())
    except Exception as e:
        raise HTTPException(503, detail=str(e))
    return {"status": "ready"}

The warmup phase, explicitly

Most ML frameworks have several first-time costs that materialize as latency on the first N requests:

  • CUDA kernel JIT. PyTorch / TensorRT compile kernels for the specific input shapes on first invocation. Run the warmup with the input shapes you'll see in production.
  • cuDNN autotune. Convolutional layers auto-select the fastest algorithm for the shape; the first call benchmarks options. Set torch.backends.cudnn.benchmark = True only if input shapes are stable; otherwise autotune runs every shape change.
  • vLLM/TGI prefix cache. LLM servers populate KV cache buffers and run prefill passes; first inference is ~3-5× slower than subsequent.
  • Connection pool establishment. Downstream HTTP clients, database connection pools, feature-store clients. Pool warmup on init.

Probe timing — the cold-start budget

Kubelet behavior to internalize:

  • startupProbe: kubelet allows up to periodSeconds × failureThreshold for the pod to become ready. If your cold-start is 90s, set periodSeconds: 5, failureThreshold: 30 for 150s of grace.
  • livenessProbe: only starts after startupProbe succeeds. Set it to a check that almost always succeeds; aggressive liveness probes kill pods that are alive but momentarily slow under load.
  • readinessProbe: also starts after startup succeeds. Failures remove the pod from Service endpoints but don't restart it. This is the right hook for "currently rebuilding KV cache" or "model swap in progress."

The gracefulness pattern

For zero-downtime deploys, pods need a clean shutdown path:

containers:
  - name: inference
    lifecycle:
      preStop:
        exec:
          # 1. Mark readiness as false so Service stops sending new requests
          # 2. Wait for in-flight requests to drain
          command:
            - sh
            - -c
            - "curl -s -X POST localhost:8000/drain && sleep 30"
spec:
  # Generous shutdown window — must exceed preStop + actual cleanup
  terminationGracePeriodSeconds: 60
The readiness-lies test

Run this test once: scale up a fresh replica, then immediately fire a synthetic load test at the new pod's IP. Plot p99 over the first 60 seconds. If p99 is >5× steady-state for any of those seconds, readiness lied — the pod was added to Service endpoints before it was actually ready. vLLM specifics worth knowing in 2026

vLLM is not just an academic project — companies like Stripe report a 73% reduction in inference costs after migrating from Hugging Face Transformers to vLLM, processing 50 million daily calls on one-third of the GPU fleet. The architectural levers:

  • PagedAttention. Manages KV cache in non-contiguous memory blocks like virtual memory, eliminating fragmentation. Enables near-optimal KV memory utilization.
  • Continuous batching. Doesn't wait for a batch to complete; new requests join in-flight batches as tokens free up. The single biggest lever for LLM throughput.
  • Prefix caching. Common prefixes (system prompts, RAG context) cache their KV across requests. For chatbot workloads, can 2-4× throughput.
  • Quantization. FP8 / INT8 / AWQ / GPTQ support; 2-4× memory reduction with minimal quality loss for most models.

Wrong runtime for a workload typically wastes 2-10× the hardware. The runtime choice deserves real benchmarking before scaling out; the cost of the choice compounds with every replica.

RUNTIME Chapter XII

Batching · static, dynamic, continuous.

The three shapes Static (wait for N, run, repeat). Dynamic (collect for up to T ms, run). Continuous (slot-by-slot, new requests join in flight).

The rule Match batching to model: encoders → dynamic, decoders → continuous, low-latency single-tenant → none.

Batching is the lever that turns GPU FLOPs into requests-per-second. The same model on the same GPU can serve 10 RPS or 1000 RPS depending entirely on how requests are batched. Three patterns dominate; matching the right pattern to the workload is most of the optimization.

Static batching


      
Triton's dynamic batcher is the canonical implementation. Configure max_queue_delay_microseconds and max_batch_size; the server collects requests up to whichever comes first, then runs a batch. Trades a small fixed latency (the wait window) for substantially higher throughput.


      
Continuous batching (LLM-specific)

      
Instead of waiting for a complete batch of requests to be fully generated, vLLM immediately funnels new requests as soon as a token from another user is ready. In your Kubernetes cluster, this leads to consistent, high GPU utilization and minimal wait times for end users.

      
This works because autoregressive decoding generates one token at a time per request; the GPU schedules many requests in parallel at the same "step." When request X completes, request Y can take its memory slot at the next step. Throughput climbs 2-5× vs static batching at similar p99 latency.

The latency cost of batching

      
Static and dynamic batching both add a structural delay — the wait window for batch formation. Three numbers to know:

      
    
        
  • The wait window cost: roughly half of max_queue_delay in expectation (uniform-distributed arrival). For max_queue_delay = 10ms, expect ~5ms added to p50.
    • 
        
  • The batch-execution amortization: a batch of 16 takes ~1.5-3× the time of a single request, not 16×. The bigger the batch, the better the per-request economics.
    • 
        
  • The variance penalty: a batched request finishes only when the whole batch finishes. If one member is 10× longer than others, all wait. Continuous batching solves this for autoregressive; for encoders, group requests with similar input shapes.
    • 
      


      

        
The "instance_group count" knob

        
Triton's instance_group { count: N } creates N independent instances of the model on the same GPU. With N=2 and dynamic batching, two batches can be processed concurrently — useful when one batch is shape A (e.g., short input) and another is shape B (long input). Tune by measuring p99 at varying N; the optimum is often 2-4 for sufficient VRAM, more for memory-light models.

The roofline model — what's the actual ceiling

Roofline analysis tells you whether you're compute-bound, memory-bound, or somewhere else. Inputs: peak FLOPS of the GPU, peak memory bandwidth, and arithmetic intensity (FLOPs / byte) of your model.

  • If model arithmetic intensity < peak FLOPS / peak bandwidth: you're memory-bound. Throughput ceiling is bandwidth × intensity. Optimize: reduce memory traffic (kernel fusion, FP8/INT8 quantization, prefix caching).
  • If model arithmetic intensity > that ratio: you're compute-bound. Ceiling is peak FLOPS. Optimize: use tensor cores effectively (FP16/BF16/FP8), batching to fill SMs.
  • Most LLM inference is memory-bound: KV cache reads dominate. This is why PagedAttention, KV cache compression, prefix caching matter so much.

The five-question diagnostic

For any inference slowness, walk these questions in order:

  1. GPU utilization >80%? If no → you're not feeding the GPU fast enough. Check batching, queue depth, preprocessing.
  2. GPU memory near full? If yes → KV cache pressure causing recompute. Reduce max_num_seqs or model size.
  3. Tensor cores active >50%? If no → you're not using the right precision. FP16/BF16/FP8 matters.
  4. PCIe TX high during inference? If yes → you're CPU↔GPU bouncing data. Pin buffers; reduce copies.
  5. Python frames hot in py-spy? If yes → preprocessing or postprocessing dominates. Move to compiled code.
The profiling escalation ladder

Continuous metrics (Prometheus + DCGM) → answer "is something wrong now?" Sampling profilers (py-spy) → answer "where is Python time going?" Framework profilers (PyTorch profiler, vLLM stats) → answer "where is GPU time going?" Tracing profilers (Nsight) → answer "exactly which kernel is slow and why?" Most teams never need the bottom rung; most performance work happens in the top three. Cross-reference Issue 03 on the broader observability story.

The two-stage scaling chain

Pod-level autoscaling adds pods; if the cluster has no spare capacity, the pods stay Pending. Node-level autoscaling (Karpenter, cluster-autoscaler) provisions new nodes. The end-to-end latency:

  • Pod request → schedule: 0-30s (HPA decision + Kube scheduler).
  • Node request → ready: 30-120s (Karpenter requests instance, OS boots, kubelet joins, image pulls).
  • Pod schedule → running: 30-180s (image pull, model load, warmup — most of this chapter's territory).

End-to-end for a fresh node + fresh pod: 60-300 seconds before serving traffic. This is why pre-pull (Chapter VIII) and lazy pulling (Chapter VII) compound — they pull node-provisioning latency below the pod cold-start.

The "scale-up cliff" pattern

For predictable spikes (daily user activity peaks, scheduled batch jobs), pre-scale rather than wait for the autoscaler to react. Two patterns:

  • Scheduled scaling. KEDA cron trigger that bumps minReplicas at 8:55am for a 9am peak.
  • Predictive scaling. Some platforms (AWS Predictive, custom ML on your traffic) forecast load and pre-scale 15 min ahead.
The autoscaler dance during deploys

During a rolling deploy, old pods are draining while new pods start. The autoscaler may panic at the temporary capacity dip and scale up — then the new replicas come up, and now you have 2× capacity. To avoid the oscillation: longer stabilization windows during deploy (set scaleDown.stabilizationWindowSeconds: 600+) or pause autoscaling during deploy via temporary minReplicas bump.

RUNTIME Chapter XV

Scale-to-zero · the real tradeoffs.

The promise No load = no cost. Idle GPUs cost $1-10/hour; scale-to-zero saves it.

The cost First request after idle waits for the full cold-start envelope. Acceptable for some workloads; catastrophic for others.

Inference workloads are extremely "spiky." At night, you often need zero capacity, while during the day, requests skyrocket. This is where Serverless AI on K8s shines. Scale-to-zero: when no requests come in, the expensive GPU nodes are shut down via Knative. Scale-to-zero converts always-on cost into per-request cost. For workloads with high idle time, this is the single biggest cost lever available. For workloads where the first-request-after-idle matters (interactive APIs, real-time inference), it's a trap.

Mitigating cold-start in STZ workloads

If you've decided STZ fits but the first-request latency is unacceptable, four mitigations stack:

  • Aggressive cold-start optimization. Everything in chapters VI-X. Drop the envelope below the user-tolerable threshold (typically <5s for interactive, <30s for back-office).
  • Warm-pool of one. Set minReplicas: 1 during business hours via a cron-based KEDA scaler; allow minReplicas: 0 overnight.
  • Predictive warm-up. A monitoring signal (user logs in, opens dashboard) preemptively scales the service before the actual inference request arrives.
  • Pre-pulled image + small artifact. The biggest leverage; cold-start of <15s makes STZ work for most semi-interactive workloads.

Karpenter + KServe — the node-side complement

Scale-to-zero saves money only if the underlying node also scales away. Karpenter's consolidation policy removes unused nodes 5-10 minutes after the last pod evacuates. The combined behavior:

      
Some teams set minReplicas: 1 to avoid STZ but configure aggressive PDB/affinity that prevents the replica from being evicted during node consolidation. Result: one zombie replica per node, never scales down, autoscaler can't consolidate, paying for permanent unused capacity. The fix: if you can't STZ, set min replicas based on actual traffic shape, not as a workaround for slow cold-start.


      

        
The STZ decision tree, summarized

        
Is the workload interactive (user waits on the response)? Yes → measure cold-start; if <5s, STZ OK; if >5s, optimize first, then STZ. No → STZ aggressively; cold-start is amortized across the batch. Is traffic predictable (diurnal / weekly)? Yes → schedule warm-up; STZ between peaks. No → conservative cold-start optimization + low min replicas. The CMMM-specific cold-start dominant stage

For most teams running LLMs, model load is the dominant cold-start cost. For CMMM, dill reconstruction is the dominant cost. The PyMC trace contains many MultiTrace objects, sample chains, posterior parameters; deserializing them and reattaching to fresh cmmm-science code is single-threaded Python work. The right optimization is therefore caching the promoted ModelInference object — which the code already supports via the inferencer.bin fallback, but production pods never hit a local cache. Persisting promoted inferencers to GCS, so every fresh pod pulls a pre-promoted artifact, would skip the most expensive step entirely.

Improvement 2 — persist promoted ModelInference to GCS

Chapter: IX. Estimated impact: Cuts dill-reconstruction cost from every pod cold-start to once-per-promotion.

The code already supports loading inferencer.bin from GCS as the preferred path. What's missing: a step in the model promotion pipeline that materializes the promoted inferencer to GCS at promotion time, not at pod startup.

from cmmm_science.serializable.serving_model.model_inference import ModelInference
from cmmm_science.serializable.serving_model.base import ServingModelBase
import dill
from google.cloud import storage

def promote_to_gcs(model_uri: str, inferencer_uri: str):
    """Run at promotion time, NOT at pod startup.
    Loads MMMMod from dill, promotes to ModelInference, persists to GCS."""
    gcs = storage.Client()
    # Pull model.dill
    with open("/tmp/m.dill", "wb") as f:
        gcs.download_blob_to_file(model_uri, f)
    with open("/tmp/m.dill", "rb") as f:
        mmm = dill.load(f)
    inferencer = ModelInference.from_model(mmm, apply_legacy_adstock_normalization=True)
    # Persist binary form for pod consumption
    inferencer.to_binary_file("/tmp/inferencer.bin")
    bucket = gcs.bucket("cmmm-models")
    bucket.blob(inferencer_uri).upload_from_filename("/tmp/inferencer.bin")

# Wire into the GitHub Action that processes model_promotions/*.yaml PRs.

This is the single biggest cold-start win for CMMM specifically: pods no longer pay the promote MMMMod → ModelInference cost (which involves reconstructing the PyMC trace and running ModelInference.from_model) on every startup. They just ModelInference.from_binary_file the pre-promoted artifact. A 30-60s reduction in the model-load stage is realistic.

Improvement 3 — parallel GCS fetch via init container

Chapter: IX. Estimated impact: 3-8× faster artifact fetch for large dills.

Currently, gcs_client.download_blob_to_file opens a single HTTPS connection and streams sequentially. For the multi-hundred-MB dill files CMMM produces, this is bandwidth-underutilized. Replace with a parallel-multipart download:

spec:
  volumes:
    - name: model-cache
      emptyDir: { sizeLimit: "4Gi", medium: "Memory" }    # tmpfs for fast reads
  initContainers:
    - name: model-fetcher
      image: google/cloud-sdk:slim
      command: ["sh", "-c", "\
        gcloud storage cp --recursive \
          \"${MODEL_BASE_URI}/*\" /cache/ \
          --num-threads=8 --read-chunk-size=16M"]
      env:
        - { name: MODEL_BASE_URI, value: "gs://cmmm-models/acme/revenue/20260512_193000" }
      volumeMounts:
        - { name: model-cache, mountPath: /cache }
      resources:
        requests: { cpu: "2", memory: "2Gi" }
  containers:
    - name: serving
      image: ghcr.io/myorg/cmmm-serving:v2.18.0
      env:
        - { name: MODEL_PATH, value: "/cache/mmm_cal.pkl" }
        - { name: INFERENCER_PATH, value: "/cache/inferencer.bin" }
      volumeMounts:
        - { name: model-cache, mountPath: /cache, readOnly: true }

Improvement 6 — explicit /ready probe that runs an inference path

Chapter: X. Estimated impact: Eliminates the "pod is healthy but first request is pathological" failure mode.

The startup_events.py file already runs real inference paths during Gunicorn preload. The improvement is making the readiness probe gate on this completing. Today, the readiness probe likely passes whenever the HTTP server binds; tomorrow, it should pass only when CMMM_STARTUP_TESTS_DONE=1 is set and a smoke endpoint succeeds.

import os
from fastapi import APIRouter, HTTPException
from app import app

router = APIRouter()

@router.get("/live")
def live():
    return {"status": "alive"}

@router.get("/ready")
def ready():
    # Gate 1: did the preload-time warmup complete?
    if os.environ.get("CMMM_STARTUP_TESTS_DONE") != "1":
        raise HTTPException(503, detail="startup tests not complete")
    # Gate 2: model state actually present
    if not hasattr(app.state, "mmm") or app.state.mmm is None:
        raise HTTPException(503, detail="model not loaded")
    # Gate 3: optional cheap smoke — current model can answer a CPT call
    if not app.state.cached_cpt_file_contents:
        raise HTTPException(503, detail="cpt cache empty")
    return {"status": "ready", "model_load_time_sec": app.state.model_load_time_sec}
