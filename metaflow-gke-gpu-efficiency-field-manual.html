FRAME Chapter I

The frame · why 8-GPU launch is so painful.

The naked truth A Metaflow step that requests 8 GPUs typically waits 30-60 minutes before its first line of Python runs.

Where the time goes Capacity acquisition. Image pull. Model weight load. Three independent cliffs.

A Metaflow step decorated @resources(gpu=8) compiles down to one Kubernetes pod with nvidia.com/gpu: 8 in its resource limits. On paper, this is a one-line declaration. In practice, on a GKE cluster that doesn't already have an idle 8-GPU node sitting empty, the pod's first second of Python runtime is preceded by a chain of waits — capacity must be acquired, a node must be provisioned, a 12GB CUDA container must be pulled, model weights must be loaded from object storage to the GPU's framebuffer memory, and only then does training begin. Each link in this chain has a different time cost, a different failure mode, and a different optimization technique. This issue is the systematic map of every cliff and every lever for flattening it.

The H100 economics

To frame the urgency. An a3-highgpu-8g node — 8× H100 80GB, 208 vCPU, 1872 GB RAM, per the Northflank GCP instance catalog — runs roughly $30-40/hr on-demand, $4-8/hr on the various Spot/preemptible programs, $15-25/hr through Dynamic Workload Scheduler Flex-Start (discounted up to 53%). An a3-ultragpu-8g with 8× H200 141GB at 224 vCPU and 2952 GB RAM is even pricier. An a4-highgpu-8g with 8× B200 192GB at 224 vCPU and 3968 GB RAM is at the very top of the price ladder.

At $30/hr × 8 GPUs = $240/hr per node = $4/minute. A 45-minute cold-launch wastes $180 per training step. Run it ten times a day, that's $1,800/day, $54,000/month, in launch overhead alone. The optimizations in this issue have first-month ROI measured in tens of thousands of dollars.

What Metaflow's @resources actually does

Per the Metaflow @resources docs: "Use @resources to specify the resource requirements independently of the specific compute layer (@batch, @kubernetes). You can choose the compute layer on the command line by executing e.g. ... Number of CPUs required for this step. ... Number of GPUs required for this step." The decorator is a declarative annotation — it doesn't specify where the step runs, just the resource needs. The compute layer is chosen at runtime via --with kubernetes or in the deployed flow's config.

When the chosen compute is Kubernetes, Metaflow translates @resources(gpu=8) into pod-spec fields. Per the Metaflow Kubernetes documentation: applications target "a custom schedulable resource such as amd.com/gpu or nvidia.com/gpu, which Metaflow's Kubernetes resources integration is already configured to call when a user specifies a decorator like @kubernetes(gpu=1)". The GPU request lands in the pod's spec.containers[].resources.limits as nvidia.com/gpu: 8 — and that's the entirety of what Metaflow itself contributes. Everything downstream — scheduling, capacity, node provisioning, data loading — is Kubernetes and GKE territory.

This issue is largely a guide to what Metaflow doesn't do for you, and the GKE features that fill the gaps. The Metaflow side is @kubernetes(annotations=..., labels=..., tolerations=..., node_selector=..., persistent_volume_claims=...) — the standard pod-spec knobs, threaded through the decorator (Chapter XVI).

Three design principles that follow

  1. Decouple capacity from launch. Don't make the training step wait for capacity. Use DWS Flex-start with queued provisioning to let the request sit in a queue; only start the pod when nodes are guaranteed ready.
  2. Pre-stage everything that can be pre-staged. Container images on secondary boot disks. Model weights on Hyperdisk ML. Datasets cached in Local SSD via GCS Fuse profiles. The pod's first job is to run training, not download.
  3. Declare the resource tiers, let the autoscaler pick. Custom ComputeClasses turn "I need an H100" into "I need an H100 from reservation, fall back to flex-start, fall back to spot" — without changing the pod spec.

FRAME Chapter II

Three bottlenecks · capacity, launch, data.

The map Capacity = "I can't get GPUs." Launch = "I have GPUs but startup is slow." Data = "Workload runs but spent 20 min loading."

Don't conflate them Each has its own toolset. DWS for capacity. Boot disks for launch. Hyperdisk ML / GCS Fuse for data.

The 30-60 minute cold-launch is a sum of three independent latencies. Treating them as one problem leads to muddled solutions; treating each on its own terms produces a clear stack of techniques. The capacity bottleneck — "GCP can't give me 8 H100s in zone us-central1-a right now" — is solved by Dynamic Workload Scheduler, ComputeClass fallbacks, and reservations. The launch bottleneck — "the node is up but the container is still pulling" — is solved by secondary boot disks, image streaming, and Autopilot's fast-starting nodes. The data bottleneck — "the container is running but it's been reading 140GB of weights for 20 minutes" — is solved by GCS Fuse with tuned caching, Hyperdisk ML in READ_ONLY_MANY mode, or the Run:ai Model Streamer. This chapter names them and shows the map; the rest of the issue dives into each.

Bottleneck #1 — capacity acquisition

The hardest problem because it's not under your control. There are only so many H100s in any given GCP zone at any given moment, and they're shared across every GCP customer. When you ask the cluster autoscaler for an 8-GPU node and there are none available, the answer is ZONE_RESOURCE_POOL_EXHAUSTED — and you wait. The 2026 toolkit:

  • Reservations. Commit to capacity in advance; it's guaranteed yours. Highest cost, highest certainty.
  • Dynamic Workload Scheduler Flex-start. Submit a capacity request; GCP fulfills it when capacity becomes available, up to 7 days. 53% billing discount. Per the DWS launch blog: "Dynamic Workload Scheduler intelligently persists the request; once the capacity becomes available, it automatically provisions your VMs enabling your workloads to run continuously for the entire duration of the capacity allocation."
  • DWS Calendar mode. Future reservations, 7 or 14 days, bookable up to 8 weeks in advance. The "hotel" model.
  • Spot/preemptible. Cheapest, but interruptible. Good for fault-tolerant training (Issue 02 checkpointing pattern).
  • ComputeClass fallback priorities. Declare "try reservation, then flex-start, then spot, then on-demand" — autoscaler iterates the list.

Bottleneck #2 — launch latency

The node exists. The kubelet has started. Now we're racing against the time-to-first-process. Five sub-components:

  • VM boot. The Compute Engine VM itself. Out of your control beyond instance type selection.
  • Node bootstrap. GKE installs kubelet, container runtime, GPU driver, GPU device plugin. ~2-5 minutes on cold A3.
  • Image pull. The big one. A modern AI/ML container is 8-15GB. Pulled from Artifact Registry over the VPC. Per the GKE data-loading blog: "inference servers such as Triton, Text Generation Inference (TGI), or vLLM are packaged as containers that are often over 10GB in size; this can make them slow to download, and extend pod startup times in Kubernetes."
  • Init containers + sidecars. Setup work before the main container starts.
  • Container startup itself. Python imports, library initialization, CUDA context setup.

Optimization toolkit:

  • Secondary boot disks for container images. Per the same blog: "GKE lets you pre-cache your container image into a secondary boot disk that is attached to your node at creation time." Bypasses the image pull entirely.
  • GKE Image Streaming. Containerd starts running before the full image is pulled, streaming layers on-demand.
  • Autopilot fast-starting nodes. Per the GKE docs: "Requesting compatible GPUs in Autopilot mode results in up to four times faster node startup time and up to two times faster Pod scheduling time than similar requests in GKE Standard mode, because the Autopilot GPU workloads can use fast-starting nodes."
  • Image optimization. Multi-stage builds, layer ordering, minimal base images. The lowest-tech but highest-ROI win.

Bottleneck #3 — data loading

The container is running. CUDA is initialized. Now the training script needs to load a multi-gigabyte model checkpoint. The naive path — torch.load("gs://bucket/model.pt") with the gsutil-style download — is the slowest. Options ranked by typical speed:

  • Run:ai Model Streamer (vLLM only). Per the GKE docs: "Instead of the download-then-load process, the model streamer streams the model directly from your Cloud Storage to the GPU or TPU memory. The streamer uses a high-performance backend to read multiple parts of the model, called tensors, in parallel." Bypasses local disk entirely.
  • Hyperdisk ML in READ_ONLY_MANY mode. Per the Hyperdisk ML docs: "When enabled in read-only-many mode, you can use Hyperdisk ML to accelerate the loading of model weights by up to 11.9X relative to loading directly from a model registry. This acceleration is made possible by the Google Cloud Hyperdisk architecture that allows scaling to 2,500 concurrent nodes at 1.2 TiB/s."
  • Managed Lustre. Sub-millisecond latency, multi-PB, up to 1 TB/s. Overkill for most teams but the right answer for very large training fleets.
  • GCS Fuse with tuned caching. Per the GCS Fuse docs: "For AI/ML workloads, you can use pre-defined Cloud Storage FUSE profiles for training, serving, and checkpointing to automatically tune performance based on your workload and node configuration."
  • Naive GCS download. The baseline. Slow.

Why the user named exactly three things

The original question asked specifically about DWS Flex-start, gke-gcsfuse/volumes, and custom ComputeClass. That triplet is not random — it's one tool from each bottleneck. Flex-start is the headline capacity solution; gke-gcsfuse/volumes is the headline data-loading mount path; ComputeClass is the orchestration layer that ties capacity strategy to specific workloads. Combining all three is the "core stack" for a 2026 GKE training cluster. This issue walks each in depth, then shows how Metaflow's @kubernetes decorator threads all three through one pod spec.

The three-bottleneck discipline

When debugging slow launches, name which bottleneck you're in before reaching for tools. A pod stuck in Pending with cluster-autoscaler logs showing capacity errors is a bottleneck #1 problem — adding GCS Fuse won't help. A pod in ContainerCreating for 12 minutes is bottleneck #2 — image streaming or secondary boot disk. A pod in Running for 18 minutes before the first loss line appears is bottleneck #3 — Hyperdisk ML or Run:ai streamer. The fastest debug is the question "where in the timeline are we stuck?"

The catalog A3 (H100), A3 Mega (H100 Mega), A3 Ultra (H200), A4 (B200), G2 (L4). Each has 8-GPU variants and different shapes.

For @resources(gpu=8) You're picking from a3-highgpu-8g, a3-megagpu-8g, a3-ultragpu-8g, or a4-highgpu-8g.

Before you can request capacity, you have to know what you're requesting. The 2026 GKE GPU catalog has three accelerator generations, each in a handful of variants. The A3 Ultra specifics — what's different

Per the A3 Ultra GKE cluster docs: "GPU drivers: The H200 GPUs in A3 Ultra VM instances require a minimum of the R550 GPU driver version, which is available in GKE 1.31 as the latest driver version. For A3 Ultra, you must set gpu-driver-version=latest with GKE 1.31. For GKE version 1.31.5-gke.1169000 or later, GKE automatically installs the R550 GPU driver versions on A3 Ultra nodes by default. Storage: For A3 Ultra node pools, you must set the disk type to hyperdisk-balanced. GPUDirect RDMA: To use GPUDirect RDMA with A3 Ultra, use GKE version 1.31.4-gke.1183000 or later. Node images: To use GPUDirect RDMA, the GKE nodes must use a Container-Optimized OS node image. Ubuntu and Windows node images are not supported."

Three constraints worth memorizing for A3 Ultra: (1) GKE 1.31+ with gpu-driver-version=latest required for R550 driver. (2) Disk type must be hyperdisk-balanced (not the older pd-* types). (3) GPUDirect RDMA requires GKE 1.31.4-gke.1183000+ and Container-Optimized OS — Ubuntu and Windows nodes can't use it. Per the AI Hypercomputer A3 Ultra cluster setup guide: "Make sure that you have a reservation for the required number of a3-ultragpu-8g machines using the DENSE deployment type."

Reservation deployment types — DENSE vs SPARSE

When you reserve A3 Ultra or A4 capacity, GCP offers two physical deployment modes. DENSE packs reserved VMs onto adjacent physical machines in the same rack — minimizing inter-node network latency for multi-node NCCL. SPARSE spreads them out — higher availability under failures, but worse all-reduce performance. For multi-node training where 8-GPU pods are nodes in a larger collective, DENSE is the right answer; the per-step communication tax is what you're paying for the hardware. For single-node 8-GPU jobs (where intra-node NVLink dominates), the difference is negligible.

The choice of variant, for Metaflow @resources(gpu=8)

The decision tree for which 8g variant to target:

  • You need every recent training optimization → A4 (B200). Tightest supply; expensive; the right answer for frontier work.
  • You're training 30B-100B parameters and need fast inter-node communication → A3 Ultra (H200) with DENSE reservation + GPUDirect RDMA.
  • You're training sub-30B or doing inference, want widest availability → A3 (H100). The 2024-2025 sweet spot; most supply; broadest ComputeClass fallback options.
  • You need NCCL-heavy multi-node, but not H200 memory → A3 Mega (H100 Mega). The interconnect-fixed H100.
  • Inference of <30B models or training of <7B → consider G2 (L4) with gpu=8 mapping to g2-standard-96. ~10x cheaper.
The supply reality

The 2026 GPU supply hierarchy in increasing scarcity: L4 (always available somewhere) → H100 (usually available with patience) → H100 Mega (somewhat tight) → H200 (often need reservation) → B200 (need reservation or be very patient). Plan ComputeClass priority lists accordingly — for an H200 workload, the fallback chain should include H100 Mega and H100 specifically because pure H200 capacity will not always be there.

CAP Chapter IV

DWS Flex-start · the ProvisioningRequest mechanism.

The promise Submit a capacity request. Wait. Get nodes when GCP can deliver them. 53% cheaper than on-demand. Up to 7-day runtime.

The mechanism A Kubernetes CRD — ProvisioningRequest — that signals "I need this much capacity"; cluster autoscaler queues it; nodes provision atomically.

Dynamic Workload Scheduler is GCP's answer to "GPU stockouts hurt me, but I don't need them in the next 5 minutes." The Flex-start mode lets you submit a capacity request that GCP fulfills opportunistically — typically minutes to hours, occasionally days, depending on supply. Once fulfilled, the nodes are guaranteed yours for up to 7 days, non-preemptible. The billing discount of up to 53% reflects that you're flexing on start time in exchange for capacity. This chapter walks the mechanism — the ProvisioningRequest CRD, the warden-policies label, the pod annotations that bind a workload to a request, the lifecycle from creation to fulfillment.

What DWS Flex-start actually is

Per the GKE Flex-start concepts page: "Flex-start, powered by Dynamic Workload Scheduler, provides a flexible and cost-effective technique to consume specialized compute resources, like GPUs or TPUs, when you need to run AI/ML workloads. Flex-start lets you dynamically provision Flex-start VMs for GPUs, TPUs, and H4D machine series as needed, for up to seven days, not bounded to a specific start time, and without the management of long-term reservations. Therefore, flex-start works well for smaller to medium-sized workloads with fluctuating demand requirements or short durations. For example, small model pre-training, model fine-tuning, or scalable serving models."

The economics from the same source: "Flex-start is recommended if your workload requires dynamically provisioned resources as needed, for up to seven days with short-term reservations, no complex quota management, and cost-effective access. Flex-start is powered by Dynamic Workload Scheduler and is billed using Dynamic Workload Scheduler pricing: Discounted (up to 53%) for vCPUs, GPUs, and TPUs. You pay as you go." The catch: "To use flex-start in GKE, your cluster must meet the following requirements: To run GPUs, use GKE version 1.32.2-gke.1652000 or later."

The mental model — a hotel vs a restaurant

Per Strzelczyk's metaphor: DWS Calendar Mode is like booking a hotel — you specify dates, location, hardware type, hardware quantity in advance; the system holds the reservation; you pay for the time regardless of utilization. Per the same source: "You tell DWS that you need hardware, let's say 10x A4 machines, to run a job that will take at most 6 days. With that knowledge, DWS goes out to the Cloud to get you your 10 A4 machines. After some time (this is where the 'flex' part comes from — it's a flexible process) the system has the 10 A4 machines you need and provides them to you all at once. This 'all-or-nothing' approach ensures you receive the full requested capacity simultaneously." Flex-start is the restaurant — show up, the host puts you on a waitlist, you wait, eventually they seat you.

The ProvisioningRequest CRD

The Kubernetes-native expression of "I want this capacity, queue me." Per the GKE provisioningrequest docs: "PROVISIONING_REQUEST_NAME: The name of the ProvisioningRequest. You'll refer to this name in the Pod annotation. MAX_RUN_DURATION_SECONDS: Optionally, the maximum runtime of a node in seconds, up to the default of seven days. To learn more, see How flex-start with queued provisioning works. You can't change this value after creation of the request. This field is available in GKE version 1.28.5-gke.1355000 or later."

The critical label · cloud.google.com/apply-warden-policies

Per the GKE provisioningrequest docs: "GKE might apply validations and mutations to Pods during their creation. The cloud.google.com/apply-warden-policies label allows GKE to apply the same validations and mutations to PodTemplate objects. This label is necessary for GKE to calculate node resource requirements for your Pods." Without this label, the ProvisioningRequest fails because GKE can't determine what shape of nodes to provision from the template alone. Easy to forget; failure mode is non-obvious.

The pod-to-request binding annotations

Once the ProvisioningRequest is fulfilled, your actual workload pod (not the template) needs to consume that capacity. Per the docs: "The Pod annotation key consume-provisioning-request defines which ProvisioningRequest to consume. GKE uses the consume-provisioning-request and provisioning-class-name annotations to do the following: To schedule the Pods only in the nodes provisioned by flex-start with queued provisioning. To avoid double counting of resource requests between Pods and flex-start with queued provisioning in the cluster autoscaler. To inject safe-to-evict: false annotation, to prevent the cluster autoscaler from moving Pods between nodes and interrupting batch computations. You can change this behavior by specifying safe-to-evict: true in the Pod annotations."

The provisioning-class options

Three classes that you can set on the ProvisioningRequest:

  • queued-provisioning.gke.io — the canonical GKE Flex-start with queued provisioning. All-or-nothing. The default for distributed training.
  • check-capacity.autoscaling.x-k8s.io — best-effort. Check if capacity is available right now; if yes, scale up; if no, fail fast (no queueing).
  • best-effort-atomic-scale-up.autoscaling.x-k8s.io — try atomic scale-up immediately; fall back to no-op if not possible.

For Metaflow's @resources(gpu=8) on a single pod, queued-provisioning.gke.io is the right choice — wait for capacity; guarantee atomicity; honor the 53% discount.

The 7-day maximum runtime

Per the DWS Flex-start technical guide: "7-Day Maximum Runtime: All resources provisioned via Flex Start have a maximum lifespan, defined by the maxRunDuration parameter, which cannot exceed seven days (168 hours). After this duration expires, the resources are automatically terminated or stopped, depending on the configuration. Non-Preemptible Guarantee: This is a foundational promise of the service. Once a Flex Start resource transitions to an ACTIVE or RUNNING state, it is guaranteed to run for its entire specified duration without being preempted by Google."

The implications: training runs longer than 7 days need to checkpoint and resume on a fresh Flex-start allocation. Inference deployments that need stability beyond 7 days need a different consumption model (reservations or on-demand). The maxRunDurationSeconds field, once set, cannot be changed — you can't extend an in-flight request. Set it generously.

The dedicated capacity pool

Per the same source: "A key architectural detail of the Dynamic Workload Scheduler is that it operates with a controlled, separate pool of hardware capacity dedicated specifically to DWS requests. This means a DWS Flex Start request is not competing for the same pool of immediately available hardware as a standard on-demand request. This separation helps DWS to manage its queue and fulfill requests more reliably over time." The implication: DWS isn't just an API in front of the on-demand market — it's its own capacity bucket, which is why it can offer the all-or-nothing guarantee that on-demand can't.

The lifecycle

From kubectl apply on a ProvisioningRequest to running pods, the states (visible via kubectl describe provreq train-8gpu-req):

  1. Accepted — GKE has received the request and queued it. Reason: SuccessfullyQueued.
  2. Provisioning — DWS is working on it. May stay here for minutes to days. Status: Provisioned=False, Reason: ResourcePoolExhausted (waiting) or Provisioning (actively allocating).
  3. Provisioned — Nodes are ready. Provisioned=True. This is when you start your workload pods (or Kueue unsuspends them automatically).
  4. Failed — Couldn't fulfill, hit a timeout, or some other terminal error.

Per the docs: "When the Provisioning Request request reaches the Provisioned=true status, you can run your Job to start the Pods. This avoids proliferation of unschedulable Pods for pending or failed requests, which can impact kube-scheduler and cluster autoscaler performance. Alternatively, if you don't care about having unschedulable Pods, you can create Pods in parallel with the Provisioning Request request."

The atomic guarantee

Per the GKE Flex-start docs: "The scheduler waits until all the requested resources are available in a single zone. All Pods of the workload can run together on newly provisioned nodes. The provisioned nodes aren't reused between workload executions. Note: Clusters running GKE version 1.29.1-gke.1708000 and later optimize zone selection for lower wait times. Earlier versions might experience longer queueing." The atomicity guarantee — all-or-nothing zone-atomic provisioning — is the key property for multi-node distributed training. You never get partial capacity.

The single-PodSet limitation

Per the same docs: "A single ProvisioningRequest object supports only one entry in the podSets list. Requests with multiple podSet entries fail." The implication for Metaflow: a single @resources(gpu=8) step maps to one PodSet, which works fine. A more complex workload with heterogeneous pod shapes (e.g., RayJob with head + workers, or JobSet with different roles) hits this limitation — covered in Chapter XIV.

The cancellation cleanup

Per the docs: "To cancel the request before it's provisioned, you can delete the ProvisioningRequest: kubectl delete provreq PROVISIONING_REQUEST_NAME -n NAMESPACE. In most cases, deleting ProvisioningRequest stops nodes from being created. However, depending on timing, for example if nodes were already being provisioned, the nodes might still end up created. In these cases, the cluster autoscaler removes the nodes after 10 minutes if no Pods are created." Set up monitoring for orphaned nodes — a cancelled request that races provisioning can leave you billed for nodes you didn't get to use.

CAP Chapter V

Best-Effort vs Queued · the critical choice.

Two configurations Best-Effort flex-start (Option A): fast start, per-node. Queued provisioning (Option B): atomic, gang scheduling.

For @resources(gpu=8) on one pod Either works. For multi-node (multiple pods, distributed training), queued is mandatory.

DWS Flex-start has two operational modes when used in GKE, and the choice between them determines whether your workload starts fast or starts atomically. The Best-Effort Flex-start mode optimizes for getting nodes up as soon as each one is available — good for single-node and best-effort workloads. The Queued Provisioning mode buffers the request until all requested nodes can come up simultaneously — mandatory for distributed training where 8 workers need to be online together for NCCL initialization. The difference is one CLI flag at node-pool creation; the implications shape every workload running on that pool.

The two configurations, side by side

Per the Startup's Guide to DWS Flex-Start on GKE (March 2026): "Mainly, there are two ways of leveraging DWS Flex-Start in your jobs, addressing different requirements for workload scheduling and resource guarantee: When configuring your GPU node pool, it is crucial to select the provisioning strategy that best matches your workload's needs: Best-Effort or All-or-Nothing. Best-Effort Provisioning (Option A) prioritizes the speed of start-up for single-node or smaller jobs by utilizing available capacity quickly. Queued Provisioning (Option B) ensures that large, interdependent workloads receive all required resources simultaneously, preventing resource starvation and guaranteeing a synchronized start for distributed training."

The atomicity-vs-availability trade.

The Best-Effort flow

Create a flex-start pool without queued provisioning. Submit a normal pod with nvidia.com/gpu: 8. Cluster autoscaler sees the unschedulable pod, sees the pool can grow, calls Compute Engine with provisioningModel: FLEX_START, waits for capacity, brings up the node. The pod schedules on the new node. This works exactly like normal autoscaling, just on flex-start pricing.

flex-besteffort-pool \
    --region us-central1 --cluster my-cluster \
    --node-locations us-central1-a \
    --accelerator type=nvidia-h100-80gb,count=8,gpu-driver-version=latest \
    --machine-type a3-highgpu-8g \
    --enable-autoscaling --num-nodes 0 --min-nodes 0 --max-nodes 10 \
    --flex-start # <-- only this flag

The Queued flow

Create a flex-start pool with queued provisioning. The pool starts at 0 nodes and can only grow via ProvisioningRequest. Submit a ProvisioningRequest with provisioningClassName: queued-provisioning.gke.io describing the pods that will eventually run. Wait for Provisioned=true. Submit the actual pods with consume-provisioning-request annotation. They schedule onto the new nodes.

flex-queued-pool \
    --region us-central1 --cluster my-cluster \
    --node-locations us-central1-a \
    --accelerator type=nvidia-h100-80gb,count=8,gpu-driver-version=latest \
    --machine-type a3-highgpu-8g \
    --enable-autoscaling --num-nodes 0 --min-nodes 0 --max-nodes 10 \
    --flex-start \
    --enable-queued-provisioning \
    --reservation-affinity=none

Note --reservation-affinity=none: per the Ray gang-scheduling tutorial: "You need to use the --reservation-affinity=none flag because GKE doesn't support Node Reservations with ProvisioningRequest." The flag is required.

The 8-GPU Metaflow question — which one?

For @resources(gpu=8) on a single Metaflow step, you have one pod requesting one node. Both modes work:

  • Best-Effort. Submit the pod; the autoscaler does its thing; pod runs. Simpler. No ProvisioningRequest to manage.
  • Queued. Wrap the pod in a ProvisioningRequest. Wait. Then submit the pod. More overhead, but you get the explicit capacity-acquisition state visible as a CRD — easier to monitor, easier to cancel cleanly.

For multi-step Metaflow workflows where multiple steps run concurrently — e.g., train and train_baseline as parallel branches each requesting gpu=8 — the choice matters more. With Best-Effort, the two pods race for capacity independently; one may succeed while the other waits. With Queued and Kueue, you can submit them as a workload group with all-or-nothing admission.

When Queued becomes mandatory — distributed training

If your workload is genuinely multi-node — multiple pods that must come up together for NCCL to work — Queued provisioning becomes the only sane option. Per the DWS technical guide: "For tightly coupled distributed workloads like large-scale model training, all worker pods must be scheduled simultaneously to establish communication and begin work. The node-by-node approach is insufficient here. GKE's Queued Provisioning mode solves this by implementing 'gang scheduling'. This mode requires creating a node pool with both the --flex-start and --enable-queued-provisioning flags. It is designed to work with a Kubernetes-native job queueing system like Kueue. The workflow is as follows: a user submits a Job (or other workload type like a RayJob) in a suspended state. Kueue intercepts this and creates a single, atomic ProvisioningRequest for all the nodes required by the job. DWS then waits until it can fulfill the entire request at once. Only when all nodes are provisioned and ready does Kueue 'unsuspend' the job, allowing the pods to be scheduled simultaneously onto the new nodes."

Metaflow's @resources(gpu=8) on a single pod is single-node — all 8 GPUs in one pod on one node. So you don't technically need Queued. But if your Metaflow flow has multiple parallel branches each requesting 8 GPUs, and you want them admitted atomically, Queued + Kueue is the right shape.

The pod template label, again

Per the GKE docs: "GKE might apply validations and mutations to Pods during their creation. The cloud.google.com/apply-warden-policies label allows GKE to apply the same validations and mutations to PodTemplate objects. This label is necessary for GKE to calculate node resource requirements for your Pods." Reminder — the PodTemplate referenced by a ProvisioningRequest needs cloud.google.com/apply-warden-policies: "true" as a metadata label or the request fails silently.

What goes into the workload pod for either mode

Best-Effort: just the normal pod spec. The pool's --flex-start flag triggers flex-start pricing automatically. You can also explicitly request flex-start via the cloud.google.com/gke-flex-start: "true" node selector.

Queued: pod must carry the consume-provisioning-request annotations. Plus the same node selector for the flex-start pool.

metadata:
  annotations:
    cluster-autoscaler.kubernetes.io/consume-provisioning-request: "train-8gpu-req"
    cluster-autoscaler.kubernetes.io/provisioning-class-name: "queued-provisioning.gke.io"
spec:
  nodeSelector:
    cloud.google.com/gke-nodepool: flex-queued-pool
    cloud.google.com/gke-flex-start: "true"
  tolerations:
    - key: "nvidia.com/gpu"
      operator: "Exists"
      effect: "NoSchedule"
The pragmatic Metaflow recommendation

For Metaflow workflows that run one or two parallel 8-GPU steps and value operational simplicity, start with Best-Effort flex-start. For workflows where multiple 8-GPU steps must succeed together (data-parallel branches that aggregate results), or where partial failure is unacceptable, invest in Queued + Kueue. The migration from Best-Effort to Queued is "add a ProvisioningRequest in front, add two annotations to the pod" — not a re-architecture.

CLS Chapter VI

Custom ComputeClass · the CRD, walked.

The CRD A Kubernetes resource that defines a prioritized list of node configurations. The autoscaler walks the list top-to-bottom looking for capacity.

The available fields Machine family, GPU type/count/topology, spot, reservations, flex-start, storage, locations, max runtime, sharing strategies.

A Custom ComputeClass is a Kubernetes CRD — cloud.google.com/v1 ComputeClass — that lets platform teams declare a prioritized list of node configurations and let the GKE cluster autoscaler walk the list when a workload needs nodes. The first priority is tried; if capacity is unavailable, the autoscaler falls through to the second; and so on. For GPU workloads, this turns "I need an H200" into a declarative search across multiple consumption strategies, reservation pools, and even GPU types — all without changing the workload manifest. Per the GKE ComputeClass concepts page: "Custom ComputeClasses are available to use in GKE Autopilot mode and GKE Standard mode in version 1.30.3-gke.1451000 and later, and offer a declarative approach to defining node attributes and autoscaling priorities."

The mental model

Think of a ComputeClass as a search policy for the autoscaler. The workload says "schedule me with compute class h200-training." When the workload can't be scheduled on existing nodes, the autoscaler consults the ComputeClass and tries to provision nodes matching priority #1, then #2, then #3 — until one succeeds, or the whenUnsatisfiable policy fires. Per the same docs: "Fallback compute priorities: Define a hierarchy of node configurations in each ComputeClass for GKE to prioritize. If the most preferred configuration is unavailable, GKE automatically chooses the next configuration in the hierarchy. This fallback model helps ensure that even when compute resources are unavailable, your workloads still run on optimized hardware with minimal scheduling delays."

The CRD shape, walked

Per the ComputeClass CRD reference, the top-level fields:

apiVersion: cloud.google.com/v1
kind: ComputeClass
metadata:
  name: h200-training
spec:
  # Move workloads to higher-priority nodes as they become available.
  activeMigration:
    optimizeRulePriority: false

  # Autoscaler tuning
  autoscalingPolicy:
    consolidationDelayMinutes: 10
    consolidationThreshold: 70
    gpuConsolidationThreshold: 100

  # Automatically create node pools that match priorities.
  nodePoolAutoCreation:
    enabled: true

  # Cluster-wide defaults applied to every priority unless overridden.
  priorityDefaults:
    location:
      zones: ["us-central1-a", "us-central1-b"]
      locationPolicy: ANY
    storage:
      bootDiskType: hyperdisk-balanced
      bootDiskSize: 200

  # The fallback list — walked top-to-bottom.
  priorities:
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8, driverVersion: latest }
      reservations: { affinity: AnyReservation }
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8, driverVersion: latest }
      flexStart: { enabled: true }
    - machineType: a3-megagpu-8g  # fallback to H100 Mega
      gpu: { type: nvidia-h100-mega-80gb, count: 8, driverVersion: latest }
      flexStart: { enabled: true }
    - machineType: a3-highgpu-8g  # further fallback to base H100
      gpu: { type: nvidia-h100-80gb, count: 8, driverVersion: latest }
      spot: true

  # What to do if nothing in the priorities list works.
  whenUnsatisfiable: DoNotScaleUp

The gpuSharing sub-block

From the CRD reference: gpuSharing: { sharingStrategy: string, maxSharedClientsPerGPU: integer, gpuPartitionSize: string }. Strategies are mig (Multi-Instance GPU, hardware partition), time-sharing (NVIDIA time-slicing, software multiplexing), mps (NVIDIA Multi-Process Service). For 8-GPU training, sharing is irrelevant — you want the whole GPU. For multi-tenant inference clusters, MIG slicing of an H100 into 7 partitions per GPU lets one node serve many lightweight inference replicas.

Where the CRD doesn't influence things — kube-scheduler

A crucial subtlety per the docs: "Custom ComputeClasses influence autoscaling decisions but are not considered by kube-scheduler. During Pod scheduling, the scheduler might not prioritize nodes with higher custom ComputeClass priorities, even when existing nodes are available across various priorities." ComputeClass is an autoscaler policy, not a scheduler policy. Once nodes exist, kube-scheduler picks among them based on its own rules — it doesn't know which priority a node came from. Practical implication: if you have nodes from priority #2 sitting idle and priority #1 capacity becomes available, kube-scheduler won't migrate workloads to priority #1 unless activeMigration.optimizeRulePriority: true is set, which makes the autoscaler proactively drain lower-priority nodes once higher-priority ones exist.

The whenUnsatisfiable choice

Two values: DoNotScaleUp (if nothing in the priority list works, leave the pod Pending) or ScaleUpAnyway (fall back to default autoscaler behavior, picking any node that satisfies the pod's other constraints). For GPU workloads with specific accelerator requirements, DoNotScaleUp is the right answer — you don't want the autoscaler creating CPU-only nodes that can't satisfy the request anyway. For mixed workloads where any node will do, ScaleUpAnyway is more permissive.

How a workload selects a ComputeClass

Two mechanisms:

  • Node selector. The workload sets cloud.google.com/compute-class: h200-training in spec.nodeSelector. The autoscaler matches.
  • Default class. Set cloud.google.com/default-compute-class at the namespace level; all unselected workloads in that namespace use it.
apiVersion: v1
kind: Pod
spec:
  nodeSelector:
    cloud.google.com/compute-class: h200-training
  tolerations:
    - key: "nvidia.com/gpu"
      operator: "Exists"
      effect: "NoSchedule"
  containers:
    - name: train
      image: ...
      resources: { limits: { nvidia.com/gpu: 8 } }

The platform-engineer / app-developer split

The ComputeClass design encourages a clean separation. Platform engineers author and maintain the ComputeClass CRDs — they own the consumption strategy, fallback chains, reservation references, and cost discipline. Application developers (data scientists writing Metaflow flows) just say "give me the h200-training compute class" without knowing whether that class fronts reservations, flex-start, spot, or a mix. The CRD is reviewable in git, version-controlled, deployable via Terraform. Changes to the consumption strategy don't require touching workload code.

CLS Chapter VII

Priority patterns · obtainability, cost, hybrid.

Three archetypes Obtainability (widest net for scarce hardware). Cost-optimized (cheapest first, on-demand last). Hybrid (best-of-both).

The pattern source Google's accelerated-platforms repo + Searce's blog — battle-tested templates.

A ComputeClass with one priority is just a node selector. A ComputeClass with a thoughtfully ordered priority list is a capacity strategy encoded as Kubernetes config. Three archetypes show up repeatedly in production: obtainability-first (we need this GPU, cost is secondary, throw the widest net to find it), cost-optimized (every dollar matters, start with spot, fall back through on-demand), and hybrid (specific reservation first for guaranteed capacity, then flex-start for opportunistic, then spot for cost). This chapter walks each pattern with full manifests and the rationale for the order. Per the GoogleCloudPlatform/accelerated-platforms repo: "You are not limited to choosing between pure obtainability or maximum optimization. Many workloads benefit from a blended strategy that finds a middle ground."

Pattern 1 — Obtainability-first

The scenario: a critical training run that absolutely needs H200s; cost is secondary; the goal is to get capacity by any reasonable means. Per the accelerated-platforms repo: "This profile is ideal for critical jobs or other workloads where securing a specific accelerator is the top priority, and cost is a secondary concern. The goal is to create the widest possible net to catch an available resource. The priority list is structured to try the most reliable long term options first, falling back to more ephemeral options. Here, we want to acquire an nvidia-h200-141gb GPU."

apiVersion: cloud.google.com/v1
kind: ComputeClass
metadata:
  name: h200-obtainability
spec:
  nodePoolAutoCreation: { enabled: true }
  activeMigration: { optimizeRulePriority: true }
  priorities:
    # 1. Try the reservation first — guaranteed if available.
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8, driverVersion: latest }
      reservations:
        affinity: SpecificReservations
        specific:
          - name: h200-reservation
            project: my-project

    # 2. DWS Flex-start — opportunistic, 53% discount, max 7-day runtime.
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8, driverVersion: latest }
      flexStart: { enabled: true }
      maxRunDurationSeconds: 604800

    # 3. Spot — cheap, preemptible. Last resort for cost-optimization too.
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8, driverVersion: latest }
      spot: true

    # 4. On-demand — full price, but always available if quota allows.
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8, driverVersion: latest }

  whenUnsatisfiable: DoNotScaleUp  # H200 only. No CPU-fallback.

Each priority is the same physical hardware (H200), differing only in consumption strategy. The autoscaler tries the reservation; if exhausted, tries DWS flex-start; if no flex-start capacity, tries spot; if no spot capacity, tries on-demand. Four distinct GCP capacity pools, walked top-to-bottom.

Pattern 1a — Obtainability across GPU types

A bolder variant. If you need an H200 but would accept an H100 Mega or H100 in a pinch, expand the search across GPU types:

spec:
  priorities:
    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8 }
      reservations: { affinity: AnyReservation }

    - machineType: a3-ultragpu-8g
      gpu: { type: nvidia-h200-141gb, count: 8 }
      flexStart: { enabled: true }

    # H100 Mega — fewer GB/GPU, faster interconnect
    - machineType: a3-megagpu-8g
      gpu: { type: nvidia-h100-mega-80gb, count: 8 }
      flexStart: { enabled: true }

    # H100 base — most-available variant of H-series
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      flexStart: { enabled: true }

    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      spot: true

Caveat: the workload must tolerate the GPU memory difference. H200 has 141GB per GPU; H100 has 80GB. A model that needs 100GB of GPU memory won't fit on H100 — adding it as a fallback wastes the autoscaler's time. Make sure the fallbacks are actually compatible with the workload.

Pattern 2 — Cost-optimized

The scenario: a fault-tolerant training job or an evaluation harness where preemption is acceptable; minimize cost. Per Searce's GKE Compute Classes guide, the cost-optimized pattern leads with Spot and falls back to on-demand:

apiVersion: cloud.google.com/v1
kind: ComputeClass
metadata:
  name: cost-optimized-h100
spec:
  activeMigration: { optimizeRulePriority: true }
  nodePoolAutoCreation: { enabled: true }
  autoscalingPolicy:
    consolidationDelayMinutes: 2
    consolidationThreshold: 70

  priorities:
    # 1. Spot H100 — cheapest. Workload must tolerate preemption.
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      spot: true

    # 2. Flex-start — 53% discount, non-preemptible once started.
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      flexStart: { enabled: true }

    # 3. On-demand — full price, only as last resort.
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }

The economics: Spot is roughly 60-70% off on-demand pricing; Flex-start is up to 53% off; on-demand is full price. Walking the list in this order produces the minimum cost subject to capacity. The activeMigration.optimizeRulePriority: true setting means when Spot capacity returns, workloads get moved back from on-demand/flex-start.

Pattern 3 — Hybrid with specific reservations

The most common production shape. Use a specific reservation as the foundation (you've committed to it; consume it first); fall back to flex-start (53% discount); ultimate fallback to on-demand. Per Searce's specific-reservations pattern:

apiVersion: cloud.google.com/v1
kind: ComputeClass
metadata:
  name: shared-specific-reservations
spec:
  priorities:
    # 1. Consume the named shared reservation in another project.
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      reservations:
        affinity: Specific
        specific:
          - name: h100-shared-reservation
            project: reservation-project

    # 2. Spot fallback within the same machine family.
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      spot: true

    # 3. On-demand fallback.
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }

  whenUnsatisfiable: DoNotScaleUp

Pattern 4 — Multi-zone obtainability

GPU availability varies by zone. The same H200 capacity that's exhausted in us-central1-a might be available in us-central1-b. A regional ComputeClass with multiple zones in location.zones and locationPolicy: ANY lets the autoscaler search across zones:

spec:
  priorityDefaults:
    location:
      zones: ["us-central1-a", "us-central1-b", "us-central1-c"]
      locationPolicy: ANY
  priorities:
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      reservations: { affinity: AnyReservation }
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8 }
      flexStart: { enabled: true }

For DWS Flex-start, the docs note that "Clusters running GKE version 1.29.1-gke.1708000 and later optimize zone selection for lower wait times. Earlier versions might experience longer queueing." The newer GKE versions search zones intelligently; older ones don't.

The fallback timing

Per the ComputeClass troubleshooting docs: "Custom ComputeClasses use prioritized fallback logic. If a Pod isn't scheduling on nodes matching the highest-priority rule, it's often due to constraints like resource unavailability or project quotas. When GKE can't provision nodes matching a specific priority rule, for example, due to a ZONE_RESOURCE_POOL_EXHAUSTED or QUOTA_EXCEEDED error from Compute Engine, the cluster autoscaler immediately tries the next rule in the priorities list. There is no waiting period before GKE falls back to the next priority, except when using TPUs or the Flex Start provisioning model, which support a configurable delay."

The implication: ZONE_RESOURCE_POOL_EXHAUSTED and QUOTA_EXCEEDED trigger instant fallback. For DWS Flex-start (which can take time), you can configure a capacityCheckWaitTimeSeconds per priority — wait this long before deciding flex-start can't satisfy and falling through.

Pattern for Metaflow per-step variation

Different Metaflow steps in the same flow can target different ComputeClasses. The preprocess step uses cheap CPU; train uses H200 with the obtainability class; evaluate uses H100 with the cost-optimized class:

class TrainFlow(FlowSpec):
    @kubernetes(cpu=8, memory=32000)
    @step
    def preprocess(self):
        ...

    @kubernetes(node_selector={"cloud.google.com/compute-class": "h200-obtainability"})
    @resources(gpu=8, cpu=200, memory=1800000)
    @step
    def train(self):
        ...

    @kubernetes(node_selector={"cloud.google.com/compute-class": "cost-optimized-h100"})
    @resources(gpu=8, cpu=200, memory=1800000)
    @step
    def evaluate(self):
        ...
The composition rule

Three or four priorities is usually the right count. Two is too narrow (one fallback). Eight is too many (autoscaler walks all of them per request, adding latency, and most never trigger). The standard pattern: reservation → flex-start → spot → on-demand (four). Drop any that don't apply to your workload. Keep the order consistent across ComputeClasses so platform users can reason about behavior without re-reading the YAML.

CLS Chapter VIII

ComputeClass + DWS · the integration.

The bridge ComputeClass.priorities[].flexStart.enabled = true tells the autoscaler to provision via DWS Flex-start for that priority.

What you don't manage The ProvisioningRequest CRD itself. The ComputeClass machinery wires it under the hood when nodePoolAutoCreation is on.

ComputeClass and DWS Flex-start were designed to compose. A ComputeClass priority can carry flexStart: { enabled: true }; when the autoscaler picks that priority, it provisions a flex-start node pool (or scales an existing one) without you needing to manually author a ProvisioningRequest. The result is a single declarative artifact — the ComputeClass YAML — that encodes "for this workload, try reservation first, then flex-start, then spot, then on-demand" — and the autoscaler turns that into the right consumption-API calls. This chapter walks the integration, the node-pool side of the picture, and the operational tradeoffs vs. managing ProvisioningRequests explicitly.

What flexStart.enabled actually triggers

When a ComputeClass priority has flexStart: { enabled: true }, two things change in autoscaler behavior:

  1. Node provisioning calls the DWS API with provisioningModel: FLEX_START rather than standard on-demand. The 53% discount kicks in automatically.
  2. The autoscaler can tolerate longer wait times at this priority (configurable via capacityCheckWaitTimeSeconds) before falling through to the next priority.

Crucially, this is best-effort flex-start, not queued flex-start. For queued (all-or-nothing for distributed training), you still need an explicit ProvisioningRequest + Kueue. For single-pod 8-GPU workloads, ComputeClass-driven flex-start is sufficient.

The two integration shapes

Two ways to combine ComputeClass with DWS:

  • Shape A — ComputeClass drives Flex-start. ComputeClass priorities include flexStart.enabled: true. Autoscaler manages everything. Workload just selects the ComputeClass. Best for single-pod workloads.
  • Shape B — ComputeClass selects pool, ProvisioningRequest manages capacity. ComputeClass priorities use nodepools: ["flex-queued-pool"] to target a pre-created queued-provisioning pool. Workload also creates a ProvisioningRequest (via Kueue) for atomic admission. Best for multi-pod gang-scheduled workloads.

Shape A — ComputeClass-driven Flex-start, end to end

apiVersion: cloud.google.com/v1
kind: ComputeClass
metadata:
  name: h100-flex-cascade
spec:
  nodePoolAutoCreation: { enabled: true }

  priorities:
    # 1. Try a specific reservation (instant if available).
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8, driverVersion: latest }
      reservations:
        affinity: SpecificReservations
        specific:
          - name: my-h100-reservation
            project: my-project
      capacityCheckWaitTimeSeconds: 30  # short — reservation succeeds fast or not

    # 2. Fall back to DWS Flex-start (wait up to 20 min for capacity).
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8, driverVersion: latest }
      flexStart: { enabled: true }
      maxRunDurationSeconds: 604800  # 7 days max
      capacityCheckWaitTimeSeconds: 1200  # 20-minute wait window

    # 3. Spot (also subject to availability, but typically faster).
    - machineType: a3-highgpu-8g
      gpu: { type: nvidia-h100-80gb, count: 8, driverVersion: latest }
      spot: true

  whenUnsatisfiable: DoNotScaleUp

The workload submits a normal pod with node_selector={"cloud.google.com/compute-class": "h100-flex-cascade"}. Done. No ProvisioningRequest YAML to author. The autoscaler walks the priorities, calling the right consumption API for each tier.

Shape B — Queued pool + manual ProvisioningRequest

For multi-pod gang scheduling. The ComputeClass references a pre-existing queued-provisioning node pool (created separately with --enable-queued-provisioning):

apiVersion: cloud.google.com/v1
kind: ComputeClass
metadata:
  name: h100-queued
spec:
  priorities:
    - nodepools: ["flex-queued-h100-pool"]   # pre-created with --enable-queued-provisioning
      gpu: { type: nvidia-h100-80gb, count: 8 }
  whenUnsatisfiable: DoNotScaleUp

The workload then carries the consume-provisioning-request annotations referencing a ProvisioningRequest that was created externally (e.g., by Kueue via AdmissionCheck). Two control planes — the ComputeClass picks the pool; the ProvisioningRequest gates atomic admission.

Per-priority capacityCheckWaitTimeSeconds tuning

Reservation lookups complete in seconds (either the reservation exists and has capacity, or it doesn't). Flex-start can take minutes to hours. Spot is usually fast (either spot capacity exists in the zone or it doesn't). The wait-time settings should reflect this. Per the ComputeClass troubleshooting docs, the default behavior is no waiting between priorities except for TPU and Flex-start. Tuning per-priority lets you say "try reservation for 30 seconds, then try flex-start for 20 minutes, then fall through to spot."

activeMigration — moving back to higher priorities

Per the GKE ComputeClass docs: "Active migration to higher priority nodes: when configured, GKE automatically replaces nodes that are lower on your list of fallback priorities with nodes that are higher on that list over time. As a result, your Pods eventually run on your most preferred nodes in a ComputeClass, even if that hardware wasn't available when you created the workload."

The setting: spec.activeMigration.optimizeRulePriority: true. With it on, if your workload landed on priority #3 (spot) because #1 (reservation) and #2 (flex-start) were exhausted, and later #1 capacity becomes available, the autoscaler proactively moves the workload back to priority #1. The trade-off: migration is disruptive — the pod restarts on the new node. For training, set safe-to-evict: false to prevent this in the middle of a run. For inference, migration during off-peak windows is often acceptable.

The node-pool auto-creation question

Two modes for how nodes actually come into existence:

  • nodePoolAutoCreation: enabled. The ComputeClass creates node pools on demand matching each priority. No manual pool creation needed. Pools are deleted when scaled to zero.
  • nodePoolAutoCreation: disabled, with nodepools[] in each priority. Pools are created and managed separately (often via Terraform); ComputeClass priorities reference them by name.

For GKE Autopilot, auto-creation is the only option. For Standard, both work. The Terraform-managed approach (Issue 13 territory) tends to be safer for production — you control pool names, sizes, and labels explicitly; the ComputeClass becomes a pure routing policy.

The autoscaling consolidation knobs

Per the CRD reference, autoscalingPolicy exposes three thresholds:

  • consolidationDelayMinutes. How long an under-utilized node sits idle before the autoscaler considers consolidating workloads off it. Default ~10 min. Lower it (1-2 min) for fast-cycling batch jobs; raise it (15-30 min) to absorb traffic spikes without reprovisioning.
  • consolidationThreshold. The CPU utilization below which a node is considered for consolidation. Default 50%. For GPU workloads, this is usually overridden by the GPU-specific threshold.
  • gpuConsolidationThreshold. The GPU utilization threshold. Default behavior consolidates nodes whose GPUs are below this percentage. For Metaflow training runs that fully utilize the GPU when active, set this to a low value (e.g., 30%) so brief gradient-accumulation pauses don't trigger consolidation.

The 8-GPU Metaflow recommendation, end of section

For a single-step @resources(gpu=8) Metaflow flow that needs H100 or H200:

  1. Author a ComputeClass (say h100-flex-cascade) with priorities: reservation → flex-start → spot → on-demand. Set flexStart.enabled: true on the flex-start priority.
  2. Set nodePoolAutoCreation.enabled: true if you trust GKE to create pools, OR pre-create pools via Terraform and reference them by name.
  3. In Metaflow: @kubernetes(node_selector={"cloud.google.com/compute-class": "h100-flex-cascade"}, tolerations=[...]).
  4. Done. The autoscaler handles capacity; the Metaflow step runs when nodes are ready.
The composition payoff

The win of ComputeClass + DWS together: capacity strategy lives in one YAML file, owned by the platform team, version-controlled, applied per environment. Workload teams (Metaflow flow authors) reference it by name. Migrating from "spot only" to "reservation-first" is editing one ComputeClass — every workload picks up the change on next scale-up. No Metaflow code changes; no per-flow consumption logic; no ProvisioningRequest YAML to manage. This is the GKE-native way to encode AI infrastructure consumption policy.

GCS Fuse CSI · gke-gcsfuse/volumes.

The pattern Mount a GCS bucket as a POSIX filesystem via the gcsfuse sidecar. No application code changes.

The annotation gke-gcsfuse/volumes: "true" on the pod. Workload Identity for auth. The CSI driver does the rest.

Training data lives in Google Cloud Storage. Training scripts expect POSIX file IO. The GCS Fuse CSI driver bridges the two — mount a bucket inside the pod via a sidecar container that translates filesystem calls into GCS API requests. Per the GKE GCS Fuse docs: "The Cloud Storage FUSE CSI driver lets you use standard file system semantics to mount and access Cloud Storage buckets. This provides a familiar interface that improves portability for ML workloads, and avoids the need for you to make extensive code or application changes. The driver is supported on all accelerators available on GKE including GPUs and TPUs." For Metaflow steps that read training shards or write checkpoints, this is the path of least resistance — no S3-API code, no manual gsutil invocations, just files at a mount path.

What gke-gcsfuse/volumes actually does

The annotation gke-gcsfuse/volumes: "true" on a pod tells GKE to inject the gcsfuse sidecar alongside your main containers. Per the same docs: "The Cloud Storage FUSE CSI driver attaches a sidecar container in your Pods to manage interactions with Cloud Storage. The sidecar handles the mounting and interaction with Cloud Storage, allowing your applications to seamlessly access data." When the main container starts, the bucket is already mounted at the specified path; the container reads/writes files normally; the sidecar translates each call.

Three pieces compose:

  1. The CSI driver. Cluster-level addon, enabled with --addons GcsFuseCsiDriver. Per the setup docs: gcloud container clusters create CLUSTER_NAME --addons GcsFuseCsiDriver --workload-pool=PROJECT_ID.svc.id.goog.
  2. The pod annotation. gke-gcsfuse/volumes: "true" opts the pod in.
  3. Workload Identity auth. The pod's KSA bound to a GSA with bucket access. Required for the sidecar to authenticate.

The minimal pod spec

apiVersion: v1
kind: Pod
metadata:
  name: train-pod
  annotations:
    gke-gcsfuse/volumes: "true"                  # enable sidecar injection
    gke-gcsfuse/ephemeral-storage-limit: "200Gi"    # size the cache disk
spec:
  serviceAccountName: train-ksa  # bound to GSA via Workload Identity
  containers:
    - name: train
      image: us-central1-docker.pkg.dev/PROJECT/repo/train:latest
      resources: { limits: { nvidia.com/gpu: 8 } }
      volumeMounts:
        - { name: training-data, mountPath: /mnt/data, readOnly: true }
        - { name: checkpoints, mountPath: /mnt/checkpoints }
  volumes:
    - name: training-data
      csi:
        driver: gcsfuse.csi.storage.gke.io
        readOnly: true
        volumeAttributes:
          bucketName: my-training-data
          mountOptions: "implicit-dirs,file-cache:max-size-mb:32768"
    - name: checkpoints
      csi:
        driver: gcsfuse.csi.storage.gke.io
        volumeAttributes:
          bucketName: my-checkpoints

Two buckets mounted — one read-only for training data, one read-write for checkpoints. The training script reads /mnt/data/shards/* and writes /mnt/checkpoints/step-1000.pt as if they were local files; the sidecar translates each call into GCS API operations.

Workload Identity — the auth layer

The CSI driver requires Workload Identity Federation for GKE. The pod's Kubernetes Service Account (KSA) is bound to a Google Service Account (GSA) that holds the GCS permissions. The KSA gets a short-lived token via the metadata server; the sidecar uses it to authenticate.

$ gcloud iam service-accounts create train-gsa --project=PROJECT_ID

# 2. Grant bucket access
$ gcloud storage buckets add-iam-policy-binding gs://my-training-data \

The PV/PVC alternative

For buckets used across many workloads, define a PV/PVC pair once and reference the PVC:

kind: PersistentVolume
metadata:
  name: training-data-pv
spec:
  accessModes: [ReadOnlyMany]
  capacity: { storage: 1Pi }
  storageClassName: ""
  claimRef: { namespace: training, name: training-data-pvc }
  csi:
    driver: gcsfuse.csi.storage.gke.io
    volumeHandle: my-training-data
    volumeAttributes:
      bucketName: my-training-data
      mountOptions: "implicit-dirs,file-cache:max-size-mb:32768"
---

kind: PersistentVolumeClaim
metadata: { name: training-data-pvc, namespace: training }
spec:
  accessModes: [ReadOnlyMany]
  resources: { requests: { storage: 1Pi } }
  storageClassName: ""
  volumeName: training-data-pv

The pod then references the PVC: volumes: [{name: data, persistentVolumeClaim: {claimName: training-data-pvc}}]. The PV is declared once per cluster; workloads reference the PVC. This is the shape Metaflow's persistent_volume_claims parameter takes.

Read vs write semantics

GCS Fuse is not POSIX — it's a translation layer with important differences:

  • Rename is not atomic — implemented as copy + delete. Don't rely on rename for crash safety.
  • Writes are not visible until file close — other readers see old contents while a write is in progress.
  • Object listing is eventually consistent — a file just written may not appear in ls immediately.
  • No file locking — multiple writers to the same key produce last-writer-wins.
  • Read-after-write IS strongly consistent — a file you wrote can be read back immediately by name.

For training, the common patterns work well: read shards in a fixed order; write checkpoints with unique names. For workloads that lock files or rely on atomic rename, you need a different primitive (Filestore, Lustre, or Hyperdisk).

The accelerator-agnostic claim

Per the GCS Fuse docs: "The driver is supported on all accelerators available on GKE including GPUs and TPUs." The same data-access pattern works for H100/H200/B200 GPU workloads and TPU workloads. A Metaflow flow ported across accelerator families doesn't need data-loading rewrites — only the resource decorator changes.

Manual file-cache tuning

When you need finer control than profiles provide, the manual knobs per the perf tuning docs:

  • file-cache:max-size-mb or volume attribute fileCacheCapacity. Maximum cache size. Default is unset (no caching). Per the docs: "You must opt-in to enable file caching with the Cloud Storage FUSE CSI driver.   • Cache medium. Per the same source: "GKE uses an emptyDir volume by default for Cloud Storage FUSE file caching backed by the ephemeral storage configured on the node.   • RAM disk option for fastest reads: "To use a RAM disk with the Cloud Storage FUSE CSI driver, add the following to your manifest: volumes: - name: gke-gcsfuse-cache emptyDir: medium: Memory"

The ephemeral-storage-limit annotation

The metadata cache

Per the same docs: "The Cloud Storage FUSE CSI driver enhances performance by caching file metadata, like size and modification time. The CSI driver enables this stat cache by default and reduces latency by storing information locally instead of repeatedly requesting it from Cloud Storage. You can configure its maximum size (the default is 32 MB) and how long the data stays in the cache (the default is 60 seconds). By fine-tuning the metadata cache, you can reduce API calls to Cloud Storage, to improve application performance and efficiency by minimizing network traffic and latency."

Default 32MB / 60s TTL is fine for most workloads. For very large directory trees (millions of files), bump the size; for workloads sensitive to staleness, lower the TTL.

Metadata prefetch — the cold-start win

Per the AI/ML optimization guide: "The metadata prefetch feature lets the Cloud Storage FUSE CSI driver proactively load relevant metadata about the objects in your Cloud Storage bucket into Cloud Storage FUSE caches. This approach reduces calls to Cloud Storage and it's especially beneficial for applications that access large datasets that have many files, such as AI and ML training workloads. To pre-populate the metadata cache, enable metadata prefetch for the given volume. Set the volume attribute gcsfuseMetadataPrefetchOnMount to true. To avoid workload interruptions, consider increasing the memory limit of the gke-gcsfuse-metadata-prefetch sidecar by configuring the sidecar resources. Enable list caching: This feature optimizes listing directories and files. It's particularly beneficial for AI and ML training workloads, which often involve repeatedly accessing and listing entire directories."

volumeAttributes:
  bucketName: my-training-data
  gcsfuseMetadataPrefetchOnMount: "true"     # eager load all metadata
  mountOptions: |
    implicit-dirs
    file-cache:max-size-mb:180000
    metadata-cache:enable-list-cache:true
    metadata-cache:ttl-secs:600
    parallel-downloads-per-file:8

Trade-off: metadata prefetch makes the first ls on a large tree slow (walks the whole tree at mount time), but every subsequent ls is instant. For training jobs that scan the dataset directory at startup, prefetch pays for itself in the first epoch.

Parallel downloads

The 2026 win for large files. Mount option parallel-downloads-per-file:N splits downloads of files larger than a threshold into N parallel HTTP requests, dramatically improving single-file throughput. For a 500MB training shard, this is the difference between 8 MB/s (single-threaded) and 80 MB/s (10 parallel streams).

The Rapid Cache option for serving

The gcsfusecsi-serving profile automatically enables Rapid Cache — a special-purpose optimization for the "load weights once at startup, serve forever" pattern. Rapid Cache reads aggressively into local memory at mount time. For inference deployments where the cold start is dominated by weight loading, this can cut startup latency dramatically.

The sidecar resource sizing

The gcsfuse sidecar runs in the pod and consumes its own CPU and memory. Default limits are conservative. For high-throughput training reading hundreds of MB/s, bump them via pod annotations:

  • gke-gcsfuse/cpu-limit: "8" — sidecar CPU.
  • gke-gcsfuse/memory-limit: "32Gi" — sidecar memory.
  • gke-gcsfuse/ephemeral-storage-limit: "200Gi" — sidecar's emptyDir.

Under-sized sidecars become the bottleneck — the training script reads slowly even though the network has capacity. Per Saikat Roychowdhury's deep dive on GCS Fuse scale: "Beyond just the control plane optimizations of the CSI driver, there have been several improvements made to the Cloud storage fuse implementation to address the data plane bottlenecks and the IO patterns seen by such workloads (which deserves a post on its own!). These optimizations collectively enhance the driver's ability to handle large-scale AI/ML workloads efficiently and reliably."

The tuning recipe for 8-GPU training

The pragmatic default for a Metaflow @resources(gpu=8) training step with GCS-resident data: (1) apply the gcsfusecsi-training profile if available, otherwise (2) set fileCacheCapacity to roughly the size of one epoch's shards (so the cache hits during epochs 2+), (3) set gcsfuseMetadataPrefetchOnMount: "true" for fast ls on dataset directories, (4) bump sidecar memory to 32Gi and CPU to 8 — A3 nodes have plenty to spare, and (5) use parallel-downloads-per-file:8 for large shards. First-epoch slowdown is unavoidable (the cache is empty); subsequent epochs are GPU-bound.

What Hyperdisk ML actually is

Block storage — not a file system, not an object store. A Hyperdisk ML volume is a network block device with provisioned throughput; mount it on one or many VMs; read like a local SSD. The write path is "hydrate once" — copy data from GCS to the disk on a single VM in read-write mode, detach, then re-attach in READ_ONLY_MANY mode to any number of consumer VMs. The aggregate throughput scales with VM count up to enormous limits.

Per the scalable AI storage guide: "Hyperdisk ML is a network block storage option that's purpose-built for AI/ML workloads, particularly for accelerating the loading of static data like model weights. Unlike Cloud Storage FUSE, which provides a file system interface to an object store, Hyperdisk ML provides a high-performance block device that can be pre-loaded, or hydrated, with model artifacts from Cloud Storage. Its standout feature for inference serving is its support for READ_ONLY_MANY access, which allows a single Hyperdisk ML volume to be attached as a read-only device to up to 2,500 VMs concurrently."

The limits and caveats

Per the Hyperdisk overview docs: "Hyperdisk Extreme, Hyperdisk ML and Hyperdisk Throughput volumes can't be used as boot disks. You can attach a Hyperdisk ML volume to up to 100 VMs at most once every 30 seconds. You can't create a Hyperdisk ML disk in read-write mode from a snapshot or a disk image. You must create the disk in read-only mode. If you enable read-only mode for a Hyperdisk ML volume, you can't re-enable read-write mode."

Per the GKE Hyperdisk concepts: "Hyperdisk ML can't be used as boot disks. Hyperdisk ML can't be used in multi-writer mode. Hyperdisk ML doesn't support Storage Pools." Plus: no resizing in place; no snapshots from read-only volumes; can't switch back from RO to RW.

The implications: Hyperdisk ML is the right choice for immutable data — model weights, foundation checkpoints, evaluation datasets. It's the wrong choice for anything you want to update (use GCS Fuse RW for checkpoints) or anything that needs to be a boot disk.

The economics

Per the Hyperdisk choice guide: "Hyperdisk ML is specifically optimized for accelerating data load times for inference, training and HPC workloads — Hyperdisk ML accelerates model load time by 3-5x compared to common alternatives. Hyperdisk ML is particularly well-suited for serving tasks compared to other storage services on Google Cloud because it can concurrently provide to many VMs exceptionally high aggregate throughput (up to 1.2 TiB/s of aggregate throughput per volume, offering greater than 100x higher performance than competitive offerings). You write once (up to 64 TiB per disk) and attach multiple VM instances to the same volume in a read-only mode."

The billing model: per the Hyperdisk overview docs: "Hyperdisk ML charges a monthly rate based on the provisioned throughput (in MiB/s). The economics favor the "many consumers" pattern — the more pods that mount the same volume, the lower the per-pod cost.

STO Chapter XII

Secondary boot disks + image streaming.

Two complementary techniques Secondary boot disks pre-stage container images on disk. Image Streaming begins running containers before the image is fully pulled.

The win Up to 4× faster node start; minutes shaved off image pull for 10GB+ AI containers.

The third bottleneck after capacity and weight loading is image pull. Modern AI containers — vLLM, Triton, TGI, custom PyTorch with frozen libraries — are 8-15GB. Pulling that from Artifact Registry every time a node starts is wasteful and slow. Two GKE features attack this from different angles: secondary boot disks (pre-stage the image on a disk attached to the node at provision time), and GKE Image Streaming (start running the container before pull completes, lazy-loading layers on demand). Both predate the AI boom but are now critical AI infrastructure.

Secondary boot disks — pre-stage at the disk layer

Per the GKE preload docs: "Starting in GKE version 1.28.3-gke.1067000 in Standard clusters and in GKE version 1.30.1-gke.1329000 in Autopilot clusters, you can configure the node pool with secondary boot disks. You can tell GKE to provision the nodes and preload them with data, such as a machine learning model, or a container image. Using preloaded container images or data in a secondary disk has the following benefits for your workloads: Reduced latency when pulling large container images, or downloading data, Faster autoscaling, Quicker recovery from disruptions like maintenance events and system errors."

The mechanism: build a disk image containing your container images (or your data); reference it when creating the node pool; GKE attaches a Persistent Disk per node from that image. When pods start, the kubelet finds the images already on disk and skips the registry pull.

Per the same docs: "Secondary boot disks are Persistent Disks which are backed by distributed block storage. The Persistent Disk is instantiated from disk images that you create ahead of time. For scalability reasons, each node gets its own Persistent Disk instance created from the disk image. These Persistent Disk instances are deleted when the node is deleted. If the disk image is already in use in the zone, the creation time of all subsequent disks created from the same disk image will be lower. The secondary boot disk type is the same as the node boot disk. Adding secondary boot disks to your node pools does not increase the node provisioning time. GKE provisions secondary boot disks from a disk image in parallel with the node provisioning process."

The build pipeline

GKE provides a tool called gke-disk-image-builder that builds the disk image from a list of container images. A typical CI pipeline:

# 1. Build container images and push to Artifact Registry
$ docker build -t us-central1-docker.pkg.dev/PROJECT/repo/train:v1 .
$ docker push us-central1-docker.pkg.dev/PROJECT/repo/train:v1

# 2. Build a disk image preloaded with these container images
$ ./gke-disk-image-builder \
    --project-name=PROJECT \
    --image-name=train-images-v1 \
    --container-image=us-central1-docker.pkg.dev/PROJECT/repo/train:v1 \
    --container-image=us-central1-docker.pkg.dev/PROJECT/repo/triton:24.10 \
    --zone=us-central1-a

# 3. Create node pool referencing this disk image
$ gcloud container node-pools create train-pool \
    --cluster=my-cluster \
    --machine-type=a3-highgpu-8g \
    --accelerator=type=nvidia-h100-80gb,count=8,gpu-driver-version=latest \
    --secondary-boot-disk=disk-image=projects/PROJECT/global/images/train-images-v1,mode=CONTAINER_IMAGE_CACHE

From the docs: "You can't update secondary boot disks for existing nodes. To attach a new disk image, create a new node pool." This means CI builds new disk images per release; you rotate node pools rather than updating in place.

The end-to-end win for inference

Per the GKE inference data-loading blog: "GKE lets you pre-cache your container image into a secondary boot disk that is attached to your node at creation time. The image above shows a secondary boot disk (1) that stores the container image ahead of time, avoiding the image download process during pod/container startup. And for AI/ML inference workloads with demanding speed and scale requirements, Cloud Storage Fuse (2) and Hyperdisk ML (3) are options to connect the pod to model + weight data stored in Cloud Storage or a network attached disk." The full picture: secondary boot disks for the image, Hyperdisk ML for the weights, GCS Fuse for hot training data.

From the same source: "Loading large AI models, weights, and container images into GKE-based AI models can delay workload startup times. By using a combination of the three methods described above — secondary boot disk for container images, Hyperdisk ML OR Cloud Storage Fuse for models + weights — get ready to accelerate data load times for your AI/ML inference applications."

GKE Image Streaming — start before pull completes

The complementary technique. Image Streaming lets containerd start running the container before the full image has been pulled from the registry — file blocks are fetched on-demand from a streaming layer cache. The first read of any file pulls it; subsequent reads hit local cache.

Enabled at cluster level via --enable-image-streaming. The cluster requires Container-Optimized OS nodes and images stored in Artifact Registry. The win is most pronounced for images where most files aren't accessed at startup — a vLLM image with many bundled examples and tools, of which only the core binary runs at boot.

Autopilot fast-starting nodes

Per the fast-starting nodes docs: "Requesting compatible GPUs in Autopilot mode results in up to four times faster node startup time and up to two times faster Pod scheduling time than similar requests in GKE Standard mode, because the Autopilot GPU workloads can use fast-starting nodes." For GKE Autopilot workloads, this is automatic — no manual config needed. For Standard clusters, the equivalent is a combination of secondary boot disks + image streaming + warmer node pools.

Constraints from the same source: "G2 with DEFAULT GPU driver version on GKE versions earlier than 1.33.0-gke.1304000. For 1.33.0-gke.1304000 or later, both LATEST and DEFAULT work with fast-starting nodes. Secondary boot disks on versions earlier than 1.33.2-gke.1015000." Both newer-GKE features.

The combined stack

For a Metaflow flow that needs the fastest possible cold start on Standard GKE:

  1. Pre-build a secondary boot disk image containing the training container (rotates per release).
  2. Pre-hydrate a Hyperdisk ML volume with foundation model weights.
  3. Mount GCS via Fuse with caching profile for training shards.
  4. Use DWS Flex-start for capacity acquisition.
  5. Image Streaming enabled at cluster level as a belt-and-suspenders fallback.

The cold-start timeline collapses from 45 minutes (capacity) + 12 minutes (pull) + 20 minutes (weights) = 77 minutes to maybe 3 minutes (capacity from reservation or warm flex-start) + 0 minutes (image already on disk) + 90 seconds (Hyperdisk ML weight load) = under 5 minutes. The optimizations compose.

Image optimization — the underloved baseline

Before reaching for boot disks and streaming, consider the basics:

  • Multi-stage builds. Build artifacts in a heavy image; copy only what's needed into a slim final image. CUDA dev images are 5GB+; runtime images can be under 2GB.
  • Layer ordering. Put rarely-changing layers (CUDA, framework) at the bottom; frequently-changing layers (your code) at the top. Layer cache reuse improves dramatically.
  • Minimal base images. nvidia/cuda:12.4.0-runtime-ubuntu22.04 not :devel. Strip dev headers, build tools, docs.
  • Distroless or Chainguard for final stage. If your runtime doesn't need apt, ship without it.

A 12GB → 4GB image reduction is often a one-day refactor with multi-stage builds. It saves time at every pull, on every node, forever.

The cache rotation discipline

Secondary boot disks freeze the container image at the time the disk image was built. If you push a new container image, existing nodes still have the old one. The discipline: tag boot disk images, rotate node pools when new container images ship, and use a CD pipeline that produces both at once. Failure to rotate means new pods run old code from the boot disk, silently.

For vLLM inference deployments specifically, there's a third option that's neither GCS Fuse nor Hyperdisk ML — the Run:ai Model Streamer. SCHED Chapter XIV

Kueue gang scheduling · all-or-nothing admission.

The problem Default kube-scheduler admits pods one at a time. For 8-pod distributed training, partial admission = deadlock.

The fix Kueue's ClusterQueue admits an entire workload or none of it. Combined with DWS ProvisioningRequest for atomic capacity.

A single Metaflow step with @resources(gpu=8) is one pod — gang scheduling is unnecessary. But the moment you have multiple parallel steps each requesting 8 GPUs, or a multi-pod distributed training job (one head, N workers), gang scheduling becomes critical. Kueue is the Kubernetes-native job queueing system that adds gang scheduling on top of kube-scheduler. It introduces three CRDs — ResourceFlavor, ClusterQueue, AdmissionCheck — that together enforce "this workload's pods are admitted together, or none of them are." Combined with DWS ProvisioningRequest, you get atomic capacity acquisition AND atomic scheduling.

Why default kube-scheduler is wrong for gangs

kube-scheduler is a pod-at-a-time scheduler. When eight pods need to run together (e.g., a multi-node training job with NCCL), the scheduler admits them in whatever order they become schedulable. If you have capacity for 6 nodes but need 8, six pods schedule and start running; two wait in Pending; the running ones burn $30/hr/GPU while waiting for the missing two. If the missing capacity never materializes, the job is stuck — you've paid for 6 pods' worth of compute that did nothing.

The Kueue model

Per CoreWeave's Kueue overview: "Kueue allows users to submit jobs which remain queued until sufficient compute is available for the job. Using gang-scheduling, Kueue also provides a mechanism to start the job on all nodes simultaneously. Unlike traditional Kubernetes workloads, where resources are generally modeled only as CPU and memory, AI/ML workloads can run on a staggering variety of hardware types. While it's possible to leverage Kubernetes labels, taints, and tolerations to target workloads to the various machine types, it is difficult, tedious and often error-prone to do so. Kueue provides an explicit and expressive formalization of the computation resource by means of a ResourceFlavor. This allows administrators to model the properties of the instances that are relevant to their specific workloads, only allowing admission to nodes which match the needs of the corresponding workload."

The three CRDs:

  • ResourceFlavor. A named bundle of node attributes (machine type, GPU type, location). Workloads request a flavor; Kueue ensures admission only happens when nodes of that flavor are available.
  • ClusterQueue. A queue with quotas. "This namespace can use up to 100 H100 GPUs." Workloads enter the queue; admission happens FIFO or by priority.
  • AdmissionCheck. A plug-in extension point that gates admission. The provisioning-request check waits for DWS to provision before admitting.

The workflow

Per the DWS technical guide: "The workflow is as follows: a user submits a Job (or other workload type like a RayJob) in a suspended state. Kueue intercepts this and creates a single, atomic ProvisioningRequest for all the nodes required by the job. DWS then waits until it can fulfill the entire request at once. Only when all nodes are provisioned and ready does Kueue 'unsuspend' the job, allowing the pods to be scheduled simultaneously onto the new nodes."

The user submits a Job with the queue label; Kueue intercepts, suspends the Job, creates a ProvisioningRequest. DWS fulfills the request when capacity arrives. The provisioning-request AdmissionCheck transitions to "True." Kueue unsuspends the Job. All pods schedule together on the freshly provisioned nodes. NCCL initializes. Training starts.

The single-PodSet limitation

The non-obvious gotcha. Per a Ray project issue from October 2025: "The tutorial Gang Scheduling with RayJob and Kueue shows how to integrate with Kueue and GKE's 'dynamic workload scheduling' feature to achieve gang scheduling. However, the tutorial should mention the constraint that this only works if the RayJob only has a head but no workers. Otherwise one gets this error: Error creating ProvisioningRequest: admission webhook 'warden-validating.common-webhooks.networking.gke.io' denied the request: GKE Warden rejected the request because it violates one or more constraints. Violations details: {'[denied by provisioning-request-cr-validation]':['the queued provisioning feature currently supports only single PodSet per request']}"

The constraint: a single ProvisioningRequest can only describe one PodSet — one shape of pod, replicated N times. RayJob has two shapes (head + worker), so it fails. Per the GKE Flex-start docs: "A single ProvisioningRequest object supports only one entry in the podSets list. Requests with multiple podSet entries fail."

For Metaflow's @resources(gpu=8) on a single pod, this is fine — one shape, one PodSet. For more complex workloads (RayJob with workers, JobSet with multiple roles), you can't use queued-provisioning today; you need either separate ProvisioningRequests or wait for the multi-PodSet feature to land.

For Metaflow specifically

Metaflow's Kubernetes integration creates a Pod per step, not a Job. Kueue supports plain pods as a workload integration, so you can still use it — but the Metaflow team isn't (yet) producing Kueue-aware manifests out of the box. The practical path: a thin admission-control layer (e.g., via a mutating webhook or a Metaflow custom step decorator) that adds the kueue.x-k8s.io/queue-name label to pods created by Metaflow. Once that label is present, Kueue intercepts and applies admission control.

For most teams running single-step @resources(gpu=8) Metaflow flows, the simpler path is to skip Kueue and rely on ComputeClass + DWS Flex-start (Best-Effort mode). Kueue becomes important when you have:

  • Multiple parallel @resources(gpu=8) branches that need atomic admission
  • Fair sharing of GPU quota across teams/namespaces
  • Priority-based preemption (production flows preempt dev flows)
  • Multi-cluster Job dispatching (MultiKueue)
The pragmatic Metaflow + Kueue recommendation

Start with ComputeClass + DWS Best-Effort flex-start; this covers single-pod 8-GPU steps cleanly. Add Kueue when you have multi-tenant GPU sharing requirements (fair share, preemption, priorities), or when concurrent parallel Metaflow branches need atomic admission. Kueue is genuinely useful but operationally non-trivial — three CRDs to author and version, a controller pod to monitor, debugging admission state when things stall. Don't reach for it for single-pod cases.

DRA + KAI · the 2026 future.

What's new Topology-aware allocation. Cross-pod device sharing. The KAI scheduler for AI-native gang scheduling.

The 2018-era device-plugin model — nvidia.com/gpu: 8 in resource limits — has been the universal way to request GPUs in Kubernetes for years. It works, but it's blind to topology (which NVLink islands does this GPU belong to?), to sharing (one workload, one whole GPU, no exceptions), and to capacity claims that span pods (this 16-GPU job needs a coherent 16-GPU placement). The 2026 replacement is DRA — Dynamic Resource Allocation — which went GA in Kubernetes 1.34. Combined with the KAI Scheduler (donated by NVIDIA to CNCF), it produces the new shape of AI scheduling: topology-aware, share-aware, and gang-native. This chapter is the orientation; full adoption is a 2026-2027 process for most teams.

What DRA is

Per the TechPlained 2026 K8s GPU scheduling deep-dive: "Kubernetes GPU scheduling used to mean 'set nvidia.com/gpu: 1 on your Pod and hope.' That model — the device-plugin framework shipped in 2018 — assumed one workload, one whole GPU, and no concept of topology, sharing, or priority. It was fine for a few Jupyter notebooks. It is catastrophically wrong for the 2026 reality of mixed training-inference-notebook clusters where an 8x H100 node costs more per hour than a mid-sized engineering team. At KubeCon North America 2025 NVIDIA donated the k8s-dra-driver to the CNCF, Dynamic Resource Allocation (DRA) went GA in Kubernetes 1.34, and the KAI Scheduler became the de facto reference for AI-workload-aware scheduling."

The mechanism: instead of device plugins exposing a flat count (nvidia.com/gpu: 8), DRA introduces ResourceClaim and ResourceClass CRDs that describe device requirements richly. "I need 8 GPUs on the same NVSwitch fabric." "I need a MIG slice with 20GB of memory." "I need to share this GPU with another pod that I can name." The DRA driver translates claims into specific device assignments, considering topology and sharing.

What KAI is

Per the same source: "KAI isn't the only option. Kueue (Kubernetes SIG, thin queue layer delegating to kube-scheduler) and Volcano (CNCF batch scheduler, older, HPC-oriented) also do gang scheduling. The practical split: KAI: best when workloads are mostly GPU AI/ML and you want topology-aware gang scheduling with DRA integration. NVIDIA-aligned. Kueue: best when you want a thin queue layer on top of the default scheduler and workloads mix CPU and GPU batch. Volcano: best for HPC-style MPI jobs and Slurm-trained operators."

KAI is a scheduler — runs alongside kube-scheduler, takes over decisions for workloads that target it. It understands DRA, NVLink topology, MIG, MPS, and priority preemption native to AI workloads.

The version gating reality

Per the TechPlained guide: "Watch out: DRA is only GA on Kubernetes 1.34+. On 1.32 and 1.33 it ships as beta behind the DynamicResourceAllocation feature gate. Most managed offerings (EKS 1.34, GKE 1.34, AKS 1.34) shipped DRA-capable control planes in Q1-Q2 2026. If your cluster is older, you're still on device plugins until you upgrade — there is no backport. DRA handles the device abstraction; KAI handles the workload abstraction."

For practical Metaflow deployments in mid-2026, this means: if you're on GKE 1.34+, DRA is available; if you're on an older release channel, you're still on device plugins. Most GKE clusters are still effectively device-plugin-based, and the nvidia.com/gpu: 8 pattern continues to work fine. DRA is the future; device plugins are the present.

MIG vs MPS vs time-slicing

For workloads where one full GPU is overkill — small-model inference, dev notebooks, eval harnesses — sharing makes sense. Three NVIDIA mechanisms:

  • MIG (Multi-Instance GPU). Hardware partitioning. A100/H100/H200/B100 GPUs split into up to 7 isolated instances per GPU. Each instance is a separate device with its own memory and compute. True hardware isolation; predictable QoS. Per the GKE MIG docs: "multi-instance GPUs allow you to partition a single supported GPU in up to seven slices. Each slice can be allocated to one container on the node independently, for a maximum of seven containers per GPU. Multi-instance GPUs provide hardware isolation between the workloads, and consistent and predictable QoS for all containers running on the GPU."
  • MPS (Multi-Process Service). NVIDIA's software multiplexing. Multiple processes share one GPU concurrently; better utilization than time-slicing; some isolation but not as strong as MIG.
  • Time-slicing. Round-robin time-share. Two pods, each thinks they have the GPU, take turns in milliseconds. No isolation; cheapest to operate; for dev clusters and notebooks only.

Per the TechPlained guide: "A typical production cluster runs MIG on inference nodes, whole-GPU on training nodes, and time-slicing on the dev namespace. Use MIG for production multi-tenant workloads where isolation matters — separate memory, SMs, and fault domains prevent one tenant from starving or crashing another. Use time-slicing for dev clusters where the GPU sits idle most of the time and oversubscription is fine. MIG only works on A100, H100, H200, and B100-class GPUs; time-slicing works on any CUDA GPU. Never rely on time-slicing for production SLOs."

Where this matters for Metaflow @resources(gpu=8)

Almost nowhere, today. @resources(gpu=8) wants a whole node — 8 full GPUs. MIG, MPS, time-slicing are for "give me less than one GPU" patterns. DRA matters when you eventually need topology-aware scheduling (e.g., "ensure all 8 GPUs are on the same NVLink fabric for fastest all-reduce") — but the GCP A3/A3 Ultra/A4 8g machine types already ship with all 8 GPUs in one NVLink island, so topology is implicitly correct.

The 2026-2027 evolution that will matter for Metaflow: multi-node distributed training where 16 GPUs span 2 nodes, and you want guarantees about the inter-node fabric. DRA + KAI will let workloads request this explicitly; today it's a manual reservation/placement-policy exercise.

The takeaway

For a 2026 Metaflow flow running @resources(gpu=8) on GKE:

  • Continue using device plugins (nvidia.com/gpu: 8 in pod limits). It works; it's what Metaflow emits.
  • Continue using kube-scheduler with Kueue if you need gang admission across parallel branches.
  • Skip DRA and KAI for now unless you're already on GKE 1.34+ and have multi-node training that needs topology-aware placement.
  • Re-evaluate in 2027 when DRA tooling matures and most managed offerings have stabilized.
The conservative posture

DRA + KAI is the right direction for the long term. The conservative production posture in mid-2026 is to use device plugins for GPU requests, Kueue for queueing where needed, and DWS + ComputeClass for capacity. Adopt DRA when (a) you upgrade to GKE 1.34+ across all clusters, (b) you have workloads where topology-aware allocation is provably needed (multi-node training with custom NVSwitch topologies), and (c) the surrounding ecosystem (Metaflow integrations, monitoring tools, debugging guides) has caught up. For most teams that's 2027.

The decorator order matters

Per the Metaflow Kubernetes tips: "When setting @resources, keep in mind the configuration of your Kubernetes cluster. Your pod will be stuck in an unschedulable state if Kubernetes is unable to provision the requested resources." Plus from the @resources docs: "In addition to cpu and memory you can specify gpu=N to request N GPUs for the instance." And from the @kubernetes docs: "Number of CPUs required for this step. If @resources is also present, the maximum value from all decorators is used."

When both @kubernetes and @resources are present and both specify CPU/memory/GPU, the maximum value wins. This means you can set conservative defaults in @kubernetes and override per-step with @resources — useful when one step needs much more than the cluster's default. For GPU specifically, putting gpu=8 in @resources and not in @kubernetes is the cleanest pattern.

Interactive decision tree
Pick the right stack for your Metaflow GPU step.

The questions, in order

  1. Workload shape. One pod or many? Single-pod 8-GPU is the simple case (no gang scheduling needed). Multi-pod distributed (one Metaflow step that fans out across multiple nodes for 16+ GPUs, or multiple parallel Metaflow branches) needs Kueue + queued provisioning.
  2. Criticality. Production training run where every minute of wait costs you, or fault-tolerant batch where preemption is OK? Critical pushes you toward reservations + obtainability-first ComputeClass; fault-tolerant pushes you toward spot + flex-start.
  3. Data shape. Static foundation weights (read many times) vs. mutable training data (read+write)? Static + reused = Hyperdisk ML. Mutable = GCS Fuse with profiles. Both = use both.
  4. Reservation availability. Do you have a reservation? If yes, lead with it in ComputeClass priorities. If no, lead with flex-start.
  5. Latency target. Cold-start within 5 minutes (need every optimization), or "minutes are fine"? The aggressive target pulls in secondary boot disks, image streaming, fast-starting nodes.

The shape of each verdict

Every verdict in the picker names five things:

  • GPU type — which 8g machine to target (or sub-8 if appropriate).
  • Consumption mode — reservation, flex-start best-effort, flex-start queued, spot, on-demand, or some combination.
  • ComputeClass shape — which of the three priority patterns (obtainability, cost-optimized, hybrid).
  • Storage stack — GCS Fuse, Hyperdisk ML, Run:ai streamer, or combinations.
  • Scheduler — kube-scheduler alone, Kueue gang admission, or DRA/KAI (for the few teams already on 1.34+).

What the picker doesn't decide

The infrastructure side — Terraform for cluster + node pools, Argo CD for ComputeClass CRDs, the hydration pipelines for Hyperdisk ML, the Workload Identity bindings — these are constant across all the verdicts; they're not workload-specific decisions. Issue 13 covers them. The picker assumes that infrastructure exists; it picks how a particular Metaflow flow consumes it.

When the picker is wrong

The picker reflects 2026 best practices but can't model every constraint. Specific situations where it's likely wrong: workloads with deeply unusual memory patterns (need to hand-pick storage), regulated environments where data residency forces specific zones, multi-cloud Metaflow deployments where the GKE side is one of several backends. For these, take the picker's verdict as a starting point and adjust. The chapters cited in each verdict explain the underlying tradeoffs.

NVIDIA_H100_GPUS, NVIDIA_H200_GPUS, etc. Quota = hard ceiling regardless of capacity availability.
4 · DWS Flex-start node pool created with --flex-start (and --enable-queued-provisioning for gang)
Pool exists at 0 nodes; auto-scaling configured; reservation-affinity=none for queued.
5 · GCS Fuse CSI driver enabled at cluster level
--addons GcsFuseCsiDriver at cluster create, or update via gcloud after.
6 · Workload Identity Federation enabled; KSA bound to GSA
workload-pool=PROJECT_ID.svc.id.goog on cluster. KSA annotation iam.gke.io/gcp-service-account=GSA_EMAIL.
7 · GCS bucket region matches cluster region
Cross-region GCS reads are slow and expensive. Multi-region buckets are an alternative.
8 · Hyperdisk ML volume pre-hydrated for foundation weights
If using foundation models, hydrate once, snapshot, reference via PVC. 11.9x weight load speedup.
9 · gke-gcsfuse/volumes: "true" annotation on every pod using GCS Fuse
Without this annotation, the sidecar isn't injected; the mount silently doesn't happen.
10 · gke-gcsfuse/ephemeral-storage-limit sized larger than fileCacheCapacity
The cache is silently capped at the ephemeral-storage limit; size both explicitly.
11 · Sidecar CPU and memory limits raised for high-throughput training
Defaults are too small. gke-gcsfuse/cpu-limit: "8", gke-gcsfuse/memory-limit: "32Gi".
12 · Pod tolerations include nvidia.com/gpu:Exists:NoSchedule
Standard GPU node taint. Pods without this toleration can't land on GPU nodes.
13 · @resources(gpu=8) plus matching cpu and memory
For a3-highgpu-8g: cpu=200, memory=1800000. Leave headroom under the 208 vCPU / 1872GB physical.
14 · service_account set explicitly in @kubernetes (not relying on default)
Default SA has no GCS or Artifact Registry permissions. Explicit KSA bound to a GSA with the right roles.
15 · shared_memory=16000 for PyTorch / DataLoader workloads
Default /dev/shm of 64MB causes silent DataLoader hangs at scale. 16GB is a safe default for 8-GPU training.
16 · persistent_volume_claims dict mounts the right PVCs at expected paths
Hyperdisk ML weights, training-data PV, checkpoints PV — all listed explicitly.
17 · Container image is multi-stage built; <5GB if possible
Multi-stage builds; strip dev headers; use runtime base images. Cuts image pull time linearly.
18 · Secondary boot disk image pre-built with container cache (optional, big win)
For tight cold-start SLOs. Requires a CI pipeline that builds disk images per release.
19 · GKE Image Streaming enabled at cluster level
Complementary to secondary boot disks. Lazy-loads image layers on demand.
20 · safe-to-evict: false annotation on long-running training pods
Prevents the cluster autoscaler from migrating mid-run. Auto-injected for consume-provisioning-request pods.
21 · DCGM exporter installed for GPU utilization metrics
Issue 16 territory. Without GPU metrics, you can't tell whether the optimizations actually helped.
22 · Cloud Logging filter saved for autoscaler events
resource.type="k8s_cluster" jsonPayload.event="PROVISIONING_TIME_LIMIT_REACHED" to see fallback decisions.
23 · Cost monitoring with breakdown by ComputeClass / Metaflow flow
Cost labels propagated from ComputeClass → node pool → resource consumption. Per-flow attribution.
24 · Runbook for stuck-Pending pods at the four diagnostic layers
Capacity / ComputeClass / GCS Fuse / Metaflow integration. The four diagnostic tables above.

The closing rule

This issue covered eighteen techniques across three bottlenecks. The 2026 reality is that not every workload needs every technique. The discipline is to name your bottleneck before reaching for a tool: pods stuck Pending = capacity work (DWS, ComputeClass priorities, reservations); pods ContainerCreating for ages = launch work (secondary boot disks, image streaming, multi-stage builds); pods Running but slow to first gradient = data work (Hyperdisk ML, GCS Fuse profiles, Run:ai streamer). Add the optimization that addresses the bottleneck you're actually in. The Metaflow flow at the end of Chapter XVI is the composition shape; not every step needs every part.

The final rule of thumb

If a Metaflow @resources(gpu=8) step takes more than 5 minutes from kubectl apply to first gradient, something in the three-bottleneck stack is mis-tuned. Walk the four diagnostic tables, find the layer, apply the optimization. Don't accept "GPUs are slow to start" as a baseline; with these techniques, sub-5-minute is achievable, sub-3-minute is achievable with reservation + Hyperdisk ML + cached image. The price of getting it right pays back the first day; the price of getting it wrong compounds for the lifetime of the cluster.
