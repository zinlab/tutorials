Part I — The object model

What you're actually holding.

Every internals conversation bottoms out here. A tensor is not an array — it's a small metadata object pointing at bytes, and almost every "why did PyTorch do that?" question is answered by its five fields.

01What a tensor actually is

torch.Tensor (Python) ──▶ at::Tensor ──▶ intrusive_ptr<TensorImpl> │ TensorImpl (c10/core/TensorImpl.h) │ ├─ storage_ ──▶ Storage ──▶ StorageImpl ──▶ DataPtr ──▶ raw bytes │ │ └─ Allocator (CUDA │ └─ nbytes caching allocator) ├─ sizes_ [2, 3, 4] ├─ strides_ [12, 4, 1] ← elements, not bytes ├─ storage_offset_ 0 ← where this tensor starts in storage ├─ dtype_ (TypeMeta), device_ ├─ key_set_ DispatchKeySet ← CUDA | Autograd | ... (see §04) ├─ version_counter_ ← autograd's mutation detector (§14) ├─ autograd_meta_ grad_fn / grad / requires_grad (nullptr if no autograd) └─ (view metadata, if a view) ───────────────────────────────────────────────────────────── KEY CONSEQUENCE: many Tensors can share ONE Storage. A view is a new TensorImpl with different sizes/strides/offset over the SAME bytes. That single fact explains views, in-place hazards, .detach(), memory "leaks" that aren't leaks, and half of autograd's rules.
FieldWhy an interviewer cares
storage + offsett[1:] costs nothing and frees nothing — the storage stays alive as long as any view does. "My memory won't drop after slicing a big tensor" is this, not a leak.
sizes/stridesLayout algebra (§02): transpose/permute/expand are stride tricks with zero data movement.

autograd_metaNullptr for most tensors — autograd metadata is optional weight, allocated only when needed. (Why inference_mode can be faster than no_grad: §15.)

02Strides, views & contiguity

Strides are the address function: element(i,j,k) = storage[offset + i·s0 + j·s1 + k·s2]. Everything below follows from letting sizes and strides lie about the bytes.

OperationWhat it does to metadataCopies?
view / reshapeNew sizes; strides recomputed. reshape falls back to a copy when the requested shape isn't expressible on the current strides; view raises instead.view: never · reshape: sometimes
transpose / permuteSwaps entries of sizes and strides. Result is non-contiguous.No
expand / broadcastSets stride = 0 on the expanded dim — the same element is read many times.No (stride-0 magic)
slice / narrowAdjusts sizes + storage_offset.No
contiguous()If already contiguous: returns self. Else materializes a fresh, densely-strided copy.Conditionally
as_stridedRaw metadata surgery — the escape hatch (and the footgun; it can alias memory arbitrarily).No
The three sentences that show you actually know this

(1) "Contiguous" means strides are exactly the row-major products of the trailing sizes — nothing more. (2) A stride-0 dimension is how broadcasting avoids materializing the broadcast — which is also why a broadcast tensor can't be written to in-place safely. (3) reshape = view when possible, contiguous().view() otherwise — so reshape can silently cost you a copy in a hot loop, and that copy shows up in a profile as a mysterious copy_ kernel.

Memory format: the second layout axis

channels_last (NHWC) is not a different tensor class — it's a stride permutation with the same logical shape (N,C,H,W). It matters because cuDNN's tensor-core convolution kernels want NHWC: feed them NCHW and the library inserts transposes around every conv. The classic perf bug is a model where some tensors were converted and others weren't, so the stack ping-pongs between formats. Memory-format propagation rules (which ops preserve channels_last) are an actual maintenance surface on this team.

03dtype, device & type promotion

Type promotion is Xiang Gao's territory — he did much of TensorIterator's promotion machinery, and there's a well-known issue in the repo about promotion causing an extra copy kernel launch. Know the rules and the cost.

THE PROMOTION LATTICE (categories): bool < integral < floating < complex PARTICIPANT TIERS (within the winning category): dimensioned tensor > 0-dim tensor > python scalar ("wrapped number") ───────────────────────────────────────────────────────────── RESULT = highest category present; within it, the dtype is decided by the HIGHEST TIER of participants — lower tiers can raise the CATEGORY but not the WIDTH. ───────────────────────────────────────────────────────────── int8_tensor + int64_0dim_tensor → int8 (0-dim can't widen int8) int8_tensor + 1.0 (py float) → float (category raised → default float) fp16_tensor + 1.0 (py float) → fp16 (scalar doesn't widen to fp32!) fp16_tensor + fp32_tensor → fp32 (both dimensioned → widen) bool_tensor + True → bool ───────────────────────────────────────────────────────────── COST: when inputs disagree, TensorIterator either fuses the cast into the kernel's load, or (if it can't) emits a separate copy kernel — which is exactly the "why are there two kernels?" bug class.
The trap worth naming out loud

Mixed-dtype arithmetic in a training loop is a silent perf tax: every promotion is either a fused cast (cheap) or an extra materialized copy (a whole kernel + a whole tensor of bandwidth). When someone says "my AMP model isn't faster," an under-appreciated cause is dtype churn — fp16 activations meeting fp32 constants and getting promoted back up. Say that, and you're speaking directly into this team's daily bug queue.

04The dispatch key set

Every tensor carries a DispatchKeySet — a bitset of the layers that apply to it. Every op call unions the key sets of its arguments, applies thread-local includes/excludes, and picks the highest-priority key present. That's the whole dispatcher in one sentence; §05–06 unpack it.

Key familyExamples · role
Backend keys (lowest priority)CPU, CUDA, Meta, SparseCUDA, QuantizedCPU, NestedTensor, XLA — where the actual math lives. A CUDA tensor carries the CUDA key; that's how a + b finds the CUDA kernel with no if device == 'cuda' anywhere.
Autograd keysAutogradCUDA/AutogradCPU/… — the layer that records the graph, then re-dispatches down (§06).
Autocast keysAutocastCUDA — casts inputs per op policy, then re-dispatches (§06). Sits above autograd, which is the whole reason AMP is differentiable.
View/in-place bookkeepingADInplaceOrView — bumps version counters and records view metadata, separately from graph recording (§14).
Functionality/otherFunctionalize (removes mutation for compile), Python/PythonTLSSnapshot (__torch_dispatch__), Batched/VmapMode (vmap), Tracer, Conjugate/Negative (lazy conj/neg views), BackendSelect (factory functions like torch.zeros that have no tensor argument to infer from).
TLS include/excludeHow no_grad works: it excludes the autograd keys for the duration. Not a flag check inside every op — a key-set edit.

The mental model to carry into every one of these interviews: PyTorch is a dispatch machine with layers, and "features" are layers. Autograd is a layer. AMP is a layer. vmap is a layer. Functionalization is a layer. Backends are the bottom layer. Once you hold that, questions that sound unrelated — "how does autocast interact with autograd?", "why does torch.compile need functionalization?", "how would a new backend get added?" — all collapse into the same answer shape: a key, a kernel registered to it, and a re-dispatch to whatever is below.

Part II — The dispatcher

How a call finds its kernel.

Five chapters on the machine that turns a + b into a launched CUDA kernel — the layering, the registration system, and the macro that bridges runtime dtypes to compile-time C++ types. This is the spine of the codebase; knowing it is what separates "PyTorch user" from "PyTorch engineer" in a room full of PyTorch engineers.

05The life of a + b

a + b (a, b: CUDA fp16, requires_grad, inside autocast) ───────────────────────────────────────────────────────────── 1 Python torch.Tensor.__add__ → C++ binding (torch/csrc) 2 at::add(a, b) → Dispatcher::call keys = a.key_set | b.key_set | TLS_included − TLS_excluded = {AutocastCUDA, AutogradCUDA, ADInplaceOrView, CUDA} 3 highest priority → AUTOCAST layer policy for `add`: promote-to-widest (not a cast-to-fp16 op) casts if needed → redispatch with Autocast EXCLUDED 4 next → AUTOGRAD layer (AutogradCUDA kernel, codegen'd) - checks requires_grad on inputs - constructs AddBackward0 node; wires next_edges to inputs' grad_fn (or their AccumulateGrad) - saves what backward needs (for add: just alpha — no tensors!) - redispatch with Autograd EXCLUDED → 5 next → ADInplaceOrView (for out-of-place add: near no-op; for add_ : bump version counter; for views: record view meta) 6 next → CUDA BACKEND kernel: at::native::add_kernel_cuda - build a TensorIterator (broadcast, promote, coalesce) §18 - AT_DISPATCH over the common dtype → scalar_t §08 - gpu_kernel(iter, [] __device__ (scalar_t x, scalar_t y) { return x + alpha * y; }) §19 - launch: vectorized_elementwise_kernel if contiguous+aligned 7 result tensor gets grad_fn = AddBackward0; returns up the stack ───────────────────────────────────────────────────────────── EVERY layer's exit move is the same: RE-DISPATCH to the next key down.
Say this and the room relaxes

"The dispatcher is a priority-ordered layer stack driven by a bitset the tensors carry, and each layer does its job and re-dispatches downward with its own key masked off — so autograd is just the layer that records a node before calling the real kernel." That one sentence answers a dozen possible questions and proves you've read the machine, not the docs.

06Keys as layers: autocast, autograd, backend

The priority order isn't arbitrary — it's forced by correctness. Derive it rather than memorizing it, because a derivation survives an interviewer's follow-up.

Why autocast must sit ABOVE autograd: autocast's job is to run the op in fp16/bf16. If autograd recorded first, it would save the fp32 inputs and build a backward expecting an fp32 op, while the forward silently ran in fp16 below — a mismatch. Because autocast is above, the casts it inserts are themselves ordinary ops that pass through the autograd layer below, so each cast gets its own backward node. That's exactly how gradients flow from an fp16 forward back to fp32 master weights with no special-casing anywhere: the cast is differentiable because it happened above autograd. (Autocast also caches the cast copy of a weight within the region, so a weight used twice isn't cast twice.)

LayerWhat it does before redispatching
Python / __torch_dispatch__Lets Python subclasses intercept every op — the mechanism behind tensor subclasses, DTensor, and much of the compile stack's tracing.
FunctionalizeRewrites mutations and views into functional equivalents so compilers see a clean, side-effect-free graph. (Answer to "why does compile need functionalization?")
AutocastPer-op casting policy: matmul/conv-family → lower precision; reductions/norms/softmax-family → fp32; others → promote to widest input. Then redispatch.
AutogradIf any input requires grad: build the backward Node, wire edges, save tensors, set the output's grad_fn. Then redispatch. Under no_grad this key is excluded, so the layer never runs.
ADInplaceOrViewVersion-counter bumps for in-place ops; view metadata for view ops. Split out from the autograd layer so inference paths can skip it cheaply.
Backend (CPU/CUDA/…)The math. Everything above is bookkeeping.

07Registration & codegen

Ops aren't written as C++ functions someone calls — they're declared in a schema file and registered against dispatch keys, with codegen generating the glue. This is the surface a life-cycle engineer touches constantly (every new op = new tests, new coverage, new CI).

# aten/src/ATen/native/native_functions.yaml — the single source of truth
- func: add.Tensor(Tensor self, Tensor other, *, Scalar alpha=1) -> Tensor
  device_check: NoCheck
  structured_delegate: add.out          # see §21 — impl lives in the out variant
  variants: function, method            # torch.add(a,b) AND a.add(b)
  dispatch:
    SparseCPU, SparseCUDA: add_sparse   # per-key kernels where needed
  tags: core, pointwise
MechanismWhat it gives you
The schema lineName, arg types, defaults, aliasing/mutation annotations (Tensor(a!) = mutated in place). The schema is what the dispatcher, the JIT, functionalization, and every serializer all read.
CodegenGenerates the C++ API (at::add), the Python bindings, the autograd kernel (from derivatives.yaml, §09), the registration boilerplate, and the meta/structured plumbing. Editing generated code is a classic newcomer mistake — you edit the yaml and the codegen.
TORCH_LIBRARY / TORCH_LIBRARY_IMPLThe out-of-tree path: define a custom op and register kernels per key from an extension — the same machinery, exposed publicly. (Your pybind11 experience lands here: this is how a real custom CUDA op joins the dispatcher rather than bolting onto the side.)
Static registration ⇒ build hazardsRegistration runs from static initializers at load time — which is why Vol. III §12's whole-archive/duplicate-registration bestiary exists. The internals and the linking problems are the same fact seen from two sides.

08AT_DISPATCH and scalar_t

You've already gone deep here — this is the bridge between a runtime ScalarType and a compile-time C++ type, and it's why one kernel source compiles into a kernel per dtype.

void add_kernel_cuda(TensorIteratorBase& iter, const Scalar& alpha_scalar) {
  AT_DISPATCH_ALL_TYPES_AND_COMPLEX_AND3(
      at::ScalarType::Half, at::ScalarType::BFloat16, at::ScalarType::Bool,
      iter.common_dtype(), "add_cuda", [&]() {          // ← the lambda is
    using opmath_t = at::opmath_type<scalar_t>;          //   instantiated ONCE
    auto alpha = alpha_scalar.to<opmath_t>();            //   PER DTYPE, with
    gpu_kernel_with_scalars(iter,                        //   scalar_t bound
      [alpha] __device__ (scalar_t a, scalar_t b) -> scalar_t {
        return static_cast<opmath_t>(a) + alpha * static_cast<opmath_t>(b);
      });
  });
}
DetailWhy it's the senior detail
It's a switch, not magicAT_DISPATCH_* expands to a switch on the runtime ScalarType; each case instantiates the lambda with scalar_t = that C++ type. Hence: compile time and binary size scale with the dtype list — a real cost this team pays and manages (and one reason adding a dtype to an op isn't free).
opmath_t / acc_typeThe compute type is deliberately not the storage type: fp16 in, fp32 math, fp16 out. This is the single most important numerics idiom in ATen — and the reason CPU-vs-GPU results can legitimately differ (§Vol. VI).
The AND3 suffixThe macro family is a menu — floating-only, all-types, plus explicitly-listed extras (Half/BFloat16/Bool/complex). Which macro an op uses is its dtype support matrix, and mismatches between that and the tests are a live CI issue class.

09Composite vs explicit; derivatives.yaml

KindMeaning · consequence
CompositeImplicitAutogradThe op is written in terms of other ops. It gets autograd for free (the constituent ops record their own nodes) and works on every backend automatically. Cost: no fused kernel, and the backward graph is the composition — often more memory, more launches.
CompositeExplicitAutogradOne implementation for all backends, but autograd must be defined explicitly (an entry in derivatives.yaml).
Per-backend kernelsdispatch: CPU: foo_cpu, CUDA: foo_cuda — the hand-written path, with an explicit derivative formula.
tools/autograd/derivatives.yamlWhere backward formulas live, as symbolic expressions over grad and saved values. Codegen turns them into the *Backward0 Node classes you see in grad_fn. This is the file to know exists — "where is the derivative of X defined?" is a question you can now answer instantly.
The design question they might actually ask

"When would you write a fused CUDA kernel with an explicit derivative instead of a composite op?" The answer is a tradeoff, not a preference: fuse when the composite's intermediates dominate (memory bandwidth and peak memory), when launch count matters (many tiny ops), or when a numerically-stable formulation needs to be computed in one pass (logsumexp, softmax, Welford variance). Stay composite when generality across backends and dtypes matters more than the last 20% — because every hand-written kernel is a permanent maintenance and testing obligation on this team. Framing the answer as a maintenance cost, not just a perf win, is precisely the lens a core team hires for.

Part III — Autograd, the machine

The engine, opened.

Eight chapters on the mechanism the user asked about and every core interviewer assumes: what the graph is, how the engine executes it, why in-place operations are dangerous, how the three "off switches" differ, and how backward interacts with CUDA streams. Mechanism first — because every autograd error message in existence is a direct readout of one of these mechanisms.

10The graph: grad_fn, Node, edges

FORWARD BUILDS THE GRAPH AS A SIDE EFFECT (no tape replay, no separate "compile" step — the graph IS built by the autograd layer of §06, one Node per differentiable op, as the op runs). x (leaf, requires_grad) w (leaf, requires_grad) │ │ └──────── MulBackward0 ◀───────┘ y = x * w │ saves: x, w (needed for the derivative) ▼ AddBackward0 z = y + b │ saves: nothing (d/dy = 1) ▼ SumBackward0 loss = z.sum() │ loss.grad_fn ─── the ROOT the engine starts from NODE ANATOMY (torch/csrc/autograd/function.h): next_edges_ : [(Node*, input_nr), ...] — where MY output grads go saved tensors: SavedVariable (records value + VERSION, see §14) apply(grads) : the derivative, from derivatives.yaml codegen (§09) LEAF EDGES point at AccumulateGrad nodes (§11) — the graph's exits. ───────────────────────────────────────────────────────────── The graph is built FORWARD and traversed BACKWARD. It is per-iteration and disposable: unless retain_graph=True, running backward FREES the saved tensors (that's the memory model of eager training).
FactConsequence you can be quizzed on
Only ops on requires-grad inputs recordThe autograd layer checks and skips — that's why inference is cheap without any special mode, and why no_grad is an optimization, not a correctness requirement.
Nodes save what the derivative needsadd saves nothing; mul saves both inputs; relu saves its output (the mask is recoverable from it). Saved tensors are why activations dominate training memory — the graph is holding them alive for backward. Checkpointing (§16) trades compute to break exactly this.
grad_fn is on the output tensorReading t.grad_fn in a debugger tells you which op produced a tensor — a genuinely useful debugging habit (and t.grad_fn.next_functions walks the graph backwards by hand).

11Leaves, requires_grad, AccumulateGrad

ConceptMechanism
Leaf tensorA tensor with no grad_fn — created by the user (or by detach), not by a differentiable op. Parameters are leaves. Only leaves (by default) get .grad populated.
AccumulateGradThe Node at every leaf edge. Its apply writes into param.grad: if .grad is None it steals the incoming buffer; otherwise it adds. That "otherwise it adds" is the entire reason you must call zero_grad() — gradient accumulation is not a feature bolted on, it's the default behavior of the leaf node.
zero_grad(set_to_none=True)Now the default. Sets .grad = None rather than zero-filling: skips a kernel, frees the memory, and makes the next accumulation a steal instead of an add. Side effect worth knowing: optimizers must (and do) handle None grads.
Non-leaf .gradNot populated (you get a warning). Use t.retain_grad() to force it, or torch.autograd.grad(out, [t]) to get it without touching .grad at all.
requires_grad propagationAny op with at least one requires-grad input produces a requires-grad output. Freezing = set requires_grad=False on params, which prunes those subgraphs from ever being built.

12The engine: dependencies & queues

This is the chapter that separates "I've used autograd" from "I know how it executes." The engine (torch/csrc/autograd/engine.cpp) is a small, elegant, parallel graph executor.

loss.backward() → Engine::execute(roots, inputs, ...) ───────────────────────────────────────────────────────────── 1. SEED grad_outputs for the roots (implicit ones_like for a scalar loss; explicit gradient= required otherwise — because "gradient of a non-scalar" needs a covector) 2. DISCOVER compute_dependencies(): DFS from the roots counting, for each reachable Node, HOW MANY incoming edges it has within this backward. That count is its "not yet ready" counter. 3. EXECUTE ready queue per DEVICE (a worker thread per CUDA device; CPU work reuses the calling thread): pop a ready Node → InputBuffer holds its accumulated incoming grads → node.apply(grads) ← the derivative formula runs → for each next_edge: add the produced grad into the consumer's InputBuffer; decrement its counter; when counter hits 0 → push to that device's queue 4. FINISH AccumulateGrad nodes write .grad on leaves (§11); if `inputs=` was given, grads are returned instead ───────────────────────────────────────────────────────────── KEY PROPERTIES: • it's a topological execution, but NOT a sequential list — nodes become ready as their gradient inputs arrive, so independent branches can run concurrently across device queues • a node with TWO consumers (a tensor used twice) waits for BOTH gradients and SUMS them — that's the multivariable chain rule, implemented as the InputBuffer accumulate • the engine is C++; Python-defined Function nodes take the GIL when they run (a real perf consideration for custom Functions) • REENTRANT backward (backward called inside backward, e.g. double backward or a custom Function that calls .backward()) is handled by depth-tracking and spilling to a thread pool past a depth limit — otherwise deep reentrancy would blow the stack
The question this chapter answers instantly

"Why does backward need to know dependencies before running?" Because a Node must not fire until all the gradients flowing into it have arrived — a tensor reused in three places contributes three gradient terms that must be summed first. The dependency count is how the engine knows when the sum is complete. Say that, and then say what it buys: correct chain rule for reused tensors, plus parallel execution of independent branches. Two sentences, and you've described a graph scheduler most candidates can't describe at all.

13backward vs grad; double backward

APIloss.backward()torch.autograd.grad(out, inputs)
Writes .grad?Yes — accumulates into leaves via AccumulateGradNo — returns the gradients as tensors
ScopeWhole graph reachable from the rootOnly what's needed for the requested inputs
Typical useTraining loopsHigher-order grads, penalties (gradient penalty in GANs), meta-learning, per-sample analysis
FlagMechanism & the error it explains
retain_graph=TrueBackward normally frees the saved tensors as it goes (that's the memory release you rely on). Retaining keeps them so you can run backward again. The famous error — "Trying to backward through the graph a second time, but the buffers have already been freed" — is not a bug in your code's logic; it's the engine telling you it already released the activations. The right fix is usually not to sprinkle retain_graph=True (which silently balloons memory) but to ask why backward is running twice.
create_graph=TrueMakes the backward pass itself differentiable — the derivative computations get recorded as ordinary ops, producing a graph you can backward through again. That's double backward, and it's how gradient penalties and Hessian-vector products work. Cost: the backward's intermediates now live in a graph too.
Double backward correctnessAn op supports double backward only if its backward is written in differentiable ops. Custom Functions marked @once_differentiable declare that they aren't. This is a genuine op-coverage matrix on a core team, and gradgradcheck is how it's tested (§16).

14In-place, version counters & views

The most-hit autograd error in the world, and it has a precise mechanism. Know it cold — Piotr has answered this question on the forum more times than anyone alive.

THE HAZARD: backward needs the VALUES the forward saw. In-place ops overwrite those values. MECHANISM: every TensorImpl has a VERSION COUNTER. • an in-place op bumps it (via the ADInplaceOrView layer) • SavedVariable records the version AT SAVE TIME • at backward, the saved version is CHECKED against the current version → mismatch ⇒ hard error, not silence THE MESSAGE (read it like a diagnostic, because it is one): "one of the variables needed for gradient computation has been modified by an inplace operation: [torch.cuda.FloatTensor [64,128]], which is output 0 of ReluBackward0, is at version 2; expected version 1 instead." → the tensor is the OUTPUT of ReLU → something wrote into the ReLU output after ReLU saved it → classic: a fused residual add done in-place, or `x += y` where x is still needed ───────────────────────────────────────────────────────────── VIEWS MAKE IT SHARPER: a view SHARES its base's version counter. Writing to a view bumps the base's version too — so an in-place write through a slice can invalidate a save of the WHOLE tensor. And the view's own grad_fn gets REBASED (to a CopySlices-style node) so the backward math stays right. ───────────────────────────────────────────────────────────── WHY ALLOW IN-PLACE AT ALL? Memory. `relu_()`, `add_()`, in-place optimizer math and foreach/fused steps (Vol. III §20-21) exist to avoid materializing another tensor. Autograd's job is to let you have it whenever it's SAFE and to fail LOUDLY when it isn't — the version counter is that guarantee, and it never gives a wrong gradient silently. That property is worth stating out loud: PyTorch chose a hard error over a silent wrong number.
SituationSafe?
x = x + y in a loopAlways safe — makes a new tensor; costs memory.
x += y where x isn't saved by any nodeSafe. Autograd allows it and the version bump bothers nobody.
relu_(x) on the output of a conv that saved xUnsafe → the error above. (This is why nn.ReLU(inplace=True) is safe after conv/BN but not everywhere — those ops don't need their output saved... whereas ReLU itself does, which is why chained in-place activations bite.)
In-place on a detach()ed tensorDangerous — detach shares storage and the version counter, so mutating the detached alias can still invalidate the original's saves. The safe idiom is .detach().clone().
In-place inside no_grad on a leaf paramSafe and normal — it's exactly what optimizers do (p.add_(d, alpha=-lr) under @torch.no_grad()).

The eval-loop bug this explains

The classic — validation without no_grad — is not just "slower": every forward builds a graph and holds activations alive until the tensors go out of scope, so memory climbs and can OOM mid-eval. And it's a different bug from forgetting model.eval(), which leaves dropout on and lets BatchNorm keep updating its running stats on validation data (silent train/val contamination). Two bugs, two mechanisms, one line apart. Vol. IV §13 plants both on purpose.

16Custom Functions & saved-tensor hooks

class MyReLU(torch.autograd.Function):
    @staticmethod
    def forward(x):                       # new style: pure forward…
        return x.clamp(min=0)

    @staticmethod
    def setup_context(ctx, inputs, output):   # …ctx setup separated
        (x,) = inputs
        ctx.save_for_backward(output)     # save the OUTPUT: mask is recoverable

    @staticmethod
    def backward(ctx, grad_out):
        (y,) = ctx.saved_tensors
        return grad_out * (y > 0)         # one grad per forward input

# verify against numerical derivatives in float64 — the standard of proof
torch.autograd.gradcheck(MyReLU.apply, (torch.randn(8, dtype=torch.double,
                                                    requires_grad=True),))
MechanismWhy it exists
save_for_backward (not ctx.x = x)Saved tensors get version-checked (§14) and participate in graph-freeing and saved-tensor hooks. Stashing tensors as plain attributes bypasses all of that — a correctness and memory bug that looks like it works.
setup_context splitSeparating the math from the ctx bookkeeping is what lets vmap/functorch and the compile stack transform custom ops. If asked "how do I make a custom op work with torch.compile," the modern answer is the custom-op registration path (torch.library) with a meta/fake implementation so tracing can compute shapes without running the kernel.
once_differentiableDeclares no double backward — the honest annotation instead of a wrong second derivative.
mark_dirtyTells autograd you mutated an input in place, so version-counter bookkeeping stays coherent.
gradcheck / gradgradcheckFinite differences in float64 vs the analytical gradient — the accepted proof that a derivative is right. On a core team, "did you gradcheck it?" is the standard first review question. Say it before they ask.
saved_tensors_hooksPack/unpack callbacks on every saved tensor — the mechanism behind activation offloading (pack to CPU, unpack on demand) and a favorite lever for memory-vs-bandwidth tradeoffs. Its cousin, torch.utils.checkpoint, instead drops saves and recomputes the forward during backward — trading FLOPs for activation memory.

17Autograd meets CUDA streams

RuleConsequence
Backward ops run on the stream their forward op ran onThe engine records the stream per node and re-establishes it, inserting the needed event syncs. So multi-stream forwards don't silently corrupt backward — but it also means your custom stream juggling in forward is inherited by backward, which is where exotic bugs live.
Backward runs on engine worker threadsNot the Python thread. So a stream you set with with torch.cuda.stream(s): around loss.backward() does not automatically apply to the backward kernels the way people assume — the engine's per-node stream bookkeeping governs.
Nondeterminism enters hereBackward for scatter-like ops (index_add, embedding, scatter_add with duplicate indices) uses atomics → the summation order varies run to run → bitwise-different gradients. This is the #1 source of "my training isn't reproducible" (Vol. VI §19).
Grad accumulation is a sumWhich means it's floating-point-order-dependent too: the same math, summed in a different order, gives a different last bit. Correct, not broken — and the distinction (§Vol. VI §17) is the whole skill.

Autograd never gives a silently wrong gradient. It gives you a hard error, an atomics-shaped nondeterminism, or the right answer — and knowing which is which is the job.

Part IV — ATen kernels

From lambda to launch.

Xiang Gao's machine. TensorIterator is the piece of PyTorch that does everything except the math — and then hands a lambda to a launcher that does the math on a few thousand threads. Five chapters: the iterator, the elementwise loop, reductions, structured kernels, and adding an op end to end.

18TensorIterator: the machine before the kernel

TensorIteratorConfig → build() → everything a kernel needs: ───────────────────────────────────────────────────────────── 1 check memory OVERLAP (is an input also the output? alias?) 2 compute SHAPE broadcasting all operands together 3 compute STRIDES per-operand, in the broadcast shape 4 REORDER dimensions sort dims so the fastest-varying is innermost → coalesced memory access 5 compute TYPES common dtype (§03), decide casts; fuse the cast into the load if it can, else emit a separate copy kernel 6 ALLOCATE outputs (if not provided) 7 COALESCE dimensions merge adjacent dims that are contiguous → a 4-D contiguous op becomes 1-D! 8 decide 32-BIT INDEXING if the tensor fits in int32 offsets, use them (faster address math); else split into sub-iterators that do ───────────────────────────────────────────────────────────── THEN: gpu_kernel(iter, lambda) launches. The KERNEL AUTHOR writes only the lambda — broadcasting, promotion, layout, and indexing are already solved. That's the whole design: ONE iterator, HUNDREDS of ops.
Why each step matters in an interview
CoalescingExplains why a contiguous elementwise op on a 4-D tensor is as fast as on a flat one — the iterator collapses it to 1-D and the kernel is a simple grid-stride loop.
Dim reorderingExplains why a transposed (non-contiguous) input isn't catastrophic: the iterator reorders for locality rather than forcing a copy.
Type computationExplains the "extra copy kernel" bug class (§03) — this is the code that decides whether your cast is free or costs a full pass over memory.
32-bit indexingA real optimization with a real cliff: cross the int32 offset limit and the kernel changes shape. A lovely detail to know exists.

19gpu_kernel & the vectorized loop

Xiang Gao's PR that added vectorized memory access to the TensorIterator GPU loop is one of those changes whose reasoning you should be able to reconstruct from first principles — because it's exactly the CUDA reasoning you already do.

gpu_kernel(iter, f) picks a launch strategy: ───────────────────────────────────────────────────────────── CONTIGUOUS + ALIGNED inputs/outputs → vectorized_elementwise_kernel each thread loads a VECTOR (e.g. 4 halves = one 64-bit load, or float4 = 128-bit), applies f element-wise, stores a vector WHY: elementwise ops are MEMORY-BOUND. One 128-bit load beats four 32-bit loads: fewer instructions, fewer memory transactions, better bandwidth utilization at the same occupancy. The math was never the bottleneck. NON-CONTIGUOUS / mixed strides → unrolled_elementwise_kernel + OffsetCalculator each thread computes per-operand byte offsets from the iterator's stride metadata (divmod-based), still unrolled to hide latency, but without vector loads. ───────────────────────────────────────────────────────────── ALIGNMENT is a precondition, not a detail: a vector load requires the base pointer to be suitably aligned — which is one reason the allocator's alignment guarantees matter (Vol. VI §02), and why an oddly-offset view can silently take the slow path.
Your genuine hook here

You've written CUDA kernels and cared about coalescing, vectorized loads, and occupancy for years — gpu_kernel is that knowledge expressed as framework infrastructure. If the conversation goes to elementwise performance, you can reason as a peer: memory-bound op → maximize bytes-in-flight per thread → vector loads when alignment permits → fall back to offset math when the layout is hostile. That's not memorized; that's your own CUDA instinct pointed at ATen.

20Reductions, the hard ones

ProblemHow ATen solves it
Parallel sum needs cross-thread communicationMulti-stage: thread-local partials → warp shuffle reduce → block shared-memory reduce → cross-block (a second pass or atomics with semaphores). gpu_reduce_kernel plus a ReduceConfig that chooses the shape of that tree from the input's geometry.
PrecisionAccumulate in a wider type (acc_type/opmath_t): summing 10⁶ fp16 values in fp16 loses catastrophically; accumulating in fp32 doesn't. This is why a GPU sum can differ from a naive CPU sum — and why it's often more accurate, not less.
Numerical stabilityWelford's algorithm for variance/std (one-pass, stable); log-sum-exp shifting for softmax. Stability is a design constraint, not an afterthought.
DeterminismTree reduction order depends on the launch geometry, which can depend on the input shape → different shapes, different rounding. Atomics-based cross-block reductions are worse: order varies run to run. Hence Vol. VI §19.
Why reductions are the flaky-test capitalEvery one of the above interacts with hardware: new arch → different occupancy → different config → different order → different last bits → a tolerance-tuned test tips over. When Wei says "this test only fails on one GPU type," a reduction is a prime suspect (Vol. II §15).

TORCH_IMPL_FUNC(add_out_cuda) (const Tensor& self, const Tensor& other,

PieceWhat it buys

Part V — Beyond eager

Where the graph goes.

Three chapters on the two things that consume the autograd graph — compilers and distributed training. Both are staffed on this exact team (Ivan Yashchuk: "PyTorch Compilers @ NVIDIA"; Xiang Gao: nvFuser; Masaki: Thunder), so know the shape and be candid about the depth.

23torch.compile: dynamo → AOT → inductor

torch.compile(model)(x) ───────────────────────────────────────────────────────────── 1 TORCHDYNAMO hooks CPython frame evaluation; symbolically executes the bytecode and extracts an FX GRAPH of tensor ops, plus GUARDS (assumptions: shapes, dtypes, types, python values). Anything it can't capture → GRAPH BREAK → that region runs eager. Guard fails at runtime → RECOMPILE. 2 AOTAUTOGRAD traces the graph AND ITS BACKWARD ahead of time (via __torch_dispatch__ + FakeTensor + meta funcs §21) into a JOINT forward+backward graph, then FUNCTIONALIZES it (§06: mutations/views removed), then PARTITIONS it into fw/bw graphs — the min-cut partitioner deciding what to SAVE vs RECOMPUTE. ⇒ autograd is now a compiled artifact, not an engine walk. This is the deep idea. 3 INDUCTOR lowers the graphs to code: TRITON kernels on GPU (fusing elementwise/reduction chains), C++/OpenMP on CPU; optionally wraps in CUDA graphs (mode="reduce-overhead") to kill launch overhead. ───────────────────────────────────────────────────────────── DEBUG SURFACE (know these exist): TORCH_LOGS=graph_breaks,recompiles, output_code · torch._dynamo.explain · TORCHINDUCTOR_CACHE_DIR · "why is my compiled model slow?" is usually GRAPH BREAKS or RECOMPILATION THRASH, not bad kernels.
The insight worth carrying
Why functionalization existsA compiler cannot reason about a graph full of aliasing in-place mutations (§14 is a runtime guarantee, not a static one). Functionalize rewrites them away so the graph is pure — and then the partitioner is free to move things.
Why the meta/fake path mattersTracing must compute shapes without executing kernels — which is exactly what the meta functions of §21 provide. An op without a meta/fake impl breaks compile. That's the connective tissue between "adding an op" and "compile works."
Save-vs-recomputeThe partitioner is making the same trade as activation checkpointing (§16), automatically: memory vs FLOPs. Naming that equivalence out loud is a strong move in a compiler-adjacent room.

25Distributed's hooks into autograd

DDP IS AN AUTOGRAD PLUGIN — this is the mechanism people miss: ───────────────────────────────────────────────────────────── construction: Reducer registers a post-accumulate-grad HOOK on every parameter, and packs params into BUCKETS (~25 MB), built in REVERSE registration order to approximate the order gradients become ready backward: the autograd engine (§12) fires each param's hook the moment that param's grad is finalized → when ALL params in a bucket are ready, DDP launches that bucket's ALLREDUCE asynchronously → so communication OVERLAPS with the rest of backward instead of waiting for it. The overlap IS the design. end of step: wait on the outstanding allreduces; grads are now the average across ranks; optimizer steps identically everywhere (no parameter broadcast needed) ───────────────────────────────────────────────────────────── FAILURE MODES THAT FALL OUT OF THIS DESIGN: • a param that gets no grad on some ranks (control flow / unused head) ⇒ its bucket never completes ⇒ THAT RANK NEVER LAUNCHES the allreduce ⇒ the others block forever = the classic DDP HANG. find_unused_parameters=True fixes it by traversing the graph to mark unused params ready — at real cost. static_graph=True is the "the graph never changes" fast path. • ranks executing DIFFERENT NUMBERS of collectives (a conditional branch, an early `continue`, uneven dataset lengths) ⇒ desync/hang • bucket size trades overlap granularity against allreduce efficiency ───────────────────────────────────────────────────────────── FSDP: shards params/grads/optimizer state across ranks; all-gathers weights just-in-time in forward, reduce-scatters grads in backward — same "hook into autograd, overlap comm with compute" idea, applied to memory rather than just gradients.
Your genuine standing

You've built distributed PyTorch/Ray systems and driven a 400× speedup — so speak to DDP from the systems side you actually lived: overlap, bucketing, stragglers, and the debugging of hangs. And cross-reference Vol. I §14–18 for NCCL depth. When the mechanism above is what you can derive rather than recite, a distributed question becomes home turf rather than a quiz.

Part VI — Drills & honesty

Say it out loud.

Fifteen questions a core interviewer actually asks, each with the shape of a strong answer; how to navigate the source live if someone shares a file; and the honest map of where your depth ends — written so you can state it before anyone has to find it.

26Fifteen core questions, answered

Q1What actually happens when I call a + b?

Walk §05's seven steps. Land on the punchline: the dispatcher is a priority-ordered layer stack driven by a key set the tensors carry; each layer does its job and re-dispatches downward.

Q2How does autograd know what to differentiate?

It doesn't "know" — the forward builds the graph as a side effect (§10). The autograd dispatch layer, for any op with a requires-grad input, constructs a Node whose apply is the codegen'd formula from derivatives.yaml, wires next_edges to the inputs' grad_fns (or their AccumulateGrad), saves what the derivative needs, and stamps grad_fn on the output.

Q3Walk me through loss.backward().

§12: seed the roots → compute_dependencies counts incoming edges per node → per-device ready queues; a node fires only when all its incoming gradients have arrived (InputBuffer sums them = the multivariable chain rule) → apply → push gradients to next_edges, decrement counters → AccumulateGrad writes leaves. Independent branches run concurrently; reentrancy spills to a thread pool.

Q4Why do I have to zero gradients?

Because AccumulateGrad adds into .grad when it isn't None (§11) — accumulation is the leaf node's defined behavior, which is precisely what makes gradient accumulation across micro-batches work for free. set_to_none=True makes the next write a steal instead of an add, skipping a kernel and freeing memory.

Q5"Modified by an inplace operation" — explain the mechanism.

§14: version counters on TensorImpl, versions recorded in SavedVariable at save time, checked at backward. Then read the message like a diagnostic — it names the tensor, its producing Node, and the two versions. Close with the design point: PyTorch chose a hard error over a silently wrong gradient.

Q6no_grad vs inference_mode vs detach?

§15. Three different mechanisms: a TLS key exclusion; a stronger mode that also drops view/version bookkeeping and marks its outputs as inference tensors (which can't later enter autograd); and a graph cut that shares storage and the version counter (hence .detach().clone()).

Q7How does AMP/autocast work, and why is it differentiable?

§06's derivation: autocast is a dispatch layer above autograd, so the casts it inserts are themselves ops recorded by the autograd layer below — which is exactly how gradients flow back to fp32 master weights. Per-op policies (matmul/conv down, reductions/norms in fp32, others promote-to-widest), plus a cast cache within the region. GradScaler handles fp16's underflow range; bf16 usually doesn't need it.

Q8What does TensorIterator do?

§18: everything except the math — overlap checks, broadcasting, stride computation, dim reordering for locality, type promotion and cast placement, output allocation, dimension coalescing, 32-bit-indexing selection. Then gpu_kernel takes a lambda and launches a vectorized elementwise kernel when contiguous and aligned, or an unrolled offset-calculator kernel otherwise (§19).

Q9Why is my GPU result different from CPU?

Not "GPUs are inaccurate." Name the real mechanisms: accumulate types (§20 — fp16 sums accumulate in fp32, often more accurate than a naive CPU loop), different reduction orders (tree vs sequential), TF32 on matmul, fused multiply-add contraction, and library algorithm selection. Then the discipline: compare both against a float64 reference and judge against the format's error bounds — "differently right" vs wrong (Vol. VI §17).

Q10How would you add a new op with a CUDA kernel?

§22's eight steps — schema, meta, kernels, derivative, OpInfo tests, fake/meta for compile, docs, CI. Then the sentence that gets you hired here: an op is not a function; it's a schema, a kernel per backend, a derivative, a test matrix, a compile path, and a permanent maintenance obligation across every GPU NVIDIA ships.

Q11What's a structured kernel, and why?

§21: split the op into a meta function (shapes/dtypes/allocation, no data) and an impl (math on a pre-built iterator). Codegen fans out functional/in-place/out variants from one implementation, and the meta function gives you the meta device — which is what lets FakeTensor and the compile stack trace without a GPU.

Q12When is a composite op the right choice?

§09's tradeoff, framed as maintenance: composite inherits autograd and all backends for free; a fused kernel wins when intermediates dominate memory/bandwidth, when launch count matters, or when stability demands a single-pass formulation — at the cost of a permanent per-backend, per-dtype obligation.

Q13How does DDP overlap communication with backward?

§25: DDP is an autograd plugin — post-accumulate-grad hooks on every param, buckets built in reverse registration order, a bucket's allreduce fires the moment its last gradient is ready, so comm rides alongside the remaining backward. Then name the hang: a param that gets no grad on some rank means that bucket never completes and that rank never launches its collective.

Q14Why does torch.compile need functionalization?

§06 + §23: a compiler can't safely reorder or partition a graph full of aliasing mutations, so Functionalize (a dispatch layer) rewrites them into pure ops. AOTAutograd then traces a joint forward+backward graph and min-cut-partitions it, deciding what to save versus recompute — the same memory-vs-FLOPs trade as checkpointing, made automatic.

Q15How do you know a derivative is correct?

gradcheck in float64 against finite differences, gradgradcheck for double backward, and an OpInfo entry so the generic suites hammer it across dtypes/devices/meta/compile (§16, §22). On this team, "did you gradcheck it?" is the first review question — so volunteer it before you're asked.
