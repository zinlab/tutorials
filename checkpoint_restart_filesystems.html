Checkpoint & restart
across file systems

A long simulation on a shared cluster is a bet against the wall-clock limit, a flaky node, and a 60-day purge. Winning the bet is less about writing state than about publishing it atomically, laying it out to match the file system's parallelism, and resuming as if nothing happened. A field guide — grounded in snapy's restart path — to the concept, the implementation, the file-system tuning, and the generalization.

Part I

The problem & the concept

What state is, why it must be saved, and why the file system underneath turns a one-line feature into a design problem with three good answers and a dozen sharp edges.

§00Synopsis & what we shipped

Running the Straka density-current case (and larger) on a Slurm cluster, we need to save state periodically so a job killed by the wall-clock limit, a preemption, or a node failure can resume from the last good point rather than from t = 0. snapy already had a working restart path; the work here hardened it for scale and for different file systems.

ChangeWhat & whyFS payoff
combine: falsefile-per-rank restart dumps — no cross-rank barrier, no serial root-rank bundlingparallel on Lustre/GPFS
atomic publishwrite to .tmp, then rename() over the destinationcrash-safe last good ckpt
per-rank resumeeach rank rewrites the .block<N>. token to its own rank and loads its own filepass any one file to --restart
signal path (pre-existing)SIGTERM → cross-rank flag → forced final checkpoint at the next safe stepgraceful preemption

A checkpoint is not “write the arrays to disk.” It is a transaction: capture a consistent state, publish it atomically, lay it out to match the storage's parallelism, and make the resume idempotent. Get those four right and the file system is just a tuning knob; get any wrong and you lose runs to silent corruption, purges, or hangs.

§01The problem: betting against the wall clock

A production CRM run can take days. The cluster gives you, at most, a 24–48 h job slot, reserves the right to preempt you for a higher-priority job, and runs on hardware where a single node fault ends the whole MPI world. Three distinct adversaries:

  • The wall-clock limit. Predictable. Slurm sends SIGTERM, waits a grace period, then SIGKILL. If you catch the signal you can flush a final checkpoint on the way out.
  • Preemption / requeue. Same signal mechanism, unpredictable timing. The defense is identical: a signal-driven final write, plus a periodic checkpoint stream as backstop.
  • The hard crash. SIGKILL, OOM, ECC error, network partition, NFS hiccup. No warning, no graceful flush. Your only defense is the most recent periodic checkpoint — and that it is not half-written.

So the requirement is two-layered: a periodic checkpoint on a sim-time cadence (survives hard crashes) and a signal-triggered final checkpoint (survives graceful preemption with the least lost work). Both must produce a file that is either complete or absent — never truncated.

The corruption trap

The naive implementation writes directly to run.00007.restart. If the job is killed mid-write — precisely the scenario checkpoints exist for — you now have a truncated file and you may have deleted the previous good one to make room. The single most important property of a checkpoint is that a kill at any instant leaves the last good state intact.

§02Anatomy of a checkpoint

What actually needs to be on disk to resume bit-for-bit? In snapy the restart record is the conserved state plus enough bookkeeping to continue the run and its output schedule without a seam.

FieldRole on resume
hydro_uconserved hydro variables — the actual physical state
scalar_sconserved passive-scalar state (when present)
last_timesimulation time to resume from
last_cyclecycle index — so nlim-based runs don't take an extra step
file_numberper-output-stream counter — so NetCDF frames keep numbering, no clobber
next_timeper-output-stream next-write time — so the cadence resumes cleanly

Each rank's record is a tensor map serialized by kintera::save_tensors. The question that drives everything downstream is how those per-rank records are arranged on disk.

Two layouts: bundle vs. file-per-rank

Combined (bundle)

Every rank writes a .part; the root rank waits on a barrier, globs them, concatenates into one run.NNNNN.restart bundle, deletes the parts. One file per checkpoint — tidy, few inodes, easy to archive.

But: a barrier + a single-stream serial copy on one rank. That throttles the whole FS to one writer.

File-per-rank (uncombined)

Every rank writes its own run.block<rank>.NNNNN.restart directly. N files per checkpoint, no barrier, no root bottleneck.

Each writer hits storage independently — the canonical parallel-I/O pattern. The cost is more inodes and a many-file listing.

COMBINED FILE-PER-RANK rank0 ─┐ .part rank0 ──► run.block0.00007.restart rank1 ─┤ .part ╮ barrier rank1 ──► run.block1.00007.restart rank2 ─┤ .part ├─► root: glob+cat rank2 ──► run.block2.00007.restart rank3 ─┘ .part ╯ ──► 1 bundle rank3 ──► run.block3.00007.restart ▲ serial, one writer ▲ N independent writers

The choice is not stylistic — §03 shows it is dictated by the file system. The contribution of this work was to make it a combine: true|false switch instead of a hard-wired true.

§03The file-system landscape

A “parallel scratch file system” is not one thing. The two you will actually meet — GPFS at U-M Great Lakes, Lustre at NERSC Perlmutter — differ in ways that flip the right answer to §02.

The detail that flips the design — Lustre default striping

On Perlmutter, files are striped across a single OST by default (out of 370). A monolithic bundle written by one root rank to one un-striped file therefore targets one server out of 370 — you would get a few GB/s out of a multi-TB/s file system. On Lustre the combined-bundle layout is an anti-pattern unless you explicitly stripe the file wide. File-per-rank is the native fast path.

Two more facts that the implementation has to respect:

  • Where files land is the process CWD. snapy's output_dir defaults to "." — nothing routes output to scratch automatically. You must cd into the scratch run directory before launching, or the run writes wherever sbatch started it (often $HOME, the worst place).
  • The combine step needs a shared FS. The root rank globs every rank's .part. On multiple nodes those parts must be visible to the root — i.e. on a globally-mounted parallel FS, never node-local /tmp. File-per-rank reads back per-rank on resume, so it has the same shared-FS requirement, but no barrier to stall on.
Part II

Strategy & implementation

The five decisions that define a checkpoint regime, the code that implements them in snapy, and the per-file-system tuning that makes them fast.

§04Strategy: the five decisions

Every checkpoint design answers the same five questions. Naming them explicitly turns “add checkpointing” into a tractable checklist.

DecisionThe questionsnapy answer
1 · CadenceHow often? In sim-time or wall-time?restart dt (sim-seconds) + final on signal
2 · LayoutOne bundle, or file-per-writer?combine switch, FS-driven (§05)
3 · AtomicityWhat does a mid-write kill leave behind?temp + rename (§06)
4 · ResumeSelf-describing & idempotent?state carries time/cycle/counters (§07)
5 · RetentionKeep how many? Archive where?keep-last-N + copy keepers off scratch

Cadence is a sim-time number; retention is a wall-time-and-quota policy; atomicity is a correctness invariant. Conflating them is the root of most checkpoint bugs.

Sizing the cadence

Pick the restart dt so that checkpoint write time « checkpoint interval. A useful rule: lose at most one checkpoint interval of work to a hard crash, and spend at most a few percent of run time writing. If a full-state write takes T_w and you tolerate overhead f, then interval ≥ T_w / f. The signal-driven final write is the safety net that lets you make the periodic cadence coarse.

§05Implementation — file-per-rank vs. bundle

The combined path was hard-wired. The fix respects the YAML combine option (default true, so nothing changes for existing runs) and branches the write:

// build the shared stem: <output_dir>/<basename>.<blockid>.<file_number|final>
std::string stem = output_dir + "/" + basename + "." + blockid + "." +
                   (final_write ? "final" : number);

if (options->combine()) {
  // Each rank writes a per-block ".part"; the root rank bundles all
  // parts into one combined ".restart" file.
  save_tensors_atomic(out_vars, stem + ".part");
  combine_blocks(pmb, final_write);
} else {
  // File-per-rank: each rank writes its own ".restart" directly, with no
  // cross-rank barrier or serial bundling. On restart every rank loads
  // its own file (the block id is rewritten to the local rank).
  save_tensors_atomic(out_vars, stem + ".restart");
}

The block id is the global block rank (proc_rank × blocks_per_process + local_block), so per-rank filenames never collide — even with several blocks per process.

Turning it on is one line of YAML:

outputs:
  - type: restart
    dt: 300.           # sim-seconds between checkpoints
    combine: false     # file-per-rank — recommended for large Lustre/GPFS runs
When to pick which

combine: false for frequent, in-run checkpoints at scale — especially on Lustre, where it is the only way to use more than one OST without manual striping. combine: true for the occasional archival snapshot you will copy to CFS/Turbo/HPSS, where one tidy file beats thousands of inodes. A common regime: uncombined periodic stream + a combined final.

§06Implementation — atomic publish

The correctness invariant from §01. Every write goes to a temporary name, then a single rename() publishes it. POSIX rename within one file system is atomic and replaces the destination in one step — so the previous good checkpoint survives until the new one is fully on disk.

// Write tensors to a temp file and atomically rename into place, so a crash
// mid-write can never corrupt an existing restart file.
void save_tensors_atomic(Variables const& vars, std::string const& final_path) {
  std::string tmp_path = final_path + ".tmp";
  kintera::save_tensors(vars, tmp_path);          // 1. whole file to a temp name
  std::error_code ec;
  std::filesystem::rename(tmp_path, final_path, ec);  // 2. atomic publish (same FS)
  if (ec) {
    std::filesystem::remove(tmp_path);                  // clean up; never leave a torn file
    throw std::runtime_error("Failed to finalize restart file '" + final_path + "'");
  }
}

The same discipline fixes the bundle path. The old code did remove(outfile) before writing the new bundle — a window where a crash left you with neither. The new code bundles into a temp and renames, and never deletes the previous bundle first:

if (file_list.size() == 1) {
  err = std::rename(file_list.front().c_str(), outfile.c_str());   // atomic replace
} else {
  std::string tmp = outfile + ".tmp";
  err = make_restart_bundle(tmp, file_list);                  // build under temp name
  if (err == 0) {
    std::filesystem::rename(tmp, outfile, rn_ec);              // then atomic publish
    if (rn_ec) { std::filesystem::remove(tmp); err = -1; }
  }
}
Atomicity is per-file, not per-checkpoint

rename makes each file appear atomically. In file-per-rank mode a checkpoint is N files, and a kill can land between rank 3's and rank 4's rename — leaving a checkpoint number that is complete for some ranks, missing for others. The loader fails fast on a missing file, so you simply fall back to the previous fully-complete number. If you need all-or-nothing across ranks, gate the resume on a tiny DONE.NNNNN sentinel that the root writes only after a barrier confirming every rank renamed.

§07Implementation — idempotent resume

Resume must be a no-seam continuation. Two pieces: the state is self-describing (§02 — it carries time, cycle, and the output counters), and each rank loads its own file. The latter is the only subtlety added by file-per-rank: the user passes one path to --restart, and every rank rewrites the block-id token to its own global rank.

// Rewrite ".block<N>." in the filename to this rank's id, so each rank loads
// its own file. Bundles / single dumps (no token) are returned unchanged.
std::string restart_path_for_rank(std::string const& path, int block_rank) {
  fs::path p(path);
  std::string f = p.filename().string();
  size_t pos = f.find(".block");
  if (pos == std::string::npos) return path;            // bundle / single — unchanged
  // ... scan the digit run after ".block", require a trailing '.' ...
  p.replace_filename(prefix + std::to_string(block_rank) + suffix);
  return p.string();
}

The mesh wires this in before resolving the path, so the existing “rebase relative to output_dir” logic works per-rank:

double MeshBlockImpl::_init_from_restart(Variables& vars, std::string fname) {
  // For file-per-rank dumps, resolve to this block's own file.
  fname = restart_path_for_rank(fname, options->layout()->rank());
  // ... load_restart() then restores hydro_u, last_time, last_cycle,
  //     file_number[], next_time[] — the output schedule resumes seamlessly.
}
Decomposition is part of the contract

Records are keyed by global block rank. You cannot reshape the run from a checkpoint — the resume must use the same nproc, nb2/nb3, and blocks_per_process. To change decomposition you need an explicit repartition/remap step (read all blocks, redistribute, rewrite) — a separate tool, not the hot restart path.

§08Per-file-system tuning

The same binary, tuned for each storage tier. The knobs are striping (Lustre), file count (GPFS), and staging (node-local / burst).

Lustre (Perlmutter $PSCRATCH)

Striping is the knob. Set it on the run directory before writing — new files inherit the directory's layout. Use file-per-rank so each writer lands on its own OST(s); if you keep a combined bundle, stripe it wide so the single file spans many OSTs.

# per-rank files: 1 stripe each is correct — they parallelize across ranks
lfs setstripe -c 1 "$PSCRATCH/run"
# OR a combined bundle: spread the single file across many OSTs
lfs setstripe -c 72 -S 16M "$PSCRATCH/run"      # 72 OSTs, 16 MiB stripe
lfs getstripe "$PSCRATCH/run"                    # verify the layout took
lfs df -h | grep -i full                       # are any OSTs near full? (skew)

GPFS (Great Lakes /scratch)

No user-facing striping; the FS handles block placement. The lever is file count and the serial-combine bottleneck. File-per-rank parallelizes the writers but creates many inodes — fine at moderate rank counts. At very high rank counts, the single-root-rank bundle is the bottleneck on both Lustre and GPFS; prefer uncombined and combine only for archival.

Node-local & burst tiers

Node-local NVMe (/tmp) and burst buffers (Great Lakes IME, >80 GB/s) are the fastest tier — but node-local is not shared, so it breaks both the combine glob and the per-rank cross-node resume. Use it only as a write-through stage that you flush to shared scratch:

# single-node only, or with an explicit gather step:
# 1. write checkpoints to node-local NVMe (fast)   2. rsync to shared scratch (durable)
export OUTPUT_DIR="$SLURM_TMPDIR"          # node-local — single node runs
trap 'rsync -a "$SLURM_TMPDIR"/*.restart "$PSCRATCH/run/"' EXIT
Retention vs. the purge

Scratch purges unaccessed files (GL 60 d, PM ~8 wk) and is not backed up. A long campaign's old checkpoints will silently vanish. Keep only the last N on scratch and archive keepers to a persistent tier:

# keep the newest 2 numbered checkpoints; archive every 10th to persistent storage
ls -t run.*.restart | tail -n +3 | xargs -r rm -f
cp run.00010.restart "$CFS/myproj/archive/"   # PM: CFS  |  GL: Turbo

§09Slurm orchestration & auto-resubmit

The signal path is already wired in snapy: a caught SIGTERM sets a flag, check_redo does a cross-rank MAX-allreduce on it, and a positive result breaks the time loop so finalize forces a final checkpoint regardless of the dt schedule. Slurm just needs to deliver the signal early enough, and the script needs to resume from the newest file.

#!/bin/bash
#SBATCH --time=24:00:00
#SBATCH --signal=B:TERM@180      # SIGTERM 180 s before the wall limit
#SBATCH --requeue               # let Slurm requeue on preemption

RUNDIR="$SCRATCH/straka_run"      # shared parallel FS, visible to all nodes
mkdir -p "$RUNDIR"; cd "$RUNDIR"
cp "$SLURM_SUBMIT_DIR/straka.yaml" .
lfs setstripe -c 1 . 2>/dev/null || true   # Lustre: per-rank files, 1 OST each

# newest checkpoint, if any — works for bundle or per-rank (mtime order)
RESTART=$(ls -t straka.*.restart 2>/dev/null | head -1)

torchrun --no-python --nproc-per-node=$SLURM_GPUS_ON_NODE \
  "$SLURM_SUBMIT_DIR/build/bin/straka.release" ./straka.yaml \
  ${RESTART:+--restart "$RESTART"}

# resubmit until the run reports completion (a DONE marker / tlim check)
if [ ! -f DONE ]; then sbatch "$SLURM_SUBMIT_DIR/submit.sh"; fi
Two signal-routing gotchas

(1) --signal=B:TERM@180 sends to the batch shell; because torchrun is the foreground child it receives and forwards the signal. If you wrap the launch in srun, drop the B: (--signal=TERM@180) so every task gets it. (2) The grace period (180 s) must exceed “finish current cycle + write + any combine + barrier.” Size it up for large grids; the periodic stream is the backstop if the grace window is ever too short.

§11Debugging approaches

Checkpoint bugs hide on disk and in the launcher, not in the solver. The first moves are filesystem forensics.

# 1. What's actually there, newest first, with sizes (spot a torn/short file)
ls -lat --time-style=long-iso *.restart *.part *.tmp 2>/dev/null
#    Leftover *.tmp or *.part  ==  a crash mid-write. The .restart is your truth.

# 2. Is the bundle / per-rank file complete & loadable? Inspect from Python.
python -c "import kintera, sys; d=kintera.load_tensors(sys.argv[1]); \
print({k: tuple(v.shape) for k,v in d.items()}); \
print('time', float(d['last_time']), 'cycle', int(d['last_cycle']))" run.final.restart

# 3. Lustre layout & OST skew (slow or ENOSPC despite quota)
lfs getstripe run.00007.restart        # how many OSTs is this file on?
lfs df -h | sort -k5 -h | tail          # any single OST near 100%?

# 4. Launcher / environment faults (exit 127, 'not found')
ldd build/bin/straka.release | grep -i 'not found'
echo "$PWD"; df -h .                     # am I even on scratch? quota left?

# 5. The ground truth: does a restart reproduce the uninterrupted run?
ctest -R restart                        # bit-for-bit resume equivalence (§14)
Debugging mindset — bisect the transaction

A checkpoint is capture → publish → resume. Localize the failure to one stage: Did it write? (ls/sizes) → Is the file complete? (load it) → Does resume read the right file? (per-rank path rewrite, decomposition) → Does it continue seamlessly? (counters, bit-for-bit test). Each stage has a distinct probe; don't jump to the solver.

§13Generalizing the pattern

Strip away snapy and the design is a domain-independent recipe. It applies verbatim to any distributed simulation, and almost verbatim to large-model training. Seven invariants:

InvariantThe general principle
1 · Separate capture from publishSerialize state to a temp object, then make it visible in one atomic step (rename, object-store PUT, a manifest commit). The visible name is always complete.
2 · Layout follows parallelismOne object per independent writer is the native fast path on parallel FS; a single shared object needs explicit striping/sharding or it serializes.
3 · Self-describing stateThe checkpoint carries everything resume needs — logical clock (time/step), and any scheduler counters — so continuation has no seam and no external metadata to drift.
4 · Idempotent, addressable resumeEach worker can find & load its own shard from a single handle; re-running resume yields the same state.
5 · Signal-driven flushCatch the preemption signal, reach the next consistent point, flush once. Decouple from the periodic cadence.
6 · Cadence ≠ retentionHow often you write (work-loss budget) is independent of how many you keep (quota/purge) and where they live (tiering).
7 · Validate by equivalenceThe acceptance test is “resume reproduces the uninterrupted run” — bit-for-bit if deterministic, within tolerance otherwise.

The abstract interface

Any checkpointable distributed component can expose four methods; everything above is policy layered on top of them:

class Checkpointable(Protocol):
    def state_dict(self) -> dict: ...               # 1. capture (+ logical clock, counters)
    def load_state_dict(self, s: dict) -> None: ...   # 3,4. idempotent resume

def save(obj, step, dir, rank):
    tmp   = f"{dir}/ckpt.step{step}.rank{rank}.tmp"     # 2. one object per writer
    final = f"{dir}/ckpt.step{step}.rank{rank}"
    serialize(obj.state_dict(), tmp)
    os.replace(tmp, final)                          # 1. atomic publish (os.replace == rename)

def resume_path(handle, rank):                       # 4. rank finds its own shard
    return handle.replace("rankANY", f"rank{rank}")
Where you've already seen this

PyTorch Distributed Checkpoint (DCP) is exactly invariants 1–4: sharded per-rank writes, a committed metadata manifest (the atomic publish), and reshardable load. MPI-IO / ADIOS2 / HDF5 give one-file-per-rank vs. shared-file-with-collective-I/O — invariant 2 again. DMTCP / CRIU generalize capture to whole-process images. The lesson transfers: the file system, not the math, decides the layout; atomicity and idempotent resume are non-negotiable everywhere.

What changes per domain

  • Determinism. CFD with fixed arithmetic resumes bit-for-bit (invariant 7 is strict). ML training with async ops resumes within tolerance — and must also checkpoint RNG state and optimizer/scheduler state, the ML analogue of file_number/next_time.
  • Resharding. Simulations usually pin decomposition (invariant 4 is “same shape”). DCP and elastic training relax it with a resharding load — worth the complexity only when you genuinely change world size.
  • Storage target. POSIX rename on Lustre/GPFS; a manifest-commit on object stores (S3 has no atomic rename — you commit by writing a small pointer object last). The invariant “publish in one step” holds; the mechanism differs.

Cross-repo note

The default behavior is unchanged: combine defaults to true, existing restart files load as before, and the per-rank path rewrite only triggers on filenames containing .block<digits>. — which only uncombined dumps emit. No change to kintera's serialization format, so stored checkpoints remain loadable.
