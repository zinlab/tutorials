Unified Communication X · Programming Field Manual

Talking to the network at the speed of the hardware.

UCX is the thin, fast layer that sits between your program and the interconnect — InfiniBand, RoCE, NVLink, shared memory, TCP — and lets you move bytes (and GPU bytes) with zero-copy, hardware offload, and automatic transport selection. This manual takes you from first principles to a running ping-pong to the internals that make MPI, NCCL, and RAPIDS all quietly stand on the same foundation.

§ 01What UCX Actually Is

UCX (Unified Communication X) is an open-source communication framework, not an application API you launch jobs with. Its job is to expose the full performance of whatever interconnect you have — without you writing transport-specific code for each one.

The cleanest way to place UCX in your head: it is a networking abstraction layer that a runtime sits on top of. You rarely run a UCX program the way you run an MPI program. Instead, MPI, NCCL, Dask, Spark RAPIDS, and custom shuffle engines (like a GPU exchange operator inside a query engine) call into UCX to do the actual byte movement. UCX answers one question extremely well: "given this source buffer, this destination, and this hardware, what is the fastest way to get the bytes there?"

It was created to solve a real fragmentation problem. Before UCX, every HPC communication library re-implemented the same low-level machinery — InfiniBand verbs handling, shared-memory transports, registration caches, rendezvous protocols — and each did it slightly differently. UCX consolidated that machinery into one well-optimized, vendor-co-developed codebase (NVIDIA/Mellanox, IBM, ORNL, ARM, and others), then exposed it through a clean API so that everyone above it could share the work.

Why you, specifically, care

If you are building a GPU-to-GPU exchange operator — moving column batches between workers over NVLink intra-node and InfiniBand inter-node — UCX is the layer that lets one code path target both, pick zero-copy RDMA for big batches, route through the NIC closest to the GPU, and skip the host bounce entirely with GPUDirect RDMA. The "logical topology compiled from physical topology" framing maps directly onto UCX's transport-and-lane selection engine.

What UCX gives you — and what it deliberately does not

UCX provides

Point-to-point messaging (tag-matched & stream), one-sided RMA (put/get) and atomics, active messages, memory registration, transport & device selection, multi-rail striping, GPU memory awareness, and a progress engine.

UCX does not provide

Process launch / job management, ranks & communicators, collectives (broadcast/allreduce — that's UCC), MPI datatypes, or an out-of-band bootstrap. You bring your own way to exchange initial addresses.

§ 02The Three-Layer Architecture

UCX is three nested libraries. Almost all application code lives at the top layer (UCP); the lower two exist so that UCP can be portable and fast. Knowing the split is essential for reading logs and config, because every environment variable and every error message is scoped to one of these layers.

The stack. You program UCP; UCT abstracts each interconnect; UCS provides shared plumbing.

UCS — Unified Communication Services

The foundation. Pure software utilities with no networking of their own: lock-free queues and lists, memory pools, the asynchronous event engine, configuration parsing (every UCX_* environment variable flows through UCS), debugging and logging, and — critically for performance — the registration cache (rcache). When you see UCX_LOG_LEVEL, UCX_RCACHE_ENABLE, or a backtrace printed on a fault, that is UCS.

UCT — Unified Communication Transport

The thin abstraction over each piece of hardware. UCT defines three object types that every transport driver implements: a memory domain (uct_md, which knows how to register/pin memory for a device), an interface (uct_iface, a communication resource on a device), and an endpoint (uct_ep, a connection to a remote interface). UCT exposes raw capabilities — "this transport can do zero-copy put up to N bytes, has hardware atomics, supports GPU memory" — but it does not decide which to use. UCT is deliberately dumb and fast; the intelligence lives in UCP.

UCP — Unified Communication Protocol

The brain and the public API. UCP takes a single logical operation ("send these 4 MB from this GPU buffer to that peer with this tag") and: fragments it if needed, selects between eager and rendezvous, picks one or several UCT transports (multi-rail), handles unexpected messages, matches tags, manages completion, and tracks the protocol state machine. This is the layer you write code against — virtually every symbol you call starts with ucp_.

Reading-the-logs payoff

An error like UCX ERROR no active messages transport is a UCT-level complaint (no driver matched your request). A message about rndv thresholds or eager is UCP. A backtrace with ucs_ frames is the services layer. The prefix tells you which layer to investigate.

§ 03The Core Objects You Hold

UCP programming is the lifecycle management of five handle types. Internalize what each one is, how many you create, and its threading rules — most beginner bugs are object-lifecycle bugs.

The skeleton of every UCP program

lifecycle.cC
/* 1. Configure: read UCX_* env vars into a config object */
ucp_config_t *config;
ucp_config_read(NULL, NULL, &config);

/* 2. Create the context: declare which features you need.
 *    UCX only initializes machinery for features you request. */
ucp_params_t ucp_params = {
    .field_mask = UCP_PARAM_FIELD_FEATURES,
    .features   = UCP_FEATURE_TAG | UCP_FEATURE_RMA,
};
ucp_context_h context;
ucp_init(&ucp_params, config, &context);
ucp_config_release(config);

/* 3. Create a worker (the progress engine for this thread) */
ucp_worker_params_t worker_params = {
    .field_mask  = UCP_WORKER_PARAM_FIELD_THREAD_MODE,
    .thread_mode = UCS_THREAD_MODE_SINGLE,   /* this worker, one thread */
};
ucp_worker_h worker;
ucp_worker_create(context, &worker_params, &worker);

/* 4. ... establish endpoints, communicate (see later sections) ... */

/* 5. Tear down in reverse order */
ucp_worker_destroy(worker);
ucp_cleanup(context);
The feature-mask discipline

You declare the features you need at context creation (UCP_FEATURE_TAG, _RMA, _AM, _STREAM, _AMO32/64, _WAKEUP). This is not bureaucracy — UCX uses it to avoid initializing transports/resources you'll never use, which trims memory and connection-setup cost. Request only what you use.

§ 04Transports & Memory Domains

A transport (TL) is a UCT driver for one way of moving bytes. UCX ships many, and at runtime it discovers which are usable on your machine, scores them, and selects per operation. You mostly don't name transports in code — you let UCX choose, and optionally constrain the set with UCX_TLS.

Memory domains and the registration problem

To let a NIC DMA directly into your buffer, the OS must pin the pages (they can't be swapped or moved) and the device must be told about them — this is memory registration (ibv_reg_mr under the hood). It is expensive: a syscall, page-table walks, and device setup. A uct_md (memory domain) is the object that performs registration for a given device class.

This single fact drives a large fraction of UCX's design and tuning. Registration cost is why UCX has a registration cache (reuse a registration for a buffer you send from repeatedly), why rendezvous exists (register once, RDMA the bulk), and why pre-registering long-lived buffers is the number-one application-level performance lever.

Memory type awareness

UCX inspects pointers to learn their memory type — host, CUDA device, CUDA managed, ROCm — using driver queries (cuPointerGetAttribute) cached in a memtype cache. This is how a single ucp_tag_send_nbx on a GPU pointer transparently routes through cuda_ipc or GPUDirect RDMA without you writing any GPU-specific transfer code. Inspect what UCX detected with ucx_info -d.

§ 05The Five Communication Primitives

UCP exposes five families of operations. Picking the right one is a design decision, not a detail — it determines your synchronization model and how much the receiver needs to know in advance.

1 · Tag-matched messaging — the MPI-like model

Two-sided. The sender attaches a 64-bit tag; the receiver posts a receive with a tag and a 64-bit mask, and UCX matches incoming messages against posted receives. This is the foundation MPI's point-to-point sits on (tag ≈ encode of rank/communicator/MPI-tag). Either side may arrive first; unmatched arrivals wait in an unexpected queue.

tag.cC
/* sender */                ucp_tag_send_nbx(ep, buf, len, tag, &param);
/* receiver (exact tag) */  ucp_tag_recv_nbx(worker, buf, len, tag, /*mask*/ -1, &param);
/* receiver (wildcard)   */ ucp_tag_recv_nbx(worker, buf, len, tag, /*mask*/ 0,  &param);
/* peek without consuming */ ucp_tag_probe_nb(worker, tag, mask, /*remove*/0, &info);

2 · Stream — ordered bytes, no tags

Two-sided but connection-oriented and byte-oriented, like a TCP socket: bytes pushed on one end arrive in order on the other, with no message boundaries or tags. Use it when you have an established peer relationship and want simple ordered flow. ucp_stream_send_nbx / ucp_stream_recv_nbx.

3 · RMA — one-sided put / get

The sender (origin) reads or writes the remote (target) memory without the target's CPU participating in each transfer. The target registers memory once and hands out an rkey; thereafter the origin issues ucp_put_nbx / ucp_get_nbx directly to a remote address. This is the model that exploits RDMA hardware most fully and underpins MPI one-sided windows and PGAS languages.

rma.cC
/* target: register & publish a key */
ucp_mem_map(context, &map_params, &memh);
void *rkey_buf; size_t rkey_sz;
ucp_rkey_pack(context, memh, &rkey_buf, &rkey_sz);   /* send rkey_buf to origin via OOB */

/* origin: unpack the key, then write directly into remote memory */
ucp_ep_rkey_unpack(ep, rkey_buf, &rkey);
ucp_put_nbx(ep, local_src, len, remote_addr, rkey, &param);
ucp_ep_flush_nbx(ep, &param);   /* ensure remote completion */

4 · Atomics — remote read-modify-write

Hardware-offloaded atomic operations on remote memory: add, and, or, xor, swap, compare-and-swap, on 32/64-bit words. ucp_atomic_op_nbx. The building block for distributed counters, locks, and work-stealing queues — done by the NIC, not the remote CPU.

5 · Active Messages — handler dispatch

The most flexible primitive and the modern favorite for framework builders. The sender attaches a small header + payload to a numeric AM id; the receiver has pre-registered a callback for that id which fires on arrival. The handler decides what to do — copy now, defer, or trigger a rendezvous pull of a large payload. RAPIDS/UCX-Py and many custom runtimes are built on AM because it cleanly separates control (header) from bulk (payload).

am.cC
/* receiver: register a handler for AM id 7 */
ucp_am_handler_param_t h = {
    .field_mask = UCP_AM_HANDLER_PARAM_FIELD_ID |
                  UCP_AM_HANDLER_PARAM_FIELD_CB,
    .id = 7, .cb = my_am_recv_cb,
};
ucp_worker_set_am_recv_handler(worker, &h);

/* sender: fire it with a header and a payload */
ucp_am_send_nbx(ep, /*id*/7, hdr, hdr_len, payload, payload_len, &param);
Choosing a primitive

Porting MPI-style code → tag. Socket-like ordered pipe → stream. Big bulk transfers where one side owns the schedule → RMA. Distributed coordination → atomics. Building a new runtime / RPC-ish protocol with control + bulk separation → active messages.

§ 06The nbx Request Model

Every modern UCP operation ends in _nbx ("non-blocking, extended") and takes a single ucp_request_param_t describing how you want it done and completed. This unified parameter struct replaced the older fixed-signature _nb / _nbi calls; learn it once and every operation looks the same.

The three-valued return — the contract you must handle

An _nbx call returns a ucs_status_ptr_t, which is one of three things. Mishandling this is the single most common UCX bug.

ReturnMeaningWhat you do
NULLCompleted immediately and in-line.Nothing — it's done. Do not free anything.
UCS_PTR_IS_ERR(p)Failed synchronously.Read UCS_PTR_STATUS(p); handle the error.
a valid request pointerIn progress. A request handle was allocated.Drive ucp_worker_progress until done, then you must ucp_request_free(p).

The canonical "wait for one request" helper

request_wait.cC
ucs_status_t request_wait(ucp_worker_h worker, void *request)
{
    /* Immediate completion: nothing was allocated. */
    if (request == NULL)
        return UCS_OK;

    /* Synchronous failure. */
    if (UCS_PTR_IS_ERR(request))
        return UCS_PTR_STATUS(request);

    /* In progress: pump the engine until the status settles. */
    ucs_status_t status;
    do {
        ucp_worker_progress(worker);              /* advance the state machine */
        status = ucp_request_check_status(request);
    } while (status == UCS_INPROGRESS);

    ucp_request_free(request);                    /* MUST free the handle */
    return status;
}

Anatomy of ucp_request_param_t

You only set the fields named in op_attr_mask; everything else is ignored. This is how one struct serves every operation.

param.cC
ucp_request_param_t param = {
    /* declare which fields below are valid */
    .op_attr_mask = UCP_OP_ATTR_FIELD_CALLBACK  |   /* I'm providing a completion cb */
                    UCP_OP_ATTR_FIELD_USER_DATA |   /* ...with this user pointer */
                    UCP_OP_ATTR_FIELD_DATATYPE  |   /* ...and a non-contiguous datatype */
                    UCP_OP_ATTR_FIELD_MEMORY_TYPE,  /* ...and I'll tell you the mem type */

    .cb.send      = my_send_complete_cb,            /* fires on async completion */
    .user_data    = my_context,                     /* passed back to the cb */
    .datatype     = ucp_dt_make_contig(1),          /* bytes; or iov/generic */
    .memory_type  = UCS_MEMORY_TYPE_CUDA,           /* skip detection: it's GPU mem */
};

/* Useful flags you can OR into op_attr_mask:                              */
/*  UCP_OP_ATTR_FLAG_NO_IMM_CMPL  -> always return a request, never NULL   */
/*  UCP_OP_ATTR_FLAG_FAST_CMPL    -> optimize for latency over bandwidth   */
/*  UCP_OP_ATTR_FIELD_FLAGS + UCP_AM_SEND_FLAG_EAGER/_RNDV (AM hinting)    */
Callback vs. polling

Two completion styles: (1) poll with ucp_request_check_status while calling ucp_worker_progress (the helper above), or (2) register a callback via .cb.send/.cb.recv that UCX invokes from inside ucp_worker_progress when the op finishes. Either way, progress must be driven — callbacks do not fire on a background thread unless you've set that up explicitly.

If you're reading old tutorials

Code calling ucp_tag_send_nb (no x) with a bare callback argument is the deprecated pre-1.9 API. The signatures differ and the docs steer you to _nbx. Don't mix the two families; port to _nbx.

§ 07Install & Build

Get UCX present and capable first; nothing else matters if your build lacks the transports your hardware needs. The configure-time flags determine which transports compile in — a UCX without --with-cuda simply cannot do GPU transfers no matter what you set at runtime.

build-from-source.shbash
# Distro packages exist (apt install libucx-dev), but source build lets you
# control which transports are enabled — important for IB + CUDA setups.
git clone https://github.com/openucx/ucx.git
cd ucx
./autogen.sh
./contrib/configure-release \
    --prefix=$HOME/ucx-install \
    --with-cuda=/usr/local/cuda \   # GPU support: cuda_copy, cuda_ipc, gdr
    --with-gdrcopy \                 # low-latency GPU memory access
    --with-verbs \                   # InfiniBand / RoCE
    --enable-mt                      # multi-threaded workers
make -j$(nproc)
make install

# Verify the install knows about your hardware:
$HOME/ucx-install/bin/ucx_info -v     # version + which features compiled in
$HOME/ucx-install/bin/ucx_info -d     # devices + transports actually available

Compile and link an application against UCP:

compile.shbash
gcc myapp.c -o myapp \
    -I$HOME/ucx-install/include \
    -L$HOME/ucx-install/lib -lucp -lucs -luct \
    -Wl,-rpath,$HOME/ucx-install/lib

# Ships with reference programs — read these, they are excellent:
#   ucx_perftest        microbenchmark for latency/bandwidth of every primitive
#   examples/ucp_hello_world.c     tag send/recv with TCP-socket OOB exchange

First thing to run on any new node

ucx_perftest tells you instantly whether the fast path works. On the server: ucx_perftest -t tag_lat; on the client: ucx_perftest <server> -t tag_lat. Compare latency/bandwidth against the hardware's spec sheet — if a 200 Gb/s NIC delivers 10 Gb/s, you're on TCP or a wrong device, and you know it before writing a line of code.

§ 08Hello World

The smallest meaningful program: bring up the stack, connect a worker to itself via the self transport, and send a tagged message from a worker to itself. It exercises the full send/recv path with zero networking complexity, so when it works you know your build, link, and lifecycle logic are sound.

hello.c — self-loopback tag send/recvC
#include <ucp/api/ucp.h>
#include <string.h>
#include <stdio.h>

static ucs_status_t wait_req(ucp_worker_h w, void *r) {
    if (r == NULL) return UCS_OK;
    if (UCS_PTR_IS_ERR(r)) return UCS_PTR_STATUS(r);
    ucs_status_t s;
    do { ucp_worker_progress(w); s = ucp_request_check_status(r); }
    while (s == UCS_INPROGRESS);
    ucp_request_free(r);
    return s;
}

int main(void) {
    /* --- bring up context + worker --- */
    ucp_config_t *cfg; ucp_config_read(NULL, NULL, &cfg);
    ucp_params_t cp = { .field_mask = UCP_PARAM_FIELD_FEATURES,
                        .features   = UCP_FEATURE_TAG };
    ucp_context_h ctx; ucp_init(&cp, cfg, &ctx); ucp_config_release(cfg);

    ucp_worker_params_t wp = { .field_mask = UCP_WORKER_PARAM_FIELD_THREAD_MODE,
                               .thread_mode = UCS_THREAD_MODE_SINGLE };
    ucp_worker_h w; ucp_worker_create(ctx, &wp, &w);

    /* --- connect the worker to ITSELF via its own address --- */
    ucp_address_t *addr; size_t addr_len;
    ucp_worker_get_address(w, &addr, &addr_len);
    ucp_ep_params_t ep_p = { .field_mask = UCP_EP_PARAM_FIELD_REMOTE_ADDRESS,
                             .address = addr };
    ucp_ep_h ep; ucp_ep_create(w, &ep_p, &ep);
    ucp_worker_release_address(w, addr);

    /* --- send "hello, ucx" to ourselves with tag 0x42 --- */
    const char *msg = "hello, ucx";
    char recv_buf[64] = {0};
    ucp_request_param_t prm = { .op_attr_mask = 0 };

    void *sreq = ucp_tag_send_nbx(ep, msg, strlen(msg)+1, 0x42, &prm);
    void *rreq = ucp_tag_recv_nbx(w, recv_buf, sizeof(recv_buf),
                                  0x42, /*mask*/ (ucp_tag_t)-1, &prm);
    wait_req(w, sreq);
    wait_req(w, rreq);
    printf("received: \"%s\"\n", recv_buf);   /* -> received: "hello, ucx" */

    /* --- teardown --- */
    void *creq = ucp_ep_close_nbx(ep, &prm); wait_req(w, creq);
    ucp_worker_destroy(w);
    ucp_cleanup(ctx);
    return 0;
}
What this proves

Address-based connection (ucp_worker_get_address → ucp_ep_create with REMOTE_ADDRESS) is exactly the mechanism used between real nodes — the only difference there is that the address travels over an out-of-band channel instead of staying in-process. Master this and the two-process version is a small step.

§ 09Full Ping-Pong: Client & Server

Now the real thing across two processes/nodes, using the listener (client-server) connection model — UCX's built-in TCP-based bootstrap, so you don't need MPI or your own socket code to exchange addresses. The client connects to host:port; UCX establishes the endpoints; then they bounce a tagged message back and forth N times.

The connection model in one picture

Listener model: the server publishes an address; UCX handles the handshake; both sides get a connected ucp_ep_h.

Shared scaffolding

pingpong.c — commonC
#include <ucp/api/ucp.h>
#include <netinet/in.h>
#include <arpa/inet.h>
#include <string.h>
#include <stdio.h>
#include <stdlib.h>

#define PORT      13337
#define ITERS     10
#define TAG       0xCAFE
#define MSG_LEN   64

/* drive a single request to completion */
static ucs_status_t wait_req(ucp_worker_h w, void *r) {
    if (r == NULL)            return UCS_OK;
    if (UCS_PTR_IS_ERR(r))    return UCS_PTR_STATUS(r);
    ucs_status_t s;
    do { ucp_worker_progress(w); s = ucp_request_check_status(r); }
    while (s == UCS_INPROGRESS);
    ucp_request_free(r);
    return s;
}

/* one blocking send + one blocking recv on the same tag */
static void send_msg(ucp_worker_h w, ucp_ep_h ep, void *buf, size_t len) {
    ucp_request_param_t p = { .op_attr_mask = 0 };
    wait_req(w, ucp_tag_send_nbx(ep, buf, len, TAG, &p));
}
static void recv_msg(ucp_worker_h w, void *buf, size_t len) {
    ucp_request_param_t p = { .op_attr_mask = 0 };
    wait_req(w, ucp_tag_recv_nbx(w, buf, len, TAG, (ucp_tag_t)-1, &p));
}

/* boot a context+worker requesting the TAG feature */
static void boot(ucp_context_h *ctx, ucp_worker_h *w) {
    ucp_config_t *cfg; ucp_config_read(NULL, NULL, &cfg);
    ucp_params_t cp = { .field_mask = UCP_PARAM_FIELD_FEATURES,
                        .features   = UCP_FEATURE_TAG };
    ucp_init(&cp, cfg, ctx); ucp_config_release(cfg);
    ucp_worker_params_t wp = { .field_mask = UCP_WORKER_PARAM_FIELD_THREAD_MODE,
                               .thread_mode = UCS_THREAD_MODE_SINGLE };
    ucp_worker_create(*ctx, &wp, w);
}

Server side

pingpong.c — serverC
/* The listener stores the incoming connection request here so main() can
 * turn it into an endpoint. (A real server would queue multiple.) */
typedef struct { volatile ucp_conn_request_h req; } server_ctx_t;

/* fires on the worker's progress thread when a client connects */
static void on_connect(ucp_conn_request_h conn_request, void *arg) {
    server_ctx_t *s = arg;
    s->req = conn_request;                 /* hand it to main loop */
}

int run_server(void) {
    ucp_context_h ctx; ucp_worker_h w;
    boot(&ctx, &w);

    server_ctx_t sctx = { .req = NULL };

    /* bind a listener to 0.0.0.0:PORT with our connect callback */
    struct sockaddr_in sa = { .sin_family = AF_INET,
                              .sin_addr.s_addr = INADDR_ANY,
                              .sin_port = htons(PORT) };
    ucp_listener_params_t lp = {
        .field_mask = UCP_LISTENER_PARAM_FIELD_SOCK_ADDR |
                      UCP_LISTENER_PARAM_FIELD_CONN_HANDLER,
        .sockaddr   = { .addr = (const struct sockaddr*)&sa, .addrlen = sizeof(sa) },
        .conn_handler = { .cb = on_connect, .arg = &sctx },
    };
    ucp_listener_h listener;
    ucp_listener_create(w, &lp, &listener);
    printf("server: listening on :%d\n", PORT);

    /* spin progress until a connection request arrives */
    while (sctx.req == NULL) ucp_worker_progress(w);

    /* accept: build an endpoint from the connection request */
    ucp_ep_params_t ep_p = {
        .field_mask   = UCP_EP_PARAM_FIELD_CONN_REQUEST,
        .conn_request = sctx.req,
    };
    ucp_ep_h ep; ucp_ep_create(w, &ep_p, &ep);
    printf("server: client connected\n");

    /* PONG: receive then echo back, ITERS times */
    char buf[MSG_LEN];
    for (int i = 0; i < ITERS; i++) {
        recv_msg(w, buf, MSG_LEN);
        printf("server: got  '%s'\n", buf);
        send_msg(w, ep, buf, MSG_LEN);     /* bounce it back */
    }

    ucp_request_param_t p = { .op_attr_mask = 0 };
    wait_req(w, ucp_ep_close_nbx(ep, &p));
    ucp_listener_destroy(listener);
    ucp_worker_destroy(w); ucp_cleanup(ctx);
    return 0;
}

Client side

pingpong.c — clientC
int run_client(const char *server_ip) {
    ucp_context_h ctx; ucp_worker_h w;
    boot(&ctx, &w);

    /* connect an endpoint straight to the server's sockaddr */
    struct sockaddr_in sa = { .sin_family = AF_INET, .sin_port = htons(PORT) };
    inet_pton(AF_INET, server_ip, &sa.sin_addr);

    ucp_ep_params_t ep_p = {
        .field_mask = UCP_EP_PARAM_FIELD_FLAGS |
                      UCP_EP_PARAM_FIELD_SOCK_ADDR,
        .flags      = UCP_EP_PARAMS_FLAGS_CLIENT_SERVER,
        .sockaddr   = { .addr = (const struct sockaddr*)&sa, .addrlen = sizeof(sa) },
    };
    ucp_ep_h ep; ucp_ep_create(w, &ep_p, &ep);
    printf("client: connected to %s:%d\n", server_ip, PORT);

    /* PING: send then await echo, timing the round trip */
    char buf[MSG_LEN];
    for (int i = 0; i < ITERS; i++) {
        snprintf(buf, MSG_LEN, "ping #%d", i);
        send_msg(w, ep, buf, MSG_LEN);
        recv_msg(w, buf, MSG_LEN);
        printf("client: echo '%s'\n", buf);
    }

    ucp_request_param_t p = { .op_attr_mask = 0 };
    wait_req(w, ucp_ep_close_nbx(ep, &p));
    ucp_worker_destroy(w); ucp_cleanup(ctx);
    return 0;
}

int main(int argc, char **argv) {
    /* usage:  ./pingpong            -> server
     *         ./pingpong 10.0.0.5   -> client connecting to that server */
    return (argc > 1) ? run_client(argv[1]) : run_server();
}

§ 10The Progress Engine

UCX is fundamentally a polled, run-to-completion state machine. Nothing advances unless someone calls ucp_worker_progress(worker). This is the mental model that resolves most "my program hangs" confusion: a posted send/recv is just a promise; progress is the engine that fulfills it — pumping the NIC completion queues, matching tags, advancing rendezvous handshakes, and firing your callbacks.

progress.cC
/* Returns nonzero if any work was performed. Cheap to call in a tight loop. */
unsigned ucp_worker_progress(ucp_worker_h worker);

/* The "I have nothing else to do, wake me on an event" path (needs
   UCP_FEATURE_WAKEUP) — avoids burning a core spinning: */
int efd; ucp_worker_get_efd(worker, &efd);     /* an epoll-able fd */
for (;;) {
    if (work_pending) { ucp_worker_progress(worker); continue; }
    ucp_worker_arm(worker);                     /* arm before blocking */
    epoll_wait(epfd, &ev, 1, -1);                /* sleep until the NIC signals */
    ucp_worker_progress(worker);
}
The #1 hang

You post a receive, then block waiting for it inside a loop that doesn't call progress. Nothing arrives because nothing is pumping the engine. Every blocking wait must include a ucp_worker_progress call (as the helper in §06 does). Symmetrically, a callback that never fires almost always means progress isn't being driven on that worker.

Threading

A default worker is UCS_THREAD_MODE_SINGLE — exactly one thread may touch it. For multi-threaded designs you either (a) give each thread its own worker (the common, scalable choice), or (b) build with --enable-mt and create the worker UCS_THREAD_MODE_MULTI, accepting the locking overhead. Per-thread workers is why the worker, not the context, is the unit of concurrency.

§ 11How It Works: Protocols & Internals

When you hand UCP a buffer, it doesn't just "send" — it chooses a protocol based on message size, memory type, and transport capabilities. The three families below are the heart of UCX's performance, and their thresholds are the knobs you tune most.

Eager — small messages, copy-based

For small payloads, UCX sends the data immediately, optionally inlined into the transport header (the "short" path) or copied into a pre-registered bounce buffer ("bcopy"). If the receiver hasn't posted a matching receive yet, the data lands in an unexpected queue and is copied to the user buffer once the receive appears. The cost is the copy; the win is no handshake — for tiny messages, latency is dominated by round trips, so eager's single one-way trip is optimal.

Rendezvous (rndv) — large messages, zero-copy

For large payloads, copying is the enemy and registration must be amortized. The rendezvous protocol negotiates first, then moves bulk data by RDMA with no intermediate copy:

Rendezvous: a control handshake, then a single zero-copy RDMA. The data never touches a bounce buffer.

The crossover between eager and rendezvous is UCX_RNDV_THRESH (default auto, derived from the hardware). The scheme — receiver-driven get_zcopy vs sender-driven put_zcopy vs auto — is UCX_RNDV_SCHEME.

Multi-rail striping

If a node has several NICs (or several ports), UCX can split one large rendezvous transfer across them in parallel — "rails" — to aggregate bandwidth. UCX_MAX_RNDV_RAILS and lane limits control this. The same machinery picks multiple lanes for eager messages too.

The registration cache (rcache)

Because ibv_reg_mr is so costly, UCX keeps a cache (in UCS) of registrations keyed by virtual address range. The first zero-copy send from a buffer registers it; subsequent sends from the same (or overlapping) buffer reuse the registration for free. This is why reusing long-lived buffers dramatically outperforms registering fresh memory per transfer — and why a buggy allocator that recycles addresses with different sizes can cause subtle rcache invalidation stalls.

Connection establishment, under the hood

Whether via address-exchange or the listener, the result is a wire-up: UCX picks transports usable between the two workers, creates the underlying UCT endpoints (e.g. IB queue pairs), and may use a scalable transport (UD/DC) for the handshake before promoting to RC for bulk data. Large-scale jobs lean on DC precisely so that connection state doesn't grow O(peers²).

§ 12Protocol Selection as Cost-Model Compilation

This is the conceptual core of modern UCX, and it will feel familiar if you think about query planners. UCX does not hardcode "use eager below X, rendezvous above X." The newer protocol framework (often called protocols v2) builds an explicit performance cost model and picks the protocol with the lowest estimated time for each operation — exactly analogous to a cost-based optimizer choosing a physical plan from logical operators.

The compilation analogy

Query plannerUCX protocol engine
Logical operation (join, scan)Logical op: "send N bytes, this mem type, this peer"
Physical plan candidates (hash vs merge join)Candidate protocols: short / eager-bcopy / eager-zcopy / rndv-get / rndv-put / multi-rail split

Choose min-cost plan, cache itChoose min-estimated-time protocol, memoize per (size-range, mem-type, peer)

Each candidate protocol contributes a piece-wise linear cost latency + size / bandwidth (plus fixed overheads like registration or handshake round trips). UCX evaluates these across the message-size axis and selects, for each size range, the protocol whose modeled time is lowest. The result is effectively a compiled dispatch table from (message size, memory type, peer) → concrete transport plan. This is precisely the "compile a logical topology/plan from a physical cost model" framing — the physical fabric's measured costs determine the logical send strategy.

Why this matters for a shuffle engine

When your exchange operator sends column batches of wildly varying sizes — tiny null-masks, medium dictionaries, huge data buffers — you don't want to choose a transfer strategy per batch. UCX's cost model does it for you, automatically promoting big buffers to multi-rail zero-copy RDMA while keeping small control messages on the eager fast path. Reading UCX_PROTO_INFO output tells you whether your batches are landing on the path you expect.

§ 13Topology & Locality

On a fat node — multiple sockets, multiple NICs, multiple GPUs hanging off a PCIe hierarchy — which device you use matters as much as which protocol. UCX builds a model of the physical topology and selects devices by locality, so a transfer to/from a given GPU prefers the NIC sharing its PCIe switch, not one across a socket.

How UCX discovers and scores topology

  • Device enumeration: at context init, UCX queries every transport for its devices and their capabilities (bandwidth, latency, max message, atomics, GPU support).
  • Distance / locality: UCX reads system topology (NUMA node, PCIe path) to estimate the "distance" between a buffer's location (a particular CPU socket or GPU) and each NIC, then scores devices so the closest, highest-bandwidth one wins.
  • Lane assignment: the winning device(s) become the lanes for that endpoint; multiple may be combined for multi-rail.

The knobs that override discovery

VariableEffect
UCX_NET_DEVICESRestrict to specific devices, e.g. mlx5_0:1,mlx5_1:1. Essential when you must bind traffic to a particular rail or keep it off the management NIC.
UCX_TLSRestrict the transport set, e.g. rc,cuda to force IB+GPU, or ^tcp to forbid the TCP fallback (a great way to catch silent slow paths).
UCX_IB_GPU_DIRECT_RDMAEnable/disable letting the NIC DMA straight to/from GPU memory (skip the host bounce).
UCX_MAX_RNDV_RAILS / UCX_MAX_EAGER_LANESHow many parallel rails/lanes a transfer may stripe across.
UCX_NUM_EPSHint about scale, influencing UD/DC vs RC selection for connection-state scalability.
Physical → logical, concretely

Say node A has GPU0 + GPU1, each behind PCIe switch S0/S1, with NIC0 on S0 and NIC1 on S1. For a GPU0→remote send, UCX scores NIC0 above NIC1 (same switch, GPUDirect-eligible), selects it as the lane, and — if the batch is large — adds NIC1 as a second rail for bandwidth while still preferring the local one. You wrote one ucp_tag_send_nbx; UCX compiled it into a topology-aware multi-rail plan. That compilation step is the logical topology derived from the physical one.

§ 14Why UCX Is Fast

Five mechanisms, each attacking a specific source of overhead. None is exotic; the value is that UCX implements all of them carefully in one shared, well-tested place.

1 · Zero-copy

Rendezvous + RDMA move bulk data directly between user buffers — the data never visits a bounce buffer or kernel socket buffer, eliminating the dominant cost for large transfers.

2 · Hardware offload

The NIC performs the transfer, RDMA, and even atomics autonomously. The CPU issues a descriptor and is free; remote CPUs aren't involved in one-sided ops at all.

3 · Kernel bypass

RDMA verbs let the application post work directly to NIC queues from user space — no syscall, no kernel network stack, no context switch on the data path.

4 · Registration caching

The expensive memory-pinning step is amortized across all transfers from a reused buffer, turning a per-message cost into a one-time one.

5 · Adaptive protocols

The cost model picks the optimal strategy per message — eager for latency-bound small messages, zero-copy multi-rail for bandwidth-bound large ones — so you never pay the wrong protocol's tax.

+ · Topology awareness

Choosing the NIC closest to the data (and skipping the host bounce for GPU memory) avoids cross-socket and PCIe-traversal penalties before a single byte moves.

The throughput intuition

For small messages, the round-trip count dominates — UCX minimizes trips (eager, inline). For large messages, copies and registration dominate — UCX eliminates copies (zero-copy RDMA) and amortizes registration (rcache), then parallelizes across rails. There is no single trick; "fast" is the union of the right choice at every size.

§ 15UCX vs MPI

The most useful framing: they live at different layers. MPI is a high-level programming model and standard; UCX is a low-level transport library. They are not competitors — most MPI implementations run on top of UCX. Comparing them is like comparing "SQL" to "the storage engine."

When to drop to UCX directly

  • You're building a runtime that isn't MPI: a distributed query engine's shuffle/exchange, a parameter server, a custom RPC fabric, a dataframe library's transport (Dask/RAPIDS).
  • You need primitives MPI handles awkwardly — fine-grained active messages, custom rendezvous, direct GPU-memory transfers without MPI's datatype machinery.
  • You want to avoid MPI's process model entirely (e.g. a Kubernetes-scheduled service mesh where there is no mpirun).

Conversely, if you have ranks, want collectives, and your problem fits SPMD — use MPI, and let it use UCX underneath. Don't reimplement MPI on UCX unless you have a reason MPI can't serve.

§ 16Coordinating With MPI

A powerful and common hybrid: use MPI for what it's good at — process launch, ranks, collectives, and especially as the out-of-band (OOB) channel to bootstrap UCX — then use UCX directly for a custom high-performance data plane. This sidesteps UCX's lack of a bootstrap by borrowing MPI's.

Pattern A — MPI bootstraps a UCX data plane

Recall the hello-world needed each peer's worker address. Across nodes you must ship those addresses somehow. MPI_Allgather is the easiest possible OOB exchange:

mpi_bootstrap_ucx.cC
MPI_Init(&argc, &argv);
int rank, size;
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

/* ... ucp_init + ucp_worker_create as usual ... */

/* 1. Get THIS worker's address */
ucp_address_t *my_addr; size_t my_len;
ucp_worker_get_address(worker, &my_addr, &my_len);

/* 2. Exchange address lengths, then the addresses themselves, over MPI */
size_t *lens = malloc(size * sizeof(size_t));
MPI_Allgather(&my_len, sizeof(size_t), MPI_BYTE,
              lens, sizeof(size_t), MPI_BYTE, MPI_COMM_WORLD);
/* (use Allgatherv to collect the variable-length address blobs) */
int *displs = ...; int total = ...;            /* compute from lens */
char *all_addrs = malloc(total);
MPI_Allgatherv(my_addr, my_len, MPI_BYTE,
               all_addrs, /*counts*/..., displs, MPI_BYTE, MPI_COMM_WORLD);

/* 3. Build a UCX endpoint to every other rank from its gathered address */
ucp_ep_h *eps = malloc(size * sizeof(ucp_ep_h));
for (int r = 0; r < size; r++) {
    if (r == rank) continue;
    ucp_ep_params_t ep_p = {
        .field_mask = UCP_EP_PARAM_FIELD_REMOTE_ADDRESS,
        .address    = (ucp_address_t*)(all_addrs + displs[r]),
    };
    ucp_ep_create(worker, &ep_p, &eps[r]);
}
/* Now every rank has UCX endpoints to all peers — use tag/RMA/AM directly,
   while still using MPI for collectives and control. */
ucp_worker_release_address(worker, my_addr);
One process, one framework on the wire — pick the rail deliberately

When MPI and your direct UCX both run in the same job, they may both want the IB device. That's fine — they're independent UCX contexts — but watch resource contention and make sure both target the intended NIC via UCX_NET_DEVICES. Also ensure the MPI you link was itself built on UCX, or you'll have two different transport stacks fighting for the HCA.

Pattern B — let MPI transparently be UCX

If you just want fast MPI, you may not write any UCX at all — you select the UCX path inside your MPI:

run-mpi-on-ucx.shbash
# Open MPI: force the UCX point-to-point component
mpirun --mca pml ucx --mca osc ucx -np 64 ./my_mpi_app

# MPICH / MVAPICH built with CH4:ucx use UCX automatically.
# UCX_* env vars still apply and tune the underlying transport:
UCX_TLS=rc,sm,cuda UCX_NET_DEVICES=mlx5_0:1 mpirun --mca pml ucx -np 64 ./app

§ 17How One Tool Backs MPI, NCCL & the Rest

The "unified" in Unified Communication X is literal: UCX is the shared transport substrate that many higher-level libraries delegate to, instead of each re-implementing IB verbs, shared-memory transports, and rendezvous. The trick is that UCX exposes a narrow, capability-rich, runtime-facing API (UCP) that is general enough to serve very different consumers.

Many runtimes, one substrate. Each consumer maps its own concepts onto UCP's primitives.

How each consumer maps on

  • Open MPI — the pml ucx component implements MPI point-to-point by encoding (communicator, source rank, MPI tag) into UCP's 64-bit tag and calling ucp_tag_send/recv_nbx. MPI one-sided maps to UCP RMA.
  • MPICH / MVAPICH — the CH4 ucx netmod does the same at MPICH's lower layer.
  • NCCL — NCCL owns intra-node GPU collectives via its own optimized rings/trees over NVLink, but its network transport is pluggable. A UCX net plugin lets NCCL's inter-node traffic ride UCX (reusing UCX's IB tuning, multi-rail, GPUDirect), instead of NCCL's built-in IB transport.
  • RAPIDS / Dask-CUDA — UCX-Py wraps UCP for Python; dask-cuda uses it so GPU dataframe shuffles travel GPU→GPU over NVLink (cuda_ipc) intra-node and GPUDirect RDMA inter-node, never staging through host memory.
  • Spark RAPIDS — its shuffle manager uses UCX to move columnar shuffle blocks between executors at RDMA speed.
The design lesson

Unification was achievable because UCP's API is capability-oriented, not policy-oriented. It exposes tag/RMA/AM and lets the consumer decide semantics — so MPI can build send/recv, NCCL can build a net transport, and a query engine can build a shuffle, all from the same five primitives. The hard, hardware-specific work (registration, rendezvous, device selection, GPU awareness) is done once below the API and inherited by everyone.

§ 18UCC — Where Collectives Live

UCX deliberately has no collectives. UCC (Unified Collective Communication) is the sibling project that does — allreduce, broadcast, alltoall, barrier, reduce-scatter — built on top of UCX's point-to-point and RMA, and able to dispatch to specialized backends.

  • Composable backends: UCC can route a collective to NCCL for intra-node GPU reductions, to UCX for inter-node movement, or to SHARP (in-network reduction performed by the InfiniBand switch itself) when available — choosing per collective and per message size.
  • Used by: Open MPI's coll ucc component offloads MPI collectives to UCC; some DL training stacks use UCC directly for allreduce.
  • The parallel to §17: just as UCP unifies transport, UCC unifies collectives over heterogeneous accelerators — same philosophy, one layer up.
Practical takeaway

If you need allreduce-style operations and you're already on UCX, reach for UCC rather than hand-rolling reductions over UCP tag messages — it will pick in-network or GPU-direct paths you'd never implement by hand.

§ 19GPU Memory, GPUDirect & RAPIDS

The piece most relevant to GPU-accelerated query execution. UCX treats GPU memory as a first-class citizen: you pass a device pointer to the same ucp_tag_send_nbx / ucp_put_nbx you'd use for host memory, and UCX figures out the right path.

The three GPU transports and when each fires

PathTransportUsed when
Intra-node GPU↔GPUcuda_ipcTwo GPUs in the same node — direct peer copy over NVLink/NVSwitch (or PCIe P2P). No host involvement.
Inter-node GPU↔GPUrc + GPUDirect RDMAThe NIC DMAs straight out of / into GPU memory across the fabric — the host never touches the payload.
GPU↔host stagingcuda_copy / gdr_copyWhen a direct path isn't available, or for small/low-latency GPU-memory access via gdrcopy.

Sending from a device pointer

gpu_send.cC
/* Build the context with the CUDA feature set available
   (requires UCX compiled --with-cuda). */
ucp_params_t cp = { .field_mask = UCP_PARAM_FIELD_FEATURES,
                    .features   = UCP_FEATURE_TAG | UCP_FEATURE_RMA };

float *d_buf;
cudaMalloc(&d_buf, n * sizeof(float));            /* device memory */

/* Option 1: let UCX auto-detect the memory type (memtype cache). */
ucp_request_param_t p1 = { .op_attr_mask = 0 };
ucp_tag_send_nbx(ep, d_buf, n*sizeof(float), TAG, &p1);

/* Option 2: tell UCX explicitly — skips detection, slightly faster. */
ucp_request_param_t p2 = {
    .op_attr_mask = UCP_OP_ATTR_FIELD_MEMORY_TYPE,
    .memory_type  = UCS_MEMORY_TYPE_CUDA,
};
ucp_tag_send_nbx(ep, d_buf, n*sizeof(float), TAG, &p2);
/* UCX now selects cuda_ipc / GPUDirect-RDMA / staging based on the peer's
   location and the cost model — same call, topology-aware result. */
For a GPU exchange / shuffle operator

This is the whole reason UCX is the right substrate for a Velox/cuDF-style GPU exchange: one send path that (a) routes intra-node GPU↔GPU over NVLink via cuda_ipc, (b) crosses nodes via GPUDirect RDMA without a host bounce, (c) picks the NIC nearest the source GPU, and (d) promotes large column buffers to multi-rail zero-copy rendezvous automatically. You express "send this batch to that peer"; UCX compiles the physical plan. Verify it with UCX_PROTO_INFO=y and confirm cuda_ipc/GPUDirect appear — if you instead see cuda_copy + tcp, you're silently bouncing through host memory and the win is gone.

GPUDirect prerequisites that bite

GPUDirect RDMA needs the kernel peer-memory module loaded (nvidia_peermem, or the older nv_peer_mem) and a NIC+GPU sharing a PCIe path. gdr_copy needs the gdrdrv module. If these aren't present, UCX silently falls back to host staging — functionally correct, dramatically slower. Always confirm with ucx_info -d | grep -i cuda and a UCX_PROTO_INFO check on a real transfer.

§ 20Debugging UCX

UCX is highly introspectable once you know the tools. The workflow is almost always: (1) confirm what's available with ucx_info, (2) confirm what got chosen with UCX_PROTO_INFO, (3) turn up the log level to see why.

A reliable triage sequence

(1) ucx_info -d — is the device/transport even present? (2) UCX_TLS=^tcp — does forbidding the slow path expose a hidden failure? (3) UCX_PROTO_INFO=y — is the chosen protocol the one you wanted? (4) UCX_LOG_LEVEL=debug — what did connection setup decide and why? Ninety percent of issues resolve in those four steps.

§ 22Common Pitfalls Checklist

  • Forgetting to drive progress. The cause of most hangs. Every blocking wait and every "why didn't my callback fire" traces back here.
  • Not freeing requests. A non-NULL, non-error _nbx return is an allocated handle you must ucp_request_free after completion. Leak them and you exhaust the request pool.
  • Mishandling the three-valued return. Treating NULL (done) as an error, or calling ucp_request_free on NULL/an error pointer. Use the helper.
  • Sharing a SINGLE-mode worker across threads. Silent corruption or deadlock. One worker per thread, or build MULTI.
  • Mismatched send/recv length or tag. A recv buffer smaller than the message truncates or errors; a tag/mask mismatch means the recv never matches and the program hangs.
  • Mixing the deprecated _nb API with _nbx. Different signatures and completion semantics. Standardize on _nbx.
  • Assuming GPUDirect is on. Without the peer-memory kernel module, UCX silently stages through host memory. Always verify with UCX_PROTO_INFO.
  • Letting TCP mask a broken RDMA path. Everything "works" but slowly. Periodically test with UCX_TLS=^tcp to force the fast path or fail loudly.
  • Registering fresh memory per transfer. Defeats the rcache. Reuse long-lived, pre-registered buffers for hot paths.
  • Closing endpoints without flushing. Outstanding RMA may not have landed remotely. ucp_ep_flush_nbx before close when correctness depends on completion.

§ 23Mental Model Recap

If you keep only a handful of sentences, keep these:

The whole guide in seven lines

1. UCX is a transport substrate, not an application framework — runtimes (MPI, NCCL, RAPIDS) sit on it.
2. Three layers: you program UCP; UCT abstracts each interconnect; UCS is shared plumbing.
3. You manage five objects: context, worker, endpoint, mem-handle, rkey — and the worker is the unit of progress and threading.
4. Every op is _nbx + a request param, returning NULL / error / a handle you must drive and free.
5. Nothing happens without ucp_worker_progress.
6. UCX compiles each logical send into a physical plan via a cost model — eager vs rendezvous, which NICs, GPU-direct or staged — exactly like a query planner over the physical fabric.
7. Debug with ucx_info -d → UCX_TLS=^tcp → UCX_PROTO_INFO=y → UCX_LOG_LEVEL=debug.

From here, the highest-leverage next steps are reading the reference programs (ucp_client_server.c, ucp_hello_world.c) and running ucx_perftest on your real hardware while watching UCX_PROTO_INFO — that loop teaches you more about your fabric than any amount of documentation.
