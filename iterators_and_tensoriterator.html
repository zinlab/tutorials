Iterators & TensorIterator

A systematic walk from the humble loop abstraction in NumPy and C++ up to PyTorch's TensorIterator — with visual guides for broadcasting, dim coalescing/reordering, cache behavior, and offset arithmetic — then down into how snapy's Weno5 CUDA kernel launches every block in parallel while you write only a per-block device function, and finally a contrast with how cuDF solves the same problem the Thrust way.

1What an iterator actually is#

An iterator is an abstraction that separates the act of traversal from the work done at each element. Instead of every algorithm hard-coding how to walk a data structure (where the next element lives, when to stop, how to handle strides/gaps), the iterator owns that knowledge and hands the algorithm one element at a time.

This buys three things that recur at every level we will discuss:

  • Decoupling — the same loop body runs over an array, a file, a strided view, or a GPU tensor.
  • Uniformity — broadcasting, type promotion, memory layout, and bounds become the iterator's job, not the kernel's.
  • Performance hooks — because the iterator plans the traversal, it can collapse/reorder dimensions, vectorize, and parallelize without the kernel author knowing.
The one idea to carry through Every iterator in this document — np.nditer, a C++ RandomAccessIterator, and ATen's TensorIterator — is doing the same job: turn an N-dimensional, possibly-strided, possibly-broadcast layout into a flat stream of "here is the address of element i for every operand," so the compute kernel can stay tiny.

2Iterators in NumPy#

NumPy's user-facing iterator is numpy.nditer, a Python wrapper over the C engine NpyIter that powers every ufunc (np.add, np.sin, …). It is the direct conceptual ancestor of ATen's TensorIterator.

Stride tricks: the 0-stride broadcast

The mechanism that makes broadcasting free is the 0 stride. A broadcast axis advances the data pointer by 0 bytes per step, so the same memory is re-read. TensorIterator and snapy's Weno5 config use this same idea (an "expanded"/squashed axis whose stride the kernel reads but does not iterate over).

"Hands the inner body a contiguous chunk instead of one scalar"

Without buffering, the iterator would invoke your kernel once per element — a function call (or Python round-trip) for every scalar, and the compiler can't vectorize across calls. With external_loop/buffering, the iterator finds the longest run of elements that is contiguous in every operand and hands your body a base pointer + stride + length n for that whole run. Your body then runs a tight inner for i in 0..n loop the compiler can SIMD-vectorize (AVX/NEON), and the per-call overhead is paid once per run instead of once per element.

This is exactly snapy's CPU path (§8): iter.for_each([](char** data, const int64_t* strides, int64_t n){ for(i<n) … }) — the iterator hands a run of n, and the inner loop is the vectorizable hot path. The "operations involved" are: (1) pick the contiguous inner dimension, (2) emit base_ptr, strides[], n, (3) your loop does data[k] + i*strides[k] addressing per element.

3Iterators in C++#

C++ took the iterator idea and made it a compile-time concept rather than a runtime object. An STL iterator is anything that supports a defined set of operations (*it, ++it, comparison). Algorithms are written against those operations, so std::sort, std::accumulate, etc. work on any container.

  return init;

Why this matters for CUDA A raw pointer char* is a contiguous random-access iterator. In the Weno5 GPU kernel you will see the iterator's job reduced to exactly this: produce a char* data[] base pointer plus a byte offset for each operand. ATen's OffsetCalculator is "random-access iteration" expressed as arithmetic on a linear index — perfect for a GPU where every thread computes its own offset independently.

"Random-access iteration expressed as arithmetic on a linear index"

A forward iterator only knows "give me the next one" (++it) — it must step from where it is. A random-access iterator can jump straight to element k in O(1) (it + k). On a GPU you need the random-access flavour: thread #7341 must find its element with no knowledge of any other thread and no sequential walking. The trick is to never keep an iterator object at all — instead, give every thread one integer (its linear index) and a pure function that turns that integer into a memory offset. That function is ATen's OffsetCalculator.

Why one offset per operand: each tensor has its own strides (a broadcast input carries a 0; a transposed view carries swapped strides), so the calculator stores a stride table per operand and returns offsets[] — one byte offset for out, one for each input — all from the same idx. That is precisely what snapy captures into its GPU lambda: auto offsets = offset_calc.get(bid); (§13). Decoding a linear index is the GPU-friendly restatement of it + k.

Pointer-arithmetic iteration = strided access

The macros in snapy's device code — INP(j,i) = inp[(j)*stride2 + (i)*stride1] — are hand-written strided iterators. The kernel is literally doing the address arithmetic an iterator would normally hide, because on the GPU you want that arithmetic inlined and register-resident.

4Iterators in PyTorch / Torch C++#

PyTorch has iterators at two altitudes:

  • Python level — for x in tensor yields slices along dim 0; DataLoader is an iterator over batches; torch.Tensor.unfold creates a strided sliding-window view (used by snapy's MPS path, §11). These are convenience iterators, not the compute engine.
  • ATen C++ level — at::TensorIterator, the workhorse behind nearly every element-wise and reduction operator (add, mul, copy_, sum, comparison ops…). This is what snapy plugs into.
The mental model When you write a + b in PyTorch on CPU or CUDA, ATen builds a TensorIterator over {out, a, b}, and a device-specific loop runner (cpu_kernel, gpu_kernel) walks it applying [](x,y){return x+y;}. snapy's Weno5 hijacks this exact machinery: it builds its own TensorIterator and supplies its own loop runner (stencil_kernel) and its own per-element body (a WENO reconstruction instead of an add).

5What TensorIterator is, precisely#

at::TensorIterator is a C++ object that, given a set of input and output tensors, plans and describes a unified iteration over them. It does not by itself run your math — it produces the metadata (shape, per-operand strides, base pointers, dtype) that a separate loop runner consumes.

6The build pipeline — what happens inside .build()#

TensorIteratorConfig(...).build() runs a fixed sequence of passes. Knowing them explains every flag snapy sets.

  1. populate_operands — record each tensor + whether it is in/out.
  2. mark_outputs / mark_resize — decide if outputs may be resized. snapy sets resize_outputs(false) because the output buffer is pre-allocated with an exact shape.
  3. compute_types — promote dtypes, optionally insert casts. snapy sets check_all_same_dtype(true): no hidden casts, fail loudly if mismatched.
  4. compute_shape — broadcast all operand shapes to one common shape. snapy instead pins it with declare_static_shape(out.sizes(), squash_dims) (next section).
  5. compute_strides — for each operand, per-dimension byte stride (0 for broadcast axes).
  6. reorder_dimensions — permute axes so the fastest-varying memory dim is innermost (cache/coalescing win).
  7. allocate_or_resize_outputs — if an output was passed as undefined, allocate it now.
  8. coalesce_dimensions — fuse adjacent dims whose strides are compatible, lowering N-D to as-few-D as possible.

The "static shape + squash" override

snapy does not want broadcasting. It wants total control of which axes are iterated and which are handed to the kernel whole. That is the role of declare_static_shape(shape, squash_dims):

  • declare_static_shape(shape) — pin the iteration shape, skip broadcast inference (faster build, deterministic).
  • squash_dims — list of axes to collapse out of the iteration. Their size becomes 1 in the iteration count, but their stride is preserved so the kernel can walk them itself. This is how snapy says: "don't iterate the variable axis or the reconstruction axis — give those to my device function as a block."

7Coalesce, reorder, squash — and why none of them copy memory#

The headline answer Coalescing, reordering, and squashing are all metadata-only. They rewrite the iterator's internal shape and stride arrays — the loop bounds and the per-step pointer increments — and never touch, move, or allocate the tensor's data buffer. The base data pointers are identical before and after. A physical copy only happens elsewhere, when an output must be allocated (§6 step 7) or when you explicitly call .contiguous() / .clone().

Each operation is a small transformation of two little arrays the iterator carries: shape[] (how many steps per axis) and, per operand, strides[] (how many bytes one step on that axis costs). The data stays put; only the description of how to walk it changes.

A C-contiguous (3,4) tensor has element strides (4,1). Walking axis 1 advances by 1, and after 4 steps you've advanced exactly to the next row — so the two axes describe one unbroken run of 12. If the tensor were a non-contiguous view (e.g. a sliced column with stride gaps), the boundary stride wouldn't match and the axes would not coalesce — the iterator keeps them separate. Coalescing is opportunistic, never forced.

Reorder: put the smallest stride innermost — why it wins the cache

CPUs (and GPUs) don't fetch one element at a time; they pull a whole cache line (~64 bytes ≈ 16 floats) on every miss. If the innermost loop walks memory with stride 1, each fetched line serves the next ~16 iterations → near-100% line utilization. If the innermost loop strides by a whole row, every iteration lands on a different line, evicting data you'll need again → "cache thrashing." Reorder sorts axes so the hot inner loop always has the smallest stride, regardless of the logical axis order you wrote.

Same idea on GPU = "memory coalescing": when the 32 threads of a warp read 32 consecutive addresses, the hardware merges them into one wide transaction. Strided/scattered reads split into many transactions. snapy's stencil_kernel deliberately maps the reconstruction axis onto threadIdx.x (§13/§16) so neighbouring threads read neighbouring memory — the GPU twin of putting the smallest stride innermost.

So: is it in place? Yes — reorder/coalesce/squash mutate the iterator's own metadata in place and reuse the existing data pointers. Your tensor is not transposed, copied, or re-laid-out. This is what makes building a TensorIterator cheap enough to do on every a + b.

8The iteration phase — how the plan gets walked#

CPU: for_each + grain size

c++ · ATen CPU loop driver pattern (mirrors snapy call_weno5_cpu)int grain = iter.numel() / at::get_num_threads();      // work per thread
iter.for_each([&](char** data, const int64_t* strides, int64_t n){
  for (int i = 0; i < n; i++) {                       // inner contiguous run of n elems
    auto out = reinterpret_cast<scalar_t*>(data[0] + i*strides[0]);
    auto w   = reinterpret_cast<scalar_t*>(data[1] + i*strides[1]);
    interp_weno5_impl(out, w, c, stride1, stride2, stride_out, nvar, scale);
  }
}, grain);                                              // >grain ⇒ at::parallel_for splits across threads

So on CPU the iterator is walked in parallel across threads (via at::parallel_for / TBB or OpenMP), each thread getting a contiguous sub-range; within a thread the inner for is serial and auto-vectorizable.

GPU: offset calculator + a launched kernel

On CUDA there is no "for loop" walking elements on the host. Instead the iterator produces an OffsetCalculator, and a kernel is launched so that every element index is computed by a different thread simultaneously. The vanilla ATen pattern (snapy's gpu_kernel in loops.cuh reproduces it):

cuda · the canonical element-wise GPU driverauto offset_calc = make_offset_calculator<Arity>(iter);   // linear idx → N offsets
int64_t numel = iter.numel();
launch_legacy_kernel<128,1>(numel, [=] __device__(int idx){
    auto offsets = offset_calc.get(idx);                  // runs on each thread
    f(data.data(), offsets.data());                       // your per-element body
});
The pivot point of this whole document On the GPU the TensorIterator never iterates. It is a compile-time-ish description that gets baked into an OffsetCalculator and shipped (by value, [=]) into a kernel. The actual "iteration" is the CUDA grid: thousands of threads each call offset_calc.get(my_index) in parallel. The iterator's role is purely addressing.

9Benefits & common issues#

Why teams reach for it

  • Write the kernel once, run everywhere — same body for CPU/CUDA, strided/contiguous, broadcast/not.
  • Free broadcasting & dtype promotion — no manual shape-matching.
  • Layout optimization — dim reorder + coalesce gives near-optimal memory access for free.
  • Built-in parallelism — CPU threading and GPU launch handled by the runner.
  • Autograd / dispatcher fit — slots straight into ATen's device dispatch (the REGISTER_*_DISPATCH macros).

Where people get burned

  • Dim coalescing surprises — you can't assume operand ndim or axis order survive; snapy sidesteps this with declare_static_shape + squash.
  • Hidden casts — if dtypes differ, buffered casting silently appears; check_all_same_dtype(true) guards it.
  • Resize footguns — an output left resizable can be reallocated under you; snapy uses resize_outputs(false).
  • 0-stride aliasing — broadcast inputs share memory; writing to a broadcast operand is undefined.
  • Not for stencils out of the box — element-wise runners give one element; neighbor access (WENO needs ±2) requires a custom runner, which is exactly why snapy wrote stencil_kernel.

10What Weno5 computes (just enough)#

WENO5 = Weighted Essentially Non-Oscillatory, 5th order reconstruction. In a finite-volume fluid solver it takes cell-averaged values along one spatial axis and reconstructs the value at a cell interface, using a 5-point stencil, blending three candidate 3-point polynomials with smoothness-dependent weights so the scheme avoids ringing near shocks.

For each interface it needs the 5 neighbors φ₀…φ₄, computes 3 candidate polynomials p0,p1,p2, 3 smoothness indicators β0,β1,β2, nonlinear weights α, and outputs the convex combination. The coefficients live in a 9×5 matrix cm (and its flip cp for the other-biased stencil):

snapy · weno5.cpp — the coefficient bufferscm = register_buffer("cm", torch::tensor({
   {-1./6, 5./6, 1./3, 0, 0},   // p0 reconstruction row
   {0, 1./3, 5./6, -1./6, 0},   // p1
   {0, 0, 11./6, -7./6, 1./3},  // p2
   {1,-2,1,0,0}, {1,-4,3,0,0},      // β0 (two quadratic forms)
   {0,1,-2,1,0}, {0,-1,0,1,0},     // β1
   {0,0,1,-2,1}, {0,0,3,-4,1}},     // β2
   torch::kFloat64));
cp = register_buffer("cp", cm.flip({1}));   // mirror stencil for right/left bias
The shape of the problem The input tensor w is laid out as (nvar, …spatial…): axis 0 is the physical variable (ρ, momentum, energy…), and one of the spatial axes dim is the reconstruction direction. The output along dim is shorter by stencils()-1 = 4. Every interface is independent — embarrassingly parallel. That independence is what the iterator exploits.

12The iterator config + the squash trick#

snapy · weno5.cpp — building the iteratorvoid Weno5InterpImpl::left(torch::Tensor w, int dim, torch::Tensor const& out) {
  std::vector<int64_t> squash_dim = {0};                 // always squash the variable axis
  if (w.device().is_cuda()) squash_dim.push_back(dim); // on GPU also squash the recon axis

  auto iter = at::TensorIteratorConfig()
      .resize_outputs(false)              // out is pre-sized exactly
      .check_all_same_dtype(true)         // no silent casts
      .declare_static_shape(out.sizes(), squash_dim)  // pin shape, remove squashed axes
      .add_output(out)
      .add_input(w)
      .build();

  at::native::call_weno5(out.device().type(), iter, cm, dim, options->scale());
}

Why CPU does not squash dim On CPU there is no shared memory and no block notion; the body just walks neighbors via raw strides (INP(j,i)=inp[j*stride2 + i*stride1]) over the full contiguous buffer. Squashing only dim 0 is enough. On GPU, squashing dim is what makes "one block = one reconstruction line" possible.

13stencil_kernel internals — the custom GPU runner#

This is the heart. ATen's stock gpu_kernel gives you one element; WENO needs a neighborhood and shared memory. So loops.cuh defines a custom runner that still consumes the TensorIterator's offsets but launches a 3-D block grid and hands each block a shared-memory scratchpad.

snapy · loops.cuh — stencil_kernel (annotated, trimmed)template <typename scalar_t, int Arity, typename func_t>
void stencil_kernel(at::TensorIterator& iter, int dim, int buffers, const func_t& f) {
  TORCH_CHECK(iter.ninputs() + iter.noutputs() == Arity);     // here Arity = 2 (out,in)

  std::array<char*, Arity> data;
  for (int i=0;i<Arity;i++) data[i] = (char*)iter.data_ptr(i);  // base ptrs from iterator

  auto offset_calc = make_offset_calculator<Arity>(iter);     // linear blockId → byte offsets
  int64_t numel = iter.input().numel();                       // # of squashed-out "blocks"

  // ---- derive a 3-D block (the squashed recon axis) and grid (the rest) ----
  int len[3] = {1,1,1};
  int ndim = iter.input().dim();
  len[3+dim-ndim] = ensure_nonempty_size(iter.input(), dim);   // recon axis length → block.x
  _left_shift<3>(len, dim+1-ndim);
  dim3 block(len[2], len[1], len[0]);                       // threads = points along recon axis
  // ... recompute len[] from the non-squashed sizes ...
  dim3 grid(len[2]/block.x, len[1]/block.y, len[0]/block.z); // blocks = independent columns

  int nvar  = ensure_nonempty_size(iter.output(), 0);
  size_t shared = (block.x*nvar + buffers) * sizeof(scalar_t);   // scratch: stencil + coeffs
  auto stream = at::cuda::getCurrentCUDAStream();                 // the op's CUDA stream

  reduce_kernel<scalar_t><<<grid, block, shared, stream>>>(       // LAUNCH all blocks at once
      numel, [=] __device__(int bid, scalar_t* smem) {
        auto offsets = offset_calc.get(bid);                     // this block's operand offsets
        f(data.data(), offsets.data(), smem);                  // → your weno5 device fn
      });
  C10_CUDA_KERNEL_LAUNCH_CHECK();
}
snapy · loops.cuh — reduce_kernel (the launched __global__)template <typename scalar_t, typename func_t>
__global__ void reduce_kernel(int64_t numel, func_t f) {
  // global thread id across the whole grid…
  int tid = /* x + y*… + z*… */;
  // …and the BLOCK id — this is the index fed to the offset calculator
  int bid = blockIdx.x + blockIdx.y*gridDim.x + blockIdx.z*gridDim.x*gridDim.y;

  extern __shared__ unsigned char memory[];        // dynamic shared mem
  scalar_t* smem = reinterpret_cast<scalar_t*>(memory);
  if (tid < numel) f(bid, smem);                  // every block runs the body concurrently
}
How the pieces line up
  • offset_calc.get(bid) turns the block index into a byte offset per operand — this is the TensorIterator doing its one job: addressing. It is what lets the kernel handle non-contiguous / arbitrarily strided tensors without knowing the layout.
  • block.x = number of points along the reconstruction axis → each thread owns one interface; neighbors live in the same block's shared memory.
  • grid = every other (non-squashed) column → all launched at once.
  • shared = block.x*nvar + buffers → room to stage the stencil for all variables plus the coefficient table.

14The device function — what one block does#

The lambda registered by call_weno5_cuda unpacks the offsets into typed pointers and calls interp_weno5_impl. The CUDA dispatch wrapper:

snapy · interp_impl.cuh — the per-block device fn (trimmed)template <typename T>
__device__ void interp_weno5_impl(T* out, T* inp, T* coeff, int nvar, ... , T* smem) {
  int id = threadIdx.x;          // this thread = one interface along recon axis
  int nt = blockDim.x;           // block width = recon-axis length

  // 1) cooperatively stage the stencil line for every variable into shared mem
  T* sinp = smem;
  for (int j=0;j<nvar;j++) sinp[id + j*nt] = INP(j, id);

  // 2) cooperatively load the 45 coefficients once per block
  T* scoeff = smem + nt*nvar;
  constexpr int N = 45;
  for (int i=id;i<N;i+=nt) scoeff[i] = coeff[i];

  if (id > nt - 5) return;       // last 4 threads have no full 5-pt stencil → drop
  __syncthreads();              // barrier: shared mem now fully populated

  // 3) each remaining thread does the full WENO5 math from shared mem
  for (int j=0;j<nvar;j++) {
    int i = id + j*nt; T phi[5];
    // optional rescale, candidates p0..p2, smoothness β0..β2, weights α…
    OUT(j,id) = vscale*(a0*p0 + a1*p1 + a2*p2)/(a0+a1+a2);
  }
}
The "you only write per-block ops" insight Your code never says "launch 4096 blocks" or "loop over every column." You write what one block of one reconstruction line does — load its stencil to shared memory, sync, compute. The TensorIterator + stencil_kernel replicate that body across the entire grid, each block addressed by offset_calc.get(bid). Composition of (iterator = where) × (device fn = what) = the full sweep.

15Why you only need to write per-block operations#

Three layers cooperate, each erasing a concern from the layer above:

Layer 3 — your body
"For one stencil line: stage to shared mem, sync, compute WENO over nvar." No indices into global memory, no grid math.
→
Layer 2 — stencil_kernel
Turns the iteration space into a grid; gives each block a shared-mem scratchpad and the right base offset via the iterator.
→
Layer 1 — TensorIterator
Knows every operand's strides & base pointer; produces offset_calc so any layout maps to correct addresses.

Because Layer 1 guarantees addressing and Layer 2 guarantees replication + parallel launch, Layer 3 can be written as if it were the only block in existence. That is the entire ergonomic payoff.

The boundary you still own Two things the iterator can't decide for you and that the device fn must handle: (1) halo/edge threads — the if (id > nt-5) return; drops threads without a full 5-point stencil; (2) shared-memory cooperation — staging + __syncthreads() is your responsibility because the iterator has no concept of a block.

16Sequential or parallel? The grid/block geometry#

Parallel — fully. The iterator itself does not loop on the GPU. The parallelism comes from the CUDA launch reduce_kernel<<<grid, block, shared, stream>>>, which schedules all blocks at once (subject to the GPU's SM occupancy). Mapping:

ConceptCUDA entityBuilt from
One reconstruction line (the squashed dim)block (blockDim.x = line length)ensure_nonempty_size(input, dim)
One interface within a linethread (threadIdx.x)—
All independent columns (non-squashed spatial axes)grid (gridDim.{x,y,z})iteration numel / block
Per-variable inner work (nvar)serial for j inside each threadsquashed dim 0

So the traversal is parallel across lines (grid) and across interfaces within a line (threads), and serial only over the handful of physical variables inside each thread. Every block runs f(bid, smem) independently and concurrently; there is no host-side loop launching them one by one.

The _left_shift rotation The fiddly len[3+dim-ndim] + _left_shift<3> dance is just packing an arbitrary-rank tensor's relevant axes into CUDA's fixed 3-slot dim3 while making the reconstruction axis land on block.x (the contiguous, warp-friendly dimension). It is bookkeeping to keep neighbors adjacent in threads.

The contrast in one line In cuDF the iterator is pure addressing + value + validity; the parallelism and kernel launch live entirely inside Thrust/CUB, not in the iterator. That mirrors §3 (the C++ iterator concept) + §5's "a runner walks it" split — whereas PyTorch fuses planning and running into one TensorIterator object and lets you swap the runner (snapy's stencil_kernel). When Thrust can't express the access pattern, cuDF — exactly like snapy — drops to a hand-written __global__ kernel that captures column_device_views directly.
Version caveat The cudf::detail::make_*_iterator names and signatures live in internal detail:: headers and have shifted across RAPIDS releases (the move toward optional-based iterators and nullate policy types is recent). The architecture above — Thrust/CUB + fancy iterators + column_device_view + type_dispatcher + RMM — is stable and correct; treat exact spellings as version-dependent.

19Generalizing the pattern — your reusable recipe#

To add a new strided / stencil operator to a repo like snapy, repeat this five-part skeleton:

  1. Declare the dispatch slot — using fn = void(*)(TensorIterator&, …); DECLARE_DISPATCH(fn, my_op); in a header.
  2. Build the iterator at the call site — TensorIteratorConfig().resize_outputs(false).check_all_same_dtype(true).declare_static_shape(out.sizes(), squash_dims).add_output(out).add_input(w).build();
  3. Write the device/CPU body — a function that takes typed pointers + strides + (GPU) shared mem and computes one block / one element.
  4. Pick a runner — element-wise → ATen gpu_kernel/cpu_kernel; neighbor-aware → a custom stencil_kernel.
  5. Register per backend — REGISTER_ALL_CPU_DISPATCH, REGISTER_CUDA_DISPATCH, optionally REGISTER_MPS_DISPATCH (high-level tensor ops).

20Assumptions & limitations to respect#

  • Dtype homogeneity — check_all_same_dtype(true) means you must pass operands of one floating type; mixed types throw. AT_DISPATCH_FLOATING_TYPES only covers float/double (no half/bfloat16 here).
  • Output is pre-sized — resize_outputs(false) assumes you allocated out with exactly the reduced shape (size(dim) − (stencils−1)). A wrong-sized buffer is UB, not an error.
  • Block dimension limits — the reconstruction axis maps to blockDim.x, capped at 1024 threads and by shared-memory budget (block.x*nvar + 45 scalars). Very long lines need tiling (not implemented here).
  • Grid divisibility — grid = len/block uses integer division; the geometry assumes the non-squashed extents divide evenly. Shapes that don't fit the assumed packing can silently drop work.
  • Stencil staging is per-block — neighbors must live within one block (one line). Cross-block halos are not exchanged; the squash puts the entire line in a block precisely to avoid that.
  • Edge threads dropped — the last stencils−1 threads return early; the output axis being shorter must match exactly or you read past the buffer.
  • No autograd through the custom kernel — this is a raw forward kernel; gradients (if needed) require a separate backward.
  • 32-bit index assumption — stencil_kernel asserts numel ≤ INT32_MAX; enormous tensors would overflow the offset math.

21Tips & gotchas for using the iterator well#

Building

  • Prefer declare_static_shape + squash when you know the layout — it skips broadcast inference and prevents dim-reorder surprises that would scramble your hand strides.
  • Pull strides from the operand tensors (ensure_nonempty_stride(iter.input(), dim)), not from assumptions — they survive coalescing.
  • Use iter.common_dtype() (GPU) vs iter.dtype() (CPU output) to match what the dispatch macro expects.

Launching

  • Always wrap CUDA work in CUDAGuard(iter.device()) and launch on getCurrentCUDAStream() — never the default stream.
  • Capture by value ([=]) into __device__ lambdas; capturing references to host stack = garbage on device.
  • Compute shared-mem size from real extents (block.x*nvar + buffers) and check it against the device limit before scaling up.
  • Keep __syncthreads() after cooperative staging and before any thread reads a neighbor's slot.

What does TensorIterator do on GPU?Nothing iterative — it produces base pointers + an OffsetCalculator (addressing only). The CUDA grid does the iterating.
Sequential or parallel?Parallel. All blocks launch at once; threads within a block run concurrently; only the nvar loop is serial per thread.
Why per-block code only?The iterator guarantees correct addressing and stencil_kernel replicates your body across the grid, so you write one block's worth of work.
What is "squash"?Axes removed from the iteration space (size→1, stride kept). nvar always; recon dim on GPU so a whole line lands in one block.
How does a block reuse work?Cooperative shared-memory staging of the stencil + 45 coefficients, one __syncthreads(), then per-thread WENO over all variables.
How is the right backend chosen?DispatchStub: call_weno5(device_type, …) jumps to the CPU/CUDA/MPS function registered with REGISTER_*_DISPATCH.

Why does reorder help cache?Smallest-stride axis innermost ⇒ consecutive accesses share a cache line (CPU) / coalesce into one warp transaction (GPU).
What is a "contiguous chunk" dispatch?The runner hands your body (ptr, stride, n) for a whole run instead of one scalar — one call, an inner SIMD-vectorizable loop.
