FRAME Chapter I

The frame · pipelines, reconcilers, provisioners.

Three roles Pipelines execute imperative steps in order. Reconcilers continuously drive state toward a declared target. Provisioners create the substrate everything else assumes.

The rule Each role has a canonical tool in 2026. Mixing them — running provisioning through a pipeline tool, or treating reconciliation like a pipeline — produces the recognizable failure modes.

Most platform engineering complexity in 2026 boils down to three orthogonal jobs. Pipelines run a sequence of steps once, in order, with explicit success and failure. Reconcilers continuously compare desired state to actual state and apply differences. Provisioners create the underlying infrastructure — clusters, VPCs, IAM, registries — that everything else assumes. Different timescales, different idempotency models, different audit shapes. Pick the right tool for each role; the toolchain shines. Conflate them; the toolchain becomes the bottleneck.

Why this matters operationally

The three roles fail differently, and the operator skill required is different. A pipeline failure is a single bounded event — you read the log of the failed step, fix the input, re-run. A reconciler failure is continuous — the cluster keeps drifting and the reconciler keeps unable-to-converge until someone notices in the dashboard. A provisioner failure can be catastrophic — a botched plan applied to production state can destroy infrastructure that took months to build. Each requires a different operational posture. Pipelines: good observability and good log retention. Reconcilers: drift dashboards and health alerts. Provisioners: plan-before-apply gating, state file backups, and very serious approval workflows.

The overlap zone — and why it confuses

The boundary is fuzzy on purpose. Argo CD reconciles, but its sync operation is pipeline-like. Argo Workflows orchestrates pipelines, but its WorkflowTemplate CRD is reconciled by a controller. Terraform's plan/apply is pipeline-shaped, but state-as-source-of-truth is reconciliation-flavored. Crossplane explicitly fuses provisioning with reconciliation. Tools borrow from each other; the roles remain distinct.

The pragmatic rule for picking: what is the natural unit of operation? If it's "trigger this once when something happens" — pipeline. If it's "this should always be true" — reconciler. If it's "this should exist and survive across the platform's lifetime" — provisioner.

The 2026 canonical stack

Most production platforms in 2026 use one tool per role:

  • Provisioner — Terraform or OpenTofu for cloud infrastructure (VPCs, clusters, IAM, databases). Crossplane increasingly for in-cluster provisioning APIs (cross-reference Issue 10).
  • Reconciler — Argo CD or Flux for Kubernetes application state. Standard K8s controllers (cert-manager, external-secrets, operators) for in-cluster state.
  • Pipeline — Argo Workflows for K8s-native pipelines (ML training, data processing, complex CI). GitHub Actions or similar for application CI. Tekton in some OpenShift shops.

FRAME Chapter II

The two Argos · confusable names, different roles.

The confusion "Argo" is a project umbrella. "Argo Workflows" and "Argo CD" share branding but solve entirely different problems.

The rule Workflows runs pipelines; CD reconciles state. Both are K8s CRDs. Both speak YAML. They are not substitutes.

"Argo" is one of the most successful CNCF umbrella brands — and one of the most confusing. The project covers four mature subprojects: Argo Workflows (pipelines), Argo CD (GitOps reconciliation), Argo Rollouts (progressive delivery), and Argo Events (event-driven triggers). The first two share little except branding; mistaking them for variants of the same thing leads to architectural confusion that takes months to unwind. This chapter pins the distinction, then walks the four together so you know what each tool is for.

How to remember

The shortest mnemonic: "Workflows is a Job; CD is a Deployment." A Job runs to completion and finishes. A Deployment maintains a stable target state indefinitely. The same dichotomy at the platform level: Workflows runs a pipeline to completion; CD maintains the cluster state indefinitely against Git as target.

The right combinations

The two often appear together in production:

  • Argo Workflows for CI, Argo CD for CD. The most common pattern. Workflows builds and tests; pushes image and updated manifest to Git; Argo CD reconciles the cluster to the new manifest.
  • Argo Workflows for data/ML pipelines, Argo CD for the platform. The workflows are the workload; Argo CD ensures the workflows are scheduled correctly and that the workflow-controller itself is running.
  • Argo Workflows + Events for event-driven pipelines. Events watches an S3 prefix, a Kafka topic, a webhook; triggers a Workflow on each event. Argo CD ensures the Events and Workflows controllers are deployed and configured.
  • Argo CD + Rollouts for progressive delivery. Argo CD syncs the Rollout CRD; Rollouts manages the canary/blue-green within each cluster.

The wrong combinations

  • "Use Argo Workflows for deployment." Workflows can apply YAML, but it runs once. The next time the cluster drifts, no one reconciles. You've lost the reconciliation property entirely. Use Argo CD.
  • "Use Argo CD for CI." Argo CD doesn't build images or run tests. It expects manifests as input. Use Workflows or another CI tool for the build phase.
  • "Use Workflows to manage long-running services." Workflows is for jobs with bounded duration. A service that should run forever is a Deployment, not a Workflow.
The naming tax

The Argo project's branding choice has cost the ecosystem many onboarding hours. The lesson for anyone designing platform tools: if two products share a brand prefix, they will be confused. The Argo team's response — better docs and emphatic explanation — works, but the cost compounds. When you adopt Argo Workflows and Argo CD together, train your team explicitly on the distinction. It does not become obvious by osmosis.

WORKFLOW Chapter III

Argo Workflows · the mechanism.

The architecture Workflow CR is the spec. The workflow-controller watches Workflow CRs and creates a pod per step. Each pod has a main container (the user step) and an init+wait sidecar (orchestration).

The strength Pure K8s-native. No external scheduler, no agent on each node. Scales by adding pods.

Argo Workflows is implemented as a Kubernetes CRD with a single controller that orchestrates pods. Understanding the mechanism — what the controller does, what each pod contains, where state lives — is the difference between operating Workflows well and being confused when something hangs at scale. This chapter walks the architecture; later chapters layer on templates, scaling, and the failure modes that come with each.

The components

Argo Workflows architecture
The controller orchestrates; each step is its own pod

What each pod actually does

Three containers per step pod:

  • init container — runs first. Downloads input artifacts (from S3, GCS, OCI, etc.) into the shared volume that main will read from.
  • main container — the user-defined image and command. Reads inputs from the shared volume; writes outputs to the shared volume. Knows nothing about Argo.
  • wait sidecar — runs alongside main. When main exits, the wait container captures the exit code, uploads output artifacts (to S3, GCS, etc.), reports status back to the controller via annotation on the pod. This is how Argo gets results without the user container needing to know about Argo.

The crucial implication: user containers are unchanged from any container you'd run elsewhere. There's no SDK to install, no client library to import, no DSL to learn. The orchestration is purely external — provided by the controller and the wait sidecar. This is why Argo Workflows fits ML and data workloads so well: existing containers run without modification.

        image: registry.example.com/publisher:v1.0
        command: [python, /app/publish.py]

The submission paths

Workflows can be triggered in four canonical ways:

  • kubectl / argo CLI. Direct argo submit workflow.yaml. Operator-driven.
  • CronWorkflow. Recurring workflows on a schedule. spec.schedule: "0 */6 * * *". The controller materializes a new Workflow CR each tick.
  • WorkflowTemplate + workflow-of-references. Reusable workflow definitions; instances reference them via templateRef. The 2026 standard for shared pipeline libraries.
  • Argo Events. EventSource (S3, Kafka, webhook, schedule) → EventBus → Sensor → fires Workflow. This is how event-driven pipelines are built.

Where state lives

This matters at scale. Three layers:

  • Live workflow status. Stored in the Workflow CR's status field, in etcd. Includes per-node status, retry counts, current phase. Updated by the controller on every transition.
  • Completed workflow archive. Optional. The controller can offload completed Workflow CRs to a SQL database (Postgres/MySQL). This dramatically reduces etcd pressure for high-volume environments.
  • Artifacts. Inputs and outputs of steps. Stored in object storage (S3, GCS, MinIO). The wait sidecar uploads on step completion; the init container of downstream steps downloads.

The etcd pressure problem

The single biggest scale issue with Argo Workflows in 2026 — and the root cause of most "the cluster slowed down after we adopted Argo Workflows" reports. Each Workflow CR's status field grows with the number of steps. Hundreds of completed Workflows × KBs of status × frequent updates = etcd write pressure. The fix is the workflow archive: enable it, set a short TTL on completed Workflow CRs, and the controller offloads to SQL. Per the CNOE scaling tests on EKS, "if not tuned properly, Argo Workflows can negatively impact the scalability of a given Kubernetes cluster, particularly if the Kubernetes cluster happens to be the control cluster managing developer workflows." Chapter VI covers the tuning in depth.

The mental model

Argo Workflows is one controller, plus one pod per step, plus optional archive. Everything else is configuration. The simplicity is what makes it scale — no separate scheduler, no agent on each node, no language-specific runtime. It's also what makes the failure modes specific: too many Workflow CRs in etcd is the single most common scale failure, and the fix is always the archive.

WORKFLOW Chapter IV

Templates, DAGs, steps · the vocabulary.

The three shapes Steps (sequential), DAG (graph of dependencies), template references (composition).

The pattern DAG for complex pipelines, steps for simple sequences, WorkflowTemplate for reuse. Mix freely.

Argo Workflows offers two ways to express the structure of a pipeline. Steps are an explicit sequence — step 1, then step 2, then step 3 — with optional parallelism within each step. DAGs are a graph where each task lists its dependencies; the controller infers execution order. Both styles compose via template references, which let you build pipeline libraries and assemble them at runtime. Choosing among the three is mostly a readability question, but each has nuances worth knowing.

Steps style — explicit sequence

templates:
  - name: ml-pipeline
    steps:
      - - name: download-data        # step 1 — single task
          template: download
      - - name: preprocess-features  # step 2 — TWO tasks in parallel
          template: preprocess
        - name: preprocess-labels
          template: preprocess
      - - name: train                # step 3 — runs after step 2
          template: train

The YAML convention: - - (two dashes) starts a new step; - (one dash) adds a task to the current step. Tasks in the same step run in parallel; steps run sequentially. Easy to read; rigid about ordering. Good for linear pipelines with occasional fan-out.

DAG style — declared dependencies

templates:
  - name: ml-pipeline-dag
    dag:
      tasks:
        - name: download-data
          template: download
        - name: preprocess-features
          template: preprocess
          dependencies: [download-data]
        - name: preprocess-labels
          template: preprocess
          dependencies: [download-data]
        - name: train
          template: train
          dependencies: [preprocess-features, preprocess-labels]
        - name: evaluate
          template: evaluate
          dependencies: [train]
        - name: deploy
          template: deploy
          dependencies: [evaluate]
          when: "{{tasks.evaluate.outputs.parameters.accuracy}} >= 0.85"

The DAG style is the right choice when fan-out and fan-in patterns become common, or when the dependency graph isn't a clean sequence. The when conditional shown above — conditional execution based on upstream task outputs — works in both styles but is more natural in DAG form.

The DAG/Steps confusion that bites

In steps style, tasks within the same step run in parallel. In DAG style, tasks with no dependencies run in parallel. Engineers coming from Airflow or Tekton sometimes expect a different semantic ("declared tasks run sequentially by default"). The Argo semantic is the opposite — parallelism is default; dependencies create ordering. Get this wrong and your pipeline either races (if you expected sequence) or serializes unnecessarily (if you over-declared dependencies).

Template references — composition

Real production pipelines are not single Workflow YAMLs. They reference reusable templates from WorkflowTemplate (namespaced) or ClusterWorkflowTemplate (cluster-scoped):

apiVersion: argoproj.io/v1alpha1
kind: WorkflowTemplate
metadata:
  name: data-prep-templates
  namespace: pipelines
spec:
  templates:
    - name: download-from-s3
      inputs:
        parameters: [{name: bucket}, {name: key}]
      outputs:
        artifacts: [{name: data, path: /out/data.parquet}]
      container:
        image: registry.example.com/s3-fetch:v1
        env:
          - name: BUCKET
            value: "{{inputs.parameters.bucket}}"
          - name: KEY
            value: "{{inputs.parameters.key}}"
apiVersion: argoproj.io/v1alpha1
kind: Workflow
spec:
  entrypoint: main
  templates:
    - name: main
      dag:
        tasks:
          - name: download
            templateRef:                       # reference the template
              name: data-prep-templates
              template: download-from-s3
            arguments:
              parameters:
                - { name: bucket, value: my-bucket }
                - { name: key, value: data/2026/jan.parquet }

The 2026 pattern for serious pipeline platforms: a shared WorkflowTemplate repository contains 50-200 reusable "step types"; individual Workflows reference them by name. Updates to the template propagate to new pipeline runs. The team running the platform owns the template library; teams using the platform compose pipelines from the library plus their own steps.

WorkflowTemplate vs ClusterWorkflowTemplate

WorkflowTemplate is namespace-scoped. ClusterWorkflowTemplate is cluster-scoped. The same composition mechanism otherwise. The 2026 default is namespace-scoped + Capsule-style tenant scoping (cross-reference Issue 12 Ch V) — each tenant's WorkflowTemplate library lives in their namespace; ClusterWorkflowTemplate is reserved for genuinely cluster-wide platform templates.

Recursion and loops

Workflows supports recursive templates (a template that references itself) and dynamic fan-out via withItems / withParam:

templates:
  - name: process-shards
    steps:
      - - name: shard
          template: process-one-shard
          arguments:
            parameters: [{name: shard-id, value: "{{item}}"}]
          withItems:                       # static list
            - shard-001
            - shard-002
            - shard-003

  - name: process-shards-dynamic
    inputs:
      parameters: [{name: shard-list}]     # JSON array string
    steps:
      - - name: shard
          template: process-one-shard
          arguments:
            parameters: [{name: shard-id, value: "{{item}}"}]
          withParam: "{{inputs.parameters.shard-list}}"

Dynamic fan-out is what enables the BYD-scale numbers (Chapter VI) — one Workflow CR with withParam over 1,000 items produces 1,000 step pods. The controller manages all of them as children of one parent Workflow.

The shape choice in practice

Default to DAG for non-trivial pipelines — even simple sequences read fine as a chain of dependencies, and you don't have to rewrite when fan-out becomes needed. Use steps for genuinely linear 2-3 step sequences where the parallel-within-step idiom is natural. Always extract reusable steps into WorkflowTemplates; never duplicate container specs across Workflows. The cost of refactoring later is far greater than the cost of templating up front.

WORKFLOW Chapter V

Artifacts, parameters, exit handlers.

The data flow Parameters pass small values (strings, numbers, JSON). Artifacts pass files via object storage. Exit handlers run cleanup or notification regardless of success or failure.

The rule Use parameters for ≤200 KB strings. Use artifacts for everything bigger or file-shaped.

A pipeline is mostly its data flow. Argo Workflows separates this into parameters (small values that flow through the controller and end up in pod annotations and env vars) and artifacts (files that flow through object storage). The split matters for performance — parameters add to etcd Workflow CR size; artifacts don't. Exit handlers wrap the whole pipeline with cleanup or notification logic. Together they cover the common pipeline patterns.

Parameters — for the small stuff

templates:
  - name: train-and-publish
    inputs:
      parameters:
        - name: epochs
          value: "10"
        - name: learning-rate
          value: "0.001"
    dag:
      tasks:
        - name: train
          template: trainer
          arguments:
            parameters:
              - name: epochs
                value: "{{inputs.parameters.epochs}}"
              - name: lr
                value: "{{inputs.parameters.learning-rate}}"

        - name: publish
          template: publisher
          dependencies: [train]
          arguments:
            parameters:
              - name: model-version
                value: "{{tasks.train.outputs.parameters.version}}"  # output → input

Parameter outputs are read by the controller from a file the user container writes:

templates:
  - name: trainer
    inputs:
      parameters: [{name: epochs}, {name: lr}]
    outputs:
      parameters:
        - name: version
          valueFrom:
            path: /tmp/model-version.txt    # wait sidecar reads this
        - name: accuracy
          valueFrom:
            path: /tmp/accuracy.txt
    container:
      image: registry.example.com/trainer:v2.1
      command: [python, train.py]
      args: ["--epochs", "{{inputs.parameters.epochs}}", "--lr", "{{inputs.parameters.lr}}"]
      # User container writes /tmp/model-version.txt and /tmp/accuracy.txt
      # wait sidecar captures content and reports to controller via pod annotation

The parameter mechanism is elegant but bounded. Pod annotations cap at 256 KB; in practice keep individual parameters under 200 KB and total per-pod parameters under 1 MB. Anything larger goes through artifacts.

Artifacts — for files

templates:
  - name: data-prep
    outputs:
      artifacts:
        - name: features
          path: /workspace/features.parquet    # wait sidecar uploads this
          archive: { none: {} }                # don't tar/gz
    container:
      image: registry.example.com/prep:v1
      command: [python, prep.py, --out, /workspace/features.parquet]

  - name: trainer
    inputs:
      artifacts:
        - name: features
          path: /workspace/features.parquet    # init container downloads to here
    container:
      image: registry.example.com/trainer:v2
      command: [python, train.py, --in, /workspace/features.parquet]

The artifact backend is configured cluster-wide (S3, GCS, MinIO, Azure Blob) and used by the wait sidecar and init container transparently. The user container reads and writes local paths; the orchestration handles upload/download. Per-Workflow artifact key prefixes prevent collision.

Artifact garbage collection

One of the more common operational pain points: artifacts accumulate forever in object storage if not actively cleaned up. Argo offers artifactGC at workflow or template level — the controller schedules a GC pod that deletes the artifact after the workflow completes (or after Workflow CR deletion). For high-volume environments, enabling artifactGC is mandatory; otherwise S3 bills compound monthly.

spec:
  entrypoint: main
  artifactGC:
    strategy: OnWorkflowDeletion         # or OnWorkflowCompletion
  templates:
    - # ...

Exit handlers

An onExit template runs when the workflow finishes, regardless of success or failure. The standard uses: send a notification (Slack, PagerDuty), clean up external resources, post metrics, gate next-stage workflows.

spec:
  entrypoint: main
  onExit: notify
  templates:
    - name: main
      # ... pipeline ...
    - name: notify
      steps:
        - - name: success
            template: slack-success
            when: "{{workflow.status}} == Succeeded"
          - name: failure
            template: slack-failure
            when: "{{workflow.status}} != Succeeded"

Retry strategies

Per-template retry policies handle the common case of transient failures (network blip, scheduling delay, spot instance preemption):

templates:
  - name: flaky-step
    retryStrategy:
      limit: 5
      retryPolicy: OnFailure              # OnFailure, OnError, Always, OnTransientError
      backoff:
        duration: "30s"
        factor: 2
        maxDuration: "10m"
      affinity:
        nodeAntiAffinity: {}              # retry on a different node — escape bad host
    container:
      # ...

Timeouts and active deadlines

Pipeline-level activeDeadlineSeconds caps the total workflow runtime. Template-level timeout caps individual step runtime. Without these, runaway workflows (infinite loop in user code, missed signal) consume cluster capacity indefinitely. The 2026 best practice is per-template timeouts with sensible defaults — e.g., 1 hour for most steps, longer only for steps known to be long-running.

The artifact size watershed

Most operational pain in production Argo Workflows comes from one of two places: too many active Workflow CRs in etcd (Chapter VI: enable the archive), or unbounded artifact growth in object storage (this chapter: enable artifactGC). Neither is hard to fix in advance; both are painful to fix in retrospect. Set both up before going to production scale.

WORKFLOW Chapter VI

Scaling Argo Workflows · BYD & the million-task frontier.

The proven ceiling BYD runs 20-40K concurrent workflows, 200K queued tasks, 99% success rate, 50ms p99 high-priority queue. Argo Workflows scales to autonomous-driving-data scale.

The levers Workflow archive, controller sharding, workflow workers, client QPS, separate node pools, semaphore-based concurrency control.

Argo Workflows has proven scalable to numbers most engineers wouldn't believe a single CRD-based system could handle. The reference 2026 data point: BYD, processing one petabyte per day of autonomous driving data. This chapter walks the levers that make that work, the failure modes when they aren't tuned, and the metrics that surface trouble before it becomes an outage.

The BYD reference point

Presented at ArgoCon Europe 2026. BYD's autonomous driving group migrated from Airflow to Argo Workflows for petabyte-per-day data processing. The transformation delivered staggering improvements in performance and operational efficiency: 11x Faster Execution — data processing pipelines now run eleven times faster than the previous Airflow-based setup. 30% Cost Reduction — smarter resource scheduling and elastic scaling significantly lowered infrastructure spend. Massive Throughput — the system now supports a queue capacity of 200,000 tasks and handles 20,000 to 40,000 concurrent workflows. Ultra-Low Latency — high-priority tasks experience queue delays as low as 50 milliseconds. 99% Success Rate — the enhanced stability ensures almost every workflow reaches completion.

The relevance: even if your scale is 1% of BYD's, the same levers apply. Tuning Argo Workflows is mostly the same problem at every scale; the operating points just shift.

Lever 1 — Workflow archive (etcd offload)

The single highest-impact tuning. Without the archive, completed Workflow CRs accumulate in etcd. With 100K+ completed workflows, etcd memory pressure leads to slow list operations, slow controller reconciliation, and eventually cluster-wide API server degradation.

apiVersion: v1
kind: ConfigMap
metadata:
  name: workflow-controller-configmap
  namespace: argo
data:
  persistence: |
    archive: true
    archiveTTL: "168h"                  # 7 days — adjust to compliance needs
    postgresql:
      host: argo-postgres.argo.svc
      port: 5432
      database: argo
      tableName: argo_workflows
      userNameSecret: { name: argo-postgres-creds, key: username }
      passwordSecret: { name: argo-postgres-creds, key: password }
  workflowDefaults: |
    spec:
      ttlStrategy:
        secondsAfterCompletion: 3600     # delete CR 1h after completion
        secondsAfterSuccess: 600          # 10min for successes
        secondsAfterFailure: 86400        # 24h for failures (debug window)

The TTL strategy ensures the controller actively deletes completed Workflow CRs from etcd; the archive holds the history in Postgres for searchability via the UI. The result: etcd Workflow CR count stays bounded, regardless of pipeline volume.

Lever 2 — workflow-controller tuning

The single controller process orchestrates everything. Its concurrency settings determine how many workflows and how many step-pod creations can happen in parallel.

args:
  - --workflow-workers=64                  # concurrent workflow reconciles (default 32)
  - --pod-workers=32                       # concurrent pod-event handlers
  - --pod-cleanup-workers=8                # pod GC workers
  - --workflow-ttl-workers=8               # TTL-based deletion workers
  - --qps=100                              # K8s client QPS (default 20)
  - --burst=200                            # K8s client burst (default 30)

Per the CNOE scaling tests: existing mechanisms for increasing workflow workers, increasing client and burst QPS settings and sharding the controller can help Argo Workflows scale better. Another interesting observation is that we saw differences in performance with how you submit your workflows.

Lever 3 — Controller sharding

For very large environments, a single controller instance can become CPU-bound. Argo Workflows supports controller sharding via --namespaced + multiple deployments, each managing a subset of namespaces:

# controller-shard-1 manages namespaces matching shard-1 label
- args:
  - workflow-controller
  - --namespaced
  - --managed-namespace=team-a
  - --managed-namespace=team-b
  # ...

# controller-shard-2 manages team-c, team-d
- args:
  - workflow-controller
  - --namespaced
  - --managed-namespace=team-c
  - --managed-namespace=team-d

Lever 4 — Separate node pools

Per the 2026 scaling guide: scaling Argo Workflows involves tuning both your Kubernetes cluster and the Argo controller. Use separate node pools for controller and executor workloads. Provision high-memory nodes for data-intensive steps. The controller pod itself is sensitive to noisy neighbors; isolating it on its own node prevents step-pod resource pressure from delaying reconciliation.

Lever 5 — Semaphore-based concurrency control

BYD's "queue capacity 200,000 tasks" works because Argo Workflows has built-in semaphores that throttle parallel execution at the workflow or template level:

apiVersion: v1
kind: ConfigMap
metadata: { name: workflow-semaphores }
data:
  gpu-pool: "8"                       # 8 concurrent users of the gpu-pool semaphore
  db-writers: "3"                     # 3 concurrent DB-writing steps cluster-wide
---
apiVersion: argoproj.io/v1alpha1
kind: WorkflowTemplate
spec:
  synchronization:
    semaphore:
      configMapKeyRef:
        name: workflow-semaphores
        key: gpu-pool

The semaphore mechanism enables the queue model: thousands of workflows can be submitted; only N actually execute concurrently; the rest wait. This is how you achieve high overall throughput without overwhelming downstream systems (databases, GPUs, external APIs).

Lever 6 — Pod cleanup

Completed pods sit in etcd until garbage collected. For high-volume pipelines:

spec:
  podGC:
    strategy: OnPodCompletion           # or OnWorkflowSuccess, OnWorkflowCompletion
    labelSelector:
      matchLabels:
        retain-for-debug: "false"

OnPodCompletion deletes the step pod as soon as it finishes — most aggressive; right for high-volume environments. OnWorkflowCompletion keeps pods for the duration of the workflow (useful for debugging). The label selector lets you keep specific pods around regardless of the strategy.

CD Chapter VII

Argo CD · the reconcile loop.

The four components argocd-server (UI/API, stateless), repo-server (fetches Git, renders manifests), application-controller (reconciles state), Redis (cache).

The loop Every 3 minutes (configurable): fetch Git → render manifests → diff against cluster → apply if drift → cache result.

Argo CD's elegance comes from a small set of components that together implement a remarkably simple loop. Understanding what each component does — and where it sits in the loop — is the foundation for both day-to-day operation and the heavy scale tuning required for thousands of applications. This chapter walks the architecture; Chapter IX walks the scale levers built on top.

The four components

Argo CD architecture
Four components, one reconcile loop, optional Redis cache

What each component does

  • argocd-server. The UI and API server. Stateless. Authenticates users (OIDC/SSO), enforces RBAC, exposes the CLI/UI. Per the official docs, "the argocd-server is stateless and probably the least likely to cause issues."
  • repo-server. Clones Git repos, runs Helm template / Kustomize build / plain YAML / Jsonnet, returns rendered manifests. Stateless; scales by adding replicas. Most CPU-intensive component for repos with complex Helm charts.
  • application-controller. The heart of the reconcile loop. Watches Application CRs; for each one, fetches current desired state from repo-server, fetches current live state from the target cluster API, diffs, applies changes if syncPolicy permits. Runs as a StatefulSet (stable identity for sharding). The bottleneck at scale.
  • Redis. Cache for rendered manifests, application trees, status data. Reduces repo-server load and speeds up the UI.

App-of-apps & ApplicationSets recap.

The two patterns App-of-apps: a parent Application whose source is a directory of child Application YAMLs. ApplicationSet: a controller-generated set of Applications from a generator + template.

The recommendation ApplicationSets are the 2026 default. App-of-apps remains useful for bootstrap.

Once you have more than a handful of Applications, managing them one-by-one becomes untenable. Two scaling patterns have emerged. App-of-apps is the older, more manual approach — a parent Application points at a directory containing many child Application YAMLs. ApplicationSets, covered extensively in Issue 12, are the controller-generated successor. Both have their place; this chapter pins where each fits.

App-of-apps

apiVersion: argoproj.io/v1alpha1

The pruning interaction

Two prune semantics interact and confuse: Application.syncPolicy.automated.prune (removes resources no longer in Git) and ApplicationSet.syncPolicy.preserveResourcesOnDeletion (controls what happens when an Application is removed from the set). The first is "remove what Git no longer declares for this app"; the second is "what to do when the app itself goes away." Set them deliberately; the defaults can surprise.

CD Chapter IX

Scaling Argo CD · sharding, repo-server, Redis.

The reference points AWS/CNOE EKS: 10K apps on 97 clusters with 10 shards = 8m30s sync time. OneUptime 2026 guide: 1000+ apps via 3 shards + 5 repo-servers + Redis HA.

The three bottlenecks Application controller (CPU/memory + reconcile concurrency), repo-server (Git clones + manifest rendering), Redis (cache hit rate, memory).

Argo CD scales further than most teams realize, but the levers must be pulled in the right order. The single most common failure mode is reaching the application-controller bottleneck and assuming "we need more clusters of Argo CD" when the answer is "we need to shard the controller." This chapter walks the levers in the order they should be applied.

Lever 1 — Vertical scaling of the controller

Before sharding, scale up. The application-controller is CPU + memory hungry. The 2026 OneUptime guidance: "Base: 500m per 100 apps cpu; Base: 1Gi per 100 apps memory." For 500 applications: request 2.5 CPU + 5 GiB memory; for 1000: 5 CPU + 10 GiB. Headroom matters; OOM on the controller restarts every Application's status tracking.

Lever 2 — Processor tuning

Two key concurrency knobs: --status-processors (concurrent diff/health checks) and --operation-processors (concurrent sync operations).

args:
  - argocd-application-controller
  - --status-processors=50                  # default 20, often raised to 50-100
  - --operation-processors=25               # default 10, often raised to 25-50
  - --repo-server-timeout-seconds=120      # raise for slow Helm repos

The OneUptime guidance: "With 500 applications and only 10 status processors, each reconciliation cycle takes much longer than necessary because most applications sit in the queue waiting. Status processors handle the comparison between live and desired state. This is the first thing to increase."

Lever 3 — Client QPS for K8s API calls

The controller makes many requests per second to each managed cluster's K8s API. Default 50 QPS / 100 burst — fine for small fleets, immediate bottleneck at scale.

env:
  - { name: ARGOCD_K8S_CLIENT_QPS, value: "100" }       # raise from 50
  - { name: ARGOCD_K8S_CLIENT_BURST, value: "200" }     # raise from 100

Watch target cluster's API server: per the Harness performance guide, "when changing these values make sure to monitor the k8s api server to make sure it can handle the increased load."

Lever 4 — Controller sharding

The biggest scale lever. Per the AWS scaling tests: "The Argo CD application controller was broken into 10 shards and showed significant improvements in sync times and reconciliation queue clear out. Sync times dropped from 43 mins to 8 mins 30 secs."

apiVersion: apps/v1
kind: StatefulSet
metadata: { name: argocd-application-controller, namespace: argocd }
spec:
  replicas: 3                                  # 3 shards for ~1000 apps
  template:
    spec:
      containers:
        - name: argocd-application-controller
          env:
            - name: ARGOCD_CONTROLLER_REPLICAS
              value: "3"                            # MUST match replicas

Pick the sharding algorithm carefully. Three options:

  • legacy. UID-based hash. Non-uniform; some shards get more clusters than others. The cause of documented production cases where "some of the Argo CD Application Controller replicas were managing more clusters in comparison to other replicas and a couple of replicas were managing no clusters at all."
  • round-robin. Even distribution by cluster count. Per the OpenShift GitOps guide: "The round-robin shard distribution algorithm is an experimental feature." Reshuffling occurs on cluster removal.
  • consistent-hashing. Recommended for stability. Per the docs: "consistent-hashing uses the consistent hashing with bounded loads algorithm which tends to equal distribution and also reduces cluster or application reshuffling in case of additions or removals of shards or clusters."
apiVersion: v1
kind: ConfigMap
metadata: { name: argocd-cmd-params-cm, namespace: argocd }
data:
  controller.sharding.algorithm: consistent-hashing
  controller.dynamic.cluster.distribution.enabled: "true"  # 2.8+ HPA

Lever 5 — Repo-server scaling

Repo-server bottleneck shows up as slow manifest rendering and high Application "Refreshing" duration. Scale horizontally:

apiVersion: apps/v1
kind: Deployment
metadata: { name: argocd-repo-server, namespace: argocd }
spec:
  replicas: 5                                     # scale for 1000+ apps
  template:
    spec:
      containers:
        - name: argocd-repo-server
          args:
            - argocd-repo-server
            - --parallelism-limit=10                  # concurrent manifest gens per replica
            - --git-shallow-clone                       # reduce clone size
            - --redis-compress=gzip                     # reduce Redis memory
          resources:
            requests: { cpu: "1", memory: 1Gi }
            limits:   { cpu: "2", memory: 3Gi }
          volumeMounts:
            - { name: tmp, mountPath: /tmp }
      volumes:
        - name: tmp
          emptyDir: { sizeLimit: 15Gi }                 # clone cache

Five replicas × parallelism-limit 10 = 50 concurrent manifest renders. Pod anti-affinity spreads them across nodes. The /tmp volume is critical — repo-server keeps cloned repos there; default emptyDir can fill up and trigger OOM.

Lever 6 — Git shallow clone

For large monorepos, full clones are expensive. --git-shallow-clone uses depth-1 clones; combined with annotation-based depth configuration:

apiVersion: v1
kind: Secret
metadata:
  name: monorepo-credentials
  namespace: argocd
  labels:
    argocd.argoproj.io/secret-type: repository
stringData:
  depth: "1"                          # shallow clone
  type: git
  url: https://github.com/myorg/monorepo.git

Per the OpenShift GitOps tuning blog, the difference for a typical repo: 1011ms with full clone vs ~150ms with depth=1. For repos with deep history (years of commits), the speedup compounds.

Lever 7 — Redis sizing and HA

Redis caches rendered manifests and application trees. Capacity planning: ~100 KB per Application × number of Applications + slack. For 1000 Applications: ~100 MB minimum; 1 GB is comfortable.

# Use the official HA install which provides Redis with Sentinel
$ kubectl apply -n argocd -f https://raw.githubusercontent.com/argoproj/argo-cd/stable/manifests/ha/install.yaml

# Set application tree shard size to reduce per-key load
env:
  - { name: ARGOCD_APPLICATION_TREE_SHARD_SIZE, value: "100" }

Per the official docs: "ARGOCD_APPLICATION_TREE_SHARD_SIZE — environment variable controlling the max number of resources stored in one Redis key. Splitting application tree into multiple keys helps to reduce the amount of traffic between the controller and Redis. The default value is 0, which means that the application tree is stored in a single Redis key. The reasonable value is 100."

The progression

The order for scaling Argo CD:

  1. Raise controller resources (vertical scale) — buys time, fast to apply.
  2. Tune status/operation processors — solves slow reconciliation queue.
  3. Raise K8s client QPS/burst — solves API throttling.
  4. Scale repo-server replicas — solves slow manifest rendering.
  5. Enable Redis HA + tune cache — solves cache pressure.
  6. Shard the controller — solves the fundamental CPU/memory bottleneck on the controller.
  7. Split into multiple Argo CD instances — when sharding within one instance isn't enough (beyond ~1000 clusters or 50K applications).

CD Chapter X

Sync waves, hooks, drift remediation.

The ordering primitives Sync waves order resources within a sync. Sync hooks insert steps before/during/after. Sync phases sequence these.

The drift question selfHeal = on or off? Each has a cost. The 2026 default is on with diff customization.

A single sync of an Application is not always one apply operation. Argo CD has rich primitives for ordering resources, running pre/post hooks, and deciding what to do when live state drifts from Git. This chapter walks the primitives and the operational consequences of each.

Sync waves — within-sync ordering

Resources within a single sync don't apply in arbitrary order. Annotations control the wave:

# Wave -10 — runs first
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  annotations:
    argocd.argoproj.io/sync-wave: "-10"
---
# Wave 0 — default — runs after wave -10
apiVersion: v1
kind: Secret
metadata:
  name: app-secrets
---
# Wave 10 — runs last
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app
  annotations:
    argocd.argoproj.io/sync-wave: "10"

Argo CD applies resources in wave order; within a wave, it waits for resources to be Healthy before advancing to the next wave. The pattern enables: CRDs deployed before CRs that use them, secrets and configmaps before the deployments that mount them, schema migration job before the app that depends on the schema.

Sync hooks — pre/post operations

Hooks are resources whose lifecycle is tied to the sync phase rather than the workload itself. Annotation:

# PreSync — runs before main resources apply
apiVersion: batch/v1
kind: Job
metadata:
  name: db-migration
  annotations:
    argocd.argoproj.io/hook: PreSync
    argocd.argoproj.io/hook-delete-policy: BeforeHookCreation,HookSucceeded
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: migrate
          image: registry.example.com/migrator:v2
          command: [flyway, migrate]
---
# PostSync — runs after main resources are Healthy
apiVersion: batch/v1
kind: Job
metadata:
  name: smoke-test
  annotations:
    argocd.argoproj.io/hook: PostSync
    argocd.argoproj.io/hook-delete-policy: HookSucceeded
spec:
  # ... runs after deployment is ready ...

Four hook types:

  • PreSync. Runs before the regular sync. Failure aborts the sync. Used for migrations, validations, preconditions.
  • Sync. Runs as part of the regular sync wave (default for any resource without a hook annotation).
  • PostSync. Runs after all resources are Healthy. Used for smoke tests, post-deployment notifications, cache warming.
  • SyncFail. Runs when a sync fails. Used for rollback triggers, error notifications.

Hook deletion policies

Hook resources accumulate in the cluster if not cleaned up. hook-delete-policy controls when:

  • HookSucceeded — delete after success.
  • HookFailed — delete after failure (often retained for debug).
  • BeforeHookCreation — delete any previous instance before creating new (safe re-runs).

The 2026 default for migration-style PreSync Jobs: BeforeHookCreation,HookSucceeded. Job is uniquely identified by Argo CD; re-runs delete previous instance and the new one cleans up on success.

The drift problem

Live state in the cluster can drift from Git. Causes: manual kubectl edit by an operator, mutating webhook (e.g., Istio sidecar injection adds fields), operator that owns part of a resource Argo CD also manages, infrastructure changes (cluster autoscaler adds annotations to nodes Argo CD doesn't expect).

Three drift responses

  • Ignore. syncPolicy.automated not enabled or selfHeal: false. Argo CD reports OutOfSync but doesn't act. Operator decides.
  • Auto-sync. syncPolicy.automated.selfHeal: false, prune: true. Re-applies Git state on next reconcile; doesn't override mutations done outside Argo CD.
  • Self-heal. syncPolicy.automated.selfHeal: true. Continuously reverts any drift. The strongest GitOps posture; can fight with operators and webhooks.

Diff customization

When self-heal fights legitimate cluster modifications, the answer is ignoreDifferences — fields Argo CD should not consider when computing drift:

spec:
  ignoreDifferences:
    - group: apps
      kind: Deployment
      jsonPointers:
        - /spec/template/spec/containers/1   # sidecar slot
        - /spec/template/metadata/annotations/sidecar.istio.io~1status
    - group: ""
      kind: Service
      jsonPointers:
        - /spec/clusterIP                    # assigned by cluster, not in Git
    - group: ""
      kind: ServiceAccount
      jsonPointers:
        - /secrets                            # auto-generated

The trap: too-loose ignoreDifferences creates blind spots. If you ignore /spec/replicas to allow HPA, you'll never notice if someone edits replicas manually. Be precise; ignore only what's legitimately managed by another controller.

Self-heal backoff

Constant self-heal attempts on a stubborn application waste resources. The backoff config:

apiVersion: v1
kind: ConfigMap
metadata: { name: argocd-cmd-params-cm, namespace: argocd }
data:
  controller.self.heal.timeout.seconds: "5"           # 5s before retry
  controller.self.heal.backoff.factor: "3"            # exponential factor

    hs = {}
    if obj.status ~= nil then

The self-heal default question

For a production fleet, the right default is selfHeal: true with carefully-tuned ignoreDifferences. The reason: drift is otherwise invisible. With selfHeal off, an engineer's manual kubectl edit silently stays in place until a real sync overrides it; you've lost reproducibility. With selfHeal on, every legitimate operator-managed field needs explicit ignoreDifferences — more configuration upfront, but the system is honest about who owns what. The default off is rarely the right call for production; it just feels safer until you accumulate enough silent drift.

for_each vs count

One of the most consequential HCL choices. count creates resources indexed by integer position; for_each indexes by string keys. The latter is far more robust to changes:

 only ["b"] is destroyed; ["a"], ["c"] untouched
  -> safe

The 2026 rule: use for_each wherever the collection is non-trivial. count = 0/1 for boolean inclusion is still fine; count = N for list-like resources is a footgun.

The S3 + DynamoDB → S3 native lockfile transition

The traditional Terraform S3 backend required a DynamoDB table for state locking — preventing two concurrent terraform apply from corrupting state. As of 2024, S3 itself supports conditional writes, enabling native locking without DynamoDB:

terraform {
  backend "s3" {
    bucket       = "acme-tf-state"
    key          = "prod/vpc/terraform.tfstate"
    region       = "us-east-1"
    encrypt      = true
    use_lockfile = true                          # S3 conditional-write lock
    kms_key_id   = "alias/tf-state-key"          # SSE-KMS
  }
}

One less moving part. The DynamoDB lock table approach still works and is fine to keep on existing setups; new setups should use use_lockfile.

IAC Chapter XIV

Module discipline · versioning, registry, composition.

The shape Modules are reusable HCL packages. Versioned via Git tags or registry. Composed in root modules.

The rule Pin module versions. Don't reference main branch from production. Treat modules like internal libraries.

A module is a directory of HCL files that can be called from other configurations. Modules are how IaC reaches scale — without them, a 100-environment fleet means 100 copies of the same VPC config. With them, one module is consumed by all 100. The discipline of writing, versioning, and consuming modules is the difference between an IaC codebase that grows manageably and one that collapses under copy-paste duplication.

Module anatomy

# 1. Git repository — tagged version (recommended for prod)
module "vpc" {
  source = "git::https://github.com/myorg/terraform-aws-vpc.git?ref=v1.4.2"
}

# 2. Terraform Registry — public
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "~> 5.13"
}

# 3. Private registry (HCP Terraform, Spacelift, etc.)
module "vpc" {
  source  = "app.terraform.io/myorg/vpc/aws"
  version = "~> 1.4"
}

# 4. Local path — for development only
module "vpc" {
  source = "../modules/vpc"
}

# 5. S3 / GCS — module as artifact
module "vpc" {
  source = "s3::https://s3.amazonaws.com/modules/vpc-1.4.2.zip"
}

The version pinning rule

The single most consequential rule: pin every module reference to a specific version. Two reasons:

  • Reproducibility. Without a pinned version, two engineers running the same code at different times get different infrastructure. CI runs differ from local. Production differs from staging. Debugging becomes impossible.
  • Safety. A module update can introduce breaking changes. With version pinning, you adopt updates intentionally; without, an upstream change can destroy resources on the next apply.

Three pin styles for Git modules:

  • ?ref=v1.4.2 — exact tag. Most strict. The 2026 default for production references.
  • ?ref=abc123def — exact commit SHA. Even stricter; survives tag deletion. Used when tags are mutable in your environment.
  • ?ref=main — branch reference. Never use in production. The infrastructure can change without any code change in the consumer.

Module interface design

Three properties of a good module interface:

  • Small surface. A module with 50 input variables is doing too much; split it. Aim for <15 inputs in most cases.
  • Sensible defaults. Most inputs should have defaults that work for common cases. Required inputs are reserved for the truly necessary (e.g., a name, an environment label).
  • Explicit outputs. Outputs are the only public surface. Don't expose internal resource references unless consumers genuinely need them; that creates implicit API contracts.
variable "environment" {
  type        = string
  description = "Environment name (dev | staging | prod)"
  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "environment must be dev, staging, or prod"
  }
}

variable "cidr_block" {
  type        = string
  description = "CIDR block for the VPC"
  default     = "10.0.0.0/16"
}

variable "availability_zones" {
  type        = list(string)
  description = "AZs to use for subnets (typically 3 for prod)"
  default     = []
  validation {
    condition     = length(var.availability_zones) == 0 || length(var.availability_zones) >= 2
    error_message = "at least 2 AZs required if specified"
  }
}

variable "tags" {
  type        = map(string)
  description = "Tags applied to all resources"
  default     = {}
}

Module versioning discipline

Modules are libraries; treat them like libraries. Semantic versioning:

  • Major (v2.0.0). Breaking changes. Removed outputs, removed inputs, resource recreate. Consumer must update code.
  • Minor (v1.5.0). Additive changes. New optional inputs with defaults; new outputs; new resources that don't affect existing.
  • Patch (v1.4.3). Bug fixes. No interface changes; no plan diff for consumers using defaults.

The CI for a module repo should at minimum: terraform fmt, terraform validate, tflint, tfsec or checkov, and a terraform plan against a test fixture. A module without CI is a module that will break consumers.

The mono-module trap

The temptation: write one giant module that creates "an environment" — VPC + clusters + databases + everything. The consequence: changes anywhere in the module trigger full re-plans across all consumers; module versioning becomes synchronized across unrelated changes; the module's interface grows to dozens of inputs to handle every variation.

The 2026 rule: one module per concept. VPC module. EKS module. RDS module. ElastiCache module. Root configurations compose them. Per the 2026 IaC patterns guide: "When you start with IaC it is common to have 'one big thing': a single state file or Terragrunt stack creating everything: VPCs, databases, queues, ECS services, Lambda functions and more. This works for early prototypes, but becomes a liability as the system grows. A more scalable approach is to slice the infrastructure into components and give each component its own state and execution pipeline."

The three approaches Terragrunt (DRY wrapper), Atlantis (PR-based self-hosted), Spacelift / env0 / Scalr (SaaS managed platforms).

Once you have more than a handful of Terraform/OpenTofu stacks, manual operation is untenable. The ecosystem in 2026 has converged on three approaches: Terragrunt to keep configurations DRY, Atlantis for self-hosted PR-based execution, and commercial platforms (Spacelift, env0, Scalr) for managed orchestration. Each fits a different scale and operational posture. This chapter walks them and the choices they encode.

Terragrunt — DRY and orchestration

Terragrunt is a thin wrapper around Terraform/OpenTofu that solves two problems: configuration duplication (the same backend, provider, and version blocks repeated across every module) and dependency orchestration (running modules in the right order). Per the Spacelift 2026 guide: "When you write Terraform configurations at scale for large cloud infrastructures, Terraform code including provider definitions and state backend configurations will repeat throughout all your configurations. Keeping things like provider versions up to date across all Terraform configurations can be tedious. A whole ecosystem of tools has grown around Terraform to help you run it at scale."

include "root" {
  path = find_in_parent_folders()
}

include "envcommon" {
  path           = "${dirname(find_in_parent_folders())}/_envcommon/vpc.hcl"
  expose         = true
  merge_strategy = "deep"
}

terraform {
  source = "git::ssh://git@github.com/myorg/terraform-aws-vpc.git?ref=v3.2.1"
}

dependency "identity" {
  config_path = "../identity"
}

inputs = {
  cidr_block         = "10.10.0.0/16"
  availability_zones = ["us-east-1a", "us-east-1b", "us-east-1c"]
  kms_key_arn        = dependency.identity.outputs.tf_state_kms_key_arn
}

Terragrunt's two killer features

  • run-all. Runs Terragrunt operations across a directory tree in dependency order. terragrunt run-all apply at the prod level orchestrates dozens of dependent module applies in the right sequence.
  • Backend generation. Generates backend.tf automatically per leaf based on parent config — the leaf only specifies its key suffix; bucket/region/encryption inherit.

Atlantis — PR-driven self-hosted execution

Atlantis is the lightweight self-hosted option. It watches a GitHub/GitLab/Bitbucket repo for PRs touching .tf or .hcl files, runs terraform plan automatically, posts the plan output as a PR comment, and lets reviewers approve with a comment. Per the 2026 Atlantis guide: "Atlantis is an automation tool that enhances the PR workflows for Terraform projects. It is used to reduce the complexity of collaboration and governance for your Terraform workflow. The tool is open-source and does not offer any SaaS, so you will have to install and configure it yourself."

Atlantis runs as a Go binary or container; teams self-host on a VM or in Kubernetes. The 2026 perspective per the comprehensive guide: "Terragrunt + Atlantis works, but isn't free. You get DRY config and dependency awareness. You also need extra pre_workflow_hooks and terragrunt run-all glue."

Atlantis limitations

Atlantis is the right choice when "self-hosted Terraform PR automation" is the entire requirement. It is the wrong choice when the requirements expand. Per the 2026 alternatives analysis: "Workflow complexity: Atlantis relies heavily on pull requests for its operations, which can be non-intuitive and cumbersome, especially when setting up multiple workspaces linked to a monorepo. Limited Native Integrations Beyond VCS: Compared to commercial platforms (Terraform Cloud, Scalr, env0, Spacelift), Atlantis has fewer built-in integrations for advanced policy checking, cost estimation, or security scanning. These can be added with custom workflows but require more setup. Basic UI/UX: Atlantis primarily operates via PR comments and has a basic web UI for viewing logs and locks. It lacks the dashboards and reporting features of commercial alternatives."

The platform pattern

Commercial platforms share a structural pattern. Per the 2026 patterns guide: "Spacelift implements this pattern out of the box. Each stack is wired to a repo, branch and directory. Pull requests create proposed runs that show plans and policy decisions inside your Git provider. Merges create tracked runs that apply changes. You can also add approval gates, policies and notifications to build the exact deployment flow you want, without breaking the separation between development and deployment."

The shared shape:

  • Stack — a (repo, branch, directory) tuple. Each stack has its own state, runs, policies.
  • Policy engine — OPA (Spacelift, env0, Scalr) or Sentinel (HCP Terraform). Policies gate plans before apply.
  • Stack dependencies — explicit graph of stacks; one stack's outputs feed another's inputs. Replaces terraform_remote_state.
  • Run history — every plan and apply recorded; rollback by re-running a previous state.
  • RBAC — SAML/OIDC integration; team-scoped access per stack.
  • Drift detection — built-in periodic plan; alerts on non-empty diff.
  • Cost estimation — Infracost integration; plan output includes estimated $ delta.

Mono-repo vs poly-repo for IaC

Per the 2026 patterns guide: "At some point, you will have to decide how many Git repositories your IaC should live in. The classic tension is between a single mono-repo and a multi-repo (polyrepo) approach. A mono-repo works well when you are still discovering patterns, have a small platform team, or want a single source of truth. It makes it easy to change a module and update all its usages in one go. It also simplifies onboarding: new team members clone one repo and see everything."

For IaC specifically, the common 2026 shape is two repos:

  • terraform-modules — reusable modules, versioned via tags.
  • terraform-live — environment configurations (Terragrunt or root modules) that consume modules. This is where stacks live.

The separation matches the library vs application distinction. Modules are libraries; environments are applications using them.

The scaling progression

  1. Small team, <10 stacks. Plain Terraform/OpenTofu + GitHub Actions. No orchestrator needed.
  2. Growing, 10-50 stacks. Add Terragrunt for DRY config. Still GitHub Actions for execution.
  3. Scaling, 50+ stacks, multiple teams. Add Atlantis for PR-based plan/apply with locking, or pick a commercial platform.
  4. Multi-team, compliance pressure. Commercial platform (Spacelift/env0/Scalr) for policy, RBAC, drift, cost.
  5. Enterprise, regulated. Commercial platform + private VCS + self-hosted runners + custom policies.
The right time to introduce orchestration

Most teams add orchestration too late, then under-utilize the tool they pick. The cleanest decision point: when you have a second team that needs to apply to a different stack, and you're solving "how do they get permission to apply without giving them cloud creds" by hand. That's the moment to introduce Atlantis or a commercial platform — the alternative is provisioning cloud credentials to a second engineer's laptop, which is a problem that compounds quickly.

IAC Chapter XVI

CI integration & policy-as-code.

The pipeline fmt → validate → lint → plan → policy gate → manual approval → apply. Each step a quality gate; failure stops the pipeline.

The policy layer OPA/Conftest (open), Sentinel (HashiCorp), Checkov/tfsec (security-focused), Infracost (cost). Different tools, different concerns.

A Terraform/OpenTofu apply triggered from a developer's laptop is a security incident waiting to happen. Production IaC runs from CI, with a sequence of automated gates that fail loudly on bad changes. This chapter walks the gates — what each one checks, when to add it, how to integrate it cleanly.

The canonical CI pipeline

name: terraform-pr
on:
  pull_request:
    paths: ["live/**", "modules/**"]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: opentofu/setup-opentofu@v1
        with: { tofu_version: "1.8.x" }
      - run: tofu fmt -check -recursive          # 1. formatting
      - run: tofu init -backend=false
      - run: tofu validate                       # 2. syntax
      - uses: terraform-linters/setup-tflint@v4
      - run: tflint --recursive                  # 3. lint

  security:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: aquasecurity/tfsec-action@v1.0.3
        with: { soft_fail: false }              # 4. security scan
      - uses: bridgecrewio/checkov-action@master
        with: { framework: terraform }           # 5. compliance scan

  plan:
    runs-on: ubuntu-latest
    needs: [validate, security]
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: "${{ secrets.PLAN_ROLE_ARN }}"     # OIDC, read-only role
          aws-region: us-east-1
      - run: tofu init
      - run: tofu plan -out=tfplan -lock=false  # 6. plan
      - uses: infracost/actions/setup@v3
      - run: infracost diff --path=tfplan        # 7. cost diff
      - run: conftest test tfplan --policy policies/  # 8. OPA policy
      - uses: actions/upload-artifact@v4
        with: { name: tfplan, path: tfplan }

The principle: each step has one concern. fmt catches whitespace; validate catches syntax; tflint catches anti-patterns; tfsec catches insecure resource configs; checkov catches compliance gaps; plan computes the diff; infracost estimates the bill change; OPA enforces organizational rules.

The OIDC credential pattern

Notice configure-aws-credentials uses role-to-assume via OIDC — no long-lived AWS access keys in GitHub secrets. The CI job is granted a short-lived token by AWS based on the GitHub workflow's signed identity. Two roles typically:

  • Plan role — read-only across the AWS account. Used in PR pipelines. Can't change anything; safe to grant to any merged PR.
  • Apply role — broader permissions. Used only by main-branch workflows after merge + approval. Scoped to specific stack via IAM conditions on tags or paths.
Statement:
  - Effect: Allow
    Principal:
      Federated: "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
    Action: "sts:AssumeRoleWithWebIdentity"
    Condition:
      StringEquals:
        "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
      StringLike:
        "token.actions.githubusercontent.com:sub":
          - "repo:myorg/terraform-live:ref:refs/heads/main"     # only main branch
          - "repo:myorg/terraform-live:environment:production"   # only via env gate

The apply pipeline

name: terraform-apply
on:
  push:
    branches: [main]
    paths: ["live/**"]

jobs:
  apply:
    runs-on: ubuntu-latest
    environment: production                  # GitHub environment with required reviewers
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: "${{ secrets.APPLY_ROLE_ARN }}"
          aws-region: us-east-1

      - run: tofu plan -out=tfplan -lock-timeout=10m

The environment: production + GitHub environment configured with required reviewers means apply waits for a human approval before proceeding — even though everything else is automated. The OIDC credentials are scoped to this environment; the role can't be assumed from PR workflows.

State drift detection in CI

on:
  schedule:
    - cron: "0 6 * * *"          # 6am UTC daily

jobs:
  drift-check:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        stack: [network, identity, eks, rds]
    steps:
      - uses: actions/checkout@v4
      - uses: aws-actions/configure-aws-credentials@v4
        with: { role-to-assume: "${{ secrets.PLAN_ROLE_ARN }}" }
      - run: cd live/prod/${{ matrix.stack }} && tofu plan -detailed-exitcode
        id: plan
        continue-on-error: true
      - if: steps.plan.outputs.exitcode == 2     # 2 = drift detected
        uses: 8398a7/action-slack@v3
        with:
          status: custom
          custom_payload: |
            { text: "Drift in prod/${{ matrix.stack }} — review immediately" }

The pattern: every stack gets a daily plan; any non-empty diff alerts the platform team. The -detailed-exitcode flag returns 0 for no changes, 1 for error, 2 for drift — perfect for CI gates.
