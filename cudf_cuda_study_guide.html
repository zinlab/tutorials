Part I

Modern C++ as the substrate

cuDF, Velox, and the broader RAPIDS surface are written in C++17/20. Performance isn't only about kernels — the host-side code orchestrates allocation, dispatches on type, manages lifetimes across streams, and is where most of the binary size and compile time lives. This section covers the C++ idioms that show up most.

§01Modern C++ foundations for performance libraries

The library is full of long-lived value types that own GPU memory, plus thin views that don't. Getting move semantics, RAII, and the rule of five right is the floor; everything above is built on it.

RAII for device memory

A canonical pattern: an owning class that holds a device pointer and frees it on destruction. cuDF's rmm::device_uvector is the production version. Here's the skeleton:

// A minimal owning device buffer (production: use rmm::device_uvector)
template<typename T>
class device_buffer {
  T*          d_ptr_   = nullptr;
  std::size_t size_    = 0;
  cudaStream_t stream_ = 0;

public:
  device_buffer() = default;

  device_buffer(std::size_t n, cudaStream_t s)
    : size_(n), stream_(s) {
    cudaMallocAsync(&d_ptr_, n * sizeof(T), s);
  }

  // Rule of five: explicit move, deleted copy
  device_buffer(device_buffer const&)            = delete;
  device_buffer& operator=(device_buffer const&) = delete;

  device_buffer(device_buffer&& o) noexcept
    : d_ptr_(o.d_ptr_), size_(o.size_), stream_(o.stream_) {
    o.d_ptr_ = nullptr; o.size_ = 0;
  }

  device_buffer& operator=(device_buffer&& o) noexcept {
    if (this != &o) {
      release();
      d_ptr_ = o.d_ptr_; size_ = o.size_; stream_ = o.stream_;
      o.d_ptr_ = nullptr; o.size_ = 0;
    }
    return *this;
  }

  ~device_buffer() { release(); }

  T*          data()       noexcept { return d_ptr_; }
  T const*    data() const noexcept { return d_ptr_; }
  std::size_t size() const noexcept { return size_; }

private:
  void release() noexcept {
    if (d_ptr_) cudaFreeAsync(d_ptr_, stream_);
    d_ptr_ = nullptr;
  }
};
Subtleties that cost you in review
  • noexcept on the move constructor is not cosmetic — without it, std::vector<device_buffer> falls back to copy on growth, which deletes.
  • Stream-ordered allocation (cudaMallocAsync) must be paired with stream-ordered free; freeing on a different stream than allocation requires a stream-ordered dependency or you have UB.
  • Don't put cudaFree in destructors that may run at static-destruction time — the CUDA driver may already be shut down. Production code structures lifetimes to avoid this.

Owning vs viewing — the column_view pattern

cuDF distinguishes column (owns memory) from column_view (does not). The view is cheap to copy, contains pointers + metadata, and is what kernels receive. This split is the single most important API design decision in the library — it lets algorithms compose without copying.

// Conceptual sketch — actual cudf::column_view is more elaborate
class column_view {
  void const*    data_;        // type-erased
  bitmask_type const* null_mask_;
  size_type        size_, offset_, null_count_;
  data_type        type_;
  std::vector<column_view> children_;  // for nested types
public:
  template<typename T>
  T const* data() const { return static_cast<T const*>(data_) + offset_; }
  size_type size() const { return size_; }
  bool has_nulls() const { return null_count_ > 0; }
  // ... null bitmap accessor, child accessors, ...
};

§02Templates, concepts & type dispatch

A dataframe library serves dozens of types (int8…int64, float32/64, bool, fixed-point, strings, lists, structs, decimals). Every algorithm must dispatch at runtime to a template instantiation chosen by the column's type_id. This is exactly what cuDF's type_dispatcher does.

// Skeleton of cudf::type_dispatcher
enum class type_id { INT8, INT16, INT32, INT64, FLOAT32, FLOAT64, BOOL8, STRING, ... };

template<type_id Id> struct id_to_type;
template<> struct id_to_type<type_id::INT32>   { using type = int32_t; };
template<> struct id_to_type<type_id::FLOAT64> { using type = double; };
// ... one per type ...

template<typename Functor, typename... Args>
decltype(auto) type_dispatcher(data_type dt, Functor&& f, Args&&... args) {
  switch (dt.id()) {
    case type_id::INT32:
      return f.template operator()<int32_t>(std::forward<Args>(args)...);
    case type_id::FLOAT64:
      return f.template operator()<double>(std::forward<Args>(args)...);
    // ... etc ...
    default: throw std::logic_error("unsupported type");
  }
}

// Usage: a generic "sum a column" functor
struct sum_functor {
  template<typename T>
  auto operator()(column_view const& col) const {
    if constexpr (std::is_arithmetic_v<T>) {
      return launch_sum_kernel<T>(col.data<T>(), col.size());
    } else {
      throw std::logic_error("sum requires arithmetic type");
    }
  }
};

auto result = type_dispatcher(col.type(), sum_functor{}, col);
The if constexpr branch lets one functor handle the "valid only for arithmetic types" case without writing a separate dispatcher for each subset.

C++20 concepts — when SFINAE finally got readable

// Before C++20: SFINAE soup

         typename = std::enable_if_t<std::is_arithmetic_v<T> &&
                                   !std::is_same_v<T, bool>>>

Compile time is a real cost

Dispatching on every type at every call site explodes instantiations. cuDF compiles for ~30 minutes on a beefy machine. Strategies that matter: extern template declarations to push instantiation to a single TU, explicit instantiation in .cu files, splitting the dispatcher into "fixed-width only" / "strings only" / "all" variants to avoid pulling in the full type matrix when an algorithm doesn't need it.

§03Compile-time computation

constexpr and consteval let you push table lookups, validation, and small computations to compile time. For data-processing libraries, the most common use is type-property tables.

constexpr size_t size_of(type_id id) {
  switch (id) {
    case type_id::INT8:    return 1;
    case type_id::INT16:   return 2;
    case type_id::INT32:   return 4;
    case type_id::INT64:   return 8;
    case type_id::FLOAT32: return 4;
    case type_id::FLOAT64: return 8;
    default: return 0;  // variable-width
  }
}

// Used in compile-time validation contexts
template<type_id Id>
struct fixed_width_only {
  static_assert(size_of(Id) > 0, "variable-width types not supported here");
};

C++20 also brings consteval (must run at compile time) and constinit (must be constant-initialized but not necessarily constexpr at use). Both occasionally useful for kernel-launch parameter validation.

Part II

The CUDA programming model, from threads to streams

CUDA's mental model is a four-level hierarchy of threads, with a memory hierarchy alongside it. Internalize both, and most performance work is straightforward bookkeeping. Misunderstand them, and every optimization is a guess.

§04The CUDA execution hierarchy

┌─────────────────────────────────────────────────────────────────┐ │ GRID (one kernel launch) │ │ ┌───────────────────────────────┐ ┌──────────────────────────┐ │ │ │ BLOCK (≤ 1024 threads) │ │ BLOCK │ │ │ │ ┌─────────────┐ ┌────────────┐│ │ ... │ │ │ │ │ WARP (32 t) │ │ WARP ││ │ │ │ │ │ │ t0…t31 │ │ t32…t63 ││ │ │ │ │ │ └─────────────┘ └────────────┘│ │ │ │ │ └───────────────────────────────┘ └──────────────────────────┘ │ │ Blocks land on SMs; warps are the unit of scheduling. │ └─────────────────────────────────────────────────────────────────┘

The non-negotiable facts:

  • A warp is 32 threads that execute in lockstep on a single SM. The warp is the unit of scheduling, divergence, and most synchronization. Every CUDA optimization is downstream of this fact.
  • A block is a group of threads (up to 1024) that share fast on-chip shared memory and can synchronize with __syncthreads(). One block executes entirely on one SM.
  • A grid is a collection of blocks. There is no built-in synchronization across blocks within a kernel (except via Cooperative Groups grid sync, which requires special launch).
  • On Hopper, a new level — thread block clusters — lets up to 16 blocks on the same GPC share Distributed Shared Memory.

Kernel launch — what the syntax means

// A vector add — the "hello world" of CUDA
__global__ void vadd(float const* a, float const* b, float* c, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < n) c[i] = a[i] + b[i];
}

// Launch — grid of ceil(n/256) blocks, each of 256 threads
int  block_size = 256;
int  grid_size  = (n + block_size - 1) / block_size;
vadd<<<grid_size, block_size, 0 /* shared mem bytes */, stream>>>(a, b, c, n);
The classic mistakes
  • Forgetting the bounds check (if (i < n)). The last block has threads with indices ≥ n, and they will write past the array.
  • Block size that's not a multiple of 32. Some threads in the last warp will be inactive, costing throughput.
  • Hardcoded block size like 1024 — uses too many registers on complex kernels and tanks occupancy. Default to 128 or 256; tune later.
  • Forgetting that cudaGetLastError() after a launch reports launch-time errors, not runtime errors from inside the kernel.

Shared memory — the workhorse

Shared memory is the staging area where blocks cooperate. Every reduction, scan, and tile of every BLAS-like kernel uses it.

// Block-level reduction using shared memory
__global__ void block_reduce_sum(float const* in, float* partial, int n) {
  __shared__ float sdata[256];
  int tid = threadIdx.x;
  int i   = blockIdx.x * blockDim.x + tid;

  sdata[tid] = (i < n) ? in[i] : 0.0f;
  __syncthreads();

  // Tree reduction in shared memory
  for (int s = blockDim.x / 2; s > 0; s >>= 1) {
    if (tid < s) sdata[tid] += sdata[tid + s];
    __syncthreads();
  }

  if (tid == 0) partial[blockIdx.x] = sdata[0];
}
This is not the fastest reduction

The shared-memory tree above is correct but suboptimal. Modern reductions use warp-level __shfl_down_sync (§12) to do the last 5 stages entirely in registers, then a single shared write. The CUB version (cub::BlockReduce) is what you actually use.

Warp-level primitives — the secret sauce

// Sum 32 values across a warp using register-only shuffles
__device__ float warp_reduce_sum(float v) {
  for (int off = 16; off > 0; off >>= 1) {
    v += __shfl_down_sync(0xffffffff, v, off);
  }
  return v;  // thread 0 of the warp holds the result
}

// Variant: every thread gets the broadcast result
__device__ float warp_reduce_sum_all(float v) {
  for (int off = 16; off > 0; off >>= 1) {
    v += __shfl_xor_sync(0xffffffff, v, off);
  }
  return v;
}

// Predicate-counting across a warp — used everywhere in stream compaction
__device__ int warp_count(bool p) {
  return __popc(__ballot_sync(0xffffffff, p));
}
Why the mask matters

The first argument to every _sync intrinsic is a 32-bit mask of which lanes participate. 0xffffffff means "all 32 lanes." Pass the wrong mask after a divergent branch and you get undefined behavior. Use __activemask() when you don't know statically which lanes are active.

Cooperative groups — the modern abstraction

#include <cooperative_groups.h>
#include <cooperative_groups/reduce.h>
namespace cg = cooperative_groups;

__global__ void reduce_kernel(float* in, float* out, int n) {
  auto block = cg::this_thread_block();
  auto warp  = cg::tiled_partition<32>(block);

  float v = (block.thread_rank() < n) ? in[block.thread_rank()] : 0;
  v = cg::reduce(warp, v, cg::plus<float>{});

  if (warp.thread_rank() == 0) out[warp.meta_group_rank()] = v;
}

§07Streams, events, and CUDA graphs

A stream is a queue of GPU work that executes in order. Operations submitted to different streams can run concurrently, subject to resource availability. Streams are the basic unit of asynchrony.

// Two streams for overlap of compute and copy
cudaStream_t compute_s, copy_s;
cudaStreamCreate(&compute_s);
cudaStreamCreate(&copy_s);

// H2D copy on copy_s
cudaMemcpyAsync(d_in, h_in, n*sizeof(float), cudaMemcpyHostToDevice, copy_s);

// Have compute_s wait on the copy via an event
cudaEvent_t copied;
cudaEventCreate(&copied);
cudaEventRecord(copied, copy_s);
cudaStreamWaitEvent(compute_s, copied, 0);

vadd<<<grid, block, 0, compute_s>>>(d_in, d_in2, d_out, n);

// D2H copy can run on copy_s, dependent on compute_s
cudaEvent_t computed;
cudaEventRecord(computed, compute_s);
cudaStreamWaitEvent(copy_s, computed, 0);
cudaMemcpyAsync(h_out, d_out, n*sizeof(float), cudaMemcpyDeviceToHost, copy_s);
Stream-zero is a global serialization point

Operations submitted to the legacy default stream (stream 0) implicitly synchronize with all other streams on the device. This is rarely what you want in a library. Use cudaStreamPerThread (compile with --default-stream per-thread) so the default stream is thread-local, or always pass an explicit stream. cuDF passes streams explicitly through every API.

CUDA graphs — amortizing launch overhead

Kernel launches cost ~5-10 µs each. A workflow with 100 small kernels burns half a millisecond just in launch overhead. CUDA graphs capture a sequence of operations once and replay them as a single submission.

cudaGraph_t graph;
cudaGraphExec_t exec;

cudaStreamBeginCapture(stream, cudaStreamCaptureModeGlobal);
kernel_a<<<g, b, 0, stream>>>(...);
kernel_b<<<g, b, 0, stream>>>(...);
kernel_c<<<g, b, 0, stream>>>(...);
cudaStreamEndCapture(stream, &graph);
cudaGraphInstantiate(&exec, graph, nullptr, nullptr, 0);

// Replay — single submission, full pipeline
for (int i = 0; i < 1000; ++i) {
  cudaGraphLaunch(exec, stream);
}

Graphs matter most for iterative workloads where the same DAG runs repeatedly (training step, inference batch, streaming pipeline). For one-shot data-processing operations, the capture overhead may exceed the savings.

Part III

Performance — what actually makes kernels fast

Most kernel-level performance work is one of four levers: get memory access patterns right, keep enough warps in flight, eliminate divergence, and hide latency with asynchrony. Everything else is a special case of these.

§08Coalescing, vectorized loads, bank conflicts

Coalescing

When a warp issues a load, the GPU coalesces consecutive thread accesses into the minimum number of memory transactions. The rule: thread i accesses element i and you get one 128-byte transaction. Stride-2 access (thread i → element 2*i) needs two transactions. Random access can need up to 32.

// Coalesced — perfect
__global__ void good(float* x) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  x[i] = x[i] * 2.0f;          // each thread reads/writes its own slot
}

// Strided — half the bandwidth
__global__ void bad(float* x) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  x[i * 2] = x[i * 2] * 2.0f;  // stride 2 — 50% bandwidth
}

Vectorized loads

Loading 4 floats with one instruction (float4) is faster than 4 separate float loads — fewer instructions, fewer transactions, more bytes per transaction.

// Each thread loads/stores 4 floats in one instruction
__global__ void scale4(float* x, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i * 4 + 3 < n) {
    float4 v = *reinterpret_cast<float4*>(&x[i * 4]);
    v.x *= 2.0f; v.y *= 2.0f; v.z *= 2.0f; v.w *= 2.0f;
    *reinterpret_cast<float4*>(&x[i * 4]) = v;
  }
}
Alignment is a prerequisite

float4 loads require 16-byte aligned addresses. cudaMalloc guarantees 256-byte alignment, but offsets into the buffer don't automatically preserve it. Use cuda::std::assume_aligned (or the compiler hint) when you know the pointer is aligned, and fall back to scalar code for the tail.

Shared memory bank conflicts

Shared memory is split into 32 banks. Consecutive 4-byte words live in consecutive banks. If two threads in a warp access the same bank (different addresses), the accesses serialize. Same address is broadcast and fine.

// Classic conflict: column access in a 32-wide tile
__shared__ float tile[32][32];
float v = tile[threadIdx.y][threadIdx.x];   // row access — fine, broadcasts
float u = tile[threadIdx.x][threadIdx.y];   // column access — 32-way conflict, 32× slower

// Fix: pad the inner dimension by 1
__shared__ float tile[32][33];                 // note the 33
// Now column j of tile lives in bank (j*33) %% 32 = (j*1) %% 32, all different

§09Occupancy and the register budget

Occupancy is the fraction of an SM's maximum warps that are actively scheduled. Higher occupancy hides latency: when one warp stalls on a memory load, another runs. But high occupancy isn't always optimal — registers are precious and a kernel that uses many registers per thread can deliver more work per warp.

The constraints per SM:

  • Registers: 65,536 32-bit registers. 64-register kernel → 1024 threads max → 32 warps max. 128-register kernel → 512 threads max → 16 warps.
  • Shared memory: 48-228 KB depending on architecture (configurable on Volta+). A kernel using 32 KB/block → 1-7 blocks/SM.
  • Threads per block: ≤ 1024.
  • Blocks per SM: ≤ 32 (Volta+) or 16.
// Tell the compiler to limit registers — enables higher occupancy
template<int BLOCK_SIZE>
__launch_bounds__(BLOCK_SIZE, 4) // max BLOCK_SIZE threads, target ≥4 blocks/SM
__global__ void my_kernel(...);
The classic occupancy debate

Volkov's "Better Performance at Lower Occupancy" (GTC 2010) made the case that high ILP (instruction-level parallelism) per thread can outperform high occupancy by keeping more memory transactions in flight per warp. Practical guidance: tune your __launch_bounds__ and measure. cuobjdump --dump-resource-usage tells you the actual register count; ncu tells you achieved occupancy.

Quick occupancy check

int max_blocks_per_sm;
cudaOccupancyMaxActiveBlocksPerMultiprocessor(
    &max_blocks_per_sm, my_kernel, block_size, dynamic_smem_bytes);

// Or let CUDA pick a launch config that maximizes occupancy
int min_grid, block_size;
cudaOccupancyMaxPotentialBlockSize(&min_grid, &block_size, my_kernel, 0, 0);

§10Warp divergence & predication

When threads in a warp take different branches, the warp executes both paths sequentially, with inactive threads masked off. Pure divergence is 50% efficiency for an if/else; 32-way divergence is 3% efficiency.

// Divergent — bad if condition varies within warp
if (x[i] > 0) {
  out[i] = expensive_a(x[i]);
} else {
  out[i] = expensive_b(x[i]);
}

// Branchless / predicated — both paths run, compiler emits select
float a = expensive_a(x[i]);
float b = expensive_b(x[i]);
out[i] = (x[i] > 0) ? a : b;

// Idea: when both branches are cheap, predication wins;
// when one branch is dramatically more expensive, divergence wins
// (because most threads skip the expensive path). Measure.

Sorting to reduce divergence

A common technique: sort by the branching condition before the kernel runs, so threads in each warp share a path.

// Before: random distribution, warps diverge ~50%
// After: cub::DeviceRadixSort sorts by category, warps converge

// In a join, you'd sort by partition id to make probe threads coherent
cub::DoubleBuffer<int32_t> keys(d_partition, d_partition_alt);
cub::DoubleBuffer<int32_t> vals(d_row_id,    d_row_id_alt);
cub::DeviceRadixSort::SortPairs(d_temp, temp_bytes, keys, vals, n);

§11Async copies, software pipelining, TMA

The pre-Ampere idiom for hiding global-memory latency was issuing loads into registers early, then doing computation while loads completed. Ampere's cp.async made this explicit: copy global → shared asynchronously, work on the previous tile, wait for the next.

#include <cuda/pipeline>

__global__ void pipelined(float const* in, float* out, int n) {
  __shared__ float tile[2][TILE_SIZE];   // double-buffered

  auto pipe = cuda::make_pipeline();
  int buf = 0;

  // Prefetch tile 0
  pipe.producer_acquire();
  cuda::memcpy_async(tile[buf], in + 0, sizeof(tile[buf]), pipe);
  pipe.producer_commit();

  for (int t = 1; t < num_tiles; ++t) {
    // Prefetch tile t into the other buffer
    pipe.producer_acquire();
    cuda::memcpy_async(tile[1 - buf], in + t * TILE_SIZE,
                       sizeof(tile[0]), pipe);
    pipe.producer_commit();

    // Wait for tile (t-1) and process it
    pipe.consumer_wait();
    process_tile(tile[buf], out + (t - 1) * TILE_SIZE);
    pipe.consumer_release();

    buf = 1 - buf;
  }
  // Drain last tile
  pipe.consumer_wait();
  process_tile(tile[buf], out + (num_tiles - 1) * TILE_SIZE);
  pipe.consumer_release();
}

TMA — Hopper's tensor memory accelerator

On H100, the TMA is a hardware DMA engine that copies multi-dimensional tiles between global and shared memory without using SMs. A single thread programs the TMA descriptor; the engine handles addressing, strides, swizzling, and barrier signaling. This is what lets H100 matmul kernels saturate HBM at unprecedented occupancy. For data processing, TMA is mostly useful for tile-based aggregations and large-tile scans.

The mental model upgrade

Pre-Ampere: threads issue loads; compiler hopes to schedule independent work between load and use. Ampere/Hopper: separate the request (async copy) from the arrival (barrier wait). Code structure becomes explicit producer-consumer pipelines. CUTLASS, CCCL, and the latest cuDF kernels are full of this pattern.

Part IV

Parallel algorithm primitives — the building blocks

Every cuDF operation reduces to a handful of primitives: reduction, scan, sort, compaction, gather/scatter, hash. Master these, and complex operations (joins, group-bys, parsers) decompose into compositions. CUB and CCCL implement them at production quality; you read their source, you understand them, you reach for them first.

§12Reductions

Combine N values via an associative operator (sum, max, min, AND, OR, custom). Three levels: warp, block, grid. Sketch each from the bottom up.

// Warp-level: 5 shuffles, no shared memory
__device__ float warp_sum(float v) {
  #pragma unroll
  for (int off = 16; off > 0; off >>= 1)
    v += __shfl_down_sync(0xffffffff, v, off);
  return v;
}

// Block-level: warp reduce, store per-warp result, warp reduce again
template<int BLOCK_SIZE>
__device__ float block_sum(float v) {
  __shared__ float warp_results[BLOCK_SIZE / 32];

  v = warp_sum(v);
  int lane = threadIdx.x %% 32;
  int warp = threadIdx.x / 32;
  if (lane == 0) warp_results[warp] = v;
  __syncthreads();

  if (warp == 0) {
    v = (lane < BLOCK_SIZE / 32) ? warp_results[lane] : 0.0f;
    v = warp_sum(v);
  }
  return v;  // thread 0 holds the block sum
}

// Grid-level: each block computes block_sum, atomically adds to global,
// OR each block writes a partial and a second kernel reduces those
__global__ void reduce_kernel(float const* in, float* result, int n) {
  int tid = blockIdx.x * blockDim.x + threadIdx.x;
  float v = (tid < n) ? in[tid] : 0.0f;
  v = block_sum<256>(v);
  if (threadIdx.x == 0) atomicAdd(result, v);
}

§13Scans and decoupled lookback

A scan (prefix sum) over [a, b, c, d] produces [a, a+b, a+b+c, a+b+c+d] for inclusive scan, or [0, a, a+b, a+b+c] for exclusive. Scan is the foundational primitive for stream compaction, radix sort, and many partition operations.

Block-level scan: Hillis-Steele

// Inclusive scan in shared memory — Hillis-Steele, log2(N) steps
template<int BLOCK_SIZE>
__device__ int block_scan_inclusive(int v) {
  __shared__ int s[BLOCK_SIZE];
  int tid = threadIdx.x;
  s[tid] = v;
  __syncthreads();

  for (int off = 1; off < BLOCK_SIZE; off <<= 1) {
    int t = (tid >= off) ? s[tid - off] : 0;
    __syncthreads();
    s[tid] += t;
    __syncthreads();
  }
  return s[tid];
}

Decoupled lookback — the killer device-wide scan

The naïve device-wide scan needs two passes: each block computes local scans, then a second pass adds the prefix from preceding blocks. Decoupled lookback (Merrill & Garland, 2016 — implemented in CUB) does it in one pass by having each block atomically publish its result and look back at predecessors. This is what makes cub::DeviceScan the fastest scan implementation extant.

Block 0: computes local sum → publishes [PREFIX] Block 1: computes local sum → publishes [AGGREGATE] looks back: sees block 0 PREFIX → updates to [PREFIX] Block 2: computes local sum → publishes [AGGREGATE] looks back: sees block 1 [PREFIX], combines → [PREFIX] ... Each block publishes a "state" (AGGREGATE → PREFIX) atomically. A trailing block can combine many AGGREGATEs without waiting for the chain.

Why this matters operationally

Scan appears wherever you need to compute output positions: "where in the output array does each thread write?" That's compaction, partition, copy_if, segmented operations, run-length encoding, sorting passes. Familiarity with scan and its complexity is a senior-level expectation.

§14Radix sort

Radix sort, not comparison sort, dominates GPU sorting. Why: it has predictable, regular memory access, no divergence in the inner loop, and complexity O(n · b/r) where b is bits per key and r is radix bits per pass. CUB's radix sort achieves >40 GB/s effective throughput on H100.

The algorithm:

  1. Take r bits of the key (e.g., 8 bits = 256 buckets).
  2. Histogram: count how many keys fall in each bucket.
  3. Scan the histogram to get the starting offset for each bucket.
  4. Scatter: each key is written to its bucket's current position.
  5. Repeat for the next r bits.
// CUB radix sort — what you actually call
void sort_uint64(uint64_t* d_keys, size_t n, cudaStream_t s) {
  uint64_t* d_keys_alt;
  cudaMallocAsync(&d_keys_alt, n * sizeof(uint64_t), s);

  cub::DoubleBuffer<uint64_t> keys(d_keys, d_keys_alt);

  size_t temp_bytes = 0;
  cub::DeviceRadixSort::SortKeys(nullptr, temp_bytes, keys, n, 0, 64, s);
  void* d_temp;
  cudaMallocAsync(&d_temp, temp_bytes, s);

  cub::DeviceRadixSort::SortKeys(d_temp, temp_bytes, keys, n, 0, 64, s);

  // If keys.Current() != d_keys, copy the result back
  if (keys.Current() != d_keys)
    cudaMemcpyAsync(d_keys, keys.Current(), n * sizeof(uint64_t),
                    cudaMemcpyDeviceToDevice, s);

  cudaFreeAsync(d_keys_alt, s);
  cudaFreeAsync(d_temp, s);
}
Floating-point radix sort needs key transformation

IEEE 754 floats can't be radix-sorted by their raw bits because of the sign bit and negative-number ordering. CUB handles this internally; if you roll your own, transform: positive floats keep their bits except flip the sign bit; negative floats flip all bits. Reverse the transform after sorting.

§16GPU hash tables

Hash tables are the heart of GPU joins and group-by aggregations. The CPU world's chained-bucket hash tables don't translate well — pointer chasing kills performance. GPU hash tables use open addressing: a flat array, probe sequence on collision. The canonical library is cuCollections (cuco).

Open-addressing hash table sketch

// Concept: a fixed-capacity hash map of K→V using linear probing
// Insert via atomicCAS on the key slot
__device__ bool insert(int32_t* keys, int32_t* vals,
                       int32_t k, int32_t v, uint32_t cap) {
  constexpr int32_t EMPTY = -1;
  uint32_t slot = hash(k) %% cap;
  while (true) {
    int32_t prev = atomicCAS(&keys[slot], EMPTY, k);
    if (prev == EMPTY) {
      vals[slot] = v;
      return true;
    }
    if (prev == k) return false;  // key already present
    slot = (slot + 1) %% cap;          // linear probe
  }
}

__device__ bool lookup(int32_t const* keys, int32_t const* vals,
                       int32_t k, int32_t& out, uint32_t cap) {
  constexpr int32_t EMPTY = -1;
  uint32_t slot = hash(k) %% cap;
  while (true) {
    int32_t kk = keys[slot];
    if (kk == k)     { out = vals[slot]; return true; }
    if (kk == EMPTY) { return false; }
    slot = (slot + 1) %% cap;
  }
}
Why this works on GPU

(1) The table is a flat array — no pointer chasing. (2) atomicCAS resolves write contention without locks. (3) Linear probing has good cache behavior (sequential reads). (4) Load factor below ~50% keeps probe lengths short. The cost is wasted space — production cuco maps default to 50-70% load factor.

Cooperative probing — using a warp per key

For workloads with long probe sequences (skewed data, high load factor), cuco offers cooperative probing: 8 or 32 lanes work together on one query, scanning a window of slots in parallel and combining via warp ballot. Crucial for join performance on real-world data.

Columnar data & the cuDF type system

The Apache Arrow columnar format is the lingua franca: cuDF, Velox, DuckDB, Polars, and Spark Photon all read and write it. Internalize Arrow, then layer cuDF's type system on top.

§17Apache Arrow columnar format

Three principles, embodied in every Arrow type:

  1. Columnar — values of one column live contiguously in memory. Cache-friendly for analytical operations; trivially SIMD-able.
  2. Null mask — a separate bitmap, one bit per row, says whether the value at that row is valid. Separate from the data buffer, so non-null values stay densely packed.
  3. Composable — strings = (offsets buffer, chars buffer); list = (offsets buffer, child column); struct = (multiple child columns).
column "x: int32[5]" (with one null) data: [10 | 20 | _ | 40 | 50] (the _ slot is undefined; mask covers it) null_mask: bit 0=1, 1=1, 2=0, 3=1, 4=1 → 0b11011 column "name: string[3]" offsets: [0, 5, 10, 13] (size = n+1) chars: "alicebobcarol" (size = offsets[n]) null_mask: all valid column "tags: list<string>[2]" offsets: [0, 2, 3] child: string column of length 3 null_mask: all valid

Why this layout dominates for analytics

  • Vectorization — kernel over a column is a tight loop on a contiguous array.
  • Compression — column-by-column compression (dictionary, RLE, bit-packing) is dramatically better than row-wise.
  • Predicate pushdown — for a filter on column X, only X needs to be read. Saves I/O proportional to num_columns / 1.
  • Zero-copy interop — Arrow buffers can be shared across languages and frameworks without serialization. cuDF's to_arrow() hands a pointer.

§19Strings, lists, structs columns

Strings — offsets + chars

The cuDF strings column is two child columns plus a null mask. offsets[i+1] - offsets[i] is the byte length of string i. chars[offsets[i] : offsets[i+1]] is the UTF-8 bytes.

// Compute the length of each string in a strings column
__global__ void string_lengths(int32_t const* offsets, size_type* out, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < n) out[i] = offsets[i + 1] - offsets[i];
}
Variable-width strings break naive parallelism

A kernel that does work proportional to string length has heavy thread imbalance — one thread per string means most threads sit idle while the longest string finishes. Three mitigations:

  • One warp per string: warp cooperatively walks the bytes.
  • Two-pass: first pass computes output offsets; second pass writes chars.
  • Sort by length: warps process strings of similar size, reducing imbalance within a warp.

cuDF uses combinations of all three depending on the operation. cudf::strings::contains (substring search) uses one warp per string with a parallel matcher.

Lists — offsets + child column

A list column of element type T is structurally identical to a strings column where the chars buffer is replaced with a column of T. Algorithms on lists therefore look a lot like algorithms on strings — process per element, then reconstruct offsets via scan.

Structs — collection of children

A struct column is N child columns of the same length, sharing a null mask. The null mask at the struct level is logically AND-ed with each child's mask to determine validity of (struct.field).

Dataframe algorithms — where the magic happens

§20Hash joins

The canonical GPU join is the hash join: build a hash table on the smaller side, probe with the larger side. The build/probe split exploits both available parallelism (independent inserts/lookups) and the asymmetry between the sides.

build side (smaller): probe side (larger): k1, v1 q1 k2, v2 ──build──→ HMAP q2 ──probe──→ for each q, k3, v3 q3 lookup q in HMAP ... ... emit matched pairs Complexity: O(|build|) to build + O(|probe| × probe_length) to probe. On uniform data, probe length is O(1); on skewed data, can degrade.

Implementation outline

// Sketch of an inner-join — actual cuDF is significantly more involved
struct join_result {
  rmm::device_uvector<size_type> left_indices;
  rmm::device_uvector<size_type> right_indices;
};

join_result inner_hash_join(
    cudf::table_view const& build, cudf::table_view const& probe,
    rmm::cuda_stream_view stream) {

  // 1. Hash each row of the build side into a multimap
  auto capacity = static_cast<size_t>(build.num_rows() * 1.5);
  cuco::static_multimap<hash_value_t, size_type> map{capacity, ...};

  auto build_hasher = cudf::row_hash{build, stream};
  map.insert(thrust::make_transform_iterator(...), ..., stream);

  // 2. For each probe row, look up matches; record (probe_idx, build_idx)
  auto probe_hasher = cudf::row_hash{probe, stream};
  auto [out_probe, out_build] = map.pair_retrieve(...);

  // 3. Verify true equality (hash collisions need a second check)
  auto row_equal = cudf::row_equality_comparator{build, probe};
  auto mask = thrust::transform_reduce(...,
      [=] __device__ (auto i) { return row_equal(out_probe[i], out_build[i]); });

  // 4. Stream-compact to keep only true matches
  return compact_pairs(out_probe, out_build, mask, stream);
}
The skew problem

If the build side is uniform but a few build keys appear billions of times, the multimap has heavy buckets and probes for those keys cause warp divergence and write contention on output writes. Mitigations: shuffle-aware joins that detect skew and route heavy keys to a separate path; broadcast the small side when one is tiny; bucketize the probe by hash range. In Spark-RAPIDS, this is one of the most actively tuned areas.

Join variants

  • Inner: only matched pairs.
  • Left outer: all probe rows; unmatched get NULL for build columns.
  • Full outer: all rows from both sides; harder because you need to track which build rows didn't match.
  • Semi/anti: filter-style — keep probe rows that do/don't have a match. Cheaper because no joined-row materialization.
  • Conditional: arbitrary predicate beyond equi-join. Nested-loop with parallelism.

§21Group-by aggregations

Two approaches; cuDF supports both.

Hash-based group-by (default)

A hash table maps group-key → row position (or accumulator). Each row hashes its key, locates or creates the group's slot, and atomically updates the aggregate.

// Atomic-accumulator hash group-by (sketch)
__global__ void groupby_sum_kernel(
    int32_t const* keys, double const* vals,
    int32_t* ht_keys, double* ht_sums, int cap, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;

  int32_t k = keys[i];
  double  v = vals[i];
  uint32_t slot = hash(k) %% cap;

  while (true) {
    int32_t prev = atomicCAS(&ht_keys[slot], EMPTY, k);
    if (prev == EMPTY || prev == k) {
      atomicAdd(&ht_sums[slot], v);
      return;
    }
    slot = (slot + 1) %% cap;
  }
}

Sort-based group-by

Sort by key, then segmented reduction over runs of equal keys. Better for very high cardinality (where the hash table doesn't fit comfortably) and gives sorted output for free.

Sum of float32 is not associative — parallel reduction can give a different result than sequential. §22Sort, top-k, window functions

Multi-column sort

Sort by columns A, B, C with mixed ascending/descending. The trick: radix-sort the keys by combining them into a single comparator with a stable sort, sorting on the least-significant column first, then the next, and so on (LSD radix).

Top-K

Top-K when K is small: a partial radix sort (only the top-K bits matter). When K is large: full sort + slice. Hybrid: partial sort with cub::DeviceRadixSort + a manual partition. cuDF picks among these adaptively.

Window functions

SQL ROW_NUMBER() OVER (PARTITION BY a ORDER BY b) and its kin: partition the data by a key, sort within partition by another key, compute an aggregate over a window. On GPU, this is sort-by-key followed by segmented scan (for cumulative aggregates) or segmented operations (for rank, lag, lead).

§24Parquet / ORC decode

Parquet is the dominant columnar file format. It stores data in column chunks divided into pages; each page can use dictionary encoding, run-length encoding, bit-packing, and a compression codec (Snappy, Zstd, LZ4) on top. Decoding is decompression → decode-encoding → materialize-into-buffer.

Parquet file ├── Row group 0 │ ├── Column chunk "a" (int32) │ │ ├── Page 0: dict-encoded, snappy-compressed │ │ ├── Page 1: plain, zstd-compressed │ │ └── ... │ ├── Column chunk "b" (string) │ └── ... ├── Row group 1 ├── ... └── Footer (schema, statistics, page locations)

Per-page parallelism

Each page is decoded by a CUDA block. Inside a page: decompression is sequential within a codec, but multiple pages decode in parallel. The challenge is that decompressed-page sizes are unknown, so the decoder must allocate conservatively or use a two-pass approach to discover sizes.

Stream-ordered semantics

The CUDA 11.2+ cudaMallocAsync family is stream-ordered: an allocation is "owned" by a stream; subsequent ops on that stream see the allocation; other streams must sync to see it. RMM's pool wraps this. Stream-ordered allocation is what lets cuDF chain kernels without host-side synchronization.

// Buffer freed asynchronously on destruction — back to the pool, not to the OS
Why this pattern dominates

Without stream-ordered allocation, every memory op needs an implicit or explicit sync, serializing the host with the device. With it, cuDF can build entire pipelines (read parquet → filter → join → group-by → write) and submit them as fire-and-forget; the host never blocks until the user asks for a result. That's where the speedups come from.

§26Memory pressure & spilling

GPU memory is finite (40-192 GB on a single device), and analytical workloads happily produce intermediate results larger than that. Strategies:

  • Unified Memory — cudaMallocManaged. OS-level paging between host and device. Performance is poor when working sets exceed device memory; works as a safety net.
  • Explicit spill to host — when memory pressure is detected, move cold buffers to pinned host memory. Dask-cuDF does this via Dask's spill-to-host mechanism.
  • Spill to disk — last resort. Throughput is bound by storage; with NVMe RAID, ~10 GB/s achievable.
  • Partitioned execution — break the operation into chunks that fit, process serially. This is what Dask-cuDF does for shuffles.

The OOM debugging workflow

  1. rmm::mr::tracking_memory_resource — records the allocation that crossed the limit, with a stack trace.
  2. nsys with the memory tracing flag — shows allocation lifecycles.
  3. Check for buffer leaks (RAII broken, exception unwinding through CUDA).
  4. Check for fragmentation — repeated alloc/free of varying sizes can leave the pool unable to satisfy a large request even when total free is sufficient.

§28Distributed — Dask-cuDF shuffles

Dask-cuDF distributes a DataFrame across workers (one GPU each, typically). Operations that are "embarrassingly parallel per partition" (map, filter, simple aggregations) are trivial. Operations that need data from multiple partitions — joins, group-bys across partitions, sorts — require shuffle.

SHUFFLE: redistribute rows by hash(key) so all rows with the same key land on the same worker. worker 0: ┐ ┌ worker 0: rows where hash(k) %% N == 0 worker 1: │ data │ worker 1: rows where hash(k) %% N == 1 worker 2: │ ─shuffle─→ worker 2: rows where hash(k) %% N == 2 worker 3: ┘ └ worker 3: rows where hash(k) %% N == 3 Then each worker performs local join/groupby — and the result is correct.

Shuffles are dominated by network bandwidth. With UCX-over-IB or NVLink-over-NVSwitch, a Dask-cuDF shuffle on a DGX moves ~10s of GB/s. With TCP, ~1-2 GB/s. The library choice is consequential.

Where the engineering complexity lives
  • Memory pressure during shuffle — each worker may need to hold (incoming + outgoing) data simultaneously. Spill-to-host or spill-to-disk during shuffle is non-optional.
  • Skew — a hot key sends 80% of rows to one worker, OOMing it. Detect and split.
  • Fault tolerance — Dask's scheduler can recompute failed tasks, but stateful shuffles need checkpointing.

§30Roofline & performance analysis

The roofline model: a kernel's max performance is bounded by either peak compute or peak bandwidth, whichever the kernel saturates first. Arithmetic intensity (FLOPs per byte loaded) determines which.

log(GFLOPS) ▲ │ _______________ peak compute │ / │ / │ / │ / ← compute-bound region (high arithmetic intensity) │ / │ / ← memory-bound region (low arithmetic intensity) │ / │ / └──────────────────────────→ log(FLOPs/byte) = arithmetic intensity Inflection point = peak_compute / peak_bandwidth For H100: ~67 FLOP/byte for FP32 (60 TF / 3 TB/s)

Most data-processing kernels are memory-bound. A reduction is ~1 FLOP per 4 bytes → intensity 0.25, deep in memory-bound. A hash probe is even worse — each byte loaded does very little arithmetic. The optimization frontier is therefore bandwidth utilization: load every byte the minimum number of times, in the most coalesced way possible.

The 80/20 of perf debugging
  1. What's the kernel achieving in GB/s? ncu reports dram__bytes_read.sum / kernel time. Compare to GPU peak (3 TB/s on H100).
  2. If ≥ 70% of peak: memory-bound and optimal. Look elsewhere — kernel launch overhead, host code, allocator.
  3. If < 30% of peak: memory access pattern problem. Check coalescing, vectorized loads, L2 hit rate.
  4. 30-70%: nuanced. Could be occupancy, instruction issue, or partial coalescing. Look at ncu's "Memory Workload Analysis" section.

The JD calls out cuDF, Velox, Presto, and Spark specifically. The story:

Velox

Meta's C++ execution engine — a vectorized, columnar runtime for SQL queries. Underlies Presto, planned for Spark and Trino. Problem 1 · Implement a histogram on GPU
The prompt

Given an array of N integers in [0, K), produce an array of K counts. K is small (256). Optimize for throughput.

Naive — slow
__global__ void histo_slow(int const* in, int* counts, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < n) atomicAdd(&counts[in[i]], 1);  // global atomic — contention
}
Production — shared-memory privatization
__global__ void histo_fast(int const* in, int* counts, int n) {
  __shared__ int local[256];
  if (threadIdx.x < 256) local[threadIdx.x] = 0;
  __syncthreads();

  int tid = blockIdx.x * blockDim.x + threadIdx.x;
  int stride = blockDim.x * gridDim.x;
  for (int i = tid; i < n; i += stride) {
    atomicAdd(&local[in[i]], 1);    // shared-mem atomic — fast
  }
  __syncthreads();

  if (threadIdx.x < 256) atomicAdd(&counts[threadIdx.x], local[threadIdx.x]);
}
Senior-level discussion points
  • Why shared-memory atomics are faster than global (lower latency, higher throughput, hardware path).
  • Grid-stride loop — handles N much larger than total threads, gives the launcher flexibility.
  • For larger K (e.g., 1M bins), shared memory doesn't fit; fall back to multiple sub-histograms or use CUB's DeviceHistogram.
  • Privatization scales — replicate the histogram per-warp for even more parallelism if K is small enough.
  • Bank conflicts on the shared histogram if many threads target the same bin — interleave or use atomic.
Problem 2 · Parallel prefix sum, from scratch
The prompt

Implement an inclusive scan on a device array, with full code. Don't use CUB.

Implementation — block scan + grid-level offset addition
__device__ int warp_inclusive_scan(int v) {
  int lane = threadIdx.x %% 32;
  #pragma unroll
  for (int off = 1; off < 32; off <<= 1) {
    int n = __shfl_up_sync(0xffffffff, v, off);
    if (lane >= off) v += n;
  }
  return v;
}

template<int BLOCK_SIZE>
__device__ int block_inclusive_scan(int v) {
  __shared__ int warp_sums[BLOCK_SIZE / 32];
  int lane = threadIdx.x %% 32;
  int warp = threadIdx.x / 32;

  v = warp_inclusive_scan(v);
  if (lane == 31) warp_sums[warp] = v;
  __syncthreads();

  if (warp == 0) {
    int w = (lane < BLOCK_SIZE / 32) ? warp_sums[lane] : 0;
    w = warp_inclusive_scan(w);
    if (lane < BLOCK_SIZE / 32) warp_sums[lane] = w;
  }
  __syncthreads();

  if (warp > 0) v += warp_sums[warp - 1];
  return v;
}

// Pass 1: each block scans its own chunk, writes block totals
__global__ void scan_block(int const* in, int* out, int* block_sums, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  int v = (i < n) ? in[i] : 0;
  v = block_inclusive_scan<256>(v);
  if (i < n) out[i] = v;
  if (threadIdx.x == blockDim.x - 1) block_sums[blockIdx.x] = v;
}

// Pass 2: scan block totals (recursive — usually one more level suffices)
// Pass 3: add block_sums[block-1] to every element in block
__global__ void add_block_offset(int* out, int const* scanned_block_sums, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (blockIdx.x > 0 && i < n) {
    out[i] += scanned_block_sums[blockIdx.x - 1];
  }
}
Senior-level discussion points
  • Three-pass structure is simple but uses extra memory. Decoupled lookback collapses to single pass — sketch it.
  • Warp-level scan with __shfl_up_sync is register-only, no shared memory.
  • For very long arrays, the recursive scan-of-block-sums might itself need a grid — at some point, just use CUB.
  • Numeric stability: this works for int; for floats, the parallel order changes the result. Document or use compensated summation.

Problem 4 · Sketch a GPU hash join
The prompt

Given two integer-keyed tables (build, probe), output pairs of (build_idx, probe_idx) where keys match.

Implementation outline
// Hash table: linear probing, atomicCAS insert (see §16)
struct slot { int32_t key; int32_t idx; };

__global__ void build_phase(int32_t const* keys, slot* ht,
                            int cap, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;

  int32_t k = keys[i];
  uint32_t s = hash(k) %% cap;
  while (true) {
    int32_t prev = atomicCAS(&ht[s].key, -1, k);
    if (prev == -1) {
      ht[s].idx = i;          // note: multimap would chain or use linked slots
      return;
    }
    s = (s + 1) %% cap;
  }
}

__global__ void probe_phase(int32_t const* keys, slot const* ht,
                            int32_t* out_build, int32_t* out_probe,
                            int* out_count, int cap, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i >= n) return;

  int32_t k = keys[i];
  uint32_t s = hash(k) %% cap;
  while (true) {
    int32_t kk = ht[s].key;
    if (kk == -1) return;
    if (kk == k) {
      int pos = atomicAdd(out_count, 1);   // crude; better: scan-based
      out_build[pos] = ht[s].idx;
      out_probe[pos] = i;
    }
    s = (s + 1) %% cap;
  }
}
Senior-level discussion points
  • This is unique-key only; multimap requires either chaining or rolling additional buckets per key.
  • Output via atomicAdd contends and produces unsorted output — production uses scan-based output position computation.
  • Hash function matters: Murmur3 or xxh32 in production; CRC32 if you have hardware support.
  • Load factor: too high → long probes; too low → wasted memory. Sweet spot ~50-75%.
  • How would you extend to multi-column keys? Combine columns into a single hash; verify true row equality after match (row equality kernel).
Problem 5 · Optimize a matrix transpose
The prompt

Transpose a NxN float matrix on GPU. Show the naïve version and an optimized one. Explain why the optimized one is ~10× faster.

Naïve — uncoalesced writes
__global__ void transpose_naive(float const* in, float* out, int N) {
  int x = blockIdx.x * blockDim.x + threadIdx.x;
  int y = blockIdx.y * blockDim.y + threadIdx.y;
  if (x < N && y < N) out[x * N + y] = in[y * N + x];
  // Read is coalesced (consecutive threads, consecutive memory)
  // Write is strided by N — terrible coalescing
}
Optimized — shared-memory tile, padded to avoid bank conflicts
constexpr int TILE = 32;

__global__ void transpose_fast(float const* in, float* out, int N) {
  __shared__ float tile[TILE][TILE + 1];   // +1 padding kills bank conflicts

  int x = blockIdx.x * TILE + threadIdx.x;
  int y = blockIdx.y * TILE + threadIdx.y;
  if (x < N && y < N) {
    tile[threadIdx.y][threadIdx.x] = in[y * N + x];   // coalesced read
  }
  __syncthreads();

  x = blockIdx.y * TILE + threadIdx.x;
  y = blockIdx.x * TILE + threadIdx.y;
  if (x < N && y < N) {
    out[y * N + x] = tile[threadIdx.x][threadIdx.y];   // coalesced write
  }
}
Senior-level discussion points
  • The shared memory tile lets both global accesses (read and write) be coalesced. Without it, one direction must be strided.
  • The [TILE+1] padding shifts banks so column accesses don't collide. Without it, the column-major shared-memory read would 32-way conflict.
  • For very large matrices, consider out-of-place vs in-place — in-place transpose is much harder and rarely worth it on GPU.
  • cuBLAS has cublasGeam for transpose; in production just use that. But this kernel is a rite of passage.
Problem 6 · A debugging story — kernel produces wrong results
The scenario

Your reduction kernel passes unit tests but produces wrong results on the production data. The output is occasionally off by a few. What do you check?

The checklist
  1. Floating-point non-associativity. Parallel reduction sums in a different order than sequential. Test against a sequential reference with fabsf(a - b) < tolerance, not ==.
  2. Race conditions. compute-sanitizer --tool racecheck finds them. Common cause: missing __syncthreads() between shared-memory writes and reads in different threads.
  3. Out-of-bounds. compute-sanitizer --tool memcheck. Often the boundary in the last block.
  4. Uninitialized shared memory. Shared memory isn't zero-initialized. Threads beyond n may read garbage from sdata[].
  5. Improper warp sync. Post-Volta, warps no longer implicitly converge after divergence. Need __syncwarp() or _sync-suffixed shuffles.
  6. Wrong stream. Result read before kernel completed because read was on a different stream without dependency.
  7. Stale cache. Rarely — but cudaDeviceSynchronize() + __threadfence() issues can show up in multi-kernel pipelines.
The senior signal

A senior candidate will reach for compute-sanitizer as the first move (free, catches 90% of bugs), then explain how to construct a minimal repro by halving the dataset until the failure isolates. Bonus points for mentioning cuda-gdb for stepping through device code, and for the observation that most production wrong-result bugs are not in the kernel — they're in how the kernel was wired into the pipeline (wrong stream, stale buffer, type punning).

A senior engineer in this space can move between the C++ template that dispatches the operation, the CUDA kernel that executes it, and the parallel algorithm that justifies it — without losing fluency at any level.
