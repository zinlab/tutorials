Part I

The system & the layered stack

A wrapper of a wrapper of a framework. Before any single competency is useful you have to see how kintera sits on top of pydisort and pyharp, both of which sit on torch — and what that costs you at every stage of the life cycle.

§00Synopsis & the competency map

The NVIDIA Senior Life Cycle Engineer brief calls out two things that pydisort barely exercises but kintera exercises constantly: “many contributing teams” and “shipping multi-cloud large-scale systems and supporting those systems in production.” kintera makes both literal. Its build links against the artifacts of three other Python packages, its wheel is the first in the workspace stack to actually ship a CUDA build, and its CD job runs inside PyTorch’s own manylinux image.

The deep truth specific to kintera: a layered native extension is a multi-team contract enforced at link time and at run time. pydisort’s contract was with torch. kintera’s contract is with torch and pydisort and pyharp and NetCDF — four independent release calendars, each with its own ABI surface. The life-cycle engineer’s job is to make every clause of all four contracts true on every release, on Linux x86_64 with and without GPUs, and on macOS arm64.

If you have already read the pydisort guide, the foundational mechanisms (the two-stage build, the dual C++ ABI, $ORIGIN RPATHs, the vendor-vs-depend packaging fork) are the same. This guide is structured to lean on those where they repeat, and spend its pages on the four things kintera adds: (a) cross-repo discovery, (b) the aggressive Linux linker, (c) a CUDA wheel that actually ships, and (d) the operational discipline of versioning and releasing a many-author repo.

§01The layered native stack

When a user types import kintera, the loader walks a strictly taller tower than pydisort’s. Knowing this map cold is again the trick to placing any bug at the right floor in seconds.

user code import kintera; kintera.ThermoX(opts).forward(T, P, X) │ ┌────▼─────────────────────────────────────────────────────────────────┐ │ kintera/__init__.py + atm2d/ (pure Python orchestration) │ Python ├────▼─────────────────────────────────────────────────────────────────┤ │ kintera/kintera*.so pybind11 module: thermo, kinetics, photo, │ C++ / Python C-API │ constants, sparsesolver, diffusion │ ├────▼─────────────────────────────────────────────────────────────────┤ │ libkintera_release.so thermo, kinetics, photochem, diffusion, │ C++17 + LibTorch │ libkintera_cuda_release.so math, vapors, utils — over 9 src dirs │ CUDA (Linux wheels) ├────▼─────────────────────────────────────────────────────────────────┤ │ SIBLING Python packages (installed in the SAME site-packages): │ │ pydisort/lib/libdisort_release.so (radiative transfer) │ │ pyharp/lib/libharp_release.so (RT kernels / infra) │ ├────▼─────────────────────────────────────────────────────────────────┤ │ libtorch_cpu.so libc10.so (+ libtorch_cuda, libc10_cuda) │ user's torch ├────▼─────────────────────────────────────────────────────────────────┤ │ SYSTEM libs: libnetcdf, libcusolver, libcusparse, libcublas, libm │ OS / driver ├────▼─────────────────────────────────────────────────────────────────┤ │ libstdc++ / libgomp + CUDA driver (libcuda.so) → PTX JIT │ └────────────────────────────────────────────────────────────────────────┘

Three properties of this taller tower drive every later chapter:

  • The sibling-package floor is new. pydisort and pyharp are not system libraries and not torch — they are Python packages installed in the same site-packages directory, and kintera resolves their shared libs by walking up to that directory and back down. This is what $ORIGIN/../pydisort/lib means in §09.
  • NetCDF is the first true system dep. Unlike torch and the siblings, NetCDF is not installed by pip — it is apt-get install libnetcdf-dev on Linux, brew install netcdf on macOS, yum install netcdf-devel in the CD container. The CI matrix has to branch on OS to install it.
  • The CUDA layer is real, not aspirational. kintera’s Linux wheels ship a libkintera_cuda_release.so built inside pytorch/manylinux2_28-builder:cuda12.8. The user does not install the CUDA toolkit — CUDA math libs (cusolver/cusparse/cublas) come from torch’s own bundled set, found via RPATH (§14).
The placement reflex, layered edition

When something breaks: “undefined symbol” with pydisort:: or harp:: in the name → sibling-floor (§03/§04); with __cxx11 → ABI (§07); with at:: or c10:: → torch drift (carry forward pydisort’s §12). “cannot open libnetcdf.so” → system dep missing (§17). “cannot open libcusolver.so.12” → torch absent or no GPU (§14). “no kernel image” → CUDA arch (§10).

§02The two-stage build, layered edition

kintera uses the same two-stage CMake→setuptools build as pydisort, with one consequential addition: Stage 1 itself depends on installed Python packages. find_package(Disort REQUIRED) and find_package(Harp REQUIRED) shell out to python -c "import pydisort" to locate their headers and shared libs. So before Stage 1 can even configure, the build environment must already have pyharp and pydisort installed in the same Python that CMake will use.

STAGE 0 — environment pip install 'pyharp>=2.2.0' 'pydisort' (implicit via pyharp) ^^^^^^^^^ pip install 'torch==2.10.0' + system: NetCDF │ STAGE 1 — CMake cmake -B build ┌── runs `python -c "import pydisort"` cmake --build build ├── runs `python -c "import pyharp"` ├── libkintera_release.so └── locates their lib & include dirs └── (optional) libkintera_cuda_release.so STAGE 2 — setuptools+torch pip install --no-build-isolation --no-deps . ├── parse_library_names(build/lib) → orders cuda last ├── compute 6 RPATHs (own + torch + pydisort + pyharp) └── kintera/kintera.cp3XX-*.so

The implication is that “pip install kintera from a clean machine” is a non-trivial chain — the build-system table in pyproject.toml bakes the chain into PEP 517 as a hard requirement:

[build-system]
requires = [
  "setuptools>=61.0.0", "setuptools-scm>=8", "wheel", "build", "ninja",
  "pyharp>=2.2.0",    # native sibling — must be importable before CMake runs
  "torch==2.10.0",    # exact pin: shared with pydisort/pyharp; the keystone
]
build-backend = "setuptools.build_meta"

Note that pydisort is not in requires: it comes in as a transitive dependency of pyharp. That choice is a tiny but important release-coordination decision (you pin the immediate parent and trust its tree), and it concentrates breakage on the pyharp release cadence.

The build environment is a contract too

A torch extension whose Stage 1 imports siblings has a fourth ABI it has to keep right: the build-time Python. The CD pipeline goes out of its way to verify this with a literal smoke test inside CIBW_BEFORE_BUILD:

# Test that pydisort imports correctly in the *same Python* that CMake will use
python - <<'PY'
import sys, os
try:
    import pydisort
    print("pydisort import OK:", pydisort.__version__)
except Exception as e:
    print("pydisort import FAILED:", e); sys.exit(1)
PY

A failed sibling import here means CMake will fail in Stage 1 with a less clear message; the fail-fast in the right Python is exactly the kind of pipeline hygiene a senior engineer adds.

Decision tree — how to depend on a sibling native package

  • Sibling is a torch extension, on PyPI → this pattern: find_package by Python import, declare in build-system.requires, RPATH to $ORIGIN/../<sibling>/lib.
  • Sibling is C++-only, on the system → classical find_package against an installed CMake config; link via system paths.
  • Sibling is git-only, no release → FetchContent (vendor the source); avoid for anything that holds global state (the §11 trap from pydisort).
  • Sibling is yours, monorepo → CMake add_subdirectory; one build, no Python handshake.
Part II

C++ build: dependencies, symbols, linking

Same brief, harder problems — because every problem from pydisort returns with one extra layer of indirection, and three new ones appear.

§03Cross-repo discovery in CMake

The CMake idiom for “find an installed Python-packaged native lib” is not standardized; kintera implements it in FindDisort.cmake and FindHarp.cmake. The pattern, distilled:

set(Python3_FIND_VIRTUALENV ONLY)              # pin to the active venv, not /usr/bin/python
find_package(Python3 QUIET COMPONENTS Interpreter)

execute_process(COMMAND ${Python3_EXECUTABLE} -c
  "import pydisort; print(pydisort.__file__)"
  OUTPUT_VARIABLE disort_init_file OUTPUT_STRIP_TRAILING_WHITESPACE)

cmake_path(REPLACE_FILENAME disort_init_file lib     OUTPUT_VARIABLE disort_lib_dir)
cmake_path(REPLACE_FILENAME disort_init_file disort  OUTPUT_VARIABLE disort_include_dir)

find_path(_DISORT_HEADER_DIR disort.hpp HINTS ${disort_include_dir} ...)
find_library(DISORT_LIBRARY disort_release  HINTS ${disort_lib_dir} ...)

__disort_determine_version()                  # import pydisort; print(pydisort.__version__)
find_package_handle_standard_args(Disort REQUIRED_VARS ... VERSION_VAR DISORT_VERSION)

The same pattern (with one ankle-grabber) is in FindHarp.cmake, which additionally calls link_directories against pyharp’s .dylibs directory on macOS — reflecting that pyharp ships some libs in pyharp/.dylibs rather than pyharp/lib:

if (APPLE)
  link_directories(${Python3_SITELIB}/pyharp/.dylibs)
else()
  link_directories(${Python3_SITELIB}/pyharp.libs)
endif()

These directories (pyharp.libs on Linux, pyharp/.dylibs on Mac) are auditwheel/delocate’s own convention for the libs they vendored into the sibling wheel — an undocumented surface that becomes load-bearing across teams.

Production issues

  • The Python version baked at configure time wins. Python3_FIND_VIRTUALENV ONLY locks to the active interpreter; if a contributor accidentally runs cmake outside their venv, the lookups silently succeed against a different Python with a different (or absent) pydisort — producing a kintera build that links against the wrong sibling. Surfacing Python3_EXECUTABLE in the CMake log is the cheap fix; refusing to configure without a venv is the better one.
  • Sibling layout is not standardized. pyharp/.dylibs vs pyharp.libs is auditwheel’s internal convention, not pyharp’s API. A pyharp release that changes its repair-tool or its layout silently breaks kintera’s build — the kind of multi-team coupling the brief warns about.
  • No installed CMake config. The cleaner pattern would be for pydisort and pyharp to ship a pydisortConfig.cmake in their wheel and have kintera do plain find_package(pydisort). Today the contract is implicit and lives in Find modules; a senior engineer would push to formalize it.
Improve — the Config.cmake upgrade

Have pydisort and pyharp install proper CMake package config files via install(EXPORT). Then kintera replaces FindDisort.cmake with find_package(pydisort CONFIG REQUIRED), and the cross-repo contract becomes a versioned, namespaced target (e.g. pydisort::disort) instead of a path-walk. This is exactly the layered-team coordination move the brief is rewarding.

§04The site-packages contract

Once Stage 1 has linked against pydisort’s and pyharp’s shared libs, Stage 2 must encode where to find them at runtime. pydisort needed two RPATH entries; kintera needs six, and they tell a complete sociological story about layered Python-native deps:

if sys.platform == "darwin":
    extra_link_args = [
        "-Wl,-rpath,@loader_path/lib",                # my own libkintera
        "-Wl,-rpath,@loader_path/../torch/lib",        # torch (sibling)
        "-Wl,-rpath,@loader_path/../pydisort/lib",     # pydisort (sibling)
        "-Wl,-rpath,@loader_path/../pyharp/lib",       # pyharp (sibling)
    ]
else:
    extra_link_args = [
        "-Wl,-rpath,$ORIGIN/lib",
        "-Wl,-rpath,$ORIGIN/../torch/lib",
        "-Wl,-rpath,$ORIGIN/../pydisort/lib",
        "-Wl,-rpath,$ORIGIN/../pyharp/lib",
        f"-Wl,-rpath,{site_dir}/torch/lib",            # ABSOLUTE fallback to site-packages
        f"-Wl,-rpath,{site_dir}/pydisort/lib",
        f"-Wl,-rpath,{site_dir}/pyharp/lib",
    ]

The Linux side has an extra trick that pydisort doesn’t: both relative and absolute paths are baked in. $ORIGIN/../torch/lib works when kintera is installed in the same site-packages as torch (the normal case). The absolute {site_dir}/torch/lib works when something has moved kintera (e.g., a vendored install, a frozen app, an unusual layout) — the loader tries both.

The absolute-path tradeoff

Baking absolute paths into an ELF binary makes it less relocatable: the wheel built on the runner’s /opt/python/cp311/.../site-packages carries that exact string forever. It happens to coincide with where pip installs in practice, but a custom prefix install will leave those entries dangling. The relative $ORIGIN entries do the real work; the absolute entries are a safety net — arguably one to retire once the relative paths are battle-tested.

Recall from the pydisort guide that -Wl,-rpath emits DT_RUNPATH on modern Linux, which is non-transitive. That means each of these six entries is consulted only for kintera’s direct NEEDED list. pyharp’s and torch’s own transitive deps rely on their own RPATHs — which is why pyharp wheels ship their auditwheel-vendored libs in pyharp.libs/.dylibs rather than pyharp/lib: those folders are where pyharp’s own RPATH points, and the §03 link_directories dance is how kintera reaches into that internal layout to link.

Improve

The cleanest long-term move is to depend only on the public portion of each sibling’s layout (pyharp/lib, not pyharp.libs). That requires pyharp to ship its libs under the public path — a coordinated change but one that turns an implicit contract into an explicit one.

§05Configure-time codegen

kintera is the first repo in the workspace stack to use CMake configure_file for build-time C++ codegen. The template:

#pragma once
#include <cstddef>
// NetCDF output (NETCDFOUTPUT or NO_NETCDFOUTPUT)
#define @NETCDF_OPTION@
#define @PNETCDF_OPTION@
#define @CUDA_OPTION@

#ifdef __CUDACC__
  #define DISPATCH_MACRO __host__ __device__
#else
  #define DISPATCH_MACRO
#endif

constexpr size_t KINTERA_CUDA_HEAP_SIZE = 256 * 1024 * 1024;  // 256 MB
constexpr char KINTERA_ROOT_DIR[] = "@CMAKE_CURRENT_SOURCE_DIR@";
constexpr char HARP_ROOT_DIR[]    = "@HARP_INCLUDE_DIR@";

CMake’s configure_file(... configure.h @ONLY) substitutes the @FOO@ tokens into build/configure.h, which the rest of the source includes. This is how feature flags (NETCDFOUTPUT/NO_NETCDFOUTPUT), the source-tree root, and the harp include root become available to C++ code at compile time without poisoning the command line with -Ds.

Production issues

  • Source-tree paths in the binary. KINTERA_ROOT_DIR is baked in at configure time. In a developer build it points to your local checkout, in CI it points to the runner’s workspace, in the wheel it points to a path that no longer exists on the user’s machine. Anything that reads files from KINTERA_ROOT_DIR at runtime in a shipped wheel will fail. Audit usages and prefer importlib.resources or a Python-side data-path lookup over a baked-in path.
  • Same for HARP_ROOT_DIR. It captures pyharp’s include dir at kintera’s build time — if the user’s pyharp is reinstalled to a different location after kintera was built, the binary still remembers the old one.
  • The mutually-exclusive define dance. @NETCDF_OPTION@ expands to either NETCDFOUTPUT or NO_NETCDFOUTPUT as a single token in the macro name — clever, but it means every guard in the codebase has to use the same constant. A typo silently disables a feature. Splitting into a boolean macro (#define KINTERA_HAVE_NETCDF 1) is the modern idiom.
The dispatch macro is the right thing

DISPATCH_MACRO expands to __host__ __device__ when compiled by nvcc and to nothing otherwise. That single token is how the same C++ function body can be called from CPU host code and from a CUDA kernel — the cornerstone of pyharp-style performance-portable code. The configure-time generation is what keeps non-CUDA builds free of nvcc keywords.

Part III

Symbols, ABIs & the linker

Every load-time failure pydisort had, kintera inherits. Two are specifically louder here: AVX-tier defines that travel through Python, and the Linux linker’s eagerness to throw away the CUDA libs you need.

§06Dependency acquisition, layered

kintera has more shapes of dependency than pydisort, and each shape has its own pinning posture:

DepMechanismPin & tradeoff
torchpip exact pin==2.10.0 — the keystone; identical to pydisort/pyharp (§07)
pyharppip range pin>=2.2.0 — trusted to honor SemVer; pulls pydisort transitively
NetCDF (system)OS package managerper-OS (apt/brew/yum); declared in CI/CD & Dockerfile, not in pyproject
fmtFetchContent + git tagpinned tag with .cache tarball; same macro as pydisort
yaml-cppFetchContent + git tagthe YAML parser for reactions/kinetics/data files
googletestFetchContent + git tagreal unit-test framework (vs pydisort’s plain executables)

The lift over pydisort: system NetCDF turns the build into a per-OS install dance, and yaml-cpp introduces a header-only-ish runtime-config story for chemistry (reactions in YAML, ingested at startup). yaml-cpp’s API has shifted over the years; pinning a specific tag is what keeps users on consistent parser semantics.

The pyharp range pin is the weak link

pyharp>=2.2.0 trusts the pyharp project to never break ABI within a major. But pyharp is a torch extension too, and (like pydisort) it reaches into ATen internals (§05 of the pydisort guide) where there is no stable ABI. A pyharp minor release that bumps its own torch pin or changes its sibling layout will break kintera even though the range says it shouldn’t. The defensive move is to use the same == discipline kintera uses for torch — pin pyharp exactly per kintera release, and bump deliberately.

§07ABI: leveraging the lesson

The _GLIBCXX_USE_CXX11_ABI handling is identical to pydisort: detect torch’s value via execute_process, propagate via add_compile_definitions. The new failure mode that only shows up in kintera is the three-way ABI alignment: kintera, pydisort, and pyharp must all have been built against the same torch ABI as the user’s torch.

This works in practice for one boring reason and one important one. The boring reason: all three repos run the same detection code against the same exact torch pin (==2.10.0), so they always agree. The important reason: PyPI delivers them as separate wheels, so an ABI mismatch can be diagnosed precisely.

The cross-team ABI failure

Suppose pyharp ships a 2.3.0 wheel built against a torch RC that flipped the ABI. The user installs kintera, pip resolves the new pyharp, and the import-time error looks like a kintera bug — undefined symbol in a libharp call. The triage reflex (§22): always check the three versions, and the ABI of each: python -c "import pydisort, pyharp, torch; print(*[m.__version__ for m in (pydisort, pyharp, torch)])"; if any disagree about _GLIBCXX_USE_CXX11_ABI you have found the bug.

§08Symbols: AVX defines from torch

In pydisort, AVX2/AVX512 capability defines were set in CMake’s compilers.cmake as static defines for non-Apple. kintera does it differently and better — in setup.py, by parsing torch’s own build configuration:

def torch_dispatch_stub_defines():
    config = torch.__config__.show()
    defines = []
    if "CPU capability usage: AVX512" in config:
        defines.append("-DHAVE_AVX512_CPU_DEFINITION")
    if ("PERF_WITH_AVX2=1" in config or
        "CPU capability usage: AVX2" in config or
        "CPU capability usage: AVX512" in config):
        defines.append("-DHAVE_AVX2_CPU_DEFINITION")
    return defines

Why this matters: REGISTER_ALL_CPU_DISPATCH (the macro pydisort and kintera both use to register CPU kernels into ATen’s DispatchStub) expands differently depending on whether HAVE_AVX2_CPU_DEFINITION / HAVE_AVX512_CPU_DEFINITION are set. If kintera disagrees with torch about which SIMD tiers exist, its registrations land in tiers the dispatcher never queries — or it tries to register tiers torch doesn’t have, causing build failures or silent no-ops at runtime. The pydisort approach (static defines based on the host CPU) is correct for most cases but is technically guessing; kintera’s approach (ask torch what it believes) is the ABI-honest answer.

Improve, both repos

Push this Python-side detection back into pydisort and pyharp too — the truth about CPU capability tiers belongs to torch, and a torch-extension build should ask torch, not the compiler. One shared helper module across the three repos is the cleanest end state.

§09Linking: order & --no-as-needed

This is the chapter where the brief’s “linking problems” bullet earns its keep. kintera’s setup.py does two non-obvious link-time things that pydisort doesn’t. Both are responses to the Linux linker behaving exactly as documented, in the way that breaks you anyway.

(a) Library ordering — consumers before providers

GNU ld resolves symbols in a single left-to-right pass: each library on the link line sees only symbols it can satisfy from itself plus libraries to its right. So if libkintera_release.so (consumer) calls into libkintera_cuda_release.so (provider), the consumer must appear first on the line, the provider after it. setup.py enforces this by sorting:

# 1) non-cuda libs first (CONSUMERS, with unresolved cuda symbols)
kintera_non_cuda = [l for l in library_names if l.startswith("kintera") and "cuda" not in l]
# 2) cuda libs last (PROVIDERS, satisfying those unresolved symbols)
kintera_cuda     = [l for l in library_names if l.startswith("kintera") and "cuda" in l]
# 3) everything else (NetCDF, torch — fmt & yaml-cpp are linked statically/header)
other = [l for l in library_names if not l.startswith("kintera")]
return kintera_non_cuda + other + kintera_cuda

The exact ordering rule encoded here — kintera non-cuda → system → kintera cuda — is a tiny but real piece of build engineering: it would not even be visible without splitting the library into a CPU and a CUDA half. The moment that split exists, the order matters.

(b) --no-as-needed — defeating the aggressive linker

Modern ld on Linux defaults to --as-needed: it drops any library from the binary’s NEEDED list if it doesn’t directly satisfy a symbol used by the binary itself. Static initializers that only register things into a global table (the §05 pattern from the pydisort guide) look unused to ld, so it strips them, the registration constructors never run, and the kernels are missing at runtime — precisely the “builds fine, fails at runtime” symbol bug.

# ubuntu system has an aggressive linker that removes unused shared libs
# add cuda library explicitly if built with cuda
cuda_libraries = [lib for lib in libraries if "cuda" in lib]
if cuda_libraries:
    for lib in cuda_libraries:
        libraries.remove(lib)
    cuda_linker = (
        ["-Wl,--no-as-needed"]            # DO keep these on NEEDED even if they look "unused"
        + [f"-l{lib}" for lib in cuda_libraries]
        + ["-Wl,--as-needed"]              # restore the default for everything that follows
    )

The wrap-and-restore pattern is the right idiom: it applies --no-as-needed only to the libs that need it, not globally (which would bloat NEEDED for everyone).

The bug this prevents, in one line

Without the wrap, import kintera on a CUDA wheel succeeds, kintera.thermo.someop on a CUDA tensor fails with RuntimeError: Could not run '...' with arguments from the 'CUDA' backend. nm libkintera_cuda_release.so | grep register shows the registration symbols are present; readelf -d libkintera.cpython-311-...so | grep NEEDED shows they are not listed — the linker stripped them. The fix is exactly the --no-as-needed wrap.

Debugging reflex

readelf -d kintera/kintera*.so | grep NEEDED                     # are the cuda libs even on the list?
nm -D -C build/lib/libkintera_cuda_release.so | grep -i register  # are the registrations present?
LD_DEBUG=libs python -c "import kintera; t = ..." 2>&1 | grep cuda  # do they actually load?
Part IV

Performance portability, for real

In pydisort the CUDA path was a scaffold and the wheels were CPU-only. In kintera the CUDA path is shipped — built inside PyTorch’s own manylinux image, with a CUDA math-library exclusion list pages long. This is where the brief’s “compilers, math libraries, drivers” layer becomes concrete shipping infrastructure.

§10CUDA that actually ships

The CMake CUDA-arch policy is identical to pydisort’s — 60-89 with version-gated 90/120 — but kintera turns it on (-DCUDA=ON) in the CD pipeline and the Linux wheels carry it. The keystone decision is the manylinux base image:

CIBW_MANYLINUX_X86_64_IMAGE: docker.io/pytorch/manylinux2_28-builder:cuda12.8

This is PyTorch’s own manylinux builder image, with the toolchain that PyTorch itself uses to produce its torch==2.10.0 wheels, plus CUDA 12.8 pre-installed at /usr/local/cuda. Two reasons it is the right choice:

  • ABI alignment by construction. Whatever GCC, libstdc++, and CUDA the user’s torch was built against, this image is the closest mirror — minimizing the §07 surface.
  • Toolkit available without you installing it. Installing a multi-GB CUDA toolkit inside a vanilla manylinux image during every CD run is wasteful and brittle; using the pre-baked one is fast and reproducible.
The CUDA-versioned wheel implication

This wheel was built against CUDA 12.8. The CUDA runtime API is forward-compatible across minors but not always across majors; the published wheel implicitly requires a user with a CUDA 12.x torch. PyPI metadata cannot express “CUDA 12 only” in the wheel tag (it’s still just manylinux_2_28_x86_64), so a CUDA 11 user installing this wheel will get a runtime mismatch with no helpful warning. The fix is convention: document the supported torch/CUDA matrix loudly in the README and the release notes, or follow PyTorch’s +cu128 local version label convention on a separate index.

The --extended-lambda requirement

kintera relies on host-device generic lambdas across the GPU/CPU boundary (the §05 DISPATCH_MACRO idiom). nvcc only accepts those with two non-default flags, which the build sets globally:

set(CMAKE_CUDA_FLAGS "${CMAKE_CUDA_FLAGS} --extended-lambda --expt-relaxed-constexpr")

Without these, you get cryptic error: __host__ __device__ lambda is not supported on the kernels that share code paths between CPU and GPU — one of those compiler-error-as-policy-decision moments where the obvious fix happens to also be the right one.

§11Device dispatch & the codegen header

The dispatch architecture re-uses pydisort’s ATen DispatchStub pattern (DECLARE/DEFINE/REGISTER across CPU and CUDA TUs — see pydisort guide §05). What kintera adds is a small but consequential bit of state: a tunable CUDA heap size:

constexpr size_t KINTERA_CUDA_HEAP_SIZE = 256 * 1024 * 1024; // 256 MB

This sits in the configure-generated header so a build can override it without code edits. CUDA device-side allocations (malloc/new inside kernels) live in a per-device heap that defaults to 8 MB; a chemistry kernel that allocates per-thread Jacobian state for a reaction network will exhaust that quickly. Bumping to 256 MB is a pragmatic ceiling; the right place to expose it is exactly this constant.

The performance-portability spine, summarized

Same single source tree compiles to CPU and CUDA because three things line up: (1) DISPATCH_MACRO from the configure header gates __host__ __device__; (2) --extended-lambda lets generic lambdas cross the boundary; (3) ATen’s DispatchStub selects between the two at runtime per tensor. This is the same pattern modern PyTorch ops follow internally — kintera reuses it deliberately.

§12The multi-module torch surface

pydisort exposed one main pybind module. kintera exposes one Python-level module backed by six binding TUs, each owning a domain:

BindingDomain
kintera.cpptop-level PYBIND11_MODULE(kintera, m) — species, reactions, the umbrella
pythermo.cppthermodynamics: ThermoX, EoS, reference states, NASA9 polynomials
pykinetics.cppreaction networks, rate evaluation, Arrhenius/falloff
pyphotolysis.cppphotochemistry: cross sections, actinic flux coupling
pysparsesolver.cppsparse linear algebra for stiff networks (cusolver-backed on GPU)
pydiffusion.cppmolecular/eddy diffusion operators
pyconstants.cppphysical constants (R, k_B, Avogadro, …)

Splitting the binding into one TU per domain is a maintainability win — each domain’s C++ side can grow without producing a 5,000-line kintera.cpp. There is a sharp edge to it though.

The static-init order trap

Each binding TU registers types and submodules with pybind11 at static-init time. If any two registrations have a dependency on each other (TU A defines a Python class that TU B’s function signature references via py::class_), the C++ static initialization order across TUs is unspecified and you can get import-time crashes like generic_type: type X is already registered or incompatible function arguments. Keep one root PYBIND11_MODULE (kintera.cpp) that calls per-domain bind_*(m) functions in a fixed order — pydisort already follows this pattern, kintera should too — rather than letting each TU register itself.

The Python side composes all of this back into one logical namespace, plus a separate kintera.atm2d pure-Python orchestration package (solver.py, assembly.py, chemistry.py, …) that uses the C++ ops as building blocks. That is the “high-level API on top of native” pattern done well: the C++ side is composable primitives, the Python side is the user-facing workflow.

Packaging & release management

A CUDA wheel under one roof with a CPU wheel, four Python versions, two operating systems, four sibling Python packages to coordinate, and a release calendar that no single person controls.

§13Wheels with the pytorch builder

The cibuildwheel surface is the same as pydisort’s. What changes is the image for Linux and the preparation:

CIBW_PLATFORM: linux
CIBW_MANYLINUX_X86_64_IMAGE: docker.io/pytorch/manylinux2_28-builder:cuda12.8
CIBW_BUILD: cp${{ env.PY_VERSION }}*           # cp310..cp313
CIBW_ARCHS: x86_64
CIBW_SKIP: "*musllinux*"
CIBW_BEFORE_BUILD: |
  # Activate the matrix Python explicitly INSIDE the container
  export PATH="/opt/python/cp${{ env.PY_VERSION }}-cp${{ env.PY_VERSION }}/bin:$PATH"
  dnf install -y epel-release ...
  yum install -y netcdf-devel                  # SYSTEM dep, installed inside the container
  pip install numpy 'cmake<=3.30' 'pyharp>=2.2.0' 'torch==2.10.0'
  python -c "import pydisort; ..."            # sibling smoke test (§02)
  export TORCH_CMAKE_PREFIX_PATH=$(python -c '...')
  cmake -B build -DCMAKE_PREFIX_PATH="$TORCH_CMAKE_PREFIX_PATH" -DCUDA=ON
  cmake --build build --parallel 3

Things to read out of this block:

  • System dep inside the container. yum install netcdf-devel from EPEL (after enabling PowerTools/CodeReady) — the CD container is RHEL-family, not Debian. The same dependency on the GitHub runner outside the container is installed via apt-get in CI. This OS-aware install matrix is exactly the cross-distro reality the brief points at.
  • The cmake<=3.30 pin. CMake 3.31+ tightened policies in ways that break some FetchContent flows; pinning is the pragmatic block on accidental upgrades inside the container. Worth revisiting deliberately.
  • PATH manipulation for the matrix Python. Unlike the macOS path (which uses the host Python directly), Linux inside the container has every cp3XX in /opt/python/...; the export PATH= line is what activates the right one before pip install and cmake.
  • Free-disk-space hammer. Linux wheels for torch+CUDA inflate fast; cd.yml nukes /usr/share/dotnet, Android SDK, Haskell, and prunes Docker before building. A senior engineer recognizes this as a runner-economics pattern, not a code-quality issue.

§14Excluding CUDA libs by soname

kintera’s wheel-repair step expands the “don’t bundle torch’s singletons” principle from pydisort into a full exclusion list of CUDA math libraries, by exact soname:

CIBW_REPAIR_WHEEL_COMMAND_LINUX: |
  export LD_LIBRARY_PATH=/usr/local/cuda/lib64:/usr/local/lib64:/usr/lib64:$LD_LIBRARY_PATH
  # Resolve CUDA runtime libraries from torch/lib at runtime instead
  # of vendoring duplicate copies into the repaired wheel.
  auditwheel repair \
    --exclude lib*.so \
    --exclude libcusolver.so.11   --exclude libcusolver.so.12 \
    --exclude libcusparse.so.11   --exclude libcusparse.so.12 \
    --exclude libcublas.so.11     --exclude libcublas.so.12 \
    --exclude libcublasLt.so.11   --exclude libcublasLt.so.12 \
    --exclude libnvJitLink.so.11  --exclude libnvJitLink.so.12 \
    --exclude libcudart.so.11.0   --exclude libcudart.so.12 \
    -w {dest_dir} {wheel}

The first line (--exclude lib*.so) is the global “don’t vendor anything by default” from pydisort’s playbook. The explicit per-soname lines are the new contribution: even though the wildcard already excludes libcusolver.so, it does not match the actual install-time soname libcusolver.so.12 (the versioned symlink target), so auditwheel would happily copy it in. The explicit lines plug that hole.

The why in one sentence: torch ships its own copies of these CUDA math libs inside torch/lib for exactly the version of CUDA torch was built against, and forcing the user’s torch and kintera to share that one set is the only way to avoid duplicate state (allocator pools, JIT caches) and version skew between the two halves of the same process.

The maintenance burden you just accepted

Every CUDA major bump — CUDA 13 will rename these (libcusolver.so.13, libcudart.so.13, etc.) — the exclude list must grow. A version of this list that derives the sonames from torch’s actual torch/lib/ contents (a shell loop over nm / readelf -d) would be self-maintaining; the cost is opacity. The hand-written list is honest about which libraries you have audited.

§15setuptools-scm: tag-driven versioning

kintera does not use bumpversion. Its version is derived from git tags by setuptools-scm:

[build-system]
requires = [..., "setuptools-scm>=8", ...]
[project]
dynamic = ["version"]                       # resolved at build time from the latest tag
[tool.setuptools_scm]
fallback_version = "0.0.0"                  # for shallow clones / no-tags edge cases

The advantages over pydisort’s __init__.py-edits-on-merge approach:

  • No bump commit clutter. The git history is reactions to actual changes; the version comes from the tag, not a parallel stream of bumps.
  • Dev builds get rich versions. Building between tags yields a version like 2.5.1.dev3+g4a7b9c1 — encoding distance from the last tag and the commit SHA. Invaluable in support: kintera.__version__ tells you exactly which commit a user ran.
  • The build-system requires it. setuptools-scm is in build-system.requires, so any build (CI, local, CD) gets the same versioning logic — no special tooling beyond pip.

The fallback_version is a real production concession: shallow clones (the common case in CI without fetch-depth: 0) lack tags and would otherwise fail. kintera’s workflows correctly set fetch-depth: 0, but the fallback keeps a half-configured CI from producing a corrupt wheel.

§16Data, YAML & the wheel payload

kintera is the first repo in the stack with non-trivial data shipping in its wheel. Reactions, kinetics, photolysis cross-sections, NASA9 thermodynamic polynomials — all live as text data the C++ code parses at runtime.

[tool.setuptools.package-data]
"kintera" = [
  "kintera/**/*",         # source tree mirror (chem tables, YAML inputs)
  "data/*.dat",            # nasa9.dat — NASA polynomial thermo data
  "lib/*.so", "lib/*.dylib",  # the native libs (the §11 pydisort strategy)
  "*.pyi", "py.typed",       # PEP 561 typing payload (1857 lines, see §20)
]

The implications for support:

  • Locating the data at runtime. Anywhere kintera reads nasa9.dat it must use importlib.resources (or torch’s equivalent) rather than relative paths or the baked-in KINTERA_ROOT_DIR from §05 — the latter is the source tree, which does not exist on the user’s machine after install. This is a real gotcha that the configure header makes easy to fall into.
  • YAML config as a release surface. Users write chapman_cycle.yaml-style reaction networks (the repo has examples). The yaml-cpp version that ingests them is a frozen release surface: bumping yaml-cpp can change anchor/alias semantics and silently alter user inputs. Pin tightly and document.
  • Data is a SemVer surface. Changing the contents of nasa9.dat (or its parsing) changes numerical results. That is at least a minor bump per §19’s label scheme — an editorial discipline that matters for reproducibility.

§18CD: tag-input, GPU pipeline

kintera’s cd.yml is named release.yml but I’ll refer to the publish workflow by its content: workflow_dispatch with a required tag input. That decision is the difference from pydisort, where the dispatch took a branch:

workflow_dispatch (tag: REQUIRED, build_os ∈ {Both, MacOS, Ubuntu}) │ concurrency: publish-pypi-, cancel-in-progress: false ← unlike pydisort ├─ build-macos matrix py3.10–3.13 → cibuildwheel → upload artifact ├─ build-linux matrix py3.10–3.13 → cibuildwheel (in pytorch image, CUDA on) └─ publish-pypi needs both; if: always() → pypa/gh-action-pypi-publish

The structural improvements over pydisort:

  • You publish a tag, not a branch. actions/checkout@v5 with: ref: ${{ github.event.inputs.tag }}. This guarantees the wheels are built from an immutable, named point in history — the foundation of reproducible release artifacts. pydisort’s branch-input approach trusts the operator not to dispatch from a moving target; kintera makes the target immovable.
  • cancel-in-progress: false. A second dispatch for the same tag doesn’t cancel the first — once you have started uploading wheels, you finish; PyPI rejects duplicate version uploads anyway, so cancelling mid-flight is worse than completing.
  • CUDA wheels on Linux. The Linux builder turns CUDA on (-DCUDA=ON), the macOS builder does not (no native CUDA on Apple Silicon). So macOS arm64 ships CPU, Linux x86_64 ships CUDA + CPU dispatch in one wheel. The user gets the right artifact from one PyPI install.
OIDC opportunity, same as pydisort

The publish job already declares permissions: id-token: write but still authenticates with PYPI_API_TOKEN. The same one-line upgrade to Trusted Publishing applies. Across three workspace repos using the same pattern, doing this once is a multiplier.

The gap the brief would surface

No pre-publish wheel smoke test. The wheels go from upload-artifact to aggregate to pypi-publish without ever being installed and imported in a clean Python in this workflow. Given the layered RPATH story (§04), this is the single highest-leverage CD improvement: pip install dist/kintera-*.whl into a clean venv with only the declared runtime deps, python -c "import kintera", smoke-test a function, and only then publish.

§19Release: label-driven SemVer + GitHub App

kintera’s release automation is the most sophisticated piece of the workspace stack. release.yml — misleadingly named, but content-accurate as “tag on PR merge” — reads SemVer intent from PR labels and authenticates as a GitHub App.

# 1) Generate a short-lived token from a GitHub App (not a PAT, not the default GITHUB_TOKEN)
- uses: actions/create-github-app-token@v2
  with:
    app-id:      ${{ vars.BUMP_BOT_APP_ID }}
    private-key: ${{ secrets.BUMP_BOT_PRIVATE_KEY }}

# 2) Read labels, classify the bump (default: patch)
LABELS_JSON: ${{ toJson(github.event.pull_request.labels) }}
BUMP_TYPE=$(python3 - <<'PY'
labels = {item["name"] for item in json.loads(os.environ["LABELS_JSON"])}
selected = [n.split(":", 1)[1] for n in ("release:major", "release:minor", "release:patch") if n in labels]
if len(selected) > 1: raise SystemExit(f"multiple release labels: {selected}")
print(selected[0] if selected else "patch")
PY)

# 3) Compute the next tag from `git describe --tags --abbrev=0`
LATEST=$(git describe --tags --abbrev=0 || echo "v0.0.0")
case "$BUMP_TYPE" in
  major) MAJOR=$((MAJOR+1)); MINOR=0; PATCH=0 ;;
  minor) MINOR=$((MINOR+1)); PATCH=0 ;;
  patch) PATCH=$((PATCH+1)) ;;
esac

# 4) Race-safe: if the tag exists remotely, bump patch and try again
while git ls-remote --tags origin "refs/tags/${NEXT_TAG}" | grep -q ...; do
  PATCH=$((PATCH+1))
done

# 5) Tag, push, and create a GitHub release with auto-generated notes
git tag "$TAG"; git push origin "$TAG"
gh release create "$TAG" --title "Release $TAG" --generate-notes

Each line earns its keep

StepWhy it’s done this way
GitHub App tokenscoped, short-lived, identifiable in audit logs as bump-bot[app]; avoids using GITHUB_TOKEN (which cannot trigger downstream workflows from a tag push) and avoids long-lived PATs
label-driven bumpSemVer intent stays attached to the PR that introduced the change; defaults to patch so no missing label is fail-closed
multi-label rejectionfail-loud if a PR carries release:major AND release:minor; the operator must decide
existing-tag short-circuitif HEAD already has a v* tag (e.g. a hotfix tagged manually), skip the bump — avoids double-tagging
remote-tag race loopif two PRs merge in quick succession, the second discovers its computed tag already exists and bumps again; idempotent
--generate-notesGitHub fills the body from PRs since the previous tag; release notes write themselves

This is one of the cleanest small examples of release engineering you will see in an open-source repo: it encodes semantic intent into a human-friendly artifact (labels), authenticates with the right primitive (an App, not a PAT), handles concurrent merges idempotently, and decouples the tag from the publish so a human still pulls the trigger in §18. The only thing missing is for tag-creation to automatically dispatch the publish workflow — the same loop-closing recommendation as pydisort.

The disciplines that survive contact with users. kintera’s investments here — a documented stub-files architecture, a real Docker dev env — map directly to the brief’s “developer productivity” and “cloud-hosted developer environments” bullets.

§20The stub-files architecture

kintera ships a deliberate separation between implementation (C++ pybind code) and API documentation (a 1,857-line kintera.pyi) — documented as a first-class architectural decision in ARCHITECTURE.md. The shape:

python/ ├── csrc/ C++ pybind11 — IMPLEMENTATION ONLY │ ├── kintera.cpp (root module, calls bind_*) │ ├── pythermo.cpp (ThermoX, EoS, NASA9) │ ├── pykinetics.cpp (reaction networks) │ ├── pyphotolysis.cpp (photochem) │ ├── pysparsesolver.cpp (cusolver/cusparse-backed) │ ├── pyconstants.cpp (physical constants) │ └── pydiffusion.cpp (diffusion ops) │ ├── kintera.pyi ← TYPE INFO + DOCSTRINGS for IDE/mypy (1857 lines) ├── py.typed ← PEP 561 marker ├── STUB_FILES.md ← contributor guide ├── __init__.py ← from .kintera import * └── atm2d/ ← pure-Python orchestration (solver, assembly, chemistry, ...)

The point of the split: pybind11 type signatures and docstrings, written in C++, are worse at every job than a Python stub — IDEs only partially read them, mypy doesn’t, refactor tooling can’t. A hand-maintained .pyi with @overloads and proper docstrings restores all of those.

The drift you must own

A 1,857-line hand-maintained .pyi describing a six-TU pybind module will drift from the actual exported signatures. The same fix recommended for pydisort applies more loudly here: run pybind11-stubgen in CI and diff against the committed stub. STUB_FILES.md mentions a validate_stub_file.py — that script should be part of the gate, not optional.

mypy.ini exists in-tree, which means the project intends to check itself. Wiring mypy into pre-commit (or CI) would close the loop — today the stubs are shipped for users but not verified for the project.

§21Docker dev env & static gates

The brief asks for “cloud-hosted developer environments” specifically. kintera is the first repo in the stack with a real one — a Docker Compose service intended to be the canonical contributor environment:

      context: .

What makes this a useful dev env, not just a Dockerfile:

  • GPU passthrough as a first-class concern. capabilities: [gpu] in deploy.resources.reservations.devices — the modern Compose-v2 way to request GPUs (the old runtime: nvidia still works but is less portable). This is the single line that makes “test the CUDA build before pushing” a local possibility, not a CI-only luxury.
  • Volumes that respect contributor identity. The host’s .gitconfig is mounted read-only so commits inside the container carry the right author. Source is bind-mounted so edits in your editor land instantly.
  • Versions baked into the image. CUDA_VER and UBUNTU_VER are build args, so the image is reproducible — you tag a Docker image alongside a kintera release if you want a frozen environment.
The gap with CI/CD

The dev image is CUDA 12.9.1 on Ubuntu 22.04; CD builds inside pytorch/manylinux2_28-builder:cuda12.8 (RHEL family). Two different toolchains with two different glibcs. “Works in Docker” does not equal “will pass CD.” A senior engineer would harmonize: either the dev image switches to the pytorch builder, or there is a second “manylinux dev” service in compose. Reproducibility is one image used by everyone, not three.

Static gates

kintera’s .pre-commit-config.yaml is lighter than pydisort’s — trailing whitespace, EOF, check-yaml, plus clang-format Google-style. The cppcheck/cpplint from pydisort’s config are absent locally but installed by CI as part of pre-commit’s side install (pip install cpplint cppcheck in ci.yml). Pulling them back into pre-commit-config.yaml would mean contributors get the same warnings locally as in CI.

§22Debugging across four libs

The pydisort guide’s native-debugging toolkit applies verbatim. What changes in kintera is the length of the chain you have to walk. The triage commands every senior engineer should have at fingertip:

# 1) The four-version sanity check (find ABI/version drift in one line)
python -c "import torch, pydisort, pyharp, kintera; print(*(m.__version__ for m in (torch, pydisort, pyharp, kintera)))"

# 2) Confirm the ABI is consistent across all native modules
for mod in torch pydisort pyharp kintera; do
  echo "== $mod =="
  python -c "import $mod, os; d=os.path.dirname($mod.__file__); print(d)" | \
    xargs -I{} sh -c 'nm -D -C {}/*.so 2>/dev/null | grep -c __cxx11'
done

# 3) What does the layered RPATH actually find?
patchelf --print-rpath $(python -c "import kintera, os; print(os.path.dirname(kintera.__file__))")/kintera*.so
ldd      $(python -c "import kintera, os; print(os.path.dirname(kintera.__file__))")/kintera*.so \
  | grep -E '(disort|harp|torch|cuda|netcdf|not found)'

# 4) Are the CUDA libs being kept on NEEDED (not stripped by --as-needed)?
readelf -d $(python -c "...")/kintera*.so | grep -E 'NEEDED.*(cusolver|cusparse|cublas|cudart|kintera_cuda)'
The three new failure modes kintera introduces

(a) Sibling-version drift: kintera 2.5.0 in a venv with pyharp 2.4.0 that imports against torch 2.11 — undefined symbol referencing a torch-internal namespace. Triage by the four-version line above. (b) System NetCDF missing: a user installs the wheel on a fresh box without libnetcdf; ldd shows libnetcdf.so.* => not found. (c) CUDA host without torch’s CUDA libs: user has a GPU but installed the CPU torch; libcusolver.so.12 => not found at first CUDA call. The exclude list (§14) is what makes this case visible — without it you’d get a duplicate-cusolver crash instead.

§23Supply chain & multi-team coordination

Three operational concerns unique to a layered stack of separately-released native packages.

The release-coordination calendar

kintera, pyharp, pydisort, and torch all release independently. A naive workflow lets each repo bump its torch pin separately, producing a window where kintera 2.5.0 wants torch 2.10.0 but pyharp 2.3.0 already wants torch 2.11.0 — an impossible install. Two operational moves close this:

  • One workspace-wide torch pin file (a shared constant or a git submodule) so all four repos bump in lockstep.
  • An integration test repo that pip-installs the four latest releases together on every nightly and runs the kintera test suite — the earliest possible discovery of a four-way drift.

§25Decision trees

Depending on a sibling native package

Same monorepo? → add_subdirectory; one build Public PyPI release? → this pattern (FindFoo.cmake importing the python module, build-system.requires, RPATH to $ORIGIN/..//lib) Internal pip index? → same as PyPI; verify the index URL in CI C++ system install? → classical find_package(Foo CONFIG); add to README install steps

Vendor vs. exclude — for CUDA math libs

torch ships its own copy in torch/lib? ├─ YES → EXCLUDE by exact soname (libcusolver.so.12, libcusparse.so.12, ...) │ resolve at runtime via the user's torch └─ NO → would need to vendor; this is the rare path For a NEW major (CUDA 13): → add libcusolver.so.13, libcudart.so.13, ... to the exclude list per §14 → bump the README's required CUDA range

SemVer label policy

PR adds public Python or C++ API (new class, new function, new keyword) → release:minor PR removes/renames anything users could be calling, OR breaks numerical contracts → release:major (and document the migration in the PR body) PR fixes a bug, refactors, internal-only changes → release:patch (default; can be omitted) PR bumps a sibling pin (pyharp, torch) → at LEAST release:minor (it changes installability constraints)

“Build fails / import fails” triage

Stage 1 CMake fails? ├─ FindDisort/FindHarp can't find header → siblings not pip-installed in this Python → §03 ├─ undefined NetCDF symbol → system dep missing → §17 └─ nvcc unsupported flag → --extended-lambda missing → §10 Stage 2 build fails? ├─ linker error in cuda symbols → library ORDER (consumer before provider) → §09 └─ undefined cuda kernel reg → --no-as-needed wrap missing → §09 Import fails on the user's machine? ├─ libnetcdf.so not found → README install step missing → §17 ├─ libcusolver.so.12 not found → user has CPU torch; need CUDA torch → §14 └─ __cxx11 symbol mismatch → ABI / version drift across pydisort/pyharp/torch → §07
