1 · What the VIC solver computes

The Jupiter model integrates the compressible Euler equations. The vertical (gravity) direction is stiff: sound waves and gravity make an explicit time step prohibitively small. snapy therefore solves the vertical dynamics implicitly — the Vertical Implicit Correction (VIC). For every vertical column of cells it forms and solves a block‑tridiagonal linear system.

The two-and-a-half jobs inside the solve

Solving one column actually involves three kinds of work, with very different parallel structure:

PhaseWhat it doesMathDependency along the column
AssemblyBuild the 3×3 blocks ai, bi, ci for every layer Roe average + eigendecomposition |A| = R|Λ|R⁻¹ at each interface none — each cell independent
SolveBlock-Thomas: forward elimination + back-substitution 3×3 inverses, a chain of mat-vec products serial recurrence (i depends on i−1)
Redistribute
(added by MS-VIC, PR #152/#153)		 Conserve column-integrated constituent tendencies column sums, then a per-cell capped_exp correction a reduction, then an independent map
The pivotal observation. Only the Thomas solve is fundamentally serial down the column. The assembly and the redistribution map are embarrassingly parallel per cell — they were just trapped inside a per-column loop.

2 · The bottleneck (before)

The original kernel assigned one GPU thread to one column. That thread ran a serial loop down the column doing everything: assemble each layer's blocks, then the Thomas sweep. Nsight Compute on the profiling grid (nx2×nx3 = 32×32 = 1024 columns) reported:

MetricValueMeaning
Launch grid8 blocks × 1281024 threads total — a near-empty GPU
Registers / thread128 (max, spilling)six live Eigen 5×5 matrices per thread
Achieved occupancy8.3 %almost no warps resident to hide latency
SM (compute) throughput2.78 %the compute units are idle 97% of the time
Duration3.38 ms / call#1 GPU-compute kernel in the run

For comparison, the WENO reconstruction kernel in the same run hits 78% SM throughput — that is what a healthy compute-bound kernel looks like. The VIC solver at 2.78% is pathological: it is not doing heavy math quickly, it is stalling.

3 · Why it was slow: occupancy & latency hiding

A GPU hides the long latency of memory and dependent arithmetic by over-subscription: it keeps many warps (groups of 32 threads) resident on each Streaming Multiprocessor (SM) and switches between them whenever one stalls. With too few resident warps, the SM simply waits. Two things starved the VIC kernel of warps:

Starvation #1 — too few threads

1 thread per column ⇒ 1024 threads ⇒ 8 blocks. A modern GPU has dozens of SMs each wanting thousands of resident threads. Most SMs had no work at all.

Starvation #2 — register pressure

Each thread held six Eigen 5×5 matrices (R, R⁻¹, Λ, |A|−, |A|+, dF/dq) ≈ 150 doubles. At 128 registers/thread the compiler spilled to local memory and capped the SM to ~4 blocks.

A failed first attempt (worth remembering). The intuitive fix — trim registers by deleting dead matrices and folding constants — was implemented and proven bit-exact… and moved occupancy by 0.0%. The six 5×5 matrices dominate the register budget; shaving the small ones changes nothing. Micro-optimization cannot fix a structural parallelism problem.

4 · The transformation (before → after)

The fix is not to make the column thread cheaper — it is to stop doing independent work inside a serial per-column loop. We split the one kernel into a pipeline of kernels, each launched at its own natural parallelism.

Passing data between the phases

The phases communicate through the workspace buffers that already exist for the tridiagonal coefficients (a, b, c, δ, all in global memory). Two tricks keep it cheap:

  • Reuse dead memory. After the forward sweep, the off-diagonal buffer b is no longer needed, so the per-column reduction sums (denom, dry_struct, species_struct[]) are stashed there — no new allocation.
  • Recompute cheap scalars instead of storing them. The redistribution kernel re-derives the per-cell quantities (Δρ·V, dry fraction) locally rather than reading them back, trading a few flops for less memory traffic. Because the arithmetic is identical, the result stays bit-for-bit the same.
// GPU pipeline (host side) — implicit_hydro.cpp
vic_assemble_partial(...);   // ① per-cell: Roe eig + build a,b,c     → 800 blocks
vic_solve_partial(...);      // ② per-column: Thomas + column reductions
vic_redistribute_partial(...);// ③ per-cell: capped_exp map → write Δu  → 800 blocks
// On CPU these collapse back into one fused call (no-op ①③) — one source, two schedules.

5 · The core concept: parallelism-aware kernel fission

Every loop nest has an iteration space (here: cells indexed by column × layer) and a dependency structure over that space. The performance mistake was letting the most serial part dictate the parallel granularity of everything. The cure is to classify each piece of work and give it the parallelism it deserves.

The principle, in one sentence: a serial recurrence should constrain only the work that truly depends on it — launch every other phase at its maximal independent parallelism.

Concretely, the assembly's (column × layer) = 1 024 × 100 ≈ 100 000 independent cells (≈ 1 000 000 at production resolution). Trapped in a per-column loop, only 1 024 ran at once; freed as a map, all of them do. That is the entire difference between 2.78% and 73% SM throughput.

6 · Generalizing the method

This is a reusable recipe — sometimes called kernel fission or loop distribution by dependency type. It applies whenever a kernel is latency-bound / low-occupancy and mixes independent work with a serial recurrence or reduction.

The recipe

  1. Profile, don't guess. Use the SpeedOfLight + Occupancy sections. The tell-tale signature is low SM % and low memory % together with low achieved occupancy — that means latency-bound, not compute- or bandwidth-bound. (A kernel already at 70–80% SM, like WENO here, is not a fission candidate.)
  2. Map the iteration space and its dependencies. For each loop dimension ask: are iterations independent (map), associatively combinable (reduce), or chained (scan/recurrence)? Draw it.
  3. Find the parallelism that was sacrificed. Look for independent work nested inside a loop that exists only because of a neighbouring serial phase. That trapped work is the prize.
  4. Fission along dependency boundaries. Split the kernel so each phase contains one dependency class. Maps and reduces become their own kernels; the irreducible recurrence is isolated and kept as small as possible.
  5. Launch each phase at its natural granularity. Maps → one thread per element (saturate the device). Reduces → tree / segmented reduction. Recurrence → one thread per independent chain (or attack it with a parallel algorithm — see below).
  6. Choose the data hand-off. Persist intermediates in global memory; reuse buffers that are dead after a phase; recompute cheap quantities instead of storing them. Mind that separate kernels serialize on the stream — that ordering is your correctness guarantee for read-after-write.
  7. Preserve semantics & verify. Keep the arithmetic identical where you can (bit-exact), and where recomputation reorders flops, bound the difference. Validate with unit tests and an end-to-end numeric comparison before trusting the speedup.
  8. Re-profile and find the new floor. Fission shifts the bottleneck; by Amdahl's law the remaining serial phase now dominates. Decide whether it is worth a deeper attack.

Attacking the irreducible part

After fission, the serial Thomas sweep is the floor. A recurrence is only apparently serial — there are parallel algorithms that trade extra work for depth:

TechniqueIdeaCost vs depth
Parallel Cyclic Reduction (PCR)repeatedly eliminate odd/even unknowns so all rows update together O(N log N) work, O(log N) depth
Recursive doubling / Hillis–Steele scancombine partial results at stride 1,2,4,…O(N log N) / O(log N)
Blelloch work-efficient scanup-sweep + down-sweep over a balanced treeO(N) / O(log N)

These were deliberately not applied here: a 3×3 block PCR is intricate and numerically delicate, and the serial sweep is no longer the run's dominant cost. Knowing where to stop is part of the method.

When fission helps — and when it doesn't

Use it when…

  • the kernel is latency-bound (low SM% & low mem%, low occupancy);
  • occupancy is capped by registers/shared memory from one heavy phase;
  • independent work is trapped inside a serial/uneven loop;
  • the launch is grid-starved (too few blocks for the device).

Be wary when…

  • the kernel is already compute- or bandwidth-bound (fission won't help);
  • the problem is tiny — extra launch overhead dominates;
  • intermediates are huge — the added global traffic outweighs the gain;
  • the phases are genuinely fused (heavy reuse of the same registers/shared tile). This is the inverse case where kernel fusion is the right move.
Fusion vs. fission are two ends of one dial. Fuse to cut launch/memory overhead and exploit reuse; fission to expose parallelism and relieve resource pressure. The skill is reading the profiler to know which way to turn it. Related levers in the same family: tiling, loop interchange, kernel specialization, and choosing the launch granularity to match the dominant parallel dimension.

Correctness

  • Bit-identical to the fused reference: a seeded 10-cycle run reproduces every printed digit of mass and energy; the result is deterministic across runs (no data race in the parallel phases).
  • Unit and integration tests pass: test_backward_substitution, test_run_hydro_cpu, test_run_hydro_cuda.
  • One source, two schedules: on the CPU the phases collapse back into the original fused routine (the extra phases are no-ops), so CPU behaviour is unchanged and the GPU split is purely a scheduling decision.
Takeaway. The 2.3× did not come from faster math — the arithmetic is unchanged. It came from scheduling the same math at the right parallel granularity: separate the embarrassingly-parallel maps from the irreducible serial recurrence, and launch each so the GPU is actually busy. That reframing — read the profiler, classify the dependency structure, fission accordingly — generalizes far beyond this solver.
