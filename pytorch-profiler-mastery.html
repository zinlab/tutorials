1 The three profiling tools compared

PyTorch Profiler vs Nsight Systems vs Nsight ComputeOverview

PyTorch Profiler

Level: Python / PyTorch operator

Question: Which PyTorch ops are slow? Where is Python spending time? What's the memory usage pattern?

Output: Chrome Trace, TensorBoard, text tables

Overhead: Low–moderate (2–10%)

Ease: ★★★★★ (no install needed)

Nsight Systems

Level: System / GPU timeline

Question: Where is the GPU idle? Is communication overlapped? What's the CPU↔GPU relationship?

Output: .nsys-rep timeline

Overhead: Low (1–5%)

Ease: ★★★☆☆ (separate install + GUI)

Nsight Compute

Level: Kernel / hardware counter

Question: Why is this specific CUDA kernel slow? Is it compute- or memory-bound? What are warps stalled on?

Output: .ncu-rep report

Overhead: Very high (10–100×)

Ease: ★★☆☆☆ (requires deep GPU knowledge)

The 80/20 rule: PyTorch Profiler answers 80% of your profiling questions in 20% of the time. It tells you which ops are slow, where memory goes, and whether the GPU is utilized — all from within your Python script, viewable in a browser. You only need Nsight Systems when you need µs-level GPU timeline accuracy (idle gap analysis, communication overlap). You only need Nsight Compute when a specific CUDA kernel is underperforming and you need to understand why at the hardware level.

2 Capture: torch.profiler

Basic profiling — your first captureCapture
# Simplest possible profile
import torch
from torch.profiler import profile, ProfilerActivity

with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    record_shapes=True,     # log tensor shapes (critical for GEMM analysis)
    profile_memory=True,     # track memory allocations
    with_stack=True,          # capture Python call stacks
) as prof:
    for step in range(5):
        train_step(model, batch)

# Print top ops by GPU time
print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=20))

# Export for Chrome trace viewer
prof.export_chrome_trace("trace.json")

# Export for TensorBoard
# (see Chapter 4 for the schedule-based approach)

activities=[CPU, CUDA]Traces CPU ops and CUDA kernels~2–5%Always — this is the minimum useful configuration
record_shapes=TrueLogs input/output tensor shapes for each op~1%Always — shapes tell you GEMM dimensions, which determines Tensor Core eligibility

Schedule-based profiling (the recommended approach)Capture

Instead of profiling every step (expensive), use a schedule that waits for warmup, profiles a few steps, then exports.

from torch.profiler import profile, schedule, tensorboard_trace_handler

# Schedule: skip 2 warmup steps, actively profile 3 steps, repeat
with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    schedule=schedule(wait=1, warmup=1, active=3, repeat=1),
    on_trace_ready=tensorboard_trace_handler("./log/profiler"),
    record_shapes=True,
    profile_memory=True,
    with_stack=True,
    with_flops=True,
) as prof:
    for step in range(10):
        train_step(model, batch)
        prof.step()  # CRITICAL — must call this every iteration

# Schedule breakdown for 7 steps:
# Step 0: wait    (no profiling, no overhead)
# Step 1: warmup  (profiling on, but data discarded — warm caches)
# Step 2: active  ← data collected
# Step 3: active  ← data collected
# Step 4: active  ← data collected + trace exported
# Steps 5+: done (repeat=1 means don't repeat)

Why the schedule matters

The wait phase lets any startup transients settle. The warmup phase warms CUDA caches and the PyTorch caching allocator so the profiled steps are representative. The active phase captures the actual data. Without this schedule, you'd profile the first step (which includes JIT compilation, memory allocation, cuDNN autotuning) — none of which is representative of steady-state performance.

Profiling with NVTX markers (bridging to Nsight)Capture

PyTorch's profiler can emit NVTX annotations, which makes the trace visible in Nsight Systems too. This is the bridge between the two tools.

# Method 1: emit_nvtx() — auto-annotate all autograd ops
with torch.autograd.profiler.emit_nvtx():
    output = model(input)
    loss = criterion(output, target)
    loss.backward()
# Every aten op becomes an NVTX range visible in Nsight Systems

# Method 2: manual NVTX markers (more control)
import torch.cuda.nvtx as nvtx

nvtx.range_push("forward")
output = model(input)
nvtx.range_pop()

nvtx.range_push("loss")
loss = criterion(output, target)
nvtx.range_pop()

nvtx.range_push("backward")
loss.backward()
nvtx.range_pop()

# Method 3: record_function (works with both PyTorch Profiler and NVTX)
with torch.profiler.record_function("my_custom_region"):
    # This appears in both PyTorch Profiler AND Nsight Systems
    output = model(input)
record_function is the universal annotation. It works with PyTorch Profiler (appears in Chrome Trace and TensorBoard), and also emits NVTX ranges that Nsight Systems can see. Use it to label the phases of your training loop — "data_load", "forward", "backward", "optimizer" — regardless of which profiler you'll ultimately use.

Chrome Trace limitation: The chrome://tracing viewer struggles with large traces (>100 MB). For big models or many steps, the file can be 500 MB+ and the viewer becomes unusable. Solutions: (1) profile fewer steps (use the schedule), (2) use TensorBoard plugin instead (handles large traces better), or (3) use Perfetto which handles large traces much better than Chrome Tracing.

Example breakdown — the 22% "other" is where performance is being lost

The Overview's "GPU utilization" number is the most important single metric. If it's above 80%, the GPU is well-utilized and you're looking at kernel-level or communication optimizations. If it's below 60%, there's a systemic issue — data loading, sync stalls, or Python overhead — that the other views will help diagnose.

How to read the columns

ColumnWhat it meansHow to use it
NamePyTorch operator name (aten namespace)Maps directly to model operations: aten::mm → matrix multiply, aten::linear → nn.Linear, aten::_softmax → softmax
Self CPU / Self CUDATime spent in this op excluding child opsThe "true" cost of this specific operation. Use self time to find the actual bottleneck, not the parent op that merely called it.

# Group by module (layer) instead of operator
print(prof.key_averages(group_by_input_shape=True).table(
    sort_by="cuda_time_total", row_limit=20
))
# This groups identical ops with different shapes separately
# — essential for finding which GEMM configurations are slow

# Group by nn.Module (requires with_modules=True)
print(prof.key_averages(
    group_by_stack_n=5  # group by top 5 stack frames
).table(sort_by="cuda_time_total", row_limit=20))
# Shows time per source code location — traces ops back to your code

The shapes tell you everything about Tensor Core utilization

When you see aten::mm with shape [256, 4096] × [4096, 4096], the M=256 dimension (batch×seq) and K=4096, N=4096 are all multiples of 8 → Tensor Cores eligible. If you see [256, 4096] × [4096, 30001], the N=30001 (vocab size) is not a multiple of 8 → Tensor Core fallback. This is exactly the kind of insight that's trivial in PyTorch Profiler but requires Nsight Compute kernel name decoding to get from Nsight.

Tensor Core utilization detectionInterpret

The TensorBoard GPU kernel view shows a "Tensor Core" column indicating whether each CUDA kernel used Tensor Cores. You can also detect this programmatically:

# Check Tensor Core usage in kernel names
for event in prof.key_averages():
    if event.device_type == torch.autograd.DeviceType.CUDA:
        name = event.key
        # Tensor Core GEMM indicators in kernel names:
        is_tc = any(marker in name.lower() for marker in [
            'h884', 'h1688', '16816', 's16816',  # Tensor Core GEMMs
            'mma', 'hmma', 'i8816',               # MMA instructions
            'bf16', 'fp16', 'tf32',                # reduced precision
        ])
        if 'gemm' in name.lower() or 'mm' in event.key:
            print(f"{name}: {'TC ✓' if is_tc else 'CUDA cores ✗'}"
                  f" time={event.cuda_time_total/1000:.1f}ms")
This is one area where Nsight Compute is strictly better. PyTorch Profiler can hint at Tensor Core usage via kernel names, but Nsight Compute directly measures Tensor Core pipe utilization as a percentage. If you need to confirm TC usage definitively (not just heuristically), use Nsight Compute on the specific GEMM kernel.

6 Memory profiling

Memory timeline and allocation trackingMemory

This is PyTorch Profiler's strongest advantage over Nsight — it understands the PyTorch caching allocator, not just raw CUDA memory.

Three levels of memory profiling

LevelToolWhat it shows
Quick snapshottorch.cuda.memory_summary()Current state: active allocations, cached blocks, fragmentation. No profiler needed.
Timelineprofile(profile_memory=True) + TensorBoard Memory viewMemory usage over time. Shows allocation/deallocation events correlated with ops.
Deep snapshottorch.cuda.memory._record_memory_history()Full allocation stack traces. Can visualize at pytorch.org/memory_viz.
# Level 1: Quick snapshot (run anytime, no profiler)
print(torch.cuda.memory_summary(device=0))
# Shows:
# |             CUDA memory |  Allocated  |   Cached   |
# | Peak usage:            |    42.3 GiB |   48.0 GiB |
# | Current usage:         |    28.1 GiB |   48.0 GiB |
# | Num alloc retries:     |           5 |            |  ← fragmentation!
# | Num OOM:               |           0 |            |

# Level 2: Timeline (in profiler)
with profile(activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
             profile_memory=True) as prof:
    train_step(model, batch)
# View in TensorBoard → Memory view

# Level 3: Deep snapshot with allocation stacks
torch.cuda.memory._record_memory_history(max_entries=100000)
train_step(model, batch)
snapshot = torch.cuda.memory._snapshot()
torch.cuda.memory._record_memory_history(enabled=None)  # stop recording

# Save and visualize
import pickle
with open("memory_snapshot.pkl", "wb") as f:
    pickle.dump(snapshot, f)
# Upload to https://pytorch.org/memory_viz to see interactive visualization
# Shows: every allocation with its Python stack trace, lifetime, and size

What the memory visualizer reveals that Nsight cannot

Nsight Systems sees cudaMalloc and cudaFree — the raw CUDA calls. But PyTorch's caching allocator intercepts these. Most PyTorch allocations reuse cached blocks and never call cudaMalloc. The PyTorch memory snapshot shows you: (1) which tensors are alive and why (full stack trace), (2) fragmentation within the caching allocator (blocks that are cached but unusable), (3) peak memory vs current memory, (4) allocation retry count (how many times the allocator had to consolidate blocks). This level of detail is simply invisible to Nsight.

8 Profiling torch.compile

Profiling compiled models and Triton kernelsCompile

When torch.compile is active, the profiler shows fused Triton kernels instead of individual PyTorch ops. This changes how you interpret the output.

# Profile compiled model
model_compiled = torch.compile(model, mode="max-autotune")

# First run triggers compilation (slow — don't profile this)
model_compiled(warmup_input)

# Profile steady-state
with profile(
    activities=[ProfilerActivity.CPU, ProfilerActivity.CUDA],
    record_shapes=True,
    with_stack=True,
) as prof:
    for _ in range(3):
        output = model_compiled(input)
        loss = criterion(output, target)
        loss.backward()

print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=20))

What changes in the profiler output

Profile AFTER compilation, not during

The first forward pass through a compiled model triggers JIT compilation, which can take 30–120 seconds and dominates the profile. Always run 1–2 warmup iterations before starting the profiler. The schedule(wait=1, warmup=1, ...) pattern handles this if the first iteration triggers compilation.

9 Pattern diagnosis

Six common problems diagnosed with PyTorch ProfilerDiagnosis

1. CPU bottleneck (data loading, preprocessing)

What you see in PyTorch Profiler

TensorBoard Overview shows high "Other" or "CPU compute" percentage (>30%). The Trace view shows long gaps on the CUDA row while the CPU row is busy. Operator table shows high CPU time for DataLoader operations or custom preprocessing functions.

What Nsight Systems would show additionally

The OS Runtime trace would reveal which syscalls are slow (read(), futex(), poll()), distinguishing disk I/O from CPU compute from thread contention. PyTorch Profiler doesn't trace OS-level calls.

Verdict: PyTorch Profiler is sufficient

PyTorch Profiler identifies the problem (CPU bottleneck). The fix (more workers, GPU augmentation, DALI) doesn't require Nsight-level detail to determine. Use Nsight Systems only if you need to distinguish between disk I/O, CPU compute, and thread contention as the specific CPU sub-bottleneck.

2. Kernel launch overhead (too many small ops)

What you see in PyTorch Profiler

Operator table shows hundreds of ops with <1ms CUDA time each, but thousands of calls. Total GPU time is dominated by many tiny ops rather than a few large ones. The GPU kernel view shows many kernels under 10µs.

What Nsight Systems would show additionally

The CUDA API row would be a solid wall of cudaLaunchKernel calls. Power draw trace would show low power despite high time-occupancy. The gap analysis would quantify exactly how much time is lost to inter-kernel gaps.

Verdict: PyTorch Profiler is sufficient for diagnosis, Nsight for quantification

PyTorch Profiler identifies the pattern (many small ops). The fix is torch.compile. If you want to precisely quantify the launch overhead cost (e.g., "launch gaps account for 35% of iteration time"), Nsight Systems is more accurate.

3. Synchronization stalls (.item(), .cpu())

What you see in PyTorch Profiler

Operator table shows aten::item or aten::_local_scalar_dense with very high CUDA time (these ops force a GPU→CPU sync). With with_stack=True, you can see exactly which line of Python code triggered the sync.

What Nsight Systems would show additionally

The CUDA API row would show cudaStreamSynchronize or cudaMemcpyAsync D2H. The GPU kernel row would show a gap during the sync. The visual pattern is more dramatic in Nsight but the information is the same.

Verdict: PyTorch Profiler is better here

PyTorch Profiler with with_stack=True tells you the exact Python line that called .item(). Nsight Systems only shows the CUDA API call — you'd have to manually trace it back to Python code. This is a case where PyTorch Profiler is strictly more useful.

4. Memory issues (OOM, fragmentation, leaks)

What you see in PyTorch Profiler

Memory view shows peak usage, allocation timeline, and per-op memory consumption. torch.cuda.memory_summary() shows "Num alloc retries" (fragmentation) and "Num OOM" events. The deep snapshot at pytorch.org/memory_viz shows every live tensor with its allocation stack.

What Nsight Systems would show additionally

Nsight sees cudaMalloc/cudaFree calls that bypass the caching allocator — useful for detecting when the allocator can't serve a request and falls back to CUDA. But it doesn't understand the allocator's internal state (cached blocks, fragmentation).

Verdict: PyTorch Profiler is strictly better for memory

For PyTorch memory issues, always use PyTorch's built-in memory tools. They understand the caching allocator, tensor lifetimes, and Python stack traces. Nsight only sees the raw CUDA allocations, which are a tiny fraction of PyTorch's actual allocation activity.

5. No Tensor Core utilization

What you see in PyTorch Profiler

TensorBoard GPU kernel view has a "TC eligible" column. Operator table with record_shapes=True shows GEMM dimensions — you can check alignment manually. Kernel names may contain sgemm (FP32, no TC) vs h884gemm (TC).

What Nsight Compute would show additionally

Direct measurement of Tensor Core pipe utilization as a percentage. Roofline analysis showing where the kernel sits relative to the TC ceiling. Instruction mix showing FMA vs Tensor Core pipe usage. This is definitive — no heuristic guessing.

Verdict: PyTorch Profiler for detection, Nsight Compute for confirmation

PyTorch Profiler is usually sufficient: check shapes for alignment (÷8), check kernel names, verify autocast is active. Use Nsight Compute only when you're sure autocast is on, shapes are aligned, but you suspect the GEMM still isn't hitting TCs (e.g., a custom kernel or unusual cuBLAS path).

6. Communication bottleneck

What you see in PyTorch Profiler

TensorBoard Overview shows high "Communication" percentage (>20%). Operator table shows nccl:allreduce or nccl:all_gather with significant CUDA time. Trace view shows NCCL kernels serialized after backward, not overlapped.

What Nsight Systems would show additionally

Cross-GPU timeline: see all 8 GPUs simultaneously, identify stragglers, measure exact overlap of NCCL kernels with compute. NCCL kernel names reveal algorithm selection (Ring vs Tree). Power trace correlation across GPUs shows load imbalance. The multi-GPU view is Nsight's killer feature for distributed profiling.

Verdict: PyTorch Profiler for detection, Nsight Systems for diagnosis

PyTorch Profiler tells you "communication is 20% of step time." Nsight Systems tells you "NCCL AllReduce on GPU 3 starts 15ms after GPU 0 because GPU 3's backward is slower due to thermal throttling, and the Ring algorithm is suboptimal for this message size — Tree would be 30% faster." If you just need to know whether communication is a problem, PyTorch Profiler is fine. If you need to fix it, Nsight Systems is required.

"I'm on a restricted cloud VM and can't install NVIDIA tools"PyTorch ProfilerBuilt into PyTorch — no additional installation, no root access needed.
"I need to compare two training configurations rigorously"Nsight Systems + Nsight ComputeNsight's A/B diff mode and deterministic counter collection give more rigorous comparisons than PyTorch Profiler's statistical traces.
The recommended workflow: (1) Start with PyTorch Profiler — it answers most questions, requires no install, and gives you Python-level insight. (2) If you see GPU idle gaps or communication issues, switch to Nsight Systems for µs-precision timeline analysis. (3) If you find a specific kernel that's underperforming, use Nsight Compute for hardware counter analysis. Most investigations stop at step 1 or 2.

The profiling mental model: Think of the three tools as a zoom lens. PyTorch Profiler is the wide-angle view — it shows the full picture at the Python/operator level, answers "what's slow?" and "where's the memory going?" with zero setup friction. Nsight Systems is the standard zoom — it shows the system-level timeline at µs precision, answers "why is the GPU idle?" and "is communication overlapped?" when PyTorch Profiler isn't detailed enough. Nsight Compute is the microscope — it shows a single kernel's hardware utilization, answers "why is this kernel at 50% of peak?" when you've already identified the bottleneck kernel. Start wide, zoom in only as needed. Most problems are visible at the PyTorch Profiler level.
