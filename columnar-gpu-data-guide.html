In-memory · On-disk · On-device

How data is laid out, and why the GPU stack is built the way it is.

One idea — store columns together, not rows — runs through almost everything here: Arrow, Parquet, cuDF, and the way a GPU actually reads memory. This guide builds it up from first principles, then walks the whole tooling chain on top.

01 · The foundation

Row-major vs column-major

Memory is one-dimensional. A CPU or GPU addresses bytes in a flat line, 0, 1, 2, …. Any structure with more than one dimension — a matrix, a table — has to be flattened onto that line, and there is more than one way to do it. The choice is invisible to your code but decisive for performance.

The 2-D array case (strides)

Take a 3×3 matrix. Row-major order (used by C, C++, and NumPy by default) walks across each row before dropping to the next. Column-major order (Fortran, MATLAB, R, BLAS/LAPACK conventions) walks down each column first.

The mechanism that makes both work without copying is the stride: the number of bytes you step to advance one index along a given axis. Change the strides and the same buffer is reinterpreted — that is how a transpose can be free (just swap the strides) and why arr.T in NumPy returns a view, not a copy.

A crucial distinction: "column-major" vs "columnar storage"

These sound identical and are constantly confused. They are related but not the same thing:

Column-major (strides)

A property of a single, homogeneously-typed N-D array (all floats, say). It is about the order of axes in one buffer. Fortran/BLAS world.

Columnar storage (Arrow/Parquet)

A property of a table with many differently-typed columns. Each column gets its own contiguous buffer. The database/analytics world.

Arrow and Parquet are "columnar" in the second sense: a table of id (int), name (string), price (float) is stored as three independent buffers, one per column, rather than as records where each row's three fields sit together. The intuition is the same — keep things you process together adjacent in memory — but applied to tables instead of matrices.

Why the layout matters so much

  • Cache locality. CPUs fetch memory in cache lines (~64 bytes), not single values. If your loop touches adjacent addresses, every fetched line is fully used. Summing one column: columnar gives you only the values you need, packed tight; row storage drags in every other field of every row and throws most of it away.
  • SIMD / vectorization. A single instruction can add 8 or 16 contiguous values at once. That only works when same-typed values are adjacent — exactly what a column buffer provides. Row storage interleaves types and breaks vectorization.
  • Compression. Values in one column are statistically similar (same type, similar magnitudes, repeats). Packing them together makes run-length, dictionary, and delta encoding dramatically more effective. Mixed-type rows compress poorly.
  • Column pruning & predicate pushdown. An analytical query like SELECT AVG(price) WHERE region='US' only needs two columns. Columnar lets you read just those, and skip whole blocks using per-block min/max stats. Row storage forces you to read everything.
  • GPU coalesced access (the punchline for later). When 32 GPU threads in a warp each read one element of a contiguous buffer, the hardware merges those into a single wide memory transaction. Columnar layout is what makes that coalescing possible — it is the reason cuDF is columnar.
The trade-off Columnar is a win for analytics (scan many rows, few columns, aggregate). Row-major wins for transactional work (read/update one entire record at a time — an OLTP database, a single user row). Neither is "better"; they optimize opposite access patterns.
02 · In-memory standard

Apache Arrow

Arrow is a language-independent specification for how columnar data lives in memory. That framing matters: it is not primarily a file format and not a library you "run." It is a precise, byte-level agreement about how a column should be laid out in RAM, so that any system — Python, C++, Java, Rust, a GPU library — that speaks Arrow can hand a buffer to any other system without serializing, copying, or converting it.

in-memory columnar zero-copy language-agnostic little-endian canonical 64-byte aligned

Before Arrow, moving a dataframe from, say, a JVM system to Python meant serializing to some intermediate form and rebuilding it on the other side — often the dominant cost in a pipeline. Arrow's bet: if everyone agrees on the exact memory layout, that whole step disappears.

Because nulls live in a separate bitmap and values are packed densely, the value buffer is pure, same-typed data — ideal for SIMD and for GPU kernels. Buffers are padded to 64-byte boundaries so vector instructions never straddle the end of an allocation.

The logical structure

  • Array / ChunkedArray — one column. A ChunkedArray is several Arrays glued logically end-to-end (lets you append without reallocating).
  • RecordBatch — a set of equal-length Arrays plus a Schema; i.e. a contiguous slab of a table.
  • Table — a Schema plus one ChunkedArray per column; a RecordBatch that may be chunked.
  • Schema — the field names and types; Arrow is self-describing.

How Arrow is actually moved around

Arrow defines the in-memory shape, then bolts on standard ways to share it:

Compute on Arrow

Several engines operate directly on Arrow buffers: Acero (Arrow's C++ streaming execution engine), Gandiva (compiles SQL-like expressions to native code via LLVM), and DataFusion (a Rust query engine built on Arrow). The point is that once data is in the Arrow layout, many tools can compute on it in place.

One-line mental model Arrow is the shared language for columnar data in memory. Parquet (next) is how that data rests on disk. They are partners, not competitors.
03 · On-disk standard

Apache Parquet

Parquet is a columnar file format for storage. Where Arrow optimizes for fast access and computation in RAM, Parquet optimizes for the opposite constraints: minimal bytes on disk and minimal bytes read during a query. It is heavily encoded and compressed — wonderful for storage, but you must decode it before you can compute, which is exactly why it pairs with Arrow.

File structure

Parquet file — nested containers, metadata at the end
Fileone logical table
Row Groupa horizontal slice of rows (~128 MB) — the unit of parallelism
Column Chunkone column's data within that row group — contiguous on disk
Pagesmallest unit (~1 MB): encoded + compressed, with min/max stats
Footer (FileMetaData)schema · row-group locations · per-chunk statistics — read first

A reader opens the footer first to learn the schema and where each column chunk lives, then seeks directly to only the chunks it needs. Two superpowers fall out of this:

  • Column pruning (projection pushdown) — need 3 of 200 columns? Read 3 column chunks, ignore the rest entirely.
  • Predicate pushdown — each chunk/page stores min & max. Filtering price > 1000? Any block whose max is 900 is skipped without being read or decompressed.

Encoding & compression (the layered savings)

Parquet first encodes (changes representation losslessly), then compresses (general-purpose codec on top):

TechniqueIdeaGreat for
DictionaryReplace repeated values with small integer codes into a per-chunk dictionaryLow-cardinality columns (country, category)
RLE + bit-packingStore "value × run length"; pack small ints into minimal bitsLong runs, dictionary codes
DeltaStore differences between consecutive valuesSorted / monotonic data (timestamps, IDs)
Codec (Snappy / ZSTD / GZIP / LZ4)General compression over the encoded bytesEverything — Snappy for speed, ZSTD for ratio

Nested and repeated fields (structs, lists) are handled with Google's Dremel encoding — repetition and definition levels — which flattens arbitrary nesting into flat columns while remembering exactly how to rebuild the structure.

04 · GPU dataframes

cuDF & how it deploys on CUDA

cuDF is a GPU DataFrame library — part of NVIDIA's RAPIDS suite — that mirrors the pandas API but runs operations on the GPU. You write df.groupby('k').sum(); cuDF executes it across thousands of GPU threads. The Python layer is thin; the real work lives in libcudf, a C++/CUDA library.

Why cuDF is columnar (the whole point)

cuDF stores each column as a contiguous device buffer in an Arrow-compatible memory layout — values buffer plus validity bitmap, exactly like Section 02. This is not an aesthetic choice; it is what makes the GPU fast. Here's the chain of reasoning:

  1. A GPU runs threads in groups of 32 called a warp, all executing the same instruction (SIMT — single instruction, multiple threads).
  2. When the 32 threads of a warp read 32 adjacent memory addresses, the hardware fuses them into one wide transaction. This is memory coalescing, and it is the difference between using and wasting the GPU's enormous memory bandwidth.
  3. A columnar buffer is 32 adjacent same-typed values. Map thread i to element i of the column and coalescing is automatic.
  4. A row-major table would scatter a column's values every row-width bytes — strided, uncoalesced, slow.

So columnar layout is the precondition for the GPU being good at this at all. Arrow just happens to be the agreed-upon columnar layout, which gives cuDF interoperability for free.

How an operation becomes a kernel

# Python — looks like pandas
df['c'] = df['a'] + df['b']

# What libcudf does, conceptually:
# launch one thread per row over the contiguous a,b buffers
__global__ void add(const float* a, const float* b, float* c, int n){
    int i = blockIdx.x * blockDim.x + threadIdx.x;   // global index
    if (i < n) c[i] = a[i] + b[i];                   // coalesced read+write
}

Elementwise ops map to one thread per element. More complex operations decompose into well-studied parallel patterns:

DataFrame opParallel primitive(s)
filter / dropnapredicate → prefix-sum (scan) → compaction (gather)
sort / sort_valuesparallel radix / merge sort
groupby + agghash-based aggregation, or sort + segmented reduce
join / mergebuild a GPU hash table, probe in parallel
sum / mean / mintree reduction across the buffer

cuDF does not hand-write all of these — it leans on battle-tested primitive libraries (Section 05) so its kernels are correct and near-optimal.

Arrow interop on the GPU

Because cuDF columns already match Arrow's layout, handing data to CuPy, PyTorch, or another RAPIDS library is a pointer hand-off, not a copy. It happens through the Arrow C Data Interface (extended for CUDA device pointers), the __cuda_array_interface__ protocol, or DLPack (Section 09). Reading a Parquet file with cudf.read_parquet can even pull bytes from storage directly into VRAM (via GPUDirect Storage) and decompress on the GPU.

05 · The supporting cast

Thrust, CUB, RMM & friends

cuDF rarely writes raw kernels for hard parallel patterns. It composes them from a layered set of NVIDIA libraries, each operating at a different level of abstraction. Understanding this stack demystifies "how does a sort actually run on a GPU."

The RAPIDS / CUDA execution stack (top = highest level)
cuDF (Python)pandas-like API you call
libcudf (C++/CUDA)dataframe algorithms: join, groupby, I/O
Thrusthigh-level parallel algorithms (STL-style)
CUBlow-level warp/block/device primitives
RMMdevice memory pool allocator
CUDA Runtime + GPU hardwarekernels, warps, global/shared memory

Thrust — the parallel STL

Thrust is a C++ template library that gives you STL-like algorithms — sort, reduce, transform, scan (prefix sum), reduce_by_key, gather/scatter — that run on the GPU. You express what ("sort this buffer", "sum these"), and Thrust handles thread configuration and the parallel algorithm. It is backend-agnostic (can target CUDA, OpenMP, or TBB) and ships with the CUDA Toolkit. libcudf uses it for things like sorts and segmented reductions inside groupby.

CUB — the cooperative primitives

CUB sits one level below. It provides reusable building blocks tuned to the GPU's execution hierarchy: WarpReduce and BlockScan (cooperation within 32 threads or a thread block) up to DeviceRadixSort and DeviceReduce (whole-device operations). Crucially, Thrust's CUDA backend is itself implemented on top of CUB — when you call a Thrust sort, a CUB device sort usually does the heavy lifting. libcudf also calls CUB directly when it needs that finer control.

Thrust

What, declaratively. Convenient, composable, productive. "Sort this array."

CUB

How, with control. Tunable, fused into custom kernels, maximal performance. "Scan within this block."

RMM — RAPIDS Memory Manager

Calling cudaMalloc / cudaFree for every intermediate is expensive — each one synchronizes the device. RMM grabs one large slab of VRAM up front and sub-allocates from a pool, so the thousands of temporary buffers a query needs are nearly free. Every RAPIDS library shares the same RMM resource, which is part of why they interoperate so smoothly.

06 · Distributed computing

Ray & Arrow — and the format change

Ray is a framework for distributed Python: it scatters tasks and objects across many processes and machines. cuDF and Ray both leaned on Arrow, but for different reasons, and Ray's relationship with Arrow has a genuinely instructive history — including a real format change, though a more nuanced one than "Ray dropped Arrow."

Why Ray chose Arrow in the first place

Ray's central problem is moving Python objects between processes that don't share memory. Plain pickle fails several of Ray's needs at once. Around 2017 the Ray team built their serialization layer on Arrow because Arrow gave them, in one package:

  • Zero-copy reads — Arrow buffers can sit in shared memory and be read by many worker processes on a node without copying. For large NumPy arrays this made deserialization essentially free (the win was eliminating data movement, not faster parsing).
  • A columnar layout efficient for numerical data — NumPy arrays and dataframes are exactly what ML workloads pass around.
  • Language independence — a path toward Python and Java workers sharing objects.

This is the same property cuDF exploits — Arrow buffers are "just memory" with an agreed layout — but aimed at shared-memory IPC rather than GPU kernels. That common need is why both projects adopted Arrow: a stable, zero-copy, language-neutral memory format is exactly what you want whether you're shipping a buffer to another process or to another library on a GPU.

The Plasma story

Ray also needed a shared-memory object store — a place to park objects on a node so any worker can grab them. The Ray team built one, called Plasma, and contributed it into Apache Arrow (2017). For a while, Plasma was part of Arrow.

It didn't stay that way. Before Ray 1.0, the Ray team forked Plasma back into Ray's own codebase, so they could evolve it for Ray's architecture and performance needs without the constraints of being an Arrow subcomponent. With its developer-maintainers gone, the Arrow copy of Plasma went unmaintained, was deprecated in Arrow 10.0.0 (2022) and removed around Arrow 12.0.0 (2023). Today the object store is fully Ray's own; Plasma-in-Arrow is history.

The actual format change: away from the Arrow serializer

Separately from the object store, Ray replaced the Arrow-based object serializer with a customized Pickle protocol 5 backport plus cloudpickle. The motivation was breadth, not speed: the Arrow serializer couldn't handle the messy variety of real Python objects Ray must ship — recursive objects, lambdas, nested functions, dynamically created classes. Pickle 5 introduced out-of-band buffers, which let Ray keep the prize feature — zero-copy reads for NumPy arrays — while cloudpickle covers everything else.

The nuance that's usually missed Ray did not abandon Arrow. It swapped the general-purpose serializer for pickle5 + cloudpickle (for flexibility) and brought the object store in-house (for control). But Arrow is still central where it fits: Ray Data uses Arrow as its in-memory block format, and the zero-copy NumPy path is still Arrow-flavored shared memory. The change was "Arrow where it's the right tool," not "Arrow out."

Why this was a net win for Ray

  • Coverage — pickle5 + cloudpickle serialize almost any Python object; the old path choked on common cases.
  • Kept the fast path — out-of-band buffers preserved zero-copy for the large numerical arrays that dominate ML payloads.
  • Independence — owning the object store let Ray build features Arrow's Plasma never would: object spilling to disk, distributed reference counting, and an ownership model for fault tolerance.
07 · CPU array memory

NumPy memory management

A NumPy ndarray is a thin Python object wrapped around one contiguous block of raw bytes. The object holds metadata; the bytes hold values. Almost everything clever about NumPy comes from separating those two.

Anatomy of an ndarray
data pointerstart of the raw byte buffer
dtypehow to interpret each element (e.g. float64)
shapesize along each dimension
stridesbytes to step to advance one index per axis
flags · baseC/F-contiguous, owns-data; base = object whose memory a view borrows

Strides make views free

Because indexing is "data pointer + sum of index×stride," NumPy can describe many views over the same buffer just by changing metadata:

  • Slicing (a[2:8]) returns a view — adjusts the pointer and shape, copies nothing.
  • Transpose (a.T) returns a view — swaps the strides; the bytes never move.
  • Reshape returns a view when the layout permits, otherwise copies.
  • Broadcasting uses a stride of 0 on a dimension, so one row appears repeated without duplicating memory.

The flip side: fancy indexing (a[[0,3,7]]) and boolean masks copy, because the result can't be expressed with regular strides.

Ownership, lifetime, and sharing

A base array owns its buffer; a view's base attribute points to whoever owns the memory, and Python's reference counting keeps that buffer alive as long as any view references it. NumPy exposes its memory to the outside world through the buffer protocol (PEP 3118) and __array_interface__, which is precisely how PyArrow, PyTorch, and others can wrap a NumPy buffer with zero copying.

Default order NumPy is row-major (C order) by default but supports column-major (order='F'). A C-contiguous array handed to a Fortran library (BLAS/LAPACK) may be silently copied/transposed — a common hidden cost in numeric code.
08 · GPU tensor memory

PyTorch tensor memory

A PyTorch tensor uses the same buffer-plus-metadata idea as NumPy, but adds device awareness (CPU or CUDA), autograd, and a custom GPU allocator. Structurally a tensor is a view onto a Storage — a flat, contiguous, untyped 1-D buffer.

Tensor = a view (metadata) onto a shared Storage (bytes)
Storage1-D contiguous buffer of raw bytes on a specific device
sizes · strides · storage_offsethow this tensor reads the storage
dtype · device · layoutfloat32 · cuda:0 · strided
requires_grad · grad_fnautograd bookkeeping for backprop

Views, contiguity, and the usual gotchas

  • .view() reinterprets the existing storage cheaply, but requires a compatible (often contiguous) layout, or it errors.
  • .reshape() returns a view if it can, otherwise copies — the forgiving cousin of view.
  • .transpose() / .permute() return non-contiguous views (strides reordered).
  • .contiguous() materializes a fresh, tightly-packed copy — needed before some kernels and before view.

The CUDA caching allocator — the big one

Raw cudaMalloc/cudaFree are slow and synchronize the GPU; a training loop allocating thousands of tensors per step couldn't afford them. PyTorch interposes a caching allocator: it requests large segments from CUDA and recycles freed blocks internally, so most tensor allocations never touch the driver.

  • Reserved memory = total the allocator holds from CUDA. Allocated = the slice currently in live tensors. Reserved − allocated is the cache sitting ready for reuse.
  • This is why nvidia-smi shows PyTorch "using" more than your tensors — that's reserved cache, not a leak.
  • torch.cuda.empty_cache() returns cached-but-unused segments to CUDA (mainly to let other processes use the GPU; it doesn't speed up your own run).
  • Fragmentation — many odd-sized allocations can leave reserved memory unusable for a big new tensor, causing OOM despite "free" memory; expandable_segments helps.

Two memory features worth knowing

  • Pinned (page-locked) memory — tensor.pin_memory() on the CPU side lets host→device copies run asynchronously and faster (the GPU can DMA directly). Standard in DataLoader(pin_memory=True).
  • Zero-copy interop — PyTorch CUDA tensors expose __cuda_array_interface__ and DLPack, so a tensor can become a cuDF column or CuPy array (and back) with no copy — they all describe the same VRAM buffer.
NumPy vs PyTorch, in one line Same strided-view-over-a-buffer design. PyTorch adds: a device, a caching allocator for GPU memory, and an autograd graph that itself consumes memory (every saved activation for the backward pass).
09 · Crossing boundaries

The interchange glue

Everything above keeps returning to one move: hand a buffer from one library to another without copying. That only works because of a handful of shared protocols — the unsung standards that let NumPy, PyTorch, cuDF, CuPy, Arrow, and JAX cooperate. They all do the same job: describe an existing buffer (pointer, dtype, shape, strides, device) so the receiver can wrap it in place.

ProtocolDomainUsed by
Buffer protocol (PEP 3118)CPU, Python-levelNumPy, memoryview, PyArrow
__array_interface__CPU arraysNumPy and friends
__cuda_array_interface__GPU arrays (CUDA pointers)Numba, CuPy, cuDF, PyTorch
DLPackCPU + GPU tensors (cross-framework)PyTorch, CuPy, cuDF, JAX, TensorFlow
Arrow C Data InterfaceColumnar tables (CPU + CUDA)Arrow ecosystem, cuDF, Polars, DuckDB

DLPack is the closest thing to a universal tensor lingua franca — a tiny C struct (DLManagedTensor) that any framework can produce or consume. It's how you do torch.from_dlpack(cupy_array) and have two libraries share the exact same VRAM with no transfer.

The unifying insight Arrow's C Data Interface, DLPack, and the array-interface protocols are all answers to the same question: "how do I describe a buffer precisely enough that you can use it without me copying it to you?" Standardizing the answer is what turns a pile of separate libraries into one composable ecosystem.
10 · Putting it together

The whole map

Read top to bottom, this is the life of analytical data on modern hardware — and every layer is the same idea (keep same-typed values contiguous so machines can stream and vectorize them):

Where each piece lives
Parqueton disk — columnar, encoded, compressed; predicate pushdown & column pruning
Arrowin memory — columnar, decoded, zero-copy interchange standard
cuDF columns / NumPy / Torch storagethe working buffers — contiguous, strided, device-aware
Thrust · CUB · libcudf · BLASthe algorithms that run over those buffers
RMM · CUDA caching allocatorwho hands out and recycles the memory

One mental model to keep

  1. Hardware reads memory in contiguous chunks (cache lines, coalesced warps). Layout is destiny.
  2. Columnar = same-typed values adjacent → great compression, SIMD, and GPU coalescing → the right shape for analytics.
  3. Parquet stores it compactly on disk; Arrow holds it ready-to-compute in memory; they convert cleanly into each other.
  4. cuDF puts Arrow-shaped columns in VRAM so each operation becomes a coalesced CUDA kernel, built from Thrust/CUB primitives and fed by RMM.
  5. Ray wanted the same zero-copy, language-neutral buffers for IPC — adopted Arrow, then kept it only where it fit (Ray Data blocks) while moving general serialization to pickle5+cloudpickle.
  6. NumPy/PyTorch are buffer + strides + metadata; PyTorch adds device, a caching allocator, and autograd.
  7. DLPack & the *-interface protocols are the glue that lets all of them pass buffers around without copying.
