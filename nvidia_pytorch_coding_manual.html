You are being scored on a process, and the process is exactly your debugging discipline wearing a different shirt. 04P1 · CI-log failure summarizer

You're given a raw CI log. Lines of interest look like [timestamp] STATUS test_file.py::Class::test_name (12.4s) with STATUS ∈ {PASS, FAIL, SKIP}; a FAIL may be followed by error lines. Interleaved noise exists. Produce: pass/fail/skip counts, failures grouped by file with their first error line, and the top-5 slowest tests.

Clarify first

  • Exact line format guaranteed, or defensive parsing? (Defensive: unmatched lines are noise — but an error line right after a FAIL attaches to it.)
  • Log size? (Could be GBs → stream line-by-line, never read-all. The solution already does.)
  • First error line or full traceback? (First line is the summary's job; say the full-capture variant is a five-line change.)

Complete solution — verified

# Streamed CI-log summary: counts, failures by file (+first error), slowest.
import re
from collections import defaultdict, Counter

LINE = re.compile(
    r"^\[(?P<ts>[^\]]+)\]\s+(?P<st>PASS|FAIL|SKIP)"
    r"\s+(?P<test>\S+)\s+\((?P<dur>[0-9.]+)s\)")

def summarize(lines):
    counts = Counter()
    fails  = []                      # [test, dur, first_err_line]
    slow   = []
    last_fail = None                 # attach point for error lines
    for raw in lines:
        m = LINE.match(raw)
        if m:
            st = m["st"]; counts[st] += 1
            dur = float(m["dur"]); test = m["test"]
            slow.append((dur, test))
            last_fail = None
            if st == "FAIL":
                fails.append([test, dur, ""])
                last_fail = fails[-1]
        elif last_fail is not None and raw.strip():
            if not last_fail[2]:              # keep only the FIRST error line
                last_fail[2] = raw.strip()
    by_file = defaultdict(list)
    for test, dur, err in fails:
        by_file[test.split("::")[0]].append((test, dur, err))
    slow.sort(reverse=True)
    return counts, dict(by_file), slow[:5]

Walkthrough

One pass, one small state variable. The named-group regex is the contract with the log format — anchored at line start, tolerant of trailing junk (the SKIP line's [magma not built] tail parses fine because there's no end anchor). last_fail implements "error lines attach to the FAIL above": any matched status line clears it; the first non-empty unmatched line after a FAIL fills the error slot. Grouping by file is a split on :: — the pytest test-id convention. Verified on the sample log: 2 PASS / 3 FAIL / 1 SKIP, test_nn.py carrying 2 failures with RuntimeError: CUDNN_STATUS_EXECUTION_FAILED captured, slowest = 44.2s test_addmm_cuda_float64.

AspectNotes
ComplexityO(n) time over lines, O(results) memory — streaming-safe for giant logs. Top-k via sort is fine here; mention heapq.nlargest if k ≪ n.
Follow-ups to expectFull tracebacks (accumulate until next match) · JSON/CSV output · nonzero exit code if any FAIL (CI contract!) · multi-file via fileinput · durations histogram.
Where it mapsWei/Ting hours; the real-world cousin is every triage script this team writes. Pairs with the bash one-liners in P4 — offer both registers.

05P2 · The flaky-test detector

Given run records (commit, test, status) with possible repeated runs per commit, classify each test and rank the worst: a test that both passed and failed at the same commit is FLAKY; failing on every run is CONSISTENT-FAIL; failing only on some commits (but deterministically within each) is COMMIT-DEPENDENT — a regression suspect. Ignore tests with too few runs.

Clarify first

  • The definition is the problem: same-commit contradiction = nondeterminism; commit-partitioned failure = the code changed → different tool (bisect, P8). Stating this taxonomy unprompted is the senior move — it's Vol. III §17 as code.
  • Minimum-run threshold to suppress noise? (Parameterize it.) Ranking key? (Failure rate, then volume, then name for determinism.)

Complete solution — verified

from collections import defaultdict, Counter

def analyze(records, min_runs=3, top=5):
    per = defaultdict(lambda: defaultdict(Counter))   # test → commit → Counter
    for commit, test, st in records:
        per[test][commit][st] += 1

    out = []
    for test, commits in per.items():
        runs  = sum(sum(c.values()) for c in commits.values())
        fails = sum(c["fail"] for c in commits.values())
        if runs < min_runs:
            continue
        contradictory = sorted(k for k, c in commits.items()
                               if c["pass"] and c["fail"])
        if contradictory:            cls = "FLAKY"
        elif fails == runs:          cls = "CONSISTENT-FAIL"
        elif fails == 0:             cls = "HEALTHY"
        else:                        cls = "COMMIT-DEPENDENT"
        out.append((fails/runs, fails, runs, test, cls, contradictory))

    out.sort(key=lambda r: (-r[0], -r[2], r[3]))      # rate ↓, volume ↓, name ↑
    return [r for r in out if r[4] != "HEALTHY"][:top]

Walkthrough

The nested structure mirrors the question: to detect same-commit contradiction you must keep per-commit tallies, not global ones — a global pass+fail count can't distinguish a flake from a regression, which is exactly the diagnostic distinction that decides whether you rerun-and-quarantine or bisect. Classification reads straight off the tallies; the sort key is a tuple so ties resolve deterministically (a CI-team reflex worth narrating). Verified: test_b → CONSISTENT-FAIL (1.0), test_c → COMMIT-DEPENDENT (0.67), test_a → FLAKY (0.33, contradiction at c1), healthy test suppressed.

AspectNotes
ComplexityO(records) build, O(tests·log) rank; memory O(tests × commits touched).
Follow-ups to expectRecency window (only last K commits) · emit disable-issue-shaped output ("DISABLED test_x — flaky, 33%, contradiction at c1") · streaming from CSV (csv.reader) · per-platform dimension (add a key level).
Where it mapsWei's hour, dead center — and your answer's taxonomy line doubles as the §17 policy discussion even if no code is written.

06P3 · The version-matrix resolver

Given per-CUDA-lane minimum driver versions and a cuDNN→minimum-CUDA compatibility table, write: (a) which CUDA lanes a machine with driver D can run, and (b) the newest valid full stack for that machine. Version strings are dotted numerics of varying length.

Clarify first

  • Version format — dotted ints only, or suffixes like rc1? (Scope to numeric; name packaging.version as the production tool.)
  • Comparison depth — is 12.8 ≥ 12.8.0? (Yes: pad with zeros. Encode it.)
  • Selection policy — newest valid, and deterministic.

Complete solution — verified

def parse_v(s):
    return tuple(int(x) for x in s.split("."))

def at_least(v, floor):
    a, b = parse_v(v), parse_v(floor)
    n = max(len(a), len(b))
    return a + (0,)*(n-len(a)) >= b + (0,)*(n-len(b))   # pad, then tuple-compare

DRIVER_FLOOR = {"12.6": "525.60", "12.8": "570.00", "12.9": "575.00"}
CUDNN_NEEDS  = {"9.5": "12.6",  "9.7": "12.8",  "9.8": "12.8"}   # cudnn → min cuda

def usable_lanes(driver, floors=DRIVER_FLOOR):
    return sorted((l for l in floors if at_least(driver, floors[l])), key=parse_v)

def pick_stack(driver):
    lanes = usable_lanes(driver)
    if not lanes:
        return None
    cuda = lanes[-1]                                   # newest usable lane
    cudnns = sorted((c for c, mn in CUDNN_NEEDS.items() if at_least(cuda, mn)),
                    key=parse_v)
    return {"cuda": cuda, "cudnn": cudnns[-1] if cudnns else None}

Walkthrough — and the trap this problem exists to spring

The entire problem hides one classic bug: string comparison of versions. "12.10" < "12.9" is True — strings compare character-by-character, and '1' < '9' — which is precisely how real systems mis-sort 12.10 below 12.9. The fix is structural: parse to int tuples, pad to equal length, let tuple ordering do the right thing (both asserted in verification). Everything else is data-as-configuration: the compatibility knowledge lives in dicts, not in code, so the tables can grow without logic changes — the shape real support-matrix tooling takes. Verified: driver 560.35 → lanes ["12.6"] → stack {cuda: 12.6, cudnn: 9.5}; driver 571.00 → {cuda: 12.8, cudnn: 9.8}; ancient driver → None.

AspectNotes
Follow-ups to expectSuffixes (rc/a — point at PEP 440 / packaging) · multi-constraint joint resolution (this greedy picks per-component; full joint solving is a resolver/SAT conversation — name it, don't build it) · "why is nvidia-smi's CUDA version not what I run?" segue to Vol. III §09.
Where it mapsTing's hour (the support matrix is her artifact) and any hour that touches drivers — the string-compare trap is also a great story about why this team pins and parses carefully.

08P5 · Build order: toposort & waves

Given packages and dependency pairs (pkg, needs), output a valid build order — deterministically — or report the cycle. Follow-up: output the waves: maximal batches that can build in parallel.

Clarify first

  • Direction convention — confirm (pkg, needs) means pkg depends on needs (so needs builds first). Getting the edge direction wrong is the classic 10-minute hole; nail it with one example on the pad.
  • Determinism required? (Volunteer it: reproducible order is a build-system value — sort every frontier.)
  • Cycle handling — error with the members, not just a boolean.

Complete solution — verified (Kahn's, level-synchronous)

from collections import defaultdict

def build_order(packages, deps):
    """deps: (pkg, needs) pairs — pkg depends on needs. Returns (order, waves)."""
    indeg = {p: 0 for p in packages}
    dependents = defaultdict(list)            # needs → [pkgs waiting on it]
    for pkg, needs in deps:
        dependents[needs].append(pkg)
        indeg[pkg] += 1

    order, waves = [], []
    frontier = sorted(p for p in packages if indeg[p] == 0)   # determinism
    while frontier:
        waves.append(frontier)
        order.extend(frontier)
        nxt = []
        for done in frontier:
            for pkg in dependents[done]:
                indeg[pkg] -= 1
                if indeg[pkg] == 0:
                    nxt.append(pkg)
        frontier = sorted(nxt)

    if len(order) != len(packages):
        cycle = sorted(p for p in packages if indeg[p] > 0)
        raise ValueError(f"dependency cycle among: {cycle}")
    return order, waves

Walkthrough

Kahn's algorithm is "repeatedly build whatever has nothing left to wait for": count each package's unmet dependencies (indeg), start from the zero-dependency frontier, and as each builds, decrement its dependents — anything hitting zero joins the next frontier. Processing frontier-by-frontier (instead of one queue-pop at a time) gives the parallel waves for free: every member of a wave has all dependencies satisfied by earlier waves, so a wave is exactly what make -j/ninja can run concurrently. Cycle detection is the leftover check — nodes never reaching indegree 0 are on or behind a cycle, and reporting them beats a bare "cycle exists". Verified on a torch-shaped graph: waves [[toolkit], [cudnn, cutlass, magma], [torch]]; the two-node cycle raises with its members named.

AspectNotes
ComplexityO(V + E) plus the sorts (O(V log V) total) bought deliberately for determinism — say that trade aloud.
Why Kahn over DFS hereIterative (no recursion limits), waves fall out naturally, cycle reporting is direct. DFS post-order is the right answer to "alternative?" — know both.
Follow-ups to expectCritical path / minimum total time with durations (longest path over the DAG — DP over the same structure) · incremental rebuild set given changed nodes (BFS over dependents) · exactly-the-cycle extraction (walk indeg>0 subgraph).
Where it mapsEverywhere in this role: package builds, docker layer ordering, CI job DAGs. The most on-theme classic in existence for this loop.

09P6 · Intervals: windows & runners

CI jobs have (start, end) times. (a) Merge them into busy windows. (b) Compute the minimum number of runners that could have executed the schedule (= peak concurrency).

Clarify first

  • Boundary semantics — if one job ends at t and another starts at t, do they overlap? Declare half-open [start, end): the runner frees exactly at end and is reusable. This single sentence is where the problem is won.
  • Merged-window touching — for "busy windows," touching intervals merge (different question than runner overlap; noticing that distinction is senior signal).

Complete solution — verified

def merge_windows(jobs):
    out = []
    for s, e in sorted(jobs):                 # by start (then end)
        if out and s <= out[-1][1]:           # overlaps or touches last window
            out[-1][1] = max(out[-1][1], e)   # extend (nesting-safe via max)
        else:
            out.append([s, e])
    return [tuple(x) for x in out]

def max_runners(jobs):
    events = []
    for s, e in jobs:
        events += [(s, 1), (e, -1)]
    events.sort(key=lambda t: (t[0], t[1]))   # at a tie: -1 (free) before +1 (claim)
    cur = best = 0
    for _, d in events:
        cur += d
        best = max(best, cur)
    return best

Walkthrough

Merging: sort by start, then a single window-extension pass; the max handles fully-nested intervals (forgetting it is the classic bug — call it out as you write it). Counting: convert to +1/−1 events and sweep; the sort key encodes the half-open semantics — at equal timestamps, tuple ordering puts −1 before +1, so a freed runner is reusable by a job starting that instant. Verified: [(0,30),(5,10),(15,20),(30,40)] merges to [(0,40)] (touching at 30) while needing only 2 runners (the end-at-30 job's runner takes the start-at-30 job); the triple-overlap case returns 3.

AspectNotes
ComplexityO(n log n) both — the sort dominates; sweep is O(n).
Follow-ups to expectActual runner assignment (min-heap of end times; pop when reusable — same count, plus a mapping) · total busy time (sum merged windows) · streaming arrivals (heap-based, no global sort) · weighted capacity (jobs needing k GPUs: delta = ±k, same sweep).
Where it mapsWei's capacity world (§15 of Vol. III) — and your closing line writes itself: "this is the toy version of the capacity-vs-coverage question."

10P7 · LRU cache, two ways

Implement an LRU cache with O(1) get/put and capacity-based eviction. (Domain framing: this is ccache, docker layer caches, and CI artifact caches in miniature.)

Clarify first

  • Does get refresh recency? Does updating an existing key? (Yes and yes — state it.)
  • Which register do they want — "use the right stdlib tool" or "build the data structure"? Ask; then ideally show both, fast-first.

Solution A — the tool (verified)

from collections import OrderedDict

class LRU:
    def __init__(self, cap):
        self.cap, self.d = cap, OrderedDict()
    def get(self, k):
        if k not in self.d:
            return -1
        self.d.move_to_end(k)                 # mark k most-recently-used
        return self.d[k]
    def put(self, k, v):
        if k in self.d:
            self.d.move_to_end(k)
        self.d[k] = v
        if len(self.d) > self.cap:
            self.d.popitem(last=False)        # evict the LRU end

Solution B — the internals (verified): dict + doubly-linked list

class Node:
    __slots__ = ("k", "v", "prev", "next")
    def __init__(self, k=None, v=None):
        self.k, self.v, self.prev, self.next = k, v, None, None

class LRU2:
    def __init__(self, cap):
        self.cap, self.map = cap, {}
        self.head, self.tail = Node(), Node()      # sentinels; head side = MRU
        self.head.next, self.tail.prev = self.tail, self.head

    def _unlink(self, n):
        n.prev.next, n.next.prev = n.next, n.prev
    def _push_front(self, n):
        n.next, n.prev = self.head.next, self.head
        self.head.next.prev = n
        self.head.next = n

    def get(self, k):
        n = self.map.get(k)
        if not n:
            return -1
        self._unlink(n); self._push_front(n)
        return n.v

    def put(self, k, v):
        n = self.map.get(k)
        if n:                                       # update = refresh + rewrite
            n.v = v; self._unlink(n); self._push_front(n); return
        if len(self.map) >= self.cap:
            lru = self.tail.prev
            self._unlink(lru); del self.map[lru.k]  # node knows its key → O(1)
        n = Node(k, v); self.map[k] = n; self._push_front(n)

Walkthrough

The invariant in both: recency order is a list; the dict jumps to any node in O(1); MRU lives at one end, eviction happens at the other. Solution A is OrderedDict doing exactly that internally — knowing move_to_end/popitem(last=False) is the tool-fluency register. Solution B is the interview classic: sentinels kill every null-edge special case (empty list, ends), _unlink/_push_front are the only two pointer operations you need, and storing the key on the node is the detail people forget — without it, evicting from the tail can't delete its dict entry in O(1). Both implementations passed the same verified sequence (evict-the-LRU, refresh-on-get, update-refreshes).

AspectNotes
ComplexityO(1) get/put both ways; O(cap) memory. __slots__ is a small "I've done this at scale" flourish.
Follow-ups to expectThread safety (one lock; or say "shard it") · TTL (store timestamps, lazy-expire on access) · LFU (the follow-up trap: needs freq buckets — outline, don't improvise) · functools.lru_cache as the decorator cousin · byte-size-based capacity (ccache reality: evict until under budget).
Where it mapsAny hour; the domain framing ("this is the docker layer cache's eviction problem") turns a stock question into a fit signal.

11P8 · Bisect the regression

Nightlies 1..N; the latest is bad and badness is monotone (once broken, stays broken). Find the first bad nightly with minimal checks. Follow-ups: the range isn't known in advance; the check itself is flaky.

Clarify first

  • Confirm monotonicity and that is_bad(N) holds — the preconditions are the algorithm's contract. (If badness isn't monotone — a flake — this is the wrong tool: that's P2's detector. Saying so is the P2↔P8 taxonomy again.)
  • Check cost — expensive (a full test run), so minimizing calls matters and retries must be budgeted.

Complete solution — verified

def first_bad(lo, hi, is_bad):
    """First i in [lo, hi] with is_bad(i); requires is_bad(hi) is True.
    Invariant: the answer is always inside [lo, hi]."""
    while lo < hi:
        mid = (lo + hi) // 2
        if is_bad(mid):
            hi = mid          # mid could BE the first bad → keep it
        else:
            lo = mid + 1      # mid is good → first bad is strictly after
    return lo

# Unknown upper bound: gallop until bad, then bisect the last doubling.
def find_range(is_bad):
    hi = 1
    while not is_bad(hi):
        hi *= 2
    return hi // 2 + 1, hi

# Flaky check: majority-vote wrapper restores (probabilistic) monotonicity.
def dampen(is_bad_once, votes=3):
    def is_bad(i):
        bad = sum(bool(is_bad_once(i)) for _ in range(votes))
        return bad * 2 > votes
    return is_bad

Walkthrough

The loop shape while lo < hi with hi = mid / lo = mid + 1 is the off-by-one-proof pattern: the invariant "answer ∈ [lo, hi]" holds at every step (a bad mid stays in range because it might be the first; a good mid excludes itself), the range strictly shrinks (since mid < hi always), and termination at lo == hi leaves exactly the answer — narrate the invariant; it's the difference between knowing binary search and reciting it. Verified: first-bad 347 in a 1000-range found in 10 checks (⌈log₂1000⌉); galloping brackets an unknown range in O(log answer); and the majority-vote wrapper found 347 correctly even with a 15%-lying predicate (7 votes). The whole chapter is git bisect — say so, including the flaky-caveat (git bisect skip exists for a reason) and the cost model (each check = a build+test, so log₂ matters in hours, not nanoseconds).

12P9 · SGD three ways

Implement SGD with momentum as (a) a for-loop reference and (b) a foreach version using torch._foreach_*; explain what a fused version adds; prove (a) ≡ (b).

Complete solution — hand-verified, canonical APIs

import torch

class ForLoopSGD:
    """Reference: one small kernel launch per op per tensor."""
    def __init__(self, params, lr, momentum=0.0):
        self.params = list(params)
        self.lr, self.momentum = lr, momentum
        self.bufs = [None] * len(self.params)

    @torch.no_grad()                          # optimizer math must not be traced
    def step(self):
        for i, p in enumerate(self.params):
            if p.grad is None:
                continue
            d = p.grad
            if self.momentum != 0.0:
                if self.bufs[i] is None:
                    self.bufs[i] = d.clone()
                else:
                    self.bufs[i].mul_(self.momentum).add_(d)
                d = self.bufs[i]
            p.add_(d, alpha=-self.lr)

class ForeachSGD:
    """Horizontally fused: one launch per OP across ALL tensors.
    Simplification: assumes all params share device & dtype and all have
    grads every step — the real optimizer groups tensor lists by
    (device, dtype) and handles sparse/None; say this caveat aloud."""
    def __init__(self, params, lr, momentum=0.0):
        self.params = list(params)
        self.lr, self.momentum = lr, momentum
        self.bufs = None

    @torch.no_grad()
    def step(self):
        ps = [p for p in self.params if p.grad is not None]
        gs = [p.grad for p in ps]
        if self.momentum != 0.0:
            if self.bufs is None:
                self.bufs = [g.clone() for g in gs]
            else:
                torch._foreach_mul_(self.bufs, self.momentum)
                torch._foreach_add_(self.bufs, gs)
            upd = self.bufs
        else:
            upd = gs
        torch._foreach_add_(ps, upd, alpha=-self.lr)

def test_equivalence():                       # run this in your env before Monday
    torch.manual_seed(0)
    shapes = [(4, 3), (10,), (2, 2, 2)]
    p1 = [torch.randn(s, requires_grad=True) for s in shapes]
    p2 = [a.detach().clone().requires_grad_(True) for a in p1]
    o1 = ForLoopSGD(p1, lr=0.1, momentum=0.9)
    o2 = ForeachSGD(p2, lr=0.1, momentum=0.9)
    for _ in range(5):
        for a, b in zip(p1, p2):
            g = torch.randn_like(a)
            a.grad, b.grad = g.clone(), g.clone()
        o1.step(); o2.step()
    for a, b in zip(p1, p2):
        torch.testing.assert_close(a, b)
    print("equivalent")

Walkthrough

The launch arithmetic is the whole point, so say it with numbers: with N parameter tensors and momentum, the for-loop issues ~3 tiny kernels per tensor per step (mul, add, add) → 3N launches, each microseconds of overhead around trivial math; the foreach version issues the same three ops once each over whole tensor lists → ~3 launches total, with multi_tensor_apply underneath packing pointer/size metadata into chunked, fixed-size kernel arguments (Vol. III §20). A fused step is the vertical completion: the entire update rule in one kernel per chunk — fewer launches still, and no foreach intermediates spiking peak memory — which is what Masaki's fused AdamW does, plus capturable device-side state for CUDA graphs (Vol. III §21). The @torch.no_grad() decorator is non-decorative: parameter updates must not be recorded by autograd. And the grouping caveat in the docstring is the honest edge — volunteering that the real implementation groups by (device, dtype) and handles Nones is precisely the calibrated-candor register for his hour.

Follow-ups to expectBe ready with
"Why can foreach change numerics?"It shouldn't for elementwise ops (same math, batched) — but ordering-sensitive reductions or dtype promotion edge cases are why foreach ops carry consistency tests against the for-loop reference (P11's pattern at scale).
"Why not always fused?"Per-optimizer CUDA implementations (coverage cost), dtype/device support boundaries, and debuggability — the taxonomy exists because each tier trades generality for speed.
"Make it capturable"Move step-dependent state to device tensors, no host branches on device values, lr as a tensor — derive it from the graph-replay contract (Vol. III §21, §23).

13P10 · Debug the training loop

"This training script behaves badly — loss is erratic, memory climbs every epoch, validation looks wrong, and it sometimes crashes with a CUDA device-side assert. Find the bugs." (Labels, per the data pipeline's docs, come from an annotation tool that 1-indexes classes.)

The buggy code

model   = SmallNet(num_classes=10).cuda()
opt     = torch.optim.AdamW(model.parameters(), lr=3e-4)
sched   = torch.optim.lr_scheduler.CosineAnnealingLR(opt, T_max=EPOCHS)
loss_fn = torch.nn.CrossEntropyLoss()

total_loss = 0.0
for epoch in range(EPOCHS):
    for x, y in train_loader:          # y ∈ {1..10} per the annotation tool
        x, y = x.cuda(), y.cuda()
        out  = model(x)
        loss = loss_fn(out, y)
        loss.backward()
        opt.step()
        sched.step()
        total_loss += loss

    correct = 0
    for x, y in val_loader:
        out = model(x.cuda())
        correct += (out.argmax(1) == y.cuda()).sum()
    print(epoch, total_loss, correct)

Read it systematically — lifecycle order, symptom by symptom

BugSymptom → mechanism → fix
A · no zero_grad()Erratic/instable loss → gradients accumulate across steps because .backward() adds into .grad; every step applies the running sum. Fix: opt.zero_grad(set_to_none=True) before each backward.
B · labels 1..10 vs C=10Intermittent device-side assert → CrossEntropy needs targets in [0, C); a batch containing label 10 hits an out-of-range gather on GPU. Async execution makes the reported line unreliable — localize with CUDA_LAUNCH_BLOCKING=1 (the ptrblck move, Vol. II §17). Fix at the source: shift labels to 0-index in the dataset (or y - 1 at ingestion) — and add an assert.
C · total_loss += lossMemory climbs → accumulating the tensor keeps every step's autograd graph alive via the running sum. Fix: += loss.item() (or .detach()), and reset per epoch (the printed number is also cumulative across epochs — a correctness bug hiding inside the leak).
D · sched.step() per batchLR collapses way too fast → CosineAnnealing with T_max=EPOCHS is an epoch-granularity schedule being stepped per iteration. Fix: step once per epoch (or set T_max in iterations — pick one, consistently).
E · validation leaks stateVal is slow, memory-heavy, and wrong-ish → no model.eval() (dropout still active, batchnorm still updating running stats on val data — silent training-set contamination) and no torch.no_grad() (autograd graphs built for nothing). Fix both; restore model.train() after.

The fixed loop

for epoch in range(EPOCHS):
    model.train()
    total_loss = 0.0
    for x, y in train_loader:
        x = x.cuda(non_blocking=True)
        y = (y - 1).cuda(non_blocking=True)        # 0-index; better: fix the dataset
        opt.zero_grad(set_to_none=True)
        out  = model(x)
        loss = loss_fn(out, y)
        loss.backward()
        opt.step()
        total_loss += loss.item()
    sched.step()                                    # epoch-granularity schedule

    model.eval()
    correct = 0
    with torch.no_grad():
        for x, y in val_loader:
            out = model(x.cuda(non_blocking=True))
            correct += (out.argmax(1) == (y - 1).cuda()).sum().item()
    print(epoch, total_loss, correct)

Don't hunt randomly — narrate the sweep in lifecycle order (data → forward → loss → backward → step → schedule → eval) and check each stage's contract; the five bugs fall out one per stage, which is why the systematic read beats staring. 14P11 · A parametrized op test

Write a proper test for an op implementation — say rms_norm — across dtypes, shapes, and devices, with defensible tolerances. (The skill under test: do you know how numerical code gets verified, not just written?)

Complete solution — hand-verified, pytest style

import pytest
import torch

def rms_norm(x, weight, eps=1e-6):
    # implementation under test
    ms = x.pow(2).mean(dim=-1, keepdim=True)
    return x * torch.rsqrt(ms + eps) * weight

def rms_norm_ref(x, weight, eps=1e-6):
    # reference: compute in float64, cast back — the trust anchor
    xd = x.double()
    ms = xd.pow(2).mean(dim=-1, keepdim=True)
    return (xd * torch.rsqrt(ms + eps) * weight.double()).to(x.dtype)

TOLS = {
    torch.float32:  dict(rtol=1e-5,  atol=1e-6),
    torch.float16:  dict(rtol=1e-3,  atol=1e-3),
    torch.bfloat16: dict(rtol=1.6e-2, atol=1e-2),
}
DEVICES = ["cpu"] + (["cuda"] if torch.cuda.is_available() else [])

@pytest.mark.parametrize("device", DEVICES)
@pytest.mark.parametrize("dtype", list(TOLS))
@pytest.mark.parametrize("shape", [(8, 64), (1, 1), (3, 5, 127)])
def test_rms_norm(device, dtype, shape):
    if device == "cpu" and dtype is torch.float16:
        pytest.skip("fp16-on-CPU coverage out of scope here")   # skip WITH a reason
    torch.manual_seed(0)
    x = torch.randn(shape, dtype=dtype, device=device)
    w = torch.randn(shape[-1], dtype=dtype, device=device)
    torch.testing.assert_close(rms_norm(x, w), rms_norm_ref(x, w), **TOLS[dtype])

Walkthrough — the five decisions that make it a real test

DecisionWhy it's the senior one
Reference in higher precisionComparing an implementation to itself in another order proves nothing; float64 reference then cast-down gives an independent trust anchor. This is how numerics people think, and it shows.
Per-dtype tolerances, justifiedTolerance isn't a knob you loosen until green — it's derived from the format (~1e-3-scale units-in-last-place for fp16; bf16's short mantissa needs ~1e-2). Being able to say why each number is what it is answers the "tolerance vs bug" question from Vol. III §17 preemptively.
Shapes that hunt bugs(1,1) probes degenerate reduction; a prime, non-multiple-of-anything last dim (127) probes vectorized-tail handling — where real kernel bugs live.
Deterministic + honest skipsSeeded inputs; device parametrization degrades gracefully without CUDA; skips carry reasons. Every skip-with-reason is future triage prepaid.
Structure scales to OpInfoSay the bridge: "this is one op hand-parametrized; PyTorch's OpInfo system is this pattern industrialized — declare the op's samples/dtypes once, inherit the whole battery" (Vol. III §18). That sentence lands the internals-fluency point in one breath.

15P12 · RAII: a move-only handle

Wrap a raw resource (a POSIX fd) in a class that cannot leak and cannot double-close: destruction releases; copying is forbidden; moving transfers ownership. (The same shape as CUDA streams/events/handles wrapped in RAII throughout real GPU codebases.)

Complete solution — compiled & run, verified

#include <fcntl.h>
#include <unistd.h>
#include <utility>
#include <cassert>
#include <cstdio>

// Move-only owner of a POSIX file descriptor.
class UniqueFd {
public:
    UniqueFd() noexcept = default;
    explicit UniqueFd(int fd) noexcept : fd_(fd) {}
    ~UniqueFd() { reset(); }

    UniqueFd(const UniqueFd&) = delete;              // owning ⇒ no copies
    UniqueFd& operator=(const UniqueFd&) = delete;

    UniqueFd(UniqueFd&& o) noexcept : fd_(o.release()) {}
    UniqueFd& operator=(UniqueFd&& o) noexcept {
        if (this != &o) reset(o.release());          // self-move safe
        return *this;
    }

    int  get() const noexcept { return fd_; }
    explicit operator bool() const noexcept { return fd_ >= 0; }
    int  release() noexcept { int f = fd_; fd_ = -1; return f; }
    void reset(int fd = -1) noexcept {
        if (fd_ >= 0) ::close(fd_);
        fd_ = fd;
    }
private:
    int fd_ = -1;
};

int main() {
    UniqueFd a(::open("/dev/null", O_WRONLY));
    assert(a);
    UniqueFd b(std::move(a));                        // move ctor: b owns, a empty
    assert(!a && b);
    a = std::move(b);                                // move assign
    assert(a && !b);
    std::puts("raii ok");
}

Walkthrough — what each line defends

ChoiceThe reasoning to narrate
Deleted copiesTwo owners of one fd = double-close. The type system now enforces the invariant humans forget — that's the entire RAII thesis, stated in two = delete lines.
Move = steal + neutralizeTake the source's fd via release(), leaving it at −1 so its destructor is a no-op. Move assignment must also release what this held — reset(o.release()) does both in one expression.
noexcept on movesNot decoration: containers like std::vector only move elements during reallocation if the move is noexcept — otherwise they copy (here: can't) or lose the strong guarantee. One keyword, real performance semantics.
Self-move guardif (this != &o) — without it, reset(o.release()) on the same object still works here (release neutralizes first), but the guard makes the invariant unconditional; mention both facts and you've shown depth, not paranoia.
release() / reset()The unique_ptr-shaped escape hatches: interop with C APIs that take ownership, and deterministic early close. Naming the analogy ("this is unique_ptr for an fd; production code uses exactly this pattern for CUDA handles") lands the transfer.
Likely follow-ups

"Templatize it" → unique_handle<T, Deleter>, i.e. re-derive unique_ptr's design and say so. "Thread safety?" → ownership transfer isn't atomic; external synchronization or single-owner discipline — same answer as unique_ptr. "Why not shared ownership?" → shared_ptr's costs (atomics, control block) and the design smell of unclear ownership for scarce resources like fds/streams. Each answer is short because the design already contains it.

16P13 · Linker-triage drills

The JD's stand-out skill as a read-and-diagnose round: three realistic artifacts, each with the diagnostic walkthrough. The full bestiary and toolbelt live in Vol. III §12 — these drills are it, performed.

DRILLImportError: undefined symbol …TensorBase…__cxx11… loading a custom extension
Read the artifact

Pipe the mangled name through c++filt: the demangled signature contains std::__cxx11::basic_string. That token is the tell — __cxx11 spellings exist only under the new libstdc++ string/list ABI, so the extension was compiled expecting one _GLIBCXX_USE_CXX11_ABI setting while the torch it loaded against exports the other. Two artifacts, two incompatible std layouts, one missing symbol.

Confirm & fix

Confirm: nm -D the torch libs for the symbol's actual spelling; check the wheel's ABI convention vs the extension's compile flags (and compiler/toolchain vintage). Fix by agreement, not patching: rebuild the extension with the matching ABI flag and a compatible toolchain — ideally via the build helpers that inherit torch's flags — and add the flag echo to the extension's build log so the mismatch can never be silent again.

DRILLLink error: multiple definition of `registerMyOps()` — or ops silently missing at runtime
Read the artifact

Two faces of one mechanism: PyTorch-style kernel/op registration runs from static initializers. "Multiple definition" means the registrar's definition landed in more than one translation unit (defined in a header without inline, or the same object archived into two static libs that both got linked) — an ODR violation. The eerier sibling — link succeeds but ops are missing — is the linker dropping the registrar's object from a static archive because nothing referenced it, so its side-effect registration never ran.

Confirm & fix

Confirm with nm across the archives (who defines it, how many times; is the object even pulled in?). Fixes by cause: header definitions become inline or move to one TU; duplicated archives get deduped in the link graph; and dropped-registrar syndrome is exactly why registration libraries link with --whole-archive — force every object in, side effects included. Closing line worth saying: "this is why PyTorch's own build treats registration libs specially — Vol. I §21–22 territory."

DRILLSame wheel, works in venv A, crashes with a cuDNN version-mismatch warning in venv B
Read the artifact

Behavior varying by environment with identical code means the dynamic loader resolved a different library: venv B's environment (an exported LD_LIBRARY_PATH, a system-installed CUDA, another package shipping its own libcudnn) is winning the search-order race against the wheel's intended nvidia-* dependency copy. The mismatch warning is torch noticing the loaded cuDNN isn't the one it was built against.

Confirm & fix

Confirm decisively: LD_DEBUG=libs python -c "import torch" in both venvs and diff which libcudnn.so path each loads (or ldd the extension libs). Fix at the environment layer: remove the shadowing path from LD_LIBRARY_PATH / the conflicting package, letting RPATH-relative resolution find the wheel's own deps — and answer the human with the mechanism (search order: RPATH vs LD_LIBRARY_PATH vs system), ptrblck-style, so venv C never happens to them.
