BUILD Chapter I

The anatomy of a container image.

The premise A container image is not a VM. It's a sandwich of read-only filesystem layers plus a small JSON config. Everything else follows from that.

Mental model Each Dockerfile line creates a layer. Every byte in any layer ships forever, even if a later layer deletes it.

A container image is a stack of read-only filesystem layers (each compressed as a tarball) plus a JSON config that lists those layers, sets entrypoint/env/labels, and declares architecture. When a container runs, the runtime stacks the layers, adds a writable top layer, and starts your process inside a set of Linux kernel namespaces. That's it. Everything else is convention.

Interactive · Image layer anatomy

Why layers matter

Layers cache. Builders re-use unchanged layers on the next build. Order Dockerfile instructions from least-frequently-changing to most-frequently-changing and your typical build drops from minutes to seconds.

Layers ship. Every layer is content-addressed by its sha256 hash. If two images share a base, pulling the second image only downloads the diff. Cluster-wide, this saves enormous bandwidth and disk.

Layers leak. Anything written in a layer is forever. RUN curl secret && rm secret still ships the secret — the file existed at one layer's commit. Use --mount=type=secret instead.

Layers are immutable. Image identity is the manifest hash. Two builds producing the same byte-content layers are the same image. This is what makes reproducible builds possible (Chapter III).

The manifest, the config, the index

Three JSON documents stitch an image together:

Image config. Architecture, OS, layer digests, environment variables, entrypoint, working directory, user, labels. ~2KB.

Manifest. Points to the config and the layer blobs by hash. The "image" in a registry is, technically, this manifest. ~1KB.

Image index (a.k.a. manifest list). For multi-arch images: points to per-platform manifests (linux/amd64, linux/arm64, etc.). The registry returns the right manifest based on what the pulling client asks for.

The one fact that explains 80% of container behavior

Image hashes are not just identifiers — they are the cache key, the deduplication key, the security verification key, and the cross-registry portability key. Anything that changes the byte content of any layer changes the hash, which changes everything downstream. Reproducible builds (Chapter III) and image signing (Chapter V) both ride on this invariant.

BUILD Chapter II

Designing the Dockerfile.

The discipline Multi-stage builds. Pinned digests. Cache mounts. Distroless runtime. Non-root user. These five conventions together produce a 5–20× size reduction and matching security improvement vs. naive Dockerfiles.

The single biggest improvement Multi-stage builds: separate the toolchain image from the runtime image.

A Dockerfile encodes a build plan. The art is making it cache-friendly, reproducible, small, and safe — all at once. Below: the bad way and the good way, side by side.

Comparison · same Python service, two Dockerfiles
A 17× size reduction in 18 lines of refactor.
✗ Naive / common
FROM python:3.12
WORKDIR /app
COPY . .
RUN apt-get update && apt-get install -y curl gcc
RUN pip install -r requirements.txt
RUN apt-get remove -y gcc && apt-get autoremove -y
EXPOSE 8000
CMD ["python", "app.py"]
✓ Production-grade
# syntax=docker/dockerfile:1.7
# ─── BUILD STAGE ──────────────────────
FROM python:3.12-slim@sha256:abc... AS build
WORKDIR /app
RUN --mount=type=cache,target=/root/.cache/pip \
    --mount=type=bind,source=requirements.txt,target=/tmp/r.txt \
    pip install --prefix=/install -r /tmp/r.txt

# ─── RUNTIME STAGE ────────────────────
FROM gcr.io/distroless/python3-debian12:nonroot
COPY --from=build /install /usr/local
COPY --chown=nonroot:nonroot app.py /app/
WORKDIR /app
USER nonroot
EXPOSE 8000
ENTRYPOINT ["python", "app.py"]
Naive
1,247 MB
Production
73 MB

The seven Dockerfile rules

1. Multi-stage. The build stage has the compiler, the package manager, the dev libraries. The runtime stage has only what's needed to run the binary. Copy the artifacts between stages with COPY --from=build.

2. Pin base images by digest, not tag. FROM python:3.12-slim@sha256:abc... guarantees byte-identical input. FROM python:3.12-slim resolves to whatever the registry returns today, which is a moving target.

3. Order layers by change frequency. Base image → system packages → language deps → app code. Code changes every commit; deps change weekly; base changes monthly. Right order = max cache hit.

4. Use cache mounts for package managers. RUN --mount=type=cache,target=/root/.cache/pip pip install ... persists pip's cache across builds without baking it into the image. Same trick works for apt, npm, go, cargo.

5. Distroless or scratch for runtime. Distroless images (gcr.io/distroless/...) contain only your app's dependencies — no shell, no package manager, no curl, no anything an attacker could use. The nonroot variants drop privileges by default.

6. Run as a non-root user. Always. Set USER nonroot (or a specific UID like USER 65532) before ENTRYPOINT. Combined with read-only root filesystem at runtime, this defeats most container-escape exploits.

7. Use BuildKit secret mounts. RUN --mount=type=secret,id=npm,target=/root/.npmrc npm ci mounts the secret only during that RUN step; the secret is never written to any layer.

Common pitfalls

# ✗ Leaks secrets into layer history forever
ENV NPM_TOKEN=secret_token_here
RUN npm install

# ✗ Invalidates cache on every code change, even for deps
COPY . .
RUN pip install -r requirements.txt

# ✗ apt-get install in one layer, cleanup in another → cleanup doesn't help
RUN apt-get install -y build-essential
RUN apt-get remove -y build-essential  # too late, prev layer keeps files

# ✗ Latest tag — non-reproducible, breaks the moment upstream changes
FROM ubuntu:latest

# ✗ ADD when COPY would do — ADD has surprises (auto-extract, URL fetch)
ADD https://example.com/file.tar.gz /tmp/
The order-of-operations heuristic

A good Dockerfile reads like a sentence: "from a pinned base, install least-changing things first, then deps, then code, then run as a non-root user." When you can't quote that summary back from looking at a Dockerfile, it has structural problems.

BUILD Chapter III

Reproducible & multi-arch builds.

The goal Build the same Dockerfile twice, anywhere, and get byte-identical images. And build for amd64 + arm64 from a single CI job.

The trick SOURCE_DATE_EPOCH for timestamps, buildx for multi-platform, native nodes per arch when possible.

Reproducible builds mean two different runs of the same Dockerfile produce byte-identical image layers. Multi-arch means one image works on both amd64 and arm64 by shipping a manifest list with both variants. Both are now standard practice in 2026.

Reproducible builds, the three sources of non-determinism

1. Timestamps. Every file written during the build gets the current time. Use SOURCE_DATE_EPOCH to override:

# Set to a fixed Unix timestamp (e.g., commit time)
export SOURCE_DATE_EPOCH=$(git log -1 --pretty=%ct)

docker buildx build \
  --build-arg SOURCE_DATE_EPOCH=$SOURCE_DATE_EPOCH \
  --output type=oci,dest=image.tar,rewrite-timestamp=true \
  --provenance=false \                # reproducibility needs no provenance
  -t myapp:repro .

# Verify byte-identical
sha256sum image.tar
# → same hash on every machine, every run

2. Network calls. Anything that fetches from a URL during build (apt-get, pip install from index, curl) can change. Mitigations: lock files (requirements.txt with hashes, Pipfile.lock, package-lock.json), pin every direct dependency, prefer offline caches.

3. File order. Some package managers write files in directory-walk order, which varies by filesystem. find ... -print | sort | tar ... patterns fix this.

Multi-arch with buildx

Buildx + BuildKit + QEMU is the standard. One command produces a manifest list that points to per-arch images:

# One-time setup
docker buildx create --name multiarch \
  --driver docker-container \
  --bootstrap --use

# Verify QEMU emulators are registered
docker run --rm --privileged tonistiigi/binfmt --install all

# Build for multiple architectures, push as manifest list
docker buildx build \
  --platform linux/amd64,linux/arm64 \
  --tag myorg/myapp:v1.2.3 \
  --tag myorg/myapp:latest \
  --push \
  --cache-to type=registry,ref=myorg/myapp:cache,mode=max \
  --cache-from type=registry,ref=myorg/myapp:cache \
  --sbom=true \
  --provenance=mode=max \
  .

# Inspect the manifest list
docker buildx imagetools inspect myorg/myapp:v1.2.3

QEMU emulation vs. native builders — when to switch

QEMU emulation works but is slow — sometimes 10–30× slower than native — especially for compute-heavy builds like compiling C/C++ or building CUDA kernels. Two ways out:

1. Cross-compilation. The compiler runs on amd64 but emits arm64 code. BUILDPLATFORM and TARGETPLATFORM args let your Dockerfile branch on this. Works well for Go, Rust; harder for Python or Java with native deps.

# syntax=docker/dockerfile:1.7
FROM --platform=$BUILDPLATFORM golang:1.22-alpine AS build
ARG TARGETOS TARGETARCH
WORKDIR /src
COPY . .
RUN GOOS=$TARGETOS GOARCH=$TARGETARCH CGO_ENABLED=0 \
    go build -ldflags="-s -w -buildid=" -trimpath -o /app .

FROM --platform=$TARGETPLATFORM gcr.io/distroless/static-debian12:nonroot
COPY --from=build /app /app
ENTRYPOINT ["/app"]

2. Native builder nodes per architecture. Register an arm64 builder (a real arm64 machine — an AWS Graviton instance, a Mac, a Raspberry Pi cluster) and let buildx route per-platform builds to native nodes. Result: full-speed builds with no emulation overhead.

# Create builder, add native arm64 node
docker buildx create --name native --node amd64-node \
  --driver docker-container ssh://build-amd64
docker buildx create --append --name native --node arm64-node \
  --driver docker-container --platform linux/arm64 ssh://build-arm64
docker buildx use native

# buildx automatically routes per-platform work
docker buildx build --platform linux/amd64,linux/arm64 ...
When reproducibility matters most

Two situations make reproducibility non-negotiable. Compliance (SLSA, FIPS, government deployments) — you need to prove the artifact comes from the source. Security forensics — when a CVE drops you want to know if you're affected, which means re-building from the same source must yield the same image. For other situations, "deterministic enough" — pinned bases + lock files — is usually fine.

HARDEN Chapter IV

Hardening the image.

The model Defense in depth. Assume the application code has a bug. Make sure that bug cannot become root on the node.

The single highest-leverage change Run as non-root with a read-only root filesystem. Defeats ~80% of escape exploits.

Hardening is layered: a small image with no shell, a non-root user, a read-only filesystem, dropped capabilities, a tight seccomp profile, and signature verification before the container ever runs. Each layer alone is partial; together they're robust.

Image-time hardening

Minimal base. Distroless (gcr.io/distroless/static-debian12, ~2MB) or scratch (~0 bytes) for static binaries. Alpine (~5MB) for dynamic ones — but watch for musl libc vs glibc subtleties. Wolfi/Chainguard images for the "distroless-plus-package-management" middle ground.

No shell. Distroless images have no shell. An attacker who lands a code execution can't easily pivot — there's nothing to exec into. (You also lose the ability to kubectl exec for debugging; the trade is usually worth it. Use ephemeral debug containers in K8s when needed.)

Non-root user. USER 65532:65532 before ENTRYPOINT. Most distroless images ship a :nonroot variant that does this for you.

No setuid binaries. Scan for them: find / -perm -4000 -type f 2>/dev/null. Remove or chmod them.

Drop write permission on app files. The app shouldn't be able to modify its own binary at runtime.

Runtime hardening (the pod / container spec)

securityContext:
  runAsNonRoot: true
  runAsUser: 65532
  runAsGroup: 65532
  readOnlyRootFilesystem: true      # mount /tmp via emptyDir if needed
  allowPrivilegeEscalation: false
  capabilities:
    drop: ["ALL"]                     # drop ALL capabilities
    add: []                              # add only what's needed
  seccompProfile:
    type: RuntimeDefault                # or Localhost with custom profile

# And at pod level
spec:
  automountServiceAccountToken: false  # if pod doesn't need K8s API
  securityContext:
    fsGroup: 65532
    seccompProfile: { type: RuntimeDefault }

readOnlyRootFilesystem. The biggest single hardening win. Combined with emptyDir volumes for paths that actually need writes (typically /tmp, /var/run), this means most exploits — which need to write tools to disk — fail.

Drop capabilities. Default container capabilities include things like NET_BIND_SERVICE, SETUID, CHOWN that most apps don't need. Drop all and add only what's required.

seccomp. Restricts which Linux syscalls the container can make. RuntimeDefault blocks dangerous ones (kernel exploits, container escapes) and is good for 95% of workloads. A custom Localhost profile lets you tighten further.

Pod Security Standards (PSS). Set restricted on every namespace that doesn't have a specific need otherwise. K8s enforces it cluster-wide via the built-in Pod Security admission plugin.

Vulnerability scanning

Three categories of issues to catch:

OS packages with CVEs. Trivy, Grype, Clair scan the image's apt/apk database against known-vulnerability lists. Run in CI; block on high/critical.

Application dependencies. Same scanners can detect Python/Node/Go packages with CVEs. Pair with npm audit / pip-audit / govulncheck.

Image misconfiguration. Trivy and others check for "runs as root," "no health check," "uses ADD with URL," and similar policy violations.

# Scan and fail build on HIGH or CRITICAL CVEs
trivy image \
  --severity HIGH,CRITICAL \
  --exit-code 1 \
  --ignore-unfixed \
  --format sarif --output trivy.sarif \
  myorg/myapp:$TAG

# Also scan the Dockerfile for misconfigs
trivy config --severity HIGH,CRITICAL Dockerfile
The principal anti-pattern

Adding tools to the production image "in case someone needs to debug." curl, netcat, bash, vim, the cluster's view of /etc/passwd — every one of these is a pivot tool for an attacker. Use ephemeral debug containers (kubectl debug) when you need a shell; never bake them into the production image.

HARDEN Chapter V

Supply chain · SBOM, sign, verify.

The threat Supply-chain attacks (compromised base image, typo-squatted dep, malicious commit) bypass code review. The defense is cryptographic provenance.

SLSA Supply-chain Levels for Software Artifacts. A 0–4 maturity model for build provenance.

Supply-chain security in 2026 has three pillars: SBOMs tell you what's in your image; signatures tell you who built it and that it hasn't been tampered with; provenance attestations tell you exactly how it was built. All three attach to the image in the registry and are verifiable in CI/CD and at deploy time.

SBOM (Software Bill of Materials)

An SBOM is a structured list of every package, library, and file in your image — OS packages from the base, language deps from your install steps, and your application's own files. Two standards dominate: SPDX (Linux Foundation) and CycloneDX (OWASP). Both work; CycloneDX is more popular for containers.

# Inline in the build — BuildKit calls syft scanner
docker buildx build \
  --sbom=true \                          # attach SBOM as in-toto attestation
  --provenance=mode=max \                # attach full SLSA provenance
  -t myorg/myapp:$TAG --push .

# Or after-the-fact with Syft
syft myorg/myapp:$TAG \
  -o cyclonedx-json=app.cdx.json \
  -o spdx-json=app.spdx.json

# Inspect what's in there
docker buildx imagetools inspect myorg/myapp:$TAG \
  --format "{{json .SBOM}}" | jq .

Sign with cosign

Cosign (Sigstore) is the dominant container-signing tool in 2026. It signs images, attaches signatures as registry artifacts, and integrates with Fulcio (a free CA that issues short-lived certs via OIDC, e.g., GitHub Actions identity) so you don't even need to manage long-lived keys.

# Keyless signing (preferred for CI) — uses OIDC identity from GitHub Actions
cosign sign \
  --identity-token $ACTIONS_ID_TOKEN_REQUEST_TOKEN \
  myorg/myapp:$TAG

# Attach the SBOM as an attestation
cosign attest --predicate app.cdx.json \
  --type cyclonedx \
  myorg/myapp:$TAG

# At deploy time / pull time — verify the signature
cosign verify \
  --certificate-identity-regexp="^https://github.com/myorg/myrepo/" \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  myorg/myapp:$TAG

# Verify the SBOM attestation
cosign verify-attestation \
  --certificate-identity-regexp="^https://github.com/myorg/" \
  --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
  --type cyclonedx \
  myorg/myapp:$TAG

SLSA provenance

Provenance is the build's autobiography: which CI ran it, which Git commit, which Dockerfile, which builder, what materials were consumed. BuildKit's --provenance=mode=max embeds a SLSA-format attestation in the image. SLSA defines levels 1–4 of provenance rigor; most teams target Level 3 (signed provenance from a hardened, isolated builder).

Admission enforcement

Generating signatures isn't enough — you need K8s to refuse to run unsigned images. Three deployable options:

Kyverno — policy engine; verifyImages rule checks cosign signatures at admission. Most flexible.

OPA / Gatekeeper — generic policy; you write the cosign check yourself.

Sigstore Policy Controller — purpose-built for cosign verification. Simplest.

apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: require-signed-images
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-image-signatures
      match:
        any:
          - resources:
              kinds: [Pod]
      verifyImages:
        - imageReferences: ["myorg/*"]
          attestors:
            - entries:
                - keyless:
                    issuer: https://token.actions.githubusercontent.com
                    subject: "https://github.com/myorg/*"
The minimum-viable supply chain

If you do nothing else: (1) pin base images by digest, (2) generate SBOMs with --sbom=true in CI, (3) sign with cosign using keyless GitHub OIDC, (4) verify at admission with Kyverno. Four steps; each costs minutes once set up; together they meet SLSA Level 3.

GPU Chapter VI

CUDA & GPU containers.

The three layers Driver on host. CUDA + cuDNN in image. App on top. Mismatch any pair and the container fails to start or crashes at first kernel launch.

NVIDIA's stack Container Toolkit on the host; CUDA base images for the build; NGC for vetted ML containers.

GPU containers are the same as CPU containers except for one critical difference: the host's NVIDIA driver must be compatible with the container's CUDA version. The NVIDIA Container Toolkit handles the runtime plumbing — mounts the driver, libraries, and device files into the container — but you, the image builder, choose the CUDA version and architecture compute capabilities.

The three layers

Host: NVIDIA driver. Installed on the bare-metal node. Version matters: a CUDA 12.x runtime needs at least driver 545.x. The driver is forward-compatible — a newer driver can run older CUDA — but never the other way around.

Host: Container Toolkit. The nvidia-container-runtime hook that detects nvidia.com/gpu requests, mounts /dev/nvidia* device files, and injects driver libraries (libcuda.so) into the container. Installed by the NVIDIA GPU Operator (covered in Issue 01).

Container: CUDA toolkit + cuDNN + your app. The image you build. CUDA libraries are baked in; the driver is injected at runtime.

The CUDA base image hierarchy

# From smallest to largest
nvcr.io/nvidia/cuda:12.6.0-base-ubuntu22.04        # ~190MB — runtime only, no toolkit
nvcr.io/nvidia/cuda:12.6.0-runtime-ubuntu22.04     # ~2GB — cuBLAS, NCCL, cuDNN runtimes
nvcr.io/nvidia/cuda:12.6.0-devel-ubuntu22.04       # ~5GB — nvcc, headers, all libs
nvcr.io/nvidia/cuda:12.6.0-cudnn-runtime-ubuntu22.04
nvcr.io/nvidia/cuda:12.6.0-cudnn-devel-ubuntu22.04

# For applications (no PyTorch/TF):
#   build stage: -devel    runtime stage: -runtime
# For training/inference:
#   prefer NGC PyTorch / TensorFlow containers (pre-tuned)
nvcr.io/nvidia/pytorch:24.10-py3                  # ~9GB — PyTorch + TRT + Apex + ...
nvcr.io/nvidia/tritonserver:24.10-py3            # Triton Inference Server
nvcr.io/nvidia/tensorrt:24.10-py3                 # TensorRT-LLM build base

The canonical CUDA Dockerfile

# syntax=docker/dockerfile:1.7

# ─── BUILD STAGE (devel image; large but only used at build) ────
FROM nvcr.io/nvidia/cuda:12.6.0-cudnn-devel-ubuntu22.04@sha256:... AS build
WORKDIR /work

# Install Python toolchain (cache mounted)
RUN --mount=type=cache,target=/var/cache/apt \
    --mount=type=cache,target=/var/lib/apt \
    apt-get update && apt-get install -y --no-install-recommends \
      python3.11 python3-pip git

# Build wheels (e.g., custom kernels) with target compute capabilities
ENV TORCH_CUDA_ARCH_LIST="8.0;8.6;8.9;9.0;10.0"     # A100, RTX30, H100, B200
COPY . .
RUN --mount=type=cache,target=/root/.cache/pip \
    pip wheel --wheel-dir=/wheels -r requirements.txt
RUN python3 setup.py bdist_wheel -d /wheels

# ─── RUNTIME STAGE (smaller runtime image) ──────────────────
FROM nvcr.io/nvidia/cuda:12.6.0-cudnn-runtime-ubuntu22.04@sha256:...
RUN --mount=type=cache,target=/var/cache/apt \
    apt-get update && apt-get install -y --no-install-recommends \
      python3.11 && rm -rf /var/lib/apt/lists/*

COPY --from=build /wheels /wheels
RUN pip install --no-index --find-links=/wheels myapp && rm -rf /wheels

# Drop privileges (non-root user)
RUN useradd -u 10001 -r -s /sbin/nologin app
USER app

# Required env vars for the NVIDIA Container Toolkit
ENV NVIDIA_VISIBLE_DEVICES=all
ENV NVIDIA_DRIVER_CAPABILITIES=compute,utility

ENTRYPOINT ["python3", "-m", "myapp.serve"]

NGC: the curated registry

NVIDIA GPU Cloud (NGC) is a public registry of GPU-optimized containers. For training and inference, NGC's PyTorch / TensorFlow / TensorRT-LLM containers are typically faster than rolling your own — they bundle proprietary optimizations (Apex, transformer_engine, NCCL tuning, NVTX hooks) that take weeks to replicate.

Pin NGC tags by digest in production. Tags like 24.10-py3 are stable, but always re-resolve to a digest in your Dockerfile for reproducibility.

The CUDA version compatibility rule

If the host driver is X.y, the container can run any CUDA toolkit ≤ X.y. NVIDIA also publishes CUDA Forward Compatibility packages that let newer CUDA toolkits run on older drivers within a major version family — useful when you can't upgrade the host driver but need newer CUDA features. Verify host vs container compatibility with nvidia-smi inside the container before debugging anything else.

GPU Chapter VII

NVIDIA NIM · the microservice.

What NIM is A pre-optimized inference container per model and GPU SKU. Pull, run, get an OpenAI-compatible API. The fastest path from "I want to serve this model" to production.

When to use it When the model is in the NIM catalog. When NVIDIA GPUs are the target. When the 2× throughput vs. vanilla vLLM is worth the AI Enterprise license cost.

A NIM is a container that bundles a foundation model, a GPU-specific inference engine (TensorRT-LLM, vLLM, or SGLang), and an OpenAI-compatible HTTP API. The whole stack is tuned by NVIDIA for the specific model + GPU pair. Pull the image, run with a GPU, point your app at the endpoint — done in minutes, not weeks.

What's inside a NIM container

The architecture, from NVIDIA's docs: NIM LLM is an enterprise orchestration layer for vLLM (or TRT-LLM / SGLang depending on the model). It packages an inference engine into a production-ready container with curated model profiles, validated configurations, and enterprise features such as health management, observability, and security hardening. The actual layout:

Inside a NIM container · the four components
nginx proxy (8000)
~10 MB
External port. Handles TLS termination, CORS, immediate liveness probes, model-aware readiness routing. Returns 503 while model loads; 200 once ready.
Inference engine (8001, loopback)
~200 MB
vLLM, TensorRT-LLM, or SGLang — selected based on model + GPU. Listens on localhost only; proxied by nginx. OpenAI-compatible API.
Model weights + engine cache
10–500 GB
Stored at NIM_CACHE_PATH. Engine compilation (TensorRT-LLM) happens on first launch and is cached for subsequent starts. Mount as PVC for warm restarts.
Process supervisor + observability
~50 MB
Fail-fast supervision: if either process exits, container shuts down for K8s to reschedule. Prometheus metrics at /v1/metrics. Structured JSON logs.
The key abstraction NIM hides the inference engine selection and tuning. You see an OpenAI-compatible API; NIM picks vLLM / TRT-LLM / SGLang for the model+GPU combination, downloads the right pre-compiled engine, and exposes it. The cost is an NVIDIA AI Enterprise license and lock-in to NVIDIA-architecture GPUs.

The NIM deployment pattern

# 1. NGC login (one-time)
helm repo add nim https://helm.ngc.nvidia.com/nim
helm repo update

# 2. Install with API key (NGC org)
helm install llama3-70b nim/meta-llama-3-3-70b-instruct \
  --namespace nim --create-namespace \
  --set image.repository=nvcr.io/nim/meta/llama-3.3-70b-instruct \
  --set image.tag=1.0.0 \
  --set model.ngcAPIKey=$NGC_API_KEY \
  --set persistence.enabled=true \                  # PVC for cache
  --set persistence.size=200Gi \
  --set resources.limits."nvidia\.com/gpu"=2 \
  --set hostNetwork=false

# 3. Hit the OpenAI-compatible API
kubectl port-forward -n nim svc/llama3-70b 8000:8000
curl http://localhost:8000/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model": "meta-llama-3-3-70b-instruct", "messages": [...]}'

Engine compilation & the cache mount

For TensorRT-LLM-backed NIMs, the first launch compiles the model into a hardware-specific engine — this can take 10–30 minutes for a 70B model. The engine file is then cached in NIM_CACHE_PATH (typically /opt/nim/.cache) so subsequent launches are fast.

Always mount a PVC at the cache path. Without it, every pod restart re-compiles. With it, restarts are 1–2 minutes. For high-availability deployments, the PVC must support ReadWriteMany (so multiple replicas share) or you use one PVC per replica (independent caches).

Tuning NIMs in production

NIM exposes most of its tuning surface as environment variables. The high-leverage ones:

env:
  # Concurrency and batching
  - name: NIM_MAX_BATCH_SIZE
    value: "256"
  - name: NIM_KV_CACHE_FREE_GPU_MEM_FRACTION
    value: "0.9"           # reclaim from default 0.85

  # Engine profile (select pre-compiled variant for your GPU)
  - name: NIM_MODEL_PROFILE
    value: "tensorrt_llm-h100-fp8-tp2-throughput"

  # Disable telemetry if compliance requires it
  - name: NIM_TELEMETRY_ENABLED
    value: "false"

  # Observability
  - name: NIM_LOG_LEVEL
    value: "INFO"
  - name: NIM_ENABLE_PROMETHEUS_METRICS
    value: "true"

The NIM_MODEL_PROFILE setting is the biggest performance lever. NIM ships multiple pre-compiled engine variants per model — different tensor-parallelism sizes, different quantization (FP8 vs FP16), different batch profiles. Pick the one that matches your GPU SKU and traffic shape; the wrong profile costs 30–50% throughput.

Build-once, promote-by-digest

The cardinal rule: the artifact that runs in production is the same artifact that ran in staging is the same artifact that ran in dev. The artifact's identity is its content digest (sha256:...), not its tag. Tags float; digests don't.

# BAD — rebuild for staging
docker build -t myapp:staging-v1.2.3 .
docker push myapp:staging-v1.2.3

# GOOD — promote the dev image by digest
DIGEST=$(crane digest ghcr.io/myorg/myapp:v1.2.3)
crane tag ghcr.io/myorg/myapp@$DIGEST staging-v1.2.3
crane tag ghcr.io/myorg/myapp@$DIGEST prod-v1.2.3
# All three tags point at the same bytes

Cache strategies that scale

GitHub Actions cache (cache-to: type=gha) is fine for small projects. For larger ones it has 10GB total cache limit and slow read speeds.

Registry-based cache (cache-to: type=registry,ref=...) stores BuildKit cache as a separate manifest in your container registry. Unlimited size, fast pulls, works across CI providers.

S3 / GCS cache (BuildKit S3 cache backend) for the biggest builds — multi-GB CUDA images benefit substantially.

The PR-preview pattern

Every PR builds and signs an image tagged pr-NNN. The image gets deployed to an ephemeral namespace via ArgoCD's ApplicationSet PR generator. QA, design review, and load tests happen against real, deployed code. When the PR merges, the same image (by digest) gets promoted to staging, then prod. The image you reviewed is bit-for-bit identical to the image that ships.

DEPLOY Chapter IX

Runtime · probes, resources, signals.

The three contracts Probes (am I healthy?), Resources (what do I need?), Signals (how do I shut down?). Get these wrong and reliability collapses regardless of how good the image is.

For ML Startup probes matter. A model can take 5+ minutes to load — the cluster needs to know it's loading vs. broken.

A container's runtime contract with the orchestrator has three parts: probes declare health, resources declare needs, signals declare lifecycle. The orchestrator (K8s) uses these to decide when to start, route, scale, restart, or terminate. Get them right and the cluster takes care of you. Get them wrong and reliability suffers regardless of how good the application code is.

Probes — three flavors, three meanings

Startup probe. "Are you still loading?" Runs until it succeeds, then never again. Liveness and readiness probes don't run until startup probe passes. Critical for slow-starting services (model loading, JIT compilation, big PVCs). Without it, K8s kills the pod 30 seconds in because liveness times out before the model has loaded.

Liveness probe. "Are you in a stuck state?" If this fails, K8s restarts the container. Use sparingly — a flaky liveness probe causes restart storms. Generally only set this for services that can deadlock (a custom inference engine with a known wedge bug) and not for services that can naturally be slow (a database, an inference engine doing a big batch).

Readiness probe. "Should I receive traffic right now?" If this fails, K8s removes the pod from Service endpoints but doesn't restart it. Use this for "I'm temporarily overloaded" or "I'm draining for shutdown" states. The most important probe in practice.

containers:
  - name: inference
    startupProbe:                          # up to 30 min for model load
      httpGet: { path: /v1/health/ready, port: 8000 }
      periodSeconds: 10
      failureThreshold: 180                 # 180 * 10s = 30 min
    readinessProbe:                        # traffic routing
      httpGet: { path: /v1/health/ready, port: 8000 }
      periodSeconds: 5
      failureThreshold: 2                   # 10s to remove from endpoints
    livenessProbe:                         # restart only on hard wedge
      httpGet: { path: /v1/health/live, port: 8000 }
      periodSeconds: 30
      failureThreshold: 5                   # 150s before killing

Resources — requests, limits, and QoS

Three QoS classes, in increasing protection from eviction:

BestEffort (no requests, no limits): killed first when the node is under pressure. Acceptable only for non-critical batch work.

Burstable (requests < limits): can use spare capacity but is killable. The default for most web services.

Guaranteed (requests == limits, all resources): protected from eviction. Required for training and inference pods. Required for kubelet's CPU/memory/topology managers to engage (see Issue 03 Case 03).

resources:
  requests:
    cpu: "16"
    memory: 128Gi
    nvidia.com/gpu: "2"                  # 2 H100 for 70B model
    ephemeral-storage: 50Gi              # for /tmp, logs
  limits:                                # MUST equal requests for Guaranteed QoS
    cpu: "16"
    memory: 128Gi
    nvidia.com/gpu: "2"
    ephemeral-storage: 50Gi

Signals — graceful shutdown

When K8s decides to stop a pod (deployment rollout, scale-down, eviction), the lifecycle is:

1. Pod removed from Service endpoints (no new traffic).
2. preStop hook executes (sync) — your chance to drain.
3. SIGTERM delivered to PID 1.
4. Wait terminationGracePeriodSeconds (default 30s).
5. SIGKILL if still running.

For inference services, "draining" means: stop accepting new requests but finish in-flight ones. The application must trap SIGTERM and do this:

import signal, asyncio
shutting_down = False

def handle_sigterm(signum, frame):
    global shutting_down
    shutting_down = True                                   # readiness probe now fails

signal.signal(signal.SIGTERM, handle_sigterm)

async def readiness():
    return {"ready": not shutting_down}

# And in K8s pod spec:
spec:
  terminationGracePeriodSeconds: 120           # long enough to drain
  containers:
    - lifecycle:
        preStop:
          httpGet: { path: /shutdown, port: 8000 }
          # Or exec: { command: ["sleep", "10"] } to give LB time to notice

What goes wrong, specifically

Pod restarts every N minutes. Liveness probe too aggressive. Loosen the failureThreshold or remove liveness entirely.

504s during deploy. No graceful shutdown. Pods get killed mid-request. Add preStop, terminationGracePeriodSeconds, and SIGTERM handling.

Pod stuck Pending. Resource requests too large for any node. Or no nodes with the right GPU label. Or no GPU device plugin.

"GPU not found" inside container. Missing NVIDIA Container Toolkit, missing NVIDIA_VISIBLE_DEVICES=all, or device plugin not running. Check nvidia-smi inside the pod first.

OOMKilled, no warning. Memory limit too tight. The kernel doesn't ramp pressure for containers — it just oom-kills at the limit. Add headroom or use memory.high via cgroups v2 for graceful degradation.

The most under-used lever

Tune the startup probe. Most teams don't set one, then panic-fix when their first 70B model takes 8 minutes to load and K8s starts oom-killing pods. Set startupProbe.failureThreshold = 180 with periodSeconds: 10 = 30 minutes for the model to load before anyone gets upset. Costs nothing; prevents a class of outages.

DEPLOY Chapter X

Disaggregated inference at scale.

The pattern Separate prefill workers (compute-bound) from decode workers (memory-bound). Ship the KV cache between them over RDMA. Independently scale each pool.

The tooling NVIDIA Dynamo on K8s. NIXL for the data transfer. KV-aware routing for cache reuse.

Disaggregated inference is the production answer to the prefill/decode interference problem (see Issue 03 Chapter VI). Instead of running both phases on the same GPU pool, you run separate worker pools and ship the KV cache between them when a request transitions from prefill to decode. Done right: tighter TTFT, tighter TPOT, higher goodput. Done wrong: KV transfer becomes the bottleneck.

Architecture · Disaggregated inference with Dynamo
The five components of a disaggregated cluster

How a request flows

1. Client to Frontend. Request hits Dynamo's frontend at /v1/chat/completions. Standard OpenAI-compatible API.

2. KV-aware routing. The router computes an "overlap score" between the incoming request and KV cache blocks across the cluster. Routes to a worker that already has the relevant prefix cached if possible — saving compute on the prefill side. As Issue 03 Case 08 demonstrated, prefix caching can yield 90%+ savings on RAG workloads.

3. Prefill phase. The chosen prefill worker processes the prompt, producing a KV cache for the entire input. This is compute-bound (matmul-dominated) — the prefill pool is tuned for throughput, often with smaller tensor parallelism but more replicas.

4. KV cache transfer. NIXL ships the KV cache from the prefill worker to a decode worker. This is where the architecture lives or dies. Without RDMA, expect 200-500x performance degradation in Time To First Token — observed ~98s TTFT with TCP vs ~200-500ms with RDMA. Production deployments require InfiniBand or RoCE.

5. Decode phase. The decode worker generates tokens one at a time. This is memory-bandwidth-bound — decode pools are tuned for memory bandwidth, often with larger tensor parallelism (TP=8) to fit big batches of long-context conversations.

6. KV offload (when needed). When GPU memory pressure rises, the Dynamo KV Block Manager offloads cooler cache blocks to CPU RAM, then NVMe, then object storage. NIXL handles every tier transparently.

The container architecture

Each worker in this picture is a container. The deployment has at least:

  • frontend Deployment (Dynamo's request router + KV-aware scheduler)
  • planner Deployment (SLO monitor, autoscaler)
  • prefill-worker StatefulSet (each pod has GPU(s); requires RDMA NIC; tuned for throughput)
  • decode-worker StatefulSet (each pod has GPU(s); requires RDMA NIC; tuned for memory)
  • etcd StatefulSet (worker registration and KV-block discovery)
  • Optional: kv-cache-store StatefulSet (CPU-RAM and NVMe offload nodes, separate from GPU workers)
deployment:
  model: deepseek-ai/DeepSeek-R1
  backend: trtllm

prefill:
  replicas: 4
  parallelism:
    tensorParallelism: 2             # 2× H100 each, throughput-tuned
  resources:
    limits:
      nvidia.com/gpu: "2"
      rdma/hca: "1"                  # required for NIXL
      memory: 256Gi

decode:
  replicas: 2
  parallelism:
    tensorParallelism: 8             # 8× H100 each, memory-tuned
  resources:
    limits:
      nvidia.com/gpu: "8"
      rdma/hca: "1"
      memory: 512Gi

router:
  kvAware: true
  prefixCacheBlockSize: 16           # tokens per cache block

planner:
  sloTargets:
    ttftP95Ms: 800
    tpotP95Ms: 40
  scaleInterval: 30s

kvBlockManager:
  tiers:
    - { type: hbm, gb: 80 }
    - { type: cpu, gb: 256 }
    - { type: nvme, path: /mnt/kv-cache, gb: 2000 }
    - { type: s3, bucket: kv-cache-cold, region: us-west-2 }

Why this is operationally complex

Disaggregation adds new failure modes to handle:

RDMA must work. If the RDMA fabric flaps, KV transfers fail, TTFT spikes 200×. Treat IB health monitoring (Issue 02 Chapter VII) as critical infrastructure.

Prefill and decode pools must scale together. Too many decode workers with too few prefills → TTFT cliff. Too many prefills with too few decoders → wasted prefill compute. The Dynamo Planner does this dynamically; without it you tune by hand.

KV cache eviction is now a cluster concern. A request's KV may be on a different GPU than the one currently serving it. Block manager logic, eviction policies, and tier promotion/demotion all matter.

Routing decisions affect cache hit rate. The router's "overlap score" is approximate; bad routing means cache miss and recomputation. Monitor cache hit rate as a primary metric.

When to disaggregate

The architecture pays back when: (a) traffic is mixed (long prompts + short prompts), (b) TTFT and TPOT SLOs are both tight, (c) you have 100+ GPUs to amortize the operational complexity, (d) you have RDMA fabric. Below that scale, aggregated serving with chunked prefill (Issue 03 Chapter VI) is simpler and competitive.

DEPLOY Chapter XI

Scaling to thousands of CPUs & GPUs.

The transitions Three breakpoints where the architecture has to change qualitatively. Know which breakpoint you're past.

The discipline Most "fix this at scale" problems are actually "stop scaling the wrong thing" problems.

Scale-out doesn't break smoothly. Specific architectural choices fail at specific sizes, and the failures are usually about the parts you forgot rather than the parts you optimized. Three breakpoints to watch.

Breakpoint 1 — Image distribution (past ~200 nodes)

Pulling a 50GB CUDA image to 1000 nodes from a single registry crushes the registry, the network, and every pod's startup time. The patterns:

Registry mirroring — a Harbor or Zot registry in each region/cluster, configured as a pull-through cache. First pod pulls from the upstream; the rest hit the local mirror. Pull time drops from minutes to seconds.

Lazy / streaming pull — tools like stargz, nydus, or SOCI let containers start before the image is fully pulled. The container reads chunks on demand. For CUDA images where 80% of bytes are libraries that aren't immediately needed, this is a 5–10× cold-start improvement.

Pre-pulling with DaemonSet — a DaemonSet on every node that pulls "expected" images. Less elegant than lazy pull but works on any K8s.

Breakpoint 2 — Scheduling (past ~64 GPU nodes)

The default kube-scheduler is per-pod, greedy, and topology-blind. Past 64 GPU nodes — especially for distributed training — this becomes a serious problem. The three classes of fix (covered in Issue 01 Chapter V):

Gang scheduling: Kueue (lightweight), Volcano (heavyweight), KAI Scheduler (NVIDIA-tuned). Holds the entire job until every pod can be placed. Without this, distributed training partially starts and burns money.

Topology-aware scheduling: nodes labeled with rack, IB switch, NVLink island. Workloads constrain to same-island placement. NCCL bandwidth stays high.

Queue policy: fair-share across teams, preemption from low to high priority, quota enforcement. Without this, a single team consumes the cluster.

Breakpoint 3 — Operational tooling (past ~1000 pods)

Hand inspection of kubectl get pods stops working. You can't grep through ten thousand log files. Specific tools must scale with you:

Metrics — Prometheus with shards or Mimir/Thanos for long-term storage. DCGM exporter for GPU metrics. NIM exposes Prometheus metrics out of the box at /v1/metrics.

Logging — Loki, Elastic, or OpenSearch. Structured JSON logs (the NIM default). Index by namespace, model, region.

Tracing — OpenTelemetry from the application; NIM forwards Traceparent and X-Request-Id headers. Critical for debugging multi-hop inference (router → prefill → decode).

Cost attribution — DCGM metrics include namespace and pod labels. Aggregated by team, multiplied by hourly GPU cost, you get chargeback per team. Without this, multi-tenant clusters fight over resources without context.

The three architectural patterns

Single-pod
One container, one or more GPUs on one node. vLLM with tensor parallelism. Simple, debuggable, low operational overhead. The right answer up to a few GPUs of demand.

The unspoken rule

At every breakpoint, half of teams optimize the wrong thing. Past 200 nodes, they tune the inference engine when they should fix image distribution. Past 64 GPU nodes, they add hardware when they should fix the scheduler. Past 1000 pods, they hire SREs when they should buy a managed metrics product.

The right question at each breakpoint: what's the new failure mode that wasn't there yesterday? Not "what's slower" — what's different. The breakpoints are sharp; the qualitative failure is the signal.
