▸ Probe · likely opener

"Walk me through what happens when I call y = torch.relu(x) on a CUDA tensor."

Answer arc

Python binding → ATen op → the dispatcher computes a DispatchKeySet from x (CUDA, plus Autograd if requires_grad, plus Autocast if enabled) → Autograd key records the node and redispatches → lands on the CUDA kernel → TensorIterator sets up shapes/dtypes → gpu_kernel launches an elementwise CUDA kernel on the current stream, with output memory from the caching allocator. Then name the failure modes: launch overhead if x is tiny (→ graphs/fusion), and the fact that the kernel is queued async so timing needs an event or synchronize(). That single answer touches half this manual — use it to signal range.

Section 01 · Orientation

The stack, end to end

Hold the whole vertical in one picture. The genius of PyTorch's design is that the same operator definition flows through optional cross-cutting layers (autograd, autocast, tracing) before resolving to a device kernel — and the CUDA team owns the bottom third of this diagram.

The five repos-within-the-repo you should name

  1. c10 — "Caffe2 + ATen 10", the core library: TensorImpl, Storage, Device, DispatchKey, ScalarType, and the CUDA runtime (c10/cuda). No autograd, no ops — just the substrate.
  2. ATen — the tensor library and the ~2,000+ operators. native_functions.yaml is its registry.
  3. torch/csrc — the C++ that binds ATen to Python and houses the autograd engine, the JIT, and distributed.
  4. torch (python) — the user-facing package, plus Dynamo/Inductor (torch/_dynamo, torch/_inductor).
  5. functorch / vmap / decompositions — the transform layer that AOTAutograd and Inductor lean on.

Section 02 · Core mechanism

Tensor · TensorImpl · Storage

Before dispatch makes sense, the data model has to. A Tensor in PyTorch is a thin, reference-counted handle; the metadata and the bytes live in two separate objects. Getting this triple right is the foundation for views, aliasing, and why the caching allocator can hand out device pointers safely.

The triple

  • at::Tensor — a c10::intrusive_ptr<TensorImpl>. Copying a Tensor bumps a refcount; it does not copy data. This is why passing tensors by value is cheap.
  • TensorImpl — the metadata: sizes, strides, storage_offset, dtype (ScalarType), device, the DispatchKeySet, autograd metadata (AutogradMeta), and a pointer to the Storage.
  • Storage — a refcounted buffer holding a DataPtr = {raw pointer, Allocator, Device}. For a CUDA tensor the raw pointer is a device address and the allocator is the CUDACachingAllocator.

A view (x.transpose(0,1), x[2:], x.view(...)) makes a new TensorImpl that points at the same Storage with different sizes/strides/offset. No bytes move. That single fact drives the ADInplaceOrView dispatch key (version counters so autograd can detect a mutation through an alias) and a whole class of correctness bugs you should be ready to discuss.

▤ Spec · contiguity, precisely

A tensor is contiguous (in the default sense) when iterating its logical indices in row-major order walks memory with monotonically packed strides — i.e. stride[i] == stride[i+1] * size[i+1] and the innermost stride is 1. Channels-last (NHWC) is a different but equally "contiguous" memory format; cuDNN convolutions are sensitive to it, which is why x.to(memory_format=torch.channels_last) is a real performance lever, not a no-op.

dtype dispatch in kernels

Inside a CUDA kernel you don't write per-dtype branches by hand — you use the AT_DISPATCH_* macros, which switch on the runtime ScalarType and instantiate the templated lambda for the concrete C++ type. This is the bridge between dynamic Python dtypes and static C++ templates.

Section 03 · Core mechanism

The dispatcher

This is the heart of PyTorch and the single most likely deep-dive in a core interview. The dispatcher is, in ezyang's framing, a per-operator table of function pointers indexed by dispatch key — and dispatch keys correspond to the cross-cutting concerns (device, autograd, autocast, tracing) that compose to define an operator's full behavior.

How a key is chosen

Every TensorImpl carries a DispatchKeySet (a 64-bit bitset). On an operator call, the dispatcher unions the key sets of the relevant tensor arguments, applies any thread-local includes/excludes (this is how autocast and no_grad toggle behavior), and selects the highest-priority key still in the set. Priority order is fixed: roughly Autocast > Autograd > ADInplaceOrView > backend (CUDA/CPU). The chosen kernel may do its work and then re-dispatch to the next key down — that's the whole trick.

Where kernels come from: codegen + native_functions.yaml

You almost never call m.impl() by hand for a standard op. Instead you declare the operator and its per-backend kernels in native_functions.yaml, and codegen emits the registration into files like RegisterCUDA.cpp. Structured kernels go further: you write a meta() function (shape/dtype inference + output allocation) once, and an impl() that fills the pre-allocated output, and codegen generates the functional/out/inplace variants for you.

Boxed vs unboxed — the one paragraph that impresses

Eager Python/C++ calls use the unboxed path: templated metaprogramming reads only the tensor arguments to build the key, then calls a concrete function pointer — fast, no allocation. JIT, mobile, and cross-cutting fallbacks (autocast, profiling, a future Lazy backend) use the boxed path: arguments are packed into a stack of IValues so one generic function can handle any operator. PyTorch maintains a bridge (make_boxed_from_unboxed_functor) so a kernel written one way can be called the other. The payoff of boxed fallbacks is binary size: one function instead of one-per-op across 2,000+ ops — which is exactly why mobile cares.

▸ Probe · design judgment

"cuDNN SDPA is just another attention backend. Why is it a separate dispatched op rather than folded into the existing attention op?"

Answer arc

This is a real tension Eddie and albanD debated on the cuDNN-SDPA PR. Adding a backend as its own op grows the operator surface (more ops, more coverage matrix) — "op bloat." The alternative is a composite explicit op that unifies many backends behind one schema (the path batch-norm consolidation took). The pragmatic call on SDPA was to accept the backend tech debt for velocity, because cuDNN-SDPA's support matrix doesn't neatly cover flash/mem-efficient anyway, so unification would be lossy. The mature answer names the tradeoff (surface area & maintainability vs. shipping speed & clean dispatch rules) and notes that consolidation is an ongoing, deliberate effort — not an accident.

Section 04 · Core mechanism

ATen CUDA kernels & TensorIterator

Once dispatch lands on the CUDA key, most pointwise and reduction ops route through TensorIterator — the component that turns "apply this scalar lambda over these tensors" into a launched, vectorized, broadcasting-aware CUDA kernel. If you've read ATen internals, this is where to show it.

What TensorIterator actually does

It is a configuration + execution engine. Given inputs and (optional) outputs it: computes the broadcasted shape; resolves type promotion and the common compute dtype; allocates outputs if absent; collapses adjacent dimensions whose strides allow it (so a contiguous N-D tensor becomes effectively 1-D); decides whether inputs are contiguous and identically-strided (the fast path) or need per-element offset computation; and picks a vectorized launch (float4-style 128-bit loads) when alignment and dtype permit.

The launcher is gpu_kernel (and gpu_kernel_with_scalars for ops with scalar operands). Under the hood it runs a grid-stride loop so any input size maps onto a fixed grid, and it dispatches to launch_vectorized_kernel when the iterator reports contiguity. When input and output dtypes differ, it inserts a dynamic cast per element instead of failing — convenient, but a measurable cost worth naming.

◈ Reductions are a different machine

Sums, norms, softmax-numerators, etc. don't fit the elementwise launcher. ATen has a dedicated reduction infrastructure (Reduce.cuh) doing hierarchical reduction — warp-shuffle within a warp, shared memory across warps, then a grid-level pass — with vectorized loads and an accumulation dtype that's often wider than the storage dtype (e.g. fp32 accumulate for fp16 inputs). This accumulate-in-fp32 detail connects straight to the precision discussion in §12.

✕ Pitfall · the fast path is fragile

Non-contiguous inputs, mixed dtypes, and broadcasting can each knock an op off the vectorized path onto the slower offset-calculating kernel. A common real-world "why is this slow" is a transpose upstream that left a tensor non-contiguous; the elementwise op then reads with strided, uncoalesced access. The fix is usually an explicit .contiguous() (paying one copy) or choosing the right memory_format — and you can confirm the uncoalesced access in Nsight Compute's memory workload analysis. Naming the profiler here is exactly Eddie's wavelength.

▸ Probe · CUDA fundamentals

"Why a grid-stride loop instead of launching exactly N threads?"

Answer arc

Decouples the launch configuration from the problem size: one tuned grid (sized to the device's SM count × occupancy) handles any N, threads are reused across multiple elements, and the kernel stays correct when N exceeds the max grid dimension. It also improves instruction-level parallelism and lets the compiler keep more work in flight. Bonus: it makes the kernel trivially compatible with CUDA-graph capture because the launch geometry is fixed.

Section 05 · Core mechanism

Autograd & autocast as dispatch keys

Both are implemented as keys layered above the device kernel — the cleanest demonstration of why the dispatcher is designed the way it is. Neither knows or cares whether the backend is CPU or CUDA; they intercept, do their slice, and re-dispatch.

Autograd: a tape woven by codegen

For each differentiable op, codegen emits a VariableType kernel registered to the Autograd key. On the forward call it: allocates a backward Node (the grad_fn) wired to the inputs' grad_fns, saves exactly the tensors the backward formula needs (from derivatives.yaml), then re-dispatches with Autograd excluded to run the real forward. loss.backward() hands the leaf nodes to the engine, which topologically walks the DAG and accumulates gradients into .grad.

The version counter on TensorImpl ties this to §02: if a saved tensor is later mutated in place through any alias, the counter advances and autograd raises rather than silently computing a wrong gradient. That is the ADInplaceOrView key earning its keep.

Autocast (AMP): a per-op casting policy

The Autocast key sits above Autograd in priority. When enabled it consults a per-operator policy and casts arguments before redispatching:

PolicyBehaviorExample ops
lower_precision_fpcast inputs to fp16/bf16 — the speed winmm, conv, linear, bmm
fp32force fp32 — numerically sensitivesoftmax, log, norm, sum, losses
promoterun at the widest input dtypecat, stack, addcdiv

It also keeps a cast cache keyed on the source tensor, so a weight that's used many times per iteration is cast fp32→fp16 once, not repeatedly. This is a small but real allocator-and-bandwidth optimization the team cares about.

◆ On the record · why this is Eddie-adjacent

Autocast's cast policy and the precision of fused backends overlap with TF32 and reduced-precision reductions (§12) — and Masaki Kozuki (@crcrpar) on the same team owns much of the AMP/foreach machinery. If asked "how do bf16 and TF32 relate," the crisp distinction is: autocast/bf16 changes the storage dtype of operands; TF32 changes the internal math precision of an fp32 op without changing its storage dtype. They're orthogonal knobs that both trade accuracy for tensor-core throughput.

Section 06 · The CUDA runtime layer

The caching allocator

If there is one component the CUDA core team is defined by, it's c10/cuda/CUDACachingAllocator.cpp. Understand why it exists and you understand most CUDA-side performance and memory behavior in PyTorch. Be ready to draw it.

"Why not just cudaMalloc per tensor?" Because cudaMalloc and cudaFree synchronize the device — and synchronization destroys the one optimization PyTorch's eager mode depends on.

The load-bearing idea: CPU runs ahead of the GPU

PyTorch hides the latency of the Python interpreter and dispatch behind GPU execution: the CPU races ahead, queuing kernels onto a stream, while the GPU chews through earlier work. cudaMalloc/cudaFree introduce implicit synchronization that stalls this pipeline. So the allocator's goal is a steady state where it never has to call into the driver for memory — it requests big blocks once and recycles them internally.

Two levels: segments and blocks

  • Segment — a contiguous region obtained from CUDA (a cudaMalloc, or a virtual mapping with expandable segments). The unit of OS-level allocation.
  • Block — a sub-region within a segment tracking one allocation. On request, the allocator finds a free block ≥ the (rounded-up) size; if it's too big it splits, returning the front and leaving the back as a new free block. On free, it tries to coalesce with immediate neighbors — but only if they're in the same segment and also free.

Allocations are rounded up (to reduce odd-shaped leftovers) and pulled from a small pool (≤ 1 MB) or large pool, and pools are per-stream. The cross-segment no-merge rule is the crux of fragmentation:

Things to be able to say cold

  • Why fragmentation happens: sizes vary across iterations (batch N then N−1), allocations interleave, and freed blocks can't merge across segment boundaries. The June 2026 eager devlog walks concrete lab cases of exactly this.
  • Expandable segments use CUDA's low-level VMM APIs (akin to mmap) to separate the reservation of virtual address space from the mapping of physical pages. Caveat to mention: tensors in expandable segments can't be shared across processes, so data-loader workers may need it off.
  • record_stream is the bridge to §07: if you allocate on stream A and use the tensor on stream B, you must tensor.record_stream(B) so the allocator doesn't recycle the block while B's queued work still reads it. Forgetting this is a classic silent corruption.
  • CUDA graphs use a private memory pool so captured allocations have stable addresses across replays; the allocator checkpoints/restores pool state around capture. Expandable segments were extended to interoperate with CUDA-graph trees.
  • NCCL interop: there's active RFC work so expandable-segment (VMM) buffers can be registered for NCCL user-buffer / NVLS paths — directly relevant to your distributed-comms background.
▸ Probe · the allocator classic

"My job OOMs at iteration 500 even though memory_allocated is well under the GPU's capacity. What's going on and what do you try?"

Answer arc

Reserved ≫ allocated means fragmentation: the allocator holds enough total free memory but no single contiguous block large enough. First confirm with memory_summary() (large reserved-minus-allocated gap) and a memory snapshot. Then: try expandable_segments:True (usually the biggest lever); cap max_split_size_mb so huge blocks aren't carved into unusable slivers; stabilize allocation sizes (fixed batch/seq, pad to buckets); and as a blunt tool, empty_cache() at a safe boundary — noting it forces sync and returns memory to the driver, so it's a steady-state killer, not a fix. Name the profiler/snapshot at each step.

Section 07 · The CUDA runtime layer

Streams, events & the async model

Almost every performance question about PyTorch on GPU reduces to one fact: the Python/C++ host thread runs ahead of the device. Operations enqueue work onto a CUDA stream and return immediately; the GPU drains that queue asynchronously. Understanding where that illusion breaks — and where it must be synchronized — is core-team table stakes.

The model in one paragraph

A CUDA stream is an ordered FIFO of work (kernels, copies). Work on one stream executes in issue order; work on different streams may overlap and has no ordering guarantee unless you create one with an event. When you call y = x.relu(), ATen launches the kernel on the current stream for that device and returns to Python before the GPU has run anything. The result tensor is real, but its contents are a promise. PyTorch keeps the promise consistent by always issuing dependent ops onto the same stream, so program-order correctness is automatic within a stream.

◈ non_blocking=True needs pinned memory to actually be async

A host→device copy is only truly asynchronous if the host buffer is page-locked (pin_memory()). From pageable memory the driver must stage through a pinned bounce buffer and the copy effectively serializes. This is the single most common reason a "async" data pipeline shows no overlap in Nsight Systems.

✕ Pitfall · the hidden synchronization

Many innocuous-looking operations force a device→host sync: print(tensor), tensor.item(), .cpu(), .numpy(), tensor.tolist(), a Python if tensor > 0:, and even nonzero() / .unique() (data-dependent output shapes). Each drains the host's run-ahead and exposes raw kernel latency. In a tight loop this can dominate. The interview-grade move: enumerate these, then say you'd confirm with an nsys timeline showing the host blocked on a cudaStreamSynchronize.

▸ Probe · the async model

"You add a print(loss) inside your training loop and step time doubles. Why, and how do you keep the logging?"

Answer arc

print(loss) calls loss.item(), which forces cudaStreamSynchronize — the host stops running ahead and waits for the whole queued step to finish, so you're now paying full latency every iteration instead of pipelining. Fixes: log every N steps; accumulate losses in a GPU tensor and sync once per epoch; or stash loss.detach() and only call .item() outside the hot loop. The general principle — keep the host ahead of the device — is what the allocator's record_stream and CUDA graphs both exist to protect.

Section 08 · The CUDA runtime layer

CUDA graphs

When a workload is launch-bound — lots of tiny kernels where the CPU can't enqueue fast enough to keep the GPU fed — CUDA graphs collapse an entire sequence of launches into a single replayable unit. This is Eddie's territory: his GTC 2025 talk was specifically about making cuDNN play well with CUDA graphs so the graph doesn't have to be rebuilt.

The mechanism

You capture a stream: instead of executing, CUDA records the topology of every kernel launch, copy, and their dependencies into a graph object. Thereafter a single cudaGraphLaunch replays the whole thing, skipping per-op Python overhead, C++ dispatch, and most driver-side launch cost. For a decoder doing dozens of small kernels per token, replay can be dramatically cheaper than re-issuing them.

The constraints — and why each exists

  • Static shapes & addresses. The graph bakes in pointers and launch geometry. New input tensors at a new address ⇒ the graph reads stale memory. You therefore allocate static input/output buffers once and copy_ each iteration's data in. This is also why the caching allocator gives graphs a private pool (§06): captured allocations must land at reproducible addresses.
  • No CPU synchronization during capture. Anything that syncs the host (an .item(), a data-dependent shape) can't be captured. Data-dependent control flow is out.
  • Warm up first. Before capturing you run a few eager iterations so cuBLAS/cuDNN finish their algorithm selection and the allocator reaches steady state — otherwise the autotuning work gets captured or the addresses churn.
◈ The ergonomic wrappers

You rarely hand-roll the above. torch.cuda.make_graphed_callables(fn, sample_args) wraps a module so capture/replay is transparent and autograd-aware — it captures forward and backward and splices them into the autograd graph. And torch.compile(model, mode="reduce-overhead") turns on Inductor's CUDA Graph Trees, which capture compiled regions and share one memory pool across many graphs so you don't pay a full static allocation per graph. Mentioning both shows you know the manual primitive and the production path.

◆ Eddie connection · cuDNN × CUDA graphs

His GTC 2025 session was about cuDNN gaining first-class CUDA-graph support so a captured graph doesn't need to be rebuilt when cuDNN would otherwise re-plan. The friction: cuDNN historically did internal work (workspace, planning) that wasn't graph-capturable, forcing rebuilds that defeat the point. If graphs come up, pivot to "the hard part is the libraries underneath being capture-safe — cuDNN, the allocator's graph pool, and NCCL all had to be taught to cooperate." That's the systems-level framing he'll recognize.

▸ Probe · when NOT to reach for graphs

"When do CUDA graphs give little or no benefit — or actively hurt?"

Answer arc

They help when you're launch-bound (many small kernels, host can't keep up). They give little when you're compute-bound — large matmuls already saturate the GPU, so launch overhead is noise. They hurt or simply don't apply when shapes vary every step (dynamic sequence lengths, variable batch), when there's data-dependent control flow, or when capture's static-buffer requirement balloons memory because you need many graphs for many shapes. The diagnostic is an nsys timeline: visible host-side gaps between kernels ⇒ launch-bound ⇒ graphs are worth it; wall-to-wall kernels ⇒ look elsewhere.

Section 09 · Eddie's home turf

cuDNN integration

This is the section to over-prepare. Eddie's PyTorch work is concentrated in cuDNN: convolution backends, the cuDNN SDPA path, precision flags, and the unglamorous-but-critical job of keeping cuDNN versions from breaking the build. If you can speak fluently about algorithm selection, the V8 graph API, and version-compat fire drills, you're speaking his language.

What cuDNN is, and where PyTorch plugs it in

cuDNN is NVIDIA's library of hand-tuned deep-learning primitives — convolutions, pooling, normalization, RNN cells, and now fused attention. PyTorch's integration lives under aten/src/ATen/native/cudnn/. The dispatcher (§03) routes, say, aten::convolution to a cuDNN-backed kernel when the tensor is on CUDA, cuDNN is available, and the configuration is supported; otherwise it falls back to a native ATen kernel or another backend. cuDNN is itself one backend among several for ops like convolution and attention.

Algorithm selection: the core idea

For a single convolution, cuDNN exposes multiple algorithms — implicit GEMM, implicit precomputed GEMM, Winograd, FFT-based, and direct — each with different speed/memory tradeoffs that depend on the exact tensor shapes, strides, dtype, and the GPU. PyTorch can either ask cuDNN for a heuristic pick or, if you opt in, empirically time them:

V7 vs V8: the legacy API and the graph frontend

cuDNN's classic ("V7") API exposed fixed entry points per primitive. The modern V8 graph API lets you describe an operation graph (e.g. conv → bias → activation) that cuDNN can fuse and plan as a unit — the same direction of travel as CUDA graphs and Inductor fusion. PyTorch consumes this through cudnn-frontend, NVIDIA's header-only C++ wrapper around the graph API (with Python bindings). Eddie's PR to always build the cuDNN V8 API was part of making this the default integration path; the frontend is also where the cuDNN SDPA graph lives.

✕ Pitfall · determinism vs speed vs memory

cudnn.deterministic = True forbids algorithms that use atomics / non-deterministic reductions, so results are reproducible run-to-run — but the allowed algorithms can be slower and use more workspace. Conversely benchmark = True is not deterministic across processes because the timed winner can vary. Know that these two flags interact, and that "reproducible," "fast," and "low-memory" are three corners you usually can't all have.

▸ Probe · the benchmark flag

"A teammate sets torch.backends.cudnn.benchmark = True and their detection model gets slower. Diagnose it."

Answer arc

Detection models often run variable input resolutions (or variable batch). The benchmark cache is keyed by the full conv configuration, so every new shape is a miss that triggers a fresh timing search over all candidate algorithms — and that search cost recurs instead of amortizing. The fix: only enable benchmark when input shapes are fixed (or bucket/pad to a small set of shapes); otherwise leave it off and rely on cuDNN's heuristics, or cap the search with the user-specified benchmark limit. Confirm with a profile showing repeated algo-search regions, or by logging cache misses.

Section 10 · Eddie's home turf

SDPA & backend dispatch

scaled_dot_product_attention is the cleanest example in PyTorch of one operator, many backends, chosen at runtime. It's also directly Eddie-adjacent: he built the cuDNN attention backend. Knowing how the selector works — and how it can silently pick the slow path — is a high-value talking point.

One op, four backends

A single call to F.scaled_dot_product_attention(q, k, v, ...) can be served by any of four implementations, in descending preference: FlashAttention, memory-efficient attention, the cuDNN backend, and a pure-ATen math fallback. Each fused backend avoids materializing the full N×N attention matrix (the math path does materialize it — correct, but O(N²) memory). The selection logic lives in aten/src/ATen/native/transformers/cuda/sdp_utils.cpp.

◆ War story to internalize · the cuDNN-SDPA stride bug

The cuDNN attention backend assumed the backward's incoming gradient dO shared the same strides as the forward output O. When an upstream op produced a differently-strided dO, the kernel read it wrong — a correctness bug fixed by a cuDNN 9.5.1 bump (§09's #137978). The lesson, and a great thing to say out loud: fused backends bake in layout assumptions, so stride/contiguity correctness is as important as the math. It also explains why a version bump can be a genuine bugfix, not housekeeping.

◈ The static-order limitation (#138907)

Because priority_order returns one global ranking, it can't be optimal for every (shape, head-dim, GPU) combination — the fastest backend for short sequences may differ from long ones. This is an open design tension, which makes it a fantastic discussion topic: it shows you read the dispatch code and can reason about where its design is imperfect. Floating a per-shape heuristic or autotuned selection is exactly the kind of contribution this team makes.

▸ Probe · the silent fallback

"Your attention layer OOMs on long sequences even though you 'use FlashAttention.' What happened and how do you prove it?"

Answer arc

Almost certainly the selector fell through to the math backend — some constraint (an unsupported attn-mask shape, a head dimension over the limit, a misaligned/odd stride, an unsupported dtype) disqualified all fused backends, so it materialized the full N×N score matrix → O(N²) memory → OOM. Prove it by wrapping the call in sdpa_kernel([SDPBackend.FLASH_ATTENTION]): if it now raises instead of OOMs, the default was silently downgrading. Then fix the disqualifying condition (mask layout, pad head_dim, make contiguous) so a fused backend qualifies. Naming the support matrix and the math-path memory cost is the whole answer.

Section 11 · Eddie's home turf

cuBLAS / cuBLASLt & matmul

Every linear layer, attention projection, and embedding matmul bottoms out in a GEMM served by cuBLAS or its lightweight successor cuBLASLt. The integration questions here mirror cuDNN's: algorithm/heuristic selection, workspace, precision flags, and epilogue fusion.

The routing

torch.mm, torch.bmm, torch.matmul, and torch.addmm dispatch (CUDA key) into aten/src/ATen/native/cuda/Blas.cpp, which calls the thin wrappers in aten/src/ATen/cuda/CUDABlas.cpp. Classic cuBLAS gives you the standard BLAS-3 GEMM. cuBLASLt ("Lt" = lightweight) is a newer API that exposes algorithm variants, better tensor-core targeting, flexible layouts, and crucially an epilogue stage that can fuse a bias add and an activation directly into the GEMM — so an nn.Linear followed by a GELU can become a single fused kernel instead of three.

OpComputesBackend note
mm(A,B)A·Bplain GEMM
addmm(b,A,B)β·b + α·(A·B)nn.Linear's fast path; bias fuses into the Lt epilogue
bmm(A,B)batched A·Bbatched GEMM; attention's QKᵀ and ·V
matmulbroadcastedreshapes to mm/bmm under the hood
◈ Epilogue fusion is the headline feature

cuBLASLt's epilogue can fold bias, ReLU/GELU, and sometimes the auxiliary outputs needed for the backward, into the matmul itself. That removes extra kernel launches and, more importantly, extra round-trips of the output tensor through global memory. When you say "the Lt path fuses the linear's bias and activation into the GEMM epilogue," you're naming the exact reason modern transformer MLP blocks are fast.

✕ Pitfall · layout and size surprises

Two recurring GEMM gotchas: (1) non-contiguous / awkwardly-transposed operands can force cuBLAS into a slower path or an implicit copy — the leading dimension and transpose flags matter, and a stray .contiguous() upstream sometimes fixes a slow matmul. (2) Many small GEMMs are launch-bound, not compute-bound — the fix is batching (bmm), fusing, or CUDA graphs (§08), not a "faster" GEMM algorithm. As always, Nsight Compute tells you which regime you're in (tensor-core utilization, achieved occupancy).

▸ Probe · addmm vs mm

"Why does nn.Linear call addmm instead of mm then a bias add?"

Answer arc

addmm expresses "bias + x·Wᵀ" as one operation, so the cuBLASLt epilogue fuses the bias add into the GEMM — no separate elementwise kernel, no extra read/write of the output through global memory. Doing mm then + bias launches a second kernel and re-streams the whole activation tensor, which for a memory-bound bias add is pure overhead. The same epilogue mechanism extends to fusing the activation. It's a clean example of why PyTorch maps a high-level layer onto the lowest-level fused primitive available.

Section 12 · Eddie's home turf

TF32 & the fp32_precision migration

Precision flags are squarely Eddie's domain, and they just went through a breaking API redesign in PyTorch 2.9 (Oct 2025). Knowing both the old and new APIs — and the regression the migration caused — signals you track the codebase as it is today, not as a tutorial from two years ago.

What TF32 actually is

TF32 ("TensorFloat-32") is a tensor-core math mode for fp32 inputs: it keeps fp32's 8-bit exponent but truncates the mantissa to 10 bits (≈19 bits total internally), so tensor cores run the multiply far faster at the cost of a little mantissa precision. Storage stays fp32 — this is the key distinction from bf16/fp16, which reduce the stored dtype. TF32 affects how fp32 matmuls and convolutions are computed, not how tensors are stored. For most training it's a large speedup with negligible quality impact; for precision-sensitive numerics it can matter.

◆ Keep this distinction crisp (it's a classic interview trap)

TF32: fp32 storage, reduced-mantissa compute on tensor cores, controlled by precision flags. AMP / bf16 / fp16: genuinely reduced-precision storage chosen by autocast (§05). A tensor can be fp32 and still be multiplied in TF32. Conflating "I'm using TF32" with "I'm in mixed precision" is exactly the kind of slip a precision-focused interviewer listens for.

The API, old → new

The pre-2.9 controls were a handful of coarse, awkwardly-named globals. PyTorch 2.9 replaced them with a hierarchical, per-backend, per-op system so you can, say, allow TF32 for cuDNN convolutions but force IEEE fp32 for matmuls.

The mental model for the new names: fp32_precision = "ieee" means "compute fp32 ops in true IEEE fp32"; "tf32" means "you may use the TF32 tensor-core path." Granularity is the whole point — previously one switch governed all matmuls; now matmul, conv, and RNN are independent knobs under their respective backends.

✕ Pitfall · the migration shipped a regression

The redesign wasn't free: FlexAttention changed its effective default from IEEE to TF32 during the transition, a measurable precision/behavior regression tracked in #161022. The takeaway to voice: cross-cutting precision-default changes are dangerous precisely because they're invisible — nothing errors, results just shift — so they demand careful per-op defaults and regression tests. That is exactly the kind of subtle, high-blast-radius change this team has to get right.

→ RFCs #121791, #76440 · doc set_float32_matmul_precision · the flag plumbing lives in aten/src/ATen/Context.cpp
▸ Probe · precision, current events edition

Answer arc

Then the news: pre-2.9 you toggled coarse globals (allow_tf32, set_float32_matmul_precision); 2.9 deprecated those for a granular fp32_precision API set per backend and per op (cuda.matmul, cudnn.conv, cudnn.rnn) taking "ieee" or "tf32". Section 13 · Compilation & lifecycle

torch.compile / Inductor

The CUDA team doesn't just hand-write kernels — increasingly the compiler generates them. Knowing how torch.compile lowers a model to Triton, where it breaks, and how a hand-written CUDA op is made visible to it, connects your low-level work to PyTorch's strategic direction.

The four stages

Making a hand-written CUDA kernel visible to the compiler

This is the bridge from your kernel work to the framework. If you write a custom CUDA op, you register it through torch.library with both a real implementation and a fake/meta function that propagates shapes & dtypes without touching data. The meta function is what lets the dispatcher (§03), AOTAutograd, and Inductor reason about your op during tracing — so it composes with torch.compile instead of forcing a graph break. NVIDIA's cudnn-frontend ships exactly such torch.compile-compatible custom ops for its fused attention.

✕ Pitfall · graph breaks & recompilation storms

Two failure modes to name. Graph breaks: data-dependent Python control flow, .item(), printing, or an unsupported op makes Dynamo split the graph and run that region in eager — each break is a fusion boundary and a perf leak; inspect with torch._dynamo.explain() or TORCH_LOGS="graph_breaks". Recompilation storms: if a tensor's shape changes every call, guards keep failing and Inductor recompiles repeatedly — mark the axis dynamic with torch._dynamo.mark_dynamic(x, 0) (or rely on automatic dynamic shapes) so one compiled artifact covers many sizes.

▸ Probe · compile gone wrong

"You wrap your model in torch.compile and it's slower, with the log full of recompiles. What's happening?"

Answer arc

First, separate one-time compile cost from steady state — the first iterations pay tracing+codegen; benchmark after warmup. If it recompiles continuously, guards are failing: almost always dynamic shapes (variable batch/seq) blowing the cache, or frequent graph breaks forcing re-entry. Fixes: mark_dynamic the varying dims (or enable dynamic shapes) so guards generalize; remove graph breaks found via torch._dynamo.explain() / TORCH_LOGS=recompiles; and only use mode="reduce-overhead" when shapes are stable, since its CUDA graphs need static buffers. The structured answer — compile-time vs runtime, guards, dynamic shapes, graph breaks — is the point.

Build, CI & release engineering

The combinatorial release matrix

A PyTorch release isn't one artifact — it's a matrix: several CUDA toolkit versions × supported Python versions × {Linux, Windows, macOS} × {CPU, CUDA, ROCm/XPU}, each a separately-built, separately-tested wheel. Every cell bundles a compatible cuDNN, cuBLAS, NCCL (shipped as the nvidia-*-cuXX pip dependencies), pinned to versions known to work with that CUDA. Keeping those pins correct — and keeping the cuDNN that CI tests against equal to the one the wheel ships — is precisely the class of problem in Eddie's recent issues (§09).

C++ build, ABI, and the linking problems they explicitly ask about

libtorch is split into shared libraries — notably libtorch_cpu and libtorch_cuda — plus libc10/libc10_cuda. The build (driven by setup.py over CMake) has to get several things right that the JD calls out by name:

  • The C++11 ABI flag (_GLIBCXX_USE_CXX11_ABI). A C++ extension built against one ABI but loaded next to a libtorch built with the other yields link-time or, worse, run-time std::string/std::list mismatches. Matching it is the first thing to check when a custom extension won't load.
  • Symbol visibility & resolution. Undefined-symbol errors at import torch usually mean a version skew (an extension referencing a symbol the installed libtorch doesn't export) or two libraries exporting the same symbol (ODR violation). The toolchain study you've done — linkers, symbol tables, nm/ldd — is exactly the debugging vocabulary here.
  • One CUDA runtime, please. Mixing two CUDA runtimes (e.g. the wheel's bundled libs and a system toolkit) in one process causes context/allocator weirdness. Static-vs-dynamic linking of the CUDA runtime is a deliberate packaging decision.
◈ "Without on-call… drives software quality and reliability"

The JD frames this as development-focused infra work: building the tools, test systems, and pipelines that keep the whole stack shippable — not paging at 3am. When you talk about it, emphasize making failures reproducible and caught early: sharded CI across the matrix, nightly builds that exercise real GPUs, version-pin audits, and bisecting a regression to the offending cuDNN/cuBLAS bump. That's the posture they're hiring for.

▸ Probe · the import-time linker failure

"A user reports ImportError: undefined symbol (or libcudnn.so.X: cannot open shared object file) on import torch. Walk me through debugging it."

Answer arc

Triage which class it is. "cannot open libcudnn" = a loader problem: the bundled or system cuDNN isn't on the search path, or the wheel's CUDA version doesn't match the installed one — check ldd for "not found," verify the nvidia-cudnn-cuXX dependency installed, confirm CUDA-version match. The discriminator is loader-can't-find-file vs symbol-not-resolved, and naming ldd/nm/ABI shows you've actually done this.

Warm-up & architecture

Q · Walk me through what happens when I call y = x.relu() on a CUDA tensor.
Python C-binding → the relu op enters the dispatcher → a DispatchKeySet is computed from the args (here just the CUDA backend key, plus Autograd/Autocast if active) → the highest-priority key selects the kernel from the op's function table → if autograd is on, the Autograd kernel records a node then redispatches → the CUDA kernel runs through TensorIterator/gpu_kernel as a grid-stride elementwise launch on the current stream → returns a tensor whose contents are a promise the GPU will fulfill asynchronously. Hitting dispatcher → TensorIterator → stream-async in one breath shows the whole stack. (§02–§07)

Q · What's the difference between a Tensor, TensorImpl, and Storage?
Tensor is an intrusive_ptr handle (cheap to copy, refcounted). TensorImpl holds the metadata — sizes, strides, dtype, device, the dispatch key set. Storage is the actual contiguous byte buffer on a device. Many Tensors/TensorImpls can view one Storage (that's what slicing and .view() do — new strides, shared bytes). This is why views are free and why .contiguous() may or may not copy. (§02)

CUDA-deep

Q · Why a grid-stride loop?
Decouples launch geometry from problem size: one occupancy-tuned grid handles any N, threads reuse across elements, correctness holds when N exceeds max grid dim, and it's CUDA-graph-friendly because the launch shape is fixed. (§04)

Q · Explain the caching allocator and why it exists.
cudaMalloc/cudaFree synchronize the device, which would stall the host's run-ahead every allocation. So PyTorch suballocates from large cached segments — split/coalesce blocks, per-stream pools, size rounding — to hit a steady state with near-zero driver calls. The cost is fragmentation (free memory stranded across segments), mitigated by expandable_segments (CUDA VMM: one virtual reservation, physical pages mapped on demand). (§06)

Q · A tensor allocated on stream A is used on stream B and you get intermittent garbage. Why?
The allocator tracks which stream a block is "live" on; if B uses it without tensor.record_stream(B), the allocator may recycle the block once A's work finishes even though B's queued kernels haven't run yet. record_stream (or an explicit event) tells the allocator to wait for B too. Classic cross-stream hazard. (§06–§07)

cuDNN, precision & SDPA — his core

Q · What does torch.backends.cudnn.benchmark do, and when does it hurt?
It empirically times cuDNN's candidate conv algorithms for each unique input configuration and caches the winner. Great for fixed shapes; harmful for varying shapes because every new shape is a cache miss that re-runs the search. (§09)

Q · What is TF32 and what changed in the 2.9 precision API?
TF32 = tensor-core mode for fp32 ops with a 10-bit mantissa: fp32 storage, reduced-mantissa compute, big speedup, small accuracy cost. 2.9 deprecated the coarse globals (allow_tf32, set_float32_matmul_precision) for a granular per-backend/per-op fp32_precision = "ieee"|"tf32" API — and the migration even shipped a FlexAttention default regression, a lesson in how invisible precision changes are. (§12)

Q · I "use FlashAttention" but my long-sequence model OOMs. Why?
The SDPA selector silently fell through to the math backend (some constraint — mask shape, head dim, stride, dtype — disqualified the fused backends), and math materializes the full N×N scores → O(N²) memory. Prove it by forcing sdpa_kernel([FLASH_ATTENTION]) so it raises instead of downgrading, then fix the disqualifying condition. (§10)

Q · Why can a cuDNN point-release bump be a real bugfix?
Because fused backends bake in layout assumptions. The cuDNN SDPA path once assumed dO shared strides with O; a differently-strided gradient produced wrong results, fixed by the 9.5.1 bump. Version bumps can carry correctness fixes, not just features. (§09–§10)

Debugging & performance

Q · Adding print(loss) in the loop doubled step time. Explain.
.item() forces cudaStreamSynchronize, killing the host run-ahead so you pay full latency every step. Log every N steps or accumulate on-GPU and sync once. (§07)

Q · A model OOMs at iteration 500 with memory_allocated well under capacity.
Reserved ≫ allocated ⇒ fragmentation. Confirm with memory_summary() + a snapshot; then expandable_segments:True, cap max_split_size_mb, stabilize allocation sizes. (§06)

Q · An elementwise op is unexpectedly slow. How do you find out why?
Suspect it fell off TensorIterator's vectorized fast path — a non-contiguous input (often a transpose upstream), mixed dtype forcing per-element dynamic casts, or broadcasting. Confirm uncoalesced access in Nsight Compute's memory workload analysis; fix with .contiguous() or the right memory_format. (§04)

Q · How do you decide whether a workload is launch-bound or compute-bound?
An nsys timeline: visible host-side gaps between kernels ⇒ launch-bound ⇒ CUDA graphs / batching / fusion help. Wall-to-wall kernels with high tensor-core utilization (per ncu) ⇒ compute-bound ⇒ look at algorithm/precision instead. Never optimize before classifying. (§08, §11)

Design judgment

Q · Should a new fused attention variant be its own ATen op + backend, or composed from existing ops?
Trade-off the team actually litigates (see PR #122510): a dedicated backend gives peak performance and clean autocast/precision control but adds dispatch surface, backend bloat, and maintenance/version-compat burden; a composite op reuses infrastructure and stays portable but may leave performance on the table. Reasonable answer: prototype composite, promote to a fused backend when profiling justifies it and the op is stable — and be explicit about the tech-debt you're accepting. (§03, §10)

Q · SDPA's backend priority order is static. How might you improve it?
Acknowledge the limitation (#138907): one global ranking can't be optimal across shapes/GPUs. Options: a lightweight per-shape heuristic keyed on (seq, head_dim, dtype, sm), an autotuning pass that times backends once and caches (à la cudnn.benchmark), or a learned/cost-model selector. Note the cost: added complexity and the need for a fallback when autotuning isn't possible. (§10)

The JD thread — build, CI, release

Q · How would you debug undefined symbol vs cannot open libcudnn.so at import torch?
Loader-can't-find-file vs symbol-not-resolved. For the former: ldd for "not found," verify the nvidia-cudnn-cuXX install and CUDA-version match. For the latter: usually an extension built against a different libtorch/ABI — nm -DC to confirm the symbol, check the C++11 ABI flag, rebuild against the installed torch. (§14)

Q · Why is keeping CI's cuDNN equal to the shipped cuDNN important?
Because divergence creates "passes in CI, breaks for users" bugs (issue #165801): the wheel can carry a different cuDNN than CI tested, so a real numerical or perf regression slips through. Part of release hygiene is auditing those pins per platform — sometimes even rolling a version back on one OS (#162322). (§09, §14)
