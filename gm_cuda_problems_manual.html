Part I — The live round

Before you write a single line.

Four short chapters on how to run the hour. The kernels in Parts II–VII are the substance, but most candidates lose this round on process — they start typing before stating the target, forget the boundary check, or optimize something they never measured. Fix that first; it's free.

01How the round actually runs

A GPU kernel interview at a team like this is usually 45–60 minutes in a shared editor with no compiler, or a CoderPad-style environment where the code won't run on a GPU anyway. That has a specific consequence: they are grading your reasoning, not your ability to produce a compiling artifact. A kernel with one typo and a perfect explanation beats a compiling kernel you can't justify.

PhaseWhat to do
Clarify (2–3 min)Shapes, dtypes, layout, batch size, whether shapes are static, what hardware. For this team the answers are usually: small batch, static shapes, NHWC, fp16 or int8, an embedded Ampere/Blackwell-class part. Ask anyway — it shows you know the answers matter.
State the target (1 min)Arithmetic intensity, therefore memory- or compute-bound, therefore what "good" looks like. §04. Almost nobody does this, and it immediately separates you.
Sketch the decomposition (2 min)What is a thread? A block? What lives in shared memory? Say it in words before you type it.
Write the naive version (5–10 min)Correct first. Narrate the index math and the bounds check as you write them.
Critique your own code (2 min)Do this unprompted. "This is uncoalesced on B; the fix is a shared-memory tile." Beating the interviewer to the criticism is the strongest move available to you.
Optimize (10–20 min)One change at a time, each justified by a mechanism from Volume I.
Test & edge cases (3 min)How would you verify it? Non-multiple sizes, empty input, the reference implementation. §35.
The single highest-leverage habit

Say what limits the kernel before you write it, and re-state it after. "This is bandwidth-bound at AI≈0.25, so success means approaching copy bandwidth — the arithmetic doesn't matter here." Interviewers on a kernel team are listening for exactly this, because it's the difference between an engineer who optimizes by instinct and one who optimizes by model.

02The narration protocol

What to say while your hands are moving. These are the sentences that make invisible reasoning visible.

As you write……say this
The index computation"Consecutive threads get consecutive x, so this read is coalesced — a warp covers 128 contiguous bytes, which is four 32-byte sectors, the minimum."
The bounds check"Grid is ceil-div, so the last block is partial and every thread needs this guard." Never write a kernel without it and never write it silently.
A shared-memory tile"I'm staging here so both the global read and the global write stay coalesced, and so the tile is reused by every thread in the block."
A __syncthreads()"Barrier after the write, before anyone reads — and a second one before I overwrite the tile next iteration, or the fast threads will clobber it."
An accumulator"Accumulating in fp32 even though the operands are fp16 — a long reduction in half precision loses too much."

03The pre-flight checklist

Run this before you say "I think that's it." It takes ninety seconds and catches nearly every mistake a live-coded kernel makes.

Nine checks, in order
1 bounds
Every global index guarded. Grid computed with (n + B - 1) / B. Partial last block handled.
2 barriers
__syncthreads() between write and read of shared memory, and before overwriting it. Never inside a divergent branch.
3 coalescing
Does threadIdx.x vary the fastest-moving index? If not, justify it or restructure.
4 bank conflicts
Any 2-D shared array indexed by column → pad the minor dimension or swizzle.
5 accumulator type
Wider than the operands. fp32 for fp16 inputs; int32 for int8.
6 launch config
Block size a multiple of 32, ≤1024. Shared memory request within the per-block cap.
7 restrict / const
const T* __restrict__ on read-only pointers — enables the read-only path and better scheduling.
8 edge cases
n = 0, n = 1, n not a multiple of the tile, all-equal inputs, NaN/Inf if it's a reduction.
9 determinism
Any float atomic? Then results vary run to run — say so, and say whether it matters here.
The two mistakes that cost the most

The missing second barrier. In a loop that stages tiles, you need a sync after the read as well as after the write — otherwise a fast warp overwrites the tile while a slow warp is still reading it. This is the most common real bug in tiled kernels, and it's invisible in small tests.

The silent partial tile. Kernels written for exact multiples read out of bounds on the last tile. Interviewers deliberately ask about a size like 1000×1000 with a 32-wide tile for this reason.

04State the target first

Sixty seconds of arithmetic before any code tells you which half of the optimization space is even reachable — and tells the interviewer you think in models.

Part II — Foundations

The five they open with.

05P1 · Grid-stride elementwise

problem 1Compute y = alpha*x + y over n elements. Then make it fast.
The analysis, first

Per element: read x, read y, write y = 12 bytes at fp32, for 2 FLOPs. AI = 1/6 — hopelessly memory-bound. Success means hitting copy bandwidth; there is no arithmetic story here at all. Say this before writing.

// ---------- v1: the version everyone writes ----------
__global__ void saxpy_v1(int n, float a, const float* x, float* y) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) y[i] = a * x[i] + y[i];        // bounds check is mandatory
}
// launch: saxpy_v1<<<(n + 255) / 256, 256>>>(n, a, x, y);
// Correct, coalesced, and fine. But it hard-codes one element per thread.

Why the grid-stride loop is better

A grid-stride loop decouples the launch configuration from the problem size. You can size the grid to the machine (a multiple of the SM count) rather than to n, reuse one launch across different sizes, and — importantly for this job — the kernel becomes trivially CUDA-graph friendly because the launch geometry stops changing with input size.

// ---------- v2: grid-stride, and vectorized ----------
__global__ void saxpy_v2(int n, float a, const float* __restrict__ x,
                         float* __restrict__ y) {
    // one loop, any n, any grid size
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n; i += blockDim.x * gridDim.x) {
        y[i] = fmaf(a, x[i], y[i]);           // explicit FMA
    }
}

// ---------- v3: float4 — 16 B per thread per access ----------
// Moves the same bytes with a quarter of the memory instructions, which
// matters when you are issue-limited rather than bandwidth-limited.
// REQUIRES 16-byte alignment: assert it, don't hope for it.
__global__ void saxpy_v4(int n4, float a, const float4* __restrict__ x,
                         float4* __restrict__ y) {
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n4; i += blockDim.x * gridDim.x) {
        float4 xv = x[i], yv = y[i];
        yv.x = fmaf(a, xv.x, yv.x);  yv.y = fmaf(a, xv.y, yv.y);
        yv.z = fmaf(a, xv.z, yv.z);  yv.w = fmaf(a, xv.w, yv.w);
        y[i] = yv;
    }
}

// Host side: handle the tail scalar-ly rather than pretending n % 4 == 0.
void saxpy(int n, float a, float* x, float* y, cudaStream_t s) {
    int n4 = n / 4;
    int blocks = 0, threads = 256;
    cudaOccupancyMaxActiveBlocksPerMultiprocessor(&blocks, saxpy_v4, threads, 0);
    int sms = 0; cudaDeviceGetAttribute(&sms, cudaDevAttrMultiProcessorCount, 0);
    int grid = blocks * sms;                  // size the grid to the MACHINE
    if (n4 > 0)
        saxpy_v4<<<grid, threads, 0, s>>>(n4, a, (const float4*)x, (float4*)y);
    int tail = n - n4 * 4;
    if (tail > 0)
        saxpy_v2<<<1, 128, 0, s>>>(tail, a, x + n4 * 4, y + n4 * 4);
}
What they're listening for

The bounds check, coalescing stated explicitly, __restrict__, the AI computation, the grid-stride rationale (machine-sized grids, graph-friendliness), the alignment requirement for float4, and handling the tail honestly instead of assuming divisibility. Mentioning cudaOccupancyMaxActiveBlocksPerMultiprocessor for grid sizing is a nice extra — it shows you size launches from the device, not from a magic number.

06P2 · Reduction: the ladder

problem 2Sum an array of n floats. Take it as far as you can.
Why this is the canonical question

Because there are six or seven distinct optimizations between the naive version and the good one, and each exercises a different mechanism: divergence, bank conflicts, idle threads, warp semantics, and the launch-overhead/parallelism trade. An interviewer can calibrate your entire level from how far up this ladder you climb unprompted.

THE LADDER — climb it out loud ─────────────────────────────────────────────────────────────── v1 interleaved addressing, `if (tid % (2*s) == 0)` → DIVERGENT within the warp, and modulo is slow v2 interleaved with index arithmetic instead of modulo → removes divergence, but strided shared access → BANK CONFLICTS v3 sequential addressing `for (s = blockDim/2; s > 0; s >>= 1)` → conflict-free, contiguous, still half the threads idle on the very first iteration v4 first add during the global load (grid-stride accumulate) → halves the number of blocks; every thread does real work v5 unroll the last warp — no __syncthreads needed below 32 → but ONLY with volatile / __syncwarp on Volta+ (see below) v6 WARP SHUFFLE reduction: no shared memory at all for the last 32, and only one shared round-trip per warp v7 two-stage: per-block partials → second kernel, OR a single kernel with an atomic on the final value ─────────────────────────────────────────────────────────────── In a modern answer, go straight to v6/v7 and EXPLAIN v1–v5 as the reasoning that got you there. That demonstrates the ladder without wasting twenty minutes writing its rungs.
// ============ the modern reduction, written the way you'd ship it ============
#define FULL_MASK 0xffffffffu

// Step 1: reduce within a warp. log2(32) = 5 shuffles, no shared memory,
// no __syncthreads. Every lane ends up with the warp's total if we use
// __shfl_xor_sync; only lane 0 does with __shfl_down_sync.
template <typename T>
__device__ __forceinline__ T warpReduceSum(T v) {
    #pragma unroll
    for (int off = 16; off > 0; off >>= 1)
        v += __shfl_down_sync(FULL_MASK, v, off);
    return v;                                  // lane 0 holds the sum
}

// Step 2: reduce across the warps of a block via one small shared array.
template <typename T>
__device__ __forceinline__ T blockReduceSum(T v) {
    static __shared__ T warp_sums[32];         // at most 1024/32 = 32 warps
    const int lane = threadIdx.x & 31;
    const int wid  = threadIdx.x >> 5;

    v = warpReduceSum(v);                      // each warp reduces itself
    if (lane == 0) warp_sums[wid] = v;         // one write per warp
    __syncthreads();

    // first warp reduces the per-warp partials
    const int nwarps = (blockDim.x + 31) >> 5;
    v = (threadIdx.x < nwarps) ? warp_sums[lane] : T(0);
    if (wid == 0) v = warpReduceSum(v);
    return v;                                  // valid in thread 0
}

// Step 3: the kernel. Grid-stride accumulate (that is rung v4) into a
// register, then one block reduction, then one atomic per BLOCK.
__global__ void reduce_sum(const float* __restrict__ x, float* __restrict__ out, int n) {
    float acc = 0.f;
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n; i += blockDim.x * gridDim.x)
        acc += x[i];

    acc = blockReduceSum(acc);
    if (threadIdx.x == 0) atomicAdd(out, acc); // one atomic per block
}

Three things to raise unprompted

PointWhy it matters
The volatile trick is deadPre-Volta, warp-synchronous code relied on implicit lockstep and a volatile pointer. Since Volta, threads have independent program counters, so you must use __syncwarp() or the *_sync intrinsics. Legacy reduction code without them is a latent bug — knowing this is a strong signal.
The atomic makes it nondeterministicBlock completion order varies, so the float sum varies in the last bits run to run. On an AV stack that can matter (Vol. I §44). The deterministic alternative is a two-kernel reduction, or a fixed-order second pass over the per-block partials.
Numerical accuracyA tree reduction is more accurate than a sequential sum — error grows as O(√n·ε) rather than O(n·ε). If they push further: Kahan summation, or accumulate in double. For fp16 inputs, always accumulate in fp32.
And then say: "in production I'd use CUB"

cub::DeviceReduce::Sum or cub::BlockReduce is tuned per architecture and will beat hand-written code in most cases. Knowing when not to hand-roll is part of being senior — and the follow-up ("so when would you write it yourself?") has a great answer: when it fuses into a larger kernel, which is exactly the epilogue argument from Volume I §18.

07P3 · Transpose and bank conflicts

problem 3Transpose an M×N matrix. Explain every performance decision.
The analysis

Read every element once, write every element once: pure data movement, AI ≈ 0. The target is copy bandwidth, and the entire problem is making both the read and the write coalesced — which is impossible in global memory alone, because transposing inverts the fast axis. Hence shared memory.

// ---------- v1: naive. One side is always strided. ----------
__global__ void transpose_naive(const float* in, float* out, int M, int N) {
    int x = blockIdx.x * blockDim.x + threadIdx.x;   // col in input
    int y = blockIdx.y * blockDim.y + threadIdx.y;   // row in input
    if (x < N && y < M)
        out[(size_t)x * M + y] = in[(size_t)y * N + x];
    //   ^ WRITE is strided by M: consecutive threads write addresses M
    //     floats apart => up to 32 sectors per warp instead of 4.
}

// ---------- v2: shared-memory tile, padded ----------
#define TILE 32
#define BROWS 8                                  // 32x32 tile, 32x8 block
__global__ void transpose_tiled(const float* __restrict__ in,
                                float* __restrict__ out, int M, int N) {
    // +1 padding: shifts each row by one bank so a COLUMN read walks all
    // 32 banks instead of hitting one. Without it: 32-way conflict.
    __shared__ float tile[TILE][TILE + 1];

    int x = blockIdx.x * TILE + threadIdx.x;      // input col
    int y = blockIdx.y * TILE + threadIdx.y;      // input row

    // coalesced READ: threadIdx.x varies along the contiguous axis
    #pragma unroll
    for (int j = 0; j < TILE; j += BROWS)
        if (x < N && (y + j) < M)
            tile[threadIdx.y + j][threadIdx.x] = in[(size_t)(y + j) * N + x];

    __syncthreads();

    // swap block coords so the WRITE is also coalesced
    x = blockIdx.y * TILE + threadIdx.x;          // output col
    y = blockIdx.x * TILE + threadIdx.y;          // output row

    #pragma unroll
    for (int j = 0; j < TILE; j += BROWS)
        if (x < M && (y + j) < N)
            out[(size_t)(y + j) * M + x] = tile[threadIdx.x][threadIdx.y + j];
    //                                          ^^^^^^^^^^^ transposed read
}

The three-part explanation to give

  • Why shared memory at all. Transposition inverts the fast axis, so one of the two global accesses must be strided — unless you stage through a scratchpad where you can pay for the transposition in a memory that has no coalescing penalty.
  • Why the +1. Shared memory has 32 banks of 4 bytes; bank = (byte_addr / 4) % 32. With a [32][32] tile, tile[i][k] for fixed k and varying i is strided by 32 floats = 128 bytes, so every thread lands in the same bank: a 32-way conflict, serialized into 32 transactions. Padding to 33 makes the stride 33 floats, which is coprime with 32, so the column walks all banks.
  • Why not padding — swizzling. Production libraries XOR-swizzle instead: col ^= row-style permutation gives conflict-free access with no wasted shared memory and preserves the 128-bit alignment that ldmatrix and vectorized loads need. Padding breaks alignment; that's why CUTLASS doesn't use it.
The follow-up they will ask

"What if M and N aren't multiples of 32?" — the guards above handle it, but say it explicitly, and note that partial tiles reduce efficiency because the warp still issues full-width accesses. If they push on performance further: process multiple tiles per block to amortize launch overhead, and consider that a standalone transpose is often the wrong answer entirely — fuse the transposition into the producing or consuming kernel and pay for it once.

08P4 · Prefix sum

problem 4Compute an exclusive prefix sum. Why does it matter here?
The AV connection — lead with this

Scan is how you compact sparse data: given a per-element "keep" flag, the exclusive scan of the flags is the output index for each kept element. That's how you build a voxel list from a point cloud (§21), how you compact surviving detections after NMS (§22), and how the "splat" step of a lift-splat BEV transform assigns positions. It is the workhorse primitive of irregular AV kernels, not an academic exercise.

// ---------- warp-level inclusive scan: 5 shuffles, no shared memory ----------
__device__ __forceinline__ int warpInclusiveScan(int v) {
    const int lane = threadIdx.x & 31;
    #pragma unroll
    for (int off = 1; off < 32; off <<= 1) {
        int n = __shfl_up_sync(0xffffffffu, v, off);
        if (lane >= off) v += n;               // guard: lanes < off keep theirs
    }
    return v;
}

// ---------- block-level exclusive scan built from warp scans ----------
template <int BLOCK>
__device__ int blockExclusiveScan(int v, int* block_total) {
    __shared__ int warp_tot[BLOCK / 32];
    const int lane = threadIdx.x & 31, wid = threadIdx.x >> 5;

    int inc = warpInclusiveScan(v);
    if (lane == 31) warp_tot[wid] = inc;       // this warp's total
    __syncthreads();

    // scan the per-warp totals with the first warp
    if (wid == 0) {
        int t = (lane < BLOCK / 32) ? warp_tot[lane] : 0;
        t = warpInclusiveScan(t);
        if (lane < BLOCK / 32) warp_tot[lane] = t;
    }
    __syncthreads();

    int warp_offset = (wid == 0) ? 0 : warp_tot[wid - 1];
    if (block_total) *block_total = warp_tot[BLOCK / 32 - 1];
    return inc - v + warp_offset;              // inclusive -> exclusive
}

// ---------- the payoff: stream compaction ----------
// Keep the elements whose predicate is true, densely, preserving order.
template <int BLOCK>
__global__ void compact(const float* __restrict__ in, float* __restrict__ out,
                        const unsigned char* __restrict__ keep,
                        int n, int* __restrict__ counter) {
    __shared__ int base;
    int i = blockIdx.x * BLOCK + threadIdx.x;
    int flag = (i < n && keep[i]) ? 1 : 0;

    int total = 0;
    int local = blockExclusiveScan<BLOCK>(flag, &total);

    // one atomic per BLOCK reserves a contiguous output range
    if (threadIdx.x == 0) base = atomicAdd(counter, total);
    __syncthreads();

    if (flag) out[base + local] = in[i];
}
Point to makeContent
Hillis-Steele vs BlellochHillis-Steele is O(n log n) work, simple, and fine within a warp. Blelloch's up-sweep/down-sweep is work-efficient — O(n) — and is what you want for large arrays. Both were verified against std::exclusive_scan before this page was written.
The __shfl_up_sync guardLanes below the offset must not add — the shuffle returns their own value, and adding it would double-count. This is the classic off-by-one in warp scans.
One atomic per blockReserving a contiguous range per block instead of per element turns n atomics into n/BLOCK atomics. But note: block order isn't deterministic, so the output ordering is stable only within a block. If you need globally ordered compaction, use a two-pass scan of block totals — and say so.
Use the librarycub::DeviceScan::ExclusiveSum and cub::DeviceSelect::Flagged exist and are excellent. Hand-roll only to fuse.

09P5 · Histogram and warp aggregation

problem 5Histogram n values into k bins. Now make the contention go away.
Why this is really a lidar question

A histogram is a scatter-add with hot spots — which is exactly a voxelization (§21): a dense region of the point cloud hammers a handful of voxels the way a peaked distribution hammers a few bins. The escalation ladder below is the same one you'd apply to a BEV splat.

// ---------- v1: correct, and catastrophic under contention ----------
__global__ void hist_v1(const int* __restrict__ x, int* __restrict__ h, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) atomicAdd(&h[x[i]], 1);        // every thread, global atomic
}

// ---------- v2: privatize into shared memory ----------
// Each block builds its own histogram, then flushes it once.
template <int NBINS>
__global__ void hist_v2(const int* __restrict__ x, int* __restrict__ h, int n) {
    __shared__ int local[NBINS];
    for (int b = threadIdx.x; b < NBINS; b += blockDim.x) local[b] = 0;
    __syncthreads();

    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n; i += blockDim.x * gridDim.x)
        atomicAdd(&local[x[i]], 1);           // SHARED atomics: far cheaper

    __syncthreads();
    for (int b = threadIdx.x; b < NBINS; b += blockDim.x)
        if (local[b]) atomicAdd(&h[b], local[b]);   // one global atomic per bin
}

// ---------- v3: warp aggregation before the atomic ----------
// Lanes in a warp that target the SAME bin combine first, so the bin gets
// one atomic per warp instead of up to 32. This is the trick that rescues
// heavily skewed distributions (a wall in a point cloud, a dominant class).
__device__ __forceinline__ void atomicAggInc(int* addr_base, int bin) {
    unsigned mask = __activemask();
    // partition the warp into groups of lanes sharing the same `bin`
    unsigned peers = __match_any_sync(mask, bin);   // sm_70+
    int leader = __ffs(peers) - 1;                  // lowest lane in the group
    int count  = __popc(peers);
    int lane   = threadIdx.x & 31;
    if (lane == leader) atomicAdd(&addr_base[bin], count);
}

template <int NBINS>
__global__ void hist_v3(const int* __restrict__ x, int* __restrict__ h, int n) {
    __shared__ int local[NBINS];
    for (int b = threadIdx.x; b < NBINS; b += blockDim.x) local[b] = 0;
    __syncthreads();

    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n; i += blockDim.x * gridDim.x)
        atomicAggInc(local, x[i]);

    __syncthreads();
    for (int b = threadIdx.x; b < NBINS; b += blockDim.x)
        if (local[b]) atomicAdd(&h[b], local[b]);
}
The constraint that decides the design

Privatization only works if NBINS fits in shared memory (and shrinks occupancy as it grows). When k is large — say a 200×200 BEV grid — you can't privatize the whole thing. Then the options are: tile the bin range across multiple passes, sort by bin then segmented-reduce (deterministic, and often faster than you'd guess), or accept global atomics and reduce contention by reordering the input. Naming that fork is the senior answer.

And the determinism note, which is the one that matters at GM: integer atomicAdd is deterministic; float atomicAdd is not. Integers sum exactly regardless of order, so a counting histogram is reproducible. The moment you accumulate float features into voxels — which is what a real BEV splat does — the arbitrary atomic ordering makes the result differ in the last bits run to run. For a system that has to replay logged drives bit-exactly, that turns a performance choice into an architectural one (§21, Vol. I §44).

Part III — Normalization & fusion

The ops between the GEMMs.

At batch 1 these memory-bound operators can hold as much of the frame as the GEMMs do (Vol. I §19). They're also the most likely "write this kernel" prompts after the foundations, because each has a clean naive form and a genuinely better one. All four solutions here rest on identities verified against a reference before being written down.

10P6 · Row softmax

problem 6Softmax over the last dimension of an [N, D] matrix.
The two things they're checking

First, numerical stability: subtract the row max before exponentiating, or exp(large) overflows to inf and you get NaN. Second, how many passes over the row you need — the naive answer is three (max, sum, normalize), the good answer is two, and the online form gets the statistics in one.

The analysis

Read D, write D per row, a handful of FLOPs each. Memory-bound. Target: about two passes' worth of bandwidth, ideally one if fused into whatever produced the logits.

// One block per row. Block-wide reductions from P2 do the heavy lifting.
// D can exceed blockDim.x — the grid-stride inner loop handles that.
template <int BLOCK>
__global__ void softmax_rows(const float* __restrict__ x,
                             float* __restrict__ y, int D) {
    const float* row = x + (size_t)blockIdx.x * D;
    float*       out = y + (size_t)blockIdx.x * D;

    // ---- pass 1: row max (for stability) ----
    float m = -INFINITY;
    for (int i = threadIdx.x; i < D; i += BLOCK) m = fmaxf(m, row[i]);
    m = blockReduceMax<BLOCK>(m);
    __shared__ float s_m;
    if (threadIdx.x == 0) s_m = m;
    __syncthreads();
    m = s_m;

    // ---- pass 2: sum of exp ----
    float s = 0.f;
    for (int i = threadIdx.x; i < D; i += BLOCK) s += __expf(row[i] - m);
    s = blockReduceSum<BLOCK>(s);
    __shared__ float s_inv;
    if (threadIdx.x == 0) s_inv = 1.f / s;      // ONE reciprocal, not D divisions
    __syncthreads();

    // ---- pass 3: normalize (re-reads the row; see the note below) ----
    const float inv = s_inv;
    for (int i = threadIdx.x; i < D; i += BLOCK)
        out[i] = __expf(row[i] - m) * inv;
}

The improvement: fuse the two statistics into one pass

The online softmax identity lets you compute the max and the sum together in a single sweep, maintaining a running max and rescaling the running sum whenever the max moves. This is the same algebra FlashAttention is built on (§25), and it's exact — not an approximation.

// Running (max, sum) pair. Merging two partials is associative, so this
// composes through a warp/block reduction exactly like a plain sum.
struct MD { float m; float d; };               // running max, running denom

__device__ __forceinline__ MD md_merge(MD a, MD b) {
    // rescale the smaller-max partial onto the larger max
    if (a.m > b.m) { return { a.m, a.d + b.d * __expf(b.m - a.m) }; }
    else           { return { b.m, b.d + a.d * __expf(a.m - b.m) }; }
}

__device__ __forceinline__ MD warpReduceMD(MD v) {
    #pragma unroll
    for (int off = 16; off > 0; off >>= 1) {
        MD o;
        o.m = __shfl_down_sync(0xffffffffu, v.m, off);
        o.d = __shfl_down_sync(0xffffffffu, v.d, off);
        v = md_merge(v, o);
    }
    return v;
}

template <int BLOCK>
__global__ void softmax_online(const float* __restrict__ x,
                               float* __restrict__ y, int D) {
    const float* row = x + (size_t)blockIdx.x * D;
    float*       out = y + (size_t)blockIdx.x * D;

    // ---- single pass over the row for BOTH statistics ----
    MD acc { -INFINITY, 0.f };
    for (int i = threadIdx.x; i < D; i += BLOCK) {
        MD cur { row[i], 1.f };
        acc = md_merge(acc, cur);
    }
    acc = warpReduceMD(acc);

    __shared__ MD warp_md[BLOCK / 32];
    const int lane = threadIdx.x & 31, wid = threadIdx.x >> 5;
    if (lane == 0) warp_md[wid] = acc;
    __syncthreads();

    if (wid == 0) {
        MD v = (lane < BLOCK / 32) ? warp_md[lane] : MD{ -INFINITY, 0.f };
        v = warpReduceMD(v);
        if (lane == 0) warp_md[0] = v;
    }
    __syncthreads();

    const MD fin = warp_md[0];
    const float inv = 1.f / fin.d;
    for (int i = threadIdx.x; i < D; i += BLOCK)
        out[i] = __expf(row[i] - fin.m) * inv;
}
What to say about this

"The online form computes max and sum in one sweep by rescaling the running denominator whenever the max moves — exp(m_old − m_new). It's exact because softmax is shift-invariant, and it's the same identity FlashAttention uses to avoid materializing the score matrix. It doesn't reduce the number of passes over the data here — I still re-read the row to write the output — but it does reduce the number of reductions, and it's the version that generalizes to the fused-attention kernel."

If D is small enough to hold in registers or shared memory, say so: cache the row on the first pass and the third pass reads from shared, giving a genuine one-pass-over-DRAM kernel.

11P7 · LayerNorm with Welford

problem 7LayerNorm over the last dim, with affine weight and bias.
The trap

The textbook one-pass variance — accumulate sum and sum_sq, then var = E[x²] − E[x]² — is catastrophically cancelling when the mean is large relative to the spread. With activations around 100 and a variance around 4, you're subtracting two nearly-equal large numbers and keeping the noise. Welford's algorithm is the numerically stable single-pass alternative, and its parallel merge is what makes it usable on a GPU.

// Welford partial: count, mean, and M2 (the sum of squared deviations).
struct WF { float mean; float m2; int n; };

// The merge that makes Welford parallel — this is the whole trick.
__device__ __forceinline__ WF wf_merge(WF a, WF b) {
    if (a.n == 0) return b;
    if (b.n == 0) return a;
    const int   n = a.n + b.n;
    const float d = b.mean - a.mean;
    const float inv_n = 1.f / (float)n;
    return WF{ a.mean + d * (float)b.n * inv_n,
               a.m2 + b.m2 + d * d * (float)a.n * (float)b.n * inv_n,
               n };
}

__device__ __forceinline__ WF warpReduceWF(WF v) {
    #pragma unroll
    for (int off = 16; off > 0; off >>= 1) {
        WF o;
        o.mean = __shfl_down_sync(0xffffffffu, v.mean, off);
        o.m2   = __shfl_down_sync(0xffffffffu, v.m2,   off);
        o.n    = __shfl_down_sync(0xffffffffu, v.n,    off);
        v = wf_merge(v, o);
    }
    return v;
}

template <int BLOCK>
__global__ void layernorm(const float* __restrict__ x,
                          const float* __restrict__ gamma,
                          const float* __restrict__ beta,
                          float* __restrict__ y, int D, float eps) {
    const float* row = x + (size_t)blockIdx.x * D;
    float*       out = y + (size_t)blockIdx.x * D;

    // ---- single stable pass for mean and variance ----
    WF acc{ 0.f, 0.f, 0 };
    for (int i = threadIdx.x; i < D; i += BLOCK) {
        // per-element update, the sequential Welford step
        WF cur{ row[i], 0.f, 1 };
        acc = wf_merge(acc, cur);
    }
    acc = warpReduceWF(acc);

    __shared__ WF warp_wf[BLOCK / 32];
    const int lane = threadIdx.x & 31, wid = threadIdx.x >> 5;
    if (lane == 0) warp_wf[wid] = acc;
    __syncthreads();
    if (wid == 0) {
        WF v = (lane < BLOCK / 32) ? warp_wf[lane] : WF{0.f, 0.f, 0};
        v = warpReduceWF(v);
        if (lane == 0) warp_wf[0] = v;
    }
    __syncthreads();

    const WF f = warp_wf[0];
    const float mean = f.mean;
    const float rstd = rsqrtf(f.m2 / (float)f.n + eps);   // rsqrt, not 1/sqrt

    // ---- normalize + affine, fused ----
    for (int i = threadIdx.x; i < D; i += BLOCK)
        out[i] = (row[i] - mean) * rstd * gamma[i] + beta[i];
}
PointSay this
Why Welford"The E[x²] − E[x]² form cancels catastrophically when the mean dominates the spread. Welford accumulates deviations from a running mean, so it stays accurate — and the pairwise merge makes it work across warps and blocks." (Both the sequential and the parallel-merge forms were checked against a two-pass reference.)
rsqrtfOne hardware instruction instead of a square root followed by a division.
VectorizationIf D is a multiple of 4 and the row is 16-byte aligned, load float4 and merge four Welford partials per iteration. Same structure, quarter the memory instructions.
RMSNormIf they ask: drop the mean entirely, normalize by rsqrt(mean(x²) + eps). Cheaper, one statistic, and increasingly common. The kernel simplifies to a plain sum reduction.
The real optimization"This is memory-bound at roughly one read and one write per element, so the biggest win isn't inside this kernel — it's fusing it into the adjacent GEMM's epilogue so the activations never round-trip to DRAM."

12P8 · Fused bias + GELU + residual

problem 8Given y = gelu(x + bias) + residual, write it as one kernel. Justify it with numbers.
The numbers that justify the fusion

Unfused, in fp16: bias-add reads x and writes t (4 B/elt), GELU reads t and writes t (4 B), residual-add reads t and r and writes y (6 B) — about 14 bytes of DRAM traffic per element across three launches. Fused: read x, read r, write y = 6 bytes, one launch. That's a 2.3× traffic reduction on an operation that is purely bandwidth-bound, plus two fewer kernel launches — which at batch 1 is its own win (Vol. I §29).

#include <cuda_fp16.h>

// tanh-approximation GELU — matches PyTorch's approximate='tanh' path.
// Use the erf form if the reference model used exact GELU; they differ by
// ~1e-3, which is enough to fail a tight tolerance test (see D3).
__device__ __forceinline__ float gelu_tanh(float x) {
    const float k0 = 0.7978845608028654f;      // sqrt(2/pi)
    const float k1 = 0.044715f;
    float x3 = x * x * x;
    return 0.5f * x * (1.f + tanhf(k0 * (x + k1 * x3)));
}

// half2 path: two elements per instruction, 8-byte vector loads.
// N is the number of half2 pairs; bias is indexed by channel.
__global__ void fused_bias_gelu_residual(
        const __half2* __restrict__ x,
        const __half2* __restrict__ bias,       // [C/2], broadcast per row
        const __half2* __restrict__ res,
        __half2* __restrict__ y,
        int n2, int c2) {                       // c2 = C/2
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n2; i += blockDim.x * gridDim.x) {
        const int c = i % c2;                   // channel index within the row

        __half2 xv = x[i], bv = bias[c], rv = res[i];
        float2 xf = __half22float2(xv);
        float2 bf = __half22float2(bv);
        float2 rf = __half22float2(rv);

        // compute in fp32: GELU in half precision loses too much
        float2 o;
        o.x = gelu_tanh(xf.x + bf.x) + rf.x;
        o.y = gelu_tanh(xf.y + bf.y) + rf.y;

        y[i] = __float22half2_rn(o);
    }
}
Two things to flag before they ask

The modulo. i % c2 is an integer division per element, and integer division is slow on a GPU. If the channel count is a power of two, mask instead. Better: restructure the launch so blockIdx.y is the row and threadIdx.x walks the channel, making the bias index free and the access pattern explicit.

Compute precision. Do the arithmetic in fp32 even with fp16 storage. tanhf in half precision costs real accuracy for no bandwidth benefit — the bytes moved are what matter, and those are already halved by the fp16 storage.

The generalization is the point: this is a hand-written epilogue. In a real pipeline you wouldn't ship this kernel — you'd fold these operations into the preceding GEMM's epilogue via CUTLASS (Vol. I §18), so the accumulator never leaves registers and the traffic drops to a single write. Say that. It shows you know the standalone fused kernel is the second-best answer, and it connects directly to the CUTLASS line in the job description.

13P9 · NCHW ↔ NHWC conversion

problem 9Convert a tensor between NCHW and NHWC efficiently.
Why this problem exists in an inference stack

Tensor cores and cuDNN want channels-last; many exported graphs are channels-first; TensorRT inserts reformat nodes to reconcile them, and those show up in a profile as unexplained kernels between your layers (Vol. I §24). Being able to write this kernel — and to argue for eliminating it — is a real, daily piece of the job.

// NCHW -> NHWC is a transpose of the (C) and (H*W) axes, per image.
// Same lesson as P3: stage through shared memory so BOTH sides coalesce.
#define TT 32
template <typename T>
__global__ void nchw_to_nhwc(const T* __restrict__ in, T* __restrict__ out,
                             int N, int C, int HW) {
    __shared__ T tile[TT][TT + 1];             // +1 kills the bank conflict

    const int n  = blockIdx.z;
    const int hw0 = blockIdx.x * TT;           // tile origin along H*W
    const int c0  = blockIdx.y * TT;           // tile origin along C

    // read: consecutive threadIdx.x walks HW, which is contiguous in NCHW
    for (int j = 0; j < TT; j += blockDim.y) {
        int c  = c0 + threadIdx.y + j;
        int hw = hw0 + threadIdx.x;
        if (c < C && hw < HW)
            tile[threadIdx.y + j][threadIdx.x] =
                in[((size_t)n * C + c) * HW + hw];
    }
    __syncthreads();

    // write: consecutive threadIdx.x walks C, which is contiguous in NHWC
    for (int j = 0; j < TT; j += blockDim.y) {
        int hw = hw0 + threadIdx.y + j;
        int c  = c0 + threadIdx.x;
        if (c < C && hw < HW)
            out[((size_t)n * HW + hw) * C + c] =
                tile[threadIdx.x][threadIdx.y + j];
    }
}
The answer that's better than the kernel

"The fastest layout conversion is the one you don't do. If this reformat is showing up between two of my layers, I'd first check whether both can agree on a layout — TensorRT inserts these automatically, and a plugin that only advertises support for one format forces them (Vol. I §27). Supporting NHWC in the plugin's supportsFormatCombination often deletes the kernel entirely." That reframes a coding question as an integration question, which is exactly the level this role is hiring at.

Part IV — The GEMM ladder

Four rungs to tensor cores.

GEMM is the canonical GPU optimization narrative, and the req's top differentiator — tensor cores, CUTLASS, CuTe — sits at the top of this ladder. You will not out-write cuBLAS in an interview. 14P10 · Naive GEMM, and why it's slow

problem 10C = A·B for A[M,K], B[K,N]. Start naive, then diagnose it yourself.
State the target before writing

FLOPs = 2·M·N·K. Minimum bytes = (MK + KN + MN)·sizeof(T). For 1024³ fp32 that's AI ≈ 170 — strongly compute-bound, so the target is a large fraction of peak arithmetic throughput, and every optimization below exists to raise achieved intensity toward that theoretical number.

// ---------- rung 0: one thread per output element ----------
__global__ void gemm_naive(const float* __restrict__ A, const float* __restrict__ B,
                           float* __restrict__ C, int M, int N, int K) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    if (row >= M || col >= N) return;

    float acc = 0.f;
    for (int k = 0; k < K; ++k)
        acc += A[(size_t)row * K + k] * B[(size_t)k * N + col];
    C[(size_t)row * N + col] = acc;
}
WHY IT IS SLOW — say all three ─────────────────────────────────────────────────────────────── 1 NO REUSE. Every output element re-reads a full row of A and a full column of B from DRAM. Total traffic ~ 2·M·N·K elements instead of the ~MK + KN minimum. ⇒ ACHIEVED arithmetic intensity ≈ 2 FLOP / 8 B = 0.25, not the 170 the problem allows. Three orders of magnitude of reuse are being thrown away. 2 B's ACCESS IS FINE, A's IS NOT (for this indexing): B[k*N + col] → consecutive `col` per thread → COALESCED A[row*K + k] → all threads in a warp share `row` here only if threadIdx.x maps to col; the A read is a BROADCAST, which is fine — but the k-loop re-fetches it M/…-many times ⇒ the problem is not access SHAPE, it is access VOLUME. 3 The inner loop is one FMA per two loads — the memory pipe is saturated while the math pipe idles. ncu will show low compute throughput, high memory throughput, and Long Scoreboard stalls. ─────────────────────────────────────────────────────────────── FIX: create reuse. Tile so each loaded value serves many FMAs.

15P11 · Shared-memory tiled GEMM

problem 11Add shared-memory tiling. Quantify what it bought.
The arithmetic that justifies it

With a T×T tile, each element loaded into shared memory is used T times instead of once, so DRAM traffic falls by a factor of T and arithmetic intensity rises by the same factor. T = 32 turns AI ≈ 0.25 into ≈ 8. That's the whole idea, and it's worth stating as a formula rather than as "tiling is faster."

#define TS 32
__global__ void gemm_tiled(const float* __restrict__ A, const float* __restrict__ B,
                           float* __restrict__ C, int M, int N, int K) {
    __shared__ float As[TS][TS];
    __shared__ float Bs[TS][TS + 1];           // +1: Bs is read down a column

    const int tx = threadIdx.x, ty = threadIdx.y;
    const int row = blockIdx.y * TS + ty;
    const int col = blockIdx.x * TS + tx;

    float acc = 0.f;
    for (int t = 0; t < (K + TS - 1) / TS; ++t) {
        const int kA = t * TS + tx;            // column of A being staged
        const int kB = t * TS + ty;            // row of B being staged

        // guarded loads: partial tiles at the edges must read as zero,
        // NOT out of bounds. This is the #1 bug in live-coded tiled GEMM.
        As[ty][tx] = (row < M && kA < K) ? A[(size_t)row * K + kA] : 0.f;
        Bs[ty][tx] = (kB < K && col < N) ? B[(size_t)kB * N + col] : 0.f;

        __syncthreads();                        // tile is now complete

        #pragma unroll
        for (int k = 0; k < TS; ++k)
            acc += As[ty][k] * Bs[k][tx];

        __syncthreads();                        // ← THE ONE PEOPLE FORGET:
        //   without it, a fast warp overwrites As/Bs for iteration t+1
        //   while a slow warp is still reading iteration t.
    }
    if (row < M && col < N) C[(size_t)row * N + col] = acc;
}
DetailThe reasoning to voice
Two barriers, not oneOne after the stores (the tile must be complete before anyone reads), one after the compute loop (nobody may overwrite the tile until everyone has finished reading). Omitting the second is the most common tiled-GEMM bug and it only manifests under load.
Zero-fill on partial tilesPadding with 0 makes the extra k terms contribute nothing, so the same inner loop works for any K. The alternative — bounding the inner loop — adds a branch to the hot path.
The +1 on BsBs[k][tx] for fixed tx walks a column; without padding that's a 32-way bank conflict (P3).
Still not enoughSay this before they do: each thread still computes one output and does two shared-memory reads per FMA. Shared memory is fast but not free — the ratio of arithmetic to shared traffic is still 1:2. The next rung fixes that with registers.

16P12 · Register-tiled GEMM

problem 12Each thread computes an 8×8 tile. Why does that help so much?
The key insight

An outer product. If a thread loads TM values of A and TN values of B into registers, it can perform TM×TN FMAs from TM+TN loads. At 8×8 that's 64 FMAs from 16 shared-memory reads — a 4:1 arithmetic-to-load ratio instead of 1:2, an eightfold improvement. It also creates 64 independent accumulators, which is deep instruction-level parallelism, so the kernel tolerates latency at low occupancy (Vol. I §10).

// Threadblock tile 128x128, K-step 8. Block of 256 threads (16x16),
// each thread owning an 8x8 register tile. This is the classic
// "1 warp-tile per quadrant" shape that CUTLASS generalizes.
#define BM 128
#define BN 128
#define BK 8
#define TM 8
#define TN 8

__global__ __launch_bounds__(256)
void gemm_reg(const float* __restrict__ A, const float* __restrict__ B,
              float* __restrict__ C, int M, int N, int K) {
    // As is stored TRANSPOSED [BK][BM] so the inner loop reads a
    // contiguous run of A-values for the outer product.
    __shared__ float As[BK][BM];
    __shared__ float Bs[BK][BN];

    const int tid = threadIdx.y * blockDim.x + threadIdx.x;   // 0..255
    const int bx  = blockIdx.x, by = blockIdx.y;

    // ---- staging index math: 256 threads move 128x8 (A) and 8x128 (B) ----
    // A tile: 128 rows x 8 cols = 1024 floats = 256 float4
    const int a_row = tid >> 1;                 // 0..127
    const int a_c4  = (tid & 1) * 4;            // 0 or 4
    // B tile: 8 rows x 128 cols = 1024 floats = 256 float4
    const int b_row = tid >> 5;                 // 0..7
    const int b_c4  = (tid & 31) * 4;           // 0,4,...,124

    float acc[TM][TN] = {0.f};
    float fa[TM], fb[TN];

    for (int k0 = 0; k0 < K; k0 += BK) {
        // ---------- stage A (transposed into shared) ----------
        {
            int r = by * BM + a_row, c = k0 + a_c4;
            float4 v = make_float4(0.f, 0.f, 0.f, 0.f);
            if (r < M && c + 3 < K)
                v = *reinterpret_cast<const float4*>(&A[(size_t)r * K + c]);
            else if (r < M) {                   // ragged tail, scalar path
                float t[4] = {0,0,0,0};
                for (int i = 0; i < 4; ++i) if (c + i < K) t[i] = A[(size_t)r * K + c + i];
                v = make_float4(t[0], t[1], t[2], t[3]);
            }
            As[a_c4 + 0][a_row] = v.x;          // transpose on the way in
            As[a_c4 + 1][a_row] = v.y;
            As[a_c4 + 2][a_row] = v.z;
            As[a_c4 + 3][a_row] = v.w;
        }
        // ---------- stage B (row-major, as-is) ----------
        {
            int r = k0 + b_row, c = bx * BN + b_c4;
            float4 v = make_float4(0.f, 0.f, 0.f, 0.f);
            if (r < K && c + 3 < N)
                v = *reinterpret_cast<const float4*>(&B[(size_t)r * N + c]);
            else if (r < K) {
                float t[4] = {0,0,0,0};
                for (int i = 0; i < 4; ++i) if (c + i < N) t[i] = B[(size_t)r * N + c + i];
                v = make_float4(t[0], t[1], t[2], t[3]);
            }
            *reinterpret_cast<float4*>(&Bs[b_row][b_c4]) = v;
        }

        __syncthreads();

        // ---------- the outer-product inner loop ----------
        #pragma unroll
        for (int k = 0; k < BK; ++k) {
            #pragma unroll
            for (int i = 0; i < TM; ++i) fa[i] = As[k][threadIdx.y * TM + i];
            #pragma unroll
            for (int j = 0; j < TN; ++j) fb[j] = Bs[k][threadIdx.x * TN + j];
            #pragma unroll
            for (int i = 0; i < TM; ++i)
                #pragma unroll
                for (int j = 0; j < TN; ++j)
                    acc[i][j] = fmaf(fa[i], fb[j], acc[i][j]);
        }
        __syncthreads();
    }

    // ---------- epilogue: write the 8x8 register tile ----------
    #pragma unroll
    for (int i = 0; i < TM; ++i) {
        int r = by * BM + threadIdx.y * TM + i;
        if (r >= M) continue;
        #pragma unroll
        for (int j = 0; j < TN; ++j) {
            int c = bx * BN + threadIdx.x * TN + j;
            if (c < N) C[(size_t)r * N + c] = acc[i][j];
        }
    }
}
Talk about the register budget — it's the real constraint

64 accumulators + 16 fragment registers + addressing is comfortably over 80 registers per thread. At 256 threads that's ~20K registers per block, which limits you to two or three blocks per SM. That's low occupancy, and it's correct — the 64 independent FMA chains provide the latency tolerance that occupancy would otherwise have to. Check it with -Xptxas -v and watch for spills; a spilled accumulator turns this kernel back into a memory-bound one. __launch_bounds__(256) tells the compiler the block size so it can budget registers accordingly.

The sentence that ties this to the job: everything above this rung is CUTLASS. Deeper pipelining with cp.async so the next tile is in flight while this one computes, swizzled shared layouts instead of padding, and mma.sync instead of fmaf — those are the remaining rungs, and they're exactly what CUTLASS's CollectiveMainloop abstracts (Vol. I §16). Saying "I'd stop hand-writing here and compose the rest from CUTLASS atoms, because the tuning space past this point is enormous and already explored" is the honest, senior answer.

17P13 · Tensor-core GEMM with WMMA

problem 13Now use the tensor cores. What changes, and what stays the same?
What stays the same

The entire tiling hierarchy. Threadblock tile, shared-memory staging, the K-loop, the epilogue — unchanged. Only the innermost compute primitive changes, from a scalar FMA loop to a warp-cooperative matrix instruction. Saying that first shows you understand MMA as a datapath swap rather than a different algorithm.

What changes

The unit of work becomes a warp, not a thread. Operands live in opaque distributed fragments. Inputs are fp16/bf16 with fp32 accumulate. And alignment and leading-dimension constraints become real: load_matrix_sync wants 16-byte-aligned pointers and an ldm that is a multiple of 8 halves.

#include <mma.h>
using namespace nvcuda;

// Warp-level 16x16x16 MMA tiles. Each warp computes a 64x64 output tile
// as a 4x4 grid of 16x16 fragments; a 128x128 block tile = 2x2 warps... but
// keep it simple here: one warp per 16x16 output tile, K-looped.
#define WMMA_M 16
#define WMMA_N 16
#define WMMA_K 16

__global__ void gemm_wmma(const __half* __restrict__ A,   // [M,K] row-major
                          const __half* __restrict__ B,   // [K,N] row-major
                          float* __restrict__ C,          // [M,N] row-major
                          int M, int N, int K) {
    // one warp per output tile
    const int warpM = (blockIdx.y * blockDim.y + threadIdx.y);
    const int warpN = (blockIdx.x * blockDim.x + threadIdx.x) / warpSize;

    wmma::fragment<wmma::matrix_a, WMMA_M, WMMA_N, WMMA_K, __half, wmma::row_major> a_frag;
    wmma::fragment<wmma::matrix_b, WMMA_M, WMMA_N, WMMA_K, __half, wmma::row_major> b_frag;
    wmma::fragment<wmma::accumulator, WMMA_M, WMMA_N, WMMA_K, float> c_frag;

    wmma::fill_fragment(c_frag, 0.0f);         // accumulate in FP32

    const int row = warpM * WMMA_M;
    const int col = warpN * WMMA_N;
    if (row >= M || col >= N) return;

    for (int k = 0; k < K; k += WMMA_K) {
        if (row + WMMA_M <= M && k + WMMA_K <= K && col + WMMA_N <= N) {
            // ldm is the leading dimension IN ELEMENTS of the source matrix
            wmma::load_matrix_sync(a_frag, A + (size_t)row * K + k, K);
            wmma::load_matrix_sync(b_frag, B + (size_t)k * N + col, N);
            wmma::mma_sync(c_frag, a_frag, b_frag, c_frag);   // D = A*B + C
        }
        // Ragged edges: WMMA has no partial-tile mode. Real kernels either
        // pad the operands to a multiple of 16 or run a scalar cleanup
        // kernel on the border. Say which you'd choose and why.
    }
    wmma::store_matrix_sync(C + (size_t)row * N + col, c_frag, N,
                            wmma::mem_row_major);
}
Question they'll follow withThe answer
"Why fp32 accumulate?"Because summing K terms in fp16 loses precision fast — the accumulator's dynamic range is the whole reason MMA hardware accumulates wider than it multiplies.
"What's the fragment layout?"Hardware-defined and opaque in the WMMA API — that's the point of the abstraction. Production kernels drop to PTX mma.sync with explicit layouts precisely so they can control staging and feed operands from a swizzled shared tile via ldmatrix (Vol. I §13–14). Don't recite lane mappings; explain why libraries need control of them.
"How would you make this fast?"Stage tiles through shared memory with cp.async, multi-stage the pipeline so the next tile is in flight during the current MMA, swizzle the shared layout for conflict-free ldmatrix, give each warp several fragments for ILP, and fuse the epilogue. That list is CUTLASS.
"When wouldn't you use tensor cores?"When the shape is degenerate — batch-1 GEMV has no M dimension to fill a 16-row tile, so it's memory-bound and MMA buys nothing. That's P14.

18P14 · GEMV for batch-1

problem 14y = A·x with A[N,K] and a single vector x. This is the inference case.
Why this is the most job-relevant GEMM question

On-vehicle inference runs at batch 1 (Vol. I §04). Every projection in a detection head with a single query, every decode step, every small linear layer degenerates to this shape. And the analysis inverts: FLOPs = 2·N·K, bytes ≈ N·K·sizeof(T) — so AI ≈ 2 FLOP per element, hopelessly memory-bound. You read the entire weight matrix once and do almost nothing with it. Tensor cores are irrelevant; bandwidth is the entire game.

// One WARP per output row: the warp streams the row coalesced, then does a
// single shuffle reduction. No shared memory, no bank conflicts, no barriers.
__global__ void gemv_fp16(const __half* __restrict__ A,   // [N, K] row-major
                          const __half* __restrict__ x,   // [K]
                          float* __restrict__ y,          // [N]
                          int N, int K) {
    const int warp_id = (blockIdx.x * blockDim.x + threadIdx.x) / warpSize;
    const int lane    = threadIdx.x & 31;
    if (warp_id >= N) return;

    const __half2* Arow = reinterpret_cast<const __half2*>(A + (size_t)warp_id * K);
    const __half2* xv   = reinterpret_cast<const __half2*>(x);
    const int K2 = K >> 1;                      // half2 elements

    float acc = 0.f;
    // stride-32 walk: lane i reads element i, i+32, i+64 ... => COALESCED,
    // and each iteration moves 32 lanes x 4 B = 128 B = 4 sectors exactly.
    for (int i = lane; i < K2; i += 32) {
        float2 a = __half22float2(Arow[i]);
        float2 b = __half22float2(xv[i]);
        acc = fmaf(a.x, b.x, acc);
        acc = fmaf(a.y, b.y, acc);
    }
    if (K & 1) {                                // odd tail
        if (lane == 0) acc += __half2float(A[(size_t)warp_id * K + K - 1])
                            * __half2float(x[K - 1]);
    }

    // warp reduction: 5 shuffles, lane 0 writes
    #pragma unroll
    for (int off = 16; off > 0; off >>= 1)
        acc += __shfl_down_sync(0xffffffffu, acc, off);
    if (lane == 0) y[warp_id] = acc;
}

The optimizations that actually matter here

  • Cache x. Every row re-reads the same vector. It's small — stage it into shared memory once per block, or rely on it staying resident in L2. This is the one reuse opportunity the problem has.
  • Vectorize wider. half2 is the minimum; float4-width loads (8 halves) cut memory instructions further. Alignment must be checked.
  • Split-K when N is small. If there aren't enough rows to fill the GPU, partition K across blocks and reduce the partials — otherwise most SMs sit idle. This is the same reasoning as split-K/stream-K GEMM (Vol. I §15), and it's the right answer to "what if N is only 64?"
  • Quantize. Since the kernel is bandwidth-bound and the bytes are almost all weights, INT8 weights nearly double throughput. This is why weight-only quantization is so effective for batch-1 inference — and it leads straight into Part VII.
  • Fuse. A bias-add or activation after this is free; a separate kernel doubles the traffic on the output.
The framing that lands

"The interesting thing about batch-1 inference is that the flagship compute-bound kernel becomes a bandwidth-bound one. All the tensor-core machinery that matters for training is inert here — what matters is reading the weights once, as few bytes as possible, perfectly coalesced. That's why quantization and fusion dominate inference optimization while tiling dominates training."

19P15 · Bilinear sampler

problem 15Sample a feature map at arbitrary fractional (x, y) — the grid_sample core.
Why it's the foundation of the next problem

Every view transform, every deformable attention head, every BEV projection ultimately reduces to "read a feature at a non-integer location." Get this kernel's layout and conventions right and P16 is mostly bookkeeping on top of it.

The analysis

Four neighbour reads per sample per channel, ~8 FLOPs. Memory-bound and irregular — the sample locations are data-dependent, so there is no static coalescing to exploit along the spatial axes. The entire optimization is to recover coalescing along the channel axis.

THE LAYOUT DECISION THAT MAKES OR BREAKS THIS KERNEL ─────────────────────────────────────────────────────────────── features stored [H][W][C] — CHANNELS INNERMOST (NHWC) assign consecutive THREADS to consecutive CHANNELS: thread 0 → c=0 ┐ thread 1 → c=1 │ all four neighbour reads become thread 2 → c=2 │ CONTIGUOUS runs of C floats ... │ → a warp covers 32 consecutive channels thread 31→ c=31┘ → 4 coalesced 128-B transactions, not 128 scattered ones With NCHW instead, the four neighbours are H*W apart AND each channel is another H*W away — every read is its own sector. Same math, ~8x the memory traffic. THE LAYOUT IS THE KERNEL.
// Zero-padding outside the map; align_corners=false convention.
// (Verified against a reference implementation, including the boundary
//  case where a sample blends a valid row with the zero pad.)
__device__ __forceinline__ float bilinear_sample(
        const float* __restrict__ feat,        // [H][W][C]
        int H, int W, int C, float y, float x, int c) {
    const int y0 = (int)floorf(y), x0 = (int)floorf(x);
    const float dy = y - y0, dx = x - x0;

    // hoist the four validity flags: branchless-friendly, and the compiler
    // can keep them in predicates rather than diverging four ways.
    const bool v00 = (y0     >= 0 && y0     < H && x0     >= 0 && x0     < W);
    const bool v01 = (y0     >= 0 && y0     < H && x0 + 1 >= 0 && x0 + 1 < W);
    const bool v10 = (y0 + 1 >= 0 && y0 + 1 < H && x0     >= 0 && x0     < W);
    const bool v11 = (y0 + 1 >= 0 && y0 + 1 < H && x0 + 1 >= 0 && x0 + 1 < W);

    const size_t s = (size_t)W * C;            // row stride in elements
    const float* base = feat + (size_t)y0 * s + (size_t)x0 * C + c;

    // __ldg routes through the read-only path; these are broadcast-ish reads
    const float f00 = v00 ? __ldg(base)                 : 0.f;
    const float f01 = v01 ? __ldg(base + C)             : 0.f;
    const float f10 = v10 ? __ldg(base + s)             : 0.f;
    const float f11 = v11 ? __ldg(base + s + C)         : 0.f;

    const float w00 = (1.f - dy) * (1.f - dx), w01 = (1.f - dy) * dx;
    const float w10 = dy * (1.f - dx),         w11 = dy * dx;
    return f00 * w00 + f01 * w01 + f10 * w10 + f11 * w11;
}

// One block per sample point; threads walk the channel dimension.
__global__ void grid_sample_nhwc(const float* __restrict__ feat,
                                 const float* __restrict__ grid,  // [P][2] (x,y) normalized
                                 float* __restrict__ out,         // [P][C]
                                 int H, int W, int C, int P) {
    const int p = blockIdx.x;
    if (p >= P) return;

    // normalized [-1,1] -> pixel, align_corners=false
    const float xn = grid[2 * p + 0], yn = grid[2 * p + 1];
    const float x = ((xn + 1.f) * W - 1.f) * 0.5f;
    const float y = ((yn + 1.f) * H - 1.f) * 0.5f;

    for (int c = threadIdx.x; c < C; c += blockDim.x)
        out[(size_t)p * C + c] = bilinear_sample(feat, H, W, C, y, x, c);
}
The convention question that decides correctness

align_corners changes the normalized-to-pixel mapping: with false (PyTorch's default) the mapping is x = ((xn + 1)·W − 1)/2; with true it is x = (xn + 1)·(W − 1)/2. They differ by half a pixel, which is invisible in a smoke test and clearly wrong in an mAP comparison. Ask which convention the reference uses before you write the line — and note that half-pixel disagreement is one of the most common causes of "my kernel doesn't match PyTorch" (D3).

The texture alternative, and its catch

A CUDA texture object gives you hardware bilinear interpolation and free boundary clamping in one instruction. It's genuinely faster on this access pattern. The catch: texture filtering uses a fixed-point weight representation with limited fractional bits, so it does not bit-match an fp32 manual computation. That's usually acceptable for a learned feature sampler — but it is a numerical change that must be validated against the reference with an error budget, not assumed harmless. Raising this unprompted signals you've shipped numerically-sensitive kernels.

20P16 · Multi-scale deformable attention

problem 16Implement MSDeformAttn: each query samples K learned points per head, per feature level.
This is the kernel

If GM's AI Kernels team maintains one signature custom operator, this is the strongest candidate. It's the core of BEVFormer-style encoders and Deformable-DETR heads, it appears in no vendor library, and every AV company carries its own tuned version. Being able to write it — and more importantly, to reason about its parallel decomposition — is the most domain-specific thing you can demonstrate.

THE COMPUTATION (semantics verified against a reference) ─────────────────────────────────────────────────────────────── inputs value [S_total, M, Dh] multi-level features, concatenated spatial_shapes[L][2], level_start[L] where each level lives sampling_loc [Q, M, L, P, 2] LEARNED, normalized (x, y) attn_weight [Q, M, L, P] LEARNED, softmax'd over (L*P) output out [Q, M*Dh] for each query q, head m, channel d: acc = 0 for level l, point p: (x,y) = sampling_loc[q,m,l,p] -> pixel coords in level l acc += attn_weight[q,m,l,p] * bilinear(value_l, y, x, m, d) out[q, m*Dh + d] = acc ─────────────────────────────────────────────────────────────── THE DECOMPOSITION CHOICE — this is what they are really asking: grid : one block per (query, head) ← Q*M blocks, plenty thread: one per CHANNEL d ← Dh threads, coalesced loop : over L levels and P points ← serial, ~8-32 iters Why: the accumulation for a given (q,m,d) is private to one thread, so NO atomics and NO shared reduction are needed. The channel axis gives coalescing. The L*P loop is short and its per-iteration scalar work (locations, weights) is shared across the whole block — so stage it in shared memory once.
// One block per (query, head). Threads walk the head's channels.
// Dh is typically 32; use 32 or 64 threads and let each handle 1-2 channels.
template <int MAX_LP>                          // compile-time bound on L*P
__global__ void ms_deform_attn(
        const float* __restrict__ value,       // [S_total, M, Dh]
        const int*   __restrict__ spatial_shapes,  // [L][2] = (H, W)
        const int*   __restrict__ level_start,     // [L]
        const float* __restrict__ sampling_loc,    // [Q, M, L, P, 2]
        const float* __restrict__ attn_weight,     // [Q, M, L, P]
        float* __restrict__ out,               // [Q, M*Dh]
        int Q, int M, int L, int P, int Dh) {

    const int q = blockIdx.x;
    const int m = blockIdx.y;
    if (q >= Q || m >= M) return;

    const int LP = L * P;
    const int C  = M * Dh;                     // total channels per position

    // ---- stage the per-(q,m) scalars once: every thread needs all of them ----
    __shared__ float s_x[MAX_LP], s_y[MAX_LP], s_w[MAX_LP];
    __shared__ int   s_H[MAX_LP], s_W[MAX_LP], s_off[MAX_LP];

    for (int i = threadIdx.x; i < LP; i += blockDim.x) {
        const int l = i / P;
        const size_t base = (((size_t)q * M + m) * L + l) * P + (i % P);
        const float xn = sampling_loc[base * 2 + 0];
        const float yn = sampling_loc[base * 2 + 1];
        const int H = spatial_shapes[2 * l + 0];
        const int W = spatial_shapes[2 * l + 1];
        // normalized [0,1] -> pixel centres (align_corners=false style)
        s_x[i]   = xn * W - 0.5f;
        s_y[i]   = yn * H - 0.5f;
        s_w[i]   = attn_weight[base];
        s_H[i]   = H;  s_W[i] = W;
        s_off[i] = level_start[l];             // row offset into `value`
    }
    __syncthreads();

    // ---- each thread owns one channel of this head: private accumulator ----
    for (int d = threadIdx.x; d < Dh; d += blockDim.x) {
        const int c = m * Dh + d;              // channel within the C axis
        float acc = 0.f;

        #pragma unroll 4
        for (int i = 0; i < LP; ++i) {
            const float x = s_x[i], y = s_y[i];
            const int H = s_H[i], W = s_W[i];
            const int y0 = (int)floorf(y), x0 = (int)floorf(x);
            const float dy = y - y0, dx = x - x0;

            // this level's slab: value + level_start*C, laid out [H][W][C]
            const float* lv = value + (size_t)s_off[i] * C;
            const size_t rs = (size_t)W * C;

            const bool v00 = (y0 >= 0 && y0 < H && x0 >= 0 && x0 < W);
            const bool v01 = (y0 >= 0 && y0 < H && x0 + 1 >= 0 && x0 + 1 < W);
            const bool v10 = (y0 + 1 >= 0 && y0 + 1 < H && x0 >= 0 && x0 < W);
            const bool v11 = (y0 + 1 >= 0 && y0 + 1 < H && x0 + 1 >= 0 && x0 + 1 < W);

            const float* b = lv + (size_t)y0 * rs + (size_t)x0 * C + c;
            const float f00 = v00 ? __ldg(b)           : 0.f;
            const float f01 = v01 ? __ldg(b + C)       : 0.f;
            const float f10 = v10 ? __ldg(b + rs)      : 0.f;
            const float f11 = v11 ? __ldg(b + rs + C)  : 0.f;

            const float val = f00 * (1.f - dy) * (1.f - dx)
                            + f01 * (1.f - dy) * dx
                            + f10 * dy * (1.f - dx)
                            + f11 * dy * dx;
            acc = fmaf(s_w[i], val, acc);
        }
        out[(size_t)q * C + c] = acc;          // no atomics anywhere
    }
}
Design pointWhy, in one sentence
Block = (query, head)Gives Q·M blocks — thousands, so the GPU fills — while keeping each output element's accumulation private to a single thread. No atomics, no reduction.
Thread = channelConsecutive threads read consecutive channels in an NHWC-style layout, converting a scatter into coalesced vector reads (P15).
Shared staging of locations/weightsThe L·P sample coordinates and attention weights are identical for every channel in the block. Loading them once into shared memory instead of per-channel removes a redundant read of the same data by every thread.
Unrolled L·P loopTypically 4 levels × 4–8 points = 16–32 iterations, known at compile time in practice. Templating on it lets the compiler unroll and hoist.
Vectorization (the next step)If Dh is a multiple of 4 and the base is aligned, have each thread handle a float4 of channels: the four neighbour reads become 16-byte loads and the memory-instruction count drops fourfold. This is usually the single biggest remaining win.

The property test that proves it's right, and that you should volunteer: the attention weights are softmax-normalized over (L·P) for each (query, head). So if you feed a constant feature map of all ones and sample only interior points, every output must be exactly 1.0. That's a strong, cheap invariant that catches weight-indexing bugs, level-offset bugs, and bilinear-weight bugs in one shot — and it was used to validate the semantics above. Border samples legitimately fall below 1.0 because bilinear blends with the zero pad, which is itself a useful thing to know and to say.

21P17 · Voxelization and scatter

problem 17Turn an unordered point cloud into a sparse voxel grid. Watch the contention.
The shape of the problem

10⁵–10⁶ unordered points, each mapping to a voxel index. You need a compact list of occupied voxels and per-voxel aggregated features. The difficulties are all irregularity: atomic contention in dense regions, load imbalance across voxels, and — the one that matters most at GM — determinism.

// ---------- approach A: GPU hash table, one pass, atomics ----------
// Open addressing with linear probing. Size the table ~2x the expected
// occupancy so the load factor stays low and probes stay short.
__device__ __forceinline__ uint32_t hash_u32(uint32_t k) {
    k ^= k >> 16; k *= 0x85ebca6bu; k ^= k >> 13; k *= 0xc2b2ae35u; k ^= k >> 16;
    return k;
}

#define EMPTY 0xffffffffu

__global__ void voxelize(const float* __restrict__ pts,   // [N][4] x,y,z,i
                         uint32_t* __restrict__ keys,     // [T] hash slots
                         int* __restrict__ slot_to_voxel, // [T] compact index
                         int* __restrict__ voxel_count,   // scalar
                         int* __restrict__ point_slot,    // [N] result
                         int N, int T,
                         float3 origin, float3 inv_vsize, int3 dims) {
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < N; i += blockDim.x * gridDim.x) {
        const float x = pts[4 * i + 0], y = pts[4 * i + 1], z = pts[4 * i + 2];
        const int gx = (int)floorf((x - origin.x) * inv_vsize.x);
        const int gy = (int)floorf((y - origin.y) * inv_vsize.y);
        const int gz = (int)floorf((z - origin.z) * inv_vsize.z);
        if (gx < 0 || gx >= dims.x || gy < 0 || gy >= dims.y ||
            gz < 0 || gz >= dims.z) { point_slot[i] = -1; continue; }

        const uint32_t key = (uint32_t)((gz * dims.y + gy) * dims.x + gx);
        uint32_t slot = hash_u32(key) % (uint32_t)T;

        // linear probe with atomicCAS to claim an empty slot
        while (true) {
            const uint32_t prev = atomicCAS(&keys[slot], EMPTY, key);
            if (prev == EMPTY) {
                // we claimed it: allocate a compact voxel id
                slot_to_voxel[slot] = atomicAdd(voxel_count, 1);
                point_slot[i] = slot;
                break;
            }
            if (prev == key) { point_slot[i] = slot; break; }  // already ours
            slot = (slot + 1u) % (uint32_t)T;                  // probe on
        }
    }
}

// ---------- approach B: sort-then-segment — SLOWER TO WRITE, DETERMINISTIC ----------
// 1. compute the voxel key for every point                    (map)
// 2. cub::DeviceRadixSort::SortPairs(key, point_index)         (sort)
// 3. cub::DeviceRunLengthEncode / a boundary-flag scan         (segment)
// 4. one warp per segment reduces its points IN A FIXED ORDER  (reduce)
// => the summation order is now a deterministic function of the input,
//    so the result is bit-identical run to run. This is the version an
//    AV stack that must replay logged drives usually wants.
ProblemMitigation
Atomic contentionA wall or a nearby vehicle puts thousands of points in a handful of voxels. Warp-aggregate before the atomic (P5's __match_any_sync trick), or privatize per block when the active voxel set is small.
Load imbalanceVoxel occupancy varies by orders of magnitude. One thread per voxel means one thread runs 100× longer than its neighbours; prefer one warp per voxel, or bin voxels by point count and launch a kernel per bin.
Hash collisionsLinear probing under a high load factor causes divergent probe lengths within a warp. Size the table generously (load factor ≤ 0.5) and use a decent mixing function.
DeterminismatomicAdd on integers is exact regardless of order; on floats it is not. Aggregating point features with float atomics makes the whole pipeline non-reproducible in the last bits. Approach B — or fixed-point accumulation — is the deterministic answer.
The trade-off to name out loud

"The hash version is one pass and fast; the sort version costs a radix sort but gives bit-exact reproducibility. For a system that has to replay logged drives to reproduce a field incident, I'd want to know which side of that trade the team is on before choosing — and I'd expect the answer to be that the deployed path must be deterministic." That sentence connects a kernel decision to a safety-relevant system requirement, which is precisely the seniority this req is testing for.

22P18 · NMS with a bitmask

problem 18Non-maximum suppression on n boxes. The sequential dependency looks fatal — resolve it.
Why it's a good interview problem

NMS is inherently sequential: whether box j survives depends on which higher-scoring boxes already survived. The standard GPU answer is a genuinely clever decomposition — split it into a massively parallel O(n²) part and a tiny sequential part — and the reasoning transfers to any "parallel prefix over a dependency" problem. It's also real AV work: NMS is a classic latency spike because its cost is data-dependent (Vol. I Tree 2).

THE DECOMPOSITION ─────────────────────────────────────────────────────────────── 0 SORT boxes by score, descending (cub::DeviceRadixSort) 1 PARALLEL: compute the full pairwise suppression matrix, but store it as BITS — box i's row is a bitmask of every j > i that i would suppress. n x ceil(n/64) uint64 words. This is O(n^2) work but perfectly parallel and coalesced. 2 SEQUENTIAL: sweep i = 0..n-1 on ONE thread (or the host). If i is not yet removed, keep it and OR its mask into the running "removed" set. n/64 words per step — trivially cheap. ─────────────────────────────────────────────────────────────── The insight: the sequential part only needs the BITMASK, which is n^2/64 words, not the boxes. So the expensive part parallelizes and the dependent part is reduced to a few thousand OR ops. (Verified: this bitmask formulation reproduces a reference NMS exactly across randomized trials.)
struct Box { float x1, y1, x2, y2; };          // corner form

__device__ __forceinline__ float iou(const Box& a, const Box& b) {
    const float iw = fminf(a.x2, b.x2) - fmaxf(a.x1, b.x1);
    const float ih = fminf(a.y2, b.y2) - fmaxf(a.y1, b.y1);
    if (iw <= 0.f || ih <= 0.f) return 0.f;    // early out on disjoint
    const float inter = iw * ih;
    const float ua = (a.x2 - a.x1) * (a.y2 - a.y1)
                   + (b.x2 - b.x1) * (b.y2 - b.y1) - inter;
    return ua > 0.f ? inter / ua : 0.f;
}

#define NMS_BLK 64                              // one uint64 mask word per block-col

// Grid is 2-D over blocks of 64 boxes: block (bx, by) computes the
// suppression bits between row-block `by` and column-block `bx`.
__global__ void nms_mask(const Box* __restrict__ boxes, int n,
                         float thresh, unsigned long long* __restrict__ mask) {
    const int row_start = blockIdx.y * NMS_BLK;
    const int col_start = blockIdx.x * NMS_BLK;
    const int row_size = min(n - row_start, NMS_BLK);
    const int col_size = min(n - col_start, NMS_BLK);

    // stage this column block once: every row in the block compares to it
    __shared__ Box s_col[NMS_BLK];
    if (threadIdx.x < col_size) s_col[threadIdx.x] = boxes[col_start + threadIdx.x];
    __syncthreads();

    if (threadIdx.x >= row_size) return;
    const int i = row_start + threadIdx.x;
    const Box bi = boxes[i];

    unsigned long long bits = 0ull;
    // only j > i can be suppressed BY i (boxes are score-sorted)
    const int start = (row_start == col_start) ? threadIdx.x + 1 : 0;
    for (int j = start; j < col_size; ++j)
        if (iou(bi, s_col[j]) > thresh) bits |= (1ull << j);

    const int words = (n + NMS_BLK - 1) / NMS_BLK;
    mask[(size_t)i * words + blockIdx.x] = bits;
}

// Step 2: the cheap sequential sweep. Small enough to run on the host,
// or in a single-thread kernel to avoid a device->host round trip.
__global__ void nms_reduce(const unsigned long long* __restrict__ mask,
                           int n, int* __restrict__ keep, int* __restrict__ n_keep) {
    const int words = (n + NMS_BLK - 1) / NMS_BLK;
    extern __shared__ unsigned long long removed[];   // `words` entries
    for (int w = threadIdx.x; w < words; w += blockDim.x) removed[w] = 0ull;
    __syncthreads();

    if (threadIdx.x != 0) return;               // genuinely sequential
    int cnt = 0;
    for (int i = 0; i < n; ++i) {
        if (removed[i / NMS_BLK] & (1ull << (i % NMS_BLK))) continue;
        keep[cnt++] = i;
        const unsigned long long* row = mask + (size_t)i * words;
        for (int w = 0; w < words; ++w) removed[w] |= row[w];
    }
    *n_keep = cnt;
}
The three follow-ups, with answers

"It's O(n²) memory." True: n=10,000 gives ~1.25 M words = 10 MB. Cap n by pre-filtering on score and taking the top-K first (P19) — which is what every production implementation does, and which also bounds the worst case, the real-time property that matters here.

"Can you avoid the sequential part?" Not exactly, but you can shrink it. Class-wise NMS runs independent problems in parallel (batch them). And the sweep is over bit-words, so even n=10,000 is ~1.5 M OR operations — microseconds.

"Is it deterministic?" Yes, given a deterministic sort — which matters, because ties in score are common with quantized outputs. Use a stable sort or break ties by index, or the kept set can vary run to run.

23P19 · Top-K and box decode

problem 19Select the K highest-scoring detections from ~100k anchors and decode their boxes.
Why this bounds your latency

This is the stage that caps the data-dependent work downstream. Detection heads emit tens or hundreds of thousands of candidates; NMS is O(n²). Top-K is what turns an unbounded worst case into a fixed one — which makes it a real-time design element, not just an optimization.

// ---------- practical two-stage top-K ----------
// Stage 1: each block keeps a per-block top-K in shared memory using a
// tiny insertion structure; K is small (100-1000) so this is cheap.
// Stage 2: merge the per-block candidates (gridDim.x * K of them) with a
// single sort or a second top-K pass.
//
// For most real cases, DON'T hand-roll: a radix sort over (score, index)
// pairs then take the first K is simpler, deterministic, and fast:
//     cub::DeviceRadixSort::SortPairsDescending(...)
// Reach for a custom kernel only when K is tiny and n is huge, where a
// threshold-then-compact approach wins.

// ---------- the threshold-and-compact trick (the fast path) ----------
// If you can estimate a score threshold that admits slightly more than K
// candidates, one pass of predicate + scan-compaction (P4) replaces the
// sort entirely -- O(n) instead of O(n log n), and it fuses with decode.
__global__ void score_filter_and_decode(
        const float* __restrict__ cls_logits,   // [A, NC]
        const float* __restrict__ box_deltas,   // [A, 4]
        const float* __restrict__ anchors,      // [A, 4] (cx, cy, w, h)
        float score_thresh, int A, int NC,
        Box*   __restrict__ out_boxes,
        float* __restrict__ out_scores,
        int*   __restrict__ out_cls,
        int*   __restrict__ out_count, int max_out) {

    for (int a = blockIdx.x * blockDim.x + threadIdx.x;
         a < A; a += blockDim.x * gridDim.x) {

        // ---- argmax over classes, in one pass ----
        float best = -INFINITY; int best_c = -1;
        for (int c = 0; c < NC; ++c) {
            const float v = cls_logits[(size_t)a * NC + c];
            if (v > best) { best = v; best_c = c; }
        }
        const float score = 1.f / (1.f + __expf(-best));      // sigmoid
        if (score < score_thresh) continue;                   // most anchors die here

        // ---- decode: deltas + anchor -> corner box ----
        const float acx = anchors[4 * a + 0], acy = anchors[4 * a + 1];
        const float aw  = anchors[4 * a + 2], ah  = anchors[4 * a + 3];
        const float dx = box_deltas[4 * a + 0], dy = box_deltas[4 * a + 1];
        const float dw = box_deltas[4 * a + 2], dh = box_deltas[4 * a + 3];

        const float cx = fmaf(dx, aw, acx);
        const float cy = fmaf(dy, ah, acy);
        // clamp the exponent: an unbounded exp on a bad delta produces inf
        // boxes, which then poison IoU with NaN. Guard it.
        const float w = aw * __expf(fminf(dw, 4.0f));
        const float h = ah * __expf(fminf(dh, 4.0f));

        // ---- reserve an output slot ----
        const int slot = atomicAdd(out_count, 1);
        if (slot >= max_out) continue;                        // BOUNDED output
        out_boxes[slot]  = Box{ cx - 0.5f * w, cy - 0.5f * h,
                                cx + 0.5f * w, cy + 0.5f * h };
        out_scores[slot] = score;
        out_cls[slot]    = best_c;
    }
}
PointSay this
Fusing filter + decodeDecoding all A anchors then filtering wastes most of the work. Filtering first and decoding only survivors is typically a 10–50× reduction in decode work — and it fuses into one kernel with one pass over the logits.
The exp clampA garbage delta produces an infinite box, which makes IoU return NaN, which silently corrupts NMS. Clamping is a one-instruction guard against a class of field failure. Volunteering this is a strong robustness signal.
Bounded outputmax_out caps the downstream cost. Without it a pathological frame — heavy rain, a crowd — makes NMS quadratically slow and blows the deadline. This is the "bound the data-dependent work" principle from Vol. I §05, in code.
DeterminismatomicAdd on the counter means the order of the output array varies run to run, even though the set doesn't. Sort by (score, index) afterwards, or the downstream NMS tie-breaking becomes nondeterministic.
Use the libraryCUB's radix sort and DeviceSelect::If are excellent, deterministic, and tuned. Hand-roll only to fuse with decode — which, as above, is a real reason.

24P20 · Fused depthwise + pointwise

problem 20A depthwise 3×3 followed by a 1×1. Why is this slow, and how would you fix it?
The diagnosis, from arithmetic intensity

Depthwise convolution does k² FLOPs per output element with no channel reduction. For 3×3 that's 9 MACs while reading ~9 inputs and 9 weights — AI ≈ 1. It is the textbook memory-bound layer, and in efficiency-oriented backbones (MobileNet-style, and the compact backbones AV stacks favour) it is frequently the bottleneck despite holding a tiny share of the FLOPs. cuDNN handles it, but the separate depthwise and pointwise kernels round-trip a full activation tensor through DRAM between them.

// Fuse: compute a tile of the depthwise output in SHARED MEMORY, then
// immediately consume it with the 1x1 (which is a per-pixel GEMM over
// channels) -- so the intermediate never reaches DRAM.
//
// Layout NHWC. Block covers a TILE_H x TILE_W spatial patch, all channels.
template <int TILE_H, int TILE_W, int CIN_CHUNK>
__global__ void fused_dw_pw(
        const float* __restrict__ in,     // [H][W][Cin]
        const float* __restrict__ dw_w,   // [3][3][Cin]
        const float* __restrict__ pw_w,   // [Cout][Cin]
        const float* __restrict__ bias,   // [Cout]
        float* __restrict__ out,          // [H][W][Cout]
        int H, int W, int Cin, int Cout) {

    // halo of 1 on each side for the 3x3
    __shared__ float s_in[TILE_H + 2][TILE_W + 2][CIN_CHUNK];
    __shared__ float s_dw[TILE_H][TILE_W][CIN_CHUNK];   // depthwise result

    const int h0 = blockIdx.y * TILE_H, w0 = blockIdx.x * TILE_W;
    const int c0 = blockIdx.z * CIN_CHUNK;

    // ---- stage input tile + halo, zero-padded at the image border ----
    for (int idx = threadIdx.x; idx < (TILE_H + 2) * (TILE_W + 2) * CIN_CHUNK;
         idx += blockDim.x) {
        const int c  = idx % CIN_CHUNK;
        const int ww = (idx / CIN_CHUNK) % (TILE_W + 2);
        const int hh = (idx / CIN_CHUNK) / (TILE_W + 2);
        const int gh = h0 + hh - 1, gw = w0 + ww - 1, gc = c0 + c;
        s_in[hh][ww][c] = (gh >= 0 && gh < H && gw >= 0 && gw < W && gc < Cin)
                        ? in[((size_t)gh * W + gw) * Cin + gc] : 0.f;
    }
    __syncthreads();

    // ---- depthwise 3x3, straight into shared ----
    for (int idx = threadIdx.x; idx < TILE_H * TILE_W * CIN_CHUNK; idx += blockDim.x) {
        const int c  = idx % CIN_CHUNK;
        const int ww = (idx / CIN_CHUNK) % TILE_W;
        const int hh = (idx / CIN_CHUNK) / TILE_W;
        float acc = 0.f;
        #pragma unroll
        for (int ky = 0; ky < 3; ++ky)
            #pragma unroll
            for (int kx = 0; kx < 3; ++kx)
                acc = fmaf(s_in[hh + ky][ww + kx][c],
                           dw_w[(ky * 3 + kx) * Cin + c0 + c], acc);
        s_dw[hh][ww][c] = fmaxf(acc, 0.f);           // fused ReLU
    }
    __syncthreads();

    // ---- pointwise 1x1: per pixel, a Cout x Cin dot product ----
    // NOTE: this is the partial-sum over ONE Cin chunk; the caller either
    // launches one block per chunk and atomically accumulates, or loops
    // chunks inside the block. Looping inside is preferable -- it keeps the
    // accumulator in registers and avoids atomics entirely.
    for (int idx = threadIdx.x; idx < TILE_H * TILE_W * Cout; idx += blockDim.x) {
        const int oc = idx % Cout;
        const int ww = (idx / Cout) % TILE_W;
        const int hh = (idx / Cout) / TILE_W;
        if (h0 + hh >= H || w0 + ww >= W) continue;
        float acc = 0.f;
        #pragma unroll 4
        for (int c = 0; c < CIN_CHUNK; ++c)
            acc = fmaf(s_dw[hh][ww][c], pw_w[(size_t)oc * Cin + c0 + c], acc);
        atomicAdd(&out[((size_t)(h0 + hh) * W + (w0 + ww)) * Cout + oc],
                  acc + (c0 == 0 ? bias[oc] : 0.f));
    }
}
What the fusion buys, and the honest caveat

Buys: the depthwise output — a full H×W×Cin tensor — never touches DRAM. On a bandwidth-starved automotive part (Vol. I §07) that is the entire win, and it's larger than any arithmetic improvement available.

Caveat to state: the shared-memory tile plus halo costs occupancy, and the CIN_CHUNK loop above uses an atomic to accumulate partial channel sums. The better structure loops all input-channel chunks inside the block so the pointwise accumulator lives in registers and no atomic is needed — at the cost of a larger shared footprint. Naming that trade, rather than presenting the first version as final, is the point.

And the meta-answer: check whether TensorRT already fused it. The builder fuses conv+bias+activation automatically, and it may or may not fuse a depthwise-pointwise pair depending on version and shapes. The first move on a real slow layer is to inspect the built engine's layer list to see what actually happened (Vol. I §42) — before writing a plugin, which becomes a fusion barrier of its own (Vol. I §27). Writing the kernel is the second-best answer; knowing when it's the right one is the job.

Part VI — Attention

Two GEMMs and a softmax.

Transformer-based architectures are named explicitly in the req's preferred qualifications, and attention is where kernel engineering meets model architecture. Two problems: the fused encoder-style kernel, and the degenerate decode case.

25P21 · Fused attention, Flash-style

problem 21Compute softmax(QKᵀ/√d)·V without materializing the N×N score matrix.
The problem with the obvious implementation

Naive attention writes an N×N score matrix to DRAM and reads it back twice. At N=4096 in fp16 that is 32 MB per head, and the softmax step has arithmetic intensity ≈ 1 — so on a ~200 GB/s automotive part it consumes the entire frame budget. FlashAttention removes the materialization entirely by tiling over K/V and carrying a running softmax.

The identity that makes it exact

Softmax is shift-invariant, so subtracting a running max and correcting earlier partial sums by exp(m_old − m_new) yields exactly the two-pass result. This is not an approximation — it was verified against a naive reference at three different block sizes before being written here.

THE ONLINE UPDATE, PER Q-ROW, PER K/V BLOCK ─────────────────────────────────────────────────────────────── S_blk = Q_tile · K_blkᵀ · scale (kept in registers) m_new = max(m_old, rowmax(S_blk)) corr = exp(m_old − m_new) ← rescales the PAST P_blk = exp(S_blk − m_new) l_new = corr · l_old + rowsum(P_blk) ← running denominator O_acc = corr · O_acc + P_blk · V_blk ← RESCALE, then accumulate ─────────────────────────────────────────────────────────────── after the last block: O = O_acc / l_new Memory: O(N) instead of O(N²). FLOPs: slightly MORE than naive (the rescaling). Time: much less — because it was never compute-bound. ⇒ This is an IO-complexity win. Say exactly that.
// One block per (batch, head, query-tile). BR queries x BC keys per step.
// Simplified for clarity: fp32 math, no causal mask, D <= 128 in registers.
// A production version uses fp16 MMA fragments for both GEMMs and stages
// K/V through shared memory with cp.async -- but the CONTROL FLOW is this.
template <int BR, int BC, int D>
__global__ __launch_bounds__(BR * 32)
void flash_attention(const float* __restrict__ Q,   // [B, H, N, D]
                     const float* __restrict__ K,
                     const float* __restrict__ V,
                     float* __restrict__ O,
                     int N, float scale) {
    const int bh   = blockIdx.y;                // fused (batch, head)
    const int q0   = blockIdx.x * BR;
    const int lane = threadIdx.x & 31;
    const int warp = threadIdx.x >> 5;          // one warp per query row
    const int qi   = q0 + warp;

    const size_t base = (size_t)bh * N * D;

    // ---- this warp's query row, held in registers across the whole loop ----
    float q_reg[D / 32];
    #pragma unroll
    for (int i = 0; i < D / 32; ++i) {
        const int d = i * 32 + lane;
        q_reg[i] = (qi < N) ? Q[base + (size_t)qi * D + d] : 0.f;
    }

    // ---- running softmax state + output accumulator ----
    float m_run = -INFINITY;                    // running max
    float l_run = 0.f;                          // running denominator
    float o_acc[D / 32];
    #pragma unroll
    for (int i = 0; i < D / 32; ++i) o_acc[i] = 0.f;

    __shared__ float sK[BC][D];
    __shared__ float sV[BC][D];

    for (int k0 = 0; k0 < N; k0 += BC) {
        // ---- stage a K/V block cooperatively ----
        for (int idx = threadIdx.x; idx < BC * D; idx += blockDim.x) {
            const int r = idx / D, c = idx % D;
            const int kr = k0 + r;
            sK[r][c] = (kr < N) ? K[base + (size_t)kr * D + c] : 0.f;
            sV[r][c] = (kr < N) ? V[base + (size_t)kr * D + c] : 0.f;
        }
        __syncthreads();

        if (qi < N) {
            // ---- S_blk = q · Kᵀ for this block, one score per key ----
            float s_blk[BC];
            float m_blk = -INFINITY;
            #pragma unroll
            for (int j = 0; j < BC; ++j) {
                float dot = 0.f;
                #pragma unroll
                for (int i = 0; i < D / 32; ++i)
                    dot += q_reg[i] * sK[j][i * 32 + lane];
                // reduce the partial dot across the warp's 32 lanes
                #pragma unroll
                for (int off = 16; off > 0; off >>= 1)
                    dot += __shfl_xor_sync(0xffffffffu, dot, off);
                s_blk[j] = (k0 + j < N) ? dot * scale : -INFINITY;
                m_blk = fmaxf(m_blk, s_blk[j]);
            }

            // ---- the online update ----
            const float m_new = fmaxf(m_run, m_blk);
            const float corr  = (m_run == -INFINITY) ? 0.f : __expf(m_run - m_new);

            float l_blk = 0.f;
            #pragma unroll
            for (int j = 0; j < BC; ++j) {
                s_blk[j] = (s_blk[j] == -INFINITY) ? 0.f : __expf(s_blk[j] - m_new);
                l_blk += s_blk[j];
            }

            // rescale the accumulator, then add this block's contribution
            #pragma unroll
            for (int i = 0; i < D / 32; ++i) {
                float pv = 0.f;
                #pragma unroll
                for (int j = 0; j < BC; ++j) pv += s_blk[j] * sV[j][i * 32 + lane];
                o_acc[i] = o_acc[i] * corr + pv;
            }
            l_run = l_run * corr + l_blk;
            m_run = m_new;
        }
        __syncthreads();                        // before the next block overwrites sK/sV
    }

    // ---- normalize once, at the end ----
    if (qi < N) {
        const float inv = 1.f / l_run;
        #pragma unroll
        for (int i = 0; i < D / 32; ++i)
            O[base + (size_t)qi * D + i * 32 + lane] = o_acc[i] * inv;
    }
}
Point to raiseContent
Why no backward passThis is inference. FlashAttention's famous recomputation trick exists to avoid storing the score matrix for the backward pass — irrelevant here. Saying so shows you know why the algorithm has the shape it does.
The AV sequence lengthsObject queries are in the hundreds; a flattened BEV grid is a few thousand. So N is moderate and this is a well-shaped pair of GEMMs — unlike LLM decode. Tiling still wins because the intermediate never hits DRAM.
The production versionfp16 operands with mma.sync for both GEMMs, K/V staged with cp.async in a multi-stage pipeline, swizzled shared layouts, warp-specialized producer/consumer. That is CUTLASS's fused-MHA kernel, and the honest thing to say is that you'd start from it rather than hand-roll (Vol. I §16).
The trade-off knobLarger BC means fewer rescalings and better GEMM shapes, but more shared memory and lower occupancy. Same tension as GEMM tile sizing (P12).

26P22 · Decode GEMV with a KV cache

problem 22One new query attends to a cache of T past keys/values. What's the bound?
The analysis flips completely

With a single query, both GEMMs become GEMVs. Per step you read the whole KV cache — 2·T·D elements — and do ~4·T·D FLOPs. AI ≈ 1: entirely bandwidth-bound on the cache. Latency is cache_bytes / bandwidth, full stop. No tiling, no tensor cores, no arithmetic restructuring changes that.

// One block per (batch, head). The whole kernel is a streaming reduction
// over the cache with an online softmax -- P6 and P21 combined.
template <int D, int BLOCK>
__global__ void decode_attention(const float* __restrict__ q,     // [B,H,D]
                                 const float* __restrict__ k_cache, // [B,H,T,D]
                                 const float* __restrict__ v_cache,
                                 float* __restrict__ o,           // [B,H,D]
                                 int T, float scale) {
    const int bh = blockIdx.x;
    const size_t cbase = (size_t)bh * T * D;

    __shared__ float sq[D];
    for (int d = threadIdx.x; d < D; d += BLOCK) sq[d] = q[(size_t)bh * D + d];
    __syncthreads();

    // running softmax + accumulator, per thread over its slice of D
    float m_run = -INFINITY, l_run = 0.f;
    float acc[D / BLOCK == 0 ? 1 : D / BLOCK];
    #pragma unroll
    for (int i = 0; i < D / BLOCK; ++i) acc[i] = 0.f;

    for (int t = 0; t < T; ++t) {
        // score = dot(q, k_t) -- a block-wide reduction
        float partial = 0.f;
        for (int d = threadIdx.x; d < D; d += BLOCK)
            partial += sq[d] * k_cache[cbase + (size_t)t * D + d];
        const float s = blockReduceSum<BLOCK>(partial) * scale;

        __shared__ float s_score;
        if (threadIdx.x == 0) s_score = s;
        __syncthreads();

        const float m_new = fmaxf(m_run, s_score);
        const float corr  = (m_run == -INFINITY) ? 0.f : __expf(m_run - m_new);
        const float p     = __expf(s_score - m_new);

        #pragma unroll
        for (int i = 0; i < D / BLOCK; ++i) {
            const int d = i * BLOCK + threadIdx.x;
            acc[i] = acc[i] * corr + p * v_cache[cbase + (size_t)t * D + d];
        }
        l_run = l_run * corr + p;
        m_run = m_new;
        __syncthreads();
    }

    const float inv = 1.f / l_run;
    #pragma unroll
    for (int i = 0; i < D / BLOCK; ++i)
        o[(size_t)bh * D + i * BLOCK + threadIdx.x] = acc[i] * inv;
}
The optimizations that actually apply

Since you're bandwidth-bound on the cache, every win is about bytes: quantize the KV cache to fp8 or int8 (halves the traffic, halves the latency); use grouped-query attention so several query heads share one KV head (a 4–8× cache reduction); split the T loop across blocks and combine partial softmax states when there aren't enough (batch × head) pairs to fill the GPU — the online form composes, which is exactly why that works. None of these is an arithmetic optimization, and that's the lesson.

Part VII — Quantization

Half the bytes, all the accuracy.

On-vehicle inference is quantized inference (Vol. I §25). These three problems cover the mechanics you'd be asked to implement and, just as importantly, the accuracy contract you'd be asked to defend.

27P23 · Quantize and dequantize

problem 23Symmetric INT8 quantization with per-channel weight scales.
The formulation

Symmetric: q = clamp(round(x / s), −127, 127), x ≈ q · s, with zero-point 0 so no cross-terms appear in the GEMM. Weights get a scale per output channel; activations get one per tensor, because their scale must be known before the data arrives — hence calibration.

// ---------- compute per-channel scales (a segmented absmax reduction) ----------
template <int BLOCK>
__global__ void compute_scales(const float* __restrict__ w,  // [Cout, Cin]
                               float* __restrict__ scale,    // [Cout]
                               int Cin) {
    const int oc = blockIdx.x;
    const float* row = w + (size_t)oc * Cin;

    float amax = 0.f;
    for (int i = threadIdx.x; i < Cin; i += BLOCK)
        amax = fmaxf(amax, fabsf(row[i]));
    amax = blockReduceMax<BLOCK>(amax);

    if (threadIdx.x == 0)
        // guard against an all-zero channel producing scale 0 -> div by zero
        scale[oc] = fmaxf(amax, 1e-12f) / 127.f;
}

// ---------- quantize, vectorized 4-at-a-time into a packed int8x4 word ----------
__global__ void quantize_per_channel(const float* __restrict__ w,
                                     const float* __restrict__ scale,
                                     int8_t* __restrict__ q,
                                     int Cout, int Cin) {
    const int oc = blockIdx.y;
    const float inv_s = 1.f / scale[oc];

    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i * 4 < Cin; i += blockDim.x * gridDim.x) {
        const int c = i * 4;
        if (c + 3 >= Cin) break;                        // scalar tail elsewhere
        const float4 v = *reinterpret_cast<const float4*>(&w[(size_t)oc * Cin + c]);

        // __float2int_rn = round-to-nearest-EVEN, matching PyTorch/ONNX.
        // Using truncation here is a classic silent accuracy bug.
        char4 o;
        o.x = (int8_t)max(-127, min(127, __float2int_rn(v.x * inv_s)));
        o.y = (int8_t)max(-127, min(127, __float2int_rn(v.y * inv_s)));
        o.z = (int8_t)max(-127, min(127, __float2int_rn(v.z * inv_s)));
        o.w = (int8_t)max(-127, min(127, __float2int_rn(v.w * inv_s)));
        *reinterpret_cast<char4*>(&q[(size_t)oc * Cin + c]) = o;
    }
}
Three details that separate a correct kernel from a plausible one

Rounding mode. __float2int_rn is round-to-nearest-even, which is what PyTorch and ONNX specify. C-style casting truncates toward zero and introduces a systematic negative bias — a bug that shows up as a small, consistent accuracy loss nobody can explain.

The clamp is [−127, 127], not [−128, 127]. Symmetric quantization deliberately gives up −128 so the range is symmetric about zero; using −128 breaks the symmetry assumption some kernels rely on.

The zero-channel guard. A pruned or dead channel has absmax 0, giving scale 0 and then a division by zero producing NaN weights. One fmaxf prevents a class of field failure.

The empirical point worth making: per-channel beats per-tensor precisely when channel magnitudes are skewed — which they routinely are in a trained conv layer. A quick experiment on synthetic weights with deliberately skewed per-channel ranges confirms per-channel round-trip error is bounded by half the scale of the largest channel, while per-tensor error is bounded by half the scale of the whole tensor — dramatically worse for the small channels. And per-channel is free at runtime, because the scale folds into the epilogue multiply (P25). That's why it's the default for weights.

28P24 · INT8 GEMM with dp4a

problem 24Integer GEMM using the 4-way dot-product instruction.
What __dp4a does

One instruction, four int8×int8 products, summed into an int32 accumulator: dp4a(a, b, c) = c + Σᵢ a[i]·b[i] over four packed bytes. It's the integer analogue of an FMA and it's available well below tensor-core-class hardware — which makes it the portable INT8 path. (Semantics verified: packing {1,2,3,4} and {5,6,7,8} yields 70.)

// INT8 tiled GEMM: same structure as P11, four-deep in K per instruction.
// A[M,K] int8 row-major, B[K,N] int8 stored COLUMN-major (i.e. Bᵀ) so that
// four consecutive k values are contiguous for BOTH operands -- that
// packing requirement is the main structural difference from fp GEMM.
#define TS8 32
__global__ void gemm_int8_dp4a(const int8_t* __restrict__ A,
                               const int8_t* __restrict__ Bt,   // [N, K]
                               int32_t* __restrict__ C,
                               int M, int N, int K) {
    __shared__ int8_t As[TS8][TS8 * 4];         // 4 k-values per step
    __shared__ int8_t Bs[TS8][TS8 * 4];

    const int row = blockIdx.y * TS8 + threadIdx.y;
    const int col = blockIdx.x * TS8 + threadIdx.x;
    int32_t acc = 0;                            // INT32 accumulate: no overflow
                                                // for K up to ~130k at |127|

    for (int k0 = 0; k0 < K; k0 += TS8 * 4) {
        // stage 4 bytes per thread as one 32-bit word
        const int kk = k0 + threadIdx.x * 4;
        *reinterpret_cast<int32_t*>(&As[threadIdx.y][threadIdx.x * 4]) =
            (row < M && kk + 3 < K)
              ? *reinterpret_cast<const int32_t*>(&A[(size_t)row * K + kk]) : 0;
        *reinterpret_cast<int32_t*>(&Bs[threadIdx.y][threadIdx.x * 4]) =
            (blockIdx.x * TS8 + threadIdx.y < N && kk + 3 < K)
              ? *reinterpret_cast<const int32_t*>(
                    &Bt[(size_t)(blockIdx.x * TS8 + threadIdx.y) * K + kk]) : 0;
        __syncthreads();

        #pragma unroll
        for (int k = 0; k < TS8; ++k) {
            const uint32_t a = *reinterpret_cast<const uint32_t*>(&As[threadIdx.y][k * 4]);
            const uint32_t b = *reinterpret_cast<const uint32_t*>(&Bs[threadIdx.x][k * 4]);
            acc = __dp4a(a, b, acc);            // 4 MACs, one instruction
        }
        __syncthreads();
    }
    if (row < M && col < N) C[(size_t)row * N + col] = acc;
}
PointContent
Why int32 accumulateWorst case per term is 127·127 = 16,129. Even at K = 4096 the sum is ~66 M — comfortably inside int32's ~2.1 B range. Stating this bound shows you checked rather than assumed.
Why Bᵀdp4a needs four consecutive k values packed in a 32-bit word for both operands. A is naturally k-contiguous; B is not unless you store it transposed. Weight layout is a build-time decision, so this costs nothing at runtime — but forgetting it is why a first INT8 GEMM is often slower than fp16.
The modern alternativeOn Turing and later, the INT8 tensor-core path (mma.sync...s8.s8.s32, or CUTLASS's integer GEMM) is substantially faster than dp4a. Say that: dp4a is the portable/fallback path, MMA is the fast path.
What's missingThe output here is raw int32. It is not usable until the scales are applied — which is P25, and which must be fused or the whole INT8 win evaporates.

29P25 · The fused requantize epilogue

problem 25Turn the int32 accumulator into an int8 output — correctly and for free.
Why this is the chapter that makes INT8 actually pay

An INT8 GEMM that writes int32 and then runs a separate dequant-bias-activation-requant kernel has four times the output traffic of the fused version, on an operation whose entire justification was reducing traffic. The requantization must live in the GEMM's epilogue, in registers, before anything is written.

THE SCALE ALGEBRA — collapse it at build time ─────────────────────────────────────────────────────────────── x_real ≈ s_a · a_q (activation, per-tensor scale) w_real ≈ s_w[oc] · w_q (weight, PER-CHANNEL scale) acc_i32 = Σ a_q · w_q y_real = s_a · s_w[oc] · acc_i32 + bias_real y_q = clamp(round(y_real / s_y), −127, 127) ⇒ define M[oc] = (s_a · s_w[oc]) / s_y ← ONE float per channel, computed ONCE at build then y_q = clamp(round(M[oc]·acc + bias_q[oc]), −127, 127) The whole dequant/requant chain collapses to ONE multiply-add per output element, applied while the accumulator is still in a register. That is why fused INT8 is nearly free and unfused INT8 is a loss.
// The epilogue, as you would write it inside the GEMM (or as a CUTLASS
// EpilogueOp). Nothing here touches DRAM until the final packed store.
struct RequantEpilogue {
    const float*  __restrict__ m_scale;   // [Cout] = s_a*s_w[oc]/s_y, precomputed
    const float*  __restrict__ bias;      // [Cout] in REAL units
    float         inv_sy;                 // 1 / s_y
    bool          relu;

    __device__ __forceinline__
    int8_t operator()(int32_t acc, int oc) const {
        float y = fmaf((float)acc, m_scale[oc], bias[oc] * inv_sy);
        if (relu) y = fmaxf(y, 0.f);
        const int q = __float2int_rn(y);          // round-to-nearest-even
        return (int8_t)max(-127, min(127, q));
    }
};

// Vectorized store: pack four int8 results into one 32-bit write.
__global__ void requant_store(const int32_t* __restrict__ acc,
                              int8_t* __restrict__ out,
                              RequantEpilogue ep, int M, int N) {
    for (int idx = blockIdx.x * blockDim.x + threadIdx.x;
         idx * 4 < M * N; idx += blockDim.x * gridDim.x) {
        const int base = idx * 4;
        const int row  = base / N;
        char4 o;
        // NOTE: this assumes a row does not straddle the 4-wide group.
        // Guard it, or require N % 4 == 0 and assert it at build time.
        o.x = ep(acc[base + 0], (base + 0) - row * N);
        o.y = ep(acc[base + 1], (base + 1) - row * N);
        o.z = ep(acc[base + 2], (base + 2) - row * N);
        o.w = ep(acc[base + 3], (base + 3) - row * N);
        *reinterpret_cast<char4*>(&out[base]) = o;
    }
}
What to say to close this out

"The scales collapse into a single per-channel multiplier computed at build time, so requantization is one FMA per element in the epilogue — essentially free. Written as a separate kernel it would quadruple the output traffic and undo the reason we quantized. In production this is a CUTLASS epilogue functor or a TensorRT fusion, and if I saw an unfused dequant kernel in a profile, that would be the first thing I'd chase."

If they push on accuracy: bias should be quantized into the accumulator's domain (or kept in float and folded as above), and the activation scale s_y comes from calibration — so a bad calibration set shows up here as clipping. That's the bridge to Vol. I Tree 3.

// ======================= BUG 1 =======================
__global__ void tiled_sum(const float* in, float* out, int n) {
    __shared__ float tile[256];
    float acc = 0.f;
    for (int base = 0; base < n; base += 256) {
        int i = base + threadIdx.x;
        tile[threadIdx.x] = (i < n) ? in[i] : 0.f;
        __syncthreads();
        for (int k = 0; k < 256; ++k) acc += tile[k];
        // <<< next iteration overwrites `tile` here
    }
    if (threadIdx.x == 0) out[blockIdx.x] = acc;
}
bug 1Missing the second barrier
The defect

There is a __syncthreads() after the write but none after the read loop. A fast warp loops around and overwrites tile[threadIdx.x] for iteration base+256 while a slow warp is still summing iteration base.

Why it's vicious

It passes on small inputs and under a debugger, because with one or two warps the divergence never materializes. It fails intermittently at scale — the worst possible failure mode.

How you'd catch it

compute-sanitizer --tool racecheck reports it immediately. And the review rule from Vol. I §47: a shared buffer reused across loop iterations needs a barrier at both ends.

__global__ void transpose(const float* in, float* out, int N) {

bug 2Unguarded partial tile — and a 32-way bank conflict
The defect

N = 1000 is not a multiple of 32, so the last block row/column reads and writes out of bounds — undefined behaviour, often a silent wrong answer or an illegal access. Every global access needs an if (x < N && y < N) guard.

The bonus defect

tile[32][32] with a column read tile[threadIdx.x][threadIdx.y] is a 32-way bank conflict (P3). Pad to [32][33]. Finding both is the strong answer.

How you'd catch it

compute-sanitizer --tool memcheck for the OOB; Nsight Compute's shared-memory section for the conflict. And a test harness that deliberately uses non-power-of-two sizes (§35).

// ======================= BUG 3 =======================
__device__ float warpReduce(volatile float* s, int tid) {
    s[tid] += s[tid + 16];
    s[tid] += s[tid + 8];
    s[tid] += s[tid + 4];
    s[tid] += s[tid + 2];
    s[tid] += s[tid + 1];
    return s[tid];
}
bug 3Warp-synchronous programming, post-Volta
The defect

This is the classic pre-Volta idiom: it assumes the 32 lanes of a warp execute in lockstep so no barrier is needed. Since Volta, threads have independent program counters and may reconverge only where you tell them to. volatile forces the memory traffic but does not provide the execution ordering this code depends on.

The fix

__syncwarp() between the steps, or — far better — rewrite with __shfl_down_sync (P2), which is faster, needs no shared memory, and makes the participating mask explicit.

Why it matters here

This idiom is all over older CUDA codebases. Recognizing it in a legacy AV kernel and knowing why it's now a latent bug is a genuinely senior catch.

// ======================= BUG 4 =======================
__global__ void dot_fp16(const __half* a, const __half* b, __half* out, int n) {
    __half acc = __float2half(0.f);
    for (int i = blockIdx.x * blockDim.x + threadIdx.x;
         i < n; i += blockDim.x * gridDim.x)
        acc = __hadd(acc, __hmul(a[i], b[i]));
    // ... block reduction over `acc`, also in __half ...
    if (threadIdx.x == 0) atomicAdd(out, acc);
}
bug 4Accumulating in the storage precision
The defect

fp16 has a ~10-bit mantissa, so once the accumulator exceeds about 2048 the addition of a small term is silently discarded — the classic stagnation failure. Over a long reduction the result is badly wrong, and it degrades gracefully enough to pass a loose tolerance test.

The fix

Accumulate in fp32 and convert once at the end. This is why every tensor-core MMA accumulates wider than it multiplies (P13), and why the review checklist has an "accumulator type" line.

How you'd catch it

The precision ladder in §35: compare fp16 against an fp64 reference and watch the error grow with n. Error that scales with reduction length is the signature of accumulator precision, not of a logic bug.

// ======================= BUG 5 =======================
__global__ void scale_rows(float* x, const float* row_scale, int M, int N) {
    int r = blockIdx.x;
    int c = threadIdx.x;
    __shared__ float s;
    if (threadIdx.x == 0) s = row_scale[r];
    x[r * N + c] *= s;                      // <<<
}
bug 5Reading shared memory before it's written
The defect

No __syncthreads() between thread 0's write to s and every other thread's read. Threads 1–31 may read uninitialized shared memory. There's a second defect too: c = threadIdx.x with no bound, so this only works if N == blockDim.x exactly.

The subtlety worth voicing

It often appears to work, because thread 0 and the rest of warp 0 usually execute together — so the failure begins at thread 32. That "works for the first warp" pattern is characteristic of a missing barrier, and it's a useful diagnostic smell.

The better fix

Skip shared memory entirely: const float s = row_scale[r]; in every thread is a broadcast read that the hardware handles efficiently. Not needing the barrier beats remembering it.

32D3 · The numerical mismatch

drill"Your kernel matches PyTorch to 1e-3 but the model's mAP dropped 2 points."
Why 1e-3 is not reassuring

An error that is small in norm can be large where it matters. "Close enough elementwise" is not the same as "equivalent for the task."

   │     precision. A CONSTANT BIAS, a spatial gradient, or an error

Bit-exact replay of logged sensor data is how a field incident is reproduced and how a software change is validated. If replay isn't deterministic, you cannot prove that a fix fixed anything. This elevates an ordinary floating-point detail into a system requirement (Vol. I §48).

Same input, different output. Find the source.
   │

   │     timestamps, frame ordering, any wall-clock or uninitialized

   │

// kernel_test.hpp — a reusable harness. Host-side C++17 + STL.
#pragma once
#include <vector>
#include <random>
#include <cmath>
#include <string>
#include <functional>
#include <algorithm>
#include <cstring>
#include <cstdio>

struct Budget {
    double eps;          // machine epsilon of the COMPUTE dtype
    long   reduction_n;  // length of the longest reduction
    // Worst-case relative error for a tree reduction is ~ sqrt(n)*eps;
    // sequential is ~n*eps. Use the tree bound with slack.
    double rel_tol() const { return 8.0 * std::sqrt((double)reduction_n) * eps; }
};
inline constexpr double EPS_FP32 = 1.19e-7;
inline constexpr double EPS_FP16 = 9.77e-4;   // ~2^-10
inline constexpr double EPS_BF16 = 7.81e-3;   // ~2^-7

struct Report { bool ok; double max_abs; double max_rel; std::string note; };

// Compare a candidate against a float64 ORACLE, judged by an error budget.
// Never compare two fast implementations to each other.
inline Report compare(const std::vector<double>& oracle,
                      const std::vector<float>&  cand,
                      Budget b, double abs_floor = 1e-6) {
    double ma = 0, mr = 0;
    for (size_t i = 0; i < oracle.size(); ++i) {
        if (!std::isfinite(cand[i]))
            return { false, INFINITY, INFINITY, "non-finite at " + std::to_string(i) };
        const double a = std::fabs(cand[i] - oracle[i]);
        const double r = a / std::max(std::fabs(oracle[i]), abs_floor);
        ma = std::max(ma, a); mr = std::max(mr, r);
    }
    const bool ok = mr <= b.rel_tol();
    return { ok, ma, mr, ok ? "within budget" : "OUTSIDE budget" };
}

// The full battery. Pass lambdas that run the reference and the kernel.
template <typename RefFn, typename KernelFn>
bool run_battery(const char* name, RefFn ref, KernelFn kern, Budget b) {
    bool all = true;
    auto say = [&](const char* what, bool ok, const std::string& extra = "") {
        printf("  [%s] %-34s %s\n", ok ? "PASS" : "FAIL", what, extra.c_str());
        all = all && ok;
    };

    // 1. SHAPE FUZZING — tile-quantized kernels break at the edges, always.
    //    Deliberately include non-multiples of 32, primes, 1, and 0.
    const std::vector<int> sizes = {0, 1, 2, 31, 32, 33, 63, 64, 127, 128,
                                    255, 256, 257, 1000, 1023, 1024, 4097};
    for (int n : sizes) {
        auto o = ref(n);
        auto c = kern(n);
        auto r = compare(o, c, b);
        if (!r.ok) { say("shape fuzz", false, "n=" + std::to_string(n) +
                          " rel=" + std::to_string(r.max_rel)); break; }
    }
    say("shape fuzz (0..4097)", all);

    // 2. EDGE VALUES — the inputs that break reductions and activations.
    {
        bool ok = true;
        for (const char* which : {"zeros", "all_equal", "large", "tiny",
                                  "mixed_sign", "single_outlier"}) {
            auto c = kern_on_pattern(kern, which);   // see note below
            for (float v : c) if (!std::isfinite(v)) { ok = false; break; }
        }
        say("edge-value patterns finite", ok);
    }

    // 3. DETERMINISM — run twice, compare BITWISE.
    {
        auto a = kern(1024), c = kern(1024);
        bool bitwise = std::equal(a.begin(), a.end(), c.begin(),
            [](float x, float y){ return std::memcmp(&x, &y, sizeof(float)) == 0; });
        say("bitwise determinism", bitwise,
            bitwise ? "" : "float atomics? tactic selection? -- decide deliberately");
    }

    // 4. PRECISION LADDER — fp32 vs fp16 vs int8, each against the fp64 oracle.
    //    Separates "quantization loss" from "kernel bug" BEFORE anyone argues.
    say("precision ladder", true, "(fp32/fp16/int8 each vs fp64 oracle)");

    printf("  %s: %s\n\n", name, all ? "OK" : "FAILURES");
    return all;
}
The four assertions that catch the most
  • Non-multiple-of-tile sizes. 1000, 1023, 4097. Every tiled kernel's first bug lives here.
  • Set equality, not tolerance, after any selection stage. Same top-K indices, same surviving boxes, same argmax class (D3).
  • Bitwise determinism. Two runs, memcmp. Cheap, and it makes an architectural property visible.
  • A property/invariant test. Softmax rows sum to 1; deformable attention with unit features and normalized weights returns 1 (P16). Invariants catch whole classes of indexing bug that a random-input comparison can miss.

36The benchmark harness

// bench.hpp — measure honestly. Reports a DISTRIBUTION, because for a
// real-time system the tail is the specification (Vol. I §05).
#pragma once
#include <vector>
#include <algorithm>
#include <cuda_runtime.h>

struct Stats { float p50, p90, p99, p999, mean, min, max; int n; };

inline Stats summarize(std::vector<float> ms) {
    std::sort(ms.begin(), ms.end());
    auto q = [&](double p) {
        if (ms.empty()) return 0.f;
        size_t i = (size_t)std::min<double>(ms.size() - 1,
                                            std::llround(p * (ms.size() - 1)));
        return ms[i];
    };
    double sum = 0; for (float v : ms) sum += v;
    return { q(0.50), q(0.90), q(0.99), q(0.999),
             (float)(sum / ms.size()), ms.front(), ms.back(), (int)ms.size() };
}

// Time a launch with CUDA EVENTS (not wall clock -- launches are async).
template <typename LaunchFn>
Stats bench(LaunchFn launch, int warmup = 50, int iters = 1000,
            cudaStream_t stream = 0) {
    cudaEvent_t beg, end;
    cudaEventCreate(&beg); cudaEventCreate(&end);

    for (int i = 0; i < warmup; ++i) launch(stream);   // clocks + caches + JIT
    cudaStreamSynchronize(stream);

    std::vector<float> ms;
    ms.reserve(iters);
    for (int i = 0; i < iters; ++i) {
        cudaEventRecord(beg, stream);
        launch(stream);
        cudaEventRecord(end, stream);
        cudaEventSynchronize(end);
        float t = 0.f; cudaEventElapsedTime(&t, beg, end);
        ms.push_back(t);
    }
    cudaEventDestroy(beg); cudaEventDestroy(end);
    return summarize(ms);
}

// Report achieved bandwidth / FLOPS against the roofline so the number
// means something. "0.42 ms" is not a result; "78% of copy bandwidth" is.
inline void report(const char* name, Stats s, double bytes, double flops) {
    const double sec = s.p50 * 1e-3;
    printf("%-28s p50 %7.3f ms  p99 %7.3f  p99.9 %7.3f  max %7.3f\n",
           name, s.p50, s.p99, s.p999, s.max);
    printf("%-28s %6.1f GB/s   %6.1f GFLOP/s\n", "",
           bytes / sec / 1e9, flops / sec / 1e9);
}
RuleWhy
CUDA events, not wall clockLaunches are asynchronous; a CPU timer around a launch measures enqueue time. Events are recorded in the stream.
Warm upClocks ramp, caches fill, and the first launch pays module-load cost. Discard 50 iterations minimum.
Report percentilesFor a vehicle, p99.9 is the number that matters. A mean hides exactly the behaviour that blows a deadline.
Consume the outputOtherwise the compiler may delete the computation. Checksum the result.
Realistic dataZeros and uniform random change branch behaviour, sparsity, and candidate counts. Benchmark on recorded frames.
Normalize against the roofline"78% of achievable bandwidth" tells you whether to keep optimizing. A raw millisecond does not.
Measure in situ tooA kernel alone gets the whole memory system; in the pipeline it competes with the rest of the SoC (Vol. I Tree 7).

The line to have ready: "I'd use CUB for this unless I needed to fuse it." It says you know the ecosystem, you don't reinvent tuned primitives, and you understand that the only durable reason to hand-write a reduction or a scan is to keep the data on chip and avoid a kernel boundary. That reason is real and common in inference — which is precisely why this team writes custom kernels at all.

38Rapid-fire bank and the plan

Answer each of these in under thirty seconds
Why is my kernel slow?
Roofline first, then Speed of Light, then stall reasons. Never guess.
What is a bank conflict?
32 banks × 4 B; bank = (addr/4) % 32; N threads hitting different addresses in one bank serialize N ways. Fix by padding or XOR swizzle; libraries swizzle to preserve alignment.
Coalescing, precisely?
A warp's 32 addresses are serviced in 32-byte sectors; the hardware fetches the minimum covering set. Ideal is 4 sectors for a 32×4 B access.
Occupancy 100% but slow?
Occupancy isn't the limiter — check which pipe is saturated, look for uncoalesced access or register spills.
When is low occupancy fine?
When ILP covers the latency — deep register tiling in a GEMM runs great at 25–50%.
Why fp32 accumulate for fp16?
~10-bit mantissa stagnates once the accumulator exceeds ~2048; error grows with reduction length.
How do tensor cores work?
Warp-cooperative matrix multiply-accumulate on register-resident fragments; the engineering problem is feeding them.
Why does batch-1 inference behave differently?
Launch-bound and memory-bound, not compute-bound. Fuse and capture a graph before optimizing any kernel.
__syncthreads() rules?
All threads in the block must reach it; never inside divergent flow; needed at both ends of a reused shared buffer.
Warp divergence cost?
Paths serialize with inactive lanes masked. Divergence across warps is free.
Volta changed what?
Independent thread scheduling — no implicit lockstep, so _sync intrinsics and __syncwarp are mandatory.
Deterministic kernel?
No float atomics, fixed reduction order, stable sorts, pinned library tactics.
How do you validate?
float64 oracle, error budget, shape fuzz, set-equality after selection stages, bitwise determinism, end-to-end metric.
Biggest inference win generally?
Fusion — it attacks bandwidth, which is the binding constraint on an automotive part.
