Part I

Foundations: the execution model

Before kernels, you need a mental model of what the GPU actually is. CUDA's abstractions — grids, blocks, threads, warps — map onto specific silicon: streaming multiprocessors, schedulers, register files. Knowing what corresponds to what is the difference between writing kernels that look reasonable and writing kernels that hit hardware peak.

§01The GPU execution model

A modern NVIDIA GPU is a collection of Streaming Multiprocessors (SMs) — independent processors that share global memory but otherwise execute independently. Each SM contains many CUDA cores (which execute floating-point and integer ops), one or more Tensor Cores (matrix-multiply units), warp schedulers, a register file, an L1 cache that doubles as shared memory, and texture units.

The SIMT abstraction

NVIDIA calls its execution model SIMT — Single Instruction, Multiple Threads. The hardware actually executes threads in groups of 32 called warps. All 32 threads in a warp execute the same instruction at the same time. When threads in a warp disagree about which branch to take, the hardware serializes the disagreeing paths — warp divergence, §16.

This is the central design tension: you write kernels as if every thread is independent, but performance comes from understanding when they're not.

CUDA's programming model gives you a billion-thread abstraction. The hardware gives you ~30 SMs each running ~64 warps at a time. Performance comes from knowing both views and writing kernels where the gap between them is small.

Host vs device

Every CUDA program has two sides:

  • Host — the CPU and its memory (DRAM). Runs C++ code; allocates GPU memory; launches kernels; copies data.
  • Device — the GPU and its memory (HBM). Runs kernels in parallel across thousands of threads.

Functions are tagged with where they run:

__host__   void cpu_only();              // runs on CPU (default)
__device__ int  gpu_helper(int x);         // callable only from device code
__global__ void kernel(int* data);          // the kernel: launched from host, runs on device
__host__ __device__ int both(int x);        // compiled for both; usable from either side

§02Anatomy of a kernel

The Hello-World of CUDA — not because it's interesting, but because every concept in the language shows up here. Read it carefully:

#include <cuda_runtime.h>
#include <cstdio>

// The kernel: runs on device, one instance per thread
__global__ void vector_add(const float* a, const float* b, float* c, int n) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;   // global thread id
  if (i < n) {                                  // boundary guard
    c[i] = a[i] + b[i];
  }
}

int main() {
  const int N = 1 << 20;                       // 1 M elements
  const size_t bytes = N * sizeof(float);

  // 1. Allocate host memory and fill
  float *h_a = new float[N], *h_b = new float[N], *h_c = new float[N];
  for (int i = 0; i < N; ++i) { h_a[i] = i; h_b[i] = 2*i; }

  // 2. Allocate device memory
  float *d_a, *d_b, *d_c;
  cudaMalloc(&d_a, bytes);
  cudaMalloc(&d_b, bytes);
  cudaMalloc(&d_c, bytes);

  // 3. Copy inputs host -> device
  cudaMemcpy(d_a, h_a, bytes, cudaMemcpyHostToDevice);
  cudaMemcpy(d_b, h_b, bytes, cudaMemcpyHostToDevice);

  // 4. Configure launch — one thread per element
  int threadsPerBlock = 256;
  int blocksPerGrid = (N + threadsPerBlock - 1) / threadsPerBlock;

  // 5. Launch the kernel with <<<grid, block>>> syntax
  vector_add<<<blocksPerGrid, threadsPerBlock>>>(d_a, d_b, d_c, N);

  // 6. Check launch errors; synchronize and check runtime errors
  cudaGetLastError();         // catches invalid configurations
  cudaDeviceSynchronize();    // waits for kernel; surfaces any runtime errors

  // 7. Copy result device -> host
  cudaMemcpy(h_c, d_c, bytes, cudaMemcpyDeviceToHost);

  // 8. Free device memory and host memory
  cudaFree(d_a); cudaFree(d_b); cudaFree(d_c);
  delete[] h_a; delete[] h_b; delete[] h_c;
  return 0;
}

The eight things every kernel launch does

  1. Allocate host memory — just standard C++ memory.
  2. Allocate device memory with cudaMalloc.
  3. Transfer inputs from host to device with cudaMemcpy.
  4. Configure the launch — how many blocks, how many threads per block.
  5. Launch with kernel<<<grid, block>>>(args...).
  6. Check for errors — CUDA's API is async, so error checking has two stages.
  7. Transfer results back to host.
  8. Free device memory.

The launch syntax decoded

kernel<<<gridDim, blockDim, sharedMemBytes, stream>>>(args...);
//      |        |          |              |
//      |        |          |              +-- which stream (default: 0)
//      |        |          +-- dynamic shared memory size in bytes (default: 0)
//      |        +-- threads per block (1D, 2D, or 3D dim3)
//      +-- blocks per grid (1D, 2D, or 3D dim3)

Error checking that doesn't hide bugs

CUDA kernel launches are asynchronous — the launch call returns immediately, and any runtime error appears later. Two checkpoints are needed:

#define CUDA_CHECK(call) do {                            \
  cudaError_t err = (call);                              \
  if (err != cudaSuccess) {                              \
    fprintf(stderr, "CUDA error %s:%d: %s\n",         \
            __FILE__, __LINE__, cudaGetErrorString(err)); \
    exit(1);                                            \
  }                                                      \
} while (0)

// Usage:
CUDA_CHECK(cudaMalloc(&d_a, bytes));

vector_add<<<grid, block>>>(d_a, d_b, d_c, N);
CUDA_CHECK(cudaGetLastError());      // catches launch-config errors
CUDA_CHECK(cudaDeviceSynchronize()); // catches any error that happened during the kernel
The async pitfall

If a kernel reads out-of-bounds memory and you don't synchronize, the error is "latent" — the next unrelated cudaMemcpy or cudaMalloc returns the error from the earlier kernel, with no useful context. Always check both cudaGetLastError (catches "I launched a bad config") and cudaDeviceSynchronize (catches "the kernel itself crashed") during development.

§03Grids, blocks, threads — the hierarchy

The CUDA execution hierarchy has three levels:

GRID +-------------------------------------------------------------+ | block(0,0) block(1,0) block(2,0) block(3,0) | | +-------+ +-------+ +-------+ +-------+ | | |t00 t01| |t00 t01| |t00 t01| |t00 t01| | | |t02 t03| |t02 t03| |t02 t03| |t02 t03| | | +-------+ +-------+ +-------+ +-------+ | | | | block(0,1) block(1,1) block(2,1) block(3,1) | | +-------+ +-------+ +-------+ +-------+ | | |t00 t01| |t00 t01| |t00 t01| |t00 t01| | | |t02 t03| |t02 t03| |t02 t03| |t02 t03| | | +-------+ +-------+ +-------+ +-------+ | +-------------------------------------------------------------+ Threads in same block: share shared memory, can __syncthreads() Threads across blocks: independent; only see each other through global memory + atomics

The rules

  • A grid is launched per kernel call. It contains all the blocks.
  • A block is a set of threads that runs on one SM and shares its resources. Block size is fixed at launch.
  • A thread is the smallest unit of execution. Each thread has its own registers and program counter.
  • Threads in the same block can cooperate via shared memory and __syncthreads.
  • Threads in different blocks cannot easily cooperate. They run independently and may not even be co-resident.

The built-in variables

Inside a kernel, four dim3 variables tell you where you are:

__global__ void kernel() {
  // Inside the kernel, these are predefined:
  threadIdx.x, threadIdx.y, threadIdx.z   // my position within my block
  blockIdx.x,  blockIdx.y,  blockIdx.z    // my block's position in the grid
  blockDim.x,  blockDim.y,  blockDim.z    // the dimensions of every block
  gridDim.x,   gridDim.y,   gridDim.z     // the dimensions of the grid
}

The classic 1D global index, used everywhere:

int i = blockIdx.x * blockDim.x + threadIdx.x;
// for 1D launch: this is the unique thread number across the whole grid

Picking block size

Block size is one of the most consequential decisions in a kernel. The good news: most kernels are insensitive to it within a window. The right answers in practice:

Block sizeWhen to pick it
128Many register-heavy kernels (lets more blocks fit per SM)
256Default for almost everything; good balance
512Memory-bound kernels with simple bodies
1024Maximum allowed; rarely better than 256

Always a multiple of 32 (warp size). Non-multiples waste threads in the last warp of each block.

Grid size

// Pattern: enough blocks to cover N elements with `threads` threads per block
int threads = 256;
int blocks  = (N + threads - 1) / threads;   // ceiling division

// Alternative: launch a fixed number of blocks and let each thread handle
// multiple elements with a grid-stride loop (see §14, Thread coarsening)
__global__ void stride_kernel(float* data, int N) {
  int tid = blockIdx.x * blockDim.x + threadIdx.x;
  int stride = gridDim.x * blockDim.x;
  for (int i = tid; i < N; i += stride) {
    data[i] = sqrtf(data[i]);
  }
}

§04Multi-dimensional indexing & row-major arrays

2D and 3D problems — images, matrices, volumes — map naturally to 2D and 3D thread grids. The indexing is straightforward but every CUDA programmer gets it wrong at least once.

2D launch and indexing

__global__ void matrix_scale(float* A, float s, int rows, int cols) {
  int col = blockIdx.x * blockDim.x + threadIdx.x;   // x dim = horizontal
  int row = blockIdx.y * blockDim.y + threadIdx.y;   // y dim = vertical

  if (row < rows && col < cols) {
    A[row * cols + col] *= s;                       // row-major: index = row*width + col
  }
}

int main() {
  const int R = 2048, C = 3000;
  ...

  dim3 block(16, 16);                          // 256 threads per block, in a 16x16 arrangement
  dim3 grid((C + block.x - 1) / block.x,        // enough blocks to cover columns
            (R + block.y - 1) / block.y);        // enough blocks to cover rows

  matrix_scale<<<grid, block>>>(d_A, 2.0f, R, C);
}
The axis-confusion classic

In a 2D launch, blockDim.x corresponds to columns and blockDim.y to rows. The mismatch with row/col mental model causes the single most common 2D-CUDA bug: swapping x and y in the kernel. Run it with a small input first and verify the output isn't transposed.

3D launches

__global__ void volume_kernel(float* V, int nx, int ny, int nz) {
  int x = blockIdx.x * blockDim.x + threadIdx.x;
  int y = blockIdx.y * blockDim.y + threadIdx.y;
  int z = blockIdx.z * blockDim.z + threadIdx.z;

  if (x < nx && y < ny && z < nz) {
    int idx = z * (nx * ny) + y * nx + x;          // linearize 3D index
    V[idx] = f(V[idx]);
  }
}

dim3 block(8, 8, 8);                              // 512 threads
dim3 grid((nx + 7) / 8, (ny + 7) / 8, (nz + 7) / 8);

Row-major vs column-major

C and C++ are row-major: A[i][j] is stored as A[i * cols + j]. Fortran and (notoriously) BLAS conventions are column-major: A[i][j] = A[i + j * rows]. Mismatch between your storage convention and the library you call is a frequent source of "results are transposed" bugs.

// Row-major (C/C++, PyTorch, NumPy default):
A[row * cols + col]

// Column-major (BLAS, cuBLAS by default, Fortran, Eigen default):
A[col * rows + row]

// Strided general case (works for any layout):
A[row * row_stride + col * col_stride]

Pitch and 2D allocations

Row-major 2D arrays sometimes need their rows aligned to the hardware's natural 128-byte boundary for coalescing (§10). cudaMallocPitch pads each row up to a "pitch":

float* d_A;
size_t pitch;                                 // bytes per row, may exceed cols*sizeof(float)
cudaMallocPitch(&d_A, &pitch, cols * sizeof(float), rows);

// Indexing with pitch:
__global__ void k(float* A, size_t pitch, int rows, int cols) {
  int row = blockIdx.y * blockDim.y + threadIdx.y;
  int col = blockIdx.x * blockDim.x + threadIdx.x;
  if (row < rows && col < cols) {
    float* row_ptr = (float*)((char*)A + row * pitch);   // note: pitch is in BYTES
    row_ptr[col] = do_work(row_ptr[col]);
  }
}

Modern CUDA has reduced the need for pitched allocations — the hardware tolerates unaligned starts better than it used to. Use plain cudaMalloc by default; reach for cudaMallocPitch only when profiling shows you'd benefit.

§05Boundary checks & ghost regions

You launch more threads than data, and the extra threads need to do nothing. The boundary guard is the most-written line of CUDA code:

int i = blockIdx.x * blockDim.x + threadIdx.x;
if (i < N) {
  c[i] = a[i] + b[i];
}

Why this pattern? Block size 256 doesn't divide N exactly. The last block of N=1000 launches threads 768..1023 but only threads 768..999 should do work. The guard prevents 1000-1023 from reading or writing out-of-bounds memory.

Ghost / halo regions in stencil computations

Many algorithms need each thread to read its neighbors. At the edges of the array, those neighbors don't exist — you need to either clip or pad. The pattern:

// 1D stencil: out[i] = (in[i-1] + in[i] + in[i+1]) / 3
__global__ void stencil_1d(const float* in, float* out, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;

  // Option 1: skip the boundary entirely
  if (i > 0 && i < N - 1) {
    out[i] = (in[i - 1] + in[i] + in[i + 1]) / 3.0f;
  }

  // Option 2: clamp the indices (repeats edge value)
  if (i < N) {
    int im = max(0, i - 1);
    int ip = min(N - 1, i + 1);
    out[i] = (in[im] + in[i] + in[ip]) / 3.0f;
  }

  // Option 3: use zero outside the array (boundary contribution suppressed)
  if (i < N) {
    float left  = (i > 0)     ? in[i - 1] : 0.0f;
    float right = (i < N - 1) ? in[i + 1] : 0.0f;
    out[i] = (left + in[i] + right) / 3.0f;
  }
}

2D stencils — ghost cells in shared memory

When you tile a stencil with shared memory (§11), each tile needs a halo of the neighboring tile's data. Loading the halo is part of the tile-loading code:

__global__ void stencil_2d_tiled(const float* in, float* out, int W, int H) {
  constexpr int TILE = 16;
  constexpr int HALO = 1;
  __shared__ float tile[TILE + 2*HALO][TILE + 2*HALO];

  int gx = blockIdx.x * TILE + threadIdx.x;       // global x
  int gy = blockIdx.y * TILE + threadIdx.y;       // global y
  int lx = threadIdx.x + HALO;                     // local x in tile
  int ly = threadIdx.y + HALO;                     // local y in tile

  // Load center
  tile[ly][lx] = (gx < W && gy < H) ? in[gy * W + gx] : 0.0f;

  // Load halo (threads on edge of block load extra ghost cells)
  if (threadIdx.x < HALO) {                       // left edge
    int gxL = gx - HALO;
    tile[ly][lx - HALO] = (gxL >= 0 && gy < H) ? in[gy * W + gxL] : 0.0f;
  }
  // ... similar for right, top, bottom edges, and four corners ...

  __syncthreads();

  // Now compute using tile[][]
  if (gx < W && gy < H) {
    out[gy * W + gx] = (tile[ly][lx] + tile[ly][lx - 1] + tile[ly][lx + 1]
                       + tile[ly - 1][lx] + tile[ly + 1][lx]) * 0.2f;
  }
}

Boundary checks aren't bureaucracy; they're correctness. A kernel without them either crashes or, worse, corrupts data that survives undetected.

Part II

Memory: allocation & transfer

Before the memory hierarchy on the GPU side, you need to understand the boundary between host and device. CPU memory and GPU memory are physically separate (mostly), and how you bridge that gap determines whether a kernel is memory-starved or fed.

§06The host-device memory model

The CPU and GPU have separate physical memory connected by PCIe (or NVLink on some systems). Anything the GPU computes must first travel from host RAM to GPU HBM; results must travel back. These transfers are often slower than the computation itself.

cudaMalloc, cudaMemcpy, cudaFree

cudaMemcpy(d_data, h_data, N * sizeof(float),       // blocking copy:
           cudaMemcpyHostToDevice);                       //   returns after data transferred

           cudaMemcpyDeviceToHost);

The directions

// cudaMemcpyKind values:

cudaMemcpyDeviceToDevice   // stays on GPU; very fast
cudaMemcpyDefault          // CUDA figures it out from pointer types (UVA required)

Allocation cost

cudaMalloc is expensive — often hundreds of microseconds, with significant variability. For repeated allocations the right answer is a pool allocator. PyTorch and RAPIDS both ship one; for hand-written code, look at cuMemAlloc* or cuda::mr::cuda_async_memory_resource in libcudacxx.

// Modern (CUDA 11.2+): stream-ordered allocator, much cheaper
float* d_data;
cudaMallocAsync(&d_data, bytes, stream);
// ... use d_data on `stream` ...
cudaFreeAsync(d_data, stream);
// Memory is reusable by other allocations on the same stream without
// expensive cudaFree-cudaMalloc round trips.

Unified Virtual Addressing (UVA)

Since Kepler, all host and device allocations share a single 64-bit address space. A pointer's value uniquely identifies whether it lives on the host or the device. This is what lets cudaMemcpyDefault work, and what enables Unified Memory (next section).

The classic bug

Passing a host pointer to a kernel doesn't fail at launch — the kernel just dereferences a host address, causing a memory fault. The error appears later, often masked by another API call. Always pass device pointers to kernels.

float h_arr[100];
kernel<<<1, 100>>>(h_arr);    // COMPILES, RUNS, CRASHES — no compile-time check

§07Unified Memory & managed allocation

cudaMallocManaged allocates memory that's logically in both host and device address spaces. The runtime migrates pages between physical locations on demand — you don't write explicit copies.

float* data;
cudaMallocManaged(&data, N * sizeof(float));

// Use directly from CPU code
for (int i = 0; i < N; ++i) data[i] = i;

// Use directly from a kernel — pages migrate transparently
my_kernel<<<grid, block>>>(data, N);
cudaDeviceSynchronize();

// Use from CPU again
printf("%f\n", data[0]);

cudaFree(data);

The trade-offs

  • Simplicity — no explicit copies, no separate pointers.
  • Demand paging — first access triggers a fault, then a migration. First touch can be expensive.
  • Implicit overhead — you don't see the transfers, so you don't optimize them.
  • Prefetching helps — tell the runtime where data should live before you need it.
// Pre-migrate to GPU before the kernel
cudaMemPrefetchAsync(data, N * sizeof(float),
                     deviceId, stream);

my_kernel<<<grid, block, 0, stream>>>(data, N);

// Pre-migrate to CPU before reading on the host
cudaMemPrefetchAsync(data, N * sizeof(float),
                     cudaCpuDeviceId, stream);
cudaStreamSynchronize(stream);

When to use Unified Memory

Three sweet spots:

  1. Prototyping — remove a class of bugs while you're figuring out your algorithm.
  2. Data structures that don't fit on the GPU — oversubscription works (with paging overhead).
  3. Coherent memory systems (Grace Hopper, MI300A, others) — on these architectures Unified Memory is the fastest option, not a convenience tradeoff.

When to avoid: throughput-critical production kernels on traditional PCIe systems — explicit transfers with overlap give better predictability.

§08Pinned memory & async transfers

Standard host memory is pageable — the OS can swap it. The GPU's DMA engine can't directly read pageable memory; the runtime has to copy it through an intermediate pinned buffer first. cudaMallocHost (or cudaHostAlloc) allocates already-pinned memory that DMA can transfer directly. The benefit: faster transfers and the ability to overlap them with kernels.

// Pinned host allocation
float* h_data;
cudaMallocHost(&h_data, N * sizeof(float));    // or cudaHostAlloc with flags

// Async copy (returns immediately, runs on stream)
cudaMemcpyAsync(d_data, h_data, N * sizeof(float),
                cudaMemcpyHostToDevice, stream);

my_kernel<<<grid, block, 0, stream>>>(d_data, N);

cudaMemcpyAsync(h_result, d_data, N * sizeof(float),
                cudaMemcpyDeviceToHost, stream);

cudaStreamSynchronize(stream);
cudaFreeHost(h_data);

Performance comparison — the typical numbers

Overlapping transfer with compute

The classic pipeline: while the GPU computes on batch n, transfer batch n+1. Done correctly, transfer time is hidden behind compute:

const int NSTREAMS = 2;
cudaStream_t streams[NSTREAMS];
for (int s = 0; s < NSTREAMS; ++s) cudaStreamCreate(&streams[s]);

size_t chunk = N / NSTREAMS;
for (int i = 0; i < NSTREAMS; ++i) {
  size_t offset = i * chunk;
  cudaStream_t s = streams[i];
  cudaMemcpyAsync(d_data + offset, h_data + offset,
                  chunk * sizeof(float),
                  cudaMemcpyHostToDevice, s);
  my_kernel<<<grid, block, 0, s>>>(d_data + offset, chunk);
  cudaMemcpyAsync(h_result + offset, d_data + offset,
                  chunk * sizeof(float),
                  cudaMemcpyDeviceToHost, s);
}
for (int s = 0; s < NSTREAMS; ++s) cudaStreamDestroy(streams[s]);
Pinning is not free

Pinned memory is locked in physical RAM — it cannot be swapped. Pinning gigabytes can starve the OS and other processes. Use pinned memory for transfer staging buffers, not for your entire dataset. A common pattern: small pinned buffer + cudaMemcpyAsync chunks through it.

Part III

The memory hierarchy on the device

Inside the GPU, memory has a strict tiered structure with capacity-vs-speed tradeoffs that span four orders of magnitude. The performance of every kernel is determined by which tier its working set lives in.

§09The memory hierarchy

Speed Capacity | | fastest ┌──────────────────────────┐ smallest │ │ Registers (per-thread) │ ~256 KB/SM │ │ ~0 cycles, dedicated │ ~64 KB per thread max │ └──────────────────────────┘ │ ┌──────────────────────────┐ │ │ Shared memory (per-blk) │ 100-228 KB/SM │ │ ~1-30 cycles │ ~48 KB per block default │ └──────────────────────────┘ │ ┌──────────────────────────┐ │ │ L1 cache (per-SM) │ shares physical RAM │ │ ~30 cycles │ with shared memory │ └──────────────────────────┘ │ ┌──────────────────────────┐ │ │ L2 cache (per-GPU) │ 40-60 MB │ │ ~200 cycles │ │ └──────────────────────────┘ │ ┌──────────────────────────┐ │ │ Global memory (HBM) │ 40-80 GB │ │ ~400-800 cycles │ ~3 TB/s bandwidth │ └──────────────────────────┘ │ ┌──────────────────────────┐ │ │ Constant memory │ 64 KB │ │ cached: ~1-30 cycles │ read-only from device │ └──────────────────────────┘ │ ┌──────────────────────────┐ │ │ Texture memory │ cached, addressed via │ │ │ texture units │ └──────────────────────────┘ slowest largest

The actual ratios that matter

Latency:

  • Register access — effectively free, baked into the instruction
  • Shared memory — ~30x slower than register
  • L2 cache — ~7x slower than shared
  • HBM — ~2-4x slower than L2

Bandwidth ratios (per-SM):

  • Shared memory: many TB/s effective per SM
  • HBM: ~25 GB/s per SM (3 TB/s / 132 SMs on H100)

This is the central economic argument for shared memory: the bandwidth you get per SM is roughly 100x higher than HBM bandwidth per SM. Tiling rewrites a kernel so its inner loop hits shared memory, not HBM.

The memory spaces, in code

__global__ void k(float* g_in, float* g_out) {
  // 1. REGISTERS — default storage for locals
  float x = 3.14f;                    // in a register
  int   i = threadIdx.x;             // in a register

  // 2. SHARED memory — explicitly declared
  __shared__ float tile[256];          // shared by all threads in the block

  // 3. GLOBAL memory — pointers passed in
  float v = g_in[i];                  // load from HBM
  g_out[i] = v;                       // store to HBM

  // 4. LOCAL memory — spills, large arrays declared per-thread
  float big_array[1024];             // likely spills to "local" (actually L1/L2/HBM)

  // 5. CONSTANT memory — declared at file scope
}

__constant__ float coeffs[64];           // at file scope; broadcast-fast in kernels

Registers — the inner sanctum

Each thread has its own private set of registers. They're the fastest memory by far. A kernel's register pressure — how many registers per thread it needs — directly affects how many threads can run concurrently. Too many registers per thread, and fewer warps fit per SM, hurting occupancy (§18).

# Check register usage of a compiled kernel
nvcc -Xptxas=-v -arch=sm_80 -c kernel.cu
# Compiles output like:
# ptxas info    : Used 32 registers, 384 bytes cmem[0]
# Used 32 regs/thread — about right for many kernels
# 64 regs/thread is the sweet spot for many small kernels
# 128+ regs/thread starts to hurt occupancy unless deliberate

If a kernel needs more state than fits in registers, the compiler spills to "local memory" — actually L1/L2/HBM. Spills are a major performance cliff. Check spills with the same -Xptxas=-v flag.

Local memory — not what its name suggests

Despite the name, "local memory" is not on the SM. It's in HBM (cached by L1/L2). Per-thread arrays larger than the register file, or arrays whose indices aren't known at compile time, end up here.

__global__ void spilly() {
  float arr[16];               // May be in regs (if all indices constant)
                              // or in local memory (if indices computed)
  int i = threadIdx.x % 16;
  arr[i] = 3.14f;             // Forces local memory (dynamic index)
}

Constant memory

64 KB of cached read-only memory. Hardware-optimized for the case where every thread in a warp reads the same address (broadcast). When threads disagree, it serializes — up to 32 accesses for a divergent warp.

__constant__ float filter[25];

// From host:
float h_filter[25] = { 1.0f/25, 1.0f/25, ... };
cudaMemcpyToSymbol(filter, h_filter, sizeof(filter));

// In the kernel:
__global__ void conv5x5(const float* in, float* out, int W, int H) {
  float sum = 0;
  for (int ky = 0; ky < 5; ++ky)
    for (int kx = 0; kx < 5; ++kx)
      sum += in[...] * filter[ky*5 + kx];     // all threads read same address
                                              // per iter — perfect broadcast
}

The decision: small read-only data (filters, coefficients, lookup tables) where threads read uniformly — constant memory. Otherwise prefer global memory with the read-only data cache (__ldg).

L1 / L2 caches

L1 is per-SM and shares physical memory with shared memory — you can rebalance the split per kernel. L2 is per-GPU, shared by all SMs. Both are transparent — you don't manage them, but you do influence them:

// __ldg() is a hint: load through the read-only data cache
float v = __ldg(&input[i]);

// Cache hint annotations (compile-time)
const float* __restrict__ in;     // __restrict__ tells compiler pointers don't alias;
                                // enables read-only cache usage automatically

// L1 cache configuration per kernel (a hint, not a guarantee)
cudaFuncSetCacheConfig(my_kernel, cudaFuncCachePreferShared);
//                                     vs cudaFuncCachePreferL1, ...PreferEqual, ...PreferNone

§10Memory coalescing — the most important optimization

The single highest-leverage rule in CUDA: threads in a warp should access consecutive memory locations. When they do, the hardware combines the 32 individual loads into one or two 128-byte transactions. When they don't, you can pay 32x the bandwidth cost.

COALESCED — 32 threads load 32 consecutive floats (128 bytes): warp threads: t0 t1 t2 t3 ... t30 t31 │ │ │ │ │ │ ▼ ▼ ▼ ▼ ▼ ▼ memory: [a0 a1 a2 a3 ... a30 a31] one transaction UNCOALESCED — threads stride through memory: warp threads: t0 t1 t2 ... │ │ │ ▼ ▼ ▼ memory: [a0 ... a16 ... a32 ...] |---128B---|---128B---|... up to 32 transactions

The example that teaches it

Same operation, two access patterns. The first is fast; the second is slow:

Coalesced — row-major, x = column
__global__ void good(float* A,
                     int rows, int cols) {
  int r = blockIdx.y * blockDim.y
                            + threadIdx.y;
  int c = blockIdx.x * blockDim.x
                            + threadIdx.x;
  if (r < rows && c < cols) {
    A[r * cols + c] *= 2;
  }
}
// Adjacent threadIdx.x =>
// adjacent c => adjacent
// memory addresses. Coalesced.
Uncoalesced — x = row
__global__ void bad(float* A,
                    int rows, int cols) {
  int r = blockIdx.x * blockDim.x
                            + threadIdx.x;
  int c = blockIdx.y * blockDim.y
                            + threadIdx.y;
  if (r < rows && c < cols) {
    A[r * cols + c] *= 2;
  }
}
// Adjacent threadIdx.x =>
// adjacent r => addresses
// are `cols` floats apart.
// Uncoalesced.

The transpose kernel — a famous performance puzzle

Naive transpose: one of the reads or writes is uncoalesced no matter what you do, because rows and columns map to different memory strides:

__global__ void transpose_naive(float* out, const float* in, int N) {
  int x = blockIdx.x * blockDim.x + threadIdx.x;
  int y = blockIdx.y * blockDim.y + threadIdx.y;
  if (x < N && y < N) {
    out[x * N + y] = in[y * N + x];
    // in[y*N + x]: coalesced (adjacent x = adjacent address)
    // out[x*N + y]: uncoalesced (adjacent x = stride-N address)
  }
}

Fix: tile into shared memory. Read coalesced (consecutive x), transpose in shared memory, then write coalesced. The shared-memory access pattern crosses the diagonal, which raises the bank-conflict problem — covered in §12.

Structures of Arrays vs Arrays of Structures

The classic data-layout decision — with a clear answer for GPUs.

AoS — uncoalesced for typical access
struct Particle {
  float x, y, z;
  float vx, vy, vz;
};
Particle p[N];

// Read all x's:
for (int i = ...) p[i].x;
// Adjacent i => address +24 bytes
// Strided => uncoalesced
SoA — coalesced naturally
struct Particles {
  float* x; float* y; float* z;
  float* vx; float* vy; float* vz;
};
Particles p;

// Read all x's:
for (int i = ...) p.x[i];
// Adjacent i => adjacent address.
// Coalesced.

For most GPU workloads, SoA is the right call. The exception: when threads need multiple fields of the same particle, AoS can be better — one transaction loads adjacent fields.

Vectorized loads

The hardware can issue 128-bit (16-byte) loads in one instruction. float4, int4, etc., trigger this:

__global__ void vec_copy(const float4* in, float4* out, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < N) {
    float4 v = in[i];               // one 16-byte load
    v.x *= 2; v.y *= 2; v.z *= 2; v.w *= 2;
    out[i] = v;                     // one 16-byte store
  }
}

// Caller passes float* cast to float4* — pointers must be 16-byte aligned
vec_copy<<<grid, block>>>((float4*)d_in, (float4*)d_out, N / 4);

Vectorized access reduces the number of load instructions by 4x, which often translates to better latency hiding and ~2x throughput on memory-bound kernels.

The bandwidth a kernel achieves vs HBM peak (3 TB/s on H100) is the headline number for memory-bound work. If you're at 5% of peak, the access pattern is wrong; coalescing is the first thing to check.

§11Shared memory & tiling

Shared memory is a small, fast, programmer-managed cache shared by threads in a block. The standard pattern: each block loads a tile of data from global memory into shared memory once, then all threads in the block reuse it many times.

Declaration & basic usage

__global__ void reverse_block(float* data) {
  __shared__ float tile[256];     // static shared memory: size known at compile time

  int tid = threadIdx.x;
  int i   = blockIdx.x * blockDim.x + tid;

  tile[tid] = data[i];               // load to shared (coalesced)
  __syncthreads();                  // wait for all threads to finish loading

  data[i] = tile[255 - tid];        // reverse within block, write back (coalesced)
}

Dynamic shared memory

When the size isn't known at compile time, declare as extern and pass the byte count at launch:

__global__ void dyn() {
  extern __shared__ float tile[];      // size unknown here
  int tid = threadIdx.x;
  tile[tid] = 0.0f;
  ...
}

// Launch with shared memory size in the third <<<>>> slot
int shared_bytes = 256 * sizeof(float);
dyn<<<grid, 256, shared_bytes>>>();

You can have multiple dynamic shared arrays by stitching one allocation into views:

__global void multi(int n_floats, int n_ints) {
  extern __shared__ unsigned char buf[];
  float* a = reinterpret_cast<float*>(buf);
  int*   b = reinterpret_cast<int*>(buf + n_floats * sizeof(float));
  ...
}

The 1D smoothing filter — complete tiled example

template<int RADIUS>
__global__ void smooth_1d(const float* in, float* out, int N) {
  constexpr int BS = 256;
  __shared__ float tile[BS + 2*RADIUS];

  int tid = threadIdx.x;
  int gid = blockIdx.x * BS + tid;
  int ltid = tid + RADIUS;             // position in tile (offset for halo)

  // Load center: each thread loads one element
  tile[ltid] = (gid < N) ? in[gid] : 0.0f;

  // Load halo: first RADIUS threads load left side, last RADIUS threads load right
  if (tid < RADIUS) {
    int g_left  = gid - RADIUS;
    tile[tid] = (g_left >= 0) ? in[g_left] : 0.0f;
    int g_right = gid + BS;
    tile[ltid + BS] = (g_right < N) ? in[g_right] : 0.0f;
  }

  __syncthreads();

  // Compute — all reads now from fast shared memory
  if (gid < N) {
    float sum = 0;
    #pragma unroll
    for (int k = -RADIUS; k <= RADIUS; ++k) {
      sum += tile[ltid + k];
    }
    out[gid] = sum / (2*RADIUS + 1);
  }
}

Why tiling wins

Without tiling: each output reads 2*RADIUS + 1 global memory locations — many of them overlapping with neighbors. BS threads do BS * (2*RADIUS+1) global reads.

With tiling: each block does BS + 2*RADIUS global reads (one per element + halo) and many shared-memory reads. The reuse factor is (2*RADIUS+1)x, which translates directly to bandwidth savings.

The dynamic shared-memory cap

Default is 48 KB per block. For more (up to ~228 KB on H100), opt in:

cudaFuncSetAttribute(my_kernel,
    cudaFuncAttributeMaxDynamicSharedMemorySize,
    100 * 1024);   // 100 KB

Larger shared memory reduces occupancy (fewer blocks fit per SM), so verify it's worth the trade.

§12Bank conflicts

Shared memory is divided into 32 banks. Each bank can service one access per cycle. The 32 threads of a warp can issue 32 shared-memory accesses in parallel only if they hit 32 different banks — or all hit the same address (broadcast). When two threads access different addresses in the same bank, the hardware serializes them: a bank conflict.

Shared memory layout (32-bit words): Address : 0 1 2 3 ... 31 32 33 34 ... Bank : 0 1 2 3 ... 31 0 1 2 ... Word at address A is in bank (A % 32). No conflict: t0->A0 t1->A1 ... t31->A31 (every bank hit once) Broadcast: t0->A5 t1->A5 ... t31->A5 (all same address: special) Conflict: t0->A0 t1->A32 t2->A64 ... (all bank 0: 32-way conflict)

The classic conflict: column access on a 32-wide tile

__shared__ float tile[32][32];        // 32 rows of 32 floats

// Row access (no conflict): each thread reads a different column
float v = tile[threadIdx.y][threadIdx.x];
// threadIdx.x = 0, 1, ..., 31  =>  columns 0, 1, ..., 31  =>  banks 0..31 OK

// Column access (32-way conflict): all threads read same column, different rows
float w = tile[threadIdx.x][threadIdx.y];
// All have threadIdx.y = ...y..., row index changes
// Stride between threads = 32 floats = 128 bytes = same bank!
// Result: warp accesses serialized 32 times. 32x slowdown.

The padding fix

Add one extra column (padding) so that the stride between rows is 33, not 32. Bank index becomes (row * 33 + col) % 32, which now varies across rows:

__shared__ float tile[32][33];        // padded! 33 instead of 32

float w = tile[threadIdx.x][threadIdx.y];
// Bank = (threadIdx.x * 33 + threadIdx.y) % 32 = threadIdx.x mod 32 + ...
// Across the warp, bank indices are all different. No conflict.

You pay 33/32 = 3% extra shared memory; you gain 32x bandwidth. Universal trick.

Verifying with the profiler

ncu --metrics l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld.sum,\
              l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_st.sum \
    ./my_program

# Output:
# l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld.sum    1,234,567
# l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_st.sum            0
#
# Any non-zero value means you have conflicts. Investigate.

The broadcast escape hatch

When all threads in a warp access the same address, the hardware broadcasts — one transaction, no conflict. This is why constant memory and uniform-address shared accesses are fast.

// All threads read tile[0]: broadcast, one cycle
float v = tile[0];

// Threads 0-15 read tile[0], threads 16-31 read tile[1]: 2 broadcasts, 2 cycles
float w = tile[threadIdx.x / 16];

// Each thread reads different bank, no conflict
float x = tile[threadIdx.x];
Part IV

Matmul & thread coarsening

Two topics chosen for their pedagogical power: tiled matrix multiplication is the textbook example that exercises every concept so far; thread coarsening is the optimization that often beats "more threads" intuition.

§13Tiled matrix multiplication

Matrix multiply (GEMM) is the workhorse of deep learning and dense linear algebra. The naive implementation is bandwidth-bound; the tiled version is the compute-bound version that hits 60-80% of peak FLOPS on modest sizes (and Tensor Cores in §25 take you the rest of the way).

Naive: each thread computes one output element

__global__ void gemm_naive(const float* A, const float* B, float* C,
                           int M, int N, int K) {
  int row = blockIdx.y * blockDim.y + threadIdx.y;
  int col = blockIdx.x * blockDim.x + threadIdx.x;

  if (row < M && col < N) {
    float sum = 0.0f;
    for (int k = 0; k < K; ++k) {
      sum += A[row * K + k] * B[k * N + col];
    }
    C[row * N + col] = sum;
  }
}

This works, and it's slow. Why? Each output element loads 2K values from global memory and does 2K FLOPs — arithmetic intensity of 1 FLOP per memory byte. For float32 (4 bytes), that's 1 FLOP per byte loaded. HBM bandwidth caps at ~3 TB/s = 3 TFLOP/s for this pattern. The hardware peak is 30+ TFLOPS — so naive uses ~10% of peak.

The tiled version

Each thread block computes a TILE x TILE output sub-matrix. Within the block, threads cooperate to load TILE x TILE sub-matrices of A and B into shared memory, then reuse them across TILE multiply-accumulates per output element.

A (M x K) B (K x N) C (M x N) ┌─────┬─────┐ ┌────┬────┐ ... ┌────┬────┐ │ A0 │ A1 │ │ B0 │ │ │ C │ │ │ │ │ ├────┤ │ │ │ │ ├─────┼─────┤ │ B1 │ │ ├────┤ │ │ │ │ ├────┤ │ │ │ │ │ │ │ │ ...│ │ │ │ │ └─────┴─────┘ └────┴────┘ └────┴────┘ One block computes C-tile = A0*B0 + A1*B1 + ... Loop over k-tiles: 1. Cooperatively load A-tile to shared 2. Cooperatively load B-tile to shared 3. __syncthreads() 4. Each thread computes its part of the partial sum 5. __syncthreads() 6. Next k-tile
template<int TILE>
__global__ void gemm_tiled(const float* A, const float* B, float* C,
                           int M, int N, int K) {
  __shared__ float Asub[TILE][TILE];
  __shared__ float Bsub[TILE][TILE];

  int ty = threadIdx.y, tx = threadIdx.x;
  int row = blockIdx.y * TILE + ty;
  int col = blockIdx.x * TILE + tx;

  float sum = 0.0f;

  // Loop over the K dimension in TILE-sized chunks
  for (int kt = 0; kt < (K + TILE - 1) / TILE; ++kt) {
    // Cooperatively load A-tile and B-tile
    int a_col = kt * TILE + tx;
    int b_row = kt * TILE + ty;
    Asub[ty][tx] = (row < M && a_col < K) ? A[row * K + a_col] : 0.0f;
    Bsub[ty][tx] = (b_row < K && col < N) ? B[b_row * N + col] : 0.0f;

    __syncthreads();

    // Compute — each thread accumulates TILE multiply-adds
    #pragma unroll
    for (int k = 0; k < TILE; ++k) {
      sum += Asub[ty][k] * Bsub[k][tx];
    }

    __syncthreads();
  }

  if (row < M && col < N) {
    C[row * N + col] = sum;
  }
}

// Launch:
//   dim3 block(TILE, TILE);
//   dim3 grid((N+TILE-1)/TILE, (M+TILE-1)/TILE);
//   gemm_tiled<16><<<grid, block>>>(A, B, C, M, N, K);

The math of the improvement

With TILE=16, each global value is reused 16 times within a block. Arithmetic intensity goes from 1 FLOP/byte to 16 FLOP/byte. The kernel is now compute-bound on most hardware. Hitting 30-40% of peak FLOPS is realistic with this code.

Why two __syncthreads?

  1. After load — ensure all threads have finished loading before any thread reads the loaded data.
  2. Before next iteration's load — ensure all threads have finished using the current tile before any thread overwrites it.

Forgetting either is a classic bug. The kernel produces wrong but plausible results — non-determinism that's painful to debug.

Going faster: register tiling

The next level: each thread computes multiple output elements, with the partial sums in registers. This is what cuBLAS does. The block tile and thread tile become two-level hierarchical decomposition:

// Each block: 128x128 output tile
// Each warp:  64x32 output sub-tile
// Each thread: 8x8 output micro-tile in registers
//
// Outer loop k_block (cooperative load to shared)
//   Inner loop k_thread (each thread does an 8x8 outer product in regs)
//
// This is how libraries get to 80%+ of Tensor Core peak.

__global__ void gemm_register_tiled(...) {
  __shared__ float As[BM][BK];
  __shared__ float Bs[BK][BN];

  float Cs[TM][TN] = {0};         // register tile per thread

  for (int k = 0; k < K; k += BK) {
    // 1. Each thread loads multiple A and B elements to shared
    // 2. __syncthreads
    // 3. Outer product loop: load BK fragments from shared to regs,
    //    multiply-accumulate into Cs[TM][TN]
    // 4. __syncthreads
  }
  // Each thread writes its TM*TN register tile to C
}

This is one step shy of using Tensor Cores directly (§25). It's the gateway from textbook performance to library performance.

§14Thread coarsening — when fewer threads are faster

The common instinct: "more parallelism is better, give each thread one element." Often wrong. Coarsening — making each thread do more work — amortizes overhead and exposes instruction-level parallelism.

Why coarsening helps

  • Amortizes thread setup — index calculation, register init, boundary check happen once per several elements.
  • Reuses loaded data — values fetched once stay in registers across iterations.
  • Reduces redundant work — for kernels where threads in a block do partially overlapping work (stencils, reductions), coarsening cuts redundancy.
  • Exposes ILP — the compiler can issue multiple independent operations from one thread's stream.

The grid-stride loop — the canonical coarsening pattern

// Fine-grained: one element per thread, total threads = N
__global__ void add_fine(const float* a, const float* b, float* c, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < N) c[i] = a[i] + b[i];
}

// Coarsened: each thread handles multiple elements via grid-stride loop
__global__ void add_coarse(const float* a, const float* b, float* c, int N) {
  int tid    = blockIdx.x * blockDim.x + threadIdx.x;
  int stride = gridDim.x * blockDim.x;
  for (int i = tid; i < N; i += stride) {
    c[i] = a[i] + b[i];
  }
}

// Launch coarsened with a fixed, modest grid:
int blocks = 160;     // e.g., enough to saturate SMs
int threads = 256;
add_coarse<<<blocks, threads>>>(d_a, d_b, d_c, N);
// Each thread processes ~N / (160*256) elements
// Works for any N without changing launch config

The portability benefit

A grid-stride loop kernel works correctly for any N, and the launch config can be tuned for the hardware (more blocks on bigger GPUs) without changing the math. The fine-grained version requires recomputing grid size on every launch.

When coarsening hurts

  • Working set explodes per thread — if each thread now needs more registers/shared, occupancy drops.
  • Hides parallelism the GPU wanted — for very small problems, you starve SMs by having too few threads.

The right answer is empirical: try 2x, 4x, 8x coarsening, measure. Most non-trivial kernels see a benefit; the sweet spot is usually 2x-4x.

Coarsening + tiling: amortizing the load

In GEMM, each thread computing an 8x8 output tile (§13) is itself a form of coarsening: one tile of A loaded to registers does 8x8 = 64 multiply-adds instead of 1.

The intuition "more threads = more parallelism" is incomplete. The right framing: pick the thread count where you've saturated the SMs and stop adding overhead per element.

Warps & the SIMT machine

Below the block lies the warp — 32 threads that execute in lockstep. Most CUDA performance puzzles ultimately become warp puzzles: what's happening across the 32 threads in a single instruction issue?

§15The warp & SIMT execution

A warp is a group of 32 consecutive threads in a block (threads with the same threadIdx / 32). The hardware schedules and executes a warp as a single SIMT unit. One instruction is fetched and issued to all 32 threads; each thread either executes it (with its own register values) or sits idle.

// Inside a kernel:
int warp_id = threadIdx.x / 32;            // which warp am I in?
int lane_id = threadIdx.x % 32;            // my position within my warp (0..31)

// Or, using the intrinsics:
int lane = lane_id;
// The warp size constant:
// warpSize  — built-in, always 32 on NVIDIA hardware

Implicit synchronization within a warp

Threads in the same warp execute the same instruction at the same time — so they're implicitly synchronized after every instruction. This historically allowed "warp-synchronous programming": skip __syncthreads when communication stays within a warp.

Modern CUDA on Volta+ changed this: warps can have independent program counters per thread (Independent Thread Scheduling). The implicit-sync guarantee is no longer reliable. Use explicit primitives (__syncwarp(), warp shuffles with the appropriate mask) for warp-level communication.

// Pre-Volta: this was OK
shared[tid] = ...;
// no sync, threads in warp see writes immediately
float v = shared[(tid + 1) % 32];

// Modern (safe) version
shared[tid] = ...;
__syncwarp();        // explicit warp-level sync
float v = shared[(tid + 1) % 32];

// Better: use warp shuffle, no shared memory at all
float v = __shfl_sync(0xFFFFFFFF, my_val, (lane + 1) % 32);

Warp scheduling

Each SM has multiple warp schedulers (4 on Ampere/Hopper). Every cycle each scheduler picks one warp that's ready to execute. Latency hiding: if a warp issues a long-latency operation (e.g., HBM load), the scheduler swaps in another ready warp.

This is the model for hiding global memory latency: the SM needs enough warps in flight that some are always ready while others wait. Hence occupancy (§18) — the fraction of max possible warps actually resident.

The single most important mental model for CUDA performance: the SM is a latency-hiding machine. It hides 400-cycle HBM latency by switching between many resident warps. If you have too few warps in flight, latency becomes the bottleneck. If you have enough, the SM is compute-bound (or shared-memory-bound).

§16Warp divergence

When threads in a warp disagree on a branch — some take the if, some take the else — the hardware serializes: it executes the if-path with the agreeing threads active and the others masked off, then the else-path the same way. Worst case: 32-way serialization.

// Divergent: ~50% slower (each branch serialized)
__global__ void divergent(float* x, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i % 2 == 0) {           // half the warp goes here
    x[i] = sqrtf(x[i]);
  } else {                  // other half goes here
    x[i] = x[i] * x[i];
  }
}

// Coherent: each warp goes one way (cheap branch)
__global__ void coherent(float* x, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  int warp_id = i / 32;
  if (warp_id % 2 == 0) {     // entire warp goes one way
    x[i] = sqrtf(x[i]);
  } else {
    x[i] = x[i] * x[i];
  }
}

What's NOT divergent

  • Loops with the same trip count across threads — all threads exit on the same iteration.
  • Branches the compiler proves uniform — e.g., based on blockIdx alone.
  • Conditionals where one side does nothing — sometimes the compiler emits predicated execution instead of branching.

Predicated execution — the cheap branches

For short bodies the compiler often emits predicated instructions: every thread executes the instruction, but the result is conditionally written based on a predicate register. No serialization. The hardware does this automatically for simple cases:

// Often compiled to predicated execution (cheap)
if (i < N) y[i] = x[i] * 2;

// Equivalent in many cases — ?: makes the compiler's job easier
y[i] = (i < N) ? x[i] * 2 : y[i];

Reducing divergence by reorganizing work

The classic example: in a reduction tree, naive code has divergence in early steps:

// Bad: divergent reduction
__shared__ float s[256];
int tid = threadIdx.x;
s[tid] = data[blockIdx.x * 256 + tid];
__syncthreads();

for (int stride = 1; stride < 256; stride *= 2) {
  if (tid % (2 * stride) == 0) {        // half of threads in warp diverge!
    s[tid] += s[tid + stride];
  }
  __syncthreads();
}
// Good: coherent reduction. Active threads form a contiguous prefix.
__shared__ float s[256];
int tid = threadIdx.x;
s[tid] = data[blockIdx.x * 256 + tid];
__syncthreads();

for (int stride = 128; stride > 0; stride >>= 1) {
  if (tid < stride) {                  // active threads are 0..stride-1
    s[tid] += s[tid + stride];         // no divergence within those warps
  }
  __syncthreads();
}

Measuring divergence

ncu --metrics smsp__sass_average_branch_targets_threads_uniform.pct \
    --metrics smsp__thread_inst_executed_per_inst_executed.ratio \
    ./my_program

# branch_targets_uniform_pct = 100% means no divergence
# warp_efficiency = 100% means all 32 lanes active on every instruction
# below ~80% suggests serious divergence somewhere

§17Warp-level primitives

Warp-level intrinsics let 32 threads communicate through registers (not shared memory). They're faster than shared and require no __syncthreads. Modern CUDA depends on them heavily; understanding them is the next level of performance.

__shfl_sync — the most useful primitive

// Each thread in the warp reads source_lane's value of `var`
int result = __shfl_sync(mask, var, source_lane);

// Variants:
__shfl_sync(mask, val, src)          // arbitrary source lane
__shfl_up_sync(mask, val, delta)     // reads from (lane - delta)
__shfl_down_sync(mask, val, delta)   // reads from (lane + delta)
__shfl_xor_sync(mask, val, lane_mask) // reads from (lane ^ lane_mask)

The mask parameter is a 32-bit bitmask of participating lanes. For "all 32 lanes participating," use 0xFFFFFFFF or __activemask().

Warp reduction in 5 instructions

__device__ float warp_sum(float val) {
  for (int offset = 16; offset > 0; offset /= 2) {
    val += __shfl_down_sync(0xFFFFFFFF, val, offset);
  }
  // After the loop, lane 0 holds the sum of all 32 values
  return val;
}

// Visualizes as a butterfly:
//
//   Initially: val_0 val_1 val_2 ... val_31
//   offset=16: val_0+val_16, val_1+val_17, ..., val_15+val_31, ...
//   offset=8:  ...
//   offset=4:  ...
//   offset=2:  ...
//   offset=1:  ...
//   Final lane 0: sum of all 32

Five shuffles, ~5 cycles, no shared memory used — far faster than the shared-memory tree reduction.

__ballot_sync & __any_sync & __all_sync

unsigned int mask = __ballot_sync(0xFFFFFFFF, predicate);
// mask bit i = 1 iff lane i's predicate was true
// Useful for counting / packing without atomics

if (__any_sync(0xFFFFFFFF, condition)) {
  // true if ANY lane in the warp has condition true
}

if (__all_sync(0xFFFFFFFF, condition)) {
  // true if ALL lanes have condition true
}

The compaction example

Stream compaction (keep elements where pred is true): a perfect use of __ballot:

__global__ void compact_warp(const int* in, int* out, int* out_count, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  int lane = threadIdx.x & 31;

  int val = (i < N) ? in[i] : 0;
  bool keep = (i < N) && (val > 0);
  unsigned mask = __ballot_sync(0xFFFFFFFF, keep);

  // Number of kept elements in this warp
  int warp_count = __popc(mask);
  // My index within the warp's kept elements
  int my_pos = __popc(mask & ((1u << lane) - 1));

  // Lane 0 reserves space in the global output
  __shared__ int warp_offset;
  if (lane == 0) warp_offset = atomicAdd(out_count, warp_count);
  __syncwarp();
  int base = __shfl_sync(0xFFFFFFFF, warp_offset, 0);

  if (keep) {
    out[base + my_pos] = val;
  }
}

This is 1-2 orders of magnitude faster than a per-element atomic. Warp-level cooperation is the unlock.

Reduce-broadcast pattern

// Compute a value at lane 0, broadcast to all lanes
int shared_value = compute_at_lane_zero();      // only lane 0's value matters
shared_value = __shfl_sync(0xFFFFFFFF, shared_value, 0);   // broadcast
// Now all lanes have the same value

§18Occupancy & resource limits

Each SM has finite resources: registers, shared memory, warp slots, block slots. The maximum number of blocks (and hence warps) that can be resident on an SM is determined by whichever resource hits its limit first. Higher occupancy gives the SM more options for latency hiding, but trying to maximize it can hurt other things.

What limits occupancy

  • Threads / SM cap — 2048 on current hardware.
  • Blocks / SM cap — 32 on current hardware.
  • Register file — 256 KB / SM. If your kernel uses 64 regs/thread and you have 1024 threads/block, that's 256 KB — one block fits per SM.
  • Shared memory — 100-228 KB / SM. If a block uses 48 KB, you fit at most 4-5 blocks/SM.

Compute occupancy yourself

int blockSize = 256;
int dynamicSmem = 0;
int maxBlocks;

cudaOccupancyMaxActiveBlocksPerMultiprocessor(
    &maxBlocks, my_kernel, blockSize, dynamicSmem);

// Or the higher-level helper that suggests block size
int blockSizeRecommended, minGridSize;
cudaOccupancyMaxPotentialBlockSize(
    &minGridSize, &blockSizeRecommended,
    my_kernel, 0, 0);

The occupancy-tuning false god

Maximizing occupancy is not the same as maximizing performance. Kernels often run faster at 50% occupancy with more registers per thread than at 100% with spills. The rules:

  • Memory-bound kernels — usually want higher occupancy. More warps in flight = more latency hiding.
  • Compute-bound kernels — often want lower occupancy with more registers. Each warp has more state, less spilling, better ILP.
  • 50% occupancy is plenty — below that, latency hiding starts to suffer; above it, marginal gains.

Trust profilers (ncu reports "achieved occupancy" and "theoretical occupancy") over instinct. The "Stall Reasons" section in Nsight Compute tells you whether occupancy is the actual bottleneck.

Controlling register usage

// Tell the compiler "use no more than 64 registers per thread"
__global__ __launch_bounds__(256, 4) void my_kernel(...) { ... }
//                            ^         ^
//                            |         +-- min blocks per SM (occupancy hint)
//                            +-- max threads per block

// Or globally via compiler flag:
// nvcc -maxrregcount=64 ...

// Or via cudaFuncSetAttribute
cudaFuncSetAttribute(my_kernel,
    cudaFuncAttributePreferredSharedMemoryCarveout, 75);
//                       larger fraction for shared (vs L1)
The "increase occupancy" trap

Forcing a kernel from 32 regs to 24 regs to get more occupancy may cause spills. Spills hit local memory (= HBM); your "more occupied" kernel may now be HBM-bound when it was compute-bound. Always profile the change end-to-end — don't chase occupancy as a number in isolation.

Synchronization & atomics

Threads on the GPU need to coordinate — sometimes within a warp, sometimes across a block, sometimes across the entire grid. CUDA provides primitives at each level, with different costs and guarantees.

__syncthreads()

The block-level barrier. Every thread in the block must reach it before any thread proceeds. Used to ensure all threads have finished writing shared memory before any reads, or finished reading before overwrites.

__shared__ float tile[256];

tile[tid] = global_array[idx];      // each thread writes shared memory
__syncthreads();                   // wait for ALL threads to finish writing

float v = tile[255 - tid];        // safe: every thread sees the final values
The deadlock you must avoid

Never call __syncthreads inside a branch unless every thread in the block enters that branch. If some threads skip the sync, the others wait forever — the kernel hangs.

// BAD: threads with tid >= 128 never reach the sync
if (tid < 128) {
  tile[tid] = ...;
  __syncthreads();      // HANGS
}

// OK: every thread reaches the sync, the conditional work is OK
if (tid < 128) {
  tile[tid] = ...;
}
__syncthreads();

Memory fences vs barriers

A barrier blocks execution until all threads arrive. A fence doesn't block — it orders memory operations. Use fences when you need writes from this thread to be visible to other threads, but the threads aren't synchronizing in lockstep.

// Producer-consumer pattern across blocks via global memory
__global__ void producer(int* flag, int* data) {
  data[0] = 42;
  __threadfence();             // ensure write to data is visible globally
  if (threadIdx.x == 0) {
    atomicExch(flag, 1);          // signal: data is ready
  }
}

__global__ void consumer(volatile int* flag, int* data) {
  if (threadIdx.x == 0) {
    while (*flag != 1);           // spin until producer signals
  }
  __syncthreads();
  __threadfence();             // ensure we see the data writes too
  // safe to read data[0] now
}

Cooperative groups: modern syncing

CUDA 9+ introduced cooperative_groups, a more flexible API:

#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ void k() {
  auto block = cg::this_thread_block();
  block.sync();                          // equivalent to __syncthreads()

  auto warp = cg::tiled_partition<32>(block);
  warp.sync();                           // equivalent to __syncwarp()

  // Partial sums within a sub-warp:
  auto sub = cg::tiled_partition<8>(warp);
  int rank = sub.thread_rank();
  int sum  = cg::reduce(sub, my_val, cg::plus<int>());
}

Cooperative kernel launches (grid-wide sync)

// Within a kernel that was launched cooperatively:
__global__ void k() {
  cg::grid_group grid = cg::this_grid();
  // phase 1
  ...
  grid.sync();      // every block in the grid waits here
  // phase 2
  ...
}

// Launch must use cudaLaunchCooperativeKernel and the kernel must
// fit on the device with full grid resident simultaneously.

Grid sync is expensive (it serializes the whole device) but useful for multi-step algorithms that would otherwise require kernel boundaries.

§20Atomics & race conditions

When multiple threads write to the same memory location, you need an atomic to avoid lost updates. CUDA atomics work at warp, block, device, and system scope (the last spans CPU and GPU).

// The common ones:
atomicAdd(int*, int);            // also float, double on sm_60+
atomicSub(int*, int);
atomicMin(int*, int);
atomicMax(int*, int);
atomicAnd(int*, int);
atomicOr(int*, int);
atomicXor(int*, int);
atomicExch(int*, int);            // exchange: old = *p; *p = new; return old
atomicCAS(int*, int compare, int val);   // compare-and-swap (the workhorse)

Histogram — the canonical atomic example

// Naive: one global atomic per element. Heavy contention.
__global__ void histogram_naive(const unsigned char* data, int* hist, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < N) {
    atomicAdd(&hist[data[i]], 1);     // every thread atomics global
  }                                       // 256 bins, N threads → contention
}

// Better: per-block histogram in shared memory, merge at end
__global__ void histogram_tiled(const unsigned char* data, int* hist, int N) {
  __shared__ int local_hist[256];

  // Zero the shared histogram
  for (int i = threadIdx.x; i < 256; i += blockDim.x) {
    local_hist[i] = 0;
  }
  __syncthreads();

  // Each thread atomics into the shared (fast) histogram
  int tid = blockIdx.x * blockDim.x + threadIdx.x;
  int stride = gridDim.x * blockDim.x;
  for (int i = tid; i < N; i += stride) {
    atomicAdd(&local_hist[data[i]], 1);
  }
  __syncthreads();

  // Merge the per-block histograms into the global one (one atomic per bin per block)
  for (int i = threadIdx.x; i < 256; i += blockDim.x) {
    atomicAdd(&hist[i], local_hist[i]);
  }
}

The shared-memory version is typically 10-100x faster than naive, even though both use atomics. The reasons: shared-memory atomics are far cheaper than global, and the global atomics are now blocks * 256 total instead of N.

atomicCAS: building any atomic you want

// Atomic float-min (not natively supported on some hardware)
__device__ float atomicMinFloat(float* addr, float value) {
  int* p = (int*) addr;
  int old = *p, expected;
  do {
    expected = old;
    float current = __int_as_float(expected);
    float updated = fminf(current, value);
    int updated_bits = __float_as_int(updated);
    old = atomicCAS(p, expected, updated_bits);
  } while (old != expected);
  return __int_as_float(old);
}

The atomic cost hierarchy

  • Shared memory atomic, no contention — ~10 cycles.
  • Shared memory atomic, contended — serialized; up to 32x slower per contending lane.
  • Global atomic, L2-resident — ~30-100 cycles.
  • Global atomic, HBM-resident — ~hundreds of cycles.
  • System atomic across host-device — much more, only on coherent systems.

Design rule: keep atomics in shared memory when possible, then merge with global atomics (or, better, hierarchical reduction) at the end.

§21Reductions in depth

Reductions compute one (or a few) values from a large array: sum, max, argmax, etc. They're the second most common pattern after matrix ops, and they're a classic test of CUDA fluency.

The three-stage strategy

  1. Per-warp reduction with shuffles — 32 values to 1.
  2. Per-block reduction by writing each warp's result to shared, then reducing again.
  3. Per-grid reduction by having each block atomic-add into a global accumulator, OR by launching a second kernel.
__inline__ __device__ float warp_reduce_sum(float val) {
  #pragma unroll
  for (int offset = 16; offset > 0; offset /= 2) {
    val += __shfl_down_sync(0xFFFFFFFF, val, offset);
  }
  return val;
}

__inline__ __device__ float block_reduce_sum(float val) {
  __shared__ float shared[32];    // one per warp
  int lane = threadIdx.x & 31;
  int wid  = threadIdx.x >> 5;

  val = warp_reduce_sum(val);     // reduce within each warp
  if (lane == 0) shared[wid] = val;
  __syncthreads();

  // First warp reduces the warp partial sums
  val = (threadIdx.x < blockDim.x / 32) ? shared[lane] : 0.0f;
  if (wid == 0) val = warp_reduce_sum(val);
  return val;                  // only valid for thread 0
}

__global__ void reduce_kernel(const float* in, float* out, int N) {
  float sum = 0.0f;
  // Grid-stride load: each thread reads many inputs and accumulates
  int tid = blockIdx.x * blockDim.x + threadIdx.x;
  int stride = gridDim.x * blockDim.x;
  for (int i = tid; i < N; i += stride) {
    sum += in[i];
  }
  // Block reduction
  sum = block_reduce_sum(sum);
  // Block 0's thread 0 has this block's contribution; atomic-add to out
  if (threadIdx.x == 0) atomicAdd(out, sum);
}

What this gets you

Within a few percent of HBM bandwidth for the load (memory-bound reduction), with very few instructions per element. This pattern (grid-stride load + warp reduce + block reduce + atomic) is the template for almost every "reduce X over array" kernel.

Argmax: the variation

__inline__ __device__
void warp_reduce_argmax(float& v, int& idx) {
  #pragma unroll
  for (int off = 16; off > 0; off /= 2) {
    float other_v = __shfl_down_sync(0xFFFFFFFF, v, off);
    int   other_i = __shfl_down_sync(0xFFFFFFFF, idx, off);
    if (other_v > v) { v = other_v; idx = other_i; }
  }
}

Use the library when you can

For production code, hand-rolled reductions rarely beat cub::DeviceReduce or thrust::reduce:

#include <cub/cub.cuh>

void* temp_storage = nullptr;
size_t temp_bytes = 0;
cub::DeviceReduce::Sum(temp_storage, temp_bytes, d_in, d_out, N);
cudaMalloc(&temp_storage, temp_bytes);
cub::DeviceReduce::Sum(temp_storage, temp_bytes, d_in, d_out, N);

CUB's reductions are aggressively tuned and very close to HBM peak. Use them. Build hand-rolled ones to understand the technique, not for production.

§22Scan / prefix sum

Prefix sum (scan) is the second great parallel primitive: out[i] = in[0] + in[1] + ... + in[i]. It looks inherently serial but parallelizes elegantly.

Inclusive vs exclusive

in        = [3, 1, 4, 1, 5, 9, 2, 6]

inclusive = [3, 4, 8, 9,14,23,25,31]      out[i] includes in[i]
exclusive = [0, 3, 4, 8, 9,14,23,25]      out[i] excludes in[i]
                                          (out[0] = identity)

The Hillis-Steele scan (illustrative; O(n log n) work)

__global__ void scan_hillis_steele(float* data, int N) {
  __shared__ float tmp[256];
  int tid = threadIdx.x;
  tmp[tid] = data[tid];
  __syncthreads();

  for (int offset = 1; offset < N; offset *= 2) {
    float v = (tid >= offset) ? tmp[tid - offset] : 0.0f;
    __syncthreads();
    tmp[tid] += v;
    __syncthreads();
  }
  data[tid] = tmp[tid];
}

Easy to read; O(n log n) work. For production, the work-efficient Blelloch scan does O(n) work but is more complex.

Warp-level scan with shuffles

__device__ int warp_inclusive_scan(int val) {
  int lane = threadIdx.x & 31;
  #pragma unroll
  for (int i = 1; i < 32; i *= 2) {
    int n = __shfl_up_sync(0xFFFFFFFF, val, i);
    if (lane >= i) val += n;
  }
  return val;
}

Block-wide scan

The composition: each warp scans in registers; lane 31 of each warp writes its total to shared; a single warp scans those totals; broadcast to add the offset back to each warp's elements. Lots of moving parts — use CUB:

#include <cub/cub.cuh>

__global__ void scan_block(int* data) {
  typedef cub::BlockScan<int, 256> BlockScan;
  __shared__ typename BlockScan::TempStorage tmp;

  int v = data[blockIdx.x * 256 + threadIdx.x];
  int result;
  BlockScan(tmp).InclusiveSum(v, result);
  data[blockIdx.x * 256 + threadIdx.x] = result;
}

// Or for a full-array scan:
cub::DeviceScan::InclusiveSum(temp, temp_bytes, d_in, d_out, N);

What scan unlocks

  • Stream compaction — scan a 0/1 keep-mask, use the result as the output index.
  • Radix sort — uses scan to compute bin offsets at each pass.
  • Sparse operations — CSR matrix construction, segment offsets.
  • Histogram-like merges — per-block totals to global offsets.

The line in any parallel-algorithms course: "scan is to parallel programming what for-loops are to sequential programming." Make peace with it.

Streams & concurrency

A kernel launch is non-blocking, but on a single stream operations execute in order. Multiple streams let you overlap compute with transfer, and compute with compute. Events provide the timing and dependency primitives that hold the pipeline together.

§23Streams & concurrency

A stream is a sequence of operations that execute in order on the device. Operations in different streams may overlap. The default stream (stream 0) has special blocking semantics that often surprise people.

The basic API

cudaStream_t s;
cudaStreamCreate(&s);

// All these run in order on stream s, but may overlap with other streams
cudaMemcpyAsync(d_a, h_a, bytes, cudaMemcpyHostToDevice, s);
my_kernel<<<grid, block, 0, s>>>(d_a);
cudaMemcpyAsync(h_r, d_r, bytes, cudaMemcpyDeviceToHost, s);

cudaStreamSynchronize(s);     // wait for everything on this stream
cudaStreamDestroy(s);

The classic three-way overlap

Modern GPUs have separate engines for compute, host-to-device transfer, and device-to-host transfer. With enough streams, all three can run simultaneously:

Without streams (serial): H2D: [###] cmp: [######] D2H: [###] total time: ============================> With 4 streams (pipelined): stream 0 H2D: [#] cmp: [##] D2H: [#] stream 1 H2D: [#] cmp: [##] D2H: [#] stream 2 H2D: [#] cmp: [##] D2H: [#] stream 3 H2D: [#] cmp: [##] D2H: [#] total time: ==========> (~3x faster on transfer-heavy work)
const int NSTREAMS = 4;
cudaStream_t streams[NSTREAMS];
for (int i = 0; i < NSTREAMS; ++i) cudaStreamCreate(&streams[i]);

size_t chunk = N / NSTREAMS;
for (int i = 0; i < NSTREAMS; ++i) {
  size_t off = i * chunk;
  cudaStream_t s = streams[i];
  cudaMemcpyAsync(d_in  + off, h_in  + off, chunk*sizeof(float), cudaMemcpyHostToDevice, s);
  my_kernel<<<grid, block, 0, s>>>(d_in + off, d_out + off, chunk);
  cudaMemcpyAsync(h_out + off, d_out + off, chunk*sizeof(float), cudaMemcpyDeviceToHost, s);
}
for (int i = 0; i < NSTREAMS; ++i) cudaStreamSynchronize(streams[i]);
Prerequisites for overlap

For H2D/D2H overlap to actually happen: (1) host buffers must be pinned (cudaMallocHost); (2) use cudaMemcpyAsync, not cudaMemcpy; (3) the operations must be on different non-default streams. Missing any of these silently serializes the pipeline.

The default stream — the synchronization landmine

Stream 0 (the default) is special. By default it's the "legacy default stream," which synchronizes with all other streams: any operation on stream 0 waits for all prior work on every stream, and subsequent work on every stream waits for stream 0.

// This serializes despite using multiple streams!
kernel_A<<<g, b, 0, sA>>>();
kernel_default<<<g, b>>>();        // stream 0: blocks until sA finishes
kernel_B<<<g, b, 0, sB>>>();       // blocks until kernel_default finishes

// Solutions:
// 1. Don't use the default stream when you want concurrency.
// 2. Compile with --default-stream=per-thread (each host thread gets its own default).
// 3. Use cudaStreamCreateWithFlags(&s, cudaStreamNonBlocking) for non-default streams.

Stream priorities

int low, high;
cudaDeviceGetStreamPriorityRange(&low, &high);
// low = 0, high typically = -1 or -5 (lower number = higher priority)

cudaStream_t high_pri;
cudaStreamCreateWithPriority(&high_pri, cudaStreamNonBlocking, high);
// Kernels on high_pri preempt waiting kernels on low-priority streams
// at block boundaries.

Stream callbacks

void CUDART_CB on_done(void* user_data) {
  printf("stream finished its prior work\n");
}

my_kernel<<<g, b, 0, s>>>();
cudaLaunchHostFunc(s, on_done, nullptr);   // runs on host after the kernel completes

CUDA Graphs — when streams aren't enough

For workloads with many small kernels, even non-blocking launches become the bottleneck (launch latency ~5 microseconds each). CUDA Graphs let you record a sequence of operations once and replay it many times with much lower overhead:

cudaGraph_t graph;
cudaGraphExec_t exec;
cudaStream_t s;
cudaStreamCreate(&s);

// Capture the operations on stream s into a graph
cudaStreamBeginCapture(s, cudaStreamCaptureModeGlobal);
kernel_A<<<g, b, 0, s>>>();
kernel_B<<<g, b, 0, s>>>();
cudaMemcpyAsync(...);
cudaStreamEndCapture(s, &graph);

cudaGraphInstantiate(&exec, graph, nullptr, nullptr, 0);

// Now launch the whole sequence in one call:
for (int step = 0; step < 10000; ++step) {
  cudaGraphLaunch(exec, s);
}
cudaStreamSynchronize(s);

cudaGraphExecDestroy(exec);
cudaGraphDestroy(graph);

For inference workloads with repeating patterns, graphs commonly give 10-30% end-to-end speedup just by removing launch overhead.

§24Events & timing

Events mark points in a stream. They're used for two things: timing intervals between points, and creating dependencies between streams.

Timing a kernel

cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start);                // mark the start point
my_kernel<<<grid, block>>>(d_a, d_b, d_c, N);
cudaEventRecord(stop);                 // mark the end point

cudaEventSynchronize(stop);            // wait for stop event to complete

float ms;
cudaEventElapsedTime(&ms, start, stop);   // difference in milliseconds
printf("kernel time: %.3f ms\n", ms);

cudaEventDestroy(start);
cudaEventDestroy(stop);

Resolution is ~0.5 microseconds. Much more accurate than CPU timers (which would include kernel-launch latency in the measurement).

Don't time with CPU clocks

Wrapping a kernel launch with std::chrono measures the time to queue the kernel, not to run it. Use events. The classic mistake: "my kernel takes 5 microseconds" when actually the kernel is async and you measured the launch.

Cross-stream dependencies

// Kernel on stream B waits for an event from stream A — without
// blocking the host or synchronizing all of stream A.

cudaEvent_t e;
cudaEventCreateWithFlags(&e, cudaEventDisableTiming);   // faster, no timestamping

kernel_A<<<g, b, 0, streamA>>>();
cudaEventRecord(e, streamA);

cudaStreamWaitEvent(streamB, e, 0);   // streamB will wait for `e` before
kernel_B<<<g, b, 0, streamB>>>();   // kernel_B runs, but not before then

Polling vs blocking event sync

// Blocking sync: thread sleeps until done (good for waits >1ms)
cudaEventSynchronize(stop);

// Polling: check without blocking
while (cudaEventQuery(stop) == cudaErrorNotReady) {
  // do CPU work while waiting
}

// Configure how cudaEventSynchronize behaves:
cudaEventCreateWithFlags(&e, cudaEventBlockingSync);    // uses OS sync (saves CPU)
cudaEventCreateWithFlags(&e, cudaEventDefault);          // spins by default (low latency)

Pattern: measuring kernel + transfer separately

cudaEvent_t t0, t1, t2, t3;
for (auto& e : {&t0, &t1, &t2, &t3}) cudaEventCreate(e);

cudaEventRecord(t0, s);
cudaMemcpyAsync(d_in, h_in, bytes, cudaMemcpyHostToDevice, s);
cudaEventRecord(t1, s);
my_kernel<<<g, b, 0, s>>>(d_in, d_out, N);
cudaEventRecord(t2, s);
cudaMemcpyAsync(h_out, d_out, bytes, cudaMemcpyDeviceToHost, s);
cudaEventRecord(t3, s);
cudaEventSynchronize(t3);

float h2d, kern, d2h;
cudaEventElapsedTime(&h2d,  t0, t1);
cudaEventElapsedTime(&kern, t1, t2);
cudaEventElapsedTime(&d2h,  t2, t3);
printf("H2D %.2fms  kernel %.2fms  D2H %.2fms\n", h2d, kern, d2h);

This is the workhorse measurement pattern. You quickly learn which leg of the pipeline dominates — and where the optimization budget goes.

Part VIII

Modern CUDA: Tensor Cores, async copy, TMA

The CUDA you learn from textbooks is the late-2010s CUDA. The CUDA running on modern hardware has additional hardware units — Tensor Cores for matmul, async copy units for memory traffic, the Tensor Memory Accelerator for bulk transfers — that change which kernels are competitive. Skipping these means leaving 5-10x performance on the table for the workloads they target.

§25Tensor Cores & WMMA

Tensor Cores are dedicated matrix-multiply units on every SM since Volta. Each one computes a small matrix multiplication-accumulate per cycle: D = A * B + C for small fixed-size tiles. Software programs them as a warp-wide operation — one warp issues one Tensor Core instruction and gets a tile of output.

The shapes

GenerationSizes (M×N×K)InputsAccumulator
Volta (1st gen)16×16×16FP16FP16/FP32
Turing (2nd)+ INT8/INT4+ integer typesINT32
Ampere (3rd)+ TF32, BF16, FP64+ TF32, BF16, FP64FP32, FP64
Hopper (4th)+ FP8 (E4M3, E5M2)+ FP8FP32
Blackwell (5th)+ FP4, MXFP+ FP4 micro-scaled formatsFP32

The WMMA API — warp matrix multiply-accumulate

#include <mma.h>
using namespace nvcuda;

// 16x16x16 tile, FP16 inputs, FP32 accumulator
__global__ void wmma_gemm(const half* A, const half* B, float* C,
                          int M, int N, int K) {
  // Tile of A: 16 rows x 16 cols of half
  // Tile of B: 16 rows x 16 cols of half (column-major for performance)
  // Tile of C/D: 16 rows x 16 cols of float

  int warpM = (blockIdx.y * blockDim.y + threadIdx.y) / 1;
  int warpN = blockIdx.x * blockDim.x / 32 + threadIdx.x / 32;

  wmma::fragment<wmma::matrix_a,   16,16,16, half,  wmma::row_major> a_frag;
  wmma::fragment<wmma::matrix_b,   16,16,16, half,  wmma::col_major> b_frag;
  wmma::fragment<wmma::accumulator,16,16,16, float>                 c_frag;

  wmma::fill_fragment(c_frag, 0.0f);

  for (int k = 0; k < K; k += 16) {
    const half* a_tile = A + warpM*16*K + k;
    const half* b_tile = B + k*N + warpN*16;

    wmma::load_matrix_sync(a_frag, a_tile, K);
    wmma::load_matrix_sync(b_frag, b_tile, N);

    wmma::mma_sync(c_frag, a_frag, b_frag, c_frag);  // the magic line
  }

  float* c_tile = C + warpM*16*N + warpN*16;
  wmma::store_matrix_sync(c_tile, c_frag, N, wmma::mem_row_major);
}

What you get

  • Volta — ~8x FP16 throughput vs CUDA cores.
  • Ampere — another 2x; structured sparsity adds another 2x for compatible weights.
  • Hopper — FP8 brings another 2x over FP16.
  • Blackwell — FP4 doubles again for inference.

Reality check: you probably won't write WMMA by hand

For real GEMMs use:

  • cuBLAS — the standard library; hits 80-95% of peak on most shapes.
  • CUTLASS — C++ templates for composing custom GEMMs; this is the path when you need a fused or non-standard variant.
  • cuDNN — convolution-specific paths, fused conv+bias+activation, etc.

The WMMA API is right when you're writing a fused kernel where a matmul is one step among others — e.g., a fused attention kernel. CUTLASS now handles most of those cases too, at higher abstraction.

TF32 — the silent precision change

TF32 is a 19-bit format: FP32-range exponent, FP16-range mantissa. Tensor Cores run TF32 inputs natively while the rest of CUDA still sees FP32. PyTorch defaults to TF32 matmul on A100+ for ~5-8x speedup, with mostly-FP32 accuracy:

// In a CUDA kernel using cuBLAS, enable TF32 for the math:
cublasSetMathMode(handle, CUBLAS_TF32_TENSOR_OP_MATH);

// In PyTorch:
// torch.backends.cuda.matmul.allow_tf32 = True   (default since 1.7 on Ampere+)
// torch.backends.cudnn.allow_tf32 = True

This is the source of "my CI runner produces slightly different numerics" bugs — an A100 with TF32 enabled gives different output than a V100 without it.

§26Async copy & cp.async

Before Ampere, loading shared memory required: global load to register, register store to shared, then __syncthreads. This pulled every byte through the register file unnecessarily. Ampere introduced cp.async — a hardware path that copies directly from L2 to shared memory, bypassing registers.

The pre-async pattern

__shared__ float tile[256];

float r = global_data[idx];     // global -> register (latency hit)
tile[tid] = r;                  // register -> shared
__syncthreads();               // wait for all threads to finish loading
// now use tile[]

The cp.async pattern

#include <cuda/barrier>
#include <cuda_pipeline.h>
namespace cg = cooperative_groups;

__shared__ float tile[256];

// Async copy: issued by the warp, doesn't block thread execution
__pipeline_memcpy_async(&tile[tid], &global_data[idx], sizeof(float));
__pipeline_commit();
__pipeline_wait_prior(0);     // wait for the most recent batch
__syncthreads();
// now use tile[]

The real win: software pipelining

Because cp.async doesn't block, you can issue the next tile's load before consuming the current tile. Two-stage pipeline overlaps load and compute:

constexpr int STAGES = 2;
__shared__ float tile[STAGES][256];

// Prime stage 0
__pipeline_memcpy_async(&tile[0][tid], &in[0 + tid], sizeof(float));
__pipeline_commit();

for (int step = 0; step < n_steps; ++step) {
  int curr = step % STAGES;
  int next = (step + 1) % STAGES;

  // Issue the next async load before consuming the current tile
  if (step + 1 < n_steps) {
    int next_off = (step + 1) * 256;
    __pipeline_memcpy_async(&tile[next][tid], &in[next_off + tid], sizeof(float));
    __pipeline_commit();
  }

  // Wait for THIS step's data, then compute
  __pipeline_wait_prior(sizeof...(1));   // keep 1 in flight
  __syncthreads();
  compute_using(tile[curr]);
}

On Ampere this typically buys 10-30% in compute-bound kernels (the loads don't get into the critical path). On Hopper, where TMA takes over the bulk-load duty, the manual cp.async pattern is less common in new code.

The library shortcut

#include <cooperative_groups/memcpy_async.h>

auto block = cg::this_thread_block();
cg::memcpy_async(block, tile, global_ptr, sizeof(float) * 256);
cg::wait(block);

The cooperative-groups wrapper hides the pipeline machinery for the common case. Reach for the lower-level intrinsics only when manually scheduling multiple in-flight stages.

§27TMA & Hopper-specific features

The Hopper architecture (sm_90, H100) adds dedicated hardware for bulk memory transfers and a new programming layer above blocks: thread block clusters.

TMA — Tensor Memory Accelerator

TMA is a dedicated hardware engine for asynchronously copying tiles between global memory and shared memory. One thread issues one TMA instruction to move a tile of any shape (up to 5D). The hardware handles addressing, bounds, and progress tracking.

#include <cuda.h>
#include <cuda/barrier>

// Host side: create a TMA tensor descriptor describing the global tensor
CUtensorMap tma_desc;
cuTensorMapEncodeTiled(
    &tma_desc,
    CU_TENSOR_MAP_DATA_TYPE_FLOAT32,
    2,                          // rank
    d_data,                     // global pointer
    global_dims, global_strides,
    box_dims,                   // tile size
    elem_strides,
    CU_TENSOR_MAP_INTERLEAVE_NONE,
    CU_TENSOR_MAP_SWIZZLE_128B,
    CU_TENSOR_MAP_L2_PROMOTION_NONE,
    CU_TENSOR_MAP_FLOAT_OOB_FILL_NONE);

// Device side: one thread issues the copy
__global__ void tma_load_kernel(const CUtensorMap* desc) {
  __shared__ alignas(128) float tile[128][128];
  __shared__ cuda::barrier<cuda::thread_scope_block> bar;

  if (threadIdx.x == 0) init(&bar, 1);
  __syncthreads();

  if (threadIdx.x == 0) {
    // Issue the TMA copy of a tile starting at (block_x*128, block_y*128)
    cp_async_bulk_tensor_2d(
        tile, desc,
        blockIdx.x * 128,
        blockIdx.y * 128,
        &bar);
  }
  bar.arrive_and_wait();         // wait for the TMA copy to finish

  // Now use tile[][] — same as if filled by a normal load loop
}

Why TMA matters

  • One instruction moves an entire tile; no per-element loop, no per-thread address math.
  • Asynchronous; the rest of the warp continues issuing other work.
  • Hardware bounds-checking; out-of-range elements are filled with 0 (or NaN) without per-thread guards.
  • Built-in swizzle: TMA can write tiles into shared memory with bank-conflict-free layout automatically.

For state-of-the-art GEMM and attention kernels, TMA replaces the manual cp.async pipeline of the Ampere generation. CUTLASS 3.x is built around it.

Thread block clusters

Hopper adds a level above blocks: a cluster. Up to 16 blocks can be grouped into a cluster that runs on the same SM-cluster (a Hopper GPC contains multiple SMs). Within a cluster, blocks can:

  • Access each other's shared memory (Distributed Shared Memory, DSMEM).
  • Synchronize via cluster-level barriers.
  • Atomic operations across the cluster's shared memory.
#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ __cluster_dims__(2, 2, 1) void cluster_kernel(...) {
  // This kernel launches with 2x2 = 4 blocks per cluster
  auto cluster = cg::this_cluster();
  cg::cluster_group cg_cluster = cg::this_cluster();

  unsigned int bid_in_cluster = cluster.block_rank();
  cluster.sync();

  // Get another block's shared memory pointer:
  __shared__ float my_data[256];
  float* peer_data = cg::cluster_group::map_shared_rank(my_data, 1);
  // peer_data points into block 1's shared memory — this thread can read it!
}

The conceptual jump

Clusters give you a working set up to 16x larger than what fits in one block's shared memory, while still keeping access fast (much faster than going through global). For Hopper-tuned kernels (FlashAttention-3, modern CUTLASS GEMM), clusters are how the algorithm fits efficiently.

When to reach for Hopper features

If you're writing a generic kernel that runs on Ampere + Hopper, stick to cp.async patterns. Reach for TMA + clusters when you're targeting Hopper specifically and the kernel's performance matters — e.g., the inner loop of an inference engine. For learning, read the CUTLASS 3.x examples; they're the canonical TMA/cluster reference code.

§28Cooperative groups

cooperative_groups is a header-only library shipped with CUDA that exposes synchronization and communication primitives at every scope: thread block, warp, sub-warp tile, cluster, grid, multi-grid. It's the modern way to write sync code that's correct, portable, and composable.

The scopes, top down

#include <cooperative_groups.h>
namespace cg = cooperative_groups;

__global__ void k() {
  // Whole grid (only if launched cooperatively):
  cg::grid_group       grid    = cg::this_grid();
  // Whole cluster (Hopper+, only if __cluster_dims__):
  cg::cluster_group    cluster = cg::this_cluster();
  // The current thread block:
  cg::thread_block     block   = cg::this_thread_block();
  // The current warp:
  auto                 warp    = cg::tiled_partition<32>(block);
  // A sub-warp tile of 8:
  auto                 octet   = cg::tiled_partition<8>(warp);

  // Each group has: .sync(), .size(), .thread_rank(), .meta_group_size(), etc.
  block.sync();              // equivalent to __syncthreads()
  warp.sync();               // equivalent to __syncwarp()
}

Sub-warp tiles — the new primitive

Tiles of 1, 2, 4, 8, 16, or 32 threads. They are statically-sized warp partitions, with their own sync and shuffle:

auto warp = cg::tiled_partition<32>(cg::this_thread_block());
auto quad = cg::tiled_partition<4>(warp);
// 4-thread groups; each warp contains 8 quads

int rank_in_quad = quad.thread_rank();         // 0..3
int total_in_warp = quad.meta_group_size();    // 8 quads per warp

// Sub-warp shuffles:
int v = quad.shfl(my_val, 0);                 // broadcast within the quad
int s = cg::reduce(quad, my_val, cg::plus<int>());    // reduction within the quad

Algorithm helpers

#include <cooperative_groups/reduce.h>
#include <cooperative_groups/scan.h>

int sum  = cg::reduce(warp, val, cg::plus<int>());
int excl = cg::exclusive_scan(warp, val, cg::plus<int>());
int incl = cg::inclusive_scan(warp, val, cg::plus<int>());

int maxv = cg::reduce(block, val, cg::greater<int>());

These compose: a function written for a generic group works at any scope. That's the design payoff — you write one reduction, you use it on tiles, warps, blocks, or clusters.

Why this matters beyond syntax

The old API had __syncthreads and __shfl_sync. The new API gives a typed handle to whichever group of threads is currently relevant, with the right sync and the right shuffle baked in. The compiler validates that you're not, e.g., calling block-level sync from inside divergent code. For complex multi-level kernels, the safety is real.

Cooperative groups are the modern lingua franca: prefer them over raw __syncthreads/__shfl_sync in new code, especially at warp and sub-warp scope.

Part IX

Profiling, debugging & pitfalls

Writing the kernel is half the work. The other half is measuring it — finding out what it actually costs, on real hardware, against the right baseline. CUDA has two excellent tools (nsys and ncu) plus the roofline model that ties measurement to ceiling. Skipping this step is how kernels with "obvious" optimizations end up slower than naive code.

§29Profiling: nsys, ncu, the roofline

The two tools, and what each is for

The workflow is two-stage: run nsys first to identify which kernel dominates wall time, then ncu on that kernel to investigate.

Nsight Systems — the timeline view

# Profile an application and write a report
nsys profile -o my_report --stats=true ./my_program

# Open the GUI for the timeline view
nsys-ui my_report.nsys-rep

# Or print a summary to stdout
nsys stats my_report.nsys-rep

What you look for in the timeline:

  • Long gaps between kernels — CPU bottleneck, sync, or launch overhead.
  • Memcpy bars that don't overlap with kernel bars — missing streams or pageable memory.
  • One kernel taking 80% of the time — that's the one to focus on.
  • Per-stream lanes — if you intended concurrency, do the lanes actually overlap?

Nsight Compute — the per-kernel deep dive

# Profile every kernel with a comprehensive metric set
ncu --set full -o report ./my_program

# Profile only one specific kernel (faster, less output)
ncu -k "gemm_tiled" --set full -o report ./my_program

# Quick command-line summary
ncu --set default ./my_program

# Cherry-pick specific metrics
ncu --metrics dram__bytes.sum,smsp__inst_executed.sum ./my_program

# Open the GUI
ncu-ui report.ncu-rep

The roofline model

The roofline plot is the single best mental tool for kernel performance. Two axes:

  • X axis: arithmetic intensity (FLOPs per byte transferred from HBM).
  • Y axis: achieved throughput (FLOP/s or GB/s).

Two ceilings:

  • Horizontal line at peak FLOP/s — compute ceiling.
  • Diagonal line: peak_bandwidth × arithmetic_intensity — memory ceiling.

The crossover (ridge point) is the AI value where you switch from memory-bound to compute-bound. For H100 SXM: peak FP32 ~67 TFLOPS, peak HBM ~3.35 TB/s, ridge point ~20 FLOP/byte.

Throughput (FLOP/s) ^ | peak ___ |................................. | compute-bound region | +-----------+---+--------- | / kernel here | / is compute-bound | / | / | / | / memory-bound region | / |/ +-----------------------------> Arithmetic Intensity ridge point (FLOPs per byte from HBM)

Putting your measured kernel on this plot tells you which ceiling it hits and how much room remains. ncu --set roofline generates the data; many third-party tools render it.

§30Debugging & common pitfalls

The debugging hierarchy

  1. Synchronous launches — CUDA_LAUNCH_BLOCKING=1 pins errors to the offending kernel.
  2. compute-sanitizer — CUDA's ASan/MSan equivalent; finds memory errors and races.
  3. cuda-gdb — symbolic debugger; step through kernels, inspect threads.
  4. printf inside kernels — sometimes the fastest tool, especially for a quick "what's this thread doing".
  5. assert inside kernels — catches assumption violations at the offending line.

compute-sanitizer — the must-run check

# memcheck: out-of-bounds, misaligned access, leaks
compute-sanitizer --tool=memcheck ./my_program

# racecheck: shared-memory hazards (read-after-write without sync, etc.)
compute-sanitizer --tool=racecheck ./my_program

# synccheck: improper use of __syncthreads in divergent code
compute-sanitizer --tool=synccheck ./my_program

# initcheck: reads of uninitialized device memory
compute-sanitizer --tool=initcheck ./my_program

Run all four before any optimization pass. They catch the bugs that "work most of the time" — the ones that show up only on different hardware, or when warps schedule differently, or after a year in production.

printf in kernels — the simple option

__global__ void k(int* data, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  if (i < N && data[i] < 0) {
    printf("thread %d block %d: unexpected negative %d at i=%d\n",
           threadIdx.x, blockIdx.x, data[i], i);
  }
}

kernel printf is buffered; the host flushes the buffer on the next cudaDeviceSynchronize or when the buffer fills. If you don't see output, sync first.

assert in kernels

#include <cassert>

__global__ void k(int* a, int N) {
  int i = blockIdx.x * blockDim.x + threadIdx.x;
  assert(i < N);            // catches off-by-one at the line, with full diag
  assert(a[i] >= 0);         // catches data-invariant violations
  ...
}

// Compile with -G or default; on assertion fail you get:
// kernel.cu:5: void k(int*, int): block: [3,0,0], thread: [123,0,0] Assertion `i < N` failed.

Assertions in production: keep them. The cost is small; the bug-catching value is huge.

The classic pitfalls catalogue

Pitfall 1 · Forgetting to check for launch errors

Symptom: the kernel "runs" but later cudaMemcpy returns a cryptic error.

Cause: kernel launched with invalid config (too many threads/block, etc.) or crashed during execution; the error appears at the next API call that synchronizes.

Fix: always cudaGetLastError() after launches; cudaDeviceSynchronize() during debug builds.

Pitfall 2 · Passing a host pointer to a kernel

Symptom: kernel "starts," then a CUDA error happens at sync.

Cause: the kernel dereferenced a host address. Compiles silently; runs as a memory fault.

Fix: discipline. Prefix host pointers h_ and device pointers d_ — the visual mismatch in kernel<<<>>>(h_data) is hard to miss.

Pitfall 3 · __syncthreads inside divergent code

Symptom: kernel hangs (no output, never returns).

Cause: some threads skip the sync; the others wait forever.

Fix: every thread in the block must reach every __syncthreads. Restructure so the conditional work is between syncs, not around them.

Pitfall 4 · Missing the second __syncthreads in tiled kernels

Symptom: wrong results, not deterministic across runs.

Cause: the next iteration's loads overwrite the previous tile while some threads are still using it.

Fix: sync before AND after each tile's compute step. Mnemonic: "sync when you fill, sync when you finish."

Pitfall 5 · Race on shared memory without atomics or sync

Symptom: histograms with totals slightly off; reductions with the wrong sum.

Cause: two threads write the same shared-memory location with no atomic/sync between them.

Fix: compute-sanitizer --tool=racecheck catches this. Use atomics in shared, or restructure so each thread writes a distinct location.

Pitfall 6 · Uncoalesced access on the wrong dimension

Symptom: kernel is 5-30x slower than peak HBM bandwidth would predict.

Cause: threadIdx.x indexed into a dimension that isn't the innermost. Adjacent threads stride through memory instead of stepping by one element.

Fix: threadIdx.x should always vary the fastest-changing index. For row-major arrays, threadIdx.x = column; for column-major, threadIdx.x = row.

Pitfall 7 · The 32-way bank conflict on column access

Symptom: shared-memory accesses 5-30x slower than expected.

Cause: a 32-wide shared-memory tile accessed by column — every thread hits bank 0.

Fix: pad to 33. __shared__ float tile[32][33] instead of [32][32].

Pitfall 8 · Implicit dependence on warp size = 32

Symptom: code works today, but is brittle. (Warp size IS 32 on all NVIDIA hardware today, but old hand-rolled tricks that assume the previous "implicit warp sync" can break on Volta+ Independent Thread Scheduling.)

Fix: use __syncwarp(), __shfl_sync with explicit masks, cooperative groups. Don't rely on warp-synchronous behavior without an explicit primitive.

Pitfall 9 · Default stream serializing concurrent work

Symptom: created multiple streams, but the timeline shows them running serially.

Cause: a single operation on the default stream (cudaMemcpy without Async, or any default-stream kernel) synchronizes with all other streams.

Fix: use only non-default streams; compile with --default-stream=per-thread if mixing libraries that may use the default stream.

Pitfall 10 · cudaMemcpyAsync from pageable memory

Symptom: Async memcpy is no faster than blocking memcpy; overlap with compute doesn't materialize.

Cause: pageable memory can't be DMA'd directly. cudaMemcpyAsync falls back to a synchronous staging copy.

Fix: allocate host buffers with cudaMallocHost.

The senior engineer's mental model

The thing that distinguishes a senior CUDA engineer from a competent one isn't knowing more tricks. It's knowing, before starting, whether a kernel is going to be compute-bound or memory-bound on the target hardware, and architecting accordingly. The roofline lives in their head:

  • Know the peak HBM bandwidth and peak FLOPS of the target GPU.
  • Estimate arithmetic intensity from the algorithm (FLOPs per byte loaded).
  • Predict which ceiling you'll hit.
  • Architect to push toward the higher ceiling: if memory-bound, focus on coalescing/reuse/cp.async/TMA; if compute-bound, focus on Tensor Cores, ILP, low occupancy with more registers.
  • Profile, compare to prediction, find the gap, fix.

CUDA performance is mostly mechanical sympathy: the kernel runs fast when the silicon's strengths match what the code is asking for. Understanding the hardware in enough detail to predict that match is the durable skill.
