Concepts & Mechanisms — Storage-Aware AI Workload Benchmarks

The systematic background for the Storage-Aware AI Workload Benchmark Suite. Each card explains what the thing is, how it works, how we use it here, and — most importantly — why it matters in MLOps / AI performance. Read top-to-bottom once, then use the sidebar as a reference.

Methodology

What a benchmark actually is Phase 0

A benchmark is a controlled, repeatable measurement of a system under a defined workload, designed so that the only thing that changes between runs is the thing you're studying. It is not "run it once and read the number" — that's an anecdote. A real benchmark controls the environment (cache state, CPU governor, neighbour load), repeats N times, and reports a distribution.

How to build one

  1. Define the metric precisely (e.g. "cold wall-clock to load 16 GB of weights into HBM", not "loading speed").
  2. Control inputs: fix the model, the file layout, the dtype; pin versions; drop caches for cold runs.
  3. Warm up: discard the first run(s) so one-off costs (allocator init, JIT, autotuning) don't pollute the steady-state number.
  4. Repeat (≥5 for loads, ≥100 steps for training) and report mean ± σ/μ.
  5. Bracket with floors/ceilings so a number means something (see roofline below).
Why it matters: infra teams make six-figure hardware and config decisions off benchmark numbers. A benchmark you can't reproduce, or whose noise you don't quantify, leads to chasing ghosts — "optimising" something that was measurement jitter. Reproducibility is the entire value proposition.

σ/μ — the coefficient of variation his key metric

The ratio sigma/avg = standard deviation ÷ mean. It is dimensionless, so you can compare the run-to-run consistency of a 0.4 s load against a 90 s training step on the same scale.

σ/μInterpretationAction
< 5%tight, trustworthya 10% change later is real signal
5–20%noisy; small deltas suspectincrease N, isolate the box
> 20%the harness itself is unstablefix the measurement before trusting any trend

In common/stats.py this is summarize().cv, reported on every result in the suite.

Why it matters: trend detection across commits (did this PR regress loading?) is only possible if your noise floor is below the effect size you care about. σ/μ is that noise floor. We saw it spike to 63% at the num_workers=2 transition in Phase 2 — the loader was sometimes keeping up and sometimes not, a bistable regime — which is precisely the situation where a single-run number would lie to you.

Floors, ceilings & the roofline model

Before optimising software, measure the hardware limits that bound it:

  • Storage ceiling: max GB/s the device delivers (measure with fio). No loader can exceed it.
  • Compute ceiling: peak/achievable FLOPS (measure with gpu_peak.py). No kernel can exceed it.
  • Link ceiling: PCIe bandwidth for host↔device transfers.

The roofline model plots achievable performance vs arithmetic intensity (FLOPs per byte): a sloped region where you're memory-bound and a flat region where you're compute-bound. Knowing which side you're on tells you what to fix.

Why it matters: Phase 1 showed the naive loader already at ~5.6 GB/s on a device that tops out near there — so "better loader code" buys almost nothing; the win is a faster tier (GDS, NVMe RAID, WEKA). Measuring the floor first stopped us from optimising the wrong layer.

Cold vs warm & the Linux page cache

When you read a file, Linux keeps its pages in the page cache (free RAM). A second read is served from RAM at tens of GB/s — a warm read — and tells you nothing about storage. A cold read (cache evicted) hits the device and is what cold-start performance actually depends on.

How to go cold

# Privileged, system-wide:
sudo sh -c 'echo 3 > /proc/sys/vm/drop_caches'
# Unprivileged, surgical (just this file) — what we default to:
posix_fadvise(fd, 0, 0, POSIX_FADV_DONTNEED)

Implemented in common/io_utils.py::cold_clear().

Why it matters: the #1 way storage benchmarks lie is by accidentally measuring the page cache. In Phase 1 our cold reads were ~5.6 GB/s and warm ~15 GB/s — a 2.7× gap. If we hadn't evicted between runs, we'd have reported the loader as 3× faster than it is. Watch out for tmpfs: /tmp is often RAM — writing checkpoints there fakes your write throughput entirely.

Storage & I/O

NVMe & queue depth

NVMe (Non-Volatile Memory express) is the protocol modern SSDs use to talk to the CPU over PCIe, replacing the ancient AHCI/SATA path. Its superpower is parallelism: up to 64K queues × 64K commands each, versus SATA's single queue of 32. A consumer NVMe drive does ~5–7 GB/s sequential and ~1M random IOPS; a datacenter NVMe more.

Queue depth (iodepth)

A single thread issuing one read at a time leaves the device idle while it waits. Keeping many requests in flight (high queue depth) lets the SSD's internal parallelism (multiple NAND dies) work — that's why fio iodepth=16 numjobs=4 and our multi-threaded loader extract more bandwidth than one sequential reader.

Why it matters: this is exactly why Phase 1's Variant C beat the naive loader (6.75 vs 5.57 GB/s): 8 threads reading different regions raised the effective queue depth on the NVMe. On spinning disks the opposite holds (seeks kill you) — so the right concurrency is a property of the tier, which is why you benchmark it.

mmap — memory-mapped files

mmap() maps a file's bytes directly into your process's virtual address space. You get a pointer; reading through it triggers a page fault that the kernel satisfies by paging in just the bytes you touched — lazily, on demand, with no explicit read() call and no copy into a user buffer.

How it works

  1. mmap sets up page-table entries pointing at (not-yet-loaded) file pages.
  2. First access to a page → fault → kernel DMAs that page from storage into the page cache.
  3. Subsequent accesses are plain memory loads; pages are shared across processes that map the same file.
import numpy as np
mm = np.memmap("shard.bin", dtype=np.uint16, mode="r")  # nothing read yet
seq = mm[start:start+2048]   # THIS triggers the page-in of just those bytes
Why it matters: (1) you can "open" a 50 GB dataset instantly and read 2 KB sequences out of it without loading 50 GB into RAM — our TokenShardDataset does exactly this. (2) safetensors uses mmap so weights load zero-copy on the CPU side; the only real copy is the H2D transfer. (3) Multiple DataLoader workers mmapping the same shard share the page cache instead of each loading their own copy. The gotcha: open the memmap inside the worker (post-fork), not before — a mapped handle pickled across fork is a classic source of bugs.

fio & dd — storage micro-benchmarks

dd is the quick-and-dirty sequential read/write timer. fio (Flexible I/O tester) is the real instrument: it models access patterns (seq/random, block size, queue depth, parallel jobs, O_DIRECT) and reports bandwidth, IOPS and latency percentiles.

[seq-read-1m]
ioengine=libaio
direct=1        ; bypass the page cache → measure the device
bs=1M           ; block size
iodepth=16      ; keep 16 IOs in flight
numjobs=4       ; 4 parallel readers
rw=read

direct=1 is the key flag — without it you measure RAM. Used in tensor_loading/fs_bandwidth.py and seq-read-1m.fio.

Why it matters: fio gives you the storage ceiling in the access pattern your real workload uses. "My NVMe does 7 GB/s" (seq) and "my loader does random 4K reads" can be a 50× difference — fio lets you measure the pattern that actually applies.

safetensors — the industry weight format

A dead-simple binary layout: [8-byte little-endian header length] + [JSON header] + [raw tensor bytes]. The JSON maps each tensor name → dtype, shape, and byte offsets [begin, end) into the data block. Because offsets are explicit, you can mmap the file and slice any tensor with zero copy. (Parsed by hand in tensor_loading/loaders.py.)

Why it replaced pickle / torch.save

pickle (.pt/.bin)safetensors
Securityarbitrary code execution on load ⚠️pure data, no code
Loaddeserialize everything, single-threadedmmap, zero-copy, lazy, partial
Layoutopaque object graphexplicit offsets → parallel/GDS reads

Why it matters: Hugging Face, vLLM, SGLang and essentially the whole open-weights ecosystem default to safetensors because (1) you can load untrusted weights without RCE risk, and (2) the explicit offset table is what makes fast loaders — multi-threaded, GDS, partial-shard — even possible. The format is the enabler for everything in Phase 1.

Data movement

PCIe & the hidden bounce buffer

The GPU talks to the CPU/storage over PCIe. An RTX 5090 is PCIe 5.0 ×16 ≈ ~64 GB/s each way. Host→Device (H2D) copies traverse this link.

DMA engines can only transfer from page-locked (pinned) memory. If you copy from ordinary pageable host memory, the driver must first stage it through a hidden internal pinned buffer — an extra CPU-side copy that roughly halves effective bandwidth and serialises the transfer.

Why it matters: H2D is on the critical path of every weight load and every training batch. Understanding the link ceiling (~64 GB/s) vs storage (~6 GB/s) tells you that for cold loads, storage, not PCIe, is the wall — but for warm/cached loads PCIe becomes the wall, which is why warm numbers plateau in Phase 1.

Pinned (page-locked) memory & async DMA

Normal host allocations are pageable — the OS may swap them out, so their physical address isn't stable, so a DMA engine can't target them. Pinned memory (cudaHostAlloc / torch.empty(..., pin_memory=True)) is locked to physical RAM. Two consequences: the copy needs no bounce-buffer staging, and it can be asynchronous (non_blocking=True) — the CPU issues the DMA and moves on while it runs.

buf = torch.empty(shape, dtype=dt, pin_memory=True)   # page-locked
buf.copy_(cpu_tensor)                                 # fill staging
gpu_tensor.copy_(buf, non_blocking=True)              # true async DMA

Pinned allocation is expensive, so production code reuses a small pool of staging buffers (our Variant C double-buffers per thread). Pinning too much RAM starves the OS.

Why it matters: pinned + async is what lets you overlap reading the next tensor with copying the current one to the GPU (Phase 1 Variant C) and lets the DataLoader stage batches while the GPU computes (Phase 2, pin_memory=True bought a few % MFU). It's also the residual cost in async checkpointing (Phase 3): pinning the optimizer-state staging buffer is the next optimisation.

CUDA streams & overlap

A stream is an ordered queue of GPU work. Operations in different streams can run concurrently (subject to hardware: copy engines vs SM compute). The default stream serialises everything; using multiple streams lets copies and kernels overlap.

Why it matters: Variant C gives each loader thread its own stream so 8 H2D copies can be in flight and overlap the CPU-side reads. In NSight you literally see the copy bands interleave — that overlap is the speedup. Remove streams (--no-streams) and they serialise; the ablation makes the mechanism visible.

GPU HBM & the memory hierarchy

HBM (High Bandwidth Memory) is the GPU's on-package DRAM — stacked dies connected by a very wide bus. The RTX 5090 has 32 GB of GDDR7 at ~1.8 TB/s; datacenter parts (H100/B200) use HBM3/3e at 3–8 TB/s. This is one to two orders of magnitude faster than the PCIe link feeding it.

TierBandwidth (order)
GPU registers / SRAM10s of TB/s
GPU HBM / GDDR1–8 TB/s
PCIe 5.0 ×16~64 GB/s
Local NVMe~3–14 GB/s
Network / parallel FS~1–100+ GB/s (aggregate)
Why it matters: the whole storage problem exists because each step down this hierarchy is ~10–100× slower. Getting weights from NVMe (~6 GB/s) into HBM (1.8 TB/s) is the bottleneck; once resident, the GPU flies. Memory-bound kernels (LayerNorm, softmax) are limited by HBM bandwidth, not FLOPS — which is why MFU < 100% even on a perfect pipeline.

GPUDirect Storage (GDS), cuFile, kvikio, nvidia-fs

Normally, NVMe→GPU goes NVMe → CPU bounce buffer (page cache / pinned) → PCIe → HBM. GDS removes the CPU hop: a direct peer-to-peer DMA path NVMe → PCIe → GPU HBM. The CPU sets up the transfer but the bytes never land in host RAM.

  • cuFile — the C API for GDS reads/writes.
  • kvikio — RAPIDS Python bindings over cuFile (Variant D uses it).
  • nvidia-fs — the kernel driver enabling the P2P path. Without it, kvikio falls back to a "compat" mode (POSIX read + copy) that works but isn't true GDS.
from kvikio import CuFile
with CuFile(path, "r") as f:
    fut = f.pread(gpu_tensor, nbytes, file_offset=off)  # NVMe → HBM, bypass CPU
    fut.get()
Why it matters: GDS saves the host-memory copy and frees CPU/DRAM bandwidth — decisive at scale where you're loading TB of weights/data and the host can't keep up, or where you simply don't have the RAM to bounce through. It needs the right hardware/driver/FS path (local NVMe, or a GDS-aware FS like WEKA or Lustre+lnet), which is why it's the "stretch" variant and why you investigate what your cluster supports before assuming it.

NUMA — Non-Uniform Memory Access

Multi-socket servers split RAM into nodes, each "local" to one CPU socket. Accessing your own node's memory is fast; reaching across the interconnect to another node's memory is slower. PCIe devices (GPUs, NICs, NVMe) are also attached to a specific socket.

Why it matters: if your DataLoader workers or pinned buffers land on a different NUMA node than the GPU's PCIe root, every H2D copy and every read crosses the interconnect — adding latency and stealing cross-socket bandwidth, and inflating σ/μ as placement varies run to run. On big hosts you pin processes with numactl / --cpu-bind so workers, their memory, and the GPU share a socket. It's a common hidden cause of "why is node 3 slower than node 1" in scaling studies.

Compute & utilization

MFU & the 6N FLOP rule

MFU (Model FLOPs Utilization) = achieved FLOPS ÷ peak FLOPS. It answers "what fraction of the GPU's compute am I actually using?". The achieved side uses the standard estimate:

flops_per_token = 6 * N + 12 * L * S * H      # N=params, L=layers, S=seq, H=hidden
flops_per_step  = flops_per_token * batch * S
TFLOPS          = flops_per_step / step_time / 1e12
MFU             = TFLOPS / peak_TFLOPS

The 6N comes from: 2 FLOPs per param for the forward multiply-add, ~4 for the backward (grad w.r.t. input + weight). The +12·L·S·H term captures the sequence-quadratic attention matmuls that 6N misses.

Choosing "peak"

Peak is accumulate-mode dependent. RTX 5090 BF16 ≈ 209.5 TFLOPS (FP32 accumulate) / ~419 (FP16 accumulate). We instead use the measured achievable roofline (gpu_peak.py → ~219 TFLOPS here) as the denominator — the honest "fraction of what this box can actually deliver". A spec number that's too low yields the nonsensical >100% MFU.

Why it matters: MFU is the headline efficiency number for any training run. Low MFU + an idle GPU in NSight = you're input-bound, and the fix is the data pipeline (Phase 2), not a bigger GPU. Real dense-transformer training lands ~40–55% MFU; our GEMM-heavy proxy hit ~98% (it's almost pure matmul), which is itself the finding that data loading wasn't the local bottleneck.

BF16 / FP8 / FP4 — low precision on Blackwell

Lower-precision formats halve memory and roughly double tensor-core throughput per step down. BF16 (8-bit exponent, 7-bit mantissa) is the training default — same dynamic range as FP32, less precision, no loss scaling needed. FP8 (E4M3/E5M2) is used for inference and increasingly training on Hopper/Blackwell. FP4 (Blackwell) pushes inference throughput further (the 5090's headline "AI TOPS" are FP4-with-sparsity numbers).

Why it matters: the format sets both your memory footprint (a 16 GB BF16 model is 8 GB in FP8) — directly the Phase 1 load size — and your peak FLOPS denominator for MFU. Quoting MFU without stating the dtype/accumulate mode is meaningless. The 5090's FP4/FP8 capability is why a 32 GB consumer card can serve surprisingly large models.

Pipelines

PyTorch DataLoader internals

KnobWhat it does
num_workerssubprocesses doing read+collate in parallel
prefetch_factorbatches each worker stages ahead of demand
pin_memorycollate into page-locked buffers → async H2D
persistent_workerskeep workers alive across epochs (skip respawn)

num_workers is the most-misset PyTorch parameter. Too few → the CPU can't produce batches fast enough and the GPU idles (high data-wait). Too many → workers contend for CPU cores and the shared FS, and it gets worse. The sweet spot is workload- and host-specific — so you sweep it (Phase 2).

Why it matters: our sweep found MFU climbing 11% → 65% as workers went 1 → 8, then flat to 16 (diminishing returns) on a 12-core host. That heatmap is the artifact MLOps teams paste into "recommended settings" docs — it directly converts to GPU-hours saved across every training job on the cluster.

Checkpointing: sync, async, and DCP

Checkpointing periodically writes model + optimizer state so a crashed/preempted job can resume. The cost has two parts: the G2C copy (GPU→CPU snapshot) and the disk write.

  • sync: torch.save inline — training freezes for copy + write.
  • async: snapshot to pinned CPU, hand the write to a background thread; training resumes after just the copy.
  • DCP (torch.distributed.checkpoint): each rank writes its shard in parallel — aggregate bandwidth scales with ranks.

The right metric is the step-time delta on the checkpoint step, plus the often-ignored background-write interference on the following steps.

Why it matters: at scale you checkpoint every N steps for the whole run; a multi-second sync stall × thousands of checkpoints × thousands of GPUs is enormous waste. Phase 3 measured async cutting the stall ~66% (1310→440 ms) but adding ~89% slowdown to the next steps — so async spreads the cost rather than erasing it, and the residual G2C copy is the next target. MTTR (mean time to recovery, the resume side) often dominates the true cost — that's resume_bench.py.

Filesystems at scale

Lustre — OSTs, OSS, MDS, striping

Lustre is the classic HPC parallel filesystem. A file's data is spread (striped) across many OSTs (Object Storage Targets, backed by OSS servers); metadata (names, permissions, layout) lives on the MDS (Metadata Server). Aggregate bandwidth = sum over OSTs, so a widely-striped file read by many clients can hit hundreds of GB/s.

lfs df -h /scratch              # per-OST capacity/usage
lfs getstripe <file>            # how is it striped?
lfs setstripe -c -1 <dir>       # stripe new files across ALL OSTs
Why it matters: the Phase 2.5 scaling wall lives here. Add nodes all reading one narrowly-striped dataset and you saturate a few OSTs while others idle, or you hammer the single MDS with metadata ops (open/stat) — per-node throughput drops. Distributed checkpointing (Phase 3.3) exploits the flip side: wide striping + parallel rank writes saturate aggregate OST bandwidth. Knowing to check lfs getstripe is the difference between "the FS is slow" and "the file is striped ×1".

WEKA

WEKA is a modern NVMe-native parallel/distributed filesystem popular in AI clusters (including at NVIDIA scale). It pools NVMe across nodes into a single namespace, is built for massive small-file + metadata throughput (the pain point for data loading), and supports GPUDirect Storage so reads can land straight in HBM.

Why it matters: the failure mode Lustre hits on metadata-heavy AI workloads (millions of small shard reads, frequent opens) is what NVMe-native systems like WEKA are engineered to avoid. When you frame a scaling finding, naming the tier (Lustre vs WEKA vs local NVMe) is essential — the contention behaviour is completely different, and "what FS tier" is the first question a storage team will ask.

Systems & tooling

vLLM & SGLang — inference engines

Both are high-throughput LLM serving engines exposing an OpenAI-compatible API.

  • vLLM: introduced PagedAttention — managing the KV cache in fixed-size pages like virtual memory, eliminating fragmentation and enabling high batch concurrency. The de-facto throughput baseline.
  • SGLang: adds RadixAttention (prefix-sharing of KV cache via a radix tree) — big wins when many requests share prompts (few-shot, agents). Also a fast structured-output / programming frontend.
# vLLM
python -m vllm.entrypoints.openai.api_server --model Qwen/Qwen3-8B --dtype bfloat16
# SGLang
python -m sglang.launch_server --model-path Qwen/Qwen3-8B

We benchmark their cold-start (launch → ready → first token) in run_vllm.sh / run_sglang.sh, because that startup is dominated by the same weight-loading path as Phase 1.

Why it matters: cold-start time = how fast an autoscaler can bring up a new replica under load. It's mostly weight loading, so the Phase 1 loader work transfers directly. Pinning engine versions matters: SGLang's loader changed between v0.5.6 (single-threaded) and v0.5.10 (multi-threaded) — exactly the kind of version-sensitive trend that motivates the σ/μ + version-pinning discipline.

Proxy models (e.g. Nemotron-5-8B-proxy)

A proxy model keeps realistic tensor shapes (hidden size, head count, vocab — so kernel sizes and memory traffic match production) but strips the depth (4 layers instead of 32–64). A step is short, so you collect hundreds of samples in minutes. You're not training to convergence — you're stress-testing the infrastructure around the model (data path, checkpoint path, collective comms).

Why it matters: you can't iterate on storage/IO benchmarks if every run needs a real 8B forward/backward. The proxy gives production-representative IO and memory behaviour at a fraction of the wall-clock and GPU cost — it's the standard tool for isolating an infra bottleneck from model compute. Our proxy_model.py (hidden=4096, 4 layers, ~945M params) is exactly this.

Slurm — the cluster scheduler

Slurm allocates nodes/GPUs and launches jobs. sbatch script.sh queues a batch job (with #SBATCH resource directives); srun launches the parallel tasks across the allocation and sets the env vars (SLURM_PROCID, SLURM_NTASKS, SLURM_LOCALID…) that common/slurm_utils.py reads to bootstrap torch.distributed.

Why it matters: benchmarking from sbatch (not an interactive shell) from day one means your numbers are reproducible by anyone with the script, the scheduler captures the exact topology, and you measure the same path production training uses (prolog, cgroups, NUMA binding, fabric). "Works in my interactive session" is not a benchmark.

NVTX, NSight Systems (nsys) & NSight Compute (ncu)

NVTX ranges are named, coloured annotations you push onto the timeline (common/nvtx_helpers.py). They turn an undifferentiated wall of kernels into a readable story.

ToolScopeUse it to…
nsyswhole-timeline (CPU+GPU, copies, NVTX) find where time goes; spot GPU idle gaps, missing overlap, sync bubbles
ncusingle-kernel deep dive find why a kernel is slow (occupancy, memory vs compute bound, stalls)
nsys profile -t cuda,nvtx,osrt -o trace python script.py   # timeline
ncu --nvtx --set full --launch-count 20 -o k python script.py  # kernels (SLOW — scope it!)
Why it matters: profiling is how you move from "it feels slow" to a falsifiable bottleneck. nsys answers the storage/pipeline questions in this project (is the GPU idle waiting for data? does the async write overlap the next step?); ncu answers the kernel-efficiency questions. ncu replays each kernel and is brutally slow — always scope it with --launch-count / --nvtx-include.

CI for benchmarks

Continuous Integration usually means "run the tests on every commit". For an infra team it also means run the benchmarks on every commit and fail/flag if a key metric regresses beyond the noise floor. The .gitlab-ci.yml stub shows the stages: lint → smoke (tiny model, asserts the pipeline runs) → benchmark (on a GPU runner) → publish dashboard.

Why it matters: performance rots silently — a dependency bump, a kernel regression, a config drift, and loading is 20% slower with no failing test to catch it. Benchmark CI catches it at the commit that caused it. This only works if your σ/μ is below your regression threshold (back to the coefficient of variation) — which is why "measure the noise" and "detect regressions" are the same discipline.
