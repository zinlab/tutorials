00

Orientation: the mental model before the details

A SQL query says what you want; it says nothing about how to get it. Between those two facts sits one of the most consequential pieces of software in any data system — the query optimizer — and after it, the execution engine that actually moves the bytes. This manual takes you from the top of that stack down to the metal, and then off the CPU entirely.

Here is the whole pipeline in one breath, so every later section has a place to attach. A query string is parsed into a syntax tree, bound against the catalog (names become real tables and typed columns), and lowered into a logical plan — a tree of relational-algebra operators that describes the answer set without committing to an algorithm. The optimizer then does two jobs that are worth separating in your head: it rewrites the logical plan into a cheaper but equivalent logical plan (rule-based), and it chooses physical algorithms and an evaluation order for that plan (cost-based). The result is a physical plan: a specific tree of operators — this hash join, that index scan, this order of tables — annotated with everything the engine needs. The execution engine runs it.

Three ideas are worth fixing now because they recur at every level:

  1. The search space is enormous and the cost surface is bumpy. For a query joining a dozen tables there are billions of valid evaluation orders, each with wildly different runtimes. The optimizer is a search procedure over that space, steered by estimates.
  2. Everything hinges on cardinality estimation. "How many rows will this operator emit?" is the question the entire cost model is built on, it is provably hard to answer well, and its errors compound multiplicatively up a join tree. Most catastrophically slow queries are cardinality-estimation failures wearing a cost-model costume.
  3. The optimal plan is hardware-relative. The "best" join algorithm on a tuple-at-a-time CPU engine, a vectorized columnar engine, and a GPU are three different choices. A cost model encodes assumptions about the machine; change the machine and you must change the model. This is the hinge the whole GPU section turns on.
The thesis of this manual

An optimizer is a bet placed under uncertainty about a machine. On the CPU the dominant cost is usually random memory access and per-tuple overhead. On the GPU the machine changes so completely — thousands of lanes, terabyte-per-second on-package memory, but a narrow PCIe straw to feed it — that the bets change too. The same operator (a hash join, a group-by) survives the move, but which implementation wins, where intermediates live, and what the optimizer should even be optimizing for are all rewritten. cuDF and the engines built on it are where you watch this happen.

One more orienting note on vocabulary, because the word "optimizer" is overloaded. In a full database (PostgreSQL, SQL Server, Spark) the optimizer is a distinct, heavyweight component that reorders joins and chooses algorithms by cost. In a dataframe library like cuDF, there is no such component by default: when you write df.merge(other) you have already chosen the join and its order — you are the optimizer. That distinction turns out to be one of the most important things to understand about GPU ETL, and §11 is devoted to it. Keep it in your pocket until then.

Part I — The Big Picture
01

From SQL text to running code

Before anything can be optimized, the query has to become a data structure. Four stages turn a string into an executable plan, and each one quietly constrains what the optimizer can later do.

We'll carry one query through the entire manual. It's deliberately mundane — a star-schema rollup that any analytics workload produces by the thousand:

SQL · the running example
SELECT c.region, SUM(l.amount) AS revenue
FROM orders o
JOIN customers c  ON o.customer_id = c.id
JOIN line_items l ON l.order_id = o.id
WHERE o.order_date >= DATE '2024-01-01'
  AND c.segment = 'enterprise'
GROUP BY c.region
ORDER BY revenue DESC;

Stage 1 — Parsing (text → syntax tree)

A lexer tokenizes the bytes into keywords, identifiers, literals, and operators; a grammar (typically LALR, as in PostgreSQL's Bison grammar) assembles those tokens into an abstract syntax tree. At this point the tree is purely syntactic: orders is just an identifier token, SUM is a function name, and nobody has checked whether any of these things exist. Parse errors ("syntax error near GROUP") come from here. The output is a faithful structural transcription of what you typed, nothing more.

Stage 2 — Binding / semantic analysis (names → typed objects)

This is where the query meets reality. The binder (also called the analyzer or resolver) walks the AST and resolves every identifier against the catalog — the system tables describing schemas, tables, columns, types, functions, and constraints. Several things happen, and each can fail:

  • Name resolution. c.region becomes a reference to a specific column of a specific table with a specific type. Ambiguity ("column id exists in two tables") is resolved or rejected here.
  • Type checking and coercion. o.order_date >= DATE '2024-01-01' requires both sides to be comparable; implicit casts are inserted, or an error is raised.
  • View and macro expansion. If orders were a view, its definition is inlined now, so the optimizer sees through it.
  • Aggregate/grouping validation. The rule that every non-aggregated SELECT column must appear in GROUP BY is enforced here.

The product is a bound tree in which every leaf is a real, typed catalog object. The optimizer can now reason about it because it knows the shape and type of everything.

Stage 3 — Logical planning (bound tree → relational algebra)

The bound tree is lowered into a logical plan: a tree of relational-algebra operators (§02). This is the optimizer's working representation — an algebra precisely because algebra comes with equivalence laws that let you transform a plan into a cheaper equivalent. Crucially the logical plan specifies operators, not algorithms: it says "join orders and customers," not "hash-join them." That freedom is exactly what the optimizer will spend.

Stage 4 — Optimization → physical plan

The two-part job from the orientation: rewrite the logical plan (rules, §03), then enumerate physical alternatives and pick the cheapest by cost (§04–07). The output is a physical plan the engine can execute directly.

Mechanism · why the split is load-bearing

Separating logical (what) from physical (how) is the architectural decision that makes optimization tractable. Logical equivalence rules can be applied without worrying about algorithms; physical algorithm choice can be made on a fixed logical shape. Volcano/Cascades-style optimizers (§06) formalize this split into two distinct rule sets — transformation rules that rewrite logical→logical, and implementation rules that map logical→physical. Hold this two-rule-set idea; it's the backbone of every modern optimizer.

02

Relational algebra & the logical plan

Relational algebra is a small set of operators that consume relations (sets/bags of tuples) and produce relations. Because operators compose and obey algebraic laws, a query plan is literally an expression in this algebra — and optimization is expression rewriting under those laws.

Equivalence rules: the algebra that licenses rewriting

The reason the plan above can be improved without changing its result is that relational-algebra operators obey identities. A handful do most of the work:

  • Join commutativity: R ⋈ S ≡ S ⋈ R. Either side can build, either can probe.
  • Join associativity: (R ⋈ S) ⋈ T ≡ R ⋈ (S ⋈ T). This is what creates the join-ordering search space (§06).
  • Selection pushdown: σp(R ⋈ S) ≡ σp(R) ⋈ S when p only references R. Filter early, join less.
  • Projection pushdown: drop columns as soon as nothing above needs them — vital for columnar and GPU engines where unused columns are pure wasted bandwidth.
  • Selection splitting: σp∧q ≡ σp(σq), so conjuncts can be pushed to different relations independently.
Mechanism · plans are DAGs, not just trees

When the same sub-expression appears twice (common in views, CTEs, and self-joins), a smart optimizer represents the plan as a DAG and computes the shared sub-result once — common subexpression elimination. This matters disproportionately on GPUs: a materialized intermediate column is expensive to recompute and expensive to keep resident, so sharing it is a real win. The Cascades "memo" (§06) is fundamentally a DAG of equivalence classes built for exactly this reason.

Relational algebra gives the optimizer a closed world: a finite operator set, a finite rule set, and a guarantee that any rule application preserves correctness. Everything from here is about which rewrites to apply and which physical algorithms to choose — and how to decide without running the query.

Part II — The Optimizer Core
03

Rule-based transformation

Rule-based (heuristic) optimization applies algebraic rewrites that are almost always beneficial, so they can be fired without consulting cost. These rewrites shrink the plan and the data flowing through it before the expensive cost-based search even begins. In a modern optimizer they're expressed as pattern→replacement rules over the plan tree (Spark's Catalyst is the canonical example — its entire optimizer is a library of such rules applied to fixed point).

The workhorse rewrites

  • Predicate pushdown. Move σ as close to the scans as possible. c.segment = 'enterprise' belongs on the customers scan; o.order_date >= … belongs on the orders scan. Pushed into the scan itself, it can become an index range or a skipped row group (Parquet/ORC predicate pushdown), so the filtered rows are never even read.
  • Projection pushdown / column pruning. Read only the columns some operator above actually needs. Our query never uses customers.phone, so a columnar engine should never touch that column file. On the GPU this is bandwidth you literally do not pay.
  • Constant folding & predicate simplification. Evaluate constant sub-expressions once (DATE '2024-01-01' is parsed to an integer day), simplify x AND TRUE, fold 1=1, normalize BETWEEN into two comparisons.
  • Predicate inference (transitive closure). From o.customer_id = c.id and c.id = 42, infer o.customer_id = 42 and push it to orders too. Generating new, cheaper predicates from existing equalities is one of the highest-leverage rewrites in real systems.
  • Limit / TopN pushdown. ORDER BY … LIMIT k becomes a bounded heap (Top-k) rather than a full sort, and the limit can sometimes pass through joins.
  • Join elimination. If a join to a table contributes no columns and the foreign key is enforced (so every row matches exactly once), drop the join entirely.

Subquery decorrelation — the one that earns its own subsection

A correlated subquery (one that references the outer row) is, evaluated literally, a nested loop that re-runs the inner query per outer row — quadratic. Decorrelation (a.k.a. unnesting) rewrites it into a join or aggregate that the optimizer can then cost and reorder normally.

SQL · correlated → decorrelated
-- correlated: inner query re-runs for every order (O(n·m))
SELECT * FROM orders o
WHERE o.amount > (SELECT AVG(amount) FROM orders x
                  WHERE x.customer_id = o.customer_id);

-- decorrelated: one grouped aggregate + one join (O(n))
WITH avg_per_cust AS (
  SELECT customer_id, AVG(amount) AS a FROM orders GROUP BY customer_id)
SELECT o.* FROM orders o
JOIN avg_per_cust ac ON ac.customer_id = o.customer_id
WHERE o.amount > ac.a;

This transformation is the difference between a query that returns in milliseconds and one that never returns. It matters enormously for GPU engines: the nested-loop form has data-dependent control flow that maps terribly to SIMT lanes, while the decorrelated form is a plain group-by + hash join — both operators GPUs execute beautifully.

Applying the always-good rules to our running query produces the plan everyone actually wants: filters sit on the scans, projections are narrow, and only the join order remains to be decided.

Caveat · not every rule is unconditionally good

A few "rewrites" are really trade-offs and belong in the cost-based search, not the always-fire list — e.g. pushing an aggregate below a join (eager aggregation) helps only if it shrinks the join input enough to pay for the extra grouping. Modern optimizers therefore blur the rule/cost boundary: Cascades expresses even heuristic rewrites as rules but lets cost decide whether to keep the result.

04

Cardinality & selectivity estimation

If you learn one thing about optimizers, learn this section. Every cost number downstream is computed from estimated row counts, those estimates rest on statistical assumptions that are routinely violated, and the errors compound. The single most reliable predictor of a bad plan is a bad cardinality estimate.

Selectivity is the fraction of rows an operator passes: a predicate with selectivity 0.02 emits 2% of its input. Cardinality is the absolute row count out. The optimizer estimates selectivity from statistics collected over the data (PostgreSQL's ANALYZE, the catalog's pg_statistic), because it cannot run the query to find out.

What the statistics actually contain

For each column, a typical catalog stores a small fixed-size summary:

  • n_distinct — number of distinct values (the NDV).
  • null_fraction — fraction of NULLs.
  • Most-Common-Values (MCV) list — the top-k values and their exact frequencies. Handles skew: a few values that dominate.
  • Histogram — for the non-MCV remainder, an equi-depth (height-balanced) histogram: bucket boundaries chosen so each bucket holds roughly the same number of rows, not the same value range. Equi-depth adapts resolution to data density, which is why it beats equi-width for skewed data.

From statistics to a selectivity number

The estimator combines these pieces with three standard assumptions. For an equality predicate col = v:

selectivity · equality predicate
# if v is in the MCV list, we know its frequency exactly:
sel(col = v)  =  mcv_freq[v]

# otherwise assume the non-MCV mass is spread uniformly
# over the non-MCV distinct values (the UNIFORMITY assumption):
sel(col = v)  =  (1 - Σ mcv_freq - null_frac) / (n_distinct - n_mcv)

For a range predicate col < v, the estimator walks the histogram: buckets entirely below v contribute their full row fraction, and the straddling bucket is linearly interpolated. With a B-bucket equi-depth histogram, each fully-below bucket contributes exactly 1/B.

Combining predicates — where independence bites

Our WHERE has two conditions on customers… well, one each on two tables, but consider a single table with WHERE city='SF' AND state='CA'. The estimator multiplies:

sel(p ∧ q) = sel(p) · sel(q) ← the INDEPENDENCE assumption

This is exactly wrong when columns are correlated. city='SF' already implies state='CA', so multiplying their selectivities underestimates the result by a large factor — the optimizer thinks far fewer rows survive than actually do. Underestimation is the dangerous direction: it leads the optimizer to pick a tiny build side or a nested-loop join that then explodes. The fix is multi-column / extended statistics: store joint distributions or functional-dependency hints for correlated column groups (PostgreSQL CREATE STATISTICS, SQL Server multi-column stats).

The compounding catastrophe

Here is why all of this is fragile. Estimation errors don't add — they multiply up the join tree, because each join's input cardinality is the previous join's (already-wrong) output. A 3× error at each of four joins is an 80× error at the top.

Why this section is the hinge

Cost is a deterministic function of cardinality, so a good cost model fed bad cardinalities still picks bad plans. This is why the frontier moved here: sampling (estimate selectivity by running the predicate on a small random sample at plan time), sketches (HLL, count-min), learned estimators (ML models trained on the data/workload to predict cardinality directly), and adaptive execution (§10 — stop guessing, measure the real cardinality mid-query and re-plan). Every one of these is a reaction to FIG 04C. And on the GPU, where re-running a misjudged operator means re-materializing a multi-gigabyte column, the cost of a cardinality miss is even higher — so GPU engines lean hard on runtime adaptivity rather than upfront estimation.

05

The cost model

A cost model turns a candidate physical plan into a single number that proxies for runtime, so the optimizer can compare alternatives without executing them. The number is fictional currency — its absolute value is meaningless; only the ordering it imposes on plans matters.

What gets costed

Classic models sum a few resource terms, each weighted into the common currency:

  • I/O — page reads, split into sequential (cheap, prefetchable) and random (expensive seeks). This dominated the disk era.
  • CPU — per-tuple work: comparisons, hash computations, expression evaluation, function calls.
  • Memory — whether a hash table or sort fits in the working-memory budget; if not, a spill penalty (extra I/O for partitioning to disk).
  • Network — in distributed engines, bytes shuffled between nodes. Often the dominant term (§16), and the analog that takes over completely on multi-GPU.

PostgreSQL makes the weights concrete and tunable — every cost is relative to one sequential page fetch defined as 1.0:

ConstantDefaultMeaning
seq_page_cost1.0read one page sequentially (the unit)
random_page_cost4.0read one page via random seek — 4× a sequential read
cpu_tuple_cost0.01process one row
cpu_index_tuple_cost0.005process one index entry
cpu_operator_cost0.0025evaluate one operator/function

A sequential scan of a 10,000-page, 1,000,000-row table costs roughly 10000·1.0 + 1000000·0.01 = 20,000. The random_page_cost/seq_page_cost ratio of 4:1 is precisely what makes the optimizer prefer a sequential scan over an index scan once a predicate matches more than a small fraction of rows — the index's random I/O stops paying off. (On SSDs people lower it to ~1.1; the constant is a hardware assumption.)

Mechanism · costs compose bottom-up

A plan's cost is computed recursively: cost(op) = cost(children) + local_work(op), where local_work is a function of the operator's estimated input and output cardinalities (§04). This is the join between the two halves of the optimizer — the cost model is just arithmetic; cardinality estimation supplies the inputs. Optimizers also track two costs separately: start-up cost (work before the first row emerges — a sort must consume everything first) and total cost. A LIMIT query weights start-up cost heavily; a full scan weights total cost.

Where the classical model breaks

These models were calibrated for tuple-at-a-time execution on spinning disks. Two shifts broke them: (1) memory got large and fast, so I/O often isn't the bottleneck — cache misses and per-tuple interpreter overhead are; and (2) vectorized and compiled engines (§08) cut per-tuple CPU cost by 10–50×, so a cost model that still charges cpu_tuple_cost per row mis-ranks plans. This is the first crack that the GPU will blow wide open: on a GPU the per-row CPU cost is effectively zero (thousands of lanes) but data movement costs dominate — a cost model has to be rebuilt around bandwidth and residency, not page fetches.

06

Plan enumeration & join ordering

Join associativity and commutativity (§02) mean the same query has astronomically many valid evaluation orders, each with a different cost. Finding the cheapest is the optimizer's hardest search problem, and it is NP-hard in general. This section is about how the search is actually structured.

The size of the haystack

For n relations, the number of distinct join trees is:

Tree shapeCountn=4n=10
Left-deep onlyn!243,628,800
Bushy (all shapes)(2(n−1))! / (n−1)!120~17.6 billion

You cannot enumerate 17 billion plans. The art is searching this space without visiting most of it.

Strategy 1 — System R bottom-up dynamic programming

The 1979 System R optimizer's insight, still the core of most cost-based optimizers: the cheapest plan for a set of relations can be built from the cheapest plans for its subsets. Enumerate subsets of relations in increasing size; for each subset, find the best way to split it into two already-solved sub-subsets and join them; memoize the winner. The principle of optimality makes this exact.

pseudocode · DP join enumeration (System R / DPsize)
for each single relation R: bestPlan[{R}] = best access path for R
for size = 2 .. n:
  for each subset S of relations with |S| = size:
    for each split of S into non-empty (S1, S2):
       # only if S1 and S2 are connected in the join graph (no cross products)
       p = makeJoin(bestPlan[S1], bestPlan[S2])
       if cost(p) < cost(bestPlan[S]): bestPlan[S] = p
return bestPlan[{all relations}]

Two refinements make this practical:

  • Join-graph pruning. Only join subsets that share a predicate — skip cross products. The state-of-the-art enumerator, DPccp, enumerates exactly the connected subgraph–complement pairs of the join graph in the correct order, which is provably the minimal DP work. (DPsize and DPsub are simpler but visit more.)
  • Interesting orders. A plan that produces output already sorted on a useful key (because it used a merge join or an ordered index) may be more valuable than a cheaper plan that doesn't — because it saves a later sort. So the DP keeps, for each subset, the best plan per relevant physical property (sort order, and in distributed/GPU settings, partitioning), not just the single cheapest. This is the seed of the "physical properties" idea that §07 formalizes.

Strategy 2 — Top-down transformation (Volcano / Cascades)

The other dominant architecture — Cascades, used by SQL Server, Greenplum/Apache ORCA, CockroachDB, and Apache Calcite's Volcano planner — searches top-down using a goal-directed, memoized structure called the memo.

  • The memo is a set of groups; each group is an equivalence class of expressions producing the same logical output (e.g. "the result of joining {R,S,T}"). Within a group are group expressions (concrete operators whose children are other groups) — so the memo is a compact DAG that shares sub-results across the whole search.
  • Optimization runs as a stack of tasks (optimize group, explore expression, apply rule, optimize inputs). Transformation rules rewrite logical→logical (join reorder); implementation rules map logical→physical (join → hash join).
  • Branch-and-bound pruning: each group is optimized against a cost upper bound; any partial plan exceeding it is abandoned before its inputs are even optimized. This is what makes top-down search tractable — it never fully expands provably-losing branches.
Mechanism · DP vs Cascades, in one line each

Bottom-up DP builds every winning sub-plan once and combines upward — exhaustive within its space, naturally exact. Top-down Cascades pursues a goal and prunes with bounds — extensible (new operators = new rules) and lazy. They explore the same space; the trade is exhaustiveness-and-simplicity vs extensibility-and-pruning. Most new systems pick Cascades because adding a rule is how you teach the optimizer about a new physical operator — including, crucially, a GPU operator or a data-transfer operator.

Strategy 3 — Giving up gracefully (randomized search)

Past ~12 relations even DP gets expensive. PostgreSQL switches to GEQO, a genetic algorithm, above geqo_threshold (default 12) — it evolves a population of join orders rather than enumerating them, trading optimality for bounded planning time. Other systems use iterative improvement or simulated annealing. The lesson: for large joins, even the optimizer admits it's making an educated guess.

07

Physical operator selection

With an order chosen, the optimizer picks a concrete algorithm for each logical operator. These algorithms are the vocabulary of the execution engine, and — this is the point the GPU sections build on — the same handful reappears at every level of the stack; only their relative costs change.

The hash join deserves a close look because it is the operator GPUs care about most. It has two phases:

When the build side doesn't fit in the memory budget, the engine falls back to a grace / partitioned hash join: hash-partition both inputs into buckets small enough to fit, then hash-join bucket-by-bucket (recursing if a bucket is still too big). Hybrid hash join keeps the first partition resident to avoid spilling it. Partitioning to fit a memory budget is the same technique GPUs use to fit a hash table in fast on-chip memory — remember it for §14.

Aggregation

Two strategies, mirroring the join choice: hash aggregation (one hash-table slot per group key, update in place — the default, order-agnostic) versus sort-based aggregation (sort by group key, then a single linear pass reduces each run — preferred when the input is already sorted, or when groups are too numerous to fit a hash table). The same hash-vs-sort dichotomy governs GPU group-by (§14), where it becomes a contention-vs-bandwidth trade.

Access paths

  • Sequential scan — read every page; best when the predicate is unselective.
  • Index scan — descend a B-tree, fetch matching rows by pointer (random I/O); best when very selective.
  • Index-only scan — answer entirely from the index without touching the heap (when the index covers all needed columns).
  • Bitmap scan — build a bitmap of matching row locations from one or more indexes, then fetch in physical order to convert random I/O back into sequential. The optimizer's hedge between the two extremes.
Physical properties — the idea that survives all the way to the GPU

The optimizer doesn't just track cost; it tracks the physical properties each plan delivers — chiefly sort order and (in parallel systems) data partitioning. A plan that delivers data already sorted or already partitioned on the right key can satisfy a downstream operator for free. This is why a slightly-more-expensive merge join can beat a hash join: it hands the next operator a useful sort order. Carry this forward: in distributed and multi-GPU execution (§16), partitioning becomes the dominant physical property, because matching it avoids a network shuffle — the single most expensive thing a distributed query can do.

Part III — Where Optimization Meets the Hardware
08

Execution models: Volcano to compiled

The optimizer hands the engine a physical plan. How the engine walks that plan — one tuple at a time, a vector at a time, or as compiled machine code — changes per-row cost by more than an order of magnitude, which is why this is where the CPU story and the GPU story finally diverge.

The Volcano / iterator model — tuple-at-a-time pull

The classical model (Graefe, 1990) gives every operator the same interface: open(), next(), close(). Calling next() on the root pulls one tuple, which recursively pulls from children. A query is a tree of these iterators; data flows up one tuple per call.

the iterator interface, conceptually
class Filter(Operator):
  def next(self):
    while True:
      row = self.child.next()           # pull one row from below
      if row is None: return None     # exhausted
      if self.predicate(row): return row  # emit or loop

Why it was loved: operators compose freely, memory use is tiny (one tuple in flight), and it pipelines — a row flows from scan to filter to join without materializing intermediate tables. Why it became a bottleneck: one virtual function call per operator per tuple. For a billion rows through five operators that's five billion indirect calls — branch mispredictions, no instruction-cache locality, and no chance for the compiler to vectorize. The actual useful work (a comparison) is dwarfed by interpreter overhead.

Vectorized execution — vector-at-a-time

MonetDB/X100 (Boncz, Zukowski, Nes — CIDR 2005) kept the iterator structure but changed the unit: each next() returns a vector of ~1,000–2,048 values from a single column, not one tuple. The per-call overhead is now amortized over a thousand rows; the inner loop is a tight, branch-free, cache-resident loop over a typed array that the compiler auto-vectorizes into SIMD. This is the model behind DuckDB, Velox, ClickHouse, and Vectorwise — and it is the conceptual stepping-stone to the GPU.

Mechanism · why 1,024 is the magic number

A vector is sized to fit comfortably in L1/L2 cache while being long enough to amortize call overhead and fill SIMD registers. Too small and overhead dominates (back toward Volcano); too large and the working set spills out of cache (back toward full materialization). The GPU takes this idea to its limit: the "vector" becomes the entire column, and the "SIMD lanes" become tens of thousands of GPU threads. cuDF is, in this lineage, a maximally-wide vectorized engine — with all the benefits and the memory-pressure costs that implies.

Data-centric compilation — fuse the whole pipeline

HyPer's approach (Neumann, "Efficiently Compiling Efficient Query Plans for Modern Hardware", VLDB 2011) flips pull to push and eliminates the interpreter entirely: the plan is compiled (via LLVM) to machine code in which a tuple is kept in CPU registers and pushed through as many operators as possible before it ever has to be written to memory. The boundaries where data must be materialized are called pipeline breakers — the build side of a hash join, a sort, an aggregation. Between two pipeline breakers, all operators are fused into one tight loop over the data.

The through-line to the GPU

This "fuse everything between pipeline breakers" idea is exactly what GPU kernel fusion does, and it's the single most important concept connecting this section to Part IV. On a CPU, fusion keeps tuples in registers to avoid memory traffic. On a GPU, fusion keeps intermediate columns in registers/shared memory to avoid (a) writing multi-gigabyte intermediates to scarce HBM and (b) paying a kernel-launch tax per operator. cuDF's default operator-at-a-time model is the un-fused extreme — every operator materializes its full output — which is why fusion is the headline optimization for GPU query engines.

Morsel-driven parallelism — how to fill many cores

Compilation makes a single pipeline fast; morsel-driven parallelism (Leis et al., SIGMOD 2014) makes it parallel. The input is sliced into small morsels (a few thousand rows). A fixed pool of worker threads pulls morsels from a queue, processes each through the compiled pipeline, and steals work when idle — all NUMA-aware, so a thread processes data on its local memory node. This dynamic, work-stealing dispatch is the CPU cousin of how a GPU schedules thousands of thread-blocks across its streaming multiprocessors: oversubscribe with small units of work and let the hardware load-balance.

Columnar & vectorized execution

Vectorized and GPU execution both assume columnar storage — values of one column laid out contiguously, rather than rows laid out field-by-field. For analytics this is the right default, and understanding why explains most of cuDF's memory layout.

Row vs column, and why analytics is columnar

  • Scan only what you need. Our query touches region, segment, amount, … but not phone, address. Column storage reads only the referenced column files; row storage drags every field of every row through the cache. For wide tables this is a 10–100× difference in bytes moved.
  • Compression that actually helps. A column holds values of one type and often low cardinality, so it compresses far better than mixed-type rows — and the encodings double as speedups (next).
  • SIMD / SIMT friendliness. A contiguous typed array is exactly what a vector unit — or a GPU warp — wants to chew through with one instruction over many lanes.

Encodings — and computing without decompressing

The encodings matter because the best engines operate on the encoded form:

  • Dictionary encoding — replace values with small integer codes into a dictionary. Filters and joins can run on the codes; group-by can aggregate by code. Decode only at the very end.
  • Run-length encoding (RLE) — store (value, count) for repeated runs. A predicate can accept/reject an entire run at once.
  • Bit-packing — use only as many bits as the value range needs (a column of values 0–7 needs 3 bits, not 32).
  • Frame-of-reference — store deltas from a base value; great for sorted or clustered numeric columns.

Late materialization — touch the fewest bytes

The defining tactic of columnar engines: don't assemble rows early. Operate on columns and pass around selection vectors (lists of qualifying row positions) instead of materializing intermediate rows; reconstruct full tuples only at the output. A highly selective filter then means later operators touch tiny position lists rather than wide rows. This is doubly vital on the GPU, where every materialized intermediate consumes scarce HBM and PCIe-adjacent bandwidth — late materialization is one of the few free lunches.

Mechanism · Apache Arrow is the bridge to cuDF

cuDF's in-memory layout is the Apache Arrow columnar format: contiguous typed value buffers plus a separate validity bitmask (one bit per element marking null/not-null). Arrow's significance is that it's a shared layout — Spark, pandas, Polars, Velox, and cuDF can hand each other column buffers with zero serialization, and on the GPU that means data can move host→device (or GPU→GPU) as raw Arrow buffers without a transcoding step. When §12 describes a cuDF column as "a data buffer + a null mask + children," that is Arrow. The validity-bitmask detail returns in §14, because handling nulls correctly inside a fused GPU kernel is a real source of complexity.

Adaptive query execution

FIG 04C established that upfront cardinality estimates are often wrong by orders of magnitude. Adaptive query execution (AQE) is the pragmatic response: stop trusting the estimate, measure the real cardinality once part of the query has run, and re-optimize the rest with facts instead of guesses.

The mechanism: re-optimize at materialization boundaries

Re-planning can only happen where the engine already has exact statistics — at pipeline breakers (§08), the points where an intermediate result is fully materialized. After a shuffle or a build phase completes, the engine knows the true row count and size of that intermediate, and can revise downstream choices before continuing.

Spark's AQE is the most widely deployed example, and its three moves map cleanly onto the problems we've built up:

  • Coalesce shuffle partitions. After a shuffle, merge tiny partitions so the next stage isn't drowned in scheduling overhead — fixing an over-partitioning estimate after the fact.
  • Switch join strategy. If a side estimated as large turns out (post-materialization) to be small enough to fit in memory, swap a sort-merge join for a broadcast hash join — exactly the §07 algorithm choice, but made with real sizes.
  • Skew handling. Detect partitions far larger than the median (the join/group-by skew problem) and split them across more tasks so one straggler doesn't dominate. This skew problem is much worse on the GPU (§14–15), where a single hot key serializes atomic updates.

Sideways information passing (runtime / dynamic filters)

A subtler adaptive technique that recurs on the GPU: while building a hash join's build side, construct a compact Bloom filter of the build keys and push it down to the probe-side scan. The scan then discards non-matching rows before they're ever read or transferred. On a distributed or GPU engine this can eliminate enormous amounts of data movement — you avoid shuffling or DMA-ing rows that can't possibly join. This is the same idea as predicate pushdown (§03), but the predicate is discovered at runtime from the other side of the join.

Why GPUs lean on adaptivity even harder

Three forces make runtime adaptation more important on the GPU than on the CPU: (1) a cardinality miss is more expensive because re-running an operator means re-materializing a multi-GB column in scarce memory; (2) the spill/no-spill decision (does this hash table fit in GPU memory?) genuinely cannot be made well upfront — it depends on the true intermediate size; and (3) the where-to-run-it decision (this operator on GPU vs CPU) depends on data sizes only known at runtime. So GPU query engines tend to combine a lightweight optimizer with aggressive runtime measurement, rather than betting everything on a static plan. AQE isn't a nicety on the GPU — it's load-bearing.

Does the optimizer matter? Engines vs. cuDF

This is the question that motivated the manual, so let's answer it squarely: a cost-based optimizer matters enormously in a database engine, matters in a different and subtler way in a library like cuDF, and — crucially — does not vanish on the GPU so much as move up one layer and acquire new variables. The rest of Part IV is the detail behind that sentence.

"Does the optimizer matter" has no single answer because "the optimizer" lives at different layers depending on what you're running. Pull them apart:

When you write pandas-style cuDF code, there is no component sitting between your call and the kernel that reconsiders your choices. a.merge(b) launches a hash join right now; if joining b into a in the other order, or filtering before the join, would have been 100× cheaper, nothing will fix that for you. Every optimization the §03–07 machinery does automatically — predicate pushdown, join reordering, build-side selection, projection pruning — becomes your responsibility, expressed as the order and shape of your API calls. The skills in Parts I–II don't stop mattering when you move to cuDF; they stop being automatic.

But libcudf is still making physical decisions for you

"No optimizer" doesn't mean "no choices." Below the API, every libcudf operator is itself a packaged physical-operator decision, frozen at library-authoring time rather than chosen per-query:

  • cudf::groupby picks between a hash-based aggregation and a sort-based one depending on the aggregations requested — the §07 hash-vs-sort dichotomy, decided inside the library.
  • cudf::hash_join commits to an open-addressing GPU hash table (via cuCollections), a build/probe split, and an output-sizing strategy — the §14 machinery, none of it your call.
  • Sorts dispatch to radix sort; nulls flow through validity-mask handling on every operator. These are real algorithmic decisions; you simply don't get a knob for them.

So the optimization that classical systems do across operators (which order? which algorithm for this specific join?) is absent, while the optimization within an operator (how do I implement a fast GPU hash join?) is present and excellent. cuDF is a library of superb physical operators with no planner on top — which is exactly why putting a planner on top is the whole business of the next layer.

The engines that put the optimizer back

The moment you want to feed cuDF a SQL string instead of imperative calls, the optimizer reappears — and now it has more to decide than a CPU optimizer, not less:

  • Spark RAPIDS Accelerator reuses Spark's Catalyst optimizer wholesale — all the logical rewrites and cost-based join planning of §03–07 still run — then adds a GPU-execution layer that, plan node by plan node, decides whether each operator runs on the GPU (as a libcudf call) or falls back to CPU, inserting host↔device transfers at the boundaries. Its planning problem is Catalyst's plus a placement-and-transfer problem.
  • BlazingSQL (historically) parsed SQL through Apache Calcite — a standalone, reusable optimizer framework — to produce a relational plan, then executed the physical operators on cuDF. The optimizer was Calcite; the executor was the GPU.
  • Dask-cuDF builds a lazy task graph and optimizes that (fusing tasks, pruning columns, pushing down filters) before any kernel runs — graph-level optimization standing in for a classical query optimizer.
  • GPU-native engines (HeavyDB, and newer GPU-first query engines such as NVIDIA's Theseus / the Velox-plus-GPU direction your own work sits in) build a real cost-based optimizer that is GPU-aware from the ground up — modeling PCIe, memory residency, and fusion as first-class costs rather than bolting them on.
The new decision variables the GPU adds

When the optimizer does exist over a GPU engine, its cost model grows several dimensions the classical model (§05) never had. These are the through-lines of the rest of Part IV:

  • Transfer cost (PCIe). Moving a column host→device can cost more than computing on it. A "cheaper" plan that forces a transfer can lose to an "expensive" plan that keeps data resident. (§13, §15)
  • Memory capacity & spill. GPU memory is 24–80 GB, not terabytes. Whether an intermediate fits is now a planning question, and "spill or not" is a runtime one. (§13, §15)
  • Fusion opportunity. Because the default model materializes every operator, the biggest single win is fusing adjacent operators to skip a multi-GB write. The optimizer wants to maximize fusable runs. (§08, §15)
  • Placement (CPU vs. GPU). In a heterogeneous plan, each operator has a where-to-run decision that depends on data size, transfer cost, and what's already resident. (§16)
  • Kernel-launch overhead. Thousands of tiny operators pay a fixed per-launch tax; the optimizer (or CUDA graphs) wants to amortize it. (§13, §15)

The verdict. Optimization matters as much as it ever did — arguably more, because the penalties (a needless multi-GB transfer, an out-of-memory spill) are harsher than a CPU plan simply running slow. What changes is where it lives: in bare cuDF it collapses onto you, the programmer; in an engine over cuDF it returns as a richer cost-based optimizer with new GPU-specific variables. The classical theory of Parts I–II is the foundation; Part IV is what you build on top of it once the target machine is a GPU.

cuDF & libcudf architecture

Before the GPU operators make sense, you need the shape of the thing they operate on and the memory system they allocate from. cuDF is a stack: a pandas-like Python API on top, a C++ library (libcudf) doing the real work, and a memory manager (RMM) underneath — all built on the Arrow columnar layout from §09.

The column model — Arrow on the device

A cuDF column is not an array of values; it is a small structure of GPU buffers, identical in spirit to Apache Arrow:

  • A contiguous data buffer in device memory holding the typed values (e.g. int64 laid end to end).
  • An optional null mask — a packed bitmask, one bit per element, 1 = valid, 0 = null. This single detail complicates every GPU kernel (§14), because each thread must check its element's validity bit.
  • For nested or variable-width types (strings, lists), one or more child columns: a strings column is an int32 offsets child plus a char data child, so the values are contiguous and indexed by offset — exactly the GPU-friendly layout, and exactly Arrow.

Because this layout is Arrow, a cuDF column can be handed to or from Spark, Polars, or pandas, or moved host→device or GPU→GPU, as raw buffers with no serialization step. A table is just a set of equal-length columns. There is no row object anywhere — which is the entire point.

Mechanism · RMM and why cudaMalloc is the enemy

A raw cudaMalloc/cudaFree is a synchronizing call: it can stall the GPU and serialize against running kernels, costing tens of microseconds each. An operator-at-a-time engine that allocated a fresh buffer for every intermediate would spend much of its life in the allocator. RMM (the RAPIDS Memory Manager) fixes this with a pool allocator: grab one big slab of device memory up front, then satisfy each operator's allocation by sub-dividing that slab — turning a kernel-synchronizing system call into a cheap pointer bump. RMM is also the seam where spilling and memory-limit policy (§15) are enforced, and where you'd plug in managed/UVM memory. When people say RAPIDS "manages GPU memory well," this pool is most of what they mean.

The stack, top to bottom

  • libcudf — the C++ core: typed columns, and the operator library (join, groupby, sort, filter, scan, string ops) implemented as CUDA kernels. This is where the algorithms of §14 live.
  • cudf (Python) — a pandas-shaped binding over libcudf. Each DataFrame method maps to one or a few libcudf calls; there is no lazy graph by default, so each call executes immediately (the "you are the optimizer" property of §11).
  • cudf.pandas — a compatibility shim that makes existing pandas code run on the GPU by intercepting pandas calls and routing them to cuDF, falling back to pandas on the CPU for anything unsupported. A drop-in accelerator, not a new API.
  • dask-cudf — partitions a logical dataframe across many GPUs/nodes, with each partition a cuDF dataframe, coordinated by a Dask task graph (which is itself optimized — §11).
  • cuCollections (cuco) — the standalone GPU data-structure library providing the concurrent hash tables that libcudf's join and hash-groupby are built on. When §14 talks about GPU hash tables, it means cuco.
The execution model, and the cost it implies

cuDF is operator-at-a-time: each operator launches one (or a few) kernels that read whole input columns and write whole output columns to device memory, then returns — the far-right end of the §08 granularity spectrum. This is what makes it simple, composable, and bandwidth-saturating. It is also what makes materialization the dominant cost: a five-operator query writes and re-reads five full intermediates through HBM, and pays five kernel launches. Hold this thought — it is the single fact that motivates fusion (§15) and explains why a chain of cuDF calls can be memory-bound even when each individual kernel is fast.

What changes on the GPU

A GPU is not a faster CPU; it is a different shape of machine, and almost every assumption baked into the classical cost model (§05) and the Volcano execution model (§08) breaks against that shape. Three facts dominate everything else: the GPU has thousands of threads but they march in lockstep; it has staggering memory bandwidth but a tiny pipe to the rest of the system; and it loves doing the same thing to a whole column at once and hates doing one thing to one tuple. Get these three and the rest of Part IV is commentary.

SIMT: thousands of threads, marching in step

A CPU has a handful of fat cores, each with deep out-of-order execution, big caches, and branch prediction tuned to make one thread fast. A GPU has tens of thousands of thin threads grouped into warps of 32 that execute in SIMT (single-instruction, multiple-thread) fashion: all 32 threads in a warp issue the same instruction each cycle, on different data. This is why a GPU can scan a billion-row column so fast — the work is identical per element and maps perfectly onto warps — and also why two pathologies appear that have no CPU analog:

  • Warp divergence. If threads in a warp take different branches (if (x > 0) A else B with mixed values), the hardware must execute both paths and mask off the inactive threads — serializing what looked like parallel work. A filter or join probe with data-dependent branching can lose much of its throughput to divergence. The columnar, branch-light operators of §09 exist partly to avoid this.
  • Coalescing. When the 32 threads of a warp read 32 consecutive addresses, the hardware fuses them into one wide memory transaction — a coalesced access at full bandwidth. When they read scattered addresses (a hash-table probe, a gather), it issues many small transactions and bandwidth collapses. Contiguous columnar layout makes scans coalesce automatically; the random access pattern of hash probing is exactly the case that does not (§14).

The memory hierarchy — and the straw

The second fact is the one that reshapes query planning. A GPU's on-package memory is extraordinarily fast, but everything that has to travel to or from the GPU goes through pipes that are one to two orders of magnitude narrower. The whole game of GPU data processing is keeping work on the fast side of this cliff:

Read the figure as a planning constraint. A GPU can chew through HBM-resident data at terabytes per second, so a scan-heavy aggregate that fits in device memory is almost absurdly fast. But the moment a plan needs to pull data across PCIe — to load it, to spill it, to hand it to the CPU — it hits a pipe so narrow that the transfer can dominate total runtime. This single asymmetry is why the new cost-model dimensions of §11 (transfer cost, residency, fusion, placement) exist, and why §15 is mostly a catalog of techniques for not paying the straw tax.

Why Volcano dies on the GPU

The iterator model of §08 calls next() once per tuple and threads control flow through a tree of operators. On a GPU this is doubly fatal. First, a per-tuple virtual call is meaningless when the execution unit is a 32-wide warp — there is no efficient way to run one tuple through one operator. Second, the model's whole premise is that control flow is cheap and data movement (of one tuple) is free; on a GPU control flow is expensive (divergence) and the only thing that is free-ish is streaming a contiguous column. So GPU engines abandon tuple-at-a-time entirely and run operator-at-a-time over whole columns (§12): each operator is a kernel that reads input columns coalesced, does identical work per element across thousands of threads, and writes output columns. The price — full materialization of every intermediate in HBM — is the central cost that fusion (§15) attacks.

Kernel-launch overhead and occupancy

Two more GPU-specific costs round out the picture. Every kernel launch has a fixed overhead of a few microseconds — negligible for one big kernel over a billion rows, ruinous for ten thousand tiny kernels over small partitions. This is why GPU engines prefer fewer, larger operators and why CUDA graphs (capture a whole sequence of launches once, replay with one call) matter (§15). And to hide HBM latency the GPU relies on occupancy: keeping enough warps resident that while some wait on memory, others compute. A kernel that uses too many registers or too much shared memory per thread limits how many warps fit, lowers occupancy, and exposes memory latency — a constraint that directly shapes how the GPU hash table of §14 is sized.

GPU operators in depth

Now the payoff: how the operators of §07 are actually built on a SIMT machine. Three matter most — hash join, group-by aggregation, and sort — and each runs into the SIMT and bandwidth realities of §13 in a characteristic way. The recurring themes are the build/probe asymmetry from FIG 07, the cost of not knowing the output size in advance, and the contention that appears whenever many threads must write the same place.

GPU hash join

Recall the asymmetry (FIG 07): build is write-heavy, probe is read-only and parallel. On a GPU that asymmetry becomes the whole design:

  • Build. Thousands of threads insert keys into a shared hash table concurrently. There is no per-thread lock; instead the table is an open-addressing array (no per-bucket chains — chasing pointers would destroy coalescing) and each insertion uses an atomic compare-and-swap (atomicCAS) to claim an empty slot, probing linearly on collision. This is exactly what cuCollections (cuco) provides, and it is what cudf::hash_join builds on. Contention on hot slots and the atomics are the cost.
  • Probe. Each probe-side row independently hashes its key and walks the open-addressing slots until it finds a match or an empty slot. The work is read-only and embarrassingly parallel — but the memory access is random, so it does not coalesce (§13). A hash-join probe is therefore latency-bound on HBM, hidden only by high occupancy. This is the single biggest reason GPU joins are slower per byte than GPU scans.
  • The output-size problem. Before probing, you do not know how many rows the join will emit (each probe row may match zero, one, or many build rows). You cannot pre-allocate the exact output. Two strategies: (a) count-then-materialize — a first pass counts matches per row, a prefix sum (scan) turns counts into output offsets, and a second pass scatters results into the exactly-sized buffer; or (b) atomic append — each match does an atomicAdd on a global output counter to claim a write position, trading a sizing pass for atomic contention on the counter. Count-then-materialize is the workhorse because it makes the writes coalesced and the allocation exact.
  • Partitioning to fit fast memory. When the build table is too large for the L2/shared budget that keeps probing fast, the GPU does the same thing the grace hash join of §07 does: radix-partition both inputs by hash bits into buckets, then join bucket-by-bucket so each build sub-table fits in fast memory. The CPU technique and the GPU technique are the same idea aimed at a different level of the hierarchy.

GPU group-by — the contention story

Aggregation is where SIMT contention is most visible, and it directly concerns streaming-aggregation work. The naive approach — every thread atomicAdds its value into a global hash table keyed by group — is correct but pathological under skew: if one key holds 30% of the rows, thousands of threads serialize on that one slot's atomic, and the "parallel" aggregation runs nearly sequentially for the hot key. The fixes mirror the §07 hash-vs-sort dichotomy plus a GPU-specific middle path:

  • Sort-based (segmented reduction). Sort rows by group key (radix sort, below), which clusters each group into a contiguous run, then do a single linear segmented reduction over runs. No atomics, perfectly coalesced, skew-immune — but pays the full sort cost. The §07 sort-aggregation, GPU edition.
  • Two-level hash (shared then global). The figure's right panel: aggregate locally in per-block shared memory, then merge partials globally. This keeps almost all the atomic traffic on-chip and is the default high-performance path for moderate group counts.
  • The optimizer-shaped decision — when groups are few, hash wins; when groups are enormous or input is pre-sorted, sort wins — is made inside libcudf, the "local choices only" row of the §11 table.

GPU sort — radix, not comparison

Comparison sorts (quicksort, mergesort) are branch-heavy and diverge badly on SIMT. GPUs instead use radix sort: process keys digit by digit, and at each digit do a counting sort built from a parallel prefix sum to compute output positions. Radix sort is all coalesced reads, histogram counting, and scatter by computed offset — no data-dependent branching, no divergence. It is the engine under cudf::sort, under sort-based group-by, and under sort-merge joins on the GPU. Note the recurring primitive: the prefix sum (scan) turns "how many of each / where does each go" into exact output offsets in both radix sort and the count-then-materialize join — it is the load-bearing parallel primitive of the entire GPU operator library.

The unifying pattern

Every GPU operator here reduces to the same three concerns from §13: keep reads coalesced (contiguous columns, radix scatter), avoid divergence (branch-light, histogram-and-scan instead of compare-and-branch), and manage contention on shared writes (open-addressing + atomics for joins, shared-memory-first for aggregation). When you read that a GPU join or group-by is "memory-bound," it is one of these three biting: random probe access that won't coalesce, or atomic serialization on a hot slot. The optimization techniques of §15 are the systemic responses.

The transfer tax, and the three ways to dodge it

This is the bottleneck that has no CPU equivalent and the one that most often makes a GPU plan lose to a CPU one. Three orthogonal techniques attack it:

  • GPUDirect Storage (GDS). Normally data goes NVMe → CPU bounce buffer in host RAM → GPU, crossing the bus twice and burning CPU. GDS establishes a direct DMA path from storage straight into GPU memory, removing the host bounce entirely. For scan-heavy analytics that stream large columns off disk, this can roughly double effective load bandwidth and free the CPU. (This is squarely the I/O-path concern; libraries like kvikIO expose GDS to cuDF.)
  • Pinned (page-locked) memory. A normal cudaMemcpy from pageable host memory is slower because the driver must stage through an internal pinned buffer. Allocating the host side as pinned lets the DMA engine read it directly, hitting full PCIe bandwidth and — crucially — enabling true asynchronous copy.
  • Overlap via streams. With pinned memory and multiple CUDA streams, you can copy chunk N+1 while computing on chunk N, hiding transfer latency behind compute. A well-pipelined loader makes the straw's width matter less because it is never idle and never the thing you're waiting on at the end.

The meta-rule behind all three: the cheapest transfer is the one you don't do. A GPU engine's first instinct is to keep data resident across as many operators as possible (which is also why fusion and residency-aware planning, §11, pay off twice — once in compute, once in avoided transfers).

Fusion — the single biggest structural win

Operator-at-a-time's defining cost is that a chain like filter → project → aggregate writes the filtered column to HBM, reads it back to project, writes that, reads it back to aggregate. Each intermediate is a full multi-GB round trip through the very bandwidth you're trying to conserve. Kernel fusion collapses adjacent operators into a single kernel that carries values from one stage to the next in registers or shared memory, never materializing the middle. This is the exact GPU analog of the compiler-based pipelining of HyPer/Neumann (§08): there, a CPU compiler fused operators within a pipeline to keep tuples in registers between pipeline breakers; here, a fused GPU kernel keeps column elements in registers between operators. The pipeline-breaker concept transfers directly — fusion runs between breakers (joins, sorts, aggregations that must see all input), and the breakers are where materialization is unavoidable. An engine over cuDF that wants performance is, more than anything else, a fusion engine.

CUDA graphs · amortizing the launch tax

When a query decomposes into many small kernels (common when partitions are small or the plan is deep), the few-microsecond launch overhead per kernel adds up to real time, and the CPU issuing launches can even become the bottleneck — the GPU sits idle between launches. CUDA graphs solve this by capturing the entire sequence of kernel launches and their dependencies once into a graph object, then replaying the whole thing with a single launch call. The per-kernel CPU overhead is paid at capture time, not on every execution, and the GPU runs the sequence back-to-back. For iterative or repeated query shapes, this turns thousands of launches into one. PyTorch uses the same mechanism to cut launch overhead in training loops — the technique generalizes well beyond databases.

Memory pressure and spilling

Because device memory is small, a real engine must answer "what if it doesn't fit?" at runtime — itself an adaptive decision (§10). cuDF's spill manager (via RMM) can evict columns from device memory to host memory (or disk) under pressure and page them back on demand, trading the straw tax for not crashing. Unified Virtual Memory (UVM) offers an automatic alternative: oversubscribe device memory and let the driver page-fault data in and out — convenient, but page-fault migration is expensive and unpredictable, so hand-managed spilling usually wins for performance-critical paths. Either way, "spill or recompute or fail" becomes a planning-and-runtime decision the classical model never had to make, because host RAM was effectively unbounded.

Distributed & multi-GPU execution

One GPU's 24–80 GB runs out, and one node's GPUs run out, long before the data does — so production GPU analytics is distributed across many GPUs and many nodes. This changes the optimization problem in one dominant way: the most expensive thing a distributed query can do is move data between partitions, the shuffle, and the optimizer's central job becomes minimizing and accelerating it. This is the layer where the physical-property idea of §07 reaches its final form, and where the GPU-interconnect and communication-library work becomes the critical path.

Partitioning is the dominant physical property

In §07 we said the optimizer tracks sort order and partitioning as physical properties a plan delivers. In a distributed engine, partitioning dominates everything else, because the rule is brutally simple: an operator can run locally on each partition only if its inputs are partitioned on the right key. A join of A and B on key k can run partition-local — with no communication — if and only if both A and B are already hash-partitioned on k across the same set of workers. If they aren't, the engine must repartition (shuffle) one or both, and that shuffle is almost always the most expensive operator in the plan. So the distributed optimizer's job is largely: choose a partitioning for every intermediate that satisfies downstream operators while triggering the fewest, smallest shuffles.

The shuffle / exchange operator

The exchange (or shuffle) operator is how data is repartitioned: every worker hash-partitions its local rows by the target key and sends each row to the worker that owns that key's range, so afterwards each worker holds all rows for its assigned keys. It is an all-to-all communication — every worker talks to every other worker — and it is the distributed analog of the join-algorithm choice from §07:

Broadcast vs. hash-partitioned join — the distributed §07 decision

Just as a single-node optimizer chooses among nested-loop / sort-merge / hash join, a distributed optimizer chooses among distributed join strategies, and the choice turns on cardinality estimates (§04) exactly as before:

  • Broadcast (replicated) join. If one side is small, send a full copy of it to every worker; each worker then joins its local big-side partition against the complete small side — no shuffle of the big side at all. This is the distributed cousin of build-side selection, and it is what Spark AQE's "switch to broadcast" move (§10) does at runtime when it learns a side is small enough.
  • Hash-partitioned (shuffle) join. If both sides are large, hash-partition both on the join key (two shuffles) so matching keys co-locate, then join locally. Unavoidable when neither side fits for broadcast — and the case where shuffle performance is everything.
  • Choosing wrong is catastrophic here in a way it isn't on one node: broadcasting a "small" side that cardinality estimation under-counted can OOM every worker, while shuffling a side that could have been broadcast wastes the most expensive resource in the system. The §04 hinge bites hardest at distributed scale.
Why the interconnect is the cost model · where the UCX work lands

On a single GPU the bottleneck was PCIe (FIG 13). Across GPUs it becomes the network and the GPU-to-GPU path, and the same "don't cross the slow link" logic governs planning — now at cluster scale. The naive path for a cross-node shuffle is GPU → host memory → NIC → network → NIC → host memory → GPU, crossing PCIe four times and copying through CPU buffers twice. GPUDirect RDMA removes the host bounce: the NIC reads and writes GPU memory directly, so a partition moves GPU→GPU across the network without ever touching host RAM. A communication library like UCX is what abstracts this — selecting NVLink, PCIe peer-to-peer, or RDMA transports automatically, and exposing tag-matched messaging plus one-sided RMA and active messages for the irregular, data-dependent send patterns a shuffle produces (each worker sends a different, unpredictable amount to each peer). This is precisely why UCX-based GPU exchange/shuffle in an engine like Velox/Presto is on the critical path: the distributed optimizer can minimize shuffles, but the shuffles that remain are bounded entirely by how fast this layer moves bytes between GPUs. Optimizer and interconnect are two halves of the same performance story — the planner decides how much to move, the communication library decides how fast it moves.

Multi-GPU adds a hierarchy to even this: GPUs within a node talk over NVLink (600–900 GB/s), GPUs across nodes over the network (often 10× slower), so a topology-aware engine prefers intra-node exchange and minimizes inter-node traffic — the NUMA and network-topology concerns of single-node tuning, lifted to the cluster. The dominant physical property (partitioning) and the dominant cost (moving between partitions) are the same idea from §07 and §16's opening, now the entire ballgame.

18

The whole machine

Step back from the parts and the shape of the whole becomes visible. A query optimizer is one idea wearing many costumes, and the move to the GPU does not replace that idea — it raises the stakes on it and adds new variables. Here is the manual compressed to the few sentences worth carrying out the door.

The optimizer is a bet under uncertainty about a machine

From the very first section: the optimizer takes a declarative query, faces a search space too large to enumerate exhaustively (§06), estimates costs from statistics that are always approximate (§04–05), and commits to a single physical plan it hopes is good. It is a wager — and like any wager, its quality depends on the odds (the cost model) being honest about the table (the hardware). Everything else in the manual is detail about how the bet is placed and what happens when the table changes.

Cardinality estimation is the hinge

If one idea earns disproportionate attention, it is this (§04). Every downstream decision — join order, join algorithm, broadcast-vs-shuffle, build-side selection, whether an intermediate fits in GPU memory — is only as good as the row-count estimate feeding it, and estimation error compounds multiplicatively up a join tree. This is why adaptive execution exists (§10), why the independence assumption is the classic failure mode, and why "the optimizer chose a bad plan" almost always traces back to "the estimate was wrong." On the GPU the hinge swings harder: a misjudged cardinality doesn't just make a plan slow, it can force a multi-GB transfer or an out-of-memory spill.

Fusion is the GPU analog of compilation

The throughline of Part IV. The CPU world learned that tuple-at-a-time iteration wastes cycles on control flow, and answered with vectorization and then compilation — fusing operators within a pipeline to keep data in registers between pipeline breakers (§08). The GPU world inherits the identical insight in a new dialect: operator-at-a-time execution wastes the GPU's scarce resource — memory bandwidth — by materializing every intermediate, and the answer is kernel fusion, keeping column elements in registers between operators across the same pipeline breakers (§15). Same problem, same solution, different hardware. Recognizing that equivalence is most of what it takes to reason about GPU query performance.

Partitioning and the shuffle dominate at scale

The physical-property idea of §07 — that a plan delivers sort order and partitioning, not just rows — becomes the entire game in distributed and multi-GPU execution (§16). The dominant cost is moving data between partitions; the dominant optimization is choosing partitionings that avoid doing so; and the dominant engineering problem is making the unavoidable shuffles fast, which is exactly what GPUDirect RDMA and a communication library like UCX exist to do. At cluster scale, the optimizer decides how much to move and the interconnect decides how fast — and the two are inseparable halves of one performance story.

Optimization moves up a layer — and gains new variables

The direct answer to the question that opened the manual (§11). A cost-based optimizer is decisive in a database engine, absent in bare cuDF (where calling an operator is the plan decision and you are the optimizer), and reconstructed — richer than before — in engines built over the GPU. What it gains on the way up are dimensions the classical model never had: transfer cost across the PCIe straw, memory capacity and spill, fusion opportunity, and CPU-vs-GPU placement. Optimization didn't vanish when execution moved to the GPU; it relocated and grew. The classical theory of Parts I–III is the foundation, and Part IV is what you build on it once the target machine is a firehose connected to the world by a straw.

The one-sentence version

A query optimizer is a bet, placed under cardinality uncertainty, about how to run a plan on a specific machine; moving that machine from CPU to GPU keeps every classical idea, swaps compilation for fusion, makes the interconnect the cost model, and pushes the optimizer one layer up — above cuDF, into the engine — where it acquires a new set of variables and matters more than ever.

Appendix — The Operator Atlas
A1

Access methods — how a table is read

Everything in Parts I–V assumed the plan was built from a small vocabulary of physical operators. This appendix draws each one. We start at the leaves of every plan: the access methods — the ways an engine turns "I need rows from this table" into actual page reads. The choice among them is pure selectivity arithmetic (§04): how many rows qualify, and how scattered are they.

How to read these figures

Neutral boxes are heap pages (the table's actual storage). Cyan structures are indexes / hash tables. Sage marks rows that survive and reach the output; rust marks work discarded. Copper arrows are the operator's hot path. Each figure ends with the rule the optimizer uses to choose it.

Sequential scan

Read every page of the heap in physical order and apply the predicate to each tuple. No index, no random I/O — just one long streaming sweep. Its cost is N_pages sequential reads regardless of how many rows actually qualify, which is exactly why it wins when the predicate is unselective (most rows pass) or the table is small enough that index bookkeeping isn't worth it. On a columnar/GPU engine this is the native, friendliest pattern: a coalesced linear read of a column (§13).

Index scan

Descend a B-tree on the predicate column to the qualifying key(s), then — for each match — follow the index entry's pointer to do one random heap fetch to retrieve the full row. The cost is the tree height (a few I/Os) plus one random read per matching row. That random-per-match term is the whole story: at low selectivity (a handful of rows) it crushes the sequential scan; as matches grow, random I/O piles up until, somewhere around a few percent of the table, a sequential scan that reads everything once becomes cheaper. This crossover is the optimizer's single most common scan decision.

Index-only scan

The optimization that removes the random heap fetch entirely. If every column the query needs is already stored in the index (a covering index), the answer can be assembled from the index leaves alone — the heap is never touched. The one wrinkle is MVCC visibility: the index doesn't record whether a row version is visible to your transaction, so the engine consults a compact visibility map; if a page is marked all-visible, it's skipped, and only for the rare not-all-visible page does it fall back to a heap fetch to check. This is why adding the one extra column a query selects can flip a plan from "index scan + thousands of random fetches" to "index-only," a routine and dramatic win.

Bitmap scan

The hybrid that captures the middle of the selectivity range. Rather than fetch the heap one row at a time as it walks the index (random, and a page can be hit repeatedly), a bitmap index scan first materializes an in-memory bitmap of which heap pages/tuples qualify. Multiple indexes can be combined with bitmap AND/OR before touching the heap at all. Then a bitmap heap scan walks the heap in physical page order, visiting each flagged page exactly once. The result reads like a partial sequential scan: it recovers I/O ordering and de-duplicates page visits, bridging the gap between a plain index scan (great at very low selectivity) and a sequential scan (great at high selectivity).

A2

Join algorithms — the family in pictures

The join is where the optimizer spends most of its intelligence (§06–07), and the physical-operator choice among these six is the most consequential single decision in most plans. They divide cleanly: nested-loop family (cheap to start, quadratic if misused), sort-merge (pays for order, then linear), and the hash family (linear, memory-hungry, with graceful fallbacks when memory runs out). Watch for the build/probe asymmetry from FIG 07 — it reappears everywhere and becomes the entire design constraint on the GPU (§14).

Nested-loop join (and the block variant)

The definitional join: for each row of the outer input, scan the inner input for matches. Naively that is |R| × |S| work — fine when the inner is tiny, ruinous otherwise. The standard refinement is the block (buffered) nested-loop: buffer a chunk of B outer rows in memory and scan the inner once per block rather than per row, cutting passes over the inner by a factor of B. Nested-loop is also the only general option for non-equi join conditions (<, ranges, inequalities), where hash and merge can't apply.

Index nested-loop join

Nested-loop's redemption, and frequently the fastest join when the outer side is small. Replace the inner scan with an index probe: for each outer row, look up the join key in an index on the inner relation and fetch only the matching rows. Cost collapses from |R|×|S| to roughly |R| × (h + matches) — a handful of I/Os per outer row. This is the natural partner of a highly selective outer (after a restrictive filter, §04) joined to a large, well-indexed inner: the optimizer probes the big table a few times rather than scanning or hashing all of it.

Sort-merge join

Sort both inputs on the join key, then merge them with two cursors that advance in lockstep: when the keys match, emit; otherwise advance whichever cursor trails. Duplicate keys are handled by mark/restore — remember where a run of equal keys began on the inner so the cursor can rewind to it for each matching outer row. The merge itself is linear; the cost is the sort. Crucially, that sort is free if the inputs already arrive sorted — from an index, or as the byproduct of an upstream operator — which is precisely the physical-property reasoning of §07. Sort-merge also leaves its output sorted, which a downstream ORDER BY or merge can reuse, and it handles range/band joins that hashing cannot.

Hash join (in-memory)

The default equi-join when neither a useful sort nor an index is available. Build a hash table from the smaller input, then stream the larger input through it, emitting matches on lookup (§07, FIG 07). It is linear — O(|R|+|S|) — with one hard precondition: the build side's hash table must fit in memory. The asymmetry is the thing to remember: build is write-heavy (insertions, collision handling, concurrent updates) while probe is read-only and embarrassingly parallel. That single fact dictates the entire GPU implementation (§14) — probes map trivially to thousands of threads; builds need atomics — and motivates the two fallbacks that follow when the build side is too large to fit.

Grace / partitioned hash join

What to do when the build side does not fit. Hash-partition both inputs by the same function h(key) into k buckets, spilling them to disk (or to separate GPU/host regions). Because identical keys hash identically, all matches for a key land in the same partition pair (R_i, S_i) — so the global join decomposes into k independent in-memory hash joins, each sized to fit. The price is two passes over both inputs (partition, then join). This is the canonical out-of-core join, and its logic is exactly the GPU spill strategy when an intermediate exceeds device memory (§15).

Hybrid hash join

Grace's refinement, and what most production engines actually run. If there's some memory but not enough for the whole build side, don't spill everything: keep partition 0's build side resident in RAM while partitioning the rest. Probe rows belonging to partition 0 are joined immediately and never spilled, so their data skips the spill-and-reload round-trip entirely; partitions 1..k spill as in grace and are processed afterward. The payoff is a smooth degradation across the memory spectrum: with abundant memory it collapses to a single in-memory hash join (one partition, no spilling); with almost none it becomes pure grace; in between it spends exactly the memory it has on avoiding I/O for the hottest fraction of the data.

A3

Join shape & distribution — related operators

Two families that sit beside the core joins and matter constantly in practice: the existence joins the optimizer synthesizes when flattening subqueries (§03), and the distribution strategies that decide how a join runs across many workers or GPUs (§16). Both are less about a new algorithm than about how the algorithms above are wired up.

Semi-join and anti-join

These are existence tests dressed as joins. A semi-join (the physical form of EXISTS / IN) emits each outer row at most once if it has any match in the inner — never inner columns, never a duplicate per extra match — so the implementation can stop at the first hit. An anti-join (NOT EXISTS / NOT IN) emits exactly the outer rows with no match. The optimizer builds both on top of a hash or merge join, and they are the targets of subquery decorrelation: turning a correlated EXISTS subquery into a single set-based semi-join is one of the highest-value rewrites in §03, replacing a per-row re-execution with one pass.

Broadcast vs shuffle (partitioned) join

The defining choice for any distributed or multi-GPU join (§16). A broadcast join replicates the small side to every worker, which then joins its local shard of the big side against the full small side — the big table never moves across the network. A shuffle (partitioned) join hash-partitions both sides by the join key across workers so that matching keys co-locate, then joins locally; its cost is dominated by the all-to-all exchange (FIG 16). The rule is a data-size threshold: broadcast when one side is small enough that shipping small × workers bytes beats reshuffling everything; shuffle when both sides are large. Getting this wrong — broadcasting a table that's too big, or shuffling when a broadcast would do — is one of the most common causes of a blown distributed query, and the cardinality estimate (§04) is what the decision rides on.

A4

Aggregation — grouping and rollup

The last operator family: collapsing many rows into per-group results for GROUP BY and aggregates. The same hash-vs-sort dichotomy that organizes joins organizes aggregation, plus a third pattern — partial aggregation — that is the workhorse of parallel and GPU group-by (§14) and the reason a billion-row aggregation can finish by moving only kilobytes between workers.

Hash aggregation

The default GROUP BY. Maintain a hash table keyed by the grouping columns; for each input row, find its group's slot and fold the row into a running accumulator (SUM, COUNT, MIN, …); when the input is exhausted, emit one row per slot. No ordering requirement and a single pass — O(rows) — sized to the number of distinct groups. Its failure mode is too many groups to fit in memory, which forces a grace-style spill/partition exactly as with hash join; and its output order is arbitrary. On the GPU this becomes the shared-memory-first, atomics-on-collision pattern of FIG 14.

Sort-based (streaming) aggregation

Sort the input by the grouping key (or inherit an existing sort) so that all rows of a group become contiguous, then stream with a single accumulator: fold rows while the key is unchanged, emit and reset at each boundary. Memory is O(1) in the number of groups — only the current group's running value is held — and the output emerges in sorted order. As with sort-merge join, the sort is the cost and it is free when the input already arrives sorted (§07); it also pairs naturally with anything downstream that wants the group key ordered. The hash-vs-sort aggregation choice mirrors the hash-vs-merge join choice exactly: hashing avoids the sort but needs group-sized memory; sorting needs order but runs in constant space.

Partial / two-phase aggregation

The move that makes parallel and distributed aggregation cheap, and the single most important pattern for GPU group-by (§14–16). Split the aggregation in two: each worker/partition first computes a partial aggregate over its local rows (pre-aggregation), then a single final step merges the partials per key. Because the partial step collapses many rows into one-per-group before any data crosses the network or the PCIe straw (FIG 13), only the tiny partial results are moved — a billion local rows can become a few thousand bytes of partials. The catch is decomposability: the aggregate must be algebraic, expressible as combine-of-partials. SUM/COUNT/MIN/MAX are trivially so; AVG works by carrying (sum, count) and dividing at the end; holistic aggregates like COUNT(DISTINCT) and MEDIAN don't decompose and need a different (often shuffle-heavy) plan.

The atlas in one line

Every plan is built from three verbs — read a table (scan family), match two tables (join family), collapse rows into groups (aggregation family) — and within each family the same axis recurs: hash (no order, costs memory) versus sort/order (costs a sort, runs lean) versus index/probe (costs a structure, wins when one side is small or selective). The optimizer's whole job (§04–07) is choosing a point on that axis for every node given the cardinalities; moving to the GPU keeps the axis intact but reprices it — sequential reads and parallel probes get cheaper, random gathers and per-partition shuffles get relatively dearer, and partial aggregation goes from helpful to essential.
