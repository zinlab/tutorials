Concurrency & Parallelism

A ground-up treatment of how programs run many things at once — the conceptual split, the four execution models, their kernel-level machinery, and the throughput / latency calculus that decides which one you actually want.

SECTION 00The Core Distinction

The single most important idea, and the one most often blurred: concurrency and parallelism are not the same axis. Concurrency is about structure — composing a program out of independently progressing tasks. Parallelism is about execution — physically running computations at the same instant. Rob Pike's slogan is the cleanest: concurrency is dealing with many things at once; parallelism is doing many things at once.

Concurrency is a property of how you decompose the problem. Parallelism is a property of the machine you run it on. The two are orthogonal, which gives four quadrants:

Not parallel (1 core)Parallel (N cores)
Not concurrentA single sequential program. One task, one timeline.SIMD / a vectorized loop — one logical task, many lanes firing together. Data parallelism with no task structure.
ConcurrentAn async event loop or threads time-sliced on one core. Tasks interleave but never overlap in time.Threads or processes each pinned to a core. Tasks both interleave and overlap. The full picture.

The figure below is the canonical way to see it. Concurrency slices one timeline into interleaved chunks; parallelism stacks timelines. You can have either without the other.

The load-bearing consequence

Concurrency is what you write; parallelism is what the runtime grants you if hardware and the scheduler allow. A well-structured concurrent program can light up 64 cores or run on one — same code. This is why "should I use threads or async?" is the wrong first question. The right first question is "is my bottleneck computation or waiting?" (Section 08). That answer dictates whether you even need parallelism, and only then which model delivers it.

Why the confusion is expensive

Treating them as one axis produces two classic mistakes. First: reaching for parallelism (more processes) when the bottleneck is I/O wait — you pay heavy context-switch and memory cost for cores that mostly sit idle. Second: reaching for concurrency (async) expecting it to speed up a CPU-bound kernel — interleaving compute on one core makes it slower, not faster, because you've added scheduling overhead to work that was never waiting. The rest of this manual is, in effect, a careful walk through which mechanism gives you which property and at what cost.

SECTION 01The Execution Substrate

Every model below is a way of multiplexing tasks onto a fixed set of physical execution resources. You can't reason about the models without first being precise about what those resources are. There are three layers, from silicon up.

Layer 1 — Hardware execution units

A physical core has its own ALUs, register file, and L1/L2 cache; it retires instructions independently. This is the only thing that delivers true parallelism. Simultaneous multithreading (Intel Hyper-Threading, AMD SMT) exposes one physical core as two logical cores that share execution units but keep separate architectural state — so a stall on one (cache miss) lets the other use the idle pipeline slots. SMT improves throughput on mixed workloads but does not double compute; two SMT siblings contending for the same FMA units run at well under 2×.

Latency hiding is the recurring theme

SMT, GPU warp scheduling (Section 09), and async I/O are all the same trick at different scales: when one unit of work stalls on a long-latency event (memory, network), switch to other ready work so the execution resource stays busy. Keep this lens — most of "concurrency" is latency hiding in disguise.

Layer 2 — OS abstractions: process & kernel thread

The kernel scheduler doesn't see your tasks; it sees kernel-schedulable entities. On Linux that's the task_struct — the unit the CFS/EEVDF scheduler places on cores. A process is a task with its own virtual address space (its own page tables), file-descriptor table, and signal handlers. A kernel thread is a task that shares the address space and fd table of its peers but has its own stack and register state. On Linux both are created by clone(); the flags decide how much is shared:

// fork(): a new process — nothing shared (copy-on-write pages)
clone(0);
// pthread_create(): a new thread — share VM, files, fd table, signals
clone(CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD);

That single line is the entire process-vs-thread distinction at the syscall level. Everything downstream — IPC cost, context-switch cost, the GIL, data-race exposure — follows from which bits are set.

Layer 3 — User-space tasks

Coroutines, futures, goroutines, green threads — these are invisible to the kernel. They live entirely inside your process's memory and are multiplexed onto kernel threads by a user-space scheduler (an event loop or runtime). Because the kernel never sees them, switching between them costs no syscall, no mode transition, no TLB churn — which is exactly why you can have millions of them. This layer is the difference between "thousands of connections" and "ten million connections."

SECTION 02Multiprocessing

Multiprocessing runs work across multiple OS processes, each with its own private virtual address space. Nothing is shared by default — that isolation is both the headline benefit (a crash or memory corruption in one process can't touch another; one runaway process can be killed cleanly) and the headline cost (communicating means crossing an address-space boundary).

Low-level mechanics

On Unix a process is born from fork(), which conceptually duplicates the parent. The duplication is lazy: copy-on-write. The child's page tables initially point at the same physical frames as the parent, all marked read-only. The first write to any page traps into the kernel, which copies that single page and remaps it private. So fork() of a 10 GB process is near-instant and only pages that are actually mutated get copied. exec() then optionally replaces the whole image with a new program. Each process carries its own page tables, so the same virtual address in two processes maps to different physical memory.

Inter-process communication (IPC)

MechanismHow it worksCost / notes
Shared memorymmap / shm_open a region into both address spacesFastest — zero copy after setup; you must add your own synchronization
Pipes / FIFOsKernel-buffered byte stream, one-directionalTwo copies (user→kernel→user); simple, stream-oriented

The practical upshot: any time you "send an object" between processes you serialize it (Python pickles it), push the bytes through the kernel, and deserialize on the far side. That copy is the dominant cost and the reason multiprocessing shines for coarse-grained, compute-heavy tasks with little data exchange, and suffers for fine-grained chatty ones.

Distributed-systems connection

The IPC table is the single-machine projection of the same design space you navigate in a GPU exchange / shuffle: shared memory ≈ same-NUMA-node zero-copy, Unix sockets ≈ same-host loopback, TCP ≈ cross-node, and your UCX path ≈ the RDMA/NVLink fast lane that bypasses copies entirely. "Avoid the copy, avoid the boundary crossing" is the same cost model whether the boundary is a page table or a NIC.

Python specifically

Because each process is a separate interpreter with its own GIL (Section 03), the multiprocessing module is the standard way to get true CPU parallelism in CPython. The tradeoffs: process start-up cost (a fresh interpreter), and the serialization tax on every argument and return value. The fork start method is cheap (COW) but unsafe with threads; spawn is safe but re-imports your module. shared_memory and memory-mapped arrays let you skip the pickle tax for large buffers.

Throughput / latency profile: high per-task startup and communication latency; excellent aggregate throughput for embarrassingly parallel CPU work; strongest fault isolation of any model.

SECTION 03Multithreading

Threads live inside one process and share its entire address space — heap, globals, code, open files. Each thread has only its own stack, register set, and thread-local storage. Sharing memory is the point (communication is just reading a shared variable, no copy) and also the peril (every shared mutable byte is a potential data race).

Threading models — how user threads map to kernel threads

ModelMappingConsequence
1:1Each thread = one kernel threadLinux NPTL, Windows, modern default. Kernel schedules them across cores → true parallelism. A blocking syscall blocks only that thread.
N:1Many user threads on one kernel threadOld "green threads." No parallelism; one blocking syscall stalls all threads. Cheap switches though.
M:NM user threads multiplexed over N kernel threadsGo, Erlang, Java virtual threads. Best of both — cheap switches and parallelism — but the runtime scheduler is complex (Section 05).

Context switch — cheaper than a process, not free

Switching between two threads of the same process skips the page-table swap, so the TLB stays warm — a real saving over a process switch. But it still costs a trap into the kernel, a run through the scheduler, and save/restore of register state, landing in the low microseconds (Section 06). The hidden cost is often cache pollution: the incoming thread's working set evicts the outgoing one's.

Synchronization primitives

Shared mutable state demands coordination. The toolbox, roughly from cheapest to most general:

  • Atomics — single read-modify-write ops (CAS, fetch-add) the hardware guarantees indivisible; the foundation everything else builds on.
  • Mutex — mutual exclusion. On Linux backed by a futex: the uncontended fast path is a userspace atomic CAS with no syscall; only on contention does it trap into the kernel to sleep the waiter.
  • Spinlock — busy-waits instead of sleeping; wins only when the critical section is shorter than a context switch.
  • RWLock — many readers or one writer; good for read-heavy data.
  • Semaphore — a counter; gate N concurrent holders.
  • Condition variable — sleep until a predicate holds; always paired with a mutex.
  • Barrier — all threads wait until everyone arrives; the backbone of bulk-synchronous parallel loops.

The GIL — why Python threads don't parallelize CPU work

CPython protects its internal state (reference counts, object memory) with one process-wide lock, the Global Interpreter Lock. Only one thread executes Python bytecode at a time. The lock is released around blocking I/O and inside many C extensions (NumPy heavy loops, for instance). The consequences are sharp and worth memorizing:

I/O-bound → threads help

While one thread waits on a socket the GIL is released, so others run. Real concurrency, no parallelism needed.

CPU-bound → threads don't help

Pure-Python compute holds the GIL; threads just time-slice one core. Reach for multiprocessing or a C/CUDA extension.

Moving target

The free-threaded build (PEP 703, experimental from CPython 3.13 via a no-GIL interpreter) removes this restriction, letting threads run Python bytecode in parallel — at the cost of finer-grained internal locking and some single-thread overhead. If you're benchmarking Python threading for CPU work, check which interpreter you're on; the answer changed.

False sharing — the silent throughput killer

Coherence operates at cache-line granularity (64 bytes on x86). If two threads on two cores write to different variables that happen to land on the same line, the MESI protocol bounces that line's exclusive ownership back and forth on every write — invalidating each core's copy though there's no logical sharing at all. The fix is padding/alignment so hot per-thread variables occupy separate lines. This is a top suspect whenever a parallel loop scales far worse than core count predicts.

Throughput / latency profile: low switch latency, zero-copy sharing, true parallelism in non-GIL runtimes — paid for with the full weight of data races, locks, and memory-model reasoning.

SECTION 04Coroutines & Async

A coroutine is a function that can suspend itself mid-execution and later resume exactly where it left off, preserving its local state. Many coroutines are multiplexed onto a single thread by an event loop, cooperatively: a coroutine runs until it voluntarily yields — at an await — and only then can another run. This is the C10K answer: tens of thousands of in-flight connections on one OS thread, because each suspended coroutine is just a small heap object, not a kernel thread with an 8 KB stack.

Cooperative, not preemptive

The scheduler can only switch at await points. The blessing: between two awaits your code runs atomically with respect to other coroutines, so data touched only there needs no lock. The curse: a coroutine that does heavy CPU work without awaiting starves the entire loop — every other connection freezes until it yields. Async buys I/O concurrency, never CPU parallelism.

How async/await actually compiles

In the stackless world the magic is a source-to-source transform. The compiler takes your linear-looking async function and turns it into a struct holding (a) a state discriminant — "which await am I parked at" — and (b) exactly the locals that must survive across suspension. Driving it forward is poll:

// Rust mental model — an async fn becomes roughly:
enum ReadAndProcess {
    Start,
    AwaitingRead { buf: Vec<u8>, fut: ReadFut },   // locals live across .await
    Done,
}
impl Future for ReadAndProcess {
    fn poll(&mut self, cx: &mut Context) -> Poll<Output> {
        match self.state {
            Start => { // kick off read, store fut, move to AwaitingRead }
            AwaitingRead => match fut.poll(cx) {
                Pending => Poll::Pending,    // yield to executor, register a Waker
                Ready(n) => { // process, move to Done }
            }
        }
    }
}

The executor (Tokio, asyncio) repeatedly polls ready futures. A future that can't progress returns Pending and registers a Waker; when the underlying I/O is ready, the Waker tells the executor to poll it again. Nothing spins — the loop sleeps in the kernel until something is ready.

The I/O multiplexing substrate

What does the event loop sleep on? A readiness syscall that watches many file descriptors at once: epoll (Linux), kqueue (BSD/macOS), IOCP (Windows). One thread registers thousands of sockets and gets woken with the list that became ready — O(ready), not O(total). The modern step-change is io_uring: paired submission/completion ring buffers shared with the kernel, so you batch many I/O operations with few (or zero) syscalls and get true async completion rather than readiness. This is the substrate that makes a single core handle enormous connection counts.

The colored-function problem

async is contagious. An async function can only be awaited from another async function, so "color" propagates up the entire call stack; bridging sync and async code requires explicit run-on-loop / block-on shims. This is a real ergonomic tax and a frequent source of architectural friction — worth naming because it's a design cost, not a bug.

Throughput / latency profile: the lowest switch cost and highest connection density of any model for I/O-bound work; near-useless for CPU-bound work; cooperative scheduling means one bad actor can wreck tail latency for everyone.

SECTION 05Green Threads & the M:N Bridge

The two prior models force a choice: kernel threads give parallelism but are heavy and few; coroutines are light and many but single-threaded and cooperative. M:N green threads refuse the choice — a user-space runtime multiplexes millions of lightweight tasks across a small pool of kernel threads, and the runtime, not the kernel, decides who runs where.

Go's GMP scheduler — the reference design

Go names three entities: G (a goroutine — ~2 KB initial, growable stack), M (an OS thread, the actual execution vehicle), and P (a processor — a scheduling context, count fixed by GOMAXPROCS, that owns a local run queue of Gs). An M must hold a P to run Go code. Two mechanisms make it fast and fair:

  • Work stealing — when a P's local queue empties, it steals half of another P's queue. Load balances without a global lock on the common path.
  • The netpoller — blocking-looking network calls are quietly registered with epoll; the goroutine is parked and its M freed to run other Gs. So idiomatic blocking code gets async I/O underneath for free — no colored functions.

Since Go 1.14, scheduling is also asynchronously preemptive: the runtime can interrupt a long-running goroutine via a signal, so one CPU-bound G can't starve the others — closing the cooperative-scheduling gap that pure coroutines leave open.

Java's parallel evolution

Virtual threads (Project Loom, stable in JDK 21) are the same idea for the JVM: cheap user-mode threads that mount onto a small pool of carrier (platform) threads and unmount when they hit blocking I/O, parking instead of holding an OS thread. Write plain blocking code; get coroutine-scale concurrency. The industry has converged here: blocking-style ergonomics, async-style scaling.

Throughput / latency profile: coroutine-class memory and switch cost, thread-class parallelism, preemptive fairness — at the price of a heavy runtime you don't control and that can interact awkwardly with C interop and tight latency tails (GC pauses, scheduler hops).

SECTION 06Context-Switch Economics

Almost everything above reduces to one number: what does it cost to switch from running task A to running task B? The cheaper the switch, the more tasks you can afford and the finer-grained you can decompose. The costs span roughly four orders of magnitude, and that span is the reason the models exist.

Switch typeOrder of magnitudeWhat it touches
Plain function call~1 nsPush/pop a few registers; nothing leaves the core
Stackless coroutine resume~10–100 nsRestore a handful of locals + advance a state enum; pure userspace
Stackful fiber / goroutine~100–200 nsSwap stack pointer + callee-saved registers; userspace
Kernel thread switch (same process)~1–5 µsTrap to kernel, run scheduler, save/restore state; TLB stays warm
Process switch~several µs directAll of the above + page-table swap → TLB flush (absent PCID/ASID)
Indirect (cache/TLB cold)up to ~10s of µsThe incoming task's working set must be re-fetched from L2/L3/RAM
Read the orders of magnitude, not the digits

Exact numbers vary wildly by CPU, kernel, and whether caches stay warm — the direct register-swap cost of a switch is often dwarfed by the indirect cost of a cold cache and flushed TLB afterward. The durable takeaway is the spread: userspace switches are ~100× cheaper than kernel switches. That single fact is why you can have ten million coroutines but only thousands of threads, and why M:N runtimes (Section 05) work so hard to keep scheduling in userspace.

SECTION 07Throughput vs Latency

These are the two metrics every concurrency decision optimizes, and they routinely trade against each other. Precise definitions first:

  • Latency — time to complete one operation, end to end. Units: seconds. "How long does this request take?"
  • Throughput — operations completed per unit time. Units: ops/s. "How many requests per second can the system clear?"

The naïve intuition is that they're reciprocals — halve latency, double throughput. That's true only for a strictly serial system. The instant you add concurrency they decouple, and the relationship is governed by one of the most useful laws in systems work.

Little's Law — the bridge between the two

For any stable system: L = λ × W, where L is the average number of in-flight items, λ the throughput (arrival = completion rate at steady state), and W the average latency. Rearranged for design: required concurrency = throughput × latency.

Worked example — this is the formula you'll actually use

You need 50,000 req/s and each request spends 20 ms waiting on a database (W = 0.02 s). Then L = 50,000 × 0.02 = 1,000 concurrent requests in flight. With blocking kernel threads that's 1,000 threads × ~8 KB stacks + scheduler pressure. With coroutines it's 1,000 cheap heap objects on a handful of threads. Little's Law tells you the concurrency a target throughput demands; the switch-cost ladder (Section 06) tells you which model can afford it.

The fundamental tradeoffs

Batching

Group work to amortize per-op overhead → throughput ↑. But items wait for the batch to fill → individual latency ↑. The classic dial.

Pipelining

Overlap stages of independent items → throughput ↑ with no increase in any single item's latency. Strictly better when applicable.

Buffering / queues

Absorb bursts → throughput stays high under load spikes. But queued items sit waiting → latency ↑, and deep queues hide overload.

Replication / parallelism

More workers → throughput ↑ (Gustafson). Doesn't shrink one item's latency unless you also split that item (Amdahl).

GPU connection

This is exactly the batching-vs-latency dial you live with on accelerators: large batches maximize occupancy and arithmetic throughput but inflate per-request latency, while small batches keep latency low at the cost of underutilized SMs. A GPU is a throughput machine that deliberately sacrifices single-thread latency — Section 09 makes that architectural bargain explicit.

Amdahl vs Gustafson — two views of speedup

If a fraction p of work is parallelizable and (1−p) is inherently serial, then on N workers:

// Amdahl — FIXED problem size (a latency view)
speedup ≤ 1 / ( (1−p) + p/N )      // capped by the serial part; N→∞ ⇒ 1/(1−p)

// Gustafson — problem GROWS with resources (a throughput view)
speedup = N − (1−p)·(N−1)        // scales nearly linearly with N

They aren't contradictory; they answer different questions. Amdahl: "I have this job — how much faster with more cores?" (a latency question, and the serial fraction is a hard ceiling). Gustafson: "I have more cores — how much more work can I do in the same time?" (a throughput question, far more optimistic). Which law applies depends on whether you're shrinking a fixed job's latency or growing aggregate throughput.

Tail latency — averages lie

The mean is a poor summary; what users feel is the tail — p99, p99.9. Two reasons it dominates real systems. First, queueing blows up near saturation: in an M/M/1 queue, wait time ≈ 1/(µ−λ), which diverges as utilization → 100%. Run a server at 95% and a tiny load bump sends latency vertical — there's a "knee" you must stay left of. Second, fan-out amplifies tails: a request that touches 100 services waits for the slowest of 100, so even a 1% per-service p99 means the overall request almost always hits someone's slow path. Tail-tolerant design (hedged requests, timeouts, load shedding) is therefore a first-class concern, not polish.

SECTION 08The Deciding Axis: CPU-bound vs I/O-bound

Return to the claim from Section 00: the first question is never "threads or async?" — it's "where does my time go: computing, or waiting?" This single axis selects the model.

CPU-boundI/O-bound
BottleneckArithmetic / the cores themselvesWaiting on network, disk, another service
What helpsParallelism — actually use more coresConcurrency — overlap the waits
Right toolMultiprocessing, true (non-GIL) threads, SIMD, GPUAsync / coroutines, thread pools, green threads
Concurrency without parallelismUseless — adds overhead to work that never waitedThe whole game — one core serves thousands of idle-waiting connections
Python noteGIL blocks threads → use multiprocessing / C extGIL released on I/O → threads and asyncio both work

The deep reason: for I/O-bound work the core is idle most of the time, so a single core can host enormous concurrency just by switching to whatever's ready — no extra cores needed, and the cheapest switch (coroutines) wins. For CPU-bound work the core is always busy, so the only lever is more cores running simultaneously — pure parallelism — and interleaving on one core merely adds scheduling cost. Mixed workloads split: do the I/O concurrently and hand the CPU-heavy chunks to a process pool or an accelerator.

SECTION 09Data Parallelism: SIMD & SIMT

Everything so far has been task parallelism — different tasks progressing independently. The orthogonal flavor is data parallelism — one operation applied across many data elements at once. Flynn's taxonomy frames it: SISD (one instruction, one datum — a scalar core), SIMD (one instruction, many data), MISD (rare), MIMD (independent cores — ordinary multithreading).

SIMD — vector lanes on the CPU

A single instruction operates on a wide register holding several values: SSE/AVX/AVX-512 on x86, NEON/SVE on ARM. One vaddps adds eight (or sixteen) floats in one issue. Compilers auto-vectorize tidy loops; hot kernels are often hand-written with intrinsics. SIMD multiplies throughput per core with no task-scheduling cost at all — there's only one instruction stream — but requires regular, contiguous, branch-light data.

SIMT — the GPU model

GPUs run Single Instruction, Multiple Threads. Threads are grouped into warps (32 on NVIDIA) that execute the same instruction in lockstep across 32 lanes. Two architectural consequences define GPU performance:

  • Branch divergence — if threads in a warp take different paths at a conditional, the hardware serializes the paths (executing each with the other lanes masked off), so divergent control flow costs throughput. Keep warps coherent.
  • Latency hiding by oversubscription — this is the central bargain. A GPU does not minimize the latency of any single thread; instead it keeps thousands of warps resident and, whenever one warp stalls on a memory access, instantly switches to another ready warp (a zero-cost switch, because all register state stays resident). With enough warps in flight, memory latency is fully hidden and the arithmetic units never idle.
The throughput-vs-latency bargain, made physical

A CPU core is a latency machine — deep caches, branch prediction, out-of-order execution all to make one thread fast. A GPU is a throughput machine — it accepts high per-thread latency and buries it under massive parallelism, spending its transistor budget on ALUs and registers rather than latency-reduction. This is Section 07's tradeoff cast in silicon, and it's why the right unit of work and the right batch size matter so much on accelerators: you're feeding a machine that's only fast in aggregate.

Profile: SIMD/SIMT deliver the highest throughput per watt and per transistor of any model for regular numeric work, with effectively zero per-element scheduling cost — at the price of demanding data regularity, coherent control flow, and (on the GPU) enough parallel work to hide latency. They sit under the task-parallel models: a thread or process is what launches and feeds the vector units.

SECTION 10Memory Models & Synchronization

Once two threads share memory, a question arises that has no answer at the single-thread level: in what order do one thread's writes become visible to another? The intuitive answer — "the order I wrote them" (sequential consistency) — is not what real hardware provides. Both the compiler and the CPU reorder memory operations for performance.

Hardware reordering

x86 offers a relatively strong model (TSO — Total Store Order: loads aren't reordered past loads, stores past stores, but a store can be delayed past a later load via the store buffer). ARM and POWER are weakly ordered — almost anything can be reordered absent explicit barriers. Code that "works on my x86 laptop" can break on ARM purely from this. The remedy is memory barriers (fences) that forbid specific reorderings.

The C++ memory-order vocabulary

Atomics let you specify exactly how much ordering you need — and paying for no more than that is a real performance lever:

OrderingGuaranteeUse
relaxedAtomicity only; no ordering vs other opsCounters/stats where order doesn't matter
acquireNo later read/write moves before this loadLock acquire; reading a published pointer
releaseNo earlier read/write moves after this storeLock release; publishing data
acq_relBoth, for read-modify-writeCAS in a lock-free structure
seq_cstSingle global total order (the default)When in doubt; correct but slowest

The key relation is happens-before: a release store paired with an acquire load of the same variable establishes that everything before the release is visible after the acquire. A data race — two unsynchronized accesses, at least one a write — is undefined behavior in C/C++/Rust, not merely a wrong value; the compiler may assume it never happens.

Lock-free & wait-free

Beyond locks lies a hierarchy of non-blocking progress guarantees, built directly on atomic CAS:

  • Obstruction-free — a thread makes progress if it runs in isolation.
  • Lock-free — some thread always makes progress system-wide (no global stall), though individual threads may starve.
  • Wait-free — every thread makes progress in bounded steps (strongest; rare and hard).

These avoid the pathologies of locks (a thread descheduled inside a critical section can't block others), but bring their own hazards — most famously the ABA problem (a value changes A→B→A and a naïve CAS thinks nothing happened), addressed with tagged pointers or hazard pointers. For read-mostly data, RCU (read-copy-update) lets readers proceed with zero synchronization while writers publish new versions and defer reclamation — the technique behind much of the Linux kernel's scalability.

Rule of thumb

Reach for lock-free only with measurement in hand. A well-implemented mutex with an uncontended futex fast path is fast and correct; hand-rolled lock-free code is a notorious source of subtle, platform-specific, once-a-week bugs. Correctness first, then profile, then optimize the contended path you can prove matters.

SECTION 11Schedulers

A scheduler decides which ready task runs next on each execution resource. The defining choice is when a switch can happen.

Preemptive vs cooperative

Preemptive

The scheduler can forcibly suspend a task at any point — a timer interrupt fires every quantum. Used by OS kernel schedulers (CFS/EEVDF on Linux) and Go 1.14+. Guarantees fairness: no task can monopolize a core. Cost: a switch can land mid-anything, so shared state always needs synchronization.

Cooperative

A task runs until it voluntarily yields (an await). Used by async event loops. Cheaper and lets you reason about atomicity between yields — but a non-yielding task starves everyone (Section 04's curse).

Work-stealing — the parallel-runtime workhorse

The dominant design for parallel task runtimes (Cilk, Intel TBB, Java Fork/Join, Rust's Rayon, Go, Tokio). Each worker thread owns a double-ended queue of tasks. It pushes and pops its own work from one end (LIFO — hot in cache, no contention). When a worker runs dry, it steals from the other end of a random victim's deque (FIFO — grabs the oldest, likely-largest task). The properties are why it's everywhere:

  • Locality — you mostly run your own recently-created tasks, warm in cache.
  • Low contention — owner and thief touch opposite ends; the common path needs no lock.
  • Automatic load balancing — idle workers find work themselves; no central dispatcher to bottleneck.

This is the engine under "just spawn tasks and the runtime figures out the cores" — the user writes fine-grained concurrency, the work-stealer maps it onto whatever parallelism the hardware offers, tying the whole manual back to Section 00's thesis.

The four Coffman conditions for deadlock

All four must hold simultaneously: mutual exclusion (resources aren't shareable), hold-and-wait (you keep what you have while waiting for more), no preemption (resources can't be forcibly taken), and circular wait (a cycle in the who-waits-for-whom graph). Deadlock is impossible if you break any one. The most practical break is the last: impose a global ordering on lock acquisition so no cycle can form.

SECTION 13Decision Framework

The whole manual, compressed into the order you should actually ask the questions.

  1. Is the bottleneck compute or waiting? (Section 08) — this comes first and selects everything after it. Measure; don't guess.
  2. If CPU-bound, do you need many cores or one fast lane? Regular numeric data → SIMD/GPU (Section 09). Coarse independent chunks → process pool or non-GIL threads. Concurrency alone will not help.
  3. If I/O-bound, how many concurrent waits? Use Little's Law (Section 07): concurrency = throughput × latency. Thousands+ → coroutines/async or green threads. Dozens → a thread pool is simpler and fine.
  4. What can you afford to switch? The cost ladder (Section 06) tells you whether your concurrency target fits in threads or demands userspace tasks.
  5. Are you optimizing one item's latency or aggregate throughput? (Section 07) — they trade off. Pick the metric, mind the saturation knee, design for the tail.

The one-paragraph version

Concurrency is how you structure work; parallelism is whether it runs simultaneously. Choose the model by your bottleneck — parallelism (processes, non-GIL threads, SIMD, GPU) for compute, concurrency (async, green threads) for waiting. The switch-cost ladder caps how many tasks each model can afford, Little's Law tells you how many you need, and almost every tuning decision is a deliberate trade between latency and throughput — made while staying left of the saturation knee and designing for the tail.
