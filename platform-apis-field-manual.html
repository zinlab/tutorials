FRAME Chapter I

The frame · platform as product.

The shift Helm charts and Operators stop being deployment tools and start being products. They have users, contracts, versions, deprecations, support windows.

The rule Every internal API has the same liabilities as a public API. Treat them that way from day one or pay later.

Up through Issue 09 we treated Helm charts and Operators as how-it-works mechanics — render templates, run reconciliation loops, deploy to a cluster. This issue treats them as product surfaces. The platform team has internal customers; the customers consume APIs; the APIs are Helm charts (config-as-data), Operators (control-as-CRDs), and the broader platform surface (Crossplane compositions, Backstage templates, golden paths). The discipline of designing those APIs well — versioned, schema'd, documented, escape-hatched, deprecation-aware — is what separates a platform that scales from a platform that becomes a ticketing queue.

What "platform as product" actually means

Three working definitions, in order of operational maturity:

  • Platform-as-tool: the platform team writes scripts and manifests; teams copy them. Knowledge transfers by Slack DM and tribal memory. Fine for <10 services; unsustainable beyond.
  • Platform-as-service: the platform team operates centralized services (CI, registries, observability) and the rules for using them. Teams self-serve via UI / API. Common at mid-scale.
  • Platform-as-product: the platform team treats its capabilities as a product, with named owners, semver-versioned APIs (Helm chart versions, CRD versions, Backstage template versions), release notes, deprecation policies, SLOs on uptime + provisioning latency, and feedback loops (surveys, DX metrics).

2026 data suggests this shift is now mainstream. DORA's 2025 report finds that internal platform quality is the strongest predictor of AI value delivery; Gartner projects 80% of large engineering orgs have dedicated platform teams by year-end.

Four product disciplines that don't transfer from "ops"

API design. Every Helm values key, every CRD field, every Backstage template parameter is part of a contract. Once exposed, breaking changes are expensive. Add fields easily; remove or rename them carefully and with deprecation windows.

Versioning. Helm charts have a version in Chart.yaml. CRDs have v1alpha1 through v1. Platform APIs (Crossplane XRDs, Backstage templates) need their own versioning policy. Without it, every change risks every consumer.

Documentation. README at the top of every chart and operator. Values schemas as JSON Schema. Backstage TechDocs co-located with the source. API references generated from CRD OpenAPI. The documentation is the API; if it isn't documented, you're going to answer the same questions forever.

Adoption metrics. Counting deployments isn't enough. Track time-to-first-deploy (TTFD) for a new service, time-to-resolution for platform support requests, NPS, "how often did you have to file a ticket this month." Without these, the platform optimizes for things nobody asked for.

The three surfaces

This issue covers three layers of platform API, increasing in abstraction:

  • Helm chart (Chapters II–VI). Config-as-data: rendered manifests with values and templates. Solves "how do I deploy this app". The lowest abstraction.
  • Operator (Chapters VII–X). Control-as-API: CRDs + reconciliation loop. Solves "how do I keep this app correct over time". A higher abstraction with operational intelligence built in.
  • Platform API (Chapters XI–XIII). Domain-as-API: a Crossplane XRD or a Backstage template that hides the underlying Helm + Operator + cloud resources behind a developer-friendly facade. The highest abstraction.

A mature platform has all three. The Helm chart deploys the Postgres operator; the Postgres operator manages the actual database; the Crossplane Database claim hides both behind a 10-line YAML developers can use without knowing Postgres exists.

The escape-hatch principle

The best platforms are opinionated AND escape-hatched. A developer who needs the standard path gets it in one click. A developer who needs something unusual can drop down to the next layer (Helm values, raw CRD spec, raw kubectl). Platforms that lock the second case out generate workarounds; platforms that have no opinionated path generate inconsistency. You need both ends.

HELM Chapter II

Chart anatomy · templates, values, releases.

The shape Chart.yaml + values.yaml + values.schema.json + templates/*.yaml + _helpers.tpl. Three of those five are non-negotiable in 2026.

The mental model A chart is a function. values are arguments. templates are the body. A release is the invocation.

Issue 08 covered Helm at the user level. This chapter zooms in on the chart-author surface — the seven pieces that compose a production chart and the design choices that make a chart maintainable or a maintenance burden.

The canonical chart layout

myapp/
├── Chart.yaml                    # identity + dependencies + version
├── Chart.lock                    # resolved subchart digests (commit this)
├── values.yaml                   # defaults — the documented surface
├── values.schema.json            # JSON schema — validation
├── README.md                     # the chart's docs
├── README.md.gotmpl              # auto-generated from values (helm-docs)
├── templates/
│   ├── _helpers.tpl              # reusable named templates
│   ├── deployment.yaml
│   ├── service.yaml
│   ├── serviceaccount.yaml
│   ├── ingress.yaml              # gated by .Values.ingress.enabled
│   ├── hpa.yaml                  # gated by .Values.autoscaling.enabled
│   ├── poddisruptionbudget.yaml
│   ├── networkpolicy.yaml
│   ├── servicemonitor.yaml       # gated by Prometheus Operator CRD presence
│   ├── configmap.yaml
│   ├── tests/
│   │   └── connection-test.yaml  # helm test — smoke after install
│   └── NOTES.txt                 # printed on install
├── crds/                         # raw CRDs (special install behavior — see below)
└── charts/                       # bundled subcharts (umbrella case)

Chart.yaml — identity and constraints

apiVersion: v2
name: myapp
description: "Production-grade chart for myapp REST API"
type: application                  # or 'library' for shared templates
version: "2.4.1"                     # SEMVER for the chart — NOT the app version
appVersion: "v1.18.0"                # the app version this chart deploys (informational)
kubeVersion: ">=1.28.0-0"             # minimum K8s version (helm install will refuse below)
home: https://example.com/myapp
sources:
  - https://github.com/myorg/myapp
maintainers:
  - { name: platform-team, email: platform@example.com }
annotations:
  artifacthub.io/changes: |                # shows in Artifact Hub changelog
    - kind: added
      description: NetworkPolicy template
    - kind: fixed
      description: PDB minAvailable now respects replica count
  category: "Integration and Delivery"
dependencies:
  - name: common
    version: "2.x.x"                  # SemVer range — patches OK, minors OK, no majors
    repository: file://../common      # or https://charts.example.com
  - name: postgresql
    version: "15.x.x"
    repository: https://charts.bitnami.com/bitnami
    condition: postgresql.enabled    # subchart toggleable from parent values
    alias: db                          # rename for cleaner values

Two non-obvious rules. First: quote the appVersion. Wrapping the version in quotes is highly recommended — leaving it unquoted can lead to parsing issues, since YAML interprets 1.0 as a floating point value, and a git commit SHA like 1234e10 as scientific notation. Second: chart version is independent of app version. A chart can bump 1.0.0 → 1.1.0 (added a new template) without the app changing. The app can bump v1.18.0 → v1.19.0 (new code) without the chart changing. The two version lines have different consumers and should be separately managed.

Template anatomy — Go templates plus Sprig

Helm renders Go templates with the Sprig template function library plus two Helm-specific functions: the include function allows you to bring in another template, and then pass the results to other template functions; the required function allows you to declare a particular values entry as required for template rendering — if the value is empty, the template rendering will fail with a user submitted error message.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "myapp.fullname" . }}
  labels:
    {{- include "myapp.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicaCount | default 3 }}
  selector:
    matchLabels:
      {{- include "myapp.selectorLabels" . | nindent 6 }}
  strategy:
    type: RollingUpdate
    rollingUpdate: { maxSurge: 25%, maxUnavailable: 0 }
  template:
    metadata:
      labels:
        {{- include "myapp.selectorLabels" . | nindent 8 }}
      annotations:
        # Force pod rollout when ConfigMap changes — the checksum pattern
        checksum/config: {{ include (print $.Template.BasePath "/configmap.yaml") . | sha256sum }}
    spec:
      serviceAccountName: {{ include "myapp.serviceAccountName" . }}
      securityContext:
        {{- toYaml .Values.podSecurityContext | nindent 8 }}
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}@{{ required "image.digest is required (pin by digest)" .Values.image.digest }}"
          imagePullPolicy: {{ .Values.image.pullPolicy | default "IfNotPresent" }}
          ports: [{ containerPort: {{ .Values.service.port }}, name: http }]
          resources: {{- toYaml .Values.resources | nindent 12 }}
          {{- with .Values.env }}
          env: {{- toYaml . | nindent 12 }}
          {{- end }}
          readinessProbe: {{- toYaml .Values.readinessProbe | nindent 12 }}
          livenessProbe:  {{- toYaml .Values.livenessProbe  | nindent 12 }}

Several patterns earn their keep here:

  • include "myapp.fullname" instead of hardcoding names — names become consistent across templates and override-able by one helper change.
  • {{ required "..." .Values.image.digest }} on the image digest — fails the install with a useful error if the consumer forgot to pin.
  • checksum/config: ... | sha256sum annotation — forces pod rollout when a referenced ConfigMap changes.
  • {{- toYaml ... | nindent N }} for nested maps — handles whitespace cleanly. Per Helm docs: using the {{- ... | nindent _n_ }} pattern makes it easier to read the include in context, because it chomps the whitespace to the left (including the previous newline), then the nindent re-adds the newline and indents the included content by the requested amount.

_helpers.tpl — the named-template stash

Every chart of meaningful size has a _helpers.tpl file with reusable named templates: fullname, labels, selectorLabels, serviceAccountName, image string assembly. These keep the rest of the templates clean.

{{/* Generate a short, DNS-safe fullname */}}
{{- define "myapp.fullname" -}}
{{- if .Values.fullnameOverride -}}
{{- .Values.fullnameOverride | trunc 63 | trimSuffix "-" -}}
{{- else -}}
{{- $name := default .Chart.Name .Values.nameOverride -}}
{{- printf "%s-%s" .Release.Name $name | trunc 63 | trimSuffix "-" -}}
{{- end -}}
{{- end -}}

{{/* Standard labels — applied to everything */}}
{{- define "myapp.labels" -}}
helm.sh/chart: {{ printf "%s-%s" .Chart.Name .Chart.Version | replace "+" "_" | trunc 63 | trimSuffix "-" }}
{{ include "myapp.selectorLabels" . }}
{{- if .Chart.AppVersion }}
app.kubernetes.io/version: {{ .Chart.AppVersion | quote }}
{{- end }}
app.kubernetes.io/managed-by: {{ .Release.Service }}
{{- end -}}

{{/* Selector labels — the subset used for matching (never include version) */}}
{{- define "myapp.selectorLabels" -}}
app.kubernetes.io/name: {{ include "myapp.name" . }}
app.kubernetes.io/instance: {{ .Release.Name }}
{{- end -}}

The selectorLabels vs labels split matters. Selector labels are used in Deployment.spec.selector and Service.selector; those are immutable after creation. Never include app.kubernetes.io/version in selectorLabels — it changes on every upgrade and breaks the selector. Always include it in labels (used for metadata, not selection).

NOTES.txt — the first-run experience

templates/NOTES.txt renders after install and shows the user what to do next: how to access the service, where to find credentials, what commands to run. The single highest-leverage piece of UX in a chart.

myapp {{ .Chart.AppVersion }} deployed as {{ .Release.Name }}.

  {{- if .Values.ingress.enabled }}
  ▸ Access at: https://{{ (index .Values.ingress.hosts 0).host }}
  {{- else }}
  ▸ Port-forward to access:
       kubectl port-forward -n {{ .Release.Namespace }} svc/{{ include "myapp.fullname" . }} 8080:{{ .Values.service.port }}
  {{- end }}

  ▸ Status:    helm status {{ .Release.Name }} -n {{ .Release.Namespace }}
  ▸ Smoke:     helm test {{ .Release.Name }} -n {{ .Release.Namespace }}
  ▸ Rollback:  helm rollback {{ .Release.Name }} -n {{ .Release.Namespace }}

tests/ — helm test as smoke gate

Templates inside templates/tests/ with the annotation "helm.sh/hook": test only run when a user invokes helm test RELEASE. The pattern: a single short-running pod that performs a real readiness check (curl the service, query the database, validate a token). Smoke test the install before declaring success.

The crds/ directory — special install

Files in crds/ get installed before everything else and are not templated, not removed on uninstall, and not upgraded. This is intentional: CRDs are cluster-scoped, often shared, and breaking them is dangerous. If your chart owns a CRD lifecycle, you generally manage CRDs separately (Operator pattern, Chapter VII) and let Helm install only the operator binary that consumes them.

The five-minute chart-quality test

Run helm lint, helm template . | kubectl apply --dry-run=server -f -, and helm install --dry-run. Run kubeconform or kube-linter against the rendered output. Render with --debug and read every YAML; any duplicated boilerplate is a candidate for _helpers.tpl. Run helm test against a fresh install. Five minutes; catches 80% of chart bugs.

HELM Chapter III

Library charts · the reuse layer.

What they are type: library charts. They render nothing themselves. They export named templates other charts use via include.

Why they matter One change in the library propagates to every consuming chart. Twenty services start sharing one definition of "what a production Deployment looks like."

When a platform team finds itself maintaining a dozen application charts, all repeating roughly the same Deployment + Service + Ingress + HPA + PDB + ServiceMonitor + NetworkPolicy boilerplate, the reuse problem becomes acute. Library charts are Helm's answer. A library chart provides utilities or functions for the chart builder. A library chart differs from an application chart because it is not installable and usually doesn't contain any resource objects.

The structure

common/
├── Chart.yaml          # type: library — the critical line
├── values.yaml         # defaults consumers can override
├── templates/
│   ├── _deployment.tpl       # named templates only — prefix with _
│   ├── _service.tpl
│   ├── _ingress.tpl
│   ├── _hpa.tpl
│   ├── _pdb.tpl
│   ├── _servicemonitor.tpl
│   ├── _networkpolicy.tpl
│   ├── _labels.tpl
│   └── _security.tpl
└── README.md

apiVersion: v2
name: common
type: library                       # NOT 'application'
version: "2.4.0"
description: "Shared templates for myorg charts"

A library template

{{/*
common.deployment renders a full Deployment given a context (.) with
keys: name, image, port, replicas, resources, probes, env, volumes.
*/}}
{{- define "common.deployment.tpl" -}}
apiVersion: apps/v1
kind: Deployment
metadata:
  name: {{ include "common.fullname" . }}
  labels: {{- include "common.labels" . | nindent 4 }}
spec:
  replicas: {{ .Values.replicas | default 3 }}
  strategy: {{- toYaml (.Values.strategy | default (dict "type" "RollingUpdate")) | nindent 4 }}
  selector:
    matchLabels:
      {{- include "common.selectorLabels" . | nindent 6 }}
  template:
    metadata:
      labels:
        {{- include "common.selectorLabels" . | nindent 8 }}
    spec:
      serviceAccountName: {{ include "common.serviceAccountName" . }}
      securityContext:   {{- include "common.podSecurity" . | nindent 8 }}
      containers:
        - name: {{ .Chart.Name }}
          image: "{{ .Values.image.repository }}@{{ required "image.digest required" .Values.image.digest }}"
          ports: [{ containerPort: {{ .Values.service.port }}, name: http }]
          resources: {{- toYaml .Values.resources | nindent 12 }}
          readinessProbe: {{- include "common.probes.readiness" . | nindent 12 }}
          livenessProbe:  {{- include "common.probes.liveness"  . | nindent 12 }}
          securityContext: {{- include "common.containerSecurity" . | nindent 12 }}
{{- end -}}

{{/* Public entry — consumers call this, optionally overriding sections */}}
{{- define "common.deployment" -}}
{{- include "common.util.merge" (append . "common.deployment.tpl") -}}
{{- end -}}

A consumer chart uses it in 5 lines

{{- include "common.deployment" (list . "myapp.deployment") -}}
{{- define "myapp.deployment" -}}
spec:
  template:
    spec:
      containers:
        - name: {{ .Chart.Name }}
          env:                                       # overrides merged into common output
            - { name: LOG_LEVEL, value: "info" }
            - { name: DB_URL, valueFrom: { secretKeyRef: { name: db, key: url } } }
{{- end -}}

The pattern from the Helm docs: use the common code from the helper chart and override specific sections per consumer. The merge helper (a small Sprig-based utility) does a deep-merge between the library's base and the consumer's overrides.

How to introduce a library chart to an existing fleet

  • Identify duplication first. Diff your 5-10 most-used charts. The 90% that's identical (selectorLabels, security context, probe shape, label conventions, common annotations) is the library candidate.
  • Library version 0.1, used by one new chart. Prove the pattern on a new service first; don't refactor 20 charts on day one.
  • Migrate one chart at a time. Each migration is a non-trivial PR; rendered output should diff to zero (the chart should produce identical YAML before and after) before merging.
  • Version the library with SemVer. Adding a new helper is a minor. Changing existing helper signatures is a major (consumers break).
  • Pin in consumers. Consumers declare dependencies: [{ name: common, version: "2.x.x" }] — patch and minor bumps auto-flow; majors require explicit upgrade.
The library / app split

Library charts contain reusable template logic; application charts contain the actual workloads. Library charts allow teams to define reusable template blocks and maintain consistency across deployments. When shared logic changes, updates happen in one place instead of dozens. This is the single highest-leverage Helm pattern for organizations with more than a few services — without it, every chart drifts.

HELM Chapter IV

Umbrella charts · multi-component bundles.

What they are A top-level chart whose only purpose is to compose subcharts. charts/ contains the bundled components.

When they fit Deploying an opinionated stack: the observability bundle, the developer-platform bundle, the "everything for a new team" bundle.

When they don't Long-lived application stacks where components evolve independently. Use multiple separate Helm releases.

An umbrella chart bundles multiple Helm charts into one deployable unit. The current best practice for composing a complex application from discrete parts is to create a top-level umbrella chart that exposes the global configurations, and then use the charts/ subdirectory to embed each of the components. The classic examples: kube-prometheus-stack (Prometheus + Alertmanager + Grafana + node-exporter + kube-state-metrics), linkerd (control plane + data plane + viz), istio (base + istiod + ingress + egress).

When NOT to use an umbrella chart

Umbrella charts have real costs. One small change in the base-chart gets propagated to all the derived charts, which requires us to be more conscious when making such changes. However, most of such problems can be mitigated by grouping our apps into smaller chunks instead of grouping dozens of apps into a single umbrella chart. Five anti-cases:

  • Long-lived microservice stacks. Twenty services bundled in one umbrella means every microservice deploy bumps the umbrella version. Each service should be its own chart, its own release, its own rollout.
  • Mixed lifecycle. Subchart A wants daily releases; subchart B is updated quarterly. Umbrella forces them onto the same release cadence.
  • Different ownership. Subcharts owned by different teams; the umbrella becomes a coordination point that doesn't add value.
  • Independent scaling. One subchart scales horizontally; another is a singleton. Bundling complicates the autoscaling story.
  • Failure isolation needed. Umbrella-level rollback rolls back everything, even healthy components.

The right model in those cases is a GitOps repo (Issue 08 Chapter IV) with N independent Argo CD Applications pointing at N separate charts — composed at the GitOps layer, not the Helm layer.

The umbrella's sweet spot

Umbrella charts shine for tightly-coupled, opinionated bundles installed together: the observability stack, the service mesh, the developer platform foundation. Bundles where "all-or-nothing" rollback is the right semantic. Bundles where global values legitimately flow across components (one TLS config, one storageClass, one cluster domain).

Umbrella chart drift

Once you have an umbrella with 5+ subcharts, version skew becomes a real failure mode. Subchart A version bumped from upstream, subchart B didn't, and they share a CRD. Pin all subchart versions to specific minors. Test umbrella upgrades end-to-end in staging. Document the test matrix. Without this discipline, umbrella charts become time bombs.

HELM Chapter V

Values schema · the public interface.

The single most under-used Helm feature values.schema.json — JSON Schema validating values at install time. Required fields, type checks, regex on strings, enums.

The argument Values is the chart's public API. Without a schema, the API is undocumented and unenforced.

A chart's values.yaml file is the contract between the chart author and every consumer. Without a schema, the contract is a documentation file the consumer may or may not read. With values.schema.json, the contract is mechanically enforced — install fails before rendering if required values are missing, types are wrong, or values fall outside allowed ranges.

The pattern

{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "type": "object",
  "required": ["image", "resources", "ingress"],
  "additionalProperties": false,                       # reject unknown top-level keys
  "properties": {
    "image": {
      "type": "object",
      "required": ["repository", "digest"],
      "properties": {
        "repository": { "type": "string", "pattern": "^[a-z0-9./-]+$" },
        "digest": {
          "type": "string",
          "pattern": "^sha256:[a-f0-9]{64}$",             # enforce digest, never tag
          "description": "Image must be pinned by digest"
        },
        "pullPolicy": { "enum": ["IfNotPresent", "Always", "Never"] }
      }
    },
    "replicaCount": { "type": "integer", "minimum": 1, "maximum": 200 },
    "resources": {
      "type": "object",
      "required": ["requests", "limits"],
      "properties": {
        "requests": {
          "type": "object",
          "required": ["cpu", "memory"],
          "properties": {
            "cpu": { "type": "string", "pattern": "^([0-9]+m|[0-9]+(\\.[0-9]+)?)$" },
            "memory": { "type": "string", "pattern": "^[0-9]+(Ki|Mi|Gi|Ti)$" }
          }
        }
      }
    },
    "ingress": {
      "type": "object",
      "required": ["enabled"],
      "properties": {
        "enabled": { "type": "boolean" },
        "className": { "enum": ["envoy", "nginx", "alb"] }
      },
      "if": { "properties": { "enabled": { "const": true } } },
      "then": { "required": ["hosts", "className"] }            # conditional required
    },
    "podSecurityContext": {
      "type": "object",
      "properties": {
        "runAsNonRoot": { "const": true }                       # platform policy
      }
    }
  }
}

Several patterns from this earn keeping:

  • additionalProperties: false at the top level rejects typos. A user writing replicaCnt: 3 instead of replicaCount: 3 would silently get the default; with this, the install fails with a clear error.
  • Regex on the image digest — refuses tags. The chart's policy is "pinned by digest only" and the schema enforces it.
  • Conditional required — if ingress.enabled: true, then ingress.hosts and ingress.className must be set. Pure JSON Schema, no template logic.
  • Const for fixed policy — runAsNonRoot: const: true. The chart will refuse to install if a consumer tries to flip this.
  • Patterns for K8s quantity strings — cpu (millicores or whole cores), memory (Ki / Mi / Gi). Catches the "cpu: 0.5" oh-no-that-rounds-to-zero class of bugs.

The full lifecycle pattern

Generate the schema, don't hand-write it. Hand-written schemas drift from values.yaml. Two options:

  • helm-schema (or helm-schema-gen): scans values.yaml for annotations like # @schema type:string; pattern:"^v\d+\.\d+" and produces values.schema.json. Commit both; CI runs the generator and fails if the schema is out of date.
  • helm-docs: a sibling tool that generates README.md from values.yaml annotations. Same source of truth; one annotation produces both the user-facing doc and the validation.

Validated values + Argo CD = early failure

Argo CD renders charts server-side; if values.schema.json exists, the validation runs there too. A bad values commit fails the Argo CD sync with a clear error, before any pods are touched. This is the difference between "deploy fails after 8 pods are rolled" and "PR fails CI before merge."

Values schema as platform policy

Schema is also where you encode platform-wide rules. Every chart we own enforces runAsNonRoot: true. Every prod chart requires an image digest, never a tag. Every chart in finance must have networkPolicy.enabled: true. Encoding these as schema constants is more enforceable than wiki docs. Combine with library charts (Chapter III) for full platform consistency.

Operator design · CRDs as public API.

The frame A CRD is an API. It has a schema, versions, semantics, and consumers. Design it like an API, not like a YAML file.

The split spec is what the user wants. status is what's actually true. Never confuse them.

Issue 08 introduced operators as the lifecycle-management pattern: CRDs define new resource types, controllers reconcile them. This chapter focuses on the API design side — what makes a CRD a good public interface for a platform consumer.

The three CRD-design principles

One. Spec and status are different. spec is the user's declared intent — fields they set. status is the controller's observation — fields the controller writes. They have separate API endpoints and separate RBAC. In general, the stored configuration depicts the full desired state, making it easier for the system to determine how to achieve the state, and for the user to know what to anticipate. Never put observed state in spec; never put desired state in status.

Two. Explicit defaults beat implicit assumptions. Set defaults in the CRD schema, not in the controller. If a default is in the schema, users see it via kubectl explain, dry-run renders include it, and the stored object is fully specified. If a default is in the controller, users have no way to know what they're getting until it runs.

Three. Make the common case short and the unusual case possible. The 10-line CR a new user writes should produce a sensible production-grade resource. The 200-line CR a power user writes should give them full control. Both must validate against the same schema.

A well-designed CRD

apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: postgresclusters.database.example.com
spec:
  group: database.example.com
  names:
    plural: postgresclusters
    singular: postgrescluster
    kind: PostgresCluster
    shortNames: [pg, pgc]
    categories: [databases]
  scope: Namespaced
  versions:
    - name: v1
      served: true
      storage: true
      # Promote subresources — REQUIRED for production CRDs
      subresources:
        status: {}
        scale:
          specReplicasPath: .spec.replicas
          statusReplicasPath: .status.readyReplicas
          labelSelectorPath: .status.selector
      # Make output meaningful in kubectl get
      additionalPrinterColumns:
        - { name: Replicas, type: integer, jsonPath: .spec.replicas }
        - { name: Ready, type: integer, jsonPath: .status.readyReplicas }
        - { name: Phase, type: string, jsonPath: .status.phase }
        - { name: Age, type: date, jsonPath: .metadata.creationTimestamp }
      schema:
        openAPIV3Schema:
          type: object
          required: [spec]
          properties:
            spec:
              type: object
              required: [replicas, storage]
              properties:
                replicas:
                  type: integer
                  minimum: 1
                  maximum: 9
                  default: 3                        # default in schema, not in code
                version:
                  type: string
                  enum: ["15", "16", "17"]            # enumerate; reject unknown
                  default: "17"
                storage:
                  type: object
                  required: [size]
                  properties:
                    size: { type: string, pattern: "^[0-9]+(Gi|Ti)$" }
                    storageClassName: { type: string }
                backup:
                  type: object
                  properties:
                    enabled: { type: boolean, default: true }
                    schedule: { type: string, default: "0 2 * * *" }
                    retention: { type: string, pattern: "^[0-9]+[hdwmy]$", default: "30d" }
            status:
              type: object
              properties:
                phase:
                  type: string
                  enum: [Pending, Provisioning, Running, Failed, Deleting]
                readyReplicas: { type: integer }
                selector: { type: string }
                conditions:
                  type: array
                  items:
                    type: object
                    required: [type, status, lastTransitionTime]
                    properties:
                      type: { type: string }
                      status: { enum: ["True", "False", "Unknown"] }
                      lastTransitionTime: { type: string, format: date-time }
                      reason: { type: string }
                      message: { type: string }
                endpoint: { type: string }      # conn string written by controller

The patterns from above earn keeping:

  • Subresources. status subresource isolates status updates from spec updates (separate RBAC, prevents controller from accidentally overwriting user spec changes). scale subresource lets HPA scale the CR directly via the standard scale API.
  • additionalPrinterColumns. kubectl get postgresclusters shows replicas, ready, phase, age. Without this, users see only NAME + AGE and have to describe for anything useful.
  • Defaults in the schema. replicas: default: 3, version: default: "17". A 5-line CR produces a 3-replica Postgres 17 cluster with 30-day backup retention — sensible production defaults.
  • Enums for closed sets. version: enum: ["15", "16", "17"] — the user can't accidentally request Postgres 9; phase is one of five values, not a free string.
  • Regex patterns. Storage size validates as Gi/Ti; retention validates as duration. Invalid values rejected at API time, not at reconcile time.

The 5-line CR a user writes

apiVersion: database.example.com/v1
kind: PostgresCluster
metadata: { name: orders }
spec:
  storage: { size: "100Gi" }

This is the entire user-facing API for the common case. Replicas default to 3. Version defaults to 17. Backup defaults to enabled, daily, 30-day retention. The Operator's controller materializes ~200 lines of StatefulSet + Services + Secrets + ConfigMaps + ServiceMonitors + NetworkPolicies + backup CronJob from this 5-line request.

Status, properly

The Kubernetes-conventional Conditions pattern is what most controllers use for status. Each Condition has type, status, reason, message, lastTransitionTime. Standard types for stateful systems include Ready, Progressing, Degraded, BackupReady.

status:
  phase: Running
  readyReplicas: 3
  selector: "app=postgrescluster,instance=orders"
  endpoint: "orders-rw.default.svc.cluster.local:5432"
  conditions:
    - { type: Ready,        status: "True",  lastTransitionTime: 2026-05-13T..., reason: AllReplicasReady }
    - { type: Progressing,  status: "False", lastTransitionTime: 2026-05-13T..., reason: NoChangeInProgress }
    - { type: BackupReady,  status: "True",  lastTransitionTime: 2026-05-13T..., reason: BackupCompleted }
    - { type: Degraded,     status: "False", lastTransitionTime: 2026-05-13T..., reason: "" }

The Conditions pattern is the Kubernetes-conventional API contract. Tooling (kubectl wait, OPA policies, dashboards) knows how to read it. Custom status enums are the per-operator deviation that breaks tooling.

The CRD as documented API

Once the schema is rich and the defaults are explicit, kubectl explain postgrescluster.spec becomes the documentation. The OpenAPI schema in the CRD provides field descriptions, types, validation, defaults. Use kubebuilder's // +kubebuilder:default=N, // +kubebuilder:validation:Minimum=1, and field godoc comments — they all flow into the OpenAPI schema in the published CRD.

The CRD design discipline

Treat a CRD release like a public API release. Every new version requires a deprecation plan for the previous version. Every field deletion or rename is breaking. Every default change is observable. Document the user-facing API (the spec) separately from the operator-internal API (the status). Run kubectl explain on your CRD before shipping; if it doesn't tell a new user how to use the thing, the schema needs more annotations.

CRD versioning · hub-and-spoke conversion.

The model One hub (storage) version, N spoke (served) versions. Every spoke converts to and from the hub. Spoke-to-spoke conversion derives.

The rule Hub-to-spoke round-trip must be lossless. If a v2 spec field has no v1 equivalent, you have to design where it goes when the object is stored as v1.

CRDs evolve. v1alpha1 ships, gets feedback, becomes v1beta1, then v1. Each stage has consumers. One can think storage version as the hub and other versions as spoke to visualize the relationship between storage and other versions. The key thing to note is that conversion between storage and other version should be lossless (round trippable). If each spoke version defines conversion function from/to the hub version, then conversion function between the spoke versions can be derived.

The hub-and-spoke model in detail

Since the conversions aren't built in for CRDs, the Kubernetes API server calls out to a webhook to do the conversion instead. For Kubebuilder, this webhook is implemented by controller-runtime, and performs the hub-and-spoke conversions.


      
Defining hub and spoke in code

      
// api/v1/postgrescluster_types.go — the HUB
package v1

// PostgresCluster is the hub. It implements no Convertible interface;
// it implements only the Hub interface (which is a tag).
func (*PostgresCluster) Hub() {}      // marks this as the hub version

// ===========================================================
// api/v2/postgrescluster_types.go — a SPOKE
package v2

import (
    v1 "github.com/example/api/v1"
    "sigs.k8s.io/controller-runtime/pkg/conversion"
)

// ConvertTo converts this v2 object to the hub (v1).
func (src *PostgresCluster) ConvertTo(dstRaw conversion.Hub) error {
    dst := dstRaw.(*v1.PostgresCluster)
    dst.ObjectMeta = src.ObjectMeta
    dst.Spec.Replicas = src.Spec.Replicas
    dst.Spec.Version = src.Spec.Version
    // v2 has Backup.Schedule + Retention; v1 has just Backup.Enabled.
    // Lossy direction — store v2-only fields as annotations for round-trip.
    dst.Spec.Backup = v1.BackupSpec{Enabled: src.Spec.Backup.Enabled}
    if dst.Annotations == nil {
        dst.Annotations = map[string]string{}
    }
    dst.Annotations["backup.schedule.v2"] = src.Spec.Backup.Schedule
    dst.Annotations["backup.retention.v2"] = src.Spec.Backup.Retention
    return nil
}

// ConvertFrom converts the hub (v1) object to this v2.
func (dst *PostgresCluster) ConvertFrom(srcRaw conversion.Hub) error {
    src := srcRaw.(*v1.PostgresCluster)
    dst.ObjectMeta = src.ObjectMeta
    dst.Spec.Replicas = src.Spec.Replicas
    dst.Spec.Version = src.Spec.Version
    dst.Spec.Backup = BackupSpec{
        Enabled:   src.Spec.Backup.Enabled,
        Schedule:  src.Annotations["backup.schedule.v2"],     // recover from annotation
        Retention: src.Annotations["backup.retention.v2"],
    }
    return nil
}


      
The conversion is lossless because the v2-only fields round-trip through annotations on the v1 hub. When the v2 client writes, fields go to annotations; when the v2 client reads, fields come back from annotations. When the v1 client reads, it doesn't see the v2-only fields, which is correct. Unit tests should ensure that every spoke version can round trip to the hub version and back again to the same version with no loss of information.

Versioning policy

      
A reasonable platform policy:

      
    
        
  • v1alpha1: new CRD, no compatibility guarantee. Free to break.
    • 
        
  • v1beta1: stabilizing. Breaking changes require deprecation notice in release notes.
    • 
        
  • v1: stable. Breaking changes require a new major version (v2) and deprecation window.
    • 
        
  • Storage version is always the latest stable. Older versions are served (for backwards compat) but converted on read.
    • 
        
  • Deprecation window: served-but-deprecated → stop-serving after N months. Communicate via release notes + a webhook that emits warnings via the Warning header on responses.
    • 
      


      
The bear traps

      
    
        
  • Storage-version migration. Changing the storage version from v1 to v2 requires migrating every existing object — read it as v1, write it back, which converts to v2 storage. The kube-storage-version-migrator tool handles this; do it explicitly as part of an upgrade.
    • 
        
  • Webhook downtime. If the conversion webhook is down, the API server can't read CRs of non-storage versions. Run the conversion webhook with HA (3 replicas), PodDisruptionBudget, and conservative resource limits.
    • 
        
  • Lossy spoke-to-spoke. v2-to-v3 conversion routes through v1 (the hub). If v3 added a field that has no v1 equivalent, and v2 reads it, the v3-only data is lost. Either keep all fields on the hub or stage version transitions carefully.
    • 
      


      

        
The forward-compatibility discipline

        
Design CRDs assuming you'll add fields, not remove them. Make new fields optional with sensible defaults. Avoid renames — add a new field, mark the old deprecated, eventually remove. Use the Discriminated Union pattern (a type field with allowed values) to extend without breaking. The cost of moving carefully is small; the cost of an inadvertent breaking change is the entire user base filing tickets.

The reconciler · idempotency, status, finalizers.

The contract Reconcile is called any time the API thinks something changed. It must converge regardless of how many times it runs.

The three properties Idempotent. Level-triggered. Status-separated.

A controller's reconcile loop is where the operator earns its keep. Three properties make a controller correct: idempotent (running Reconcile twice yields the same end state as running it once), level-triggered (Reconcile acts on the current state of the world, not on the sequence of past events), and status-separated (Reconcile writes status to the status subresource; it never touches spec).

The reconcile shape

func (r *PostgresClusterReconciler) Reconcile(ctx context.Context, req ctrl.Request) (ctrl.Result, error) {
    log := logr.FromContextOrDiscard(ctx).WithValues("name", req.NamespacedName)

    // 1. Fetch the custom resource. Not found = deleted, return cleanly.
    var cluster databasev1.PostgresCluster
    if err := r.Get(ctx, req.NamespacedName, &cluster); err != nil {
        return ctrl.Result{}, client.IgnoreNotFound(err)
    }

    // 2. Handle deletion if a DeletionTimestamp is set.
    if !cluster.DeletionTimestamp.IsZero() {
        return r.handleDelete(ctx, &cluster)
    }

    // 3. Ensure the finalizer is on the object. Idempotent — checks then adds.
    if !controllerutil.ContainsFinalizer(&cluster, postgresFinalizer) {
        controllerutil.AddFinalizer(&cluster, postgresFinalizer)
        if err := r.Update(ctx, &cluster); err != nil {
            return ctrl.Result{}, err
        }
        return ctrl.Result{Requeue: true}, nil            // requeue with new resourceVersion
    }

    // 4. Build desired state from spec. Pure function of the spec.
    desiredSTS := buildStatefulSet(&cluster)
    desiredSVC := buildService(&cluster)
    desiredCM  := buildConfigMap(&cluster)
    desiredPDB := buildPDB(&cluster)

    // 5. Upsert each owned resource. SetControllerReference establishes ownership.
    for _, obj := range []client.Object{desiredSTS, desiredSVC, desiredCM, desiredPDB} {
        if err := controllerutil.SetControllerReference(&cluster, obj, r.Scheme); err != nil {
            return ctrl.Result{}, err
        }
        if err := r.upsert(ctx, obj); err != nil {
            return r.degraded(ctx, &cluster, err)
        }
    }

    // 6. Compute and write status. Always via Status() subresource.
    cluster.Status.Phase = computePhase(ctx, r, &cluster)
    cluster.Status.ReadyReplicas = countReadyReplicas(ctx, r, &cluster)
    cluster.Status.Endpoint = computeEndpoint(&cluster)
    cluster.Status.Conditions = computeConditions(ctx, r, &cluster)
    if err := r.Status().Update(ctx, &cluster); err != nil {
        return ctrl.Result{}, err
    }

    // 7. Periodic re-check to catch drift the watch wouldn't surface.
    return ctrl.Result{RequeueAfter: 30 * time.Second}, nil
}

Idempotency in practice

Three rules:

  • Build desired state from spec only. The function that produces the desired StatefulSet should not read the actual cluster state. It's a pure function of the CR spec.
  • Compare and upsert. Fetch the current resource, diff against desired, apply. controllerutil.CreateOrUpdate or server-side apply (SSA) handles this — SSA is preferred in 2026 because it knows which fields the controller owns and won't fight other actors.
  • No "have I already done this?" branches. The reconciler doesn't track its own history. If a StatefulSet already exists matching the desired spec, the upsert is a no-op; if it doesn't, the upsert creates it. Same code path either way.

Level-triggered, not edge-triggered

Edge-triggered means "when X happens, do Y." Level-triggered means "while X is the case, ensure Y is the case." Kubernetes controllers must be level-triggered: missed events still converge because the next reconcile sees the current state and acts.

Practical implication: never store "I sent the create event" in memory. If your operator restarts, it has no memory; it gets a list of all CRs and reconciles each one. If reconcile depends on past events, the restart breaks things. If reconcile depends only on current state, the restart is transparent.

Finalizers and deletion

When a user deletes a CR (kubectl delete), the API server sets metadata.deletionTimestamp. The object continues to exist until all finalizers are removed. This is the operator's chance to clean up external state — cloud resources, DNS records, observability registrations, S3 buckets — before the object disappears.

func (r *PostgresClusterReconciler) handleDelete(ctx context.Context, c *databasev1.PostgresCluster) (ctrl.Result, error) {
    if !controllerutil.ContainsFinalizer(c, postgresFinalizer) {
        return ctrl.Result{}, nil                // nothing to do
    }

    // Clean up external state (snapshots, DNS, cloud resources).
    if err := r.cleanupBackups(ctx, c); err != nil {
        return ctrl.Result{RequeueAfter: 30 * time.Second}, err
    }
    if err := r.cleanupDNS(ctx, c); err != nil {
        return ctrl.Result{RequeueAfter: 30 * time.Second}, err
    }

    // Remove finalizer — only after external state is clean.
    controllerutil.RemoveFinalizer(c, postgresFinalizer)
    if err := r.Update(ctx, c); err != nil {
        return ctrl.Result{}, err
    }
    return ctrl.Result{}, nil                    // object now disappears
}

The contract: idempotent cleanup. cleanupBackups must succeed even if it's been called before. cleanupDNS must handle "DNS record already absent" as success, not error. Otherwise a stuck finalizer means a stuck delete — and the only fix is editing metadata, which is operationally awful.

Owner references and garbage collection

SetControllerReference writes metadata.ownerReferences on the child resource pointing at the parent CR. When the parent is deleted, the K8s garbage collector deletes the children automatically (the foreground propagation mode). For most operators, this is enough — you don't need a finalizer for cleaning up in-cluster resources, only for external state.

Backoff and requeueing

  • Return error: exponential backoff. Use for transient errors that should retry.
  • RequeueAfter: N: requeue after N. Use for periodic re-checks ("re-verify state every 30s").
  • Requeue: true: requeue with current backoff. Use when something legitimately needs another loop (e.g., you just added a finalizer and need the new resourceVersion).
  • Return ctrl.Result{}, nil: success, watch-driven re-runs only. The common end state for a converged loop.

Eventing — surface what happened

The Recorder API emits Kubernetes events visible via kubectl describe and kubectl get events. Use them for state transitions and recoverable errors.

// Reasons follow Kubernetes conventions: CamelCase
r.Recorder.Event(&cluster, corev1.EventTypeNormal, "BackupCompleted", "Daily backup completed")
r.Recorder.Eventf(&cluster, corev1.EventTypeWarning, "BackupFailed",
    "Failed to upload backup: %v", err)
The reconcile mental model

"Given the current spec and the current observed state, what action would move the world one step closer to convergence?" Answer that question, do that one action, return. The runtime will call you again. Don't try to drive multi-step state machines inside one Reconcile call; let each call be a single step, and let the requeue mechanism handle the loop. Operators that follow this discipline are debuggable; operators that don't become inscrutable.

Operator anti-patterns · and the fix.

Read this as a triage list Common operator failure modes and their corrections. Each one shows up in operators that grew organically rather than to a design.

The frequency rule Most operators have at least three of these. Fixing them in order of pain is the typical road to maturity.

Patterns that look fine in the small operator become liabilities once usage scales or the CRD surface broadens. Catalogued below with concrete corrections.

Operator design anti-patterns
Recognize → fix in order of operational pain
Anti-pattern
Why it hurts
Fix
Writing spec from the controller
Controller modifies .spec fields (e.g., "auto-fill defaults" by writing them in). Conflicts with user's kubectl apply; spec changes become controller-author and user-author races.
Defaults belong in the CRD schema. Observed state belongs in .status. Controller never writes .spec after creation.
Updating status via main client
r.Update(ctx, &cluster) when you meant r.Status().Update(). The full update includes any in-memory spec changes; you can overwrite a concurrent user edit.
Always r.Status().Update() for status writes. Configure the subresource (subresources: { status: {} }) so RBAC and updates are separated.
In-memory state machine
Controller stores "I already did step X" in a Go map. Restart wipes it; reconcile re-runs from scratch and either re-does work or skips work.
Level-triggered. Reconcile inspects current state and acts. If you need to remember progress, write it to .status.phase or annotations.
Long-running synchronous work
Reconcile blocks for 5 minutes copying data. The work queue stalls; other CRs starve; pods OOM.
Kick off a Job; record the Job's name in status; subsequent Reconciles check Job status and return quickly. Or use RequeueAfter with short polls.
Cluster-scoped, namespace-leaking
Operator watches all namespaces; one user's CR triggers actions in another's namespace; multi-tenant boundary broken.
Run namespace-scoped operators where possible. If cluster-scoped, enforce RBAC + admission policies on who can create which CRDs in which namespaces.
Stuck finalizer
Finalizer cleanup fails permanently (S3 bucket missing, IAM key revoked). Object can't be deleted; user has to kubectl patch ... --remove-finalizer.

Watches that miss owned resources
Operator updates the StatefulSet; nothing watches the StatefulSet → reconcile doesn't re-trigger when it goes Ready. Status sits stale for 30s.
Set up explicit watches: For(&PostgresCluster{}).Owns(&appsv1.StatefulSet{}).Owns(&corev1.Service{}). Owns sets up the watch automatically.
Single replica, no leader election
One operator pod = single point of failure. Pod restart = stale reconciles until it's back. Two pods without coordination = double-write conflicts.
Run 2 or 3 replicas with leader election enabled (--enable-leader-election via controller-runtime manager options). Only the leader reconciles; standbys wait.
Logging at INFO for every reconcile
One CR reconciled every 30s = 2880 log lines per day per CR. 100 CRs = 288k lines. Loki costs spike; useful signal drowns.
INFO on state transitions only. DEBUG for per-reconcile detail. Use structured logging with the CR name as a key; query by that.
No metrics
Operator silently fails. Reconcile errors aren't aggregated; queue depth isn't visible; you find out via user complaint.
controller-runtime exposes controller metrics by default. Scrape controller_runtime_reconcile_errors_total, controller_runtime_reconcile_time_seconds, workqueue_depth. Alert when error rate > threshold or queue depth grows monotonically.
CRD installed by application Helm chart
Two consumers install two charts that both ship the same CRD. Helm fights over ownership; uninstall of one breaks the other.
CRDs live in the operator's chart only. Or ship them out-of-band (Helm crds/ dir, kustomize). Document the install order: CRDs first, operator second, CRs third.
No webhook auth/cert rotation
Admission/conversion webhook cert expires. CRD writes fail until the cert is regenerated.
Use cert-manager to issue + rotate webhook certs. Set up caBundle via the cert-manager injectAnnotations. Test cert rotation in staging.

The fix order

When you inherit an operator with multiple issues, fix in this order:

  1. Leader election + 2+ replicas. Removes the SPOF; immediate operational win.
  2. Status subresource + RBAC split. Stops the most common bug class (spec-overwrite races).
  3. Metrics + alerts. Now you can see what's happening.
  4. Idempotent reconcile. Audit each branch; remove in-memory state; build desired-state functions as pure.
  5. Idempotent finalizers. Re-runnable cleanup; bounded retries; status condition for partial-cleanup state.
  6. CRD schema tightening. Add validation, defaults, printer columns. Now kubectl get is informative.
The operator-burnout pattern

The most common end state for a poorly-designed operator: nobody trusts it, everyone uses the YAML it generates as a starting point and edits by hand. The whole point — declarative, self-healing, self-correcting — is gone. If you see your users bypassing the operator with kubectl edit on the generated resources, the operator has failed its product job. Either fix the API (the CRD doesn't let users express what they need) or fix the reconciler (it fights what users do). Don't ship more features until the trust problem is resolved.

Platform APIs · compositions & Crossplane.

The leap From Operator-style CRDs to platform-defined CRDs. The platform team writes a high-level API; Crossplane materializes it into Helm + Operator + cloud resources.

The point Developers request "a database," not "an RDS instance with these specific settings." The platform abstracts.

A Helm chart abstracts manifests. An Operator abstracts lifecycle. A platform composition abstracts the whole domain — a developer requests "a service with a Postgres database, exposed via HTTPS, with monitoring and backups" and the platform materializes everything underneath. Crossplane extends Kubernetes to manage cloud resources as custom resources. The power of Crossplane for platform engineering is Compositions — custom API abstractions that hide cloud complexity. Your platform team defines a "Database" composition that provisions an RDS instance, configures backups, sets up monitoring, and creates connection secrets. Developers request a database by submitting a 10-line YAML file. The platform handles everything else.

The three Crossplane concepts

Provider: a controller that knows how to manage one type of resource. provider-aws manages AWS resources, provider-gcp manages GCP, provider-kubernetes manages in-cluster K8s objects. Providers ship as Crossplane packages.

CompositeResourceDefinition (XRD): defines a new high-level CRD that the platform exposes. This is the API surface developers see.

Composition: a recipe that says "given a Composite (an instance of an XRD), create these N concrete resources." The recipe binds platform-level abstractions to concrete cloud + K8s + Helm resources.

The platform-team / app-team contract

Templates are the contract surface between the platform team and consumers. The platform team owns the template; the app team consumes it. Runtime enforcement (admission policies, OPA) tells app teams "no" after they've already written the wrong thing. Template-driven scaffolding tells them "yes, here's the right thing" up front. Golden paths · templated service creation.

The definition The opinionated default path to ship something. Not the only path. The fastest path.

The promise A new developer joins on Monday; ships a production service on Wednesday. Without the platform, that takes weeks.

You identify the highest-friction, highest-volume patterns — service creation, database provisioning, CI pipeline setup, environment cloning. Then you encode those patterns into a reproducible template that any developer can invoke through the portal with a few clicks, getting a compliant, production-ready result without ever needing to understand the underlying infrastructure. That's the golden-path framing.

What "golden" means concretely

A golden path is an opinionated workflow that produces a compliant, production-ready outcome with minimal user input. Five properties:

  • Discoverable. Visible in the developer portal (Backstage). Searchable.
  • Self-service. Click-driven, parameter-driven. No tickets, no Slack pings.
  • Compliant. The output passes whatever policies your org enforces (security, observability, cost tagging).
  • Composable. The output integrates with the rest of the platform — auto-registers in the catalog, auto-creates dashboards, auto-deploys via GitOps.
  • Escape-hatched. The output is regular code/YAML the developer can edit. They aren't locked in.

Adoption metrics — measuring the path

Per 2026 platform-engineering practice, track:

  • Time-to-first-deploy (TTFD) for new services. Before the platform: weeks. After: hours.
  • Template usage rate: % of new services created via a template. Above 80% = healthy adoption.
  • NPS: How likely are you to recommend our platform to a new hire? Above 30 is excellent; negative means developers actively work around you.
  • Time-to-resolution: how long it takes the platform team to fix or unblock a developer issue. Track as a service SLO.
  • Cost per service onboard: combined platform-team hours per new service. The denominator the platform exists to push down.

The hardest part — the user research

Most platforms fail not at the technology but at the prioritization. The #1 mistake: spending 6 months building a service catalog that developers don't ask for while teams are actually drowning in manual certificate renewals. Two disciplines avoid this:

  • Quarterly developer surveys. 5 questions max: deployment satisfaction (1-5), service setup time, biggest friction, platform capability requested, NPS. Track the numbers.
  • Shadow weeks. Platform engineers sit with app teams during real work; observe where they struggle. The friction the platform team would never guess at becomes obvious in an afternoon.
The golden-path discipline

Build one golden path completely before starting the second. From-developer-click to running-compliant-service. Deploy to a willing pilot team. Measure onboarding time before and after. That single metric, clearly evidenced, is how you justify the next quarter's investment. Then build the next path. The "build six things badly" trap is real and patient.

TEAM Chapter XIV

Team interaction models · topologies.

The framework Team Topologies by Skelton & Pais. Four team types, three interaction modes. The platform is one of the four.

The point The same platform tech fails or succeeds depending on how the platform team relates to app teams. Topology beats technology.

The Team Topologies framework (Skelton & Pais, 2019, now near-canonical in 2026 platform practice) names four team types: stream-aligned (the app teams), platform (the team you're on), enabling (consultants who teach), and complicated-subsystem (specialist work too deep to put in a stream-aligned team). The interactions between them — three modes: X-as-a-Service, Facilitating, Collaboration — determine whether the platform succeeds.

Interactive · The four team types
Click to see how each relates to the platform team

The three interaction modes

X-as-a-Service is the platform team's default mode with stream-aligned teams. Low-bandwidth, asynchronous, contract-mediated. App teams consume the platform via its API (Helm chart, Crossplane claim, Backstage template); platform team responds to issues but is not in standup with the app team. This is what scales — one platform team can serve fifty app teams in X-as-a-Service mode.

Facilitating is how enabling teams work with stream-aligned teams. High-bandwidth, time-bounded, knowledge-transfer-focused. An enabling team embeds with an app team for 2-6 weeks to teach a new practice (testing, observability instrumentation, a new framework) then leaves. The goal is to make the stream-aligned team self-sufficient on the new capability, not to do the work for them.

Collaboration is high-bandwidth, sustained, two-way. Use it sparingly — it's expensive and doesn't scale. Reserve for new-domain discovery: the platform team and one stream-aligned team co-design a new capability for several weeks, then transition to X-as-a-Service once the abstraction is stable.

The platform team's mode allocation

A healthy platform team in 2026 spends time roughly like this:

  • ~60% X-as-a-Service: shipping API versions, fixing bugs, monitoring SLOs, responding to support tickets.
  • ~25% Facilitating: pair-debugging with app teams; office hours; documentation; demo videos.
  • ~10% Collaboration: co-designing a new platform capability with one or two pilot teams.
  • ~5% with complicated-subsystem teams: integrating their work (Kubernetes upgrades, custom schedulers, ML platform internals) into the platform surface.

When the X-as-a-Service ratio is below 40%, the platform team has been pulled into Collaboration on too many tracks; symptom is "platform team is the bottleneck on every release." When it's above 80%, the team has stopped engaging with users and is at risk of building features nobody asked for.

The platform-team-as-product-team

Several practical implications follow:

  • Product manager. Even a part-time PM dedicated to the platform produces better prioritization than engineer-led prioritization. Roadmap based on developer pain, not technical interest.
  • Office hours. A weekly drop-in slot where any app team can ask about anything. Far more efficient than DM-driven support.
  • Release notes per API. Every chart version, every CRD version, every template version has release notes. What's new, what's deprecated, what to migrate to.
  • Deprecation policy. Explicit support windows. v1 is supported until July 2026; v2 was released; please migrate by then.
  • SLOs on the platform itself. Backstage uptime, Argo CD sync latency, Crossplane claim provisioning time. Cross-reference Issue 06 Chapter VIII on SLO discipline.

Common org-design failure modes

Four patterns that show up repeatedly and break platform efforts:

  • Platform team is just SRE renamed. Still firefighting; no product discipline; no roadmap; no users-as-customers framing. Same outcomes as before.
  • Platform team has no charter. Reports into the same VP as app teams; treated as another delivery team; gets pulled into app work. Solution: separate management line + dedicated budget.
  • Mandatory platform with no escape hatch. App teams forced onto the platform before it's mature. Resistance, workarounds, and political damage. Solution: opt-in pilots, expand based on adoption.
  • Platform team builds in isolation. Months of architecture work with no user feedback. Lands a release; nobody uses it. Solution: pilot every new capability with one real app team before broad release.
The 1-to-N ratio

A reasonable target: one platform engineer for every 20-50 application developers, depending on platform maturity and complexity. Starting too small — expecting one or two people to build a comprehensive platform — leads to burnout and incomplete solutions. The platform team is itself a leverage point; under-invest and the leverage goes negative (the team becomes a bottleneck rather than an accelerant).

Many teams; many environments; standardization needed.

The composition pattern

In practice these layer. A typical platform offering at scale: Backstage template on top, that creates a Crossplane claim, that composes existing Helm charts for in-cluster work and existing Operators for stateful systems and cloud provider resources for managed services. Four layers; each delegates to the one below.

Don't try to skip layers. A Backstage template that doesn't go through Crossplane or Helm but directly calls cloud APIs becomes a bespoke pile of code the platform team alone maintains. A Crossplane composition that re-implements what a mature operator already does duplicates work and creates drift. The win comes from using each layer for what it's good at.

The pragmatic order of construction

Most platforms build in this order. 1. Library Helm chart (Chapter III) for shared deployment shape. 2. Application charts for each major service type. 3. Operators for genuinely-stateful systems with operational behavior. 4. Crossplane compositions to combine the above with cloud resources. 5. Backstage templates to invoke the compositions via a UI. Each layer is useful on its own; each layer becomes more useful when the layers below exist. Skipping layer 1 and going straight to layer 5 is the most common failure mode — the templates produce inconsistent code because the underlying primitives aren't standardized.
