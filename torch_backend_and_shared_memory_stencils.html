Torch as a backend, & the kernels you write by hand

The same algorithm has to run on a laptop CPU, a datacenter GPU, and an Apple M-series chip. You don’t want to write it three times. You also don’t want to give up peak performance on the device where it matters most. This is a tutorial — starting from zero — in the architecture that lets you have both: use PyTorch as the device-dispatching tensor backend, then drop down to a hand-written shared-memory CUDA kernel exactly where it pays. The case studies are pydisort (a C radiative-transfer solver wrapped as torch ops), kintera (chemistry + thermodynamics), and snapy (a fluid-dynamics code whose WENO5 reconstruction is the cleanest shared-memory stencil in the workspace).

Part I

The problem & the idea

No code, no jargon yet. First we get clear on what we are trying to accomplish and what trick makes it possible. Once those two things are in your head, every later chapter is a slot the trick clicks into.

§00Synopsis: the one-sentence answer

The one-sentence answer to “how does one codebase run fast on every device?” is:

Write the algorithm once as a function that takes typed memory pointers; let PyTorch wrap that function in a tensor-shaped, device-aware op; and on the one device where peak performance matters most, write a second version of the same function using the device’s native primitives.

That’s it. The rest of this tutorial unpacks each phrase: typed memory pointers, tensor-shaped op, device-aware, native primitives. Then we look at three real codebases that follow exactly this recipe — with progressively harder kernels — and end with a deep dive on the most interesting case: a hand-written CUDA kernel that uses on-chip shared memory to outperform every pure-torch alternative on the same algorithm.

Audience note: this tutorial assumes nothing beyond “an array is a contiguous block of typed numbers” and “a GPU runs many threads in parallel.” If you know more, parts will feel slow; the deep-dive in §14–§16 should still be new.

§01The problem: one algorithm, three devices

Concretely: you have a numerical algorithm — say, a discrete-ordinates radiative transfer solver, or a chemical-equilibrium calculation, or a high-order spatial reconstruction for a fluid solver. You want users to:

  1. Run it from Python.
  2. Pass in PyTorch tensors and get back PyTorch tensors (so it composes with the rest of their pipeline).
  3. Have it run on whatever device their tensors live on — CPU, NVIDIA GPU (CUDA), or Apple Silicon GPU (Metal Performance Shaders, “MPS”) — without changing a line of their code.
  4. Get speeds that are competitive with hand-written native code on whichever device matters most.

The naive approach is to write three implementations of every function (one per device) and a dispatcher that picks based on the tensor’s .device. That works, technically, but it has three problems:

  • Tripled maintenance. A bug fix in one path has to be applied to three.
  • Per-device specialism. CPU benefits from cache-blocking, GPU from coalesced loads and shared memory, Apple Silicon from different shared-memory rules — you can’t use one mental model.
  • Discovery problem. Most algorithms don’t need three implementations; the loops are simple and bandwidth-limited. Writing three copies of a loop is waste. Only the few inner kernels that truly want device-specific tuning deserve a custom version.

So we want a design where:

  • The default per-device version of an op is generated from one source — you write the loop body once, the framework runs it CPU-parallel or in CUDA threads.
  • For the hot kernels you care about, you can override the default with a hand-written device kernel that uses shared memory, warp shuffles, or whatever the hardware exposes.
  • Either way, the user sees the same Python function and the same tensor in, tensor out, regardless of device.

That design exists. PyTorch is one of its most polished implementations — not because of deep learning, but because deep-learning kernels happen to need exactly this structure (one op, many devices, often with hand-tuned inner loops). The three case-study repos in this tutorial use PyTorch’s machinery for non-ML problems (radiation, chemistry, fluid dynamics), demonstrating that the architecture is general.

§02What “torch as backend” really means

The most common source of confusion: PyTorch is two things layered on top of each other, and only the bottom layer matters here.

What you probably know PyTorch as

A deep-learning framework. Autograd, neural-net modules, optimizers, distributed training, the whole stack used to train large models.

What we’re using PyTorch for here

The tensor library underneath: a typed n-dimensional array (at::Tensor) that knows its dtype, shape, strides, and which device it lives on, plus a machine for writing operations on those tensors that dispatch to the right device implementation. Deep learning is one user of this layer; you can be another.

Specifically, three pieces of the lower layer do all the work. We’ll meet each in Part II; the names worth memorizing now are:

PieceIn one sentence
at::Tensora typed pointer + shape + strides + device label — what gets passed in and out.
at::TensorIteratorgiven some input/output tensors and a description of how they broadcast, hand you a way to walk every element exactly once.
DispatchStuba small registry mapping (operation name, device) to a function pointer. The call call_my_op(device_type, iter, ...) looks up the right function and runs it.

The user-facing convenience — binding a C++ class as a Python type with PyTorch’s nn::Module — is icing. We’ll meet it last (§06). It is not load-bearing for performance; it’s load-bearing for ergonomics.

You can think of PyTorch (for our purposes) as a library that handles four things you would otherwise build yourself: device-aware memory, broadcasting iteration, a function-pointer registry per device, and a Python binding. Building any of those well is months of work. Using PyTorch’s, your day-one is “write the loop body.”

Part II

The integration surface, end to end

Now we open the lid and look at each of the three pieces in turn. By the end of this part you’ll have read a complete, minimal example — about fifty lines of C++ — that takes one C function and turns it into a torch op runnable on CPU and CUDA. Every other chapter is a riff on this skeleton.

§03A tensor is a small contract

Forget what tensors mean in math for a moment. In PyTorch’s C++ API an at::Tensor is a tiny C++ object that, at runtime, knows five things:

FieldWhat it is
data pointerraw memory address of the first element (tensor.data_ptr<float>())
dtypewhich scalar type the bytes encode — kFloat32, kFloat64, kInt32, …
shapea small array of dimension sizes, e.g. {nwave, ncol, nlyr, nprop}
strideshow many elements to step to advance each dimension; lets you walk non-contiguous views without copying
devicewhere the memory lives: CPU host, CUDA device 0, MPS, …

That’s the entire user-visible surface. Two important consequences:

  • A tensor is not a number array you have to translate to and from. Once you have the data pointer and the strides, the loop body sees plain typed pointers — identical to the code you would write for a raw C array. PyTorch is just bookkeeping around the pointer.
  • The device is just a label, until you act on it. A tensor on CUDA is the same C++ object as one on CPU — same class, same methods. The dispatch (§05) is what reads the label and routes to the right implementation. This is what makes “one function, many devices” possible.
What this means in practice

If you already have a C function void my_solver(double* in, double* out, int n), you are 80% done. The remaining 20% is teaching PyTorch (a) how to walk the tensor and call your function with the right pointers, and (b) which version of my_solver to use for which device.

§04TensorIterator: the loop machine

at::TensorIterator answers a deceptively hard question: given some input tensors that may broadcast against each other, and one or more outputs, what is the loop that visits every element once with the right offsets? If you have ever written a loop over a multidim array that was strided differently from another array of the same shape, you have written a custom loop machine. PyTorch wrote that machine for you.

You configure one with a builder. Here is a real example, from pydisort’s radiative-transfer forward pass, lightly trimmed:

auto iter = at::TensorIteratorConfig()
    .resize_outputs(false)
    .check_all_same_dtype(true)
    .declare_static_shape({nwave, ncol, ds().ntau, 2},
                          /*squash_dims=*/{2, 3})
    .add_output(flx)
    .add_input(prop)
    .add_owned_input(umu0_broadcast)        // each boundary-condition
    .add_owned_input(phi0_broadcast)        // is broadcast up to (nwave,ncol,...)
    // ... more inputs, all broadcastable to the static shape ...
    .build();

The builder takes a few important options:

OptionWhat it does
add_output / add_inputregister the tensors. add_owned_input lets you pass a temporary (e.g. a broadcast view) without managing its lifetime yourself.
declare_static_shapetell the iterator the conceptual shape of the computation. Broadcasting expands smaller inputs to this shape.
squash_dimsdimensions to treat as a single contiguous unit inside the kernel — “don’t parallelize over these; the kernel will loop over them itself.” In the example above, dims 2 and 3 (the inner two) are squashed because disort_impl consumes them as one column at a time.

Once built, the iterator can be walked by a CPU loop or fed to a CUDA kernel launcher. In both cases the callback you supply receives raw char* base pointers and per-element strides — exactly the “typed memory pointer” from the synopsis.

int grain_size = iter.numel() / at::get_num_threads();
iter.for_each([&](char** data, const int64_t* strides, int64_t n) {
    for (int i = 0; i < n; i++) {
      auto out  = reinterpret_cast<scalar_t*>(data[0] + i * strides[0]);
      auto prop = reinterpret_cast<scalar_t*>(data[1] + i * strides[1]);
      // ... more reinterpret_casts to recover typed pointers from char* + stride ...
      disort_impl(out, prop, /*…,*/ ds[idx], ds_out[idx], nprop);  // the actual solver
    }
  }, grain_size);

Three things to notice:

  • The callback is a plain C++ lambda. No torch types appear inside it.
  • iter.for_each automatically multi-threads across the iteration with the supplied grain size. You write the inner loop; PyTorch parallelizes it.
  • The kernel body — disort_impl(...) here — is the same function that would be called from a CUDA kernel (§05). One source of truth.

§05DispatchStub: per-device function pointers

DispatchStub is a small idiom you will see in every torch-extension repo. It looks like three macros in three different files, doing one thing: maintain a table of function pointers, indexed by device.

// 1) HEADER: declare the dispatch stub (a global function-pointer table)
using disort_fn = void (*)(at::TensorIterator&, int,
                           disort_state*, disort_output*);
DECLARE_DISPATCH(disort_fn, call_disort);

// 2) CPU CPP: define the table, then register the CPU implementation
DEFINE_DISPATCH(call_disort);
REGISTER_ALL_CPU_DISPATCH(call_disort, &disort::call_disort_cpu);

// 3) CUDA CU:  register the CUDA implementation into the SAME table
REGISTER_CUDA_DISPATCH(call_disort, &disort::call_disort_cuda);

At call time the table is keyed by a DeviceType:

at::native::call_disort(flx.device().type(),   // kCPU or kCUDA — picked at runtime
                        iter, upward, ds_.data(), ds_out_.data());

What just happened mechanically:

  1. DECLARE_DISPATCH creates a global object call_disort with a function-pointer slot per device.
  2. DEFINE_DISPATCH actually allocates that object in one translation unit (otherwise you’d get linker duplicates).
  3. Each REGISTER_*_DISPATCH macro expands to a static object whose constructor writes the function pointer into the right slot. The constructor runs when the .so loads.
  4. Calling call_disort(device_type, ...) looks up the slot and invokes the function.
Why this design wins

Two reasons. First, zero overhead at the call site — just an indirect function call. Second, the CUDA implementation can be in a separate .cu file, which only the CUDA-enabled build of the project even compiles. The CPU build doesn’t need an nvcc dependency; the CUDA-included build automatically picks up the GPU variant when present.

One subtle trap, mentioned for completeness

Because registration happens via a static constructor, if the linker decides the registration object is “unused” (nothing references its symbol) and strips it out, the constructor never runs and the device slot stays null — you get “Could not run with backend CUDA” at runtime even though the code compiled fine. The fix is a link-time flag (-Wl,--no-as-needed wrapping the CUDA library on the link line), which kintera handles explicitly in its setup.py. You don’t need to act on this in a tutorial reading, but knowing it exists saves an hour the first time you hit it.

§06nn::Module: a class that lives in two languages

The last piece is the user-facing wrapper. You want users to write:

disort = pydisort.Disort(opts)
flux   = disort.forward(properties, fbeam=..., umu0=...)    # returns a torch.Tensor

To make this work, you wrap your C++ solver class as a torch::nn::Module and use PyTorch’s pybind11 helper to bind it as a Python class. The mechanism, from pydisort:

class DisortImpl : public torch::nn::Cloneable<DisortImpl> {
  DisortOptions options;                       // shared_ptr to a config struct
  void reset() override;                       // (re)allocate internal state
  torch::Tensor forward(torch::Tensor prop,
                        std::map<std::string, torch::Tensor>* bc,
                        std::string bname = "", ...);
};
TORCH_MODULE(Disort);                            // alias: shared_ptr<DisortImpl> with sugar
PYBIND11_MODULE(pydisort, m) {
  ...
  ADD_DISORT_MODULE(Disort, DisortOptions)     // macro: bind_module + ctor + repr
      .def("forward", [](DisortImpl& self, torch::Tensor prop, ...,
                          const py::kwargs& kwargs) {
            // build boundary-condition map from Python kwargs
            // broadcast inputs to (nwave, ncol, ...)
            return self.forward(prop, &bc, bname, temf);
          });
}

Two things this buys you:

  • The C++ class is the Python class. A user’s disort.forward(...) call dispatches directly to the C++ method. No marshaling, no glue.
  • It plays well with torch. Because Disort is an nn::Module, it has .to(device), .state_dict(), can sit in an nn::Sequential — it acts like any other torch module, and torch handles parameter bookkeeping for you.

That said, nn::Module is convenience, not necessity. You could expose a plain pybind11 class. Using nn::Module is the choice you make when you want the wrapped op to live in the torch ecosystem rather than alongside it.

§07The recipe end to end, in 50 lines

Putting Parts II together. Here is the minimum viable torch-backed op. Imagine you have a C function void my_solver(double* out, const double* in, int nprop) that does some per-column calculation. The complete wrapping in this style is roughly:

using my_fn_t = void (*)(at::TensorIterator&);
DECLARE_DISPATCH(my_fn_t, call_my_solver);
void my_solver_cpu(at::TensorIterator& iter) {
  AT_DISPATCH_FLOATING_TYPES(iter.dtype(), "my_solver_cpu", [&] {
    iter.for_each([&](char** data, const int64_t* strides, int64_t n) {
      for (int64_t i = 0; i < n; i++) {
        auto out = reinterpret_cast<scalar_t*>(data[0] + i * strides[0]);
        auto in  = reinterpret_cast<const scalar_t*>(data[1] + i * strides[1]);
        my_solver(out, in, nprop);                  // the same C function
      }
    });
  });
}
DEFINE_DISPATCH(call_my_solver);
REGISTER_ALL_CPU_DISPATCH(call_my_solver, &my_solver_cpu);
void my_solver_cuda(at::TensorIterator& iter) {
  at::cuda::CUDAGuard g(iter.device());
  AT_DISPATCH_FLOATING_TYPES(iter.dtype(), "my_solver_cuda", [&] {
    native::gpu_kernel<2>(iter, [=] GPU_LAMBDA(char* const data[2], unsigned strides[2]) {
      auto out = reinterpret_cast<scalar_t*>(data[0] + strides[0]);
      auto in  = reinterpret_cast<const scalar_t*>(data[1] + strides[1]);
      my_solver(out, in, nprop);                  // SAME function — if it’s __host__ __device__
    });
  });
}
REGISTER_CUDA_DISPATCH(call_my_solver, &my_solver_cuda);
auto iter = at::TensorIteratorConfig()
   .add_output(out).add_input(in).build();

at::native::call_my_solver(out.device().type(), iter);     // dispatches CPU or CUDA

That is the whole skeleton. Notice that my_solver appears identically in both the CPU and CUDA paths. This is the “write once” promise of the architecture. The only thing that has to be different is the surrounding boilerplate that walks the iterator — and TensorIterator + DispatchStub give you both versions for free.

How does the same C function compile for both the CPU and the GPU? With a portability macro that expands to __host__ __device__ under nvcc and to nothing otherwise. kintera does this explicitly in its configure.h:

#ifdef __CUDACC__
  #define DISPATCH_MACRO __host__ __device__
#else
  #define DISPATCH_MACRO
#endif
template <typename T, int N>
DISPATCH_MACRO void interp_poly_impl(T* out, T* inp, T* coeff, ...) {
  for (int j = 0; j < nvar; ++j) {
    OUT(j) = 0.;
    for (int i = 0; i < N; ++i) OUT(j) += coeff[i] * INP(j, i);
  }
}

That single function body runs as serial C++ on CPU (inside iter.for_each) and as device-callable code on GPU (inside gpu_kernel). One file, two compilers, two devices.

Everything in this guide is a riff on this 50-line skeleton. The three case studies in Part III show how it scales to a real C solver, to a multi-domain module, and to a stencil that wants more. The deep dive in Part IV shows what happens when the default gpu_kernel isn’t enough and you replace it with a hand-written kernel that uses shared memory.

The pattern across three repos

§09pydisort: wrap a 1980s C solver

pydisort takes the 1980s discrete-ordinates radiative-transfer solver (cdisort, written in plain C) and exposes it as a torch op. The key observation: radiative transfer over (wavelength, column) pairs is embarrassingly parallel — each pair is one independent solve. So one element of the iteration = one DISORT call.

void call_disort_cpu(at::TensorIterator& iter, int upward,
                     disort_state* ds, disort_output* ds_out) {
  AT_DISPATCH_FLOATING_TYPES(iter.dtype(), "call_disort_cpu", [&] {
    int grain_size = iter.numel() / at::get_num_threads();
    iter.for_each([&](char** data, const int64_t* strides, int64_t n) {
        for (int i = 0; i < n; i++) {
          auto out  = reinterpret_cast<scalar_t*>(data[0] + i * strides[0]);
          auto prop = reinterpret_cast<scalar_t*>(data[1] + i * strides[1]);
          // ...11 more inputs unpacked similarly (boundary conditions, index)...
          int idx = static_cast<int>(*idxf);
          disort_impl(out, prop, /*…*/, ds[idx], ds_out[idx], nprop);
        }
      }, grain_size);
  });
}

Three details worth pulling out:

  • The C solver state is per-element. ds[idx] and ds_out[idx] are preallocated arrays of DISORT state structs, one per (wave, col). The kernel passes the right element’s state in. This is how a legacy C solver with mutable state becomes thread-safe in a torch-parallel context: you give each “element” its own copy.
  • The index input is a trick. A small int tensor whose ith entry is just i, broadcast into the iteration so each element knows its flat index for the state-array lookup.
  • The same body works on CUDA (in principle). pydisort’s CUDA registration is wired and uses gpu_kernel<12> over 12 broadcast inputs — though the disort_impl body itself is commented out in the .cu file today, marking it as a scaffold the GPU build registers but does not yet compute through. The infrastructure works; the kernel body is a TODO.

This case is the simplest possible application of the recipe: no shared state, no neighbor access, no stencils. Default machinery is enough.

§10kintera: chemistry as torch ops, multi-domain

kintera scales the pattern in two dimensions. First, the binding is split into six pybind11 translation units (one per scientific domain — thermo, kinetics, photochem, sparse-solver, diffusion, constants), each owning its own slice of the Python API. Second, the kernels actually are portable C++ that compiles for both CPU and GPU thanks to the DISPATCH_MACRO trick.

You can see the per-element bodies are pure C-style code that compiles for either device:

template <typename T>
DISPATCH_MACRO void interp_weno3_impl(T* out, T* inp, T* coeff, ...,
                                      int nvar, double scale) {
  for (int j = 0; j < nvar; ++j) {
    T vscale = scale ? (fabs(INP(j,0)) + fabs(INP(j,1)) + fabs(INP(j,2))) / 3.0 : 1.0;
    // ... WENO algorithm using normal C math: dots, squares, divisions ...
    OUT(j) = (alpha0*p0 + alpha1*p1) / (alpha0 + alpha1) * vscale;
  }
}

One template, two compilers. The CPU path wraps it with iter.for_each, the GPU path wraps it with gpu_kernel. The actual numerics — the only thing that needed thinking — lives in one file.

This is also the case where a non-torch question matters: some sub-ops need device-native libraries that PyTorch does not expose. The sparse linear solver for stiff chemical networks wants cusolver/cusparse on GPU. kintera links those directly (its setup.py includes them in the library list) and the sparse-solver binding calls them from inside its kernel. The torch surface still presents one Python op; underneath, the GPU path reaches into NVIDIA’s math libraries.

The progression so far

pydisort: one element = one heavy C call; default machinery suffices. kintera: one element = one C++ body, shared between devices via the portability macro; default machinery again, plus direct links to NVIDIA math libs where needed. snapy is the next step — the default machinery doesn’t suffice for the inner stencil kernel, and you write the CUDA part by hand.

§11snapy: setting up the stencil call

snapy is a fluid-dynamics code. Its high-order reconstruction step (WENO5) computes, for each cell face, an interpolated value from a 5-cell stencil — the cell to its left and right, two cells out on each side, plus the face itself. The interface looks identical to the previous repos:

auto iter = at::TensorIteratorConfig()
    .resize_outputs(false)
    .check_all_same_dtype(true)
    .declare_static_shape(out.sizes(), squash_dim)
    .add_output(out)
    .add_input(w)
    .build();

at::native::call_weno5(out.device().type(), iter, cm, dim, options->scale());

Same shape as §07. Same DispatchStub call. What changes is what happens inside the registered functions on different devices. The CPU and MPS paths use what torch gives you. The CUDA path drops down to a hand-written kernel that uses on-chip shared memory. That difference is the whole story of Part IV.

One small but important wrinkle visible above: squash_dim. For CPU, only dim 0 (the “number of variables” axis) is squashed — the iteration walks the spatial axes. For CUDA, the stencil axis is also squashed:

std::vector<int64_t> squash_dim = {0};
if (w.device().is_cuda()) {
  squash_dim.push_back(dim);                  // don’t parallelize across the stencil axis
}

Why: the CUDA kernel will parallelize within the stencil axis using thread indices, and use neighbor access along that axis — it doesn’t want TensorIterator to chunk that axis. The CPU kernel has no such constraint and would rather just see a flat sequence of points. This is a window into how the same op gets different framing on different devices — even though the user-visible API is identical.

Part IV

Snapy’s shared-memory stencil

The deep dive. What a stencil is, how snapy implements WENO5 three different ways on three different devices, why the CUDA version uses shared memory, and exactly what the dimension layout looks like — with diagrams.

§12What a stencil is (and the reuse it leaves)

A stencil is an operation where each output point reads from a small fixed window of input points. The simplest stencil — a 3-point average — reads in[i-1], in[i], in[i+1] for each output out[i]. WENO5 (Weighted Essentially Non-Oscillatory, 5th order) reads five contiguous input cells per output:

WENO5 stencil — to compute one output value at face i+½, read 5 cells: input cells: [ i-2 ][ i-1 ][ i ][ i+1 ][ i+2 ] │ │ │ │ │ └─────┴──┐ ┌┴──────┘ │ │ │ │ (nonlinear weights, smoothness measures, …) │ │ ▼ ▼ output: reconstructed value at face i+½

Now slide that window along the input array. For an input array of length N, you produce ~N outputs, each reading 5 cells. Counted naively, that’s 5N reads from input memory.

But look at the diagram: input cell i participates in five different output computations — the ones at faces i-2.5, i-1.5, i-0.5, i+0.5, i+1.5. If each output is computed independently in its own thread, that one input value is fetched from memory five times. That redundancy is what makes a stencil memory-bandwidth-bound, and it is the gap a good kernel closes.

Stencil kernels are the canonical example of an operation that is arithmetically light per memory load: each input byte is touched by ~5 outputs (for WENO5) but only a handful of floating-point ops — dot products, squares, divisions — happen once per output. The kernel’s performance is dominated not by how fast it computes but by how often each input value travels from global memory to a register. Bring each value across that wire once, reuse it five times near the ALU, and you go five times faster. That is what shared memory is for.

§13Three flavors of WENO5 side by side

snapy implements the same WENO5 algorithm three times — one for each device class. They are pedagogically perfect because they let you see what each approach asks of the device.

CPU: per-element loop

void call_weno5_cpu(at::TensorIterator& iter, torch::Tensor coeff, int dim, bool scale) {
  int grain_size = iter.numel() / at::get_num_threads();
  AT_DISPATCH_FLOATING_TYPES(iter.dtype(), "call_weno5_cpu", [&] {
    iter.for_each([&](char** data, const int64_t* strides, int64_t n) {
        for (int i = 0; i < n; i++) {
          auto out = reinterpret_cast<scalar_t*>(data[0] + i * strides[0]);
          auto w   = reinterpret_cast<scalar_t*>(data[1] + i * strides[1]);
          interp_weno5_impl(out, w, c, stride1, stride2, stride_out, nvar, scale);
        }
      }, grain_size);
  });
}

What it does: walk every output cell, call interp_weno5_impl with a pointer to where the 5-cell window starts. The CPU’s caches do the reuse work implicitly — if the array fits in L1/L2, repeated reads of the same value are cheap. You don’t explicitly manage reuse. This is why CPU code can be naive and still fast on stencils.

MPS: pure torch ops

void call_weno5_mps(at::TensorIterator& iter, torch::Tensor coeff, int dim, bool scale) {
  auto result = iter.output();
  auto w      = iter.input();

  auto wu = w.unfold(dim, 5, 1);                   // create the 5-wide stencil as a strided view
  auto beta1 = 13./12. * wu.matmul(c4).square() + 1./4. * wu.matmul(c5).square();
  auto beta2 = 13./12. * wu.matmul(c6).square() + 1./4. * wu.matmul(c7).square();
  auto beta3 = 13./12. * wu.matmul(c8).square() + 1./4. * wu.matmul(c9).square();
  auto alpha1 = 0.3 / (beta1 + 1e-6).square();
  auto alpha2 = 0.6 / (beta2 + 1e-6).square();
  auto alpha3 = 0.1 / (beta3 + 1e-6).square();
  torch::div_out(result,
      alpha1*wu.matmul(c1) + alpha2*wu.matmul(c2) + alpha3*wu.matmul(c3),
      alpha1 + alpha2 + alpha3);
}

What it does: composes the entire WENO5 algorithm out of existing torch ops. unfold(dim, 5, 1) creates a strided view that adds a new last dimension of size 5 — the stencil window — without copying. Then a sequence of matmul / square / divide ops, each backed by Apple’s Metal Performance Shaders, computes the answer. No custom kernel was written. This is the “just use torch” approach in its purest form, and it works because each op (unfold, matmul, square, divide) is already implemented on MPS by PyTorch. We will return to its cost in §19.

void call_weno5_cuda(at::TensorIterator& iter, at::Tensor coeff, int dim, bool scale) {
  at::cuda::CUDAGuard g(iter.device());
  AT_DISPATCH_FLOATING_TYPES(iter.common_dtype(), "call_weno5_cuda", [&]() {

      });

}

What it does: launches a custom CUDA kernel (native::stencil_kernel) that allocates shared memory, loads the stencil window into it cooperatively, synchronizes, and then has each thread compute its output from shared memory. The user lambda receives a scalar_t* smem argument that the kernel passes through — that’s the on-chip scratchpad. The reusable work happens explicitly. The rest of the chapters in this part dissect exactly how this kernel works.

§14The CUDA shared-mem kernel, annotated

The whole kernel lives in two files: a generic launcher (snapy/src/utils/loops.cuh) and the per-algorithm body (snapy/src/recon/interp_impl.cuh). We’ll read them in order: the body first (what each thread does), then the launcher (how blocks are sized and how shared memory is allocated).

The per-thread body

template <typename T>
__device__ void interp_weno5_impl(T* out, T* inp, T* coeff, int nvar,
                                  int stride_in1, int stride_in2,
                                  int stride_out1, int stride_out2,
                                  bool scale, T* smem) {
  int id = threadIdx.x;                  // MY index along the stencil axis
  int nt = blockDim.x;                   // number of threads in my block along that axis

  // PHASE 1 — cooperative load into shared memory --------------------------
  // Every thread loads ONE input value per variable into shared memory.
  T* sinp = smem;                          // shared-mem region 1: input cache
  for (int j = 0; j < nvar; ++j) {
    sinp[id + j * nt] = INP(j, id);       // global → shared, one read per thread per var
  }

  // Coefficients (45 doubles) are constant per block — load them once cooperatively.
  T* scoeff = smem + nt * nvar;            // shared-mem region 2: coefficient cache
  constexpr int N = 45;                  // 9 vectors × 5 coefficients
  for (int i = id; i < N; i += nt) {
    scoeff[i] = coeff[i];
  }

  if (id > nt - 5) return;                 // last 4 threads only loaded; no output for them
  __syncthreads();                         // barrier — all writes to smem are visible to every thread

  // PHASE 2 — compute, reading 5 neighbors from SHARED memory --------------
  T* c1 = scoeff;       // 5 coefficients for substencil-0 reconstruction
  T* c2 = c1 + 5;       // 5 for substencil-1
  T* c3 = c2 + 5;       // 5 for substencil-2
  T* c4 = c3 + 5;       // 5 for smoothness indicator 0 (part a)
  T* c5 = c4 + 5;       // 5 for smoothness indicator 0 (part b)
  // ... c6..c9 similarly

  T phi[5];
  for (int j = 0; j < nvar; ++j) {
    int i = id + j * nt;                 // my 5-wide window start in shared mem
    T vscale = scale ? (fabs(sinp[i]) + ... + fabs(sinp[i+4])) / 5.0 : 1.0;
    for (int k = 0; k < 5; ++k) phi[k] = sinp[i + k] / vscale;   // 5 reads from SHARED MEM

    T p0 = _vvdot<5>(phi, c1);              // substencil reconstructions
    T p1 = _vvdot<5>(phi, c2);
    T p2 = _vvdot<5>(phi, c3);
    T beta0 = 13./12. * SQR(_vvdot<5>(phi, c4)) + .25 * SQR(_vvdot<5>(phi, c5));
    // ... beta1, beta2 ...
    T alpha0 = .3 / SQR(beta0 + 1e-6);
    // ... alpha1, alpha2 ...
    OUT(j, id) = vscale * (alpha0*p0 + alpha1*p1 + alpha2*p2) / (alpha0+alpha1+alpha2);
  }
}

Four properties of this kernel are worth naming explicitly:

  • One global-memory read per (thread, variable). Each thread fetches its own input value once into sinp. There is no redundant load anywhere.
  • Coefficients are loaded cooperatively. Only one thread per index loads each coefficient; all threads then read from scoeff. Coefficients are the same for every output point, so this is a pure broadcast.
  • The barrier __syncthreads() is non-negotiable. Without it, thread i might read sinp[i+4] before thread i+4 has written it — a race. The barrier guarantees all writes to shared memory are visible to all threads in the block before any reads.
  • The last 4 threads load but don’t compute (the if (id > nt-5) return;). They are halo loaders: their job is to ensure threads before them have the right neighbors in shared memory. Their wasted output cell at the block boundary is the cost of doing business.

The launcher

template <typename scalar_t, int Arity, typename func_t>
void stencil_kernel(at::TensorIterator& iter, int dim, int buffers, const func_t& f) {
  // Figure out tensor lengths along each axis; rotate so the stencil axis is the inner one
  int len[3] = {1,1,1};
  len[3 + dim - ndim] = ensure_nonempty_size(iter.input(), dim);
  _left_shift<3>(len, dim + 1 - ndim);
  dim3 block(len[2], len[1], len[0]);     // stencil axis → block.x (threadIdx.x)

  // Grid covers the other axes
  dim3 grid(len[2]/block.x, len[1]/block.y, len[0]/block.z);

  // Shared-mem size: input cache (nt × nvar) + coefficient cache (buffers)
  int nvar = ensure_nonempty_size(iter.output(), 0);
  size_t shared = (block.x * nvar + buffers) * sizeof(scalar_t);

  auto stream = at::cuda::getCurrentCUDAStream();
  reduce_kernel<scalar_t><<<grid, block, shared, stream>>>(numel,
    [=] __device__ (int bid, scalar_t* smem) {
      auto offsets = offset_calc.get(bid);
      f(data.data(), offsets.data(), smem);    // hand smem to the user lambda
    });
}

The launcher does three things: (1) rotate the iteration extents so the stencil axis ends up as threadIdx.x (the fastest-varying), (2) compute the shared-memory size needed per block (per-thread input cache + coefficient buffer), and (3) launch a kernel whose lambda forwards the shared-memory pointer to the user’s function body. Everything else is plumbing.

What this picture says, in words:

  • Each block handles one row of x. The block has 256 threads — one per cell along x. There are 128 blocks (one per y row). All blocks run in parallel on the GPU’s streaming multiprocessors.
  • Within a block, every thread loads exactly one input cell into shared memory. 256 threads × 5 variables = 1280 doubles cached per block. That is one global-memory read per (thread, variable) — the minimum possible.
  • After the barrier, each thread reads its 5-element window from shared memory. Reads from shared memory are ~100× faster than reads from global memory. The reuse pattern from §12 (each input cell touched by 5 outputs) is now resolved on-chip instead of crossing the wire 5 times.
  • The last 4 threads can’t produce a full window (their id+4 would walk off the end of the loaded region), so they exit early. Their loads were still useful — they populated the halo their predecessors needed.
What if the row is bigger than a block can hold?

The launcher assumes the stencil axis fits in one block (block.x = len[2]). For very long rows, you would split into multiple blocks per row, each loading its own halo — that’s the “halo region” extension every shared-memory stencil tutorial talks about. snapy keeps it simple: rows fit. In a production-scale code, the extension is a fixed cost (16 extra threads per block load the halo, then return).

§16Why this is efficient: bandwidth math

Let’s put numbers on it. For our (5 vars, 256, 128) tensor doing WENO5 along x:

  • Total output cells: 5 × 256 × 128 ≈ 164,000.
  • Naive kernel (no shared memory, each thread reads its own 5-cell window): each output reads 5 doubles, so 5 × 164,000 = 820,000 global-memory reads = ~6.5 MB.
  • Shared-memory kernel: each thread loads exactly one input cell once. 5 × 256 × 128 = 164,000 input cells = ~1.3 MB. Coefficients are loaded once per block by 45 of the threads — effectively free.
  • Speedup factor, just from this: 5× less global-memory traffic, assuming the kernel was bandwidth-bound (which stencil kernels almost always are).

That’s the headline. But there’s a second, subtler win:

Memory tierLatency (cycles)BandwidthCost of a single read
Register1very highessentially free
Shared memory (L1-equivalent SRAM, on the SM)~20–30~10–15 TB/s on H100one cycle of work, then ready
L2 cache~200a few TB/stens of cycles — pipelined, but real
HBM global memory~500–700~3 TB/s (H100)hundreds of cycles per miss

Even when a naive kernel’s redundant reads are caught by L2 cache, each L2 hit still costs ~10× more than a shared-memory read. The shared-memory kernel moves the 4 redundant reads from L2 (slow) to shared (fast). For stencils, this is what closes the gap to peak bandwidth.

The win in one sentence: a shared-memory stencil kernel turns a memory-bandwidth-bound op into one that is bottlenecked by the fastest memory tier on the chip. Each input value crosses the global-memory wire exactly once, then the redundant reads required by the stencil’s shape are served from on-chip SRAM at ~10× the bandwidth and ~25× less latency than even L2. Stencils are the canonical case where this design pays for the extra programming effort.

Part V

Why not just use torch on CUDA?

The fairest possible question. If torch on CUDA already has fast unfold and matmul kernels — and they’re what the MPS path uses — why doesn’t snapy use them on CUDA too? Three reasons, each one a useful design lesson.

§17Why not just use torch ops?

You absolutely can, and you sometimes should — the MPS path proves it. But for a stencil on CUDA, three costs add up:

CostWhat it is & how it hurts
Kernel launcheseach torch op (matmul, square, add, divide) is a separate CUDA kernel launch. Each launch costs ~10µs of fixed overhead. The MPS WENO5 path has ~15–20 ops. For a small grid, the overhead can dominate the actual work.
Intermediate tensorseach op produces a full-size intermediate tensor written to global memory and read back by the next op. wu.matmul(c1) is a (5, 256, 128) tensor written to HBM; the next op reads it back. Bandwidth scales with the number of ops, not with what the algorithm fundamentally needs.
No on-chip reuseunfold creates a strided view, not a copy — good — but each downstream op still reads through global memory. The stencil’s natural 5× reuse is invisible to torch’s individual ops; they don’t share a shared-memory cache across op boundaries.

To answer the user’s specific question directly: yes, this is fundamentally because torch’s op interface doesn’t let you express “these N ops all want to share a chunk of shared memory inside a single fused kernel.” Each torch op is its own kernel. To get the shared-memory locality, you have to write the fused kernel yourself.

§18The kernel-launch & tensor-materialization tax

The cleanest mental model:

Pure torch on CUDA

Compute graph: 15 ops × 1 launch each = 15 kernel launches.

Bandwidth: every intermediate result is materialized in HBM — ~15 round trips of (5, 256, 128) doubles each = ~20 MB of bandwidth.

Reuse: the 5× stencil reuse is paid for in L2 cache hits at best — still 10× slower than shared memory.

Custom shared-memory kernel

Compute graph: 1 kernel launch. Total.

Bandwidth: ~1.3 MB read from HBM (each input once), result written once. ~2.6 MB total — about 8× less than the torch path.

Reuse: 5× reuse paid for in shared-memory hits — effectively register-speed.

For a (small) 256×128 grid, the launch overhead alone (15 × 10µs = 150µs) can be more than the actual compute. For a (large) 4096×4096 grid, bandwidth dominates and the 8×-less-traffic difference is what wins. Either way, the custom kernel is the better answer on CUDA — provided you’re willing to write and maintain it.

§19The MPS path is the counterfactual

Now — why use pure torch on MPS but not on CUDA? Two reasons:

  • The cost shape is different on Apple Silicon. Apple Silicon has unified memory, no discrete HBM, and very different shared-memory semantics; the “every op writes back to global memory” tax is smaller because there is no distinct global memory. The launch overhead is real but lower in absolute terms. The win from a custom shared-memory kernel is correspondingly smaller.
  • Writing Metal/Apple-Silicon shader code is an additional language and toolchain. CUDA gives you __global__/__device__ in your C++ source. Metal is Objective-C++ or Metal Shading Language plus a runtime. For a small marginal win, the engineering cost isn’t worth it — especially when the same algorithm composed out of torch ops just works on MPS for free.

So the MPS path is the answer to “what does pure torch give you for free?” It is, deliberately, the baseline you decided to beat on CUDA. Read the MPS code as “here is what we’d ship if we didn’t care about peak performance.” Compare its 15 lines to the CUDA kernel’s 100, and you see what the extra 85 lines bought you.

This also explains the squash_dim trick

Recall from §11 that the CUDA path squashes the stencil axis but the CPU/MPS paths don’t. The reason: the CUDA kernel handles the stencil axis itself via shared memory and thread indexing — it doesn’t want TensorIterator to chunk it. The CPU/MPS paths walk per-element through TensorIterator’s normal machinery, so they want the stencil axis treated normally. One line of code (squash_dim.push_back(dim) only on CUDA) encodes the entire structural difference.

§20When to stay on torch, when to drop down

The general decision rule, distilled from the three repos:

┌─ Is the kernel a hot inner loop you call millions of times? ──┐ │ │ NO YES │ │ stay on torch: ┌── Does it have spatial reuse ──┐ compose with torch ops │ (stencils, gather, attention, │ ── easiest, portable across │ shared accumulators)? │ every device torch supports │ │ ── default for pydisort whole- NO YES column ops & kintera per- │ │ element thermo use torch's default write a shared-memory gpu_kernel pattern CUDA kernel (snapy's ── same body for CPU+GPU stencil_kernel) ── pydisort & kintera ── pay the maintenance cost do this for the worst inner loop ── leave torch on for CPU / MPS / others

The honest costs of dropping down to a custom CUDA kernel:

  • Maintenance: two implementations of the same algorithm to keep in sync. snapy mitigates this by using the same per-thread body in CUDA and CPU; only the cooperation pattern (load to shared, syncthreads) differs.
  • ABI coupling: the kernel reaches into ATen-internal types (at::TensorIterator, at::native::ensure_nonempty_size) that have no stability guarantee across torch versions. Hence the exact torch==2.10.0 pin you see in all three repos.
  • Platform-specific: the CUDA kernel doesn’t run on MPS, AMD ROCm, or anywhere else. You need a torch-op fallback (the MPS path) for every other device.

So the architecture you end up with is a hybrid: torch handles 90% of the surface area for free, you pay the cost of a custom kernel only on the device + kernel combination that bottlenecks your application. That hybrid is performance-portability done correctly.

§22Decision trees

Should I wrap my algorithm as a torch op?

Will users want to compose its output with other torch tensors? ├─ YES → wrap as a torch op (this guide) └─ NO → just bind via pybind11 directly; skip TensorIterator + DispatchStub

How many device implementations do I write?

Does my per-element body work as plain C++ (no neighbor access, no shared state)? ├─ YES → write ONE body with DISPATCH_MACRO; let TensorIterator+gpu_kernel handle CPU+CUDA+... │ (pydisort, kintera's per-element ops) └─ NO → write the body once for CPU/MPS via torch ops, and once for CUDA as a custom kernel (snapy's WENO5: CPU+MPS use torch composition, CUDA uses shared_memory_kernel)

When to write a hand kernel for CUDA?

Bottleneck profiling shows the op is bandwidth-bound AND has spatial reuse (stencil/gather/attention)? ├─ YES → custom shared-memory kernel (~5–10× speedup typical) └─ NO → stay on torch; reach for cuBLAS/cuSOLVER calls if you need GEMM-shaped ops
