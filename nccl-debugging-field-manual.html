Chapter I

What NCCL is & how it moves bytes.

Read time 5 min

The premise Every synchronous distributed step ends in a collective. Every collective is a barrier. Every barrier can hang.

Mantra A hang is not a bug. It's a missing rank.

NCCL (NVIDIA Collective Communications Library) is the layer underneath your torch.distributed calls. PyTorch hands it a tensor and an opcode — "all-reduce, sum, fp16" — and NCCL takes care of choosing rings, picking transports, dispatching to NVLink or InfiniBand, and barrier-synchronizing all ranks until everyone has the same answer.

That last part — "barrier-synchronizing" — is the entire story of NCCL debugging. A collective is an all-or-nothing handshake. If one rank shows up at the meeting and the others don't, that rank waits. Forever, by default. The hang you see at 3 a.m. is one rank, somewhere, that never made it to the meeting.

The three-layer stack

Your code. You write loss.backward() or dist.all_reduce(tensor). Under the hood, PyTorch's DDP/FSDP hooks queue work into a process group.

The c10d process group. PyTorch's distributed coordinator. It owns the watchdog (the thing that fires the timeout you see in the stack trace), it owns the Flight Recorder (the thing that tells you why the hang happened), and it owns the call into NCCL.

NCCL itself. A C++ library that talks to the GPUs through CUDA streams. It picks a topology-aware algorithm (ring, tree, or NVLS), serializes the tensor through whichever transport is fastest between each pair of GPUs (NVLink → PCIe → SHM → IB → TCP), and waits for every peer's contribution before producing the output.

Why so many things break

Because NCCL spans all of: CUDA streams, NVIDIA drivers, kernel networking, RDMA verbs, InfiniBand fabric, Ethernet switches, NUMA topology, container namespaces, and your application code's collective-ordering logic. A failure in any of these surfaces as the same symptom: a hang followed, ten or thirty minutes later, by a watchdog timeout.

This is the heart of NCCL debugging — the symptom is uniform, but the cause is not. You need telemetry to localize where the chain broke.

The single most important sentence in this manual

NCCL failures are almost never bugs in NCCL. They are symptoms of: a missing peer (rank crashed elsewhere), a collective-ordering bug (your code), a hardware fault (GPU XID, IB link flap), or a misconfiguration (wrong interface, blocked port, missing GDR). Debug accordingly.

Chapter II

The collectives, visually.

Click Each tab. The illustration shows the data movement; the caption tells you which parallelism strategy uses it.

Why you care Knowing the access pattern tells you which collective will hang first. AllReduce hangs uniformly. AllGather hangs asymmetrically.

Six collectives carry essentially all of distributed training. Each has a distinct data-flow pattern and a distinct failure mode.

Interactive · Collective Patterns

What each one is used for

AllReduce is the workhorse of data parallelism. Each rank computes a local gradient; AllReduce sums them and gives every rank the result. PyTorch DDP fires one AllReduce per parameter bucket per step.

AllGather and ReduceScatter together implement ZeRO/FSDP. AllGather rebuilds the full parameter from shards on each rank before a forward pass; ReduceScatter distributes gradient shards back. A single FSDP step fires hundreds of these.

Broadcast sends one rank's value to everyone. Used for parameter initialization, sending hyperparameter updates, and to synchronize random seeds.

AllToAll is the MoE collective. Each rank sends a slice of its data to every other rank. Expert parallelism, sequence parallelism, and some forms of tensor parallelism use it. It's the most bandwidth-hungry collective and the first to expose a weak network link.

Barrier is a pure synchronization point. No data moves; ranks just confirm "we're all here." Often inserted around checkpoint writes or evaluation.

Pattern → likely failure mode

If AllReduce hangs, suspect a missing rank (one peer died) or a collective-ordering mismatch (your code dispatched different collectives on different ranks). If AllGather/AllToAll hangs in the middle of a healthy run, suspect transient network issues (IB link flap, RoCE packet drops). If Broadcast hangs at start-up, suspect bootstrap connectivity (wrong NCCL_SOCKET_IFNAME, blocked port).

Chapter III

Anatomy of a hang.

The taxonomy Four classes, four signatures. Learn to recognize each from the first 20 lines of log.

Don't waste time Restarting blindly works ~30% of the time and teaches you nothing. Diagnose first.

Per the PyTorch team's published taxonomy, every NCCL hang you'll ever see falls into one of four buckets. The buckets are distinguishable in the trace data, and each has its own remedy.

Class 1
Collective mismatch

Different ranks called different collectives, or called them in different orders, or with different shapes/dtypes. Application bug.

Class 2
GPU hang

One rank's GPU got stuck doing pre-collective work (a slow custom kernel, a CPU/GPU sync, an unrelated CUDA call). Peers waiting at the barrier. Latency bug.

Class 3
Misconfigured args

All ranks called the same collective, but with mismatched dtype/size — e.g. an fp16 tensor on rank 0, fp32 on rank 1. Type bug.

Class 4
Faulty hardware / network

Link flap, NIC reset, GPU XID, ECC error. Symptoms can mimic the others — distinguishing feature is repeat patterns across jobs.

The signature of each class

Class 1 — Collective mismatch. Flight Recorder shows ranks executing different collective opcodes at the same sequence ID. Rank 0 ran AllReduce; ranks 1–7 ran AllGather. The hang is at the first collective that diverged.

Class 2 — GPU hang. All ranks have the same collective queued, but one rank's collective is in state scheduled while the others are in state started. The culprit hasn't reached its ncclAllReduce call yet — something upstream is taking too long. Often a Python-level CPU sync, a profiler attached, or an unrelated CUDA kernel.

Class 3 — Misconfigured args. Same opcode, same sequence ID, but Flight Recorder dumps show mismatched dtype, count, or root rank. This is detected and reported by NCCL itself in many recent versions: NCCL WARN Mismatched collective ...

Class 4 — Hardware/network. All ranks at the same collective, all in state started, none completing. ibstat shows port DOWN, dmesg shows GPU XID 79 (GPU lost from bus) or XID 13/31 (graphics engine errors). The diagnostic move is to repeat the job — if it fails on the same node, swap the node.

The signature you must learn first

When you see Watchdog caught collective operation timeout in the log, look at which ranks survived. If all 8 ranks of one node are missing and the other 56 are stuck, that's a node failure (Class 4). If 1 rank of each node is missing, that's a process or GPU failure within each node. If all 64 ranks are stuck and none crashed, that's Class 1, 2, or 3 — Flight Recorder will tell you which.

Chapter IV

The debugging telemetry.

Set once Bake these env vars into every job. The overhead is <1% and a single avoided debugging session pays for it forever.

Without telemetry You are guessing. With telemetry, you are diagnosing.

Three sources of telemetry, layered. NCCL logs tell you what NCCL did. The PyTorch Flight Recorder tells you what every rank's process group was doing at the moment of the hang. Stack dumps tell you where the Python code was. With all three, the hang's cause is usually obvious within 10 minutes.

Layer 1 — NCCL logs

# Always-on, low-overhead
export NCCL_DEBUG=INFO
export NCCL_DEBUG_SUBSYS=INIT,ENV,GRAPH,TUNING

# Critical: per-rank file output, never to stdout in prod
export NCCL_DEBUG_FILE=/var/log/nccl/nccl.%h.%p.log
# %h = hostname, %p = pid. Per-rank file or output gets mangled.

# For triage of an active issue, also add:
export NCCL_DEBUG_SUBSYS=INIT,ENV,GRAPH,TUNING,COLL,NET

The default NCCL_DEBUG_SUBSYS is INIT,BOOTSTRAP,ENV — enough to confirm setup but blind to collectives. Add COLL only when you're actively debugging; it logs every collective and explodes log size 100×.

Layer 2 — PyTorch Flight Recorder

The Flight Recorder is a ring buffer inside c10d that records every collective each rank issues: opcode, input/output sizes, dtype, sequence ID, state (scheduled, started, completed), C++ stack, Python stack. When a hang triggers the watchdog, every rank dumps its buffer to disk. The fr_trace tool aligns the dumps across ranks and tells you exactly where they diverged.

# Set in env BEFORE process group init
export TORCH_NCCL_TRACE_BUFFER_SIZE=2000        # keep last 2000 ops
export TORCH_NCCL_DUMP_ON_TIMEOUT=1              # dump on hang
export TORCH_NCCL_DEBUG_INFO_TEMP_FILE=/var/log/fr/dump_%h_%p
export TORCH_NCCL_ENABLE_MONITORING=1            # watchdog

# For deeper diagnosis (more overhead, only when investigating):
export TORCH_NCCL_TRACE_CPP_STACK=1
export TORCH_NCCL_ENABLE_TIMING=1

# Then, after a hang, analyze the dumps:
python -m torch.distributed.checkpoint.fr_trace \
  /var/log/fr/ --selected-ranks=0,1,7,15

Layer 3 — In-process stack dumps

Flight Recorder tells you what collective hung. Sometimes you also need to know where in Python the offending rank was when it issued the divergent call. The lightweight option: install py-spy in the image and run, against a hung rank's PID:

# From the host of a hung rank:
sudo py-spy dump --pid <rank_pid>          # Python stack
sudo gdb -p <rank_pid> --batch -ex bt    # C++ stack

You learn whether the rank is inside Python (likely Class 1 or 2), inside CUDA (likely Class 2), or inside NCCL's commWait loop (likely Class 4).

The minimum-viable telemetry

If you do nothing else: set NCCL_DEBUG=INFO, NCCL_DEBUG_FILE, TORCH_NCCL_DUMP_ON_TIMEOUT=1, and TORCH_NCCL_TRACE_BUFFER_SIZE=2000. That's four lines in your launch script and it pays for itself the first time you hang.

Reading a log like a coroner.

The order Read top-down. The first anomaly is the cause. Everything below it is a consequence.

A real Flight Recorder dump from a 64-GPU job that hung at step 4,128. Click any line to read what it tells you and what to do about it.

Excerpt · nccl.log + flight recorder · step 4128

[2026-04-12 14:23:17.401] gpu-node-07:1834:1956 [3] NCCL INFO Using network IB
[2026-04-12 14:23:17.402] gpu-node-07:1834:1956 [3] NCCL INFO NET/IB : Using [0]mlx5_0:1/IB [RO]; OOB ib0:10.0.7.4
[2026-04-12 14:23:17.512] gpu-node-07:1834:1956 [3] NCCL INFO Channel 00/0 : 27[3] -> 28[4] via NET/IB/0/GDRDMA
[2026-04-12 14:23:18.001] gpu-node-07:1834:1956 [3] NCCL INFO comm 0x7f9c... rank 27 nranks 64 cudaDev 3 busId 47000 - Init COMPLETE
[2026-04-12 16:48:13.882] gpu-node-07:1834:1956 [3] NCCL WARN Watchdog caught collective operation timeout: WorkNCCL(SeqNum=18244, OpType=ALLREDUCE, Timeout(ms)=600000)
[2026-04-12 16:48:13.884] gpu-node-07:1834:1956 [3] FR DUMP rank=27 sequence=18244 state=started input=[131072000] dtype=bfloat16
[2026-04-12 16:48:13.886] gpu-node-31:8821:8902 [7] FR DUMP rank=39 sequence=18243 state=completed last_seq=18243
[2026-04-12 16:48:14.001] gpu-node-04:2105:2230 [0] NCCL ERROR Bootstrap : send failed : Connection reset by peer (104)
Line 1 — Setup confirmation NCCL chose the IB transport. If you wanted IB but see Using network Socket here, your IB stack didn't initialize — check ibstat, the device plugin, and NCCL_IB_HCA.
Line 2 — Which NIC is being used mlx5_0:1 means port 1 of the first Mellanox HCA. OOB ib0 is the out-of-band channel for bootstrap. If you have multiple HCAs and you're not seeing all of them, set NCCL_IB_HCA=mlx5_0,mlx5_1,mlx5_2,mlx5_3 explicitly. Wrong HCA selection halves bandwidth silently.
Line 3 — A channel is set up over GDRDMA GDRDMA means GPUDirect RDMA — GPU memory talks directly to the NIC without bouncing through host RAM. This is what you want. If you see NET/IB/0/0 (no GDRDMA suffix), GPUDirect is NOT engaged and you'll lose 3–10× bandwidth.
Line 4 — Communicator initialized successfully The rank joined the global communicator. If init fails, you'll see this line missing for one or more ranks; check the bootstrap section (line 8).
Line 5 — THE HANG · 2 hours 25 minutes after init The watchdog fired after 600 seconds with no progress on collective sequence 18244 (an AllReduce on a 131M-element bfloat16 tensor — that's a gradient bucket of ~250MB). This is the symptom. The next two lines are the diagnosis.
Line 6 — Rank 27's view Rank 27 issued AllReduce sequence 18244 and the collective is in state started. It's waiting for peers.
Line 7 — Rank 39's view · THE SMOKING GUN Rank 39 (on a different node) shows its last completed sequence was 18243. It never issued sequence 18244. Rank 39 is the culprit — something prevented it from reaching the next AllReduce. This narrows the search to "what happened on gpu-node-31 between 14:23 and 16:48?"
Line 8 — The root cause A different rank on a third node logged Connection reset by peer. This is a TCP-level failure on the bootstrap channel — almost always means the peer process died. Cross-reference: was a rank OOM-killed? Did a node go NotReady? Did dmesg show an Xid error? The hang is downstream of a process death somewhere else.

The reading method

Walk the log from top to bottom. Skip anything labeled INFO once init looks clean. Stop at the first WARN or ERROR. From there, you're looking for two things: what sequence number was active, and which rank diverged from the others. Flight Recorder gives you both. The rest is figuring out why that rank diverged — and the answer is almost always elsewhere in the log (another rank's earlier failure) or on the system (kubectl describe, dmesg, ibstat).

Chapter VII

Eight real hangs, annotated.

The cases All seen in production. Each case: context, symptom, diagnosis, fix.

Pattern recognition After reading these, you'll recognize ~80% of the hangs you'll encounter.

Eight hangs from real training clusters, in roughly increasing order of subtlety. Each is presented as a case file: the context, the symptom, how to debug, and the fix.

01
The OOM elsewhere
Class 1 · Missing rank Severity: high Frequency: very common
Context

64-GPU pretraining run, FSDP. Has been running 18 hours. Suddenly hangs.

Symptom

Watchdog timeout on rank 0. Stack trace shows it's stuck in AllReduce. All other ranks log identical messages. Logs look identical and unremarkable for the 30 minutes prior.

Debug

Check kubectl get pods: rank 47's pod is in OOMKilled status. dmesg on its node shows kernel OOM killer fired ~2 minutes before the hang. Rank 47 died; the other 63 ranks waited at the next collective. The hang report on rank 0 is downstream noise.

Fix
# Increase memory request on the pod spec
resources:
  requests: { memory: 320Gi }    # was 256Gi
  limits:   { memory: 320Gi }

# Lower per-rank activation memory in PyTorch
fsdp_config = FullyShardedDataParallelPlugin(
    activation_checkpointing=True,
    cpu_offload=CPUOffload(offload_params=False),
)

Permanent fix: enable activation checkpointing at the right granularity (transformer blocks usually), validate the new memory ceiling with a 1-hour smoke test, and add a memory-usage Prometheus alert so the next OOM is caught before it cascades.

02
The conditional collective
Class 1 · Collective mismatch Severity: med Frequency: common
Context

RLHF training. Custom logic that occasionally logs eval metrics only from rank 0.

Symptom

Run hangs once every ~500 steps, never the same step. Flight Recorder shows rank 0 stuck in all_gather sequence 9842; all other ranks stuck in all_reduce sequence 9842.

Debug

The collective opcode mismatch at the same sequence ID is the giveaway — Class 1. Search the code for any if rank == 0: or if some_condition: blocks containing a collective. Found:

if step % 100 == 0 and should_eval(metrics):
    # rank 0 gathers eval samples
    samples = dist.all_gather(local_samples, ...)
else:
    # normal step
    dist.all_reduce(grads, ...)

should_eval() uses local rank-0 metrics, so different ranks make different branch choices. They dispatch different collectives at the same sequence ID. Hang.

Fix

Two rules, never broken: (1) every rank must dispatch the same sequence of collectives in the same order; (2) any branch that contains a collective must be decided by a broadcast from one rank, not by local state.

if rank == 0:
    do_eval = should_eval(metrics)
else:
    do_eval = False
do_eval_tensor = torch.tensor([do_eval], device=device)
dist.broadcast(do_eval_tensor, src=0)   # everyone learns
if do_eval_tensor.item():
    dist.all_gather(...)
else:
    dist.all_reduce(...)
03
The dtype drift
Class 3 · Misconfigured args Severity: med Frequency: uncommon
Context

Mixed-precision training. One rank uses torch.autocast from inside a helper module; another rank's code path doesn't reach that module on certain steps.

Symptom

NCCL itself prints a warning: NCCL WARN Mismatched collective: type fp16 on rank 3, type fp32 on rank 5. Then hangs. (In newer NCCL versions; older ones just hang silently.)

Debug

The error message is precise. Flight Recorder confirms: same opcode, same sequence, different dtype. Search the code for places where a tensor's dtype depends on local state — autocast contexts, conditional .float() casts, model-side type coercion.

Fix

Cast the gradient bucket explicitly to a known dtype before the collective. PyTorch's DDP/FSDP do this automatically for the gradient sync, but custom dist.all_reduce calls in your code don't. Add .to(torch.bfloat16) before the call, or wrap with a dist.barrier() + dtype assertion in dev mode.

04
The IB link flap
Class 4 · Network Severity: high Frequency: common at scale
Context

512-GPU run. Healthy for 30 hours. Suddenly all ranks stall.

Symptom

Flight Recorder: all 512 ranks at the same collective, all in state started, none completing. No rank crashed. No collective mismatch. The cluster is just frozen mid-AllReduce.

Debug

The pattern — everyone at the same state with no rank missing — points at network. Run on each node:

# Check link state on every node
ibstat | grep -E "State|Rate"
# If any port shows "Initializing" or "Down" → flap

# Check switch counters (requires fabric mgmt access)
ibdiagnet --pc

# Kernel ring buffer for NIC events
dmesg -T | grep -iE "mlx5|infiniband|link"

Result: gpu-node-23's mlx5_2 port flapped at 15:42:18, came back at 15:42:23. NCCL's connections through that NIC died but never reconnected. The hang is permanent because NCCL doesn't auto-retry transport-level failures by default.

Fix

Immediate: kill the job and restart from the last checkpoint. NCCL cannot recover from a transport reset.

Mid-term: enable NCCL fault tolerance via NCCL_IB_RETRY_CNT=12 and NCCL_IB_TIMEOUT=24, and at the PyTorch level use torchft or fault-tolerant collectives that can re-form the process group around a failed node.

Long-term: install IB health monitoring — Prometheus exporters for port_xmit_discards, symbol_errors, and link state changes. Alert the moment any of these climb, swap the offending hardware before the next hang.

05
The slow rank (straggler)
Class 2 · GPU hang Severity: med Frequency: very common
Context

Vision training. Most steps take 1.2s; occasionally one takes 35s and the job nearly times out.

Symptom

No hang per se — just severe step-time variance. Flight Recorder (with TORCH_NCCL_ENABLE_TIMING=1) shows that during the slow steps, one rank's start time for the AllReduce is 30 seconds after the others. The collective itself is fast; the rank's arrival at it is slow.

Debug

Class 2 signature: peers waiting, one rank not yet at the collective. The slow rank's GPU is busy on something before the collective. Causes (in order of likelihood): a CPU sync (tensor.item(), tensor.tolist(), print(tensor)), a slow data-loader returning, a custom kernel with awful occupancy, or a profiler attached.

Tool of choice: NVIDIA Nsight Systems profiler on the slow rank for one step. The gap before AllReduce will reveal what's filling it.

Fix

Most common cause: a .item() or print(loss) inside the training step on a rank whose loss happens to vary more. The implicit cudaStreamSynchronize blocks the CPU. Move logging to a fire-and-forget background thread, or only log every N steps from rank 0 after a fast dist.reduce.

If it's a slow data loader: bigger prefetch_factor, more workers, persistent workers, and verify /dev/shm is big enough (mount as emptyDir: { medium: Memory } in K8s).

06
The wrong network interface
Class 4 · Misconfig Severity: high Frequency: at every new cluster
Context

New cluster deployed. Single-node 8-GPU runs work. Multi-node runs hang immediately at init_process_group.

Symptom

Bootstrap : send failed : Connection refused (111), or hangs forever during init. Sometimes a verbose could not find a working interface warning.

Debug

NCCL picked the wrong interface for the bootstrap (rendezvous) channel — probably a Docker bridge, a management LAN, or a flannel/cilium overlay that can't reach the other nodes' pod IPs. Show interfaces:

# From inside the pod
ip addr show
getent hosts <other-rank-hostname>

# What did NCCL pick?
grep -E "OOB|using.*interface" /var/log/nccl/*.log

NCCL log shows it picked docker0 on one rank and eth0 on another. They can't reach each other.

Fix
# Force NCCL to use the right Ethernet interface for bootstrap
export NCCL_SOCKET_IFNAME=eth0

# Force which IB HCAs to use for data
export NCCL_IB_HCA=mlx5_0,mlx5_1,mlx5_2,mlx5_3

# Exclude Docker / overlay interfaces explicitly
export NCCL_SOCKET_IFNAME=^docker,lo,veth

Many cluster operators bake the right values into a /etc/nccl.conf on every node so jobs don't have to know.

07
The GPU XID
Class 4 · Hardware Severity: high Frequency: 1 per 100 GPU-weeks
Context

256-GPU job. Hangs at step 12,000. Restart from checkpoint — hangs again at step 12,000-ish. Restart on different nodes — runs fine.

Symptom

One node's pod logs show CUDA error: an illegal memory access was encountered moments before the hang. Other nodes show the standard watchdog timeout. dmesg on the suspect node shows NVRM: Xid (PCI:0000:5e:00): 79, pid='<unknown>', name=<unknown>, GPU has fallen off the bus.

Debug

Xid 79 = GPU lost from PCIe bus, almost always fatal hardware. Other useful Xids to recognize: 13 (graphics engine illegal access), 31 (GPU memory page fault), 48 (double-bit ECC error), 63/64 (page retirement). Check DCGM:

nvidia-smi --query-gpu=index,name,gpu_uuid,gpu_serial,inforom.ecc \
  --format=csv

dcgmi diag -r 3      # run a thorough hardware diag
nvidia-smi -q -d ECC | grep -i error
Fix

Cordon and drain the node in Kubernetes (kubectl cordon <node> && kubectl drain <node>). File a hardware RMA. Add the node's GPU UUID to a "known-bad" list so the scheduler avoids re-using it even after reboot. Some teams maintain an automated "bad GPU" admission webhook that blocks pods from running on flagged hardware until a tech replaces it.

08
The CUDA-graph capture deadlock
Class 2 · GPU hang Severity: subtle Frequency: rare, but vicious
Context

Tensor-parallel training using CUDA graphs to reduce kernel-launch overhead. Two communicators in use (TP group + DP group). Hangs intermittently inside a graph launch.

Symptom

The hang is rare and not at a reproducible step. py-spy on the hung rank shows the Python thread inside cudaGraphLaunch; another thread holds the GIL inside a CUDA API call. The watchdog timeout fires but the dump is incomplete because the watchdog itself can't proceed.

Debug

This is a known interaction between CUDA graph launches and multiple NCCL communicators in the same process. The NCCL docs specifically call it out — if multiple ranks launch the same graph with NCCL inside from the same thread, the CUDA launches themselves can deadlock.

Fix
# Disable cross-communicator graph optimization
export NCCL_GRAPH_MIXING_SUPPORT=0

# Or use explicit grouped semantics
export NCCL_LAUNCH_MODE=PARALLEL

# And ensure the watchdog can dump even when blocked
export TORCH_NCCL_DESYNC_DEBUG=1
export TORCH_NCCL_DUMP_ON_TIMEOUT=1

Long-term: avoid using CUDA graphs across NCCL communicator boundaries until you have measured a real speedup that justifies the operational cost.

What these eight have in common

Every case here was diagnosed by the same method: read the Flight Recorder, find the divergent rank, look on that rank's node. The path from symptom to fix takes 10 minutes when telemetry is enabled and 4 hours when it isn't.

Chapter VIII

Environment variables that matter.

The discipline Most env vars are for tuning, not debugging. Keep tuning vars in /etc/nccl.conf; keep debugging vars in your launch script and remove them when issue is resolved.

NCCL's own advice Don't keep debug vars set in production — they can degrade performance or trigger hangs.

Twenty env vars cover essentially every real-world NCCL situation. Memorize the first eight; reference the rest.

Variable
What it does · when to set
NCCL_DEBUG
VERSION | WARN | INFO | TRACE. Default is silent. Set to INFO always in dev/staging; WARN in prod (keeps warnings but trims volume).
NCCL_DEBUG_SUBSYS
INIT, COLL, NET, GRAPH, TUNING, ENV, ALLOC, P2P, NVLS, BOOTSTRAP, REG, ALL. Default INIT,BOOTSTRAP,ENV. For debugging hangs add COLL,NET. Prefix with ^ to exclude.
NCCL_DEBUG_FILE
Per-rank file output. Use template like /var/log/nccl/nccl.%h.%p.log. Without this, all ranks write to stdout and you get a tangle.
NCCL_SOCKET_IFNAME
Which Ethernet interface for bootstrap. Comma-separated prefix list. Prefix with ^ to exclude (^docker,lo,veth). The #1 cause of init hangs.
NCCL_IB_HCA
Which IB adapters for data. Set explicitly (mlx5_0,mlx5_1,mlx5_2,mlx5_3) — auto-discovery sometimes misses HCAs or picks the wrong ones.
NCCL_IB_DISABLE
0 or 1. Set to 1 to force fallback to TCP over Ethernet. Use as a test: if the hang goes away with IB disabled, the IB fabric is the culprit. Never run production this way — 10–50× bandwidth loss.
NCCL_P2P_DISABLE
0 or 1. Disables NVLink/PCIe direct GPU-to-GPU. Use only to test whether a P2P path is broken. Default 0.
NCCL_ALGO
Tree | Ring | CollnetDirect | NVLS. Forces a specific algorithm. Useful to bisect: if hang only happens with Ring, suspect a stuck peer in the ring; if only with Tree, suspect topology miscalculation.
NCCL_PROTO
Simple | LL | LL128. Protocol. LL128 is fastest on NVLink-class fabrics. Force Simple if you suspect a protocol-related corruption.
NCCL_NET_GDR_LEVEL
LOC | PIX | PXB | PHB | SYS. Maximum NIC-to-GPU distance for using GPUDirect RDMA. Default is auto. Set to PHB or higher if GDR is being skipped.
NCCL_IB_TIMEOUT
IB retransmit timeout exponent. Default 18 (~4 sec). Increase to 22 in noisy fabrics. Tuning, not debugging.
NCCL_IB_RETRY_CNT
Number of retries per packet. Default 7. Bump to 12 if you observe transient link issues that aren't true link-downs.
NCCL_BUFFSIZE
Per-channel buffer size in bytes. Default 4MB. Increase for large messages (8MB or 16MB) if all-reduces dominate; never decrease without measurement.
NCCL_NTHREADS
Number of CUDA threads per block in NCCL kernels. Don't touch unless guided by NCCL-tests output.
NCCL_ASYNC_ERROR_HANDLING
0 or 1. When 1, NCCL errors are surfaced to Python as exceptions instead of aborting the process. PyTorch sets this automatically. Useful for fault-tolerant frameworks.
TORCH_NCCL_BLOCKING_WAIT
0 or 1. When 1, work.wait() blocks the CPU instead of waiting on the CUDA stream. Combine with TORCH_NCCL_ASYNC_ERROR_HANDLING=1 for fault tolerance.
TORCH_NCCL_DUMP_ON_TIMEOUT
0 or 1. Flight Recorder dumps trace buffers when watchdog fires. Set to 1 always. Cost is near zero; debugging payoff is enormous.
TORCH_NCCL_TRACE_BUFFER_SIZE
Ring buffer size in collectives. Default 0 (off). Set to 2000+ in production. Each entry is ~few KB, so 2000 ≈ 10MB per rank — trivial.
TORCH_NCCL_ENABLE_MONITORING
0 or 1. Enables the heartbeat watchdog. Default 1. Don't disable unless debugging the watchdog itself.
TORCH_NCCL_DESYNC_DEBUG
0 or 1. Extra diagnostics when collectives desync. Set to 1 during incident investigation; the overhead is non-trivial so don't leave on.

A word on the timeout

The default NCCL watchdog timeout in PyTorch is 30 minutes for some operations, 10 minutes for others. Long-running collectives can legitimately exceed this — particularly the first AllReduce after model load (can take minutes for very large models). If you see "false positive" timeouts during init, raise TORCH_NCCL_HEARTBEAT_TIMEOUT_SEC and the per-PG timeout. But: if you're raising it because timeouts happen mid-training, you have a real hang and you're hiding it. Don't do that.

The smoke test you should always run

Before any multi-day run on N nodes, run a 5-minute scaled-down version of the same code on N nodes. This catches 90% of integration bugs before they cost you a checkpoint's worth of compute. The smoke test below is the canonical one:

import torch, torch.distributed as dist, time
dist.init_process_group(backend="nccl")
rank, world = dist.get_rank(), dist.get_world_size()
torch.cuda.set_device(rank % torch.cuda.device_count())

# 1. all-reduce smoke test, sized like a real gradient bucket
t = torch.randn(256_000_000, dtype=torch.bfloat16, device="cuda")
torch.cuda.synchronize()
t0 = time.time()
for _ in range(20):
    dist.all_reduce(t)
torch.cuda.synchronize()
bw = 20 * t.numel() * 2 / (time.time() - t0) / 1e9
if rank == 0:
    print(f"AllReduce: {bw:.1f} GB/s busbw")

# 2. barrier sanity
dist.barrier()

# 3. broadcast sanity (catches NIC asymmetry)
t2 = torch.zeros(1024, device="cuda") if rank else torch.ones(1024, device="cuda")
dist.broadcast(t2, src=0)
assert t2.sum().item() == 1024, f"rank {rank} got wrong broadcast"

dist.destroy_process_group()

Run it across your full cluster. If the bandwidth is <80% of theoretical or if it hangs, you have a problem to fix now, not at hour 47 of a real run.
