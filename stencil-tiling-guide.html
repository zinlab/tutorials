01

What a stencil actually is

A stencil updates every point of a grid using a small, fixed pattern of neighbours. Same shape, applied everywhere.

The canonical 1D version — a 3-point stencil with radius R = 1:

math · 1D 3-point radius 1 → 2R+1 = 3 taps
out[i] = c0·in[i-1] + c1·in[i] + c2·in[i+1]

Two numbers define the whole problem:

  • Radius R — how far each output reaches. A 3-point stencil has R = 1; a 5-point 1D stencil has R = 2.
  • Tap count — the number of inputs read per output. In 1D that's 2R+1; for a 2D cross (von Neumann) stencil it's 4R+1 (5 taps at R = 1).

That tap count is the source of all the inefficiency we're about to fight: neighbouring outputs overlap heavily, so the same input element gets read from global memory many times.

02

Row-major arrays & the index algebra

GPU memory is a flat 1D address space. A 2D grid is stored row by row — that single fact drives every index you'll write.

In row-major order, element (row, col) of a height × width grid lives at linear offset:

row-major flattening width = elements per row
idx = row * width + col      // width is the *stride* between rows

Once you internalise the two strides, the neighbour offsets fall out for free:

selfidx left / rightidx − 1  /  idx + 1 up / downidx − width  /  idx + width diagonalsidx − width ± 1  /  idx + width ± 1

Thread & block indices → data indices

A kernel launch is a grid of blocks; each block is a grid of threads. The job of the first lines of every stencil kernel is to convert the thread's coordinates into a data coordinate:

the universal opening lines
// 1D
int i   = blockIdx.x * blockDim.x + threadIdx.x;   // global element

// 2D — note: x ↔ column, y ↔ row
int col = blockIdx.x * blockDim.x + threadIdx.x;   // horizontal
int row = blockIdx.y * blockDim.y + threadIdx.y;   // vertical
int idx = row * width + col;                       // row-major linearisation
Watch the axis swap

threadIdx.x is the column (fastest-varying, contiguous in memory), and threadIdx.y is the row. Getting this backwards is the single most common stencil bug — it produces a transposed result and destroys memory coalescing. x goes with width.

03

The 1D naïve kernel & the reuse problem

Each thread computes one output and reads its own neighbourhood straight from global memory. Correct — but wasteful.

stencil_1d_naive.cu
__global__ void stencil1d_naive(const float* in, float* out, int n){
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i >= R && i < n - R){               // skip the edges (no neighbours)
        float acc = 0.f;
        for (int r = -R; r <= R; ++r)
            acc += coef[r + R] * in[i + r]; // each in[] read from DRAM
        out[i] = acc;
    }
}
04

1D tiling: the tile, the halo, the ghost

A block cooperatively loads its slice once into shared memory — plus a thin halo of R cells on each side so edge threads still find their neighbours — then everyone computes from that fast on-chip copy.

Shared memory needed per block: BLOCK + 2R elements. Two indices now live side by side:

gindexblockIdx.x·blockDim.x + threadIdx.x   — position in the global array lindexthreadIdx.x + R   — position in the shared tile (shifted right by R to leave room for the left halo)
stencil_1d_tiled.cu load → __syncthreads → compute
#define R     1
#define BLOCK 256            // must equal blockDim.x
__constant__ float coef[2*R + 1];

__global__ void stencil1d_tiled(const float* in, float* out, int n){
    __shared__ float tile[BLOCK + 2*R];

    int gindex = blockIdx.x * blockDim.x + threadIdx.x; // global element
    int lindex = threadIdx.x + R;                     // slot in tile

    // (1) every thread loads its own element
    tile[lindex] = (gindex < n) ? in[gindex] : 0.f;

    // (2) the first R threads also load both halos
    if (threadIdx.x < R){
        int L = gindex - R;       // left  source
        int Rt = gindex + BLOCK;   // right source
        tile[lindex - R]     = (L  >= 0) ? in[L]  : 0.f; // left  ghost if OOB
        tile[lindex + BLOCK] = (Rt <  n) ? in[Rt] : 0.f; // right ghost if OOB
    }

    __syncthreads();   // tile is now fully populated & visible to all threads

    // (3) compute purely from shared memory — no global reads here
    if (gindex < n){
        float acc = 0.f;
        for (int r = -R; r <= R; ++r)
            acc += coef[r + R] * tile[lindex + r];
        out[gindex] = acc;
    }
}
Why __syncthreads is non-negotiable

Thread j reads tile[lindex-1], which thread j−1 wrote. Without the barrier, j might read that slot before j−1 has filled it — a classic read-before-write race. The barrier guarantees the entire tile (interior + halo) is written before anyone starts computing.

05

Boundary checks: halo vs ghost (they're different)

These two words get used interchangeably, but keeping them apart makes boundary logic obvious.

// the two kinds of "extra" cells around a tile
TermWhat it isWhere the value comes from
HaloValid neighbour data that lives inside the global domain but is owned by an adjacent block.A real in[] load — just from a position outside this block's main slice.
GhostPositions outside the global domain entirely (index < 0 or ≥ n).A boundary condition: usually 0, or clamp / reflect / wrap.
Rule of thumb

Halo is about blocks not seeing each other's data — solved by loading a few extra real cells. Ghost is about the domain ending — solved by a boundary condition. A tile at the array's interior has halo but no ghost; a tile at the corner of the domain has both.

06

Stepping up: 2D indexing & the 5-point stencil

The 2D cross stencil reads a center plus its four orthogonal neighbours. In row-major memory those neighbours are ±1 and ±width away.

stencil_2d_naive.cu
__global__ void stencil2d_naive(const float* in, float* out,
                                int width, int height){
    int col = blockIdx.x * blockDim.x + threadIdx.x;
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    if (row > 0 && row < height-1 && col > 0 && col < width-1){
        int o = row * width + col;
        out[o] = cC*in[o]
               + cN*in[o - width] + cS*in[o + width]   // up / down
               + cW*in[o - 1]     + cE*in[o + 1];      // left / right
    }
}

Here each interior element is read by up to 5 different output computations — even worse pressure than 1D.

07

2D tiling: a tile wrapped in a halo border

A TY × TX block of threads loads a (TY+2R) × (TX+2R) tile: the interior, plus a one-cell-thick border on all sides.

stencil_2d_tiled.cu 5-point · corners skipped
#define R  1
#define TX 16
#define TY 16

__global__ void stencil2d_tiled(const float* in, float* out,
                                int width, int height){
    __shared__ float tile[TY + 2*R][TX + 2*R];

    int col = blockIdx.x * blockDim.x + threadIdx.x;  // global x
    int row = blockIdx.y * blockDim.y + threadIdx.y;  // global y
    int lx  = threadIdx.x + R;                        // local x
    int ly  = threadIdx.y + R;                        // local y

    // (1) interior
    if (row < height && col < width)
        tile[ly][lx] = in[row * width + col];

    // (2) left/right column halos
    if (threadIdx.x < R){
        int cl = col - R, cr = col + TX;
        tile[ly][lx - R]  = (cl >= 0)    ? in[row*width + cl] : 0.f;
        tile[ly][lx + TX] = (cr < width) ? in[row*width + cr] : 0.f;
    }
    // (3) top/bottom row halos
    if (threadIdx.y < R){
        int rt = row - R, rb = row + TY;
        tile[ly - R][lx]  = (rt >= 0)     ? in[rt*width + col] : 0.f;
        tile[ly + TY][lx] = (rb < height) ? in[rb*width + col] : 0.f;
    }

    __syncthreads();

    // (4) compute from shared memory
    if (row > 0 && row < height-1 && col > 0 && col < width-1){
        out[row*width + col] =
              cC*tile[ly][lx]
            + cN*tile[ly-1][lx] + cS*tile[ly+1][lx]
            + cW*tile[ly][lx-1] + cE*tile[ly][lx+1];
    }
}
Two things to keep in mind

Bank conflicts: padding the inner tile dimension to TX + 2R + 1 (an odd stride) often removes shared-memory bank conflicts — the same trick you used to speed up the transpose kernel. Corners for 9-point: a diagonal stencil also needs the four corner cells, loaded by the single thread where tx < R && ty < R.

08

How much faster — and the honest caveat

Stencils are almost always memory-bound, so the figure that matters is global-memory traffic. Tiling collapses the redundant reads toward the theoretical floor of "read each element once."

Global reads per output element:

  • Naïve: exactly the tap count — 2R+1 (1D) or 4R+1 (2D cross).
  • Tiled: tile size ÷ outputs — (B+2R)/B (1D) or (T+2R)²/T² (2D), which → 1 as the tile grows.
// theoretical global-read reduction from tiling
Stenciltile / blocknaïve reads/elemtiled reads/elemreduction
1D 3-pointB = 2563258/256 ≈ 1.0082.98×
2D 5-pointT = 165324/256 ≈ 1.273.95×
2D 5-pointT = 3251156/1024 ≈ 1.134.43×
2D 9-pointT = 169324/256 ≈ 1.277.11×
The honest caveat — don't expect the full factor in wall-clock time

Those numbers are reductions in global-memory traffic, not guaranteed speedups. On modern GPUs the L1/L2 caches already capture much of the naïve kernel's reuse, so a hand-written shared-memory tile sometimes wins only modestly — often ~1.5–3× for simple low-radius stencils, and occasionally close to nothing when the caches were already doing the job. Shared-memory tiling pays off most clearly when: the radius is large (more reuse to harvest), the tile is sized for high occupancy, you've eliminated bank conflicts, and the working set would otherwise thrash the cache. Always measure (e.g. Nsight Compute's memory-throughput and L2-hit-rate counters) rather than trusting the arithmetic alone.

Other real-world costs the formula ignores: the __syncthreads() barrier, the shared-memory capacity limit (it caps occupancy), bank conflicts on the inner tile dimension, and the extra index arithmetic in the load phase. None of these are large for well-tuned kernels, but they're why the measured curve sits below the theoretical one.

The four-beat pattern, every time

  • 1 · Map — turn (blockIdx, threadIdx) into gindex and lindex.
  • 2 · Load — every thread copies its interior cell; edge threads also copy halo cells, substituting the ghost value when the source is out of bounds. All bounds checks live here.
  • 3 · Sync — __syncthreads() so the whole tile is visible before anyone reads a neighbour.
  • 4 · Compute — the weighted sum, reading only from shared memory, writing one output. Branch-free inner loop.
The one-sentence version

Tiling trades a little redundant shared-memory storage (the halo) for a large reduction in redundant global-memory traffic — and the bigger the stencil and the tile, the better that trade gets.
