Fusing evolve_implicit
into a per-cell kernel

How one batched matrix multiply that PyTorch quietly split into 19 GPU launches per timestep became a single fused kernel — and the Torch/AVX ABI rabbit hole that nearly broke the build. A worked case study, a transferable recipe, and a packaging field guide.

model: Jupiter CRM 100³ device: RTX 5090 per-cycle GPU 250.9 → 192.0 ms kinetics solve ~4.2× numerics ~1e-11 rel

§0TL;DR & results

The Jupiter cloud-resolving simulation (run_jupiter_crm.py -c jupiter_crm_100.yaml) spent ~24% of every timestep inside one helper, kintera::evolve_implicit, that advances cloud chemistry implicitly. Profiling showed the cost was not really "matrix math" — it was the shape of the math: a multiply of ~1.2 million tiny 8×8-ish matrices, expressed as a PyTorch batched matmul + linalg_solve, which cuBLAS executes as dozens of poorly-utilized launches.

Replacing it with a single hand-written kernel — one GPU thread per grid cell, each building and solving its own little linear system in scratch memory — removed the GEMM and the batched LU entirely.

The one-sentence lesson

When you find yourself doing many independent, tiny linear-algebra problems, a general BLAS/LAPACK library is the wrong tool: give each problem one thread and solve it in registers/scratch. The speed-up comes from parallel granularity, not from cleverer arithmetic.

§1The science: implicit cloud chemistry

The Jupiter CRM resolves convecting weather on a 100×100×100 grid. At each cell it tracks a handful of chemical species — the dry H₂/He background plus volatiles and their condensates: H₂O, NH₃, H₂S and the cloud phases H₂O(l), H₂O(l,p), NH₃(s), NH₃(s,p), NH₄SH(s). They interconvert through reactions: nucleation, growth/precipitation, and evaporation (one of them, NH₃ + H₂S ⇌ NH₄SH(s), is reversible).

1.1 Why implicit, and the linear system it produces

Let c be the vector of species concentrations and r(c) the vector of reaction rates. The stoichiometric matrix S turns reaction rates into species production, so the chemistry ODE is

      dc/dt = S · r(c)            S : (nspecies × nreaction),  r : (nreaction)

Cloud chemistry is stiff: nucleation and evaporation have timescales far shorter than the fluid timestep dt. An explicit update (c += dt·S·r) would demand absurdly small steps to stay stable. The cure is an implicit (backward) Euler step, which evaluates the rate at the new time:

      cⁿ⁺¹ = cⁿ + dt · S · r(cⁿ⁺¹)

That is nonlinear in the unknown. Linearizing r about cⁿ with the reaction-space Jacobian J = ∂r/∂c and writing the update as Δc = cⁿ⁺¹ − cⁿ gives a linear system for each cell:

      (I − dt · S·J) · Δc = dt · S · r        ⟺        ( I/dt − S·J ) · Δc = S · r
Science note — what each operator is

S·J maps a concentration perturbation through the reaction network back into species space — it is the chemical "feedback" matrix (nspecies × nspecies). I/dt is the inertia of the timestep: large 1/dt makes the system diagonally dominant and well-conditioned, which (foreshadowing) is exactly why a simple solver is safe here. S·r is the explicit tendency we are implicitly correcting.

This is precisely what evolve_implicit(rate, stoich, jacobian, dt) computes. The original PyTorch implementation reads almost like the math:

kintera/src/kinetics/evolve_implicit.hpp (original)C++ / libtorch
auto SJ = stoich.matmul(jacobian);                // (…, nsp, nsp)   batched over cells
auto SR = stoich.matmul(rate.unsqueeze(-1)).squeeze(-1);
auto A  = eye / dt - SJ;
try {
  return torch::linalg_solve(A, SR.unsqueeze(-1)).squeeze(-1);   // batched LU
} catch (const c10::Error&) {
  return std::get<0>(torch::linalg_lstsq(A, SR.unsqueeze(-1))).squeeze(-1); // SVD fallback
}

Beautiful, terse, correct — and, at production scale, a performance trap. The whole report is about why, and what to do instead.

§2Finding the bottleneck (profiling honestly)

Optimization without measurement is folklore. The workflow used two NVIDIA tools:

  • Nsight Systems (nsys) — a timeline: which kernels ran, when, how long, how they interleave with CPU/launches. Best for "where does wall-clock go?"
  • Nsight Compute (ncu) — a microscope on a single kernel: occupancy, memory vs compute throughput, registers, stalls. Best for "why is this kernel slow?"

2.1 The mystery of the 19-launch matrix multiply

576 ms over 190 launches — but the code calls matmul once per timestep. Where do 190 launches come from? Looking at one timestep window in the timeline, there are exactly 18 launches of a 65,535-wide kernel + 1 launch of an 11,386-wide kernel, back-to-back, immediately followed by the LU solve. The arithmetic is the giveaway:

      18 × 65535  +  1 × 11386  =  1,191,016  =  106³  =  ncell
What's going on

A CUDA grid's z dimension is capped at 65,535 blocks. cuBLAS maps the batch dimension of a batched GEMM onto gridDim.z. With 1.19M matrices in the batch, it cannot launch them in one grid, so it chops the batch into ~19 sequential kernel launches. So the "190 GEMM launches over 10 cycles" is really one logical matmul per cycle, shredded by a hardware limit. That single matmul is SJ = stoich.matmul(jacobian).

2.2 The honest part: separating init from steady state

The biggest single kernel by name was actually the batched_svd at 388 ms — and it would have been tempting to "optimize" it. But a timeline check showed every SVD launch occurs before the very first GEMM, i.e. before timestep 1. It belongs to the one-time pseudo-adiabat initialization (linalg_pinv while building the starting atmosphere), not the per-cycle loop. Spending effort there would speed up startup once and help steady-state throughput not at all.

Pitfall avoided

"Biggest bar in the profile" ≠ "biggest recurring cost." Always attribute a kernel to a phase (init / per-step / teardown). A 388 ms one-time cost is noise across a multi-thousand-step run; a 70 ms/step cost is the whole ballgame. The GEMM + LU were the real target precisely because they repeat every cycle.

§3Why tiny batched GEMM/LU is slow

The system is small (≈1.4 GFLOP of actual arithmetic per timestep) yet took ~70 ms — under 0.02 TFLOP/s on a card rated for tens of TFLOP/s FP64. The inefficiency is structural, not arithmetic:

Arithmetic intensity is tiny

Each 8×8×10 matrix does ~640 multiply-adds but reads/writes ~150 doubles. The op is memory-bound: you move a lot of bytes to do almost no math. BLAS GEMM kernels are tuned for the opposite regime (large M,N,K, data reused many times).

Occupancy & tiling collapse

A GEMM kernel tiles M×N and streams K. With M=N≈8, K≈10 there is nothing to tile — most of a warp's lanes and most of the shared-memory staging sit idle. Batched GEMM amortizes this poorly when each matrix is sub-tile sized.

Launch & sync overhead

Splitting into 19 launches means 19 grid launches, 19 tails, and a serialized dependency into the LU. The fixed per-launch overhead becomes a real fraction of a sub-millisecond kernel.

LAPACK batched LU is general

getrf/trsm carry pivoting bookkeeping, info arrays, and host round-trips designed for arbitrary sizes. For an 8×8 that never goes singular, that machinery is pure overhead.

Reframing

The problem is not "make the GEMM faster." It is "stop expressing 1.19M independent 8×8 solves as library calls." The data is embarrassingly parallel across cells; the right unit of work is one cell per thread, where an 8×8 build-and-solve lives entirely in fast per-thread memory and every one of the GPU's thousands of lanes is doing useful work.

§4The idea: kernel fusion + the map pattern

Two classic ideas combine here.

Kernel fusion

The original path materializes large intermediate tensors and crosses the GPU multiple times: SJ (1.19M×8×8 doubles ≈ 0.6 GB written then re-read), then A, then the LU's internal buffers. Fusion means computing A = I/dt − S·J, the right-hand side S·r, and the solve without ever writing the intermediates to global memory — they live and die inside one thread's scratch.

The map pattern (data parallelism by cell)

Each cell's solve is independent of every other cell's. That is a textbook map: apply the same function to every element. The natural GPU expression is one thread (or small group) per element. With 1.19M cells we launch 1.19M threads — saturating all 170 SMs of an RTX 5090 with light, uniform work.

Provenance

This is the same move that earlier sped up the fluid solver in this codebase (the "vic" implicit block-tridiagonal solve): replace batched library calls over many tiny systems with a custom per-cell kernel. The recipe is reusable — §9 distills it.

§5Implementation, in coding detail

5.1 Architecture: dispatch + device function

libtorch's idiom for "same op, different backends" is a DispatchStub: declare a typed function pointer, register a CPU and a CUDA implementation, and call through a thin stub that picks the right one at runtime by device. We mirror the existing thermo_dispatch in the codebase. Four pieces:

evolve_implicit_dispatch.hpp — the stub declarationC++
namespace at::native {
  using evolve_implicit_fn = void (*)(at::TensorIterator& iter,
                                      at::Tensor const& stoich, double dt);
  DECLARE_DISPATCH(evolve_implicit_fn, call_evolve_implicit);
}

The actual numerical work lives in a header-only device function, evolve_implicit_cell, marked DISPATCH_MACRO (which expands to __host__ __device__ under CUDA, nothing on CPU). Writing the math once, compilable for both backends, is how we keep a single source of truth and guarantee CPU≡GPU behavior.

5.2 Feeding it: a squashed TensorIterator

The three operands have different last dimensions — rate is (…, nreaction), jacobian is (…, nreaction, nspecies), output delta is (…, nspecies). We flatten the jacobian's trailing pair, then build a TensorIterator that iterates over cells and squashes the trailing dimension, so each thread receives a base pointer to its cell's contiguous block:

evolve_implicit_dispatch.cpp — the host entry (out-of-line; see §10)C++
torch::Tensor evolve_implicit(torch::Tensor rate, torch::Tensor stoich,
                              torch::Tensor jacobian, double dt) {
  auto nspecies = stoich.size(0);
  auto rate_c   = rate.contiguous();
  auto jac2d    = jacobian.contiguous().flatten(-2, -1);  // (…, nrxn*nsp)
  auto stoich_c = stoich.contiguous();

  auto out_sizes = rate_c.sizes().vec();
  out_sizes.back() = nspecies;
  auto delta = torch::empty(out_sizes, rate_c.options());

  auto iter = at::TensorIteratorConfig()
      .resize_outputs(false)
      .check_all_same_dtype(false)
      .declare_static_shape(delta.sizes(), /*squash_dims=*/{delta.dim() - 1})
      .add_output(delta)
      .add_input(rate_c)
      .add_input(jac2d)
      .build();

  at::native::call_evolve_implicit(rate_c.device().type(), iter, stoich_c, dt);
  return delta;
}

The CUDA launcher uses the project's gpu_mem_kernel<Threads, Arity> helper, which hands every thread a private slice of dynamic shared memory (work) for scratch:

evolve_implicit_dispatch.cu — the CUDA launcherCUDA C++
AT_DISPATCH_FLOATING_TYPES(iter.dtype(), "call_evolve_implicit_cuda", [&] {
  int nspecies  = ensure_nonempty_size(stoich, 0);
  int nreaction = ensure_nonempty_size(stoich, 1);
  auto stoich_ptr = stoich.data_ptr<scalar_t>();
  scalar_t inv_dt   = scalar_t(1.0 / dt);
  int mem = evolve_implicit_space<scalar_t>(nspecies, nreaction);

  native::gpu_mem_kernel<32, 3>(iter, mem,
    [=] GPU_LAMBDA(char* const data[3], unsigned strides[3], char* work) {
      auto delta = reinterpret_cast<scalar_t*>(data[0] + strides[0]);
      auto rate  = reinterpret_cast<scalar_t*>(data[1] + strides[1]);
      auto jac   = reinterpret_cast<scalar_t*>(data[2] + strides[2]);
      evolve_implicit_cell(delta, rate, jac, stoich_ptr,
                           nspecies, nreaction, inv_dt, work);
    });
});

5.3 The per-cell kernel

This is the heart. Per cell: build A and the right-hand side sr from the shared constant stoich, the cell's jac and rate; then solve. We reuse the codebase's tiny dense primitives mmdot (matrix·matrix) and mvdot (matrix·vector).

evolve_implicit_impl.h — evolve_implicit_cellC++ (host+device)
template <typename T>
DISPATCH_MACRO void evolve_implicit_cell(T* delta, const T* rate, const T* jac,
                                         const T* stoich, int nspecies,
                                         int nreaction, T inv_dt, char* work) {
  char* cur = work;
  T* A   = alloc_from<T>(cur, nspecies * nspecies);  // kept for the singular fallback
  T* Alu = alloc_from<T>(cur, nspecies * nspecies);  // destroyed by the LU solve
  T* sr  = alloc_from<T>(cur, nspecies);             // rhs, kept
  T* x   = alloc_from<T>(cur, nspecies);             // rhs → solution
  char* pw = cur;                                   // remaining = psolve workspace

  // A = S·J  (then turned into I/dt − S·J below);   sr = S·rate
  mmdot(A, stoich, jac, nspecies, nreaction, nspecies);
  mvdot(sr, stoich, rate, nspecies, nreaction);
  for (int i = 0; i < nspecies; ++i)
    for (int k = 0; k < nspecies; ++k)
      A[i*nspecies + k] = (i == k ? inv_dt : T(0)) - A[i*nspecies + k];

  // primary path: dense LU with partial pivoting on a destroyable copy
  for (int t = 0; t < nspecies*nspecies; ++t) Alu[t] = A[t];
  for (int t = 0; t < nspecies; ++t)          x[t]  = sr[t];

  if (!dsolve_lu(Alu, x, nspecies)) {
    // rare singular cell: minimum-norm least-squares via pseudo-inverse,
    // matching the original linalg_lstsq fallback — but per cell, no full-batch SVD.
    for (int t = 0; t < nspecies; ++t) x[t] = sr[t];
    psolve(x, A, nspecies, pw);
  }
  for (int t = 0; t < nspecies; ++t) delta[t] = x[t];
}
Reading the indexing

Everything is row-major and flat. stoich[i*nreaction + r] is S(species i, reaction r); jac[r*nspecies + k] is J(reaction r, species k). The product A(i,k)=Σ_r S(i,r)·J(r,k) is exactly mmdot. Subtracting from I/dt is a single diagonal touch. This is the entire "GEMM" — done by one thread, in scratch, for its one cell.

5.4 The solver: dsolve_lu

For an 8×8 dense system that the I/dt term keeps well-conditioned, the right solver is the simplest stable one: Gaussian elimination with partial pivoting — the same algorithm cuBLAS getrf/getrs use, so results match to rounding.

math/core.h — dsolve_lu (new)C++ (host+device)
template <typename T>
DISPATCH_MACRO bool dsolve_lu(T* A, T* b, int n) {        // A,b destroyed; b ← solution
  for (int k = 0; k < n; ++k) {
    int piv = k; T maxv = fabs(A[k*n + k]);          // partial pivot: largest |col|
    for (int i = k+1; i < n; ++i) {
      T v = fabs(A[i*n + k]); if (v > maxv) { maxv = v; piv = i; }
    }
    if (maxv == 0.0) return false;                 // exactly singular → caller falls back
    if (piv != k) { /* swap rows piv,k in A and b */ }
    T akk = A[k*n + k];
    for (int i = k+1; i < n; ++i) {              // eliminate below pivot
      T f = A[i*n + k] / akk;
      for (int j = k+1; j < n; ++j) A[i*n + j] -= f * A[k*n + j];
      b[i] -= f * b[k];
    }
  }
  for (int i = n-1; i >= 0; --i) {                 // back substitution
    T s = b[i];
    for (int j = i+1; j < n; ++j) s -= A[i*n + j] * b[j];
    b[i] = s / A[i*n + i];
  }
  return true;
}
Why an exact-zero singularity test

cuBLAS getrf reports failure only on an exact zero pivot; near-singular matrices still "succeed" (with large round-off). To match the original behavior bit-for-bit on the common path, we use the same threshold. In the real run the fallback never fires (the I/dt term keeps every cell non-singular), so it is purely a safety net.

5.5 Scratch memory: the layout and the sizing function

Each thread needs space for A, the destroyable copy Alu, the kept rhs sr, the working solution x, and — only if a cell is singular — the pseudo-inverse workspace. A small sizing function carves it deterministically (matching the existing equilibrate_*_space convention):

evolve_implicit_impl.h — evolve_implicit_spaceC++
template <typename T>
size_t evolve_implicit_space(int nspecies, int nreaction) {
  size_t bytes = 0;
  auto bump = [&](size_t align, size_t nb){ bytes = align_up(bytes, align) + nb; };
  bump(alignof(T), nspecies*nspecies*sizeof(T));  // A
  bump(alignof(T), nspecies*nspecies*sizeof(T));  // Alu
  bump(alignof(T), nspecies*sizeof(T));            // sr
  bump(alignof(T), nspecies*sizeof(T));            // x
  return bytes + psolve_space<T>(nspecies);         // pseudo-inverse fallback
}

A CPU registration mirrors the launcher with iter.for_each and a stack buffer, calling the same evolve_implicit_cell — so CPU and CUDA are guaranteed to agree, and the CPU path doubles as the validation oracle.

§6The decision tree

Several forks shaped the design. Each is a transferable judgment call.

Cautionary tale — PCR backfired

An earlier sibling effort (the fluid solver) tried Parallel Cyclic Reduction to parallelize inside each system. On a small grid it was 2.7× faster; at production scale it was 2.35× slower. Reason: when you already have millions of independent systems, the GPU is saturated by the simple "one-thread-each" map. PCR does ~7× more total work (O(N log N) vs O(N)) to win latency you no longer need. Match the parallelism you add to the parallelism you lack.

Bit-close, not bit-exact

There is no way to reorder a matrix solve and keep every bit (floating-point is non-associative). We targeted bit-close (~1e-11 relative) and validated rigorously, rather than chasing an impossible bit-exact goal. Knowing which guarantee is achievable — and acceptable — is part of the design.

Other forks

  • psolve (pinv) vs. Tikhonov for the singular fallback. psolve reproduces the original lstsq minimum-norm semantics exactly; Tikhonov would be cheaper on scratch. Since the fallback never fires in this run, correctness-fidelity won — but it costs occupancy (§9.4).
  • Unify CPU+CUDA on one cell function vs. keep torch on CPU. Unifying gives a single source of truth and a free oracle; the small CPU numeric shift is covered by the test suite. Chosen.
  • Out-of-line vs. inline definition. Forced by the ABI saga in §10 — a decision that looks like style but is really about which binary a symbol lands in.

§7Validation: a correctness ladder

Performance work earns trust only with proof it didn't change the answer. Five rungs, cheap to expensive:

  1. Unit tests — test_kinetics (the evolve_implicit case): pass on cpu/cuda × float/double.
  2. CPU ≡ CUDA — same inputs through both registrations agree to ~1e-17 (machine epsilon): proves determinism and a race-free kernel.
  3. Fused ≡ original formula — compare against a literal linalg_solve(I/dt−S·J, S·r) on random prod-shaped inputs: ~1e-11 relative (GE vs cuBLAS-LU rounding). This is the key numeric check.
  4. Stiff chemistry end-to-end — the Chapman-cycle time-marching test (24 cases incl. reversible reactions, which stress conditioning) converges to identical final states.
  5. Full simulation — a capped Jupiter CRM run completes with no NaNs, mass conserved, energy rising from the prescribed heating; deterministic across repeats.
A subtlety worth stating

The driver adds an unseeded random perturbation to the initial wind field, so absolute mass/energy cannot bit-match a previous run regardless of code changes. That is why rung 3 (a controlled comparison against the original formula on identical inputs) is the load-bearing numeric test — not "diff the run logs."

§8Results, examined

Both profiles span init + 10 cycles. After the change, the chemistry GEMM and batched LU are absent from the timeline, replaced by one fused kernel:

Why the clean-room microbenchmark "only" shows 1.3×

Benchmarking the operation on a tidy contiguous tensor gives old=23 ms, new=17.6 ms (1.3×). The real run shows ~4× because the production tensor layout triggers cuBLAS's pathological 19-launch split, which the clean tensor avoids. Lesson: microbenchmarks can flatter the baseline; trust the in-situ profile for the real verdict.

Headroom: the kernel is occupancy-limited

17 ms is far above the ~0.6 ms memory-bound floor for this data volume. The cause is the scratch: ~2.4 KB/thread × 32 threads/block ≈ 78 KB shared/block caps occupancy near 5%. The biggest lever left is shrinking per-thread scratch — chiefly the rarely-used pseudo-inverse workspace — to let far more warps hide latency. A clear, deliberate follow-up (see §9.4).

§9Generalizing the technique

This pattern — call it "batched-tiny → fused per-element kernel" — recurs across scientific GPU codes far beyond chemistry. Here is the reusable form.

The recipe

  1. Recognize the shape. Profile shows time in batched GEMM / batched LU / batched SVD with a huge batch and tiny per-element matrices. A tell-tale sign: a library op fragmenting into many launches (the grid-Z cap), or low occupancy / low arithmetic intensity in ncu.
  2. Confirm independence. Each element's computation must not depend on its neighbors (a map). If there's coupling, this still works per group but needs care.
  3. Write one device function that does the whole per-element computation in scratch — assemble, transform, solve — with no global intermediates. Make it __host__ __device__ so a CPU registration shares the code.
  4. Launch one thread (or small cooperative group) per element.
  5. Pick the in-thread algorithm by conditioning & size, not by habit (GE for small/dominant; a fallback for the rare degenerate case).
  6. Validate on the correctness ladder (§7); accept bit-close.

9.1 Assumptions that must hold

  • Many elements — enough to fill the GPU (tens of thousands+). With few elements, see §9.2.
  • Small per-element state — it must fit in registers/scratch (roughly ≤ a few dozen × a few dozen). Beyond that, scratch pressure destroys occupancy and a library may win again.
  • A shared, constant operand is a bonus — here stoich is the same for every cell, so it streams from cache, not per-cell memory.
  • Uniform work — if some elements need wildly more work (data-dependent iteration counts), warp divergence erodes the gain.

9.2 When it wins, when it loses

The trap to internalize

"More parallelism is always better" is false. PCR added intra-system parallelism the workload didn't need and lost 2.35×. The question is never "can I parallelize this further?" but "is the GPU already full?" If yes, reduce work, don't add it.

9.3 A back-of-envelope performance model

Predict before you optimize:

  • Memory floor = bytes touched ÷ bandwidth. Here ≈ (read jac 762 MB + rate 95 MB + write Δ 76 MB) ÷ ~1.5 TB/s ≈ 0.6 ms. That is the best achievable; 17 ms means we're ~28× off the floor and the lever is occupancy, not arithmetic.
  • Compute floor = FLOPs ÷ FP64 rate. ~1.4 GFLOP ÷ tens of TFLOP/s ≈ tens of µs — negligible. So this kernel is memory/occupancy bound, never compute bound. Optimize accordingly.
  • Occupancy gate = limited by registers and shared memory per thread. If a scratch-heavy kernel is far from its memory floor, shrink scratch first.

9.4 Limitations & honest caveats

  • Not bit-exact — acceptable here, possibly not for a regression suite that hashes outputs.
  • Occupancy-limited as written — the pinv fallback workspace inflates per-thread scratch to ~78 KB/block. Dropping it from the hot path (Tikhonov fallback, or bounded local arrays + more threads/block) is the next several-fold win, at the cost of changing rare-singular semantics.
  • Hard-coded smallness — the approach silently assumes the per-element matrix stays tiny. If a future config adds many species, revisit the scratch budget and the GE choice.
  • Maintenance cost — you now own a numerical kernel that a library used to own. Worth it only when the speed-up is real and the math is stable (it is here).

§10The build caveat: a Torch / AVX / ABI field guide

The optimization was the easy half. Getting the rebuilt extension to import exposed a deep, instructive trap in mixed C++/CUDA/PyTorch packaging. This chapter is a standalone tutorial: even if you never touch chemistry, you will meet this.

The symptom 
ImportError: …/kintera.cpython-311-x86_64-linux-gnu.so: undefined symbol:
  _ZN2at6native16DispatchStubImpl12get_call_ptrEN3c1010DeviceTypeEPv

The module compiled and linked fine, then failed at import on a missing PyTorch symbol — one that demonstrably exists in the installed libtorch_cpu.so.

10.1 First principles: what an ABI is, and name mangling

The API is the source-level contract (function names, signatures). The ABI (Application Binary Interface) is the binary contract: how names are encoded into symbols, how arguments are passed, how structs are laid out. Two object files link only if their ABIs agree.

C++ encodes types into symbol names — name mangling — so overloads stay distinct. Decode with c++filt:

_ZN2at6native16DispatchStubImpl12get_call_ptrEN3c1010DeviceTypeEPv
   └─ at::native::DispatchStubImpl::get_call_ptr(c10::DeviceType, void*)        // 2 args

_ZN2at6native16DispatchStubImpl12get_call_ptrEN3c1010DeviceTypeEPvS4_S4_
   └─ at::native::DispatchStubImpl::get_call_ptr(c10::DeviceType, void*, void*, void*)  // 4 args
The whole bug in one observation

The same function name get_call_ptr exists with two different arities. The module asked the linker for the 2-argument version; libtorch_cpu.so only provides the 4-argument version. Different mangled names ⇒ "undefined symbol." The question becomes: why did two parts of the same build disagree on how many arguments this function takes?

10.2 The mechanism: ATen's DispatchStub

PyTorch routes a backend-agnostic op to a CPU or CUDA kernel via a DispatchStub. Calling the stub invokes a templated, header-inlined operator() that calls into DispatchStubImpl::get_call_ptr to fetch the right function pointer. Our new code uses this mechanism (the call_evolve_implicit stub from §5.1). So any translation unit that calls our stub emits a reference to get_call_ptr.

10.3 Torch & AVX: runtime CPU-feature dispatch

PyTorch ships one binary that runs on many CPUs, so it compiles hot CPU kernels several times — a baseline build, an AVX2 build, an AVX-512 build — and picks the best at runtime. That choice is the extra arguments to get_call_ptr: one function pointer per compiled capability. The arity is gated by preprocessor macros in the very header you compile against:

ATen/native/DispatchStub.h (paraphrased)C++ header
  void* get_call_ptr(
      c10::DeviceType device_type
      , void* DEFAULT
  #ifdef HAVE_AVX512_CPU_DEFINITION
      , void* AVX512
  #endif
  #ifdef HAVE_AVX2_CPU_DEFINITION
      , void* AVX2
  #endif
      /* … VSX, ZVECTOR, SVE256 … */ );
So the arity is configuration, not code

With both AVX macros defined ⇒ 4 args (DEFAULT + AVX512 + AVX2). With neither ⇒ 2 args. The installed libtorch_cpu.so (torch 2.10+cu128) was built with them ⇒ it exports the 4-arg symbol. Any consumer that includes this header must define the same macros or it will reference a phantom 2-arg symbol.

10.4 How the symbol leaked into the Python module

This package builds in two stages with two systems — and they disagreed:

  • The C++ libraries are built by CMake. CMake's find_package(Torch) (via Caffe2) propagates -DHAVE_AVX512_CPU_DEFINITION -DHAVE_AVX2_CPU_DEFINITION automatically ⇒ the libs correctly reference the 4-arg symbol.
  • The pybind module (python/csrc/*.cpp) is built by setuptools via torch.utils.cpp_extension, which did not pass those macros ⇒ it referenced the 2-arg symbol.

But why did the module reference get_call_ptr at all? Because evolve_implicit was an inline function in a header, and the binding takes its address:

m.def("evolve_implicit", &kintera::evolve_implicit);   // forces the inline body into kintera.o

Taking the address of an inline function forces the compiler to emit its body in that translation unit. That body calls the dispatch stub → emits a get_call_ptr reference → into the module, compiled without the AVX macros → 2-arg → unresolved. A symbol probe confirmed it precisely:

$ for o in build/.../python/csrc/*.o; do echo "$o: $(nm $o | grep -c get_call_ptr)"; done
  kintera.o:        1      ← the only object referencing it — and only via the inline body
  pykinetics.o:     0
  pythermo.o:       0   …
Root cause, stated plainly

The caveat was introduced by this change: moving evolve_implicit onto the DispatchStub mechanism, while it lived inline in a header consumed by the module, dragged a macro-gated Torch symbol into a binary built without the matching macros.

10.5 The debugging trail (and the dead ends)

Worth walking through, because each dead end teaches a packaging fact:

  1. Read the symbol, don't guess. nm -D --defined-only libtorch_cpu.so | c++filt | grep get_call_ptr showed torch defines only the 4-arg; nm -D module.so showed it wants the 2-arg. The mismatch is now concrete, not mysterious.
  2. Dead end — "stale object files." Hypothesis: old .o from a previous torch. Forced a clean recompile of all csrc → still 2-arg. So it's not staleness; it's the compile flags.
  3. Found the gate. Inspecting DispatchStub.h revealed the #ifdef HAVE_AVX… arity switch. Now the fix space is clear: make the module's macros match torch's.
  4. Workaround — inject the macros. CFLAGS="-DHAVE_AVX512_CPU_DEFINITION -DHAVE_AVX2_CPU_DEFINITION". It works, but it's machine-specific (wrong on a non-AVX-512 host) and edits the build env.
  5. Dead end — distutils didn't rebuild. setup.py build_ext skipped recompiling because distutils tracks .cpp timestamps, not header dependencies. Editing only the header changed nothing. You must touch the .cpp or delete the output .so and .o to force it. (A genuinely surprising, very common footgun.)
  6. The clean fix. Rather than patch flags, remove the need for them — see §10.6.
Tooling reflexes worth keeping

nm -D file.so list dynamic symbols · nm … | c++filt demangle · U = undefined (needed) · T = defined (provided) · ldd shared-lib resolution · readelf -d rpath/needed. When a dynamic-link error appears, read the symbols; the answer is almost always literally printed.

10.6 The fix: move the definition out-of-line

The leak happened because the dispatch-calling code was compiled into the module. So compile it into the library instead. We changed evolve_implicit from an inline header function to a declaration + an out-of-line definition in evolve_implicit_dispatch.cpp (which the CMake build compiles with the AVX macros). Now the module only references the high-level evolve_implicit symbol — exported by the library — and the macro-gated get_call_ptr reference never enters the module:

evolve_implicit.hpp — header now declares onlyC++
// Defined out-of-line in evolve_implicit_dispatch.cpp (NOT inline): it calls a
// DECLARE_DISPATCH stub whose get_call_ptr arity is gated by torch's AVX macros.
// Keeping the definition in the library (built WITH those macros via Caffe2)
// keeps that reference out of downstream consumers built without them.
torch::Tensor evolve_implicit(torch::Tensor rate, torch::Tensor stoich,
                              torch::Tensor jacobian, double dt);

Verification — the module's reference count went to zero, and it imports built with the unmodified setup.py (no macro injection):

$ nm -D module.so | grep -c get_call_ptr
  0      ← nothing to resolve ⇒ AVX macro arity is now irrelevant to the module
The durable principle

Macro-gated or backend-specific symbols belong in the binary that's compiled with the matching flags. Keep them out of headers that downstream, differently-configured consumers compile. Inline + address-of is the classic way to accidentally smear such a symbol across binaries.

10.7 A general guide: building packages on top of Torch

  • Prefer one build system. Mixing CMake (libs) and setuptools (extension) means two places to keep torch flags consistent. If you must mix, deliberately propagate the same defines, std version, and ABI flags to both (e.g. _GLIBCXX_USE_CXX11_ABI, the AVX macros, -std=c++17).
  • Let the library own torch-coupled code. Header-only code that touches ATen internals (dispatch, TensorIterator guts) is fragile across consumers. Expose a narrow, stable, out-of-line API; keep torch internals behind the library boundary.
  • Match the ABI of the torch you link. The pre-CXX11 vs CXX11 string ABI and the CPU-capability macros must match the installed wheel. Read them from torch.__config__.show() / torch.utils.cpp_extension rather than hard-coding.
  • Know your build's dependency tracking. distutils/setuptools track source timestamps, not headers. CMake with CONFIGURE_DEPENDS tracks files but globs need reconfigure. When a header change "does nothing," force the rebuild.
  • Mind rpath and editable installs. A module's $ORIGIN/lib rpath must point at the actual libs; a failed pip install can leave you in a half-uninstalled or editable-redirected state where import resolves somewhere surprising. Verify with python -c "import pkg, os; print(os.path.dirname(pkg.__file__))".
  • Reach for nm/c++filt/ldd early. "undefined symbol" at import is a dynamic-link fact, not a Python bug — the binaries will tell you exactly what disagrees.

§11Related topics worth your time

Occupancy, registers & shared memory

The ceiling on resident warps per SM, set by registers/thread and shared-mem/block. Latency-bound kernels live or die by it. Tools: ncu's occupancy section, the CUDA occupancy calculator. Our §8 headroom is entirely an occupancy story.

Opt-in dynamic shared memory

Past ~48 KB/block you must opt in via cudaFuncSetAttribute(MaxDynamicSharedMemorySize). The codebase's gpu_mem_kernel handles this; it's why 78 KB/block even runs on the 5090.

CUDA Graphs for launch-bound loops

When per-step time is dominated by many small launches + syncs (the broader Jupiter CRM picture), a CUDA Graph records the launch sequence once and replays it, cutting CPU-side launch overhead. Complementary to fusion.

Numerical stability: GE vs SVD

Partial-pivot GE is backward-stable for well-conditioned systems; SVD/pinv is the robust choice for rank-deficient ones (and gives minimum-norm solutions). Choosing per-conditioning — fast path + robust fallback — is a recurring pattern.

Batched solver libraries

cuSOLVER/cuBLAS batched APIs and MAGMA exist precisely for "many small matrices" — but they're tuned for medium sizes and add host bookkeeping. For the tiny-and-many extreme, a custom kernel still wins, as here.

Companion case: the implicit fluid solver

The same fission/fusion logic was applied to the block-tridiagonal "vic" solve in this codebase (assemble / solve / redistribute split into per-cell kernels) — including the PCR negative result that informs §9.2. Cross-reading the two makes the principles stick.
