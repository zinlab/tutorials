ARCH Chapter I

Anatomy · control plane & data plane.

The split Two planes. The control plane decides what should be running. The data plane runs it. Every K8s issue is an issue in one of the two.

The point You debug Kubernetes by knowing which component to interrogate. The architecture is the diagnostic map.

Kubernetes is an orchestration system with two clearly separated planes. The control plane is a quartet of processes that hold cluster state, decide what should be running, and reconcile desired against actual. The data plane is every other node — running an agent (kubelet), a container runtime, and a network proxy. The control plane never executes your workloads. The data plane does nothing except execute and report.

The two planes · what runs where

Control plane components

kube-apiserver. The only component that talks to etcd. Every other component talks to apiserver. Stateless; horizontally scalable. Handles auth (TLS client certs, OIDC, service accounts), authorization (RBAC), and admission (validating + mutating webhooks). The watch mechanism — long-lived HTTP streaming connections — is how controllers and kubelets get notified of changes.

etcd. The Raft-replicated key-value store. The single source of truth for cluster state. Typically 3 or 5 members for HA. Backups of etcd are backups of the cluster. Restore etcd, and the cluster reconciles itself back to whatever was there.

kube-scheduler. Watches for pods with no nodeName assigned. Runs each through a filter phase (which nodes are even eligible) then a score phase (which eligible node is best). Writes the chosen nodeName back through apiserver. Does nothing else.

kube-controller-manager. Bundles dozens of controllers — Deployment, ReplicaSet, Job, CronJob, Node, Endpoints, Namespace, ServiceAccount, and more. Each runs a reconcile loop watching its resource type. Add a Deployment with 3 replicas, the Deployment controller creates a ReplicaSet, the ReplicaSet controller creates 3 Pods.

Cloud-provider integrations split into a separate cloud-controller-manager on managed offerings (EKS, GKE, AKS). It owns load balancer creation, route reconciliation, and node lifecycle from the cloud's side.

Data plane components

kubelet. The node agent. Watches apiserver for pods assigned to its node, pulls images, asks the runtime to start containers, runs probes, reports status. Talks to the runtime via CRI (Container Runtime Interface), to the network via CNI (Container Network Interface), and to storage via CSI (Container Storage Interface).

container runtime. containerd is the default; CRI-O is the alternative. Docker is gone — dockershim was removed in 1.24 (May 2022). The runtime handles image pulls, container creation, cgroup setup, namespace isolation.

kube-proxy. Programs iptables, ipvs, or eBPF rules so that traffic to a Service ClusterIP is load-balanced to the backing pods. On most modern clusters, this is replaced or supplemented by the CNI (Cilium does its own routing entirely).

CNI plugin. Cilium, Calico, Flannel, AWS VPC CNI, etc. Assigns pod IPs, sets up the virtual networking, implements NetworkPolicies. We cover this in Chapter VIII.

The diagnostic model

When something goes wrong, locate the symptom in the architecture. Pod stuck Pending → scheduler. Pod stuck ContainerCreating → kubelet, CRI, CSI, or CNI. Pod CrashLooping → application or its config. NetworkPolicy not working → CNI. Service unreachable → kube-proxy or CNI. Deployment not updating → controller-manager. The two-plane model isn't theoretical; it's the map you use at 3 AM.

OBJECTS Chapter II

Workload objects · Pod to Job.

The hierarchy Pods are the atom. Everything else creates Pods (or other workload objects that create Pods).

The rule You almost never write Pod manifests directly in production. You write a higher-level controller's object; the controller creates the Pods.

Kubernetes objects form a hierarchy. Pods are the unit of execution — one or more containers sharing a network and storage namespace. Above Pods sit the workload controllers that create and manage them: Deployments for stateless replicas, StatefulSets for stable identity, DaemonSets for one-per-node, Jobs for run-to-completion, CronJobs for scheduled work. Around Pods sit Services for stable networking, ConfigMaps and Secrets for configuration, PersistentVolumeClaims for storage. Above everything sit Namespaces for tenancy.

Deployment vs StatefulSet — when each fits

Deployment creates Pods that are interchangeable. They have random suffixes (api-7f8d-x2k9c). They can land on any node, in any order. Scale up/down freely. Roll out by killing old, creating new, in batches. Use for: stateless APIs, workers, anything where the identity of any specific replica doesn't matter.

StatefulSet creates Pods with stable, predictable names (db-0, db-1, db-2) and stable per-Pod storage (each Pod gets its own PVC). They start in order, scale down in reverse order, and the network identity persists across restarts. Use for: databases, message brokers, distributed systems that elect leaders or shard by identity.

DaemonSet — the cluster's standing army

Every CNI, every node monitor, every log forwarder, every GPU device plugin is a DaemonSet. One Pod per matching node. Add a node, the DaemonSet controller schedules a Pod there. Drain a node, the Pod tolerates the taint (DaemonSets typically tolerate NoSchedule) so the system keeps working during upgrades.

Job — the unit of batch

A Job creates Pods that run to completion. parallelism: N runs N Pods simultaneously; completions: M requires M successful completions; backoffLimit: K retries failed Pods up to K times. Indexed mode (completionMode: Indexed) gives each Pod a unique index — the base for fan-out training and parameter sweeps.

apiVersion: batch/v1
kind: Job
metadata: { name: train-shard }
spec:
  parallelism: 16                    # 16 pods running at a time
  completions: 64                    # 64 total successful runs needed
  completionMode: Indexed
  backoffLimit: 3
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: worker
          image: ghcr.io/myorg/trainer@sha256:...
          command: [python, train.py]
          args: ["--shard=$(JOB_COMPLETION_INDEX)", "--total=64"]
The selection heuristic

Stateless replicas → Deployment. Stable identity → StatefulSet. One per node → DaemonSet. Run to completion → Job. Scheduled → CronJob. Fan-out batch → Indexed Job. Long-running multi-step DAG → don't use K8s natively; use Argo Workflows (Issue 08 Chapter II). This list covers ~95% of production cases. The remaining 5% are CRDs and Operators (Issue 08 Chapters V–VII).

Where each step fails — and how you'd know

Step 1 (apply) fails when authentication or authorization rejects you. kubectl apply returns a non-zero exit and the error tells you which step: "unauthorized" (no valid creds), "forbidden" (creds but no RBAC), "invalid" (manifest fails schema validation).

Step 2 (admission) fails when a validating webhook rejects (signed image policy, pod security standard, OPA policy) or a mutating webhook errors. kubectl describe on the deployment shows the rejection. kubectl get events in the namespace echoes it.

Step 3 (controllers) fails when the controller can't progress — Deployment says it wants 3 ReplicaSets but only 0 exist. Run kubectl describe deployment X; look at the Events at the bottom; look at controller-manager logs.

Step 4 (scheduling) fails when no node can accept the Pod. kubectl describe pod X shows FailedScheduling events with the exact reason: 0/12 nodes are available: 3 Insufficient cpu, 5 Insufficient memory, 4 node(s) didn't match Pod's node affinity. The most common reasons: resource shortage, taint mismatch, affinity rules.

Step 5 (kubelet) fails when the kubelet can't proceed. Pull errors (ErrImagePull, ImagePullBackOff) show up as kubectl describe events. Secret-mount errors (CreateContainerConfigError) too. Permission errors when mounting volumes.

Step 6 (runtime/CNI/CSI) fails when one of these subsystems errors. Long ContainerCreating with no obvious reason usually means CSI taking forever to attach a volume, or CNI failing to assign an IP. kubectl get events --field-selector involvedObject.name=POD and the kubelet logs reveal it.

Step 7 (readiness) fails as CrashLoopBackOff (process keeps dying) or perpetually-not Ready (readiness probe fails). kubectl logs POD + previous container logs (--previous) get you the application's perspective.

The five-command triage

For any "my pod isn't working": kubectl get pod -o wide (state + node), kubectl describe pod NAME (events at bottom), kubectl logs NAME --all-containers --previous (application output), kubectl get events -A --sort-by=.lastTimestamp | tail -20 (cluster-wide recent events), kubectl get pod NAME -o yaml (full state). These five answer ~80% of "why is this pod broken" questions in under a minute.

SCHED Chapter IV

Scheduling · the bin-packer in detail.

Two phases Filter (which nodes can accept this pod) and Score (which acceptable node is best). Both phases are pluggable; defaults work for most cases.

The reality Most scheduling problems are misunderstood. Misconfigured taints, missing topology spread, wrong affinity rules. The scheduler is honest; the manifests usually aren't.

The kube-scheduler runs two phases for every pending pod. Filter: walk all nodes, eliminate ones that fail any predicate (insufficient resources, taint mismatch, affinity unsatisfied). Score: for the remaining nodes, run scoring plugins (least-allocated, balanced-allocation, image-locality, inter-pod affinity), sum the scores, pick the highest. Bind: write spec.nodeName back through apiserver. The kubelet on that node takes over from there.

Resource requests vs limits — the most important manifest fields

requests are what the scheduler uses to decide if a pod fits. requests.cpu: 200m means "this pod needs at least 0.2 cores"; the scheduler subtracts that from the node's allocatable CPU. Requests are guaranteed: the pod will get at least that much.

limits are the runtime cap. limits.cpu: 1000m means "this pod can burst up to 1 core but no more." Once a CPU limit is hit, the kernel throttles the cgroup. For memory, hitting the limit means OOMKill.

The three QoS classes derived from these:

  • Guaranteed: requests == limits, set for every container. Last to be evicted under node pressure. Use for latency-sensitive services.
  • Burstable: requests < limits, or only one of the two set. Evicted before Guaranteed. The default for most services.
  • BestEffort: no requests, no limits. First to be evicted. Don't use for anything that matters.

nodeSelector, nodeAffinity, taints

nodeSelector: simple label match. nodeSelector: { gpu: "true" } only schedules on nodes labeled gpu=true. Hard requirement; if no matching node, pod stays Pending.

nodeAffinity: richer expressions. requiredDuringSchedulingIgnoredDuringExecution is a hard requirement (same as nodeSelector but more expressive). preferredDuringSchedulingIgnoredDuringExecution is a preference (scheduler scores higher but doesn't require).

taints + tolerations: the reverse direction. A taint on a node says "I'm not generally available." A toleration on a pod says "I'm OK with that taint." GPU nodes are usually tainted nvidia.com/gpu:NoSchedule; only pods that explicitly tolerate it land there.

# Node taint (set by GPU node provisioning or by hand)
$ kubectl taint nodes h100-node-1 nvidia.com/gpu=true:NoSchedule

# Pod tolerating it AND requesting GPU
spec:
  tolerations:
    - { key: nvidia.com/gpu, operator: Exists, effect: NoSchedule }
  nodeSelector:
    nvidia.com/gpu.product: "NVIDIA-H100-80GB-HBM3"     # pick specific GPU SKU
  containers:
    - name: trainer
      image: ...
      resources:
        limits: { nvidia.com/gpu: "8" }                       # request 8 GPUs

podAffinity, podAntiAffinity

Scheduling decisions relative to other pods. podAntiAffinity with topologyKey: kubernetes.io/hostname says "don't put me on the same node as another pod matching this label" — the canonical pattern for "spread replicas across nodes for HA." podAffinity with topologyKey: topology.kubernetes.io/zone says "co-locate with these pods in the same zone" — useful for chatty services.

topologySpreadConstraints — modern HA placement

The clearer way to say "spread across zones, racks, or nodes." Replaces most uses of podAntiAffinity for HA.

spec:
  topologySpreadConstraints:
    - maxSkew: 1                          # max imbalance between zones
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: DoNotSchedule       # hard requirement
      labelSelector:
        matchLabels: { app: api }
    - maxSkew: 1
      topologyKey: kubernetes.io/hostname
      whenUnsatisfiable: ScheduleAnyway        # prefer; fallback if can't
      labelSelector:
        matchLabels: { app: api }

Priority & preemption

PriorityClass objects assign numeric priority to pods. When a high-priority pod can't fit anywhere, the scheduler may preempt (evict) lower-priority pods to make room. Used carefully: production-critical pods get high priority; speculative/exploratory work gets low; preemption flow handles the rest.

Gang scheduling for distributed training

Standard scheduling is per-pod. A distributed training job needs all N pods to start together — schedule 7 of 8 and the job blocks waiting for the 8th, holding 7 GPUs idle. Gang scheduling via Volcano, kueue, kube-scheduler-plugins, or KAI Scheduler treats N pods as one unit: schedule all-or-none. Cross-reference Issue 01 Chapter V; Issue 02 explains why this matters for NCCL.

The scheduling rule of thumb

Every Pod needs requests. Without them, the scheduler can't bin-pack and the cluster fills with thrashing BestEffort pods. Every workload that matters needs topologySpreadConstraints across zones. Every node-class that's not for general use needs a taint. Every distributed-training pod needs a gang-scheduling backend. Skip any of these and you'll spend more time in kubectl describe than in code.

GPU Chapter V

GPU scheduling · device plugin, MIG, DRA.

The three eras Device plugin (legacy but dominant): whole GPUs only. MIG: hardware partitioning. DRA: the new ResourceClaim API replacing device plugins.

In 2026 DRA graduated to GA in Kubernetes 1.34. Device plugins are still everywhere; DRA is the future.

GPUs are not first-class Kubernetes resources. The scheduler natively knows about CPU and memory; everything else — GPUs, FPGAs, TPUs, network interfaces — is plumbed through extension mechanisms. The legacy mechanism, the device plugin framework, exposes GPUs as opaque extended resources (e.g., nvidia.com/gpu) that the scheduler bin-packs as integer counts. The new mechanism, Dynamic Resource Allocation (DRA), makes devices first-class API objects with rich claims and parameters.

The NVIDIA GPU stack on Kubernetes

Three components must be on every GPU node:

NVIDIA driver. The kernel module + userspace libraries. Installed on the host OS or as a DaemonSet via the GPU Operator.

NVIDIA Container Toolkit. Hooks into containerd/CRI-O to make GPU devices and CUDA libraries available inside containers. Without it, your container sees no GPU even on a GPU node.

NVIDIA device plugin. A DaemonSet that registers with kubelet and advertises GPU resources (nvidia.com/gpu: 8) to the API server.

The NVIDIA GPU Operator bundles all three plus DCGM (metrics), MIG manager, and the validator. Install once via Helm; every GPU node gets configured automatically.

# One-shot install — drivers + toolkit + device plugin + DCGM + MIG manager
helm repo add nvidia https://helm.ngc.nvidia.com/nvidia
helm install gpu-operator nvidia/gpu-operator \
    --namespace gpu-operator --create-namespace \
    --set driver.enabled=true \
    --set toolkit.enabled=true \
    --set mig.strategy=mixed \
    --wait

# Verify
kubectl get pods -n gpu-operator
kubectl get nodes -L nvidia.com/gpu.product,nvidia.com/gpu.count -l nvidia.com/gpu.present=true

The legacy device-plugin model

A Pod requests GPUs via resources.limits.nvidia.com/gpu: N. The scheduler treats this exactly like CPU and memory — an integer count, bin-packed. Requests are not used for extended resources; limits is what counts. GPUs are not divisible at this layer — request 1, get a whole GPU; request 2, get two whole GPUs.

This worked when every GPU was identical and whole. It broke once teams wanted: (a) to share a GPU across multiple small workloads, (b) to express NVLink topology preferences, (c) to mix whole-GPU and partitioned-GPU pods in the same cluster.

MIG — hardware partitioning on A100/H100/H200

Multi-Instance GPU lets you partition a physical A100 or H100 into up to 7 hardware-isolated instances. Each instance gets its own memory slice, compute SMs, and PCIe bandwidth. MIG partitions operate independently with full hardware-level fault isolation. Each partition is allocated its own private share of the GPU's memory and dedicated paths to GPU resources, such as caches, memory controllers, and compute components. This is the strongest form of GPU sharing.

Common MIG profiles on an 80GB H100:

  • 1g.10gb — 1/7 compute, 10GB memory. 7 of these per H100.
  • 2g.20gb — 2/7 compute, 20GB memory. 3 of these per H100.
  • 3g.40gb — 3/7 compute, 40GB memory. 2 of these per H100.
  • 7g.80gb — the whole GPU as one MIG instance (effectively no partitioning).

You can mix profiles on one physical GPU. The GPU Operator's MIG manager handles the (somewhat involved) MIG layout reconfiguration.

# With --set mig.strategy=mixed, slices appear as separate resources:
$ kubectl describe node h100-node-1 | grep nvidia.com
  nvidia.com/mig-1g.10gb:  7
  nvidia.com/mig-2g.20gb:  0
  nvidia.com/gpu:          0

# A pod requesting a 1g.10gb slice
spec:
  containers:
    - name: inference
      resources:
        limits:
          nvidia.com/mig-1g.10gb: "1"      # one 1g.10gb slice

Time-slicing — software multiplexing

Time-slicing lets multiple pods share a GPU by context-switching. Configure the device plugin to advertise replicas: N for each GPU; N pods can land on each physical GPU and the driver schedules them onto the GPU in time slices. Time-slicing trades the memory and fault-isolation that is provided by MIG for the ability to share a GPU by a larger number of users. There is no memory or fault-isolation between replicas, but for some workloads this is better than not being able to share at all.

Use cases: dev/CI environments where many small experiments share a GPU; older GPUs (V100, T4) that don't support MIG; many low-utilization inference replicas where some interference is tolerable.

apiVersion: v1
kind: ConfigMap
metadata: { name: time-slicing, namespace: gpu-operator }
data:
  any: |-
    version: v1
    sharing:
      timeSlicing:
        resources:
          - name: nvidia.com/gpu
            replicas: 4                # 4 pods per GPU; -SHARED suffix appears on node product label

MPS — Multi-Process Service

MPS is NVIDIA's CUDA-level multiplexing: multiple processes share a GPU through a daemon that arbitrates. Tighter than time-slicing (cooperative kernel scheduling) but still no memory isolation. Useful where time-slicing's context-switch overhead hurts. As of recent device-plugin versions, MPS support is available but considered experimental; not for production-critical workloads.

DRA — the future of device scheduling

Dynamic Resource Allocation graduated to GA in Kubernetes 1.34 (March 2026). It replaces the device-plugin framework with a ResourceClaim API — the same way PersistentVolumeClaim replaced hardcoded volume mounts.

The old model extended nvidia.com/gpu the same way it extended cpu or memory: an integer count the kube-scheduler bin-packed onto nodes. DRA makes the device a first-class API object with its own lifecycle and parameters. Pods declare ResourceClaim objects that a DRA driver materializes at bind time.

# ResourceClaimTemplate — what kind of GPU we want
apiVersion: resource.k8s.io/v1beta1
kind: ResourceClaimTemplate
metadata: { name: h100-claim }
spec:
  spec:
    devices:
      requests:
        - name: gpu
          deviceClassName: nvidia.com/gpu
          selectors:
            - cel:
                expression: "device.attributes['nvidia.com'].productName == 'H100'"
          count: 1

# Pod consuming the claim
apiVersion: v1
kind: Pod
spec:
  resourceClaims:
    - { name: trainer-gpu, resourceClaimTemplateName: h100-claim }
  containers:
    - name: trainer
      image: ...
      resources:
        claims:
          - { name: trainer-gpu }

The GPU Operator installs the driver stack and manages MIG; DRA is delivered by a separate NVIDIA component — the k8s-dra-driver — installed via its own Helm chart. As of GPU Operator 24.9, both remain separate charts; NVIDIA plans tighter integration.

KAI & Volcano — gang & queue scheduling

Standard scheduling doesn't understand "this batch of 8 pods must all start together." For ML training that requires it, third-party schedulers help:

Volcano: CNCF Incubating. Gang scheduling, queue management, fair-share allocation, preemption based on priorities. The most mature option.

kube-scheduler-plugins: the k8s-sigs reference. Includes Coscheduling (gang), CapacityScheduling, NodeResourceTopology.

KAI Scheduler: newer entrant. Logical fractional GPUs, AI-aware queues, integrates with DRA.

Kueue: K8s-native job queueing. Doesn't replace the scheduler; sits above it and admits Jobs into queues with capacity caps. Pairs well with default scheduler + Volcano.

The 2026 GPU operator's playbook

Install the NVIDIA GPU Operator. Taint GPU nodes (the Operator does this automatically). Set MIG strategy: none for whole GPUs, single for one MIG profile cluster-wide, mixed for per-node profiles. Configure node labels for MIG profile selection. Add a gang scheduler (Volcano or KAI) if you run distributed training. Plan a migration to DRA over the next 12 months. Cross-reference Issue 01 Chapters V–VI for the kernel-level details.

GPU Chapter VI

CUDA in containers · driver, runtime, toolkit.

The split Driver lives on the host. CUDA runtime + libraries live in the container. The container toolkit bridges the two.

The compatibility rule Host driver version sets a maximum CUDA version. Mismatch → cryptic errors. Pin both.

Running CUDA workloads in Kubernetes containers requires understanding which pieces live where. The kernel module and the libcuda.so userspace library — together "the driver" — must be on the host. The CUDA runtime (libcudart), cuDNN, NCCL, and the rest of the CUDA toolkit live inside the container. The NVIDIA Container Toolkit is the glue that exposes the host driver to the container.

The layered stack

+-------------------------------------------------------------+
|  Container (your pod)                                       |
|  ├─ Your app (PyTorch, vLLM, custom CUDA kernel)            |
|  ├─ CUDA Toolkit: libcudart, cuBLAS, cuDNN, NCCL            |  [bundled in image]
|  └─ /usr/local/cuda/lib64/* (toolkit version, e.g. 12.6)    |
+-------------------------------------------------------------+
                              ↓ bind-mount + LD_LIBRARY_PATH
+-------------------------------------------------------------+
|  Host                                                       |
|  ├─ /usr/lib/x86_64-linux-gnu/libcuda.so.560.35.03          |  [driver userspace]

The compatibility matrix

A driver version supports CUDA versions up to a maximum. Driver R535 → CUDA 12.2 max. R545 → CUDA 12.3. R560 → CUDA 12.6. R570 → CUDA 12.8. Your container's CUDA toolkit must be ≤ the host driver's supported version. Run a CUDA 12.6 container on a CUDA 12.2 host driver and you get CUDA driver version is insufficient for CUDA runtime version — fatally cryptic the first time you see it.

NVIDIA provides a forward-compatibility layer (cuda-compat packages) that lets a newer container run on a slightly older driver, but with limits. The clean rule: pin both. Pin the driver via the GPU Operator's driver.version. Pin the CUDA version in your container image (FROM nvcr.io/nvidia/pytorch:24.10-py3).

The NVIDIA Container Toolkit

Three pieces:

nvidia-container-runtime: a containerd/CRI-O runtime hook. When a container has nvidia.com/gpu in its resources, this hook bind-mounts the driver libraries from the host into the container and exposes the GPU device nodes.

nvidia-ctk: a CLI for configuring containerd to use the NVIDIA runtime.

libnvidia-container: the C library implementing the bind-mount logic.

Result: a container image that knows nothing about NVIDIA can run CUDA workloads because the host's driver libraries appear inside it at runtime, with matching versions.

# On a GPU node, this should work end-to-end
$ docker run --rm --gpus all nvcr.io/nvidia/cuda:12.6.0-base-ubuntu24.04 nvidia-smi

CUDA_VISIBLE_DEVICES and the device plugin contract

When a pod requests nvidia.com/gpu: 4, the device plugin chooses which 4 GPUs to expose and sets NVIDIA_VISIBLE_DEVICES in the container's environment. The container runtime hook reads that env var, mounts the corresponding device nodes (/dev/nvidia0, /dev/nvidia1...) into the container, and exposes the right libraries.

Inside the container, nvidia-smi shows only those 4 GPUs (numbered 0–3 from the container's perspective, even if they're physically GPUs 4–7 on the host). Your CUDA code sees only those 4. This is the isolation.

NCCL and topology

For multi-GPU training (NCCL collectives), the device plugin matters less than what gets exposed. NCCL discovers GPU topology via /proc, /sys, and libnvidia-ml. For NVLink to work between GPUs, all participating GPUs must be on the same node AND the NCCL initialization must see the NVLink topology.

Cross-reference Issue 02 for the deep NCCL field manual. For the K8s slice of this:

  • Multi-GPU training jobs should land on a single node (use resources.limits.nvidia.com/gpu: 8 for a full DGX-style node).
  • For multi-node training (4 nodes × 8 GPUs = 32 GPUs), the inter-node link is RDMA (InfiniBand or RoCE). Surface this via the network plugin (Multus + SR-IOV for HCAs).
  • Set NCCL_IB_HCA and friends per Issue 02 Chapter VII; the cluster operator must ensure pods can see the HCA devices.

NVIDIA's Data Center GPU Manager exposes GPU metrics (utilization, memory, temperature, power, ECC errors, MIG state). The GPU Operator deploys dcgm-exporter as a DaemonSet; it scrapes DCGM and exposes Prometheus metrics. Per-pod attribution requires correlating Pod UUIDs with GPU UUIDs — DCGM-Exporter handles this in most configurations.

The driver-toolkit pin discipline

Pin the driver version in the GPU Operator. Pin the CUDA toolkit version in your container image. Document both in your runbook. Upgrade them together, not separately. Use NVIDIA's NGC base images (nvcr.io/nvidia/pytorch:24.10-py3 etc.) — they're built against specific driver versions and tested for compatibility. The single biggest source of GPU-on-K8s pain is driver/runtime drift; one paragraph of discipline prevents it.

SCALE Chapter VII

Autoscaling · HPA, VPA, KEDA, Karpenter.

Two layers Pod scaling (HPA/VPA/KEDA) decides replica count or per-pod size. Node scaling (CA/Karpenter) decides node count.

The combinator None of these tools work in isolation. Pick the right pairs.

Kubernetes scaling happens on two distinct layers. Pod scaling (application layer): adjusting the number of pod replicas or the size of individual pods — that's HPA, VPA, KEDA. Node scaling (infrastructure layer): adjusting the number of virtual machines in the cluster to support the pods — that's Cluster Autoscaler or Karpenter. If you scale your pods but have no nodes to place them on, your scaling fails (Pending Pods).

The four autoscaler types

HPA mathematics

HPA runs a control loop every 15 seconds. It fetches the current metric value from the Kubernetes Metrics API (for CPU/memory) or the Custom Metrics API (for application metrics like RPS or queue depth). It then applies the formula: desiredReplicas = ceil(currentReplicas × currentMetricValue / targetMetricValue). For example, if the target CPU utilization is 70% and 10 pods are running at 105% average utilization, desiredReplicas = ceil(10 × 105/70) = 15 — HPA creates 5 new pods. HPA never scales below minReplicas or above maxReplicas.

For scale-down, the stabilization window (default 5 minutes) prevents rapid oscillation: HPA tracks the maximum recommended replica count over the window and does not scale down faster than that.

KEDA — the event-driven extension

KEDA extends HPA with 60+ external scalers (Kafka lag, SQS depth, Redis list length, Prometheus queries, cron schedules, RabbitMQ, GCS Pub/Sub, NATS, Azure Service Bus, and more). The killer feature is scale-to-zero — HPA's minimum is 1 replica; KEDA can scale to 0 pods when there are no events and back up when events arrive.

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata: { name: order-processor }
spec:
  scaleTargetRef: { name: order-processor }
  minReplicaCount: 0                  # scale to zero when idle
  maxReplicaCount: 100
  pollingInterval: 10                  # check trigger every 10s
  cooldownPeriod: 300                  # wait 5 min before scaling down
  triggers:
    - type: aws-sqs-queue
      metadata:
        queueURL: https://sqs.us-east-1.amazonaws.com/.../orders
        queueLength: "5"                # 1 replica per 5 messages
        awsRegion: us-east-1

The combinator — pick the right pairs

None of these tools work alone. Production patterns:

  • Stateless API on AWS: HPA on CPU + Karpenter. Sub-minute scale-up; sub-minute node provision. The default for new EKS workloads.
  • Queue worker: KEDA + Karpenter (or KEDA + CA). Scale-to-zero when the queue is empty; spin up nodes only when work arrives.
  • Batch/ML training: Job + Karpenter with spot priority. Spot nodes for the bulk; on-demand fallback. Eviction-tolerant by design (checkpointing).
  • Stateful (DB, broker): VPA in recommendation mode (not Auto — Auto evicts pods which breaks state). Manually right-size based on VPA recommendations.

The bear traps

HPA + VPA on the same metric: they fight. HPA scales replicas to lower per-pod CPU; VPA increases per-pod CPU because it thinks the pod is undersized. Result: oscillation. Either use VPA on different metrics than HPA (memory only if HPA is on CPU), or use VPA in recommendation mode and apply changes manually.

No resource requests: HPA can't calculate percentage usage without requests in the container spec. Karpenter makes bin-packing decisions based on requests. Without requests, scheduling becomes random, autoscaling becomes a guessing game.

No PodDisruptionBudget: when Karpenter or CA scales down, they evict pods. Without a PDB, they might kill all replicas of your service simultaneously.

maxReplicas too low: if your peak traffic requires 100 pods but maxReplicas is 20, autoscaling silently degrades service.

Polling interval too low on KEDA: a pollingInterval of 5 seconds on external sources (SQS, external APIs) generates too many API calls and possible throttling.

The autoscaling discipline

Every workload has resource requests. Every workload has a PDB. Every Deployment has an HPA (or KEDA for event-driven). Every cluster has a node autoscaler (Karpenter on AWS; CA elsewhere). Production maxReplicas accounts for 3× peak; CA/Karpenter max-node-count accounts for 1.5× the maxReplicas worth of capacity. These six rules cover ~90% of autoscaling-related incidents.

NET Chapter VIII

Networking · CNI, Services, Ingress, NetPol.

The four layers Pod-to-pod (CNI). Pod-to-service (kube-proxy/CNI). Service-to-outside (Ingress/Gateway). Pod-to-pod policy (NetworkPolicy).

The 2026 default Cilium for CNI, Gateway API for L7, Cilium Network Policies for L3-L7 policy. eBPF eats everything.

Kubernetes networking layers stack: every pod gets its own routable IP (CNI's job); Services give stable virtual IPs for groups of pods (kube-proxy or CNI); Ingress and Gateway API route external HTTP traffic to Services; NetworkPolicies restrict which pods can talk to which. Each layer is replaceable, and the 2026 production stack increasingly is Cilium for everything except the actual Ingress controller.

The pod network — CNI

The Container Network Interface (CNI) is a small spec: when kubelet starts a pod, it calls the configured CNI plugin with the pod's namespace; the plugin assigns an IP, sets up veth pairs / vxlan tunnels / IPIP / native routing, and reports back. Every pod gets a unique IP across the cluster.

The dominant CNIs:

  • Cilium: eBPF-based. Replaces kube-proxy with eBPF programs. Implements L3/L4/L7 NetworkPolicies natively. Best observability via Hubble. The 2026 default for new clusters.
  • Calico: BGP-based for native routing or VXLAN. Mature. Strong NetworkPolicy support. Used in older / multi-cloud deployments.
  • AWS VPC CNI: assigns real VPC IPs to pods. No overlay; native routing. Default on EKS. Limits per node based on ENIs.
  • Flannel: simple VXLAN overlay. Legacy; not common in new deployments.
  • Multus: meta-plugin that attaches multiple networks to a pod. Used for SR-IOV / RDMA HCAs.

Services — stable IPs for groups of pods

A Service is a virtual IP plus a DNS name that load-balances to a set of pods (selected by label). Five types:

  • ClusterIP (default): only reachable from inside the cluster. app.namespace.svc.cluster.local.
  • NodePort: exposes the Service on a port (30000–32767) on every node. Mostly historical; ALB / Ingress is better.
  • LoadBalancer: cloud controller creates a real LB (AWS NLB/ALB, GCP LB, Azure LB). Maps to one Service.
  • ExternalName: DNS alias to an external service. No proxying.
  • Headless (clusterIP: None): DNS returns A records for each pod directly. Used by StatefulSets for per-pod DNS.
# Stable IP for the cluster — load-balances to all matching pods
apiVersion: v1
kind: Service
metadata: { name: api, namespace: prod }
spec:
  selector: { app: api }
  ports: [{ port: 80, targetPort: 8080, name: http }]

---
# Headless — for db StatefulSet, get db-0.db, db-1.db, db-2.db DNS
apiVersion: v1
kind: Service
metadata: { name: db, namespace: prod }
spec:
  clusterIP: None
  selector: { app: db }
  ports: [{ port: 5432, targetPort: 5432, name: pg }]

kube-proxy modes — and how Cilium replaces it

kube-proxy programs the Service-to-Pod load balancing. Three implementations:

  • iptables: default. Rules grow O(N×M) with services × backends; performance degrades at scale.
  • ipvs: kernel-level L4 LB. Better performance than iptables at scale.
  • eBPF: Cilium replaces kube-proxy entirely with eBPF programs. Best performance; lowest latency; rich observability.

Ingress and the Gateway API

Ingress (the original) is a deprecated-but-still-everywhere resource for HTTP routing. Gateway API (GA in K8s 1.31, March 2025) is the modern replacement with three resources: GatewayClass (the implementation), Gateway (the listener), and HTTPRoute / GRPCRoute / TCPRoute (the routing rules).

Common implementations: nginx-ingress (Ingress), Envoy Gateway (Gateway API), Istio (both), AWS Load Balancer Controller, Contour, Traefik. Pick one per cluster; running two is asking for trouble.

apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata: { name: prod-gateway }
spec:
  gatewayClassName: envoy
  listeners:
    - { name: https, port: 443, protocol: HTTPS, tls: { mode: Terminate, certificateRefs: [{ name: prod-cert }] } }

---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata: { name: api-route }
spec:
  parentRefs: [{ name: prod-gateway }]
  hostnames: ["api.example.com"]
  rules:
    - matches: [{ path: { type: PathPrefix, value: /v1 } }]
      backendRefs: [{ name: api, port: 80, weight: 100 }]
    - matches: [{ path: { type: PathPrefix, value: /v2 } }]
      backendRefs:
        - { name: api-v2-canary, port: 80, weight: 10 }   # 10% to canary
        - { name: api-v2, port: 80, weight: 90 }

NetworkPolicy — the firewall inside the cluster

By default, all pods can talk to all other pods. NetworkPolicy changes that: declare what traffic is allowed for a pod, and everything else is denied. Implementation-dependent — the CNI must support NetworkPolicy (Cilium, Calico do; AWS VPC CNI needs Calico added for it).

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: api-allow, namespace: prod }
spec:
  podSelector:
    matchLabels: { app: api }
  policyTypes: [Ingress, Egress]
  ingress:
    - from:
        - namespaceSelector: { matchLabels: { name: ingress-system } }
          podSelector: { matchLabels: { app: envoy-gateway } }
        - namespaceSelector: { matchLabels: { name: observability } }
      ports: [{ port: 8080 }]
  egress:
    - to:
        - namespaceSelector: { matchLabels: { name: prod } }
          podSelector: { matchLabels: { app: db } }
      ports: [{ port: 5432 }]
    - to:                                       # DNS — required
        - namespaceSelector: { matchLabels: { name: kube-system } }
          podSelector: { matchLabels: { k8s-app: kube-dns } }
      ports: [{ port: 53, protocol: UDP }]

DNS — CoreDNS

CoreDNS is the default DNS server in K8s. It resolves service.namespace.svc.cluster.local and the shorter forms (service.namespace, service in the same namespace). It also forwards external lookups to upstream resolvers.

The two failures: (1) CoreDNS pod count too low for the cluster's QPS — symptom is slow DNS resolution and timeouts; fix is HPA on CoreDNS. (2) ndots: 5 in /etc/resolv.conf causing every external lookup to be tried as 5 internal forms first — symptom is slow external lookups; fix is dnsConfig: { options: [{ name: ndots, value: "1" }] } in pod specs that talk to many external hosts.

The 2026 network stack

Cilium for CNI, kube-proxy replacement, NetworkPolicy enforcement, and observability via Hubble. Envoy Gateway or AWS LB Controller for Ingress/Gateway. CoreDNS HPA-scaled. NetworkPolicy default-deny per namespace, then explicit allow rules per app. This combination gives sub-millisecond pod-to-pod latency, observable traffic, enforced policy, and modern L7 routing. Cross-reference Issue 06 Chapter IX for the deeper networking performance discussion.

STORAGE Chapter IX

Storage · volumes, CSI, StatefulSets.

The split Ephemeral volumes (lifecycle = pod). Persistent volumes (lifecycle = independent). CSI is the plug-in spec for all of them.

For ML Model weights, training checkpoints, dataset caches. Choose volumes by access pattern, not by size.

Kubernetes storage layers like compute storage in general: volumes attach to pods; some volumes outlive their pods, some don't; the CSI (Container Storage Interface) is how all storage drivers — AWS EBS, GCP PD, NFS, Ceph, Lustre, S3-as-FUSE — plug into the cluster.

Volume types by lifecycle

  • emptyDir: a scratch directory created when the pod starts, deleted when the pod terminates. Backed by the node's disk by default; back with medium: Memory for a tmpfs (RAM-backed). Used for scratch space.
  • configMap / secret: project a ConfigMap or Secret into the pod as files.
  • downwardAPI: project pod metadata (labels, annotations, resources) into files.
  • projected: combine multiple of the above into one mount path.
  • hostPath: mount a directory from the host's filesystem. Use sparingly (DaemonSets needing access to /var/log); a security risk if misused.
  • persistentVolumeClaim: the durable case — survives pod restarts, can re-attach to a new pod.
  • ephemeral CSI: like PVC but lifecycle = pod. Useful for short-lived workloads needing CSI features (encryption, snapshots) without managing a separate PVC.

PersistentVolumes — the durable layer

A PersistentVolume (PV) is a piece of storage in the cluster (an EBS volume, an NFS export, a Ceph RBD image). A PersistentVolumeClaim (PVC) is a request for storage with a size and access mode. A StorageClass is a template: "provision an EBS gp3 volume of this size with these settings on demand." Most production clusters use dynamic provisioning — you create a PVC, the StorageClass + CSI driver create a matching PV automatically.

# StorageClass — defines how to provision volumes
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata: { name: fast-ssd }
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: "10000"
  throughput: "500"
  encrypted: "true"
volumeBindingMode: WaitForFirstConsumer       # key — bind only after pod scheduling
reclaimPolicy: Retain                          # keep volume on PVC deletion (production)
allowVolumeExpansion: true

---
# PVC — a request for 100Gi of fast-ssd
apiVersion: v1
kind: PersistentVolumeClaim
metadata: { name: postgres-data, namespace: prod }
spec:
  accessModes: [ReadWriteOnce]
  storageClassName: fast-ssd
  resources:
    requests: { storage: "100Gi" }

WaitForFirstConsumer is the critical setting. With Immediate binding, the PV is created when the PVC is created — which means it lands in whatever AZ the volume provisioner picks. Then the pod might get scheduled in a different AZ and can't attach the volume. WaitForFirstConsumer delays PV creation until the pod is scheduled, so the volume lands in the right AZ. Set this on every cloud-based StorageClass.

Access modes

  • ReadWriteOnce (RWO): one node can mount r/w at a time. EBS, GCP PD, Azure Disk. The most common.
  • ReadOnlyMany (ROX): multiple nodes can mount r/o. Used for shared base data.
  • ReadWriteMany (RWX): multiple nodes can mount r/w. NFS, EFS, Azure Files, CephFS. Slower than RWO; use only when you need it.
  • ReadWriteOncePod (RWOP): only one pod can mount, even on the same node. The strictest.

StatefulSets and volume claim templates

StatefulSets use volumeClaimTemplates to generate a PVC per replica automatically. Replica db-0 gets PVC data-db-0; db-1 gets data-db-1. Scale up creates new PVCs. Scale down does not delete PVCs (so data isn't lost on accidental scaledown).

apiVersion: apps/v1
kind: StatefulSet
metadata: { name: db, namespace: prod }
spec:
  serviceName: db                          # headless Service for stable DNS
  replicas: 3
  selector: { matchLabels: { app: db } }
  template:
    metadata: { labels: { app: db } }
    spec:
      containers:
        - name: postgres
          image: postgres:16
          volumeMounts: [{ name: data, mountPath: /var/lib/postgresql/data }]
  volumeClaimTemplates:
    - metadata: { name: data }
      spec:
        accessModes: [ReadWriteOnce]
        storageClassName: fast-ssd
        resources: { requests: { storage: "500Gi" } }

The ML storage profile

ML workloads have three storage axes:

Model weights: large (10s–100s of GB), read-heavy, immutable. Pattern: bake into the image (for small models), or stage to a fast local volume on pod start, or use FUSE-mounted object storage (s3fs, mountpoint-s3, gcsfuse). Cross-reference Issue 06 Chapter IV on cold starts.

Training checkpoints: large, written periodically, read on resume. Pattern: parallel filesystems (Lustre via FSx, WekaFS, Vast Data). For smaller scales, NFS or EFS works.

Dataset caches: huge, read-heavy, sometimes shared across pods. Pattern: JuiceFS or Alluxio as a caching layer over object storage; RWX volume mounted into all training pods.

Volume snapshots and clones

CSI supports VolumeSnapshot and VolumeSnapshotClass for point-in-time snapshots, plus dataSource on a PVC to clone from a snapshot or another PVC. Used for: pre-prod databases cloned from prod (with PII scrubbed), training-checkpoint snapshots for rollback, disaster recovery.

The hostPath / local volume case

For workloads needing the absolute fastest local storage (NVMe-backed checkpointing, scratch for shuffling), use local PV. The volume binds to a specific node; pods using it must land there. Manage carefully — losing the node means losing the volume.

The storage decision

EBS/PD/AzureDisk + RWO for almost everything stateful (databases, single-writer workloads). EFS/Azure Files + RWX when you genuinely need shared filesystem semantics. Local NVMe + nodeAffinity for the highest-performance ephemeral workloads. Object storage + FUSE or DataSet caching layer for ML datasets. WaitForFirstConsumer on every cloud StorageClass. reclaimPolicy: Retain in production. Snapshot before any destructive operation.

MULTI Chapter X

Multi-cluster · fleet architecture & rollouts.

Why multi-cluster Blast radius, region, sovereignty, cost. One cluster per region is the floor; per environment + region is common.

The model Hub-and-spoke: a management cluster controls many workload clusters. GitOps spans all of them. Same artifacts everywhere.

Once you have more than one cluster — and most production deployments do — fleet management becomes the central problem. Single-cluster patterns don't scale: you don't want to maintain 12 separate Argo CD installations or 12 separate Prometheus instances. The mature pattern is a hub cluster running fleet-wide controllers (GitOps, observability, policy, secret distribution) and spoke clusters running workloads.

Why multiple clusters

  • Blast radius: control-plane upgrades, CNI changes, RBAC changes — one mistake takes down one cluster, not your whole footprint.
  • Regional latency: workloads near users. Multi-region implies multi-cluster.
  • Data sovereignty: EU data stays in EU clusters; PCI workloads in dedicated clusters.
  • Capacity / cost: one cluster per cloud account when accounts have separate cost centers.
  • Workload class: training clusters with H100s separate from inference clusters separate from CI clusters.
  • Failure domains: hot standby clusters that take over when the primary fails.

Argo CD multi-cluster — the canonical setup

apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata: { name: api-fleet, namespace: argocd }
spec:
  generators:                                    # where to find clusters
    - clusters:
        selector:
          matchLabels: { env: prod }
  template:
    metadata:
      name: 'api-{{name}}'                  # one Application per cluster
    spec:
      project: production
      source:
        repoURL: https://github.com/myorg/deployments
        targetRevision: main
        path: 'clusters/{{metadata.labels.region}}/api'

        server: '{{server}}'                       # cluster's API server URL
        namespace: api

        automated: { prune: true, selfHeal: true }

Result: register N clusters with Argo CD (label them with env+region); the ApplicationSet generates one Application per cluster automatically. Add a new cluster → it gets the standard workloads without manual setup. Delete a cluster → its Applications disappear.

Rollout strategies across clusters

Progressive cluster waves: dev → staging → one prod region → all prod regions. Each wave gated by health checks. Argo Rollouts + ApplicationSet does this; Flagger is the analog.

Time-bake: a change sits in staging clusters for N hours with no alerts before promotion is auto-merged to prod. Cross-reference Issue 08 Chapter IV.

Region-by-region: us-east-1 first, then us-west-2, then eu-west-1, then ap-southeast-1. Each gated by canary success.

Blue-green per cluster: stand up parallel cluster, drain traffic over, decommission old. Used for K8s version upgrades when in-place upgrades are too risky.

Service connectivity across clusters

Three patterns:

Per-cluster ingress + DNS routing: each cluster has its own external LB; a global DNS service (Route53, Cloud DNS) routes users to the nearest healthy cluster. Simplest; what most teams start with.

Service mesh federation: Istio multi-cluster, Linkerd multi-cluster, or Cilium Cluster Mesh. Pods in cluster A can call svc.namespace.svc.cluster.local and reach pods in cluster B. Used when services need to be reachable across clusters with mTLS.

API gateway in front: a single global gateway routes per-route to per-cluster backends. Common for SaaS architectures.

Centralized observability

Per-cluster Prometheus scrapes locally; forwards to Thanos receive, Mimir, or Cortex in the hub. The hub runs Grafana with a single data source spanning all clusters. Same for logs (Loki / Grafana Cloud / Splunk) and traces (Tempo / Jaeger). Cross-reference Issue 06 Chapters XI–XII.

Cluster API — Kubernetes managing Kubernetes

Cluster API (CAPI) is the CNCF-graduated standard for declarative cluster lifecycle. Define a Cluster CR, the operator provisions the actual cluster on AWS / GCP / Azure / vSphere / bare metal. Upgrade by editing the CR. The hub cluster runs CAPI controllers; spokes are reconciled.

Alternative: Karmada, OpenClusterManagement, Crossplane for multi-cluster orchestration with different abstractions.

The two-cluster minimum

Even small organizations should run at least two clusters: management (hub) + workloads. Adding a third for staging is the next natural step. From there, scale by environment × region. The hub stays small (15-25 nodes for most orgs); spokes scale with workload. Use ApplicationSet for declarative multi-cluster deployment; use Cluster API for declarative cluster lifecycle; use mesh federation only when services genuinely need to call across clusters.

OBS Chapter XII

Observability · metrics, logs, traces, events.

The four signals Metrics, logs, traces, events. K8s emits all four natively; you collect, store, and correlate.

The default stack Prometheus + Loki + Tempo + Grafana. Or LGTM stack. Or cloud-native (CloudWatch / Cloud Monitoring / Azure Monitor).

Operating Kubernetes without observability is operating blind. Four signal types — metrics (rates and counts), logs (text events), traces (request flows), and Kubernetes events (state transitions) — together tell you what the cluster is doing. The 2026 default stack is Prometheus + Loki + Tempo (the LGTM stack), or a managed equivalent. Cross-reference Issue 06 Chapters XI–XII for the deeper performance-instrumentation discussion; this chapter covers the K8s-specific layer.

Metrics — the prometheus stack

kube-state-metrics exports the state of every K8s object as metrics: kube_pod_status_phase, kube_deployment_status_replicas_available, kube_node_status_condition. Different from metrics-server which exposes per-pod CPU/memory for HPA.

node-exporter as a DaemonSet exports host-level metrics: CPU, memory, disk, network at the node level. cAdvisor (embedded in kubelet) exports per-container resource usage.

Application metrics exposed at /metrics; Prometheus scrapes via a ServiceMonitor or PodMonitor (Prometheus Operator CRDs). Standard naming: http_requests_total, http_request_duration_seconds, business metrics like orders_processed_total.

apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: api-metrics
  namespace: prod
  labels: { release: kube-prometheus-stack }       # Prometheus selects via label
spec:
  selector:
    matchLabels: { app: api }
  endpoints:
    - port: metrics
      interval: 15s
      path: /metrics
      scheme: http

The four golden signals — and where they live

  • Latency: histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))
  • Traffic: sum(rate(http_requests_total[5m]))
  • Errors: sum(rate(http_requests_total{code=~"5.."}[5m])) / sum(rate(http_requests_total[5m]))
  • Saturation: 1 - (avg(node_cpu_seconds_total{mode="idle"}[1m])) for CPU; analogous for memory, disk, network.

Alert on these per SLO; everything else is supporting context.

Logs — Loki + Promtail or Fluent Bit

Container logs go to /var/log/pods/... on the node. A DaemonSet (Promtail, Fluent Bit, Vector) tails them, parses, adds K8s metadata (namespace, pod, container, labels), forwards to a backend (Loki, Elasticsearch, Splunk).

Structured logging is non-negotiable: applications emit JSON; the collector adds K8s metadata; Loki indexes labels but stores log lines as text. Querying via LogQL: {namespace="prod",app="api"} |= "error" | json | duration_ms > 1000.

        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ],
)

log = structlog.get_logger()
log.info("request.received", method="POST", path="/v1/predict", duration_ms=42)
# Emits: {"event":"request.received","method":"POST","path":"/v1/predict",
#         "duration_ms":42,"level":"info","timestamp":"2026-05-13T..."}

Traces — OpenTelemetry + Tempo / Jaeger

Auto-instrument with OpenTelemetry SDKs (Python, Go, Java, Node, Rust). Spans flow to a collector (otel-collector DaemonSet or Deployment); the collector forwards to Tempo, Jaeger, Honeycomb, Datadog, etc. Trace context propagates via HTTP headers (W3C Trace Context) so a request crossing 5 services shows as one trace.

The critical OTel discipline: span on every external call (DB query, HTTP call to downstream service, queue publish/consume) with attributes (db.statement, http.url, messaging.destination). Cross-reference Issue 06 Chapter XI for the deeper instrumentation discussion.

Events — Kubernetes' built-in changelog

kubectl get events shows what's happening: pod scheduled, image pulled, container started, liveness probe failed, autoscaler triggered. Events are first-class K8s objects with a 1-hour default TTL. Forward to your log aggregator (with event-exporter or kube-eventer) for long-term retention and alerting.

Events are how you reconstruct "what was happening at 3:47 AM" post-incident. Critical, often forgotten in observability setups.

Profiles — eBPF-based continuous profiling

Parca, Pyroscope, Polar Signals use eBPF to continuously profile every process in the cluster. CPU profiles, off-CPU profiles, memory allocation. Per-pod attribution; cross-correlate with metrics and logs. Cross-reference Issue 06 Chapter XII.

DEBUG Chapter XIV

Communication issues · DNS, MTU, kube-proxy.

The four layers DNS, kube-proxy, CNI, application. Most "network" problems are in one of these. Bisect.

The first tool nslookup + curl -v from inside a debug pod. If those work, the problem is application-side.

"My service can't talk to that service" is the most common K8s incident. The cause is almost always one of four layers: DNS not resolving, Service IP not routing, NetworkPolicy blocking, or the application itself misbehaving. Triage by bisection.

The bisection procedure

# Spawn a debug pod in the namespace
$ kubectl run -it --rm debug --image=nicolaka/netshoot -n NAMESPACE -- bash

# Step 1 — DNS resolution
$ nslookup target-service.target-namespace.svc.cluster.local
# If FAILS: DNS issue. Check CoreDNS. Try resolving an external name (google.com).
# If WORKS: continue.

# Step 2 — TCP reachability to ClusterIP
$ nc -zv target-service.target-namespace.svc.cluster.local 8080
# If FAILS: routing issue. Check Service endpoints (next step).
# If WORKS: continue.

# Step 3 — endpoints exist for the Service
$ kubectl get endpoints -n target-namespace target-service
# If empty: no pods match selector OR pods not Ready. Check pod labels & readiness.
# If populated: continue.

# Step 4 — application-level
$ curl -v http://target-service.target-namespace:8080/healthz
# If TCP works but HTTP 5xx: application issue, not network.
# If hangs: NetworkPolicy may be blocking egress/ingress. Check policies.

DNS issues — the most common

CoreDNS overloaded: symptoms are random NXDOMAIN responses or slow lookups. Default CoreDNS replica count (2) is too low for large clusters. Run HPA on CoreDNS; aim for one replica per ~30-50 nodes.

ndots: 5 blowing up external lookups: K8s's default /etc/resolv.conf has options ndots:5. Any name with fewer than 5 dots gets tried as 5 internal forms first. api.stripe.com resolves as api.stripe.com.NAMESPACE.svc.cluster.local, then api.stripe.com.svc.cluster.local, then api.stripe.com.cluster.local, then api.stripe.com. Slow.

spec:
  dnsPolicy: ClusterFirst
  dnsConfig:
    options:
      - { name: ndots, value: "1" }              # 1 dot → external lookup first
      - { name: single-request-reopen }              # mitigate conntrack races

conntrack DNS race: classic K8s bug where rapid DNS queries trip a kernel conntrack race, manifesting as 5s timeout on lookups. Mitigations: single-request-reopen as above, or switch to nodelocal DNS cache.

NetworkPolicy issues

If you've turned on NetworkPolicy in a namespace (or set a default-deny), pods can't talk to anything not explicitly allowed. Forgetting to allow DNS is the most common bug: pods can't resolve names because egress to kube-system/kube-dns:53 isn't allowed.

apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata: { name: allow-dns, namespace: prod }
spec:
  podSelector: {}                              # applies to all pods in namespace
  policyTypes: [Egress]
  egress:
    - to:
        - namespaceSelector: { matchLabels: { kubernetes.io/metadata.name: kube-system } }
          podSelector: { matchLabels: { k8s-app: kube-dns } }
      ports:
        - { protocol: UDP, port: 53 }
        - { protocol: TCP, port: 53 }

kube-proxy and Service routing

If TCP to a ClusterIP fails but the pods are Ready and Endpoints exist, kube-proxy isn't programming rules correctly. With iptables mode: iptables -t nat -L KUBE-SERVICES | grep IP on the node. With Cilium: cilium service list in the cilium agent pod.

For Cilium specifically: cilium status, cilium connectivity test for end-to-end checks. Hubble for flow observability: hubble observe --from-pod prod/api --verdict DROPPED shows you exactly which packets got blocked and why.

MTU issues — the silent killer

MTU (Maximum Transmission Unit) mismatch between the pod network and the underlying network causes silent packet drops on large packets — small packets work, large ones don't. Symptoms: TLS handshake works (small) but HTTP responses time out (large). ssh connects but scp hangs.

Common causes: VXLAN overlay adds ~50 bytes; if pod MTU = node MTU, encapsulated packets exceed wire MTU. Fix: set pod MTU 50 bytes below node MTU. Cilium auto-detects; Calico needs configuration; AWS VPC CNI matches by default.

# Inside the source pod
$ ping -M do -s 1472 target-pod-ip                 # 1500 - 28 = 1472; should work
$ ping -M do -s 8972 target-pod-ip                 # 9000 - 28 = 8972; only with jumbo

# If 1472 fails, MTU is mismatched somewhere. Check actual MTU:
$ ip link show eth0
2: eth0: mtu 1450                            # lower than expected → fix CNI config

# For Cilium
$ cilium config get mtu

Inter-cluster connectivity

For multi-cluster (Chapter X), service-to-service across clusters depends on the chosen federation:

Cilium Cluster Mesh: pods in cluster A can call service.namespace.svc.cluster.local with cross-cluster name resolution. Failure modes: DNS not federated, mTLS not configured, BGP peering down between clusters.

Istio multi-cluster: similar but mesh-managed. Failure modes: control planes out of sync, certificate issues.

Per-cluster ingress: just plain external HTTP. Failure modes: DNS routing wrong, regional cert mismatch.

PERF Chapter XV

Performance & scalability · the knobs.

The two axes Cluster scale (nodes, pods, API ops/sec). Workload performance (latency, throughput per pod).

The bottleneck For large clusters, it's almost always etcd and apiserver. For workloads, it's almost always resource configuration.

Kubernetes scales much further than most assume — production clusters with 5,000+ nodes exist — but only with deliberate tuning. The knobs cluster into two groups: scaling the cluster itself (control plane capacity), and tuning workload performance (data plane behavior).

Control plane scaling

etcd is the bottleneck for large clusters. Tune:

  • Dedicated SSDs with fsync < 10ms. etcd_disk_backend_commit_duration_seconds should be at p99 < 25ms.
  • 3-member or 5-member cluster — never more (Raft consensus cost grows).
  • Compact regularly (controller-manager handles this, but watch etcd_mvcc_db_total_size_in_bytes; keep under 8GB).
  • Defragment periodically — etcdctl defrag can be needed on busy clusters.

kube-apiserver:

  • Scale horizontally (3+ replicas). Stateless except for cache.
  • Increase --max-requests-inflight and --max-mutating-requests-inflight for large clusters.
  • Enable --enable-priority-and-fairness (default in modern versions) — fair scheduling of API requests.
  • Watch apiserver_request_duration_seconds p99 per verb; LIST and WATCH are the expensive ones.

controller-manager:

  • --kube-api-qps and --kube-api-burst raise default rate limits.
  • Per-controller concurrency: --concurrent-deployment-syncs, --concurrent-replicaset-syncs.
  • workqueue_depth per controller is the saturation signal.

Cluster-wide limits

Documented Kubernetes scaling thresholds (the ones you should know):

  • 5,000 nodes per cluster (default supported).
  • 150,000 pods total.
  • 110 pods per node (default; configurable up to ~250 with tuning).
  • 300,000 containers total.

Beyond these, you split into multiple clusters (Chapter X). Big organizations run dozens of clusters, not one giant one.

Workload performance knobs

Resource requests & limits: the single biggest lever. Too-low requests cause throttling / OOM; too-high requests cause underutilization. Use VPA in recommendation mode to find the right values. Cross-reference Issue 06 Chapters V–VII.

QoS class: Guaranteed (requests == limits) for latency-sensitive services. Burstable for general workloads. BestEffort never in prod.

Pod density: more pods per node = better bin-packing but more cgroup pressure and slower kubelet ops. The K8s default of 110/node is a reasonable upper bound; many production deployments run 30-60.

Image pull optimization: registry caching, image streaming (lazy-loading), pre-pulling common base images via DaemonSets. Cross-reference Issue 07 Chapter VII.

Startup & readiness probes: probe period too short = false negatives; period too long = slow detection. Start with periodSeconds: 5 for readiness, periodSeconds: 10 for liveness. Use startup probe for slow-starting apps to avoid false liveness failures during cold start.

Network performance

CNI choice: Cilium (eBPF) is fastest for pod-to-pod and Service routing. Replace kube-proxy with Cilium's eBPF replacement for ~2× throughput on heavy traffic.

conntrack table size: default 65535 — small. Bump net.netfilter.nf_conntrack_max on nodes handling many connections. Cilium with eBPF avoids conntrack entirely for many paths.

MTU: see Chapter XIV. Jumbo frames (MTU 9000) on data-heavy workloads with capable networking.

topologyAware hints: keep traffic in-zone. Annotate Services with service.kubernetes.io/topology-aware-hints: auto; kube-proxy/Cilium prefer same-zone endpoints. Reduces cross-zone bandwidth costs.

Storage performance

Per Issue 06 Chapter VI: IOPS and throughput are provisioned on cloud disks. Provision them. gp3 SSD with 16,000 IOPS, 1000 MB/s throughput, $$ — vs gp2's hidden surprise throttling.

For ML datasets: object storage + caching layer (JuiceFS, Alluxio) outperforms NFS for read-heavy workloads. For shared datasets, mount the cache RWX into many pods.

GPU performance

Cross-reference Issues 02–03 for the deep performance discussion. K8s-specific:

  • Avoid MIG / time-slicing for latency-sensitive inference. Use whole GPUs.
  • For training: gang scheduling, node affinity for NVLink-connected GPUs, dedicated GPU node pools (taints).
  • DCGM metrics gating: alert if DCGM_FI_PROF_SM_ACTIVE < 50% on training pods — they're not using the GPU you paid for.
  • Pre-pull large CUDA images via DaemonSet to avoid 5-minute pulls when scaling.

Scheduling performance

For clusters with thousands of pods scheduled per second:

  • --kube-scheduler-percentage-of-nodes-to-score: by default scheduler scores all nodes; lowering this to 50 (50%) drops scheduling latency significantly.
  • Tune scoring plugins; disable ones you don't need.
  • For batch workloads: use Volcano/Kueue/KAI for higher-throughput batch scheduling than the default scheduler.
The performance hierarchy

At the workload level: get resource requests right (VPA-driven), then probes, then CNI and pod density. At the cluster level: etcd disk performance, apiserver replicas, controller-manager concurrency. The 80/20: right-sizing pods catches 80% of workload performance issues; sizing the control plane catches 80% of cluster-scale issues. Cross-reference Issue 06 entire for the deeper performance methodology.
