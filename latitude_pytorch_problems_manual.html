01How a PyTorch round runs

For an infrastructure role the coding round is rarely an algorithms puzzle. It is usually one of four shapes, and knowing which one you're in tells you what "good" looks like.

ShapeWhat they're grading
"Write a Dataset / DataLoader for this data"Do you handle ragged shapes, sharding, and worker semantics? Do you think about throughput? Part II.
"Here's a training loop — improve it"Can you spot the leak, the missing set_epoch, the wrong accumulation scaling, the un-guarded NaN? Parts III and VI.
"This is slow / OOMs — debug it"Do you diagnose before treating? Run the trees from Volume I aloud.
"Design the system"Storage layout, throughput arithmetic, failure modes, cost. Part VII.
The habit that carries every one of them

State the constraint before you write. "How many samples per second do we need, and how many bytes is a sample?" turns an open-ended coding question into an engineering one, and it is the single most senior-sounding move available. Almost nobody does it.

02The pre-flight checklist

Run this before you say "done"
1 · shapes
Ragged inputs handled? Padding and a mask? Does any downstream reduction respect the mask?
2 · sharding
IterableDataset: sharded across both ranks and workers? Map-style: DistributedSampler with set_epoch?
3 · the leak
Any tensor accumulated across steps without .item() or .detach()?
4 · accumulation
Loss divided by the accumulation count? no_sync() on the non-final micro-batches?
5 · state
Checkpoint includes optimizer and scheduler and scaler — not just weights.
6 · rank guards
Logging/checkpointing on rank 0 only — but no collective inside a rank-0 branch.
7 · determinism
Seeds set, worker_init_fn provided, and num_workers recorded (it changes the RNG stream).
8 · finite guard
Check torch.isfinite(grad_norm) before stepping.
9 · eval mode
model.eval() + torch.no_grad() around validation, and back to train() after.

03P1 · A multi-sensor Dataset

problem 1Write a Dataset for samples of 6 cameras + a lidar sweep + labels.
State the constraint first

"A sample is ~3 MB — six JPEGs plus a sweep. At 100 samples/s per GPU across 8 GPUs that's 2.4 GB/s from storage and roughly 3 CPU cores of decode per GPU. So the design has to avoid small-file access and keep decode cheap." Then write the class.

from dataclasses import dataclass
from typing import Dict, Any
import numpy as np, torch
from torch.utils.data import Dataset

@dataclass
class SampleRef:
    sample_id: str
    shard_uri: str
    offset: int
    num_points: int          # carried in the manifest so we can bucket (P5)

class MultiSensorDataset(Dataset):
    # Manifest-driven: a Parquet index gives random access, while the
    # bytes live in large sequential shards. See Vol I §15.
    def __init__(self, manifest_path, transform=None, cameras=6):
        import pyarrow.parquet as pq
        tbl = pq.read_table(manifest_path,
                            columns=["sample_id", "shard_uri", "offset", "num_points"])
        # NUMPY-BACKED, not a list of python objects: DataLoader workers are
        # forked, and refcounting python objects copies them into every worker.
        self.sample_id = tbl["sample_id"].to_numpy(zero_copy_only=False)
        self.shard_uri = tbl["shard_uri"].to_numpy(zero_copy_only=False)
        self.offset    = tbl["offset"].to_numpy()
        self.num_points = tbl["num_points"].to_numpy()
        self.transform = transform
        self.cameras = cameras
        self._readers: Dict[str, Any] = {}      # lazily opened, per worker

    def __len__(self):
        return len(self.offset)

    def _reader(self, uri):
        # Open lazily INSIDE the worker. File handles do not survive fork
        # cleanly, so never open them in __init__.
        r = self._readers.get(uri)
        if r is None:
            r = self._readers[uri] = open_shard(uri)
        return r

    def __getitem__(self, i):
        rec = self._reader(self.shard_uri[i]).read_at(self.offset[i])

        # Keep images as uint8 all the way to the GPU: 4x less transfer than
        # float32. Normalize on device (Vol I §05).
        imgs = torch.from_numpy(
            np.stack([decode_jpeg(rec[f"cam{c}"]) for c in range(self.cameras)])
        )                                        # [6, H, W, 3] uint8

        pts = torch.from_numpy(rec["lidar"]).float()      # [N, 4] ragged
        target = torch.from_numpy(rec["boxes"]).float()   # [M, 7] ragged

        sample = {
            "sample_id": str(self.sample_id[i]),  # carried through for eval dedup (P15)
            "img": imgs,
            "pts": pts,
            "target": target,
            "calib": torch.from_numpy(rec["calib"]).float(),
        }
        return self.transform(sample) if self.transform else sample
DecisionWhy, out loud
Manifest + shardsRandom access for sampling, sequential reads for throughput. Millions of small objects would be request-rate-bound, not bandwidth-bound.
Numpy-backed indexVerified concern: DataLoader workers are forked, and refcount touches on Python objects trigger copy-on-write duplication into every worker. A numpy array is one buffer that stays shared.
Lazy readersFile/S3 handles opened in __init__ don't survive forking cleanly. Open on first use inside the worker.
uint8 to the GPU4× less host-to-device traffic; cast and normalize on device.
sample_id in the batchCosts nothing and makes correct distributed evaluation possible (P15). Add it before you need it.

04P2 · collate_fn for ragged data

problem 2Batch samples whose point clouds and box counts differ.
Two valid answers — know both

Pad + mask is simplest and keeps a dense [B, N, C] layout. Concatenate + batch index wastes no memory and is what sparse/scatter ops want. Offer both and say when each wins.

def collate_pad(batch):
    # Verified: mask.sum(1) equals the true lengths, the padded region is
    # exactly zero, and fixed-size modalities still stack normally.
    lens = torch.tensor([b["pts"].shape[0] for b in batch])
    max_n, C = int(lens.max()), batch[0]["pts"].shape[1]

    pts  = torch.zeros(len(batch), max_n, C)
    mask = torch.zeros(len(batch), max_n, dtype=torch.bool)
    for i, b in enumerate(batch):
        n = b["pts"].shape[0]
        pts[i, :n], mask[i, :n] = b["pts"], True

    return {
        "sample_id": [b["sample_id"] for b in batch],   # keep as a list
        "img":  torch.stack([b["img"] for b in batch]), # fixed shape
        "pts":  pts, "mask": mask, "lens": lens,
        "target": [b["target"] for b in batch],         # ragged, keep listed
    }

def collate_concat(batch):
    # No padding at all: one flat point tensor plus the owning batch index.
    # This is what production lidar pipelines usually do, and it is the
    # layout scatter/segment ops want.
    pts = torch.cat([b["pts"] for b in batch], dim=0)                # [sum_N, C]
    bidx = torch.cat([torch.full((b["pts"].shape[0],), i, dtype=torch.long)
                      for i, b in enumerate(batch)])                 # [sum_N]
    return {"pts": pts, "batch_idx": bidx,
            "img": torch.stack([b["img"] for b in batch])}

# THE CONSEQUENCE OF PADDING, verified to differ:
#   masked_mean = (pts * mask.unsqueeze(-1)).sum(1) / lens.unsqueeze(1)   correct
#   naive_mean  = pts.mean(1)                                            WRONG
# The naive version averages the zero padding in, so the answer depends on
# whichever sample in the batch happened to be longest.
The follow-up to expect

"What's the memory cost of padding?" With point counts uniform in [50, 1000] and random batching, measured padding waste was 44.8% — nearly half the tensor is zeros. That number is the motivation for P5's bucketing, which cut it to 2.0%. Having a real figure rather than "some overhead" is what makes the answer land.

05P3 · Sharded IterableDataset

problem 3Stream from sharded object storage. Don't duplicate anything.
The measured failure

Executed: a naive 100-element IterableDataset with num_workers=4 yielded 400 items — every worker replayed the whole dataset. Nothing errors. The loss curve looks fine. The model just silently sees each sample four times per epoch. Add distributed ranks and it multiplies again.

import torch.distributed as dist
from torch.utils.data import IterableDataset, get_worker_info

class ShardedStream(IterableDataset):
    # Shards across BOTH axes: distributed rank and DataLoader worker.
    # Verified: yields each item exactly once (naive yields it num_workers times).
    def __init__(self, shard_uris, shuffle_buffer=1000, seed=0):
        self.shards = sorted(shard_uris)      # sorted => deterministic assignment
        self.buf_size = shuffle_buffer
        self.seed = seed
        self.epoch = 0

    def set_epoch(self, e):
        self.epoch = e                        # call this, like DistributedSampler

    def _my_shards(self):
        rank, world = (dist.get_rank(), dist.get_world_size()) \
            if dist.is_available() and dist.is_initialized() else (0, 1)
        info = get_worker_info()
        wid, nw = (info.id, info.num_workers) if info else (0, 1)

        shard_id, num_shards = rank * nw + wid, world * nw
        shards = list(self.shards)
        random.Random(self.seed + self.epoch).shuffle(shards)   # shuffle level 1
        return shards[shard_id::num_shards]

    def __iter__(self):
        rng = random.Random(self.seed + self.epoch)
        buf = []
        for uri in self._my_shards():
            for rec in read_shard_stream(uri):
                buf.append(rec)
                if len(buf) >= self.buf_size:                    # shuffle level 2
                    j = rng.randrange(len(buf))
                    buf[j], buf[-1] = buf[-1], buf[j]
                    yield buf.pop()
        rng.shuffle(buf)
        yield from buf
Three constraints to state

Shard count must exceed world_size × num_workers, ideally by a healthy multiple, or some workers idle. Shard granularity is a dataset-build decision that constrains your training topology.

Uneven shards cause hangs, not just imbalance: a rank that runs out of data stops participating in collectives and the job deadlocks (P17, D4).

Two levels of shuffling. Shard order per epoch and a shuffle buffer within a shard — because consecutive AV frames are near-duplicates. Verified: the buffer preserves every element exactly once and does reorder, but displacement is bounded by the buffer size, so it is not a global shuffle. Say that limitation out loud.

06P4 · A scenario-balanced sampler

problem 4Rare events are 0.1% of the data. Oversample them, reproducibly, in distributed training.
Why this is a real AV question

Petabytes of highway driving; the frames that matter are cut-ins, construction, night rain. Sampling strategy is the pipeline, and a custom Sampler is the hook.

class DistributedWeightedSampler(Sampler):
    # Weighted oversampling that is (a) reproducible per epoch and
    # (b) correctly partitioned across ranks.
    def __init__(self, weights, num_replicas, rank, num_samples=None, seed=0):
        self.weights = torch.as_tensor(weights, dtype=torch.double)
        self.num_replicas, self.rank, self.seed = num_replicas, rank, seed
        total = num_samples or len(self.weights)
        # round UP so every rank gets the same count -- unequal counts hang
        self.per_rank = math.ceil(total / num_replicas)
        self.total = self.per_rank * num_replicas
        self.epoch = 0

    def set_epoch(self, epoch):
        self.epoch = epoch

    def __iter__(self):
        # ONE generator seeded identically on every rank => all ranks draw the
        # SAME global sequence, then each slices its own disjoint stride.
        # (Sampling independently per rank would give overlapping draws.)
        g = torch.Generator()
        g.manual_seed(self.seed + self.epoch)
        idx = torch.multinomial(self.weights, self.total,
                                replacement=True, generator=g)
        return iter(idx[self.rank::self.num_replicas].tolist())

    def __len__(self):
        return self.per_rank

# Build the weights from scenario tags in the manifest:
#   w_i = 1 / count(tag_of_i)  -> equalizes tags
#   or a capped boost so rare tags are oversampled but not to absurdity
The two subtleties worth voicing

Seed identically, slice disjointly. Every rank must generate the same global draw and then take its own stride. If each rank samples independently, ranks overlap and the effective epoch composition is not what you designed.

Oversampling changes your epoch semantics. With replacement, "an epoch" no longer means "each sample once." Be explicit about whether you're comparing runs at equal steps or equal unique samples — otherwise ablations are meaningless.

07P5 · Length bucketing

problem 5Cut the padding waste from P2 without losing shuffling.
The measured result

Executed on 2000 samples with lengths uniform in [50, 1000], batch size 16: random batching wasted 44.8% of the padded tensor. Pool-and-sort bucketing cut that to 2.0% — a ~22× reduction in wasted memory and compute, while still covering every sample exactly once and still reshuffling per epoch.

class BucketBatchSampler(Sampler):
    # Shuffle globally, sort WITHIN a pool, form batches, then shuffle the
    # batch ORDER. Keeps randomness while grouping similar lengths together.
    # Verified: covers every sample once; waste 44.8% -> 2.0%.
    def __init__(self, lengths, batch_size, pool_mult=50, drop_last=False, seed=0):
        self.lengths, self.bs = list(lengths), batch_size
        self.pool = batch_size * pool_mult
        self.drop_last, self.seed, self.epoch = drop_last, seed, 0

    def set_epoch(self, e): self.epoch = e

    def __iter__(self):
        idx = list(range(len(self.lengths)))
        random.Random(self.seed + self.epoch).shuffle(idx)
        batches = []
        for i in range(0, len(idx), self.pool):
            pool = sorted(idx[i:i + self.pool], key=lambda j: self.lengths[j])
            for b in range(0, len(pool), self.bs):
                chunk = pool[b:b + self.bs]
                if len(chunk) == self.bs or not self.drop_last:
                    batches.append(chunk)
        random.Random(self.seed + self.epoch + 1).shuffle(batches)
        return iter(batches)

    def __len__(self):
        n = len(self.lengths)
        return n // self.bs if self.drop_last else (n + self.bs - 1) // self.bs

# loader = DataLoader(ds, batch_sampler=BucketBatchSampler(lengths, 16), ...)
# NOTE: batch_sampler is mutually exclusive with batch_size/shuffle/sampler.

The bonus argument, and it's the one that connects to performance: bucketing also stabilizes shapes. Fewer distinct max_n values means fewer recompilations under torch.compile and less allocator fragmentation (Vol. I §24, §40). Round the bucket maximum to a fixed set of sizes and you get a small number of static shapes — turning a data-layout choice into a compile-time and memory win. That cross-layer connection is exactly the kind of thing this team is hiring for.

08P6 · Streaming with a shuffle buffer

problem 6You can't random-access the stream. How do you shuffle?
The mechanism and its honest limit

Fill a buffer of size K, then repeatedly emit a random element and refill from the stream. Verified: every element is preserved exactly once and the order genuinely changes — but an element can only move within roughly K positions of where it arrived, so this is local shuffling. With temporally correlated AV frames, that matters: pair it with shard-level shuffling so the correlation is broken at the coarse scale too.

def shuffle_buffer(stream, size, seed=0):
    # Verified: preserves every element exactly once, reorders, and bounds
    # displacement by ~size (a real limitation, not a bug).
    rng = random.Random(seed)
    buf = []
    for item in stream:
        buf.append(item)
        if len(buf) >= size:
            j = rng.randrange(len(buf))
            buf[j], buf[-1] = buf[-1], buf[j]   # swap-to-end, then pop: O(1)
            yield buf.pop()
    rng.shuffle(buf)                            # drain the tail
    yield from buf

# Sizing: buffer_size * sample_size * num_workers must fit in host RAM.
#   1000 samples x 3 MB x 8 workers = 24 GB -- too big.
#   Buffer the RECORDS (compressed bytes), not decoded tensors, and the
#   same buffer costs ~10x less. Decode after the shuffle, not before.

09P7 · Instrument the input path

problem 7Prove whether the job is input-bound, in under five minutes.
The two-experiment diagnosis
import time, itertools, contextlib

def bench_loader(loader, n=200, warmup=20):
    # EXPERIMENT 1: how fast can the pipeline deliver, with no model at all?
    it = iter(loader)
    for _ in range(warmup): next(it)
    t0, count = time.perf_counter(), 0
    for _ in range(n):
        b = next(it)
        count += b["img"].shape[0]
    dt = time.perf_counter() - t0
    return count / dt                          # samples/sec available

def bench_model(step_fn, batch, n=50, warmup=10):
    # EXPERIMENT 2: how fast is the model, fed from a CACHED batch?
    for _ in range(warmup): step_fn(batch)
    torch.cuda.synchronize()
    t0 = time.perf_counter()
    for _ in range(n): step_fn(batch)
    torch.cuda.synchronize()
    return n * batch["img"].shape[0] / (time.perf_counter() - t0)

data_rate  = bench_loader(loader)              # samples/sec the pipeline gives
model_rate = bench_model(train_step, cached)   # samples/sec the GPU can eat

print(f"pipeline {data_rate:8.1f} samples/s")
print(f"model    {model_rate:8.1f} samples/s")
print("INPUT BOUND" if data_rate < model_rate else "COMPUTE BOUND")
print(f"ceiling  {min(data_rate, model_rate):8.1f} samples/s")

# Then break the pipeline number down -- where in __getitem__ does it go?
class Timed:
    def __init__(self): self.t = {}
    @contextlib.contextmanager
    def stage(self, name):
        t0 = time.perf_counter(); yield
        self.t[name] = self.t.get(name, 0.0) + time.perf_counter() - t0
# usage inside __getitem__:
#   with timer.stage("read"):   rec = reader.read_at(off)
#   with timer.stage("decode"): imgs = decode(rec)
#   with timer.stage("augment"):sample = transform(sample)
What the numbers tell you to do

If data_rate ≪ model_rate, every GPU-side optimization is wasted effort until the pipeline is fixed — and the per-stage breakdown says which fix: read time means storage layout (shard bigger, cache locally), decode time means format (pre-resize, GPU decode), augment time means moving work to the GPU. If data_rate ≫ model_rate, go to Volume I's throughput checklist. Reporting both numbers and the resulting ceiling is a complete answer to "is this slow?"

10P8 · A production training loop

problem 8Write the training loop you'd actually ship.
Every line below earns its place — be ready to justify each one
def train(cfg, model, train_loader, val_loader, opt, sched, scaler, device):
    is_dist = dist.is_available() and dist.is_initialized()
    is_main = (not is_dist) or dist.get_rank() == 0
    step = start_step

    for epoch in range(start_epoch, cfg.epochs):
        model.train()
        # REQUIRED, and silently wrong without it: reshuffles per epoch.
        if hasattr(train_loader.sampler, "set_epoch"):
            train_loader.sampler.set_epoch(epoch)

        for i, batch in enumerate(train_loader):
            batch = to_device(batch, device, non_blocking=True)  # needs pin_memory
            sync = ((i + 1) % cfg.accum == 0)

            # skip the all-reduce on non-final micro-batches, or you pay
            # accum x the gradient communication for no benefit
            ctx = contextlib.nullcontext() if (sync or not is_dist) else model.no_sync()
            with ctx:
                with torch.autocast("cuda", dtype=torch.bfloat16, enabled=cfg.amp):
                    loss = criterion(model(batch), batch["target"])
                scaler.scale(loss / cfg.accum).backward()   # the /accum is required

            if not sync:
                continue

            scaler.unscale_(opt)                            # BEFORE clipping
            gnorm = torch.nn.utils.clip_grad_norm_(model.parameters(), cfg.clip)
            if torch.isfinite(gnorm):                       # clip does NOT sanitise
                scaler.step(opt)
            else:
                log.warning("step %d: non-finite grad norm, skipped", step)
            scaler.update()
            opt.zero_grad(set_to_none=True)
            sched.step()                                    # per OPTIMIZER step
            step += 1

            if is_main and step % cfg.log_every == 0:
                # .item() detaches AND syncs -- fine here, fatal in the inner loop
                log.info("step %d loss %.4f lr %.2e gnorm %.2f mem %.1fGB",
                         step, loss.item(), sched.get_last_lr()[0], gnorm.item(),
                         torch.cuda.max_memory_allocated() / 1e9)

            if step % cfg.ckpt_every == 0:
                if is_dist: dist.barrier()          # barrier OUTSIDE the rank guard
                if is_main: save_checkpoint(...)

        metrics = evaluate(model, val_loader, device, is_dist)   # P15
        if is_main:
            log.info("epoch %d %s", epoch, metrics)
LineWhy it's there
sampler.set_epoch(epoch)Without it every epoch replays the same permutation. Silent, and it degrades training.
model.no_sync()Gradient accumulation without it multiplies communication by the accumulation factor.
loss / cfg.accumVerified: omitting it scales gradients by exactly the accumulation count.
unscale_ then clipClipping scaled gradients clips the wrong magnitude.
isfinite(gnorm)Verified: clip_grad_norm_ returns nan and does not clean the gradients. Stepping on NaN poisons the weights permanently.
set_to_none=TrueFrees gradient memory; one fewer kernel per parameter.
barrier outside the rank guardA collective inside if is_main: deadlocks every other rank — a classic hang (D4).
max_memory_allocated in the logMakes a memory regression visible on the day it lands, not the week it OOMs.

11P9 · Accumulation + AMP + clipping

problem 9"We doubled the accumulation steps and training got worse. Why?"
Four candidate causes, in order of likelihood

1. The LR schedule ran twice as fast in wall-clock terms — if sched.step() is called per micro-batch instead of per optimizer step, doubling accumulation halves your actual schedule length. 2. The effective batch doubled without an LR change (Vol. I §21): larger batch needs a larger LR and warmup. 3. Missing loss/accum — verified to scale gradients by exactly the accumulation count, which looks like a badly tuned LR. 4. Warmup is counted in optimizer steps, so doubling accumulation doubles the wall-clock warmup and everything shifts.

The verified anchor

Four micro-batches of 16 with loss/4 reproduced a single batch of 64 to 3.0e-08. Accumulation is exact when the scaling is right — so if results changed, the scaling or the schedule changed, not the mathematics.

12P10 · Checkpoint and resume

problem 10Write checkpointing such that a preempted job resumes exactly.
The measured proof

Executed: 20 steps straight through, versus 10 steps → checkpoint → resume. With model + optimizer + scheduler state, the resumed losses matched the uninterrupted run exactly (difference 0.0). With model weights only and a fresh optimizer, they diverged immediately (1.8e-01). AdamW's moments and the schedule position are training state, not decoration.

def save_checkpoint(path, *, model, opt, sched, scaler, step, epoch, cfg, metrics):
    core = model.module if hasattr(model, "module") else model   # strip DDP prefix
    ckpt = {
        "model": core.state_dict(), "opt": opt.state_dict(),
        "sched": sched.state_dict(), "scaler": scaler.state_dict(),
        "step": step, "epoch": epoch,
        "cfg": OmegaConf.to_container(cfg, resolve=True),   # RESOLVED, embedded
        "data_version": cfg.data_version, "git_sha": git_sha(),
        "torch": torch.__version__,
        "world_size": dist.get_world_size() if dist.is_initialized() else 1,
        "num_workers": cfg.num_workers,     # part of the RNG stream (P11)
        "rng": {"torch": torch.get_rng_state(),
                "cuda": torch.cuda.get_rng_state_all(),
                "numpy": np.random.get_state(), "python": random.getstate()},
        "metrics": metrics,
    }
    tmp = f"{path}.tmp"
    torch.save(ckpt, tmp)
    os.replace(tmp, path)            # ATOMIC: a crash mid-write never truncates

def load_checkpoint(path, *, model, opt, sched, scaler, strict=True):
    ck = torch.load(path, map_location="cpu", weights_only=False)
    core = model.module if hasattr(model, "module") else model
    missing, unexpected = core.load_state_dict(ck["model"], strict=strict)
    opt.load_state_dict(ck["opt"]); sched.load_state_dict(ck["sched"])
    scaler.load_state_dict(ck["scaler"])
    torch.set_rng_state(ck["rng"]["torch"]); random.setstate(ck["rng"]["python"])
    np.random.set_state(ck["rng"]["numpy"])
    if torch.cuda.is_available():
        torch.cuda.set_rng_state_all(ck["rng"]["cuda"])
    # WARN LOUDLY if the environment differs -- resume will not be exact
    if ck["world_size"] != (dist.get_world_size() if dist.is_initialized() else 1):
        log.warning("resuming at a DIFFERENT world size: effective batch changed")
    if ck.get("num_workers") != cfg.num_workers:
        log.warning("num_workers changed: the data RNG stream will differ")
    return ck["step"], ck["epoch"]
The part most answers miss

Resuming mid-epoch is genuinely hard: you must either fast-forward the sampler (expensive) or accept restarting the epoch (simplest, and usually fine). Say which you'd choose and why. Checkpointing on a step cadence rather than an epoch cadence makes the choice cheap, because you lose at most one checkpoint interval either way.

13P11 · Deterministic seeding

problem 11Make two runs of the same config produce the same result.
The verified footgun

Same seed and same worker count reproduced the batch stream exactly. Changing num_workers from 2 to 4 changed the stream — each worker derives its seed from the base seed and its own id, so the worker count is effectively part of your seed. "I tuned the DataLoader for throughput" silently changes your augmentation sequence, and two runs with different worker counts are not comparable.

def seed_everything(seed: int):
    random.seed(seed); np.random.seed(seed)
    torch.manual_seed(seed); torch.cuda.manual_seed_all(seed)

def seed_worker(worker_id: int):
    s = torch.initial_seed() % 2**32       # torch already derived a per-worker seed
    np.random.seed(s); random.seed(s)      # propagate it to numpy and python

g = torch.Generator(); g.manual_seed(cfg.seed)

                    num_workers=cfg.num_workers, ...)

14P12 · Streaming metrics

problem 12Accumulate a metric over an epoch without leaking memory or lying.
Two bugs in one problem

The leak: total += loss retains the autograd graph for every step. The lie: averaging per-batch means is wrong when batches differ in size. Verified: three batches of sizes 3, 7, 1 gave mean-of-means 2.667 against a true mean of 2.000. Sum-and-count is exact.

class StreamingMean:
    # Sum + count, not mean-of-means. Verified exact for unequal batch sizes.
    # Keeps a device tensor so there is no per-step host sync.
    def __init__(self, device):
        self.total = torch.zeros((), dtype=torch.float64, device=device)
        self.count = torch.zeros((), dtype=torch.float64, device=device)

    @torch.no_grad()
    def update(self, value: torch.Tensor, n: int):
        self.total += value.detach().double() * n     # .detach() => no graph
        self.count += n

    def compute(self, distributed=False):
        if distributed and dist.is_initialized():
            buf = torch.stack([self.total, self.count])
            dist.all_reduce(buf, op=dist.ReduceOp.SUM)   # reduce SUM and COUNT
            return (buf[0] / buf[1]).item()              # ...then divide. Order matters.
        return (self.total / self.count).item()

The distributed subtlety, and it's the same bug one level up: all-reduce the sum and the count separately, then divide. Averaging each rank's local mean is only correct when every rank saw exactly the same number of samples — which is precisely what DistributedSampler padding and uneven shards break. Reducing (sum, count) as a pair is correct unconditionally, and it costs one extra float.

15P13 · EMA of weights

problem 13Maintain an exponential moving average of the model for evaluation.
Verified behaviour

Starting from zeros and stepping toward ones with decay 0.9, one update produced exactly 0.1 (the 1 − decay step), and ten updates reached 0.651 — converging as expected. The shadow buffers do not alias the live parameters.

class ModelEMA:
    # Shadow weights updated as: shadow = decay*shadow + (1-decay)*live
    def __init__(self, model, decay=0.9999, device=None):
        self.decay = decay
        core = model.module if hasattr(model, "module") else model
        self.shadow = {k: v.detach().clone().float().to(device or v.device)
                       for k, v in core.state_dict().items()
                       if v.dtype.is_floating_point}   # skip int buffers
        self.n = 0

    @torch.no_grad()
    def update(self, model):
        self.n += 1
        # warmup: early on, trust the live weights more, or EMA lags badly
        d = min(self.decay, (1 + self.n) / (10 + self.n))
        core = model.module if hasattr(model, "module") else model
        for k, v in core.state_dict().items():
            if k in self.shadow:
                self.shadow[k].mul_(d).add_(v.detach().float(), alpha=1 - d)

    def state_dict(self):  return {"shadow": self.shadow, "n": self.n}
    def load_state_dict(self, sd):
        self.shadow, self.n = sd["shadow"], sd["n"]

    @torch.no_grad()
    def copy_to(self, model):
        core = model.module if hasattr(model, "module") else model
        sd = core.state_dict()
        for k in self.shadow:
            sd[k].copy_(self.shadow[k].to(sd[k].dtype))
Three things to mention

Checkpoint it. The EMA is training state — resume without it and you restart the average from scratch, losing everything it had accumulated.

Only rank 0 needs it in DDP, since replicas are identical — but keep the update on all ranks if that keeps the code simpler and branch-free; the cost is small and rank-divergent code paths are how hangs happen.

Buffers are the trap. BatchNorm running statistics are float buffers, so they get EMA'd too — which is usually what you want. Integer buffers (like num_batches_tracked) must be excluded, which is what the dtype.is_floating_point filter does.

The four setup mistakes, ranked by how confusing they are

1. Missing set_device → every rank allocates on GPU 0 → OOM that looks like a memory bug. 2. Missing drop_last → unequal batch counts → hang at the end of the epoch. 3. Same seed on every rank → identical augmentation, so you've reduced your effective data diversity without noticing. 4. 17P15 · Eval without double counting

problem 15Compute a validation metric across ranks. Correctly.
The bug, measured

DistributedSampler pads the index list up to a multiple of the world size by wrapping the first indices onto the trailing ranks. Verified with 10 samples across 4 ranks: 12 indices dispatched, and the naive gathered mean came out 3.833 against a true value of 4.500. Those duplicated samples are counted twice. On a small validation set, or a rare-event slice, this visibly moves the number — and nothing warns you.

@torch.no_grad()
def evaluate(model, loader, device, is_dist):
    model.eval()
    ids, preds, targets = [], [], []

    for batch in loader:
        batch = to_device(batch, device, non_blocking=True)
        with torch.autocast("cuda", dtype=torch.bfloat16):
            out = model(batch)
        ids.extend(batch["sample_id"])          # <-- carried from P1, this is the fix
        preds.append(out.float().cpu())
        targets.append(batch["target"].float().cpu())

    preds = torch.cat(preds); targets = torch.cat(targets)

    if is_dist:
        # gather variable-length results from every rank
        gathered = [None] * dist.get_world_size()
        dist.all_gather_object(gathered, (ids, preds, targets))
        ids     = [i for g in gathered for i in g[0]]
        preds   = torch.cat([g[1] for g in gathered])
        targets = torch.cat([g[2] for g in gathered])

        # *** DEDUPLICATE BY SAMPLE ID ***  verified to recover the true mean
        seen, keep = set(), []
        for k, sid in enumerate(ids):
            if sid not in seen:
                seen.add(sid); keep.append(k)
        idx = torch.tensor(keep)
        preds, targets = preds[idx], targets[idx]

    model.train()
    return compute_metrics(preds, targets)
Alternative fixTrade-off
Dedup by sample ID (above)Always correct, needs IDs carried through the batch. The recommended default — and it's why P1 puts sample_id in every sample.
drop_last=True on the eval samplerVerified to make ranks truly disjoint — but it silently drops up to world_size − 1 samples. Wrong for a small or precious validation set.
Reduce (sum, count) per rankCorrect for simple averages, and cheap. Doesn't work for metrics that aren't decomposable — mAP needs the full prediction set.
Evaluate on one rankTrivially correct, and slow. Fine for a small validation set; wasteful for a large one.

Raising this unprompted is one of the strongest moves available in this interview. It's a real bug, it is present in an enormous amount of production training code, and most engineers confidently believe DistributedSampler partitions disjointly. Being able to say "it pads by wrapping the first indices onto the trailing ranks, so distributed eval double-counts unless you deduplicate" — with the mechanism, not just the folklore — demonstrates exactly the depth this team needs.

18P16 · Reduce a custom metric

# Decomposable metrics (mean, accuracy, IoU numerator/denominator) reduce
# cheaply. Reduce the COMPONENTS, then combine -- never reduce the ratio.
@torch.no_grad()
def reduce_confusion(conf: torch.Tensor) -> torch.Tensor:
    # conf: [C, C] integer confusion matrix on this rank
    if dist.is_initialized():
        dist.all_reduce(conf, op=dist.ReduceOp.SUM)     # integers: EXACT, order-free
    return conf

def iou_from_confusion(conf):
    tp = conf.diag().float()
    fp = conf.sum(0).float() - tp
    fn = conf.sum(1).float() - tp
    return tp / (tp + fp + fn).clamp(min=1)             # per-class IoU

# WHY THIS SHAPE: all-reducing per-rank IoU and averaging is WRONG whenever
# class frequencies differ across ranks -- which they always do. Reducing the
# confusion matrix and computing the ratio afterwards is exact.
# BONUS: integer all-reduce is deterministic regardless of arrival order,
# whereas float all-reduce is not.

19P17 · Uneven data, clean epoch end

problem 17Ranks have different numbers of batches. The job hangs. Fix it three ways.
Why it hangs

A rank that exhausts its data leaves the loop and stops issuing collectives. Every other rank blocks forever at the next all-reduce. Nothing errors — the job just stops making progress and burns cluster time until someone notices.

# FIX 1 -- drop_last everywhere (simplest, loses a few samples)
sampler = DistributedSampler(ds, drop_last=True)
loader  = DataLoader(ds, sampler=sampler, drop_last=True, ...)

# FIX 2 -- agree a global step count up front (best for streaming data)
local_batches  = torch.tensor([len(loader)], device=device)
dist.all_reduce(local_batches, op=dist.ReduceOp.MIN)   # everyone stops together
steps_per_epoch = int(local_batches.item())
for i, batch in enumerate(loader):
    if i >= steps_per_epoch:
        break

# FIX 3 -- DDP's join context (handles genuinely uneven inputs)
from torch.distributed.algorithms.join import Join
with Join([model]):
    for batch in loader:
        loss = criterion(model(batch), batch["target"])
        loss.backward(); opt.step(); opt.zero_grad(set_to_none=True)
# Join makes finished ranks keep participating in collectives with dummy
# contributions until everyone is done.

# AND ALWAYS, so a hang is loud instead of silent:
dist.init_process_group("nccl", timeout=datetime.timedelta(minutes=30))

The two traps with compile on this workload

Recompilation on ragged shapes. Every distinct input shape triggers a recompile, and variable point counts mean many shapes. Pad to bucketed sizes in collate (P5) so the shape set is small and fixed — this is where the data pipeline and compile performance turn out to be the same problem.

Measuring during warmup. The first steps include compilation, which can be tens of seconds. Benchmark after warmup or you'll conclude compile made things slower. Verified separately: a compiled model reproduces eager output exactly, so correctness isn't the concern — measurement discipline is.

24P22 · Overlap H2D with compute

# pin_memory=True in the DataLoader is REQUIRED -- without page-locked host
# memory, non_blocking=True is a promise the driver cannot keep.
def to_device(batch, device, non_blocking=True):
    if torch.is_tensor(batch):
        return batch.to(device, non_blocking=non_blocking)
    if isinstance(batch, dict):
        return {k: to_device(v, device, non_blocking) for k, v in batch.items()}
    if isinstance(batch, (list, tuple)):
        return type(batch)(to_device(v, device, non_blocking) for v in batch)
    return batch                                    # strings, ints pass through

# A prefetcher that copies batch N+1 on a side stream while batch N computes.
class Prefetcher:
    def __init__(self, loader, device):
        self.it, self.device = iter(loader), device
        self.stream = torch.cuda.Stream()
        self._preload()

    def _preload(self):
        try:
            self.next = next(self.it)
        except StopIteration:
            self.next = None; return
        with torch.cuda.stream(self.stream):
            self.next = to_device(self.next, self.device, non_blocking=True)

    def __iter__(self):
        while self.next is not None:
            # make the default stream wait for the copy to land
            torch.cuda.current_stream().wait_stream(self.stream)
            batch = self.next
            # keep the tensors alive until the copy stream is done with them
            for v in batch.values():
                if torch.is_tensor(v):
                    v.record_stream(torch.cuda.current_stream())
            self._preload()
            yield batch
Be honest about when this is worth it

An explicit prefetcher buys real overlap, but the DataLoader's own worker prefetching already hides most of it. Measure first — if the timeline shows the H2D copy is not on the critical path, this is complexity for nothing. And the record_stream call is not optional: without it the allocator can hand a buffer to someone else while the copy stream is still reading it, which is a silent corruption bug rather than an error.

25D1 · Find the bug ×6

Six snippets, each with one defect of a different class. Find it, and say how you'd have caught it.

# ===================== BUG 1 =====================
for epoch in range(epochs):
    for batch in loader:                 # loader uses a DistributedSampler

bug 1Missing sampler.set_epoch(epoch)
The defect

Without it, the sampler's shuffle seed never changes, so every epoch replays the identical permutation. Verified: set_epoch(0) and set_epoch(1) produce different orders; the same epoch reproduces the same one.

Why it's vicious

Nothing errors. Training "works," converges worse than it should, and no one can explain the gap. It's invisible unless you look for it.

How to catch it

A unit test that asserts two epochs produce different index sequences. Three lines, catches it forever.

# ===================== BUG 2 =====================
total_loss = 0

The defect

total_loss += loss retains the entire autograd graph for every step. Memory grows monotonically until the job OOMs, with a traceback pointing somewhere innocent.

The fix

total_loss += loss.item() (or .detach()). And the second, subtler bug: dividing by len(loader) is a mean-of-means, which is wrong for unequal batch sizes — verified to give 2.667 against a true 2.000. Use sum-and-count (P12).

How to catch it

# ===================== BUG 3 =====================
class Stream(IterableDataset):

The defect

Every worker iterates the whole file list. Verified with 4 workers on 100 items: 400 items emitted. With 8 workers and 8 ranks it's 64× duplication.

Why it's the worst bug in this list

It doesn't error, doesn't leak, doesn't slow anything down. The loss curve looks normal. You just trained on 1/64th of your data, repeated — and you'll only discover it when the model underperforms for reasons nobody can find.

The fix

Shard on both axes: rank * num_workers + worker_id, stride world_size * num_workers (P3).

How to catch it

A test that collects the loader's output with num_workers ∈ {0, 2, 4} and asserts the multiset is identical. Would have caught it instantly.

The defect

The second defect

Under DDP, every backward() triggers a full gradient all-reduce. Four micro-batches means 4× the communication for one optimizer step. model.no_sync() on the first three fixes it.

The defect

Rank 0 enters barrier() and waits for everyone. Nobody else ever calls it. Permanent deadlock, no error, and the job silently burns cluster hours.

The fix

Barrier outside the guard, side effect inside it (P18). And save model.module.state_dict() under DDP, or every key carries a module. prefix that breaks single-GPU loading.

How to catch it

A collective timeout turns a silent hang into an attributed failure. py-spy dump on each rank finds it in seconds — the rank with a different stack is the answer.

# ===================== BUG 6 =====================

        preds.append(model(batch))
    return compute_metric(torch.cat(preds), targets)
bug 6No eval(), no no_grad(), and it never goes back
Three defects

1. No model.eval() — BatchNorm keeps updating its running statistics from validation data and Dropout stays active, so the metric is wrong and validation silently corrupts the model. 2. No torch.no_grad() — the full autograd graph for the whole validation set is retained, which OOMs. 3. Even fixed, it never calls model.train() afterwards, so the next epoch trains in eval mode.

The fix

Grows across epochs, not stepsA new DataLoader/iterator per epoch holding worker processes, or per-epoch objects retaining CUDA tensors. persistent_workers interacts here.
Stop guessing

torch.cuda.memory._record_memory_history() then _dump_snapshot(), viewed in the memory visualizer: the surviving allocations carry their own allocation stacks, so the leak names itself. Two minutes, and it replaces an afternoon of hypotheses.

28D4 · The distributed hang

drill"The job stopped progressing at step 40,000. No error. GPUs at 100%."
The investigation, in order
  • py-spy dump --pid on every rank. No instrumentation needed, works on a running job. The rank whose Python stack differs is almost always the culprit. This is the first move, every time.
  • Check the collective sequence. Every rank must issue the same collectives, in the same order, with the same shapes. Divergence is the universal cause.
  • Rank-conditional code? A collective inside if rank == 0: (D1 bug 5), or an early continue/break on a bad batch that only one rank hit.
  • Uneven data? One rank exhausted its shard and left the loop (P17). At step 40,000 specifically, suspect the end of an epoch or an unbalanced shard.
  • DDP unused parameters? A conditional branch in the model means some rank's bucket never completes, so it never launches its all-reduce.
  • A rank died? Check for an OOM-kill in dmesg or the pod events. The survivors block forever on a peer that no longer exists.
  • Then harden. A collective timeout plus async error handling converts every future occurrence from a silent stall into a loud, attributed failure. That change is the actual deliverable — the fix is the prevention.

29D5 · The irreproducible run

drill"I reran the exact same config and got a different number."
Ask the meta-question first

"Do you need bitwise determinism, or statistical reproducibility?" The first costs real performance; the second usually just needs seeds, a pinned dataset, and a pinned environment. Most people asking for the first actually want the second — and establishing that reframes the whole conversation.

CheckDetail
Seedspython, numpy, torch, cuda — plus the DataLoader generator and worker_init_fn.
num_workersVerified: the worker count is part of the RNG stream. Two runs with different worker counts are not comparable. Almost nobody knows this, and raising it unprompted is a strong signal.
set_epochMissing it produces a different bug with the same smell — same order every epoch.
Dataset versionDid an upstream job re-materialize a partition underneath you? This is why data versioning is a correctness tool (Vol. I §33).
EnvironmentContainer digest, not tag. Library versions. GPU model. World size — which changes the effective batch and so genuinely changes training.
Nondeterministic opstorch.use_deterministic_algorithms(True) raises and names the op. Atomics-based ops (scatter_add, index_add, embedding backward) sum in arbitrary order. cudnn.benchmark=True picks algorithms by timing.

Bitwise or statistical? Answer that before you pay for either.

Part VII — System design

Draw the whole thing.

Four design prompts, each with the structure of a strong answer. The pattern is always the same: clarify the requirements, do the arithmetic, propose a design, name the failure modes, state what you'd measure. Skipping the arithmetic is the most common way these go badly.

30S1 · The training data pipeline

design"Design the pipeline from vehicle logs to trained model."
1 · Clarify

Fleet size and data rate per vehicle? Sensor set? How much is retained versus discarded? How many models train concurrently, on how many GPUs? What's the current experiment turnaround time — and what would good look like?

2 · Do the arithmetic, out loud

"A sample is ~3 MB. Eight GPUs at 100 samples/s each needs 2.4 GB/s sustained, and roughly 3 CPU cores of decode per GPU. That immediately tells me the design has to avoid small-file access and probably needs a local cache tier."

3 · The design
INGEST ──► RAW LOG STORE ──► CURATION ──► DATASET BUILD ──► TRAIN (immutable, (mine rare (materialize object store) events, shards + auto-label) manifest) ─────────────────────────────────────────────────────────────── STORAGE LAYOUT — the load-bearing decision manifest : Parquet, partitioned by date/drive columns: sample_id, shard_uri, offset, sensor tags, weather, scenario labels, num_points → curation queries hit ONLY this: predicate pushdown and column projection make "all night rain with a cut-in" a fast scan over a small fraction of bytes payload : large sequential shards (~1 GB), sample bytes inside → object storage likes big sequential reads; it does NOT like millions of small objects (request-rate bound, and billed per request) shards : enough that world_size x num_workers divides in comfortably -- shard count constrains topology ─────────────────────────────────────────────────────────────── ORCHESTRATION Dagster assets, partitioned by drive/date, so a new auto-labeller rebuilds only what it affects VERSIONING LakeFS: build on a branch, validate, merge atomically. The commit ID goes in the checkpoint. CACHING local NVMe read-through on training nodes: first epoch pays object storage, later epochs are local SCALING decouple CPU transform from GPU train (Ray Data) so decode parallelism is not capped by the training node's core count
4 · Failure modes, unprompted

Small-file explosion (request-rate bound); shard/worker count mismatch (idle workers or a hang); temporal leakage in the train/val split; a partially written dataset consumed mid-build (hence atomic merges); shard skew making one rank the straggler; and unbounded checkpoint/dataset retention as a cost problem. Naming failure modes before being asked is what distinguishes a senior design answer.

5 · What you'd measure

Samples/sec delivered vs. samples/sec consumed (P7); per-stage timing inside __getitem__; GPU utilization distribution across ranks; dataset build wall-clock; and experiment turnaround time end to end — which is the metric that actually matters to the team's users.

31S2 · Distributed training infrastructure

design"Support 30 ML engineers running jobs from 1 to 64 GPUs."
The shape of a good answer
ConcernDesign
Job submissionA declarative spec (config + image digest + data version) rather than a shell script. Reproducibility must be structural, not disciplinary.
SchedulingGang scheduling — a 16-GPU job needs all 16 simultaneously or it deadlocks the queue. Quotas and priorities per team; backfill for small jobs.
Launchtorchrun with a c10d rendezvous, or Ray Train. Never hand-rolled rank assignment.
Fault toleranceStep-cadence checkpointing plus automatic restart; target losing ≤ 15–30 minutes on failure. Collective timeouts so hangs fail loudly.
ObservabilityPer-rank step time, GPU utilization, memory, and data-wait time — shipped to one dashboard. Straggler detection is impossible from rank 0 alone.
Cost controlIdle-GPU alerts, a maximum job duration, and spot instances for preemptible work. An input-bound 64-GPU job is a large, invisible bill.
Developer experienceThe same code runs on 1 GPU locally and 64 in the cluster. If it doesn't, engineers debug in the cluster and everyone's throughput drops.

The senior framing: the hard part is not launching distributed jobs, it's making the platform's failures legible. A job that crashes is easy. A job that hangs, or silently trains on duplicated data, or runs at 35% GPU utilization for three days, is expensive precisely because nothing alerts. So the design should include automatic detection of the three most common silent failures: low GPU utilization, straggler ranks, and stalled progress. That's the difference between infrastructure that runs jobs and infrastructure that makes a team fast.

32S3 · Reproducibility and lineage

design"A model shipped six months ago behaves oddly. Reconstruct how it was made."
The four-part answer

Code: git SHA plus a dirty flag, recorded automatically. Data: a dataset version — LakeFS commit or content digest — so "which data?" is answerable. Environment: container image digest, library versions, GPU model, world size. Config: the fully resolved config embedded in the artifact, not a path to a file that has since changed. Seeds and deterministic algorithms sit on top of those four; without them, bitwise reproduction isn't even attemptable.

The design principle

Capture must be a side effect of running, not a discipline. If an engineer can forget to record the data version, it will be missing on precisely the run you later need to audit. Bake it into the training entry point so an untracked run is impossible.

The lineage question

"Which models were trained on data produced by the labeller version we just found a bug in?" should be a query, not an investigation. That requires the dataset version in the checkpoint and an asset graph upstream (Vol. I §32–33). In a safety-relevant domain this is the requirement that justifies the whole versioning layer.

33S4 · Model promotion

design"Design the automated path from trained artifact to vehicle-ready model."
Gates, not vibes
GateWhat it asserts
Export equivalenceThe exported graph matches eager within a stated tolerance on real data. Tracing silently bakes in branches and constants — this is where models quietly change behaviour.
Post-optimization accuracyAfter quantization or compilation for the target runtime, re-measure. A separate gate, because it fails separately.
Per-slice regressionNo safety-relevant slice may regress beyond threshold: night, rain, glare, construction, cut-in, pedestrian, low-speed. Aggregate metrics can improve while a critical slice degrades — gating on the mean is the failure mode this exists to prevent.
Resource budgetLatency and memory on the actual target hardware, not a proxy.
DeterminismSame input, same output, bitwise — so a field incident can be reproduced from logs.
Provenance completeCode, data, environment, and config all recorded (S3). A model that cannot be traced does not ship.
The staging you should propose

dev → shadow (runs alongside the incumbent on logged data, predictions compared but not acted on) → candidate → deploy. Shadow mode is the highest-value stage because it evaluates on the real distribution rather than a curated validation set — and it costs nothing but compute.

Part VIII — Practice

Say it fast.

34Rapid-fire bank

Thirty seconds each, out loud
Map-style vs IterableDataset?
Random access + sampler shuffling vs a stream you must shard yourself. Streaming wins for sharded cloud data.
What does set_epoch do?
Changes the shuffle seed. Without it every epoch replays the same order.
Is DistributedSampler disjoint?
No. It pads by wrapping the first indices onto trailing ranks, so those samples appear twice. drop_last=True makes it disjoint.
Why does distributed eval over-count?
That padding. Fix by deduplicating on sample ID before reducing.
pin_memory + non_blocking?
Async H2D copy needs page-locked host memory. One without the other does nothing.
Gradient accumulation scaling?
Divide the loss by the accumulation count; use no_sync() on non-final micro-batches.
What's in a checkpoint?
Model, optimizer, scheduler, scaler, step, RNG, config, data version. Weights alone do not resume.
bf16 vs fp16?
bf16 has fp32's exponent range — no GradScaler needed. fp16 has more mantissa, narrower range.
Why is my job slow?
Run with synthetic data. If step time collapses, it's the input pipeline.
DDP vs FSDP?
DDP replicates; FSDP shards params/grads/optimizer state. For perception-scale models, DDP plus a good pipeline.
Why does DDP hang?
Ranks disagree on the collective sequence — uneven data, unused parameters, or a collective inside a rank guard.
clip_grad_norm_ returns what?
The pre-clip norm — and it does not sanitize NaN. Check isfinite before stepping.
Biggest memory leak in training code?
total += loss without .item().
Why won't my run reproduce?
Seeds, set_epoch, dataset version, environment — and num_workers, which is part of the RNG stream.
Small files on object storage?
Request-rate bound, not bandwidth bound. Shard into large sequential files.
Scale batch by 8 — what else?
Scale the LR and add warmup, or it diverges.
