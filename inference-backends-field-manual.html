FRAME Chapter I

The frame · engine, protocol, gateway.

Three layers Engine (vLLM / SGLang / TRT-LLM / hosted) executes inference. Protocol (OpenAI Chat Completions) speaks. Gateway (LiteLLM / Portkey / cloud) routes.

The rule Decouple them. The engine should be swappable. The protocol should be uniform. The gateway should never be the bottleneck.

Building applications on LLMs in 2026 is a three-layer concern. The engine layer is what actually executes inference — vLLM, SGLang, TensorRT-LLM if you self-host; OpenAI, Anthropic, Bedrock, Vertex if you call hosted models. The protocol layer is what your application speaks — OpenAI-compatible Chat Completions has become the lingua franca; every credible engine and hosted service implements it. The gateway layer is what sits between, routing requests to the right engine based on cost, latency, quality, availability. Get the layers right and engines become swappable; get them wrong and the application is coupled to a specific provider's quirks for the lifetime of the codebase.

Why decoupling matters operationally

The application code knows about protocol. The deployment knows about engine. The platform knows about gateway. If your app imports the OpenAI SDK and calls client.chat.completions.create(...), that same code can run against:

  • OpenAI's real API at api.openai.com
  • An internal vLLM serving Llama 3.3 70B at http://llm.internal/v1
  • A LiteLLM gateway that fans out across five providers based on rules
  • An Azure OpenAI deployment for compliance reasons
  • A Bedrock Anthropic endpoint via the gateway's protocol translation

One base_url change. No SDK rewrite. This is what people mean by "OpenAI is the lingua franca" — the application doesn't have to know what's running underneath. The whole 2026 toolchain is structured around this property.

The wrong-layer trap

The biggest failure mode in 2026 inference projects: putting concerns at the wrong layer.

  • Engine choice in application code. Hardcoding vllm.LLM(...) or tensorrt_llm.LLM(...) into application logic. The app is now coupled to the engine; swapping requires a code change.
  • Gateway concerns in the engine. Spend tracking inside vLLM. Multi-provider fallback inside SGLang. These engines are not gateways; the engine should serve one model on one set of GPUs.
  • Protocol negotiation in the application. Applications calling provider-specific APIs (Bedrock Anthropic, Vertex Gemini) directly. Every provider switch becomes a rewrite.

The fix is always the same: collapse decisions to the right layer. Application code speaks OpenAI protocol. Gateway handles routing/fallback/spend. Engine serves models.

The honest 2026 default

Per the 2026 practitioner take: "Our practice: we default to vLLM for new deployments until we establish the workload profile. Once we confirm prefix reuse patterns, structured output load, and agent workflow complexity, we reassess. For teams building agent-heavy stacks from scratch in 2026, we increasingly start with SGLang."

The progression in practice:

  1. Hosted API first. Build the product on OpenAI / Anthropic. Validate that LLM is the right tool. Don't self-host until volume justifies it.
  2. vLLM second. When self-hosting starts to make economic sense, start with vLLM. Broadest support, simplest deployment, OpenAI-compatible API means no app code change.
  3. SGLang or TRT-LLM third. Once you understand your workload — prefix patterns, structured output needs, model stability — reassess. Move to SGLang for prefix-heavy. Move to TRT-LLM for throughput-ceiling.
The economics framing

But cold-start cost matters too — a 28-minute TensorRT-LLM compile makes model iteration far more expensive than the per-token savings buy back if you ship model updates weekly.

ENGINE Chapter III

vLLM · the mechanism.

The core idea PagedAttention manages the KV cache like virtual memory — fixed blocks, no fragmentation. Continuous batching plus prefix caching plus chunked prefill produce the rest.

The mental model One process per GPU group. OpenAI-compatible HTTP server. Scheduler decides which sequences advance each step.

vLLM is the engine to understand first because its mechanism became the template the rest of the field followed. PagedAttention, continuous batching, OpenAI-compatible API, broad model support — all originated or matured here. This chapter walks the mechanism and the knobs that matter in production, building on the inference fundamentals from Issue 11.

The architecture

A vLLM serving process is one Python process that owns a set of GPUs (typically one node, optionally with tensor parallelism across multiple GPUs on the node). Inside it: a tokenizer, a scheduler, a model executor, a KV cache manager, and an HTTP server exposing the OpenAI Chat Completions API.

vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --tensor-parallel-size 4             # split model across 4 GPUs on this node \
  --gpu-memory-utilization 0.90       # 90% of GPU mem for weights+KV; rest = headroom \
  --max-model-len 32768                # max context window; sets KV budget \
  --max-num-seqs 256                   # max concurrent sequences \
  --enable-prefix-caching                # share KV blocks across requests with same prefix \
  --enable-chunked-prefill               # interleave prefill chunks with decode steps \
  --dtype auto                           # bfloat16 on H100/H200; fp16 elsewhere \
  --quantization fp8                     # or awq, gptq; trades quality for memory \
  --kv-cache-dtype fp8                   # shrink KV by 2× vs bf16 \
  --served-model-name llama-70b          # the name clients use in `model` param \
  --api-key $VLLM_API_KEY                # bearer token auth on /v1 endpoints \
  --host 0.0.0.0 --port 8000

The three most consequential settings — and the ones that bite when wrong:

  • --gpu-memory-utilization. Default 0.90; lower to 0.85 if you OOM during peak. The remaining 10% is headroom for activations and CUDA workspace. Going to 0.95 in pursuit of more KV space frequently triggers OOM under load.
  • --max-model-len. Sets the KV cache budget. Halving it doubles your concurrent sequences for the same memory. Set to the actual max you serve, not the model's theoretical max — Llama 3.3 70B supports 128K but if your application caps at 8K, set 8K.
  • --tensor-parallel-size. Splits weights across GPUs on one node. Must divide attention heads evenly. Use 1 for models that fit on one GPU; 2/4/8 for larger. Crosses NVLink; don't cross nodes (use pipeline parallelism instead).

Prefix caching — the major 2026 default

With --enable-prefix-caching, vLLM hashes the prefix of each request and reuses KV blocks for requests that share it. The two scenarios that benefit massively:

  • System prompts. Most chat applications send a fixed system prompt (500-2000 tokens). With prefix caching, that prompt's KV is computed once and shared across all subsequent requests.
  • RAG with shared context. When users query the same document repeatedly, the document's KV is computed once.
  • Multi-turn chat. Each user turn re-sends the conversation; everything except the last user message hits the cache.

Per the 2026 production deployment guide: "Enable PagedAttention optimizations including prefix caching and chunked prefill for production workloads." Treat both as production defaults.

Chunked prefill — the latency lever

Prefill (processing the prompt) is compute-intensive and blocks decode (generating tokens for already-active sequences). For a long prompt, a single batch that includes the prefill stalls every other sequence's decode while the prefill completes. Chunked prefill splits the prefill across multiple iterations, interleaving it with decode steps. The effect: lower decode latency variance, more predictable TTFT, at minor cost to prefill speed.

The KV cache dtype

--kv-cache-dtype fp8 stores the KV cache at 8-bit precision instead of bf16. Halves KV memory — doubles concurrent sequences. The quality cost is typically small for chat (perplexity delta < 0.5%); measure on your eval set before adopting. Combined with weight quantization (--quantization fp8 or awq), the same GPU can serve a model that wouldn't fit at full precision.

Tensor parallelism vs pipeline parallelism

For multi-GPU deployment:

  • Tensor parallelism (TP). Splits each layer across GPUs. Requires fast interconnect (NVLink); doesn't cross nodes well. Use within a node.
  • Pipeline parallelism (PP). Splits layers across GPUs (early layers on GPU 0, later on GPU 1). Tolerates slower interconnect; can cross nodes. Use across nodes when needed.
  • Mixed. TP within a node, PP across nodes for very large models.

The OpenAI-compatible API

vLLM ships an OpenAI-compatible HTTP server. The endpoints:

# the main endpoint; streaming via SSE
POST /v1/completions         # legacy text completion
POST /v1/embeddings          # if model is an embedding model
GET  /v1/models              # list available models

Any client that speaks OpenAI works — change base_url to your vLLM endpoint and call client.chat.completions.create(model="llama-70b", ...). This decoupling — engine swappable, application unchanged — is the property that makes vLLM dominant.

Hardware breadth

Per the 2026 Morph comparison: "vLLM has production-grade support for Google TPU v4-v6e, Trainium, Inferentia, and Gaudi with active corporate contributions from Google, AWS, and Intel." This is the biggest single argument for vLLM over SGLang in multi-cloud or non-NVIDIA environments. If you're forced onto Trainium or TPU for cost reasons, vLLM is the only credible production engine in 2026.

The mental model

vLLM is one process per GPU group, holding model weights + paged KV cache, with a continuous-batching scheduler that decides which sequences advance each decode step. Configuration is mostly about memory math — pick max-model-len, gpu-memory-utilization, and max-num-seqs together so the implied KV budget actually fits. Almost every "vLLM OOM under load" report traces to setting these inconsistently with the underlying memory.

ENGINE Chapter IV

SGLang · RadixAttention & structured output.

The differentiator RadixAttention reuses KV cache across requests with shared prefixes via a radix tree. Up to 6.4× on prefix-heavy workloads.

The second differentiator Structured output (JSON schema, regex, grammar) compiled into the decoding loop with mask generation overlapping inference compute.

SGLang's reputation in 2026 rests on two specific wins. RadixAttention extends prefix caching from "same prefix → reuse blocks" to "any shared prefix anywhere in a tree of requests → reuse blocks" — a step-function improvement for workloads where the same prefixes get reused across many concurrent users. Structured output is the second: SGLang treats JSON schemas and regex grammars as first-class scheduling concerns, with mask generation overlapping the GPU's inference compute. Both wins compound for the workloads that matter in agent-heavy 2026 production.

RadixAttention — what it is

vLLM's prefix caching uses a hash table keyed by prefix block hash. Two requests with the same first N tokens hit the same cached KV blocks. Useful, but it misses cases like:

  • Many users querying the same long RAG document with different questions: each user's query starts with the same N-thousand-token document context.
  • An agent running 20 parallel tool calls, each starting from the same conversation history.
  • A chain of thought where intermediate states diverge and rejoin.

RadixAttention models the cache as a radix tree (prefix tree). Every active sequence's KV is a path through the tree; branches share nodes when sequences share prefixes. New requests walk the tree to find the longest matching path, attach to it, and continue from there. The cache hit rate stays high even when prefixes are partial-shared rather than identical.

The performance numbers

Per the 2026 particula.tech benchmark: "SGLang's RadixAttention gives it a 29% throughput edge over vLLM on H100 GPUs (16,200 vs 12,500 tokens/sec) and up to 6.4x gains on prefix-heavy workloads like RAG and multi-turn chat. Default to SGLang for DeepSeek deployments, conversational AI, and structured output."

The 6.4× number is the headline but conditional — it's prefix-heavy workloads. For one-off completions with no shared prefix (bulk document summarization where each doc is unique), the advantage shrinks. Per the 2026 comparison: "When every request has a distinct prompt with no shared prefix (bulk document summarization, one-off completions, evaluation harnesses), RadixAttention's cache provides no benefit and adds memory overhead. vLLM's clean page-and-free model is simpler and avoids the cache management cost."

Structured output — the second win

Forcing an LLM to emit valid JSON or match a regex used to mean post-hoc retry loops. SGLang compiles structured-output constraints into the decoding loop directly:

curl http://sglang.internal:30000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "llama-70b",
    "messages": [
      {"role": "user", "content": "Extract structured data from: order #1234 for 2 widgets at $25 each"}
    ],
    "response_format": {
      "type": "json_schema",
      "json_schema": {
        "schema": {
          "type": "object",
          "properties": {
            "order_id": {"type": "string"},
            "quantity": {"type": "integer"},
            "unit_price": {"type": "number"}
          },
          "required": ["order_id", "quantity", "unit_price"]
        }
      }
    }
  }'

Per the morphllm 2026 deep dive: "The key optimization: mask generation (computing which tokens are valid at each step) overlaps with the GPU's inference computation." Naive structured output stalls between tokens to compute the valid next-token set; SGLang overlaps this with the GPU's compute, hiding the cost almost entirely.

SGLang configuration

python -m sglang.launch_server \
  --model-path meta-llama/Llama-3.3-70B-Instruct \
  --tp-size 4                               # tensor parallel \
  --mem-fraction-static 0.85                 # fraction of GPU mem for weights \
  --max-running-requests 256                 # concurrent sequences \
  --context-length 32768                     # max context \
  --enable-radix-cache                          # default on; explicit for clarity \
  --quantization fp8                            # or awq \
  --schedule-policy lpm                         # longest-prefix-match scheduler \
  --host 0.0.0.0 --port 30000

The novel knob: --schedule-policy lpm (longest-prefix-match). The scheduler prioritizes batching requests with shared prefixes to maximize RadixAttention hits. Other policies: fcfs (first-come-first-served), random. The default lpm is correct for almost all production workloads.

Multi-LoRA scheduling

SGLang's other underrated win for 2026: efficient multi-LoRA serving. If you serve dozens of fine-tuned variants from one base model, SGLang's adapter scheduling outperforms vLLM at high adapter counts. Per the turion analysis: "You need multi-LoRA efficiency. SGLang's multi-LoRA scheduling outperforms vLLM at high adapter counts, which matters if you serve dozens of fine-tuned variants from one base model."

The hardware caveat

The major operational tradeoff: hardware breadth. Per morphllm: "SGLang's TPU support (via SGLang-Jax) is experimental." If you're NVIDIA-only, this is a non-issue. If you have any chance of moving to TPU, Trainium, or Gaudi, vLLM is the safer choice.

The RadixArk spinout

SGLang's commercial footing strengthened materially in 2026. Per the 2026 analysis: "SGLang emerged as its [TGI's] primary alternative — backed by a reported $400M spinout as RadixArk, led by Accel." The implication for production: enterprise support is available; the project has runway for sustained engineering investment; the supply-chain risk that hangs over smaller OSS projects is reduced.

The SGLang default decision

Pick SGLang from day one for: agent workloads with shared system prompts, RAG over a fixed corpus, multi-turn chat applications, structured-output-heavy pipelines (JSON extraction at scale), DeepSeek deployments specifically (3.1× advantage there). Pick vLLM for: NVIDIA + TPU/Trainium/Gaudi heterogeneous deployments, encoder-decoder models, bulk one-off completions, broadest community/integration ecosystem. When in doubt, start vLLM; reassess at the 6-month operating mark.

ENGINE Chapter V

TensorRT-LLM · compiled throughput ceiling.

The bet Ahead-of-time compile the model into a TRT engine. Fused kernels, picked algorithms, baked-in shapes. Peak throughput on NVIDIA hardware.

The cost ~28-minute compile per model. NVIDIA-only. Compilation pipeline to maintain. The right answer when stability and ceiling matter more than agility.

TensorRT-LLM occupies a distinct position. Where vLLM and SGLang are general-purpose Python-Python engines, TRT-LLM is a compiler. You hand it model weights and target shape/precision, it produces an optimized engine plan binary, that binary serves with the highest measured throughput on NVIDIA hardware. The price is operational: every model change means a re-compile; the toolchain is heavier; and you can only run on NVIDIA. For long-lived stable production at the throughput ceiling, that price is worth it. For most other situations, it isn't.

    ENGINE Chapter VI

    Benchmarks & the honest picker.

    Three benchmark suites Throughput (tok/s peak), latency (TTFT, ITL), cost (per-million-tokens). Each tells a different story.

    The trap Benchmarks generalize poorly. Engine X wins on workload A and loses on workload B. Measure on YOUR traffic shape.

    The non-obvious details that bite

    The /dev/shm volume. Per the 2026 K8s deployment guide: "OOMKilled on startup. Two causes: missing shared memory volume at /dev/shm for tensor parallelism, or the model is too large for available GPU VRAM. Fix: add the emptyDir with medium: Memory, or use quantization (--quantization awq) to reduce memory footprint." Tensor parallelism uses POSIX shared memory for inter-process communication; the default container /dev/shm is 64 MB and overflows immediately. The Memory-backed emptyDir gives several GB.

    Pin the image tag. Never vllm/vllm-openai:latest in production. A nightly build can ship a regression that breaks your service silently. Pin to a specific tag (v0.7.3), test the upgrade in staging, then promote.

    Probe timeouts must accommodate model load. A 70B model loading from a fast PVC takes 60-180 seconds; from a slow object store, several minutes. Set initialDelaySeconds to comfortably cover the slow case. The most common production incident in early vLLM deployments: liveness probe failing during model load, kubelet restarts the pod, model loads again, fails again, CrashLoopBackoff.

    Recreate strategy, not RollingUpdate. A new vLLM replica needs full GPU memory to load the model. RollingUpdate tries to start the new pod before terminating the old; if both can't fit on the cluster's GPU capacity, the rollout stalls. Recreate terminates the old replica first; you accept a brief unavailability window for the upgrade. For HA, run multiple replicas and roll them one at a time via separate deployments or via an operator like KServe.

    KServe, llm-d & the production stack.

    The roles KServe = Kubernetes-native model lifecycle (scale-to-zero, canary, revisions). llm-d = distributed inference intelligence (prefix-aware routing, prefill/decode disaggregation).

    K8S Chapter IX

    Autoscaling inference · KEDA, queue depth, scale-to-zero.

    The wrong metric CPU utilization. GPU utilization. Both are misleading for LLM inference.

    The right metric Request queue depth from vLLM's Prometheus metrics, fed to KEDA, which scales pods on it.

    Autoscaling LLM inference is where most teams discover that the Kubernetes patterns they know don't apply. HPA on CPU is wrong; HPA on GPU utilization is wrong; even HPA on request rate is wrong. The signal that actually matters is the engine's internal queue depth, exposed as a Prometheus metric, consumed by KEDA. This chapter walks the autoscaling stack and the specific pitfalls.

    Why standard HPA fails for LLM inference

    Per the 2026 GKE best-practice analysis: "This is the most common mistake in LLM deployment. Standard HPA scales on CPU utilization, but LLM inference is GPU-bound. Your CPU can sit at 5% while your inference queue backs up with 50 waiting requests. Google's GKE best practices document this clearly: GPU Utilization (DCGM_FI_DEV_GPU_UTIL) is a duty cycle measurement. A GPU at '100% utilization' could be processing 10 requests or 100. This metric won't tell you the difference. GPU Memory is pre-allocated by vLLM for the KV cache. Memory usage stays constant regardless of load, so it never triggers scale-down."

    The breakdown:

    • CPU. Mostly idle (~5%) during inference. Scaling on it never triggers.
    • GPU utilization. Binary in effect — "100%" with 5 active requests is the same as "100%" with 500. The metric is duty cycle, not throughput.
    • GPU memory. vLLM pre-allocates the KV cache pool at startup; memory usage stays roughly constant regardless of load.
    • Request rate (RPS). Closer but still wrong — RPS doesn't know how long each request takes. 10 short requests vs 10 long requests have very different load.

    The right metric — queue depth

    vLLM exposes vllm:num_requests_running and vllm:num_requests_waiting as Prometheus metrics. The waiting count is the queue depth — sequences submitted but not yet being decoded because the engine is at capacity. When this grows, you need more replicas. When it drops to zero, you have excess capacity.

    The KEDA pattern

    KEDA (Kubernetes Event-Driven Autoscaling) extends HPA to scale on arbitrary external metrics — including Prometheus metrics. The standard 2026 pattern: vLLM exposes queue depth → Prometheus scrapes → KEDA reads via Prometheus scaler → scales the Deployment.

    Scale-to-zero — the cost lever

    For workloads with intermittent traffic (dev/internal tools, low-RPS production services), scale-to-zero saves substantial GPU cost. minReplicaCount: 0 tells KEDA to remove all pods when no traffic for cooldownPeriod. First request after scale-down triggers spin-up; cold start latency applies (60-180s for vLLM).

    Per the 2026 scaling guide: "Install KEDA with a Prometheus scaler to scale from zero, avoiding always-on GPU costs." The trade-off is starkly traffic-dependent. For 24/7 production: scale-to-zero costs more than it saves (cold starts on every dawn traffic spike). For dev or weekend-idle services: scale-to-zero can cut GPU bill 60-80%.

    The cooldown trap

    Cooldown is the time after the last scale-up before KEDA will scale down. Too short → flapping (scale up, scale down, scale up, scale down). Too long → wasted capacity. The sweet spot for inference: ≥5 minutes. Inference cold-starts are expensive; you'd rather pay 5 minutes of extra GPU than pay another cold start.

    The full Red Hat / KServe pattern

    Per the Red Hat KServe+KEDA guide: "We will walk through the process of setting up KServe autoscaling by leveraging the power of vLLM, KEDA (Kubernetes Event-driven Autoscaling), and the custom metrics autoscaler operator in Open Data Hub. This powerful combination allows us to scale our vLLM services on a wide range of custom, application-specific signals, not just generic metrics."

    The Red Hat pattern adds the Custom Metrics Autoscaler Operator (CMAO) — a wrapper that exposes any Prometheus metric as a K8s metric API. KEDA then consumes it via the standard mechanism. This layer is required on OpenShift; on vanilla Kubernetes, KEDA's Prometheus scaler reads Prometheus directly.

    The TGI situation

    Hugging Face Text Generation Inference (TGI) was for years the third leg of "vLLM vs TGI vs Triton" comparisons. Per the 2026 inference engine landscape: "Hugging Face put TGI into maintenance mode in December 2025, and SGLang emerged as its primary alternative — backed by a reported $400M spinout as RadixArk, led by Accel."

    The practical implications:

    • If you're running TGI in production today. It continues to work; HF will accept critical bug fixes. New feature development has effectively stopped. Plan migration to vLLM or SGLang over the next year.
    • If you're choosing an engine for new deployments. Don't pick TGI. The community has consolidated around vLLM and SGLang.
    • HF Inference Endpoints. The managed product continues. Under the hood it may use TGI, vLLM, or other engines depending on the model. The OpenAI-compatible surface stays stable; the runtime underneath has shifted.

    HF Serverless Inference API

    Distinct from Inference Endpoints: HF also offers a pay-per-call Serverless Inference API for many Hub models. Convenient for prototyping; rate-limited and shared-tenant; not for production. The 2026 use case: rapid model evaluation. Try a model via the serverless API, decide if it works for your problem, then either commit to a dedicated Inference Endpoint or to self-hosting.

    The model card discipline

    Every model on the Hub has a model card describing license, intended use, training data, evaluations. For production deployments:

    • Read the license. Llama is community-license (some restrictions); Mistral varies by model; Apache 2.0 and MIT are permissive. Some "open" models prohibit commercial use.
    • Check the architecture. Some Hub models use experimental architectures that vLLM or SGLang don't support yet. The model card or community discussion will note compatibility.
    • Note the precision. bf16 weights are the standard; some models ship as fp16 only; some as 4-bit or 8-bit quantized variants in separate repos.

    Fine-tuned variants and LoRA

    The Hub hosts thousands of LoRA fine-tunes of base models. Serving them efficiently is where SGLang's multi-LoRA support and AIBrix's LoRA management shine (Ch VIII). The pattern: base model loaded once into the GPU; per-tenant or per-task LoRA adapters loaded into a small additional memory pool; the serving engine routes requests to the right adapter. Hub provides the registry; the serving infrastructure provides the runtime composition.

    The HF role in 2026

    Treat Hugging Face Hub primarily as the model registry. Treat HF Inference Endpoints as one option among hosted serving providers — competing with OpenAI/Anthropic for proprietary models and with self-hosted vLLM/SGLang for open models. Treat TGI as legacy; don't start new deployments on it. The HF brand is strongest at the registry layer; the serving competition has moved to vLLM and SGLang.

    API Chapter XII

    LiteLLM & the gateway layer.

    The gateway role One OpenAI-compatible endpoint in front of 100+ providers. Virtual keys, spend tracking, fallbacks, guardrails, load balancing.

    The 2026 leader LiteLLM as the most feature-complete OSS gateway. Portkey for governance-heavy use. OpenRouter for zero-setup prototyping. Cloudflare AI Gateway for observability-only.

    An LLM gateway is the routing fabric between application code and the engine layer. The application speaks OpenAI; the gateway decides where each request goes. Fallbacks, retries, spend tracking, virtual API keys, multi-provider load balancing, content filtering — all of these belong here, not in the application and not in the engine. This chapter walks the 2026 gateway landscape and the configuration patterns that matter.

    The gateway role, named

    A gateway sits between application code and one or more model endpoints. It does five things:

    • Protocol normalization. Application speaks OpenAI; gateway translates to provider-specific protocols (Anthropic Messages, Bedrock InvokeModel, Vertex predict) underneath.
    • Routing. Decides which provider/model handles each request based on rules (cost, latency, model availability, request content).
    • Fallback. If primary provider fails, retry on a secondary. Definitions of "fail": HTTP errors, content policy violations, rate limits, latency timeouts.
    • Spend tracking. Captures per-request token usage and computes cost. Aggregates by virtual key, team, project.
    • Access control. Virtual API keys with budgets, rate limits, allowed-model restrictions, per-team scoping.

    LiteLLM — the de facto OSS gateway

    Per the LiteLLM project description: "Python SDK, Proxy Server (AI Gateway) to call 100+ LLM APIs in OpenAI (or native) format, with cost tracking, guardrails, loadbalancing and logging. [Bedrock, Azure, OpenAI, VertexAI, Cohere, Anthropic, Sagemaker, HuggingFace, VLLM, NVIDIA NIM]." LiteLLM ships as two artifacts:

    • Python SDK. Import litellm.completion() in your application code. Direct integration; no separate service to operate.
    • Proxy Server (AI Gateway). Standalone service. Application speaks OpenAI to the proxy; proxy handles routing.

    The proxy is the production pattern. SDK is fine for prototypes; once multiple teams or services consume LLMs, the proxy provides the central control plane.

    The killer feature for multi-team deployments: virtual keys. The admin issues a virtual key per team with a budget, rate limit, allowed-model list, and metadata. Teams use that virtual key as their api_key; LiteLLM enforces the constraints and tracks spend per key.

    Fallback patterns

    Per the LiteLLM fallbacks documentation: "Fallback across providers (e.g. from Azure OpenAI to Anthropic) if you hit content policy violation errors." The fallback chain handles several failure modes:

    • Primary provider rate-limited → fall back to secondary provider with capacity.
    • Primary provider down → fall back to alternative.
    • Content policy violation on primary → try a model with different content policy.
    • Primary latency timeout → retry on faster provider.

    Routing strategies

    LiteLLM supports several routing modes:

    • simple-shuffle — round-robin across replicas of the same model_name.
    • least-busy — pick the deployment with the smallest in-flight request count.
    • usage-based-routing-v2 — route based on token usage; honors rpm/tpm limits per deployment.
    • latency-based-routing — pick the deployment with lowest p99 latency from recent history.
    • cost-based-routing — pick the cheapest available deployment that satisfies the request.

    API Chapter XIII

    Application patterns · RAG, agents, streaming.

    Three patterns dominate 2026 RAG (retrieval-augmented generation), agents (tool-using loops), streaming UX. Each has infrastructure consequences beyond the inference call itself.

    The lesson The inference engine is one component. The pattern around it determines whether the system actually works.

    Production LLM applications in 2026 are dominated by three patterns. RAG augments inference with retrieved context — pull relevant documents, stuff them into the prompt, generate. Agents loop the LLM through tool calls, executing actions in the world. Streaming pushes tokens to the user as they're generated, masking latency. Each pattern has implications for the inference stack that go beyond the engine choice. This chapter walks how each pattern interacts with the layers built up in Chapters I-XII.

    RAG — retrieval-augmented generation

    Agents — tool-using loops
    
      
    The architecture for autonomous LLM systems. The LLM generates not just text but tool calls; the application executes the tools; results feed back into the conversation; loop until the model emits a final answer.
    
      
    messages = [system_prompt, user_query]

while True:
    response = client.chat.completions.create(
        model="llama-70b",
        messages=messages,
        tools=available_tools,
        tool_choice="auto",
    )

    msg = response.choices[0].message
    messages.append(msg)

    if msg.tool_calls:
        for tool_call in msg.tool_calls:
            result = execute_tool(tool_call.function.name, tool_call.function.arguments)
            messages.append({
                "role": "tool",
                "tool_call_id": tool_call.id,
                "content": result,
            })
    elif msg.content:
        return msg.content   # final answer
    else:
        break                  # shouldn't happen; error

The reason: agent loops have huge prefix overlap between iterations. Each loop iteration sends the entire conversation; everything before the last assistant turn is identical to the previous iteration. RadixAttention turns this from "recompute on every iteration" to "reuse cache between iterations."
    

      
    Agent infrastructure consequences
    
      
      
        
    • Long-running connections. An agent loop may take 30 seconds to several minutes. Configure ingress timeouts accordingly — defaults are often 60s.
      • 
        
    • State management. Conversation state lives somewhere — Redis, Postgres, in-memory. Per-conversation state is a database load distinct from inference load.
      • 
        
    • Tool execution latency. Tools can be fast (a math computation) or slow (a web search, a database query). The agent's perceived latency is dominated by the slowest tool.
      • 
        
    • Failure modes. Infinite loops (model can't decide). Timeouts. Tool failures. Each needs explicit handling.
      • 
        
    • Cost. Each iteration is a full inference call. Agent loops are 3-10× the cost of single-shot completions.
      •

    The application/infrastructure division

When you find application code touching engine-specific knobs, you've broken the abstraction. When you find infrastructure code understanding agent semantics, you've also broken it.     K8S Chapter XIV

    Cost & throughput optimization.

    The economic equation Cost per million tokens = GPU $ per hour ÷ tokens per second per replica × 3600.

    The levers Quantization (FP8, AWQ, GPTQ), batching, prefix caching, hardware selection, spot capacity, scale-to-zero, model size choice.

    • Latency SLA (high concurrency increases tail latency).
    • Workload shape (chat tolerates more queue; agent loops less).

    Lever 6 — model size

    Often overlooked: pick the smallest model that hits quality bar. A 7B fine-tuned for your specific task can match a 70B at 10× lower cost. The 2026 pattern at most teams: routing layer (Inworld-style, or rules-based in LiteLLM) sends easy requests to a cheap 7B and hard requests to the 70B.     Lever 7 — prompt caching at the gateway

    For hosted providers that support prompt caching (Anthropic, OpenAI), enable it. Repeated long-context calls (RAG with shared corpus, agent loops) benefit massively. Hosted prompt caching can be 90% cost reduction on cached portions.

    The cost dashboard

    The metrics every production team should track weekly:

    • $ per million tokens — overall and per-route.
    • Token volume — input vs output, by application.
    • Cache hit rate — prefix cache hit % per engine.
    • GPU utilization actual — not duty cycle; tokens served vs theoretical max.
    • Spend by virtual key / team — for chargeback and quota.
    The order of operations for cost reduction

    Don't optimize prematurely. The order at most teams: (1) measure your current $ / million tokens; (2) check engine — pick the right one for your workload; (3) enable prefix caching and prefix-aware routing — biggest single win for chat/RAG/agents; (4) quantize to FP8; (5) right-size your model — try a smaller model fine-tuned on your task; (6) mix spot capacity; (7) scale-to-zero for intermittent traffic. Each step yields a measured multiplier; compounded, the 5-10× savings vs naive baseline materialize.

    API Chapter XVI

    Multi-tenancy & quota for LLM APIs.

    The problem Multiple teams sharing inference infrastructure. Each team needs isolation, fair quota, predictable cost, and the ability to fail without blast-radius into others.

    The pattern Virtual keys at the gateway + namespace-scoped engine deployments + quota at both layers.

This chapter walks the patterns that handle multi-tenancy without surprises.

    The LoRA pattern for tenant fine-tunes

    Many tenants want their own fine-tuned variant of the base model. Naively, this means 1 base model copy per tenant — prohibitively expensive in GPU memory. The 2026 pattern: one base model copy shared, per-tenant LoRA adapters loaded on demand. SGLang's multi-LoRA scheduling (Ch IV) and AIBrix's LoRA management make this efficient.

    Content filtering and guardrails

    Multi-tenant deployments need content moderation as a tenant-isolatable concern. The 2026 pattern: gateway-level guardrails. LiteLLM supports input/output guardrails as plugins; Portkey makes governance its centerpiece. The hooks:

    • Input scanning. PII detection (Presidio), prompt injection detection, jailbreak attempts. Block or sanitize before the engine sees the prompt.
    • Output scanning. Toxic content, regulated content, sensitive information. Block or redact before returning to user.
    • Per-tenant policy. Different tenants may have different sensitivity thresholds (a child-safety product needs stricter filtering than an internal engineering tool).

    The blast radius pattern

    The thing you really need from multi-tenancy: one tenant's failure doesn't take down others. The checklist:

    • Per-tenant quota enforced at gateway — runaway tenant gets 429, others continue.
    • Engine replicas separated where SLA matters — bad prompt that takes 30s on tenant A's replica doesn't affect tenant B.
    • Network isolation between tenants — tenant A can't reach tenant B's engine endpoints.
    • Spend caps that auto-disable a tenant when exhausted — runaway costs are contained.
    • Per-tenant alerting routes — pages go to the right team.
    The multi-tenancy maturity curve

    Start with shared-engine + gateway quota — sufficient for most teams under 5 tenants. Graduate to per-tenant engine replicas when SLA contention starts. Graduate to per-tenant namespaces with NetworkPolicies when compliance demands it. Graduate to per-tenant clusters only when regulation requires full physical isolation. Each step adds operational complexity; only escalate when forced by real requirements, not preemptively.

    Rolling update stalls
    New pod can't start before old terminates; GPU capacity insufficient for both.
    Switch strategy to Recreate, or scale cluster first, or use blue-green via KServe revisions.
    Streaming responses truncated
    Ingress buffering SSE; default NGINX proxy_buffering on.
    Set nginx.ingress.kubernetes.io/proxy-buffering: "off"; increase proxy-read-timeout.
    Spot preemption causes failed requests
    No graceful drain on SIGTERM; in-flight requests killed.
    Set terminationGracePeriodSeconds: 60; ensure engine handles SIGTERM (vLLM does).

    The 24-item production checklist

    Engine pinned to specific image tag
    No :latest; specific version like vllm/vllm-openai:v0.7.3; upgrade tested in staging first.
    /dev/shm Memory emptyDir mounted
    Required for tensor parallel; default 64MB shared memory overflows immediately. Size ≥32Gi.
    Probe timeouts cover model load
    readinessProbe initialDelaySeconds ≥120; livenessProbe ≥300. Prevents CrashLoopBackOff during slow load.
    Prefix caching enabled
    --enable-prefix-caching on; verify vllm:gpu_prefix_cache_hit_rate metric >0.
    Chunked prefill enabled
    --enable-chunked-prefill on; smooths TTFT variance under load.
    FP8 quantization where supported
    --quantization fp8 --kv-cache-dtype fp8 on H100/H200; halves memory; quality measured on eval set.
    Memory math validated
    max-model-len × max-num-seqs implied KV fits in (gpu-memory-utilization × VRAM − weights). Loaded under peak traffic before going live.
    Autoscale on queue depth, not CPU
    KEDA Prometheus scaler on vllm:num_requests_waiting; threshold ~5; cooldown ≥300s.
    Cold-start mitigation in place
    Model pre-staged on PVC or baked into image; not downloaded per pod start.
    Prefix-aware routing if chat/RAG/agents
    llm-d or vLLM Production Stack router. Tesla case study: 3× output tok/s, 2× TTFT just from this.
    Ingress configured for SSE pass-through
    proxy-buffering: "off"; long proxy-read-timeout; tested with streaming curl.
    LiteLLM gateway in front of engines
    Virtual keys per team; spend tracking; fallback chain; OpenAI-compatible to app code.
    Virtual key quotas set per team
    rpm, tpm, max_parallel_requests, max_budget. Runaway tenant contained; spend predictable.
    Fallback chain defined
    Primary → secondary mapping in gateway config; tested via fault injection.
    Application speaks OpenAI protocol only
    No provider-specific SDK in app code; base_url is the engine/gateway swap point.
    Token usage logged per request
    prompt_tokens + completion_tokens captured to spend dashboard; per-team aggregation.
    Engine /metrics scraped by Prometheus
    ServiceMonitor or scrape annotations in place; Grafana dashboards live; alerts firing.
    Distributed tracing instrumented
    OpenTelemetry spans for retrieval / tool calls / LLM completion; end-to-end trace visible.
    Prompt-level capture (Langfuse / Helicone)
    Full request/response pairs captured for debug, eval, prompt iteration. Sampled if volume high.
    Alerts on TTFT p99, queue depth, error rate, spend
    Per Ch XV alert table; PagerDuty/Opsgenie routes by severity.
    Agent iteration cap
    Agent loops bounded (10-20 iterations); repetition detection; user-facing timeout.
    Spot/on-demand mix if cost-sensitive
    Baseline on on-demand for stability; burst capacity on spot for 30-70% cost reduction.
    Engine choice matches workload profile
    SGLang for prefix-heavy; vLLM for general; TRT-LLM for stable-model throughput-ceiling; reassessed quarterly.
    Disaster runbook documented
    What to do when engine OOMs, when provider down, when state corrupts, when spend spikes. Rehearsed.
