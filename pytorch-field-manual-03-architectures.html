Part I — Convolutional networks

Networks that see structure.

Images have structure a plain MLP throws away: nearby pixels relate, and a cat is a cat wherever it sits in the frame. Convolutional networks bake in both facts — locality and translation-invariance — with a small sliding filter that shares its weights across the whole image. The result is dramatically fewer parameters and dramatically better vision. This part builds the CNN from the convolution up.

01Convolution mechanics

Feeding a 224×224 color image to an nn.Linear means 150,000 inputs and, for a modest hidden layer, tens of millions of weights — most spent relearning the same edge detector at every position. A convolution refuses that waste. It slides a small window (the kernel, say 3×3) across the image and, at each position, computes a weighted sum of the pixels under it. The same kernel weights are reused at every location, so a feature learned in one corner is detected everywhere — that's weight sharing and translation invariance in one stroke.

A 3x3 KERNEL SLIDING OVER AN INPUT -> ONE OUTPUT PIXEL EACH STOP: input (one channel) kernel output feature map ┌──┬──┬──┬──┬──┐ ┌──┬──┬──┐ ┌──┬──┬──┐ │ │██│██│██│ │ │w1│w2│w3│ │ │▓ │ │ ├──┼──┼──┼──┼──┤ dot with ├──┼──┼──┤ = ├──┼──┼──┤ │ │██│██│██│ │ ─────────► │w4│w5│w6│ │ │ │ │ ├──┼──┼──┼──┼──┤ ├──┼──┼──┤ └──┴──┴──┘ │ │██│██│██│ │ │w7│w8│w9│ each output = Σ (pixel × weight) └──┴──┴──┴──┴──┘ └──┴──┴──┘ over the 3x3 window, + bias

Real images have channels — a color image is 3 (red, green, blue) — and a conv layer produces many output channels, one per learned filter. So a conv layer's weight is a 4-D tensor of shape (out_channels, in_channels, kernel_h, kernel_w), and it maps a (batch, in_ch, H, W) tensor to (batch, out_ch, H', W'). Three knobs control the output size:

KnobEffect
kernel_sizeThe window's size. 3×3 is the modern default; stacks of small kernels beat one big one.
strideHow far the window jumps each step. Stride 2 halves the spatial size — a way to downsample.
paddingZeros added around the border so edge pixels get full windows. padding=1 with a 3×3 kernel keeps H, W unchanged.
 
import torch
import torch.nn as nn

conv = nn.Conv2d(in_channels=3, out_channels=16,
                 kernel_size=3, stride=1, padding=1)   # 'same' spatial size

x = torch.randn(8, 3, 32, 32)      # batch 8, RGB, 32x32
y = conv(x)                        # -> (8, 16, 32, 32): 16 feature maps
print(y.shape)
print(conv.weight.shape)           # (16, 3, 3, 3) = (out, in, kh, kw)

The output-size formula is worth memorizing because shape bugs here are constant: out = (in + 2·padding − kernel) / stride + 1. With a 3×3 kernel, padding=1, stride=1, the size is preserved; bump stride to 2 and it roughly halves.

02Building a CNN & the receptive field

A convolutional network interleaves three moves: convolutions that extract features, nonlinear activations, and downsampling that shrinks the spatial size while growing the channel count. Downsampling is done either by strided convolutions or by pooling — nn.MaxPool2d keeps the strongest response in each little region, discarding precise position for robustness. As the map shrinks and deepens, each neuron's receptive field — the patch of original image it can "see" — grows, so early layers detect edges and textures while later layers assemble them into shapes and objects.

import torch.nn as nn

class SmallCNN(nn.Module):
    def __init__(self, num_classes=10):
        super().__init__()
        self.features = nn.Sequential(
            nn.Conv2d(3, 32, 3, padding=1), nn.BatchNorm2d(32), nn.ReLU(),
            nn.MaxPool2d(2),                        # 32x32 -> 16x16
            nn.Conv2d(32, 64, 3, padding=1), nn.BatchNorm2d(64), nn.ReLU(),
            nn.MaxPool2d(2),                        # 16x16 -> 8x8
            nn.Conv2d(64, 128, 3, padding=1), nn.BatchNorm2d(128), nn.ReLU(),
            nn.AdaptiveAvgPool2d(1),                # 8x8 -> 1x1, any input size
        )
        self.head = nn.Linear(128, num_classes)     # classify from 128 features

    def forward(self, x):
        x = self.features(x)            # (B, 128, 1, 1)
        x = torch.flatten(x, 1)         # (B, 128)
        return self.head(x)             # (B, num_classes) logits

model = SmallCNN()
print(model(torch.randn(4, 3, 32, 32)).shape)   # (4, 10)

Two patterns here are worth adopting as habits. First, channels grow as space shrinks (32→64→128 while 32×32→16×16→8×8): you trade spatial resolution for semantic richness. Second, nn.AdaptiveAvgPool2d(1) collapses whatever spatial size remains to 1×1, which makes the network accept variable input sizes and feeds a fixed-length vector to the linear head — far more robust than hard-coding a flatten size, a frequent source of shape errors.

Conv → Norm → Activation, in that order

The near-universal block ordering is convolution, then BatchNorm2d, then the activation. The conv layer usually sets bias=False when a batch-norm follows it, because the norm's learned shift makes the conv bias redundant. Memorize the triple; you'll stack it hundreds of times.

03Residual networks & the classics

Once people tried stacking many layers, they hit a wall: past a certain depth, deeper networks trained worse — not from overfitting, but because the gradient signal degraded on its way back through dozens of layers. Residual connections (ResNet, 2015) solved it with an almost trivial change: instead of a block computing y = F(x), it computes y = x + F(x). The block only has to learn the difference from its input, and — crucially — the + x gives gradients a direct highway backward, skipping the transformation entirely if need be.

import torch.nn as nn
import torch.nn.functional as F

class ResidualBlock(nn.Module):
    def __init__(self, channels):
        super().__init__()
        self.conv1 = nn.Conv2d(channels, channels, 3, padding=1, bias=False)
        self.bn1   = nn.BatchNorm2d(channels)
        self.conv2 = nn.Conv2d(channels, channels, 3, padding=1, bias=False)
        self.bn2   = nn.BatchNorm2d(channels)

    def forward(self, x):
        identity = x                          # save the input
        out = F.relu(self.bn1(self.conv1(x)))
        out = self.bn2(self.conv2(out))
        out = out + identity                  # the residual "skip" connection
        return F.relu(out)                    # activate AFTER adding

This one idea unlocked networks hundreds of layers deep and is now everywhere — including inside every transformer block in Part III. It's arguably the most important architectural primitive of the last decade.

The lineage in one breath: LeNet (1998) proved convs work on digits; AlexNet (2012) scaled them to ImageNet on GPUs and started the deep-learning boom; VGG (2014) showed depth with uniform 3×3 kernels; ResNet (2015) made real depth trainable. Modern vision backbones (Volume VI) are refinements on this arc — you rarely implement them by hand, but you'll recognize their parts.

Add the input back to the output, and suddenly a hundred layers train as easily as ten.

04Transfer learning & fine-tuning

You almost never train a vision model from scratch. A network pretrained on a huge dataset (ImageNet) has already learned general visual features — edges, textures, parts — in its early layers, and those transfer to almost any image task. Transfer learning reuses that backbone and adapts only what's task-specific, which turns a project that would need a million labeled images into one that works with a few thousand.

Two modes, chosen by how much data you have:

Feature extraction (little data)

Freeze the whole pretrained backbone, replace the final classification layer, and train only that new head. Fast, cheap, and hard to overfit because almost nothing is learnable.

Fine-tuning (more data)

Replace the head, then unfreeze some or all of the backbone and train it too — at a low learning rate so you nudge the good pretrained weights rather than wrecking them.

import torch
import torch.nn as nn
from torchvision import models

# Load a model pretrained on ImageNet.
model = models.resnet50(weights=models.ResNet50_Weights.DEFAULT)

# --- feature extraction: freeze everything first ---
for p in model.parameters():
    p.requires_grad = False

# Replace the 1000-class head with one for OUR task. New layers default to
# requires_grad=True, so only these train.
num_features = model.fc.in_features            # 2048 for resnet50
model.fc = nn.Linear(num_features, 10)         # 10 classes

# Give the optimizer ONLY the parameters that require grad.
opt = torch.optim.AdamW(
    (p for p in model.parameters() if p.requires_grad), lr=1e-3)

# --- to fine-tune later: unfreeze and use a much smaller LR ---
# for p in model.parameters(): p.requires_grad = True
# opt = torch.optim.AdamW(model.parameters(), lr=1e-5)
Match the preprocessing to the pretrained model

A pretrained network expects inputs normalized exactly as it was trained — the ImageNet mean/std from Volume II, and the right input size. Feed it differently-scaled images and its learned features misfire. The weights object carries the correct transforms: use weights.transforms() to get them rather than guessing.

Start from someone else's millions of images. Transfer learning is the highest-leverage move in applied deep learning: a frozen pretrained backbone plus a fresh head, trained on a few thousand examples, routinely beats a bespoke model trained from scratch on the same data. Reach for a pretrained model first; justify training from scratch only when your domain is genuinely unlike anything pretrained.

Part II — Sequence models

Data that comes in order.

Text, audio, time series, and DNA share a trait images don't: order matters, and length varies. Handling them needs two new ideas — a way to turn discrete tokens into vectors (embeddings), and a way to carry information along the sequence (recurrence, then attention). This part walks that path from embeddings through LSTMs to the doorstep of the transformer.

05Embeddings: representing discrete things

Neural networks eat vectors of numbers, but a word, a user ID, or a product category is a discrete symbol with no natural numeric value. The naive fix — a one-hot vector, all zeros but a single 1 — is hopeless at scale: a 50,000-word vocabulary means 50,000-dimensional, almost-entirely-zero inputs, and every word sits equidistant from every other, encoding no notion of similarity.

An embedding replaces that with a learned, dense vector per symbol. nn.Embedding is literally a lookup table of shape (vocab_size, embed_dim): give it a token's integer ID and it returns that row, and those rows are trained by gradient descent like any other parameter. The network discovers, on its own, to place similar tokens near each other.

import torch
import torch.nn as nn

vocab_size, embed_dim = 10_000, 128
embed = nn.Embedding(vocab_size, embed_dim)   # a (10000, 128) learnable table

# A batch of 3 sentences, each 5 tokens, as integer IDs:
tokens = torch.randint(0, vocab_size, (3, 5))  # (batch=3, seq_len=5)
vectors = embed(tokens)                         # -> (3, 5, 128)
print(vectors.shape)   # each token ID replaced by its 128-dim learned vector

Note the shape transformation: (B, T) integers become (B, T, embed_dim) floats — a batch of sequences of vectors, which is exactly what recurrent layers and transformers consume. Embeddings aren't only for text: any high-cardinality categorical feature (users, items, locations) is best fed to a network through an embedding table. The learned vectors are themselves useful — Chapter 15 returns to them as representations.

06RNNs & the vanishing-gradient problem

To process a sequence in order, a recurrent neural network maintains a hidden state — a running summary — and updates it one element at a time: h_t = f(h_{t−1}, x_t). The same weights f are applied at every time step (weight sharing across time), and the final hidden state is a compressed memory of everything seen. Conceptually you "unroll" the loop across the sequence and backpropagate through it (called backpropagation through time).

AN RNN UNROLLED OVER A SEQUENCE: x1 x2 x3 x4 (inputs, one per step) │ │ │ │ ▼ ▼ ▼ ▼ ┌────┐ ┌────┐ ┌────┐ ┌────┐ │ f │──► │ f │──► │ f │──► │ f │──► h4 (final "memory") └────┘ h1 └────┘ h2 └────┘ h3 └────┘ same weights reused at every step; state h carries the past forward

The trouble is long-range memory. Gradients flowing back through many steps get multiplied by the same factors over and over; if those factors are slightly less than one they shrink toward zero (vanishing gradients), and the network effectively forgets anything more than a handful of steps back. Slightly greater than one and they blow up (exploding gradients, which gradient clipping from Volume II tames). Vanilla RNNs therefore struggle to connect information across long spans — the very thing sequences often demand.

Plain RNNs are mostly of historical interest

You'll rarely deploy a vanilla nn.RNN today — it's the conceptual stepping stone. Its gated descendants (next chapter) fixed the memory problem, and attention (Part III) largely replaced recurrence altogether for the biggest tasks. Understand the RNN to understand why the others exist.

07LSTM & GRU

The fix for vanishing gradients is gating: instead of overwriting the memory every step, learn to control what gets remembered, forgotten, and output. The LSTM (Long Short-Term Memory) adds a separate cell state — a memory highway that information can travel along nearly unchanged — governed by three gates: a forget gate (what to discard), an input gate (what to add), and an output gate (what to expose). Because the cell state is edited additively rather than multiplied through a squashing function, gradients survive across long spans.

The GRU (Gated Recurrent Unit) is a streamlined cousin with two gates and no separate cell state — fewer parameters, often just as good. In PyTorch both are one layer that processes an entire sequence at once, in optimized C++/CUDA:

import torch
import torch.nn as nn

lstm = nn.LSTM(input_size=128, hidden_size=256,
               num_layers=2, batch_first=True, dropout=0.2)

x = torch.randn(32, 50, 128)     # (batch, seq_len, features) with batch_first
output, (h_n, c_n) = lstm(x)
print(output.shape)   # (32, 50, 256): hidden state at EVERY time step
print(h_n.shape)      # (2, 32, 256): final hidden state per layer
print(c_n.shape)      # (2, 32, 256): final cell state per layer

Two return values matter. output is the hidden state at every time step — use it when you need a prediction per element (tagging each word, forecasting each future point). h_n is just the final states — use it when you need one summary of the whole sequence (classifying a sentence's sentiment). Set batch_first=True so tensors are the intuitive (batch, time, features); the default puts time first and trips everyone up.

08Batching sequences: padding & masking

Images in a batch share a shape; sentences don't. To stack sequences of different lengths into one tensor, you pad the short ones with a filler token up to the batch's longest length — and then you must mask so the model and the loss ignore that padding, or it will happily learn from meaningless filler.

import torch
from torch.nn.utils.rnn import pad_sequence

# Three sequences of different lengths (as embedded vectors or IDs):
seqs = [torch.randn(4, 8), torch.randn(7, 8), torch.randn(3, 8)]  # len 4,7,3
batch = pad_sequence(seqs, batch_first=True)   # -> (3, 7, 8), short ones 0-padded

# A boolean mask marking REAL positions (True) vs padding (False):
lengths = torch.tensor([4, 7, 3])
mask = torch.arange(7)[None, :] < lengths[:, None]   # (3, 7)
print(batch.shape, mask.shape)

That mask is the workhorse. For a loss, you compute it only over real positions (or pass ignore_index to CrossEntropyLoss so a designated pad label contributes nothing). For RNNs specifically, pack_padded_sequence lets the layer skip padded steps entirely for both speed and correctness. For transformers (next part), the same mask becomes the attention mask that stops a token from attending to padding.

Unmasked padding silently poisons training

Forget the mask and the model treats pad tokens as real data: the loss rewards predicting padding, attention spreads onto it, and metrics quietly degrade with no error thrown. Whenever sequences vary in length, the mask is not optional — it's part of the correctness of the model.

Recurrence processes a sequence strictly left to right, which is inherently serial and slow, and — even with gates — strains to link very distant elements. The next part introduces the mechanism that fixed both at once, and reshaped the field around it.

Part III — Attention & transformers

The idea that ate the field.

Attention lets every element of a sequence look directly at every other element and pull in what's relevant — no step-by-step recurrence, no distance penalty, and fully parallel. The transformer, built almost entirely from attention, now powers language models, vision, audio, and more. This part builds it from the dot product up; Volume V scales it into a GPT.

09Attention from scratch

The core question attention answers: for a given element, which other elements matter, and how much? It formalizes this with three learned projections of each element — a query (what am I looking for?), a key (what do I offer?), and a value (what will I contribute if chosen?). An element's query is compared against every key to score relevance; those scores become weights; and the output is the weighted sum of the values. Relevant elements dominate the sum; irrelevant ones fade.

SCALED DOT-PRODUCT ATTENTION FOR ONE QUERY: query q · k1 k2 k3 k4 → scores → softmax → weights 0.1 0.6 0.2 0.1 │ │ │ │ output = 0.1·v1 + 0.6·v2 + 0.2·v3 + 0.1·v4 ◄─────────┴───┴───┴───┘ (a blend of the values, weighted by how relevant each key was to q)

The whole operation for a sequence is three matrix multiplies and a softmax. The one non-obvious detail is the √d divisor: dot products grow with the vector dimension, and without rescaling they'd push the softmax into a near-one-hot regime where gradients vanish. Dividing by the square root of the key dimension keeps the scores in a sane range.

import torch
import torch.nn.functional as F

def attention(Q, K, V, mask=None):
    # Q,K,V: (batch, seq_len, d).  scores: (batch, seq_len, seq_len)
    d = Q.size(-1)
    scores = (Q @ K.transpose(-2, -1)) / d ** 0.5    # relevance of every pair
    if mask is not None:
        scores = scores.masked_fill(mask == 0, float("-inf"))  # block positions
    weights = F.softmax(scores, dim=-1)              # each row sums to 1
    return weights @ V                               # weighted blend of values

x = torch.randn(2, 5, 64)              # batch 2, 5 tokens, d=64
out = attention(x, x, x)               # self-attention: Q=K=V=x
print(out.shape)                       # (2, 5, 64)

Because attention compares every position with every other in one matrix multiply, it connects distant elements at constant path length — the long-range problem that crippled RNNs simply doesn't arise — and it's massively parallel, which is why transformers train so efficiently on modern hardware.

10Self-attention & multi-head

When the queries, keys, and values all come from the same sequence, it's self-attention: every token gathers context from every other token in its own sequence. That's the engine of the transformer. But a single attention computes just one kind of relationship. Multi-head attention runs several attentions in parallel — each with its own learned Q/K/V projections into a lower-dimensional subspace — so different heads can specialize (one tracks syntax, another long-range reference, another local adjacency). Their outputs are concatenated and projected back.

import torch
import torch.nn as nn
import torch.nn.functional as F

class MultiHeadSelfAttention(nn.Module):
    def __init__(self, dim, n_heads):
        super().__init__()
        assert dim % n_heads == 0
        self.n_heads, self.head_dim = n_heads, dim // n_heads
        self.qkv = nn.Linear(dim, dim * 3)   # project to Q, K, V at once
        self.proj = nn.Linear(dim, dim)      # final output projection

    def forward(self, x, mask=None):
        B, T, D = x.shape
        # make Q,K,V, then split the channels into heads:
        qkv = self.qkv(x).reshape(B, T, 3, self.n_heads, self.head_dim)
        q, k, v = qkv.permute(2, 0, 3, 1, 4)   # each: (B, n_heads, T, head_dim)
        # PyTorch's fused, memory-efficient attention (uses FlashAttention):
        out = F.scaled_dot_product_attention(q, k, v, attn_mask=mask)
        out = out.transpose(1, 2).reshape(B, T, D)   # merge heads back
        return self.proj(out)

In production you rarely hand-roll the softmax from Chapter 09. F.scaled_dot_product_attention is a single fused kernel that computes the whole thing far faster and with much less memory (it dispatches to FlashAttention on capable GPUs) — Volume IV and V lean on it heavily. Learn the manual version for understanding; call the fused one for speed.

Causal masking makes it a generator

For language modeling, a token must not peek at future tokens. A causal mask sets the attention scores for all future positions to −∞ before the softmax, so each position attends only to itself and earlier ones. That single mask is the difference between a bidirectional encoder (BERT-style) and an autoregressive decoder (GPT-style) — the subject of Volume V.

11The transformer block

A transformer is a stack of identical blocks, each two sub-layers wrapped in the residual-plus-normalization pattern from Volume II. The first sub-layer is multi-head self-attention (tokens exchange information); the second is a small position-wise feed-forward network — two linear layers with an activation — that processes each token independently. Both sub-layers get a residual skip and a LayerNorm. Modern transformers put the norm before each sub-layer ("pre-norm"), which trains more stably than the original "post-norm" design.

import torch.nn as nn

class TransformerBlock(nn.Module):
    def __init__(self, dim, n_heads, mlp_ratio=4, p=0.1):
        super().__init__()
        self.norm1 = nn.LayerNorm(dim)
        self.attn  = MultiHeadSelfAttention(dim, n_heads)
        self.norm2 = nn.LayerNorm(dim)
        self.mlp = nn.Sequential(               # position-wise feed-forward
            nn.Linear(dim, dim * mlp_ratio), nn.GELU(),
            nn.Linear(dim * mlp_ratio, dim), nn.Dropout(p),
        )

    def forward(self, x, mask=None):
        x = x + self.attn(self.norm1(x), mask)  # pre-norm + residual
        x = x + self.mlp(self.norm2(x))         # pre-norm + residual
        return x

One thing is missing, and it's essential: attention is permutation-invariant — shuffle the tokens and, since every token attends to every other symmetrically, the outputs just shuffle too. The block has no inherent sense of order. So we inject position explicitly with a positional encoding added to the token embeddings before the first block — either fixed sinusoids or learned position vectors (Volume V covers the modern rotary variant, RoPE). Without it, "dog bites man" and "man bites dog" look identical.

12Build a small transformer

Assembled, a transformer classifier is: embed tokens, add positions, run a stack of blocks, pool to one vector, classify. Here it is end to end — a complete, trainable sequence classifier that is, in miniature, the same machine as a large language model.

import torch
import torch.nn as nn

class TinyTransformer(nn.Module):
    def __init__(self, vocab, dim=128, n_heads=4, depth=4,
                 num_classes=2, max_len=512):
        super().__init__()
        self.tok = nn.Embedding(vocab, dim)
        self.pos = nn.Parameter(torch.randn(1, max_len, dim) * 0.02)  # learned
        self.blocks = nn.ModuleList(
            [TransformerBlock(dim, n_heads) for _ in range(depth)])
        self.norm = nn.LayerNorm(dim)
        self.head = nn.Linear(dim, num_classes)

    def forward(self, ids, mask=None):
        T = ids.size(1)
        x = self.tok(ids) + self.pos[:, :T]     # embed + position
        for blk in self.blocks:
            x = blk(x, mask)
        x = self.norm(x).mean(dim=1)            # mean-pool over tokens
        return self.head(x)                     # (B, num_classes) logits

model = TinyTransformer(vocab=10_000)
ids = torch.randint(0, 10_000, (8, 50))         # batch 8, 50 tokens
print(model(ids).shape)                          # (8, 2)

For real work you can also assemble this from PyTorch's built-in nn.TransformerEncoderLayer and nn.TransformerEncoder rather than hand-writing the block — but building it once, as here, is what makes the built-ins (and Volume V's GPT) legible.

One block, stacked, is most of modern AI. The transformer you just built — embed, add position, attention-and-MLP with residuals, repeat — is architecturally the same as a frontier language model; the differences are scale, a causal mask, and training data. Understanding this 40-line model is the foundation on which Volume V's LLMs and Volume VI's vision transformers both stand.

Attention plus a feed-forward, wrapped in residuals and repeated — that stack is the workhorse of the decade.

Part IV — Heads, losses & representations

From backbone to working model.

A CNN or transformer is a backbone that turns raw input into rich features. What makes it solve a task is the small head on top and the loss that scores it. This part catalogs the heads and losses you'll wire onto any backbone, then steps back to the deeper payoff: the features themselves are reusable representations — the idea that powers retrieval, transfer, and the generative models that close the series.

13Task heads

Almost every model factors into a backbone (produces a feature vector) and a head (maps features to the task's output shape). Swapping the head — and its loss — is how one backbone serves many tasks, and it's exactly what transfer learning exploited in Chapter 04.

import torch.nn as nn

feat_dim = 512   # whatever the backbone outputs

# Multi-class classification: one logit per class.
clf_head = nn.Linear(feat_dim, num_classes)          # + CrossEntropyLoss

# Regression: raw real number(s), no activation.
reg_head = nn.Linear(feat_dim, 1)                    # + MSELoss

# Multi-label (each label independent): one logit per label.
multilabel_head = nn.Linear(feat_dim, num_labels)   # + BCEWithLogitsLoss

# Per-token / sequence labeling: apply a linear at EVERY position.
token_head = nn.Linear(feat_dim, num_tags)          # over (B, T, feat_dim)

The pattern generalizes to structured outputs. Dense prediction (a label per pixel, Volume VI) uses a head that upsamples back to image resolution; detection uses heads that emit boxes and class scores. But the principle is constant: the backbone learns what's there, the head expresses it in the shape the task needs, and the loss grades that shape.

Output raw logits, let the loss do the rest

Classification heads should emit unbounded scores (logits), not probabilities — CrossEntropyLoss and BCEWithLogitsLoss apply softmax/sigmoid internally in a numerically stable way. Add your own softmax before them and you get a subtly wrong, hard-to-debug loss. Apply the activation only at inference, when you actually want probabilities.

14The loss-function catalog

Choosing a loss is choosing what "wrong" means (Volume I, Chapter 13). Match it to the task and the output format, and mind the input conventions — most beginner loss bugs are a dtype or a logits-vs-probabilities mismatch.

worked case "0D or 1D target tensor expected" from CrossEntropyLoss
The setup

A multi-class model errors at the loss, or trains but never learns, because the targets were one-hot encoded as floats.

The fix

CrossEntropyLoss wants class indices, not one-hot: a target of shape (N,) and dtype int64, where each value is the correct class number. If your labels are one-hot, convert with labels.argmax(dim=1). And pass logits, shape (N, C) — never softmaxed probabilities.

The lesson

When a loss errors, print pred.shape, pred.dtype, target.shape, target.dtype and check them against the loss's documented contract. Ninety percent of loss bugs are shape or dtype mismatches, not math.

15Representation learning & embeddings

Here's the shift in perspective that unlocks a lot of modern practice: the features a trained backbone produces — the vector just before the head — are valuable in their own right. A well-trained network maps inputs into a space where geometry means something: similar inputs land near each other, and directions can encode attributes. That vector is an embedding of the input, and you can use it far beyond the original task.

  • Similarity & retrieval. Embed everything, then find nearest neighbors by cosine similarity — the basis of semantic search, recommendation, and deduplication.
  • Transfer. Freeze the backbone and train a tiny head on a new task (Chapter 04) — you're reusing the representation.
  • Clustering & visualization. Group or plot (with UMAP/t-SNE) the embeddings to see structure the labels never named.
import torch.nn.functional as F

# Cosine similarity between two embedding vectors: 1 = identical direction.
def cosine_sim(a, b):
    return F.cosine_similarity(a, b, dim=-1)

# Retrieval: find the closest database vectors to a query embedding.
query = model.backbone(x)               # (1, D)
db    = model.backbone(corpus)          # (N, D)
scores = F.cosine_similarity(query, db) # (N,)
top5 = scores.topk(5).indices           # the 5 most similar items

You can also train specifically to shape this space. Metric learning losses — contrastive and triplet — take pairs or triplets and explicitly pull matching items together while pushing mismatches apart, producing embeddings tuned for similarity search rather than classification. This same instinct — learn a good representation without task labels — is the heart of self-supervised learning, which Volume VI develops for vision.

The representation often outlasts the task. Models get retrained and replaced, but a good embedding space is reusable infrastructure — it powers search, recommendation, transfer, and clustering long after the head that trained it is gone. When you build a model, ask not only "does it classify well?" but "is the space it learned useful?"

16Autoencoders: a first generative taste

Everything so far mapped input to a label. An autoencoder maps input back to itself, through a narrow bottleneck: an encoder compresses the input to a small latent vector, a decoder reconstructs the input from it, and the loss is reconstruction error. With no labels at all, the network is forced to discover an efficient encoding — the essence of the data — because the bottleneck won't fit anything wasteful.

import torch
import torch.nn as nn

class Autoencoder(nn.Module):
    def __init__(self, in_dim=784, latent=32):
        super().__init__()
        self.encoder = nn.Sequential(
            nn.Linear(in_dim, 128), nn.ReLU(),
            nn.Linear(128, latent),                 # the bottleneck
        )
        self.decoder = nn.Sequential(
            nn.Linear(latent, 128), nn.ReLU(),
            nn.Linear(128, in_dim), nn.Sigmoid(),   # back to pixel range [0,1]
        )

    def forward(self, x):
        z = self.encoder(x)          # compress to latent code
        return self.decoder(z), z    # reconstruction, and the code itself

model = Autoencoder()
recon, z = model(torch.rand(16, 784))
loss = nn.functional.mse_loss(recon, torch.rand(16, 784))   # reconstruction loss

Autoencoders are useful directly — for denoising (train to reconstruct a clean input from a corrupted one), anomaly detection (things that reconstruct badly are outliers), and dimensionality reduction. But their deeper importance is conceptual: they're the gateway to generative models. Constrain the latent space to a smooth probability distribution and the decoder becomes a sampler that generates new data — the variational autoencoder — and from there the line runs to GANs and diffusion models.
