Part I — Building models

From one layer to real networks.

A model is just many small functions composed together, each with its own parameters. PyTorch's nn.Module is the container that makes composition clean: it tracks every parameter automatically, moves them to a device together, and saves them as one unit. Master the module and you can assemble any architecture in the series from parts.

01nn.Module in depth

Everything you build in PyTorch is an nn.Module: a single layer, a block of layers, or an entire hundred-layer network — all the same type, nestable inside each other. A module bundles two things: state (its learnable parameters and any fixed buffers) and behavior (a forward method that says how to turn inputs into outputs).

You define one by subclassing nn.Module, creating child modules and parameters in __init__, and describing the computation in forward. Here is Linear rebuilt from scratch to expose the machinery:

import torch
import torch.nn as nn

class MyLinear(nn.Module):
    def __init__(self, in_features, out_features):
        super().__init__()                 # ALWAYS call this first
        # nn.Parameter marks a tensor as learnable -> auto-registered.
        self.weight = nn.Parameter(torch.randn(out_features, in_features) * 0.01)
        self.bias = nn.Parameter(torch.zeros(out_features))

    def forward(self, x):                   # x: (batch, in_features)
        return x @ self.weight.t() + self.bias   # -> (batch, out_features)

layer = MyLinear(4, 3)
out = layer(torch.randn(2, 4))              # call the module, NOT .forward()
print(out.shape)                            # torch.Size([2, 3])

Two rules do a lot of quiet work here. First, assigning an nn.Parameter (or a child nn.Module) as an attribute registers it automatically — PyTorch now knows to track it, include it in parameters(), move it with .to(device), and save it in the checkpoint. Second, you call the module (layer(x)), never layer.forward(x) directly, because __call__ runs hooks and bookkeeping around your forward.

Inspecting a module's state

Because registration is automatic, you can always enumerate what a module contains — indispensable for debugging, freezing layers, and counting parameters.

for name, p in layer.named_parameters():
    print(name, tuple(p.shape), p.requires_grad)
# weight (3, 4) True
# bias   (3,)   True

n_params = sum(p.numel() for p in layer.parameters())   # total learnable numbers
print("params:", n_params)                              # 15

layer.to("cuda")     # moves weight AND bias together, in one call

Buffers: state that isn't learned

Some modules need persistent state that is not a parameter — it's part of the model and must be saved and moved with it, but the optimizer must never update it. Batch-norm's running mean and variance are the canonical example. Register these with register_buffer:

class RunningMean(nn.Module):
    def __init__(self, size):
        super().__init__()
        self.register_buffer("total", torch.zeros(size))  # saved & moved,
        self.register_buffer("count", torch.zeros(1))     # but not a parameter

    def forward(self, x):
        self.total += x.sum(dim=0)
        self.count += x.size(0)
        return self.total / self.count
Parameter vs buffer, in one line

If gradient descent should update it, make it an nn.Parameter. If it's stateful but learned by some other rule (or fixed), make it a buffer. Both are saved in state_dict() and moved by .to(); only parameters appear in .parameters() and get gradients.

02Layers, activations & your first MLP

You rarely write layers from scratch — PyTorch ships them. The one you'll use most is nn.Linear (a matrix multiply plus bias, the "fully-connected" layer). Between linear layers you insert a nonlinear activation function, and that nonlinearity is non-negotiable: stack linear layers with nothing between them and the whole stack algebraically collapses to a single linear layer. The activation is what lets depth buy you expressive power.

ActivationWhat it does · when to use
nn.ReLUmax(0, x). The default workhorse — cheap, effective. Kills negatives, which can occasionally "die," but rarely a problem in practice.
nn.GELUA smooth ReLU. The default inside transformers (Volumes III, V); slightly better in deep nets, marginally costlier.
nn.SiLU / SwishAnother smooth gate, common in modern CNNs (EfficientNet). Interchangeable with GELU in practice.
nn.SigmoidSquashes to (0,1). Use only for binary output probabilities, not hidden layers (it saturates and stalls gradients).
nn.TanhSquashes to (−1,1). Historically common in RNNs; mostly superseded by ReLU-family for feed-forward nets.

Composing layers: Sequential and ModuleList

nn.Sequential chains modules so the output of each feeds the next — perfect for a simple straight-through stack. nn.ModuleList holds a list of modules you call yourself in forward, for when the flow isn't a simple chain.

# A multi-layer perceptron (MLP): the classic first neural network.
# Input: a flattened 28x28 image (784 numbers). Output: 10 class scores.
mlp = nn.Sequential(
    nn.Linear(784, 256),   # 784 -> 256
    nn.ReLU(),             # nonlinearity
    nn.Linear(256, 128),   # 256 -> 128
    nn.ReLU(),
    nn.Linear(128, 10),    # 128 -> 10 class logits (no activation on output)
)

x = torch.randn(64, 784)   # a batch of 64 flattened images
logits = mlp(x)            # -> (64, 10): ten raw scores per image
print(logits.shape)

Note the output has no activation: for classification you emit raw scores called logits, and the loss function (nn.CrossEntropyLoss) applies the softmax internally for numerical stability. Adding a softmax yourself before cross-entropy is a classic double-application bug.

Flatten before a Linear

nn.Linear expects its input's last dimension to match in_features. Image data arrives as (batch, channels, height, width); you must flatten it to (batch, features) first — nn.Flatten() as the first layer, or x.view(x.size(0), -1) in forward. A shape mismatch here is the #1 first-MLP error.

03Weight initialization

Before training, every parameter needs a starting value, and the choice matters more than beginners expect. Initialize weights too large and the signal amplifies layer over layer until activations and gradients explode to NaN; too small and the signal shrinks to nothing and the network can't learn. Good initialization keeps the scale of activations and gradients roughly constant as they pass through the network.

The good news: PyTorch's built-in layers already initialize themselves sensibly (nn.Linear and the conv layers use a Kaiming-style scheme by default). You often don't touch it. But when you build custom modules, or want to match a paper, you set it explicitly with nn.init.

import torch.nn as nn

def init_weights(m):
    if isinstance(m, nn.Linear):
        # Kaiming (He) init: tuned for ReLU-family activations.
        nn.init.kaiming_normal_(m.weight, nonlinearity="relu")
        if m.bias is not None:
            nn.init.zeros_(m.bias)      # biases almost always start at 0

model = nn.Sequential(nn.Linear(784, 256), nn.ReLU(), nn.Linear(256, 10))
model.apply(init_weights)             # applies the fn to every submodule

The two schemes you'll meet by name encode the same idea for different activations. Xavier/Glorot init assumes a symmetric activation (tanh, sigmoid); Kaiming/He init accounts for ReLU zeroing out half the values, so it scales up to compensate. The rule of thumb: ReLU-family activations → Kaiming; tanh/sigmoid → Xavier.

Initialization sets the stage; it doesn't run the play. Good init makes training possible and fast to start, but it's a one-time scale-setting, not a substitute for normalization layers (Chapter 15) which keep activations well-behaved throughout training. Reach for explicit init when you write custom layers or reproduce a paper; trust the defaults otherwise.

04A reusable model pattern

Two ways exist to express the same computation: the module API (nn.ReLU() — an object you store) and the functional API (F.relu(x) — a plain function). Stateless operations like activations and pooling can go either way; stateful ones with parameters (linear, conv, batch-norm) should be modules so their parameters register. The idiomatic pattern uses modules for anything with weights and functional calls for the stateless glue in forward.

import torch
import torch.nn as nn
import torch.nn.functional as F

class MLP(nn.Module):
    def __init__(self, in_dim=784, hidden=256, out_dim=10, p_drop=0.2):
        super().__init__()
        self.fc1 = nn.Linear(in_dim, hidden)   # stateful -> module
        self.fc2 = nn.Linear(hidden, out_dim)
        self.drop = nn.Dropout(p_drop)         # stateful behavior -> module

    def forward(self, x):
        x = torch.flatten(x, 1)                # (B,1,28,28) -> (B,784)
        x = F.relu(self.fc1(x))                # stateless relu -> functional
        x = self.drop(x)
        return self.fc2(x)                     # logits, no final activation

model = MLP()
print(model)                                   # prints the module tree
print("params:", sum(p.numel() for p in model.parameters()))

This subclass-and-forward template is the shape of virtually every model in this series. Its virtues: the architecture is readable top to bottom, hyperparameters are constructor arguments (so you can sweep them), and print(model) gives you the whole tree. When something is the wrong shape, you add a print(x.shape) line in forward and run once — because it's just Python, there's nothing more exotic to debug.

Make the constructor configurable

Pass sizes and rates as arguments with defaults (hidden=256, p_drop=0.2) rather than hard-coding them. Future-you tuning hyperparameters, and Volume IV scaling this model up, will both thank you. A model whose shape is data, not literals, is a model you can experiment on.

Part II — The data pipeline

Feeding the model without starving it.

A GPU that finishes a batch and waits for the next one is money burning. PyTorch splits data handling into two clean pieces — a Dataset that knows how to fetch one example, and a DataLoader that batches, shuffles, and loads many in parallel. Get this pipeline right and your expensive compute stays fed; get it wrong and no model tweak will save your throughput.

05Dataset & DataLoader

A Dataset answers two questions about your data: how many examples are there (__len__), and give me example number i (__getitem__). That's the entire contract. It says nothing about batching or shuffling — those are the DataLoader's job — which keeps each piece simple.

import torch
from torch.utils.data import Dataset, DataLoader

class PointsDataset(Dataset):
    def __init__(self, X, y):
        self.X, self.y = X, y            # tensors, arrays, or paths to files

    def __len__(self):
        return len(self.X)               # how many examples

    def __getitem__(self, i):
        return self.X[i], self.y[i]      # ONE (input, target) pair

X = torch.randn(1000, 20); y = torch.randint(0, 2, (1000,))
ds = PointsDataset(X, y)
print(len(ds), ds[0][0].shape)           # 1000, torch.Size([20])

The DataLoader wraps a dataset and turns single examples into shuffled mini-batches, iterable in a for loop. It's what Volume I's loop consumed.

loader = DataLoader(
    ds,
    batch_size=64,      # examples per batch
    shuffle=True,       # reshuffle every epoch (train only!)
    drop_last=False,    # keep the smaller final batch
)

for xb, yb in loader:            # each iteration yields one batch
    print(xb.shape, yb.shape)    # (64, 20)  (64,)  -- stacked automatically
    break

The loader collates a list of individual examples into batched tensors by stacking along a new first dimension — the batch dimension, by convention always first. For unusual data (variable-length sequences, say), you supply a collate_fn to control that stacking; Volume III uses one for padding sequences. For the common case of fixed-shape tensors, the default just works. And when your data already lives in tensors, skip the custom class entirely:

from torch.utils.data import TensorDataset
ds = TensorDataset(X, y)         # wraps tensors as a Dataset in one line

06Transforms & augmentation

Raw data rarely enters the model unchanged. Two kinds of processing happen in __getitem__: preprocessing that every example needs (convert to tensor, normalize to a standard scale) and augmentation that randomly perturbs training examples to combat overfitting. Both are expressed as transforms — small callables you compose into a pipeline. For images, use the modern torchvision.transforms.v2 API.

from torchvision.transforms import v2

# TRAIN pipeline: augment (random) + preprocess (deterministic).
train_tf = v2.Compose([
    v2.RandomResizedCrop(224),          # random crop + resize -> more views
    v2.RandomHorizontalFlip(p=0.5),     # mirror half the time
    v2.ToDtype(torch.float32, scale=True),   # uint8 [0,255] -> float [0,1]
    v2.Normalize(mean=[0.485, 0.456, 0.406], # per-channel standardization
                 std=[0.229, 0.224, 0.225]),  # (ImageNet statistics)
])

# VAL/TEST pipeline: preprocess ONLY. No randomness at evaluation.
eval_tf = v2.Compose([
    v2.Resize(256), v2.CenterCrop(224),
    v2.ToDtype(torch.float32, scale=True),
    v2.Normalize(mean=[0.485, 0.456, 0.406], std=[0.229, 0.224, 0.225]),
])

Normalization rescales each input feature (here, each color channel) to roughly zero mean and unit variance. Models train faster and more stably when inputs are on a consistent, modest scale — it keeps the optimization landscape well-conditioned. The mean/std above are the standard ImageNet statistics; for your own data, compute them on the training set only (Chapter 08 explains why).

Augmentation is free regularization: by showing the model a randomly cropped, flipped, color-jittered version of each image every epoch, you multiply the effective size of your dataset and force the model to learn features that survive those perturbations rather than memorizing exact pixels.

Never augment the validation or test set

Augmentation is a training-time trick to improve generalization. Applying random crops or flips at evaluation makes your metrics noisy and not comparable across runs. Evaluation transforms must be deterministic — resize, center-crop, normalize, nothing random.

07Fast input pipelines

By default the DataLoader fetches data in the main process, one batch at a time, between training steps — so your GPU sits idle while the CPU reads and transforms the next batch. The fix is to load in parallel background workers that prepare upcoming batches while the GPU is busy on the current one.

loader = DataLoader(
    ds,
    batch_size=64,
    shuffle=True,
    num_workers=8,            # background processes fetching batches in parallel
    pin_memory=True,          # stage batches in page-locked RAM -> faster to GPU
    persistent_workers=True,  # keep workers alive across epochs (avoid respawn)
    prefetch_factor=2,        # each worker preloads this many batches ahead
)

Then, in the loop, copy each batch to the GPU with non_blocking=True so the transfer overlaps with computation instead of stalling it:

for xb, yb in loader:
    xb = xb.to(device, non_blocking=True)   # overlaps with GPU work
    yb = yb.to(device, non_blocking=True)   # (pairs with pin_memory=True)
    # ... forward / loss / backward / step ...
KnobWhat it buys · how to set it
num_workersParallel loading. Start around the number of CPU cores; too many can thrash. 0 means load in the main process (simplest, slowest, easiest to debug).
pin_memoryFaster CPU→GPU copies. Turn on when training on a GPU; pairs with non_blocking=True.
persistent_workersAvoids the cost of respawning workers each epoch. Set True when num_workers > 0.
prefetch_factorHow far ahead workers read. Raise it if the GPU still waits; costs memory.
Diagnose before you tune

If GPU utilization (watch nvidia-smi) sits low and spiky, you're input-bound — the pipeline, not the model, is the bottleneck, and the knobs above are your fix. If it's pinned near 100%, data loading isn't your problem; look to the model or batch size instead. Volume IV profiles this properly.

08Splits, leakage & reproducibility

You split data into three parts, each with a distinct job. Train fits the parameters. Validation is checked during development to tune hyperparameters and decide when to stop. Test is touched exactly once, at the very end, to estimate real-world performance. The moment you use the test set to make decisions, it stops being an honest estimate.

from torch.utils.data import random_split

g = torch.Generator().manual_seed(42)          # reproducible split
n = len(full_ds)
n_val = int(0.15 * n); n_test = int(0.15 * n)
n_train = n - n_val - n_test
train_ds, val_ds, test_ds = random_split(
    full_ds, [n_train, n_val, n_test], generator=g)

Data leakage: the silent metric-inflater

Leakage is when information from outside the training set sneaks into training, making your validation and test numbers optimistic — sometimes wildly so — and then production performance craters. It is the single most common way honest-looking projects fail. The usual culprits:

  • Fitting preprocessing on all the data. Compute normalization mean/std, vocabulary, or feature scalers on the training set only, then apply those fixed statistics to val and test. Fit them on the full dataset and you've leaked test information into training.
  • Duplicates or near-duplicates across splits. The same image (or an augmented copy, or the same patient's two scans) in both train and test lets the model "recognize" test examples. De-duplicate, and split by group (patient, user, document) when examples cluster.
  • Temporal leakage. For time-ordered data, training on the future to predict the past is cheating. Split by time: train on earlier, validate on later.
worked case 98% validation accuracy that collapsed to 61% in production
The setup

A classifier scored 98% on validation but barely beat a coin flip once deployed. The pipeline normalized features using statistics computed over the entire dataset before splitting.

The investigation

Because the mean and standard deviation were computed over train + val together, every validation example had already influenced its own normalization. The model was implicitly told something about the validation distribution during training.

The fix

Split first, compute normalization statistics on the training split alone, and apply those exact numbers to validation and test. Validation accuracy dropped to a truthful 62% — which matched production, and could then be honestly improved.

The lesson

Any quantity fit from data — scalers, vocabularies, imputation values, class weights — must be learned on train only. If validation looks too good to be true, suspect leakage before you celebrate.

Finally, reproducibility: seed the split generator (above), seed the framework (torch.manual_seed), and for fully deterministic worker shuffling, pass a seeded generator to the DataLoader. You won't get bit-identical results across different GPUs, but you'll remove the everyday run-to-run randomness that makes debugging maddening.

Part III — Optimization

Turning gradients into good steps.

Autograd hands you a gradient; the optimizer decides what to do with it. The plain "subtract learning-rate times gradient" of Volume I is the floor, not the ceiling. Momentum smooths the path, adaptive methods tune a rate per parameter, schedules change the step over time, and clipping keeps a bad batch from blowing up the run. This part is how modern models actually converge.

09SGD & momentum

The "stochastic" in stochastic gradient descent refers to the mini-batch: each step estimates the true gradient (over all data) from just one batch. That estimate is noisy — and the noise is a feature, not a bug. It jostles the parameters out of shallow bad spots and acts as a mild regularizer. But pure SGD also zig-zags across narrow valleys and crawls along gentle slopes.

Momentum fixes both. Instead of stepping by the raw gradient, you accumulate a running "velocity" — an exponential average of past gradients — and step by that. Consistent directions build up speed; oscillating directions cancel out. The physical picture is a heavy ball rolling downhill: it powers through small bumps and doesn't rattle side to side.

import torch

optimizer = torch.optim.SGD(
    model.parameters(),
    lr=0.1,
    momentum=0.9,        # keep 90% of last step's velocity -> smooth, faster
    nesterov=True,       # a "look-ahead" variant, usually a small free win
    weight_decay=5e-4,   # L2 regularization (Chapter 10 explains the subtlety)
)

The velocity update is, conceptually, v ← momentum·v + gradient, then param ← param − lr·v. A momentum of 0.9 (the near-universal default) means the velocity remembers roughly the last ten gradients. SGD with momentum remains, to this day, a top choice for convolutional networks — often reaching a slightly better final accuracy than adaptive methods, at the cost of needing its learning rate and schedule tuned more carefully.

Why noise helps

A perfectly computed full-dataset gradient would march straight into the nearest minimum — which might be sharp and generalize poorly. Mini-batch noise nudges the optimizer toward flatter, wider minima that tend to generalize better. This is part of why very large batches (smoother gradients) sometimes need extra care to match small-batch generalization.

10Adam, AdamW & weight decay

SGD uses one learning rate for every parameter. Adam adapts a rate per parameter from the running statistics of its gradients: it keeps an exponential average of the gradient (like momentum) and of the gradient's square (a measure of how noisy/large that parameter's gradients are), and divides the step by the latter. Parameters with big, noisy gradients get smaller steps; quiet ones get larger steps. The upshot is an optimizer that works reasonably out of the box across a huge range of problems.

optimizer = torch.optim.AdamW(
    model.parameters(),
    lr=3e-4,              # Adam's "sweet spot" is ~1e-3; 3e-4 is a safe default
    betas=(0.9, 0.999),   # decay rates for the gradient / gradient-squared avgs
    eps=1e-8,             # numerical floor to avoid divide-by-zero
    weight_decay=0.01,    # decoupled L2 (see below) -- the reason it's AdamW
)

Weight decay, and why it's AdamW not Adam

Weight decay gently pulls every parameter toward zero each step, discouraging the model from relying on any one large weight — a form of regularization that improves generalization. The subtlety: the "obvious" way to add it (fold an L2 penalty into the loss) interacts badly with Adam's per-parameter scaling, effectively decaying big-gradient weights less. AdamW "decouples" weight decay — applying it directly to the parameters, separately from the adaptive gradient step — which is the mathematically correct version and now the default choice everywhere.

The practical default

Starting a new problem, reach for AdamW at lr=3e-4. It converges reliably with little fuss. Once things work, if you're training a CNN and chasing the last point of accuracy, try SGD+momentum with a cosine schedule — it may edge ahead. Don't agonize over the optimizer early; the learning rate matters far more.

11The learning rate: schedules & warmup

A fixed learning rate is a compromise: large enough to make progress early is too large to settle precisely late. So we schedule it — start higher to cover ground, then decay toward zero to fine-tune. The most common modern choice is cosine decay, often with a short warmup that ramps the rate up from near-zero over the first few hundred steps. Warmup matters because a fresh model with random weights produces wild gradients; hitting it with a full learning rate immediately can destabilize or diverge it, especially for transformers and large batches.

LEARNING RATE OVER TRAINING (warmup + cosine decay): lr │ ___ │ / ‾‾‾--__ │ / ‾‾--__ │ / ‾‾--___ │ / ‾‾‾----____ └─┴────────┬──────────────────────────────────► step warmup peak cosine decay to ~0 (ramp up) (long, smooth wind-down)
import math
from torch.optim.lr_scheduler import LambdaLR

optimizer = torch.optim.AdamW(model.parameters(), lr=3e-4)
warmup, total = 500, 10_000          # steps

def lr_factor(step):
    if step < warmup:                # linear ramp 0 -> 1
        return step / warmup
    # cosine from 1 -> 0 over the remaining steps
    progress = (step - warmup) / (total - warmup)
    return 0.5 * (1 + math.cos(math.pi * progress))

scheduler = LambdaLR(optimizer, lr_factor)

for step in range(total):
    # ... forward / loss / zero_grad / backward ...
    optimizer.step()
    scheduler.step()                 # advance the LR schedule EACH step

Note the ordering: optimizer.step() then scheduler.step(). And decide up front whether your schedule advances per step (as here — the modern norm for large models) or per epoch (common with StepLR / CosineAnnealingLR on smaller vision jobs); mixing them up silently ruins the schedule.

Finding a learning rate fast

Don't guess blindly. The "LR range test" trains for a few hundred steps while exponentially increasing the LR and plots loss against it: the good range is the steepest downhill stretch, and a rule of thumb is to pick roughly one-tenth of the LR where the loss bottomed out. Even cruder but effective: try 1e-3, 3e-4, 1e-4 and keep whichever trains fastest without diverging.

12Batch size, accumulation & clipping

Batch size trades three things off at once: memory (bigger batch, more GPU RAM), gradient quality (bigger batch, less noisy estimate), and speed (bigger batch, better hardware utilization — to a point). When you change it, the learning rate usually needs to move with it: the linear scaling rule says if you double the batch, roughly double the learning rate (with warmup to keep the now-larger early steps stable).

Gradient accumulation: a big batch on a small GPU

Want the smoother gradient of a batch of 256 but only have memory for 64? Run four batches of 64, summing their gradients, and step once. Because gradients accumulate by default (Volume I, Chapter 11), this is almost free — just don't zero them until after the fourth.

accum_steps = 4                        # effective batch = batch_size x 4
optimizer.zero_grad()
for i, (xb, yb) in enumerate(loader):
    loss = loss_fn(model(xb), yb)
    (loss / accum_steps).backward()    # scale so the sum matches a mean
    if (i + 1) % accum_steps == 0:
        optimizer.step()               # step once per 4 micro-batches
        optimizer.zero_grad()

Gradient clipping: a seatbelt for the update

Occasionally one bad batch produces an enormous gradient that, applied, blows the model to NaN in a single step — endemic to RNNs (Volume III) and large transformers (Volume V). Gradient clipping caps the total gradient size before the update: if the combined norm exceeds a threshold, scale it back down. It costs nothing and prevents a whole class of training explosions.

loss.backward()
# clip BEFORE stepping: rescales all grads so their global norm <= 1.0
torch.nn.utils.clip_grad_norm_(model.parameters(), max_norm=1.0)
optimizer.step()

The learning rate dominates; everything else is refinement. If you tune one thing, tune the learning rate and its schedule — they decide whether training converges at all and how well. Optimizer choice, batch size, and clipping matter, but a right-sized, well-scheduled learning rate is worth more than any of them. When a run misbehaves, look there first.

Autograd finds the direction; the optimizer, schedule, and clip decide how bravely to walk it.

Part IV — Generalization & the real loop

Doing well on data you've never seen.

Fitting the training set is easy; a big enough model can memorize it outright. The real goal is generalization — performing well on new data — and everything in this part serves it: diagnosing over- and underfitting, the regularizers that close the gap, the normalization layers that make deep training stable, and finally a production loop that ties Volume II together and won't lose your work.

13Over/underfitting & capacity

A model has capacity — roughly, how flexible it is, driven by its size and architecture. Too little capacity and it can't even fit the training data (underfitting); too much, and it fits the training data too well, memorizing noise and quirks that don't transfer (overfitting). The art is landing in between, and you diagnose where you are by watching training loss and validation loss together.

READING THE TWO CURVES: loss│ loss│ │╲ │╲ │ ╲___ val │ ╲ ___---- val ← rising = memorizing │ ╲__ train GOOD FIT │ ╲___--- OVERFIT │ ‾‾‾‾‾‾‾‾ (small gap) │ ‾‾‾___ train (→0) └──────────────► epoch └──────────────► epoch both low, close together train falls, val turns UP → stop / regularize

train low, val much higherOverfitting. Add regularization (Chapter 14), get more/augmented data, or reduce capacity.
train low, val low & closeGood fit. Ship it, or push capacity up while watching the gap.
train won't drop at allNot a capacity issue — a bug. Check the learning rate, data, and loss (Chapter 16's decision tree).

The distance between the curves is the generalization gap. A little gap is normal and fine. A gap that widens as training continues — validation loss turning upward while training loss keeps falling — is the signature of overfitting, and the cue to regularize or stop.

14Regularization that works

Regularization is any technique that trades a little training-set fit for better generalization. You have a toolbox; these are the ones that earn their keep, roughly in the order to reach for them.

  • More (or augmented) data. The most reliable regularizer of all. If you can't get more, augmentation (Chapter 06) manufactures variety for free. Always try this first.
  • Weight decay. Already in your AdamW (Chapter 10). Penalizes large weights, favoring simpler functions. A default of 0.01–0.1 for transformers, 1e-4–5e-4 for CNNs.
  • Dropout. During training, randomly zero a fraction of activations each step, so the network can't lean on any single unit and must learn redundant, robust features. At evaluation it's turned off automatically (this is why model.eval() matters).
  • Early stopping. Keep the checkpoint with the best validation score and stop when it hasn't improved for a while. Free, and it directly targets the gap.
  • Label smoothing. Soften one-hot targets slightly (e.g. 0.9 for the true class instead of 1.0), which curbs overconfidence and often nudges accuracy up. Built into CrossEntropyLoss(label_smoothing=0.1).
import torch.nn as nn

# Dropout is just another layer; it self-disables under model.eval().
net = nn.Sequential(
    nn.Linear(784, 256), nn.ReLU(),
    nn.Dropout(0.3),                # zero 30% of activations while training
    nn.Linear(256, 10),
)

# Label smoothing is a one-argument change to the loss.
loss_fn = nn.CrossEntropyLoss(label_smoothing=0.1)
Dropout and BatchNorm make train()/eval() mandatory

These layers behave differently in the two modes. Forget model.eval() before validating and dropout keeps randomly zeroing activations while batch-norm uses batch statistics — your metrics turn noisy and pessimistic. Forget model.train() when you resume training and the model stops regularizing. Set the mode explicitly at the top of every train and eval phase.

15Normalization: BatchNorm & LayerNorm

Deep networks train poorly when the scale of activations drifts from layer to layer — gradients vanish or explode, and you're forced to use a tiny learning rate. Normalization layers fix this by re-centering and re-scaling activations to a standard distribution partway through the network, then letting the model learn a per-feature scale and shift back. The payoff is faster, more stable training and tolerance for higher learning rates. Two variants dominate, differing only in which numbers they average over.

BatchNorm — normalize each feature across the batch

nn.BatchNorm standardizes each feature using the mean and variance of the current batch. It's a staple of convolutional networks and often lets you train them noticeably faster. Its quirk: because it depends on batch statistics, it behaves differently at evaluation, where there's no meaningful "batch" — so it keeps a running average of the statistics during training and uses those frozen numbers in eval() mode. That's the buffer state from Chapter 01, and the reason batch-norm is sensitive to very small batch sizes.

LayerNorm — normalize each example across its features

nn.LayerNorm standardizes across the features of a single example, so it doesn't depend on the batch at all — identical behavior in training and eval, and unbothered by batch size. That independence is why it's the normalization of choice in transformers and sequence models (Volumes III and V), where batch composition varies and sequence lengths differ.

import torch.nn as nn

# BatchNorm for a conv/MLP feature map (num_features = channels/width):
bn_block = nn.Sequential(nn.Linear(256, 256), nn.BatchNorm1d(256), nn.ReLU())

# LayerNorm for a transformer-style feature vector (normalized_shape = width):
ln_block = nn.LayerNorm(512)   # normalizes the last dim of each token vector

16A production training loop

Here is Volume II assembled: device handling, the schedule and clipping from Part III, evaluation with an accuracy metric, checkpointing so a crash doesn't cost you the run, and early stopping. This is the template to copy for real projects.

import torch

def train(model, train_dl, val_dl, epochs=50, patience=5, device="cuda"):
    model.to(device)
    opt = torch.optim.AdamW(model.parameters(), lr=3e-4, weight_decay=0.01)
    sched = torch.optim.lr_scheduler.CosineAnnealingLR(opt, T_max=epochs)
    loss_fn = torch.nn.CrossEntropyLoss(label_smoothing=0.1)
    best_val, waited = float("inf"), 0

    for epoch in range(epochs):
        model.train()                                  # regularizers ON
        for xb, yb in train_dl:
            xb, yb = xb.to(device), yb.to(device)
            loss = loss_fn(model(xb), yb)
            opt.zero_grad()
            loss.backward()
            torch.nn.utils.clip_grad_norm_(model.parameters(), 1.0)
            opt.step()
        sched.step()                                   # per-epoch schedule

        # ---- validation ----
        model.eval()                                   # regularizers OFF
        val_loss, correct, total = 0.0, 0, 0
        with torch.no_grad():
            for xb, yb in val_dl:
                xb, yb = xb.to(device), yb.to(device)
                out = model(xb)
                val_loss += loss_fn(out, yb).item() * len(xb)
                correct += (out.argmax(1) == yb).sum().item()
                total += len(xb)
        val_loss /= total
        print(f"epoch {epoch}  val_loss {val_loss:.4f}  acc {correct/total:.3f}")

        # ---- checkpoint best + early stop ----
        if val_loss < best_val:
            best_val, waited = val_loss, 0
            torch.save({"epoch": epoch,
                        "model": model.state_dict(),
                        "opt": opt.state_dict()}, "best.pt")   # save STATE, not the object
        else:
            waited += 1
            if waited >= patience:
                print("early stop"); break
Save the state_dict, not the model object

Persist model.state_dict() (a plain dict of tensors), not the whole model via pickle — the latter breaks when your code moves or refactors. To resume training, save the optimizer and scheduler states and the epoch too; to just deploy, the model state_dict alone is enough. Load with model.load_state_dict(torch.load("best.pt")["model"]).

When it won't train: a triage path

Training is misbehaving — what do you see?
   │
   ├─ loss is NaN/inf → lower the LR; add grad clipping; check for
   │        bad inputs (NaNs, /0); confirm you're not softmaxing before CE loss
   │
   ├─ loss flat, never drops → LR too low or a wiring bug: is the
   │        optimizer holding the right params? are labels/inputs aligned?
   │        overfit a SINGLE batch first — a correct model can memorize it
   │
   ├─ train drops, val rises → overfitting → regularize (Ch.14),
   │        more data/augmentation, early stop
   │
   └─ both stuck high → underfitting → more capacity, train longer,
            higher LR, less regularization

Overfit one batch before you trust anything. The fastest sanity check in deep learning: take a single batch and train on it repeatedly. A correctly wired model will drive that loss to near zero within a few dozen steps. If it can't, no amount of data or tuning will help — you have a bug in the model, loss, or data plumbing, and you've just localized it cheaply.

A model that fits everything and generalizes to nothing has learned the answers, not the lesson.
