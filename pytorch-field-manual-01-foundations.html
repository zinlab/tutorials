Part I — The mental model

What you are actually building.

Before a single tensor, get the picture straight. A neural network is a function with adjustable knobs; training is the search for knob settings that make the function's outputs match your data. PyTorch exists to make that search fast, automatic, and expressible as ordinary Python. This part gives you the whole loop at a glance so the details in Parts II–IV have somewhere to land.

01What deep learning really is

Strip away the mystique and a neural network is a function with tunable numbers inside it. You feed it an input, it produces an output, and buried in the middle are thousands (or billions) of adjustable values called parameters. "Learning" is nothing more exotic than nudging those parameters until the function's outputs line up with examples you already have the answers to.

Take the smallest possible case: predicting a house's price from its size. Guess that price is roughly w · size + b for some slope w and offset b. Those two numbers are the parameters. If you have a spreadsheet of past sales, learning means choosing w and b so the line passes as close as possible through your data. That is the entire idea — deep learning just does it with far more parameters and far more flexible functions.

Why "deep"

A straight line can only model straight-line relationships. Stack many simple functions on top of each other — each one a linear step followed by a small nonlinear bend — and the combination can bend and fold to approximate almost any relationship at all. Each layer feeds the next; "deep" just means "many layers." The nonlinear bend between layers is what makes depth worth having: without it, a stack of linear layers collapses back into a single line.

THE LEARNING LOOP, IN ONE PICTURE: data ──► model(parameters) ──► prediction │ ▼ loss = how wrong? ◄── true answer │ adjust parameters ◄── gradients (which way is "less wrong"?) │ └──────► repeat, thousands of times

Three ingredients drive that loop, and they map one-to-one onto the three ideas in PyTorch you'll meet next:

IngredientWhat it does
the modelA parameterized function: turns inputs into predictions. In PyTorch, built from tensors and nn modules.
the lossA single number measuring how wrong the predictions are on your data. Smaller is better.
the optimizerThe rule that changes parameters to shrink the loss, using gradients — computed for you by autograd.

Crucially, you never hand-write the rules. You don't tell the model "big houses cost more." You give it examples and a way to measure error, and the optimization discovers the relationship by adjusting parameters. That inversion — from programming rules to fitting parameters — is the whole conceptual shift of machine learning.

Learning is optimization. A model is a function with parameters; a loss scores its output against reality; training repeatedly adjusts the parameters in the direction that lowers the loss. Every technique in this entire series — every architecture, optimizer, and trick — is a variation on making that one loop work better.

02PyTorch's three ideas — and getting set up

PyTorch gives you exactly what the learning loop needs, in three layers that stack cleanly:

  • Tensors — n-dimensional arrays of numbers that can live on a CPU or a GPU and, optionally, remember how they were computed. This is your data and your parameters. (Part II.)
  • Autograd — an engine that watches the math you do on tensors and can then compute every derivative you need, automatically. This is what makes gradients free. (Part III.)
  • nn & optim — ready-made building blocks (layers, losses) and optimizers, so you assemble models like Lego instead of wiring math by hand. (Part IV and Volume II.)

The defining trait is that PyTorch is "define-by-run" (eager): there is no separate compile step or graph to declare up front. You write ordinary Python — loops, ifs, prints — and PyTorch executes it line by line, building its record of the computation as it goes. That is why PyTorch feels like NumPy that happens to do calculus and run on GPUs, and why debugging it is just debugging Python.

Installing it

Install the build that matches your hardware. On a machine with an NVIDIA GPU you want a CUDA build; on a laptop, the CPU build (or, on Apple Silicon, the default build with the "MPS" backend) is fine. The canonical command comes from the selector on pytorch.org; a CUDA 12.4 example looks like this:

# CPU-only (works everywhere):
pip install torch torchvision torchaudio

# NVIDIA GPU, CUDA 12.4 build (pick the URL from pytorch.org for your CUDA):
pip install torch torchvision torchaudio --index-url https://download.pytorch.org/whl/cu124
Match the build to your driver

The single most common setup failure is a CUDA build that doesn't match the installed GPU driver, which shows up later as torch.cuda.is_available() returning False. Check your driver first, then pick the matching CUDA version. When in doubt, the CPU build always runs — just slowly.

Verifying and choosing a device

Every PyTorch program starts the same way: import, check what hardware you have, and pick a device to put tensors on. Memorize this idiom — you'll type it a thousand times.

import torch

print(torch.__version__)          # e.g. 2.13.0+cu124
print(torch.cuda.is_available())  # True if a usable NVIDIA GPU is present

# Pick the best available device, in order of preference.
if torch.cuda.is_available():
    device = torch.device("cuda")          # NVIDIA GPU
elif torch.backends.mps.is_available():
    device = torch.device("mps")           # Apple Silicon GPU
else:
    device = torch.device("cpu")           # always works

print("Using:", device)

A device is just a label saying where a tensor's numbers physically live. CPUs are flexible and always available; GPUs are massively parallel and can run the matrix math of deep learning tens to hundreds of times faster, which is why almost all real training happens on them. You move data and models onto a device with .to(device), and the rule is simple: everything that touches in a computation must be on the same device. We'll return to the mechanics in Chapter 08.

You don't need a GPU to start

Every example in this volume runs on a CPU in seconds. Learn the ideas on your laptop; rent a GPU only when your models get big enough to need one. Free hosted notebooks (Colab, Kaggle) give you a GPU with PyTorch pre-installed if you want to try one.

03A whole training loop in fifteen lines

Here is a complete, runnable program that learns. It manufactures noisy data from the hidden rule y = 2x + 1, then trains a one-parameter-pair model to rediscover that rule from the data alone. Read it once for the shape; the line-by-line follows.

import torch

# 1. Data: 100 points from y = 2x + 1, plus a little noise.
X = torch.randn(100, 1)                 # 100 inputs, shape (100, 1)
y = 2 * X + 1 + 0.1 * torch.randn(100, 1)   # true rule + noise

# 2. Model: a single linear layer, 1 input -> 1 output. Its parameters
#    (a weight and a bias) start random; training will fit them.
model = torch.nn.Linear(1, 1)

# 3. Loss + optimizer: mean squared error, and plain gradient descent.
loss_fn = torch.nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)

# 4. The loop: do this many times.
for epoch in range(200):
    pred = model(X)               # forward: predictions from current params
    loss = loss_fn(pred, y)       # how wrong are we, as one number?

    optimizer.zero_grad()         # clear old gradients
    loss.backward()               # autograd: compute d(loss)/d(each param)
    optimizer.step()              # nudge every param downhill

    if epoch % 50 == 0:
        print(f"epoch {epoch:3d}  loss {loss.item():.4f}")

# 5. Inspect what it learned (should be close to weight=2, bias=1).
w, b = model.weight.item(), model.bias.item()
print(f"learned:  y = {w:.2f} x + {b:.2f}")

Run it and the loss falls from something large toward ~0.01, and the learned weight and bias land near 2 and 1. The model recovered the rule it was never told. Now the walk-through, because these five steps are the spine of every PyTorch program you will ever write:

LineWhat's happening
Linear(1,1)Creates the model. Behind it are two parameters — weight and bias — initialized randomly. Calling model(X) computes X · weight + bias.
MSELossMean squared error: average of (pred − y)². One number; zero means perfect. This is our definition of "wrong."
SGD(lr=0.1)The optimizer. It holds references to the model's parameters and knows how to change them. lr (learning rate) is the step size.
forwardpred = model(X) pushes data through the model. Autograd quietly records every operation.
zero_gradGradients accumulate by default, so you must clear last step's gradients before computing this step's. Forget this and training silently breaks — see Chapter 12.
backwardThe magic line. Autograd walks the recorded computation backward and fills in d(loss)/d(param) for every parameter. (Part III explains exactly how.)
stepThe optimizer uses those gradients to move each parameter a little in the direction that lowers the loss.
The order is not negotiable

Forward → loss → zero_grad → backward → step. Call step before backward and you update with stale gradients; forget zero_grad and gradients from every past iteration pile up. Both produce a model that "runs" but never learns — the most demoralizing bug for a beginner because nothing errors.

Five lines — forward, loss, zero, backward, step — are the heartbeat of PyTorch. Everything else is elaboration.

04The workflow you'll repeat forever

Chapter 03 was a whole project in miniature. Every serious training run is the same skeleton with more muscle on each bone. Internalize the checklist and new problems stop being mysterious — they become "which box needs work?"

Building any model? Walk the six boxes:
   │
   ├─ 1. data      → get inputs and targets into tensors of known shape
   ├─ 2. model     → a function from input to prediction (nn.Module)
   ├─ 3. loss      → one number scoring prediction against target
   ├─ 4. optimizer → the rule that updates parameters
   ├─ 5. loop      → forward, loss, zero_grad, backward, step
   └─ 6. evaluate  → measure on held-out data you didn't train on

The habit that saves you: print shapes

The overwhelming majority of beginner bugs are shape bugs — a tensor is (100, 1) when the next step expected (100,), and PyTorch either errors cryptically or, worse, "broadcasts" the mismatch into a silently wrong result. The fix is a reflex: print the shape whenever you're unsure.

print(X.shape)          # torch.Size([100, 1])
print(pred.shape, y.shape)   # both should match before a loss

Throughout this manual, shapes are written as comments like # (B, C, H, W) — batch, channels, height, width. Reading and predicting shapes is a real skill; treat those comments as the most important part of the code.

Two modes, and turning off gradients for inference

Models behave differently while training versus while being used, because some layers (dropout, batch-norm — Volume II) switch behavior. You signal intent with model.train() and model.eval(). And when you're only using a model, not training it, you wrap the call in torch.no_grad() so autograd doesn't waste time and memory recording a backward pass you'll never run.

model.eval()                     # switch to evaluation behavior
with torch.no_grad():            # don't track gradients here
    test_pred = model(X_test)    # pure inference, faster and lighter
Reproducibility: seed everything

Random initialization and shuffling make each run slightly different. When you need runs to match — for debugging or fair comparison — set the seeds at the top: torch.manual_seed(0). It won't make training deterministic to the last bit across GPUs (Volume IV covers full determinism), but it removes the everyday randomness.

That is the whole workflow. The rest of Volume I earns you the right to write each box fluently: Part II makes tensors second nature, Part III demystifies backward(), and Part IV rebuilds the training loop from first principles so that when you write those five lines, you know exactly what each one does and why.

Part II — Tensors

The one data structure everything rides on.

A tensor is an n-dimensional array of numbers — a scalar is a 0-D tensor, a vector 1-D, a matrix 2-D, and an image batch is a 4-D tensor. Your data is tensors, your parameters are tensors, your predictions are tensors. Master how they're shaped, stored, and combined and PyTorch stops surprising you. This part is the grammar; autograd and training are the sentences.

05Making tensors: shape, dtype, device

There are two families of ways to make a tensor: wrap existing numbers, or generate new ones. Wrapping uses torch.tensor(...); generating uses factory functions named for what they produce.

import torch

# From existing Python data:
a = torch.tensor([1.0, 2.0, 3.0])          # 1-D, three elements
b = torch.tensor([[1, 2], [3, 4]])         # 2-D, shape (2, 2)

# Generated:
z = torch.zeros(2, 3)        # (2,3) of 0.0
o = torch.ones(2, 3)         # (2,3) of 1.0
r = torch.randn(2, 3)        # (2,3) standard-normal random
u = torch.rand(2, 3)         # (2,3) uniform in [0, 1)
n = torch.arange(0, 10, 2)   # [0, 2, 4, 6, 8]
l = torch.linspace(0, 1, 5)  # 5 points evenly from 0 to 1
e = torch.eye(3)             # 3x3 identity

Shape, dtype, device — a tensor's three facts

Every tensor answers three questions: what shape is it, what type are its numbers, and where do they live. You will query these constantly.

x = torch.randn(4, 3)
print(x.shape)     # torch.Size([4, 3])  -> rows, cols
print(x.ndim)      # 2                    -> number of dimensions
print(x.numel())   # 12                   -> total elements
print(x.dtype)     # torch.float32        -> the number type
print(x.device)    # cpu                  -> where it lives

dtype is the numeric type. The default for decimals is float32 — the workhorse of deep learning, a good balance of range and speed. Integers default to int64. You'll mostly care about float32 (activations and parameters), int64 (class labels and indices), bool (masks), and later float16/bfloat16 (mixed-precision training, Volume IV).

i = torch.tensor([1, 2, 3])              # int64 by default
f = torch.tensor([1, 2, 3], dtype=torch.float32)   # force float
g = i.float()                            # convert: int64 -> float32
h = f.to(torch.float16)                  # convert dtype with .to()
ConstructorUse it for
torch.tensor(data)Wrapping known values from lists/NumPy. Infers dtype (watch: ints stay int).
zeros / ones / fullBuffers you'll fill in, or constant-initialized tensors.
randn / randRandom init and test data. randn is Gaussian, rand is uniform.
arange / linspaceRegular sequences — axes, time steps, grids.
zeros_like(t)A new tensor matching t's shape, dtype, and device. The safe way to make companions.
The float-vs-int trap

torch.tensor([1, 2, 3]) is int64, and most neural-network math needs floats. Passing integer tensors into a model that expects floats is a frequent first bug. When in doubt for numeric data, be explicit: dtype=torch.float32.

06The memory model: storage, strides & views

To predict when PyTorch copies data and when it doesn't — which drives both speed and a whole class of bugs — you need the picture underneath. A tensor is not the numbers; it is a small bundle of metadata (shape, strides, dtype, an offset) pointing into a flat, one-dimensional block of memory called the storage.

ONE STORAGE, VIEWED AS A (2 x 3) TENSOR: storage (flat): [ 10 11 12 13 14 15 ] index: 0 1 2 3 4 5 │ │ shape (2,3) └─ row 0 ──────┴─ row 1 ──► strides = (3, 1) "move 3 to change row, tensor[1, 2] = storage[ 1*3 + 2*1 ] = 15 1 to change column"

The stride of a dimension is how many storage steps you take to move one index along that dimension. This indirection is powerful: many operations just hand you a new metadata bundle over the same storage — a view — with zero copying.

x = torch.arange(6)          # [0,1,2,3,4,5], storage of 6 ints
y = x.reshape(2, 3)          # a VIEW: same storage, new shape
y[0, 0] = 99                 # writing through the view...
print(x[0])                  # ...changes x too!  -> 99
print(y.storage().data_ptr() == x.storage().data_ptr())  # True: shared

Slicing, reshape, view, transpose, permute, and unsqueeze all return views when they can. That's great for performance — you reshape a million-element tensor for free — but it means a view aliases its parent: modify one, you modify the other. When you truly need an independent copy, call .clone().

Contiguity: the catch

Because a view only reshuffles metadata, some views end up describing memory in a non-sequential order. After a transpose, walking the tensor row-by-row jumps around in storage — the tensor is non-contiguous. Most operations handle this fine, but .view() insists on contiguous memory and will raise an error; .reshape() is the forgiving version that copies only if it must.

worked case "view size is not compatible with input tensor's size and stride"
The setup

You transpose a tensor, then try to flatten it with .view(), and PyTorch throws the error above.

x = torch.randn(3, 4)
xt = x.t()                 # transpose -> shape (4, 3), now NON-contiguous
# xt.view(12)              # RuntimeError: view needs contiguous memory
The investigation

xt.is_contiguous() returns False. The transpose only swapped the strides; the underlying storage is still in the original order, so a plain view can't reinterpret it as a flat run.

The fix

Either make it contiguous first, or use reshape, which does that for you when needed:

flat = xt.contiguous().view(12)   # explicit: copy into contiguous order
flat = xt.reshape(12)             # idiomatic: reshape handles it
The lesson

Reach for reshape by default. Use view only when you want the guarantee of no copy, and remember transpose/permute break contiguity.

07Indexing, reshaping & broadcasting

Indexing works like NumPy: integers select, slices take ranges, and you can index each dimension independently.

x = torch.arange(12).reshape(3, 4)
x[0]           # first row -> shape (4,)
x[:, 1]        # second column -> shape (3,)
x[1, 2]        # single element -> a 0-D tensor
x[0:2, 1:3]    # a sub-block -> shape (2, 2)
x[x > 5]       # boolean mask -> 1-D tensor of the matching elements

Reshaping and the size-juggling verbs

Four verbs cover almost all reshaping. reshape changes the shape; squeeze removes size-1 dimensions; unsqueeze inserts one; and a -1 in any reshape means "infer this dimension."

v = torch.randn(10)
v.reshape(2, 5)     # (10,)   -> (2, 5)
v.reshape(-1, 5)    # infer the 2 -> (2, 5)
v.unsqueeze(0)      # (10,)   -> (1, 10)   add a batch dimension at front
v.unsqueeze(1)      # (10,)   -> (10, 1)   make it a column
img = torch.randn(1, 1, 28, 28)
img.squeeze()       # (1,1,28,28) -> (28, 28)   drop all size-1 dims

Broadcasting: the rule that saves and bites

Broadcasting lets tensors of different shapes combine without you writing loops or manually copying data. PyTorch compares shapes from the right; two dimensions are compatible if they're equal or one of them is 1, and a size-1 (or missing) dimension is virtually stretched to match.

BROADCASTING (3,1) WITH (1,4) -> (3,4): (3,1) (1,4) result (3,4) ┌───┐ ┌───┬───┬───┬───┐ each cell = row_val + col_val │ a │ │ p q r s │ ┌───────────────┐ │ b │ + └───────────────┘ = │ a+p a+q a+r a+s│ │ c │ │ b+p b+q b+r b+s│ └───┘ stretched down │ c+p c+q c+r c+s│ stretched right └───────────────┘
col = torch.tensor([[1.], [2.], [3.]])   # (3, 1)
row = torch.tensor([[10., 20., 30., 40.]])   # (1, 4)
col + row      # -> (3, 4) via broadcasting, no loops

# Everyday use: subtract a per-column mean from a data matrix.
data = torch.randn(100, 5)     # 100 samples, 5 features
mean = data.mean(dim=0)        # (5,)  -> one mean per feature
centered = data - mean         # (100,5) - (5,) broadcasts cleanly
The (N,1)-vs-(N,) silent bug

Broadcasting is so accommodating it will happily combine shapes you didn't mean to. Subtracting predictions of shape (N, 1) from targets of shape (N,) broadcasts to (N, N) — a hundred-fold-bigger tensor and a completely wrong loss, with no error raised. When a result is mysteriously the wrong size or the loss is nonsense, print shapes and look for a stray singleton dimension. Fix with squeeze/unsqueeze.

08Math, reductions & the NumPy bridge

Elementwise math is what it looks like, and it broadcasts. The one operator worth calling out is matrix multiplication, @ (or torch.matmul) — the single most important operation in deep learning, since every linear layer is a matmul.

a = torch.randn(3, 4)
b = torch.randn(4, 2)
a @ b            # matrix multiply -> (3, 4) x (4, 2) = (3, 2)
a * 2            # elementwise scale
a + b.t()        # broadcast/elementwise (shapes must be compatible)
a.exp(); a.sqrt(); a.abs()   # elementwise math, also as torch.exp(a) etc.

Reductions: collapsing a dimension

Reductions (sum, mean, max, min, std) collapse a tensor along a chosen dimension. The dim argument names which axis to collapse; keepdim=True leaves a size-1 placeholder so the result still broadcasts against the original.

x = torch.randn(100, 5)      # 100 rows, 5 columns
x.mean()                     # -> scalar: mean of all 500 numbers
x.mean(dim=0)                # -> (5,): mean down each column
x.mean(dim=1)                # -> (100,): mean across each row
x.sum(dim=1, keepdim=True)   # -> (100, 1): keep the axis for broadcasting

# max returns BOTH values and their indices along a dim:
vals, idx = x.max(dim=1)     # per-row maximum and where it was

Reading dim correctly is a common stumbling block. The rule: dim=k means "reduce along axis k, so axis k disappears." For a (rows, cols) matrix, dim=0 collapses rows and leaves one value per column; dim=1 collapses columns and leaves one per row.

In-place ops, and why to be wary

Operations with a trailing underscore modify the tensor in place instead of returning a new one: x.add_(1) versus x + 1. They save memory, but they overwrite data that autograd may still need for the backward pass, which can corrupt gradients. As a beginner, prefer the out-of-place versions; reach for in-place only when profiling says memory demands it (Volume IV).

The NumPy bridge

PyTorch and NumPy interoperate tightly, which matters because so much of the data world speaks NumPy. On the CPU they can even share memory, making conversion free — and occasionally surprising.

import numpy as np

arr = np.array([1.0, 2.0, 3.0])
t = torch.from_numpy(arr)     # SHARES memory with arr (no copy, CPU only)
back = t.numpy()              # SHARES memory the other way

arr[0] = 99.0
print(t[0])                   # -> 99.0 : they alias the same buffer!

# A GPU tensor must come home before it can become a NumPy array:
gpu = torch.randn(3, device="cpu")   # (pretend this is on cuda)
gpu.cpu().numpy()             # .cpu() first, or you'll get an error

Shapes are the language; know them cold. Most PyTorch debugging is shape debugging. If you can look at a line and predict the output shape — accounting for broadcasting, reductions, and views — you can read any model. Print shapes liberally, annotate them in comments, and the framework stops fighting you.

Part III — Autograd

The engine that computes every derivative.

Training needs gradients — for every parameter, the answer to "which way do I nudge you to lower the loss?" Computing those by hand for a million parameters is impossible; autograd does it automatically by recording your math and replaying it backward. Understand this part and loss.backward() stops being magic and becomes a tool you can reason about, debug, and control.

09Gradients, with only the calculus you need

You need exactly one idea from calculus: the derivative is a slope — it tells you how much the output changes when you nudge the input a tiny bit. If a function's slope at some point is +6, then increasing the input increases the output about six times as fast; to decrease the output, you move the input the other way.

That is the whole basis of training. The loss is a function of the parameters. Its gradient is just the collection of slopes — one partial derivative per parameter — bundled into a vector. The gradient points in the direction that increases the loss fastest, so to reduce the loss we step in the opposite direction. That single move, repeated, is gradient descent.

WHY THE SLOPE TELLS YOU WHICH WAY TO GO: f(x) = x² loss │ · · │ · · │ · slope −ve │ · slope +ve │ · (go right) │ · (go left) │ · │· │ · · · · · ← minimum, slope = 0 └────────────────────────────────────► x At x = 3 the slope is +6, so decreasing x lowers f. Step: x ← x − (learning_rate × slope).

The last piece is the chain rule: the derivative of nested functions is the product of the derivatives of each step. If loss depends on b which depends on a which depends on a parameter w, then d(loss)/dw is d(loss)/db · db/da · da/dw. Neural networks are deeply nested functions, so the chain rule is the mechanism by which a change deep inside the network is traced out to its effect on the loss. Autograd is, at heart, an automatic, exact chain-rule machine.

You will essentially never differentiate by hand. But this mental model — loss is a landscape, the gradient is the local uphill direction, we walk downhill — is the intuition every training decision draws on.

10requires_grad and the computational graph

Autograd only tracks tensors you ask it to. Set requires_grad=True on a tensor and PyTorch starts recording every operation that involves it, building a computational graph — a record of what was computed from what. That graph is exactly what backward() later walks in reverse.

import torch

x = torch.tensor(3.0, requires_grad=True)   # a parameter we can differentiate
y = x ** 2                                   # y = x², autograd records this
print(y)              # tensor(9., grad_fn=<PowBackward0>)
print(y.grad_fn)      # <PowBackward0> : the op that made y, for the backward pass

y.backward()          # compute dy/dx and store it in x.grad
print(x.grad)         # tensor(6.) — matches d(x²)/dx = 2x = 2·3

Notice y carries a grad_fn. Every tensor produced from a tracked tensor remembers the operation that created it. Chain those together and you have the graph: a directed record from your inputs, through each operation, to the final output. When you call .backward() on that output, autograd uses the stored grad_fns to apply the chain rule step by step, depositing the result in each leaf tensor's .grad.

Leaves, and where gradients land

A leaf tensor is one you created directly (not computed from others) that has requires_grad=True — your parameters. These are the tensors autograd fills .grad for, because they're what you'll update. Intermediate results (like y above, when it's not the final output) don't retain a .grad by default; they only pass gradient through.

# A two-parameter example, closer to a real model.
w = torch.tensor(2.0, requires_grad=True)   # leaf / parameter
b = torch.tensor(1.0, requires_grad=True)   # leaf / parameter
x = torch.tensor(4.0)                        # data: no grad needed

pred = w * x + b        # graph: mul -> add
loss = (pred - 10.0) ** 2   # pred=9, loss=(9-10)² = 1

loss.backward()         # fills w.grad and b.grad via the chain rule
print(w.grad, b.grad)   # d(loss)/dw = -8 , d(loss)/db = -2

Those gradients say: to lower this loss, increasing w helps eight times as much per unit as increasing b (both have negative slope, so both should increase). An optimizer will act on exactly this information. In a real model w and b are inside nn.Linear, which sets requires_grad=True for you — but the machinery is identical to what you just watched.

Data doesn't need gradients

Only mark parameters with requires_grad=True. Your inputs and targets are fixed; you never adjust them, so leaving them untracked saves memory and makes the graph smaller. nn layers handle this distinction automatically.

11Reverse-mode autodiff: what backward() does

The forward pass runs your operations and, along the way, saves the intermediate values it will need. The backward pass then starts at the output with a seed gradient of 1 and walks the graph in reverse, and at each node multiplies by that operation's local derivative — the chain rule, applied mechanically. Because it flows from the single output backward to all inputs, one backward pass computes the gradient for every parameter at once.

FORWARD BUILDS THE GRAPH, BACKWARD TRAVERSES IT: forward ► w ─┐ ├─(×)─► h ─┐ x ─┘ ├─(+)─► pred ─►(−t)²─► loss b ─────────┘ backward ◄ d loss/dw ◄──── d loss/dh ◄──── d loss/dpred ◄─── 1 (each arrow multiplies by that step's local slope)

This direction — output-to-input — is called reverse-mode automatic differentiation, and it is why deep learning is feasible. Networks have one loss (one output) and millions of parameters (many inputs); reverse mode gets all those millions of derivatives in a single traversal, at roughly the cost of one extra forward pass. Doing it forward-mode, one parameter at a time, would take millions of passes.

Why gradients accumulate

Here is a design choice that trips up everyone once. backward() does not overwrite .grad — it adds to it. Call backward() twice without clearing, and the gradients sum together.

x = torch.tensor(3.0, requires_grad=True)

(x ** 2).backward()
print(x.grad)     # 6.0
(x ** 2).backward()
print(x.grad)     # 12.0 — accumulated, not replaced!

x.grad.zero_()    # reset to zero
(x ** 2).backward()
print(x.grad)     # 6.0 again

This accumulation is deliberate and useful (it lets you split a big batch into pieces and sum their gradients — Volume II's gradient accumulation). But in the normal loop it means you must zero gradients each step, which is what optimizer.zero_grad() does. This is the single mechanical reason that line exists.

Non-scalar outputs

backward() with no argument only works when the output is a single number — which the loss always is, so in practice you rarely hit this. If you call it on a non-scalar tensor, autograd doesn't know how to weight each element and asks you for a gradient argument. The takeaway for now: always call backward() on a scalar loss.

12no_grad, detach & the gradient gotchas

Sometimes you want tensor math without autograd watching — during inference, or when manually updating parameters (the update itself must not be recorded, or you'd differentiate the optimizer). Two tools turn tracking off.

# 1. no_grad: a block where nothing is recorded. Faster, less memory.
with torch.no_grad():
    prediction = model(x)      # pure inference; no graph built

# 2. detach: get a copy of a tensor cut out of the graph.
frozen = some_tensor.detach()  # shares data, but stops gradient flow
value  = loss.detach().item()  # pull a Python float out for logging

You saw why no_grad matters for inference in Chapter 04. It's equally essential in the manual update you'll write next chapter: adjusting a parameter is bookkeeping, not part of the model's math, so it belongs inside no_grad.

The gotchas, collected

Almost every autograd error a beginner hits is one of these:

SymptomCause & fix
loss never dropsForgot zero_grad() — gradients accumulate — or forgot backward()/step() entirely. Check all five loop lines are present and ordered.
"backward through the graph a second time"You called backward() twice on the same graph. The graph is freed after one backward by default. Usually means you reused a loss tensor; recompute the forward pass, or pass retain_graph=True (rarely what you want).
"a leaf Variable that requires grad is being used in an in-place operation"You modified a parameter in place outside no_grad. Wrap manual updates in with torch.no_grad():.
"can't call numpy() on Tensor that requires grad"Detach first: t.detach().cpu().numpy().
grad is NoneThe tensor wasn't a leaf with requires_grad=True, or no backward reached it. Check it's actually a parameter.
Use .item() to log, not the tensor

When you accumulate a running loss for printing, add loss.item() (a plain Python float), not loss (a graph-attached tensor). Summing the tensors keeps the entire computation graph alive across iterations — a classic memory leak that slowly eats all your RAM.

backward() is not magic — it is the chain rule, run once from the loss back to every parameter.

Part IV — Learning by gradient descent

Assembling the whole loop.

You have the three pieces: tensors hold the numbers, autograd supplies the gradients, and a loss turns wrongness into something to minimize. Now we bolt them together — first by hand, so nothing is hidden, then the PyTorch way, so you see exactly what nn and optim automate. By the end, the five-line loop from Chapter 03 will hold no secrets.

13Loss: turning "wrong" into a number

Optimization needs a target to minimize, so we must compress "how wrong is the model right now?" into one differentiable number: the loss. Differentiable matters — autograd has to be able to take its gradient — which is why we use smooth formulas rather than, say, raw accuracy (which has no useful slope).

The right loss depends on the task. For predicting continuous numbers (regression), the default is mean squared error: average the squared differences. Squaring makes all errors positive and punishes large misses disproportionately, which pulls the model hard away from big mistakes.

pred = torch.tensor([2.5, 0.0, 2.0])
target = torch.tensor([3.0, -0.5, 2.0])

# By hand — MSE is literally the mean of the squared errors:
mse_manual = ((pred - target) ** 2).mean()      # tensor(0.1667)

# The built-in, which you'll actually use:
loss_fn = torch.nn.MSELoss()
mse_builtin = loss_fn(pred, target)             # tensor(0.1667) — identical

For classification (predicting a category), the standard loss is cross-entropy, which scores how much probability the model put on the correct class. It has its own subtleties — Volume II and III go deep — but the principle is the same: one differentiable number, smaller is better.

TaskUsual loss
regressionnn.MSELoss (squared error) or nn.L1Loss (absolute error, less sensitive to outliers).
binary classificationnn.BCEWithLogitsLoss — binary cross-entropy, numerically stable.
multi-class classificationnn.CrossEntropyLoss — expects raw scores (logits) and integer class labels.
The loss encodes your goal

Choosing a loss is choosing what "good" means. MSE says "big errors are much worse than small ones." Swap the loss and you change what the model optimizes for. Most of the time you'll pick the standard loss for your task — but know that it's a lever, not a law.

14Gradient descent from scratch

Here is training with nothing hidden — no nn, no optimizer. Just tensors, autograd, and a hand-written update. This is the version to truly understand; everything after it is convenience wrapped around this core.

import torch
torch.manual_seed(0)

# Data from the hidden rule y = 2x + 1.
X = torch.randn(100, 1)
y = 2 * X + 1 + 0.1 * torch.randn(100, 1)

# Parameters — the numbers we will learn. Start them at zero.
w = torch.zeros(1, 1, requires_grad=True)
b = torch.zeros(1, requires_grad=True)
lr = 0.1

for epoch in range(200):
    # forward: our current guess of y
    pred = X @ w + b                    # (100,1)@(1,1) + (1,) -> (100,1)
    loss = ((pred - y) ** 2).mean()     # MSE, by hand

    # backward: autograd fills w.grad and b.grad
    loss.backward()

    # update: step each parameter downhill. NOT tracked by autograd.
    with torch.no_grad():
        w -= lr * w.grad                # w ← w − lr · dloss/dw
        b -= lr * b.grad
        w.grad.zero_()                  # clear for the next iteration
        b.grad.zero_()

print(f"learned  w={w.item():.3f}  b={b.item():.3f}")   # ≈ 2.00, 1.00

Every line of the canonical loop is visible here. loss.backward() computes the slopes. The with torch.no_grad() block is the "optimizer": it moves each parameter against its gradient by a step of size lr, then zeroes the gradients so they don't accumulate. That's it — that's what training is.

The learning rate: the one knob you'll always tune

lr scales every step. It is the most important hyperparameter in all of deep learning, and getting it wrong is the most common reason training fails.

Too large

Steps overshoot the minimum and bounce out. The loss oscillates, explodes to inf, or becomes NaN. If your loss goes to NaN in the first few steps, lower the learning rate first.

Too small

Steps are safe but tiny; training crawls and may stall in a bad spot. Wastes time and compute. If loss decreases but agonizingly slowly, try raising it.

There's no universal right value — it depends on the model, data, and optimizer. Typical starting points are 0.1 for plain SGD and 1e-3 for Adam (Volume II). The practical method: try a few values spaced by factors of ten (1e-1, 1e-2, 1e-3) and watch the loss curve. Volume II turns this into a systematic recipe with schedules and warmup.

15The same model, the PyTorch way

Now watch the boilerplate vanish. nn.Linear holds the weight and bias for you (already tracked), nn.MSELoss is the loss, and optim.SGD is precisely the manual update block from Chapter 14 — packaged so it works for two parameters or two billion.

import torch
torch.manual_seed(0)

X = torch.randn(100, 1)
y = 2 * X + 1 + 0.1 * torch.randn(100, 1)

model = torch.nn.Linear(1, 1)                 # holds weight (1,1) and bias (1,)
loss_fn = torch.nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)

for epoch in range(200):
    pred = model(X)                # forward
    loss = loss_fn(pred, y)        # loss

    optimizer.zero_grad()          # = w.grad.zero_(); b.grad.zero_()
    loss.backward()                # = fill .grad for every parameter
    optimizer.step()               # = the no_grad update block, generalized

w, b = model.weight.item(), model.bias.item()
print(f"learned  w={w:.3f}  b={b:.3f}")        # ≈ 2.00, 1.00 — same result

Line for line, this is the manual version with three substitutions. Understanding that equivalence is the payoff of Part IV:

w, b = torch.zeros(...)nn.Linear(1, 1)Creating & tracking parameters, doing the matmul.

model.parameters() is the connective tissue: it yields every learnable tensor in the model, which is exactly what you hand the optimizer so it knows what to update. When you build hundred-layer networks in later volumes, this one call still gathers every parameter automatically — you never track them by hand again.

This is why nn exists

The manual version is perfect for one linear layer and hopeless for a real network. nn.Module lets you compose layers, and parameters() plus an optimizer scales the update to any size — while the five-line loop stays exactly the same. 16Anatomy of the canonical training loop

Real training adds two things to the loop: mini-batches (process a chunk of data at a time, not all of it at once) and a proper evaluation pass on held-out data. Here is the fully-formed loop, annotated. This template — with richer models and data — is what you'll write for the rest of the series.

import torch
from torch.utils.data import TensorDataset, DataLoader
torch.manual_seed(0)

# --- data: split into train and validation ---
X = torch.randn(1000, 1); y = 2 * X + 1 + 0.1 * torch.randn(1000, 1)
n_train = 800
train_ds = TensorDataset(X[:n_train], y[:n_train])
val_ds   = TensorDataset(X[n_train:], y[n_train:])
train_dl = DataLoader(train_ds, batch_size=32, shuffle=True)   # mini-batches
val_dl   = DataLoader(val_ds, batch_size=32)

# --- model / loss / optimizer ---
model = torch.nn.Linear(1, 1)
loss_fn = torch.nn.MSELoss()
optimizer = torch.optim.SGD(model.parameters(), lr=0.1)

# --- the loop: an EPOCH is one full pass over the training data ---
for epoch in range(10):
    model.train()                          # training mode
    for xb, yb in train_dl:                # one mini-batch at a time
        pred = model(xb)
        loss = loss_fn(pred, yb)
        optimizer.zero_grad()
        loss.backward()
        optimizer.step()

    # --- evaluate on data the model didn't train on ---
    model.eval()                           # evaluation mode
    val_loss = 0.0
    with torch.no_grad():                  # no gradients while evaluating
        for xb, yb in val_dl:
            val_loss += loss_fn(model(xb), yb).item() * len(xb)
    val_loss /= len(val_ds)
    print(f"epoch {epoch}  val_loss {val_loss:.4f}")

Two vocabulary words make the loop legible. A step (or iteration) is one mini-batch: one forward, one backward, one update. An epoch is one full sweep through the training set — many steps. You shuffle the training data each epoch (shuffle=True) so the model doesn't learn the order, and you never shuffle or train on the validation set — it exists solely to estimate how the model does on data it hasn't seen.

THE LOOP, NESTED: for each EPOCH (full pass over data): model.train() for each BATCH (a handful of samples): forward ─► loss ─► zero_grad ─► backward ─► step model.eval() with no_grad: measure loss on the held-out validation set │ ▼ watch this number — it's the honest one

The five lines never change; everything else is what you feed them. Forward, loss, zero_grad, backward, step — from this linear toy to a billion-parameter transformer, the loop is the same. The rest of your growth as a practitioner is learning richer models to put in the forward pass, better losses, smarter optimizers, and the judgment to read that validation number. That is precisely the road Volumes II through VI walk.

Learn the loop once, and every model you'll ever train is already half-familiar.
